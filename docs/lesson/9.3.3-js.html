Design a Hotel Booking System (Booking.com)

Intuition

Think of a hotel booking system like a physical reservation book at a front desk. The book has one page per room type per night. When someone books, the receptionist crosses off one room on each relevant page. If the last room is crossed off, that room type is full for that night. Two receptionists cannot cross off the same room simultaneously -- one must wait for the other to finish writing. The search page in a travel website is like a lobby display board that updates every minute -- it might briefly show a room as available after the last one was just booked, but the actual reservation book (database) is always correct.

Design a Hotel Booking System (Booking.com)

Introduction

A hotel booking system like Booking.com must handle property listings, room availability, reservation management, pricing, and guest reviews across millions of properties worldwide. The core technical challenge is managing room inventory -- ensuring that the same room is not double-booked while maintaining high read throughput for search queries. Unlike e-commerce where you can ship another unit, a hotel room on a specific night is a unique, perishable resource.

Step 1: Requirements Gathering

Functional Requirements

  • Search: Search hotels by location, dates, guests, filters (price, stars, amenities)
  • Availability: Show real-time room availability for date ranges
  • Booking: Reserve rooms with date range, guest count, and payment
  • Pricing: Dynamic pricing based on demand, season, day of week
  • Property Management: Hotel owners manage rooms, rates, availability, photos
  • Reviews: Guests can review hotels after stay
  • Cancellation: Support cancellation policies (free, partial refund, non-refundable)
  • Notifications: Booking confirmations, reminders, cancellation notices

Non-Functional Requirements

  • Availability: 99.99% uptime
  • Latency: Search results < 500ms, booking confirmation < 2s
  • Consistency: Strong consistency for bookings (no double booking), eventual for search
  • Scale: 1M+ properties, 50M+ listings, 5M bookings/day at peak

Step 2: Capacity Estimation

Assumptions:
- 1M properties, average 50 rooms each = 50M rooms
- 30M DAU searching, 5% conversion = 1.5M bookings/day
- Average search: 3 queries per session (refine dates, filters)
- Average booking: 2.5 nights stay

Traffic:
- Search queries: 30M * 3 = 90M/day = ~1,040 QPS
- Peak (3x): ~3,120 QPS
- Availability checks: 2x searches = ~2,080 QPS
- Bookings: 1.5M/day = ~17 TPS
- Peak bookings: ~50 TPS

Storage:
- Property data: 1M * 20KB = ~20GB
- Room types: 5M * 5KB = ~25GB
- Availability (365 days * 50M rooms): 18.25B rows * 50 bytes = ~900GB
- Bookings: 1.5M/day * 2KB * 365 = ~1TB/year
- Reviews: 100M * 1KB = ~100GB
- Images: 1M properties * 20 images * 500KB = ~10TB (CDN)

Step 4: API Design

Search API

import express from 'express';
const router = express.Router();

// GET /api/search?location=paris&checkin=2026-06-01&checkout=2026-06-04
//   &guests=2&minPrice=50&maxPrice=300&stars=4&amenities=wifi,pool&page=1
router.get('/search', async (req, res) => {
  const {
    location, checkin, checkout, guests,
    minPrice, maxPrice, stars, amenities,
    sort = 'relevance', page = 1, limit = 20,
  } = req.query;

  // Validate dates
  const checkinDate = new Date(checkin);
  const checkoutDate = new Date(checkout);
  if (checkinDate >= checkoutDate) {
    return res.status(400).json({ error: 'Checkout must be after checkin' });
  }

  const nights = Math.ceil((checkoutDate - checkinDate) / (1000 * 60 * 60 * 24));

  // Search Elasticsearch for matching properties
  const esQuery = {
    bool: {
      must: [
        { multi_match: { query: location, fields: ['city^3', 'country', 'name'] } },
      ],
      filter: [
        { range: { maxGuests: { gte: parseInt(guests) || 1 } } },
        ...(stars ? [{ term: { starRating: parseInt(stars) } }] : []),
        ...(amenities ? [{ terms: { amenities: amenities.split(',') } }] : []),
      ],
    },
  };

  const searchResults = await elasticsearch.search({
    index: 'properties',
    body: {
      from: (page - 1) * limit,
      size: limit,
      query: esQuery,
      sort: sort === 'price' ? [{ basePrice: 'asc' }] : ['_score'],
    },
  });

  const propertyIds = searchResults.hits.hits.map(h => h._id);

  // Check availability and get pricing for the date range
  const availableProperties = await availabilityService.checkBulkAvailability(
    propertyIds, checkinDate, checkoutDate, parseInt(guests) || 1
  );

  // Get dynamic pricing
  const pricedResults = await pricingService.getPrices(
    availableProperties, checkinDate, checkoutDate
  );

  // Filter by price range
  const filtered = pricedResults.filter(p => {
    if (minPrice && p.totalPrice / nights < parseFloat(minPrice)) return false;
    if (maxPrice && p.totalPrice / nights > parseFloat(maxPrice)) return false;
    return true;
  });

  res.json({
    data: filtered,
    meta: {
      page, limit, total: searchResults.hits.total.value,
      nights, checkin, checkout,
    },
  });
});

Booking API

// POST /api/bookings
router.post('/bookings', authenticate, async (req, res) => {
  const {
    propertyId, roomTypeId, checkin, checkout,
    guests, paymentMethodId, specialRequests,
  } = req.body;

  const checkinDate = new Date(checkin);
  const checkoutDate = new Date(checkout);
  const nights = Math.ceil((checkoutDate - checkinDate) / (1000 * 60 * 60 * 24));

  // 1. Check availability with lock
  const booking = await prisma.$transaction(async (tx) => {
    // Lock availability rows for the date range
    const availability = await tx.$queryRaw`
      SELECT * FROM room_availability
      WHERE room_type_id = ${roomTypeId}
        AND date >= ${checkinDate}
        AND date < ${checkoutDate}
      FOR UPDATE
    `;

    // Verify every night has rooms available
    if (availability.length < nights) {
      throw new Error('Room not available for all requested dates');
    }
    const unavailable = availability.find(a => a.available_rooms <= 0);
    if (unavailable) {
      throw new Error(`No rooms available on ${unavailable.date}`);
    }

    // 2. Calculate total price from availability records
    const totalPrice = availability.reduce((sum, a) => sum + parseFloat(a.price), 0);

    // 3. Decrement available rooms for each night
    await tx.$executeRaw`
      UPDATE room_availability
      SET available_rooms = available_rooms - 1,
          updated_at = NOW()
      WHERE room_type_id = ${roomTypeId}
        AND date >= ${checkinDate}
        AND date < ${checkoutDate}
        AND available_rooms > 0
    `;

    // 4. Create booking record
    const booking = await tx.booking.create({
      data: {
        userId: req.user.id,
        propertyId,
        roomTypeId,
        checkinDate,
        checkoutDate,
        guests,
        nights,
        totalPrice,
        status: 'CONFIRMED',
        specialRequests,
        confirmationCode: generateConfirmationCode(),
      },
    });

    return booking;
  });

  // 5. Process payment
  await paymentService.charge({
    amount: booking.totalPrice,
    currency: 'USD',
    paymentMethodId,
    bookingId: booking.id,
  });

  // 6. Send confirmation
  await notificationQueue.add('booking-confirmation', {
    bookingId: booking.id,
    userId: req.user.id,
    email: req.user.email,
  });

  res.status(201).json(booking);
});

function generateConfirmationCode() {
  const chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789';
  let code = '';
  for (let i = 0; i < 8; i++) {
    code += chars[Math.floor(Math.random() * chars.length)];
  }
  return code;
}

CREATE TABLE room_types (

    amenities     TEXT[]
);

CREATE INDEX idx_room_types_property ON room_types(property_id);

-- Room Availability (one row per room_type per date)
CREATE TABLE room_availability (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    room_type_id    UUID REFERENCES room_types(id),

    available_rooms INTEGER NOT NULL,
    price           DECIMAL(10,2) NOT NULL,  -- price for this specific date

CREATE INDEX idx_availability_lookup
    ON room_availability(room_type_id, date, available_rooms)
    WHERE available_rooms > 0;

Step 6: Deep Dive - Availability Management

The availability table is the most critical and most queried table. Each row represents one room type on one date.

Pre-population: When a hotel adds 10 "Deluxe King" rooms,
generate 365 rows (one per day) with available_rooms = 10.

Booking:   available_rooms -= 1 (with FOR UPDATE lock)
Cancel:    available_rooms += 1
Date pass: Old rows archived, new rows generated (rolling window)

// Availability service with Redis caching
class AvailabilityService {
  async checkBulkAvailability(propertyIds, checkin, checkout, guests) {
    const results = [];

    for (const propertyId of propertyIds) {
      const cacheKey = `avail:${propertyId}:${checkin}:${checkout}:${guests}`;
      const cached = await redis.get(cacheKey);

      if (cached) {
        const parsed = JSON.parse(cached);
        if (parsed.available) results.push(parsed);
        continue;
      }

      // Query DB for room types with availability on ALL dates in range
      const available = await prisma.$queryRaw`
        SELECT rt.id, rt.name, rt.max_guests, rt.base_price,
               MIN(ra.available_rooms) as min_available,
               SUM(ra.price) as total_price,
               AVG(ra.price) as avg_nightly_price
        FROM room_types rt
        JOIN room_availability ra ON ra.room_type_id = rt.id
        WHERE rt.property_id = ${propertyId}
          AND rt.max_guests >= ${guests}
          AND ra.date >= ${checkin}
          AND ra.date < ${checkout}
          AND ra.available_rooms > 0
        GROUP BY rt.id, rt.name, rt.max_guests, rt.base_price
        HAVING COUNT(ra.date) = ${nights}
           AND MIN(ra.available_rooms) > 0
      `;

      const result = {
        propertyId,
        available: available.length > 0,
        roomTypes: available,
      };

      // Cache for 60 seconds (availability changes frequently)
      await redis.setex(cacheKey, 60, JSON.stringify(result));

      if (result.available) results.push(result);
    }

    return results;
  }
}

Step 7: Dynamic Pricing

class PricingService {
  async calculatePrice(roomTypeId, checkin, checkout) {
    const baseAvailability = await prisma.roomAvailability.findMany({
      where: {
        roomTypeId,
        date: { gte: checkin, lt: checkout },
      },
    });

    let totalPrice = 0;
    for (const day of baseAvailability) {
      let price = parseFloat(day.price);

      // Demand-based adjustment
      const occupancyRate = 1 - (day.availableRooms / day.totalRooms);
      if (occupancyRate > 0.9) price *= 1.3;       // 90%+ occupied: +30%
      else if (occupancyRate > 0.7) price *= 1.15;  // 70%+ occupied: +15%

      // Day-of-week adjustment
      const dayOfWeek = new Date(day.date).getDay();
      if (dayOfWeek === 5 || dayOfWeek === 6) price *= 1.2; // Fri/Sat: +20%

      totalPrice += price;
    }

    return { totalPrice, avgNightlyPrice: totalPrice / baseAvailability.length };
  }
}

Step 8: Cancellation Handling

// POST /api/bookings/:id/cancel
router.post('/bookings/:id/cancel', authenticate, async (req, res) => {
  const booking = await prisma.booking.findUnique({
    where: { id: req.params.id },
    include: { property: true },
  });

  if (!booking || booking.userId !== req.user.id) {
    return res.status(404).json({ error: 'Booking not found' });
  }
  if (booking.status === 'CANCELLED') {
    return res.status(400).json({ error: 'Already cancelled' });
  }

  // Calculate refund based on cancellation policy
  const daysUntilCheckin = Math.ceil(
    (new Date(booking.checkinDate) - new Date()) / (1000 * 60 * 60 * 24)
  );
  const refundAmount = calculateRefund(
    booking.property.cancellationPolicy,
    booking.totalPrice,
    daysUntilCheckin
  );

  await prisma.$transaction(async (tx) => {
    // 1. Update booking status
    await tx.booking.update({
      where: { id: booking.id },
      data: {
        status: 'CANCELLED',
        cancelledAt: new Date(),
        refundAmount,
      },
    });

    // 2. Restore room availability
    await tx.$executeRaw`
      UPDATE room_availability
      SET available_rooms = available_rooms + 1,
          updated_at = NOW()
      WHERE room_type_id = ${booking.roomTypeId}
        AND date >= ${booking.checkinDate}
        AND date < ${booking.checkoutDate}
    `;
  });

  // 3. Process refund
  if (refundAmount > 0) {
    await paymentService.refund(booking.id, refundAmount);
  }

  // 4. Invalidate availability cache
  await invalidateAvailabilityCache(booking.propertyId);

  res.json({ status: 'CANCELLED', refundAmount });
});

function calculateRefund(policy, totalPrice, daysUntilCheckin) {
  switch (policy) {
    case 'FREE_CANCELLATION':
      return daysUntilCheckin >= 1 ? totalPrice : totalPrice * 0.5;
    case 'MODERATE':
      if (daysUntilCheckin >= 5) return totalPrice;
      if (daysUntilCheckin >= 1) return totalPrice * 0.5;
      return 0;
    case 'STRICT':
      return daysUntilCheckin >= 14 ? totalPrice * 0.5 : 0;
    case 'NON_REFUNDABLE':
      return 0;
    default:
      return 0;
  }
}

Step 9: Scaling Strategies

Search:       Elasticsearch cluster (3+ nodes, replicated)
              - Geo queries for "near me" searches
              - Denormalized property + room data for fast search

Availability: Read replicas + Redis cache (60s TTL)
              - Sharded by property_id region
              - Hot properties get shorter cache TTL

Bookings:     Write-primary Postgres with read replicas
              - Partitioned by checkin_date (monthly)
              - Old partitions moved to cold storage

Images:       S3 + CloudFront CDN
              - Multiple resolutions generated on upload

Step 10: Trade-offs

Decision Option A Option B Recommendation
Availability model Row per room per date Row per room type per date Row per room type per date (less storage)
Availability cache No cache (always fresh) Redis 60s TTL Redis cache (accept slight staleness for search)
Double-booking prevention Pessimistic lock (FOR UPDATE) Optimistic (check + retry) Pessimistic for bookings (critical correctness)
Search PostgreSQL + PostGIS Elasticsearch Elasticsearch (better full-text + geo)
Pricing Static (set by hotel) Dynamic (algorithm) Hybrid (hotel sets base, system suggests adjustments)

Summary

A hotel booking system centers on the availability management problem: one row per room type per date, locked during booking to prevent double-booking, cached aggressively for search. Separate the search path (Elasticsearch, eventually consistent, cached) from the booking path (PostgreSQL with transactions, strongly consistent). Dynamic pricing adjusts based on occupancy and demand. Cancellation policies determine refund amounts and restore inventory. Partition the availability table by date for efficient cleanup of past dates.

Common Problems & Solutions

1 Double bookings occurring despite database constraints
Symptoms: Two confirmed bookings for the same room on the same night. Hotel reports overbooking. available_rooms column shows negative values.
Cause: The application reads availability and creates the booking in separate, non-transactional steps. Between the availability check and the booking insert, another request books the last room.
Solution: Use SELECT FOR UPDATE within a single database transaction. The lock prevents concurrent transactions from reading the same availability row until the first transaction commits or rolls back. Add a CHECK constraint (available_rooms >= 0) as a safety net.
2 Search results showing sold-out hotels as available
Cause: The availability cache (Redis, 60s TTL) still holds stale data after the last room was booked. Search results use cached data that has not yet expired.
Solution: Accept this as a trade-off: search showing briefly stale data is OK. Always re-check availability at booking time (with locking). Optionally, use cache invalidation on booking to reduce staleness. Show 'limited availability' warnings when available_rooms <= 3.
3 Slow search queries during peak booking season
Cause: The availability join query across millions of rows is expensive, especially when filtering by date range across many properties. Elasticsearch index is not properly tuned for concurrent queries.
Solution: Denormalize availability into the Elasticsearch index (update async via CDC). Add read replicas for the availability database. Use result caching with short TTL. Pre-compute popular search combinations (top cities + next 30 days).
