Read Replicas and Master-Slave Replication

Intuition

Database replication is like a professor (primary) giving a lecture that is live-streamed to overflow rooms (replicas). Students in overflow rooms see the same content with a slight delay. If students have questions (writes), they must go to the main lecture hall. If the professor is sick, a teaching assistant from an overflow room can take over.

Read Replicas and Master-Slave Replication

Introduction

Database replication copies data from one database server (the primary/master) to one or more database servers (replicas/slaves). This is the first step in scaling your database layer — read replicas handle read queries, freeing the primary to focus on writes.

Why Replication?

Without Replication:                 With Replication:
┌──────────┐                         ┌──────────┐     ┌──────────┐
│ App      │──── ALL queries ──────> │ Primary  │────>│ Replica 1│
│ Server   │                         │ (writes) │     │ (reads)  │
└──────────┘                         └──────────┘     └──────────┘
    │                                     │           ┌──────────┐
    ▼                                     └──────────>│ Replica 2│
Single DB handles everything                          │ (reads)  │
= bottleneck                                          └──────────┘

Benefits: - Read scaling: Distribute reads across replicas (80%+ of queries are reads) - High availability: If primary fails, promote a replica - Geographic distribution: Put replicas closer to users - Analytics isolation: Run heavy reports on a replica without impacting production

PostgreSQL Streaming Replication

┌────────────┐   WAL Stream    ┌────────────┐
│  Primary   │ ──────────────> │  Replica   │
│            │                 │            │
│ Accepts    │   Continuous    │ Read-only  │
│ reads +    │   async/sync   │ queries    │
│ writes     │   replication  │ only       │
└────────────┘                 └────────────┘

WAL = Write-Ahead Log
Every change is written to WAL first
Replicas receive and replay WAL entries

Docker Compose Setup

# docker-compose.yml
version: '3.8'
services:
  postgres-primary:
    image: postgres:16
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: myapp
    command: >
      postgres
      -c wal_level=replica
      -c max_wal_senders=3
      -c max_replication_slots=3
      -c hot_standby=on
    ports:
      - "5432:5432"
    volumes:
      - primary_data:/var/lib/postgresql/data

  postgres-replica:
    image: postgres:16
    environment:
      PGUSER: app
      PGPASSWORD: secret
    command: >
      bash -c "
        until pg_basebackup --pgdata=/var/lib/postgresql/data -R
          --host=postgres-primary --port=5432 --username=app; do
          echo 'Waiting for primary...'
          sleep 2
        done
        postgres
      "
    ports:
      - "5433:5432"
    depends_on:
      - postgres-primary
    volumes:
      - replica_data:/var/lib/postgresql/data

  primary_data:
  replica_data:

Read/Write Splitting in Node.js

With Prisma

import { PrismaClient } from '@prisma/client';

// Primary for writes
const primaryPrisma = new PrismaClient({
  datasources: {
    db: { url: process.env.DATABASE_PRIMARY_URL },
  },
});

// Replica for reads
const replicaPrisma = new PrismaClient({
  datasources: {
    db: { url: process.env.DATABASE_REPLICA_URL },
  },
});

class UserRepository {
  // Reads go to replica
  async findById(id) {
    return replicaPrisma.user.findUnique({ where: { id } });
  }

  async findMany(filter) {
    return replicaPrisma.user.findMany({ where: filter });
  }

  async search(query) {
    return replicaPrisma.user.findMany({
      where: { name: { contains: query, mode: 'insensitive' } },
    });
  }

  // Writes go to primary
  async create(data) {
    return primaryPrisma.user.create({ data });
  }

  async update(id, data) {
    return primaryPrisma.user.update({ where: { id }, data });
  }

  async delete(id) {
    return primaryPrisma.user.delete({ where: { id } });
  }

  // Read-after-write: use primary to avoid replication lag
  async createAndReturn(data) {
    const user = await primaryPrisma.user.create({ data });
    // Read from primary because replica may not have it yet
    return primaryPrisma.user.findUnique({
      where: { id: user.id },
      include: { profile: true },
    });
  }
}

Generic Read/Write Router

class DatabaseRouter {
  constructor(primaryUrl, replicaUrls) {
    this.primary = new PrismaClient({
      datasources: { db: { url: primaryUrl } },
    });

    this.replicas = replicaUrls.map(url =>
      new PrismaClient({ datasources: { db: { url } } })
    );

    this.currentReplica = 0;
  }

  // Round-robin replica selection
  getReadClient() {
    if (this.replicas.length === 0) return this.primary;
    const client = this.replicas[this.currentReplica];
    this.currentReplica = (this.currentReplica + 1) % this.replicas.length;
    return client;
  }

  getWriteClient() {
    return this.primary;
  }

  // Force primary reads (after writes, for consistency)
  getPrimaryReadClient() {
    return this.primary;
  }
}

const db = new DatabaseRouter(
  process.env.DATABASE_PRIMARY_URL,
  [
    process.env.DATABASE_REPLICA_1_URL,
    process.env.DATABASE_REPLICA_2_URL,
  ]
);

// Usage
app.get('/api/users', async (req, res) => {
  const users = await db.getReadClient().user.findMany();
  res.json(users);
});

app.post('/api/users', async (req, res) => {
  const user = await db.getWriteClient().user.create({ data: req.body });
  res.status(201).json(user);
});

Replication Lag

The time between a write on the primary and it appearing on the replica is called replication lag. It is the biggest challenge with read replicas.

Timeline:
T0: Write to primary (user.name = "Alice")
T1: WAL entry created on primary
T2: WAL streamed to replica (+1-50ms typically)
T3: Replica applies WAL entry

Between T0 and T3: reading from replica returns stale data!

Handling Replication Lag

// Strategy 1: Read-after-write from primary
app.post('/api/users', async (req, res) => {
  const user = await db.getWriteClient().user.create({ data: req.body });
  // Immediately read from primary, not replica
  const fullUser = await db.getPrimaryReadClient().user.findUnique({
    where: { id: user.id },
    include: { profile: true },
  });
  res.status(201).json(fullUser);
});

// Strategy 2: Session-based routing
// After a write, route that user's reads to primary for N seconds
const recentWriters = new Map(); // userId -> timestamp

app.post('/api/users/:id', async (req, res) => {
  await db.getWriteClient().user.update({
    where: { id: req.params.id },
    data: req.body,
  });
  recentWriters.set(req.user.id, Date.now());
  res.json({ success: true });
});

app.get('/api/users/:id', async (req, res) => {
  const lastWrite = recentWriters.get(req.user.id);
  const usesPrimary = lastWrite && (Date.now() - lastWrite < 5000);

  const client = usesPrimary ? db.getPrimaryReadClient() : db.getReadClient();
  const user = await client.user.findUnique({ where: { id: req.params.id } });
  res.json(user);
});

// Strategy 3: Check replication lag before routing
async function getReplicationLag() {
  const result = await db.getReadClient().$queryRaw`
    SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp())) AS lag_seconds
  `;
  return result[0]?.lag_seconds || 0;
}

Synchronous vs Asynchronous Replication

Asynchronous (default):
  Primary ──> Write to WAL ──> ACK to client ──> Stream to replica (later)
  Pros: No write latency impact
  Cons: Data loss possible if primary crashes before streaming

Synchronous:
  Primary ──> Write to WAL ──> Stream to replica ──> Replica ACK ──> ACK to client
  Pros: No data loss, strong consistency
  Cons: Write latency increases, replica failure blocks writes

Monitoring Replication

// Monitor replication status
async function checkReplicationHealth() {
  // On primary
  const primaryStatus = await primaryPrisma.$queryRaw`
    SELECT client_addr, state, sent_lsn, replay_lsn,
           (sent_lsn - replay_lsn) AS lag_bytes
    FROM pg_stat_replication
  `;

  // On replica
  const replicaLag = await replicaPrisma.$queryRaw`
    SELECT CASE
      WHEN pg_last_wal_receive_lsn() = pg_last_wal_replay_lsn() THEN 0
      ELSE EXTRACT(EPOCH FROM now() - pg_last_xact_replay_timestamp())
    END AS lag_seconds
  `;

  return {
    primary: primaryStatus,
    replicaLag: replicaLag[0]?.lag_seconds,
  };
}

// Health check endpoint
app.get('/health/replication', async (req, res) => {
  const health = await checkReplicationHealth();
  const isHealthy = health.replicaLag < 5; // Less than 5 seconds lag
  res.status(isHealthy ? 200 : 503).json(health);
});

Common Pitfalls

  1. Reading your own writes from replica: User creates a post, page refreshes, post is missing (not yet replicated). Always read from primary after writes.
  2. Replica as backup: Replicas are not backups. If you delete data on primary, it is deleted on replicas too. Use pg_dump or point-in-time recovery for backups.
  3. All reads to one replica: Without load balancing across replicas, one replica gets overloaded. Use round-robin or least-connections.
  4. Not monitoring replication lag: If lag grows unnoticed, users see increasingly stale data.

Summary

Read replicas scale read-heavy workloads by distributing read queries across multiple database copies. PostgreSQL streaming replication continuously sends WAL changes to replicas. The main challenge is replication lag — handle it with read-after-write consistency, session-based routing, or lag monitoring. For high availability, combine replication with automatic failover using tools like Patroni or cloud-managed solutions.

Common Problems & Solutions

1 User creates a record but cannot see it immediately
Symptoms: Users report 'missing data' that appears after refreshing a few seconds later.
Cause: The write goes to the primary, but the subsequent read goes to a replica that has not yet received the replication stream.
Solution: Implement read-after-write consistency: after a write, route that user's reads to the primary for a few seconds.
2 Replica falls far behind primary
Cause: Replica is under heavy read load, has insufficient resources, or network between primary and replica is slow.
Solution: Monitor replication lag continuously. Add more replicas to distribute read load. Ensure replica has similar hardware to primary. Check for long-running queries on the replica that block WAL replay.
