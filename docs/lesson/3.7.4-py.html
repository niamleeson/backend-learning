Redis in Python: redis-py and Async Redis

Intuition

redis-py wraps the Redis protocol in Pythonic APIs. Pipelines are like batching HTTP requests, and WATCH/MULTI gives you optimistic transactions.

Redis in Python: redis-py

redis-py is the standard Python client for Redis, supporting sync/async operations, connection pooling, pipelines, and all Redis data types.

import redis

pool = redis.ConnectionPool(
    host='localhost', port=6379, db=0,
    max_connections=20, decode_responses=True
)
r = redis.Redis(connection_pool=pool)

# Pipeline: batch commands for fewer round trips
with r.pipeline() as pipe:
    pipe.set('key1', 'val1')
    pipe.set('key2', 'val2')
    pipe.get('key1')
    results = pipe.execute()  # [True, True, 'val1']

# Transaction with optimistic locking
with r.pipeline(transaction=True) as pipe:
    pipe.watch('balance:1')
    balance = int(r.get('balance:1'))
    pipe.multi()
    pipe.set('balance:1', balance - 100)
    pipe.set('balance:2', balance + 100)
    pipe.execute()

# Async support
import redis.asyncio as aioredis

async def async_example():
    r = aioredis.from_url('redis://localhost')
    await r.set('async_key', 'value')
    val = await r.get('async_key')
    await r.aclose()

Key Takeaways: - Use ConnectionPool for multi-threaded apps; redis-py is thread-safe with pooling. - Pipelines batch multiple commands into a single round trip, reducing latency. - Use pipeline(transaction=True) with WATCH for optimistic concurrency control.

Common Problems & Solutions

1 WatchError during transaction.
Cause: Another client modified the watched key between WATCH and EXEC.
Solution: Retry the transaction in a loop. This is expected with optimistic locking; keep the WATCH-EXEC window small.
2 Connection refused or timeout errors in production.
Cause: Redis server is down, network issues, or connection pool exhausted.
Solution: Use retry_on_timeout=True, health_check_interval, and monitor pool usage. Add circuit breaker logic.
