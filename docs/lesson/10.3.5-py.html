Design a Food Delivery System (DoorDash)

Intuition

A food delivery system is a real-time logistics platform that continuously solves a three-sided matching problem between customers, restaurants, and drivers while optimizing for speed, cost, and driver utilization. The geospatial matching and ETA prediction are the core algorithmic challenges.

Design a Food Delivery System (DoorDash)

Requirements

  • Match customer orders with nearby restaurants and delivery drivers
  • Provide real-time order tracking from placement to delivery
  • Optimize delivery routes and estimated arrival times
  • Handle dynamic pricing (surge) during peak demand
  • Support restaurant menu management and order preparation workflows

Architecture

A customer-facing service handles browsing and ordering. An order service manages the order lifecycle as a state machine. A matching service assigns drivers to orders using location and optimization algorithms. A real-time tracking service uses WebSockets to push driver locations. A logistics engine optimizes batching and routing.

Key Components

  • Order Service: State machine managing order states (placed, confirmed, preparing, picked_up, en_route, delivered)
  • Matching/Dispatch Service: Assigns drivers based on proximity, current load, and estimated pickup time using geospatial indexing
  • Location Service: Ingests driver GPS updates (every 5 seconds) and stores in a geospatial index (Redis GEO or PostGIS)
  • ETA Service: Predicts preparation time, pickup time, and delivery time using historical data and real-time traffic
  • Surge Pricing Engine: Adjusts delivery fees based on supply (available drivers) and demand (order volume) per zone

Scaling

Partition geospatially by delivery zone. Use a geospatial index for efficient nearest-driver queries. Batch nearby orders for the same driver to improve efficiency. Use event sourcing for the order lifecycle to enable replay and analytics.

Trade-offs

  • Immediate dispatch reduces wait time but may not find the optimal driver; batching orders improves efficiency but adds delay
  • Surge pricing balances supply and demand but frustrates customers; subsidizing drivers during peaks is costly

Exercises

  1. Design a Driver Matching Algorithm

    Build a dispatch system that assigns the best available driver to a new order. Consider: distance to restaurant, driver's current delivery status, estimated restaurant preparation time, and driver acceptance rate. Score and rank candidates.

    Hint

    Use Redis GEO to find drivers within a radius of the restaurant. Score each driver: score = w1(1/distance) + w2(1/current_orders) + w3acceptance_rate + w4match_with_prep_time. Select the highest-scoring available driver.

  2. Implement Order State Machine

    Create an order lifecycle state machine with states: placed, restaurant_confirmed, preparing, ready_for_pickup, driver_assigned, picked_up, en_route, delivered, cancelled. Define valid transitions and the events that trigger them. Handle edge cases like cancellation at any stage.

    Hint

    Use an enum for states and a transition table mapping (current_state, event) -> next_state. Validate transitions before applying. Emit events on each transition for downstream services. Cancellation has different compensation logic depending on the current state.

Common Problems & Solutions

1 During dinner rush, there are not enough drivers and delivery times spike to over an hour, causing order cancellations.
Cause: Demand exceeds driver supply in certain zones during peak hours, and the system has no mechanism to balance supply and demand.
Solution: Implement surge pricing to incentivize more drivers to work during peak hours and to moderate demand. Use predictive models to forecast demand and pre-position drivers. Allow order batching where one driver picks up multiple nearby orders.
2 ETA estimates are consistently inaccurate, showing 30 minutes when actual delivery takes 50+ minutes, frustrating customers.
Cause: The ETA model uses static estimates for restaurant preparation time and does not account for real-time factors like restaurant backlog, traffic conditions, or driver delays.
Solution: Build an ML-based ETA model trained on historical delivery data incorporating: real-time restaurant queue length, live traffic data, time of day, weather, and driver speed patterns. Update ETAs dynamically as the order progresses through each stage.
