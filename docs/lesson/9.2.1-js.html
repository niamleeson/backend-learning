Design a File Storage System (Dropbox)

Intuition

Think of Dropbox like a post office with lockers. When you upload a file, the clerk (API) does not personally store the package -- they give you a locker number (presigned URL) and you put it in the locker (S3) yourself. To sync, each device keeps a receipt number (cursor). They ask: 'What happened after receipt #4527?' The clerk replies with a list of new packages, moves, and deletions. Chunking is like sending a large parcel in multiple boxes -- if one gets lost in transit, you only resend that box.

Design a File Storage System (Dropbox)

Introduction

A cloud file storage system allows users to upload, download, sync, and share files across devices. Think Dropbox, Google Drive, or OneDrive. This is a classic system design problem that tests your understanding of large-scale object storage, metadata management, chunked uploads, deduplication, and synchronization protocols.

Step 1: Requirements Gathering

Functional Requirements

  • Upload files: Users can upload files of any size (up to 10 GB)
  • Download files: Users can download files to any device
  • Sync across devices: Changes on one device reflect on all others
  • File versioning: Keep history of file changes, allow rollback
  • Sharing: Share files/folders with other users via links or permissions
  • Offline editing: Users can edit files offline and sync when reconnected

Non-Functional Requirements

  • Availability: 99.9% uptime (less than 8.76 hours of downtime per year)
  • Durability: 99.999999999% (eleven 9s) -- no data loss
  • Latency: Uploads should start immediately; downloads < 200ms TTFB for small files
  • Consistency: Eventual consistency for sync, strong consistency for metadata
  • Scalability: Support 500M users, 100M DAU

Step 2: Capacity Estimation

Users:             500M total, 100M DAU
Avg files/user:    200 files
Avg file size:     500 KB
Total storage:     500M * 200 * 500KB = 50 PB
Daily uploads:     100M * 2 files/day = 200M files/day
Upload throughput: 200M * 500KB / 86400 = ~1.16 TB/s (peak: ~3 TB/s)
Read:Write ratio:  3:1 (more downloads than uploads)

Metadata per file: ~1KB (name, path, size, timestamps, permissions)
Metadata storage:  500M * 200 * 1KB = 100 TB

API requests:      ~10K uploads/sec, ~30K downloads/sec (avg)

Step 4: API Design

import express from 'express';
import crypto from 'node:crypto';

const router = express.Router();

// Initiate a file upload (returns presigned URL for chunked upload)
router.post('/api/v1/files/upload/init', authenticate, async (req, res) => {
  const { fileName, fileSize, parentFolderId, checksum } = req.body;

  // Check for deduplication
  const existing = await metadataService.findByChecksum(checksum, req.user.id);
  if (existing) {
    return res.json({ status: 'duplicate', fileId: existing.id });
  }

  // Calculate chunks
  const CHUNK_SIZE = 4 * 1024 * 1024; // 4MB chunks
  const totalChunks = Math.ceil(fileSize / CHUNK_SIZE);

  const upload = await uploadService.initiate({
    userId: req.user.id,
    fileName,
    fileSize,
    parentFolderId,
    totalChunks,
    checksum,
  });

  // Generate presigned URLs for each chunk
  const chunkUrls = await Promise.all(
    Array.from({ length: totalChunks }, (_, i) =>
      storageService.getPresignedUploadUrl(upload.id, i)
    )
  );

  res.status(201).json({
    uploadId: upload.id,
    chunkSize: CHUNK_SIZE,
    totalChunks,
    chunkUrls, // Client uploads directly to S3
  });
});

// Complete upload after all chunks are uploaded
router.post('/api/v1/files/upload/:uploadId/complete', authenticate, async (req, res) => {
  const { uploadId } = req.params;
  const { chunkChecksums } = req.body;

  const file = await uploadService.complete(uploadId, chunkChecksums);
  await syncService.notifyDevices(req.user.id, {
    type: 'FILE_CREATED',
    file,
  });

  res.json({ file });
});

// Download a file (returns presigned download URL)
router.get('/api/v1/files/:fileId/download', authenticate, async (req, res) => {
  const file = await metadataService.getFile(req.params.fileId, req.user.id);
  if (!file) return res.status(404).json({ error: 'File not found' });

  const downloadUrl = await storageService.getPresignedDownloadUrl(file.storageKey);
  res.json({ downloadUrl, expiresIn: 3600 });
});

// Get sync delta (changes since last sync)
router.get('/api/v1/sync/delta', authenticate, async (req, res) => {
  const { cursor } = req.query; // Last sync timestamp or sequence number

  const changes = await syncService.getChangesSince(req.user.id, cursor);
  res.json({
    changes,        // Array of file/folder change events
    cursor: changes.length ? changes[changes.length - 1].sequence : cursor,
    hasMore: changes.length === 1000,
  });
});

// Share a file/folder
router.post('/api/v1/files/:fileId/share', authenticate, async (req, res) => {
  const { targetUserId, permission } = req.body; // 'view' | 'edit'
  const share = await sharingService.share({
    fileId: req.params.fileId,
    ownerId: req.user.id,
    targetUserId,
    permission,
  });
  res.status(201).json({ share });
});

// List file versions
router.get('/api/v1/files/:fileId/versions', authenticate, async (req, res) => {
  const versions = await metadataService.getVersions(req.params.fileId, req.user.id);
  res.json({ versions });
});

Step 5: Database Schema

-- Users table
CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         VARCHAR(255) UNIQUE NOT NULL,
  display_name  VARCHAR(255) NOT NULL,
  storage_used  BIGINT DEFAULT 0,
  storage_limit BIGINT DEFAULT 2147483648,  -- 2GB free tier
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- File/Folder metadata
CREATE TABLE files (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id        UUID NOT NULL REFERENCES users(id),
  parent_id       UUID REFERENCES files(id),       -- NULL for root
  name            VARCHAR(255) NOT NULL,
  is_folder       BOOLEAN DEFAULT FALSE,
  size            BIGINT DEFAULT 0,
  mime_type       VARCHAR(127),
  checksum        VARCHAR(64),                      -- SHA-256
  storage_key     VARCHAR(512),                     -- S3 object key
  current_version INT DEFAULT 1,
  is_deleted      BOOLEAN DEFAULT FALSE,
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(owner_id, parent_id, name)
);

CREATE INDEX idx_files_owner ON files(owner_id);
CREATE INDEX idx_files_parent ON files(parent_id);
CREATE INDEX idx_files_checksum ON files(checksum);

-- File versions for history
CREATE TABLE file_versions (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  file_id     UUID NOT NULL REFERENCES files(id),
  version     INT NOT NULL,
  size        BIGINT NOT NULL,
  checksum    VARCHAR(64) NOT NULL,
  storage_key VARCHAR(512) NOT NULL,
  created_by  UUID NOT NULL REFERENCES users(id),
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(file_id, version)
);

-- Chunk tracking for resumable uploads
CREATE TABLE upload_chunks (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  upload_id   UUID NOT NULL,
  chunk_index INT NOT NULL,
  size        BIGINT NOT NULL,
  checksum    VARCHAR(64) NOT NULL,
  storage_key VARCHAR(512) NOT NULL,
  uploaded_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(upload_id, chunk_index)
);

-- Sharing permissions
CREATE TABLE shares (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  file_id      UUID NOT NULL REFERENCES files(id),
  owner_id     UUID NOT NULL REFERENCES users(id),
  shared_with  UUID REFERENCES users(id),       -- NULL for public link
  permission   VARCHAR(10) NOT NULL DEFAULT 'view',
  link_token   VARCHAR(64) UNIQUE,
  expires_at   TIMESTAMPTZ,
  created_at   TIMESTAMPTZ DEFAULT NOW()
);

-- Sync journal for change tracking
CREATE TABLE sync_journal (
  id          BIGSERIAL PRIMARY KEY,              -- monotonic sequence
  user_id     UUID NOT NULL REFERENCES users(id),
  file_id     UUID NOT NULL REFERENCES files(id),
  action      VARCHAR(20) NOT NULL,               -- CREATE, UPDATE, DELETE, MOVE
  metadata    JSONB,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_sync_user_seq ON sync_journal(user_id, id);

Step 6: Deep Dive -- Chunked Upload and Deduplication

Chunked Uploads

Large files are split into fixed-size chunks (typically 4 MB). This enables: - Resumable uploads: If the connection drops, only the incomplete chunk needs re-uploading - Parallel uploads: Multiple chunks can upload concurrently - Deduplication: Identical chunks are stored only once

class ChunkedUploadService {
  constructor(storageClient, metadataDb) {
    this.storage = storageClient;
    this.db = metadataDb;
    this.CHUNK_SIZE = 4 * 1024 * 1024; // 4MB
  }

  async initiateUpload(userId, fileName, fileSize) {
    const totalChunks = Math.ceil(fileSize / this.CHUNK_SIZE);
    const uploadId = crypto.randomUUID();

    await this.db.query(
      `INSERT INTO uploads (id, user_id, file_name, file_size, total_chunks, status)
       VALUES ($1, $2, $3, $4, $5, 'in_progress')`,
      [uploadId, userId, fileName, fileSize, totalChunks]
    );

    return { uploadId, totalChunks, chunkSize: this.CHUNK_SIZE };
  }

  async uploadChunk(uploadId, chunkIndex, data) {
    const checksum = crypto.createHash('sha256').update(data).digest('hex');

    // Deduplication: check if this chunk already exists
    const existing = await this.db.query(
      'SELECT storage_key FROM chunks WHERE checksum = $1',
      [checksum]
    );

    let storageKey;
    if (existing.rows.length > 0) {
      storageKey = existing.rows[0].storage_key; // Reuse existing chunk
    } else {
      storageKey = `chunks/${checksum}`;
      await this.storage.putObject(storageKey, data);
    }

    await this.db.query(
      `INSERT INTO upload_chunks (upload_id, chunk_index, size, checksum, storage_key)
       VALUES ($1, $2, $3, $4, $5)
       ON CONFLICT (upload_id, chunk_index) DO NOTHING`,
      [uploadId, chunkIndex, data.length, checksum, storageKey]
    );

    return { chunkIndex, checksum };
  }

  async completeUpload(uploadId) {
    const chunks = await this.db.query(
      'SELECT * FROM upload_chunks WHERE upload_id = $1 ORDER BY chunk_index',
      [uploadId]
    );

    // Create a manifest that references all chunks
    const manifest = chunks.rows.map(c => ({
      index: c.chunk_index,
      storageKey: c.storage_key,
      size: c.size,
    }));

    const fileChecksum = this.computeFileChecksum(chunks.rows);
    return { manifest, checksum: fileChecksum };
  }
}

Sync Protocol

Dropbox uses a delta-sync protocol. The client maintains a cursor (sequence number), and the server returns all changes since that cursor:

class SyncService {
  constructor(db, notificationService) {
    this.db = db;
    this.notify = notificationService;
  }

  async recordChange(userId, fileId, action, metadata = {}) {
    const result = await this.db.query(
      `INSERT INTO sync_journal (user_id, file_id, action, metadata)
       VALUES ($1, $2, $3, $4) RETURNING id`,
      [userId, fileId, action, JSON.stringify(metadata)]
    );

    // Notify other devices in real-time
    await this.notify.push(userId, {
      type: 'SYNC_UPDATE',
      sequence: result.rows[0].id,
    });

    return result.rows[0].id;
  }

  async getChangesSince(userId, cursor = 0, limit = 1000) {
    const result = await this.db.query(
      `SELECT id, file_id, action, metadata, created_at
       FROM sync_journal
       WHERE user_id = $1 AND id > $2
       ORDER BY id ASC
       LIMIT $3`,
      [userId, cursor, limit]
    );

    return result.rows;
  }
}

Step 7: Scaling Strategies

  1. Object Storage: Use S3 or equivalent for virtually unlimited file storage with 11 nines durability
  2. Metadata Sharding: Shard the metadata database by user_id to distribute load
  3. CDN for Downloads: Cache frequently accessed files at edge locations
  4. Presigned URLs: Offload bandwidth from your API servers to S3 directly
  5. Async Processing: Thumbnail generation, virus scanning, and indexing run in background workers via BullMQ
  6. Compression: Compress chunks before storage to reduce costs (delta compression for file versions)
  7. Block-Level Deduplication: At the chunk level, only store unique blocks across all users

Step 8: Trade-offs

Decision Option A Option B Recommendation
Upload approach Direct to API server Presigned URL to S3 Presigned URL (offloads bandwidth)
Chunk size Small (1 MB) Large (8 MB) 4 MB (balance between overhead and resume granularity)
Sync model Polling WebSocket push WebSocket + long-polling fallback
Dedup scope Per-user Global (cross-user) Per-user initially, global later (privacy concerns)
Consistency Strong everywhere Eventual for sync Eventual sync + strong metadata writes
Versioning Full file copies Delta encoding Delta for text, full for binary

Why Presigned URLs? Your API servers become a bottleneck if every byte of file data flows through them. With presigned URLs, clients upload and download directly from S3 while your servers only handle metadata operations. This reduces API server bandwidth from terabytes to megabytes.

Why Eventual Consistency for Sync? With millions of devices syncing simultaneously, enforcing strong consistency on every sync check would create enormous contention. Eventual consistency with conflict detection provides a better user experience.

Conflict Resolution: When two devices edit the same file offline, you cannot merge binary files. The typical strategy is to keep both versions and let the user choose ("conflicted copy"). For text files, operational transform or CRDT-based merging is possible.

Exercises

  1. Build a Chunked Upload Service

    Implement a Node.js service that accepts a file, splits it into 1 MB chunks, computes SHA-256 for each chunk, and uploads them to a local directory. Implement deduplication: if a chunk with the same hash exists, skip the write. Build a reassembly endpoint that reconstructs the original file from chunks.

    Hint

    Use node:crypto for hashing and node:fs/promises for file I/O. Store chunk metadata in a Map or SQLite. Use createReadStream with highWaterMark to read in chunks.

  2. Implement Delta Sync

    Build a sync service with a PostgreSQL-backed journal. Clients POST file changes and GET deltas using a cursor. Write a test that simulates two clients syncing through the server, verifying they converge to the same state.

    Hint

    Use a BIGSERIAL column as the cursor. Return changes WHERE id > cursor ORDER BY id LIMIT 100. The client stores the last received id.

  3. Presigned URL Flow

    Using the AWS SDK (@aws-sdk/s3-request-presigner), implement the full presigned URL upload flow: (1) client requests an upload URL, (2) client uploads directly to S3, (3) client notifies your server of completion, (4) server verifies the upload exists in S3 and records metadata.

    Hint

    Use getSignedUrl with PutObjectCommand. Set URL expiry to 15 minutes. Use HeadObjectCommand to verify the upload on completion.

Common Problems & Solutions

1 Large file uploads fail intermittently on slow connections
Cause: Single-request uploads time out or consume too much memory when buffering the entire file.
Solution: Implement chunked uploads. Split files into 4 MB chunks, upload each independently with retry logic. Use presigned URLs so clients upload directly to object storage. Track chunk completion server-side and reassemble only on final confirmation.
2 Users see stale file listings after editing on another device
Cause: The sync mechanism relies on polling with a long interval, or the notification channel (WebSocket) has silently disconnected.
Solution: Use WebSocket push notifications with heartbeat pings to detect disconnections. Implement automatic reconnection with exponential backoff. On reconnect, immediately fetch the delta using the last known cursor.
3 Storage costs are growing faster than user growth
Cause: No deduplication -- multiple users storing the same files (e.g., common documents, OS files) results in redundant storage.
Solution: Implement content-addressable storage at the chunk level. Use SHA-256 of chunk content as the storage key. Before writing a chunk, check if it already exists. Maintain a reference count and only delete chunks when the count reaches zero.
