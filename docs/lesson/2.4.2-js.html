Query Building: select, insert, update, delete

Intuition

Knex's query builder is like building with LEGO blocks. Each method (where, join, orderBy) snaps another block onto your query. You can build simple or complex structures, rearrange blocks, and conditionally add pieces — and when you're done, Knex assembles the final SQL from all the pieces.

Query Building: select, insert, update, delete

Introduction

Knex's query builder lets you construct SQL queries with a chainable JavaScript API. Each method call adds a clause to the query, and the query is only executed when you call .then(), await it, or call a terminal method. This gives you type-safe, parameterized queries without writing raw SQL strings.

SELECT Queries

import knex from './db.js';

// Basic select
const users = await knex('users').select('*');
// SELECT * FROM users

// Select specific columns
const names = await knex('users').select('id', 'name', 'email');
// SELECT id, name, email FROM users

// With WHERE
const alice = await knex('users')
  .select('id', 'name', 'email')
  .where('email', 'alice@example.com')
  .first(); // returns single object instead of array
// SELECT id, name, email FROM users WHERE email = ? LIMIT 1

// Complex WHERE
const results = await knex('users')
  .select('*')
  .where('age', '>', 25)
  .andWhere('city', 'New York')
  .orWhere('role', 'admin')
  .orderBy('name', 'asc')
  .limit(20)
  .offset(40);

WHERE Variations

// Object syntax (equality)
await knex('users').where({ status: 'active', role: 'admin' });

// Comparison operators
await knex('users').where('age', '>=', 21);

// WHERE IN
await knex('users').whereIn('id', [1, 2, 3]);
await knex('users').whereNotIn('status', ['banned', 'suspended']);

// WHERE NULL
await knex('users').whereNull('deleted_at');
await knex('users').whereNotNull('email_verified_at');

// WHERE BETWEEN
await knex('orders').whereBetween('total', [10, 100]);

// WHERE LIKE
await knex('users').where('name', 'like', '%alice%');
await knex('users').where('name', 'ilike', '%alice%'); // PostgreSQL

// WHERE EXISTS
await knex('users').whereExists(function() {
  this.select('*').from('orders').whereRaw('orders.user_id = users.id');
});

// Grouped conditions
await knex('users').where(function() {
  this.where('role', 'admin').orWhere('role', 'moderator');
}).andWhere('status', 'active');
// WHERE (role = ? OR role = ?) AND status = ?

JOINs

const orders = await knex('orders')
  .select(
    'orders.id',
    'orders.total',
    'users.name as customer_name',
    'users.email'
  )
  .join('users', 'orders.user_id', '=', 'users.id')
  .where('orders.status', 'completed')
  .orderBy('orders.created_at', 'desc');
// SELECT orders.id, orders.total, users.name as customer_name, users.email
// FROM orders INNER JOIN users ON orders.user_id = users.id
// WHERE orders.status = ? ORDER BY orders.created_at DESC

// LEFT JOIN
await knex('users')
  .leftJoin('orders', 'users.id', 'orders.user_id')
  .select('users.name', knex.raw('COUNT(orders.id) as order_count'))
  .groupBy('users.id', 'users.name');

Aggregations

const stats = await knex('orders')
  .count('id as total_orders')
  .sum('total as revenue')
  .avg('total as avg_order')
  .min('total as min_order')
  .max('total as max_order')
  .where('status', 'completed')
  .first();

// Group by
const byCity = await knex('users')
  .select('city')
  .count('* as user_count')
  .groupBy('city')
  .having('user_count', '>', 10)
  .orderBy('user_count', 'desc');

INSERT

// Single insert
const [newUser] = await knex('users')
  .insert({
    name: 'Alice',
    email: 'alice@example.com',
    role: 'customer',
  })
  .returning('*');
// INSERT INTO users (name, email, role) VALUES (?, ?, ?) RETURNING *

// Batch insert
const users = await knex('users')
  .insert([
    { name: 'Bob', email: 'bob@example.com' },
    { name: 'Carol', email: 'carol@example.com' },
    { name: 'Dan', email: 'dan@example.com' },
  ])
  .returning('id');

// Upsert (ON CONFLICT)
const result = await knex('users')
  .insert({ email: 'alice@example.com', name: 'Alice Updated' })
  .onConflict('email')
  .merge(['name']) // update name on conflict
  .returning('*');

// Insert...ignore (ON CONFLICT DO NOTHING)
await knex('users')
  .insert({ email: 'alice@example.com', name: 'Alice' })
  .onConflict('email')
  .ignore();

UPDATE

// Basic update
const updatedCount = await knex('users')
  .where('id', userId)
  .update({
    name: 'New Name',
    updated_at: knex.fn.now(),
  });
// Returns number of affected rows

// Update with returning
const [updated] = await knex('users')
  .where('id', userId)
  .update({ name: 'New Name' })
  .returning('*');

// Increment/decrement
await knex('products')
  .where('id', productId)
  .increment('stock_quantity', 5);

await knex('products')
  .where('id', productId)
  .decrement('stock_quantity', 1);

DELETE

// Delete rows
const deletedCount = await knex('users')
  .where('status', 'banned')
  .delete();
// DELETE FROM users WHERE status = ?

// Soft delete pattern
await knex('users')
  .where('id', userId)
  .update({ deleted_at: knex.fn.now() });

// Delete with returning
const [deleted] = await knex('orders')
  .where('id', orderId)
  .delete()
  .returning('*');

Dynamic Query Builder Pattern

function buildUserQuery(filters) {
  let query = knex('users').select('id', 'name', 'email', 'city', 'age');

  if (filters.search) {
    query = query.where(function() {
      this.where('name', 'ilike', `%${filters.search}%`)
          .orWhere('email', 'ilike', `%${filters.search}%`);
    });
  }

  if (filters.city) {
    query = query.where('city', filters.city);
  }

  if (filters.minAge) {
    query = query.where('age', '>=', filters.minAge);
  }

  if (filters.role) {
    query = query.whereIn('role', Array.isArray(filters.role) ? filters.role : [filters.role]);
  }

  // Sorting
  const sortColumn = filters.sortBy || 'created_at';
  const sortDir = filters.sortDir || 'desc';
  query = query.orderBy(sortColumn, sortDir);

  // Pagination
  const page = filters.page || 1;
  const perPage = Math.min(filters.perPage || 20, 100);
  query = query.limit(perPage).offset((page - 1) * perPage);

  return query;
}

// Usage
const users = await buildUserQuery({
  search: 'alice',
  city: 'New York',
  minAge: 21,
  page: 2,
  perPage: 10,
});

Raw Queries When Needed

// Full raw query
const result = await knex.raw(
  'SELECT * FROM users WHERE email = ? AND created_at > ?',
  ['alice@example.com', '2024-01-01']
);

// Raw expression within a query
await knex('orders')
  .select(
    'user_id',
    knex.raw('SUM(total) as total_spent'),
    knex.raw('COUNT(*) as order_count')
  )
  .groupBy('user_id')
  .having(knex.raw('SUM(total) > ?', [1000]));

Exercises

  1. CRUD API Functions

    Build a complete set of CRUD functions for a products table using Knex: create, findById, findAll (with pagination/filtering), update, softDelete. Include proper error handling.

    Hint

    Use .returning('*') for create/update. Use .first() for findById. Build dynamic where clauses for findAll.

  2. Report Query Builder

    Create a function that builds a sales report query with configurable: date range, grouping (daily/weekly/monthly), filters (category, min amount), and sorting. Use Knex's query builder.

    Hint

    Use knex.raw for date truncation: knex.raw("DATE_TRUNC(?, created_at)", [grouping]). Chain conditionally based on filters.

  3. Batch Operations

    Write functions for: (a) batch insert 10,000 rows efficiently, (b) batch update based on a list of {id, newValue} pairs, (c) bulk delete with a subquery. Measure performance.

    Hint

    For batch insert, use knex.batchInsert or chunk manually. For batch update, use a CTE with VALUES or multiple CASE WHEN.

Common Problems & Solutions

1 Query returns strings instead of numbers for COUNT/SUM
Cause: PostgreSQL returns bigint for COUNT, which Knex/pg serializes as a string to avoid JavaScript precision issues.
Solution: Cast in the query: `.count(knex.raw('id::int as count'))` or parse in JavaScript: `parseInt(result.count, 10)`.
2 .first() returns undefined but rows exist
Cause: The WHERE conditions don't match any rows, or column names are wrong (case sensitivity).
Solution: Debug by logging the query: `console.log(query.toSQL())`. Check column names match exactly. Try removing WHERE clauses one by one.
3 Knex silently builds wrong SQL with chained .orWhere()
Cause: `.orWhere` applies at the top level, not grouped with preceding `.where`. `where('a', 1).orWhere('b', 2)` becomes `WHERE a = 1 OR b = 2`, not grouped.
Solution: Use a callback for grouping: `.where(function() { this.where('a', 1).orWhere('b', 2); })` produces `WHERE (a = 1 OR b = 2)`.
