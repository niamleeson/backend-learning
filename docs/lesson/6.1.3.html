Pagination Patterns: Offset, Cursor, Keyset

Intuition

Offset pagination is like telling a librarian 'skip the first 200 books and give me the next 20' -- she has to count past 200 every time. Cursor pagination is like placing a bookmark: 'give me the next 20 books after this bookmark,' which is instant regardless of position.

Pagination Patterns: Offset, Cursor, Keyset

Pagination is essential for any API returning collections. Choosing the right strategy affects performance, consistency, and usability.

Offset Pagination

Simple but has performance issues on large datasets:

from fastapi import FastAPI, Query

app = FastAPI()

@app.get("/items")
def list_items(offset: int = Query(0, ge=0), limit: int = Query(20, le=100)):
    # SQL: SELECT * FROM items ORDER BY id LIMIT :limit OFFSET :offset
    items = db.execute(
        "SELECT * FROM items ORDER BY id LIMIT :limit OFFSET :offset",
        {"limit": limit, "offset": offset}
    ).fetchall()
    total = db.execute("SELECT COUNT(*) FROM items").scalar()
    return {
        "data": items,
        "total": total,
        "offset": offset,
        "limit": limit
    }

Cursor-Based Pagination

Uses an opaque token for consistent traversal:

import base64, json

@app.get("/items")
def list_items(cursor: str | None = None, limit: int = Query(20, le=100)):
    if cursor:
        decoded = json.loads(base64.b64decode(cursor))
        last_id = decoded["last_id"]
        items = db.execute(
            "SELECT * FROM items WHERE id > :last_id ORDER BY id LIMIT :limit",
            {"last_id": last_id, "limit": limit}
        ).fetchall()
    else:
        items = db.execute(
            "SELECT * FROM items ORDER BY id LIMIT :limit",
            {"limit": limit}
        ).fetchall()

    next_cursor = None
    if items:
        next_cursor = base64.b64encode(
            json.dumps({"last_id": items[-1]["id"]}).encode()
        ).decode()
    return {"data": items, "next_cursor": next_cursor}

Common Mistakes

  • Using offset pagination on tables with millions of rows (O(n) skip cost)
  • Not handling deleted/inserted rows between page requests with offset
  • Exposing internal cursor details instead of opaque tokens

Key Takeaways

  • Offset: simple, supports jump-to-page, but slow on large datasets
  • Cursor: fast and consistent, but no random page access
  • Keyset: similar to cursor but uses actual column values as the boundary

Exercises

  1. Implement Both Pagination Styles

    Create a /products endpoint that supports both offset and cursor pagination. Use a query parameter 'strategy' to toggle between them. Compare the SQL generated for each approach.

    Hint

    For offset, use LIMIT/OFFSET. For cursor, use WHERE id > :cursor_id with LIMIT. Log or return the generated SQL to compare.

  2. Cursor Pagination with Sorting

    Extend cursor pagination to work with a user-selected sort field (e.g., created_at, price). The cursor must encode both the sort value and the ID to handle ties correctly.

    Hint

    Encode both the sort column value and the primary key in the cursor. Use a compound WHERE clause: (sort_col > :val) OR (sort_col = :val AND id > :id).

  3. Pagination Response Envelope

    Design a reusable Pydantic model for paginated responses that includes data, pagination metadata (has_next, has_prev, total if available), and HATEOAS-style links.

    Hint

    Use a generic Pydantic model with TypeVar so the same envelope works for any resource type.

Common Problems & Solutions

1 API becomes extremely slow on later pages
Symptoms: ['Response time increases linearly with page number', 'Database CPU spikes on deep page requests']
Cause: Offset pagination with large offsets causes the database to scan and discard thousands of rows.
Solution: Switch to cursor/keyset pagination for large datasets. If offset is required, add a maximum offset limit.

Bad: OFFSET 100000 scans 100K rows

SELECT * FROM items ORDER BY id LIMIT 20 OFFSET 100000;

Good: keyset is O(1)

SELECT * FROM items WHERE id > 100000 ORDER BY id LIMIT 20;

2 Users see duplicate or missing items when paginating
Cause: New rows inserted or deleted between offset-based page requests shift the result set.
Solution: Use cursor-based pagination which is stable across mutations. Alternatively, snapshot the query with a timestamp filter.
