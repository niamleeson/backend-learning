Pino: High-Performance JSON Logging

Intuition

If Winston is a full-service mail sorting facility, Pino is like a high-speed conveyor belt with a barcode printer. It slaps a barcode (JSON) on every package (log entry) with minimal handling and immediately sends it down the belt to the destination. The speed comes from doing less work in the critical path -- no fancy formatting, no multiple stops, just structured data out the door as fast as possible. Processing and pretty-printing happen downstream, not in your app's hot path.

Pino: High-Performance JSON Logging

Why Pino?

Pino is the fastest JSON logger for Node.js. While Winston is the most flexible, Pino is the most performant -- it produces structured JSON logs with minimal overhead, making it ideal for high-throughput applications where every millisecond counts.

The key difference: Winston formats and processes logs synchronously in the main thread. Pino defers serialization and output to a worker thread or separate process (via pino transports), keeping the event loop free.

┌─────────────────────────────────────────────────────┐
│              Logging Performance Comparison           │
├─────────────────────────────────────────────────────┤
│                                                     │
│  console.log    ████████████████████████  ~100%      │
│  Winston JSON   ██████████████████       ~75%        │
│  Bunyan         ██████████████           ~60%        │
│  Pino           ████                     ~15%        │
│                                                     │
│  (lower is better - % of event loop time consumed)  │
│                                                     │
│  Pino achieves this by:                             │
│  - Minimal serialization in the main thread         │
│  - No string formatting or interpolation            │
│  - Worker thread transports (pino.transport)        │
│  - Flat JSON output (no nested transforms)          │
│                                                     │
└─────────────────────────────────────────────────────┘

Installing Pino

npm install pino
npm install pino-pretty --save-dev  # pretty-print for development

Basic Setup

import pino from 'pino';

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
});

logger.info('Server started');
logger.info({ port: 3000 }, 'Server started on port');
logger.error({ err: new Error('Connection failed') }, 'Database error');
logger.warn({ userId: 'u_123', limit: 100 }, 'Rate limit approaching');
logger.debug({ query: 'SELECT * FROM users' }, 'Query executed');

Output (newline-delimited JSON):

{"level":30,"time":1710500000000,"pid":12345,"hostname":"server-1","msg":"Server started"}
{"level":30,"time":1710500000001,"pid":12345,"hostname":"server-1","port":3000,"msg":"Server started on port"}

Pino's API Convention

Pino uses a different argument order than Winston. The metadata object comes first, the message string comes second:

// Pino convention: context object FIRST, message SECOND
logger.info({ userId: 'u_123', action: 'login' }, 'User logged in');

// Just a message (no context)
logger.info('Simple message');

// Just context (no message)
logger.info({ event: 'startup', port: 3000 });

// WRONG: This does NOT work like Winston
// logger.info('message', { key: 'value' });  // second arg is ignored as merge object

Log Levels

Pino uses numeric levels (lower = more severe):

┌──────────┬────────┬──────────────────────────────────┐
│  Level   │ Number │ Usage                            │
├──────────┼────────┼──────────────────────────────────┤
│ fatal    │   60   │ App is about to crash            │
│ error    │   50   │ Runtime errors, failed ops        │
│ warn     │   40   │ Potential problems                │
│ info     │   30   │ Normal operations                 │
│ debug    │   20   │ Detailed debugging                │
│ trace    │   10   │ Very detailed tracing             │
│ silent   │   --   │ Disable all logging               │
└──────────┴────────┴──────────────────────────────────┘

// Custom levels
const logger = pino({
  level: 'info',
  customLevels: {
    audit: 35,  // between info (30) and warn (40)
  },
});

logger.audit({ userId: 'u_123', action: 'delete_account' }, 'Audit event');

Pretty Printing for Development

Pino's raw JSON output is optimized for machines. For development, use pino-pretty:

import pino from 'pino';

const logger = pino({
  level: 'debug',
  transport: process.env.NODE_ENV !== 'production'
    ? {
        target: 'pino-pretty',
        options: {
          colorize: true,
          translateTime: 'HH:MM:ss',
          ignore: 'pid,hostname',
          singleLine: false,
        },
      }
    : undefined,
});

// Development output:
// 10:30:00 INFO: Server started
//     port: 3000

// Production output (raw JSON):
// {"level":30,"time":1710500000000,"port":3000,"msg":"Server started"}

Child Loggers

Child loggers add persistent context without copying the parent:

import pino from 'pino';

const logger = pino({ level: 'info' });

// Module-level child loggers
const authLogger = logger.child({ module: 'auth' });
const dbLogger = logger.child({ module: 'database' });

authLogger.info({ userId: 'u_123' }, 'Login successful');
// {"level":30,"time":...,"module":"auth","userId":"u_123","msg":"Login successful"}

dbLogger.warn({ query: 'slow query', duration: 5200 }, 'Slow query detected');
// {"level":40,"time":...,"module":"database","query":"slow query","duration":5200,"msg":"Slow query detected"}

// Request-scoped child loggers
function requestMiddleware(req, res, next) {
  req.log = logger.child({
    requestId: req.headers['x-request-id'] || crypto.randomUUID(),
    method: req.method,
    url: req.url,
  });
  next();
}

Serializers

Serializers control how objects are converted to JSON. This is critical for performance and security:

import pino from 'pino';

const logger = pino({
  serializers: {
    // Custom error serializer (redact sensitive info)
    err: pino.stdSerializers.err,  // built-in: includes message, stack, type

    // Custom request serializer
    req: (req) => ({
      method: req.method,
      url: req.url,
      headers: {
        host: req.headers.host,
        'user-agent': req.headers['user-agent'],
        // Deliberately exclude authorization, cookie headers
      },
    }),

    // Custom response serializer
    res: (res) => ({
      statusCode: res.statusCode,
      headers: {
        'content-type': res.getHeader('content-type'),
        'content-length': res.getHeader('content-length'),
      },
    }),

    // Redact user objects
    user: (user) => ({
      id: user.id,
      role: user.role,
      // Omit email, password hash, etc.
    }),
  },
});

// Serializers are triggered by matching key names
logger.info({ req, res, user }, 'Request processed');

Pino with Express: pino-http

npm install pino-http

import express from 'express';
import pino from 'pino';
import pinoHttp from 'pino-http';

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
});

const app = express();

// Automatic request/response logging
app.use(pinoHttp({
  logger,

  // Custom request ID
  genReqId: (req) => req.headers['x-request-id'] || crypto.randomUUID(),

  // Custom log level based on status code
  customLogLevel: (req, res, err) => {
    if (res.statusCode >= 500 || err) return 'error';
    if (res.statusCode >= 400) return 'warn';
    return 'info';
  },

  // Custom success message
  customSuccessMessage: (req, res) => {
    return `${req.method} ${req.url} completed`;
  },

  // Custom error message
  customErrorMessage: (req, res, err) => {
    return `${req.method} ${req.url} failed: ${err.message}`;
  },

  // Customize what gets logged from req/res
  serializers: {
    req: (req) => ({
      method: req.method,
      url: req.url,
      query: req.query,
      params: req.params,
    }),
    res: (res) => ({
      statusCode: res.statusCode,
    }),
  },

  // Redact sensitive headers
  redact: {
    paths: ['req.headers.authorization', 'req.headers.cookie'],
    censor: '[REDACTED]',
  },
}));

// Use req.log in route handlers
app.get('/api/users/:id', (req, res) => {
  req.log.info({ userId: req.params.id }, 'Fetching user');
  // ...
});

app.listen(3000, () => {
  logger.info({ port: 3000 }, 'Server started');
});

Redaction

Pino has built-in redaction via fast-redact, which is much faster than post-processing:

const logger = pino({
  redact: {
    paths: [
      'password',
      'secret',
      'token',
      'req.headers.authorization',
      'req.headers.cookie',
      'user.email',
      'creditCard',
      '*.ssn',           // wildcard: any object's ssn field
    ],
    censor: '[REDACTED]',
  },
});

logger.info({
  user: { id: 'u_123', email: 'test@example.com', name: 'Alice' },
  token: 'eyJhbG...',
}, 'User action');
// user.email and token will show as '[REDACTED]'

Transports (Pino v7+)

Pino transports run in worker threads, keeping the main thread fast:

import pino from 'pino';

const logger = pino({
  level: 'info',
  transport: {
    targets: [
      // Pretty print to console in development
      {
        target: 'pino-pretty',
        level: 'debug',
        options: { colorize: true, translateTime: true },
      },
      // Write to file
      {
        target: 'pino/file',
        level: 'info',
        options: { destination: './logs/app.log', mkdir: true },
      },
      // Write errors to separate file
      {
        target: 'pino/file',
        level: 'error',
        options: { destination: './logs/error.log', mkdir: true },
      },
    ],
  },
});

Async-Safe Logging and Process Exit

Pino's async transports buffer logs. Ensure logs are flushed before process exit:

import pino from 'pino';

const logger = pino({
  level: 'info',
  transport: {
    target: 'pino/file',
    options: { destination: './logs/app.log' },
  },
});

// Flush logs on graceful shutdown
const destination = pino.destination('./logs/app.log');

process.on('SIGTERM', () => {
  logger.info('Shutting down gracefully');
  logger.flush();  // synchronous flush
  // or with async destination:
  // destination.flushSync();
  process.exit(0);
});

// For final logging (pino.final creates a special logger)
const finalLogger = pino.final(logger, (err, finalLogger, evt) => {
  finalLogger.info(`${evt} caught`);
  if (err) finalLogger.error(err, 'error caused exit');
  process.exit(err ? 1 : 0);
});

process.on('uncaughtException', (err) => finalLogger(err, 'uncaughtException'));
process.on('unhandledRejection', (err) => finalLogger(err, 'unhandledRejection'));

Winston vs Pino Decision Guide

┌─────────────────────────────────────────────────────┐
│              Choose Pino when:                       │
│  - Performance is critical (high-throughput APIs)    │
│  - You want minimal overhead on the event loop       │
│  - Your log pipeline handles JSON natively           │
│  - You prefer convention over configuration          │
│                                                     │
│              Choose Winston when:                    │
│  - You need multiple complex transports              │
│  - You want maximum format flexibility               │
│  - You need custom log levels or formats             │
│  - Your team is already familiar with it             │
│                                                     │
│              For most new Node.js projects:          │
│  Pino is the recommended choice.                    │
└─────────────────────────────────────────────────────┘

Summary

Pino is a high-performance structured JSON logger that minimizes impact on your application's event loop. Its worker-thread transports, built-in redaction, and pino-http Express integration make it the best choice for performance-sensitive Node.js backends. Use child loggers for contextual data, serializers for controlling output shape, and pino-pretty for readable development output. In production, pipe raw JSON to your log aggregation system for maximum efficiency.

Common Problems & Solutions

1 Pino logs show numeric levels instead of level names
Cause: Pino uses numeric levels in JSON output for performance. This is by design, but it can be confusing when reading raw logs.
Solution: For development, use pino-pretty which translates levels to names. For production, your log aggregation tool (ELK, Datadog) should map the numeric levels. If you absolutely need string levels in JSON, use the `formatters` option: `formatters: { level: (label) => ({ level: label }) }`.
2 Logs are lost on process crash or SIGTERM
Cause: Pino's async transports buffer log entries. On abrupt termination, buffered entries may not be flushed to disk.
Solution: Use `pino.final()` to create a special logger for shutdown scenarios. Register handlers for 'uncaughtException', 'unhandledRejection', 'SIGTERM', and 'SIGINT' that call the final logger. For synchronous flushing, use `pino.destination({ sync: true })` but note this reduces performance.
3 pino-pretty slows down the application significantly
Cause: pino-pretty is doing formatting work that adds overhead. Using it in production defeats Pino's performance advantage.
Solution: Only use pino-pretty in development. In production, output raw JSON and let your log aggregation tool handle formatting. Set the transport conditionally: `transport: process.env.NODE_ENV !== 'production' ? { target: 'pino-pretty' } : undefined`.
