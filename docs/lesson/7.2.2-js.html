OpenID Connect and Identity Providers

Intuition

If OAuth2 is a valet parking ticket that says what the valet can do with your car, then OpenID Connect is a driver's license taped to that ticket. The driver's license (ID token) tells you exactly who the person is, and because it is cryptographically signed by a trusted authority (the identity provider), you can verify it is genuine without calling the DMV every time.

OpenID Connect and Identity Providers

From Authorization to Authentication

OAuth2 was designed for authorization (granting access to resources), not authentication (verifying who a user is). This distinction matters enormously. OAuth2 alone tells your application "this token has permission to access these resources" but does NOT tell you "the person holding this token is Alice from Engineering."

OpenID Connect (OIDC) is a thin identity layer built on top of OAuth2 that adds authentication. When someone says "Login with Google" or "Sign in with Microsoft," they are using OpenID Connect.

OAuth2 gives you an access token (opaque, for API calls). OIDC adds an ID token (a JWT containing user identity claims).

The ID Token

The ID Token is the core addition of OIDC. It is a JSON Web Token (JWT) that contains claims about the authenticated user.

ID Token (JWT):
┌─────────────────────────────────────────────┐
│ Header (algorithm, type)                     │
├─────────────────────────────────────────────┤
│ Payload (claims):                            │
│   iss: "https://accounts.google.com"         │
│   sub: "110169484474386276334"               │
│   aud: "your-client-id.apps.googleusercontent│
│   exp: 1714500000                            │
│   iat: 1714496400                            │
│   nonce: "abc123"                            │
│   email: "alice@example.com"                 │
│   email_verified: true                       │
│   name: "Alice Smith"                        │
│   picture: "https://...photo.jpg"            │
├─────────────────────────────────────────────┤
│ Signature (verifiable by provider's keys)    │
└─────────────────────────────────────────────┘

Standard Claims

Claim Description
iss Issuer - who issued this token (e.g., Google, Auth0)
sub Subject - unique identifier for the user at this provider
aud Audience - your application's client_id
exp Expiration time (Unix timestamp)
iat Issued at time
nonce Replay attack prevention (you set this)
email User's email address
name User's display name
picture URL to user's profile picture

Validating the ID Token

You MUST validate the ID token before trusting it. Never blindly decode and use the claims.

// id-token-validator.js
const jwt = require('jsonwebtoken');
const jwksClient = require('jwks-rsa');

class IdTokenValidator {
  constructor({ issuer, audience, jwksUri }) {
    this.issuer = issuer;
    this.audience = audience;
    this.client = jwksClient({
      jwksUri: jwksUri,
      cache: true,
      cacheMaxAge: 600000 // Cache keys for 10 minutes
    });
  }

  // Retrieve the signing key from the JWKS endpoint
  getSigningKey(kid) {
    return new Promise((resolve, reject) => {
      this.client.getSigningKey(kid, (err, key) => {
        if (err) return reject(err);
        resolve(key.getPublicKey());
      });
    });
  }

  async validate(idToken, expectedNonce) {
    // Step 1: Decode header to get the key ID (kid)
    const decoded = jwt.decode(idToken, { complete: true });
    if (!decoded) {
      throw new Error('Invalid token: cannot decode');
    }

    // Step 2: Get the public key for this kid
    const publicKey = await this.getSigningKey(decoded.header.kid);

    // Step 3: Verify signature, expiration, issuer, and audience
    const payload = jwt.verify(idToken, publicKey, {
      algorithms: ['RS256'],
      issuer: this.issuer,
      audience: this.audience,
      clockTolerance: 30 // 30 seconds tolerance for clock skew
    });

    // Step 4: Verify nonce (prevents replay attacks)
    if (expectedNonce && payload.nonce !== expectedNonce) {
      throw new Error('Invalid nonce: possible replay attack');
    }

    return payload;
  }
}

// Usage with Google
const googleValidator = new IdTokenValidator({
  issuer: 'https://accounts.google.com',
  audience: process.env.GOOGLE_CLIENT_ID,
  jwksUri: 'https://www.googleapis.com/oauth2/v3/certs'
});

module.exports = { IdTokenValidator, googleValidator };

OIDC Scopes

OIDC defines standard scopes that determine what claims appear in the ID token and UserInfo response.

The openid scope is required. Without it, the provider will not return an ID token and you are doing plain OAuth2.

OIDC Discovery

Every OIDC provider publishes a discovery document at a well-known URL. This tells your application everything it needs to know to integrate.

// oidc-discovery.js
async function discoverOIDCProvider(issuerUrl) {
  const discoveryUrl = `${issuerUrl}/.well-known/openid-configuration`;
  const response = await fetch(discoveryUrl);
  const config = await response.json();

  // The discovery document contains:
  console.log(config.authorization_endpoint); // Where to send users to log in
  console.log(config.token_endpoint);         // Where to exchange codes for tokens
  console.log(config.userinfo_endpoint);      // Where to get user profile
  console.log(config.jwks_uri);               // Where to get signing keys
  console.log(config.scopes_supported);       // What scopes are available
  console.log(config.response_types_supported); // code, id_token, etc.
  console.log(config.id_token_signing_alg_values_supported); // RS256, etc.

  return config;
}

// Try it:
// Google:  discoverOIDCProvider('https://accounts.google.com')
// Auth0:   discoverOIDCProvider('https://your-tenant.auth0.com')
// Keycloak: discoverOIDCProvider('https://keycloak.example.com/realms/myrealm')

This is enormously useful. Instead of hardcoding endpoint URLs, you can dynamically configure your application from the discovery document.

Complete OIDC Integration with Express

// oidc-auth.js
const express = require('express');
const crypto = require('node:crypto');
const session = require('express-session');
const jwt = require('jsonwebtoken');
const jwksClient = require('jwks-rsa');

const app = express();

app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: { httpOnly: true, secure: process.env.NODE_ENV === 'production' }
}));

// Configuration (could be loaded from discovery document)
const OIDC_CONFIG = {
  clientId: process.env.OIDC_CLIENT_ID,
  clientSecret: process.env.OIDC_CLIENT_SECRET,
  issuer: 'https://accounts.google.com',
  authorizationEndpoint: 'https://accounts.google.com/o/oauth2/v2/auth',
  tokenEndpoint: 'https://oauth2.googleapis.com/token',
  userInfoEndpoint: 'https://openidconnect.googleapis.com/v1/userinfo',
  jwksUri: 'https://www.googleapis.com/oauth2/v3/certs',
  redirectUri: 'http://localhost:3000/auth/callback',
  scopes: ['openid', 'profile', 'email']
};

// JWKS client for verifying ID tokens
const jwks = jwksClient({
  jwksUri: OIDC_CONFIG.jwksUri,
  cache: true
});

function getSigningKey(kid) {
  return new Promise((resolve, reject) => {
    jwks.getSigningKey(kid, (err, key) => {
      if (err) return reject(err);
      resolve(key.getPublicKey());
    });
  });
}

// Step 1: Initiate login
app.get('/auth/login', (req, res) => {
  const state = crypto.randomBytes(16).toString('hex');
  const nonce = crypto.randomBytes(16).toString('hex');

  req.session.oauthState = state;
  req.session.oauthNonce = nonce;

  const params = new URLSearchParams({
    response_type: 'code',
    client_id: OIDC_CONFIG.clientId,
    redirect_uri: OIDC_CONFIG.redirectUri,
    scope: OIDC_CONFIG.scopes.join(' '),
    state: state,
    nonce: nonce,
    prompt: 'consent'  // Always show consent screen
  });

  res.redirect(`${OIDC_CONFIG.authorizationEndpoint}?${params}`);
});

// Step 2: Handle callback
app.get('/auth/callback', async (req, res) => {
  const { code, state, error } = req.query;

  if (error) {
    return res.status(400).json({ error: req.query.error_description });
  }

  // Validate state
  if (state !== req.session.oauthState) {
    return res.status(403).json({ error: 'State mismatch' });
  }

  const expectedNonce = req.session.oauthNonce;
  delete req.session.oauthState;
  delete req.session.oauthNonce;

  try {
    // Exchange code for tokens
    const tokenResponse = await fetch(OIDC_CONFIG.tokenEndpoint, {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: new URLSearchParams({
        grant_type: 'authorization_code',
        code: code,
        redirect_uri: OIDC_CONFIG.redirectUri,
        client_id: OIDC_CONFIG.clientId,
        client_secret: OIDC_CONFIG.clientSecret
      })
    });

    const tokens = await tokenResponse.json();

    // Validate the ID token
    const decoded = jwt.decode(tokens.id_token, { complete: true });
    const publicKey = await getSigningKey(decoded.header.kid);

    const idTokenPayload = jwt.verify(tokens.id_token, publicKey, {
      algorithms: ['RS256'],
      issuer: OIDC_CONFIG.issuer,
      audience: OIDC_CONFIG.clientId
    });

    // Verify nonce
    if (idTokenPayload.nonce !== expectedNonce) {
      return res.status(403).json({ error: 'Invalid nonce' });
    }

    // The ID token contains the user's identity
    const user = {
      providerId: idTokenPayload.sub,    // Unique ID at this provider
      email: idTokenPayload.email,
      emailVerified: idTokenPayload.email_verified,
      name: idTokenPayload.name,
      picture: idTokenPayload.picture
    };

    // In production: find-or-create user in your database
    // const dbUser = await findOrCreateUser(user);

    req.session.user = user;
    req.session.accessToken = tokens.access_token;
    req.session.refreshToken = tokens.refresh_token;

    res.redirect('/profile');
  } catch (error) {
    console.error('OIDC authentication failed:', error);
    res.status(500).json({ error: 'Authentication failed' });
  }
});

// Fetch additional user info from the UserInfo endpoint
app.get('/profile', async (req, res) => {
  if (!req.session.user) {
    return res.redirect('/auth/login');
  }

  // The ID token has basic info, but UserInfo may have more
  const userInfoResponse = await fetch(OIDC_CONFIG.userInfoEndpoint, {
    headers: { 'Authorization': `Bearer ${req.session.accessToken}` }
  });

  const fullProfile = await userInfoResponse.json();
  res.json({ idTokenUser: req.session.user, userInfoProfile: fullProfile });
});

app.listen(3000);

Using the openid-client Library

Instead of manually building HTTP requests, use the openid-client library which handles discovery, token management, and validation.

// oidc-with-library.js
const { Issuer, generators } = require('openid-client');
const express = require('express');
const session = require('express-session');

const app = express();
app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false
}));

let oidcClient;

async function initializeOIDC() {
  // Auto-discover all endpoints from the provider
  const issuer = await Issuer.discover('https://accounts.google.com');
  console.log('Discovered issuer:', issuer.metadata.issuer);

  oidcClient = new issuer.Client({
    client_id: process.env.OIDC_CLIENT_ID,
    client_secret: process.env.OIDC_CLIENT_SECRET,
    redirect_uris: ['http://localhost:3000/auth/callback'],
    response_types: ['code']
  });
}

app.get('/auth/login', (req, res) => {
  const nonce = generators.nonce();
  const state = generators.state();
  const codeVerifier = generators.codeVerifier();
  const codeChallenge = generators.codeChallenge(codeVerifier);

  req.session.oidc = { nonce, state, codeVerifier };

  const authUrl = oidcClient.authorizationUrl({
    scope: 'openid profile email',
    state: state,
    nonce: nonce,
    code_challenge: codeChallenge,
    code_challenge_method: 'S256'
  });

  res.redirect(authUrl);
});

app.get('/auth/callback', async (req, res) => {
  const { nonce, state, codeVerifier } = req.session.oidc;
  delete req.session.oidc;

  const params = oidcClient.callbackParams(req);

  // This handles code exchange, token validation, and nonce checking
  const tokenSet = await oidcClient.callback(
    'http://localhost:3000/auth/callback',
    params,
    { nonce, state, code_verifier: codeVerifier }
  );

  // tokenSet.id_token - the raw JWT
  // tokenSet.claims() - decoded and validated claims
  const claims = tokenSet.claims();

  req.session.user = {
    sub: claims.sub,
    email: claims.email,
    name: claims.name
  };

  // Fetch additional user info
  const userInfo = await oidcClient.userinfo(tokenSet.access_token);
  console.log('Full user info:', userInfo);

  res.redirect('/dashboard');
});

initializeOIDC().then(() => {
  app.listen(3000, () => console.log('Server running on port 3000'));
});

Multi-Provider Authentication

Real applications often support multiple login providers. Here is a pattern for handling that.

// multi-provider-auth.js
class OIDCProviderRegistry {
  constructor() {
    this.providers = new Map();
  }

  register(name, config) {
    this.providers.set(name, config);
  }

  get(name) {
    const provider = this.providers.get(name);
    if (!provider) throw new Error(`Unknown provider: ${name}`);
    return provider;
  }

  listProviders() {
    return Array.from(this.providers.keys());
  }
}

const registry = new OIDCProviderRegistry();

registry.register('google', {
  issuer: 'https://accounts.google.com',
  clientId: process.env.GOOGLE_CLIENT_ID,
  clientSecret: process.env.GOOGLE_CLIENT_SECRET,
  scopes: ['openid', 'profile', 'email']
});

registry.register('github', {
  // GitHub supports OAuth2 but not full OIDC
  authorizationUrl: 'https://github.com/login/oauth/authorize',
  tokenUrl: 'https://github.com/login/oauth/access_token',
  userInfoUrl: 'https://api.github.com/user',
  clientId: process.env.GITHUB_CLIENT_ID,
  clientSecret: process.env.GITHUB_CLIENT_SECRET,
  scopes: ['read:user', 'user:email']
});

// Dynamic login route
app.get('/auth/:provider', (req, res) => {
  const provider = registry.get(req.params.provider);
  // Build and redirect to the provider's auth URL...
});

// Dynamic callback route
app.get('/auth/:provider/callback', async (req, res) => {
  const provider = registry.get(req.params.provider);
  // Handle the callback for this specific provider...
});

Account Linking

When a user logs in with different providers, you need to link those identities to a single account in your database.

// account-linking.js
async function findOrCreateUser(providerName, providerProfile) {
  // 1. Check if this provider identity already exists
  let identity = await db.identity.findUnique({
    where: {
      provider_providerId: {
        provider: providerName,
        providerId: providerProfile.sub
      }
    },
    include: { user: true }
  });

  if (identity) {
    return identity.user; // Existing user, known identity
  }

  // 2. Check if a user with this email already exists
  if (providerProfile.email && providerProfile.email_verified) {
    const existingUser = await db.user.findUnique({
      where: { email: providerProfile.email }
    });

    if (existingUser) {
      // Link this new provider to the existing account
      await db.identity.create({
        data: {
          provider: providerName,
          providerId: providerProfile.sub,
          userId: existingUser.id,
          profile: providerProfile
        }
      });
      return existingUser;
    }
  }

  // 3. Create a brand new user
  const newUser = await db.user.create({
    data: {
      email: providerProfile.email,
      name: providerProfile.name,
      identities: {
        create: {
          provider: providerName,
          providerId: providerProfile.sub,
          profile: providerProfile
        }
      }
    }
  });

  return newUser;
}

The Nonce Parameter

The nonce prevents replay attacks where an attacker captures an ID token and replays it.

Flow without nonce:
  Attacker captures: id_token = eyJhbGci...
  Attacker replays id_token to your callback
  Your server accepts it (no way to know it is replayed)

Flow with nonce:
  Your server generates nonce = "abc123" and stores in session
  Auth server includes nonce: "abc123" in the ID token
  Your server verifies id_token.nonce === session.nonce
  Attacker replays id_token but session has a DIFFERENT nonce
  Replay is detected and rejected

Session Management and Logout

OIDC defines several logout mechanisms.

// Local logout: only clears your application's session
app.post('/auth/logout', (req, res) => {
  req.session.destroy();
  res.redirect('/');
});

// Federated logout: also logs out of the identity provider
app.post('/auth/logout/full', (req, res) => {
  const idToken = req.session.idToken;
  req.session.destroy();

  // Redirect to the provider's logout endpoint
  const params = new URLSearchParams({
    id_token_hint: idToken,
    post_logout_redirect_uri: 'http://localhost:3000'
  });

  res.redirect(`https://accounts.google.com/o/oauth2/logout?${params}`);
});

Common OIDC Pitfalls

  1. Confusing OAuth2 with authentication - An OAuth2 access token does NOT prove identity. You need the ID token from OIDC.
  2. Not validating the ID token - Always verify the signature, issuer, audience, and expiration. Never just base64-decode it.
  3. Using unverified emails for account linking - Only link accounts based on email if email_verified is true. Otherwise, an attacker could create a provider account with your email.
  4. Hardcoding provider URLs - Use the OIDC discovery document to dynamically fetch endpoints.
  5. Forgetting the nonce - Without it, your app is vulnerable to token replay attacks.

Common Problems & Solutions

1 ID token validation fails with 'invalid signature' even though the token looks correct
Cause: The JWKS keys have been rotated by the provider but your application is using a cached old key. Or you are using the wrong key because the 'kid' (Key ID) in the token header does not match any cached key.
Solution: Ensure your JWKS client has a cache expiry (10-15 minutes). When a 'kid' is not found in cache, fetch the JWKS again. The jwks-rsa library handles this with the `cache` and `cacheMaxAge` options. Also verify you are fetching keys from the correct JWKS URI for your provider.
2 User gets a different 'sub' claim when logging into staging vs production
Cause: Each OAuth2 client registration (client_id) at the provider may receive a different 'sub' value for the same user, depending on the provider's policy. Or you are hitting different provider environments (sandbox vs production).
Solution: Use the same client_id across environments where possible, or use email (if verified) as the stable identifier for linking. Document your provider's 'sub' stability guarantees. Some providers guarantee sub is stable per client, others per tenant.
3 Account linking creates duplicate accounts for the same person
Cause: The user logged in with Google (email: alice@gmail.com) and later with GitHub (email: alice@gmail.com) but the linking logic did not match them because the GitHub email was not verified, or the emails were compared case-sensitively.
Solution: Normalize emails to lowercase before comparison. Only auto-link when email_verified is true from the provider. For unverified emails, prompt the user to manually link accounts by logging into both providers in the same session. Add a unique constraint on the email column.
