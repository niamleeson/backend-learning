Worker Threads: CPU-Bound Tasks in Node.js

Intuition

Worker threads are like hiring assistant cooks (threads) in the same kitchen (process). They share the pantry (memory) but work at their own stations. The head chef (main thread) delegates the heavy chopping (CPU work) to assistants, staying free to take new orders (handle requests). A worker pool is pre-hiring a fixed team rather than calling temp workers for each dish.

Worker Threads: CPU-Bound Tasks in Node.js

Introduction

While the cluster module creates separate processes for handling concurrent requests, worker_threads creates threads within a single process for offloading CPU-intensive work. This is the correct solution when you need to run expensive computations (encryption, image processing, data parsing) without blocking the main event loop.

Cluster vs Worker Threads

Cluster Module:                    Worker Threads:
- Separate processes               - Threads in same process
- Each has own V8 heap              - Can share memory (SharedArrayBuffer)
- IPC for communication             - MessagePort for communication
- For scaling HTTP servers           - For CPU-intensive tasks
- Higher memory overhead             - Lower memory overhead

Use cluster for:                   Use worker_threads for:
  Multiple HTTP listeners            Image/video processing
  Multi-core HTTP serving             JSON parsing large files
  Process isolation                   Cryptographic operations
                                      Data compression
                                      Machine learning inference

Basic Worker Thread

// main.js
import { Worker } from 'node:worker_threads';

function runWorker(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./worker.js', {
      workerData: data,
    });

    worker.on('message', resolve);
    worker.on('error', reject);
    worker.on('exit', (code) => {
      if (code !== 0) {
        reject(new Error(`Worker exited with code ${code}`));
      }
    });
  });
}

// Usage in Express
app.post('/api/process-data', async (req, res) => {
  try {
    const result = await runWorker(req.body);
    res.json(result);
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

// worker.js
import { workerData, parentPort } from 'node:worker_threads';

Worker Thread Pool

Creating a new thread per request is expensive. Use a pool:

import { Worker } from 'node:worker_threads';
import os from 'node:os';

class WorkerPool {
  constructor(workerPath, poolSize = os.cpus().length) {
    this.workerPath = workerPath;
    this.poolSize = poolSize;
    this.workers = [];
    this.freeWorkers = [];
    this.taskQueue = [];

    for (let i = 0; i < poolSize; i++) {
      this._addWorker();
    }
  }

  _addWorker() {
    const worker = new Worker(this.workerPath);

    worker.on('message', (result) => {
      // Resolve the pending task
      worker._resolve(result);
      worker._resolve = null;
      worker._reject = null;

      // Check for queued tasks
      if (this.taskQueue.length > 0) {
        const { data, resolve, reject } = this.taskQueue.shift();
        this._runTask(worker, data, resolve, reject);
      } else {
        this.freeWorkers.push(worker);
      }
    });

    worker.on('error', (err) => {
      if (worker._reject) {
        worker._reject(err);
      }
      // Replace dead worker
      this.workers = this.workers.filter(w => w !== worker);
      this._addWorker();
    });

    this.workers.push(worker);
    this.freeWorkers.push(worker);
  }

  _runTask(worker, data, resolve, reject) {
    worker._resolve = resolve;
    worker._reject = reject;
    worker.postMessage(data);
  }

  execute(data) {
    return new Promise((resolve, reject) => {
      const freeWorker = this.freeWorkers.pop();

      if (freeWorker) {
        this._runTask(freeWorker, data, resolve, reject);
      } else {
        this.taskQueue.push({ data, resolve, reject });
      }
    });
  }

  async destroy() {
    for (const worker of this.workers) {
      await worker.terminate();
    }
  }

  get stats() {
    return {
      totalWorkers: this.workers.length,
      freeWorkers: this.freeWorkers.length,
      busyWorkers: this.workers.length - this.freeWorkers.length,
      queuedTasks: this.taskQueue.length,
    };
  }
}

// Usage
const pool = new WorkerPool('./hash-worker.js', 4);

app.post('/api/hash', async (req, res) => {
  const result = await pool.execute({
    algorithm: 'bcrypt',
    data: req.body.password,
    rounds: 12,
  });
  res.json(result);
});

app.get('/api/pool-stats', (req, res) => {
  res.json(pool.stats);
});

// Cleanup on shutdown
process.on('SIGTERM', () => pool.destroy());

Shared Memory with SharedArrayBuffer

For high-performance scenarios, threads can share memory:

// main.js
import { Worker } from 'node:worker_threads';

// Create shared buffer
const sharedBuffer = new SharedArrayBuffer(1024 * 1024); // 1MB
const sharedArray = new Int32Array(sharedBuffer);

// Initialize data
for (let i = 0; i < sharedArray.length; i++) {
  sharedArray[i] = i;
}

// Spawn workers that operate on shared memory
const workers = [];
const chunkSize = Math.ceil(sharedArray.length / 4);

for (let i = 0; i < 4; i++) {
  const worker = new Worker('./sum-worker.js', {
    workerData: {
      sharedBuffer,
      start: i * chunkSize,
      end: Math.min((i + 1) * chunkSize, sharedArray.length),
    },
  });
  workers.push(new Promise(resolve => worker.on('message', resolve)));
}

const partialSums = await Promise.all(workers);
const totalSum = partialSums.reduce((a, b) => a + b, 0);
console.log('Total sum:', totalSum);

// sum-worker.js
import { workerData, parentPort } from 'node:worker_threads';

const { sharedBuffer, start, end } = workerData;
const arr = new Int32Array(sharedBuffer);

let sum = 0;
for (let i = start; i < end; i++) {
  sum += arr[i];
}

parentPort.postMessage(sum);

Using Atomics for Thread-Safe Operations

import { Worker, isMainThread, workerData, parentPort } from 'node:worker_threads';

if (isMainThread) {
  const sharedBuffer = new SharedArrayBuffer(4);
  const counter = new Int32Array(sharedBuffer);

  const workers = Array.from({ length: 4 }, () =>
    new Worker(import.meta.filename, {
      workerData: { sharedBuffer, iterations: 100000 },
    })
  );

  await Promise.all(workers.map(w => new Promise(r => w.on('exit', r))));

  console.log('Counter value:', counter[0]);
  // Should be exactly 400000 with Atomics
  // Would be less without Atomics due to race conditions
} else {
  const counter = new Int32Array(workerData.sharedBuffer);

  for (let i = 0; i < workerData.iterations; i++) {
    Atomics.add(counter, 0, 1); // Thread-safe increment
  }
}

Real-World Use Cases

Image Processing

// image-worker.js
import { parentPort } from 'node:worker_threads';
import sharp from 'sharp';

parentPort.on('message', async ({ imagePath, operations }) => {
  let pipeline = sharp(imagePath);

  for (const op of operations) {
    switch (op.type) {
      case 'resize':
        pipeline = pipeline.resize(op.width, op.height);
        break;
      case 'format':
        pipeline = pipeline.toFormat(op.format, { quality: op.quality });
        break;
      case 'blur':
        pipeline = pipeline.blur(op.sigma);
        break;
    }
  }

  const buffer = await pipeline.toBuffer();
  parentPort.postMessage({ buffer, size: buffer.length });
});

JSON Parsing of Large Files

// parse-worker.js
import { parentPort } from 'node:worker_threads';
import { createReadStream } from 'node:fs';
import { parser } from 'stream-json';
import { streamArray } from 'stream-json/streamers/StreamArray.js';

parentPort.on('message', async ({ filePath, filter }) => {
  const results = [];
  const pipeline = createReadStream(filePath)
    .pipe(parser())
    .pipe(streamArray());

  for await (const { value } of pipeline) {
    if (matchesFilter(value, filter)) {
      results.push(value);
    }
  }

  parentPort.postMessage({ count: results.length, results });
});

Common Pitfalls

  1. Using worker threads for I/O: Worker threads are for CPU-bound work. I/O is already non-blocking via the event loop. Don't use workers for database queries or HTTP requests.
  2. Transferring large data via postMessage: Each postMessage serializes and copies data. For large data, use SharedArrayBuffer or transferable objects.
  3. Not pooling workers: Creating a new worker per request is expensive (~50ms startup). Always use a pool.
  4. Deadlocks with Atomics.wait: On the main thread, Atomics.wait blocks the event loop. Only use it in worker threads.

Summary

Worker threads enable CPU-intensive operations in Node.js without blocking the event loop. Use them for image processing, data parsing, cryptography, and other compute-heavy tasks. Always use a worker pool (custom or Piscina) to amortize thread creation costs. Use SharedArrayBuffer for zero-copy data sharing between threads, and Atomics for thread-safe operations.

    Hint

    Track free/busy workers. Queue tasks as Promises. On worker message, resolve the pending task and pick next from queue.

  2. Password Hashing Service

    Hint

    Use a worker pool with 4 threads. Compare with autocannon: main thread version will block under load.

  3. Parallel Data Processing

    Split a large array into chunks and process each chunk in a separate worker thread using SharedArrayBuffer. Combine results on the main thread.

    Hint

    Create a SharedArrayBuffer, divide into chunks by index range, spawn workers for each chunk, aggregate results.

Common Problems & Solutions

1 Worker thread creation is slow under load
Cause: Creating a new Worker for each request has ~50ms overhead for V8 initialization.
Solution: Use a worker pool (Piscina or custom). Pre-create workers at startup and reuse them for subsequent tasks.
2 Large data transfer between main and worker is slow
Cause: postMessage serializes (structured clone) the data, which is slow for large objects.
Solution: Use SharedArrayBuffer for large numeric data. Use transferable objects (ArrayBuffer, MessagePort) to transfer ownership without copying.

worker.postMessage({ buffer }, [buffer]); // Transfer, not copy

3 Worker thread causes main thread to hang
Cause: Using Atomics.wait() on the main thread blocks the event loop.
Solution: Never use Atomics.wait() on the main thread. Use Atomics.waitAsync() instead, or use message passing for synchronization.
