Design a Rate Limiter

Intuition

A rate limiter is like a bouncer at a nightclub with a clicker counter. The bouncer counts everyone entering in the last hour. Once the count hits capacity, new arrivals are turned away with a polite 'please come back later.' The bouncer does not care who you are by default, but VIP list members (premium API keys) get a higher personal limit.

Design a Rate Limiter

Introduction

A rate limiter controls how many requests a client can make to an API within a given time window. It protects backend services from being overwhelmed by excessive traffic, whether from legitimate heavy users, bugs in client code, or malicious denial-of-service attacks. Rate limiting is a fundamental building block of any production API and a favorite system design interview topic.

Step 1: Requirements Gathering

Functional Requirements

  1. Limit the number of requests a client can make in a configurable time window
  2. Support multiple rate limiting rules (e.g., 100 requests/minute per user, 1000 requests/hour per IP)
  3. Return informative headers telling clients their limit, remaining quota, and reset time
  4. Support different limiting strategies: fixed window, sliding window, token bucket
  5. Configurable per-route and per-user limits

Non-Functional Requirements

  1. Very low latency -- the rate limiter sits in the critical path of every request (<1ms overhead)
  2. Distributed -- must work correctly across multiple API server instances
  3. Highly available -- if the rate limiter fails, the system should degrade gracefully (allow traffic)
  4. Accurate -- should not allow significantly more requests than the configured limit

Step 2: Capacity Estimation

Traffic Estimates

  • API traffic: 50,000 requests/second across all endpoints
  • Unique clients (by user ID or IP): 10M active per day
  • Rate limit rules to evaluate per request: 2-3 rules
  • Rate limit checks per second: 50,000 x 3 = 150,000 Redis operations/second

Storage Estimates

  • Per client rate limit state: ~100 bytes (counter, timestamps)
  • Active clients in a 1-hour window: ~2M
  • Memory needed: 2M x 100 bytes x 3 rules = 600 MB
  • Fits comfortably in a single Redis instance

Step 3: Rate Limiting Algorithms

Algorithm 1: Fixed Window Counter

Time:    |-------- Window 1 --------|-------- Window 2 --------|
         0:00                       1:00                       2:00

Requests: |||||||||||                 |||||||
          (11 requests)               (7 requests)

Limit: 10 per minute
Result: 11th request in Window 1 is REJECTED

async function fixedWindowCheck(clientId, limit, windowSizeMs) {
  const redis = getRedisClient();
  const windowKey = Math.floor(Date.now() / windowSizeMs);
  const key = `ratelimit:${clientId}:${windowKey}`;

  const current = await redis.incr(key);
  if (current === 1) {
    // Set expiry on first request in window
    await redis.pexpire(key, windowSizeMs);
  }

  return {
    allowed: current <= limit,
    remaining: Math.max(0, limit - current),
    resetAt: (windowKey + 1) * windowSizeMs
  };
}
  • Pros: Simple, low memory
  • Cons: Boundary burst problem -- a client can send 2x the limit at the boundary of two windows

Algorithm 2: Sliding Window Log

Time:    .......|====== sliding 60s window ======|...
                |  (counts all requests in this  |
                |   sliding window of time)      |

Each request timestamp is stored. On each new request,
remove timestamps older than (now - windowSize) and count.

async function slidingWindowLogCheck(clientId, limit, windowSizeMs) {
  const redis = getRedisClient();
  const key = `ratelimit:${clientId}`;
  const now = Date.now();
  const windowStart = now - windowSizeMs;

  const pipeline = redis.pipeline();
  // Remove expired entries
  pipeline.zremrangebyscore(key, 0, windowStart);
  // Add current request
  pipeline.zadd(key, now, `${now}:${Math.random()}`);
  // Count requests in window
  pipeline.zcard(key);
  // Set expiry on the whole key
  pipeline.pexpire(key, windowSizeMs);

  const results = await pipeline.exec();
  const count = results[2][1];

  return {
    allowed: count <= limit,
    remaining: Math.max(0, limit - count),
    resetAt: now + windowSizeMs
  };
}
  • Pros: Precise, no boundary burst problem
  • Cons: High memory usage (stores every request timestamp)

Algorithm 3: Sliding Window Counter (Recommended)

A hybrid that approximates the sliding window using two fixed windows:

Previous Window    Current Window
|--- 60s ---|--- 60s ---|
  (8 reqs)   (5 reqs so far)
                   ^
                   now (30s into current window)

Weighted count = 8 * (30/60) + 5 = 4 + 5 = 9
If limit is 10, this request is ALLOWED (9 < 10)

async function slidingWindowCounterCheck(clientId, limit, windowSizeMs) {
  const redis = getRedisClient();
  const now = Date.now();
  const currentWindow = Math.floor(now / windowSizeMs);
  const previousWindow = currentWindow - 1;

  const currentKey = `ratelimit:${clientId}:${currentWindow}`;
  const previousKey = `ratelimit:${clientId}:${previousWindow}`;

  const [previousCount, currentCount] = await Promise.all([
    redis.get(previousKey).then(v => parseInt(v) || 0),
    redis.get(currentKey).then(v => parseInt(v) || 0)
  ]);

  // Weight the previous window by how much of it overlaps
  const elapsedInCurrentWindow = now % windowSizeMs;
  const previousWeight = 1 - (elapsedInCurrentWindow / windowSizeMs);
  const estimatedCount = Math.floor(previousCount * previousWeight) + currentCount;

  if (estimatedCount >= limit) {
    return { allowed: false, remaining: 0, resetAt: (currentWindow + 1) * windowSizeMs };
  }

  // Increment current window counter
  const pipeline = redis.pipeline();
  pipeline.incr(currentKey);
  pipeline.pexpire(currentKey, windowSizeMs * 2);
  await pipeline.exec();

  return {
    allowed: true,
    remaining: limit - estimatedCount - 1,
    resetAt: (currentWindow + 1) * windowSizeMs
  };
}

Algorithm 4: Token Bucket

Bucket Capacity: 10 tokens
Refill Rate: 1 token per second

[||||||||||]  <- Full bucket (10 tokens)

Request arrives -> consume 1 token
[|||||||||.]  <- 9 tokens remaining

3 more requests arrive
[||||||....]  <- 6 tokens remaining

4 seconds pass -> 4 tokens refilled
[||||||||||]  <- Full again (capped at 10)

async function tokenBucketCheck(clientId, bucketSize, refillRate) {
  const redis = getRedisClient();
  const key = `ratelimit:bucket:${clientId}`;

  // Lua script for atomic token bucket operation
  const luaScript = `
    local key = KEYS[1]
    local bucket_size = tonumber(ARGV[1])
    local refill_rate = tonumber(ARGV[2])
    local now = tonumber(ARGV[3])

    local bucket = redis.call('hmget', key, 'tokens', 'last_refill')
    local tokens = tonumber(bucket[1]) or bucket_size
    local last_refill = tonumber(bucket[2]) or now

    -- Calculate tokens to add based on elapsed time
    local elapsed = now - last_refill
    local new_tokens = math.min(bucket_size, tokens + (elapsed * refill_rate / 1000))

    if new_tokens < 1 then
      return {0, math.ceil((1 - new_tokens) * 1000 / refill_rate)}
    end

    new_tokens = new_tokens - 1
    redis.call('hmset', key, 'tokens', new_tokens, 'last_refill', now)
    redis.call('pexpire', key, math.ceil(bucket_size / refill_rate * 1000) * 2)

    return {1, 0}
  `;

  const [allowed, retryAfterMs] = await redis.eval(
    luaScript, 1, key, bucketSize, refillRate, Date.now()
  );

  return { allowed: allowed === 1, retryAfterMs };
}
  • Pros: Smooth rate, allows short bursts, widely used (AWS, Stripe)
  • Cons: More complex, parameters (bucket size + refill rate) less intuitive than "N per window"

Where to Place the Rate Limiter

  1. API Gateway level (e.g., Kong, AWS API Gateway): Best for global limits, applied before your code runs
  2. Middleware level (Express middleware): Best for per-route, per-user rules
  3. Service mesh (Istio, Envoy): Best for inter-service rate limiting in microservices

For most Node.js APIs, an Express middleware backed by Redis is the right approach.

Step 5: Express Middleware Implementation

import Redis from 'ioredis';

const redis = new Redis({ host: 'redis-cluster', port: 6379 });

function rateLimiter(options = {}) {
  const {
    windowMs = 60 * 1000,      // 1 minute
    max = 100,                  // 100 requests per window
    keyGenerator = (req) => req.ip,
    handler = (req, res) => {
      res.status(429).json({
        error: 'Too Many Requests',
        retryAfter: res.getHeader('Retry-After')
      });
    },
    skipFailedRequests = false,
    skip = () => false
  } = options;

  return async (req, res, next) => {
    if (skip(req)) return next();

    const clientKey = keyGenerator(req);
    const result = await slidingWindowCounterCheck(clientKey, max, windowMs);

    // Always set rate limit headers
    res.setHeader('X-RateLimit-Limit', max);
    res.setHeader('X-RateLimit-Remaining', result.remaining);
    res.setHeader('X-RateLimit-Reset', Math.ceil(result.resetAt / 1000));

    if (!result.allowed) {
      const retryAfter = Math.ceil((result.resetAt - Date.now()) / 1000);
      res.setHeader('Retry-After', retryAfter);
      return handler(req, res);
    }

    next();
  };
}

// Usage
const app = express();

// Global: 1000 requests per minute per IP
app.use(rateLimiter({ max: 1000, windowMs: 60000 }));

// Per-route: 10 login attempts per 15 minutes per IP
app.post('/auth/login', rateLimiter({
  max: 10,
  windowMs: 15 * 60 * 1000,
  keyGenerator: (req) => `login:${req.ip}`
}), loginHandler);

// Per-user: 50 API calls per minute per authenticated user
app.use('/api', rateLimiter({
  max: 50,
  windowMs: 60000,
  keyGenerator: (req) => `user:${req.user?.id || req.ip}`
}));

Step 6: Database Schema for Rate Limit Rules

-- Rate limit rules (configurable without code changes)
CREATE TABLE rate_limit_rules (
  id            SERIAL PRIMARY KEY,
  name          VARCHAR(100) NOT NULL,
  route_pattern VARCHAR(200),           -- e.g., '/api/*' or '/auth/login'
  key_type      VARCHAR(20) NOT NULL,   -- 'ip', 'user', 'api_key'
  max_requests  INT NOT NULL,
  window_ms     BIGINT NOT NULL,
  algorithm     VARCHAR(30) DEFAULT 'sliding_window',
  enabled       BOOLEAN DEFAULT true,
  created_at    TIMESTAMP DEFAULT NOW()
);

-- Rate limit overrides for specific clients (premium users, partners)
CREATE TABLE rate_limit_overrides (
  id            SERIAL PRIMARY KEY,
  client_id     VARCHAR(100) NOT NULL,  -- user ID or API key
  rule_id       INT REFERENCES rate_limit_rules(id),
  max_requests  INT NOT NULL,           -- overridden limit
  expires_at    TIMESTAMP
);

Step 7: Scaling Strategies

Distributed Rate Limiting with Redis Cluster

When you have multiple API servers, all rate limit state must be shared. Redis is the standard solution:

  API Server 1 ---+
  API Server 2 ---+--->  Redis Cluster  (shared state)
  API Server 3 ---+

Use Redis Cluster mode for high availability:

const redis = new Redis.Cluster([
  { host: 'redis-1', port: 6379 },
  { host: 'redis-2', port: 6379 },
  { host: 'redis-3', port: 6379 }
]);

Graceful Degradation

If Redis is down, do not block all traffic:

async function rateLimitCheck(clientKey, limit, windowMs) {
  try {
    return await slidingWindowCounterCheck(clientKey, limit, windowMs);
  } catch (error) {
    // Redis is down -- fail open (allow request)
    console.error('Rate limiter Redis error:', error.message);
    return { allowed: true, remaining: limit, resetAt: Date.now() + windowMs };
  }
}

Local + Distributed Hybrid

For ultra-low latency, use a two-tier approach:

  1. Local in-memory counter (per server): catches obvious abusers without network round-trip
  2. Redis distributed counter: provides global accuracy
const localCounters = new Map();

function hybridRateLimitCheck(clientKey, localLimit, globalLimit, windowMs) {
  // Tier 1: Fast local check
  const localKey = `${clientKey}:${Math.floor(Date.now() / windowMs)}`;
  const localCount = (localCounters.get(localKey) || 0) + 1;
  localCounters.set(localKey, localCount);

  // If local limit exceeded, reject immediately (no Redis call)
  if (localCount > localLimit) {
    return { allowed: false };
  }

  // Tier 2: Distributed check via Redis
  return slidingWindowCounterCheck(clientKey, globalLimit, windowMs);
}

Decision Option A Option B Recommendation
Algorithm Fixed window Sliding window counter Sliding window counter (best accuracy/memory balance)
Storage Redis In-memory per server Redis (distributed correctness)
Failure Mode Fail closed (block) Fail open (allow) Fail open (availability over protection)
Granularity Per IP Per user + per IP Both (layered rules)

Race Conditions in Distributed Rate Limiting

With multiple Redis operations (read count, check limit, increment), race conditions can let extra requests through. The solution is atomic operations via Lua scripts, which Redis executes as a single unit:

// Atomic sliding window counter in Lua
const atomicCheck = `
  local key = KEYS[1]
  local limit = tonumber(ARGV[1])
  local window = tonumber(ARGV[2])
  local now = tonumber(ARGV[3])

  local current = tonumber(redis.call('get', key) or '0')
  if current >= limit then
    return {0, current}
  end

  current = redis.call('incr', key)
  if current == 1 then
    redis.call('pexpire', key, window)
  end
  return {1, current}
`;

Summary

A rate limiter is a critical infrastructure component that protects APIs from abuse and ensures fair resource allocation. The sliding window counter algorithm offers the best balance of accuracy and memory efficiency. In a distributed environment, use Redis with Lua scripts for atomic operations, implement fail-open behavior for resilience, and expose rate limit information via standard HTTP headers. Layer multiple rules (global, per-route, per-user) and provide overrides for premium clients.

Exercises

  1. Implement Fixed Window Rate Limiter

    Build an Express middleware that implements fixed window rate limiting using Redis. It should take configuration for max requests and window size, generate keys based on client IP, set appropriate X-RateLimit-* headers on every response, and return 429 with a Retry-After header when the limit is exceeded. Test it with a loop of rapid requests.

    Hint

    Use redis.incr() for atomic increment and redis.pexpire() for window expiry. The window key should include Math.floor(Date.now() / windowMs) to create discrete time buckets.

  2. Token Bucket with Lua Script

    Implement the token bucket algorithm using a Redis Lua script for atomicity. The bucket should have a configurable capacity and refill rate. Tokens are consumed on each request and replenished based on elapsed time. Write a test that sends a burst of requests, verifies the burst is allowed up to bucket capacity, and then verifies subsequent requests are throttled until tokens refill.

    Hint

    Store tokens and last_refill_time in a Redis hash. In the Lua script, calculate elapsed time, add refilled tokens (capped at bucket_size), then try to consume one token. Use redis.eval() to run the script.

  3. Multi-Tier Rate Limiting

    Create a rate limiting system with three tiers: global per-IP limit (1000/min), per-route limit (100/min for /api/*, 10/15min for /auth/login), and per-user limit (50/min for authenticated users). Load rules from a database table and apply them as Express middleware. Ensure the most restrictive limit wins.

    Hint

    Create a middleware factory that accepts an array of rules. For each request, evaluate all matching rules and reject if any one is exceeded. Use route pattern matching (e.g., minimatch or path-to-regexp) to match rules to routes.

Common Problems & Solutions

1 Rate limiter allows double the configured limit at window boundaries
Cause: The fixed window counter algorithm resets at exact window boundaries. A client can send max requests at the end of one window and max more at the start of the next, resulting in 2x the limit in a short period.
Solution: Switch to the sliding window counter algorithm, which weights the previous window's count by how much of it overlaps with the current sliding window. This smooths out the boundary burst problem.
2 Rate limiter blocks all traffic when Redis goes down
Cause: The middleware throws an unhandled exception when the Redis connection fails, causing Express to return 500 errors for every request.
Solution: Wrap Redis calls in try/catch and implement fail-open behavior: if Redis is unreachable, allow the request through. Log the error for alerting. Optionally, fall back to an in-memory counter as a degraded mode.
3 Race condition allows more requests than the limit under high concurrency
Cause: The check-then-increment pattern (read count, compare to limit, then increment) is not atomic. Multiple concurrent requests can read the same count before any of them increment it.
Solution: Use Redis Lua scripts to make the check-and-increment operation atomic. Redis executes Lua scripts as a single, uninterruptible operation, eliminating race conditions.
