Sharding Strategies: Hash, Range, and Directory

Intuition

Sharding is like splitting a library into multiple branches by subject. Hash sharding assigns books by a formula (unpredictable but even). Range sharding assigns by alphabet (predictable but some letters are busier). Directory sharding keeps a master catalog telling you which branch has which book.

Sharding Strategies: Hash, Range & Directory

Sharding horizontally partitions data across multiple database instances, allowing systems to scale beyond the capacity of a single server. The sharding strategy determines how data is distributed and accessed.

Core Concepts

  • Hash-Based Sharding: Apply a hash function to the shard key (e.g., user_id % N). Provides even distribution but makes range queries difficult and resharding painful.
  • Range-Based Sharding: Partition data by contiguous ranges of the shard key (e.g., users A-M on shard 1, N-Z on shard 2). Supports range queries but risks hot spots if data distribution is uneven.
  • Directory-Based Sharding: A lookup table maps each shard key to its shard. Maximally flexible but introduces the directory as a single point of failure and potential bottleneck.
  • Consistent Hashing: A hash-based approach that minimizes data movement when shards are added or removed. Used by distributed caches and databases like Cassandra.
  • Shard Key Selection: The most critical decision. A good shard key has high cardinality, even distribution, and is present in most queries to avoid cross-shard joins.

Key Takeaways

  1. Choose your shard key carefully—it is nearly impossible to change later without a major data migration.
  2. Hash sharding is the default for even distribution; range sharding is better for time-series or alphabetical access patterns.
  3. Cross-shard queries and joins are expensive; design your data model to keep related data on the same shard.

Exercises

  1. Implement Application-Level Hash Sharding

    Create a Python class that manages connections to 4 PostgreSQL instances. Implement hash-based routing using hash(user_id) % 4 to determine the target shard. Write functions for insert, read, and cross-shard aggregation queries.

    Hint

    Use a dictionary of database connections keyed by shard index. For cross-shard queries, fan out to all shards in parallel using concurrent.futures.ThreadPoolExecutor and merge results.

  2. Build a Directory-Based Shard Router

    Implement a directory service backed by Redis that maps tenant IDs to shard identifiers. Support shard migrations by updating the directory and provide a migration tool that moves a tenant's data between shards.

    Hint

    Use redis.hset('shard_directory', tenant_id, shard_id). For migration, copy data to the new shard, update the directory atomically, then clean up the old shard.

Common Problems & Solutions

1 One shard receives significantly more traffic than others, causing performance degradation.
Cause: The shard key has low cardinality or skewed distribution (e.g., sharding by country where one country dominates).
Solution: Choose a higher-cardinality shard key, add a composite key to improve distribution, or implement shard splitting to subdivide the hot shard into smaller partitions.
2 Adding a new shard requires migrating a large amount of data and causes downtime.
Cause: Simple modulo-based hash sharding (`key % N`) remaps most keys when N changes.
Solution: Use consistent hashing to minimize data movement. Alternatively, pre-provision more shards than currently needed (virtual shards) and map multiple virtual shards to each physical server.
