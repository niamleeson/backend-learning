Design a Video Streaming Platform (YouTube)

Intuition

Think of YouTube like a TV broadcast network. When a creator uploads a raw tape, it goes to the editing room (transcoding pipeline) where copies are made for different screen sizes. These copies are shipped to local TV stations (CDN edges) around the world. When a viewer tunes in, they get the feed from their nearest station, and the station automatically adjusts the picture quality based on the viewer's signal strength (adaptive bitrate).

Design a Video Streaming Platform (YouTube)

Introduction

A video streaming platform is one of the most bandwidth-intensive systems you can design. YouTube serves over 1 billion hours of video per day. This problem tests your understanding of video encoding pipelines, adaptive bitrate streaming, content delivery networks, and storage optimization at massive scale.

Step 1: Requirements Gathering

Functional Requirements

  • Upload videos: Creators upload videos (up to 12 hours, any common format)
  • Transcode videos: Convert to multiple resolutions and codecs
  • Stream videos: Viewers watch with adaptive bitrate streaming
  • Search and discovery: Search by title, tags; recommendation feed
  • Engagement: Like, comment, subscribe, view count
  • Channels: Creator profiles with video management

Non-Functional Requirements

  • Availability: 99.99% for video playback
  • Latency: Video playback starts within 2 seconds
  • Scalability: 2B monthly active users, 500 hours of video uploaded per minute
  • Durability: Zero data loss for uploaded videos
  • Global reach: Low latency across all continents

Step 4: API Design

import express from 'express';
const router = express.Router();

// Initiate video upload
router.post('/api/v1/videos/upload', authenticate, async (req, res) => {
  const { title, description, tags, category } = req.body;

  const video = await videoService.create({
    creatorId: req.user.id,
    title,
    description,
    tags,
    category,
    status: 'uploading',
  });

  // Generate presigned URL for direct upload to S3
  const uploadUrl = await storageService.getPresignedUploadUrl(
    `raw/${video.id}/original`
  );

  res.status(201).json({
    videoId: video.id,
    uploadUrl,
    expiresIn: 3600,
  });
});

// Callback after upload complete -- triggers transcoding pipeline
router.post('/api/v1/videos/:videoId/uploaded', authenticate, async (req, res) => {
  const { videoId } = req.params;

  await videoService.updateStatus(videoId, 'processing');

  // Push to transcoding queue
  await transcodingQueue.add('transcode', {
    videoId,
    sourceKey: `raw/${videoId}/original`,
    resolutions: [360, 480, 720, 1080, 1440, 2160],
  });

  res.json({ status: 'processing' });
});

// Stream a video (returns manifest URL)
router.get('/api/v1/videos/:videoId/stream', async (req, res) => {
  const video = await videoService.getById(req.params.videoId);
  if (!video || video.status !== 'ready') {
    return res.status(404).json({ error: 'Video not found or not ready' });
  }

  // Increment view count asynchronously
  viewCountQueue.add('increment', { videoId: video.id, userId: req.user?.id });

  // Return HLS manifest URL from CDN
  const manifestUrl = cdnService.getUrl(
    `transcoded/${video.id}/manifest.m3u8`
  );

  res.json({
    manifestUrl,
    title: video.title,
    duration: video.duration,
    resolutions: video.availableResolutions,
  });
});

// Search videos
router.get('/api/v1/videos/search', async (req, res) => {
  const { q, category, sort, page = 1, limit = 20 } = req.query;

  const results = await searchService.search({
    query: q,
    filters: { category },
    sort: sort || 'relevance',
    from: (page - 1) * limit,
    size: limit,
  });

  res.json(results);
});

// Engagement: like a video
router.post('/api/v1/videos/:videoId/like', authenticate, async (req, res) => {
  await engagementService.like(req.params.videoId, req.user.id);
  res.json({ liked: true });
});

// Get video details with engagement stats
router.get('/api/v1/videos/:videoId', async (req, res) => {
  const video = await videoService.getById(req.params.videoId);
  const stats = await engagementService.getStats(req.params.videoId);

  res.json({ ...video, stats });
});

Step 6: Deep Dive -- Video Transcoding Pipeline

The transcoding pipeline is the most complex component. Raw uploads must be converted into multiple formats and resolutions for adaptive streaming.

Adaptive Bitrate Streaming (HLS)

HTTP Live Streaming (HLS) works by splitting videos into small segments (2-10 seconds each) and providing a manifest file that lists available quality levels:

manifest.m3u8 (Master Playlist)
  ├── 360p/playlist.m3u8  → segment_000.ts, segment_001.ts, ...
  ├── 480p/playlist.m3u8  → segment_000.ts, segment_001.ts, ...
  ├── 720p/playlist.m3u8  → segment_000.ts, segment_001.ts, ...
  └── 1080p/playlist.m3u8 → segment_000.ts, segment_001.ts, ...

The player starts at a medium quality, then adapts up or down based on bandwidth.

Transcoding Worker

import { Worker } from 'bullmq';
import { execFile } from 'node:child_process';
import { promisify } from 'node:util';

const execFileAsync = promisify(execFile);

const worker = new Worker('transcode', async (job) => {
  const { videoId, sourceKey, resolutions } = job.data;

  // 1. Download raw video from S3
  const localPath = `/tmp/${videoId}/original`;
  await storageService.download(sourceKey, localPath);

  // 2. Extract metadata
  const probe = await execFileAsync('ffprobe', [
    '-v', 'quiet',
    '-print_format', 'json',
    '-show_format',
    '-show_streams',
    localPath,
  ]);
  const metadata = JSON.parse(probe.stdout);
  const duration = Math.floor(parseFloat(metadata.format.duration));

  // 3. Generate thumbnail
  await execFileAsync('ffmpeg', [
    '-i', localPath,
    '-ss', String(Math.floor(duration * 0.25)),
    '-vframes', '1',
    `-/tmp/${videoId}/thumbnail.jpg`,
  ]);

  // 4. Transcode to each resolution
  const resolutionConfigs = {
    360:  { width: 640,  height: 360,  bitrate: '800k' },
    480:  { width: 854,  height: 480,  bitrate: '1400k' },
    720:  { width: 1280, height: 720,  bitrate: '2800k' },
    1080: { width: 1920, height: 1080, bitrate: '5000k' },
  };

  for (const res of resolutions) {
    const config = resolutionConfigs[res];
    if (!config) continue;

    const outputDir = `/tmp/${videoId}/${res}p`;
    await execFileAsync('ffmpeg', [
      '-i', localPath,
      '-vf', `scale=${config.width}:${config.height}`,
      '-b:v', config.bitrate,
      '-c:v', 'libx264',
      '-preset', 'medium',
      '-c:a', 'aac',
      '-b:a', '128k',
      '-hls_time', '6',
      '-hls_playlist_type', 'vod',
      '-hls_segment_filename', `${outputDir}/segment_%03d.ts`,
      `${outputDir}/playlist.m3u8`,
    ]);

    // Upload segments to S3
    await storageService.uploadDirectory(outputDir, `transcoded/${videoId}/${res}p`);
  }

  // 5. Generate master manifest
  const masterManifest = generateMasterPlaylist(videoId, resolutions);
  await storageService.putObject(
    `transcoded/${videoId}/manifest.m3u8`,
    masterManifest
  );

  // 6. Update video status
  await videoService.update(videoId, {
    status: 'ready',
    duration,
    resolutions,
    manifestKey: `transcoded/${videoId}/manifest.m3u8`,
  });

}, { connection: redisConnection, concurrency: 2 });

Step 7: Scaling Strategies

  1. CDN is king: 90%+ of video traffic should be served from CDN edge servers, not origin. Popular videos get cached at edges worldwide.
  2. Tiered storage: Hot videos on SSD-backed storage, cold videos on cheaper storage (S3 Glacier).
  3. View count aggregation: Do not UPDATE a counter on every view. Buffer counts in Redis and flush to PostgreSQL in batches.
  4. Shard by video ID: Metadata reads are heavily skewed (popular videos). Cache aggressively in Redis.
  5. Separate read and write paths: Upload/transcode pipeline is completely separate from the viewing pipeline.
  6. Pre-warm popular content: Push trending videos to CDN edges proactively.
View Count Pipeline (avoiding write hotspots):

Viewer ──> API ──> Redis INCR ──> Periodic Flush ──> PostgreSQL
                   (real-time)    (every 30s)        (durable)

  Redis: INCR video:{id}:views
  Flush: UPDATE videos SET view_count = view_count + $delta WHERE id = $id

Transcoding On-upload On-demand On-upload for common resolutions, on-demand for rare ones

Exercises

  1. Build a Video Transcoding Worker

    Create a BullMQ worker that takes a video file path, transcodes it to 360p and 720p using FFmpeg, splits it into HLS segments, generates a master playlist, and stores everything in a local directory. Verify playback with an HLS player.

    Hint

    Install ffmpeg locally. Use child_process.execFile to run ffmpeg commands. The HLS master playlist is a text file referencing each resolution's playlist.m3u8.

  2. Implement Buffered View Counting

    Build a system where view events are recorded in Redis using INCR and periodically flushed to PostgreSQL. Implement the flush as a scheduled BullMQ job that runs every 30 seconds. Verify that total counts in PostgreSQL match Redis after multiple flush cycles.

    Hint

    Use Redis GETSET to atomically read and reset the counter during flush. Use ioredis.pipeline() to batch reads for multiple videos.

  3. Adaptive Bitrate Simulation

    Write a Node.js simulation of an adaptive bitrate algorithm. Given an array of available bitrates and a fluctuating bandwidth (random values between 1-10 Mbps), implement the logic that selects the highest quality that fits within 80% of available bandwidth. Log quality switches over a 5-minute simulated viewing session.

    Hint

    Buffer-based ABR: if buffer > 30s, try upgrading quality; if buffer < 10s, downgrade. Bandwidth-based: use exponential moving average of recent download speeds.

Common Problems & Solutions

1 Video playback frequently buffers for users in certain regions
Cause: CDN cache miss rate is high in those regions, causing viewers to fetch segments from the origin server with high latency.
Solution: Analyze CDN cache hit rates by region. Pre-warm popular content at edge locations. Add more CDN points of presence in underserved regions. Consider a multi-CDN strategy where you route to the CDN with best performance per region.
2 Transcoding queue grows unboundedly during peak upload hours
Cause: Fixed number of transcoding workers cannot keep up with upload spikes. Some videos are very long (hours) and block workers.
Solution: Use auto-scaling worker pools (ECS/Kubernetes HPA). Prioritize shorter videos and popular creators. Split long videos into segments for parallel transcoding. Set a maximum queue depth with back-pressure that temporarily rejects new uploads.
3 View count shows different numbers on the video page vs the creator dashboard
Cause: Buffered view counting creates a delay between Redis (real-time) and PostgreSQL (flushed every 30s). Dashboard reads from PostgreSQL while the video page reads from Redis.
Solution: Read from Redis for the video page (real-time approximate count). Read from PostgreSQL for analytics dashboards (accurate, slightly delayed). Document the expected delay. Alternatively, merge both: PostgreSQL count + Redis unflushed delta.
