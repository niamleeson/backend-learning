OAuth2 Flows: Authorization Code, Client Credentials, PKCE

Intuition

OAuth2 is like a valet parking system. You (the resource owner) give the valet (the client app) a special ticket (access token) that only lets them park and retrieve your car, but they never get a copy of your house keys (password). The Authorization Code flow adds a layer: the valet first gets a claim ticket (authorization code) from the parking desk, then exchanges it at a secure back office window (server-to-server) for the actual car key.

OAuth2 Flows: Authorization Code, Client Credentials, PKCE

Why OAuth2 Matters

OAuth2 is the industry-standard protocol for authorization. Every time a user clicks "Sign in with Google" or "Login with GitHub," OAuth2 is working behind the scenes. As a backend developer, understanding OAuth2 is essential because you will either integrate with third-party identity providers or build your own authorization server.

OAuth2 solves a critical problem: how do you let a third-party application access a user's resources without sharing the user's password? Before OAuth, applications literally asked for your username and password to other services. OAuth2 replaces that dangerous pattern with short-lived, scoped tokens.

Core OAuth2 Concepts

Before diving into flows, you need to understand the actors and artifacts in OAuth2.

The Four Roles

┌──────────────────┐     ┌───────────────────────┐
│  Resource Owner   │     │  Client (Your App)     │
│  (The User)       │     │  - client_id           │
│                   │     │  - client_secret       │
│  Owns the data    │     │  - redirect_uri        │
└────────┬─────────┘     └───────────┬───────────┘
         │                           │
         │    Grants access          │   Requests access
         ▼                           ▼
┌──────────────────┐     ┌───────────────────────┐
│  Authorization    │     │  Resource Server       │
│  Server           │     │  (API)                 │
│                   │     │                        │
│  Issues tokens    │     │  Hosts protected       │
│  (Google, GitHub) │     │  resources             │
└──────────────────┘     └───────────────────────┘
  • Resource Owner - The user who owns the data (e.g., a GitHub user's repositories)
  • Client - Your application that wants to access the user's data
  • Authorization Server - The server that authenticates the user and issues tokens (e.g., Google's OAuth server)
  • Resource Server - The API that holds the protected resources (e.g., GitHub API)

Tokens

  • Authorization Code - A short-lived, one-time-use code exchanged for tokens. Never exposed to the browser's URL bar for long.
  • Access Token - A credential used to access protected resources. Typically short-lived (15 minutes to 1 hour).
  • Refresh Token - A long-lived token used to obtain new access tokens without re-authenticating the user.

Scopes

Scopes limit what the access token can do. Think of them as permissions:

scope: "read:user repo:read notifications"

This token can read user profile info, read repositories, and access notifications, but cannot write to repositories or delete anything.

Flow 1: Authorization Code Flow

This is the most common and most secure flow for web applications with a backend server. It is the flow you should use for any application where you have a server-side component.

How It Works

┌────────┐     ┌────────┐     ┌───────────┐     ┌──────────┐
│Browser  │     │Your    │     │Auth       │     │Resource  │
│(User)   │     │Server  │     │Server     │     │Server    │
└───┬────┘     └───┬────┘     └─────┬─────┘     └────┬─────┘
    │              │                │                 │
    │ 1. Click     │                │                 │
    │  "Login"     │                │                 │
    │─────────────►│                │                 │
    │              │                │                 │
    │ 2. Redirect to Auth Server    │                 │
    │◄─────────────│                │                 │
    │              │                │                 │
    │ 3. User logs in + consents    │                 │
    │─────────────────────────────►│                 │
    │              │                │                 │
    │ 4. Redirect back with code    │                 │
    │◄────────────────────────────│                 │
    │              │                │                 │
    │ 5. Send code │                │                 │
    │─────────────►│                │                 │
    │              │                │                 │
    │              │ 6. Exchange    │                 │
    │              │ code + secret  │                 │
    │              │ for tokens     │                 │
    │              │───────────────►│                 │
    │              │                │                 │
    │              │ 7. Access +    │                 │
    │              │ Refresh Token  │                 │
    │              │◄───────────────│                 │
    │              │                │                 │
    │              │ 8. Use access token              │
    │              │─────────────────────────────────►│
    │              │                │                 │
    │              │ 9. Protected resource             │
    │              │◄─────────────────────────────────│
    │              │                │                 │
    │ 10. Response │                │                 │
    │◄─────────────│                │                 │

The key security feature: the authorization code is exchanged for tokens on the server side (step 6), where the client_secret is safe. The browser never sees the access token.

Implementation in Node.js

Let us build a complete OAuth2 Authorization Code flow with Express and GitHub as the provider.

// oauth-demo/server.js
const express = require('express');
const crypto = require('node:crypto');
const session = require('express-session');

const app = express();

// Session to store state and tokens
app.use(session({
  secret: process.env.SESSION_SECRET || 'dev-secret-change-me',
  resave: false,
  saveUninitialized: false,
  cookie: { secure: false, httpOnly: true, maxAge: 3600000 }
}));

const GITHUB_CLIENT_ID = process.env.GITHUB_CLIENT_ID;
const GITHUB_CLIENT_SECRET = process.env.GITHUB_CLIENT_SECRET;
const REDIRECT_URI = 'http://localhost:3000/auth/github/callback';

// Step 1 & 2: Redirect user to GitHub's authorization page
app.get('/auth/github', (req, res) => {
  // Generate a random state parameter to prevent CSRF
  const state = crypto.randomBytes(16).toString('hex');
  req.session.oauthState = state;

  const params = new URLSearchParams({
    client_id: GITHUB_CLIENT_ID,
    redirect_uri: REDIRECT_URI,
    scope: 'read:user user:email',
    state: state,
    response_type: 'code'
  });

  res.redirect(`https://github.com/login/oauth/authorize?${params}`);
});

// Steps 4-7: Handle the callback with authorization code
app.get('/auth/github/callback', async (req, res) => {
  const { code, state } = req.query;

  // Verify state to prevent CSRF attacks
  if (state !== req.session.oauthState) {
    return res.status(403).json({ error: 'State mismatch. Possible CSRF attack.' });
  }
  delete req.session.oauthState;

  if (!code) {
    return res.status(400).json({ error: 'No authorization code received' });
  }

  try {
    // Step 6: Exchange code for tokens (server-to-server, secret is safe)
    const tokenResponse = await fetch('https://github.com/login/oauth/access_token', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Accept': 'application/json'
      },
      body: JSON.stringify({
        client_id: GITHUB_CLIENT_ID,
        client_secret: GITHUB_CLIENT_SECRET,
        code: code,
        redirect_uri: REDIRECT_URI
      })
    });

    const tokenData = await tokenResponse.json();

    if (tokenData.error) {
      return res.status(400).json({ error: tokenData.error_description });
    }

    // Step 7: Store tokens securely in session (never send to client)
    req.session.accessToken = tokenData.access_token;
    // GitHub does not issue refresh tokens in this flow

    // Step 8: Use token to get user info
    const userResponse = await fetch('https://api.github.com/user', {
      headers: {
        'Authorization': `Bearer ${tokenData.access_token}`,
        'Accept': 'application/vnd.github.v3+json'
      }
    });

    const user = await userResponse.json();

    // In production: find or create user in your DB
    req.session.user = {
      id: user.id,
      login: user.login,
      email: user.email,
      avatar: user.avatar_url
    };

    res.redirect('/dashboard');
  } catch (error) {
    console.error('OAuth error:', error);
    res.status(500).json({ error: 'Authentication failed' });
  }
});

// Protected route
app.get('/dashboard', (req, res) => {
  if (!req.session.user) {
    return res.redirect('/auth/github');
  }
  res.json({ message: `Welcome ${req.session.user.login}!`, user: req.session.user });
});

app.listen(3000, () => console.log('Server running on http://localhost:3000'));

The State Parameter (CSRF Protection)

The state parameter is critical. Without it, an attacker could:

  1. Start an OAuth flow on their own browser
  2. Get the authorization code
  3. Trick the victim into visiting the callback URL with the attacker's code
  4. The victim's account gets linked to the attacker's identity
// Always generate a cryptographically random state
const state = crypto.randomBytes(16).toString('hex');

// Store it in the session before redirecting
req.session.oauthState = state;

// Verify it matches when the callback arrives
if (state !== req.session.oauthState) {
  throw new Error('CSRF detected');
}

Flow 2: Client Credentials Flow

This flow is for machine-to-machine (M2M) communication where no user is involved. Your backend server authenticates itself to access another service's API.

┌────────────────┐                    ┌───────────────────┐
│  Your Server    │                    │  Auth Server       │
│  (Client)       │                    │                    │
│                 │  1. client_id +    │                    │
│                 │     client_secret  │                    │
│                 │───────────────────►│                    │
│                 │                    │                    │
│                 │  2. Access Token   │                    │
│                 │◄───────────────────│                    │
└────────┬───────┘                    └───────────────────┘
         │
         │  3. Use access token
         ▼
┌────────────────┐
│ Resource Server │
│ (API)           │
└────────────────┘

No browser, no redirects, no user consent. Just server-to-server authentication.

Implementation

// client-credentials.js
class OAuth2ClientCredentials {
  constructor({ tokenUrl, clientId, clientSecret, scopes }) {
    this.tokenUrl = tokenUrl;
    this.clientId = clientId;
    this.clientSecret = clientSecret;
    this.scopes = scopes;
    this.accessToken = null;
    this.tokenExpiry = null;
  }

  async getAccessToken() {
    // Return cached token if still valid (with 60s buffer)
    if (this.accessToken && this.tokenExpiry > Date.now() + 60000) {
      return this.accessToken;
    }

    const response = await fetch(this.tokenUrl, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/x-www-form-urlencoded',
        // Some providers prefer Basic auth header
        'Authorization': 'Basic ' + Buffer.from(
          `${this.clientId}:${this.clientSecret}`
        ).toString('base64')
      },
      body: new URLSearchParams({
        grant_type: 'client_credentials',
        scope: this.scopes.join(' ')
      })
    });

    if (!response.ok) {
      const error = await response.json();
      throw new Error(`Token request failed: ${error.error_description || error.error}`);
    }

    const data = await response.json();
    this.accessToken = data.access_token;
    this.tokenExpiry = Date.now() + (data.expires_in * 1000);

    return this.accessToken;
  }

  async authenticatedFetch(url, options = {}) {
    const token = await this.getAccessToken();
    return fetch(url, {
      ...options,
      headers: {
        ...options.headers,
        'Authorization': `Bearer ${token}`
      }
    });
  }
}

// Usage: calling an external API from your backend
const apiClient = new OAuth2ClientCredentials({
  tokenUrl: 'https://auth.example.com/oauth/token',
  clientId: process.env.API_CLIENT_ID,
  clientSecret: process.env.API_CLIENT_SECRET,
  scopes: ['read:data', 'write:data']
});

// In your service layer
async function fetchExternalData() {
  const response = await apiClient.authenticatedFetch(
    'https://api.example.com/v1/resources'
  );
  return response.json();
}

Common Use Cases

  • Microservice-to-microservice communication
  • Backend cron jobs accessing APIs
  • CLI tools that run as service accounts
  • Server-side integrations (payment processors, email services)

Flow 3: Authorization Code with PKCE

PKCE (Proof Key for Code Exchange, pronounced "pixy") was originally designed for mobile and single-page apps that cannot securely store a client_secret. It is now the recommended approach for ALL public clients and is considered best practice even for server-side apps.

The Problem PKCE Solves

In a mobile app or SPA, you cannot store a client_secret securely. The app's code is visible to anyone. Without PKCE, an attacker who intercepts the authorization code could exchange it for tokens.

How PKCE Works

┌────────────┐        ┌──────────────┐        ┌───────────────┐
│  Client     │        │  Auth Server  │        │  Attacker      │
└─────┬──────┘        └──────┬───────┘        └───────┬───────┘
      │                      │                        │
      │ 1. Generate:         │                        │
      │ code_verifier (rand) │                        │
      │ code_challenge =     │                        │
      │   SHA256(verifier)   │                        │
      │                      │                        │
      │ 2. Auth request +    │                        │
      │    code_challenge    │                        │
      │─────────────────────►│                        │
      │                      │                        │
      │ 3. Auth code         │                        │
      │◄─────────────────────│                        │
      │                      │        ┌───────────────┤
      │                      │        │ Intercepts    │
      │                      │        │ auth code     │
      │                      │        └───────────────┤
      │                      │                        │
      │ 4. Exchange code +   │   Attacker tries code  │
      │    code_verifier     │   but does NOT have     │
      │─────────────────────►│   code_verifier!        │
      │                      │◄───────────────────────│
      │ 5. Tokens!           │   REJECTED!             │
      │◄─────────────────────│───────────────────────►│
      │                      │                        │

The attacker intercepts the code but cannot exchange it without the code_verifier, which was never transmitted.

Implementation

// pkce-helpers.js
const crypto = require('node:crypto');

function generateCodeVerifier() {
  // 32 bytes = 43 base64url characters (within the 43-128 char spec)
  return crypto.randomBytes(32)
    .toString('base64url');
}

function generateCodeChallenge(verifier) {
  return crypto
    .createHash('sha256')
    .update(verifier)
    .digest('base64url');
}

module.exports = { generateCodeVerifier, generateCodeChallenge };

// pkce-oauth-server.js
const express = require('express');
const session = require('express-session');
const { generateCodeVerifier, generateCodeChallenge } = require('./pkce-helpers');

const app = express();

app.use(session({
  secret: process.env.SESSION_SECRET || 'dev-secret',
  resave: false,
  saveUninitialized: false
}));

const AUTH_CONFIG = {
  clientId: process.env.OAUTH_CLIENT_ID,
  authorizationUrl: 'https://auth.example.com/authorize',
  tokenUrl: 'https://auth.example.com/oauth/token',
  redirectUri: 'http://localhost:3000/auth/callback',
  scopes: ['openid', 'profile', 'email']
};

// Step 1: Start the PKCE flow
app.get('/auth/login', (req, res) => {
  // Generate PKCE pair
  const codeVerifier = generateCodeVerifier();
  const codeChallenge = generateCodeChallenge(codeVerifier);

  // Store verifier in session (needed for token exchange)
  req.session.codeVerifier = codeVerifier;

  // Generate state for CSRF protection
  const state = require('node:crypto').randomBytes(16).toString('hex');
  req.session.oauthState = state;

  const params = new URLSearchParams({
    response_type: 'code',
    client_id: AUTH_CONFIG.clientId,
    redirect_uri: AUTH_CONFIG.redirectUri,
    scope: AUTH_CONFIG.scopes.join(' '),
    state: state,
    code_challenge: codeChallenge,
    code_challenge_method: 'S256'
  });

  res.redirect(`${AUTH_CONFIG.authorizationUrl}?${params}`);
});

// Step 2: Handle callback and exchange code + verifier for tokens
app.get('/auth/callback', async (req, res) => {
  const { code, state } = req.query;

  // Validate state
  if (state !== req.session.oauthState) {
    return res.status(403).json({ error: 'Invalid state parameter' });
  }

  // Retrieve the stored code verifier
  const codeVerifier = req.session.codeVerifier;
  if (!codeVerifier) {
    return res.status(400).json({ error: 'Missing code verifier. Start the flow again.' });
  }

  // Clean up session
  delete req.session.oauthState;
  delete req.session.codeVerifier;

  try {
    // Exchange code + code_verifier for tokens
    const tokenResponse = await fetch(AUTH_CONFIG.tokenUrl, {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: new URLSearchParams({
        grant_type: 'authorization_code',
        client_id: AUTH_CONFIG.clientId,
        code: code,
        redirect_uri: AUTH_CONFIG.redirectUri,
        code_verifier: codeVerifier  // This proves we started the flow
      })
    });

    const tokens = await tokenResponse.json();

    if (tokens.error) {
      return res.status(400).json({
        error: tokens.error,
        description: tokens.error_description
      });
    }

    // Store tokens securely
    req.session.accessToken = tokens.access_token;
    req.session.refreshToken = tokens.refresh_token;
    req.session.idToken = tokens.id_token; // If using OpenID Connect

    res.redirect('/dashboard');
  } catch (error) {
    console.error('Token exchange failed:', error);
    res.status(500).json({ error: 'Authentication failed' });
  }
});

app.listen(3000);

Token Refresh Flow

Access tokens expire. Rather than forcing the user to log in again, use the refresh token to get a new access token.

// token-refresh.js
class TokenManager {
  constructor({ tokenUrl, clientId, clientSecret }) {
    this.tokenUrl = tokenUrl;
    this.clientId = clientId;
    this.clientSecret = clientSecret;
  }

  async refreshAccessToken(refreshToken) {
    const response = await fetch(this.tokenUrl, {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: new URLSearchParams({
        grant_type: 'refresh_token',
        refresh_token: refreshToken,
        client_id: this.clientId,
        client_secret: this.clientSecret
      })
    });

    const data = await response.json();

    if (data.error) {
      // Refresh token is expired or revoked
      // User must re-authenticate
      throw new Error(`Refresh failed: ${data.error}`);
    }

    return {
      accessToken: data.access_token,
      refreshToken: data.refresh_token || refreshToken, // Some providers rotate refresh tokens
      expiresIn: data.expires_in
    };
  }
}

// Express middleware for automatic token refresh
function autoRefreshTokens(tokenManager) {
  return async (req, res, next) => {
    if (!req.session.accessToken || !req.session.tokenExpiry) {
      return next();
    }

    // Check if token is about to expire (within 5 minutes)
    const fiveMinutes = 5 * 60 * 1000;
    if (req.session.tokenExpiry - Date.now() < fiveMinutes) {
      try {
        const newTokens = await tokenManager.refreshAccessToken(
          req.session.refreshToken
        );
        req.session.accessToken = newTokens.accessToken;
        req.session.refreshToken = newTokens.refreshToken;
        req.session.tokenExpiry = Date.now() + (newTokens.expiresIn * 1000);
      } catch (error) {
        // Refresh failed, force re-login
        req.session.destroy();
        return res.redirect('/auth/login');
      }
    }
    next();
  };
}

Security Best Practices

  1. Always use the state parameter to prevent CSRF attacks.
  2. Always use PKCE for public clients (SPAs, mobile). Consider it even for confidential clients.
  3. Never store tokens in localStorage - Use httpOnly cookies or server-side sessions.
  4. Keep access tokens short-lived (15 minutes to 1 hour).
  5. Store refresh tokens securely and rotate them when possible.
  6. Validate the redirect_uri on the authorization server to prevent open redirect attacks.
  7. Use scopes conservatively - Request only the permissions you need.
  8. Implement token revocation for logout functionality.
// Token revocation endpoint
app.post('/auth/logout', async (req, res) => {
  if (req.session.refreshToken) {
    // Revoke the refresh token at the provider
    await fetch('https://auth.example.com/oauth/revoke', {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: new URLSearchParams({
        token: req.session.refreshToken,
        token_type_hint: 'refresh_token',
        client_id: process.env.OAUTH_CLIENT_ID,
        client_secret: process.env.OAUTH_CLIENT_SECRET
      })
    });
  }

  req.session.destroy();
  res.json({ message: 'Logged out successfully' });
});

Common Problems & Solutions

1 State parameter mismatch error on OAuth callback
Cause: The session storing the state value was lost between the redirect and the callback. This commonly happens when: (1) the session cookie is not being set due to SameSite or Secure attributes, (2) a load balancer routes the callback to a different server instance without shared sessions, or (3) the user took too long and the session expired.
Solution: Ensure your session store is shared across instances (use Redis or a database-backed session store). Check that your session cookie settings are correct for your domain. For local development, set `cookie.secure = false`. For production behind a proxy, set `app.set('trust proxy', 1)` and use secure cookies.
2 Authorization code exchange returns 'invalid_grant' error
Cause: The authorization code has already been used (codes are single-use), the code has expired (typically 10 minutes), or the redirect_uri in the token request does not exactly match the one used in the authorization request.
Solution: Ensure you exchange the code immediately upon receiving it. Verify the redirect_uri is identical in both requests (including trailing slashes). Do not retry the code exchange if it fails; redirect the user to start the flow again.
3 Access token works locally but fails in production with CORS errors
Cause: When making API calls from the browser, CORS policies block the request. Tokens in the Authorization header trigger a preflight OPTIONS request that the API server may not handle.
Solution: For server-side OAuth flows, make API calls from your backend server, not from the browser. If you must call from the browser, ensure the resource server has proper CORS headers. Better yet, create a proxy endpoint on your server that adds the token and forwards the request.
