SELECT, WHERE, and Basic Queries

Intuition

SELECT WHERE is like telling a librarian exactly which shelf to look on and which books to pull. You describe the data you want and the database fetches only matching rows.

SELECT WHERE Basic Queries

The SELECT statement is the foundation of SQL. Combined with WHERE, it lets you retrieve exactly the rows you need from a database.

-- Basic select with filtering
SELECT id, name, email, created_at
FROM users
WHERE status = 'active'
  AND created_at >= '2024-01-01'
ORDER BY created_at DESC
LIMIT 20;

-- Using IN, BETWEEN, LIKE
SELECT product_name, price
FROM products
WHERE category IN ('electronics', 'books')
  AND price BETWEEN 10 AND 100
  AND product_name LIKE '%Pro%';

-- NULL handling
SELECT name FROM customers
WHERE phone IS NOT NULL;

Key Takeaways: - WHERE filters rows before they are returned; use comparison operators, IN, BETWEEN, LIKE, and IS NULL. - ORDER BY sorts results; add DESC for descending. LIMIT caps the result set. - Always be explicit about columns instead of using SELECT * in production code.

Exercises

  1. Filter and Sort Users

    Write a query to find all users whose email ends with '@gmail.com', created in 2024, sorted by name alphabetically.

    Hint

    Use LIKE '%@gmail.com' combined with a date range in WHERE, then ORDER BY name ASC.

  2. NULL-Safe Search

    Write a query that returns products where the description is NULL or the price is under 5, showing product_name and price.

    Hint

    Combine IS NULL and a comparison with OR; remember OR needs parentheses when mixed with AND.

Common Problems & Solutions

1 Query returns no rows even though data exists.
Cause: Comparing against NULL with = instead of IS NULL, or mismatched string casing.
Solution: Use IS NULL / IS NOT NULL for null checks. Use ILIKE or LOWER() for case-insensitive matching in PostgreSQL.
2 Query is very slow on a large table.
Cause: No index on the columns used in the WHERE clause, or using functions on indexed columns.
Solution: Create indexes on frequently filtered columns. Avoid wrapping indexed columns in functions like LOWER() unless you have a functional index.
