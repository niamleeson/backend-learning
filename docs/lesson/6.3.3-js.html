Automated Deployment Strategies

Intuition

Deployment strategies are like different ways to renovate a restaurant while keeping it open. Recreate deployment closes the restaurant for renovation (downtime). Rolling deployment renovates one table at a time while serving guests at the others. Blue/Green is like having two identical restaurants -- renovate one while guests eat at the other, then move everyone over. Canary deployment is like inviting a few taste-testers to the new menu before offering it to all guests.

Automated Deployment Strategies

From CI to CD

Continuous Deployment (CD) extends CI by automatically deploying every change that passes tests. The deployment strategy you choose determines the risk, speed, and complexity of releases.

Deployment Strategy Comparison

Strategy         Risk    Downtime    Complexity    Rollback Speed
──────────────────────────────────────────────────────────────────
Recreate         High    Yes         Low           Slow
Rolling          Medium  No          Medium        Medium
Blue/Green       Low     No          Medium        Instant
Canary           Low     No          High          Fast
Feature Flags    Low     No          High          Instant

Recreate Deployment

Version 1 (stop all) ──────── downtime ──────── Version 2 (start all)

┌─────┐ ┌─────┐ ┌─────┐                  ┌─────┐ ┌─────┐ ┌─────┐
│v1   │ │v1   │ │v1   │   ──────────►    │v2   │ │v2   │ │v2   │
└─────┘ └─────┘ └─────┘    (downtime)    └─────┘ └─────┘ └─────┘

Rolling Deployment

┌─────┐ ┌─────┐ ┌─────┐    ┌─────┐ ┌─────┐ ┌─────┐    ┌─────┐ ┌─────┐ ┌─────┐
│v1   │ │v1   │ │v1   │ ─► │v2   │ │v1   │ │v1   │ ─► │v2   │ │v2   │ │v1   │ ─► all v2
└─────┘ └─────┘ └─────┘    └─────┘ └─────┘ └─────┘    └─────┘ └─────┘ └─────┘

Blue/Green Deployment

Load Balancer
      │
      ├──── Blue (v1) ◄── current traffic
      │
      └──── Green (v2) ◄── deploy here, test, then switch

After switch:
      │
      ├──── Blue (v1) ◄── standby (instant rollback)
      │
      └──── Green (v2) ◄── current traffic

Canary Deployment

Load Balancer
      │
      ├──── 95% ──► v1 instances (stable)
      │
      └──── 5%  ──► v2 instance  (canary)

                    If OK, gradually increase to 100%

GitHub Actions Deployment Workflows

Deploy to VPS/EC2 via SSH

# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  test:
    uses: ./.github/workflows/test.yml
    secrets: inherit

  deploy:
    name: Deploy to Production
    needs: test
    runs-on: ubuntu-latest
    environment: production        # Requires approval if configured

    steps:
      - uses: actions/checkout@v4

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /var/www/myapp
            git pull origin main
            npm ci --production
            npx prisma migrate deploy
            pm2 reload ecosystem.config.js --env production

Deploy Docker Image

name: Build and Deploy Docker

on:
  push:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Login to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: production
    steps:
      - name: Deploy to server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: deploy
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            docker pull ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
            docker compose -f docker-compose.prod.yml up -d --no-deps api
            docker image prune -f

Deploy to AWS Lambda (Serverless)

name: Deploy Serverless

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci
      - run: npm run build

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Deploy with Serverless Framework
        run: npx serverless deploy --stage production

Blue/Green with AWS

name: Blue/Green Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Deploy to Green environment
        run: |
          # Build and push Docker image
          aws ecr get-login-password | docker login --username AWS --password-stdin $ECR_REGISTRY
          docker build -t $ECR_REGISTRY/myapp:${{ github.sha }} .
          docker push $ECR_REGISTRY/myapp:${{ github.sha }}

          # Update ECS service (green)
          aws ecs update-service \
            --cluster myapp-cluster \
            --service myapp-green \
            --task-definition myapp:${{ github.sha }}

          # Wait for deployment
          aws ecs wait services-stable \
            --cluster myapp-cluster \
            --services myapp-green

      - name: Health check green
        run: |
          for i in $(seq 1 10); do
            if curl -sf https://green.myapp.com/health; then
              echo "Green is healthy!"
              exit 0
            fi
            sleep 5
          done
          echo "Green health check failed"
          exit 1

      - name: Switch traffic to green
        run: |
          # Update ALB target group weights
          aws elbv2 modify-rule \
            --rule-arn ${{ secrets.ALB_RULE_ARN }} \
            --actions '[{"Type":"forward","TargetGroupArn":"'$GREEN_TG_ARN'","Weight":100}]'

      - name: Notify
        if: always()
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {"text": "Deployment ${{ job.status }}: ${{ github.repository }}@${{ github.sha }}"}
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}

Rollback Strategies

# Manual rollback workflow
name: Rollback

on:
  workflow_dispatch:
    inputs:
      version:
        description: 'Version to rollback to (git SHA or tag)'
        required: true

jobs:
  rollback:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - name: Rollback deployment
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: deploy
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /var/www/myapp
            git checkout ${{ github.event.inputs.version }}
            npm ci --production
            npx prisma migrate deploy
            pm2 reload all

Database Migration Safety

# Run migrations as a separate step with safeguards
- name: Run database migrations
  run: |
    # Backup before migration
    ssh deploy@$SERVER "pg_dump myapp > /tmp/pre-migration-backup.sql"

    # Run migration
    ssh deploy@$SERVER "cd /var/www/myapp && npx prisma migrate deploy"

    # Verify application health after migration
    sleep 10
    if ! curl -sf https://myapp.com/health; then
      echo "App unhealthy after migration, check logs!"
      exit 1
    fi

Common Pitfalls

  1. No rollback plan - Every deployment should have a tested rollback procedure. Practice rollbacks regularly.
  2. Deploying database migrations and code simultaneously - Migrate first (forward-compatible), deploy code second. This avoids a window where new code runs against old schema.
  3. No health checks after deployment - Always verify the new version is actually working after deploy.
  4. Deploying on Fridays - Avoid deploying before weekends or holidays when the team is unavailable to fix issues.
  5. No deployment notifications - The team should know when deployments happen. Use Slack/Teams notifications.
  6. No approval gates for production - Use GitHub Environments with required reviewers for production deployments.

Exercises

  1. Docker-Based CD Pipeline

    Create a complete GitHub Actions workflow that: builds a Docker image, pushes it to GitHub Container Registry (ghcr.io), SSHs into a server and pulls the new image, and does a rolling restart with Docker Compose. Include health checks and Slack notification.

    Hint

    Use docker/login-action for registry auth. Tag images with both 'latest' and the git SHA. Use 'docker compose up -d --no-deps service' for rolling restart.

  2. Blue/Green Deploy Script

    Write a deployment script that implements blue/green deployment on a single server using Docker Compose. Maintain two compose files (blue and green). The script should: deploy to the inactive environment, run health checks, switch Nginx to point to the new environment, and keep the old one as rollback.

    Hint

    Use Nginx upstream blocks with different ports for blue (3001) and green (3002). Swap the Nginx config and reload to switch traffic.

  3. Rollback Workflow

    Create a manual GitHub Actions workflow (workflow_dispatch) that accepts a git SHA or tag and rolls back the production deployment to that version. Include database migration rollback considerations.

    Hint

    Use workflow_dispatch with inputs for the version. SSH to the server, checkout the specific version, and restart. For database rollbacks, consider that not all migrations are reversible.

Common Problems & Solutions

1 Deployment succeeds but application returns 502 Bad Gateway
Cause: The application started but crashed shortly after, or Nginx/load balancer cannot reach the new application instance.
Solution: Check application logs (`pm2 logs` or `docker logs`). Verify the app is listening on the correct port. Ensure health check passes. Common causes: missing environment variables, database connection failure, port mismatch between app and Nginx.
2 Database migration fails during deployment
Cause: The migration has a destructive change (dropping column) while old code is still running, or the migration has a syntax error.
Solution: Always test migrations in staging first. Use expand-contract pattern: (1) add new column, (2) deploy code that writes to both, (3) migrate data, (4) deploy code that reads from new column, (5) drop old column. Never drop columns in the same deployment that stops using them.
3 Deployment works but SSH action times out
Cause: The SSH connection is blocked by a firewall, security group, or the key/host configuration is wrong.
Solution: Verify the server's IP is correct in secrets. Check that port 22 is open in the security group/firewall. Ensure the SSH key in secrets matches the authorized_keys on the server. Test SSH access manually first.
