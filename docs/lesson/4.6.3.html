Connection Pooling and Performance

Intuition

Connection pooling is like a car rental fleet -- instead of buying a new car for every trip and scrapping it after, you return it to the lot for the next customer. This reuse saves the expensive overhead of establishing new database connections.

Connection Pooling and Performance

Database connection pooling reuses existing connections instead of opening a new one for every request, dramatically improving performance. SQLAlchemy provides built-in pool management, and understanding its configuration is crucial for production Flask applications.

from flask import Flask, g, jsonify
from flask_sqlalchemy import SQLAlchemy
from sqlalchemy import event, text
import time

db = SQLAlchemy()

def create_app():
    app = Flask(__name__)
    app.config['SQLALCHEMY_DATABASE_URI'] = 'postgresql://user:pass@localhost/mydb'
    app.config['SQLALCHEMY_ENGINE_OPTIONS'] = {
        'pool_size': 10,           # Number of persistent connections
        'max_overflow': 20,        # Extra connections when pool is full
        'pool_timeout': 30,        # Seconds to wait for a connection
        'pool_recycle': 1800,      # Recycle connections after 30 min
        'pool_pre_ping': True,     # Verify connections before use
    }
    db.init_app(app)

    @app.before_request
    def before_request():
        g.start_time = time.time()

    @app.after_request
    def after_request(response):
        duration = time.time() - g.start_time
        response.headers['X-Response-Time'] = f'{duration:.3f}s'
        return response

    return app

# Monitor slow queries with SQLAlchemy events
@event.listens_for(db.engine, 'before_cursor_execute')
def receive_before_execute(conn, cursor, stmt, params, context, executemany):
    conn.info.setdefault('query_start', []).append(time.time())

@event.listens_for(db.engine, 'after_cursor_execute')
def receive_after_execute(conn, cursor, stmt, params, context, executemany):
    total = time.time() - conn.info['query_start'].pop()
    if total > 0.5:
        print(f'SLOW QUERY ({total:.2f}s): {stmt[:200]}')

Key Takeaways: - pool_pre_ping=True prevents errors from stale connections by testing them before use. - pool_recycle should be shorter than your database's connection timeout (MySQL default is 8 hours). - Monitor query performance with SQLAlchemy events and optimize N+1 queries with eager loading.

Exercises

  1. Pool Configuration Testing

    Create a Flask app with a small pool (pool_size=2, max_overflow=0) and write a script that sends concurrent requests to observe pool timeout behavior. Log when connections are acquired and released.

    Hint

    Use threading or concurrent.futures to send multiple requests simultaneously. Set pool_timeout to a low value like 5 seconds to trigger TimeoutError quickly.

  2. N+1 Query Detection

    Build a route that lists users with their posts, deliberately triggering an N+1 query. Then fix it using joinedload() or subqueryload(). Use SQLAlchemy events to count and log the queries.

    Hint

    Compare User.query.all() followed by user.posts access (N+1) versus User.query.options(joinedload(User.posts)).all() (1 query with JOIN).

Common Problems & Solutions

1 QueuePool limit overflow: connection refused after sustained traffic.
Cause: The pool_size and max_overflow are too small for the application's concurrent database needs.
Solution: Increase pool_size and max_overflow, but also check for connection leaks. Ensure every session is properly closed after use. Flask-SQLAlchemy handles this automatically with teardown_appcontext.
2 OperationalError: server closed the connection unexpectedly after idle period.
Cause: The database server closed idle connections, but the pool still holds stale connection references.
Solution: Set pool_pre_ping=True to validate connections before use, and set pool_recycle to a value shorter than the database's connection timeout.
