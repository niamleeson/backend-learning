Window Functions: ROW_NUMBER, RANK, LAG, LEAD

Intuition

Window functions are like looking at a spreadsheet where each row can peek at other rows in its group to compute relative values, without merging the rows together.

Window Functions: ROW_NUMBER, RANK, LAG, LEAD

Window functions perform calculations across a set of rows related to the current row without collapsing them into a single output row.

-- ROW_NUMBER and RANK
SELECT name, department, salary,
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rn,
  RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rnk
FROM employees;

-- LAG and LEAD for comparing adjacent rows
SELECT date, revenue,
  LAG(revenue, 1) OVER (ORDER BY date) AS prev_day,
  revenue - LAG(revenue, 1) OVER (ORDER BY date) AS daily_change,
  LEAD(revenue, 1) OVER (ORDER BY date) AS next_day
FROM daily_sales;

-- Running total
SELECT date, amount,
  SUM(amount) OVER (ORDER BY date ROWS UNBOUNDED PRECEDING) AS running_total
FROM transactions;

Key Takeaways: - PARTITION BY divides rows into groups; ORDER BY sets the sequence within each partition. - ROW_NUMBER gives unique sequential numbers; RANK allows ties with gaps; DENSE_RANK has no gaps. - LAG looks back, LEAD looks forward. Both take an optional offset and default value.

Exercises

  1. Top 3 Per Department

    Find the top 3 highest-paid employees in each department using ROW_NUMBER.

    Hint

    Use a CTE with ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC), then filter WHERE rn <= 3.

  2. Month-over-Month Growth

    Calculate the percentage change in monthly revenue compared to the previous month using LAG.

    Hint

    Use (revenue - LAG(revenue) OVER (ORDER BY month)) / LAG(revenue) OVER (ORDER BY month) * 100.

Common Problems & Solutions

1 Cannot use window function in WHERE clause.
Cause: SQL evaluates WHERE before window functions are computed.
Solution: Wrap the window function in a CTE or subquery, then filter in the outer query's WHERE clause.
2 ROW_NUMBER gives unexpected ordering with ties.
Cause: ROW_NUMBER assigns arbitrary order to tied values.
Solution: Add a tiebreaker column to ORDER BY, or use RANK/DENSE_RANK if ties should share the same number.
