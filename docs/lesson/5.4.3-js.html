gRPC vs REST vs tRPC: When to Use Which

Intuition

Choosing between REST, GraphQL, tRPC, and gRPC is like choosing between a bicycle, car, motorcycle, and truck. A bicycle (REST) works everywhere and everyone knows how to ride one. A car (GraphQL) is comfortable and flexible for passengers (clients) who want control. A motorcycle (tRPC) is the fastest for a solo TypeScript rider. A truck (gRPC) carries the heaviest loads between warehouses (services) with maximum efficiency.

gRPC vs REST vs tRPC: When to Use Which

Choosing the right API paradigm is a critical architectural decision. Each has distinct strengths and trade-offs. This lesson provides a framework for making that choice.

Feature Comparison

+-------------------+----------+-----------+----------+-----------+
| Feature           |  REST    |  GraphQL  |  tRPC    |   gRPC    |
+-------------------+----------+-----------+----------+-----------+
| Protocol          | HTTP/1.1 | HTTP/1.1  | HTTP/1.1 | HTTP/2    |
| Data Format       | JSON     | JSON      | JSON     | Protobuf  |
| Type Safety       | Manual   | Schema    | Auto     | Generated |
| Code Generation   | Optional | Optional  | None     | Required  |
| Browser Support   | Native   | Native    | Native   | Limited*  |
| Streaming         | Limited  | Subscript.| Limited  | Full      |
| Performance       | Good     | Good      | Good     | Excellent |
| Caching           | HTTP     | Complex   | Manual   | Manual    |
| Learning Curve    | Low      | Medium    | Low      | High      |
| Language Agnostic | Yes      | Yes       | No (TS)  | Yes       |
+-------------------+----------+-----------+----------+-----------+
* gRPC-Web or Connect protocol for browser support

When to Use REST

REST is the best choice when:

  1. Public APIs consumed by third-party developers
  2. Simple CRUD operations on resources
  3. Browser-based consumption without libraries
  4. HTTP caching is important (CDN, browser cache)
  5. Team is familiar with REST conventions
  6. Hypermedia-driven workflows (HATEOAS)
// REST shines for straightforward resource operations
// GET /api/products/123
// PUT /api/products/123
// DELETE /api/products/123
// GET /api/products?category=electronics&sort=-price

app.get('/api/products/:id', async (req, res) => {
  const product = await productService.findById(req.params.id);
  // HTTP caching works out of the box
  res.set('Cache-Control', 'public, max-age=300');
  res.json(product);
});

Strengths: - Universal understanding - Built-in caching via HTTP headers - Easy to debug (curl, browser, Postman) - Mature tooling and documentation (OpenAPI/Swagger)

Weaknesses: - Over-fetching / under-fetching - No built-in type safety - Multiple round trips for complex data

When to Use GraphQL

GraphQL excels when:

  1. Multiple frontends need different data shapes (mobile vs web)
  2. Complex, nested data with relationships
  3. Reducing round trips is critical (mobile apps)
  4. Rapid frontend iteration without backend changes
  5. API aggregation across multiple microservices
# Mobile app: minimal data
query {
  user(id: "1") {
    name
    avatarUrl
  }
}

# Web dashboard: full data, one request
query {
  user(id: "1") {
    name
    email
    posts {
      title
      comments { text author { name } }
    }
    notifications(unreadOnly: true) { message }
  }
}

Strengths: - Client controls response shape - Single endpoint, single request - Strong schema with introspection - Great developer experience (Apollo Studio, GraphiQL)

Weaknesses: - N+1 query problem (needs DataLoader) - No HTTP caching by default - Query complexity attacks - Steeper learning curve

When to Use tRPC

tRPC is ideal when:

  1. Full-stack TypeScript (you control both ends)
  2. Internal APIs (not consumed by third parties)
  3. Rapid development with auto-complete and type checking
  4. Monorepos where client and server share types
  5. Small to medium teams moving fast
// tRPC: instant type safety, zero overhead
const user = await trpc.user.getById.query({ id: '1' });
user.name;  // TypeScript knows this is string
user.foo;   // Compile error! Property 'foo' does not exist

Strengths: - Zero-cost type safety - No code generation or schemas to maintain - Instant auto-complete on the client - Easy to set up and learn - Request batching built-in

Weaknesses: - TypeScript only (no other language clients) - Not suitable for public APIs - No standard like OpenAPI for documentation - Tightly couples frontend and backend

When to Use gRPC

gRPC excels when:

  1. Microservice communication (service-to-service)
  2. High performance is critical (binary protocol, HTTP/2)
  3. Streaming requirements (real-time data, uploads)
  4. Polyglot services (different languages per service)
  5. Strong contracts with backward compatibility
// gRPC: binary, fast, multi-language
service OrderService {
  rpc CreateOrder(CreateOrderRequest) returns (Order);
  rpc StreamOrderUpdates(OrderFilter) returns (stream OrderUpdate);
}

Strengths: - 3-10x smaller payloads (protobuf) - Native streaming (all four patterns) - HTTP/2 multiplexing - Excellent for service-to-service - Strong backward compatibility via field numbers

Weaknesses: - Not browser-native (needs gRPC-Web) - Binary format is hard to debug - Requires proto compilation toolchain - Higher learning curve - Not human-readable

Decision Framework

Are you building a public API?
  YES -> REST (with OpenAPI documentation)
  NO  ->
    Is it service-to-service?
      YES ->
        Need streaming or high performance?
          YES -> gRPC
          NO  -> REST or gRPC (team preference)
      NO  ->
        Is everything TypeScript?
          YES -> tRPC
          NO  ->
            Multiple frontends with different data needs?
              YES -> GraphQL
              NO  -> REST

Hybrid Architectures

In practice, many systems use multiple paradigms:

+---------------------------------------------------+
|                                                   |
|  [Mobile App] --GraphQL--> [API Gateway]          |
|  [Web App]    --tRPC-----> [BFF Service]          |
|  [Partner API] --REST----> [Public API]           |
|                                                   |
|       Internal microservices communicate          |
|              via gRPC                             |
|                                                   |
|  [User Service] <--gRPC--> [Order Service]        |
|  [Order Service] <-gRPC--> [Payment Service]      |
|  [Notification Service] <-gRPC--> [Email Service] |
|                                                   |
+---------------------------------------------------+

// NestJS hybrid: REST + gRPC
// REST for external clients
@Controller('api/products')
export class ProductsController {
  @Get(':id')
  findOne(@Param('id') id: string) {
    return this.productsService.findById(id);
  }
}

// gRPC for internal service communication
@GrpcMethod('ProductService', 'GetProduct')
findOneGrpc(data: { id: string }) {
  return this.productsService.findById(data.id);
}

Common Anti-Patterns

  1. Using gRPC for browser-facing APIs without gRPC-Web
  2. Using GraphQL for simple CRUD with no relationships
  3. Using tRPC for public APIs that need multiple language clients
  4. Using REST for real-time streaming (use WebSocket or gRPC instead)
  5. Choosing based on hype instead of actual requirements

Exercises

  1. Architecture Decision Document

    You are designing a system with: a public mobile app, an internal admin dashboard, and 5 backend microservices. Write an architecture decision document that recommends which API paradigm to use for each communication path (mobile-to-backend, admin-to-backend, service-to-service) with justification.

    Hint

    Consider GraphQL or REST for mobile, tRPC for admin dashboard (if TypeScript), and gRPC for service-to-service. Justify each based on the criteria in this lesson.

  2. Build the Same API in REST, tRPC, and gRPC

    Implement a simple 'todo' CRUD API (create, read, update, delete, list) using all three paradigms. Compare the code complexity, type safety, and developer experience of each.

    Hint

    Start with the REST version as a baseline. Note which paradigm requires the most boilerplate and which provides the best auto-complete.

  3. Performance Comparison

    Build a benchmark that sends 1000 sequential requests for the same data using REST (JSON), GraphQL (JSON), and gRPC (protobuf). Measure average latency, payload size, and throughput.

    Hint

    Use console.time for timing. Use JSON.stringify().length for JSON payload size. For gRPC, inspect the serialized buffer length.

Common Problems & Solutions

1 Team chose gRPC for everything including the frontend
Cause: gRPC was selected based on performance benchmarks without considering browser compatibility.
Solution: Use gRPC for service-to-service communication and REST/GraphQL/tRPC for browser-facing APIs. If you must use gRPC from browsers, use gRPC-Web or the Connect protocol.
2 GraphQL API is slower than the REST API it replaced
Cause: N+1 queries, lack of DataLoader, and no response caching.
Solution: Implement DataLoader for all relationship resolvers. Add query depth limiting. Use Apollo response cache or persisted queries.
3 tRPC types are not available to non-TypeScript consumers
Cause: tRPC's type inference only works within the TypeScript ecosystem.
Solution: For non-TypeScript clients, generate an OpenAPI spec from your tRPC routes using trpc-openapi, or provide a separate REST API for external consumers.
