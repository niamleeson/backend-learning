Kubernetes Introduction: Pods, Services, Deployments

Intuition

Kubernetes is like a fleet manager for delivery trucks (containers). You tell the manager 'I need 3 trucks running route A.' If one breaks down, the manager automatically dispatches a replacement. If demand increases, you say 'make it 5 trucks' and two more are deployed. The Service is like a dispatch phone number that always reaches an available truck.

Kubernetes Introduction: Pods, Services, Deployments

Kubernetes (K8s) orchestrates containerized applications at scale, handling deployment, scaling, and self-healing automatically.

Core Concepts

  • Pod: Smallest unit. One or more containers sharing network/storage
  • Service: Stable endpoint routing traffic to pods
  • Deployment: Declares desired state and manages rollouts

Deployment Manifest

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels: { app: myapp }
  template:
    metadata:
      labels: { app: myapp }
    spec:
      containers:
        - name: web
          image: myapp:1.2.0
          ports: [{ containerPort: 8000 }]
          resources:
            requests: { memory: "128Mi", cpu: "250m" }
            limits: { memory: "256Mi", cpu: "500m" }
          readinessProbe:
            httpGet: { path: /health, port: 8000 }
            initialDelaySeconds: 5
---
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  type: LoadBalancer
  selector: { app: myapp }
  ports: [{ port: 80, targetPort: 8000 }]

Essential kubectl Commands

kubectl apply -f deployment.yaml
kubectl get pods
kubectl logs myapp-abc123 -f
kubectl exec -it myapp-abc123 -- bash
kubectl scale deployment myapp --replicas=5
kubectl rollout undo deployment myapp  # Rollback

Common Mistakes

  • Not setting resource requests/limits
  • Missing readiness probes, sending traffic to unready pods
  • Using latest image tag, making rollbacks impossible

Key Takeaways

  • Always set resource limits and readiness probes
  • Use specific image tags for reproducible deployments
  • K8s self-heals: crashed pods are automatically restarted

Exercises

  1. Deploy to Kubernetes

    Write Kubernetes manifests to deploy a Python web application with 3 replicas, a Service exposing port 80, resource limits, readiness probes, and environment variables from a Secret. Apply to a local cluster (minikube or kind).

    Hint

    Create the Secret first with kubectl create secret. Use readinessProbe with httpGet to your health endpoint.

  2. Rolling Update and Rollback

    Deploy version 1 of an application, then update to version 2 using kubectl set image. Monitor the rollout, then rollback to version 1. Document the process and verify zero downtime.

    Hint

    Use kubectl rollout status, kubectl rollout history, and kubectl rollout undo.

  3. ConfigMap and Secrets

    Create a ConfigMap for non-sensitive configuration (app settings, feature flags) and a Secret for sensitive data (database URL, API keys). Mount them as environment variables in your deployment.

    Hint

    Use kubectl create configmap and kubectl create secret generic. Reference with envFrom in the pod spec.

Common Problems & Solutions

1 Pod stuck in CrashLoopBackOff
Symptoms: ['STATUS: CrashLoopBackOff', 'Restart count keeps increasing']
Cause: The container keeps crashing on startup. Common causes: missing environment variables, failed database connection, application error.
Solution: Check logs: `kubectl logs --previous` to see the crash output. Check events: `kubectl describe pod `. Fix the application error or missing configuration.
2 Pod stuck in Pending state
Symptoms: ['STATUS: Pending', '0/3 nodes are available: insufficient memory']
Cause: Not enough resources (CPU/memory) available on any node to schedule the pod.
Solution: Check events: `kubectl describe pod `. Reduce resource requests, add more nodes, or remove other workloads. Check for node taints or affinity rules.
3 Service not routing traffic to pods
Symptoms: ['No endpoints available for service', "curl to service returns 'connection refused'"]
Cause: Label selector mismatch between the Service and the Deployment pods.
Solution: Verify labels match: `kubectl get pods --show-labels` and compare with `kubectl describe service myapp-service`. The Service selector must match the pod template labels exactly.
