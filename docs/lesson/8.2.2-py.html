OpenID Connect and Identity Providers

Intuition

Think of OAuth 2.0 as a hotel key card: it gets you into specific rooms (resources), but it does not say who you are. OpenID Connect is like also receiving a passport at check-in — a tamper-proof document (the ID token) that proves your identity. The hotel (your application) can look at the passport to know who checked in, and use the key card to open the right doors.

OpenID Connect and Identity Providers

OAuth 2.0 solved the authorization problem — allowing third-party applications to access resources on behalf of a user — but it deliberately left out one critical piece: identity. OAuth 2.0 tells you what someone is allowed to do, but not who they are. OpenID Connect (OIDC) is a thin identity layer built on top of OAuth 2.0 that fills this gap, providing a standardized way to authenticate users and retrieve their profile information.

In this lesson, we will explore how OIDC extends OAuth 2.0, understand the structure and purpose of ID tokens, integrate with real identity providers, and implement robust token validation in Python.

How OIDC Extends OAuth 2.0

OpenID Connect is not a replacement for OAuth 2.0 — it is an extension. Every OIDC flow is a valid OAuth 2.0 flow with additional conventions layered on top. The key additions are:

  1. The openid scope: When a client includes openid in its authorization request, it signals that this is an OIDC authentication request, not just an authorization request.
  2. The ID Token: A new token type (a signed JWT) that contains claims about the authenticated user.
  3. The UserInfo Endpoint: A standardized API endpoint that returns profile information about the authenticated user.
  4. Discovery: A well-known endpoint that allows clients to automatically discover provider configuration.

Here is a simplified comparison:

Feature OAuth 2.0 OIDC
Purpose Authorization Authentication + Authorization
Token types Access token, Refresh token Access token, Refresh token, ID token
User identity Not standardized Standardized via ID token and UserInfo
Discovery Not standardized .well-known/openid-configuration
Scopes Application-defined openid, profile, email, address, phone

OIDC Flows

OIDC defines three flows, corresponding to OAuth 2.0 response types:

  • Authorization Code Flow (response_type=code): The most common and most secure flow for server-side applications. The client receives an authorization code and exchanges it for tokens at the token endpoint.
  • Implicit Flow (response_type=id_token token): Tokens are returned directly in the URL fragment. This flow is now discouraged due to security concerns.
  • Hybrid Flow (response_type=code id_token): Combines elements of both. The client gets an ID token immediately and an authorization code to exchange for an access token.

For backend applications, the Authorization Code Flow is almost always the correct choice.

ID Tokens vs Access Tokens

This distinction is fundamental and frequently misunderstood:

ID Token: A JWT intended for the client application. It contains claims about the user's identity — who they are, when they authenticated, and how. The client consumes this token to establish a session. An ID token should never be sent to an API as a bearer token.

Access Token: A token intended for the resource server (API). It represents the user's authorization to access specific resources. Access tokens may or may not be JWTs — the format is opaque to the client.

ID Token:    Client <--- Identity Provider ("Here is who the user is")
Access Token: Client ---> Resource Server  ("I am allowed to access this")

A common mistake is using the ID token as an access token, or vice versa. They serve different purposes and have different audiences.

JWT Structure and Claims

ID tokens are always JWTs (JSON Web Tokens). A JWT consists of three base64url-encoded parts separated by dots:

header.payload.signature

Header

The header specifies the signing algorithm and, optionally, a key ID:

{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "abc123"
}

Payload (Claims)

The payload contains claims — assertions about the user and the authentication event. OIDC defines several standard claims:

Claim Name Description
iss Issuer The URL of the identity provider that issued the token
sub Subject A unique identifier for the user (stable across sessions)
aud Audience The client ID of the application the token was issued for
exp Expiration Unix timestamp after which the token is invalid
iat Issued At Unix timestamp when the token was issued
nonce Nonce A random value tied to the authentication request to prevent replay attacks
auth_time Auth Time When the user last actively authenticated
azp Authorized Party The client ID of the party the token was issued to (used when aud contains multiple values)

A typical ID token payload looks like this:

{
  "iss": "https://accounts.google.com",
  "sub": "110248495921238986420",
  "aud": "your-client-id.apps.googleusercontent.com",
  "exp": 1679012345,
  "iat": 1679008745,
  "nonce": "abc123random",
  "email": "user@example.com",
  "email_verified": true,
  "name": "Jane Doe",
  "picture": "https://lh3.googleusercontent.com/a/photo.jpg"
}

Signature

The signature ensures the token has not been tampered with. For RS256, the identity provider signs the header and payload with its private key, and the client verifies the signature using the provider's public key (obtained from the JWKS endpoint).

The OIDC Discovery Endpoint

One of OIDC's most powerful features is automatic discovery. Every OIDC provider publishes a JSON document at a well-known URL:

https://{issuer}/.well-known/openid-configuration

This document contains everything a client needs to interact with the provider:

import requests

def fetch_oidc_config(issuer_url: str) -> dict:
    """Fetch the OIDC discovery document for a given issuer."""
    discovery_url = f"{issuer_url.rstrip('/')}/.well-known/openid-configuration"
    response = requests.get(discovery_url, timeout=10)
    response.raise_for_status()
    return response.json()

# Example: Google's OIDC configuration
config = fetch_oidc_config("https://accounts.google.com")
print(f"Authorization endpoint: {config['authorization_endpoint']}")
print(f"Token endpoint: {config['token_endpoint']}")
print(f"UserInfo endpoint: {config['userinfo_endpoint']}")
print(f"JWKS URI: {config['jwks_uri']}")
print(f"Supported scopes: {config['scopes_supported']}")

Key fields in the discovery document include:

  • authorization_endpoint: Where to send the user for authentication
  • token_endpoint: Where to exchange the authorization code for tokens
  • userinfo_endpoint: Where to fetch user profile information
  • jwks_uri: Where to fetch the provider's public keys for token verification
  • scopes_supported: What scopes the provider supports
  • response_types_supported: What response types are available
  • id_token_signing_alg_values_supported: What signing algorithms the provider uses

This means you can write provider-agnostic code — just point it at an issuer URL and let discovery handle the rest.

The UserInfo Endpoint

The UserInfo endpoint returns claims about the authenticated user. You call it with a valid access token (not the ID token):

def fetch_user_info(userinfo_endpoint: str, access_token: str) -> dict:
    """Fetch user profile information from the UserInfo endpoint."""
    headers = {"Authorization": f"Bearer {access_token}"}
    response = requests.get(userinfo_endpoint, headers=headers, timeout=10)
    response.raise_for_status()
    return response.json()

# Usage
user_info = fetch_user_info(
    config["userinfo_endpoint"],
    access_token="eyJhbGciOiJSUzI1NiIs..."
)
print(f"User: {user_info['name']} ({user_info['email']})")

The claims returned depend on the scopes requested during authorization:

  • openid: Returns sub only
  • profile: Adds name, family_name, given_name, picture, etc.
  • email: Adds email and email_verified
  • address: Adds structured address information
  • phone: Adds phone_number and phone_number_verified

Token Validation

Validating an ID token properly is critical for security. Here is a comprehensive validation function using the PyJWT library:

import jwt
import requests
from jwt import PyJWKClient
from typing import Optional

class OIDCTokenValidator:
    def __init__(self, issuer_url: str, client_id: str):
        self.issuer_url = issuer_url.rstrip('/')
        self.client_id = client_id
        self._config = None
        self._jwk_client = None

    @property
    def config(self) -> dict:
        if self._config is None:
            discovery_url = f"{self.issuer_url}/.well-known/openid-configuration"
            resp = requests.get(discovery_url, timeout=10)
            resp.raise_for_status()
            self._config = resp.json()
        return self._config

    @property
    def jwk_client(self) -> PyJWKClient:
        if self._jwk_client is None:
            self._jwk_client = PyJWKClient(
                self.config["jwks_uri"],
                cache_keys=True,
                lifespan=3600  # Cache keys for 1 hour
            )
        return self._jwk_client

    def validate_id_token(
        self,
        id_token: str,
        nonce: Optional[str] = None
    ) -> dict:
        """
        Validate an OIDC ID token and return the decoded claims.

        Performs the following checks:
        1. Verifies the signature using the provider's public keys
        2. Validates the issuer (iss) matches the expected provider
        3. Validates the audience (aud) matches our client ID
        4. Checks the token has not expired (exp)
        5. Optionally verifies the nonce to prevent replay attacks
        """
        # Fetch the signing key based on the token's 'kid' header
        signing_key = self.jwk_client.get_signing_key_from_jwt(id_token)

        # Decode and validate the token
        try:
            claims = jwt.decode(
                id_token,
                signing_key.key,
                algorithms=["RS256", "RS384", "RS512"],
                issuer=self.issuer_url,
                audience=self.client_id,
                options={
                    "verify_exp": True,
                    "verify_iss": True,
                    "verify_aud": True,
                    "require": ["iss", "sub", "aud", "exp", "iat"]
                }
            )
        except jwt.ExpiredSignatureError:
            raise ValueError("ID token has expired")
        except jwt.InvalidIssuerError:
            raise ValueError(f"Invalid issuer. Expected {self.issuer_url}")
        except jwt.InvalidAudienceError:
            raise ValueError(f"Invalid audience. Expected {self.client_id}")
        except jwt.InvalidTokenError as e:
            raise ValueError(f"Invalid token: {e}")

        # Verify nonce if provided
        if nonce is not None:
            token_nonce = claims.get("nonce")
            if token_nonce != nonce:
                raise ValueError(
                    "Nonce mismatch: possible replay attack"
                )

        return claims

Usage:

validator = OIDCTokenValidator(
    issuer_url="https://accounts.google.com",
    client_id="your-client-id.apps.googleusercontent.com"
)

try:
    claims = validator.validate_id_token(
        id_token=raw_id_token,
        nonce=session_nonce  # The nonce you stored when initiating the flow
    )
    print(f"Authenticated user: {claims['sub']}")
    print(f"Email: {claims.get('email')}")
except ValueError as e:
    print(f"Authentication failed: {e}")

Full Integration Example with Flask

Here is a complete example integrating OIDC with a Flask application using the Authorization Code Flow:

import secrets
from flask import Flask, redirect, request, session, url_for, jsonify
import requests as http_requests
from urllib.parse import urlencode

app = Flask(__name__)
app.secret_key = secrets.token_hex(32)

# Configuration — works with any OIDC provider
OIDC_ISSUER = "https://accounts.google.com"
CLIENT_ID = "your-client-id.apps.googleusercontent.com"
CLIENT_SECRET = "your-client-secret"
REDIRECT_URI = "http://localhost:5000/callback"

# Initialize the validator
validator = OIDCTokenValidator(
    issuer_url=OIDC_ISSUER,
    client_id=CLIENT_ID
)


@app.route("/login")
def login():
    """Initiate the OIDC Authorization Code Flow."""
    # Generate a nonce and state for security
    nonce = secrets.token_urlsafe(32)
    state = secrets.token_urlsafe(32)

    # Store in session for later verification
    session["oidc_nonce"] = nonce
    session["oidc_state"] = state

    # Build the authorization URL
    config = validator.config
    params = {
        "client_id": CLIENT_ID,
        "response_type": "code",
        "scope": "openid profile email",
        "redirect_uri": REDIRECT_URI,
        "state": state,
        "nonce": nonce,
    }
    auth_url = f"{config['authorization_endpoint']}?{urlencode(params)}"
    return redirect(auth_url)


@app.route("/callback")
def callback():
    """Handle the OIDC callback after user authentication."""
    # Verify state to prevent CSRF
    if request.args.get("state") != session.pop("oidc_state", None):
        return "State mismatch — possible CSRF attack", 403

    # Check for errors from the provider
    if "error" in request.args:
        error = request.args["error"]
        desc = request.args.get("error_description", "Unknown error")
        return f"Authentication error: {error} — {desc}", 400

    # Exchange the authorization code for tokens
    code = request.args["code"]
    config = validator.config

    token_response = http_requests.post(
        config["token_endpoint"],
        data={
            "grant_type": "authorization_code",
            "code": code,
            "redirect_uri": REDIRECT_URI,
            "client_id": CLIENT_ID,
            "client_secret": CLIENT_SECRET,
        },
        timeout=10
    )
    token_response.raise_for_status()
    tokens = token_response.json()

    # Validate the ID token
    nonce = session.pop("oidc_nonce", None)
    try:
        claims = validator.validate_id_token(
            tokens["id_token"],
            nonce=nonce
        )
    except ValueError as e:
        return f"Token validation failed: {e}", 401

    # Create a session for the user
    session["user"] = {
        "sub": claims["sub"],
        "email": claims.get("email"),
        "name": claims.get("name"),
    }
    session["access_token"] = tokens.get("access_token")

    return redirect(url_for("profile"))


@app.route("/profile")
def profile():
    """Display the authenticated user's profile."""
    user = session.get("user")
    if not user:
        return redirect(url_for("login"))
    return jsonify(user)


@app.route("/logout")
def logout():
    """Log the user out by clearing the session."""
    session.clear()
    # Optionally redirect to the provider's end_session_endpoint
    config = validator.config
    if "end_session_endpoint" in config:
        params = {
            "post_logout_redirect_uri": "http://localhost:5000/",
            "client_id": CLIENT_ID,
        }
        logout_url = f"{config['end_session_endpoint']}?{urlencode(params)}"
        return redirect(logout_url)
    return redirect("/")

Integrating with Different Providers

Because OIDC is standardized, switching providers is largely a configuration change.

Google

OIDC_ISSUER = "https://accounts.google.com"
# Discovery: https://accounts.google.com/.well-known/openid-configuration

Auth0

OIDC_ISSUER = "https://your-tenant.auth0.com"
# Discovery: https://your-tenant.auth0.com/.well-known/openid-configuration

Keycloak

OIDC_ISSUER = "https://keycloak.example.com/realms/your-realm"
# Discovery: https://keycloak.example.com/realms/your-realm/.well-known/openid-configuration

The only things that change are the issuer URL, client ID, and client secret. The rest of the code remains identical. This is the power of the OIDC standard.

Session Management

Once a user is authenticated, you need to manage their session. There are several approaches:

Server-Side Sessions

Store session data on the server (in memory, Redis, or a database) and use a session cookie to identify the user:

import redis
import json
import secrets
from functools import wraps
from flask import Flask, request, g, make_response

app = Flask(__name__)
redis_client = redis.Redis(host="localhost", port=6379, db=0)

SESSION_TTL = 3600  # 1 hour


def create_session(user_claims: dict) -> str:
    """Create a server-side session and return the session ID."""
    session_id = secrets.token_urlsafe(32)
    session_data = {
        "sub": user_claims["sub"],
        "email": user_claims.get("email"),
        "name": user_claims.get("name"),
        "authenticated_at": user_claims.get("auth_time"),
    }
    redis_client.setex(
        f"session:{session_id}",
        SESSION_TTL,
        json.dumps(session_data)
    )
    return session_id


def require_auth(f):
    """Decorator that enforces authentication on a route."""
    @wraps(f)
    def decorated(*args, **kwargs):
        session_id = request.cookies.get("session_id")
        if not session_id:
            return {"error": "Not authenticated"}, 401

        session_data = redis_client.get(f"session:{session_id}")
        if not session_data:
            return {"error": "Session expired"}, 401

        g.user = json.loads(session_data)

        # Extend the session TTL on each request (sliding window)
        redis_client.expire(f"session:{session_id}", SESSION_TTL)

        return f(*args, **kwargs)
    return decorated


@app.route("/protected")
@require_auth
def protected():
    return {"message": f"Hello, {g.user['name']}"}

Token-Based Sessions

For stateless APIs, you might issue your own short-lived JWT after OIDC authentication:

import jwt
import datetime

APP_SECRET = secrets.token_hex(32)

def create_app_token(user_claims: dict) -> str:
    """Issue an application-specific JWT after OIDC authentication."""
    now = datetime.datetime.utcnow()
    payload = {
        "sub": user_claims["sub"],
        "email": user_claims.get("email"),
        "iat": now,
        "exp": now + datetime.timedelta(hours=1),
        "iss": "my-application",
    }
    return jwt.encode(payload, APP_SECRET, algorithm="HS256")


def verify_app_token(token: str) -> dict:
    """Verify an application-issued JWT."""
    return jwt.decode(
        token,
        APP_SECRET,
        algorithms=["HS256"],
        issuer="my-application"
    )

Security Considerations

When implementing OIDC, pay close attention to these security requirements:

Always validate the state parameter: This prevents CSRF attacks. Generate a random value before the authentication request, store it in the session, and verify it matches when the callback is invoked.

Always validate the nonce: This prevents token replay attacks. The nonce binds a specific ID token to a specific authentication request.

Validate all standard claims: Do not skip any validation step. Verify issuer, audience, expiration, and signature on every token.

Use HTTPS everywhere: Tokens are bearer credentials. Transmitting them over HTTP exposes them to interception.

Never expose tokens to the frontend unnecessarily: Keep tokens server-side when possible. If you must send them to a browser, use httpOnly and secure cookie flags.

Handle key rotation: Identity providers rotate their signing keys periodically. Use the kid (Key ID) header claim to select the correct key, and cache keys with a reasonable TTL so your application picks up rotations.

# Example: Handling key rotation gracefully
try:
    claims = validator.validate_id_token(token)
except jwt.exceptions.PyJWKClientError:
    # Key not found — provider may have rotated keys
    # Clear the cache and retry once
    validator._jwk_client = None
    claims = validator.validate_id_token(token)

Summary

OpenID Connect provides a standardized, interoperable identity layer on top of OAuth 2.0. By understanding the distinction between ID tokens and access tokens, properly validating JWTs, leveraging the discovery endpoint for provider-agnostic integration, and implementing secure session management, you can build authentication systems that are both robust and portable across identity providers. The protocol's design means that switching from Google to Auth0 to Keycloak — or supporting all three simultaneously — requires only configuration changes, not code changes.

Common Problems & Solutions

1 ID token validation fails with 'Invalid audience' even though the correct client ID is configured.
Cause: Some providers include multiple values in the 'aud' claim (e.g., when the token is issued for both a web and mobile client). The PyJWT library's audience validation may behave unexpectedly if the 'aud' claim is an array and your client_id is only one of the entries. Additionally, trailing slashes or case differences in the client ID can cause mismatches.
Solution: Ensure you are passing the exact client_id string that the provider expects. If the provider returns an array for 'aud', PyJWT handles this correctly as long as your client_id is one of the values. Also check the 'azp' (authorized party) claim — if present, it should match your client_id. Debug by decoding the token without verification first to inspect the actual 'aud' value.

Debug: inspect the raw claims before validation

import jwt as pyjwt unverified = pyjwt.decode(id_token, options={"verify_signature": False}) print(f"aud claim: {unverified['aud']}") print(f"aud type: {type(unverified['aud'])}") print(f"Expected client_id: {CLIENT_ID}")

2 Token signature verification fails intermittently with 'Unable to find a signing key that matches kid'.
Symptoms: jwt.exceptions.PyJWKClientError: Unable to find a signing key that matches: 'new-kid-after-rotation'
Cause: Identity providers rotate their signing keys periodically. If your application caches the JWKS (JSON Web Key Set) indefinitely, it will eventually hold stale keys that do not match the 'kid' in newly issued tokens.
Solution: Implement a cache-with-retry strategy. Cache the JWKS for a reasonable period (e.g., 1 hour), but if signature verification fails due to an unknown kid, clear the cache and refetch the JWKS once before failing permanently. PyJWT's PyJWKClient supports a lifespan parameter for this purpose.

from jwt import PyJWKClient

Configure caching with automatic refresh

jwk_client = PyJWKClient( jwks_uri, cache_keys=True, lifespan=3600 # Refetch keys after 1 hour )

On kid mismatch, force a refresh

try: key = jwk_client.get_signing_key_from_jwt(token) except Exception: jwk_client = PyJWKClient(jwks_uri, cache_keys=True, lifespan=3600) key = jwk_client.get_signing_key_from_jwt(token)

3 After a successful OIDC login, the user's session is lost on the next request or the callback fails with a 'state mismatch' error.
Cause: The session cookie is not being persisted between requests. This commonly happens when: (1) the Flask secret_key is regenerated on each restart (e.g., using secrets.token_hex() at module level without persisting it), (2) the cookie's SameSite attribute prevents it from being sent on the redirect back from the provider, or (3) the application is behind a reverse proxy and the Secure flag prevents the cookie from being sent over the proxy's HTTP connection.
Solution: Set a stable secret_key from an environment variable. Configure session cookies with appropriate SameSite and Secure settings. If behind a reverse proxy, ensure the proxy forwards the correct protocol headers and configure Flask to trust them.

import os from flask import Flask

app = Flask(name)

Use a stable secret key from the environment

app.secret_key = os.environ["FLASK_SECRET_KEY"]

Configure session cookies

app.config.update( SESSION_COOKIE_SECURE=True, # Only send over HTTPS SESSION_COOKIE_HTTPONLY=True, # Not accessible via JavaScript SESSION_COOKIE_SAMESITE="Lax", # Allow redirects from IdP )

If behind a reverse proxy:

from werkzeug.middleware.proxy_fix import ProxyFix app.wsgi_app = ProxyFix(app.wsgi_app, x_proto=1, x_host=1)
