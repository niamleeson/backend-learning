SLIs, SLOs, SLAs: Defining Reliability Targets

Intuition

SLIs, SLOs, and SLAs are like a driving analogy. SLIs are your speedometer reading (actual measurement). SLOs are the speed limit you set for yourself, say 65 mph (internal target). SLAs are the posted legal speed limit of 70 mph (contractual, with penalties). You aim for 65 so you have buffer before getting a ticket at 70. The error budget is how many mph you can go over your personal limit before hitting the legal limit.

SLIs, SLOs, SLAs: Defining Reliability Targets

Introduction

How reliable should your system be? 99.9%? 99.99%? The answer depends on user expectations, business impact, and engineering cost. SLIs (Service Level Indicators), SLOs (Service Level Objectives), and SLAs (Service Level Agreements) provide a framework for defining, measuring, and communicating reliability targets.

The Reliability Pyramid

SLIs (Service Level Indicators)

SLIs are quantitative measures of service behavior:

Common SLIs:
┌─────────────────┬──────────────────────────────────────┐
│ SLI Type        │ Measurement                          │
├─────────────────┼──────────────────────────────────────┤
│ Availability    │ % of requests that succeed            │
│ Latency         │ % of requests faster than threshold   │
│ Throughput      │ Requests processed per second          │
│ Error rate      │ % of requests that return errors       │
│ Correctness     │ % of responses with correct data       │
│ Freshness       │ % of data updated within threshold     │
└─────────────────┴──────────────────────────────────────┘

Measuring SLIs in Node.js

import { metrics } from '@opentelemetry/api';
import client from 'prom-client';

// Prometheus metrics
const httpRequestDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration in seconds',
  labelNames: ['method', 'route', 'status_code'],
  buckets: [0.01, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10],
});

const httpRequestsTotal = new client.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'route', 'status_code'],
});

const httpErrorsTotal = new client.Counter({
  name: 'http_errors_total',
  help: 'Total HTTP errors (5xx)',
  labelNames: ['method', 'route'],
});

// Middleware to collect SLIs
app.use((req, res, next) => {
  const start = process.hrtime.bigint();

  res.on('finish', () => {
    const durationMs = Number(process.hrtime.bigint() - start) / 1e6;
    const durationS = durationMs / 1000;
    const route = req.route?.path || req.path;

    httpRequestDuration.observe(
      { method: req.method, route, status_code: res.statusCode },
      durationS
    );

    httpRequestsTotal.inc(
      { method: req.method, route, status_code: res.statusCode }
    );

    if (res.statusCode >= 500) {
      httpErrorsTotal.inc({ method: req.method, route });
    }
  });

  next();
});

// Expose metrics endpoint
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', client.register.contentType);
  res.end(await client.register.metrics());
});

SLOs (Service Level Objectives)

SLOs are targets for your SLIs:

Example SLOs for an API service:

┌─────────────────────────────────────────────────────┐
│ SLI               │ SLO                             │
├───────────────────┼─────────────────────────────────┤
│ Availability      │ 99.9% of requests return 2xx/4xx│
│ Latency (p50)     │ 50% of requests < 100ms         │
│ Latency (p99)     │ 99% of requests < 500ms         │
│ Error rate        │ < 0.1% of requests return 5xx   │
└─────────────────────────────────────────────────────┘

99.9% availability = "three nines"
  = 8.76 hours of downtime per year
  = 43.8 minutes per month
  = 1.44 minutes per day

99.99% = "four nines"
  = 52.6 minutes per year
  = 4.38 minutes per month

Error Budget

Error Budget = 1 - SLO target

If SLO = 99.9% availability:
  Error Budget = 0.1% = 43.8 minutes/month

Month starts: 43.8 minutes of allowed downtime
Week 1: 5-minute outage → 38.8 minutes remaining
Week 2: 2-minute outage → 36.8 minutes remaining
Week 3: No outages → 36.8 minutes remaining

Error budget remaining: 36.8 minutes
  → Safe to deploy risky changes

If error budget is exhausted:
  → Freeze feature deployments
  → Focus on reliability improvements

// Error budget calculator
class ErrorBudgetTracker {
  constructor(sloTarget, windowDays = 30) {
    this.sloTarget = sloTarget;
    this.windowMs = windowDays * 24 * 60 * 60 * 1000;
  }

  async calculate(prometheusClient) {
    const windowStart = Date.now() - this.windowMs;

    // Query actual availability from Prometheus
    const totalRequests = await prometheusClient.query(
      `sum(increase(http_requests_total[${this.windowMs / 1000}s]))`
    );
    const errorRequests = await prometheusClient.query(
      `sum(increase(http_errors_total[${this.windowMs / 1000}s]))`
    );

    const actualAvailability = 1 - (errorRequests / totalRequests);
    const errorBudgetTotal = 1 - this.sloTarget;
    const errorBudgetUsed = 1 - actualAvailability;
    const errorBudgetRemaining = errorBudgetTotal - errorBudgetUsed;

    return {
      sloTarget: this.sloTarget,
      actualAvailability: (actualAvailability * 100).toFixed(3) + '%',
      errorBudgetTotal: (errorBudgetTotal * 100).toFixed(3) + '%',
      errorBudgetUsed: (errorBudgetUsed * 100).toFixed(3) + '%',
      errorBudgetRemaining: (errorBudgetRemaining * 100).toFixed(3) + '%',
      errorBudgetRemainingMinutes: (errorBudgetRemaining * this.windowMs / 60000).toFixed(1),
      canDeploy: errorBudgetRemaining > 0,
    };
  }
}

SLAs (Service Level Agreements)

SLAs are contractual commitments with financial consequences:

Example SLA (simplified):
┌─────────────────────────────────────────────────────────┐
│ Service Availability │ Service Credit                    │
├──────────────────────┼───────────────────────────────────┤
│ >= 99.9%             │ None (within target)              │
│ 99.0% - 99.9%       │ 10% credit on monthly bill        │
│ 95.0% - 99.0%       │ 25% credit on monthly bill        │
│ < 95.0%              │ 50% credit on monthly bill        │
└──────────────────────┴───────────────────────────────────┘

SLO should always be stricter than SLA:
  SLA: 99.9% availability (contractual)
  SLO: 99.95% availability (internal target)
  Gap: 0.05% = buffer before SLA breach

Choosing the Right SLOs

Questions to ask:
1. What do users actually care about?
   → Availability and latency, usually

2. What latency is "good enough"?
   → Measure current performance, set targets slightly above

3. What's the cost of each nine?
   → Each additional nine roughly doubles engineering cost

Service Type and Suggested SLOs:
┌────────────────────────────────────────────────────────┐
│ Service Type          │ Availability │ Latency (p99)   │
├───────────────────────┼──────────────┼─────────────────┤
│ Payment processing    │ 99.99%       │ < 1s            │
│ User-facing API       │ 99.9%        │ < 500ms         │
│ Internal API          │ 99.5%        │ < 1s            │
│ Batch processing      │ 99%          │ N/A             │
│ Analytics dashboard   │ 99%          │ < 5s            │
│ Email sending         │ 99.5%        │ < 30s           │
└────────────────────────────────────────────────────────┘

Prometheus queries for SLO dashboard:

Availability (last 30 days):
  1 - (
    sum(rate(http_errors_total[30d]))
    /
    sum(rate(http_requests_total[30d]))
  )

Latency P99 (last 5 minutes):
  histogram_quantile(0.99,
    sum(rate(http_request_duration_seconds_bucket[5m])) by (le)
  )

Error budget remaining:
  (1 - 0.999) - (
    sum(rate(http_errors_total[30d]))
    /
    sum(rate(http_requests_total[30d]))
  )

Burn rate (how fast error budget is being consumed):
  sum(rate(http_errors_total[1h])) / sum(rate(http_requests_total[1h]))
  /
  (1 - 0.999)

Common Pitfalls

  1. Too many SLOs: Start with 2-3 SLOs per service. More creates confusion and alert fatigue.
  2. SLOs that don't reflect user experience: "CPU usage < 80%" is not an SLO. "99% of requests < 500ms" is.
  3. No error budget policy: Without consequences for exhausting the error budget, SLOs become meaningless.
  4. Setting SLOs without data: Measure first, then set targets based on actual performance and user expectations.

Summary

SLIs measure your service's behavior (latency, availability, error rate). SLOs set targets for those measurements ("99.9% availability"). SLAs are contractual commitments with financial consequences. Error budgets translate SLOs into actionable deployment decisions. Start by measuring SLIs, set SLOs based on user expectations, and use error budgets to balance reliability with feature velocity.

Common Problems & Solutions

1 Alert fatigue from too many SLO-based alerts
Cause: Alerting on every SLO violation creates noise. Minor, transient blips trigger alerts.
Solution: Use burn rate alerting: alert when the error budget is being consumed faster than expected (e.g., would exhaust in 6 hours at current rate), not on individual violations.
2 SLO shows 99.95% but users are complaining
Cause: SLI doesn't capture what users actually experience. Average latency is good but p99 is terrible, or errors only affect a specific user segment.
Solution: Measure SLIs at the user-journey level, not just the endpoint level. Include p99 latency, not just p50. Segment by user type or geography.
