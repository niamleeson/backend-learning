Client-Server Model and Request Lifecycle

Intuition

Think of a restaurant: the client is a customer placing an order, the waiter (HTTP) carries it to the kitchen (server), which coordinates cooks (services), pantry (database), and prep station (cache) to assemble the meal (response) and send it back.

Client-Server Model and Request Lifecycle

Introduction

Every backend system you will ever build rests on the client-server model — a foundational architectural pattern where clients send requests and servers produce responses. Understanding the full lifecycle of a request, from the moment a user clicks a button to the instant a response renders on screen, is the single most important mental model for system design.

The Client-Server Model

At its core, the client-server model divides computing into two roles:

  • Client: Initiates requests (browser, mobile app, CLI tool, another service)
  • Server: Listens for requests, processes them, and returns responses
┌──────────┐         HTTP Request          ┌──────────┐
│          │ ──────────────────────────────>│          │
│  Client  │                               │  Server  │
│          │ <─────────────────────────────-│          │
└──────────┘         HTTP Response          └──────────┘

This seems simple, but in modern systems the "server" is rarely a single machine. It is a distributed system of load balancers, application servers, caches, databases, and message queues.

DNS Resolution

When a client types https://api.example.com/users into a browser, the very first step is DNS resolution:

┌────────┐    ┌──────────────┐    ┌────────────┐    ┌──────────────┐
│ Client │───>│ DNS Resolver  │───>│ Root NS    │───>│ .com NS      │
│        │    │ (ISP/Google)  │    │            │    │              │
│        │    └──────────────┘    └────────────┘    └──────────────┘
│        │           │                                      │
│        │           │         ┌─────────────────┐          │
│        │           └────────>│ example.com NS  │<─────────┘
│        │                     │ Returns IP:     │
│        │<────────────────────│ 93.184.216.34   │
└────────┘                     └─────────────────┘

In Node.js, DNS resolution happens automatically in http and https modules, but you can inspect it:

import dns from 'node:dns/promises';

async function resolveDomain() {
  const addresses = await dns.resolve4('api.example.com');
  console.log('IP addresses:', addresses);

  const full = await dns.resolveAny('api.example.com');
  console.log('All records:', full);
}

resolveDomain();

TCP Connection and TLS Handshake

Once the IP is resolved, the client establishes a TCP connection via a three-way handshake, followed by a TLS handshake for HTTPS:

Client                          Server
  │                               │
  │──── SYN ─────────────────────>│   TCP
  │<─── SYN-ACK ─────────────────│   Three-Way
  │──── ACK ─────────────────────>│   Handshake
  │                               │
  │──── ClientHello ─────────────>│   TLS
  │<─── ServerHello + Cert ───────│   Handshake
  │──── Key Exchange ────────────>│   (adds ~1-2 RTT)
  │<─── Finished ────────────────-│
  │                               │
  │──── HTTP Request ────────────>│   Application
  │<─── HTTP Response ───────────-│   Data

The TCP + TLS handshake adds latency. This is why connection reuse (HTTP keep-alive, HTTP/2 multiplexing) matters enormously for performance.

The HTTP Request

An HTTP request consists of:

GET /api/users?page=1&limit=20 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Accept: application/json
Connection: keep-alive

In Express.js, you can inspect every part of this:

import express from 'express';
const app = express();

app.use((req, res, next) => {
  console.log({
    method: req.method,           // GET
    path: req.path,               // /api/users
    query: req.query,             // { page: '1', limit: '20' }
    headers: req.headers,         // all headers
    ip: req.ip,                   // client IP
    protocol: req.protocol,       // https
    hostname: req.hostname,       // api.example.com
  });
  next();
});

Server-Side Request Lifecycle

Once the request hits your Node.js server, it goes through a well-defined pipeline:

┌─────────────────────────────────────────────────────────┐
│                    Node.js Server                        │
│                                                         │
│  Incoming ──> Middleware ──> Route ──> Controller ──>    │
│  Request      Pipeline      Match     Logic             │
│                                                         │
│               │ CORS        │         │ Validate        │
│               │ Auth        │         │ Query DB        │
│               │ Logging     │         │ Transform       │
│               │ Body Parse  │         │ Respond         │
│                                                         │
│  <── Response <── Error Handler <── Service Layer <──   │
└─────────────────────────────────────────────────────────┘

Here is a complete Express example showing the full lifecycle:

import express from 'express';
import { PrismaClient } from '@prisma/client';

const app = express();
const prisma = new PrismaClient();

// 1. Body parsing middleware
app.use(express.json());

// 2. Logging middleware
app.use((req, res, next) => {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    console.log(`${req.method} ${req.path} ${res.statusCode} ${duration}ms`);
  });
  next();
});

// 3. Auth middleware
function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'No token' });
  try {
    req.user = verifyToken(token);
    next();
  } catch {
    res.status(401).json({ error: 'Invalid token' });
  }
}

// 4. Route handler (controller)
app.get('/api/users', authenticate, async (req, res, next) => {
  try {
    const page = parseInt(req.query.page) || 1;
    const limit = parseInt(req.query.limit) || 20;
    const skip = (page - 1) * limit;

    const [users, total] = await Promise.all([
      prisma.user.findMany({ skip, take: limit }),
      prisma.user.count(),
    ]);

    res.json({
      data: users,
      meta: { page, limit, total, pages: Math.ceil(total / limit) },
    });
  } catch (err) {
    next(err);
  }
});

// 5. Global error handler
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ error: 'Internal server error' });
});

app.listen(3000);

The Event Loop's Role

Node.js processes requests using a single-threaded event loop. Understanding this is critical for system design:

   ┌───────────────────────────┐
┌─>│           timers          │  setTimeout, setInterval
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks     │  I/O callbacks
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │  internal use
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           poll            │  incoming connections, data
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           check           │  setImmediate
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │      close callbacks      │  socket.on('close')
│  └─────────────┬─────────────┘
└─────────────────┘

This single-threaded model means a CPU-heavy synchronous operation blocks ALL concurrent requests. This is why you must never do CPU-intensive work on the main thread.

// BAD: Blocks the event loop
app.get('/hash', (req, res) => {
  const result = computeExpensiveHash(req.query.data); // blocks!
  res.json({ result });
});

// GOOD: Offload to worker thread
import { Worker } from 'node:worker_threads';

app.get('/hash', (req, res) => {
  const worker = new Worker('./hash-worker.js', {
    workerData: req.query.data,
  });
  worker.on('message', (result) => res.json({ result }));
  worker.on('error', (err) => res.status(500).json({ error: err.message }));
});

In a real production system, the request lifecycle is much more complex:

  ▼

  │
  ▼
Load Balancer (L4/L7 - Nginx, ALB, HAProxy)
  │
  ▼
API Gateway (rate limiting, auth, routing)
  │
  ▼
Application Server (Node.js)
  │
  ├──> In-Memory Cache (node-cache, LRU)
  ├──> Distributed Cache (Redis via ioredis)
  ├──> Database (PostgreSQL via Prisma)
  ├──> Message Queue (BullMQ / Kafka)
  └──> External Services (payment, email, etc.)
  │
  ▼
Response travels back through the same chain

Keep-Alive and Connection Pooling

Connection reuse is a key optimization. Node.js HTTP clients support keep-alive:

import http from 'node:http';

// Create a reusable agent with keep-alive
const agent = new http.Agent({
  keepAlive: true,
  maxSockets: 50,
  maxFreeSockets: 10,
  timeout: 60000,
});

// Use with fetch or http.request
const response = await fetch('http://internal-service/api/data', {
  agent,
});

For database connections, use connection pooling (Prisma does this automatically):

// Prisma connection pool configuration
// In schema.prisma:
// datasource db {
//   provider = "postgresql"
//   url      = env("DATABASE_URL")  // ?connection_limit=20&pool_timeout=10
// }

Common Pitfalls

  1. Forgetting DNS caching: DNS lookups for external services add latency on every request. Use cacheable-lookup or configure DNS TTL.
  2. Not reusing connections: Creating new TCP connections per request wastes time on handshakes.
  3. Blocking the event loop: Synchronous file reads, JSON parsing of huge payloads, or regex on user input.
  4. Ignoring response lifecycle: Not calling res.end() or next() causes requests to hang.

Summary

The client-server request lifecycle involves DNS resolution, TCP/TLS handshake, HTTP request/response, middleware pipeline, business logic, database queries, and response serialization. In Node.js, the single-threaded event loop handles concurrency through async I/O. Understanding each step lets you identify bottlenecks and design systems that scale.

    Hint

Common Problems & Solutions

1 Requests to the server are timing out intermittently
Symptoms: Some requests complete in <50ms while others take 5+ seconds. Server CPU shows 100% on one core.
Cause: A synchronous operation (large JSON parse, crypto, or regex) is blocking the event loop, preventing other requests from being processed.
Solution: Profile with `clinic doctor` or `--prof` flag. Offload CPU-intensive work to worker threads. Use streaming JSON parsers for large payloads.
2 High latency on first request after idle period
Cause: TCP connections and DNS cache have expired. The first request must re-establish connections (DNS lookup + TCP handshake + TLS handshake).
Solution: Enable HTTP keep-alive with appropriate timeouts. Use DNS caching libraries. Implement health check endpoints that keep connections warm.
3 Express app crashes with 'socket hang up' errors
Cause: Upstream service or database closed the connection before the response was sent, often due to timeout mismatch.
Solution: Set appropriate timeouts on all HTTP clients and database connections. Implement retry logic with exponential backoff. Ensure server timeout > upstream timeout.
