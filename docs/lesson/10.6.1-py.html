Design a Distributed Message Queue (Kafka)

Intuition

A distributed message queue is an append-only distributed log where the key insight is that sequential disk I/O (appending) is fast enough to rival network throughput, eliminating the need for complex in-memory caching. Partitions are the unit of parallelism, ordering, and fault tolerance all at once.

Design a Distributed Message Queue (Kafka)

Requirements

  • Publish and consume millions of messages per second with low latency
  • Guarantee ordering within a partition and at-least-once delivery
  • Support consumer groups, offset management, and message replay
  • Retain messages for configurable durations (hours to weeks)

Architecture

Producers send messages to a cluster of brokers. Each topic is split into partitions, distributed across brokers. Messages within a partition are appended to an immutable, sequential log. Consumers in a group are each assigned a subset of partitions and track their offset. Replication across brokers provides fault tolerance.

Key Components

  • Broker: Stores partition log segments on disk, serves produce/consume requests
  • Partition Log: Append-only file with sequential offsets; zero-copy reads via sendfile()
  • Consumer Group Coordinator: Assigns partitions to consumers, manages rebalancing
  • Replication Manager: Maintains in-sync replicas (ISR) with leader-follower protocol

Scaling

Add partitions to increase parallelism; add brokers and rebalance partitions. Producers partition by key hash for ordering guarantees. Use tiered storage to offload old segments to S3.

Trade-offs

  • More partitions increase parallelism but add rebalancing overhead and metadata cost
  • At-least-once is default; exactly-once requires idempotent producers and transactional consumers
  • Long retention enables replay but increases storage cost; tiered storage mitigates this

Exercises

  1. Implement a Partition Log with Segment Rotation

    Build an append-only log that writes to segment files. When a segment exceeds 1MB, rotate to a new segment. Support reading from any offset across segments.

    Hint

    Maintain an index mapping offset ranges to segment files. On read, binary search the index to find the right segment, then seek to the offset within it.

  2. Build a Consumer Group Rebalancer

    Implement partition assignment for a consumer group: when consumers join or leave, reassign partitions evenly. Test with 12 partitions and dynamically adding/removing consumers.

    Hint

    Use the round-robin or range assignor strategy. On rebalance, sort partitions and consumers, then assign partitions[i] to consumer[i % num_consumers].

Common Problems & Solutions

1 Consumer rebalances cause processing pauses of 30+ seconds during deployments
Cause: The stop-the-world rebalance protocol revokes all partitions before reassigning, causing a full pause
Solution: Use incremental cooperative rebalancing: only revoke partitions that are moving, allowing consumers to continue processing their retained partitions.
2 A slow consumer causes the entire consumer group to lag behind
Cause: One consumer cannot keep up with its assigned partitions, and the group cannot make progress past those offsets
Solution: Monitor per-consumer lag and reassign partitions from slow consumers. Consider splitting the slow consumer's partitions to other group members or scaling out.
