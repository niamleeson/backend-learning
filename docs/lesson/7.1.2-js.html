SQL Injection: Attack Vectors and Prevention

Intuition

Imagine a bank teller who reads instructions from a slip of paper the customer fills out. SQL injection is like a customer writing 'withdraw $100 from account 123 and also transfer all money from every account to account 999' on the slip, and the teller blindly following every instruction. Parameterized queries are like giving the teller a pre-printed form with fixed instructions and blank fields -- no matter what the customer writes in the blanks, the teller only performs the pre-defined action.

SQL Injection: Attack Vectors and Prevention

What Is SQL Injection?

SQL injection (SQLi) is one of the oldest and most devastating web application vulnerabilities. It occurs when an attacker can insert or manipulate SQL statements through user-controlled input that gets incorporated into database queries. Despite being well-understood for over two decades, SQL injection remains in the OWASP Top 10 and continues to cause real-world breaches.

The core problem is simple: treating user input as code instead of data.

How SQL Injection Works

Normal Login Flow:

User Input:  email = "alice@example.com", password = "secret123"
Built Query: SELECT * FROM users WHERE email = 'alice@example.com' AND password = '...'
Result:      Returns Alice's row (or nothing if password is wrong)


SQL Injection Attack:

User Input:  email = "' OR '1'='1' --", password = "anything"
Built Query: SELECT * FROM users WHERE email = '' OR '1'='1' --' AND password = '...'
                                                   ^^^^^^^^^^  ^^
                                                   Always true  Comment (ignores rest)
Result:      Returns ALL users -- attacker bypasses authentication

The -- is a SQL comment that causes the database to ignore the rest of the query, including the password check.

Types of SQL Injection

1. Classic (In-Band) SQL Injection

The attacker gets results directly in the application response.

// VULNERABLE: String concatenation builds a dynamic query
app.get('/api/users', async (req, res) => {
  const { search } = req.query;
  // If search = "'; DROP TABLE users; --"
  // The query becomes: SELECT * FROM users WHERE name = ''; DROP TABLE users; --'
  const result = await pool.query(
    `SELECT * FROM users WHERE name = '${search}'`
  );
  res.json(result.rows);
});

2. Blind SQL Injection

The attacker does not see query results directly but infers information based on application behavior.

Boolean-based blind injection:

Request:  /api/users?id=1 AND 1=1    --> Returns user (condition true)
Request:  /api/users?id=1 AND 1=2    --> Returns nothing (condition false)

The attacker now knows they can inject SQL and extract data one bit at a time:
/api/users?id=1 AND (SELECT LENGTH(password) FROM users WHERE id=1) > 10

Time-based blind injection:

Request:  /api/users?id=1; SELECT CASE WHEN (1=1) THEN pg_sleep(5) ELSE pg_sleep(0) END--
Result:   Response takes 5 seconds --> condition is true

The attacker extracts data by measuring response times.

3. Second-Order SQL Injection

The malicious input is stored first and executed later in a different query.

// Step 1: Attacker registers with a crafted username
// Username: admin'--
await pool.query(
  `INSERT INTO users (username, email) VALUES ($1, $2)`,
  [username, email]  // This insert is safe -- parameterized
);

// Step 2: Later, a different part of the app uses the stored username unsafely
const user = await pool.query('SELECT * FROM users WHERE id = $1', [userId]);
const username = user.rows[0].username; // "admin'--"

// VULNERABLE: Using the stored value without parameterization
const logs = await pool.query(
  `SELECT * FROM audit_log WHERE username = '${username}'`
  // Becomes: SELECT * FROM audit_log WHERE username = 'admin'--'
);

4. UNION-Based SQL Injection

The attacker appends a UNION SELECT to extract data from other tables.

Normal:   SELECT name, email FROM users WHERE id = 1
Attack:   SELECT name, email FROM users WHERE id = 1 UNION SELECT username, password FROM admin_users--

Result: The response now contains admin credentials

// VULNERABLE
app.get('/api/products', async (req, res) => {
  const { category } = req.query;
  // category = "electronics' UNION SELECT username, password FROM users--"
  const result = await pool.query(
    `SELECT name, price FROM products WHERE category = '${category}'`
  );
  res.json(result.rows);
  // Response contains usernames and passwords!
});

Attack Vectors in Node.js Applications

Raw Query String Concatenation

This is the most obvious vector:

// VULNERABLE: Every string interpolation in a query is a potential injection point
const query = `SELECT * FROM products
  WHERE name LIKE '%${search}%'
  AND category = '${category}'
  AND price < ${maxPrice}
  ORDER BY ${sortColumn} ${sortDirection}`;

Every single variable in that query is an injection point.

Dynamic Table and Column Names

Parameterized queries cannot protect table or column names:

// VULNERABLE: Parameterization does NOT work for identifiers
const result = await pool.query(
  `SELECT * FROM ${tableName} ORDER BY ${sortColumn}`,
);
// tableName = "users; DROP TABLE users--"

// SECURE: Whitelist allowed values
const ALLOWED_TABLES = ['products', 'categories', 'reviews'];
const ALLOWED_SORT_COLUMNS = ['name', 'price', 'created_at'];

if (!ALLOWED_TABLES.includes(tableName)) {
  return res.status(400).json({ error: 'Invalid table' });
}
if (!ALLOWED_SORT_COLUMNS.includes(sortColumn)) {
  return res.status(400).json({ error: 'Invalid sort column' });
}

const result = await pool.query(
  `SELECT * FROM ${tableName} ORDER BY ${sortColumn}`
);

NoSQL Injection (MongoDB)

SQL injection concepts apply to NoSQL databases too:

// VULNERABLE: MongoDB operator injection
app.post('/api/login', async (req, res) => {
  const { username, password } = req.body;
  // If body is: { "username": {"$gt": ""}, "password": {"$gt": ""} }
  // This matches any user with a non-empty username and password
  const user = await db.collection('users').findOne({
    username: username,
    password: password,
  });
});

// SECURE: Validate types before querying
app.post('/api/login', async (req, res) => {
  const { username, password } = req.body;

  if (typeof username !== 'string' || typeof password !== 'string') {
    return res.status(400).json({ error: 'Invalid input' });
  }

  const user = await db.collection('users').findOne({
    username: username,
    password: password,  // In real code, compare hashed passwords
  });
});

Prevention: Defense in Depth

Layer 1: Parameterized Queries (Primary Defense)

Parameterized queries (also called prepared statements) separate SQL code from data. The database treats parameters as literal values, never as SQL code.

Parameterized Query Flow:

+------------------+     +-----------------+     +------------------+
|  Application     |     |  Database       |     |  Query Plan      |
|                  |     |                 |     |                  |
|  Query Template: |---->|  Parse + Plan   |---->|  Cached plan     |
|  SELECT * FROM   |     |  (no user data) |     |  for template    |
|  users WHERE     |     |                 |     |                  |
|  email = $1      |     +-----------------+     +------------------+
|                  |                                     |
|  Parameter:      |     +-----------------+             |
|  "alice@test.com"|---->|  Bind + Execute |<------------+
|                  |     |  (data only)    |
+------------------+     +-----------------+

The user input NEVER becomes part of the SQL syntax.
Even if input is: "'; DROP TABLE users--"
It is treated as the literal string: '; DROP TABLE users--

With pg (node-postgres)

const { Pool } = require('pg');
const pool = new Pool();

// Parameterized query with positional parameters ($1, $2, ...)
async function findUser(email) {
  const result = await pool.query(
    'SELECT id, email, name FROM users WHERE email = $1',
    [email]
  );
  return result.rows[0];
}

// Multiple parameters
async function findProducts(category, minPrice, maxPrice) {
  const result = await pool.query(
    `SELECT * FROM products
     WHERE category = $1
     AND price BETWEEN $2 AND $3
     ORDER BY price ASC`,
    [category, minPrice, maxPrice]
  );
  return result.rows;
}

// LIKE queries with parameterization
async function searchUsers(searchTerm) {
  const result = await pool.query(
    'SELECT * FROM users WHERE name ILIKE $1',
    [`%${searchTerm}%`]  // The % wildcards go in the parameter, not the query
  );
  return result.rows;
}

// IN clause with parameterization
async function findUsersByIds(ids) {
  // Generate $1, $2, $3... for the number of IDs
  const placeholders = ids.map((_, i) => `$${i + 1}`).join(', ');
  const result = await pool.query(
    `SELECT * FROM users WHERE id IN (${placeholders})`,
    ids
  );
  return result.rows;
}

With Knex.js

const knex = require('knex')(config);

// Knex automatically parameterizes
async function findUser(email) {
  return knex('users')
    .where({ email })
    .first();
}

// Complex queries
async function searchProducts(filters) {
  let query = knex('products').select('*');

  if (filters.category) {
    query = query.where('category', filters.category);
  }
  if (filters.minPrice) {
    query = query.where('price', '>=', filters.minPrice);
  }
  if (filters.search) {
    query = query.where('name', 'ilike', `%${filters.search}%`);
  }

  return query;
}

// Raw queries in Knex -- use bindings!
async function customQuery(userId) {
  // SECURE: Using ? placeholders with bindings array
  return knex.raw(
    'SELECT * FROM users WHERE id = ? AND active = ?',
    [userId, true]
  );
}

With Prisma

Prisma parameterizes all queries by default through its query builder:

const { PrismaClient } = require('@prisma/client');
const prisma = new PrismaClient();

// Always safe -- Prisma parameterizes automatically
async function findUser(email) {
  return prisma.user.findUnique({ where: { email } });
}

async function searchProducts(search, category) {
  return prisma.product.findMany({
    where: {
      name: { contains: search, mode: 'insensitive' },
      category: category,
    },
  });
}

// If you must use raw SQL with Prisma, use Prisma.sql or $queryRaw
async function rawSearch(term) {
  // SECURE: Tagged template literal parameterizes automatically
  return prisma.$queryRaw`
    SELECT * FROM users WHERE name ILIKE ${`%${term}%`}
  `;
}

// VULNERABLE: Using $queryRawUnsafe with string interpolation
async function unsafeSearch(term) {
  // NEVER do this:
  return prisma.$queryRawUnsafe(
    `SELECT * FROM users WHERE name = '${term}'`
  );
}

Layer 2: Input Validation

Validation is your second line of defense. Even with parameterized queries, validating input catches malformed data early.

const { z } = require('zod');

const ProductSearchSchema = z.object({
  search: z.string().max(100).optional(),
  category: z.enum(['electronics', 'clothing', 'books', 'food']).optional(),
  minPrice: z.coerce.number().min(0).max(1000000).optional(),
  maxPrice: z.coerce.number().min(0).max(1000000).optional(),
  sortBy: z.enum(['name', 'price', 'created_at']).optional().default('name'),
  order: z.enum(['asc', 'desc']).optional().default('asc'),
  page: z.coerce.number().int().min(1).optional().default(1),
  limit: z.coerce.number().int().min(1).max(100).optional().default(20),
});

app.get('/api/products', async (req, res) => {
  const parsed = ProductSearchSchema.safeParse(req.query);

  if (!parsed.success) {
    return res.status(400).json({ errors: parsed.error.issues });
  }

  const { search, category, minPrice, maxPrice, sortBy, order, page, limit } = parsed.data;
  const offset = (page - 1) * limit;

  // All values are now validated and safe to use
  const result = await pool.query(
    `SELECT * FROM products
     WHERE ($1::text IS NULL OR name ILIKE $1)
     AND ($2::text IS NULL OR category = $2)
     AND ($3::numeric IS NULL OR price >= $3)
     AND ($4::numeric IS NULL OR price <= $4)
     ORDER BY ${sortBy} ${order}
     LIMIT $5 OFFSET $6`,
    [
      search ? `%${search}%` : null,
      category || null,
      minPrice ?? null,
      maxPrice ?? null,
      limit,
      offset,
    ]
  );

  res.json(result.rows);
});

Layer 3: Principle of Least Privilege

-- Create a restricted database user for the application
CREATE USER app_user WITH PASSWORD 'strong_random_password';

-- Grant only what is needed
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;

-- NEVER grant these to the application user:
-- DROP, CREATE, ALTER, GRANT, TRUNCATE
-- The app should never need to modify schema

-- For read-only endpoints, use a read-only user
CREATE USER readonly_user WITH PASSWORD 'another_strong_password';
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_user;

Layer 4: Web Application Firewall (WAF)

A WAF can catch common injection patterns before they reach your application:

// Simple SQLi detection middleware (complement to parameterized queries)
function sqlInjectionDetector(req, res, next) {
  const suspiciousPatterns = [
    /('|\")(\s)*(OR|AND)(\s)*('|\"|\d)/i,
    /UNION(\s)+SELECT/i,
    /;\s*(DROP|DELETE|UPDATE|INSERT)/i,
    /\/\*.*\*\//,             // SQL comments
    /--\s/,                   // SQL line comments
    /xp_cmdshell/i,
  ];

  const inputsToCheck = [
    ...Object.values(req.query),
    ...Object.values(req.params),
  ];

  if (req.body && typeof req.body === 'object') {
    inputsToCheck.push(...Object.values(req.body).filter(v => typeof v === 'string'));
  }

  for (const input of inputsToCheck) {
    if (typeof input !== 'string') continue;
    for (const pattern of suspiciousPatterns) {
      if (pattern.test(input)) {
        console.warn('Potential SQLi detected:', { input, pattern: pattern.source, ip: req.ip });
        return res.status(400).json({ error: 'Invalid input detected' });
      }
    }
  }

  next();
}

Note: a WAF or pattern detection is a supplementary defense, not a replacement for parameterized queries. Attackers can bypass pattern matching with encoding tricks, case variations, and other evasion techniques.

Testing for SQL Injection

// Integration test to verify injection resistance
const request = require('supertest');

describe('SQL Injection Prevention', () => {
  const injectionPayloads = [
    "' OR '1'='1",
    "'; DROP TABLE users;--",
    "' UNION SELECT username, password FROM users--",
    "1; DELETE FROM products",
    "' AND 1=1--",
    "admin'--",
    "1 OR 1=1",
  ];

  test.each(injectionPayloads)(
    'should safely handle payload: %s',
    async (payload) => {
      const res = await request(app)
        .get('/api/products')
        .query({ search: payload });

      // Should return 200 with 0 results or 400 for invalid input
      // Should NEVER return all records or cause a 500
      expect([200, 400]).toContain(res.status);

      if (res.status === 200) {
        // Verify it treated the payload as a literal search string
        expect(res.body.length).toBe(0);
      }
    }
  );

  test('should not expose database errors', async () => {
    const res = await request(app)
      .get('/api/products')
      .query({ search: "' OR pg_sleep(5)--" });

    expect(res.status).not.toBe(500);
    expect(res.text).not.toContain('syntax error');
    expect(res.text).not.toContain('pg_catalog');
  });
});

Summary of Defenses

+-----+------------------------------+-------------------------------+
| #   | Defense                      | Protects Against              |
+-----+------------------------------+-------------------------------+
| 1   | Parameterized queries        | All SQL injection types       |
| 2   | ORM (Prisma, Knex)           | Most injection by default     |
| 3   | Input validation (Zod/Joi)   | Malformed data, type coercion |
| 4   | Least privilege DB user      | Limits damage if injected     |
| 5   | WAF / pattern detection      | Common automated attacks      |
| 6   | Whitelist for identifiers    | Table/column name injection   |
| 7   | Error message sanitization   | Information disclosure        |
+-----+------------------------------+-------------------------------+

Exercises

  1. Identify the Injection Points

    Review the following Express route and identify every SQL injection vulnerability. Then rewrite the entire handler using parameterized queries with node-postgres:

    app.get('/api/reports', async (req, res) => {
  const { startDate, endDate, department, sortBy, limit } = req.query;
  const query = `
    SELECT * FROM reports
    WHERE created_at BETWEEN '${startDate}' AND '${endDate}'
    AND department = '${department}'
    ORDER BY ${sortBy}
    LIMIT ${limit}
  `;
  const result = await pool.query(query);
  res.json(result.rows);
});
    Hint

    There are 5 injection points: startDate, endDate, department (use $1, $2, $3), sortBy (must be whitelisted since column names cannot be parameterized), and limit (parameterize as integer). Add Zod validation for all inputs.

  2. Build a Safe Dynamic Query Builder

    Write a function buildProductQuery(filters) that accepts an object with optional fields (search, category, minPrice, maxPrice, inStock, sortBy, order) and returns a { text, values } object for use with pool.query(). The function must: (1) use parameterized queries for all data values, (2) whitelist sortBy and order, (3) dynamically build the WHERE clause only for provided filters, and (4) handle the LIKE operator safely.

    Hint

    Track a parameter counter (let paramIndex = 1) and push to a values array as you build conditions. Increment paramIndex each time you add a parameter. For LIKE, put the % in the value, not the query: name ILIKE $${paramIndex} with value %${search}%.

  3. NoSQL Injection Prevention

    Create a login endpoint for a MongoDB-based Express app that is resistant to NoSQL injection. The endpoint should: (1) validate that username and password are strings (not objects), (2) use bcrypt for password comparison (not database-level comparison), (3) return consistent error messages to prevent user enumeration, and (4) include a test suite with injection payloads like {"$gt": ""} and {"$ne": null}.

    Hint

    Never pass user input directly to MongoDB's find() without type checking. Fetch the user by username (validated as string), then compare the password in application code with bcrypt.compare().

Common Problems & Solutions

1 ORM raw query methods bypass built-in parameterization
Cause: Developers use ORM escape hatches like Prisma's $queryRawUnsafe, Knex's knex.raw() with string interpolation, or Sequelize's sequelize.query() with template literals instead of parameter bindings. These raw methods do not auto-parameterize.
Solution: For Prisma, always use `$queryRaw` with tagged template literals (not `$queryRawUnsafe`). For Knex, always pass a bindings array as the second argument to `knex.raw('SELECT * FROM users WHERE id = ?', [id])`. For Sequelize, use replacements: `sequelize.query('SELECT * FROM users WHERE id = :id', { replacements: { id } })`. Audit your codebase with grep for `queryRawUnsafe`, `knex.raw` without bindings, and template literals inside `.query()` calls.
2 Parameterized queries do not work for ORDER BY, table names, or column names
Cause: SQL parameter placeholders ($1, ?) can only be used for data values, not SQL identifiers (table names, column names) or keywords (ASC/DESC). Databases interpret parameters as literal strings in those positions, causing syntax errors.
Solution: Maintain a strict whitelist of allowed column names and sort directions. Validate user input against the whitelist before inserting into the query string. Example: `const ALLOWED = ['name','price','created_at']; if (!ALLOWED.includes(sortBy)) throw new Error('Invalid sort');`. Never use user input directly for identifiers, even if you think you have escaped it.
3 LIKE queries with user input return unexpected results or are slow
Cause: User input containing SQL wildcard characters (% and _) causes unexpected pattern matching. Input like '%' matches everything, and complex patterns can cause slow sequential scans.
Solution: Escape wildcard characters in user input before using them in LIKE parameters: `const escaped = input.replace(/%/g, '\\%').replace(/_/g, '\\_');`. Then use: `WHERE name LIKE $1` with value `%${escaped}%`. For performance, consider PostgreSQL full-text search (`tsvector`/`tsquery`) or a dedicated search engine (Elasticsearch) instead of LIKE for user-facing search.
