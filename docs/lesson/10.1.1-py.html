Design a URL Shortener (TinyURL)

Intuition

A URL shortener is fundamentally a distributed hash map that converts long URLs into short keys and retrieves them with single-digit millisecond latency. The core challenge is generating globally unique short codes at scale without coordination bottlenecks.

Design a URL Shortener (TinyURL)

Requirements

  • Shorten long URLs to compact aliases (e.g., 7-character base62 codes)
  • Redirect short URLs to original destinations with minimal latency
  • Support custom aliases, expiration, and analytics
  • Handle 100M+ URLs with 10:1 read-to-write ratio

Architecture

A stateless API layer sits behind a load balancer, backed by a distributed key-value store for URL mappings. A separate ID generation service produces unique short codes. A cache layer (Redis) handles hot URLs to reduce database reads.

Key Components

  • ID Generator: Base62 encoding of auto-increment IDs or pre-generated random keys stored in a key range table
  • Database: NoSQL store (DynamoDB) mapping short code to long URL, creation time, and expiry
  • Cache: LRU cache for frequently accessed URLs
  • Analytics Service: Async event pipeline tracking clicks, referrers, and geolocation

Scaling

Horizontally scale the API tier. Partition the database by hash of short code. Use consistent hashing for cache distribution. Pre-generate key ranges to avoid collisions without coordination.

Trade-offs

  • Sequential IDs are predictable but simple; random codes need collision checks but are secure
  • 301 (permanent) redirects reduce server load but lose analytics; 302 (temporary) keeps tracking intact

Exercises

  1. Implement Base62 Encoding

    Write a service that takes a numeric ID and converts it to a base62 string (a-z, A-Z, 0-9). Then implement the reverse decoding. Ensure the output is always exactly 7 characters by zero-padding.

    Hint

    Use modulo and division operations iteratively, mapping remainders to characters from your 62-character alphabet.

  2. Design a Key Range Pre-generation Service

    Build a service that pre-generates batches of unique 7-character keys and stores them in a database. Worker nodes request a batch of unused keys, mark them as allocated, and use them for new URL mappings without coordination.

    Hint

    Use a two-table approach: one for unused keys and one for used keys. Each worker takes a range atomically using a compare-and-swap operation.

Common Problems & Solutions

1 Multiple servers generate the same short code simultaneously, causing URL overwrites.
Cause: Lack of coordination in distributed ID generation leads to collisions when using random generation or shared auto-increment counters.
Solution: Use a key pre-generation service (KGS) that allocates unique key ranges to each server. Alternatively, use a Zookeeper-based counter with range reservations so each node gets a dedicated block of IDs.
2 Database reads become a bottleneck as popular URLs get millions of redirects per second.
Cause: Hot keys in the datastore overwhelm a single partition, causing high latency and timeouts.
Solution: Place a multi-tier caching layer (local in-memory + distributed Redis) in front of the database. Use consistent hashing to spread cache load and implement read replicas for the database.
