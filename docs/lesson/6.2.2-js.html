Docker Compose: Multi-Container Applications

Intuition

Docker Compose is like a conductor's sheet music for an orchestra. Each service (instrument) has its own part, but the compose file coordinates them all -- ensuring the database (rhythm section) starts before the API (melody), that they can all hear each other (networking), and that the music is preserved (volumes) even if the orchestra takes a break.

Docker Compose: Multi-Container Applications

Why Docker Compose?

Real applications rarely run alone. A typical Node.js backend needs PostgreSQL, Redis, maybe Elasticsearch and a message broker. Docker Compose lets you define and run all these services together with a single YAML file and one command.

Complete docker-compose.yml

# docker-compose.yml
version: '3.8'

services:
  # ================================
  # Node.js API
  # ================================
  api:
    build:
      context: .
      dockerfile: Dockerfile
      target: production        # Multi-stage build target
    container_name: myapp-api
    ports:
      - "3000:3000"
    environment:
      NODE_ENV: production
      PORT: 3000
      DATABASE_URL: postgresql://myapp:secretpassword@postgres:5432/myapp_db
      REDIS_URL: redis://redis:6379
    env_file:
      - .env                    # Additional env vars from file
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    restart: unless-stopped
    networks:
      - app-network
    volumes:
      - uploads:/app/uploads    # Persistent uploads
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "--spider", "http://localhost:3000/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s

  # ================================
  # PostgreSQL Database
  # ================================
  postgres:
    image: postgres:16-alpine
    container_name: myapp-postgres
    environment:
      POSTGRES_DB: myapp_db
      POSTGRES_USER: myapp
      POSTGRES_PASSWORD: secretpassword
    ports:
      - "5432:5432"             # Expose for local development tools
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./init-scripts:/docker-entrypoint-initdb.d  # Auto-run SQL on first start
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myapp -d myapp_db"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped
    networks:
      - app-network

  # ================================
  # Redis Cache
  # ================================
  redis:
    image: redis:7-alpine
    container_name: myapp-redis
    command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped
    networks:
      - app-network

  # ================================
  # Nginx Reverse Proxy
  # ================================
  nginx:
    image: nginx:1.25-alpine
    container_name: myapp-nginx
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/conf.d:/etc/nginx/conf.d:ro
      - ./certbot/conf:/etc/letsencrypt:ro
      - ./certbot/www:/var/www/certbot:ro
    depends_on:
      - api
    restart: unless-stopped
    networks:
      - app-network

# ================================
# Volumes (persistent data)
# ================================
volumes:
  pgdata:
    driver: local
  redis-data:
    driver: local
  uploads:
    driver: local

# ================================
# Networks
# ================================
networks:
  app-network:
    driver: bridge

Development vs Production Compose Files

Base file (docker-compose.yml)

version: '3.8'

services:
  api:
    environment:
      DATABASE_URL: postgresql://myapp:password@postgres:5432/myapp_db
      REDIS_URL: redis://redis:6379
    depends_on:
      - postgres
      - redis
    networks:
      - app-network

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: myapp_db
      POSTGRES_USER: myapp
      POSTGRES_PASSWORD: password
    networks:
      - app-network

  redis:
    image: redis:7-alpine
    networks:
      - app-network

networks:
  app-network:

Development overrides (docker-compose.dev.yml)

version: '3.8'

services:
  api:
    build:
      context: .
      dockerfile: Dockerfile.dev
    volumes:
      - .:/app                  # Mount source for hot reload
      - /app/node_modules       # Exclude node_modules from mount
    environment:
      NODE_ENV: development
    ports:
      - "3000:3000"
      - "9229:9229"            # Node.js debugger port
    command: npx nodemon --inspect=0.0.0.0:9229

  postgres:
    ports:
      - "5432:5432"            # Expose for local DB tools
    volumes:
      - pgdata-dev:/var/lib/postgresql/data

  redis:
    ports:
      - "6379:6379"            # Expose for local Redis tools

  # Additional dev services
  adminer:
    image: adminer
    ports:
      - "8080:8080"            # Database web UI
    networks:
      - app-network

  mailhog:
    image: mailhog/mailhog
    ports:
      - "1025:1025"            # SMTP
      - "8025:8025"            # Web UI
    networks:
      - app-network

volumes:
  pgdata-dev:

Running with overrides

# Development (merges both files)
docker compose -f docker-compose.yml -f docker-compose.dev.yml up

# Or use the COMPOSE_FILE env var
export COMPOSE_FILE=docker-compose.yml:docker-compose.dev.yml
docker compose up

# Production
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

Docker Compose Commands

# Start all services
docker compose up                   # Foreground (see all logs)
docker compose up -d                # Detached (background)

# Build and start
docker compose up --build           # Rebuild images before starting

# Stop services
docker compose down                 # Stop and remove containers
docker compose down -v              # Also remove volumes (deletes data!)

# View logs
docker compose logs                 # All services
docker compose logs api             # Specific service
docker compose logs -f api          # Follow logs

# Service management
docker compose ps                   # List running services
docker compose exec api sh          # Shell into running service
docker compose run api npm test     # Run one-off command
docker compose restart api          # Restart a service

# Scaling
docker compose up -d --scale api=3  # Run 3 API instances

# Cleanup
docker compose down --rmi all       # Remove images too
docker system prune -a              # Nuclear option: clean everything

Environment Variable Management

services:
  api:
    # Method 1: Inline
    environment:
      NODE_ENV: production
      PORT: 3000

    # Method 2: From .env file
    env_file:
      - .env
      - .env.local             # Override with local values

    # Method 3: From host environment
    environment:
      API_KEY: ${API_KEY}      # Must be set on host
      DEBUG: ${DEBUG:-false}   # Default value if not set

Health Checks and Dependencies

services:
  api:
    depends_on:
      postgres:
        condition: service_healthy    # Wait for healthy status
      redis:
        condition: service_started    # Just wait for start

  postgres:
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myapp"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

Named Volumes vs Bind Mounts

services:
  api:
    volumes:
      # Bind mount: host path -> container path
      - ./src:/app/src              # Development: live code reload
      - ./nginx.conf:/etc/nginx/nginx.conf:ro  # :ro = read-only

      # Named volume: managed by Docker
      - uploads:/app/uploads        # Persistent data
      - /app/node_modules           # Anonymous volume (exclude from bind)

volumes:
  uploads:                          # Declare named volumes

Practical Patterns

Database Initialization

-- init-scripts/01-create-extensions.sql
-- Files in /docker-entrypoint-initdb.d/ run on first start
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

Wait-for-it Pattern

services:
  api:
    command: >
      sh -c "
        echo 'Waiting for database...' &&
        npx prisma migrate deploy &&
        echo 'Starting server...' &&
        node dist/server.js
      "
    depends_on:
      postgres:
        condition: service_healthy

Running Migrations

# Run migrations as a one-off command
docker compose exec api npx prisma migrate deploy

# Or as a separate service
docker compose run --rm api npx prisma migrate deploy

Common Pitfalls

  1. Using depends_on without health checks - depends_on alone only waits for container start, not for the service to be ready. Always add health checks.
  2. Storing secrets in docker-compose.yml - Use .env files (gitignored) or Docker secrets for sensitive data.
  3. Bind-mounting node_modules - On macOS/Windows, this destroys performance. Use a named volume for node_modules.
  4. Forgetting docker compose down -v deletes data - Named volumes persist across down but not down -v.
  5. Not using restart: unless-stopped - Without restart policies, containers do not restart after host reboot.

Exercises

  1. Full-Stack Docker Compose

    Create a docker-compose.yml for a full-stack application with: Node.js API, PostgreSQL, Redis, and Nginx as reverse proxy. Include health checks for all services, proper dependency ordering, persistent volumes for database and uploads, and a shared network.

    Hint

    Use depends_on with condition: service_healthy. Map Nginx port 80 to the API. Use pg_isready for Postgres health check and redis-cli ping for Redis.

  2. Dev vs Prod Setup

    Create a base docker-compose.yml and two override files: docker-compose.dev.yml (with hot reload, debug ports, adminer for DB management) and docker-compose.prod.yml (with resource limits, no exposed DB ports, production env vars). Write a Makefile with 'make dev' and 'make prod' commands.

    Hint

    Use bind mounts in dev for hot reload. In prod, use 'deploy.resources.limits' for resource constraints. The Makefile should compose the correct override file.

  3. Seed and Migrate

    Add a migration service to your compose file that runs Prisma migrations and seeds the database on startup, then exits. The API service should wait for the migration service to complete successfully before starting.

    Hint

    Use a separate service with 'profiles: [migrate]' or depends_on with condition: service_completed_successfully (Compose v2).

Common Problems & Solutions

1 API container starts before database is ready, connection refused
Symptoms: Error: connect ECONNREFUSED 172.18.0.2:5432
Cause: depends_on without health check conditions only waits for container creation, not for the service inside to be ready.
Solution: Add healthcheck to the database service and use `depends_on: postgres: condition: service_healthy`. For PostgreSQL, use `test: ["CMD-SHELL", "pg_isready -U username"]`.
2 File changes not reflected in container during development
Cause: The source code is baked into the image at build time. Without bind mounts, changes require a rebuild.
Solution: Add a bind mount in your dev compose file: `volumes: ['.:/app', '/app/node_modules']`. The anonymous volume for node_modules prevents the host mount from overwriting container dependencies. Use nodemon or ts-node-dev for auto-restart.
3 Docker Compose extremely slow on macOS
Cause: File system sharing between macOS and Linux containers is slow, especially with many files (node_modules).
Solution: Use a named volume for node_modules instead of bind-mounting it. Enable VirtioFS in Docker Desktop settings (much faster than gRPC-FUSE). Consider using Docker's `watch` mode or mounting only `src/` instead of the entire project.
