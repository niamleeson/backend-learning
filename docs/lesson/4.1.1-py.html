App Factory Pattern and Configuration

Intuition

The app factory pattern decouples your application's creation from its configuration, making it easy to spin up different versions for testing, staging, or production. Think of it like a blueprint for building houses -- same plan, different finishes.

App Factory Pattern Configuration

The app factory pattern is the recommended way to structure Flask applications. Instead of creating a global app object, you define a function that creates and configures the application instance. This allows multiple configurations (testing, development, production) and avoids circular imports.

import os
from flask import Flask

def create_app(config_name=None):
    app = Flask(__name__, instance_relative_config=True)

    if config_name is None:
        config_name = os.environ.get('FLASK_CONFIG', 'development')

    app.config.from_object(f'config.{config_name.capitalize()}Config')

    # Ensure the instance folder exists
    try:
        os.makedirs(app.instance_path)
    except OSError:
        pass

    # Register blueprints
    from .main import main_bp
    app.register_blueprint(main_bp)

    # Initialize extensions
    from .extensions import db, migrate
    db.init_app(app)
    migrate.init_app(app, db)

    return app

Key Takeaways: - The factory function create_app() returns a fully configured Flask instance. - Extensions are initialized with init_app() so they are not bound to a single app. - instance_relative_config=True lets you load sensitive config from an instance folder outside version control.

Exercises

  1. Build a Basic App Factory

    Create a create_app() function that accepts a config dictionary, sets app.config values from it, and registers a simple blueprint with one route returning JSON.

    Hint

    Use app.config.from_mapping(config) to apply the dictionary and create the blueprint in a separate module.

  2. Multiple Configurations

    Define DevelopmentConfig, TestingConfig, and ProductionConfig classes. Modify your factory to select the right config based on an environment variable FLASK_ENV.

    Hint

    Each config class can inherit from a BaseConfig and override DEBUG, TESTING, and SECRET_KEY attributes.

Common Problems & Solutions

1 RuntimeError: Working outside of application context.
Cause: You tried to access current_app or g outside of a request or app context, often when using the factory pattern without pushing a context.
Solution: Wrap the code in `with app.app_context():` or ensure you are inside a request handler. In tests, use `app.test_client()` which manages the context.
2 Circular imports when registering blueprints.
Cause: Modules import the app object at module level, but the factory hasn't created it yet.
Solution: Import blueprints inside the factory function rather than at the top of the module. Use late binding for extensions with init_app().
