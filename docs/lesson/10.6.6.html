Design a Multi-Tenant SaaS Platform

Intuition

Multi-tenancy is the art of sharing infrastructure across customers while making each customer feel like they have their own dedicated system. The core architectural decision is the data isolation model, which exists on a spectrum from fully shared (cheap, risky) to fully isolated (expensive, safe), and most real systems offer tiered options.

Design a Multi-Tenant SaaS Platform

Requirements

  • Serve thousands of tenants (organizations) from a shared infrastructure
  • Isolate tenant data, enforce per-tenant resource quotas, and prevent noisy-neighbor effects
  • Support tenant-specific customization (branding, feature flags, integrations)
  • Provide tenant onboarding, billing metering, and admin dashboards

Architecture

A shared application tier routes requests by tenant ID (extracted from subdomain or JWT). A data tier supports multiple isolation models: shared database with tenant_id column, schema-per-tenant, or database-per-tenant for premium tiers. A control plane manages tenant configuration, feature flags, and quota enforcement.

Key Components

  • Tenant Router: Extracts tenant context from request, applies rate limits, routes to appropriate resources
  • Data Isolation Layer: Row-level security (RLS) in shared DB or separate schemas/databases per tenant
  • Quota/Rate Limiter: Per-tenant rate limiting and resource quotas (API calls, storage, compute)
  • Billing Meter: Tracks usage events per tenant for usage-based billing; emits to billing system

Scaling

Use shared infrastructure for small tenants (cost-efficient) and dedicated resources for large/enterprise tenants (performance isolation). Shard the shared database by tenant_id. Cache tenant configs aggressively.

Trade-offs

  • Shared DB is cost-efficient but risks data leaks via bugs; separate DBs are safer but expensive
  • Uniform infrastructure simplifies ops but cannot meet diverse tenant SLAs; tiered offerings add complexity
  • Tenant customization increases flexibility but multiplies testing surface area

Exercises

  1. Implement Row-Level Tenant Isolation

    Create a shared PostgreSQL database with a tenant_id column on all tables. Implement row-level security policies so that queries from tenant A can never see tenant B's data, even with SQL injection.

    Hint

    Use PostgreSQL RLS: CREATE POLICY tenant_isolation ON table USING (tenant_id = current_setting('app.current_tenant')). Set the session variable on each request before executing queries.

  2. Build a Per-Tenant Rate Limiter

    Implement a token bucket rate limiter that enforces different rate limits per tenant (e.g., free: 100 req/min, pro: 1000 req/min). Test with concurrent requests from multiple tenants.

    Hint

    Store a token bucket per tenant (tokens, last_refill_time) in Redis. On each request, refill based on elapsed time, then decrement. Reject if tokens <= 0. Load the rate limit from tenant config.

Common Problems & Solutions

1 A large tenant's heavy queries slow down the shared database for all other tenants
Cause: All tenants share the same database connection pool and the large tenant's queries consume most connections and I/O
Solution: Implement per-tenant connection pool limits, add query timeouts, and migrate large tenants to dedicated database instances with a seamless routing change.
2 A developer accidentally deploys a migration that exposes tenant A's data to tenant B
Cause: A new query was added without a WHERE tenant_id clause, bypassing the application-level isolation
Solution: Enforce row-level security at the database level (not just application level) as a safety net, and add automated tests that verify cross-tenant data isolation on every migration.
