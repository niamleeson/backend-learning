Type Annotations: Basics and Benefits

Intuition

Type annotations are like labels on containers telling you what goes inside. They do not change how Python runs your code, but they help tools and humans catch mistakes early.

Type Annotations Basics and Benefits

Type annotations let you declare expected types for variables, function parameters, and return values. They make code self-documenting and enable static analysis tools to catch bugs before runtime.

def greet(name: str, times: int = 1) -> str:
    return (f"Hello, {name}! " * times).strip()

age: int = 30
scores: list[float] = [95.5, 87.3, 91.0]
user_map: dict[str, int] = {"alice": 1, "bob": 2}

def process_items(items: list[str]) -> dict[str, int]:
    return {item: len(item) for item in items}

# Type aliases for complex types
Matrix = list[list[float]]
def transpose(m: Matrix) -> Matrix:
    return [list(row) for row in zip(*m)]

Key Takeaways: - Annotations are hints, not enforced at runtime by default - Use built-in generics (list[int], dict[str, int]) in Python 3.9+ - Type annotations improve IDE autocomplete, refactoring, and documentation - They enable static checkers like mypy to find type errors before execution - Annotate public APIs first, then gradually add to internal code

Exercises

  1. Annotate a Data Pipeline

    Write a function parse_records(raw: list[str]) -> list[dict[str, str]] that splits each CSV-like string into a dictionary using the first row as headers. Annotate every variable.

    Hint

    Use str.split(',') and zip the header list with each row's values to build dictionaries.

  2. Create Type Aliases

    Define type aliases for UserID = int, UserProfile = dict[str, str | int], and UserDB = dict[UserID, UserProfile]. Write a function get_user(db: UserDB, uid: UserID) -> UserProfile | None.

    Hint

    Use the pipe | syntax for unions (Python 3.10+) or Union from typing for older versions.

Common Problems & Solutions

1 Type annotations cause `NameError` at runtime when referencing a class not yet defined.
Cause: Forward references require the class name as a string or `from __future__ import annotations`.
Solution: Add `from __future__ import annotations` at the top of the file to defer all annotation evaluation.
2 Using `list[int]` syntax causes `TypeError` in Python 3.8 or earlier.
Cause: Built-in generic syntax (`list[int]`, `dict[str, int]`) is only available from Python 3.9+.
Solution: Use `from typing import List, Dict` and write `List[int]`, `Dict[str, int]` for older Python versions.
