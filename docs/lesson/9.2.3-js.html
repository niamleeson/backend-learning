Design a Key-Value Store (Redis/DynamoDB)

Intuition

Think of a key-value store as a network of filing cabinets (nodes), each responsible for a range of alphabetical folders (partitions). When you store a document, a receptionist (coordinator) figures out which cabinets it belongs in (consistent hashing), makes three copies (replication factor), and files them in three different cabinets. When you retrieve it, the receptionist asks two cabinets and takes the most recent copy (quorum read). If a cabinet is locked (node down), someone else holds the document temporarily and files it later (hinted handoff).

Design a Key-Value Store (Redis/DynamoDB)

Introduction

A key-value store is the simplest yet most fundamental distributed data system. It stores data as key-value pairs and supports get(key) and put(key, value) operations. Systems like Redis, DynamoDB, Memcached, and etcd power caching layers, session stores, configuration management, and more. Designing one from scratch tests your understanding of data partitioning, replication, consistency, failure detection, and storage engines.

Step 1: Requirements Gathering

Functional Requirements

  • put(key, value): Store a key-value pair
  • get(key): Retrieve the value for a given key
  • delete(key): Remove a key-value pair
  • TTL support: Keys can expire after a specified duration
  • Automatic data partitioning: Data distributed across multiple nodes
  • Tunable consistency: Choose between strong and eventual consistency per request

Non-Functional Requirements

  • Latency: < 10ms for get and put at p99
  • Throughput: 1M+ operations per second
  • Availability: 99.99% (partition-tolerant, favoring availability)
  • Scalability: Linear horizontal scaling by adding nodes
  • Durability: Configurable (in-memory for speed, WAL for durability)

Step 2: Capacity Estimation

Data size:          100 TB total

Total pairs:        100 TB / 1 KB = ~100B key-value pairs
Operations:         1M ops/sec (70% reads, 30% writes)

Per-node:
  Storage:          ~1 TB
  Ops:              ~10K ops/sec
  Network:          ~10 MB/s per node

Step 4: API Design

import express from 'express';
const router = express.Router();

// Put a key-value pair
router.put('/api/v1/kv/:key', async (req, res) => {
  const { key } = req.params;
  const { value, ttl, consistency } = req.body;
  // consistency: 'one' | 'quorum' | 'all'

  try {
    const partition = consistentHash.getPartition(key);
    const replicas = partitionMap.getReplicas(partition);
    const writeConsistency = consistency || 'quorum';
    const requiredAcks = getRequiredAcks(writeConsistency, replicas.length);

    const timestamp = Date.now();
    const entry = { key, value, timestamp, ttl: ttl || null };

    // Write to replicas in parallel
    const results = await Promise.allSettled(
      replicas.map(node => node.write(entry))
    );

    const acks = results.filter(r => r.status === 'fulfilled').length;
    if (acks >= requiredAcks) {
      res.json({ success: true, acks, timestamp });
    } else {
      res.status(503).json({ error: 'Failed to achieve write quorum' });
    }
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

// Get a value by key
router.get('/api/v1/kv/:key', async (req, res) => {
  const { key } = req.params;
  const consistency = req.query.consistency || 'quorum';

  const partition = consistentHash.getPartition(key);
  const replicas = partitionMap.getReplicas(partition);
  const requiredReads = getRequiredAcks(consistency, replicas.length);

  const results = await Promise.allSettled(
    replicas.map(node => node.read(key))
  );

  const responses = results
    .filter(r => r.status === 'fulfilled')
    .map(r => r.value)
    .filter(v => v !== null);

  if (responses.length < requiredReads) {
    return res.status(503).json({ error: 'Failed to achieve read quorum' });
  }

  // Return the value with the highest timestamp (last-write-wins)
  const latest = responses.reduce((a, b) => a.timestamp > b.timestamp ? a : b);

  // Check TTL
  if (latest.ttl && Date.now() > latest.timestamp + latest.ttl * 1000) {
    return res.status(404).json({ error: 'Key expired' });
  }

  res.json({ key, value: latest.value, timestamp: latest.timestamp });
});

// Delete a key
router.delete('/api/v1/kv/:key', async (req, res) => {
  const { key } = req.params;

  // Use a tombstone (soft delete) to propagate deletion
  const partition = consistentHash.getPartition(key);
  const replicas = partitionMap.getReplicas(partition);
  const tombstone = { key, value: null, timestamp: Date.now(), deleted: true };

  const results = await Promise.allSettled(
    replicas.map(node => node.write(tombstone))
  );

  const acks = results.filter(r => r.status === 'fulfilled').length;
  res.json({ deleted: acks >= Math.floor(replicas.length / 2) + 1 });
});

function getRequiredAcks(consistency, replicaCount) {
  switch (consistency) {
    case 'one': return 1;
    case 'all': return replicaCount;
    case 'quorum': default: return Math.floor(replicaCount / 2) + 1;
  }
}

Step 5: Database Schema / Storage Engine

A key-value store IS the database, so instead of a schema we design the storage engine.

LSM Tree (Log-Structured Merge Tree)

Used by LevelDB, RocksDB, Cassandra, and DynamoDB.

class LSMStorage {
  constructor(dataDir) {
    this.dataDir = dataDir;
    this.memTable = new Map();       // In-memory sorted map
    this.memTableSize = 0;
    this.maxMemTableSize = 4 * 1024 * 1024; // 4MB
    this.wal = new WriteAheadLog(`${dataDir}/wal`);
    this.sstables = [];              // Sorted on-disk files
    this.bloomFilters = [];          // One per SSTable
  }

  async put(key, value, timestamp) {
    // 1. Write to WAL first (durability)
    await this.wal.append({ key, value, timestamp });

    // 2. Write to MemTable
    this.memTable.set(key, { value, timestamp });
    this.memTableSize += key.length + (value?.length || 0);

    // 3. Flush to SSTable if MemTable is full
    if (this.memTableSize >= this.maxMemTableSize) {
      await this.flushMemTable();
    }
  }

  async get(key) {
    // 1. Check MemTable first (most recent writes)
    if (this.memTable.has(key)) {
      return this.memTable.get(key);
    }

    // 2. Check SSTables from newest to oldest
    for (let i = this.sstables.length - 1; i >= 0; i--) {
      // Check bloom filter first to skip SSTables that definitely
      // don't contain the key
      if (!this.bloomFilters[i].mightContain(key)) {
        continue;
      }

      const result = await this.sstables[i].get(key);
      if (result !== null) return result;
    }

    return null;
  }

  async flushMemTable() {
    // Sort entries and write to disk as SSTable
    const entries = [...this.memTable.entries()]
      .sort(([a], [b]) => a.localeCompare(b));

    const sstable = new SSTable(`${this.dataDir}/sst_${Date.now()}`);
    const bloom = new BloomFilter(entries.length);

    for (const [key, val] of entries) {
      await sstable.write(key, val);
      bloom.add(key);
    }

    this.sstables.push(sstable);
    this.bloomFilters.push(bloom);

    // Clear MemTable and WAL
    this.memTable.clear();
    this.memTableSize = 0;
    await this.wal.truncate();
  }
}

Consistent Hashing for Partitioning

import crypto from 'node:crypto';

class ConsistentHashRing {
  constructor(virtualNodes = 150) {
    this.virtualNodes = virtualNodes;
    this.ring = new Map();          // hash -> node
    this.sortedHashes = [];         // sorted hash positions
  }

  addNode(nodeId) {
    for (let i = 0; i < this.virtualNodes; i++) {
      const hash = this._hash(`${nodeId}:vn${i}`);
      this.ring.set(hash, nodeId);
      this.sortedHashes.push(hash);
    }
    this.sortedHashes.sort((a, b) => a - b);
  }

  removeNode(nodeId) {
    for (let i = 0; i < this.virtualNodes; i++) {
      const hash = this._hash(`${nodeId}:vn${i}`);
      this.ring.delete(hash);
    }
    this.sortedHashes = this.sortedHashes.filter(h => this.ring.has(h));
  }

  getNode(key) {
    const hash = this._hash(key);
    for (const nodeHash of this.sortedHashes) {
      if (hash <= nodeHash) return this.ring.get(nodeHash);
    }
    return this.ring.get(this.sortedHashes[0]);
  }

  // Get N replica nodes (walk clockwise on the ring)
  getReplicas(key, count = 3) {
    const hash = this._hash(key);
    const replicas = new Set();
    let startIdx = this.sortedHashes.findIndex(h => h >= hash);
    if (startIdx === -1) startIdx = 0;

    for (let i = 0; replicas.size < count && i < this.sortedHashes.length; i++) {
      const idx = (startIdx + i) % this.sortedHashes.length;
      replicas.add(this.ring.get(this.sortedHashes[idx]));
    }
    return [...replicas];
  }

  _hash(key) {
    return crypto.createHash('md5').update(key).digest().readUInt32BE(0);
  }
}

Step 6: Deep Dive -- Replication and Consistency

Quorum Reads and Writes

With N replicas, W write acknowledgments needed, and R read replicas queried: - Strong consistency: W + R > N (e.g., N=3, W=2, R=2) - Eventual consistency: W=1, R=1 (fast but may read stale data) - Write-heavy: W=1, R=3 (fast writes, reads check all replicas)

N=3 replicas, W=2, R=2 (quorum):

Write "user:123" = "Alice":
  Node A: ACK (1)    ✓
  Node B: ACK (2)    ✓  → quorum reached, return success
  Node C: timeout         → will catch up via anti-entropy

Read "user:123":
  Node A: "Alice" (ts: 100)  ✓
  Node B: "Alice" (ts: 100)  ✓  → quorum reached
  Since W + R (2+2=4) > N (3), guaranteed to read latest write

Conflict Resolution

When replicas diverge, resolve conflicts using: - Last-Write-Wins (LWW): Use timestamps; highest timestamp wins. Simple but can lose data. - Vector Clocks: Track causal history; detect true conflicts. Complex but accurate.

class VectorClock {
  constructor(clocks = {}) {
    this.clocks = { ...clocks };
  }

  increment(nodeId) {
    this.clocks[nodeId] = (this.clocks[nodeId] || 0) + 1;
    return this;
  }

  merge(other) {
    const merged = new VectorClock(this.clocks);
    for (const [node, ts] of Object.entries(other.clocks)) {
      merged.clocks[node] = Math.max(merged.clocks[node] || 0, ts);
    }
    return merged;
  }

  isConcurrentWith(other) {
    let thisGreater = false, otherGreater = false;
    const allNodes = new Set([
      ...Object.keys(this.clocks),
      ...Object.keys(other.clocks),
    ]);
    for (const node of allNodes) {
      const a = this.clocks[node] || 0;
      const b = other.clocks[node] || 0;
      if (a > b) thisGreater = true;
      if (b > a) otherGreater = true;
    }
    return thisGreater && otherGreater;
  }
}

Step 7: Scaling Strategies

  1. Consistent hashing: Adding/removing nodes only remaps K/N keys (K total keys, N nodes)
  2. Virtual nodes: Each physical node owns many positions on the ring for even distribution
  3. Hinted handoff: When a replica is down, another node temporarily accepts writes and forwards them when the replica recovers
  4. Anti-entropy (Merkle trees): Replicas periodically compare Merkle trees of their data to find and repair inconsistencies
  5. Compaction: Periodically merge SSTables to reduce read amplification
  6. Bloom filters: Avoid unnecessary disk reads for keys that do not exist in an SSTable

Step 8: Trade-offs

Decision Option A Option B Recommendation
Storage engine LSM Tree B+ Tree LSM for write-heavy, B+ for read-heavy
Consistency Strong (quorum) Eventual (single) Tunable per-request
Conflict resolution LWW Vector clocks LWW for simplicity, vector clocks when data loss is unacceptable
Partitioning Consistent hashing Range-based Consistent hashing (better load distribution)
In-memory only Yes (Redis-like) Disk-backed (DynamoDB-like) Depends on data size and durability requirements
Failure detection Heartbeat Gossip protocol Gossip for large clusters (O(log N) propagation)

Exercises

  1. Build an In-Memory Key-Value Store

    Implement a single-node key-value store in Node.js with get, put, delete, and TTL support. Use a Map for storage and a separate data structure for TTL expiration. Expose it via an Express REST API. Benchmark it with autocannon to measure throughput.

    Hint

    For TTL, use a min-heap (priority queue) sorted by expiration time. Run a setInterval that pops expired keys. Alternatively, check TTL lazily on read.

  2. Implement Consistent Hashing with Virtual Nodes

    Build a consistent hash ring in Node.js. Add 5 physical nodes with 100 virtual nodes each. Hash 10,000 random keys and verify distribution is within 20% of perfect balance. Then remove one node and verify that only ~20% of keys are remapped.

    Hint

    Use MD5 for hashing. Count keys per physical node before and after removal. Calculate standard deviation to measure balance.

  3. Simulate Quorum Reads and Writes

    Create a simulation of 3 replica nodes with configurable network delay and failure probability. Implement quorum writes (W=2) and reads (R=2). Inject a node failure and verify that the system still operates correctly. Test that W+R > N guarantees reading the latest write.

    Hint

    Simulate nodes as objects with a write(key, value) and read(key) method that randomly delay or throw. Use Promise.allSettled to collect results.

  4. Build a Write-Ahead Log

    Implement a simple WAL that appends key-value operations to a file. On startup, replay the WAL to reconstruct the in-memory state. Benchmark the performance difference between fsync on every write vs. batched fsync every 100ms.

    Hint

    Use fs.appendFile or fs.createWriteStream in append mode. For replay, read line by line with readline. Use fs.fdatasync for fsync.

Common Problems & Solutions

1 Hot keys causing uneven load across nodes
Cause: Certain keys (e.g., a viral post's like counter) receive millions of operations per second, overloading the nodes responsible for that key's partition.
Solution: Implement key splitting: append a random suffix (e.g., `key:0` through `key:9`) to spread the hot key across multiple nodes. On read, query all shards and aggregate. Alternatively, use a local cache on coordinator nodes for read-heavy hot keys.
2 Data inconsistency after network partition heals
Cause: During a network partition, both sides accepted writes for the same keys. When the partition heals, replicas have conflicting values.
Solution: Use vector clocks to detect true conflicts. For last-write-wins, ensure all nodes use synchronized clocks (NTP with bounded drift). Run anti-entropy repair using Merkle tree comparison to identify and resolve divergent keys.
3 Read latency increasing as data grows
Cause: Too many SSTables on disk. Each read must check multiple SSTables (read amplification). Bloom filters help but do not eliminate all unnecessary reads.
Solution: Implement tiered or leveled compaction to merge SSTables. Leveled compaction (LevelDB-style) bounds read amplification to O(log N). Monitor compaction throughput and ensure it keeps pace with write throughput.
