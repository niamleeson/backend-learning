Redis Data Structures: Strings, Lists, Hashes, Sets, Sorted Sets

Intuition

Redis data structures are like specialized containers in a workshop. Strings are labeled jars (one thing per jar). Lists are tubes where you push items in one end and pop from the other. Sets are bags of unique marbles. Sorted sets are trophy shelves where each trophy has a score determining its position. Hashes are filing cabinets with labeled drawers. Each container is optimized for specific operations, and choosing the right one makes your work dramatically faster.

Redis Data Structures: Strings, Lists, Hashes, Sets, Sorted Sets

Introduction

Redis is an in-memory data structure store used as a database, cache, and message broker. Unlike PostgreSQL which stores data on disk in tables, Redis keeps everything in RAM, making it extraordinarily fast — typical operations complete in microseconds. Redis is not a replacement for your primary database; it complements it for caching, sessions, real-time features, and high-throughput use cases.

Setting Up ioredis

npm install ioredis

import Redis from 'ioredis';

const redis = new Redis({
  host: '127.0.0.1',
  port: 6379,
  password: process.env.REDIS_PASSWORD,
  db: 0,
  maxRetriesPerRequest: 3,
  retryStrategy(times) {
    return Math.min(times * 50, 2000);
  },
});

redis.on('connect', () => console.log('Redis connected'));
redis.on('error', (err) => console.error('Redis error:', err));

Strings

The simplest type — a key maps to a value (up to 512MB).

// SET and GET
await redis.set('user:1:name', 'Alice');
const name = await redis.get('user:1:name'); // 'Alice'

// SET with expiration (TTL)
await redis.set('session:abc123', JSON.stringify({ userId: 1 }), 'EX', 3600); // 1 hour
// or
await redis.setex('session:abc123', 3600, JSON.stringify({ userId: 1 }));

// SET only if not exists (for locks/deduplication)
const acquired = await redis.set('lock:order:42', 'worker-1', 'EX', 30, 'NX');
// Returns 'OK' if set, null if key already exists

// INCR / DECR (atomic counters)
await redis.set('page:views', '0');
await redis.incr('page:views');      // 1
await redis.incrby('page:views', 10); // 11
await redis.decr('page:views');       // 10

// MGET / MSET (batch operations)
await redis.mset('key1', 'val1', 'key2', 'val2', 'key3', 'val3');
const values = await redis.mget('key1', 'key2', 'key3');
// ['val1', 'val2', 'val3']

Common String patterns:
  Session storage:    session:<token> -> JSON user data
  Rate limiting:      rate:<ip>:<endpoint> -> count
  Caching:            cache:<query_hash> -> JSON result
  Distributed locks:  lock:<resource> -> owner_id
  Counters:           stats:page_views:<page> -> count

Lists

Ordered collections. Great for queues, recent items, and activity feeds.

// LPUSH / RPUSH (add to left/right)
await redis.lpush('notifications:1', JSON.stringify({ type: 'like', postId: 42 }));
await redis.lpush('notifications:1', JSON.stringify({ type: 'comment', postId: 42 }));

// LRANGE (get range, 0-based)
const recent = await redis.lrange('notifications:1', 0, 9); // first 10
const all = await redis.lrange('notifications:1', 0, -1);    // all items

// LPOP / RPOP (remove and return)
const item = await redis.rpop('job_queue');  // remove from right (FIFO queue)

// LLEN (length)
const count = await redis.llen('notifications:1');

// LTRIM (keep only a range, great for capping lists)
await redis.lpush('recent:products', productId);
await redis.ltrim('recent:products', 0, 99); // keep only last 100

// Blocking pop (for worker queues)
const [queue, job] = await redis.brpop('jobs:email', 'jobs:sms', 30);
// Blocks for up to 30 seconds waiting for a job

List as a Queue:
  Producer                Queue                   Consumer
  LPUSH -->  [new, ..., old]  --> BRPOP

List as a Stack:
  LPUSH -->  [new, ..., old]  --> LPOP

Hashes

Maps of field-value pairs under a single key. Perfect for objects.

// HSET (set fields)
await redis.hset('user:1', {
  name: 'Alice',
  email: 'alice@example.com',
  age: '30',
  city: 'New York',
});

// HGET (single field)
const name = await redis.hget('user:1', 'name'); // 'Alice'

// HGETALL (all fields)
const user = await redis.hgetall('user:1');
// { name: 'Alice', email: 'alice@example.com', age: '30', city: 'New York' }

// HMGET (multiple fields)
const [name, email] = await redis.hmget('user:1', 'name', 'email');

// HINCRBY (increment numeric field)
await redis.hincrby('user:1', 'login_count', 1);

// HDEL (delete field)
await redis.hdel('user:1', 'city');

// HEXISTS
const exists = await redis.hexists('user:1', 'email'); // 1 or 0

Hash vs String for objects:

String approach: SET user:1 '{"name":"Alice","age":30}'
  - Must serialize/deserialize entire object
  - Can't update a single field

Hash approach: HSET user:1 name Alice age 30
  - Update individual fields
  - Read individual fields
  - Atomic field increment

Sets

Unordered collections of unique values.

// SADD (add members)
await redis.sadd('tags:post:42', 'javascript', 'nodejs', 'backend');
await redis.sadd('tags:post:43', 'javascript', 'react', 'frontend');

// SMEMBERS (get all members)
const tags = await redis.smembers('tags:post:42');
// ['javascript', 'nodejs', 'backend'] (unordered)

// SISMEMBER (check membership)
const isMember = await redis.sismember('tags:post:42', 'nodejs'); // 1

// SCARD (count)
const count = await redis.scard('tags:post:42'); // 3

// Set operations
const common = await redis.sinter('tags:post:42', 'tags:post:43'); // ['javascript']
const all = await redis.sunion('tags:post:42', 'tags:post:43');
const onlyIn42 = await redis.sdiff('tags:post:42', 'tags:post:43');

// SREM (remove)
await redis.srem('tags:post:42', 'backend');

// SRANDMEMBER (random)
const random = await redis.srandmember('tags:post:42', 2); // 2 random members

Set use cases:
  - Unique visitors: SADD visitors:2024-01-15 user_id
  - Tags/categories: SADD tags:post:42 tag1 tag2
  - Online users: SADD online user_id
  - Mutual friends: SINTER friends:alice friends:bob

Sorted Sets

Like sets, but each member has a score for ordering. The killer data structure for leaderboards, priority queues, and time-series.

// ZADD (add with score)
await redis.zadd('leaderboard', 1500, 'alice', 1200, 'bob', 1800, 'carol');

// ZRANGE (by rank, ascending)
const bottom3 = await redis.zrange('leaderboard', 0, 2); // ['bob', 'alice', 'carol']

// ZREVRANGE (by rank, descending)
const top3 = await redis.zrevrange('leaderboard', 0, 2, 'WITHSCORES');
// ['carol', '1800', 'alice', '1500', 'bob', '1200']

// ZSCORE (get score)
const score = await redis.zscore('leaderboard', 'alice'); // '1500'

// ZRANK / ZREVRANK (position)
const rank = await redis.zrevrank('leaderboard', 'alice'); // 1 (0-based)

// ZINCRBY (increment score)
await redis.zincrby('leaderboard', 50, 'alice'); // 1550

// ZRANGEBYSCORE (by score range)
const elite = await redis.zrangebyscore('leaderboard', 1500, '+inf', 'WITHSCORES');

// ZCARD (count)
const total = await redis.zcard('leaderboard');

// ZREM (remove)
await redis.zrem('leaderboard', 'bob');

Sorted Set use cases:
  Leaderboards:       ZADD leaderboard score player
  Priority queue:     ZADD tasks priority taskId
  Rate limiting:       ZADD requests:user:1 timestamp requestId
  Time-series:        ZADD events timestamp eventData
  Trending topics:    ZINCRBY trending 1 "topic"

Key Expiration and Management

// Set TTL
await redis.expire('session:abc', 3600);     // 1 hour
await redis.pexpire('key', 5000);            // 5000 milliseconds

// Check TTL
const ttl = await redis.ttl('session:abc');  // seconds remaining (-1 = no expiry, -2 = doesn't exist)

// Remove expiration
await redis.persist('key');

// Delete keys
await redis.del('key1', 'key2');

// Check existence
const exists = await redis.exists('key'); // 1 or 0

// Find keys (avoid in production — use SCAN instead)
// await redis.keys('user:*'); // DANGEROUS on large datasets

// SCAN (safe iteration)
let cursor = '0';
do {
  const [nextCursor, keys] = await redis.scan(cursor, 'MATCH', 'session:*', 'COUNT', 100);
  cursor = nextCursor;
  if (keys.length) console.log('Found keys:', keys);
} while (cursor !== '0');

Exercises

  1. Build a Leaderboard

    Implement a game leaderboard using Redis sorted sets: addScore(userId, score), getTopN(n), getRank(userId), getAroundMe(userId, range). Handle ties and real-time updates.

    Hint

    Use ZADD for scores, ZREVRANGE for top N, ZREVRANK for rank, ZREVRANGEBYRANK for surrounding players. Use ZINCRBY for adding to existing scores.

  2. Session Store

    Build a Redis-backed session store: createSession(userId) -> sessionToken, getSession(token) -> userData, updateSession(token, data), destroySession(token), cleanExpiredSessions(). Use hashes for session data with 24-hour TTL.

    Hint

    Use HSET for session data, EXPIRE for TTL, HGETALL to retrieve. Generate random tokens with crypto.randomUUID().

  3. Unique Visitor Counter

    Build a system that tracks unique daily visitors per page using sets. Implement: trackVisit(page, userId), getUniqueVisitors(page, date), getMostVisitedPages(date). Automatically expire old data.

    Hint

    Use SADD visitors:{date}:{page} userId. Use SCARD for counts. Use KEYS or SCAN to find all pages for a date.

Common Problems & Solutions

1 Redis returns all values as strings
Cause: Redis stores everything as strings internally. Numbers, booleans, and objects are all returned as strings.
Solution: Parse values after retrieval: `parseInt(await redis.get('count'))`, `JSON.parse(await redis.get('data'))`. For hashes, all field values are strings too.
2 KEYS command blocks Redis for seconds
Cause: KEYS iterates through ALL keys in the database. With millions of keys, this blocks the single-threaded Redis server.
Solution: Never use KEYS in production. Use SCAN with a cursor for safe iteration. Or use naming conventions and direct key access instead of searching.
3 Memory usage grows unbounded
Cause: No TTL set on cached data, or lists/sets growing without limits.
Solution: Always set TTL on cache keys. Use LTRIM to cap list sizes. Monitor with `redis-cli INFO memory`. Set maxmemory and eviction policy in redis.conf.
