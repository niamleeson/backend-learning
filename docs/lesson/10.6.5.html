Design a CI/CD Pipeline (GitHub Actions)

Intuition

A CI/CD pipeline is a distributed job execution engine triggered by source control events, where the central challenge is balancing the throughput of thousands of concurrent builds against the isolation guarantees each build needs. Ephemeral runners provide the cleanest isolation but make caching the critical optimization to avoid rebuilding the world every time.

Design a CI/CD Pipeline (GitHub Actions)

Requirements

  • Trigger builds on push, pull request, schedule, or manual dispatch
  • Execute multi-step workflows with parallel and sequential jobs on isolated runners
  • Support artifact caching, secret management, and environment-specific deployments
  • Scale to thousands of concurrent builds with job queuing and prioritization

Architecture

A webhook listener receives events from the source control system and enqueues workflow runs. A scheduler matches jobs to available runners from a pool of ephemeral VMs/containers. Each runner executes steps in an isolated environment, streaming logs back to the control plane. An artifact store holds build outputs and caches.

Key Components

  • Webhook/Event Listener: Receives SCM events, evaluates trigger conditions, creates workflow runs
  • Scheduler: Matches pending jobs to runners by label (OS, architecture), manages queue priority
  • Runner Pool: Ephemeral VMs or containers provisioned on demand; cleaned up after each job
  • Artifact & Cache Store: S3-backed storage for build artifacts, dependency caches, and logs

Scaling

Auto-scale runner pools based on queue depth. Use spot/preemptible instances for cost savings with checkpointing for interrupted jobs. Cache dependency downloads aggressively to reduce build times.

Trade-offs

  • Ephemeral runners are clean and secure but have cold-start overhead; persistent runners are faster but risk contamination
  • Shared caches speed up builds but introduce cache invalidation complexity
  • Self-hosted runners give control but require maintenance; managed runners are convenient but limit customization

Exercises

  1. Build a DAG-Based Workflow Executor

    Implement a workflow engine that parses a YAML-defined job DAG (with 'needs' dependencies), runs independent jobs in parallel, and respects sequential dependencies. Track pass/fail status.

    Hint

    Topologically sort the job DAG. Use a thread pool: enqueue all jobs with zero unmet dependencies. When a job completes, decrement dependency counts of its children and enqueue newly-unblocked jobs.

  2. Implement Dependency Caching for Builds

    Create a cache layer keyed by a hash of lock files (e.g., package-lock.json). On cache hit, restore the dependency directory; on miss, run install and upload the result.

    Hint

    Compute SHA-256 of the lock file as cache key. Check the artifact store for that key. On hit, download and extract. On miss, run install, tar the result, and upload with the key.

Common Problems & Solutions

1 Builds queue for 10+ minutes during peak hours despite available cloud capacity
Cause: The runner auto-scaler reacts too slowly, waiting for sustained queue depth before provisioning new VMs
Solution: Implement predictive scaling based on historical patterns (e.g., post-standup push spikes), and reduce the scale-up cooldown period. Pre-warm a baseline pool for expected peak.
2 A secret is exposed in build logs, creating a security incident
Cause: A build step accidentally prints an environment variable containing a secret to stdout
Solution: Implement automatic secret masking in the log streaming layer: scan output for known secret values and replace them with '***'. Also audit and rotate the exposed secret.
