SELECT, WHERE, and Basic Queries

Introduction

SQL (Structured Query Language) is the universal language for communicating with relational databases. Whether you're using PostgreSQL, MySQL, SQLite, or SQL Server, the fundamentals remain the same. As a backend developer working with Node.js, you'll write SQL directly or through ORMs — but understanding raw SQL is non-negotiable.

This lesson covers the foundational building blocks: retrieving data with SELECT, filtering with WHERE, and combining these with sorting, limiting, and aliasing.

Setting Up: Connecting to PostgreSQL from Node.js

Before diving into SQL syntax, let's establish a connection using the pg library:

// db.js
import pg from 'pg';
const { Pool } = pg;

const pool = new Pool({
  host: 'localhost',
  port: 5432,
  database: 'learning_sql',
  user: 'postgres',
  password: 'secret',
});

export async function query(text, params) {
  const start = Date.now();
  const result = await pool.query(text, params);
  const duration = Date.now() - start;
  console.log('Executed query', { text, duration, rows: result.rowCount });
  return result;
}

export default pool;

The SELECT Statement

SELECT is how you read data from a database. At its simplest:

SELECT * FROM users;

This retrieves every column and every row from the users table. In Node.js:

const result = await query('SELECT * FROM users');
console.log(result.rows);
// [{ id: 1, name: 'Alice', email: 'alice@example.com', age: 30 }, ...]

Selecting Specific Columns

Never use SELECT * in production code. Always specify the columns you need:

SELECT name, email FROM users;

const result = await query('SELECT name, email FROM users');
// [{ name: 'Alice', email: 'alice@example.com' }, ...]

Why? Because SELECT *: - Fetches unnecessary data, wasting bandwidth - Breaks if table schema changes - Prevents index-only scans (a major performance optimization)

Column Aliases

Use AS to rename columns in the result:

SELECT name AS username, email AS contact_email FROM users;

DISTINCT

Remove duplicate rows:

SELECT DISTINCT city FROM users;
SELECT DISTINCT city, state FROM users;  -- distinct combinations

The WHERE Clause

WHERE filters rows based on conditions:

SELECT name, email FROM users WHERE age > 25;

Comparison Operators

=     Equal to
<>    Not equal to (also !=)
<     Less than
>     Greater than
<=    Less than or equal to
>=    Greater than or equal to

Logical Operators

-- AND: both conditions must be true
SELECT * FROM users WHERE age > 25 AND city = 'New York';

-- OR: either condition must be true
SELECT * FROM users WHERE city = 'New York' OR city = 'Boston';

-- NOT: negates a condition
SELECT * FROM users WHERE NOT city = 'New York';

IN, BETWEEN, LIKE, IS NULL

-- IN: matches any value in a list
SELECT * FROM users WHERE city IN ('New York', 'Boston', 'Chicago');

-- BETWEEN: inclusive range
SELECT * FROM users WHERE age BETWEEN 25 AND 35;

-- LIKE: pattern matching (% = any chars, _ = single char)
SELECT * FROM users WHERE name LIKE 'A%';       -- starts with A
SELECT * FROM users WHERE email LIKE '%@gmail%'; -- contains @gmail

-- ILIKE: case-insensitive LIKE (PostgreSQL)
SELECT * FROM users WHERE name ILIKE 'alice';

-- IS NULL / IS NOT NULL
SELECT * FROM users WHERE phone IS NULL;
SELECT * FROM users WHERE phone IS NOT NULL;

Parameterized Queries (CRITICAL for Security)

Never concatenate user input into SQL strings. Use parameterized queries:

// DANGEROUS - SQL injection vulnerability!
const bad = await query(`SELECT * FROM users WHERE name = '${userInput}'`);

// SAFE - parameterized query
const safe = await query('SELECT * FROM users WHERE name = $1', [userInput]);

// Multiple parameters
const result = await query(
  'SELECT * FROM users WHERE age > $1 AND city = $2',
  [25, 'New York']
);

Query Flow with Parameters:

  Node.js App          PostgreSQL         
  +----------+         +----------+       
  | query()  |---SQL-->| Parser   |       
  | params[] |---$1--->| Planner  |       
  +----------+         | Executor |       
                       +----------+       

  The database handles escaping, preventing injection.

ORDER BY

Sort results:

-- Ascending (default)
SELECT * FROM users ORDER BY name ASC;

-- Descending
SELECT * FROM users ORDER BY created_at DESC;

-- Multiple columns
SELECT * FROM users ORDER BY city ASC, name DESC;

LIMIT and OFFSET

Control how many rows to return:

-- First 10 rows
SELECT * FROM users ORDER BY id LIMIT 10;

-- Skip first 20, then take 10 (pagination)
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 20;

In Node.js for pagination:

async function getUsers(page = 1, pageSize = 20) {
  const offset = (page - 1) * pageSize;
  const result = await query(
    'SELECT id, name, email FROM users ORDER BY id LIMIT $1 OFFSET $2',
    [pageSize, offset]
  );
  return result.rows;
}

Expressions and Computed Columns

-- String concatenation
SELECT first_name || ' ' || last_name AS full_name FROM users;

-- Math
SELECT name, price, quantity, price * quantity AS total FROM order_items;

-- CASE expressions
SELECT name, age,
  CASE
    WHEN age < 18 THEN 'minor'
    WHEN age < 65 THEN 'adult'
    ELSE 'senior'
  END AS age_group
FROM users;

Putting It All Together

// A realistic query combining everything
async function searchUsers({ search, minAge, city, page = 1, limit = 20 }) {
  const conditions = [];
  const params = [];
  let paramIndex = 1;

  if (search) {
    conditions.push(`(name ILIKE $${paramIndex} OR email ILIKE $${paramIndex})`);
    params.push(`%${search}%`);
    paramIndex++;
  }

  if (minAge) {
    conditions.push(`age >= $${paramIndex}`);
    params.push(minAge);
    paramIndex++;
  }

  if (city) {
    conditions.push(`city = $${paramIndex}`);
    params.push(city);
    paramIndex++;
  }

  const whereClause = conditions.length > 0
    ? 'WHERE ' + conditions.join(' AND ')
    : '';

  const offset = (page - 1) * limit;
  params.push(limit, offset);

  const sql = `
    SELECT id, name, email, age, city
    FROM users
    ${whereClause}
    ORDER BY name ASC
    LIMIT $${paramIndex} OFFSET $${paramIndex + 1}
  `;

  const result = await query(sql, params);
  return result.rows;
}

Common Pitfalls

  1. NULL comparisons: WHERE age = NULL does NOT work. Use WHERE age IS NULL.
  2. String comparison: SQL strings use single quotes 'text', not double quotes.
  3. OFFSET performance: Large offsets are slow. For big datasets, use cursor-based pagination instead.
  4. SELECT *: Tempting for development but harmful in production.
  5. Missing ORDER BY with LIMIT: Without ORDER BY, the database can return rows in any order, making LIMIT unpredictable.

Exercises

  3. CASE Expression Report

    Hint

    Use a CASE WHEN expression in the SELECT clause. Join with a customers table if needed.

Common Problems & Solutions

1 Query returns no rows when filtering for NULL values
Cause: Using `WHERE column = NULL` instead of `WHERE column IS NULL`. In SQL, NULL is not equal to anything, not even itself.
Solution: Always use `IS NULL` or `IS NOT NULL` for NULL comparisons. Example: `SELECT * FROM users WHERE phone IS NULL`.
2 SQL injection vulnerability in dynamic queries
Cause: Concatenating user input directly into SQL strings instead of using parameterized queries.
Solution: Always use parameterized queries: `pool.query('SELECT * FROM users WHERE id = $1', [userId])`. Never use template literals or string concatenation with user input.

// Bad: query(SELECT * FROM users WHERE id = ${id}) // Good: query('SELECT * FROM users WHERE id = $1', [id])

3 OFFSET-based pagination becomes extremely slow on large tables
Symptoms: Query time increases linearly with page number. Page 1 is fast, page 1000 is very slow.
Cause: PostgreSQL must scan and discard all rows before the offset, so OFFSET 100000 scans 100,000 rows just to throw them away.
Solution: Use cursor-based (keyset) pagination: `WHERE id > $lastSeenId ORDER BY id LIMIT 20`. This uses the index directly and is O(1) regardless of page number.
