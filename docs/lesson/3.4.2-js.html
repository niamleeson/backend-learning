JWT Authentication: Tokens, Refresh, and Revocation

Intuition

A JWT is like a concert wristband. Once the bouncer (server) gives you the wristband (token), you can show it at any stage (service) without going back to the box office. But if the wristband is stolen, it works until the concert ends (expiry) — you cannot cancel a wristband that is already on someone's wrist.

JWT Authentication: Tokens, Refresh, and Revocation

Introduction

JSON Web Tokens (JWT) provide a stateless authentication mechanism. Unlike sessions, JWTs do not require server-side storage — the token itself contains all the information needed to verify a user's identity. This makes JWT ideal for APIs, microservices, and mobile applications where statelessness and scalability are priorities.

How JWT Works

Login Flow:
  Client sends credentials (email + password)
      |
      v
  Server verifies credentials
      |
      v
  Server creates JWT (signed with secret)
      |
      v
  Server returns JWT to client
      |
      v
  Client stores JWT (memory, localStorage, httpOnly cookie)

Request Flow:
  Client sends request with JWT in Authorization header
      Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
      |
      v
  Server verifies JWT signature and expiration
      |
      v
  Server extracts user info from token payload
      |
      v
  req.user is populated, request proceeds

JWT Structure

A JWT has three parts separated by dots: Header.Payload.Signature

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.    <- Header (algorithm, type)
eyJ1c2VySWQiOjEsImVtYWlsIjoiYUBiLmNvbSJ9. <- Payload (claims)
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c <- Signature

The payload contains claims — statements about the user:

{
  "userId": 1,
  "email": "alice@example.com",
  "role": "admin",
  "iat": 1704067200,
  "exp": 1704070800
}

Token Service

// src/modules/auth/token.service.ts
import jwt from 'jsonwebtoken';
import { config } from '../../config';

interface TokenPayload {
  userId: number;
  email: string;
  role: string;
}

export const tokenService = {
  generateAccessToken(payload: TokenPayload): string {
    return jwt.sign(payload, config.jwt.secret, {
      expiresIn: config.jwt.accessExpiresIn, // e.g., '15m'
    });
  },

  generateRefreshToken(payload: TokenPayload): string {
    return jwt.sign(payload, config.jwt.refreshSecret, {
      expiresIn: config.jwt.refreshExpiresIn, // e.g., '7d'
    });
  },

  verifyAccessToken(token: string): TokenPayload {
    return jwt.verify(token, config.jwt.secret) as TokenPayload;
  },

  verifyRefreshToken(token: string): TokenPayload {
    return jwt.verify(token, config.jwt.refreshSecret) as TokenPayload;
  },

  generateTokenPair(payload: TokenPayload) {
    return {
      accessToken: this.generateAccessToken(payload),
      refreshToken: this.generateRefreshToken(payload),
    };
  },
};

Auth Routes

// src/modules/auth/auth.routes.ts
import { Router } from 'express';
import bcrypt from 'bcrypt';
import { tokenService } from './token.service';
import { userService } from '../users/users.service';

const router = Router();

// POST /auth/login
router.post('/login', async (req, res, next) => {
  try {
    const { email, password } = req.body;

    const user = await userService.findByEmail(email);
    if (!user || !(await bcrypt.compare(password, user.passwordHash))) {
      return res.status(401).json({ error: 'Invalid credentials' });
    }

    const payload = { userId: user.id, email: user.email, role: user.role };
    const tokens = tokenService.generateTokenPair(payload);

    // Store refresh token hash in database
    await userService.storeRefreshToken(user.id, tokens.refreshToken);

    res.json({
      data: {
        user: { id: user.id, email: user.email, name: user.name },
        ...tokens,
      }
    });
  } catch (error) {
    next(error);
  }
});

// POST /auth/refresh
router.post('/refresh', async (req, res, next) => {
  try {
    const { refreshToken } = req.body;
    if (!refreshToken) {
      return res.status(400).json({ error: 'Refresh token required' });
    }

    // Verify the refresh token
    const payload = tokenService.verifyRefreshToken(refreshToken);

    // Check if refresh token is still valid in database
    const isValid = await userService.verifyStoredRefreshToken(payload.userId, refreshToken);
    if (!isValid) {
      return res.status(401).json({ error: 'Refresh token revoked' });
    }

    // Generate new token pair (rotate refresh token)
    const newPayload = { userId: payload.userId, email: payload.email, role: payload.role };
    const tokens = tokenService.generateTokenPair(newPayload);

    // Replace old refresh token with new one
    await userService.storeRefreshToken(payload.userId, tokens.refreshToken);

    res.json({ data: tokens });
  } catch (error) {
    if (error instanceof jwt.TokenExpiredError) {
      return res.status(401).json({ error: 'Refresh token expired' });
    }
    next(error);
  }
});

// POST /auth/logout
router.post('/logout', authenticate, async (req, res, next) => {
  try {
    // Revoke refresh token
    await userService.removeRefreshToken(req.user!.id);
    res.json({ message: 'Logged out' });
  } catch (error) {
    next(error);
  }
});

export default router;

Authentication Middleware

// src/middleware/authenticate.ts
import { Request, Response, NextFunction } from 'express';
import { tokenService } from '../modules/auth/token.service';

export function authenticate(req: Request, res: Response, next: NextFunction) {
  const authHeader = req.get('authorization');

  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Access token required' });
  }

  const token = authHeader.slice(7);

  try {
    const payload = tokenService.verifyAccessToken(token);
    req.user = { id: payload.userId, email: payload.email, role: payload.role };
    next();
  } catch (error) {
    if (error instanceof jwt.TokenExpiredError) {
      return res.status(401).json({ error: 'Token expired', code: 'TOKEN_EXPIRED' });
    }
    if (error instanceof jwt.JsonWebTokenError) {
      return res.status(401).json({ error: 'Invalid token' });
    }
    next(error);
  }
}

Refresh Token Flow

Access Token: Short-lived (15 min), used for API requests
Refresh Token: Long-lived (7 days), used only to get new access tokens

Normal flow:
1. Client logs in -> receives access + refresh tokens
2. Client uses access token for API calls
3. Access token expires (15 min)
4. Client sends refresh token to /auth/refresh
5. Server validates refresh token, returns new access + refresh tokens
6. Client continues with new access token

Security: If access token is stolen, it expires quickly.
          If refresh token is stolen, it can be revoked server-side.

Token Revocation

JWTs are stateless — you cannot invalidate them once issued. Common revocation strategies:

1. Blocklist (Token Blacklist)

// Store revoked tokens in Redis with TTL matching token expiry
import { redis } from '../config/redis';

async function revokeToken(token: string, expiresIn: number) {
  await redis.set(`blacklist:${token}`, '1', 'EX', expiresIn);
}

async function isTokenRevoked(token: string): Promise<boolean> {
  const result = await redis.get(`blacklist:${token}`);
  return result !== null;
}

// Check in authenticate middleware
export async function authenticate(req: Request, res: Response, next: NextFunction) {
  const token = extractToken(req);
  if (await isTokenRevoked(token)) {
    return res.status(401).json({ error: 'Token revoked' });
  }
  // ... verify token
}

2. Token Versioning

// Store a version number per user; increment on logout/password change
// Include version in token payload; reject if version mismatch
const payload = { userId: user.id, tokenVersion: user.tokenVersion };
// In middleware: if payload.tokenVersion !== user.tokenVersion -> reject

Security Best Practices

// 1. Use strong secrets (at least 256 bits)
// node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"

// 2. Keep access tokens short-lived
const accessToken = jwt.sign(payload, secret, { expiresIn: '15m' });

// 3. Never store sensitive data in JWT payload (it's base64, not encrypted)
// BAD: { password: '...', ssn: '...' }
// GOOD: { userId: 1, role: 'admin' }

// 4. Use different secrets for access and refresh tokens

// 5. Always verify token algorithm
jwt.verify(token, secret, { algorithms: ['HS256'] });

// 6. Send tokens over HTTPS only

JWT vs Sessions: When to Use Which

JWT:
  + Stateless (no server storage for access tokens)
  + Works across services (microservices)
  + Good for mobile/SPA clients
  - Cannot revoke immediately
  - Larger request size
  - Must handle refresh logic

Sessions:
  + Easy to revoke (delete from store)
  + Smaller cookie size
  + Simple server-side control
  - Requires server-side storage
  - Sticky sessions or shared store needed for scaling
  - CSRF protection needed

Common Pitfalls

Storing JWTs in localStorage

// VULNERABLE to XSS: any script can read localStorage
localStorage.setItem('token', jwt);

// SAFER: httpOnly cookie (not accessible via JavaScript)
res.cookie('accessToken', token, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict',
  maxAge: 15 * 60 * 1000,
});

Not Handling Token Expiration on the Client

Always check for TOKEN_EXPIRED errors and trigger a refresh flow rather than forcing the user to log in again.

Summary

JWT authentication provides stateless, scalable auth for APIs. Use short-lived access tokens with longer-lived refresh tokens. Implement token rotation on refresh, revocation via blocklists or versioning, and always follow security best practices. Choose JWT when you need stateless auth across services, and sessions when you need simple, revocable server-side control.

Exercises

  1. Complete JWT Auth System

    Implement JWT auth with login, register, refresh, and logout. Use access tokens (15min) and refresh tokens (7 days). Store refresh tokens in a database. Implement token rotation on refresh.

    Hint

    Use different secrets for access and refresh tokens. On refresh, validate the stored refresh token, generate a new pair, and replace the stored refresh token.

  2. Token Revocation with Redis

    Add a token blocklist using Redis. When a user logs out or changes password, add their current access token to the blocklist with TTL matching the remaining token lifetime. Check the blocklist in the auth middleware.

    Hint

    Calculate remaining TTL from the token's exp claim: Math.ceil(payload.exp - Date.now()/1000). Use Redis SET with EX option.

  3. Role-Based Access Control

    Build authorization middleware that works with JWT. Implement roles (admin, editor, user) and permissions. Create a middleware factory: authorize('admin', 'editor') that checks the role from the JWT payload.

    Hint

    The authorize factory returns middleware that checks req.user.role against the allowed roles array. Chain with authenticate: router.delete('/users/:id', authenticate, authorize('admin'), handler).

Common Problems & Solutions

1 JWT token verification fails with 'invalid signature'
Cause: The secret used to verify differs from the one used to sign, or the token was modified/corrupted.
Solution: Ensure the same secret is used for signing and verification. Check for extra whitespace or encoding issues in the secret. Use different secrets for access and refresh tokens but be consistent.
2 User stays logged in even after changing password
Cause: Existing JWTs remain valid until they expire. Password change does not invalidate outstanding tokens.
Solution: Implement token versioning: increment a version counter on password change and include it in the JWT. Reject tokens with outdated versions. Or use a blocklist to revoke all existing tokens.
3 Token expired errors flood the client
Cause: The client does not implement automatic token refresh, so every request after expiry fails.
Solution: Implement an HTTP interceptor (axios/fetch) that catches 401 TOKEN_EXPIRED errors, calls the refresh endpoint, and retries the original request with the new token.
