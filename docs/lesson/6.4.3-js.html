GCP and Heroku: Alternative Platforms

Intuition

Choosing a cloud platform is like choosing a restaurant for a group dinner. AWS is the massive buffet with hundreds of options but overwhelming choices. GCP is the modern fusion restaurant with some standout dishes (Cloud Run, BigQuery). Heroku is the fast-casual chain where you order at the counter and get food in minutes -- less customizable but incredibly convenient.

GCP and Heroku: Alternative Platforms

Google Cloud Platform (GCP)

Cloud Run (Serverless Containers)

Cloud Run is one of GCP's best offerings: deploy any Docker container and it scales automatically, including to zero.

# Dockerfile for Cloud Run
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
RUN adduser -D appuser
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package.json ./
USER appuser
# Cloud Run sets PORT environment variable
CMD ["node", "dist/server.js"]

# Deploy to Cloud Run
gcloud run deploy myapp \
  --source . \
  --region us-central1 \
  --allow-unauthenticated \
  --set-env-vars NODE_ENV=production \
  --set-env-vars DATABASE_URL="$DATABASE_URL" \
  --memory 512Mi \
  --cpu 1 \
  --min-instances 0 \
  --max-instances 10 \
  --port 3000

Cloud Functions (Serverless Functions)

// index.ts - Google Cloud Function
import { HttpFunction } from '@google-cloud/functions-framework';

export const helloWorld: HttpFunction = (req, res) => {
  const name = req.query.name || 'World';
  res.json({ message: `Hello, ${name}!` });
};

# Deploy Cloud Function
gcloud functions deploy helloWorld \
  --runtime nodejs20 \
  --trigger-http \
  --allow-unauthenticated \
  --region us-central1

Cloud SQL (Managed PostgreSQL)

# Create Cloud SQL instance
gcloud sql instances create myapp-db \
  --database-version POSTGRES_16 \
  --tier db-f1-micro \
  --region us-central1

# Create database
gcloud sql databases create myapp --instance myapp-db

# Connect from Cloud Run (use Cloud SQL proxy)
gcloud run deploy myapp \
  --add-cloudsql-instances myproject:us-central1:myapp-db \
  --set-env-vars DATABASE_URL="postgresql://user:pass@localhost:5432/myapp?host=/cloudsql/myproject:us-central1:myapp-db"

Heroku

Heroku is a PaaS (Platform as a Service) that makes deployment extremely simple. It is ideal for prototypes, MVPs, and small to medium applications.

Project Setup for Heroku

// package.json
{
  "name": "myapp",
  "version": "1.0.0",
  "engines": {
    "node": "20.x"
  },
  "scripts": {
    "start": "node dist/server.js",
    "build": "tsc",
    "heroku-postbuild": "npm run build && npx prisma migrate deploy"
  }
}

# Procfile
web: node dist/server.js
worker: node dist/workers/queue.js
release: npx prisma migrate deploy

Heroku CLI Commands

# Login
heroku login

# Create app
heroku create myapp-api

# Add PostgreSQL
heroku addons:create heroku-postgresql:essential-0

# Add Redis

# Set environment variables
heroku config:set NODE_ENV=production

# View logs

Platform Decision Matrix

Choose AWS when:
  - Enterprise scale requirements
  - Need specific services (Lambda, SQS, etc.)
  - Cost optimization is priority at scale
  - Team has AWS expertise

Choose GCP when:
  - Using Kubernetes (GKE is excellent)
  - Machine learning workloads
  - Cloud Run fits your use case
  - BigQuery for analytics

Choose Heroku when:
  - Speed of deployment is priority
  - Small team / solo developer
  - Prototype or MVP
  - Don't want to manage infrastructure
  - Budget is not the primary concern

Choose Vercel/Railway/Render when:
  - Modern JS/TS stack
  - Want Heroku-like simplicity
  - Better pricing than Heroku
  - GitHub-integrated deployment

Common Pitfalls

  1. Vendor lock-in - Design your application with clean abstractions so you can switch platforms. Use environment variables for all configuration.
  2. Not understanding pricing - Each platform charges differently. Heroku charges per dyno/hour, AWS per resource, GCP per invocation/second.
  3. Ignoring data residency - Know which region your data lives in for compliance (GDPR, etc.).
  4. Over-engineering for small apps - A simple Heroku or Railway deployment is often better than a complex AWS setup for apps with < 10,000 users.
  5. Not using managed databases - Running your own PostgreSQL on a VM is asking for trouble. Use RDS, Cloud SQL, or Heroku Postgres.

Exercises

  1. Deploy to Cloud Run

    Containerize a Node.js Express application and deploy it to Google Cloud Run. Configure it to scale to zero when idle and up to 5 instances under load. Connect it to a Cloud SQL PostgreSQL instance.

    Hint

    Use gcloud run deploy --source . for automatic Dockerfile detection. Set --min-instances 0 and --max-instances 5. Use Cloud SQL proxy for database connection.

  2. Heroku Pipeline

    Deploy a Node.js app to Heroku with a pipeline: staging and production environments. Configure review apps for pull requests. Set up automatic deploys from the main branch to staging and manual promotion to production.

    Hint

    Use heroku pipelines:create. Add staging and production apps. Enable review apps in the Heroku Dashboard under Pipeline settings.

  3. Multi-Platform Comparison

    Deploy the same Node.js application to AWS (EC2 or Lambda), GCP (Cloud Run), and Heroku. Document: deployment time, monthly cost estimate for 10,000 requests/day, cold start latency, and developer experience rating.

    Hint

    Use free tiers where possible. Use curl to measure response times. Document each platform's deployment steps and count them.

Common Problems & Solutions

1 Heroku app crashes with H10 error
Symptoms: heroku[router]: at=error code=H10 desc="App crashed"
Cause: The app is not binding to the PORT environment variable provided by Heroku (it uses a dynamic port, not 3000).
Solution: Use `process.env.PORT || 3000` in your app: `app.listen(process.env.PORT || 3000)`. Check `heroku logs --tail` for the actual error. Ensure your start script in package.json is correct.
2 Cloud Run request timeout (504)
Cause: The request took longer than the configured timeout (default 300 seconds), or the container is not responding to health checks.
Solution: Ensure your app listens on the PORT environment variable (Cloud Run sets this). Increase the timeout with --timeout flag. Check container logs: `gcloud run services logs read myapp`. Verify the container starts within 4 minutes.
3 Database connection fails after platform sleep/idle
Cause: The platform shut down the dyno/instance, and the database connection pool was lost. On restart, stale connections cause errors.
Solution: Configure connection pool with proper idle timeout and retry logic. Use Prisma's built-in connection management. For Heroku, connections drop after 30 minutes of inactivity -- handle reconnection gracefully.
