GraphQL Fundamentals: Schema, Queries, and Mutations

Intuition

GraphQL is like a custom sandwich shop where you pick every ingredient from a menu, versus REST which is like a fixed-menu restaurant where you get the chef's predetermined combo meal -- even if you only wanted the salad.

GraphQL Fundamentals: Schema, Queries, and Mutations

GraphQL is a query language for APIs that lets clients request exactly the data they need. Unlike REST, where the server determines the response shape, GraphQL puts the client in control.

Why GraphQL?

REST has fundamental limitations for complex UIs:

REST: Multiple requests for one screen
+--------+     GET /users/1         +--------+
| Client | -----------------------> | Server |
|        |     GET /users/1/posts   |        |
|        | -----------------------> |        |
|        |     GET /posts/5/comments|        |
|        | -----------------------> |        |
+--------+                          +--------+
  3 round trips, over-fetching on each

GraphQL: One request for exactly what you need
+--------+     POST /graphql        +--------+
| Client | -----------------------> | Server |
|        |     { user, posts,       |        |
|        |       comments }         |        |
+--------+                          +--------+
  1 round trip, exact data

Schema Definition Language (SDL)

GraphQL APIs are defined by a schema using SDL:

# Types define the shape of data
type User {
  id: ID!
  email: String!
  name: String!
  bio: String
  posts: [Post!]!
  createdAt: DateTime!
}

type Post {
  id: ID!
  title: String!
  body: String!
  published: Boolean!
  author: User!
  comments: [Comment!]!
  tags: [String!]!
  createdAt: DateTime!
  updatedAt: DateTime!
}

type Comment {
  id: ID!
  text: String!
  author: User!
  post: Post!
  createdAt: DateTime!
}

# Custom scalar
scalar DateTime

# Enums
enum PostStatus {
  DRAFT
  PUBLISHED
  ARCHIVED
}

# Input types for mutations
input CreatePostInput {
  title: String!
  body: String!
  tags: [String!]
  status: PostStatus = DRAFT
}

input UpdatePostInput {
  title: String
  body: String
  tags: [String!]
  status: PostStatus
}

# Query type: read operations
type Query {
  user(id: ID!): User
  users(limit: Int = 20, offset: Int = 0): [User!]!
  post(id: ID!): Post
  posts(
    limit: Int = 20
    offset: Int = 0
    status: PostStatus
    authorId: ID
  ): [Post!]!
  searchPosts(query: String!): [Post!]!
}

# Mutation type: write operations
type Mutation {
  createUser(email: String!, name: String!, bio: String): User!
  createPost(input: CreatePostInput!): Post!
  updatePost(id: ID!, input: UpdatePostInput!): Post!
  deletePost(id: ID!): Boolean!
  addComment(postId: ID!, text: String!): Comment!
}

Key SDL Concepts

  • ! means non-nullable (field is guaranteed to have a value)
  • [Post!]! means a non-nullable array of non-nullable Posts
  • ID is a special scalar type for identifiers
  • input types are used for mutation arguments
  • Default values: limit: Int = 20

Setting Up Apollo Server

import { ApolloServer } from '@apollo/server';
import { expressMiddleware } from '@apollo/server/express4';
import express from 'express';
import { readFileSync } from 'fs';

// Load schema from .graphql file
const typeDefs = readFileSync('./schema.graphql', 'utf-8');

// Or define inline
const typeDefs = `#graphql
  type Query {
    hello: String!
  }
`;

// Resolvers implement the schema
const resolvers = {
  Query: {
    hello: () => 'Hello, GraphQL!',
  },
};

async function startServer() {
  const app = express();
  const server = new ApolloServer({ typeDefs, resolvers });
  await server.start();

  app.use('/graphql', express.json(), expressMiddleware(server));
  app.listen(4000, () => console.log('Server running on port 4000'));
}

startServer();

Writing Queries

Clients send queries to request specific fields:

# Basic query
query {
  users {
    id
    name
    email
  }
}

# Query with arguments
query {
  user(id: "1") {
    name
    posts {
      title
      createdAt
    }
  }
}

# Query with variables (recommended for production)
query GetUser($userId: ID!) {
  user(id: $userId) {
    name
    email
    posts(status: PUBLISHED) {
      title
      body
      comments {
        text
        author {
          name
        }
      }
    }
  }
}
# Variables: { "userId": "1" }

Aliases and Fragments

# Aliases: query the same field with different arguments
query {
  publishedPosts: posts(status: PUBLISHED, limit: 5) {
    ...PostFields
  }
  draftPosts: posts(status: DRAFT, limit: 5) {
    ...PostFields
  }
}

# Fragment: reusable field selection
fragment PostFields on Post {
  id
  title
  createdAt
  author {
    name
  }
}

Writing Mutations

mutation CreatePost($input: CreatePostInput!) {
  createPost(input: $input) {
    id
    title
    status
    author {
      name
    }
  }
}

# {
#   "input": {
#     "title": "GraphQL Fundamentals",
#     "body": "Learn GraphQL from scratch...",
#     "tags": ["graphql", "api"],
#     "status": "DRAFT"
#   }
# }

Implementing Resolvers

import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

const resolvers = {
  Query: {
    users: async (_parent, args) => {
      return prisma.user.findMany({
        take: args.limit,
        skip: args.offset,
      });
    },

    user: async (_parent, args) => {
      return prisma.user.findUnique({ where: { id: args.id } });
    },

    posts: async (_parent, args) => {
      const where: any = {};
      if (args.status) where.status = args.status;
      if (args.authorId) where.authorId = args.authorId;

      return prisma.post.findMany({
        where,
        take: args.limit,
        skip: args.offset,
        orderBy: { createdAt: 'desc' },
      });
    },

    post: async (_parent, args) => {
      return prisma.post.findUnique({ where: { id: args.id } });
    },

    searchPosts: async (_parent, args) => {
      return prisma.post.findMany({
        where: {
          OR: [
            { title: { contains: args.query, mode: 'insensitive' } },
            { body: { contains: args.query, mode: 'insensitive' } },
          ],
        },
      });
    },
  },

  Mutation: {
    createPost: async (_parent, args, context) => {
      if (!context.userId) throw new Error('Authentication required');

      return prisma.post.create({
        data: {
          ...args.input,
          authorId: context.userId,
        },
      });
    },

    updatePost: async (_parent, args, context) => {
      const post = await prisma.post.findUnique({ where: { id: args.id } });
      if (!post) throw new Error('Post not found');
      if (post.authorId !== context.userId) throw new Error('Not authorized');

      return prisma.post.update({
        where: { id: args.id },
        data: args.input,
      });
    },

    deletePost: async (_parent, args, context) => {
      const post = await prisma.post.findUnique({ where: { id: args.id } });
      if (!post) throw new Error('Post not found');
      if (post.authorId !== context.userId) throw new Error('Not authorized');

      await prisma.post.delete({ where: { id: args.id } });
      return true;
    },
  },

  // Field-level resolvers for relationships
  User: {
    posts: async (parent) => {
      return prisma.post.findMany({ where: { authorId: parent.id } });
    },
  },

  Post: {
    author: async (parent) => {
      return prisma.user.findUnique({ where: { id: parent.authorId } });
    },
    comments: async (parent) => {
      return prisma.comment.findMany({ where: { postId: parent.id } });
    },
  },

  Comment: {
    author: async (parent) => {
      return prisma.user.findUnique({ where: { id: parent.authorId } });
    },
    post: async (parent) => {
      return prisma.post.findUnique({ where: { id: parent.postId } });
    },
  },
};

Resolver Execution Flow

query {
  user(id: "1") {        <-- Query.user resolver runs
    name                  <-- returned from Query.user
    posts {               <-- User.posts resolver runs
      title               <-- returned from User.posts
      author {            <-- Post.author resolver runs (for EACH post)
        name
      }
    }
  }
}

Execution:
1. Query.user(_, { id: "1" }) -> User
2. User.posts(user) -> [Post, Post, ...]
3. For each Post: Post.author(post) -> User   <-- N+1 problem!

Error Handling in GraphQL

import { GraphQLError } from 'graphql';

const resolvers = {
  Mutation: {
    createPost: async (_parent, args, context) => {
      if (!context.userId) {
        throw new GraphQLError('You must be logged in', {
          extensions: {
            code: 'UNAUTHENTICATED',
            http: { status: 401 },
          },
        });
      }

      try {
        return await prisma.post.create({ data: args.input });
      } catch (err) {
        if (err.code === 'P2002') {
          throw new GraphQLError('A post with this title already exists', {
            extensions: { code: 'CONFLICT' },
          });
        }
        throw new GraphQLError('Failed to create post', {
          extensions: { code: 'INTERNAL_SERVER_ERROR' },
        });
      }
    },
  },
};

GraphQL errors are returned in the errors array alongside partial data:

{
  "data": { "createPost": null },
  "errors": [
    {
      "message": "You must be logged in",
      "extensions": { "code": "UNAUTHENTICATED" },
      "path": ["createPost"]
    }
  ]
}

GraphQL vs REST: When to Choose GraphQL

Factor Use GraphQL Use REST
Multiple frontends with different data needs Yes Maybe
Complex, nested data relationships Yes Harder
Simple CRUD operations Overkill Yes
File upload/download Harder Yes
Caching More complex Easy (HTTP cache)
Team familiarity Learning curve Well understood

Exercises

  1. Build a Blog GraphQL API

    Create an Apollo Server with types for User, Post, and Comment. Implement Query resolvers for fetching users, posts (with filtering by status), and a single post by ID. Implement Mutation resolvers for creating and updating posts. Use Prisma for the data layer.

    Hint

    Start with the schema SDL, then implement resolvers one by one. Test queries in Apollo Sandbox at http://localhost:4000/graphql.

  2. Implement Input Validation

    Add input validation to mutations using Zod. Validate that post titles are between 5-200 characters, body is at least 50 characters, and tags array has at most 10 items. Return proper GraphQL errors with validation details.

    Hint

    Parse input with zodSchema.safeParse() in the resolver and throw a GraphQLError with the issues in extensions.

  3. Add Pagination to Queries

    Implement Relay-style cursor pagination for the posts query. Return a Connection type with edges (node + cursor), pageInfo (hasNextPage, hasPreviousPage, startCursor, endCursor), and totalCount.

    Hint

    Define PostConnection, PostEdge, and PageInfo types in SDL. Use base64-encoded IDs as cursors.

Common Problems & Solutions

1 N+1 query problem with nested resolvers
Cause: Each field-level resolver (e.g., Post.author) makes a separate database query for every parent record.
Solution: Use DataLoader to batch and cache database queries. This is covered in depth in lesson 5.2.3.
2 GraphQL queries can be arbitrarily deep and expensive
Cause: Clients can request deeply nested data (user -> posts -> comments -> author -> posts -> ...) causing massive database load.
Solution: Implement query depth limiting, query complexity analysis, and field-level cost directives. Apollo Server supports plugins for this.
3 Cannot use HTTP caching with GraphQL
Cause: All GraphQL requests go to the same POST /graphql endpoint, so CDNs and HTTP caches cannot differentiate between queries.
Solution: Use Apollo Server's built-in response caching plugin, persisted queries with GET requests, or a CDN that supports GraphQL-aware caching.
