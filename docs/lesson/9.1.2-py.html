Load Balancing: Algorithms and Strategies

Intuition

A load balancer is like a restaurant host seating guests at tables. Round robin seats one group per table in order; least connections checks which waiter has the fewest tables before assigning the next group.

Load Balancing Algorithms & Strategies

Load balancing distributes incoming traffic across multiple servers to improve reliability, scalability, and performance. Choosing the right algorithm depends on your workload characteristics and infrastructure constraints.

Core Concepts

  • Round Robin: Requests are distributed sequentially across servers. Simple but ignores server capacity and current load.
  • Weighted Round Robin: Assigns weights to servers based on capacity, directing more traffic to stronger machines.
  • Least Connections: Routes to the server with the fewest active connections, ideal for long-lived requests with variable processing times.
  • IP Hash / Consistent Hashing: Maps client IPs to specific servers, providing session affinity without sticky session cookies.
  • Layer 4 vs Layer 7: L4 balancers route based on IP/TCP information (fast, protocol-agnostic). L7 balancers inspect HTTP content for smarter routing (path-based, header-based).

Key Takeaways

  1. No single algorithm fits all cases; choose based on whether requests are uniform or variable in cost.
  2. Health checks are essential regardless of algorithm to avoid routing to failed servers.
  3. Layer 7 load balancing enables advanced patterns like canary deployments and A/B testing.

Exercises

  1. Configure Nginx as a Load Balancer

    Set up Nginx with an upstream block containing three backend servers. Experiment with round_robin, least_conn, and ip_hash directives. Send 100 requests with curl and observe the distribution.

    Hint

    Use upstream backend { least_conn; server 127.0.0.1:8001; server 127.0.0.1:8002; server 127.0.0.1:8003; } and add logging to each backend to track request counts.

  2. Implement Consistent Hashing

    Write a Python class that implements consistent hashing with virtual nodes. Support adding/removing servers and demonstrate that only a fraction of keys are remapped when a server is added or removed.

    Hint

    Use hashlib.md5 to hash both server names and keys onto a ring. Create 150 virtual nodes per server and use bisect to find the nearest node.

Common Problems & Solutions

1 One server in the pool is receiving disproportionately more traffic than others.
Cause: Using round robin with servers of unequal capacity, or sticky sessions causing hot spots.
Solution: Switch to weighted round robin with weights proportional to server CPU/memory, or use least-connections to naturally balance load.
2 Users lose their session state after being routed to a different server.
Cause: Session data is stored in-memory on individual servers, and the load balancer does not guarantee session affinity.
Solution: Externalize session storage to Redis or a shared database, or enable sticky sessions via cookies at the load balancer level.
