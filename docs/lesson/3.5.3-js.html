Connection Pooling and Performance

Intuition

A connection pool is like a fleet of rental cars at an airport. Instead of buying a new car (opening a connection) every time someone lands (request arrives) and scrapping it when they leave (closing connection), you maintain a fleet (pool) of cars that travelers check out and return. The fleet size must match demand without exceeding parking capacity (database limits).

Connection Pooling and Performance

Introduction

Database connections are expensive resources. Every connection consumes memory on both the application and database server, and opening/closing connections adds latency. Connection pooling reuses a fixed set of connections across requests, dramatically improving performance and resource utilization. This lesson covers how Prisma manages connections, how to tune pool settings, and performance optimization techniques.

How Connection Pooling Works

Without Pooling:
  Request 1 -> Open connection -> Query -> Close connection  (50ms overhead)
  Request 2 -> Open connection -> Query -> Close connection  (50ms overhead)
  Request 3 -> Open connection -> Query -> Close connection  (50ms overhead)

With Pooling:
  Pool: [conn1, conn2, conn3, conn4, conn5]

  Request 1 -> Borrow conn1 -> Query -> Return conn1  (0ms overhead)
  Request 2 -> Borrow conn2 -> Query -> Return conn2  (0ms overhead)
  Request 3 -> Borrow conn1 -> Query -> Return conn1  (0ms overhead)
  (connections are reused)

  If all connections busy:
  Request 6 -> Wait for available connection -> Borrow -> Query -> Return

Prisma Connection Pool

Prisma manages a connection pool automatically. Configure it via the connection URL:

# .env
DATABASE_URL="postgresql://user:pass@localhost:5432/mydb?connection_limit=10&pool_timeout=10"

Key Parameters

// Connection string parameters:
// connection_limit  - Max connections in pool (default: num_cpus * 2 + 1)
// pool_timeout      - Seconds to wait for a connection (default: 10)
// connect_timeout   - Seconds to wait for DB connection (default: 5)

// For a typical web server:
// Development:  connection_limit=5
// Production:   connection_limit=10-20 (depends on DB max_connections)

Sizing the Pool

Database max_connections (default PostgreSQL: 100)

If you have 3 app server instances:
  Each instance pool: 100 / 3 = ~30 connections max
  Recommended: 20 connections per instance (leave headroom)

Rule of thumb: pool_size = (CPU cores * 2) + effective_spindle_count
For most web apps: 10-20 connections per instance works well

Query Performance Optimization

1. Select Only Needed Fields

// SLOW: fetches all columns including large text fields
const users = await prisma.user.findMany();

// FAST: fetch only needed fields
const users = await prisma.user.findMany({
  select: { id: true, name: true, email: true },
});

2. Avoid N+1 Queries

// N+1 PROBLEM: 1 query for posts + N queries for authors
const posts = await prisma.post.findMany();
for (const post of posts) {
  const author = await prisma.user.findUnique({ where: { id: post.authorId } });
  // This makes a separate query for EACH post!
}

// FIXED: Eager load with include (1 query for posts + 1 query for authors)
const posts = await prisma.post.findMany({
  include: { author: { select: { id: true, name: true } } },
});

3. Use Pagination

// NEVER: fetch all records
const allPosts = await prisma.post.findMany();

// ALWAYS: paginate
const posts = await prisma.post.findMany({
  skip: (page - 1) * limit,
  take: limit,
  orderBy: { createdAt: 'desc' },
});

4. Parallel Queries with Promise.all

// SEQUENTIAL: ~200ms (100ms + 100ms)
const users = await prisma.user.count();
const posts = await prisma.post.count();

// PARALLEL: ~100ms (both run simultaneously)
const [userCount, postCount] = await Promise.all([
  prisma.user.count(),
  prisma.post.count(),
]);

5. Batch Operations

// SLOW: individual creates
for (const user of users) {
  await prisma.user.create({ data: user });
}

// FAST: batch create
await prisma.user.createMany({
  data: users,
  skipDuplicates: true,
});

Prisma Query Logging

const prisma = new PrismaClient({
  log: [
    { level: 'query', emit: 'event' },
    { level: 'error', emit: 'stdout' },
  ],
});

// Log slow queries
prisma.$on('query', (e) => {
  if (e.duration > 100) { // > 100ms
    console.warn(`Slow query (${e.duration}ms): ${e.query}`);
  }
});

Middleware for Response Time Monitoring

// Track and log response times
app.use((req, res, next) => {
  const start = process.hrtime.bigint();

  res.on('finish', () => {
    const duration = Number(process.hrtime.bigint() - start) / 1_000_000;

    if (duration > 500) {
      console.warn(`Slow request: ${req.method} ${req.path} ${duration.toFixed(0)}ms`);
    }
  });

  next();
});

Connection Management in Production

Health Check with Database

app.get('/health', async (req, res) => {
  try {
    await prisma.$queryRaw`SELECT 1`;
    res.json({ status: 'healthy', database: 'connected' });
  } catch {
    res.status(503).json({ status: 'unhealthy', database: 'disconnected' });
  }
});

Graceful Shutdown

async function gracefulShutdown(signal: string) {
  console.log(`${signal} received. Shutting down gracefully.`);

  // Stop accepting new requests
  server.close(async () => {
    // Disconnect Prisma (returns connections to pool, then closes them)
    await prisma.$disconnect();
    console.log('Database connections closed.');
    process.exit(0);
  });

  // Force exit after timeout
  setTimeout(() => {
    console.error('Forced shutdown.');
    process.exit(1);
  }, 15000);
}

Using PgBouncer for Advanced Pooling

For high-scale deployments, use PgBouncer as an external connection pooler between your app and PostgreSQL:

App Instance 1 (pool: 10) --|
App Instance 2 (pool: 10) --|--> PgBouncer (pool: 50) --> PostgreSQL (max: 100)
App Instance 3 (pool: 10) --|

# Connection string with PgBouncer
DATABASE_URL="postgresql://user:pass@pgbouncer-host:6432/mydb?pgbouncer=true"

When using PgBouncer with Prisma, add ?pgbouncer=true to your connection string to disable features incompatible with PgBouncer.

Common Pitfalls

Connection Pool Exhaustion

// WRONG: forgetting to await causes connection leaks
prisma.user.findMany(); // Promise not awaited, connection not returned

// WRONG: long-running transactions hold connections
await prisma.$transaction(async (tx) => {
  await tx.user.findMany();
  await someSlowExternalApiCall(); // Connection held during external call!
  await tx.post.create({ data: {} });
});

// CORRECT: minimize transaction scope
const data = await someSlowExternalApiCall(); // Outside transaction
await prisma.$transaction(async (tx) => {
  await tx.user.findMany();
  await tx.post.create({ data }); // Quick DB operations only
});

Over-Provisioning the Pool

If 5 instances each have pool_size=50:
  Total connections: 250
  PostgreSQL default max_connections: 100
  Result: "too many connections" errors!

Always: total_pools * pool_size < database_max_connections

Summary

Connection pooling is essential for production database performance. Prisma manages pools automatically, but you must tune the pool size based on your deployment topology. Optimize queries by selecting only needed fields, avoiding N+1 problems, using pagination, running parallel queries, and batching operations. Monitor query performance, implement health checks, and handle shutdown gracefully.

Common Problems & Solutions

1 'Timed out fetching a new connection from the connection pool' error
Cause: All connections in the pool are in use. Requests are waiting longer than pool_timeout for an available connection.
Solution: Increase connection_limit in DATABASE_URL. Check for connection leaks (unawaited queries, long transactions). Reduce transaction scope. Consider using PgBouncer.
2 Database 'too many connections' error in production
Cause: Total connections across all application instances exceed the database's max_connections setting.
Solution: Calculate: num_instances * pool_size must be less than max_connections. Reduce pool_size per instance or increase database max_connections. Use PgBouncer for external pooling.
3 API responses are slow despite simple queries
Cause: N+1 query problem — loading relations in a loop instead of using eager loading or joins.
Solution: Use Prisma's include or select for eager loading. Enable query logging to identify N+1 patterns. Use Promise.all for independent parallel queries.
