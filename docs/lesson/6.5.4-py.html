CQRS and Event Sourcing Introduction

Intuition

CQRS is like a restaurant where the kitchen (writes) and the menu board (reads) are separate systems. The kitchen handles complex cooking operations while the menu board is optimized for customers to quickly see what is available. Event sourcing is like a bank ledger: instead of just showing your current balance, it records every deposit and withdrawal, so you can reconstruct the balance at any point in time.

CQRS and Event Sourcing Introduction

CQRS (Command Query Responsibility Segregation) separates read and write models. Event Sourcing stores state changes as a sequence of events rather than overwriting current state.

CQRS: Separate Read and Write Models

from dataclasses import dataclass
from abc import ABC, abstractmethod

# Command side: handles writes
@dataclass
class CreateOrderCommand:
    customer_id: int
    items: list[dict]

class CommandHandler:
    def __init__(self, uow: AbstractUnitOfWork):
        self.uow = uow

    def handle_create_order(self, cmd: CreateOrderCommand) -> int:
        with self.uow:
            order = Order(customer_id=cmd.customer_id)
            for item in cmd.items:
                order.add_line(**item)
            order.submit()
            self.uow.orders.add(order)
        return order.id

# Query side: optimized for reads
@dataclass
class OrderSummary:
    id: int
    customer_name: str
    total: float
    item_count: int
    status: str

class OrderQueryService:
    def __init__(self, read_db):
        self.read_db = read_db

    def get_order_summary(self, order_id: int) -> OrderSummary:
        # Reads from a denormalized view optimized for this query
        row = self.read_db.execute(
            "SELECT o.id, c.name, o.total, o.item_count, o.status "
            "FROM order_summaries o JOIN customers c ON o.customer_id = c.id "
            "WHERE o.id = :id", {"id": order_id}
        ).fetchone()
        return OrderSummary(*row)

Event Sourcing Basics

@dataclass
class Event:
    timestamp: datetime

@dataclass
class OrderCreated(Event):
    order_id: int
    customer_id: int

@dataclass
class ItemAdded(Event):
    order_id: int
    product_id: int
    quantity: int

class Order:
    def __init__(self):
        self.events: list[Event] = []
        self.items = []
        self.status = "draft"

    def apply(self, event: Event):
        if isinstance(event, OrderCreated):
            self.id = event.order_id
        elif isinstance(event, ItemAdded):
            self.items.append({"product_id": event.product_id, "qty": event.quantity})
        self.events.append(event)

    def rebuild(self, events: list[Event]):
        for event in events:
            self.apply(event)

Common Mistakes

  • Applying CQRS everywhere when simple CRUD is sufficient
  • Not keeping the read model eventually consistent with the write model
  • Event sourcing without proper event versioning and schema evolution

Key Takeaways

  • CQRS allows independent optimization of reads and writes
  • Event sourcing provides a complete audit trail and enables temporal queries
  • Start simple (shared model) and introduce CQRS only when read/write needs diverge significantly

Exercises

  1. Implement Simple CQRS

    Build an application where orders are written to a normalized database (commands) and read from a denormalized view (queries). Write a command handler for creating orders and a query service for fetching order summaries with customer details pre-joined.

    Hint

    Use two separate models: a rich Order domain model for writes and a flat OrderSummary DTO for reads. Update the read model after each successful write.

  2. Event-Sourced Shopping Cart

    Build a shopping cart using event sourcing. Events: CartCreated, ItemAdded, ItemRemoved, CartCheckedOut. Rebuild cart state by replaying events. Implement a method to get the cart state at any point in time.

    Hint

    Store events in a list. The Cart class has no mutable state except what is built by replaying events. To get state at time T, replay only events with timestamp <= T.

Common Problems & Solutions

1 Read model is stale after write operations
Symptoms: ['User creates an order but does not see it immediately in the list', 'Dashboard totals lag behind actual values']
Cause: The read model is updated asynchronously and has eventual consistency lag.
Solution: For cases where immediate consistency is required, read from the write model after writes. For normal reads, accept eventual consistency and document the expected lag. Consider using database triggers or change data capture for faster propagation.
2 Cannot deserialize old events after schema change
Symptoms: ['Application crashes when replaying historical events', 'Missing fields in reconstructed aggregate state']
Cause: Event classes were modified (fields renamed/removed) without a migration strategy.
Solution: Version your events. Use upcasting to convert old event versions to the current schema. Never modify existing event schemas; create new versions instead.
