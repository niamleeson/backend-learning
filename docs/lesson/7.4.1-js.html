Zod: Schema Validation and Type Inference

Intuition

Zod is like a bouncer at a nightclub with a detailed guest list. It checks every person (piece of data) against the list (schema), and anyone who does not match the exact criteria gets turned away with a clear explanation of why. The best part is the bouncer also tells the staff inside exactly what type of guests to expect, so everyone is on the same page.

Zod: Schema Validation and Type Inference

Why Input Validation Matters

Every piece of data entering your backend is a potential attack vector or source of bugs. User input, API payloads, query parameters, webhook bodies, configuration files -- none of it can be trusted. Without validation, you risk SQL injection, type coercion bugs, crashes from unexpected undefined values, and data corruption that silently spreads through your system.

Zod is a TypeScript-first schema validation library that solves two problems at once: runtime validation and static type inference. You define your schema once, and Zod gives you both a validator and a TypeScript type -- no duplication, no drift.

Installing Zod

npm install zod

Zod has zero dependencies and works in Node.js, browsers, and edge runtimes. It requires TypeScript 4.5+ for full type inference.

Basic Schema Definition

A Zod schema describes the shape of valid data. When you parse data against a schema, Zod either returns the validated data or throws a detailed error.

import { z } from 'zod';

// Primitive schemas
const nameSchema = z.string();
const ageSchema = z.number();
const isActiveSchema = z.boolean();
const createdAtSchema = z.date();

// Parsing valid data
const name = nameSchema.parse('Alice');  // 'Alice'
const age = ageSchema.parse(30);         // 30

// Parsing invalid data throws ZodError
try {
  nameSchema.parse(42);  // throws!
} catch (err) {
  console.log(err.errors);
  // [{ code: 'invalid_type', expected: 'string', received: 'number', path: [], message: 'Expected string, received number' }]
}

Safe Parsing

In API handlers, you often want validation errors without exceptions. Use safeParse:

const result = nameSchema.safeParse(42);

if (!result.success) {
  console.log(result.error.flatten());
  // { formErrors: [], fieldErrors: {} }
} else {
  console.log(result.data);  // typed as string
}

This pattern is essential for returning structured error responses to clients.

Object Schemas

Object schemas are where Zod shines for API validation:

const UserSchema = z.object({
  name: z.string().min(1, 'Name is required').max(100),
  email: z.string().email('Invalid email address'),
  age: z.number().int().min(13, 'Must be at least 13').max(120),
  role: z.enum(['user', 'admin', 'moderator']),
  bio: z.string().max(500).optional(),
  tags: z.array(z.string()).max(10).default([]),
});

// TypeScript type is inferred automatically
type User = z.infer<typeof UserSchema>;
// { name: string; email: string; age: number; role: 'user' | 'admin' | 'moderator'; bio?: string; tags: string[] }

Type Inference: The Killer Feature

Zod eliminates the need to maintain separate TypeScript interfaces:

┌─────────────────────────────┐
│     Zod Schema Definition   │
│  (single source of truth)   │
├─────────────┬───────────────┤
│             │               │
▼             ▼               │
Runtime       TypeScript      │
Validation    Type            │
(parse)       (z.infer)       │
│             │               │
▼             ▼               │
Safe data     Autocomplete    │
at runtime    at dev time     │
└─────────────┴───────────────┘

// Define once
const CreatePostSchema = z.object({
  title: z.string().min(1).max(200),
  content: z.string().min(10),
  published: z.boolean().default(false),
  categoryId: z.string().uuid(),
});

// Infer the type -- no manual interface needed
type CreatePostInput = z.infer<typeof CreatePostSchema>;

// Use in your service layer with full type safety
function createPost(input: CreatePostInput) {
  // input.title is string, input.published is boolean, etc.
  // TypeScript knows all the types without you writing them
}

String Validations

Zod provides a rich set of string validators:

const stringSchemas = {
  email: z.string().email(),
  url: z.string().url(),
  uuid: z.string().uuid(),
  cuid: z.string().cuid(),
  datetime: z.string().datetime(),        // ISO 8601
  ip: z.string().ip(),                    // v4 or v6
  ipv4: z.string().ip({ version: 'v4' }),
  trimmed: z.string().trim(),             // trims whitespace
  lower: z.string().toLowerCase(),        // transforms to lowercase
  regex: z.string().regex(/^[a-z0-9-]+$/), // custom pattern
  length: z.string().length(10),          // exact length
  range: z.string().min(3).max(50),       // length range
  starts: z.string().startsWith('sk_'),   // API key prefix
  ends: z.string().endsWith('.json'),     // file extension
};

Number Validations

const numberSchemas = {
  integer: z.number().int(),
  positive: z.number().positive(),
  nonNegative: z.number().nonnegative(),
  range: z.number().min(1).max(100),
  finite: z.number().finite(),
  multipleOf: z.number().multipleOf(5),   // must be divisible by 5
  port: z.number().int().min(1).max(65535),
};

Arrays and Tuples

// Array of strings with constraints
const tagsSchema = z.array(z.string()).min(1).max(10);

// Array of objects
const itemsSchema = z.array(
  z.object({
    productId: z.string().uuid(),
    quantity: z.number().int().positive(),
    price: z.number().positive(),
  })
).nonempty('Cart cannot be empty');

// Tuple (fixed-length array with specific types)
const coordinateSchema = z.tuple([
  z.number(),  // latitude
  z.number(),  // longitude
]);

Unions, Discriminated Unions, and Enums

// Simple union
const idSchema = z.union([z.string().uuid(), z.number().int().positive()]);

// Discriminated union (much better error messages)
const EventSchema = z.discriminatedUnion('type', [
  z.object({ type: z.literal('click'), x: z.number(), y: z.number() }),
  z.object({ type: z.literal('scroll'), direction: z.enum(['up', 'down']) }),
  z.object({ type: z.literal('keypress'), key: z.string() }),
]);

// Native enum
const RoleEnum = z.enum(['user', 'admin', 'moderator']);
type Role = z.infer<typeof RoleEnum>;  // 'user' | 'admin' | 'moderator'

// Access enum values
RoleEnum.options;  // ['user', 'admin', 'moderator']

Transforms and Preprocessing

Zod can transform data during validation:

// Coerce string to number (common for query params)
const pageSchema = z.coerce.number().int().positive().default(1);
pageSchema.parse('5');  // 5 (number, not string)
pageSchema.parse(undefined);  // 1 (default)

// Custom transform
const slugSchema = z.string()
  .transform(val => val.toLowerCase().replace(/\s+/g, '-').replace(/[^a-z0-9-]/g, ''));

slugSchema.parse('Hello World!');  // 'hello-world'

// Preprocess: run logic before validation
const dateSchema = z.preprocess(
  (val) => (typeof val === 'string' ? new Date(val) : val),
  z.date()
);

dateSchema.parse('2025-01-15');  // Date object

Composing Schemas

Zod schemas are composable and reusable:

// Base schema
const BaseEntitySchema = z.object({
  id: z.string().uuid(),
  createdAt: z.string().datetime(),
  updatedAt: z.string().datetime(),
});

// Extend with merge
const UserEntitySchema = BaseEntitySchema.merge(
  z.object({
    name: z.string(),
    email: z.string().email(),
  })
);

// Pick specific fields
const UserSummarySchema = UserEntitySchema.pick({
  id: true,
  name: true,
});

// Omit fields
const CreateUserSchema = UserEntitySchema.omit({
  id: true,
  createdAt: true,
  updatedAt: true,
});

// Partial (all fields optional)
const UpdateUserSchema = CreateUserSchema.partial();

// Deep partial
const PatchSchema = UserEntitySchema.deepPartial();

Express.js Integration

Here is a middleware that validates request bodies, query params, and route params:

import { z } from 'zod';
import express from 'express';

// Generic validation middleware
function validate(schemas) {
  return (req, res, next) => {
    const errors = {};

    if (schemas.body) {
      const result = schemas.body.safeParse(req.body);
      if (!result.success) {
        errors.body = result.error.flatten().fieldErrors;
      } else {
        req.body = result.data;
      }
    }

    if (schemas.query) {
      const result = schemas.query.safeParse(req.query);
      if (!result.success) {
        errors.query = result.error.flatten().fieldErrors;
      } else {
        req.query = result.data;
      }
    }

    if (schemas.params) {
      const result = schemas.params.safeParse(req.params);
      if (!result.success) {
        errors.params = result.error.flatten().fieldErrors;
      } else {
        req.params = result.data;
      }
    }

    if (Object.keys(errors).length > 0) {
      return res.status(400).json({
        status: 'error',
        message: 'Validation failed',
        errors,
      });
    }

    next();
  };
}

// Define schemas
const CreateUserBodySchema = z.object({
  name: z.string().min(1).max(100),
  email: z.string().email(),
  password: z.string().min(8).max(128),
});

const PaginationQuerySchema = z.object({
  page: z.coerce.number().int().positive().default(1),
  limit: z.coerce.number().int().min(1).max(100).default(20),
  sort: z.enum(['asc', 'desc']).default('desc'),
});

const UUIDParamsSchema = z.object({
  id: z.string().uuid('Invalid ID format'),
});

// Use in routes
const app = express();
app.use(express.json());

app.post(
  '/api/users',
  validate({ body: CreateUserBodySchema }),
  (req, res) => {
    // req.body is validated and typed
    const { name, email, password } = req.body;
    res.status(201).json({ message: 'User created' });
  }
);

app.get(
  '/api/users',
  validate({ query: PaginationQuerySchema }),
  (req, res) => {
    const { page, limit, sort } = req.query;
    // page and limit are numbers (coerced from strings)
    res.json({ page, limit, sort });
  }
);

app.get(
  '/api/users/:id',
  validate({ params: UUIDParamsSchema }),
  (req, res) => {
    const { id } = req.params;
    res.json({ id });
  }
);

Custom Validations with Refine and Superrefine

// Simple refinement
const PasswordSchema = z.string()
  .min(8)
  .refine(
    (val) => /[A-Z]/.test(val),
    { message: 'Must contain at least one uppercase letter' }
  )
  .refine(
    (val) => /[0-9]/.test(val),
    { message: 'Must contain at least one number' }
  );

// Cross-field validation with superRefine
const RegisterSchema = z.object({
  password: z.string().min(8),
  confirmPassword: z.string(),
}).superRefine((data, ctx) => {
  if (data.password !== data.confirmPassword) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: 'Passwords do not match',
      path: ['confirmPassword'],
    });
  }
});

// Async refinement (e.g., check database uniqueness)
const UniqueEmailSchema = z.string().email().refine(
  async (email) => {
    const existing = await db.user.findUnique({ where: { email } });
    return !existing;
  },
  { message: 'Email already registered' }
);

// Use parseAsync for async schemas
const result = await UniqueEmailSchema.safeParseAsync('test@example.com');

Error Formatting

Zod provides multiple ways to format errors for API responses:

const schema = z.object({
  name: z.string().min(1),
  email: z.string().email(),
  age: z.number().min(13),
});

const result = schema.safeParse({ name: '', email: 'bad', age: 5 });

if (!result.success) {
  // Flat format -- great for API responses
  console.log(result.error.flatten());
  // {
  //   formErrors: [],
  //   fieldErrors: {
  //     name: ['String must contain at least 1 character(s)'],
  //     email: ['Invalid email'],
  //     age: ['Number must be greater than or equal to 13']
  //   }
  // }

  // Formatted (nested)
  console.log(result.error.format());

  // Issues array (raw)
  console.log(result.error.issues);

  // Custom error map
  const customMap = z.setErrorMap((issue, ctx) => {
    if (issue.code === z.ZodIssueCode.too_small) {
      return { message: `Minimum length is ${issue.minimum}` };
    }
    return { message: ctx.defaultError };
  });
}

Real-World Pattern: API Request/Response Schemas

// Shared schemas for a REST API
const ProductSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1).max(200),
  price: z.number().positive(),
  currency: z.enum(['USD', 'EUR', 'GBP']),
  stock: z.number().int().nonnegative(),
  metadata: z.record(z.string(), z.unknown()).optional(),
  createdAt: z.string().datetime(),
});

const CreateProductSchema = ProductSchema.omit({
  id: true,
  createdAt: true,
});

const UpdateProductSchema = CreateProductSchema.partial();

const ProductListResponseSchema = z.object({
  data: z.array(ProductSchema),
  pagination: z.object({
    page: z.number(),
    limit: z.number(),
    total: z.number(),
    totalPages: z.number(),
  }),
});

// Type exports for use throughout the app
type Product = z.infer<typeof ProductSchema>;
type CreateProductInput = z.infer<typeof CreateProductSchema>;
type UpdateProductInput = z.infer<typeof UpdateProductSchema>;

Performance Considerations

Zod validation is fast, but keep these in mind:

  • Schema creation is cheap -- create schemas at module level, not per-request
  • safeParse is marginally faster than parse (no exception overhead)
  • For hot paths, consider z.coerce instead of .transform() where possible
  • Zod adds roughly 0.1-0.5ms per parse for typical API payloads
  • For extremely performance-sensitive code, consider pre-compiling schemas with @zod-accelerator

Summary

Zod gives you a single source of truth for both runtime validation and TypeScript types. Define your schema once, and you get type-safe validators that produce clear error messages. The safeParse pattern integrates naturally with Express error handling, and schema composition (pick, omit, partial, merge) keeps your code DRY as your API grows.

Common Problems & Solutions

1 Query parameters fail number validation even though the value is correct
Cause: Express query parameters are always strings. `z.number()` expects an actual number type, so '5' fails validation.
Solution: Use `z.coerce.number()` instead of `z.number()` for query parameters. Coerce mode attempts to convert the value to the target type before validating: `z.coerce.number().int().positive()` will correctly parse the string '5' into the number 5.
2 TypeScript types do not update after changing the Zod schema
Cause: You are maintaining a separate TypeScript interface alongside the Zod schema instead of using z.infer. The two have drifted out of sync.
Solution: Delete the manual interface and use `type MyType = z.infer` exclusively. This ensures the type always matches the schema. If you need to export types for other modules, export the inferred type.
3 Custom refinements do not show up in the correct field's errors
Cause: When using .refine() on an object schema, the error is added to the form-level errors, not to a specific field.
Solution: Use .superRefine() instead and call ctx.addIssue() with a specific path: `ctx.addIssue({ code: z.ZodIssueCode.custom, message: 'Passwords must match', path: ['confirmPassword'] })`. This places the error on the correct field.
