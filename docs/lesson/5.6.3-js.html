CQRS and Event Sourcing Introduction

Intuition

CQRS is like separating a library's 'check-out desk' from the 'catalog search computer.' The search system can be optimized with indexes and caches without affecting how books are checked in/out. Event Sourcing is like keeping a ledger instead of just a bank balance -- you can always recalculate the balance from the history, and you have a complete record of every transaction.

CQRS and Event Sourcing Introduction

CQRS (Command Query Responsibility Segregation) separates read and write operations into different models. Event Sourcing stores state as a sequence of events rather than current snapshots. Together, they enable scalable, auditable, and flexible systems.

CQRS: Separate Reads and Writes

Traditional (single model):
+--------+      +--------+      +--------+
| Client | ---->| Service |----->|  DB    |
|        | <----|  (CRUD) |<-----|        |
+--------+      +--------+      +--------+
  Same model for reading and writing

CQRS:
+--------+      +---------------+      +-----------+
| Client | ---->| Command Side  |----->| Write DB  |
|        |      | (write model) |      | (normalized)|
+--------+      +---------------+      +-----------+
    |                                       |
    |                                   [events]
    |                                       |
    |           +---------------+      +-----------+
    +---------->| Query Side    |<-----| Read DB   |
                | (read model)  |      | (denormalized)|
                +---------------+      +-----------+

Why CQRS?

  • Read and write scaling independently -- most apps read 90%+ of the time
  • Optimized read models -- denormalized for fast queries
  • Different storage engines -- Postgres for writes, Elasticsearch for reads
  • Simplified models -- each side only handles its concern

Implementing CQRS in Node.js

Command Side (Write)

// commands/create-post.command.ts
export class CreatePostCommand {
  constructor(
    public readonly authorId: string,
    public readonly title: string,
    public readonly body: string,
    public readonly tags: string[],
  ) {}
}

// commands/publish-post.command.ts
export class PublishPostCommand {
  constructor(
    public readonly postId: string,
    public readonly userId: string,
  ) {}
}

// command-handlers/create-post.handler.ts
export class CreatePostHandler {
  constructor(
    private postRepo: IPostRepository,
    private eventBus: EventBus,
  ) {}

  async execute(command: CreatePostCommand): Promise<string> {
    const post = new PostEntity(
      generateId(),
      command.title,
      command.body,
      PostStatus.DRAFT,
      command.authorId,
      command.tags,
      new Date(),
      new Date(),
      null,
    );

    await this.postRepo.create(post);

    // Emit event for read side to consume
    this.eventBus.emit(new PostCreatedEvent({
      postId: post.id,
      title: post.title,
      authorId: post.authorId,
      tags: post.tags,
      createdAt: post.createdAt,
    }));

    return post.id;
  }
}

Query Side (Read)

// queries/get-post-detail.query.ts
export class GetPostDetailQuery {
  constructor(public readonly postId: string) {}
}

// Read model (denormalized, optimized for display)
interface PostDetailView {
  id: string;
  title: string;
  body: string;
  authorName: string;      // denormalized
  authorAvatar: string;    // denormalized
  commentCount: number;    // pre-computed
  tags: string[];
  readTimeMinutes: number; // pre-computed
  createdAt: Date;
  publishedAt: Date | null;
}

// query-handlers/get-post-detail.handler.ts
export class GetPostDetailHandler {
  constructor(private readDb: ReadDatabase) {}

  async execute(query: GetPostDetailQuery): Promise<PostDetailView | null> {
    // Query from the read-optimized store
    return this.readDb.postViews.findById(query.postId);
  }
}

// Event handler that updates the read model
export class PostCreatedEventHandler {
  constructor(private readDb: ReadDatabase) {}

  async handle(event: PostCreatedEvent): Promise<void> {
    const author = await this.readDb.users.findById(event.authorId);

    await this.readDb.postViews.create({
      id: event.postId,
      title: event.title,
      body: event.body,
      authorName: author?.name || 'Unknown',
      authorAvatar: author?.avatarUrl || '',
      commentCount: 0,
      tags: event.tags,
      readTimeMinutes: Math.ceil(event.body.split(/\s+/).length / 200),
      createdAt: event.createdAt,
      publishedAt: null,
    });
  }
}

Simple Event Bus

// infrastructure/event-bus.ts
type EventHandler = (event: any) => Promise<void>;

export class EventBus {
  private handlers = new Map<string, EventHandler[]>();

  subscribe(eventType: string, handler: EventHandler): void {
    const existing = this.handlers.get(eventType) || [];
    existing.push(handler);
    this.handlers.set(eventType, existing);
  }

  async emit(event: { type: string; [key: string]: any }): Promise<void> {
    const handlers = this.handlers.get(event.type) || [];
    await Promise.all(handlers.map(h => h(event)));
  }
}

// Usage
const eventBus = new EventBus();
eventBus.subscribe('PostCreated', postCreatedHandler.handle.bind(postCreatedHandler));
eventBus.subscribe('PostPublished', postPublishedHandler.handle.bind(postPublishedHandler));

NestJS CQRS Module

NestJS has a built-in @nestjs/cqrs package:

import { CommandHandler, ICommandHandler, QueryHandler, IQueryHandler, EventsHandler, IEventHandler } from '@nestjs/cqrs';

@CommandHandler(CreatePostCommand)
export class CreatePostHandler implements ICommandHandler<CreatePostCommand> {
  constructor(private postRepo: IPostRepository) {}

  async execute(command: CreatePostCommand): Promise<string> {
    // write logic
  }
}

@QueryHandler(GetPostDetailQuery)
export class GetPostDetailHandler implements IQueryHandler<GetPostDetailQuery> {
  constructor(private readDb: ReadDatabase) {}

  async execute(query: GetPostDetailQuery): Promise<PostDetailView> {
    // read logic
  }
}

@EventsHandler(PostCreatedEvent)
export class PostCreatedHandler implements IEventHandler<PostCreatedEvent> {
  handle(event: PostCreatedEvent) {
    // update read model
  }
}

// In controller
@Controller('posts')
export class PostsController {
  constructor(
    private commandBus: CommandBus,
    private queryBus: QueryBus,
  ) {}

  @Post()
  async create(@Body() dto: CreatePostDto, @CurrentUser() user: AuthUser) {
    return this.commandBus.execute(new CreatePostCommand(user.id, dto.title, dto.body, dto.tags));
  }

  @Get(':id')
  async findOne(@Param('id') id: string) {
    return this.queryBus.execute(new GetPostDetailQuery(id));
  }
}

Event Sourcing

Instead of storing current state, store every change as an event:

Traditional: Store current state
+---------+
| Orders  |
| id: 1   |
| status: |
|  SHIPPED|
| total:  |
|  $50    |
+---------+

Event Sourcing: Store all events
+---+------------------+----------+
| # | Event            | Data     |
+---+------------------+----------+
| 1 | OrderCreated     | items: []|
| 2 | ItemAdded        | sku: A   |
| 3 | ItemAdded        | sku: B   |
| 4 | OrderPaid        | $50      |
| 5 | OrderShipped     | track: X |
+---+------------------+----------+
Current state = replay all events

// Event store interface
interface EventStore {
  append(streamId: string, events: DomainEvent[]): Promise<void>;
  getEvents(streamId: string): Promise<DomainEvent[]>;
  getEventsSince(streamId: string, version: number): Promise<DomainEvent[]>;
}

// Domain events
interface DomainEvent {
  type: string;
  aggregateId: string;
  version: number;
  timestamp: Date;
  data: Record<string, any>;
}

class OrderCreatedEvent implements DomainEvent {
  type = 'OrderCreated';
  constructor(
    public aggregateId: string,
    public version: number,
    public timestamp: Date,
    public data: { userId: string; items: OrderItem[] },
  ) {}
}

// Event-sourced aggregate
class OrderAggregate {
  private id: string;
  private status: OrderStatus;
  private items: OrderItem[] = [];
  private total: number = 0;
  private version: number = 0;
  private uncommittedEvents: DomainEvent[] = [];

  // Rebuild state from events
  static fromEvents(events: DomainEvent[]): OrderAggregate {
    const order = new OrderAggregate();
    for (const event of events) {
      order.apply(event);
    }
    order.uncommittedEvents = []; // clear after rebuild
    return order;
  }

  private apply(event: DomainEvent): void {
    switch (event.type) {
      case 'OrderCreated':
        this.id = event.aggregateId;
        this.status = OrderStatus.CREATED;
        this.items = event.data.items;
        break;
      case 'OrderPaid':
        this.status = OrderStatus.PAID;
        this.total = event.data.amount;
        break;
      case 'OrderShipped':
        this.status = OrderStatus.SHIPPED;
        break;
    }
    this.version = event.version;
  }

  pay(amount: number): void {
    if (this.status !== OrderStatus.CREATED) {
      throw new DomainError('Can only pay for created orders');
    }
    const event: DomainEvent = {
      type: 'OrderPaid',
      aggregateId: this.id,
      version: this.version + 1,
      timestamp: new Date(),
      data: { amount },
    };
    this.apply(event);
    this.uncommittedEvents.push(event);
  }

  getUncommittedEvents(): DomainEvent[] {
    return [...this.uncommittedEvents];
  }
}

When to Use CQRS and Event Sourcing

Use CQRS when: - Read and write workloads have very different characteristics - You need different read models for different consumers - You want to scale reads and writes independently

Use Event Sourcing when: - You need a complete audit trail - You need to reconstruct past states - The domain is event-driven (finance, logistics, collaborative editing)

Skip both when: - Simple CRUD applications - Small teams that would be burdened by the complexity - The domain does not benefit from event history

Exercises

  1. Implement CQRS for a Blog

    Separate your blog API into command handlers (CreatePost, PublishPost, AddComment) and query handlers (GetPostDetail, ListPosts, GetUserProfile). Use a simple event bus to sync the write model to a denormalized read model.

    Hint

    The write side uses normalized tables. The read side has a 'post_views' table with pre-joined author name, comment count, etc.

  2. Build a Simple Event Store

    Create an event store backed by PostgreSQL with a table: (stream_id, version, event_type, data, timestamp). Implement append (with optimistic concurrency), getEvents, and getEventsSince. Build an OrderAggregate that rebuilds from events.

    Hint

    For optimistic concurrency, include a version check in the INSERT: WHERE version = expected_version.

  3. NestJS CQRS Module

    Use @nestjs/cqrs to implement commands, queries, and events for a task management system. Create commands for creating and completing tasks, queries for listing and filtering, and events that update a read model with task statistics.

    Hint

    Install @nestjs/cqrs and register CqrsModule. Use CommandBus and QueryBus in controllers.

Common Problems & Solutions

1 Read model is out of sync with write model
Cause: Events are lost or event handlers fail, leaving the read model stale.
Solution: Use reliable event delivery (message queue with acknowledgment). Implement a 'rebuild' mechanism that replays all events to reconstruct the read model. Add monitoring for event processing lag.
2 Event sourced aggregate has too many events, slow to load
Cause: Long-lived aggregates accumulate thousands of events that must be replayed on every load.
Solution: Implement snapshots: periodically save the current aggregate state. On load, start from the latest snapshot and replay only events since then.
3 CQRS adds unnecessary complexity to a simple CRUD app
Cause: CQRS was applied where a simple service + repository would suffice.
Solution: Only use CQRS when you have different read/write scaling needs or complex read models. Start simple and refactor to CQRS when the pain justifies the complexity.
