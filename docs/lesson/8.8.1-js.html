CAP Theorem and Consistency Models

Intuition

The CAP theorem is like a package delivery service during a snowstorm (network partition). You can either guarantee the package arrives with the correct contents (consistency) but it might not arrive at all, or guarantee the package arrives (availability) but it might contain yesterday's version of the contents. You cannot guarantee both during the storm. The best services offer different guarantees for different package types — express (strong consistency) for legal documents, standard (eventual consistency) for catalogs.

CAP Theorem and Consistency Models

Introduction

The CAP theorem is the foundational concept for understanding trade-offs in distributed systems. It states that a distributed data store can provide only two of three guarantees: Consistency, Availability, and Partition Tolerance. Since network partitions are inevitable in distributed systems, you must choose between consistency and availability.

The CAP Theorem

        Consistency (C)
           /\
          /  \
         /    \
        / CP   \   CA (theoretical only
       /  sys   \   in practice - no
      /    /    \  partition tolerance)
     /            \
    /      AP      \
   /      sys       \
  /_________________ \
Availability (A)    Partition
                    Tolerance (P)

CP Systems: MongoDB, HBase, Redis (in some modes)
  → Strong consistency, may become unavailable during partition

AP Systems: Cassandra, DynamoDB, CouchDB
  → Always available, may return stale data during partition

CA Systems: Single-node PostgreSQL (not distributed)
  → Not applicable to distributed systems

Understanding Each Property

Consistency

Every read receives the most recent write or an error.

Consistent System:
  T0: Client A writes x = 5 to Node 1
  T1: Client B reads x from Node 2
  T2: Client B gets x = 5 (or error if not yet replicated)

Inconsistent System:
  T0: Client A writes x = 5 to Node 1
  T1: Client B reads x from Node 2
  T2: Client B gets x = 3 (stale value!)

Availability

Every request receives a non-error response, without guarantee that it contains the most recent write.

Partition Tolerance

The system continues to operate despite network partitions between nodes.

Network Partition:
┌──────────┐         X         ┌──────────┐
│  Node 1  │ ─ ─ ─ ─ X ─ ─ ─ ─│  Node 2  │
│  (x = 5) │    network        │  (x = 3) │
└──────────┘     broken         └──────────┘

CP choice: Node 2 refuses reads (unavailable) until partition heals
AP choice: Node 2 returns x = 3 (stale but available)

Consistency Models

Strong Consistency (Linearizability)

After a write completes, all subsequent reads reflect that write. This is what single-database applications provide.

// Strong consistency with PostgreSQL transaction
async function transferMoney(fromId, toId, amount) {
  return prisma.$transaction(async (tx) => {
    const from = await tx.account.findUnique({
      where: { id: fromId },
    });

    if (from.balance < amount) throw new Error('Insufficient funds');

    await tx.account.update({
      where: { id: fromId },
      data: { balance: { decrement: amount } },
    });

    await tx.account.update({
      where: { id: toId },
      data: { balance: { increment: amount } },
    });
    // After COMMIT, all reads see the updated balances
  });
}

Eventual Consistency

If no new updates are made, eventually all reads will return the last updated value. There is a window where reads may return stale data.

// Eventual consistency with cache
async function updateUserProfile(userId, data) {
  // Write to primary DB
  const user = await primaryDB.user.update({
    where: { id: userId },
    data,
  });

  // Cache update (eventually consistent)
  await redis.setex(`user:${userId}`, 3600, JSON.stringify(user));

  // Replica will eventually get the update
  // Reads from replica may return stale data for a few ms

  return user;
}

// Read may return stale data
async function getUserProfile(userId) {
  // Check cache (may be stale)
  const cached = await redis.get(`user:${userId}`);
  if (cached) return JSON.parse(cached);

  // Read from replica (may be slightly behind primary)
  return replicaDB.user.findUnique({ where: { id: userId } });
}

Causal Consistency

Preserves the order of causally related operations, but allows concurrent operations to be seen in different orders.

// Causal consistency example
// If User A posts a message, then replies to it,
// all readers must see the post before the reply.

class CausallyConsistentStore {
  constructor(redis) {
    this.redis = redis;
  }

  async write(key, value, causalDependencies = []) {
    const version = Date.now();

    // Wait for causal dependencies to be visible
    for (const dep of causalDependencies) {
      await this.waitForVersion(dep.key, dep.version);
    }

    await this.redis.hset(`data:${key}`, 'value', JSON.stringify(value), 'version', version);
    return version;
  }

  async read(key) {
    const data = await this.redis.hgetall(`data:${key}`);
    return {
      value: JSON.parse(data.value),
      version: parseInt(data.version),
    };
  }

  async waitForVersion(key, minVersion, timeoutMs = 5000) {
    const start = Date.now();
    while (Date.now() - start < timeoutMs) {
      const data = await this.redis.hget(`data:${key}`, 'version');
      if (parseInt(data) >= minVersion) return;
      await new Promise(r => setTimeout(r, 50));
    }
    throw new Error(`Timeout waiting for ${key} version ${minVersion}`);
  }
}

Read-Your-Writes Consistency

A user always sees their own writes, but may see stale data from other users.

// Read-your-writes with version tracking
class ReadYourWritesProxy {
  constructor(primaryDB, replicaDB, redis) {
    this.primaryDB = primaryDB;
    this.replicaDB = replicaDB;
    this.redis = redis;
  }

  async write(userId, table, data) {
    const result = await this.primaryDB[table].update(data);

    // Track user's last write timestamp
    await this.redis.setex(
      `last_write:${userId}`,
      30, // 30 second window
      Date.now().toString()
    );

    return result;
  }

  async read(userId, table, query) {
    const lastWrite = await this.redis.get(`last_write:${userId}`);

    if (lastWrite && Date.now() - parseInt(lastWrite) < 5000) {
      // User wrote recently — read from primary for freshness
      return this.primaryDB[table].findMany(query);
    }

    // No recent writes — safe to read from replica
    return this.replicaDB[table].findMany(query);
  }
}

Practical CAP Trade-offs

┌──────────────────────────────────────────────────────────────┐
│ Use Case                    │ Consistency Model              │
├─────────────────────────────┼────────────────────────────────┤
│ Bank account balance        │ Strong (ACID transactions)     │
│ Shopping cart               │ Eventual (AP, merge conflicts) │
│ User profile                │ Read-your-writes               │
│ Social media feed           │ Eventual (seconds of lag OK)   │
│ Inventory count             │ Strong for decrements          │
│ Analytics dashboard         │ Eventual (minutes of lag OK)   │
│ Distributed counter (likes) │ Eventual (with CRDT or merge)  │
│ Chat messages               │ Causal (order matters)         │
└──────────────────────────────────────────────────────────────┘

PACELC Theorem

An extension of CAP that also considers latency:

If Partition:
  Choose Availability or Consistency (PA or PC)
Else (normal operation):
  Choose Latency or Consistency (EL or EC)

PACELC Classification:
  DynamoDB: PA/EL (always available, low latency, eventual consistency)
  PostgreSQL: PC/EC (consistent, higher latency for distributed writes)
  MongoDB: PA/EC (available during partition, consistent normally)

Common Pitfalls

  1. Assuming strong consistency everywhere: Many systems don't need it. Eventual consistency with 100ms lag is fine for most read workloads.
  2. Ignoring partition scenarios: Test what happens when the network between your app and database breaks. Does your system fail gracefully?
  3. Not documenting consistency guarantees: Your API consumers need to know if a read might return stale data after a write.
  4. Trying to have all three: Every distributed system must sacrifice something. Understand and accept the trade-offs.

Summary

The CAP theorem forces a choice between consistency and availability during network partitions. Most real-world systems choose eventual consistency for reads (AP) while maintaining strong consistency for critical writes. Understanding consistency models (strong, eventual, causal, read-your-writes) lets you make informed trade-offs. Choose the consistency model based on the business requirement of each operation, not as a system-wide policy.

Exercises

  1. Consistency Model Simulator

    Build a simulation with two Node.js processes representing database nodes. Simulate a network partition and demonstrate the difference between CP and AP behavior.

    Hint

    Use two Express servers with shared state via HTTP. Simulate partition by blocking inter-node communication. CP: return error on reads. AP: return stale data.

  2. Read-Your-Writes Implementation

    Build a data access layer that provides read-your-writes consistency using a primary database and replica. Track per-user write timestamps in Redis.

    Hint

    After a write, store userId:timestamp in Redis with 30s TTL. Before reads, check if user wrote recently — if so, read from primary.

  3. Eventual Consistency Demo

    Create a system where writes go to a primary and reads go to a replica. Measure replication lag and demonstrate how stale reads occur. Add a UI that shows the consistency gap.

    Hint

    Write to primary, immediately read from replica and primary. Compare results. Show the lag decreasing over time.

Common Problems & Solutions

1 Users see stale data after updating their profile
Cause: Writes go to primary database, but reads come from a replica with replication lag.
Solution: Implement read-your-writes consistency: after a write, route that user's subsequent reads to the primary for a few seconds.
2 Inventory oversold during flash sale
Cause: Eventual consistency in the inventory system allows two servers to both read 'in stock' before either write 'out of stock'.
Solution: Use strong consistency (database transaction with row locking) for inventory decrements. Use SELECT ... FOR UPDATE or atomic decrement.
