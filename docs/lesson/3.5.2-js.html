Repository Pattern in Express

Intuition

The Repository Pattern is like having a personal librarian. Instead of searching the stacks yourself (writing Prisma queries everywhere), you tell the librarian what you need (call repository methods). You can replace the librarian with a different one (swap implementations) and your reading habits (business logic) do not change.

Repository Pattern in Express

Introduction

The Repository Pattern provides an abstraction layer between your business logic and data access logic. Instead of scattering Prisma queries throughout controllers and services, you encapsulate all data operations in repository classes. This makes your code more testable, swappable, and maintainable.

Why Use the Repository Pattern?

Without Repository Pattern:

  Controller -> Prisma Client (directly)
  - Hard to test (requires database)
  - Hard to switch databases
  - Business logic mixed with data access
  - Prisma API scattered everywhere

With Repository Pattern:

  Controller -> Service -> Repository -> Prisma Client
  - Easy to test (mock the repository)
  - Easy to switch data sources
  - Clear separation of concerns
  - Data access logic in one place

Basic Repository

// src/modules/users/users.repository.ts
import { prisma } from '../../config/database';
import { Prisma, User } from '@prisma/client';

export type CreateUserData = {
  email: string;
  name: string;
  passwordHash: string;
};

export type UpdateUserData = Partial<Omit<CreateUserData, 'email'>>;

export type UserFilter = {
  search?: string;
  role?: string;
};

export class UserRepository {
  async findAll(filter: UserFilter, page: number, limit: number) {
    const where = this.buildWhereClause(filter);

    const [data, total] = await Promise.all([
      prisma.user.findMany({
        where,
        skip: (page - 1) * limit,
        take: limit,
        orderBy: { createdAt: 'desc' },
        select: {
          id: true, email: true, name: true, role: true,
          createdAt: true, updatedAt: true,
        },
      }),
      prisma.user.count({ where }),
    ]);

    return { data, total };
  }

  async findById(id: number) {
    return prisma.user.findUnique({
      where: { id },
      select: {
        id: true, email: true, name: true, role: true,
        createdAt: true, updatedAt: true,
      },
    });
  }

  async findByEmail(email: string) {
    return prisma.user.findUnique({ where: { email } });
  }

  async create(data: CreateUserData) {
    return prisma.user.create({
      data,
      select: { id: true, email: true, name: true, role: true, createdAt: true, updatedAt: true },
    });
  }

  async update(id: number, data: UpdateUserData) {
    return prisma.user.update({
      where: { id },
      data,
      select: { id: true, email: true, name: true, role: true, createdAt: true, updatedAt: true },
    });
  }

  async delete(id: number) {
    return prisma.user.delete({ where: { id } });
  }

  async exists(id: number): Promise<boolean> {
    const count = await prisma.user.count({ where: { id } });
    return count > 0;
  }

  private buildWhereClause(filter: UserFilter): Prisma.UserWhereInput {
    const where: Prisma.UserWhereInput = {};

    if (filter.search) {
      where.OR = [
        { name: { contains: filter.search, mode: 'insensitive' } },
        { email: { contains: filter.search, mode: 'insensitive' } },
      ];
    }

    if (filter.role) {
      where.role = filter.role as any;
    }

    return where;
  }
}

export const userRepository = new UserRepository();

Interface-Based Repository (For Testability)

// src/modules/users/users.repository.interface.ts
export interface IUserRepository {
  findAll(filter: UserFilter, page: number, limit: number): Promise<{ data: SafeUser[]; total: number }>;
  findById(id: number): Promise<SafeUser | null>;
  findByEmail(email: string): Promise<User | null>;
  create(data: CreateUserData): Promise<SafeUser>;
  update(id: number, data: UpdateUserData): Promise<SafeUser>;
  delete(id: number): Promise<void>;
  exists(id: number): Promise<boolean>;
}

export class PrismaUserRepository implements IUserRepository {

}

// Test implementation
export class InMemoryUserRepository implements IUserRepository {
  private users: User[] = [];
  private nextId = 1;

  async findAll(filter: UserFilter, page: number, limit: number) {
    let filtered = [...this.users];
    if (filter.search) {
      const s = filter.search.toLowerCase();
      filtered = filtered.filter(u =>
        u.name.toLowerCase().includes(s) || u.email.toLowerCase().includes(s)
      );
    }
    const total = filtered.length;
    const data = filtered.slice((page - 1) * limit, page * limit);
    return { data: data.map(this.toSafeUser), total };
  }

  async findById(id: number) {
    const user = this.users.find(u => u.id === id);
    return user ? this.toSafeUser(user) : null;
  }

  async create(data: CreateUserData) {
    const user = { id: this.nextId++, ...data, role: 'USER', createdAt: new Date(), updatedAt: new Date() } as User;
    this.users.push(user);
    return this.toSafeUser(user);
  }

  // ... other methods
}

Service Layer Using Repository

// src/modules/users/users.service.ts
import { IUserRepository } from './users.repository.interface';
import { userRepository } from './users.repository';
import bcrypt from 'bcrypt';
import { ConflictError, NotFoundError } from '../../utils/ApiError';

export class UserService {
  constructor(private repo: IUserRepository = userRepository) {}

  async getUsers(filter: UserFilter, page: number, limit: number) {
    return this.repo.findAll(filter, page, limit);
  }

  async getUserById(id: number) {
    const user = await this.repo.findById(id);
    if (!user) throw new NotFoundError('User', id);
    return user;
  }

  async createUser(data: { email: string; name: string; password: string }) {
    const existing = await this.repo.findByEmail(data.email);
    if (existing) throw new ConflictError('Email already registered');

    const passwordHash = await bcrypt.hash(data.password, 12);
    return this.repo.create({ email: data.email, name: data.name, passwordHash });
  }

  async updateUser(id: number, data: UpdateUserData) {
    if (!(await this.repo.exists(id))) {
      throw new NotFoundError('User', id);
    }
    return this.repo.update(id, data);
  }

  async deleteUser(id: number) {
    if (!(await this.repo.exists(id))) {
      throw new NotFoundError('User', id);
    }
    await this.repo.delete(id);
  }
}

export const userService = new UserService();

Testing with Mock Repository

// src/modules/users/__tests__/users.service.test.ts
import { describe, it, expect, beforeEach } from 'vitest';
import { UserService } from '../users.service';
import { InMemoryUserRepository } from '../users.repository.memory';

describe('UserService', () => {
  let service: UserService;
  let repo: InMemoryUserRepository;

  beforeEach(() => {
    repo = new InMemoryUserRepository();
    service = new UserService(repo);
  });

  it('should create a user', async () => {
    const user = await service.createUser({
      email: 'alice@example.com',
      name: 'Alice',
      password: 'password123',
    });

    expect(user.email).toBe('alice@example.com');
    expect(user.name).toBe('Alice');
    expect(user).not.toHaveProperty('passwordHash');
  });

  it('should throw ConflictError for duplicate email', async () => {
    await service.createUser({ email: 'alice@example.com', name: 'Alice', password: 'pass123' });
    await expect(
      service.createUser({ email: 'alice@example.com', name: 'Bob', password: 'pass456' })
    ).rejects.toThrow('Email already registered');
  });

  it('should throw NotFoundError for missing user', async () => {
    await expect(service.getUserById(999)).rejects.toThrow('User with ID 999 not found');
  });
});

Generic Base Repository

// src/utils/BaseRepository.ts
export interface PaginatedResult<T> {
  data: T[];
  total: number;
}

export interface BaseRepository<T, CreateData, UpdateData> {
  findAll(page: number, limit: number): Promise<PaginatedResult<T>>;
  findById(id: number): Promise<T | null>;
  create(data: CreateData): Promise<T>;
  update(id: number, data: UpdateData): Promise<T>;
  delete(id: number): Promise<void>;
  exists(id: number): Promise<boolean>;
}

When to Use the Repository Pattern

USE when:
  - You want to test business logic without a database
  - You might switch databases or ORMs
  - You have complex query logic that benefits from encapsulation
  - Your team values clear architectural boundaries

SKIP when:
  - Small project / prototype
  - Prisma is tightly coupled and you will not switch
  - The extra abstraction layer adds unnecessary complexity
  - YAGNI (You Aren't Gonna Need It)

Summary

The Repository Pattern abstracts data access behind an interface. This separates concerns, improves testability (inject mock repositories), and centralizes query logic. Use interface-based repositories when you need flexibility, or concrete repositories for simpler apps. The pattern shines in larger applications with complex business logic that needs thorough testing.

Exercises

  1. Build a Repository Layer

    Create repository classes for User and Post models with all CRUD operations. Define interfaces for each repository. Create both Prisma and in-memory implementations. Wire them into services.

    Hint

    Define the interface first (IUserRepository), then implement PrismaUserRepository and InMemoryUserRepository. The service constructor should accept IUserRepository.

  2. Test Service with Mock Repository

    Write comprehensive unit tests for a UserService that uses an in-memory repository. Test: create user, duplicate email handling, find by ID (found and not found), update, and delete. No database needed.

    Hint

    Instantiate InMemoryUserRepository in beforeEach and pass it to new UserService(repo). Tests run in memory with no database setup.

  3. Generic Base Repository

    Create a generic base repository class that implements common operations (findAll, findById, create, update, delete) for any Prisma model. Extend it for specific models that need custom queries.

    Hint

    Use generics: BaseRepository. Pass the Prisma delegate (prisma.user, prisma.post) to the constructor.

Common Problems & Solutions

1 Repository adds boilerplate without clear benefit
Cause: The project is small and the abstraction layer feels like over-engineering.
Solution: For small projects, it is OK to skip the repository pattern and use Prisma directly in services. Adopt it when testability or data source swapping becomes a priority.
2 Repository interface does not match all Prisma query capabilities
Cause: Prisma offers rich filtering, including, and aggregation that is hard to abstract into a generic interface.
Solution: Do not try to abstract every Prisma feature. Use the repository for common CRUD and simple queries. For complex, one-off queries, it is acceptable to use Prisma directly in the service layer.
