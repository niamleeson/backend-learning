Design a Payment System (Stripe)

Intuition

A payment system is a ledger-based transaction processor where idempotency and exactly-once semantics are non-negotiable requirements because duplicate charges directly cost money and erode trust. The double-entry bookkeeping model ensures every financial movement is traceable and balanced.

Design a Payment System (Stripe)

Requirements

  • Process credit card, debit card, and bank transfer payments
  • Ensure exactly-once payment processing with idempotency
  • Support refunds, chargebacks, and multi-currency transactions
  • Comply with PCI-DSS security standards
  • Achieve 99.999% availability for payment processing

Architecture

A payment gateway receives payment requests, tokenizes card data, and routes to a payment processor. A ledger service maintains double-entry bookkeeping for all transactions. An idempotency layer prevents duplicate charges. A reconciliation service matches internal records with bank statements. All card data flows through PCI-compliant isolated infrastructure.

Key Components

  • Payment Gateway: Entry point that validates, rate-limits, and routes payment requests
  • Tokenization Service: Replaces card numbers with tokens in a PCI-compliant vault
  • Ledger Service: Double-entry accounting (every debit has a corresponding credit)
  • Idempotency Layer: Stores idempotency keys with request hashes to detect and deduplicate retries
  • Reconciliation Engine: Batch process that compares internal ledger with external bank/PSP records

Scaling

Use async processing for non-critical paths (notifications, analytics). Synchronous processing only for the critical payment path. Shard the ledger by merchant ID. Use active-active deployment across regions for availability.

Trade-offs

  • Synchronous payment processing ensures immediate confirmation but adds latency; async is faster but requires webhooks
  • Double-entry ledger provides audit trail but doubles write volume compared to single-entry

Exercises

  1. Implement Idempotent Payment Processing

    Build a payment endpoint that accepts an idempotency key with each request. If the same key is seen again, return the original response without reprocessing. Handle edge cases: concurrent requests with the same key, and requests that failed midway.

    Hint

    Store idempotency_key -> {status, response} in a database. Use a unique constraint on the key. For concurrent requests, the second insert fails and waits for the first to complete before returning its result.

  2. Design a Double-Entry Ledger

    Create a ledger system where every payment creates two entries: a debit from the payer's account and a credit to the payee's account. Ensure the sum of all debits always equals the sum of all credits. Support querying account balances.

    Hint

    Each transaction creates a pair of entries in a single atomic database transaction. Balance = SUM(credits) - SUM(debits) for an account. Never update balances directly; always derive from the ledger entries.

Common Problems & Solutions

1 A customer is charged twice for the same purchase because the client retried after a network timeout, but the first request had already succeeded.
Cause: The payment service processed the first request but the response was lost due to a network issue. The client retry created a second charge without idempotency protection.
Solution: Require an idempotency key with every payment request. Before processing, check if the key exists in the idempotency store. If found, return the stored response. Use a database unique constraint on the key to handle races.
2 End-of-day reconciliation reveals a mismatch between the internal ledger and the bank's settlement report.
Cause: Some transactions were recorded internally but failed at the payment processor level, or vice versa, due to network issues or delayed processing.
Solution: Run automated reconciliation that flags discrepancies. For internal-only records, verify with the payment processor API. For bank-only records, create compensating ledger entries. Implement real-time event streaming from the processor to catch discrepancies early.
