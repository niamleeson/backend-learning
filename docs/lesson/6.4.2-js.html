AWS Lambda and Serverless Node.js

Intuition

Lambda functions are like vending machines. You do not run the factory (server) -- you just put your product (code) in the machine, and it dispenses results whenever someone presses a button (triggers an event). You pay per transaction, not for keeping the factory open 24/7. The downside is the first customer after the machine has been idle needs to wait while it warms up (cold start).

AWS Lambda and Serverless Node.js

What is Serverless?

Serverless does not mean "no servers." It means you do not manage servers. AWS Lambda runs your code in response to events, automatically scaling from zero to thousands of concurrent executions. You pay only for the compute time you consume.

Serverless vs Traditional Architecture

Traditional (EC2):
┌────────────┐
│  EC2 Server │  Always running, always costing money
│  ┌────────┐ │  Handles 0-100 requests/sec
│  │Node.js │ │  You manage: OS, patching, scaling
│  │        │ │  Fixed capacity
│  └────────┘ │
└────────────┘

Serverless (Lambda):
┌──────┐ ┌──────┐ ┌──────┐
│ fn() │ │ fn() │ │ fn() │  Spins up per-request
└──────┘ └──────┘ └──────┘  Scales to 1000s automatically
   │         │         │     Costs $0 when idle
   └─────────┼─────────┘     AWS manages everything
             │
        ┌────┴────┐
        │API GW   │
        └─────────┘

Lambda Function Basics

Simple Lambda Handler

// src/handlers/hello.ts
import { APIGatewayProxyEvent, APIGatewayProxyResult } from 'aws-lambda';

export const handler = async (
  event: APIGatewayProxyEvent
): Promise<APIGatewayProxyResult> => {
  const name = event.queryStringParameters?.name || 'World';

  return {
    statusCode: 200,
    headers: {
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      message: `Hello, ${name}!`,
      timestamp: new Date().toISOString(),
    }),
  };
};

CRUD API with Lambda

// src/handlers/users.ts
import { APIGatewayProxyEvent, APIGatewayProxyResult } from 'aws-lambda';
import { PrismaClient } from '@prisma/client';

// Initialize outside handler for connection reuse
const prisma = new PrismaClient();

export const getUsers = async (
  event: APIGatewayProxyEvent
): Promise<APIGatewayProxyResult> => {
  try {
    const limit = parseInt(event.queryStringParameters?.limit || '20');
    const offset = parseInt(event.queryStringParameters?.offset || '0');

    const users = await prisma.user.findMany({
      take: limit,
      skip: offset,
      select: { id: true, name: true, email: true, createdAt: true },
    });

    return {
      statusCode: 200,
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ data: users }),
    };
  } catch (error) {
    console.error('Error fetching users:', error);
    return {
      statusCode: 500,
      body: JSON.stringify({ error: 'Internal server error' }),
    };
  }
};

export const createUser = async (
  event: APIGatewayProxyEvent
): Promise<APIGatewayProxyResult> => {
  try {
    const body = JSON.parse(event.body || '{}');

    const user = await prisma.user.create({
      data: {
        name: body.name,
        email: body.email,
      },
    });

    return {
      statusCode: 201,
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ data: user }),
    };
  } catch (error) {
    console.error('Error creating user:', error);
    return {
      statusCode: 500,
      body: JSON.stringify({ error: 'Internal server error' }),
    };
  }
};

export const getUserById = async (
  event: APIGatewayProxyEvent
): Promise<APIGatewayProxyResult> => {
  const id = event.pathParameters?.id;

  const user = await prisma.user.findUnique({
    where: { id },
  });

  if (!user) {
    return {
      statusCode: 404,
      body: JSON.stringify({ error: 'User not found' }),
    };
  }

  return {
    statusCode: 200,
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ data: user }),
  };
};

Serverless Framework

The Serverless Framework simplifies Lambda deployment with a YAML configuration file.

# serverless.yml
service: myapp-api

frameworkVersion: '3'

provider:
  name: aws
  runtime: nodejs20.x
  region: us-east-1
  stage: ${opt:stage, 'dev'}
  memorySize: 256
  timeout: 10
  environment:
    DATABASE_URL: ${env:DATABASE_URL}
    NODE_ENV: production
  iam:
    role:
      statements:
        - Effect: Allow
          Action:
            - s3:GetObject
            - s3:PutObject
          Resource: arn:aws:s3:::${self:service}-uploads-${self:provider.stage}/*

plugins:
  - serverless-offline         # Local development
  - serverless-prune-plugin    # Clean old Lambda versions

custom:
  serverless-offline:
    httpPort: 3000
  prune:
    automatic: true
    number: 3

functions:
  getUsers:
    handler: dist/handlers/users.getUsers
    events:
      - httpApi:
          path: /users
          method: get

  createUser:
    handler: dist/handlers/users.createUser
    events:
      - httpApi:
          path: /users
          method: post

  getUserById:
    handler: dist/handlers/users.getUserById
    events:
      - httpApi:
          path: /users/{id}
          method: get

  # Scheduled function (cron)
  dailyCleanup:
    handler: dist/handlers/cron.cleanup
    events:
      - schedule: rate(1 day)

  # S3 event trigger
  processUpload:
    handler: dist/handlers/s3.processUpload
    events:
      - s3:
          bucket: ${self:service}-uploads-${self:provider.stage}
          event: s3:ObjectCreated:*

resources:
  Resources:
    UploadsBucket:
      Type: AWS::S3::Bucket
      Properties:
        BucketName: ${self:service}-uploads-${self:provider.stage}

Deploy and Manage

# Install Serverless Framework
npm install -g serverless

# Deploy to AWS
serverless deploy --stage production

# Deploy single function (faster)
serverless deploy function -f getUsers

# Invoke function locally
serverless invoke local -f getUsers -d '{"queryStringParameters": {"limit": "10"}}'

# Run locally with serverless-offline
serverless offline

# View logs
serverless logs -f getUsers -t    # Tail logs

# Remove entire stack
serverless remove

Express on Lambda

You can run your entire Express application on Lambda using a wrapper:

// src/lambda.ts
import serverless from 'serverless-http';
import app from './app';     // Your existing Express app

export const handler = serverless(app);

# serverless.yml
functions:
  api:
    handler: dist/lambda.handler
    events:
      - httpApi: '*'          # Catch all routes

Cold Starts

Cold Start (first request to a new instance):
┌─────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────┐
│ Download     │ │ Init runtime │ │ Init handler  │ │ Execute  │
│ code package │ │ (Node.js)    │ │ (your code)   │ │ handler  │
│  ~200ms      │ │  ~100ms      │ │  ~300-2000ms  │ │  ~50ms   │
└─────────────┘ └──────────────┘ └──────────────┘ └──────────┘
     ▲                                  ▲
     │                                  │
Total cold start: 300-2500ms     DB connections, imports

Warm Start (reusing existing instance):
┌──────────┐
│ Execute  │
│ handler  │
│  ~50ms   │
└──────────┘

Mitigating Cold Starts

// 1. Initialize outside the handler (reused across invocations)
import { PrismaClient } from '@prisma/client';
const prisma = new PrismaClient();     // Created once, reused

export const handler = async (event) => {
  // prisma is already initialized on warm starts
  const users = await prisma.user.findMany();
  return { statusCode: 200, body: JSON.stringify(users) };
};

# 2. Provisioned Concurrency (keeps instances warm)
functions:
  api:
    handler: dist/lambda.handler
    provisionedConcurrency: 5    # Keep 5 instances always warm

# 3. Scheduled warming (cheaper alternative)
functions:
  api:
    handler: dist/lambda.handler
  warmer:
    handler: dist/handlers/warmer.handler
    events:
      - schedule: rate(5 minutes)
    environment:
      FUNCTION_TO_WARM: ${self:service}-${self:provider.stage}-api

Common Pitfalls

  1. Not reusing database connections - Initialize clients outside the handler. Lambda reuses the execution context across invocations.
  2. Packaging node_modules in the zip - Use Lambda layers or bundlers (esbuild) to reduce package size and cold starts.
  3. Ignoring cold starts - For user-facing APIs, cold starts of 2+ seconds are unacceptable. Use provisioned concurrency or warming.
  4. Setting timeout too low - Lambda default is 3 seconds. Database queries and external API calls may need 10-30 seconds.
  5. Not handling Lambda lifecycle - Lambda freezes the process between invocations. Timers, setInterval, and open connections behave unexpectedly.

Exercises

  1. Build a Serverless REST API

    Create a simple CRUD API using the Serverless Framework with Lambda and API Gateway. Implement endpoints for creating, reading, updating, and deleting items. Use DynamoDB or connect to an RDS PostgreSQL instance.

    Hint

    Start with serverless create --template aws-nodejs-typescript. Use serverless-offline for local testing. Define each CRUD operation as a separate function.

  2. Express App on Lambda

    Take an existing Express application and deploy it to Lambda using serverless-http. Compare the local Express response times vs Lambda cold start and warm start times. Measure with curl and CloudWatch.

    Hint

    Install serverless-http. Create a lambda.ts that wraps your Express app. Use a single catch-all route in serverless.yml.

  3. Event-Driven Processing

    Create a Lambda function that triggers on S3 uploads, processes the uploaded image (resize using sharp), and saves the result to a different S3 prefix. Use the Serverless Framework to configure the S3 event trigger.

    Hint

    Use the s3 event in serverless.yml. The event object contains the bucket name and object key. Use sharp for image processing. Write the result to a 'thumbnails/' prefix.

Common Problems & Solutions

1 Lambda function times out
Cause: The function execution exceeds the configured timeout, often due to database connection issues or slow external API calls.
Solution: Increase timeout in serverless.yml (max 900 seconds). Check if the Lambda can reach the database (VPC configuration, security groups). Initialize database connections outside the handler. Add logging to identify the slow operation.
2 Lambda package too large to deploy
Cause: node_modules includes unnecessary packages, or native modules add bulk.
Solution: Use a bundler like esbuild to tree-shake and bundle only used code. Move large dependencies to Lambda Layers. Exclude dev dependencies. Use serverless-esbuild plugin for automatic bundling.
3 Database connections exhausted under load
Cause: Each Lambda instance creates its own database connection. With 100 concurrent Lambdas, that is 100 connections.
Solution: Use connection pooling with RDS Proxy (AWS managed). Or use Prisma Data Proxy. Set connection limits in your Prisma schema. Consider using DynamoDB for serverless-native workloads.
