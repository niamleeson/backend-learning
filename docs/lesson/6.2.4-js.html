Docker Networking and Volumes

Intuition

Docker networks are like private phone lines between containers -- they can call each other by name (DNS), but outsiders cannot listen in. Volumes are like external hard drives that you plug into containers: the data on the drive persists even if you throw away the container (computer) and plug the drive into a new one.

Docker Networking and Volumes

Docker Networking

Networking is how containers communicate with each other, the host, and the outside world. Understanding Docker networking is critical for building multi-container applications.

Bridge Networks (Default)

# Create a custom bridge network
docker network create app-network

# Run containers on the network
docker run -d --name api --network app-network myapp
docker run -d --name postgres --network app-network postgres:16

# Inside the 'api' container, connect to postgres using container name:
# postgresql://user:pass@postgres:5432/mydb
# Docker's embedded DNS resolves 'postgres' to the container's IP

# List networks
docker network ls

# Inspect a network
docker network inspect app-network

# Connect/disconnect running container
docker network connect app-network existing-container
docker network disconnect app-network existing-container

Why Custom Bridge Networks?

Default bridge:
- Containers communicate by IP only (no DNS)
- All containers on same network (no isolation)
- Must use --link (deprecated)

Custom bridge:
- Automatic DNS resolution by container name
- Network isolation between different apps
- Better security

Host Network (Linux only)

# Container shares host's network stack
docker run --network host myapp

# No port mapping needed - container uses host ports directly
# Pros: Best performance (no NAT overhead)
# Cons: Port conflicts possible, less isolation

Port Mapping

# Map host port to container port
docker run -p 3000:3000 myapp          # host:container
docker run -p 8080:3000 myapp          # Different host port
docker run -p 127.0.0.1:3000:3000 myapp  # Bind to localhost only
docker run -P myapp                      # Random host port for all EXPOSE'd ports

Docker Volumes

Volumes persist data beyond container lifecycles. Without volumes, all data inside a container is lost when the container is removed.

Volume Types

Named Volumes:         Bind Mounts:           tmpfs:
┌──────────┐          ┌──────────┐          ┌──────────┐
│ Container │          │ Container │          │ Container │
│ /data     │          │ /app/src  │          │ /tmp      │
└────┬─────┘          └────┬─────┘          └────┬─────┘
     │                      │                      │
┌────┴──────────┐    ┌────┴──────────┐    ┌────┴──────────┐
│ Docker manages│    │ Host filesystem│    │ RAM only      │
│ /var/lib/     │    │ ./src on host  │    │ (no disk)     │
│ docker/volumes│    │                │    │               │
└───────────────┘    └───────────────┘    └───────────────┘
Best for: data      Best for: dev code   Best for: secrets

Named Volumes

# Create a volume
docker volume create pgdata

# Use in a container
docker run -v pgdata:/var/lib/postgresql/data postgres:16

# List volumes
docker volume ls

# Inspect volume
docker volume inspect pgdata

# Remove volume
docker volume rm pgdata

# Remove all unused volumes
docker volume prune

Bind Mounts

# Mount host directory into container
docker run -v $(pwd)/src:/app/src myapp

# Read-only mount
docker run -v $(pwd)/config:/app/config:ro myapp

# Mount single file
docker run -v $(pwd)/nginx.conf:/etc/nginx/nginx.conf:ro nginx

Volume Patterns in Docker Compose

version: '3.8'

services:
  api:
    volumes:
      # Development: bind mount for hot reload
      - ./src:/app/src
      # Exclude node_modules from bind mount
      - /app/node_modules
      # Named volume for uploads
      - uploads:/app/uploads
      # Read-only config
      - ./config/production.json:/app/config/production.json:ro

  postgres:
    volumes:
      # Named volume for database data (persistent)
      - pgdata:/var/lib/postgresql/data
      # Init scripts (bind mount, read-only)
      - ./db/init:/docker-entrypoint-initdb.d:ro

  redis:
    volumes:
      - redis-data:/data

volumes:
  pgdata:
    driver: local
  redis-data:
    driver: local
  uploads:
    driver: local

Backup and Restore Volumes

# Backup a named volume to a tar file
docker run --rm \
  -v pgdata:/source:ro \
  -v $(pwd):/backup \
  alpine tar czf /backup/pgdata-backup.tar.gz -C /source .

# Restore from backup
docker run --rm \
  -v pgdata:/target \
  -v $(pwd):/backup \
  alpine sh -c "cd /target && tar xzf /backup/pgdata-backup.tar.gz"

Practical Networking Scenarios

Multiple Environments on One Host

# Project A
# docker-compose.yml
services:
  api:
    networks:
      - project-a-net
  postgres:
    networks:
      - project-a-net
networks:
  project-a-net:

# Project B (separate compose file, isolated network)
services:
  api:
    networks:
      - project-b-net
  postgres:
    networks:
      - project-b-net
networks:
  project-b-net:

# Projects are completely isolated from each other

Connecting to Host Services

services:
  api:
    # Docker Desktop (Mac/Windows):
    extra_hosts:
      - "host.docker.internal:host-gateway"
    environment:
      # Connect to PostgreSQL running on host machine
      DATABASE_URL: postgresql://user:pass@host.docker.internal:5432/mydb

Exposing Services Safely

services:
  postgres:
    ports:
      # BAD: accessible from any network interface
      - "5432:5432"

      # GOOD: only accessible from localhost
      - "127.0.0.1:5432:5432"

  api:
    ports:
      # Public-facing service
      - "3000:3000"

Debugging Network Issues

# Check container's network settings
docker inspect --format='{{json .NetworkSettings.Networks}}' myapp | jq

# Test connectivity between containers
docker exec api ping postgres
docker exec api nc -zv postgres 5432

# Check DNS resolution
docker exec api nslookup postgres

# View container logs for connection errors
docker logs api 2>&1 | grep -i "connect\|ECONNREFUSED\|timeout"

# List port mappings
docker port myapp

# Check which containers are on a network
docker network inspect app-network --format='{{range .Containers}}{{.Name}} {{end}}'

Common Pitfalls

  1. Using the default bridge network - Default bridge does not support DNS. Always create custom bridge networks.
  2. Hardcoding container IPs - Container IPs change on restart. Always use container names for DNS resolution.
  3. Exposing database ports to 0.0.0.0 - Bind to 127.0.0.1 to prevent external access: 127.0.0.1:5432:5432.
  4. Not understanding volume persistence - docker compose down keeps volumes. docker compose down -v deletes them. Know the difference.
  5. Bind mounting over important container paths - Mounting an empty host directory over /app/node_modules effectively deletes all modules inside the container.

Exercises

  1. Network Isolation Experiment

    Create two separate Docker networks (frontend-net and backend-net). Run an Nginx container on frontend-net, a Node.js API on both networks, and PostgreSQL on backend-net only. Verify that Nginx can reach the API but NOT the database directly.

    Hint

    Use docker network create for each network. Connect the API container to both networks. Use docker exec to test connectivity with ping or nc.

  2. Volume Backup Pipeline

    Create a script that backs up a PostgreSQL Docker volume to a timestamped tar.gz file, uploads it to a local backup directory, and prunes backups older than 7 days. Test it by creating data, backing up, destroying the volume, and restoring.

    Hint

    Use a temporary Alpine container to access the volume and create the tar. For restore, create a fresh volume and extract the tar into it.

  3. Debug a Networking Problem

    Set up a Docker Compose file with an intentional networking bug (e.g., services on different networks, wrong hostname, missing port exposure). Document the symptoms and walk through the debugging process using docker inspect, docker exec, and docker logs.

    Hint

    Common bugs to introduce: using 'localhost' instead of service name, putting services on different networks, forgetting to expose the right port.

Common Problems & Solutions

1 Container cannot resolve hostname of another container
Cause: The containers are on different networks or using the default bridge network (which does not support DNS).
Solution: Ensure both containers are on the same custom bridge network. In Docker Compose, services on the same compose file share a default network automatically. For manual docker run, use `--network mynetwork` on both containers.
2 Data lost after docker compose down
Cause: You used `docker compose down -v` which removes named volumes, or you did not declare a named volume (used anonymous volume instead).
Solution: Always declare named volumes in the `volumes:` section of docker-compose.yml. Use `docker compose down` (without -v) to preserve data. Back up volumes regularly for critical data.
3 Port already allocated error when starting container
Symptoms: Error: Bind for 0.0.0.0:3000 failed: port is already allocated
Cause: Another process (or another container) is already using the host port you are trying to map.
Solution: Find what is using the port: `lsof -i :3000` or `docker ps` to check running containers. Either stop the conflicting process or map to a different host port: `-p 3001:3000`.
