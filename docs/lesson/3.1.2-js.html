Routing: Methods, Parameters, and Query Strings

Intuition

Routing in Express is like a mail sorting facility: each letter (request) has an address (URL path) and a type (HTTP method). The sorting machine (Express router) checks each bin (route) in order until it finds a match, then delivers the letter to the right handler.

Routing: Methods, Parameters, and Query Strings

Introduction

Routing is the mechanism by which an Express application determines how to respond to a client request at a particular URL (path) and HTTP method. Every interaction your API has with the outside world begins with a route. Understanding Express routing deeply — from basic path matching to advanced parameter patterns — is essential for building clean, predictable APIs.

Basic Route Definition

An Express route combines an HTTP method, a path, and one or more handler functions:

import express, { Request, Response } from 'express';

const app = express();

// GET request to the root path
app.get('/', (req: Request, res: Response) => {
  res.json({ message: 'Hello, World!' });
});

// POST request to /users
app.post('/users', (req: Request, res: Response) => {
  res.status(201).json({ message: 'User created' });
});

// PUT request to /users/123
app.put('/users/:id', (req: Request, res: Response) => {
  res.json({ message: `User ${req.params.id} updated` });
});

// DELETE request
app.delete('/users/:id', (req: Request, res: Response) => {
  res.status(204).send();
});

HTTP Methods in Express

Express provides methods for all standard HTTP verbs:

app.get(path, handler)      // Retrieve a resource
app.post(path, handler)     // Create a resource
app.put(path, handler)      // Replace a resource entirely
app.patch(path, handler)    // Partially update a resource
app.delete(path, handler)   // Remove a resource
app.options(path, handler)  // CORS preflight
app.head(path, handler)     // Like GET but no body

// Match ALL HTTP methods
app.all('/api/*', (req, res, next) => {
  console.log(`${req.method} ${req.path}`);
  next();
});

The app.all() method is particularly useful for middleware that should run on every method for a given path.

Route Parameters

Route parameters are named URL segments that capture values. They are defined with a colon prefix and accessed via req.params:

// Single parameter
app.get('/users/:userId', (req: Request, res: Response) => {
  const { userId } = req.params;
  res.json({ userId });
});
// GET /users/42 -> { userId: "42" }

// Multiple parameters
app.get('/users/:userId/posts/:postId', (req: Request, res: Response) => {
  const { userId, postId } = req.params;
  res.json({ userId, postId });
});
// GET /users/42/posts/7 -> { userId: "42", postId: "7" }

Important: Parameters Are Always Strings

app.get('/users/:id', (req: Request, res: Response) => {
  const id = req.params.id;      // "42" (string, not number!)
  const numId = parseInt(id, 10); // 42 (number)

  if (isNaN(numId)) {
    return res.status(400).json({ error: 'Invalid user ID' });
  }

  res.json({ id: numId });
});

Typed Parameters with TypeScript

interface UserParams {
  userId: string;
}

app.get('/users/:userId', (req: Request<UserParams>, res: Response) => {
  const { userId } = req.params; // TypeScript knows this is a string
  res.json({ userId });
});

Query Strings

Query strings pass optional data in the URL after the ? character. They are accessed via req.query:

// GET /search?q=express&page=2&limit=10
app.get('/search', (req: Request, res: Response) => {
  const { q, page, limit } = req.query;
  res.json({
    query: q,         // "express"
    page: page,       // "2" (string!)
    limit: limit      // "10" (string!)
  });
});

Parsing Query Parameters Safely

interface PaginationQuery {
  page?: string;
  limit?: string;
  sort?: string;
  order?: 'asc' | 'desc';
}

app.get('/api/posts', (req: Request<{}, {}, {}, PaginationQuery>, res: Response) => {
  const page = parseInt(req.query.page || '1', 10);
  const limit = Math.min(parseInt(req.query.limit || '20', 10), 100);
  const sort = req.query.sort || 'createdAt';
  const order = req.query.order === 'asc' ? 'asc' : 'desc';

  res.json({ page, limit, sort, order });
});

Array and Nested Query Parameters

// GET /filter?tags=node&tags=express&tags=typescript
app.get('/filter', (req: Request, res: Response) => {
  const tags = req.query.tags;
  // tags could be a string (single value) or string[] (multiple values)
  const tagArray = Array.isArray(tags) ? tags : tags ? [tags] : [];
  res.json({ tags: tagArray });
});

Route Path Patterns

Express supports several path matching patterns:

String Patterns

// Exact match
app.get('/about', handler);

// Pattern with ? (optional character)
app.get('/colou?r', handler);  // matches /color and /colour

// Pattern with + (one or more)
app.get('/ab+cd', handler);    // matches /abcd, /abbcd, /abbbcd

// Pattern with * (any characters)
app.get('/ab*cd', handler);    // matches /abcd, /abXcd, /ab123cd

Regular Expression Routes

// Match any route containing "user"
app.get(/user/, (req, res) => {
  res.json({ matched: req.path });
});

// Match routes ending in .json
app.get(/\.json$/, (req, res) => {
  res.json({ format: 'json' });
});

Parameter Constraints with Regex

// Only match numeric IDs
app.get('/users/:id(\\d+)', (req: Request, res: Response) => {
  res.json({ id: parseInt(req.params.id, 10) });
});
// GET /users/42   -> matches
// GET /users/abc  -> does NOT match (falls through to 404)

Route Order Matters

Express matches routes in the order they are defined. The first match wins:

// WRONG ORDER: /users/profile will never be reached
app.get('/users/:id', (req, res) => {
  res.json({ user: req.params.id });
});
app.get('/users/profile', (req, res) => {
  res.json({ profile: 'me' });  // Never reached!
});

// CORRECT ORDER: specific routes before parameterized routes
app.get('/users/profile', (req, res) => {
  res.json({ profile: 'me' });
});
app.get('/users/:id', (req, res) => {
  res.json({ user: req.params.id });
});

This is a very common source of bugs. Always place specific (static) routes before parameterized routes.

Route Matching Order:

Request: GET /users/profile

  /users/profile  --> MATCH (if defined first)
  /users/:id      --> MATCH (with id = "profile" -- probably wrong!)

Rule: Specific routes BEFORE parameterized routes

Multiple Handlers (Middleware Chain)

A single route can have multiple handler functions that execute in sequence:

const validateId = (req: Request, res: Response, next: Function) => {
  const id = parseInt(req.params.id, 10);
  if (isNaN(id) || id < 1) {
    return res.status(400).json({ error: 'Invalid ID' });
  }
  next();
};

const getUser = (req: Request, res: Response) => {
  res.json({ user: { id: req.params.id, name: 'Alice' } });
};

// Chain multiple handlers
app.get('/users/:id', validateId, getUser);

// Or pass an array
app.get('/users/:id', [validateId, getUser]);

app.route() for Chaining Methods

When multiple HTTP methods share the same path, use app.route():

app.route('/api/articles')
  .get((req, res) => {
    res.json({ articles: [] });
  })
  .post((req, res) => {
    res.status(201).json({ article: req.body });
  });

app.route('/api/articles/:id')
  .get((req, res) => {
    res.json({ article: { id: req.params.id } });
  })
  .put((req, res) => {
    res.json({ updated: req.params.id });
  })
  .delete((req, res) => {
    res.status(204).send();
  });

Real-World Routing Example

import { Router, Request, Response } from 'express';

const router = Router();

// GET /api/products?category=electronics&minPrice=100&maxPrice=500&page=1&limit=20
router.get('/products', async (req: Request, res: Response) => {
  const {
    category,
    minPrice,
    maxPrice,
    page = '1',
    limit = '20',
    search
  } = req.query as Record<string, string>;

  const filters = {
    ...(category && { category }),
    ...(minPrice && { minPrice: parseFloat(minPrice) }),
    ...(maxPrice && { maxPrice: parseFloat(maxPrice) }),
    ...(search && { search }),
  };

  const pagination = {
    page: Math.max(1, parseInt(page, 10)),
    limit: Math.min(100, Math.max(1, parseInt(limit, 10))),
  };

  // In a real app, pass filters and pagination to a service
  res.json({ filters, pagination, products: [] });
});

// GET /api/products/:id
router.get('/products/:id(\\d+)', async (req: Request, res: Response) => {
  const id = parseInt(req.params.id, 10);
  res.json({ product: { id, name: 'Sample Product' } });
});

// POST /api/products
router.post('/products', async (req: Request, res: Response) => {
  const { name, price, category } = req.body;
  if (!name || !price) {
    return res.status(400).json({ error: 'Name and price are required' });
  }
  res.status(201).json({ product: { id: 1, name, price, category } });
});

export { router as productRoutes };

Common Pitfalls

Forgetting to Return After Sending a Response

// WRONG: sends two responses, crashes with "headers already sent"
app.get('/users/:id', (req, res) => {
  if (!req.params.id) {
    res.status(400).json({ error: 'Missing ID' });
    // Execution continues!
  }
  res.json({ user: {} }); // Error: headers already sent
});

// CORRECT: return after sending response
app.get('/users/:id', (req, res) => {
  if (!req.params.id) {
    return res.status(400).json({ error: 'Missing ID' });
  }
  res.json({ user: {} });
});

Not Handling Missing Query Parameters

// WRONG: crashes if page is not provided
const page = parseInt(req.query.page as string, 10);
// NaN if page is undefined

// CORRECT: provide defaults
const page = parseInt((req.query.page as string) || '1', 10);

Summary

Express routing maps HTTP methods and URL paths to handler functions. Route parameters capture dynamic URL segments as strings, query strings provide optional filtering and pagination data, and route order determines which handler responds first. Master these fundamentals and you have the backbone for any Express API.

Exercises

  1. Build a RESTful Resource

    Create a complete set of routes for a 'books' resource: GET /books (list with pagination via query params), GET /books/:id (single book), POST /books (create), PUT /books/:id (update), DELETE /books/:id (delete). Store books in an in-memory array. Support ?genre= and ?author= query filters on the list endpoint.

    Hint

    Use app.route('/books') to chain GET and POST, and app.route('/books/:id') to chain GET, PUT, and DELETE. Filter the array using Array.filter() based on query parameters.

  2. Route Parameter Validation

    Create a middleware function that validates route parameters are positive integers. Apply it to all routes that use :id parameters. If the ID is invalid, respond with a 400 error. Test with /users/abc, /users/-1, and /users/42.

    Hint

    Create a reusable validateId middleware that calls parseInt on req.params.id and checks isNaN and > 0. Use next() to continue or res.status(400) to reject.

  3. Query String Parser Utility

    Write a utility function parsePagination(query) that takes req.query and returns a validated { page, limit, sort, order } object with defaults and bounds checking. Page minimum is 1, limit range is 1-100, sort defaults to 'createdAt', order must be 'asc' or 'desc'.

    Hint

    Use Math.max and Math.min for bounds. Default values with || or nullish coalescing ??. Return an object with the parsed values.

  4. Nested Resource Routes

    Implement nested routes for a blog: GET /users/:userId/posts (all posts by user), POST /users/:userId/posts (create post for user), GET /users/:userId/posts/:postId (specific post). Ensure both userId and postId are validated as numbers.

    Hint

    You can access multiple params with req.params.userId and req.params.postId. Create a validation middleware that checks all numeric params in the route.

Common Problems & Solutions

1 A specific route like /users/me is being caught by /users/:id
Cause: Route order matters in Express. If /users/:id is defined before /users/me, the parameterized route matches first with id='me'.
Solution: Always define specific (static) routes before parameterized routes. Move app.get('/users/me', ...) above app.get('/users/:id', ...).
2 req.params.id is a string but you need a number
Cause: Express route parameters are always strings. There is no automatic type coercion.
Solution: Explicitly parse with parseInt(req.params.id, 10) or Number(req.params.id). Always validate the result with isNaN() before using it.
3 ERR_HTTP_HEADERS_SENT: Cannot set headers after they are sent
Cause: The route handler sends a response (e.g., res.json()) but does not return, so execution continues and tries to send another response.
Solution: Always use 'return' before res.json(), res.send(), etc. when there is code that follows: return res.status(400).json({ error: '...' });
