PM2: Process Management, Clustering, and Zero-Downtime Reload

Intuition

PM2 is like a restaurant manager who oversees multiple chefs (Node.js processes). If one chef takes a break (crashes), the manager immediately brings in a replacement. During a menu change (code update), the manager swaps out chefs one at a time so customers (requests) are always being served. The manager also keeps track of each chef's performance and writes everything in a log book.

PM2: Process Management, Clustering, and Zero-Downtime Reload

Why PM2?

Node.js runs on a single thread by default. If your server has 4 CPU cores, a single Node.js process uses only one of them. PM2 (Process Manager 2) solves this by managing multiple Node.js processes, providing clustering, zero-downtime reloads, log management, and monitoring.

PM2 Architecture

                 Port 3000

Getting Started

# Install PM2 globally
npm install -g pm2

# Start an application
pm2 start dist/server.js --name myapp

# Start with cluster mode (use all CPU cores)
pm2 start dist/server.js --name myapp -i max

# Start with specific number of instances
pm2 start dist/server.js --name myapp -i 4

# Common management commands
pm2 list                    # List all processes
pm2 status                  # Same as list
pm2 show myapp              # Detailed info about a process
pm2 logs                    # View all logs
pm2 logs myapp              # View specific app logs
pm2 logs myapp --lines 100  # Last 100 lines
pm2 monit                   # Real-time monitoring dashboard
pm2 restart myapp           # Restart (with downtime)
pm2 reload myapp            # Graceful reload (zero downtime)
pm2 stop myapp              # Stop
pm2 delete myapp            # Remove from PM2 list

# Scale instances
pm2 scale myapp 6           # Scale to 6 instances
pm2 scale myapp +2          # Add 2 more instances

Ecosystem Configuration File

// ecosystem.config.js
module.exports = {
  apps: [
    {
      name: 'myapp-api',
      script: 'dist/server.js',
      instances: 'max',           // Use all CPU cores
      exec_mode: 'cluster',       // Enable cluster mode

      // Environment
      env: {
        NODE_ENV: 'development',
        PORT: 3000,
      },
      env_staging: {
        NODE_ENV: 'staging',
        PORT: 3000,
      },
      env_production: {
        NODE_ENV: 'production',
        PORT: 3000,
      },

      // Restart behavior
      max_memory_restart: '512M',  // Restart if memory exceeds 512MB
      restart_delay: 3000,         // Wait 3s between restarts
      max_restarts: 10,            // Max restarts before stopping
      min_uptime: 5000,            // Min uptime to consider "started"

      // Graceful shutdown
      kill_timeout: 5000,          // Time to wait for graceful shutdown
      listen_timeout: 10000,       // Time to wait for app to listen
      shutdown_with_message: true, // Send shutdown message

      // Logging
      log_file: '/var/log/myapp/combined.log',
      out_file: '/var/log/myapp/out.log',
      error_file: '/var/log/myapp/error.log',
      log_date_format: 'YYYY-MM-DD HH:mm:ss Z',
      merge_logs: true,            // Merge cluster logs into one file

      // Advanced
      node_args: '--max-old-space-size=512',
      watch: false,                // Don't watch in production
      autorestart: true,
      cron_restart: '0 3 * * *',   // Restart daily at 3 AM
    },

    // Worker process (not cluster mode)
    {
      name: 'myapp-worker',
      script: 'dist/workers/queue.js',
      instances: 1,
      exec_mode: 'fork',
      env_production: {
        NODE_ENV: 'production',
      },
      max_memory_restart: '256M',
    },
  ],
};

Using the Ecosystem File

# Start all apps
pm2 start ecosystem.config.js

# Start with specific environment
pm2 start ecosystem.config.js --env production

# Reload (zero downtime)
pm2 reload ecosystem.config.js --env production

# Restart specific app
pm2 restart myapp-api

Zero-Downtime Reload

Standard Restart (with downtime):
  Stop all ──── gap (no service) ──── Start all

Graceful Reload (zero downtime):
  Worker 0: serving ─── stop ─── start (new) ─── serving
  Worker 1: serving ─────────── serving ── stop ─── start (new) ─── serving
  Worker 2: serving ─────────── serving ────────── serving ── stop ─── start

  At least one worker is always serving!

Implementing Graceful Shutdown

// server.ts
import express from 'express';

const app = express();

// ... routes ...

const server = app.listen(process.env.PORT || 3000, () => {
  console.log(`Worker ${process.pid} listening`);

  // Tell PM2 the app is ready
  if (process.send) {
    process.send('ready');
  }
});

// Handle graceful shutdown
process.on('SIGINT', gracefulShutdown);
process.on('SIGTERM', gracefulShutdown);

function gracefulShutdown() {
  console.log(`Worker ${process.pid}: Received shutdown signal`);

  // Stop accepting new connections
  server.close(() => {
    console.log(`Worker ${process.pid}: Closed all connections`);

    // Close database, redis, etc.
    // ...

    process.exit(0);
  });

  // Force shutdown after timeout
  setTimeout(() => {
    console.error(`Worker ${process.pid}: Forced shutdown`);
    process.exit(1);
  }, 10000);
}

// ecosystem.config.js - enable wait_ready
{
  wait_ready: true,            // Wait for process.send('ready')
  listen_timeout: 10000,       // Max time to wait for ready signal
  kill_timeout: 10000,         // Time for graceful shutdown
  shutdown_with_message: true,
}

Custom Metrics

// In your application
import pmx from '@pm2/io';

const reqCounter = pmx.counter({
  name: 'Requests/min',
  type: 'counter',
});

const responseTime = pmx.histogram({
  name: 'Response Time',
  measurement: 'mean',
});

app.use((req, res, next) => {
  reqCounter.inc();
  const start = Date.now();

  res.on('finish', () => {
    responseTime.update(Date.now() - start);
  });

  next();
});

Common Pitfalls

  1. Using pm2 restart instead of pm2 reload - restart kills all processes at once (downtime). reload does zero-downtime rolling restart.
  2. Not using pm2 save after changes - If the server reboots, PM2 restores the last saved state. Always save after adding/removing apps.
  3. Running too many instances - i max on a 2-core machine creates 2 workers. More instances than cores leads to context switching overhead.
  4. Not implementing graceful shutdown - Without SIGTERM handling, connections are dropped during reload.
  5. Forgetting pm2 startup - Without this, PM2 and your apps do not start after a server reboot.

Exercises

  1. Cluster Mode Setup

    Create an Express app that logs the worker PID for each request. Deploy it with PM2 in cluster mode with 4 instances. Send multiple requests and verify that different workers handle them (round-robin). Then perform a zero-downtime reload and confirm no requests are dropped.

    Hint

    Use process.pid in the response. Use 'ab' (Apache Bench) or 'autocannon' to send concurrent requests during reload. Check that no 502 errors occur.

  2. Ecosystem Configuration

    Create a PM2 ecosystem.config.js for a production application with: API server (cluster mode, 4 instances), background worker (fork mode, 1 instance), and a scheduled job. Configure memory limits, log rotation, graceful shutdown, and environment-specific variables.

    Hint

    Use exec_mode: 'cluster' for the API and 'fork' for the worker. Set max_memory_restart and cron_restart. Configure wait_ready and kill_timeout for graceful shutdown.

  3. Monitoring Dashboard

    Set up PM2 monitoring with custom metrics: request count, average response time, and error rate. Install pm2-logrotate and configure it to keep 7 days of logs with max 50MB per file.

    Hint

    Use @pm2/io for custom metrics. Install pm2-logrotate with 'pm2 install pm2-logrotate'. Use 'pm2 monit' for the terminal dashboard.

Common Problems & Solutions

1 PM2 cluster mode not distributing load evenly
Cause: Node.js cluster module uses round-robin by default on Linux, but OS-level scheduling on other platforms. Some requests may be heavier than others.
Solution: On Linux, round-robin is the default and works well. Ensure all workers are healthy with `pm2 list`. If one worker consistently has higher CPU, check for memory leaks or blocking operations. Use `pm2 monit` to compare workers.
2 PM2 reload causes brief 502 errors
Cause: The application is not signaling 'ready' to PM2, so PM2 starts routing traffic before the app is fully initialized (database connected, etc.).
Solution: Add `process.send('ready')` after your server starts listening and all connections are established. Set `wait_ready: true` and `listen_timeout: 10000` in ecosystem config. Implement proper SIGTERM handling for graceful shutdown.
3 PM2 processes not starting after server reboot
Cause: PM2 startup script was not generated or the process list was not saved.
Solution: Run `pm2 startup` and execute the printed sudo command. Then run `pm2 save` to persist the current process list. Verify with `pm2 list` after a reboot (or `sudo reboot` to test).
