GitHub Actions: Workflows, Jobs, and Steps

Intuition

GitHub Actions is like a factory assembly line that starts automatically when you push code. Each job is a separate workstation, and each step is a specific task at that station. The matrix strategy is like running parallel assembly lines -- one for each configuration you need to test.

GitHub Actions: Workflows, Jobs, and Steps

GitHub Actions automates your build, test, and deployment pipeline. Every push, PR, or schedule can trigger workflows.

Workflow Structure

# .github/workflows/ci.yml
name: CI Pipeline
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.11', '3.12']
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: testdb
        ports: ['5432:5432']
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-retries 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}
      - run: pip install -r requirements.txt
      - run: pytest --tb=short -q
        env:
          DATABASE_URL: postgresql://postgres:testpass@localhost:5432/testdb

Key Concepts

  • Workflow: YAML file in .github/workflows/ triggered by events
  • Job: Steps running on the same runner; jobs run in parallel by default
  • Step: Shell command (run:) or reusable action (uses:)
  • Matrix: Run the same job across multiple configurations

Common Mistakes

  • Not caching dependencies, causing slow CI builds
  • Hardcoding secrets instead of using ${{ secrets.MY_SECRET }}

Key Takeaways

  • Use services to spin up databases for integration tests
  • Cache pip dependencies with actions/cache to cut build times
  • Use matrix strategy to test across multiple Python versions

Exercises

  1. Build a CI Workflow

    Create a GitHub Actions workflow that runs on PRs to main, sets up Python 3.12, installs dependencies, runs linting with ruff, type checking with mypy, and tests with pytest. Include pip caching.

    Hint

    Use actions/checkout@v4, actions/setup-python@v5, actions/cache@v4, and separate run steps for each tool.

  2. Matrix Testing with Services

    Extend the CI workflow to test against Python 3.11 and 3.12, with both PostgreSQL 15 and 16. Use the matrix strategy to cover all combinations.

    Hint

    Use strategy.matrix with python-version and postgres-version arrays, and reference them in the services section.

  3. Reusable Workflow

    Extract the test job into a reusable workflow that can be called from other repositories. Accept Python version and test command as inputs.

    Hint

    Use workflow_call trigger with inputs and secrets.

Common Problems & Solutions

1 Service container isn't ready when tests start
Symptoms: ['psycopg2.OperationalError: could not connect to server', 'Connection refused']
Cause: The service container has started but the service inside (e.g., PostgreSQL) isn't ready to accept connections yet.
Solution: Add health check options to the service definition: `--health-cmd pg_isready --health-interval 10s --health-retries 5`.
2 Cache not being restored
Symptoms: ['Cache not found for input keys', 'pip install takes 2+ minutes every run']
Cause: The cache key doesn't match any existing cache. Often caused by changing the key format or requirements file.
Solution: Use `hashFiles('requirements.txt')` in the cache key. Add a restore-keys fallback: `restore-keys: ${{ runner.os }}-pip-`.
