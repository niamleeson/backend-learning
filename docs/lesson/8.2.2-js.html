Cache Invalidation and TTL Strategies

Intuition

Cache invalidation is like keeping a phone book up to date. TTL is reprinting it every year (simple but outdated between prints). Event-based is updating individual entries whenever someone moves (accurate but labor-intensive). The best approach is to update critical entries immediately and reprint periodically as a safety net.

Cache Invalidation and TTL Strategies

Introduction

Phil Karlton famously said: "There are only two hard things in Computer Science: cache invalidation and naming things." Cache invalidation — deciding when to remove or update cached data — is genuinely one of the hardest problems in distributed systems. Get it wrong and users see stale data; invalidate too aggressively and you lose all caching benefits.

The Fundamental Problem

Timeline of a stale cache bug:

T0: User A reads product (price: $10) ──> Cached in Redis
T1: Admin updates price to $15 ──> DB updated
T2: User B reads product ──> Gets $10 from cache (STALE!)
T3: Cache expires (TTL) ──> Next read gets $15

The window between T1 and T3 is the staleness window. Your invalidation strategy determines how long this window is.

TTL (Time-to-Live) Strategies

TTL is the simplest invalidation strategy: cached data expires after a set duration.

Fixed TTL

import Redis from 'ioredis';
const redis = new Redis();

// Fixed TTL for different data types
const TTL = {
  USER_PROFILE: 3600,      // 1 hour - changes infrequently
  PRODUCT_LISTING: 300,     // 5 minutes - moderate changes
  SEARCH_RESULTS: 60,       // 1 minute - changes often
  REAL_TIME_SCORE: 10,      // 10 seconds - near real-time
  CONFIG: 86400,            // 24 hours - rarely changes
};

async function cacheWithTTL(key, ttl, fetchFn) {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const data = await fetchFn();
  await redis.setex(key, ttl, JSON.stringify(data));
  return data;
}

// Usage
const user = await cacheWithTTL(
  `user:${userId}`,
  TTL.USER_PROFILE,
  () => prisma.user.findUnique({ where: { id: userId } })
);

Jittered TTL (Prevents Thundering Herd)

When many keys expire simultaneously, all requests hit the database at once.

function jitteredTTL(baseTTL, jitterPercent = 0.1) {
  const jitter = baseTTL * jitterPercent;
  return Math.floor(baseTTL + (Math.random() * 2 - 1) * jitter);
}

// Instead of all keys expiring at exactly 300s:
// Key 1 expires at 278s, Key 2 at 312s, Key 3 at 295s, etc.
await redis.setex(key, jitteredTTL(300), JSON.stringify(data));

Sliding TTL

Reset TTL on every read to keep frequently accessed data warm:

async function getWithSlidingTTL(key, ttl, fetchFn) {
  const cached = await redis.get(key);
  if (cached) {
    // Reset TTL on hit (sliding window)
    await redis.expire(key, ttl);
    return JSON.parse(cached);
  }

  const data = await fetchFn();
  await redis.setex(key, ttl, JSON.stringify(data));
  return data;
}

Stale-While-Revalidate

Serve stale data immediately while refreshing in the background:

async function staleWhileRevalidate(key, ttl, staleTTL, fetchFn) {
  const pipeline = redis.pipeline();
  pipeline.get(key);
  pipeline.ttl(key);
  const [[, cached], [, remainingTTL]] = await pipeline.exec();

  if (cached) {
    const data = JSON.parse(cached);

    // If within stale window, refresh in background
    if (remainingTTL <= staleTTL && remainingTTL > 0) {
      // Don't await — fire and forget
      refreshCache(key, ttl, fetchFn).catch(console.error);
    }

    return data; // Return possibly stale data immediately
  }

  // Cache miss — must wait for fresh data
  return refreshCache(key, ttl, fetchFn);
}

async function refreshCache(key, ttl, fetchFn) {
  const data = await fetchFn();
  await redis.setex(key, ttl, JSON.stringify(data));
  return data;
}

// Usage: 5 min cache, start refreshing in last 60 seconds
const products = await staleWhileRevalidate(
  'products:featured',
  300,   // 5 min TTL
  60,    // refresh when < 60s remaining
  () => prisma.product.findMany({ where: { featured: true } })
);

Event-Based Invalidation

Instead of waiting for TTL, invalidate immediately when data changes.

Direct Invalidation

async function updateProduct(productId, data) {
  const product = await prisma.product.update({
    where: { id: productId },
    data,
  });

  // Invalidate all related caches
  const keysToDelete = [
    `product:${productId}`,
    `product:${productId}:details`,
    'products:featured',
    'products:latest',
  ];

  // Find pattern-based keys
  const searchKeys = await redis.keys(`products:category:${product.categoryId}:*`);
  keysToDelete.push(...searchKeys);

  if (keysToDelete.length > 0) {
    await redis.del(...keysToDelete);
  }

  return product;
}

Pub/Sub Invalidation (Multi-Instance)

When running multiple app instances, one instance's invalidation must notify others:

import Redis from 'ioredis';

const publisher = new Redis();
const subscriber = new Redis();

// Local in-memory cache (L1 cache)
const localCache = new Map();

// Subscribe to invalidation events
subscriber.subscribe('cache-invalidation');
subscriber.on('message', (channel, message) => {
  const { pattern } = JSON.parse(message);
  // Clear matching keys from local cache
  for (const key of localCache.keys()) {
    if (key.startsWith(pattern)) {
      localCache.delete(key);
    }
  }
});

// When data changes, publish invalidation
async function invalidateCache(pattern) {
  // Clear Redis cache
  const keys = await redis.keys(`${pattern}*`);
  if (keys.length) await redis.del(...keys);

  // Notify all instances to clear local caches
  await publisher.publish('cache-invalidation', JSON.stringify({ pattern }));
}

// Two-tier cache: local (L1) + Redis (L2)
async function getWithL1L2(key, ttl, fetchFn) {
  // L1: Local memory check
  if (localCache.has(key)) {
    const { data, expiry } = localCache.get(key);
    if (Date.now() < expiry) return data;
    localCache.delete(key);
  }

  // L2: Redis check
  const cached = await redis.get(key);
  if (cached) {
    const data = JSON.parse(cached);
    localCache.set(key, { data, expiry: Date.now() + 10000 }); // 10s local
    return data;
  }

  // Cache miss
  const data = await fetchFn();
  await redis.setex(key, ttl, JSON.stringify(data));
  localCache.set(key, { data, expiry: Date.now() + 10000 });
  return data;
}

Database Trigger-Based Invalidation

Use PostgreSQL LISTEN/NOTIFY for real-time invalidation:

import pg from 'pg';

const client = new pg.Client(process.env.DATABASE_URL);
await client.connect();

// Listen for changes
await client.query('LISTEN cache_invalidation');

client.on('notification', async (msg) => {
  const { table, id, operation } = JSON.parse(msg.payload);
  console.log(`${operation} on ${table}:${id}`);

  // Invalidate related cache keys
  await redis.del(`${table}:${id}`);
});

SQL trigger:

CREATE OR REPLACE FUNCTION notify_cache_invalidation()
RETURNS trigger AS $$
BEGIN
  PERFORM pg_notify('cache_invalidation',
    json_build_object(
      'table', TG_TABLE_NAME,
      'id', COALESCE(NEW.id, OLD.id),
      'operation', TG_OP
    )::text
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER products_cache_trigger
AFTER INSERT OR UPDATE OR DELETE ON products
FOR EACH ROW EXECUTE FUNCTION notify_cache_invalidation();

Tag-Based Invalidation

Group related cache entries with tags so you can invalidate them together:

class TaggedCache {
  constructor(redis) {
    this.redis = redis;
  }

  async set(key, data, ttl, tags = []) {
    const pipeline = this.redis.pipeline();

    // Store the data
    pipeline.setex(key, ttl, JSON.stringify(data));

    // Associate key with each tag
    for (const tag of tags) {
      pipeline.sadd(`tag:${tag}`, key);
      pipeline.expire(`tag:${tag}`, ttl + 60); // Tag lives slightly longer
    }

    await pipeline.exec();
  }

  async get(key) {
    const cached = await this.redis.get(key);
    return cached ? JSON.parse(cached) : null;
  }

  async invalidateByTag(tag) {
    const keys = await this.redis.smembers(`tag:${tag}`);
    if (keys.length > 0) {
      await this.redis.del(...keys, `tag:${tag}`);
    }
    return keys.length;
  }
}

const cache = new TaggedCache(redis);

// Cache a product with multiple tags
await cache.set(
  `product:${product.id}`,
  product,
  300,
  ['products', `category:${product.categoryId}`, 'featured']
);

// Invalidate all products in a category
await cache.invalidateByTag(`category:${categoryId}`);
// Invalidate all featured products
await cache.invalidateByTag('featured');

Version-Based Invalidation

Use a version number to invalidate all cached data at once:

async function versionedCache(entity, id, fetchFn, ttl = 300) {
  const version = await redis.get(`version:${entity}`) || '1';
  const key = `${entity}:v${version}:${id}`;

  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const data = await fetchFn();
  await redis.setex(key, ttl, JSON.stringify(data));
  return data;
}

// Invalidate ALL products by incrementing version
async function invalidateAllProducts() {
  await redis.incr('version:products');
  // Old keys with old version naturally expire via TTL
}

Choosing an Invalidation Strategy

┌──────────────────────┬──────────────────────────────────────┐
│ Strategy             │ Best For                             │
├──────────────────────┼──────────────────────────────────────┤
│ TTL only             │ Data where staleness is acceptable   │
│ TTL + Event-based    │ Most applications (recommended)      │
│ Pub/Sub invalidation │ Multi-instance deployments           │
│ Tag-based            │ Complex relationships between data   │
│ Version-based        │ Bulk invalidation (deploys, imports) │
│ DB triggers          │ When writes bypass the app layer     │
└──────────────────────┴──────────────────────────────────────┘

Common Pitfalls

  1. Using KEYS command in production: redis.keys('pattern*') blocks Redis. Use SCAN instead for pattern matching.
  2. Race condition on invalidation: Write completes, cache invalidated, but a concurrent read re-caches stale data. Use short TTLs as a safety net.
  3. Cascading invalidation: One write invalidates hundreds of cache keys, causing a stampede. Use tag-based invalidation with staggered refilling.
  4. No invalidation strategy at all: Relying solely on TTL means data can be stale for the full TTL duration.

Summary

Cache invalidation requires balancing freshness against performance. TTL provides a safety net but allows staleness. Event-based invalidation provides freshness but adds complexity. The best approach combines both: use TTL as a safety net and event-based invalidation for immediate freshness. For multi-instance deployments, add pub/sub to propagate invalidations. Tag-based and version-based strategies simplify invalidation of related data.

Common Problems & Solutions

1 Cache stampede after invalidation
Cause: A popular key is invalidated and hundreds of concurrent requests all hit the database simultaneously to rebuild the cache.
Solution: Implement a distributed lock so only one request rebuilds the cache. Others wait and read the fresh cache. Use stale-while-revalidate to serve old data during refresh.

async function lockAndFetch(key, ttl, fetchFn) {\n const lockKey = `lock:\${key}`;\n const locked = await redis.set(lockKey, '1', 'NX', 'EX', 5);\n if (!locked) {\n await new Promise(r => setTimeout(r, 100));\n return redis.get(key).then(JSON.parse);\n }\n const data = await fetchFn();\n await redis.setex(key, ttl, JSON.stringify(data));\n await redis.del(lockKey);\n return data;\n}

2 Redis KEYS command causing latency spikes
Cause: Using `redis.keys('pattern*')` in production blocks the single-threaded Redis server while it scans all keys.
Solution: Replace KEYS with SCAN, which iterates incrementally without blocking. Or use tag-based invalidation to avoid pattern matching entirely.

async function scanKeys(pattern) {\n const keys = [];\n let cursor = '0';\n do {\n const [next, batch] = await redis.scan(cursor, 'MATCH', pattern, 'COUNT', 100);\n cursor = next;\n keys.push(...batch);\n } while (cursor !== '0');\n return keys;\n}
