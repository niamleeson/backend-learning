Distributed Caching with Redis and Memcached

Intuition

Distributed caching is like having multiple refrigerators in different rooms of a restaurant. Redis Sentinel is having a backup fridge that automatically takes over when the main one breaks. Redis Cluster is having specialized fridges — one for meats, one for vegetables, one for dairy — so each holds less but together they store everything.

Distributed Caching with Redis and Memcached

Introduction

When a single cache server is not enough — because of memory limits, throughput requirements, or reliability needs — you need distributed caching. This lesson covers Redis Cluster, Redis Sentinel, and Memcached, comparing their architectures and showing how to use them from Node.js.

Redis Sentinel (High Availability)

Sentinel provides automatic failover for standalone Redis:

┌────────────┐  ┌────────────┐  ┌────────────┐
│ Sentinel 1 │  │ Sentinel 2 │  │ Sentinel 3 │
└─────┬──────┘  └──────┬─────┘  └──────┬─────┘
      │                │               │
      │    monitors    │    monitors   │
      ▼                ▼               ▼
┌──────────┐      ┌──────────┐
│  Master  │─────>│  Replica │
│  (R/W)   │ repl │  (Read)  │
└──────────┘      └──────────┘

If Master dies:
  Sentinels elect new master
  Replica promoted automatically
  Clients redirected

ioredis with Sentinel

import Redis from 'ioredis';

const redis = new Redis({
  sentinels: [
    { host: 'sentinel-1', port: 26379 },
    { host: 'sentinel-2', port: 26379 },
    { host: 'sentinel-3', port: 26379 },
  ],
  name: 'mymaster', // Sentinel master name
  role: 'master',   // Connect to master
  sentinelRetryStrategy: (times) => Math.min(times * 100, 3000),
});

// Read from replica for read-heavy workloads
const readReplica = new Redis({
  sentinels: [
    { host: 'sentinel-1', port: 26379 },
    { host: 'sentinel-2', port: 26379 },
    { host: 'sentinel-3', port: 26379 },
  ],
  name: 'mymaster',
  role: 'slave',    // Connect to replica
  preferredSlaves: [
    { ip: 'replica-1', port: 6379, prio: 1 },
  ],
});

// Writes go to master
await redis.set('key', 'value');

// Reads can go to replica
const value = await readReplica.get('key');

Redis Cluster (Horizontal Scaling)

Redis Cluster shards data across multiple masters using 16384 hash slots:

Key "user:123" ──> CRC16("user:123") % 16384 = 5649 ──> Master 2

┌─────────────────────────────────────────────────────┐
│ Slot Range    │ Master  │ Replicas                   │
├───────────────┼─────────┼────────────────────────────┤
│ 0 - 5460     │ Node 1  │ Node 4                     │
│ 5461 - 10922 │ Node 2  │ Node 5                     │
│ 10923 - 16383│ Node 3  │ Node 6                     │
└─────────────────────────────────────────────────────┘

ioredis Cluster Client

import Redis from 'ioredis';

const cluster = new Redis.Cluster([
  { host: 'redis-node-1', port: 6379 },
  { host: 'redis-node-2', port: 6379 },
  { host: 'redis-node-3', port: 6379 },
], {
  redisOptions: {
    password: process.env.REDIS_PASSWORD,
  },
  scaleReads: 'slave',         // Read from replicas
  maxRedirections: 16,          // Handle MOVED/ASK redirections
  retryDelayOnMoved: 100,
  retryDelayOnFailover: 200,
  clusterRetryStrategy: (times) => Math.min(times * 100, 3000),

  // Natural reconnect on node failures
  enableOfflineQueue: true,
  lazyConnect: false,
});

cluster.on('error', (err) => console.error('Cluster error:', err));
cluster.on('+node', (node) => console.log('Node added:', node.options.host));
cluster.on('-node', (node) => console.log('Node removed:', node.options.host));

// Basic operations work the same as standalone
await cluster.set('user:123', JSON.stringify({ name: 'Alice' }));
const user = JSON.parse(await cluster.get('user:123'));

Hash Tags for Multi-Key Operations

Redis Cluster only supports multi-key operations when all keys are on the same slot. Use hash tags:

// These keys may be on different slots - CROSSSLOT error
await cluster.mget('user:1', 'user:2', 'user:3'); // ERROR!

// Hash tags force keys to the same slot
// {user}.1, {user}.2, {user}.3 all hash based on "user"
await cluster.mset(
  '{user}.1', JSON.stringify({ name: 'Alice' }),
  '{user}.2', JSON.stringify({ name: 'Bob' }),
  '{user}.3', JSON.stringify({ name: 'Charlie' })
);
await cluster.mget('{user}.1', '{user}.2', '{user}.3'); // Works!

// Pipeline with hash tags
const pipeline = cluster.pipeline();
pipeline.get('{order:42}.items');
pipeline.get('{order:42}.total');
pipeline.get('{order:42}.status');
const results = await pipeline.exec();

// Transactions require same slot
const tx = cluster.multi();
tx.decr('{inventory}.product:1');
tx.incr('{inventory}.product:1:sold');
await tx.exec();

Memcached

Memcached is a simpler, purely in-memory distributed cache. It has no persistence, no data structures, just key-value pairs.

┌──────────┐  ┌──────────┐  ┌──────────┐
│Memcached │  │Memcached │  │Memcached │
│ Node 1   │  │ Node 2   │  │ Node 3   │
└──────────┘  └──────────┘  └──────────┘
     ▲              ▲              ▲
     │              │              │
     └──────────────┼──────────────┘
                    │
          Client-side hashing
          (consistent hashing ring)

Key difference: Memcached nodes are independent. The client decides which node to talk to using consistent hashing.

Memcached with Node.js

import Memcached from 'memcached';

const memcached = new Memcached([
  'memcached-1:11211',
  'memcached-2:11211',
  'memcached-3:11211',
], {
  retries: 2,
  timeout: 1000,
  poolSize: 25,
});

// Promisify for async/await
function memGet(key) {
  return new Promise((resolve, reject) => {
    memcached.get(key, (err, data) => {
      if (err) reject(err);
      else resolve(data);
    });
  });
}

function memSet(key, value, ttl) {
  return new Promise((resolve, reject) => {
    memcached.set(key, value, ttl, (err) => {
      if (err) reject(err);
      else resolve();
    });
  });
}

// Usage
await memSet('user:123', JSON.stringify({ name: 'Alice' }), 3600);
const user = JSON.parse(await memGet('user:123'));

Multi-Tier Caching Architecture

Production systems often use multiple cache layers:

┌──────────┐    ┌───────────┐    ┌───────────┐    ┌──────────┐
│ L1 Cache │───>│ L2 Cache  │───>│ L3 Cache  │───>│ Database │
│ In-Memory│    │ Redis     │    │ Redis     │    │ Postgres │
│ (per     │    │ (local    │    │ Cluster   │    │          │
│  process)│    │  region)  │    │ (global)  │    │          │
│ ~1ms     │    │ ~2-5ms    │    │ ~10-50ms  │    │ ~50-200ms│
└──────────┘    └───────────┘    └───────────┘    └──────────┘

class MultiTierCache {
  constructor({ l1MaxSize = 1000, redis, prisma }) {
    this.l1 = new Map(); // In-memory LRU
    this.l1MaxSize = l1MaxSize;
    this.redis = redis;
    this.prisma = prisma;
  }

  async get(key, fetchFn, { l1TTL = 10000, l2TTL = 300 } = {}) {
    // L1: In-memory
    const l1Entry = this.l1.get(key);
    if (l1Entry && Date.now() < l1Entry.expiry) {
      return l1Entry.data;
    }

    // L2: Redis
    const l2Data = await this.redis.get(key);
    if (l2Data) {
      const data = JSON.parse(l2Data);
      this._setL1(key, data, l1TTL);
      return data;
    }

    // L3: Database
    const data = await fetchFn();
    if (data) {
      await this.redis.setex(key, l2TTL, JSON.stringify(data));
      this._setL1(key, data, l1TTL);
    }

    return data;
  }

  _setL1(key, data, ttl) {
    if (this.l1.size >= this.l1MaxSize) {
      const firstKey = this.l1.keys().next().value;
      this.l1.delete(firstKey);
    }
    this.l1.set(key, { data, expiry: Date.now() + ttl });
  }

  async invalidate(key) {
    this.l1.delete(key);
    await this.redis.del(key);
  }
}

Connection Pooling and Pipelining

For high-throughput caching, use pipelining to batch commands:

// Without pipelining: 100 round trips
for (const id of userIds) {
  const user = await redis.get(`user:${id}`); // 100 network round trips
}

// With pipelining: 1 round trip
const pipeline = redis.pipeline();
for (const id of userIds) {
  pipeline.get(`user:${id}`);
}
const results = await pipeline.exec();
const users = results.map(([err, data]) => data ? JSON.parse(data) : null);

Common Pitfalls

  1. Hot key problem: A single popular key overwhelms one Redis node. Use local caching or key replication.
  2. Large values: Storing multi-MB values blocks Redis. Compress large values or split into smaller keys.
  3. Connection exhaustion: Too many connections to Redis. Use connection pooling (ioredis handles this).
  4. Not handling cluster MOVED errors: ioredis handles this automatically, but custom clients must follow redirections.

Summary

Redis Sentinel provides high availability through automatic failover. Redis Cluster provides horizontal scaling by sharding data across nodes. Memcached is simpler but lacks Redis's rich features. Most Node.js applications should use Redis (with ioredis), adding Sentinel for HA or Cluster for scale. Multi-tier caching (L1 in-memory + L2 Redis) provides the best performance for high-throughput applications.

Common Problems & Solutions

1 CROSSSLOT error in Redis Cluster
Cause: Attempting multi-key operations (MGET, transactions) on keys that map to different hash slots.
Solution: Use hash tags to ensure related keys map to the same slot. `{user:123}.profile` and `{user:123}.settings` will be on the same slot.

// Bad: different slots\nawait cluster.mget('user:1', 'user:2');\n// Good: same slot via hash tag\nawait cluster.mget('{user}.1', '{user}.2');

2 Redis Cluster failover causes brief data loss
Cause: Asynchronous replication means a replica may not have received the latest writes when promoted to master.
Solution: Use WAIT command for critical writes to ensure replication. Accept that Redis Cluster provides 'best effort' durability, not strong consistency.

await redis.set('critical:key', value);\nawait redis.wait(1, 1000); // Wait for 1 replica, 1s timeout
