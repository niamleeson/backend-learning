concurrent.futures: ThreadPoolExecutor and ProcessPoolExecutor

Intuition

concurrent.futures is like a job dispatch board. You submit tasks, get tickets (Futures), and can either wait for all results or handle them as each one completes.

concurrent.futures: ThreadPoolExecutor and ProcessPoolExecutor

The concurrent.futures module provides a high-level interface for asynchronously executing callables using thread or process pools.

from concurrent.futures import (
    ThreadPoolExecutor, ProcessPoolExecutor,
    as_completed, wait, Future
)
import requests

def fetch_url(url: str) -> tuple[str, int]:
    resp = requests.get(url, timeout=10)
    return url, resp.status_code

# ThreadPoolExecutor for I/O-bound work
urls = ["https://python.org", "https://pypi.org", "https://docs.python.org"]

with ThreadPoolExecutor(max_workers=5) as executor:
    futures = {executor.submit(fetch_url, url): url for url in urls}
    for future in as_completed(futures):
        try:
            url, status = future.result(timeout=15)
            print(f"{url}: {status}")
        except Exception as e:
            print(f"{futures[future]}: Error - {e}")

# ProcessPoolExecutor for CPU-bound work
def compute(n: int) -> int:
    return sum(i ** 2 for i in range(n))

with ProcessPoolExecutor(max_workers=4) as executor:
    results = list(executor.map(compute, [10**6] * 4))

Key Takeaways: - ThreadPoolExecutor for I/O-bound; ProcessPoolExecutor for CPU-bound - executor.submit() returns a Future object for async result retrieval - as_completed() yields futures as they finish (not submission order) - executor.map() preserves order and is simpler for uniform tasks - Use future.result(timeout=N) to avoid hanging on slow tasks

Exercises

  1. Parallel Web Scraper

    Build a function that fetches 10 URLs concurrently with ThreadPoolExecutor. Collect results as they complete, handle errors per-URL, and return a dict of {url: content_length}.

    Hint

    Use as_completed() to process results eagerly. Wrap future.result() in try/except to handle per-URL failures gracefully.

  2. Compare Thread vs Process Pools

    Write the same CPU-bound task using both executors. Time each and demonstrate that ProcessPoolExecutor is faster for CPU work while ThreadPoolExecutor is faster for I/O work.

    Hint

    For CPU work, use a tight computation loop. For I/O work, use time.sleep() to simulate. Measure with time.perf_counter().

Common Problems & Solutions

1 `BrokenProcessPool: A process in the process pool was terminated abruptly` during execution.
Cause: A worker process crashed (e.g., segfault, out-of-memory) or the task raised an unhandled exception.
Solution: Wrap task functions in try/except. Check for memory-heavy operations. Use `future.exception()` to inspect errors.
2 Program hangs when using `executor.map()` because one task takes much longer than others.
Cause: `executor.map()` returns results in order, so it blocks waiting for the first result even if later ones are ready.
Solution: Use `executor.submit()` + `as_completed()` to process results as they finish, or set `timeout` on `map()`.
