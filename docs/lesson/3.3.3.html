Constraints: PRIMARY KEY, FOREIGN KEY, UNIQUE, CHECK

Intuition

Constraints are guardrails for your data. They ensure the database rejects bad data rather than trusting every application to validate correctly.

Constraints: PK, FK, UNIQUE, CHECK

Constraints enforce data integrity at the database level, preventing invalid data from ever being stored.

CREATE TABLE employees (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    department_id INTEGER NOT NULL
        REFERENCES departments(id) ON DELETE RESTRICT,
    salary NUMERIC CHECK (salary >= 0),
    start_date DATE NOT NULL,
    end_date DATE,
    CONSTRAINT valid_dates CHECK (end_date IS NULL OR end_date > start_date)
);

-- Composite unique constraint
CREATE TABLE enrollments (
    student_id INTEGER REFERENCES students(id),
    course_id INTEGER REFERENCES courses(id),
    enrolled_at TIMESTAMPTZ DEFAULT now(),
    PRIMARY KEY (student_id, course_id)
);

-- Exclusion constraint (no overlapping date ranges)
CREATE TABLE bookings (
    room_id INTEGER,
    during TSTZRANGE,
    EXCLUDE USING gist (room_id WITH =, during WITH &&)
);

Key Takeaways: - PRIMARY KEY = UNIQUE + NOT NULL. It identifies each row uniquely. - FOREIGN KEY enforces referential integrity; the referenced row must exist. - CHECK constraints validate column values. Exclusion constraints prevent overlapping ranges.

Exercises

  1. Add Multi-Column Constraints

    Create a reservations table where (room_id, date) must be unique and check_out must be after check_in.

    Hint

    UNIQUE(room_id, date) and CHECK(check_out > check_in) as table-level constraints.

  2. Deferrable Foreign Keys

    Create two tables with circular references using DEFERRABLE constraints to allow inserting both rows in one transaction.

    Hint

    REFERENCES players(id) DEFERRABLE INITIALLY DEFERRED allows inserting both rows in one transaction.

Common Problems & Solutions

1 INSERT fails with duplicate key violation.
Cause: Attempting to insert a row with a primary key or unique value that already exists.
Solution: Use INSERT ... ON CONFLICT DO UPDATE (upsert) or ON CONFLICT DO NOTHING.
2 Foreign key constraint blocks DELETE of referenced row.
Cause: Default ON DELETE behavior is RESTRICT.
Solution: Change to ON DELETE CASCADE or ON DELETE SET NULL, or delete dependent rows first.
