Nginx as Reverse Proxy and Load Balancer

Intuition

Nginx is like a receptionist at a busy office. They greet visitors (handle connections), hand out brochures (static files) directly without bothering the staff, and route complex requests to the right employee (application server). The receptionist also manages the queue so no single employee gets overwhelmed (load balancing).

Nginx as Reverse Proxy and Load Balancer

Nginx sits in front of your Python app server, handling static files, SSL, load balancing, and buffering.

Basic Reverse Proxy

server {
    listen 80;
    server_name myapp.com;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /static/ {
        alias /opt/myapp/static/;
        expires 30d;
    }

    client_max_body_size 10M;
}

Load Balancing

upstream app_servers {
    least_conn;
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
    server 127.0.0.1:8004 backup;
}

server {
    listen 80;
    location / {
        proxy_pass http://app_servers;
        proxy_next_upstream error timeout http_502;
    }
}

Essential Commands

sudo nginx -t           # Test configuration syntax
sudo nginx -s reload    # Reload without downtime
sudo ln -s /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/
sudo tail -f /var/log/nginx/access.log

Common Mistakes

  • Not passing X-Forwarded-For headers, so the app sees 127.0.0.1 for all requests
  • Serving static files through Python instead of Nginx
  • Not running nginx -t before reloading

Key Takeaways

  • Always nginx -t before nginx -s reload
  • Serve static files from Nginx, not Python
  • Use least_conn for APIs with varying request durations

Exercises

  1. Set Up Nginx Reverse Proxy

    Configure Nginx as a reverse proxy for a Gunicorn-served Python application. Serve static files directly from Nginx, set proper proxy headers, and configure access logging.

    Hint

    Use proxy_pass to Gunicorn's port. Use the alias directive for static files.

  2. Load Balancer Configuration

    Set up Nginx to load balance across 3 Gunicorn instances on different ports. Implement health checks and configure a backup server. Test failover by stopping one instance.

    Hint

    Use the upstream block with least_conn and proxy_next_upstream for failover.

  3. Rate Limiting

    Configure Nginx rate limiting to allow 10 requests per second per IP for the API and 100 requests per second for static assets. Return a custom 429 response.

    Hint

    Use limit_req_zone and limit_req directives with different zones for different locations.

Common Problems & Solutions

1 502 Bad Gateway error
Symptoms: ['502 Bad Gateway in browser', 'connect() failed (111: Connection refused) in error.log']
Cause: Nginx cannot connect to the upstream application server (Gunicorn/Uvicorn isn't running or is on a different port).
Solution: Verify the application server is running: `systemctl status myapp`. Check that the proxy_pass port matches the application server's bind port. Check Nginx error logs: `tail /var/log/nginx/error.log`.
2 File uploads fail with 413 error
Symptoms: ['413 Request Entity Too Large']
Cause: Nginx's default `client_max_body_size` is 1MB.
Solution: Add `client_max_body_size 10M;` (or your desired limit) in the server or location block.

client_max_body_size 10M;

3 WebSocket connections fail through Nginx
Symptoms: ['WebSocket connection failed', '400 Bad Request on ws:// endpoint']
Cause: Nginx doesn't forward WebSocket upgrade headers by default.
Solution: Add upgrade headers to the proxy location: ``` proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade"; ```
