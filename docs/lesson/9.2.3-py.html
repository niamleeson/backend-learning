Distributed Caching with Redis and Memcached

Intuition

Redis is like a Swiss Army knife for caching—it does many things beyond simple key-value storage. Memcached is like a sharp kitchen knife—it does one thing extremely well and with minimal overhead.

Distributed Caching: Redis & Memcached

Distributed caching places a shared, in-memory data store between your application and database, dramatically reducing read latency and database load. Redis and Memcached are the two dominant solutions, each with distinct strengths.

Core Concepts

  • Redis: An in-memory data structure store supporting strings, hashes, lists, sets, sorted sets, streams, and more. Offers persistence (RDB snapshots, AOF), replication, Lua scripting, pub/sub, and clustering.
  • Memcached: A simpler, high-performance distributed memory cache. Supports only string key-value pairs with LRU eviction. Multi-threaded by design, making it efficient for simple caching at scale.
  • Redis Cluster: Provides automatic sharding across multiple nodes using hash slots (16384 slots). Supports automatic failover with replica promotion.
  • Eviction Policies: Redis supports allkeys-lru, volatile-lru, allkeys-lfu, volatile-ttl, and more. Memcached uses LRU only.
  • Serialization: Choose efficient serialization (MessagePack, Protocol Buffers) over JSON for cache values to reduce memory and network overhead.

Key Takeaways

  1. Use Redis when you need rich data structures, persistence, or pub/sub. Use Memcached for simple, high-throughput key-value caching.
  2. Monitor memory usage and eviction rates to right-size your cache cluster.
  3. Redis Cluster provides horizontal scaling but requires client library support for hash slot-aware routing.

Exercises

  1. Benchmark Redis vs Memcached

    Using redis-benchmark and memtier_benchmark or mcperf, run identical read/write workloads against Redis and Memcached. Compare throughput, latency percentiles (p50, p99), and memory usage under identical data sets.

    Hint

    Use redis-benchmark -t set,get -n 100000 -q for Redis. For Memcached, use memtier_benchmark --protocol=memcache_text -s 127.0.0.1 -p 11211.

  2. Set Up Redis Cluster

    Create a 6-node Redis Cluster (3 masters, 3 replicas) using Docker. Write a Python client using redis-py-cluster or redis.cluster.RedisCluster that sets and gets keys, demonstrating automatic slot-based routing.

    Hint

    Use docker run with different ports (7000-7005). Initialize the cluster with redis-cli --cluster create. Use RedisCluster(startup_nodes=[...]) in Python.

Common Problems & Solutions

1 Redis memory usage grows unbounded and the instance is eventually OOM-killed.
Cause: No `maxmemory` policy is configured, or keys are written without TTLs, causing indefinite accumulation.
Solution: Set `maxmemory` to a safe limit (e.g., 75% of available RAM) and configure an eviction policy like `allkeys-lfu`. Ensure all cache keys have appropriate TTLs.
2 Redis Cluster reports MOVED or ASK errors in the application logs.
Cause: The client is not cluster-aware and is sending commands to the wrong node for the key's hash slot.
Solution: Use a cluster-aware Redis client (e.g., `redis.cluster.RedisCluster` in Python, `ioredis` in Node.js) that automatically follows redirections and caches slot mappings.
