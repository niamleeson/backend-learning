Cache-Aside, Write-Through, and Write-Behind

Intuition

Cache-aside is like checking your notebook before looking something up in a textbook. Write-through is like updating both your notebook and the textbook every time you learn something new. Write-behind is updating your notebook now and transcribing to the textbook later.

Cache-Aside, Write-Through & Write-Behind

Caching strategies define how data flows between your application, cache, and database. Choosing the right pattern determines your consistency guarantees, read/write performance, and failure behavior.

Core Concepts

  • Cache-Aside (Lazy Loading): The application checks the cache first. On a miss, it reads from the database, writes to the cache, and returns the result. The cache is populated on demand.
  • Write-Through: Every write goes to the cache and the database synchronously. Ensures the cache is always consistent but adds write latency.
  • Write-Behind (Write-Back): Writes go to the cache immediately, and a background process asynchronously flushes changes to the database. Offers low write latency but risks data loss on cache failure.
  • Read-Through: Similar to cache-aside but the cache itself is responsible for loading data from the database on a miss, simplifying application code.
  • Choosing a Strategy: Cache-aside is the most common for read-heavy workloads. Write-through suits workloads requiring strong consistency. Write-behind is used when write performance is critical and eventual consistency is acceptable.

Key Takeaways

  1. Cache-aside is the default choice for most applications due to its simplicity and read optimization.
  2. Write-through and write-behind are complementary to cache-aside and solve different write-path challenges.
  3. Every caching strategy involves a trade-off between consistency, latency, and durability.

Exercises

  1. Implement Cache-Aside with Redis

    Build a Python function that implements cache-aside for a user profile lookup. On cache miss, query PostgreSQL, serialize the result, store it in Redis with a 5-minute TTL, and return it. Measure cache hit rate over 1000 requests.

    Hint

    Use redis.get(key) first; if None, query the DB, then redis.setex(key, 300, json.dumps(data)). Track hits/misses with counters.

  2. Build a Write-Behind Queue

    Implement a write-behind cache using Redis and a background worker. Writes update Redis immediately and push the key to a Redis list. A worker process polls the list and batches writes to PostgreSQL every 5 seconds.

    Hint

    Use redis.lpush('write_queue', key) on writes. The worker does redis.brpop('write_queue', timeout=5) and batches keys for a bulk DB update.

Common Problems & Solutions

1 Cache and database have different values for the same record.
Cause: In cache-aside, a race condition occurs when two processes read a cache miss simultaneously, both query the DB, and one writes stale data to the cache after the other has updated the DB.
Solution: Use distributed locks or conditional SET (SET NX) when populating the cache, or apply short TTLs to limit the staleness window.
2 Write-behind cache loses recent writes after a Redis crash.
Cause: Writes in the buffer had not yet been flushed to the database when the cache node failed.
Solution: Enable Redis persistence (AOF with `appendfsync everysec`), use Redis replication for failover, or implement a WAL-style approach where writes are logged durably before acknowledging.
