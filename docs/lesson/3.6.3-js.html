Testing the API and Deployment Prep

Intuition

Testing is like a dress rehearsal before opening night. Supertest acts as a mock audience that watches every performance (endpoint), checking that actors (handlers) hit their marks (status codes), deliver their lines (response bodies), and handle hecklers (error cases) gracefully. Deployment prep is making sure the theater (production) has lighting, sound, and emergency exits ready.

Testing the API and Deployment Prep

Introduction

An API is not done when the code works — it is done when it is tested and ready to deploy. This lesson covers writing integration tests for your Express API using Supertest and Vitest, preparing your application for production deployment, and setting up the tooling that ensures your API stays reliable.

Integration Tests with Supertest

Supertest lets you make HTTP requests to your Express app without starting a real server:

// src/modules/auth/__tests__/auth.integration.test.ts
import { describe, it, expect, beforeEach } from 'vitest';
import request from 'supertest';
import app from '../../../app';
import { prisma } from '../../../config/database';

describe('Auth API', () => {
  beforeEach(async () => {
    // Clean up test data
    await prisma.user.deleteMany();
  });

  describe('POST /api/auth/register', () => {
    it('should register a new user', async () => {
      const res = await request(app)
        .post('/api/auth/register')
        .send({
          email: 'test@example.com',
          password: 'Password123',
          name: 'Test User',
        });

      expect(res.status).toBe(201);
      expect(res.body.success).toBe(true);
      expect(res.body.data.user.email).toBe('test@example.com');
      expect(res.body.data.accessToken).toBeDefined();
      expect(res.body.data.refreshToken).toBeDefined();
      // Should NOT return password
      expect(res.body.data.user.passwordHash).toBeUndefined();
    });

    it('should reject duplicate email', async () => {
      await request(app).post('/api/auth/register').send({
        email: 'test@example.com', password: 'Password123', name: 'User 1',
      });

      const res = await request(app).post('/api/auth/register').send({
        email: 'test@example.com', password: 'Password456', name: 'User 2',
      });

      expect(res.status).toBe(409);
    });

    it('should reject invalid email', async () => {
      const res = await request(app).post('/api/auth/register').send({
        email: 'not-an-email', password: 'Password123', name: 'User',
      });

      expect(res.status).toBe(422);
    });

    it('should reject weak password', async () => {
      const res = await request(app).post('/api/auth/register').send({
        email: 'test@example.com', password: 'weak', name: 'User',
      });

      expect(res.status).toBe(422);
    });
  });

  describe('POST /api/auth/login', () => {
    beforeEach(async () => {
      await request(app).post('/api/auth/register').send({
        email: 'test@example.com', password: 'Password123', name: 'Test User',
      });
    });

    it('should login with valid credentials', async () => {
      const res = await request(app).post('/api/auth/login').send({
        email: 'test@example.com', password: 'Password123',
      });

      expect(res.status).toBe(200);
      expect(res.body.data.accessToken).toBeDefined();
    });

    it('should reject invalid password', async () => {
      const res = await request(app).post('/api/auth/login').send({
        email: 'test@example.com', password: 'WrongPassword',
      });

      expect(res.status).toBe(401);
    });
  });
});

Testing Protected Endpoints

describe('Posts API', () => {
  let authToken: string;

  beforeEach(async () => {
    await prisma.comment.deleteMany();
    await prisma.post.deleteMany();
    await prisma.user.deleteMany();

    // Register and get token
    const res = await request(app).post('/api/auth/register').send({
      email: 'author@example.com', password: 'Password123', name: 'Author',
    });
    authToken = res.body.data.accessToken;
  });

  describe('POST /api/posts', () => {
    it('should create a post when authenticated', async () => {
      const res = await request(app)
        .post('/api/posts')
        .set('Authorization', `Bearer ${authToken}`)
        .send({ title: 'My Post', content: 'Post content here' });

      expect(res.status).toBe(201);
      expect(res.body.data.title).toBe('My Post');
      expect(res.body.data.author.name).toBe('Author');
    });

    it('should reject unauthenticated requests', async () => {
      const res = await request(app)
        .post('/api/posts')
        .send({ title: 'My Post', content: 'Content' });

      expect(res.status).toBe(401);
    });
  });

  describe('GET /api/posts', () => {
    it('should return paginated posts', async () => {
      // Create 5 posts
      for (let i = 0; i < 5; i++) {
        await request(app)
          .post('/api/posts')
          .set('Authorization', `Bearer ${authToken}`)
          .send({ title: `Post ${i}`, content: `Content ${i}` });
      }

      const res = await request(app)
        .get('/api/posts?page=1&limit=3');

      expect(res.status).toBe(200);
      expect(res.body.data).toHaveLength(3);
      expect(res.body.meta.total).toBe(5);
      expect(res.body.meta.totalPages).toBe(2);
    });
  });
});

Test Helpers

// src/test/helpers.ts
import request from 'supertest';
import app from '../app';

export async function createTestUser(data = {
  email: 'test@example.com',
  password: 'Password123',
  name: 'Test User',
}) {
  const res = await request(app).post('/api/auth/register').send(data);
  return {
    user: res.body.data.user,
    accessToken: res.body.data.accessToken,
    refreshToken: res.body.data.refreshToken,
  };
}

export function authRequest(method: 'get' | 'post' | 'put' | 'patch' | 'delete', url: string, token: string) {
  return request(app)[method](url).set('Authorization', `Bearer ${token}`);
}

# Dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npx prisma generate
RUN npm run build

FROM node:20-alpine AS production
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules/.prisma ./node_modules/.prisma
COPY prisma ./prisma

ENV NODE_ENV=production
EXPOSE 3000

CMD ["sh", "-c", "npx prisma migrate deploy && node dist/server.js"]

Pre-Deployment Checklist

1. [ ] All tests pass (npm run test:ci)
2. [ ] TypeScript compiles without errors (npm run typecheck)
3. [ ] Linting passes (npm run lint)
4. [ ] Environment variables documented in .env.example
5. [ ] Database migrations are committed
6. [ ] Sensitive data not in source code or logs
7. [ ] CORS configured for production domain
8. [ ] Rate limiting enabled
9. [ ] Helmet security headers enabled
10. [ ] Error handler does not leak internal details in production
11. [ ] Health check endpoint works
12. [ ] Graceful shutdown handles SIGTERM
13. [ ] Database connection pool sized correctly
14. [ ] Logging configured for production (JSON format, no debug)

Summary

Testing and deployment prep transform working code into production-ready software. Use Supertest for integration tests that exercise the full request/response cycle. Create test helpers for common operations like authentication. Prepare for production with Docker, health checks, proper configuration, and a thorough deployment checklist. The goal is confidence that your API works correctly and can be deployed reliably.

Exercises

  1. Write Full Test Suite

    Write integration tests for the Blog API covering: auth (register, login, invalid credentials), posts CRUD (create, read, update, delete with authorization), comments (create, delete), and edge cases (404, validation errors, unauthorized access). Aim for 80%+ coverage.

    Hint

    Use beforeEach to clean the database and create test users. Use helper functions for authenticated requests. Test both happy paths and error cases.

  2. Dockerize the Application

    Create a Dockerfile using multi-stage builds for the Blog API. Include a docker-compose.yml with the API, PostgreSQL, and Redis services. Ensure migrations run on startup.

    Hint

    Use node:20-alpine for small images. Copy only package*.json first for layer caching. Run prisma generate during build and prisma migrate deploy at startup.

  3. CI/CD Pipeline

    Create a GitHub Actions workflow that: checks out code, installs dependencies, runs type checking, linting, and tests with a PostgreSQL service container. Use matrix testing for Node 18 and 20.

    Hint

    Use services: postgres in the workflow. Set DATABASE_URL as an environment variable. Run npx prisma migrate deploy before tests.

Common Problems & Solutions

1 Tests interfere with each other — order-dependent failures
Cause: Tests share database state. A test that creates data affects subsequent tests.
Solution: Use beforeEach to clean the database (deleteMany on all tables in reverse dependency order). Or use transactions that roll back after each test.
2 Tests are slow due to bcrypt hashing
Cause: bcrypt with high cost factor (12) takes ~250ms per hash, slowing down tests that create users.
Solution: Use a lower bcrypt cost factor in tests: bcrypt.hash(password, 1). Set via environment variable: BCRYPT_ROUNDS=1 in test config.
3 Docker build fails with 'prisma generate' error
Cause: The Prisma schema or dependencies are not available during the Docker build step.
Solution: Copy prisma/ directory and package.json before running npm ci. Run npx prisma generate after npm ci. Ensure the .prisma/client directory is copied to the production stage.
