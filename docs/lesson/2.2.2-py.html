pytest: Fixtures, Markers, and Parametrize

Intuition

pytest fixtures are like a catering service for tests: declare what you need as parameters, and pytest prepares and delivers it. Parametrize is like a test template that runs with different data.

pytest: Fixtures, Markers, and Parametrize

pytest is the most popular Python testing framework. It uses plain functions, powerful fixtures for setup/teardown, and decorators for test organization.

import pytest

@pytest.fixture
def sample_users():
    return [
        {"name": "Alice", "age": 30},
        {"name": "Bob", "age": 25},
    ]

def test_user_count(sample_users):
    assert len(sample_users) == 2

def test_user_names(sample_users):
    names = [u["name"] for u in sample_users]
    assert "Alice" in names

@pytest.mark.parametrize("input_val,expected", [
    ("hello", 5),
    ("", 0),
    ("pytest", 6),
])
def test_string_length(input_val, expected):
    assert len(input_val) == expected

@pytest.mark.slow
def test_expensive_operation():
    import time
    time.sleep(0.1)
    assert True

# Run with: pytest -m "not slow" to skip slow tests

Key Takeaways: - Fixtures replace setUp/tearDown with dependency injection - @pytest.mark.parametrize generates multiple test cases from data - Custom markers (@pytest.mark.slow) let you selectively run test subsets - Fixtures can have scopes: function, class, module, session - Use conftest.py to share fixtures across multiple test files

Exercises

  1. Build a Database Fixture

    Create a db_connection fixture with scope='module' that yields a mock database connection and cleans up after all tests in the module. Use it in three test functions.

    Hint

    Use yield in the fixture to provide the value, then add cleanup code after yield. The scope='module' ensures it runs once per module.

  2. Parametrize Edge Cases

    Write a parse_email function and use @pytest.mark.parametrize to test at least 6 cases including valid emails, missing @, missing domain, and empty strings.

    Hint

    Parametrize can include expected exceptions. Use pytest.raises inside the test for invalid cases, or parametrize with an expected_valid boolean.

Common Problems & Solutions

1 Fixture not found error: `fixture 'my_fixture' not found`.
Cause: The fixture is defined in a different file and not imported or placed in `conftest.py`.
Solution: Move shared fixtures to `conftest.py` in the test directory. pytest auto-discovers fixtures from conftest files.
2 Parametrized test IDs are unreadable in output.
Cause: pytest auto-generates IDs from parameter values which can be long or unclear.
Solution: Use the `ids` parameter: `@pytest.mark.parametrize('x', [1, 2], ids=['one', 'two'])` for readable test names.
