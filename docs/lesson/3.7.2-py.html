Caching Patterns: Cache-Aside, Write-Through, TTL

Intuition

Caching is a trade-off between speed and freshness. Cache-aside is reactive, write-through is proactive. TTL ensures nothing stays stale forever.

Caching Patterns: Cache-Aside, Write-Through, TTL

Caching reduces database load by storing frequently accessed data in Redis. The caching pattern determines when and how the cache is populated.

import redis, json

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

# Cache-Aside (Lazy Loading)
def get_user(user_id: int) -> dict:
    cache_key = f'user:{user_id}'
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)
    user = db.query(User).get(user_id)
    user_data = {'id': user.id, 'name': user.name}
    r.set(cache_key, json.dumps(user_data), ex=300)
    return user_data

# Write-Through
def update_user(user_id: int, data: dict):
    db.query(User).filter_by(id=user_id).update(data)
    db.commit()
    r.set(f'user:{user_id}', json.dumps(data), ex=300)

# Cache Invalidation
def delete_user(user_id: int):
    db.query(User).filter_by(id=user_id).delete()
    db.commit()
    r.delete(f'user:{user_id}')

Key Takeaways: - Cache-aside loads data into cache only on a miss; simple but can serve stale data. - Write-through updates cache on every write; consistent but adds write latency. - TTL is your safety net: even if invalidation fails, stale data expires automatically.

Exercises

  1. Implement Cache-Aside Decorator

    Write a cache-aside decorator that checks Redis before calling the database function, caching the result with a configurable TTL.

    Hint

    Use functools.wraps, generate a cache key from function name + args, json serialize the result.

  2. Cache Stampede Prevention

    Implement a locking mechanism to prevent multiple processes from rebuilding the same cache key simultaneously.

    Hint

    Use r.set(lock_key, 1, nx=True, ex=10) as a distributed lock. Only the lock holder rebuilds the cache.

Common Problems & Solutions

1 Stale data persists in cache after a database update.
Cause: The cache was not invalidated or updated when the database was written to.
Solution: Invalidate or update the cache key on every write. Use TTL as a fallback safety net.
2 Cache stampede: many requests rebuild the same key simultaneously.
Cause: A popular cache key expires and all concurrent requests hit the database at once.
Solution: Use a distributed lock or probabilistic early expiration to stagger cache rebuilds.
