Design a Ticket Booking System (Ticketmaster)

Intuition

A ticket booking system is an inventory reservation problem under extreme concurrency where a virtual queue is essential to manage demand that can exceed capacity by 100x in the first seconds of an on-sale event. Seat-level locking with time-bounded holds is the mechanism that prevents both overselling and indefinite resource blocking.

Design a Ticket Booking System (Ticketmaster)

Requirements

  • Sell tickets for events with specific seat assignments
  • Handle extreme traffic spikes when popular events go on sale (millions of users in seconds)
  • Prevent overselling and ensure fair access
  • Support seat selection, holds, and time-limited checkout
  • Manage a virtual waiting queue for high-demand events

Architecture

A virtual queue (waiting room) absorbs traffic spikes and meters users into the booking flow. A seat map service shows real-time availability. A booking service uses optimistic locking with seat-level granularity. A timer service releases held seats after a checkout timeout. A separate inventory service pre-loads seat data into Redis for fast lookups.

Key Components

  • Virtual Queue: Assigns queue positions and gradually admits users based on available capacity
  • Seat Map Service: Real-time seat availability backed by Redis bitmaps per event section
  • Hold Service: Temporarily locks selected seats for a user with a 5-10 minute expiration
  • Booking Service: Confirms purchase, charges payment, and issues e-tickets
  • Inventory Service: Redis-backed seat status (available/held/sold) with atomic transitions

Scaling

Use the virtual queue to flatten traffic spikes. Pre-load event inventory into Redis before on-sale. Shard by event ID so different events scale independently. Use WebSockets for real-time seat map updates. Separate read and write paths.

Trade-offs

  • Seat holds improve UX but reduce effective inventory while users browse; short timeouts frustrate users, long ones block others
  • A strict FIFO queue is fair but slower to fill seats; allowing parallel access fills faster but feels less fair

Exercises

  1. Design a Virtual Waiting Queue

    Build a queue system that assigns each user a position when they arrive for a popular event. Admit users in batches (e.g., 1000 at a time) based on system capacity. Show estimated wait times. Handle users who leave the queue.

    Hint

    Assign monotonically increasing queue numbers. A background process advances the 'admitted up to' counter based on current system load and completed checkouts. Use WebSocket to notify users when it's their turn.

  2. Implement Seat Hold with TTL

    Create a system where selecting a seat places a temporary hold (5 minutes). If the user does not complete checkout, the hold expires and the seat becomes available again. Handle the race condition of hold expiration during payment processing.

    Hint

    Store holds in Redis with TTL. Before charging payment, verify the hold still exists and extend it briefly. Use a Lua script for atomic check-and-extend. If the hold expired, abort the payment and inform the user.

Common Problems & Solutions

1 Popular event on-sale crashes the system as millions of users flood in simultaneously within the first second.
Cause: The system tries to serve all users immediately without traffic shaping, overwhelming the booking service, database, and seat map.
Solution: Implement a virtual waiting room that queues users before the booking flow. Meter admission based on backend capacity. Use pre-warmed Redis instances with event inventory loaded beforehand. Scale the queue infrastructure (static pages) separately from booking.
2 Users hold seats but abandon checkout, causing popular seats to appear unavailable for extended periods while nobody actually buys them.
Cause: Hold timeouts are too generous or not enforced, and users open multiple tabs holding different seats without completing any purchase.
Solution: Set aggressive hold timeouts (5 minutes). Limit holds to one active session per user per event. Run a background reaper that releases expired holds. Show a countdown timer to the user to encourage completion.
