Full-Text Search in PostgreSQL

Intuition

Full-text search breaks text into normalized word stems and matches against search patterns. It is like having a mini search engine inside your database.

Full-Text Search in PostgreSQL

PostgreSQL provides built-in full-text search with tsvector and tsquery, supporting stemming, ranking, and GIN-indexed fast lookups.

-- Create a tsvector column and index
ALTER TABLE articles ADD COLUMN search_vector tsvector;
UPDATE articles SET search_vector =
  to_tsvector('english', title || ' ' || body);
CREATE INDEX idx_fts ON articles USING gin (search_vector);

-- Search with tsquery
SELECT title, ts_rank(search_vector, q) AS rank
FROM articles, to_tsquery('english', 'python & web') q
WHERE search_vector @@ q
ORDER BY rank DESC;

-- Auto-update with trigger
CREATE TRIGGER trg_search_update
BEFORE INSERT OR UPDATE ON articles
FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'pg_catalog.english', title, body);

Key Takeaways: - tsvector is a sorted list of normalized lexemes; tsquery defines the search terms with boolean operators. - GIN index on tsvector makes full-text search performant on millions of rows. - ts_rank and ts_rank_cd score relevance; use weights (A-D) to prioritize title over body.

Exercises

  1. Weighted Search

    Set up full-text search where title matches are weighted higher than body matches using setweight.

    Hint

    setweight(to_tsvector(title), 'A') || setweight(to_tsvector(body), 'B') then use ts_rank.

  2. Search with Phrase Matching

    Write a query using phraseto_tsquery to find articles containing the exact phrase 'machine learning'.

    Hint

    WHERE search_vector @@ phraseto_tsquery('english', 'machine learning').

Common Problems & Solutions

1 Search returns no results for words that exist in the text.
Cause: Stemming reduces words to root forms; the exact word may not match the stemmed lexeme.
Solution: Check to_tsvector output to see the actual lexemes. Use plainto_tsquery for simpler matching.
2 Full-text search is slow on large tables.
Cause: Missing GIN index or computing tsvector on the fly in the query.
Solution: Store tsvector in a dedicated column, create a GIN index, and keep it updated with a trigger.
