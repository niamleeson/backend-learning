Terraform Basics: Providers, Resources, State

Intuition

Terraform is like architectural blueprints for your cloud infrastructure. Instead of building a house by making decisions on the spot (clicking in the AWS console), you draw detailed blueprints first (write .tf files), review them (terraform plan), then build exactly what is specified (terraform apply). If you want to renovate, you update the blueprints and Terraform figures out what needs to change. The state file is like a property deed that records exactly what was built.

Terraform Basics: Providers, Resources, State

What is Infrastructure as Code (IaC)?

Infrastructure as Code means defining your servers, databases, networks, and other infrastructure in declarative configuration files instead of manually clicking through a cloud console. Terraform is the most popular IaC tool, supporting AWS, GCP, Azure, and dozens of other providers.

Why Terraform?

Manual (Console/CLI):                 Terraform (IaC):
─────────────────────                 ─────────────────
Click "Create Instance"               Write .tf file
Fill in form fields                   terraform plan (preview)
Click "Launch"                        terraform apply (create)
Hope you remember settings            Config is version-controlled
No audit trail                        Full change history in git
Hard to replicate                     Exact replication every time
Hard to tear down                     terraform destroy

Core Concepts

Providers

# providers.tf
terraform {
  required_version = ">= 1.5"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      Environment = var.environment
      ManagedBy   = "terraform"
      Project     = "myapp"
    }
  }
}

Resources

# main.tf

# VPC
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name = "${var.project}-vpc"
  }
}

# Public Subnet
resource "aws_subnet" "public" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 1}.0/24"
  availability_zone = data.aws_availability_zones.available.names[count.index]

  map_public_ip_on_launch = true

  tags = {
    Name = "${var.project}-public-${count.index + 1}"
  }
}

# Security Group
resource "aws_security_group" "app" {
  name_prefix = "${var.project}-app-"
  vpc_id      = aws_vpc.main.id

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = [var.admin_cidr]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# EC2 Instance
resource "aws_instance" "app" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type
  subnet_id     = aws_subnet.public[0].id
  key_name      = var.key_pair_name

  vpc_security_group_ids = [aws_security_group.app.id]

  user_data = templatefile("${path.module}/scripts/setup.sh", {
    node_version = var.node_version
    app_name     = var.project
  })

  root_block_device {
    volume_size = 20
    volume_type = "gp3"
  }

  tags = {
    Name = "${var.project}-server"
  }
}

# RDS PostgreSQL
resource "aws_db_instance" "postgres" {
  identifier     = "${var.project}-db"
  engine         = "postgres"
  engine_version = "16.1"
  instance_class = var.db_instance_class

  allocated_storage     = 20
  max_allocated_storage = 100
  storage_type          = "gp3"

  db_name  = var.db_name
  username = var.db_username
  password = var.db_password

  multi_az               = var.environment == "production"
  db_subnet_group_name   = aws_db_subnet_group.main.name
  vpc_security_group_ids = [aws_security_group.db.id]

  backup_retention_period = 7
  skip_final_snapshot     = var.environment != "production"
  deletion_protection     = var.environment == "production"

  storage_encrypted = true

  tags = {
    Name = "${var.project}-db"
  }
}

# S3 Bucket
resource "aws_s3_bucket" "uploads" {
  bucket = "${var.project}-uploads-${var.environment}"
}

resource "aws_s3_bucket_versioning" "uploads" {
  bucket = aws_s3_bucket.uploads.id
  versioning_configuration {
    status = "Enabled"
  }
}

Variables

# variables.tf
variable "aws_region" {
  description = "AWS region"
  type        = string
  default     = "us-east-1"
}

variable "environment" {
  description = "Environment (dev, staging, production)"
  type        = string
  validation {
    condition     = contains(["dev", "staging", "production"], var.environment)
    error_message = "Environment must be dev, staging, or production."
  }
}

variable "project" {
  description = "Project name"
  type        = string
  default     = "myapp"
}

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t3.small"
}

variable "db_password" {
  description = "Database password"
  type        = string
  sensitive   = true       # Won't show in logs
}

variable "db_instance_class" {
  description = "RDS instance class"
  type        = string
  default     = "db.t3.micro"
}

# terraform.tfvars (gitignored!)
aws_region      = "us-east-1"
environment     = "production"
instance_type   = "t3.small"
db_password     = "super-secret-password"

Terraform State

State is how Terraform tracks which real resources correspond to your configuration.

# backend.tf - Store state remotely (recommended)
terraform {
  backend "s3" {
    bucket         = "myapp-terraform-state"
    key            = "production/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"     # State locking
    encrypt        = true
  }
}

Terraform Commands

# Initialize (download providers, configure backend)
terraform init

# Preview changes
terraform plan

# Apply changes
terraform apply

# Apply with auto-approve (CI/CD)
terraform apply -auto-approve

# Destroy all resources
terraform destroy

# Show current state
terraform show

# List resources in state
terraform state list

# Import existing resource
terraform import aws_instance.app i-1234567890abcdef0

# Format files
terraform fmt

# Validate configuration
terraform validate

Common Pitfalls

  1. Storing state locally - Local state cannot be shared across teams and is not backed up. Always use remote state (S3, GCS).
  2. Committing terraform.tfvars or .tfstate - These contain secrets. Add to .gitignore.
  3. Not using terraform plan before apply - Always review what will change before applying.
  4. Modifying infrastructure outside Terraform - Manual changes create drift between state and reality. Always change through Terraform.
  5. Not using state locking - Without DynamoDB locking, two people running terraform apply simultaneously can corrupt state.

Exercises

  1. Basic Infrastructure

    Write Terraform configuration to create: a VPC with public and private subnets, an EC2 instance with a security group allowing HTTP and SSH, and an RDS PostgreSQL instance in the private subnet. Use variables for all configurable values.

    Hint

    Start with the VPC, then subnets, internet gateway, route tables, security groups, and finally EC2 and RDS. Use count or for_each for multiple subnets.

  2. Remote State Setup

    Configure Terraform to store state in an S3 bucket with DynamoDB locking. Create the S3 bucket and DynamoDB table manually (or with a separate Terraform config), then configure the backend.

    Hint

    The S3 bucket for state must be created before configuring the backend. Use 'terraform init -migrate-state' to move existing local state to the remote backend.

  3. Multi-Environment Setup

    Create a Terraform configuration that can deploy the same infrastructure to dev, staging, and production environments using different .tfvars files. Each environment should have different instance sizes and database configurations.

    Hint

    Create dev.tfvars, staging.tfvars, and production.tfvars. Use 'terraform apply -var-file=production.tfvars'. Use conditionals for environment-specific settings (e.g., multi_az only in production).

Common Problems & Solutions

1 Terraform destroy fails with dependency errors
Cause: Resources have dependencies that prevent deletion (e.g., a security group is still attached to a running instance).
Solution: Terraform usually handles dependency ordering. If it fails, try targeted destroy: `terraform destroy -target=aws_instance.app` first, then destroy the rest. Check for resources created outside Terraform that reference your resources.
2 State file out of sync with actual infrastructure
Cause: Someone modified infrastructure manually through the AWS console, creating drift.
Solution: Run `terraform plan` to detect drift. Use `terraform refresh` to update state with real-world data. For resources created outside Terraform, use `terraform import` to bring them under management. Going forward, enforce that all changes go through Terraform.
3 Terraform wants to destroy and recreate a resource instead of updating it
Cause: Some resource attributes are immutable (forces new). Changing them requires destroying and recreating the resource.
Solution: Check the `plan` output for 'forces replacement'. For databases, this is dangerous -- it will delete your data. Use lifecycle { prevent_destroy = true } to prevent accidental destruction. For immutable changes, use create_before_destroy lifecycle.
