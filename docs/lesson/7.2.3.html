Multi-Stage Builds and Image Optimization

Intuition

Multi-stage builds are like building furniture in a workshop (stage 1) and then moving only the finished piece to your apartment (stage 2). You don't bring the table saw, wood shavings, and screwdriver home -- you only bring the chair. The build tools stay behind, resulting in a cleaner, smaller home.

Multi-Stage Builds and Image Optimization

Production Docker images should be small, secure, and fast to deploy. Multi-stage builds separate build-time dependencies from the runtime image.

Multi-Stage Build

# Stage 1: Build
FROM python:3.12-slim AS builder
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends gcc libpq-dev
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# Stage 2: Runtime
FROM python:3.12-slim AS runtime
WORKDIR /app
COPY --from=builder /install /usr/local
RUN apt-get update && apt-get install -y --no-install-recommends libpq5 \
    && rm -rf /var/lib/apt/lists/*
COPY . .
RUN adduser --disabled-password appuser
USER appuser
EXPOSE 8000
CMD ["gunicorn", "-w", "4", "-b", "0.0.0.0:8000", "app:create_app()"]

Optimization Techniques

# Compare image sizes
# python:3.12        ~1.0GB
# python:3.12-slim   ~150MB
# python:3.12-alpine ~50MB (compatibility issues with some packages)

# Analyze layers
docker history myapp:latest

# Combine RUN commands to reduce layers
RUN apt-get update \
    && apt-get install -y --no-install-recommends gcc \
    && pip install --no-cache-dir -r requirements.txt \
    && apt-get purge -y gcc && apt-get autoremove -y \
    && rm -rf /var/lib/apt/lists/*

# Pin base image versions
FROM python:3.12.1-slim

Common Mistakes

  • Including build tools (gcc) in the final image
  • Using Alpine for Python -- musl libc causes compatibility issues
  • Not pinning base image versions

Key Takeaways

  • Multi-stage builds can reduce image sizes by 50-80%
  • Prefer slim over alpine for Python images
  • Use --no-cache-dir with pip to avoid storing caches in the image

Exercises

  1. Optimize an Image

    Take a basic Dockerfile that uses python:3.12 (full image), installs gcc and dev libraries, and produces a 1.2GB image. Refactor it with multi-stage builds to get it under 200MB.

    Hint

    Use a builder stage for compilation, copy only /install to a slim runtime stage, and remove apt caches.

  2. Layer Analysis

    Build an image, use docker history to identify the largest layers, and refactor the Dockerfile to minimize total size. Document the before and after sizes.

    Hint

    Combine RUN commands, clean up in the same layer, and use --no-cache-dir.

Common Problems & Solutions

1 Multi-stage build copies packages but they fail at runtime
Symptoms: ['ImportError: libpq.so.5: cannot open shared object file']
Cause: Build-time packages were compiled against libraries that aren't in the runtime stage.
Solution: Ensure runtime system libraries (like libpq5 for psycopg2) are installed in the final stage. Only build-time headers (libpq-dev) can be excluded.
2 Docker build takes too long in CI
Symptoms: ['Every CI build reinstalls all pip packages from scratch']
Cause: Layer cache is not being reused between CI runs.
Solution: Use Docker BuildKit cache mounts or push/pull cache images: `docker build --cache-from myapp:latest`.
