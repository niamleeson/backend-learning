Async SQLAlchemy with FastAPI

Intuition

Async SQLAlchemy lets your FastAPI server handle other requests while waiting for the database to respond, like a waiter taking new orders while the kitchen prepares existing ones instead of standing idle at the kitchen window.

Async SQLAlchemy with FastAPI

Using SQLAlchemy's async engine with FastAPI enables non-blocking database operations. This combination provides the full power of SQLAlchemy's ORM with the performance benefits of async I/O.

from sqlalchemy.ext.asyncio import (
    create_async_engine, async_sessionmaker, AsyncSession
)
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column
from sqlalchemy import select
from fastapi import FastAPI, Depends

class Base(DeclarativeBase):
    pass

class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(index=True)
    email: Mapped[str] = mapped_column(unique=True)

engine = create_async_engine("sqlite+aiosqlite:///./app.db")
async_session = async_sessionmaker(engine, expire_on_commit=False)

async def get_db():
    async with async_session() as session:
        yield session

app = FastAPI()

@app.get("/users/")
async def list_users(db: AsyncSession = Depends(get_db)):
    result = await db.execute(select(User).order_by(User.name))
    users = result.scalars().all()
    return users

Key Takeaways: - Use create_async_engine with an async-compatible driver (aiosqlite, asyncpg) - Mapped[] and mapped_column() are SQLAlchemy 2.0's typed mapping syntax - Always use expire_on_commit=False to avoid lazy loading issues after session close - Async sessions require await for all database operations

Exercises

  1. CRUD Operations with Async SQLAlchemy

    Build a complete async CRUD for a Product model (id, name, price, category). Implement list (with filtering by category), create, read by ID, update, and delete endpoints using async sessions.

    Hint

    For filtering, use select(Product).where(Product.category == category) conditionally. For update, fetch the object, modify attributes, then await session.commit().

  2. Relationship Loading Strategies

    Create Author and Book models with a one-to-many relationship. Implement an endpoint that returns authors with their books using selectinload to avoid N+1 queries. Compare with lazy loading behavior.

    Hint

    Use select(Author).options(selectinload(Author.books)) to eagerly load books in a single query. Without it, accessing author.books in async mode raises a MissingGreenlet error.

Common Problems & Solutions

1 MissingGreenlet error when accessing a relationship
Cause: SQLAlchemy async mode does not support implicit lazy loading of relationships. Accessing an unloaded relationship triggers this error.
Solution: Use eager loading strategies: `selectinload()`, `joinedload()`, or `subqueryload()` in your query's `.options()`. Alternatively, explicitly load the relationship with `await session.run_sync(lambda s: s.refresh(obj, ['relationship']))`.
2 Tables not created on application startup
Cause: The `Base.metadata.create_all()` call is synchronous but needs to run in an async context with the async engine.
Solution: Use `async with engine.begin() as conn: await conn.run_sync(Base.metadata.create_all)` in your lifespan startup handler. This runs the sync DDL operations through the async engine.
