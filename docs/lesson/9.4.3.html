Connection Pooling with PgBouncer

Intuition

Connection pooling is like a car-sharing service. Instead of everyone buying their own car (opening a connection) for occasional trips, a fleet of shared cars is maintained. You grab one when you need it and return it when done.

Connection Pooling & PgBouncer

Database connections are expensive to create (TCP handshake, TLS, authentication, memory allocation). Connection pooling reuses existing connections across requests, dramatically reducing overhead and enabling applications to scale.

Core Concepts

  • Connection Pooling: A pool of pre-established database connections shared across application threads or processes. Instead of opening a new connection per request, the application borrows one from the pool and returns it when done.
  • PgBouncer: A lightweight connection pooler for PostgreSQL that sits between the application and database. It supports three pooling modes: session, transaction, and statement.
  • Session Pooling: A connection is assigned to a client for the entire session. Safe but limits concurrency to the pool size.
  • Transaction Pooling: A connection is assigned only for the duration of a transaction, then returned to the pool. Most efficient for web workloads but incompatible with session-level features (prepared statements, LISTEN/NOTIFY).
  • Pool Sizing: The optimal pool size is typically (core_count * 2) + effective_spindle_count. Too many connections cause contention; too few cause queuing.

Key Takeaways

  1. Connection pooling is essential for any production PostgreSQL deployment serving web traffic.
  2. Transaction pooling in PgBouncer offers the best connection utilization for typical web applications.
  3. Right-size your pool: more connections does not mean better performance—it often means worse due to contention.

Exercises

  1. Set Up PgBouncer with Transaction Pooling

    Deploy PgBouncer in front of PostgreSQL with transaction pooling mode. Configure max_client_conn=200 and default_pool_size=20. Run a load test and compare connection count on PostgreSQL with and without PgBouncer.

    Hint

    Configure pgbouncer.ini with pool_mode = transaction, max_client_conn = 200, default_pool_size = 20. Monitor PostgreSQL connections with SELECT count(*) FROM pg_stat_activity.

  2. Implement Application-Level Pooling

    Using SQLAlchemy, configure a connection pool with pool_size=10, max_overflow=20, and pool_timeout=30. Write a load test that simulates 100 concurrent requests and monitor pool utilization using SQLAlchemy's pool events.

    Hint

    Use create_engine(url, pool_size=10, max_overflow=20, pool_timeout=30, pool_pre_ping=True). Listen to checkout and checkin events to track active connections.

Common Problems & Solutions

1 Application throws 'connection pool exhausted' errors under load.
Cause: All connections in the pool are in use and the overflow limit is reached. Often caused by long-running queries or connections not being returned to the pool.
Solution: Investigate long-running queries with `pg_stat_activity`. Ensure connections are properly closed/returned (use context managers). Increase pool size cautiously. Set `pool_timeout` to fail fast rather than block indefinitely.
2 Prepared statements fail with 'unnamed prepared statement does not exist' when using PgBouncer.
Cause: Transaction pooling mode reassigns server connections between transactions, invalidating server-side prepared statements.
Solution: Disable prepared statements at the client level (`prepare_threshold=0` in psycopg2), switch to session pooling if prepared statements are critical, or use PgBouncer 1.21+ which supports protocol-level prepared statement tracking.
