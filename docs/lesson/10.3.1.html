Design an E-Commerce Platform (Amazon)

Intuition

An e-commerce platform is a choreography of microservices where the checkout flow is the most critical path, requiring saga-based distributed transactions to coordinate inventory, payment, and order creation. Inventory management under concurrent access is the hardest correctness problem.

Design an E-Commerce Platform (Amazon)

Requirements

  • Support product catalog with search, browsing, and filtering
  • Handle shopping cart, checkout, and order management
  • Process millions of concurrent users during flash sales
  • Manage inventory in real-time to prevent overselling
  • Support personalized recommendations and reviews

Architecture

A microservices architecture with dedicated services for catalog, cart, order, payment, inventory, and recommendation. An API gateway routes requests and handles authentication. An event bus (Kafka) enables async communication between services. Each service owns its data store, following the database-per-service pattern.

Key Components

  • Catalog Service: Product data in Elasticsearch for full-text search, with a relational DB as source of truth
  • Cart Service: Redis-backed session store for active carts with persistence to database
  • Order Service: Orchestrates the checkout workflow using a saga pattern across inventory, payment, and shipping
  • Inventory Service: Manages stock counts with optimistic locking to prevent overselling
  • Recommendation Engine: Collaborative filtering using user purchase and browsing history

Scaling

Use CQRS to separate read and write models for the catalog. Cache product pages at the CDN level. Use event-driven inventory updates. Pre-compute and cache recommendation results. Auto-scale services independently based on load.

Trade-offs

  • Microservices enable independent scaling but introduce distributed transaction complexity (saga vs. 2PC)
  • Optimistic locking for inventory is performant but may reject valid orders under extreme contention

Exercises

  1. Implement a Checkout Saga

    Design a saga orchestrator for the checkout process: reserve inventory -> process payment -> create order -> schedule shipping. Implement compensating transactions for each step (e.g., release inventory if payment fails).

    Hint

    Use a state machine where each state represents a saga step. On failure at any step, execute compensating actions in reverse order. Persist saga state so recovery can resume after crashes.

  2. Design an Inventory System for Flash Sales

    Build an inventory management system that handles 100,000 concurrent purchase attempts for a limited-stock item (e.g., 100 units). Ensure no overselling and fair ordering. Minimize contention on the stock counter.

    Hint

    Use Redis DECR atomically to reduce stock. If the result is >= 0, the purchase succeeds. If negative, increment back and reject. This avoids database-level locking entirely for the hot path.

Common Problems & Solutions

1 During a flash sale, the inventory database becomes a bottleneck and orders are rejected or duplicated due to lock contention.
Cause: Thousands of concurrent transactions compete for row-level locks on the same inventory record, causing timeouts and deadlocks.
Solution: Move hot inventory counts to Redis with atomic DECR operations. Only write to the database asynchronously after successful Redis decrement. Use a queue to serialize database updates and prevent contention.
2 A payment service failure leaves orders in an inconsistent state: inventory reserved but payment not processed, causing phantom stock reductions.
Cause: The distributed transaction lacks proper compensation logic, so a failure mid-checkout leaves partial state across services.
Solution: Implement a saga with compensating transactions: if payment fails after inventory reservation, automatically release the reserved stock. Use a saga orchestrator that tracks state and retries or compensates on failure.
