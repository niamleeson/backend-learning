SSH, Permissions, and File System

Intuition

SSH is like having a secure private tunnel between your laptop and a remote building. The private key is your unique badge that never leaves your pocket, while the public key is a copy of your badge reader installed at the building entrance. Only your badge opens your reader.

SSH, Permissions, and File System

Secure Shell (SSH) is how you access remote servers. Understanding file permissions and the Linux filesystem hierarchy keeps your deployments secure and organized.

SSH Essentials

# Generate an SSH key pair
ssh-keygen -t ed25519 -C "dev@example.com"
# Creates ~/.ssh/id_ed25519 (private) and ~/.ssh/id_ed25519.pub (public)

# Connect to a remote server
ssh user@192.168.1.100
ssh -i ~/.ssh/mykey.pem ubuntu@ec2-instance.amazonaws.com

# Copy files securely
scp app.tar.gz user@server:/opt/apps/
rsync -avz ./deploy/ user@server:/var/www/app/

# SSH config for convenience (~/.ssh/config)
# Host myserver
#   HostName 192.168.1.100
#   User deploy
#   IdentityFile ~/.ssh/deploy_key

File Permissions

# Permission format: rwxrwxrwx (owner/group/others)
ls -la
# -rw-r--r-- 1 deploy www-data 4096 Jan 15 config.yaml

chmod 755 deploy.sh    # rwxr-xr-x (owner full, others read+execute)
chmod 600 ~/.ssh/id_ed25519  # rw------- (owner only -- REQUIRED for SSH keys)
chown deploy:www-data /var/www/app  # Change owner and group

Linux Filesystem Hierarchy

/etc/          # Configuration files (nginx.conf, crontab)
/var/log/      # Application and system logs
/var/www/      # Web server document root
/opt/          # Optional/third-party software
/tmp/          # Temporary files (cleared on reboot)
/home/user/    # User home directories
/usr/local/bin # Locally installed binaries

Common Mistakes

  • Setting SSH private keys to world-readable (must be 600 or 400)
  • Running everything as root instead of using a dedicated deploy user
  • Forgetting to add your public key to ~/.ssh/authorized_keys on the server

Key Takeaways

  • Always use SSH keys, never password auth on production servers
  • Follow the principle of least privilege for file permissions
  • Use ~/.ssh/config to simplify connections to multiple servers

Exercises

  1. SSH Key Setup

    Generate an Ed25519 SSH key pair, add it to a remote server's authorized_keys, and verify passwordless login. Then create an SSH config entry for the server.

    Hint

    Use ssh-keygen, ssh-copy-id, and edit ~/.ssh/config.

  2. Permission Audit

    Write a find command that locates all world-writable files (o+w) in /var/www/ and generates a report. Then fix the permissions to be owner-writable only.

    Hint

    Use find /var/www -perm -o+w and chmod o-w.

Common Problems & Solutions

1 SSH connection refused
Symptoms: ['ssh: connect to host X port 22: Connection refused']
Cause: SSH daemon is not running or is on a non-standard port, or a firewall is blocking port 22.
Solution: Check the SSH service status: `sudo systemctl status sshd`. Verify the port in /etc/ssh/sshd_config. Check firewall rules: `sudo ufw status` or `sudo iptables -L`.
2 Permissions too open error for SSH key
Symptoms: ['WARNING: UNPROTECTED PRIVATE KEY FILE!', "Permissions 0644 for '~/.ssh/id_ed25519' are too open."]
Cause: Your private key file has permissions that are too permissive.
Solution: Run `chmod 600 ~/.ssh/id_ed25519` to restrict to owner-only read/write.

chmod 600 ~/.ssh/id_ed25519

3 Cannot write to /var/www as deploy user
Cause: The directory is owned by root and the deploy user lacks write permission.
Solution: Change ownership: `sudo chown -R deploy:www-data /var/www/app` and set group-write: `chmod -R g+w /var/www/app`.
