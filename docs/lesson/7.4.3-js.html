File Upload Security and Content Validation

Intuition

Think of file upload validation like airport security. The client-provided filename and Content-Type are like the label on your suitcase -- anyone can write anything on it. The real security comes from opening the suitcase (reading magic bytes), running it through the X-ray machine (content validation with sharp/file-type), and checking for prohibited items (malware, scripts). Just like airports do not let you carry your bag directly to the plane, you should never let uploaded files land directly in a publicly accessible directory.

File Upload Security and Content Validation

Why File Uploads Are Dangerous

File uploads are one of the most exploited attack surfaces in web applications. A poorly secured upload endpoint can lead to remote code execution, denial of service, server compromise, and data exfiltration. Unlike text input validation where you check strings and numbers, file uploads require you to validate binary content, enforce size limits, verify MIME types, scan for malware signatures, and store files safely.

Setting Up Multer for File Uploads

Multer is the standard multipart/form-data middleware for Express:

npm install multer
npm install --save-dev @types/multer  # TypeScript types

import express from 'express';
import multer from 'multer';
import path from 'node:path';
import crypto from 'node:crypto';

const app = express();

// INSECURE: never use the original filename directly
// const upload = multer({ dest: 'uploads/' });

// SECURE: custom storage with sanitized filenames
const storage = multer.diskStorage({
  destination: (req, file, cb) => {
    cb(null, '/var/uploads/tmp');
  },
  filename: (req, file, cb) => {
    // Generate a random filename, preserve extension
    const ext = path.extname(file.originalname).toLowerCase();
    const randomName = crypto.randomBytes(16).toString('hex');
    cb(null, `${randomName}${ext}`);
  },
});

// Configure limits
const upload = multer({
  storage,
  limits: {
    fileSize: 5 * 1024 * 1024,     // 5MB max
    files: 5,                        // max 5 files per request
    fields: 10,                      // max 10 non-file fields
    fieldNameSize: 100,              // max field name length
    fieldSize: 1024,                 // max field value size
  },
  fileFilter: (req, file, cb) => {
    // Whitelist allowed MIME types
    const allowedMimes = [
      'image/jpeg',
      'image/png',
      'image/gif',
      'image/webp',
      'application/pdf',
    ];

    if (allowedMimes.includes(file.mimetype)) {
      cb(null, true);
    } else {
      cb(new Error(`File type ${file.mimetype} is not allowed`));
    }
  },
});

Why MIME Type Checking Is Not Enough

The file.mimetype from Multer comes from the Content-Type header set by the client. An attacker can trivially set this to image/jpeg while uploading a PHP script. You must validate the actual file content.

│  Client sends  │     │  Multer checks  │     │  You must check  │
│  Content-Type  │────►│  file.mimetype  │────►│  actual bytes    │
│  (untrusted!)  │     │  (untrusted!)   │     │  (magic numbers) │

Magic Number Validation

Every file format has a signature (magic bytes) at the beginning of the file. Use the file-type library to detect the real type:

npm install file-type

import { fileTypeFromBuffer, fileTypeFromFile } from 'file-type';
import fs from 'node:fs/promises';

async function validateFileContent(filePath, allowedTypes) {
  const fileType = await fileTypeFromFile(filePath);

  if (!fileType) {
    throw new Error('Unable to determine file type');
  }

  if (!allowedTypes.includes(fileType.mime)) {
    throw new Error(
      `File content is ${fileType.mime}, expected one of: ${allowedTypes.join(', ')}`
    );
  }

  return fileType;
}

// Usage
const allowedImageTypes = ['image/jpeg', 'image/png', 'image/gif', 'image/webp'];

try {
  const type = await validateFileContent('/var/uploads/tmp/abc123.jpg', allowedImageTypes);
  console.log(`Validated: ${type.mime} (.${type.ext})`);
} catch (err) {
  // Delete the suspicious file
  await fs.unlink('/var/uploads/tmp/abc123.jpg');
  console.error(`Rejected: ${err.message}`);
}

Filename Sanitization

Filenames from clients can contain path traversal attacks, null bytes, and special characters:

import path from 'node:path';
import crypto from 'node:crypto';

function sanitizeFilename(originalName) {
  // Remove path components (prevent ../../etc/passwd)
  const basename = path.basename(originalName);

  // Remove null bytes
  const cleaned = basename.replace(/\0/g, '');

  // Extract and validate extension
  const ext = path.extname(cleaned).toLowerCase();
  const allowedExtensions = ['.jpg', '.jpeg', '.png', '.gif', '.webp', '.pdf'];

  if (!allowedExtensions.includes(ext)) {
    throw new Error(`Extension ${ext} is not allowed`);
  }

  // Generate a safe filename
  const safeName = crypto.randomBytes(16).toString('hex');
  return `${safeName}${ext}`;
}

// Dangerous inputs and their sanitized outputs:
// '../../etc/passwd'           -> Error (no allowed extension)
// 'shell.php.jpg'              -> 'a1b2c3d4e5f6...jpg'
// 'photo.jpg\0.php'            -> 'a1b2c3d4e5f6...php' -> Error
// '../../../tmp/evil.jpg'      -> 'a1b2c3d4e5f6...jpg'

Image-Specific Validation with Sharp

For image uploads, use sharp to validate and re-process images, which strips any embedded malicious content:

npm install sharp

import sharp from 'sharp';
import fs from 'node:fs/promises';

async function processAndValidateImage(inputPath, outputPath, options = {}) {
  const {
    maxWidth = 2048,
    maxHeight = 2048,
    maxSizeBytes = 5 * 1024 * 1024,
    quality = 85,
  } = options;

  // Check file size first
  const stats = await fs.stat(inputPath);
  if (stats.size > maxSizeBytes) {
    throw new Error(`File size ${stats.size} exceeds maximum ${maxSizeBytes}`);
  }

  try {
    const metadata = await sharp(inputPath).metadata();

    // Validate dimensions
    if (metadata.width > maxWidth * 4 || metadata.height > maxHeight * 4) {
      throw new Error('Image dimensions are unreasonably large');
    }

    // Re-encode the image (strips EXIF, embedded scripts, etc.)
    await sharp(inputPath)
      .resize(maxWidth, maxHeight, {
        fit: 'inside',
        withoutEnlargement: true,
      })
      .jpeg({ quality, mozjpeg: true })  // or .png(), .webp()
      .toFile(outputPath);

    // Delete the original unprocessed file
    await fs.unlink(inputPath);

    return {
      width: metadata.width,
      height: metadata.height,
      format: metadata.format,
      processedPath: outputPath,
    };
  } catch (err) {
    // If sharp cannot process it, it is not a valid image
    await fs.unlink(inputPath).catch(() => {});
    throw new Error(`Invalid image file: ${err.message}`);
  }
}

Re-encoding through sharp is one of the best defenses: if a file is not truly a valid image, sharp will fail to process it. And even if the image contains embedded malicious payloads in metadata or trailing bytes, re-encoding strips them.

Complete Secure Upload Endpoint

import express from 'express';
import multer from 'multer';
import sharp from 'sharp';
import { fileTypeFromFile } from 'file-type';
import crypto from 'node:crypto';
import path from 'node:path';
import fs from 'node:fs/promises';

const UPLOAD_DIR = '/var/uploads';
const TEMP_DIR = path.join(UPLOAD_DIR, 'tmp');
const FINAL_DIR = path.join(UPLOAD_DIR, 'images');

// Ensure directories exist
await fs.mkdir(TEMP_DIR, { recursive: true });
await fs.mkdir(FINAL_DIR, { recursive: true });

const upload = multer({
  storage: multer.diskStorage({
    destination: TEMP_DIR,
    filename: (req, file, cb) => {
      const name = crypto.randomBytes(16).toString('hex');
      const ext = path.extname(file.originalname).toLowerCase();
      cb(null, `${name}${ext}`);
    },
  }),
  limits: {
    fileSize: 10 * 1024 * 1024,  // 10MB raw upload limit
    files: 1,
  },
  fileFilter: (req, file, cb) => {
    const allowed = ['image/jpeg', 'image/png', 'image/webp', 'image/gif'];
    if (allowed.includes(file.mimetype)) {
      cb(null, true);
    } else {
      cb(new multer.MulterError('LIMIT_UNEXPECTED_FILE', file.fieldname));
    }
  },
});

const app = express();

app.post('/api/upload/avatar', upload.single('avatar'), async (req, res) => {
  if (!req.file) {
    return res.status(400).json({ error: 'No file uploaded' });
  }

  const tempPath = req.file.path;

  try {
    // Step 1: Validate magic bytes
    const detectedType = await fileTypeFromFile(tempPath);
    const allowedMimes = ['image/jpeg', 'image/png', 'image/webp', 'image/gif'];

    if (!detectedType || !allowedMimes.includes(detectedType.mime)) {
      await fs.unlink(tempPath);
      return res.status(400).json({
        error: 'Invalid file type',
        detail: `Detected: ${detectedType?.mime || 'unknown'}`,
      });
    }

    // Step 2: Process with sharp (validates, resizes, strips metadata)
    const outputName = `${crypto.randomBytes(16).toString('hex')}.jpg`;
    const outputPath = path.join(FINAL_DIR, outputName);

    const metadata = await sharp(tempPath)
      .resize(800, 800, { fit: 'cover' })
      .jpeg({ quality: 85 })
      .toFile(outputPath);

    // Step 3: Clean up temp file
    await fs.unlink(tempPath);

    // Step 4: Return the URL
    res.status(201).json({
      url: `/uploads/images/${outputName}`,
      width: metadata.width,
      height: metadata.height,
      size: metadata.size,
    });
  } catch (err) {
    // Clean up on any error
    await fs.unlink(tempPath).catch(() => {});
    res.status(500).json({ error: 'Upload processing failed' });
  }
});

// Handle Multer errors
app.use((err, req, res, next) => {
  if (err instanceof multer.MulterError) {
    const messages = {
      LIMIT_FILE_SIZE: 'File is too large (max 10MB)',
      LIMIT_FILE_COUNT: 'Too many files',
      LIMIT_UNEXPECTED_FILE: 'Unexpected file type',
    };
    return res.status(400).json({
      error: messages[err.code] || err.message,
    });
  }
  next(err);
});

Preventing Zip Bombs and Decompression Attacks

If you accept ZIP, GZIP, or other compressed files, validate the decompressed size before extracting:

import { createReadStream } from 'node:fs';
import { createGunzip } from 'node:zlib';
import { pipeline } from 'node:stream/promises';

async function safeDecompress(inputPath, maxDecompressedSize = 100 * 1024 * 1024) {
  let totalSize = 0;

  const sizeChecker = new (await import('node:stream')).Transform({
    transform(chunk, encoding, callback) {
      totalSize += chunk.length;
      if (totalSize > maxDecompressedSize) {
        callback(new Error(
          `Decompressed size exceeds limit of ${maxDecompressedSize} bytes (zip bomb?)`
        ));
        return;
      }
      callback(null, chunk);
    },
  });

  try {
    await pipeline(
      createReadStream(inputPath),
      createGunzip(),
      sizeChecker,
      // ... write stream or further processing
    );
  } catch (err) {
    throw new Error(`Decompression failed: ${err.message}`);
  }
}

SVG Sanitization

SVG files are XML and can contain embedded JavaScript, external references, and other attack vectors:

import { JSDOM } from 'jsdom';
import createDOMPurify from 'dompurify';

function sanitizeSVG(svgContent) {
  const window = new JSDOM('').window;
  const DOMPurify = createDOMPurify(window);

  const clean = DOMPurify.sanitize(svgContent, {
    USE_PROFILES: { svg: true, svgFilters: true },
    FORBID_TAGS: ['script', 'foreignObject', 'use'],
    FORBID_ATTR: ['onload', 'onerror', 'onclick', 'onmouseover', 'xlink:href'],
  });

  return clean;
}

Storing Files Securely

┌──────────────────────────────────────────────────────┐
│              Secure File Storage Strategy             │
├──────────────────────────────────────────────────────┤
│                                                      │
│  DO:                                                 │
│  - Store outside the web root                        │
│  - Use random filenames (no user input)              │
│  - Serve through a controller (access control)       │
│  - Use a CDN or object storage (S3) for production   │
│  - Set Content-Disposition: attachment for downloads  │
│  - Set X-Content-Type-Options: nosniff               │
│                                                      │
│  DON'T:                                              │
│  - Store in public/ or static/ directory             │
│  - Use original filenames from users                 │
│  - Serve uploaded files without Content-Type check   │
│  - Allow directory listing on upload directories     │
│  - Trust the Content-Type header from the client     │
│                                                      │
└──────────────────────────────────────────────────────┘

// Serve files through a controlled endpoint
app.get('/api/files/:id', authenticate, async (req, res) => {
  const file = await db.file.findUnique({ where: { id: req.params.id } });

  if (!file) return res.status(404).json({ error: 'File not found' });

  // Check access permissions
  if (file.userId !== req.user.id && req.user.role !== 'admin') {
    return res.status(403).json({ error: 'Access denied' });
  }

  // Set security headers
  res.set({
    'Content-Type': file.mimeType,
    'Content-Disposition': `attachment; filename="${file.originalName}"`,
    'X-Content-Type-Options': 'nosniff',
    'Cache-Control': 'private, max-age=3600',
  });

  const stream = fs.createReadStream(file.storagePath);
  stream.pipe(res);
});

Upload to S3 with Validation

In production, use object storage instead of local disk:

npm install @aws-sdk/client-s3

import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
import crypto from 'node:crypto';

const s3 = new S3Client({ region: process.env.AWS_REGION });

async function uploadToS3(fileBuffer, originalName, mimeType) {
  const key = `uploads/${crypto.randomBytes(16).toString('hex')}${path.extname(originalName)}`;

  await s3.send(new PutObjectCommand({
    Bucket: process.env.S3_BUCKET,
    Key: key,
    Body: fileBuffer,
    ContentType: mimeType,
    ContentDisposition: 'attachment',
    ServerSideEncryption: 'AES256',
    Metadata: {
      'original-name': encodeURIComponent(originalName),
    },
  }));

  return {
    key,
    url: `https://${process.env.S3_BUCKET}.s3.${process.env.AWS_REGION}.amazonaws.com/${key}`,
  };
}

Summary

File upload security requires defense in depth: limit file sizes at the web server and application level, validate MIME types by reading magic bytes (not trusting headers), sanitize filenames, re-encode images through sharp to strip embedded payloads, protect against zip bombs, store files outside the web root with random names, serve them through controlled endpoints with proper headers, and in production use object storage like S3. Never trust any aspect of a file upload from the client.

Common Problems & Solutions

1 Uploaded images are served with wrong Content-Type, causing the browser to download instead of display
Cause: The serving endpoint is not setting the Content-Type header based on the actual file type, or it is always setting Content-Disposition: attachment.
Solution: When serving images for display (like avatars), use Content-Disposition: inline and set the correct Content-Type based on the stored MIME type. For downloads, use Content-Disposition: attachment. Always set X-Content-Type-Options: nosniff to prevent MIME sniffing.
2 Temp files accumulate and fill up disk space
Cause: When upload processing fails (validation error, sharp crash, S3 timeout), the temporary file is not cleaned up.
Solution: Always wrap upload processing in try/catch/finally and delete the temp file in the finally block. Additionally, set up a cron job or periodic cleanup that removes files older than 1 hour from the temp directory: `find /var/uploads/tmp -mmin +60 -delete`.
3 CORS errors when uploading from the frontend
Cause: The browser sends a preflight OPTIONS request for multipart uploads, and the server does not handle it.
Solution: Configure CORS middleware before your upload routes: `app.use(cors({ origin: 'https://yourfrontend.com', methods: ['POST'], allowedHeaders: ['Content-Type', 'Authorization'] }))`. Make sure the CORS middleware handles OPTIONS requests.
