Standardized Error Responses (RFC 7807)

Intuition

RFC 7807 is like standardized shipping labels. Without a standard, every warehouse puts the address in a different spot and you waste time figuring out where to look. With a standard label, any handler anywhere in the world can immediately find the destination, return address, and tracking number in the same place every time.

Standardized Error Responses (RFC 7807)

Inconsistent error responses are one of the most common API design failures. RFC 7807 (Problem Details for HTTP APIs) defines a standard format that makes errors machine-readable and developer-friendly.

The Problem with Ad-Hoc Errors

Without a standard, every endpoint returns errors differently:

// Endpoint A
{ "error": "Not found" }

// Endpoint B
{ "message": "User not found", "code": 404 }

// Endpoint C
{ "errors": [{ "msg": "Invalid email", "field": "email" }] }

// Endpoint D
"Something went wrong"

Clients cannot reliably parse errors, write generic error handlers, or provide good user feedback.

RFC 7807: Problem Details

The standard defines these fields:

interface ProblemDetails {
  type: string;       // URI identifying the error type (default: "about:blank")
  title: string;      // Short, human-readable summary
  status: number;     // HTTP status code
  detail?: string;    // Human-readable explanation specific to this occurrence
  instance?: string;  // URI reference identifying this specific occurrence
  [key: string]: any; // Extension fields
}

Example:

{
  "type": "https://api.example.com/errors/insufficient-funds",
  "title": "Insufficient Funds",
  "status": 422,
  "detail": "Your account balance of $10.00 is insufficient for the $25.00 purchase.",
  "instance": "/transactions/abc123",
  "balance": 10.00,
  "required": 25.00
}

The Content-Type should be application/problem+json.

Implementation in Express

// errors/problem-details.ts
export class ProblemDetails extends Error {
  public readonly type: string;
  public readonly title: string;
  public readonly status: number;
  public readonly detail?: string;
  public readonly instance?: string;
  public readonly extensions: Record<string, any>;

  constructor(opts: {
    type?: string;
    title: string;
    status: number;
    detail?: string;
    instance?: string;
    extensions?: Record<string, any>;
  }) {
    super(opts.title);
    this.type = opts.type || 'about:blank';
    this.title = opts.title;
    this.status = opts.status;
    this.detail = opts.detail;
    this.instance = opts.instance;
    this.extensions = opts.extensions || {};
  }

  toJSON() {
    return {
      type: this.type,
      title: this.title,
      status: this.status,
      ...(this.detail && { detail: this.detail }),
      ...(this.instance && { instance: this.instance }),
      ...this.extensions,
    };
  }
}

// Predefined error factories
export const Errors = {
  notFound: (resource: string, id: string | number) =>
    new ProblemDetails({
      type: 'https://api.example.com/errors/not-found',
      title: 'Resource Not Found',
      status: 404,
      detail: `${resource} with id '${id}' was not found.`,
      instance: `/api/${resource.toLowerCase()}s/${id}`,
    }),

  validationFailed: (errors: Array<{ field: string; message: string }>) =>
    new ProblemDetails({
      type: 'https://api.example.com/errors/validation-failed',
      title: 'Validation Failed',
      status: 422,
      detail: 'One or more fields failed validation.',
      extensions: { errors },
    }),

  unauthorized: (detail?: string) =>
    new ProblemDetails({
      type: 'https://api.example.com/errors/unauthorized',
      title: 'Unauthorized',
      status: 401,
      detail: detail || 'Authentication is required to access this resource.',
    }),

  forbidden: (detail?: string) =>
    new ProblemDetails({
      type: 'https://api.example.com/errors/forbidden',
      title: 'Forbidden',
      status: 403,
      detail: detail || 'You do not have permission to perform this action.',
    }),

  conflict: (detail: string) =>
    new ProblemDetails({
      type: 'https://api.example.com/errors/conflict',
      title: 'Conflict',
      status: 409,
      detail,
    }),

  rateLimited: (retryAfter: number) =>
    new ProblemDetails({
      type: 'https://api.example.com/errors/rate-limited',
      title: 'Too Many Requests',
      status: 429,
      detail: 'You have exceeded the rate limit.',
      extensions: { retryAfter },
    }),
};

Error Handling Middleware

import { Request, Response, NextFunction } from 'express';

function errorHandler(err: Error, req: Request, res: Response, next: NextFunction) {
  // If it's already a ProblemDetails, send it
  if (err instanceof ProblemDetails) {
    return res
      .status(err.status)
      .type('application/problem+json')
      .json(err.toJSON());
  }

  // Handle Zod validation errors
  if (err.name === 'ZodError') {
    const zodErr = err as any;
    const problem = Errors.validationFailed(
      zodErr.issues.map((issue: any) => ({
        field: issue.path.join('.'),
        message: issue.message,
      }))
    );
    return res
      .status(problem.status)
      .type('application/problem+json')
      .json(problem.toJSON());
  }

  // Handle Prisma errors
  if (err.constructor.name === 'PrismaClientKnownRequestError') {
    const prismaErr = err as any;
    if (prismaErr.code === 'P2002') {
      const fields = prismaErr.meta?.target?.join(', ');
      const problem = Errors.conflict(
        `A record with this ${fields} already exists.`
      );
      return res.status(409).type('application/problem+json').json(problem.toJSON());
    }
    if (prismaErr.code === 'P2025') {
      const problem = Errors.notFound('Record', 'unknown');
      return res.status(404).type('application/problem+json').json(problem.toJSON());
    }
  }

  // Generic 500 error (never expose stack traces in production)
  console.error('Unhandled error:', err);
  const problem = new ProblemDetails({
    type: 'https://api.example.com/errors/internal',
    title: 'Internal Server Error',
    status: 500,
    detail: process.env.NODE_ENV === 'production'
      ? 'An unexpected error occurred.'
      : err.message,
  });

  res.status(500).type('application/problem+json').json(problem.toJSON());
}

// Register as last middleware
app.use(errorHandler);

Using in Routes

app.get('/api/users/:id', async (req, res, next) => {
  try {
    const user = await prisma.user.findUnique({
      where: { id: parseInt(req.params.id) },
    });

    if (!user) {
      throw Errors.notFound('User', req.params.id);
    }

    res.json({ data: user });
  } catch (err) {
    next(err);
  }
});

app.post('/api/users', async (req, res, next) => {
  try {
    const result = userSchema.safeParse(req.body);

      throw Errors.validationFailed(
        result.error.issues.map(i => ({
          field: i.path.join('.'),
          message: i.message,
        }))
      );
    }

    const user = await prisma.user.create({ data: result.data });
    res.status(201).json({ data: user });
  } catch (err) {
    next(err);
  }
});

NestJS Implementation

NestJS has built-in exception filters that can be customized:

import { ExceptionFilter, Catch, ArgumentsHost, HttpException } from '@nestjs/common';

@Catch()
export class ProblemDetailsFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse();
    const request = ctx.getRequest();

    let problem: any;

    if (exception instanceof ProblemDetails) {
      problem = exception.toJSON();
    } else if (exception instanceof HttpException) {
      const status = exception.getStatus();
      const exResponse = exception.getResponse();
      problem = {
        type: 'about:blank',
        title: typeof exResponse === 'string' ? exResponse : (exResponse as any).error,
        status,
        detail: typeof exResponse === 'string' ? exResponse : (exResponse as any).message,
        instance: request.url,
      };
    } else {
      problem = {
        type: 'https://api.example.com/errors/internal',
        title: 'Internal Server Error',
        status: 500,
        detail: 'An unexpected error occurred.',
        instance: request.url,
      };
    }

    response
      .status(problem.status)
      .header('Content-Type', 'application/problem+json')
      .json(problem);
  }
}

// Register globally in main.ts
app.useGlobalFilters(new ProblemDetailsFilter());

Validation Error Format

For validation errors, include a structured errors array:

{
  "type": "https://api.example.com/errors/validation-failed",
  "title": "Validation Failed",
  "status": 422,
  "detail": "One or more fields failed validation.",
  "errors": [
    {
      "field": "email",
      "message": "Must be a valid email address"
    },
    {
      "field": "password",
      "message": "Must be at least 8 characters"
    },
    {
      "field": "age",
      "message": "Must be a positive integer"
    }
  ]
}

Error Type Registry

Maintain a page/document listing all your error types:

https://api.example.com/errors/not-found
https://api.example.com/errors/validation-failed
https://api.example.com/errors/unauthorized
https://api.example.com/errors/forbidden
https://api.example.com/errors/conflict
https://api.example.com/errors/rate-limited
https://api.example.com/errors/insufficient-funds

Each URI should resolve to a human-readable page explaining the error, possible causes, and how to fix it.

Exercises

  1. Build a ProblemDetails Error Library

    Create a reusable npm-style module with a ProblemDetails class and factory functions for common HTTP errors (400, 401, 403, 404, 409, 422, 429, 500). Include proper TypeScript types and a toJSON() method. Write tests to verify the output structure.

    Hint

    Each factory function should accept context-specific parameters (e.g., notFound takes resource type and id) and return a ProblemDetails instance.

  2. Integrate with Zod Validation

    Write Express middleware that catches Zod validation errors and transforms them into RFC 7807 responses with a structured errors array. Each validation issue should map to { field, message, code }.

    Hint

    Zod errors have an .issues array with path (string[]) and message properties.

  3. NestJS Global Exception Filter

    Implement a NestJS exception filter that catches all exceptions (HttpException, ProblemDetails, PrismaClientKnownRequestError, and generic Error) and returns RFC 7807 formatted responses. Register it globally.

    Hint

    Use @Catch() with no arguments to catch all exceptions. Check instanceof for each error type.

  4. Error Type Documentation Page

    Create an endpoint GET /errors/:type that returns HTML documentation for each error type. When a client follows the 'type' URI from an error response, they should see a description, possible causes, and example resolution steps.

    Hint

    Store error type docs in a Map and render them as HTML.

Common Problems & Solutions

1 Stack traces leak to production clients
Cause: The error handler includes err.stack or err.message from unexpected errors in the response.
Solution: Only include detailed error messages in development. In production, return a generic 'An unexpected error occurred' detail and log the actual error server-side.
2 Validation errors return inconsistent formats
Cause: Different validation libraries (Zod, Joi, class-validator) produce different error structures.
Solution: Normalize all validation errors in a single error handler into the standard RFC 7807 format with a consistent errors array.
3 The 'type' URI in error responses returns 404
Cause: The error type URIs are defined but no documentation page exists at that URL.
Solution: Either host actual documentation at each error type URI, or use 'about:blank' as the type (which is the RFC default for errors where the type URI is just the HTTP status definition).
