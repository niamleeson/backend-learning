Database Transactions in NestJS

Intuition

A transaction is like a group photo. Everyone must be in the frame for the photo to be taken (commit). If anyone blinks or looks away (error), you retake the whole photo (rollback). You don't accept a photo where half the group is missing -- it's all or nothing.

Database Transactions in NestJS

Introduction

Transactions ensure that a series of database operations either all succeed or all fail together -- the ACID property of atomicity. Without transactions, a failure midway through a multi-step operation can leave your database in an inconsistent state. This lesson covers implementing transactions with both TypeORM and Prisma in NestJS.

Why Transactions Matter

Consider a money transfer:

1. Deduct $100 from Account A  -- SUCCESS
2. Add $100 to Account B       -- FAILURE (database timeout)

Without a transaction: Account A lost $100, Account B gained nothing.
With a transaction: Both operations are rolled back. No money lost.

Transactions with Prisma

Interactive Transactions

The most common pattern -- a callback that receives a transaction client:

@Injectable()
export class TransferService {
  constructor(private prisma: PrismaService) {}

  async transfer(fromId: string, toId: string, amount: number) {
    return this.prisma.$transaction(async (tx) => {
      // All operations inside use the same transaction
      const sender = await tx.account.update({
        where: { id: fromId },
        data: { balance: { decrement: amount } },
      });

      if (sender.balance < 0) {
        throw new BadRequestException('Insufficient funds');
        // Transaction automatically rolls back on throw
      }

      const receiver = await tx.account.update({
        where: { id: toId },
        data: { balance: { increment: amount } },
      });

      // Log the transaction
      await tx.transferLog.create({
        data: {
          fromAccountId: fromId,
          toAccountId: toId,
          amount,
          status: 'COMPLETED',
        },
      });

      return { sender, receiver };
    });
  }
}

Transaction Options

await this.prisma.$transaction(
  async (tx) => {
    // ... operations
  },
  {
    maxWait: 5000,     // Max time to wait for a connection (ms)
    timeout: 10000,    // Max transaction duration (ms)
    isolationLevel: Prisma.TransactionIsolationLevel.Serializable,
  },
);

Batch Transactions

For simple operations that don't depend on each other:

const [users, posts, tags] = await this.prisma.$transaction([
  this.prisma.user.findMany(),
  this.prisma.post.findMany(),
  this.prisma.tag.findMany(),
]);

Nested Operations in Transactions

async createOrderWithItems(dto: CreateOrderDto) {
  return this.prisma.$transaction(async (tx) => {
    // Create the order
    const order = await tx.order.create({
      data: {
        userId: dto.userId,
        status: 'PENDING',
      },
    });

    // Create order items and update stock
    for (const item of dto.items) {
      const product = await tx.product.findUnique({
        where: { id: item.productId },
      });

      if (!product || product.stock < item.quantity) {
        throw new BadRequestException(
          `Insufficient stock for product ${item.productId}`,
        );
      }

      await tx.orderItem.create({
        data: {
          orderId: order.id,
          productId: item.productId,
          quantity: item.quantity,
          price: product.price,
        },
      });

      await tx.product.update({
        where: { id: item.productId },
        data: { stock: { decrement: item.quantity } },
      });
    }

    // Calculate total
    const total = dto.items.reduce((sum, item) => {
      return sum + item.quantity * item.price;
    }, 0);

    return tx.order.update({
      where: { id: order.id },
      data: { total, status: 'CONFIRMED' },
      include: { items: true },
    });
  });
}

Transactions with TypeORM

QueryRunner Transactions

@Injectable()
export class TransferService {
  constructor(private dataSource: DataSource) {}

  async transfer(fromId: string, toId: string, amount: number) {
    const queryRunner = this.dataSource.createQueryRunner();
    await queryRunner.connect();
    await queryRunner.startTransaction();

    try {
      const sender = await queryRunner.manager.findOne(Account, {
        where: { id: fromId },
      });

      if (!sender || sender.balance < amount) {
        throw new BadRequestException('Insufficient funds');
      }

      await queryRunner.manager.update(Account, fromId, {
        balance: () => `balance - ${amount}`,
      });

      await queryRunner.manager.update(Account, toId, {
        balance: () => `balance + ${amount}`,
      });

      await queryRunner.manager.save(TransferLog, {
        fromAccountId: fromId,
        toAccountId: toId,
        amount,
        status: 'COMPLETED',
      });

      await queryRunner.commitTransaction();
    } catch (error) {
      await queryRunner.rollbackTransaction();
      throw error;
    } finally {
      await queryRunner.release();
    }
  }
}

Transaction Decorator

TypeORM also supports a simpler transaction pattern:

@Injectable()
export class OrdersService {
  constructor(private dataSource: DataSource) {}

  async createOrder(dto: CreateOrderDto) {
    return this.dataSource.transaction(async (manager) => {
      const order = manager.create(Order, {
        userId: dto.userId,
        status: 'PENDING',
      });
      await manager.save(order);

      for (const item of dto.items) {
        await manager.decrement(Product, { id: item.productId }, 'stock', item.quantity);
        await manager.save(OrderItem, {
          orderId: order.id,
          productId: item.productId,
          quantity: item.quantity,
        });
      }

      return order;
    });
  }
}

Transaction Isolation Levels

+--------------------+-------------------+------------------+----------------+
| Isolation Level    | Dirty Read        | Non-repeatable   | Phantom Read   |
|                    |                   | Read             |                |
+--------------------+-------------------+------------------+----------------+
| Read Uncommitted   | Possible          | Possible         | Possible       |
| Read Committed     | Not possible      | Possible         | Possible       |
| Repeatable Read    | Not possible      | Not possible     | Possible       |
| Serializable       | Not possible      | Not possible     | Not possible   |
+--------------------+-------------------+------------------+----------------+
                                                                  ^
                                            Most strict, lowest concurrency

PostgreSQL default is Read Committed. Use Serializable for critical operations like financial transactions.

Optimistic Locking

Instead of database locks, use a version column:

// With Prisma
async updateWithOptimisticLock(id: string, data: any, expectedVersion: number) {
  const result = await this.prisma.product.updateMany({
    where: {
      id,
      version: expectedVersion, // Only update if version matches
    },
    data: {
      ...data,
      version: { increment: 1 },
    },
  });

  if (result.count === 0) {
    throw new ConflictException(
      'Record was modified by another transaction. Please retry.',
    );
  }

  return this.prisma.product.findUnique({ where: { id } });
}

Transaction Patterns in NestJS

Service Method Transaction

@Injectable()
export class OrdersService {
  constructor(private prisma: PrismaService) {}

  // Each public method manages its own transaction
  async placeOrder(dto: PlaceOrderDto) {
    return this.prisma.$transaction(async (tx) => {
      // All logic inside the transaction
    });
  }
}

Passing Transaction Client

@Injectable()
export class UsersService {
  constructor(private prisma: PrismaService) {}

  // Accept optional transaction client
  async create(
    data: Prisma.UserCreateInput,
    tx?: Prisma.TransactionClient,
  ) {
    const client = tx || this.prisma;
    return client.user.create({ data });
  }
}

@Injectable()
export class RegistrationService {
  constructor(
    private prisma: PrismaService,
    private usersService: UsersService,
    private profileService: ProfileService,
  ) {}

  async register(dto: RegisterDto) {
    return this.prisma.$transaction(async (tx) => {
      const user = await this.usersService.create(
        { email: dto.email, password: dto.hashedPassword },
        tx, // Pass transaction client
      );

      await this.profileService.create(
        { userId: user.id, name: dto.name },
        tx,
      );

      return user;
    });
  }
}

Retry Logic for Transactions

async executeWithRetry<T>(
  fn: () => Promise<T>,
  maxRetries = 3,
  delay = 100,
): Promise<T> {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (error) {
      const isRetryable =
        error instanceof Prisma.PrismaClientKnownRequestError &&
        error.code === 'P2034'; // Transaction conflict

      if (!isRetryable || attempt === maxRetries) throw error;

      await new Promise((resolve) =>
        setTimeout(resolve, delay * Math.pow(2, attempt - 1)),
      );
    }
  }
  throw new Error('Max retries exceeded');
}

// Usage
async transfer(fromId: string, toId: string, amount: number) {
  return this.executeWithRetry(() =>
    this.prisma.$transaction(async (tx) => {
      // ... transfer logic
    }),
  );
}

Common Pitfalls

  1. Long-running transactions -- Hold locks for as little time as possible. Don't do HTTP calls or heavy computation inside a transaction.
  2. Forgetting to roll back on error -- With TypeORM QueryRunner, always use try/catch/finally. Prisma handles this automatically with interactive transactions.
  3. Deadlocks -- Two transactions waiting for each other's locks. Always access tables and rows in a consistent order.
  4. Not handling serialization failures -- With Serializable isolation, transactions can fail due to conflicts. Implement retry logic.

Exercises

  1. Implement a Money Transfer with Prisma Transactions

    Build a banking service with Account and Transfer models. Implement a transfer method that: debits the sender, credits the receiver, logs the transfer, and rolls back everything if any step fails. Add insufficient balance validation inside the transaction.

    Hint

    Use prisma.$transaction(async (tx) => { ... }). Check balance after decrement to handle race conditions. Throw inside the transaction callback to trigger rollback.

  2. Build an Order System with Inventory Management

    Create an order placement system that: creates an order, creates order items, decrements product stock for each item, calculates the total, and updates the order status. Everything must be atomic. Handle out-of-stock scenarios gracefully.

    Hint

    Query the product inside the transaction to get current stock. If stock < requested quantity, throw BadRequestException. The entire transaction rolls back, and no stock is decremented.

  3. Implement Optimistic Locking

    Add a version column to a Product entity. Implement an update method that uses optimistic locking -- it only succeeds if the version matches. If it doesn't match (another update happened), throw a ConflictException and let the client retry.

    Hint

    Use updateMany with where: { id, version: expectedVersion } and check result.count. If 0, the version didn't match. Increment version on successful update.

Common Problems & Solutions

1 Transaction deadlock between two operations
Symptoms: Error: deadlock detected. One of the transactions is rolled back by the database.
Cause: Two transactions lock rows in different orders: Transaction A locks Row 1 then waits for Row 2, while Transaction B locks Row 2 and waits for Row 1.
Solution: Always lock resources in a consistent order (e.g., by ID ascending). If deadlocks still occur, implement retry logic with exponential backoff.
2 Prisma transaction times out
Cause: The transaction takes longer than the default timeout (5 seconds), or it's waiting for a lock held by another transaction.
Solution: Increase the timeout option in $transaction(). Keep transactions short -- move non-database work outside. Check for lock contention with database monitoring tools.
3 Data inconsistency despite using transactions
Cause: Some operations are happening outside the transaction context (e.g., using this.prisma instead of the tx client inside the callback).
Solution: Ensure ALL operations inside a $transaction callback use the `tx` parameter, not the original PrismaService. Pass `tx` to called methods that need to participate in the transaction.
