Prisma Studio and Debugging Queries

Intuition

Prisma Studio and query logging are like a dashboard and black box recorder for your database operations. The studio lets you peek inside the database visually, while query logging records every SQL statement for post-mortem analysis. Together, they take the mystery out of what your ORM is actually doing.

Prisma Studio and Debugging Queries

Introduction

Prisma Studio is a visual database browser that lets you view, create, edit, and delete records through a web UI. Combined with Prisma's query logging and debugging tools, you get a complete toolkit for understanding what your ORM is doing under the hood.

Prisma Studio

npx prisma studio
# Opens a browser at http://localhost:5555

Prisma Studio provides: - Browse all tables and view records - Filter and sort data - Create, edit, and delete records - Navigate relations (click on related records) - View record counts

Query Logging

// Enable all logging
const prisma = new PrismaClient({
  log: [
    { emit: 'stdout', level: 'query' },   // SQL queries
    { emit: 'stdout', level: 'info' },     // Info messages
    { emit: 'stdout', level: 'warn' },     // Warnings
    { emit: 'stdout', level: 'error' },    // Errors
  ],
});

// Event-based logging (more control)
const prisma = new PrismaClient({
  log: [
    { emit: 'event', level: 'query' },
  ],
});

prisma.$on('query', (e) => {
  console.log('Query:', e.query);
  console.log('Params:', e.params);
  console.log('Duration:', e.duration, 'ms');
  if (e.duration > 100) {
    console.warn('SLOW QUERY detected!');
  }
});

Structured Query Logging

import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient({
  log: [{ emit: 'event', level: 'query' }],
});

interface QueryLog {
  query: string;
  params: string;
  duration: number;
  timestamp: Date;
}

const queryLog: QueryLog[] = [];

prisma.$on('query', (e) => {
  queryLog.push({
    query: e.query,
    params: e.params,
    duration: e.duration,
    timestamp: new Date(e.timestamp),
  });
});

// After running queries, analyze:
function getQueryReport() {
  return {
    totalQueries: queryLog.length,
    totalTime: queryLog.reduce((sum, q) => sum + q.duration, 0),
    slowQueries: queryLog.filter(q => q.duration > 50),
    queryCounts: queryLog.reduce((acc, q) => {
      const key = q.query.split(' ').slice(0, 3).join(' ');
      acc[key] = (acc[key] || 0) + 1;
      return acc;
    }, {} as Record<string, number>),
  };
}

Debugging N+1 Queries

// BAD: N+1 pattern
const users = await prisma.user.findMany({ take: 100 });
for (const user of users) {
  const orders = await prisma.order.findMany({
    where: { userId: user.id },
  });
  // 101 queries!
}

// GOOD: Include relations
const users = await prisma.user.findMany({
  take: 100,
  include: {
    orders: true,
  },
});
// 2 queries: one for users, one for all related orders

// GOOD: Select only what you need
const users = await prisma.user.findMany({
  take: 100,
  select: {
    id: true,
    name: true,
    _count: { select: { orders: true } },
  },
});
// 2 queries: users + count subquery

Using $queryRaw for Complex Queries

// Type-safe raw queries
const result = await prisma.$queryRaw<
  { id: string; name: string; order_count: bigint }[]
>`
  SELECT u.id, u.name, COUNT(o.id) AS order_count
  FROM users u
  LEFT JOIN orders o ON u.id = o.user_id
  GROUP BY u.id, u.name
  HAVING COUNT(o.id) > ${minOrders}
  ORDER BY order_count DESC
  LIMIT ${limit}
`;

// Note: $queryRaw uses tagged template literals for parameterization
// The values are automatically parameterized (NOT string interpolation)

Debugging Checklist

1. Enable query logging to see generated SQL
2. Check query count — look for N+1 patterns
3. Check query duration — identify slow queries
4. Use EXPLAIN ANALYZE on the raw SQL
5. Verify indexes exist for filtered/joined columns
6. Use select instead of include to reduce data
7. Use Prisma Studio to verify data state
8. Check for missing relations in the schema

Error Handling

import { Prisma } from '@prisma/client';

async function createUser(data: CreateUserInput) {
  try {
    return await prisma.user.create({ data });
  } catch (err) {
    if (err instanceof Prisma.PrismaClientKnownRequestError) {
      switch (err.code) {
        case 'P2002': // Unique constraint violation
          const field = (err.meta?.target as string[])?.[0];
          throw new Error(`A user with this ${field} already exists`);
        case 'P2003': // Foreign key constraint violation
          throw new Error('Referenced record does not exist');
        case 'P2025': // Record not found
          throw new Error('Record not found');
        default:
          throw new Error(`Database error: ${err.code}`);
      }
    }
    throw err;
  }
}

Exercises

  1. N+1 Detector

    Write a middleware or wrapper that detects N+1 query patterns by analyzing the query log. If the same query template runs more than N times in rapid succession, log a warning with the query and suggestion to use include/select.

    Hint

    Use prisma.$on('query') to track queries. Group by query template (ignoring parameter values). Alert when a template appears > 5 times within 100ms.

  2. Query Performance Dashboard

    Build an endpoint that returns query performance metrics: total queries, slow queries (> 50ms), average query time, most frequent queries, and current connection pool status. Reset metrics on demand.

    Hint

    Collect metrics via prisma.$on('query'). Store in a Map. Expose via a /debug/queries endpoint (protected by an admin check).

  3. Error Handler Integration

    Create a comprehensive Prisma error handler that maps all PrismaClientKnownRequestError codes to appropriate HTTP status codes and user-friendly messages. Integrate it as Express middleware.

    Hint

    Map P2002 to 409 Conflict, P2003 to 400 Bad Request, P2025 to 404 Not Found. Use err.meta for details about which field/constraint failed.

Common Problems & Solutions

1 Cannot see which SQL queries Prisma generates
Cause: Query logging is not enabled by default.
Solution: Pass `log: ['query']` to PrismaClient constructor. For event-based logging with timing, use `{ emit: 'event', level: 'query' }` and `prisma.$on('query', ...)`.
2 Prisma Studio shows stale data
Cause: Prisma Studio caches data and may not reflect very recent changes.
Solution: Refresh the page. If using Prisma Studio while your app is also writing to the database, Studio may need a manual refresh to show the latest state.
3 $queryRaw is vulnerable to SQL injection
Cause: Using string concatenation or regular template literals instead of tagged template literals for raw queries.
Solution: Always use tagged template literals: `prisma.$queryRaw\`SELECT * FROM users WHERE id = ${id}\``. The tagged literal parameterizes the values. NEVER use `prisma.$queryRawUnsafe` with user input.

// SAFE (tagged template):\nprisma.$queryRawSELECT * FROM users WHERE id = ${id}\n// DANGEROUS:\nprisma.$queryRawUnsafe(SELECT * FROM users WHERE id = '${id}')
