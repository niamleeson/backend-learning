Design a Distributed Task Scheduler

Intuition

Think of a distributed task scheduler like a postal sorting center. Letters (tasks) arrive with delivery dates written on them. A sorting machine (scheduler) checks the clock, pulls out letters that are due for delivery, and distributes them to mail carriers (workers). If a carrier gets sick (worker fails), the letter is returned and given to another carrier. A supervisor (leader) coordinates the sorting so no letter gets sorted twice.

Design a Distributed Task Scheduler

Introduction

A distributed task scheduler is a system that reliably executes jobs at specified times or intervals across a cluster of worker machines. Think of cron jobs, but distributed, fault-tolerant, and capable of handling millions of scheduled tasks. Systems like Apache Airflow, Temporal, and cloud services like AWS EventBridge solve variants of this problem. This design tests your understanding of distributed coordination, exactly-once execution, and failure recovery.

Step 1: Requirements Gathering

Functional Requirements

  • Schedule one-time tasks to execute at a specific future time
  • Schedule recurring tasks with cron-like expressions (e.g., every 5 minutes, daily at 3am)
  • Support task priorities (high, medium, low)
  • Allow task cancellation and modification
  • Provide visibility into task status (pending, running, completed, failed)
  • Support retries with configurable backoff on failure
  • Guarantee at-least-once execution (no task is silently dropped)

Non-Functional Requirements

  • Scale: 100 million scheduled tasks, 10,000 task executions per second
  • Latency: Tasks execute within 5 seconds of their scheduled time
  • Reliability: No task loss even during node failures
  • Availability: 99.99% uptime
  • Consistency: Exactly-once execution (or at-least-once with idempotency)

Out of Scope

  • Complex DAG-based workflow orchestration (Airflow territory)
  • Task dependency management
  • Multi-region deployment

Step 2: Capacity Estimation

Assumptions:
- 100 million total scheduled tasks in the system
- 10,000 tasks execute per second on average
- Peak: 50,000 tasks/second (batch jobs at midnight)
- Average task metadata: 500 bytes (id, schedule, payload, status)
- Task execution logs: 1 KB per execution

Storage:
- Task metadata: 100M x 500 bytes = 50 GB
- Daily executions: 10,000/s x 86,400 = 864M executions/day
- Execution logs: 864M x 1 KB = 864 GB/day (need retention policy)
- 30-day retention: ~26 TB of execution logs

Compute:
- 10,000 tasks/second across worker pool
- If average task takes 2 seconds: need 20,000 concurrent worker slots
- With 4 workers per machine (Node.js cluster): 5,000 worker machines
- More realistically, most tasks are fast (<100ms): ~500 machines

Database:
- Write QPS: 10,000 (task status updates) + 10,000 (execution logs) = 20,000 QPS
- Read QPS: Scheduler polling for due tasks = varies by implementation

import express from 'express';
import { z } from 'zod';

const app = express();

// Schema validation
const CreateTaskSchema = z.object({

});

  const parsed = CreateTaskSchema.safeParse(req.body);
  if (!parsed.success) {
    return res.status(400).json({ errors: parsed.error.issues });
  }

  const task = await taskService.create(parsed.data);

// GET /api/v1/tasks/:id - Get task details
app.get('/api/v1/tasks/:id', async (req, res) => {
  const task = await taskService.getById(req.params.id);
  if (!task) return res.status(404).json({ error: 'Task not found' });
  res.json(task);
});

// PATCH /api/v1/tasks/:id - Update a task
app.patch('/api/v1/tasks/:id', async (req, res) => {
  const task = await taskService.update(req.params.id, req.body);
  res.json(task);
});

// DELETE /api/v1/tasks/:id - Cancel a task
app.delete('/api/v1/tasks/:id', async (req, res) => {
  await taskService.cancel(req.params.id);
  res.status(204).end();
});

// GET /api/v1/tasks/:id/executions - Get execution history
app.get('/api/v1/tasks/:id/executions', async (req, res) => {
  const { page = 1, limit = 20 } = req.query;
  const executions = await taskService.getExecutions(
    req.params.id, parseInt(page), parseInt(limit)
  );
  res.json(executions);
});

Step 5: Database Schema

-- Core task definition
CREATE TABLE tasks (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(200) NOT NULL,
    type            VARCHAR(20) NOT NULL CHECK (type IN ('one_time', 'recurring')),
    status          VARCHAR(20) NOT NULL DEFAULT 'active'
                    CHECK (status IN ('active', 'paused', 'cancelled', 'completed')),
    cron_expression VARCHAR(100),
    payload         JSONB NOT NULL DEFAULT '{}',
    callback_url    VARCHAR(2048) NOT NULL,
    priority        SMALLINT NOT NULL DEFAULT 5, -- 1=highest, 10=lowest
    max_retries     SMALLINT NOT NULL DEFAULT 3,
    timeout_ms      INTEGER NOT NULL DEFAULT 30000,
    next_run_at     TIMESTAMP NOT NULL,
    last_run_at     TIMESTAMP,
    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    created_by      VARCHAR(100)
);

CREATE INDEX idx_tasks_next_run ON tasks(next_run_at)
    WHERE status = 'active';
CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_tasks_priority_next_run ON tasks(priority, next_run_at)
    WHERE status = 'active';

-- Execution history
CREATE TABLE task_executions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id         UUID NOT NULL REFERENCES tasks(id),
    status          VARCHAR(20) NOT NULL
                    CHECK (status IN ('pending', 'running', 'completed', 'failed', 'timed_out')),
    attempt         SMALLINT NOT NULL DEFAULT 1,
    scheduled_at    TIMESTAMP NOT NULL,
    started_at      TIMESTAMP,
    completed_at    TIMESTAMP,
    error_message   TEXT,
    worker_id       VARCHAR(100),
    result          JSONB,
    created_at      TIMESTAMP NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

CREATE INDEX idx_executions_task_id ON task_executions(task_id);
CREATE INDEX idx_executions_status ON task_executions(status);

-- Distributed lock table (alternative to Redis-based locking)
CREATE TABLE scheduler_locks (
    lock_name       VARCHAR(100) PRIMARY KEY,
    locked_by       VARCHAR(100) NOT NULL,
    locked_at       TIMESTAMP NOT NULL DEFAULT NOW(),
    expires_at      TIMESTAMP NOT NULL
);

Step 6: Core Scheduler Logic

Leader Election

Only one scheduler node should poll for due tasks at a time to avoid duplicate execution:

import Redis from 'ioredis';

class LeaderElection {
  constructor(redis, nodeId, lockKey = 'scheduler:leader') {
    this.redis = redis;
    this.nodeId = nodeId;
    this.lockKey = lockKey;
    this.ttlMs = 10000; // 10-second lease
    this.isLeader = false;
  }

  async tryBecomeLeader() {
    // SET NX with expiry - atomic leader election
    const result = await this.redis.set(
      this.lockKey,
      this.nodeId,
      'PX', this.ttlMs,
      'NX'
    );
    this.isLeader = result === 'OK';
    return this.isLeader;
  }

  async renewLease() {
    // Only renew if we still hold the lock
    const script = `
      if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('pexpire', KEYS[1], ARGV[2])
      else
        return 0
      end
    `;
    const result = await this.redis.eval(script, 1, this.lockKey, this.nodeId, this.ttlMs);
    this.isLeader = result === 1;
    return this.isLeader;
  }

  async start(onLeader, onFollower) {
    setInterval(async () => {
      if (this.isLeader) {
        const renewed = await this.renewLease();
        if (!renewed) {
          this.isLeader = false;
          onFollower();
        }
      } else {
        const acquired = await this.tryBecomeLeader();
        if (acquired) onLeader();
      }
    }, this.ttlMs / 3); // Renew at 1/3 of TTL
  }
}

Task Polling and Dispatch

import { Queue } from 'bullmq';
import { PrismaClient } from '@prisma/client';
import cronParser from 'cron-parser';

const prisma = new PrismaClient();
const taskQueue = new Queue('task-execution', {
  connection: { host: 'redis', port: 6379 },
});

class TaskScheduler {
  constructor() {
    this.batchSize = 500;
    this.pollIntervalMs = 1000;
    this.running = false;
  }

  async start() {
    this.running = true;
    while (this.running) {
      try {
        const dispatched = await this.pollAndDispatch();
        if (dispatched === 0) {
          // No tasks ready, wait before next poll
          await this.sleep(this.pollIntervalMs);
        }
      } catch (err) {
        console.error('Scheduler error:', err);
        await this.sleep(5000);
      }
    }
  }

  async pollAndDispatch() {
    // Atomically claim due tasks using UPDATE ... RETURNING
    const dueTasks = await prisma.$queryRaw`
      UPDATE tasks
      SET status = 'dispatched',
          updated_at = NOW()
      WHERE id IN (
        SELECT id FROM tasks
        WHERE status = 'active'
          AND next_run_at <= NOW()
        ORDER BY priority ASC, next_run_at ASC
        LIMIT ${this.batchSize}
        FOR UPDATE SKIP LOCKED
      )
      RETURNING *
    `;

    // Enqueue each task for worker execution
    for (const task of dueTasks) {
      await taskQueue.add(
        task.name,
        {
          taskId: task.id,
          payload: task.payload,
          callbackUrl: task.callback_url,
          attempt: 1,
          maxRetries: task.max_retries,
        },
        {
          priority: task.priority,
          attempts: task.max_retries + 1,
          backoff: { type: 'exponential', delay: 5000 },
          timeout: task.timeout_ms,
        }
      );
    }

    return dueTasks.length;
  }

  sleep(ms) {
    return new Promise(resolve => setTimeout(resolve, ms));
  }
}

Worker Implementation

import { Worker } from 'bullmq';

const worker = new Worker(
  'task-execution',
  async (job) => {
    const { taskId, payload, callbackUrl, attempt, maxRetries } = job.data;

    // Record execution start
    const execution = await prisma.taskExecution.create({
      data: {
        taskId,
        status: 'running',
        attempt,
        scheduledAt: new Date(),
        startedAt: new Date(),
        workerId: process.env.WORKER_ID,
      },
    });

    try {
      // Execute the task by calling the webhook
      const controller = new AbortController();
      const timeout = setTimeout(() => controller.abort(), job.data.timeoutMs || 30000);

      const response = await fetch(callbackUrl, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(payload),
        signal: controller.signal,
      });
      clearTimeout(timeout);

      if (!response.ok) {
        throw new Error(`Callback returned ${response.status}`);
      }

      const result = await response.json();

      // Mark execution as completed
      await prisma.taskExecution.update({
        where: { id: execution.id },
        data: { status: 'completed', completedAt: new Date(), result },
      });

      // Update recurring task's next_run_at
      await updateNextRun(taskId);

      return result;
    } catch (err) {
      await prisma.taskExecution.update({
        where: { id: execution.id },
        data: {
          status: 'failed',
          completedAt: new Date(),
          errorMessage: err.message,
        },
      });
      throw err; // BullMQ will retry based on config
    }
  },
  {
    connection: { host: 'redis', port: 6379 },
    concurrency: 50,
  }
);

async function updateNextRun(taskId) {
  const task = await prisma.task.findUnique({ where: { id: taskId } });
  if (task.type === 'recurring' && task.cronExpression) {
    const interval = cronParser.parseExpression(task.cronExpression);
    const nextRun = interval.next().toDate();
    await prisma.task.update({
      where: { id: taskId },
      data: { nextRunAt: nextRun, status: 'active', lastRunAt: new Date() },
    });
  } else {
    await prisma.task.update({
      where: { id: taskId },
      data: { status: 'completed', lastRunAt: new Date() },
    });
  }
}

Step 8: Trade-offs and Decisions

Decision Option A Option B Recommendation
Task store PostgreSQL DynamoDB PostgreSQL for ACID guarantees and complex queries; DynamoDB for massive scale
Queue BullMQ (Redis) Kafka BullMQ for simplicity and delayed jobs; Kafka for extreme throughput
Execution model Webhook callback Embedded function Webhooks for decoupling; embedded for low-latency tasks
Leader election Redis SETNX ZooKeeper Redis for simplicity; ZooKeeper for stronger guarantees
Exactly-once Idempotency keys Distributed locks Idempotency keys on the consumer side; locks add latency

Summary

A distributed task scheduler combines a task store (PostgreSQL), a coordinator with leader election (Redis), a distributed queue (BullMQ), and a pool of workers. The scheduler leader polls for due tasks, enqueues them, and workers execute them by calling webhooks or running embedded functions. Key challenges include exactly-once execution (solved via FOR UPDATE SKIP LOCKED and idempotency), failure recovery (retries with exponential backoff), and scaling (horizontal worker scaling with queue-depth-based autoscaling).

Exercises

  1. Build a Simple Task Scheduler with BullMQ

    Create a task scheduling service using Express and BullMQ that accepts task definitions via REST API, stores them in PostgreSQL, and executes them at the scheduled time. Implement both one-time and recurring (cron) task types.

    Hint

    Use BullMQ's delayed jobs feature for one-time tasks and the repeat option for cron tasks. Store task metadata in PostgreSQL for durability.

  2. Implement Leader Election with Redis

    Build a leader election mechanism using Redis SET NX with expiry. Run 3 Node.js processes that compete for leadership. Only the leader should poll for due tasks. Verify that leadership transfers when the leader crashes.

    Hint

    Use SET key value PX ttl NX for atomic lock acquisition. Renew the lease at 1/3 of the TTL interval. Use a Lua script for safe lease renewal.

  3. Retry Logic with Exponential Backoff

    Implement a task executor that retries failed tasks with exponential backoff (1s, 2s, 4s, 8s, 16s, max 60s). Track retry counts, failure reasons, and final status in the database. Add a dead letter queue for tasks that exhaust all retries.

    Hint

    Use BullMQ's built-in backoff option or implement manual retry scheduling by computing the next retry time and re-enqueuing.

  4. Build a Task Dashboard API

    Create API endpoints that return scheduler health metrics: tasks pending, tasks running, tasks completed in the last hour, tasks failed, average execution time, and queue depth. Use PostgreSQL aggregate queries.

    Hint

    Use GROUP BY status with COUNT for task breakdowns. Use AVG(completed_at - started_at) for average execution time.

Common Problems & Solutions

1 Tasks are being executed multiple times (duplicate execution)
Symptoms: Webhook endpoints receive the same payload multiple times. Task execution logs show duplicate entries with overlapping timestamps.
Cause: Multiple scheduler nodes are polling for due tasks simultaneously without proper coordination, or a task completes but the status update fails, causing it to be picked up again.
Solution: Use leader election so only one scheduler polls at a time. Use SELECT ... FOR UPDATE SKIP LOCKED to atomically claim tasks. Implement idempotency keys on the consumer side so duplicate executions are harmless.
2 Scheduled tasks are consistently late (missing their scheduled time by minutes)
Cause: The scheduler poll interval is too long, or the task queue is backed up because workers cannot process tasks fast enough.
Solution: Reduce poll interval to 1 second. Monitor queue depth and auto-scale workers based on queue size. Use priority queues so high-priority tasks are not blocked by low-priority ones.
3 Database connection pool exhaustion during peak scheduling periods
Cause: Each worker holds a database connection while executing long-running tasks, and the connection pool runs out when many tasks execute concurrently.
Solution: Separate the task execution (webhook call) from the database operations. Use a dedicated small pool for status updates. Consider batching status updates with a write-behind buffer.
