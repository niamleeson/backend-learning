Design a Hotel Booking System (Booking.com)

Intuition

A hotel booking system is an inventory management problem where the core challenge is preventing double-bookings under concurrent access while maintaining responsive search across millions of room-night combinations. Date-range availability queries are the distinguishing complexity.

Design a Hotel Booking System (Booking.com)

Requirements

  • Search available rooms by location, dates, guests, and price range
  • Handle concurrent booking attempts without double-booking
  • Support cancellations, modifications, and dynamic pricing
  • Display real-time availability across thousands of hotels
  • Handle peak loads during holiday seasons

Architecture

A search service backed by Elasticsearch handles availability queries. A booking service manages reservations with pessimistic locking on room inventory. A pricing service computes dynamic rates based on demand, season, and competition. An event-driven architecture publishes booking events for analytics, notifications, and partner systems.

Key Components

  • Search Service: Elasticsearch with filters for location (geo queries), dates, price, amenities, and ratings
  • Availability Service: Manages room inventory using a date-range availability table with row-level locking
  • Booking Service: Handles reservation creation, confirmation, modification, and cancellation
  • Pricing Engine: Dynamic pricing based on occupancy rate, day of week, season, and competitor rates
  • Hotel Management Portal: Interface for hotels to update room details, pricing rules, and blackout dates

Scaling

Cache search results with short TTLs for popular destinations. Use read replicas for search queries. Shard availability data by hotel ID. Pre-compute availability for popular date ranges. Use a CDN for hotel images and static content.

Trade-offs

  • Pessimistic locking prevents double-booking but reduces concurrency; optimistic locking improves concurrency but increases retry rates
  • Showing cached availability is fast but may lead to failed bookings; real-time checks are accurate but slow

Exercises

  1. Implement Room Availability Checking

    Design a database schema and query that efficiently checks if a specific room type is available for a given date range. Handle partial overlaps correctly. The system should support checking availability across hundreds of rooms for a 1-7 night stay.

    Hint

    Create an availability table with columns: hotel_id, room_type, date, total_rooms, booked_rooms. A room type is available for a date range if booked_rooms < total_rooms for ALL dates in the range. Use a single query with GROUP BY and HAVING.

  2. Design a Concurrent Booking Handler

    Build a booking endpoint that handles multiple users trying to book the last available room simultaneously. Ensure exactly one booking succeeds and others receive a sold-out response. Use database-level locking.

    Hint

    Use SELECT ... FOR UPDATE to lock the availability rows for the requested date range. Within the transaction, verify availability, increment booked_rooms, create the reservation, and commit. Failed transactions retry or return sold-out.

Common Problems & Solutions

1 Two users simultaneously book the last available room for the same dates, resulting in a double-booking.
Cause: Both transactions read availability as 1 room remaining, then both decrement it, resulting in -1 available (oversold).
Solution: Use pessimistic locking (SELECT FOR UPDATE) on the availability rows for the requested date range. The second transaction blocks until the first commits, then sees the updated (zero) availability and rejects the booking.
2 Search results show rooms as available, but when users try to book, they get 'sold out' errors, creating a frustrating experience.
Cause: Search results are served from a cache with a stale TTL, and high-demand rooms sell out between the time of search and booking attempt.
Solution: Use short cache TTLs (30-60 seconds) for high-demand properties. Show approximate availability indicators (e.g., 'Only 2 rooms left'). Implement a soft-hold mechanism that temporarily reserves a room when a user begins checkout.
