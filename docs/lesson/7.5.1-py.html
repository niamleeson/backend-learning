WSGI vs ASGI: Gunicorn and Uvicorn

Intuition

WSGI is like a restaurant where each waiter (worker) handles one table at a time -- simple and reliable. ASGI is like a waiter who can take drink orders from table 1, check on table 2, and serve food to table 3 without waiting idle. For a busy restaurant with lots of quick orders (API calls), async waiters handle more tables.

WSGI vs ASGI: Gunicorn and Uvicorn

WSGI and ASGI are interfaces between your Python web application and the web server. Understanding them is crucial for choosing the right application server.

WSGI - Synchronous

def application(environ, start_response):
    start_response('200 OK', [('Content-Type', 'text/plain')])
    return [b'Hello, World!']

# Gunicorn (standard WSGI server)
gunicorn app:create_app() \
  --workers 4 \
  --threads 2 \
  --bind 0.0.0.0:8000 \
  --timeout 30 \
  --access-logfile -
# Worker count rule: (2 * CPU cores) + 1

ASGI - Asynchronous

async def application(scope, receive, send):
    if scope['type'] == 'http':
        await send({'type': 'http.response.start', 'status': 200,
                    'headers': [[b'content-type', b'text/plain']]})
        await send({'type': 'http.response.body', 'body': b'Hello, async!'})

# Development
uvicorn app.main:app --reload

# Production: Gunicorn with Uvicorn workers
gunicorn app.main:app \
  -k uvicorn.workers.UvicornWorker \
  --workers 4 \
  --bind 0.0.0.0:8000

When to Use Which

Feature WSGI (Gunicorn) ASGI (Uvicorn)
Frameworks Flask, Django FastAPI, Starlette
Async/WebSockets No Yes
Use case Traditional APIs Real-time, async APIs

Common Mistakes

  • Using uvicorn --reload in production
  • Setting too many workers, exhausting server memory
  • Not using Gunicorn as process manager for Uvicorn in production

Key Takeaways

  • Use Gunicorn + Uvicorn workers for production ASGI apps
  • Workers = (2 * CPU cores) + 1 is a good starting point
  • ASGI supports WebSockets; WSGI does not

Exercises

    Hint

Common Problems & Solutions

1 Gunicorn workers keep getting killed
Symptoms: ['[CRITICAL] WORKER TIMEOUT', 'Worker with pid XXXX was terminated due to signal 9']
Cause: Workers exceed the timeout (default 30s) because of long-running requests, or the server runs out of memory.
Solution: Increase timeout for long operations: `--timeout 120`. Reduce workers if memory is the issue. For truly long tasks, use a background worker (Celery).
2 Uvicorn reload doesn't pick up changes
Symptoms: ['Code changes not reflected', 'Have to manually restart the server']
Cause: The reload watcher doesn't detect changes in certain file types or mounted volumes (Docker).
Solution: Use `--reload-dir src` to specify which directory to watch. In Docker, ensure the volume mount is working and use `--reload-delay 0.5`.
