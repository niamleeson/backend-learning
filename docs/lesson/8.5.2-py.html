ELK Stack: Elasticsearch, Logstash, Kibana

Intuition

Think of the ELK stack like a library system. Logstash is the librarian who receives incoming books, catalogs them, and assigns metadata tags. Elasticsearch is the catalog system that indexes every word in every book so you can find any passage instantly. Kibana is the reading room dashboard that shows you trends — which topics are most popular, which sections have damage reports — giving you a bird's-eye view of everything in the collection.

ELK Stack: Elasticsearch, Logstash, Kibana

When your backend application runs across dozens of servers processing thousands of requests per second, understanding what is happening inside your system becomes a serious engineering challenge. You cannot SSH into each server and tail log files manually. You need a centralized logging and observability platform, and the ELK Stack is one of the most widely adopted solutions for this problem.

What Is the ELK Stack?

The ELK Stack is a collection of three open-source tools developed by Elastic:

  • Elasticsearch — A distributed search and analytics engine built on Apache Lucene. It stores, indexes, and enables fast full-text search over your log data.
  • Logstash — A server-side data processing pipeline that ingests data from multiple sources, transforms it, and sends it to Elasticsearch.
  • Kibana — A web-based visualization layer that lets you explore, analyze, and create dashboards from data stored in Elasticsearch.

In modern deployments, a fourth component called Beats (lightweight data shippers like Filebeat) is commonly added, making it the "Elastic Stack" or sometimes "ELK+B." Beats handle the lightweight collection of data on each host, while Logstash handles heavier transformations centrally.

Architecture and Data Flow

The typical data flow through an ELK stack looks like this:

Application Logs --> Filebeat (on each host)
                        |
                        v
                    Logstash (central processing)
                        |
                        v
                    Elasticsearch (storage + indexing)
                        |
                        v
                    Kibana (visualization + dashboards)

Here is what happens at each stage:

  1. Your Python application writes structured log entries (ideally JSON) to a file or stdout.
  2. Filebeat, running as a lightweight agent on the same host, watches those log files and forwards new entries to Logstash.
  3. Logstash receives the raw log data, parses it, enriches it (e.g., adding geo-IP data, extracting fields), and sends the structured documents to Elasticsearch.
  4. Elasticsearch indexes the documents and makes them searchable within seconds.
  5. Engineers and SREs use Kibana to search logs, build dashboards, and set up alerts.

Elasticsearch Fundamentals

Elasticsearch is the heart of the stack. Understanding its core concepts is essential.

Documents and Indices

Elasticsearch stores data as documents — JSON objects. Each document belongs to an index, which is analogous to a database table. For example, you might have an index called app-logs-2026.03.17 that holds all application logs for a given day.

{
  "@timestamp": "2026-03-17T14:23:01.123Z",
  "level": "ERROR",
  "service": "payment-service",
  "message": "Failed to process payment for order 98712",
  "trace_id": "abc-def-123",
  "response_time_ms": 3420
}

Mappings

A mapping defines how fields in a document are stored and indexed. Elasticsearch can auto-detect types, but explicit mappings give you control over how data is analyzed and searched.

{
  "mappings": {
    "properties": {
      "@timestamp": { "type": "date" },
      "level": { "type": "keyword" },
      "service": { "type": "keyword" },
      "message": { "type": "text" },
      "trace_id": { "type": "keyword" },
      "response_time_ms": { "type": "integer" }
    }
  }
}

The distinction between keyword and text is important. A keyword field is stored as-is and supports exact matching and aggregations. A text field is analyzed (tokenized, lowercased) and supports full-text search. Use keyword for identifiers like service names and log levels. Use text for free-form messages you want to search within.

Shards and Replicas

Elasticsearch distributes data across shards. Each index is split into one or more primary shards, and each primary shard can have replica shards for redundancy and read throughput. For example, an index with 3 primary shards and 1 replica has 6 total shards across your cluster.

A common beginner mistake is creating too many shards. Each shard consumes memory and file handles. For logging workloads, start with 1 primary shard per 50 GB of expected daily data.

Logstash Pipelines

Logstash processes data through a pipeline with three stages: input, filter, and output.

A Complete Pipeline Configuration

# /etc/logstash/conf.d/app-logs.conf

input {
  beats {
    port => 5044
  }
}

filter {
  # Parse JSON log messages
  json {
    source => "message"
  }

  # Parse the timestamp
  date {
    match => ["timestamp", "ISO8601"]
    target => "@timestamp"
  }

  # Add geo-IP information if an IP field exists
  if [client_ip] {
    geoip {
      source => "client_ip"
      target => "geo"
    }
  }

  # Remove unnecessary fields to save storage
  mutate {
    remove_field => ["host", "agent", "ecs"]
  }

  # Tag errors for easier filtering
  if [level] == "ERROR" or [level] == "CRITICAL" {
    mutate {
      add_tag => ["error"]
    }
  }
}

output {
  elasticsearch {
    hosts => ["http://elasticsearch:9200"]
    index => "app-logs-%{+YYYY.MM.dd}"
    user => "elastic"
    password => "${ES_PASSWORD}"
  }
}

This pipeline receives data from Filebeat on port 5044, parses JSON-formatted log messages, enriches them with geo-IP data when applicable, cleans up unnecessary fields, and sends the result to a date-stamped Elasticsearch index.

Key Logstash Filters

  • grok — Parse unstructured text using regular expression patterns. Useful when you cannot control the log format.
  • json — Parse JSON strings into structured fields.
  • mutate — Rename, remove, or modify fields.
  • date — Parse date strings into proper timestamp fields.
  • geoip — Enrich IP addresses with geographic data.

Here is a grok example for parsing a traditional Apache access log:

filter {
  grok {
    match => {
      "message" => "%{IPORHOST:client_ip} - - \[%{HTTPDATE:timestamp}\] \"%{WORD:method} %{URIPATHPARAM:request} HTTP/%{NUMBER:http_version}\" %{NUMBER:status_code} %{NUMBER:bytes}"
    }
  }
}

Shipping Logs from Python Applications

There are two primary approaches to getting logs from your Python application into the ELK stack.

Approach 1: Structured File Logging with Filebeat (Recommended)

The most robust approach is to write structured JSON logs to a file and let Filebeat ship them. This decouples your application from the logging infrastructure.

import logging
import json
from datetime import datetime, timezone


class JSONFormatter(logging.Formatter):
    def format(self, record):
        log_entry = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
            "module": record.module,
            "function": record.funcName,
            "line": record.lineno,
        }
        # Include exception info if present
        if record.exc_info and record.exc_info[0] is not None:
            log_entry["exception"] = self.formatException(record.exc_info)
        # Include any extra fields
        if hasattr(record, "trace_id"):
            log_entry["trace_id"] = record.trace_id
        if hasattr(record, "user_id"):
            log_entry["user_id"] = record.user_id
        return json.dumps(log_entry)


# Configure logging
handler = logging.FileHandler("/var/log/myapp/app.log")
handler.setFormatter(JSONFormatter())

logger = logging.getLogger("myapp")
logger.setLevel(logging.INFO)
logger.addHandler(handler)

# Usage
logger.info("Payment processed", extra={"trace_id": "abc-123", "user_id": 42})

Then configure Filebeat to watch this file:

# /etc/filebeat/filebeat.yml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/myapp/app.log
    json.keys_under_root: true
    json.add_error_key: true
    json.message_key: message

output.logstash:
  hosts: ["logstash-server:5044"]

Approach 2: Direct Logging with python-logstash

For simpler setups or development environments, you can send logs directly to Logstash over the network:

import logging
import logstash

logger = logging.getLogger("myapp")
logger.setLevel(logging.INFO)

# Send logs directly to Logstash via TCP
logger.addHandler(
    logstash.TCPLogstashHandler(
        host="logstash-server",
        port=5959,
        version=1
    )
)

logger.info("User login successful", extra={
    "user_id": 42,
    "ip_address": "192.168.1.100"
})

The corresponding Logstash input configuration:

input {
  tcp {
    port => 5959
    codec => json_lines
  }
}

Approach 1 is preferred in production because Filebeat provides buffering, back-pressure handling, and retry logic. If Logstash is temporarily down, Filebeat remembers its position in the file and resumes when the connection is restored. The direct approach risks losing logs during network issues.

Kibana Dashboards and Visualizations

Kibana is where your logs become actionable. After your data flows into Elasticsearch, you create an index pattern in Kibana (e.g., app-logs-*) to tell it which indices to query.

Key Kibana Features

  • Discover — Search and browse raw log entries with filters and a time picker. This is your go-to tool for ad-hoc investigation.
  • Visualize — Create charts, tables, and gauges. Common visualizations include error rate over time (line chart), top error messages (data table), and response time percentiles (area chart).
  • Dashboard — Combine multiple visualizations into a single view. A typical operations dashboard might show request rate, error rate, p95 latency, and top error messages.
  • Alerts — Configure rules that trigger notifications (Slack, email, PagerDuty) when conditions are met, such as error rate exceeding a threshold.

Search Queries

Elasticsearch provides a powerful query DSL. Here are the most commonly used queries for log analysis.

Match Query (Full-Text Search)

{
  "query": {
    "match": {
      "message": "payment failed"
    }
  }
}

This searches the analyzed message field for documents containing "payment" or "failed" (or both).

Bool Query (Combining Conditions)

{
  "query": {
    "bool": {
      "must": [
        { "match": { "message": "timeout" } }
      ],
      "filter": [
        { "term": { "level": "ERROR" } },
        { "term": { "service": "payment-service" } },
        {
          "range": {
            "@timestamp": {
              "gte": "2026-03-17T00:00:00Z",
              "lte": "2026-03-17T23:59:59Z"
            }
          }
        }
      ]
    }
  }
}

The must clause affects relevance scoring. The filter clause is a yes/no check that does not affect scoring and is cached for performance. Always use filter for exact matches on keywords and date ranges.

Range Query

{
  "query": {
    "range": {
      "response_time_ms": {
        "gte": 5000
      }
    }
  }
}

This finds all requests that took 5 seconds or longer.

Aggregations for Metrics

Aggregations let you compute analytics over your log data. They are the foundation of Kibana visualizations.

Error Count by Service (Terms Aggregation)

{
  "size": 0,
  "query": {
    "term": { "level": "ERROR" }
  },
  "aggs": {
    "errors_by_service": {
      "terms": {
        "field": "service",
        "size": 10
      }
    }
  }
}

Response Time Percentiles Over Time

{
  "size": 0,
  "aggs": {
    "over_time": {
      "date_histogram": {
        "field": "@timestamp",
        "fixed_interval": "5m"
      },
      "aggs": {
        "latency_percentiles": {
          "percentiles": {
            "field": "response_time_ms",
            "percents": [50, 90, 95, 99]
          }
        }
      }
    }
  }
}

This creates 5-minute buckets and computes p50, p90, p95, and p99 latency within each bucket — exactly what you need for an SLA dashboard.

Index Lifecycle Management (ILM)

Log data grows fast. Without management, your Elasticsearch cluster will run out of disk. Index Lifecycle Management (ILM) automates the aging and deletion of indices through phases.

{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_size": "50gb",
            "max_age": "1d"
          },
          "set_priority": { "priority": 100 }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "forcemerge": { "max_num_segments": 1 },
          "shrink": { "number_of_shards": 1 },
          "set_priority": { "priority": 50 }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "freeze": {},
          "set_priority": { "priority": 0 }
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}

This policy keeps data on fast "hot" nodes for 7 days, moves it to cheaper "warm" nodes and optimizes it, freezes it on "cold" storage after 30 days, and deletes it after 90 days. This tiered approach balances query performance with storage costs.

Performance Tuning

As your logging volume grows, you will need to tune the stack. Here are the most impactful optimizations.

Elasticsearch Tuning

  • Bulk indexing — Always index documents in bulk. Logstash and Filebeat do this by default, but if you are using the Elasticsearch client directly, batch your writes.
  • Refresh interval — Elasticsearch makes new documents searchable every 1 second by default. For logging, increasing this to 30 seconds reduces I/O significantly: "index.refresh_interval": "30s".
  • Disable replicas during initial load — When backfilling historical data, set replicas to 0 and restore them after.
  • Use keyword over text when possible — text fields create inverted indexes with analysis, which is expensive. Only use text for fields you need full-text search on.
  • Heap sizing — Give Elasticsearch no more than 50% of available RAM (and never more than 31 GB) for its JVM heap. The rest is used by the OS for filesystem cache, which Lucene relies on heavily.

Logstash Tuning

  • Pipeline workers — Set pipeline.workers to match the number of CPU cores.
  • Pipeline batch size — Increase pipeline.batch.size (default 125) to 500 or more for high-throughput scenarios.
  • Persistent queues — Enable persistent queues to prevent data loss during Logstash restarts: queue.type: persisted.
# /etc/logstash/logstash.yml
pipeline.workers: 8
pipeline.batch.size: 500
pipeline.batch.delay: 50
queue.type: persisted
queue.max_bytes: 4gb

Filebeat Tuning

# Increase the bulk size for high-volume hosts
output.logstash:
  hosts: ["logstash:5044"]
  bulk_max_size: 2048
  worker: 4

Common Pitfalls

Mapping explosions. If your application logs arbitrary key-value pairs (e.g., user-submitted form fields), Elasticsearch creates a mapping entry for each unique field name. Thousands of fields consume massive memory. Use "dynamic": "strict" or "dynamic": false on your mappings to prevent this.

Unstructured logs. Relying on grok to parse unstructured text is fragile and slow. Always emit structured JSON from your applications when possible.

Not using filter context in queries. Putting everything in must means Elasticsearch computes relevance scores unnecessarily. Use filter for exact matches — it is faster and cacheable.

Ignoring disk watermarks. Elasticsearch stops allocating shards when disk usage hits 85% (low watermark) and becomes read-only at 95% (flood watermark). Monitor disk usage and set up ILM before you get there.

Alternatives to the ELK Stack

The ELK stack is powerful but operationally complex. Here are alternatives worth considering.

Grafana Loki

Loki is a log aggregation system designed by Grafana Labs that indexes only metadata (labels), not the full text of log lines. This makes it dramatically cheaper to operate at scale. It integrates natively with Grafana for visualization. Loki is an excellent choice if you already use Grafana for metrics and want to add logging without the operational burden of Elasticsearch.

# Example Promtail config (Loki's log shipper)
server:
  http_listen_port: 9080

positions:
  filename: /tmp/positions.yaml

clients:
  - url: http://loki:3100/loki/api/v1/push

scrape_configs:
  - job_name: myapp
    static_configs:
      - targets: ["localhost"]
        labels:
          job: myapp
          __path__: /var/log/myapp/*.log

AWS CloudWatch Logs

If you are running on AWS, CloudWatch Logs provides a fully managed solution with no infrastructure to maintain. It integrates with CloudWatch Metrics for alerting and CloudWatch Insights for querying. The trade-off is less flexibility in querying and visualization compared to Elasticsearch and Kibana.

When to Choose What

Criteria ELK Stack Loki CloudWatch
Full-text search Excellent Basic (grep-like) Moderate
Operational cost High (self-managed) Low None (managed)
Storage cost High Low Moderate
Query flexibility Excellent Moderate Moderate
Dashboard ecosystem Kibana (rich) Grafana (rich) CloudWatch (basic)

For most backend engineering teams, the ELK stack remains the gold standard when you need powerful full-text search and complex aggregations across your logs. If your primary need is correlating logs with metrics and you want simpler operations, Loki with Grafana is increasingly the pragmatic choice.

Putting It All Together

Here is a minimal Docker Compose setup to run the full stack locally for development:

# docker-compose.yml
version: "3.8"
services:
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.12.0
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false
      - "ES_JAVA_OPTS=-Xms1g -Xmx1g"
    ports:
      - "9200:9200"
    volumes:
      - es-data:/usr/share/elasticsearch/data

  logstash:
    image: docker.elastic.co/logstash/logstash:8.12.0
    ports:
      - "5044:5044"
    volumes:
      - ./logstash/pipeline:/usr/share/logstash/pipeline
    depends_on:
      - elasticsearch

  kibana:
    image: docker.elastic.co/kibana/kibana:8.12.0
    ports:
      - "5601:5601"
    depends_on:
      - elasticsearch

volumes:
  es-data:

With this running, you can point your Filebeat or Python application at localhost:5044 (Logstash) and view your logs at http://localhost:5601 (Kibana).

The ELK stack is a foundational skill for backend engineers. Whether you are debugging a production incident at 2 AM or building an operations dashboard for your team, understanding how to collect, process, store, and query logs at scale will make you dramatically more effective.

Exercises

  1. Set Up a Local ELK Stack with Docker

    Use the Docker Compose configuration from the lesson to spin up Elasticsearch, Logstash, and Kibana locally. Write a Python script that generates 1,000 structured JSON log entries with varying log levels (INFO, WARNING, ERROR), random response times (50-5000ms), and multiple service names. Ship these logs to Logstash using either Filebeat or the python-logstash library. Verify in Kibana's Discover tab that all 1,000 documents appear and that fields are correctly parsed.

    Hint

    Start the stack with 'docker compose up -d', wait about 60 seconds for all services to initialize, then create your index pattern in Kibana at http://localhost:5601. Use Python's random module to generate varied log data and the logging module with a JSONFormatter to write structured output.

  2. Build a Logstash Pipeline with Grok Parsing

    Create a Logstash pipeline that ingests unstructured Nginx access logs from a file input. Use the grok filter to extract fields: client_ip, timestamp, method, request_path, status_code, bytes_sent, and user_agent. Add a geoip filter on the client_ip field. Conditionally tag any request with a status_code >= 500 as 'server_error'. Output to Elasticsearch with a daily index pattern. Test with at least 20 sample Nginx log lines covering 2xx, 4xx, and 5xx responses.

    Hint

    Use the built-in grok pattern %{COMBINEDAPACHELOG} which handles standard Nginx/Apache log formats. Test your grok patterns at https://grokdebugger.com before putting them in production config. Remember that grok is CPU-intensive, so structured JSON logging is always preferred when you control the source.

  3. Create a Kibana Operations Dashboard

    Using the log data from the first exercise, build a Kibana dashboard with at least four visualizations: (1) a line chart showing request volume over time with 1-minute intervals, (2) a pie chart breaking down log entries by level, (3) a data table showing the top 10 error messages, and (4) an area chart showing p50, p90, and p99 response time over time. Add a filter control that lets you select a specific service name to narrow all visualizations.

    Hint

    In Kibana, go to Dashboard > Create. Use the Lens editor for each visualization. For percentile charts, choose the 'Percentile' aggregation on the response_time_ms field. For the filter control, add an 'Options list' control panel at the top of the dashboard linked to the service.keyword field.

  4. Implement Index Lifecycle Management

    Create an ILM policy called 'app-logs-policy' that rolls over indices at 10GB or 1 day, moves data to a warm phase after 3 days with a force merge to 1 segment, and deletes data after 14 days. Create an index template that applies this policy to all indices matching 'app-logs-*'. Verify the policy is working by using the Elasticsearch _ilm/explain API to check the current phase of your index.

    Hint

    Use the Elasticsearch REST API: PUT _ilm/policy/app-logs-policy for the policy, then PUT _index_template/app-logs-template for the template. The template needs 'index.lifecycle.name' set to your policy name and 'index.lifecycle.rollover_alias' set to your write alias. Check status with GET app-logs-*/_ilm/explain.

Common Problems & Solutions

1 Elasticsearch rejects new documents with a 'cluster_block_exception' and index is read-only
Symptoms: You see errors like 'blocked by: [FORBIDDEN/12/index read-only / allow delete (api)]' in your application logs or Logstash output. New log entries are not being indexed.
Cause: Elasticsearch automatically sets indices to read-only when disk usage exceeds the flood-stage watermark (default 95%). This is a protective mechanism to prevent the node from running completely out of disk and crashing.
Solution: First, free up disk space by deleting old indices: 'DELETE /app-logs-2026.01.*'. Then release the read-only block on affected indices. Finally, set up ILM to prevent this from recurring.

Release the read-only block

PUT /app-logs-*/_settings { "index.blocks.read_only_allow_delete": null }

Check current disk usage

GET _cat/allocation?v

Adjust watermarks if needed (not recommended long-term)

PUT _cluster/settings { "persistent": { "cluster.routing.allocation.disk.watermark.flood_stage": "97%" } }

2 Logstash grok filter fails to parse log lines and tags them with '_grokparsefailure'
Symptoms: Documents in Elasticsearch have a 'tags' field containing '_grokparsefailure'. The 'message' field contains the raw unparsed log line, and extracted fields like status_code or client_ip are missing.
Cause: The grok pattern does not match the actual format of the incoming log lines. This commonly happens when the log format changes slightly (e.g., an extra field, different timestamp format) or when there are multi-line log entries like stack traces that break the pattern.
Solution: Use the Kibana Grok Debugger (Dev Tools > Grok Debugger) to test your pattern against actual failing log lines. Add a conditional to handle parse failures gracefully, and consider using a fallback pattern or switching to structured JSON logging to eliminate grok entirely.

filter { grok { match => { "message" => [ "%{COMBINEDAPACHELOG}", "%{GREEDYDATA:raw_message}" ] } tag_on_failure => ["_grok_parse_error"] }

# Route unparseable messages to a separate index if "_grok_parse_error" in [tags] { mutate { add_field => { "parse_status" => "failed" } } } }

3 Mapping explosion causes Elasticsearch to consume excessive memory and reject new fields
Symptoms: Elasticsearch logs show warnings like 'Limit of total fields [1000] has been exceeded'. New documents with previously unseen field names are rejected. Cluster memory usage is abnormally high.
Cause: Your application logs dynamic key-value pairs (e.g., arbitrary request parameters, user-submitted form data) as top-level fields. Each unique field name creates a mapping entry that consumes heap memory. With thousands of unique fields, this exhausts cluster resources.
Solution: Restructure your logging to avoid dynamic top-level fields. Put variable data in a nested structure or a single text field. Set the mapping to strict mode to prevent unknown fields from being indexed, or increase the field limit only if you genuinely need many fields.

BAD: Dynamic fields cause mapping explosion

logger.info("Request received", extra={ f"param_{key}": value for key, value in request.params.items() })

GOOD: Store dynamic data in a single field

logger.info("Request received", extra={ "request_params": json.dumps(dict(request.params)) })

Or set strict mapping to catch issues early

PUT /app-logs-*/_mapping

{

"dynamic": "strict"

}
