Certificates, Certificate Authorities, and mTLS

Intuition

A certificate is like a government-issued passport: the passport itself (certificate) contains your photo and name (public key and identity), and it is stamped by the government (Certificate Authority). When you cross a border, the guard verifies the stamp is legitimate. Mutual TLS is like a border where BOTH sides must show their passports before either is allowed through.

Certificates, Certificate Authorities, and mTLS

Introduction

Every time you visit a website over HTTPS, an invisible handshake takes place: the server presents a digital certificate to prove its identity, your browser verifies that certificate against a trusted authority, and then both sides negotiate an encrypted connection. This system -- called the Public Key Infrastructure (PKI) -- is the foundation of secure communication on the internet.

As a backend developer, you need to understand certificates for configuring HTTPS servers, setting up TLS termination, debugging connection failures, and implementing mutual TLS (mTLS) for service-to-service authentication in microservice architectures. In this lesson, you will learn how certificates work, how the chain of trust is established, and how to implement both standard TLS and mutual TLS in Node.js.

Public Key Cryptography: The Foundation

Before we can understand certificates, we need to understand asymmetric (public key) cryptography. Unlike symmetric encryption (one key for both encrypt and decrypt), asymmetric cryptography uses a key pair:

ASYMMETRIC CRYPTOGRAPHY:

  +-------------------+       +--------------------+
  |   Public Key      |       |   Private Key      |
  |   (share freely)  |       |   (keep secret)    |
  +-------------------+       +--------------------+

  Use Case 1: ENCRYPTION
  Anyone can encrypt with the public key.
  Only the private key holder can decrypt.

    Plaintext --[Public Key]--> Ciphertext --[Private Key]--> Plaintext

  Use Case 2: DIGITAL SIGNATURES
  The private key holder signs a message.
  Anyone with the public key can verify the signature.

    Message --[Private Key]--> Signature
    Message + Signature --[Public Key]--> Valid / Invalid

Generating a Key Pair with node:crypto

const { generateKeyPairSync } = require('node:crypto');

const { publicKey, privateKey } = generateKeyPairSync('rsa', {
  modulusLength: 2048,
  publicKeyEncoding: {
    type: 'spki',
    format: 'pem'
  },
  privateKeyEncoding: {
    type: 'pkcs8',
    format: 'pem'
  }
});

console.log(publicKey);
// -----BEGIN PUBLIC KEY-----
// MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8A...
// -----END PUBLIC KEY-----

console.log(privateKey);
// -----BEGIN PRIVATE KEY-----
// MIIEvQIBADANBgkqhkiG9w0BAQEFAASC...
// -----END PRIVATE KEY-----

Digital Signatures in Node.js

const { createSign, createVerify, generateKeyPairSync } = require('node:crypto');

const { publicKey, privateKey } = generateKeyPairSync('rsa', {
  modulusLength: 2048,
  publicKeyEncoding: { type: 'spki', format: 'pem' },
  privateKeyEncoding: { type: 'pkcs8', format: 'pem' }
});

// Sign a message
const message = 'This message is from the server';

const signer = createSign('SHA256');
signer.update(message);
const signature = signer.sign(privateKey, 'hex');
console.log('Signature:', signature);

// Verify the signature
const verifier = createVerify('SHA256');
verifier.update(message);
const isValid = verifier.verify(publicKey, signature, 'hex');
console.log('Valid:', isValid); // true

// Tampered message fails verification
const verifier2 = createVerify('SHA256');
verifier2.update('This message was tampered');
const isValid2 = verifier2.verify(publicKey, signature, 'hex');
console.log('Tampered valid:', isValid2); // false

What Is a Digital Certificate?

A digital certificate is a document that binds a public key to an identity (like a domain name). It is signed by a Certificate Authority (CA) to assert: "I have verified that the holder of this public key is indeed the entity they claim to be."

ANATOMY OF AN X.509 CERTIFICATE:

+----------------------------------------------------------+
| Certificate                                              |
|                                                          |
|  Subject:    CN=api.example.com                          |
|  Issuer:     CN=Let's Encrypt Authority X3               |
|  Valid From: 2025-01-01                                  |
|  Valid To:   2025-04-01                                  |
|  Public Key: RSA 2048-bit                                |
|  Serial:     03:a1:b2:c3:d4:e5:f6                       |
|                                                          |
|  Extensions:                                             |
|    Subject Alternative Names (SAN):                      |
|      DNS: api.example.com                                |
|      DNS: www.example.com                                |
|    Key Usage: Digital Signature, Key Encipherment        |
|                                                          |
|  Signature: (signed by the Issuer's private key)         |
+----------------------------------------------------------+

Key fields: - Subject -- the entity the certificate identifies (domain, organization) - Issuer -- the CA that signed the certificate - Validity period -- the certificate's start and expiration dates - Public Key -- the subject's public key - Subject Alternative Names (SAN) -- additional domains covered - Signature -- the issuer's digital signature over all the above fields

The Chain of Trust

Certificates form a chain from the server's certificate (leaf) up to a trusted root CA:

CHAIN OF TRUST:

+---------------------------+
| Root CA Certificate       |  Pre-installed in OS/browser
| (Self-signed)             |  trust stores. Highly secured.
| e.g., DigiCert Root       |
+---------------------------+
        |
        | signs
        v
+---------------------------+
| Intermediate CA Cert      |  Issued by the Root CA.
| e.g., Let's Encrypt R3   |  Adds a layer of security.
+---------------------------+
        |
        | signs
        v
+---------------------------+
| Leaf Certificate          |  Your server's certificate.
| CN=api.example.com        |  Sent to clients during TLS.
+---------------------------+


VERIFICATION PROCESS:

  1. Server sends: [Leaf Cert] + [Intermediate Cert]
  2. Client checks: Leaf signed by Intermediate? (verify signature)
  3. Client checks: Intermediate signed by Root? (verify signature)
  4. Client checks: Root is in my trust store? YES -> TRUSTED
  5. Client checks: Leaf cert's domain matches requested domain?
  6. Client checks: Certificate is not expired?
  7. Client checks: Certificate is not revoked? (CRL/OCSP)
  8. ALL checks pass -> TLS connection established

Certificate Formats

You will encounter several certificate file formats:

+----------+-----------------------------------------------------------+
| Format   | Description                                               |
+----------+-----------------------------------------------------------+
| PEM      | Base64-encoded, text format. Most common for web servers. |
|          | Files: .pem, .crt, .cer, .key                            |
|          | Begins with -----BEGIN CERTIFICATE-----                   |
+----------+-----------------------------------------------------------+
| DER      | Binary format. Common on Windows and Java.                |
|          | Files: .der, .cer                                        |
+----------+-----------------------------------------------------------+
| PKCS#12  | Binary bundle of cert + private key + chain.              |
|          | Files: .p12, .pfx                                        |
+----------+-----------------------------------------------------------+

Creating a Self-Signed Certificate

For development and testing, you can generate a self-signed certificate using OpenSSL:

# Generate a private key and self-signed certificate in one command
openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem \
  -sha256 -days 365 -nodes \
  -subj "/CN=localhost" \
  -addext "subjectAltName=DNS:localhost,IP:127.0.0.1"

Breaking down the options: - -x509 -- output a self-signed certificate (not a CSR) - -newkey rsa:2048 -- generate a new 2048-bit RSA key - -keyout key.pem -- write the private key here - -out cert.pem -- write the certificate here - -nodes -- do not encrypt the private key (no passphrase) - -subj -- set the subject fields - -addext -- add Subject Alternative Name extension

HTTPS Server in Node.js

Basic HTTPS Server

const https = require('node:https');
const fs = require('node:fs');
const express = require('express');

const app = express();

app.get('/', (req, res) => {
  res.json({ message: 'Hello over HTTPS!' });
});

const options = {
  key: fs.readFileSync('key.pem'),
  cert: fs.readFileSync('cert.pem')
};

https.createServer(options, app).listen(443, () => {
  console.log('HTTPS server running on port 443');
});

HTTPS with Certificate Chain

In production, you typically have a leaf certificate and an intermediate CA certificate. They must be provided together:

const options = {
  key: fs.readFileSync('/etc/ssl/private/server.key'),
  cert: fs.readFileSync('/etc/ssl/certs/server.crt'),
  ca: fs.readFileSync('/etc/ssl/certs/intermediate.crt')
};

// Alternatively, concatenate leaf + intermediate into one PEM file:
// cat server.crt intermediate.crt > fullchain.pem
const optionsAlt = {
  key: fs.readFileSync('/etc/ssl/private/server.key'),
  cert: fs.readFileSync('/etc/ssl/certs/fullchain.pem')
};

TLS Handshake

Understanding the TLS handshake helps with debugging connection issues:

TLS 1.3 HANDSHAKE (simplified):

  Client                                    Server
    |                                          |
    |--- ClientHello --------------------------->|
    |    (supported ciphers, TLS version,       |
    |     client random, key share)             |
    |                                          |
    |<--- ServerHello --------------------------|
    |     (chosen cipher, server random,        |
    |      key share)                           |
    |<--- Certificate -------------------------|
    |     (server's cert chain)                 |
    |<--- CertificateVerify --------------------|
    |     (signature proving key ownership)     |
    |<--- Finished -----------------------------|
    |                                          |
    |--- Finished ----------------------------->|
    |                                          |
    |<========= Encrypted Data ===============>|

  Total: 1 round trip (TLS 1.3) vs 2 round trips (TLS 1.2)

Mutual TLS (mTLS)

Standard TLS is one-way: only the server proves its identity. In mutual TLS (mTLS), both the client and the server present certificates to authenticate each other. This is widely used for:

  • Service-to-service communication in microservices
  • Zero-trust architectures where every connection is verified
  • API authentication as an alternative to API keys
STANDARD TLS vs MUTUAL TLS:

Standard TLS:
  Client -------> Server
  "Show me your cert"     Server proves identity
  Client trusts server    Client is anonymous

mTLS:
  Client <------> Server
  "Show me your cert"     Server proves identity
  "Show ME your cert"     Client proves identity
  Both sides verified     Both sides trusted

Setting Up mTLS in Node.js

First, generate certificates for the CA, server, and client:

# 1. Create a Certificate Authority (CA)
openssl req -x509 -newkey rsa:2048 -keyout ca-key.pem -out ca-cert.pem \
  -sha256 -days 3650 -nodes -subj "/CN=My Internal CA"

# 2. Generate server key and Certificate Signing Request (CSR)
openssl req -newkey rsa:2048 -keyout server-key.pem -out server-csr.pem \
  -nodes -subj "/CN=localhost"

# 3. Sign the server CSR with the CA
openssl x509 -req -in server-csr.pem -CA ca-cert.pem -CAkey ca-key.pem \
  -CAcreateserial -out server-cert.pem -days 365 -sha256 \
  -extfile <(echo "subjectAltName=DNS:localhost,IP:127.0.0.1")

# 4. Generate client key and CSR
openssl req -newkey rsa:2048 -keyout client-key.pem -out client-csr.pem \
  -nodes -subj "/CN=service-a"

# 5. Sign the client CSR with the same CA
openssl x509 -req -in client-csr.pem -CA ca-cert.pem -CAkey ca-key.pem \
  -CAcreateserial -out client-cert.pem -days 365 -sha256

mTLS Server

const https = require('node:https');
const fs = require('node:fs');
const express = require('express');

const app = express();

app.get('/', (req, res) => {
  // Access client certificate information
  const clientCert = req.socket.getPeerCertificate();

  if (!clientCert || !clientCert.subject) {
    return res.status(401).json({ error: 'No client certificate provided' });
  }

  res.json({
    message: 'mTLS connection established',
    client: {
      commonName: clientCert.subject.CN,
      issuer: clientCert.issuer.CN,
      validFrom: clientCert.valid_from,
      validTo: clientCert.valid_to,
      fingerprint: clientCert.fingerprint
    }
  });
});

const options = {
  key: fs.readFileSync('server-key.pem'),
  cert: fs.readFileSync('server-cert.pem'),
  ca: fs.readFileSync('ca-cert.pem'),         // Trust this CA for client certs
  requestCert: true,                           // Request client certificate
  rejectUnauthorized: true                     // Reject connections without valid cert
};

https.createServer(options, app).listen(8443, () => {
  console.log('mTLS server running on https://localhost:8443');
});

Key options: - requestCert: true -- the server asks the client for a certificate - rejectUnauthorized: true -- connections without a valid client cert are rejected - ca -- the CA certificate(s) used to validate client certificates

mTLS Client

const https = require('node:https');
const fs = require('node:fs');

const options = {
  hostname: 'localhost',
  port: 8443,
  path: '/',
  method: 'GET',
  key: fs.readFileSync('client-key.pem'),
  cert: fs.readFileSync('client-cert.pem'),
  ca: fs.readFileSync('ca-cert.pem')           // Trust the server's CA
};

const req = https.request(options, (res) => {
  let data = '';
  res.on('data', (chunk) => data += chunk);
  res.on('end', () => {
    console.log('Response:', JSON.parse(data));
  });
});

req.on('error', (err) => {
  console.error('Connection failed:', err.message);
});

req.end();

mTLS with Axios (more practical for services)

const axios = require('axios');
const https = require('node:https');
const fs = require('node:fs');

const httpsAgent = new https.Agent({
  key: fs.readFileSync('client-key.pem'),
  cert: fs.readFileSync('client-cert.pem'),
  ca: fs.readFileSync('ca-cert.pem')
});

async function callSecureService() {
  try {
    const response = await axios.get('https://localhost:8443/', {
      httpsAgent
    });
    console.log('Response:', response.data);
  } catch (error) {
    console.error('Request failed:', error.message);
  }
}

callSecureService();

Certificate-Based Authorization

With mTLS, you can authorize requests based on the client certificate's identity:

const ALLOWED_SERVICES = ['service-a', 'service-b', 'admin-service'];

function mTLSAuthMiddleware(req, res, next) {
  const cert = req.socket.getPeerCertificate();

  if (!cert || !cert.subject) {
    return res.status(401).json({ error: 'Client certificate required' });
  }

  const clientName = cert.subject.CN;

  if (!ALLOWED_SERVICES.includes(clientName)) {
    return res.status(403).json({
      error: 'Service not authorized',
      client: clientName
    });
  }

  // Attach client identity to request for downstream use
  req.clientService = clientName;
  next();
}

app.use(mTLSAuthMiddleware);

app.get('/internal/data', (req, res) => {
  console.log(`Request from: ${req.clientService}`);
  res.json({ data: 'sensitive internal data' });
});

Certificate Rotation and Management

Certificates expire, and rotating them without downtime is critical:

const tls = require('node:tls');
const fs = require('node:fs');
const https = require('node:https');

// Dynamic certificate loading -- reload certs without restart
function loadCertificates() {
  return {
    key: fs.readFileSync('/etc/ssl/private/server.key'),
    cert: fs.readFileSync('/etc/ssl/certs/server.crt'),
    ca: fs.readFileSync('/etc/ssl/certs/ca.crt')
  };
}

const server = https.createServer({
  SNICallback: (hostname, cb) => {
    // Reload certificates on each connection
    const certs = loadCertificates();
    const ctx = tls.createSecureContext(certs);
    cb(null, ctx);
  }
}, app);

server.listen(443);

For production environments, you should: 1. Automate renewal -- use tools like certbot (for Let's Encrypt) or Vault 2. Monitor expiration -- alert well before certificates expire (30 days out) 3. Use short-lived certificates -- especially for mTLS between services (e.g., 24-hour certs from a Vault PKI)

Inspecting Certificates

Useful commands for debugging certificate issues:

# View certificate details
openssl x509 -in cert.pem -text -noout

# Check what certificate a server presents
openssl s_client -connect api.example.com:443 -showcerts

# Verify a certificate against a CA
openssl verify -CAfile ca-cert.pem server-cert.pem

# Check certificate expiration
openssl x509 -in cert.pem -noout -dates

# View the full chain
openssl s_client -connect api.example.com:443 2>/dev/null | openssl x509 -noout -subject -issuer

Inspecting Certificates Programmatically

const tls = require('node:tls');
const { X509Certificate } = require('node:crypto');
const fs = require('node:fs');

// Read and inspect a local certificate
const certPem = fs.readFileSync('cert.pem');
const cert = new X509Certificate(certPem);

console.log('Subject:', cert.subject);
console.log('Issuer:', cert.issuer);
console.log('Valid From:', cert.validFrom);
console.log('Valid To:', cert.validTo);
console.log('Serial:', cert.serialNumber);
console.log('Fingerprint:', cert.fingerprint256);

// Check if certificate is currently valid
const now = new Date();
const notBefore = new Date(cert.validFrom);
const notAfter = new Date(cert.validTo);
const isValid = now >= notBefore && now <= notAfter;
console.log('Currently valid:', isValid);

// Check days until expiration
const daysLeft = Math.floor((notAfter - now) / (1000 * 60 * 60 * 24));
console.log('Days until expiration:', daysLeft);

// Verify a certificate was signed by a CA
const caCertPem = fs.readFileSync('ca-cert.pem');
const caCert = new X509Certificate(caCertPem);
console.log('Signed by CA:', cert.checkIssued(caCert));

Certificate Monitoring Endpoint

A health check endpoint that monitors certificate expiration:

const { X509Certificate } = require('node:crypto');
const fs = require('node:fs');

app.get('/health/tls', (req, res) => {
  try {
    const certPem = fs.readFileSync('/etc/ssl/certs/server.crt');
    const cert = new X509Certificate(certPem);
    const notAfter = new Date(cert.validTo);
    const daysLeft = Math.floor((notAfter - new Date()) / (1000 * 60 * 60 * 24));

    const status = daysLeft > 30 ? 'healthy' : daysLeft > 7 ? 'warning' : 'critical';

    res.json({
      status,
      certificate: {
        subject: cert.subject,
        issuer: cert.issuer,
        validTo: cert.validTo,
        daysUntilExpiry: daysLeft
      }
    });
  } catch (error) {
    res.status(500).json({ status: 'error', message: error.message });
  }
});

Let's Encrypt and ACME Protocol

Let's Encrypt provides free, automated TLS certificates. The typical setup for a Node.js server:

PRODUCTION SETUP WITH NGINX + LET'S ENCRYPT:

  Internet
     |
     v
+-----------+     +------------------+
|  Nginx    |---->| Node.js App      |
|  (port 443)     | (port 3000)      |
|  TLS termination| Plain HTTP       |
|  Let's Encrypt  |                  |
+-----------+     +------------------+

  Nginx handles:
  - TLS termination (decrypts HTTPS)
  - Certificate renewal (certbot)
  - Reverse proxy to Node.js

  Node.js handles:
  - Application logic
  - Receives plain HTTP from Nginx
  - Trusts Nginx for client info (X-Forwarded-For)

This is the recommended production pattern. Let Nginx handle TLS termination rather than terminating TLS directly in Node.js, as Nginx is significantly faster at TLS operations.

Summary

Digital certificates bind public keys to identities and are verified through a chain of trust from leaf certificates to trusted root CAs. In Node.js, you can create HTTPS servers by providing a private key and certificate to https.createServer. Mutual TLS (mTLS) extends standard TLS by requiring both parties to present certificates, making it ideal for service-to-service authentication. Use requestCert: true and rejectUnauthorized: true on the server side, and provide client certificates on the client side. For production, terminate TLS at a reverse proxy like Nginx, automate renewal with Let's Encrypt, and monitor certificate expiration proactively. When building microservices, consider mTLS as a strong alternative to API keys or JWT tokens for inter-service authentication.

Common Problems & Solutions

1 HTTPS server fails with 'error:0906D06C:PEM routines:PEM_read_bio:no start line'
Cause: The certificate or key file is not in PEM format, is empty, or the file path is wrong. This also occurs when the key and cert files are swapped (key passed as cert or vice versa).
Solution: Verify the file begins with '-----BEGIN CERTIFICATE-----' (for certs) or '-----BEGIN PRIVATE KEY-----' (for keys). Check that fs.readFileSync paths are correct and the files are readable. Ensure you are not passing the key file as the cert option.
2 mTLS client connection fails with 'UNABLE_TO_VERIFY_LEAF_SIGNATURE'
Cause: The client does not trust the CA that signed the server's certificate. The ca option is missing from the client's HTTPS agent, or the wrong CA certificate file was provided.
Solution: Pass the CA certificate that signed the server's cert in the client's ca option: new https.Agent({ ca: fs.readFileSync('ca-cert.pem'), key: ..., cert: ... }). If the server uses a different CA than the client certificates, both CAs must be included.
3 req.socket.getPeerCertificate() returns an empty object even though requestCert is true
Cause: The rejectUnauthorized option is set to false, which allows connections without valid client certificates. In this case, getPeerCertificate() returns an empty object for unauthenticated clients.
Solution: Set rejectUnauthorized: true to require valid client certificates. If you need to allow optional client certs, check if the returned object has a 'subject' property before accessing cert fields: if (cert && cert.subject) { ... }.
