Alerting Strategies and On-Call Best Practices

Intuition

Alerting is like a home security system. Bad alerting is a system that beeps every time a leaf blows past the window (alert fatigue). Good alerting only sounds when someone is actually trying to break in (user impact). The burn rate approach is like tracking how many suspicious events happened in the last hour — one leaf is fine, but a pattern of increasing activity triggers an alert.

Alerting Strategies and On-Call Best Practices

Introduction

Monitoring without alerting is just data collection. Alerting without strategy is noise. Effective alerting wakes you up when users are impacted and stays quiet when they are not. This lesson covers alerting strategies based on SLOs, on-call best practices, and building a sustainable incident response culture.

SLO-Based Alerting (Burn Rate)

Instead of alerting on individual metric thresholds, alert when the error budget is being consumed too quickly:

Error Budget Burn Rate:
  If SLO = 99.9% over 30 days
  Error budget = 0.1% = 43.2 minutes

  Burn rate 1x: Using budget at normal rate (sustainable)
  Burn rate 2x: Would exhaust budget in 15 days (warning)
  Burn rate 10x: Would exhaust budget in 3 days (critical)
  Burn rate 100x: Would exhaust budget in 7.2 hours (PAGE!)

Multi-Window Burn Rate Alerts

// Prometheus alerting rules (alertmanager)
// Save as alerting-rules.yml

/*
groups:
  - name: slo-burn-rate
    rules:
      # Fast burn: 14.4x burn rate over 1 hour (pages)
      - alert: HighErrorBurnRate
        expr: |
          (
            sum(rate(http_errors_total[1h]))
            / sum(rate(http_requests_total[1h]))
          ) > (14.4 * 0.001)
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Error budget burning 14.4x normal rate"
          description: "At this rate, 30-day error budget exhausts in ~2 days"

      # Slow burn: 3x burn rate over 6 hours (ticket)
      - alert: ElevatedErrorBurnRate
        expr: |
          (
            sum(rate(http_errors_total[6h]))
            / sum(rate(http_requests_total[6h]))
          ) > (3 * 0.001)
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Error budget burning 3x normal rate"
          description: "At this rate, 30-day error budget exhausts in ~10 days"

      # Latency SLO: p99 > 500ms for 5+ minutes
      - alert: HighLatency
        expr: |
          histogram_quantile(0.99,
            sum(rate(http_request_duration_seconds_bucket[5m])) by (le)
          ) > 0.5
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "P99 latency exceeds 500ms SLO"
*/

Implementing Alerts in Node.js

import client from 'prom-client';

// Health check with detailed status
class HealthChecker {
  constructor() {
    this.checks = new Map();
  }

  register(name, checkFn, { critical = false, timeout = 5000 } = {}) {
    this.checks.set(name, { checkFn, critical, timeout });
  }

  async runAll() {
    const results = {};
    let healthy = true;

    for (const [name, { checkFn, critical, timeout }] of this.checks) {
      try {
        const start = Date.now();
        await Promise.race([
          checkFn(),
          new Promise((_, reject) =>
            setTimeout(() => reject(new Error('Timeout')), timeout)
          ),
        ]);
        results[name] = {
          status: 'healthy',
          duration: Date.now() - start,
        };
      } catch (err) {
        results[name] = {
          status: 'unhealthy',
          error: err.message,
          critical,
        };
        if (critical) healthy = false;
      }
    }

    return { healthy, checks: results };
  }
}

const health = new HealthChecker();
health.register('database', () => prisma.$queryRaw`SELECT 1`, { critical: true });
health.register('redis', () => redis.ping(), { critical: true });
health.register('kafka', () => producer.send({ topic: '__healthcheck', messages: [] }));

// Health endpoint
app.get('/health', async (req, res) => {
  const result = await health.runAll();
  res.status(result.healthy ? 200 : 503).json(result);
});

// Liveness probe (is the process alive?)
app.get('/health/live', (req, res) => {
  res.json({ status: 'alive', uptime: process.uptime() });
});

// Readiness probe (is the service ready to accept traffic?)
app.get('/health/ready', async (req, res) => {
  const result = await health.runAll();
  res.status(result.healthy ? 200 : 503).json(result);
});

Alerting Anti-Patterns

BAD Alerts (cause alert fatigue):
  ❌ CPU > 80% (may be normal during peak hours)
  ❌ Memory > 70% (applications use memory, that's fine)
  ❌ Any 5xx error (one error is not an outage)
  ❌ Latency > 200ms (one slow request is not an issue)

GOOD Alerts (actionable, user-impacting):
  ✓ Error budget burn rate > 14.4x for 2 minutes
  ✓ P99 latency > 500ms for 5 minutes
  ✓ Health check failing for 30 seconds
  ✓ Database replication lag > 30 seconds
  ✓ Disk usage > 90% (will cause outage if not addressed)
  ✓ Certificate expiring in < 7 days

On-Call Best Practices

Rotation Structure

Primary on-call: First responder
Secondary on-call: Backup if primary doesn't respond

Weekly rotation:
  Week 1: Alice (primary), Bob (secondary)
  Week 2: Bob (primary), Charlie (secondary)
  Week 3: Charlie (primary), Alice (secondary)

Rules:
  - Max 1 week on-call per 3-4 weeks
  - Handoff meeting at rotation change
  - On-call compensation (time off or pay)
  - Escalation policy if no response in 15 min

Incident Response

Incident Timeline:
  T+0min: Alert fires
  T+5min: On-call acknowledges
  T+10min: Initial assessment, severity assigned
  T+15min: Communication posted in incident channel
  T+30min: Mitigation attempted
  T+60min: Escalate if not mitigated
  T+?: Incident resolved
  T+1-3 days: Blameless post-mortem

Runbooks

// Automated runbook execution
class RunbookExecutor {
  constructor(runbooks) {
    this.runbooks = runbooks;
  }

  async execute(alertName) {
    const runbook = this.runbooks[alertName];
    if (!runbook) {
      return { status: 'no-runbook', message: 'No automated runbook available' };
    }

    const results = [];
    for (const step of runbook.steps) {
      try {
        const result = await step.action();
        results.push({ step: step.name, status: 'success', result });

        if (step.resolves && result.resolved) {
          return { status: 'auto-resolved', steps: results };
        }
      } catch (err) {
        results.push({ step: step.name, status: 'failed', error: err.message });
        break;
      }
    }

    return { status: 'needs-human', steps: results };
  }
}

const runbooks = {
  'HighDatabaseConnections': {
    steps: [
      {
        name: 'Check active queries',
        action: async () => {
          const queries = await prisma.$queryRaw`
            SELECT pid, now() - pg_stat_activity.query_start AS duration, query
            FROM pg_stat_activity
            WHERE state = 'active' AND query_start < now() - interval '1 minute'
          `;
          return { longRunningQueries: queries };
        },
      },
      {
        name: 'Kill long-running queries',
        action: async () => {
          const killed = await prisma.$queryRaw`
            SELECT pg_terminate_backend(pid)
            FROM pg_stat_activity
            WHERE state = 'active' AND query_start < now() - interval '5 minutes'
            AND pid <> pg_backend_pid()
          `;
          return { resolved: true, queriesKilled: killed.length };
        },
        resolves: true,
      },
    ],
  },
};

Common Pitfalls

  1. Too many alerts: If the on-call person gets > 2 pages per week, alert quality is too low. Reduce alerts.
  2. Alerts without runbooks: Every alert should link to a runbook explaining what to check and how to mitigate.
  3. No post-mortems: Without learning from incidents, the same issues recur. Blameless post-mortems are essential.
  4. On-call without support: On-call should always have a secondary, an escalation path, and authority to make changes.

Summary

Effective alerting is based on SLOs and error budget burn rates, not arbitrary thresholds. Good alerts are actionable, user-impacting, and rare enough to avoid fatigue. On-call rotations should be sustainable with proper compensation, runbooks, and escalation paths. Post-mortems focus on systemic improvements, not blame. The goal is a system that wakes you up only when users are meaningfully impacted.

Common Problems & Solutions

1 On-call person paged 10+ times per week
Cause: Alerts are too sensitive (low thresholds), too many non-actionable alerts, or flapping alerts that trigger repeatedly.
Solution: Review and tune alert thresholds. Remove alerts that are never acted upon. Add appropriate 'for' duration (e.g., 5m) to prevent flapping. Use burn rate alerting instead of threshold alerts.
2 Incidents keep recurring without improvement
Cause: No post-mortem process, or post-mortems happen but action items are not tracked.
Solution: Mandate blameless post-mortems for all P1/P2 incidents. Track action items in a project management tool. Review completion in weekly team meetings.
