Providers and Services: Dependency Injection Basics

Intuition

Think of the DI container as a staffing agency. You tell it what roles you need (providers), it finds or creates the right workers, and assigns them to teams (modules). When a manager (controller) needs help, the agency automatically supplies the right workers -- the manager never has to recruit directly.

Providers and Services: Dependency Injection Basics

Introduction

Providers are the backbone of NestJS. They're plain TypeScript classes decorated with @Injectable() that can be injected into other classes through the constructor. Services, repositories, factories, helpers -- they're all providers. The NestJS Dependency Injection (DI) container manages their lifecycle and wiring.

What Is Dependency Injection?

Dependency Injection is a design pattern where a class receives its dependencies from an external source rather than creating them itself.

// WITHOUT DI -- tightly coupled
class UsersController {
  private usersService: UsersService;

  constructor() {
    this.usersService = new UsersService(
      new UsersRepository(new DatabaseConnection('localhost', 5432))
    );
  }
}

// WITH DI -- loosely coupled
@Controller('users')
class UsersController {
  constructor(private readonly usersService: UsersService) {}
  // NestJS handles creating and wiring everything
}

The DI container creates instances, manages their lifecycle, and injects them where needed.

How NestJS DI Works

+-------------------------------------------+
|           NestJS DI Container              |
|                                            |
|  1. Scan all @Module() declarations        |
|  2. Find all providers and controllers     |
|  3. Resolve dependency graph               |
|  4. Create instances (singletons)          |
|  5. Inject dependencies via constructors   |
+-------------------------------------------+
        |              |              |
        v              v              v
  +-----------+  +-----------+  +-----------+
  | Controller|  |  Service  |  |  Service  |
  |  (has     |  |  (has     |  |  (has     |
  |  Service) |  |  Repo)    |  |  other)   |
  +-----------+  +-----------+  +-----------+

When the app starts, NestJS: 1. Reads the @Module() metadata 2. Identifies all providers and controllers 3. Analyzes constructor parameters using TypeScript metadata 4. Creates instances in the correct order 5. Caches them as singletons (by default)

Creating a Service

nest g service users

// users/users.service.ts
import { Injectable } from '@nestjs/common';

@Injectable()
export class UsersService {
  private users = [
    { id: '1', name: 'Alice', email: 'alice@example.com' },
    { id: '2', name: 'Bob', email: 'bob@example.com' },
  ];

  findAll() {
    return this.users;
  }

  findOne(id: string) {
    return this.users.find(user => user.id === id);
  }

  create(data: { name: string; email: string }) {
    const user = { id: String(this.users.length + 1), ...data };
    this.users.push(user);
    return user;
  }

  update(id: string, data: Partial<{ name: string; email: string }>) {
    const user = this.findOne(id);
    if (user) Object.assign(user, data);
    return user;
  }

  remove(id: string) {
    const index = this.users.findIndex(user => user.id === id);
    if (index > -1) return this.users.splice(index, 1)[0];
    return null;
  }
}

Registering and Injecting

// users/users.module.ts
@Module({
  controllers: [UsersController],
  providers: [UsersService],  // Register the provider
  exports: [UsersService],     // Make available to other modules
})
export class UsersModule {}

// users/users.controller.ts
@Controller('users')
export class UsersController {
  // Constructor injection -- the standard pattern
  constructor(private readonly usersService: UsersService) {}

  @Get()
  findAll() {
    return this.usersService.findAll();
  }
}

The private readonly modifier is a TypeScript shorthand that declares and assigns a class property in one step.

Provider Types

NestJS supports several provider patterns beyond simple classes:

Standard Provider (Class-based)

// This shorthand:
providers: [UsersService]

// Is equivalent to:
providers: [
  {
    provide: UsersService,
    useClass: UsersService,
  },
]

Value Provider

providers: [
  {
    provide: 'API_KEY',
    useValue: 'my-secret-api-key',
  },
]

Inject with @Inject():

@Injectable()
export class ApiService {
  constructor(@Inject('API_KEY') private apiKey: string) {}
}

Factory Provider

providers: [
  {
    provide: 'DATABASE_CONNECTION',
    useFactory: async (config: ConfigService) => {
      const connection = await createConnection({
        host: config.get('DB_HOST'),
        port: config.get('DB_PORT'),
      });
      return connection;
    },
    inject: [ConfigService], // Dependencies for the factory
  },
]

Existing Provider (Alias)

providers: [
  UsersService,
  {
    provide: 'AliasedUsersService',
    useExisting: UsersService, // Same instance as UsersService
  },
]

Injection Tokens

When the provider isn't a class, you need an injection token. You can use strings or Symbols:

// Using a string token
@Inject('DATABASE_CONNECTION') private db: Connection

// Using a Symbol (recommended for uniqueness)
export const DATABASE_TOKEN = Symbol('DATABASE_CONNECTION');

// Register
providers: [{ provide: DATABASE_TOKEN, useValue: dbConnection }]

// Inject
constructor(@Inject(DATABASE_TOKEN) private db: Connection) {}

Service-to-Service Injection

Services can depend on other services:

@Injectable()
export class UsersRepository {
  constructor(private readonly db: DatabaseService) {}

  async findById(id: string): Promise<User | null> {
    return this.db.query('SELECT * FROM users WHERE id = $1', [id]);
  }
}

@Injectable()
export class UsersService {
  constructor(private readonly usersRepo: UsersRepository) {}

  async getUser(id: string): Promise<User> {
    const user = await this.usersRepo.findById(id);
    if (!user) throw new NotFoundException(`User ${id} not found`);
    return user;
  }
}

@Injectable()
export class AuthService {
  constructor(private readonly usersService: UsersService) {}

  async validateUser(email: string, password: string): Promise<User | null> {
    const user = await this.usersService.findByEmail(email);
    if (user && await bcrypt.compare(password, user.password)) {
      return user;
    }
    return null;
  }
}

The DI container resolves the full dependency chain automatically:

AuthService -> UsersService -> UsersRepository -> DatabaseService

Property-Based Injection

While constructor injection is preferred, NestJS also supports property injection:

@Injectable()
export class UsersService {
  @Inject(LoggerService)
  private readonly logger: LoggerService;
}

Use this sparingly -- constructor injection makes dependencies explicit and easier to test.

Optional Dependencies

import { Injectable, Optional, Inject } from '@nestjs/common';

@Injectable()
export class HttpService {
  constructor(
    @Optional() @Inject('HTTP_OPTIONS') private options?: HttpOptions,
  ) {
    this.options = options || { timeout: 5000 };
  }
}

@Optional() prevents NestJS from throwing if the dependency isn't available.

The Service Layer Pattern

A well-structured NestJS app separates concerns:

Controller (HTTP layer)
    |
    v
Service (Business logic)
    |
    v
Repository (Data access)
    |
    v
Database

// users/users.repository.ts
@Injectable()
export class UsersRepository {
  constructor(private readonly prisma: PrismaService) {}

  async findAll(page: number, limit: number) {
    return this.prisma.user.findMany({
      skip: (page - 1) * limit,
      take: limit,
    });
  }

  async findById(id: string) {
    return this.prisma.user.findUnique({ where: { id } });
  }
}

// users/users.service.ts
@Injectable()
export class UsersService {
  constructor(private readonly usersRepo: UsersRepository) {}

  async findAll(page: number, limit: number) {
    const users = await this.usersRepo.findAll(page, limit);
    return users.map(user => this.sanitize(user));
  }

  async findById(id: string) {
    const user = await this.usersRepo.findById(id);
    if (!user) throw new NotFoundException();
    return this.sanitize(user);
  }

  private sanitize(user: User) {
    const { password, ...result } = user;
    return result;
  }
}

// users/users.controller.ts
@Controller('users')
export class UsersController {
  constructor(private readonly usersService: UsersService) {}

  @Get()
  findAll(@Query() query: PaginationDto) {
    return this.usersService.findAll(query.page, query.limit);
  }
}

Testing with DI

DI makes testing straightforward -- inject mocks instead of real implementations:

describe('UsersService', () => {
  let service: UsersService;
  let repository: jest.Mocked<UsersRepository>;

  beforeEach(async () => {
    const module = await Test.createTestingModule({
      providers: [
        UsersService,
        {
          provide: UsersRepository,
          useValue: {
            findAll: jest.fn(),
            findById: jest.fn(),
          },
        },
      ],
    }).compile();

    service = module.get(UsersService);
    repository = module.get(UsersRepository);
  });

  it('should throw NotFoundException when user not found', async () => {
    repository.findById.mockResolvedValue(null);
    await expect(service.findById('999')).rejects.toThrow(NotFoundException);
  });
});

Common Pitfalls

  1. Forgetting @Injectable() -- Without this decorator, TypeScript doesn't emit metadata, and DI can't resolve constructor params.
  2. Business logic in controllers -- Controllers should only handle HTTP concerns. Move logic to services.
  3. Not registering providers -- A provider must be listed in a module's providers array before it can be injected.
  4. Circular service dependencies -- Service A injects Service B which injects Service A. Use forwardRef() or restructure.

Exercises

  1. Build a Service Layer

    Create a TasksModule with TasksController, TasksService, and TasksRepository. The repository should manage an in-memory array of tasks. The service should add business logic (validation, status transitions). The controller should only handle HTTP concerns.

    Hint

    Define a Task interface with id, title, description, status. The repository handles CRUD on the array. The service validates status transitions (e.g., can't go from DONE back to TODO).

  2. Use Multiple Provider Types

    Create a NotificationModule that uses: a class-based EmailService, a value provider for 'NOTIFICATION_CONFIG', and a factory provider that creates a MailerService based on the config. Inject everything into a NotificationService.

    Hint

    Use useClass for EmailService, useValue for config, and useFactory with inject: ['NOTIFICATION_CONFIG'] for MailerService.

  3. Write Unit Tests with Mock Providers

    Write comprehensive unit tests for a UsersService that depends on UsersRepository and CacheService. Use NestJS's Test.createTestingModule to provide mock implementations.

    Hint

    Use jest.fn() to create mock methods. Verify that the service calls the repository with correct arguments and handles null results by throwing NotFoundException.

Common Problems & Solutions

1 Nest can't resolve dependencies of the MyService (?)
Symptoms: Error message shows '?' at the position of the unresolvable parameter.
Cause: The question mark (?) indicates NestJS can't determine the type of a constructor parameter. Common causes: missing @Injectable(), missing @Inject() for non-class tokens, or a circular dependency.
Solution: 1) Ensure the provider class has @Injectable(). 2) For non-class tokens (strings, symbols), use @Inject('TOKEN_NAME'). 3) For circular deps, use forwardRef(). 4) Check that the provider is registered in the module.
2 Provider instance is different across modules
Cause: The provider is registered separately in multiple modules' providers arrays, creating a new instance per module instead of sharing one.
Solution: Register the provider in one module and export it. Other modules should import that module, not re-register the provider. Alternatively, make the owning module @Global().
3 Factory provider returns undefined
Cause: The useFactory function doesn't return a value, or the async factory's promise resolves to undefined.
Solution: Ensure the factory function has an explicit return statement. For async factories, verify the awaited operation produces a value. Add error handling inside the factory.
