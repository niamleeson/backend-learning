Error Handling and API Error Responses

Intuition

Error responses are part of your API contract, not afterthoughts. A well-structured error tells the client what went wrong, why, and often how to fix it -- this dramatically improves developer experience and debugging speed.

Error Handling and API Error Responses

Consistent error handling is critical for API usability. Instead of returning bare status codes or HTML error pages, APIs should return structured JSON error responses with useful details. Flask's error handler system lets you customize error responses globally.

from flask import Flask, jsonify
from werkzeug.exceptions import HTTPException

app = Flask(__name__)

class APIError(Exception):
    def __init__(self, message, status_code=400, details=None):
        self.message = message
        self.status_code = status_code
        self.details = details or {}

@app.errorhandler(APIError)
def handle_api_error(error):
    return jsonify({
        'error': error.message,
        'status': error.status_code,
        'details': error.details
    }), error.status_code

@app.errorhandler(HTTPException)
def handle_http_error(error):
    return jsonify({
        'error': error.description,
        'status': error.code
    }), error.code

@app.errorhandler(500)
def handle_internal_error(error):
    return jsonify({
        'error': 'Internal server error',
        'status': 500
    }), 500

@app.route('/api/items/<int:item_id>')
def get_item(item_id):
    items = {1: 'Widget'}
    if item_id not in items:
        raise APIError('Item not found', status_code=404,
                       details={'item_id': item_id})
    return jsonify({'id': item_id, 'name': items[item_id]})

@app.route('/api/items', methods=['POST'])
def create_item():
    from flask import request
    data = request.get_json(silent=True)
    if not data or 'name' not in data:
        raise APIError('Validation failed', status_code=422,
                       details={'name': 'This field is required'})
    return jsonify(data), 201

Key Takeaways: - Create a custom exception class (APIError) for application-specific errors with structured details. - Register handlers for both custom exceptions and Werkzeug's HTTPException to catch all error types. - Always return JSON error responses from API endpoints, never HTML.

Exercises

  1. Standardized Error Responses

    Create an error handling system that returns RFC 7807 Problem Details (application/problem+json) format with type, title, status, detail, and instance fields for all errors.

    Hint

    Set the response Content-Type to 'application/problem+json' and include a 'type' field that points to a URI describing the error category.

  2. Validation Error Aggregation

    Build a form/JSON validation system that collects all field errors before returning them in a single 422 response, rather than stopping at the first error.

    Hint

    Validate each field independently, collect errors in a dictionary, and raise your custom error only if the dictionary is non-empty.

Common Problems & Solutions

1 Flask returns HTML error pages instead of JSON for 404 and 500 errors.
Cause: Default Flask error handlers return HTML. Without custom handlers, API clients receive HTML they cannot parse.
Solution: Register @app.errorhandler for 404, 500, and HTTPException to return jsonify() responses. Handle the generic HTTPException to catch all Werkzeug errors.
2 Custom exception handler is not being called.
Cause: The exception class used in @app.errorhandler does not match the raised exception type.
Solution: Ensure the raised exception is an instance of the exact class registered with errorhandler. Check for inheritance mismatches.
