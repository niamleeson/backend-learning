Design an Online Code Editor (Replit)

Intuition

An online code editor is fundamentally a thin browser client connected to a per-user sandboxed compute environment, where the main challenge is provisioning and isolating thousands of concurrent sandboxes with low startup latency. Pre-warming container pools is the critical optimization that makes the experience feel instant.

Design an Online Code Editor (Replit)

Requirements

  • Provide a browser-based IDE with syntax highlighting, autocomplete, and file management
  • Execute user code in isolated sandboxed environments per session
  • Support multiple programming languages with language-specific toolchains
  • Allow users to share and fork projects; persist workspace state

Architecture

A web frontend delivers the editor UI (Monaco/CodeMirror). Each user session spins up an isolated container (Docker/microVM) with the appropriate language runtime. A WebSocket connection streams terminal I/O between browser and container. A workspace service persists files to object storage.

Key Components

  • Editor Frontend: Monaco editor in browser with LSP integration for autocomplete
  • Container Orchestrator: Provisions, monitors, and recycles per-user sandboxes (Kubernetes/Firecracker)
  • Execution Engine: Runs code inside the container, captures stdout/stderr, enforces resource limits
  • Workspace Storage: Git-backed or object-store-backed file persistence per project

Scaling

Pre-warm container pools per language to reduce cold start. Use Firecracker microVMs for fast boot (<150ms) and strong isolation. Auto-scale container hosts based on active session count.

Trade-offs

  • Containers are flexible but heavier; microVMs are lighter and more secure but less compatible
  • Full LSP per session provides rich IDE features but is resource-intensive
  • Persistent storage per project increases cost; ephemeral workspaces are cheaper but frustrate users

Exercises

  1. Build a WebSocket-Based Terminal Relay

    Create a server that spawns a subprocess (e.g., Python REPL) per WebSocket connection, relays stdin/stdout between the browser and the process, and cleans up on disconnect.

    Hint

    Use pty.openpty() to create a pseudo-terminal, spawn the process attached to it, and use select/poll to shuttle bytes between the pty and the WebSocket.

  2. Implement Resource-Limited Code Execution

    Build an execution engine that runs user-submitted code with a 5-second timeout, 256MB memory limit, and no network access. Test with CPU bombs and fork bombs.

    Hint

    Use cgroups (Linux) or ulimit to enforce memory and CPU limits. Use seccomp or a network namespace with no interfaces to block network access.

Common Problems & Solutions

1 Cold start takes 10+ seconds when a user selects an uncommon language
Cause: No pre-warmed containers exist for that language, so the system must pull the image and start fresh
Solution: Maintain a small pre-warmed pool for all supported languages, and use layered container images with a shared base to speed up image pulls.
2 A malicious user escapes the container and accesses host resources
Cause: The container runs with excessive privileges or shares the host network namespace
Solution: Use rootless containers or Firecracker microVMs, apply seccomp profiles, drop all capabilities, and run with a read-only root filesystem.
