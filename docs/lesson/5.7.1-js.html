Hexagonal Architecture: Ports and Adapters

Intuition

Hexagonal Architecture is like an electrical appliance with standardized plugs (ports). Your toaster's heating logic (domain) does not care whether electricity comes from a wall socket, a generator, or a battery (adapters). The plug shape (port interface) is the contract. You can test the toaster with a test power supply, and deploy it in any country by swapping the adapter.

Hexagonal Architecture: Ports and Adapters

Hexagonal Architecture (also called Ports and Adapters) organizes code so that business logic is completely isolated from external concerns like databases, HTTP frameworks, and third-party services. This makes your application testable, flexible, and technology-agnostic.

The Core Idea

              +---------------------------+
              |      External World       |
              |  (HTTP, CLI, Messages)    |
              +----------+----------------+
                         |
                    [Driving Adapters]
                    (Controllers, CLI)
                         |
              +----------v----------------+
              |       Input Ports         |
              |     (Use Case interfaces) |
              +----------+----------------+
                         |
              +----------v----------------+
              |    APPLICATION CORE       |
              |                           |
              |  Domain Entities          |
              |  Business Rules           |
              |  Use Cases                |
              |                           |
              +----------+----------------+
                         |
              +----------v----------------+
              |      Output Ports         |
              |  (Repository interfaces)  |
              +----------+----------------+
                         |
                   [Driven Adapters]
                   (Prisma, Redis, S3)
                         |
              +----------v----------------+
              |      External World       |
              |  (DB, Cache, Email, etc)  |
              +---------------------------+

Key Terminology

  • Ports: Interfaces that define how the application communicates with the outside
  • Input Ports (Driving): How the outside calls the application (use cases)
  • Output Ports (Driven): How the application calls the outside (repositories, services)
  • Adapters: Implementations of ports
  • Driving Adapters: Controllers, CLI commands, GraphQL resolvers
  • Driven Adapters: Database repositories, email senders, API clients

Implementation

Domain Layer (Zero Dependencies)

// domain/entities/post.entity.ts
// NO imports from Prisma, Express, NestJS, etc.
export class Post {
  constructor(
    public readonly id: string,
    public title: string,
    public body: string,
    public status: PostStatus,
    public readonly authorId: string,
    public tags: string[],
    public readonly createdAt: Date,
    public updatedAt: Date,
    public publishedAt: Date | null,
  ) {}

  publish(): void {
    if (this.status !== PostStatus.DRAFT) {
      throw new DomainError('Only drafts can be published');
    }
    this.status = PostStatus.PUBLISHED;
    this.publishedAt = new Date();
    this.updatedAt = new Date();
  }

  isOwnedBy(userId: string): boolean {
    return this.authorId === userId;
  }
}

Output Ports (Interfaces)

// application/ports/output/post.repository.port.ts
// Defines what the application NEEDS from persistence
export interface PostRepositoryPort {
  findById(id: string): Promise<Post | null>;
  findAll(options: FindOptions): Promise<{ posts: Post[]; total: number }>;
  save(post: Post): Promise<Post>;
  update(id: string, data: Partial<Post>): Promise<Post>;
  delete(id: string): Promise<void>;
}

// application/ports/output/notification.port.ts
export interface NotificationPort {
  notifyUser(userId: string, message: string): Promise<void>;
  notifyFollowers(userId: string, message: string): Promise<void>;
}

// application/ports/output/event-bus.port.ts
export interface EventBusPort {
  publish(event: DomainEvent): Promise<void>;
}

Input Ports (Use Case Interfaces)

// application/ports/input/create-post.port.ts
export interface CreatePostPort {
  execute(command: CreatePostCommand): Promise<Post>;
}

export interface CreatePostCommand {
  title: string;
  body: string;
  tags: string[];
  authorId: string;
}

Use Cases (Application Layer)

// application/use-cases/create-post.use-case.ts
import { CreatePostPort, CreatePostCommand } from '../ports/input/create-post.port';
import { PostRepositoryPort } from '../ports/output/post.repository.port';
import { EventBusPort } from '../ports/output/event-bus.port';
import { Post, PostStatus } from '../../domain/entities/post.entity';

export class CreatePostUseCase implements CreatePostPort {
  constructor(
    private postRepo: PostRepositoryPort,   // Output port
    private eventBus: EventBusPort,         // Output port
  ) {}

  async execute(command: CreatePostCommand): Promise<Post> {
    const post = new Post(
      generateId(),
      command.title,
      command.body,
      PostStatus.DRAFT,
      command.authorId,
      command.tags,
      new Date(),
      new Date(),
      null,
    );

    const saved = await this.postRepo.save(post);
    await this.eventBus.publish(new PostCreatedEvent(saved));
    return saved;
  }
}

// application/use-cases/publish-post.use-case.ts
export class PublishPostUseCase implements PublishPostPort {
  constructor(
    private postRepo: PostRepositoryPort,
    private notifier: NotificationPort,
    private eventBus: EventBusPort,
  ) {}

  async execute(command: PublishPostCommand): Promise<Post> {
    const post = await this.postRepo.findById(command.postId);
    if (!post) throw new NotFoundError('Post', command.postId);
    if (!post.isOwnedBy(command.userId)) throw new ForbiddenError();

    post.publish(); // Domain logic

    const updated = await this.postRepo.update(post.id, {
      status: post.status,
      publishedAt: post.publishedAt,
    });

    await this.eventBus.publish(new PostPublishedEvent(updated));
    await this.notifier.notifyFollowers(post.authorId, `New post: ${post.title}`);

    return updated;
  }
}

Driven Adapters (Implement Output Ports)

// infrastructure/adapters/driven/persistence/prisma-post.repository.ts
import { PrismaClient } from '@prisma/client';
import { PostRepositoryPort } from '../../../../application/ports/output/post.repository.port';
import { Post } from '../../../../domain/entities/post.entity';

export class PrismaPostRepository implements PostRepositoryPort {
  constructor(private prisma: PrismaClient) {}

  async findById(id: string): Promise<Post | null> {
    const record = await this.prisma.post.findUnique({ where: { id } });
    return record ? this.toDomain(record) : null;
  }

  async save(post: Post): Promise<Post> {
    const record = await this.prisma.post.create({
      data: this.toPersistence(post),
    });
    return this.toDomain(record);
  }

  // Map between persistence and domain models
  private toDomain(record: any): Post {
    return new Post(
      record.id, record.title, record.body, record.status,
      record.authorId, record.tags, record.createdAt,
      record.updatedAt, record.publishedAt,
    );
  }

  private toPersistence(post: Post): any {
    return {
      id: post.id, title: post.title, body: post.body,
      status: post.status, authorId: post.authorId, tags: post.tags,
    };
  }

  // ... other methods
}

Driving Adapters (Call Input Ports)

// infrastructure/adapters/driving/rest/post.controller.ts
import { CreatePostPort } from '../../../../application/ports/input/create-post.port';
import { PublishPostPort } from '../../../../application/ports/input/publish-post.port';

export class PostController {
  constructor(
    private createPostUseCase: CreatePostPort,
    private publishPostUseCase: PublishPostPort,
  ) {}

  async create(req: Request, res: Response) {
    const post = await this.createPostUseCase.execute({
      title: req.body.title,
      body: req.body.body,
      tags: req.body.tags || [],
      authorId: req.user.id,
    });
    res.status(201).json(post);
  }

  async publish(req: Request, res: Response) {
    const post = await this.publishPostUseCase.execute({
      postId: req.params.id,
      userId: req.user.id,
    });
    res.json(post);
  }
}

Composition Root

// main.ts - Wire everything together
const prisma = new PrismaClient();

// Create driven adapters
const postRepo = new PrismaPostRepository(prisma);
const notifier = new EmailNotificationAdapter(emailClient);
const eventBus = new RedisEventBus(redis);

// Create use cases
const createPost = new CreatePostUseCase(postRepo, eventBus);
const publishPost = new PublishPostUseCase(postRepo, notifier, eventBus);

// Create driving adapters
const postController = new PostController(createPost, publishPost);

// Mount routes
app.post('/api/posts', (req, res) => postController.create(req, res));
app.post('/api/posts/:id/publish', (req, res) => postController.publish(req, res));

The Dependency Rule

Dependencies point INWARD:

  Infrastructure -> Application -> Domain

  Domain depends on NOTHING
  Application depends only on Domain
  Infrastructure depends on Application and Domain

Never import from infrastructure in domain or application layers.

Exercises

  1. Refactor to Hexagonal Architecture

    Take an existing Express API where controllers directly call Prisma and refactor it to hexagonal architecture. Create domain entities, define input/output ports, implement use cases, and create adapters. Verify the domain layer has zero external dependencies.

    Hint

    Start from the inside out: define domain entities first, then ports, then use cases, then adapters. The domain folder should have zero imports from node_modules.

  2. Multiple Driving Adapters

    Create a single set of use cases that can be driven by both an Express REST controller and a GraphQL resolver. Both adapters should use the same CreatePostPort interface.

    Hint

    Define the input port once. Create a REST controller adapter and a GraphQL resolver adapter that both call the same use case implementation.

  3. Swap Driven Adapters

    Implement the NotificationPort with three adapters: EmailNotificationAdapter, SlackNotificationAdapter, and ConsoleNotificationAdapter (for development). Configure which one is used based on environment variables.

    Hint

    Create a factory function that reads process.env.NOTIFICATION_ADAPTER and returns the appropriate implementation.

Common Problems & Solutions

1 Too many files and interfaces for a simple app
Cause: Hexagonal architecture was applied to a small CRUD application that does not benefit from the abstraction.
Solution: Use hexagonal architecture for complex domains with multiple adapters or changing requirements. For simple CRUD, a simpler layered architecture (controller -> service -> repository) is fine.
2 Domain entities import from Prisma or NestJS
Cause: Developers accidentally used @nestjs/common decorators or Prisma types in domain entities.
Solution: Enforce the dependency rule with linting rules or TypeScript project references that prevent domain from importing infrastructure. The domain folder should have zero imports from node_modules except standard lib.
3 Mapping between domain and persistence models is tedious
Cause: Every repository method needs to convert between Prisma models and domain entities.
Solution: Create mapper classes (PostMapper.toDomain(), PostMapper.toPersistence()) to centralize the conversion. Accept the mapping overhead as the cost of decoupling.
