Triggers, Views, and Stored Procedures

Intuition

Triggers, views, and procedures push logic into the database itself, so the rules are enforced no matter which application connects to it.

Triggers, Views, and Stored Procedures

These database objects encapsulate logic at the database level, running automatically (triggers), simplifying queries (views), or providing reusable routines (procedures).

-- View: virtual table from a query
CREATE VIEW active_orders AS
SELECT o.id, c.name, o.total, o.created_at
FROM orders o JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'active';

-- Materialized view (cached)
CREATE MATERIALIZED VIEW monthly_stats AS
SELECT DATE_TRUNC('month', created_at) AS month, SUM(total)
FROM orders GROUP BY 1;

-- Trigger: auto-update timestamp
CREATE FUNCTION update_modified() RETURNS trigger AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END; $$ LANGUAGE plpgsql;

CREATE TRIGGER set_updated
BEFORE UPDATE ON orders
FOR EACH ROW EXECUTE FUNCTION update_modified();

-- Stored procedure
CREATE PROCEDURE transfer_funds(from_id INT, to_id INT, amt NUMERIC)
LANGUAGE plpgsql AS $$
BEGIN
    UPDATE accounts SET balance = balance - amt WHERE id = from_id;
    UPDATE accounts SET balance = balance + amt WHERE id = to_id;
END; $$;

Key Takeaways: - Views simplify complex queries; materialized views cache results for performance. - Triggers fire automatically on INSERT/UPDATE/DELETE for auditing and timestamps. - Stored procedures run multi-statement logic in a single CALL, reducing round trips.

Exercises

  1. Audit Trigger

    Create an audit_log table and a trigger that records every UPDATE to the users table, storing old and new values.

    Hint

    Use OLD and NEW in a BEFORE UPDATE trigger. Insert a row into audit_log with OLD., NEW., and now().

  2. Updatable View

    Create a view joining two tables and add INSTEAD OF triggers to allow INSERT/UPDATE through the view.

    Hint

    CREATE VIEW, then CREATE TRIGGER INSTEAD OF INSERT ON the view that inserts into the underlying tables.

Common Problems & Solutions

1 Trigger causes infinite recursion.
Cause: An UPDATE trigger modifies the same table, re-firing the trigger.
Solution: Use pg_trigger_depth() to prevent recursion, or restructure to avoid self-referencing updates.
2 Materialized view has stale data.
Cause: Materialized views do not auto-refresh; they show data from the last REFRESH.
Solution: Schedule REFRESH MATERIALIZED VIEW CONCURRENTLY via pg_cron or application-level scheduling.
