kafkajs: Producers, Consumers, Topics, and Partitions

Intuition

Kafka is like a newspaper distribution system. The printing press (producer) publishes editions (messages) organized by section (topic). Each section has multiple delivery routes (partitions). Subscribers (consumer groups) each get their own copy of every edition. Unlike a mailbox where mail is removed after reading, Kafka keeps all editions in the archive for a configured retention period, allowing new subscribers to read from the beginning.

kafkajs: Producers, Consumers, Topics, and Partitions

Introduction

Apache Kafka is a distributed event streaming platform designed for high-throughput, fault-tolerant data pipelines. Unlike message queues (RabbitMQ, BullMQ) where messages are consumed and removed, Kafka is an append-only log where events are retained and can be replayed. kafkajs is the leading Node.js client for Kafka.

Setup and Connection

import { Kafka, logLevel } from 'kafkajs';

const kafka = new Kafka({
  clientId: 'my-app',
  brokers: ['kafka-1:9092', 'kafka-2:9092', 'kafka-3:9092'],
  logLevel: logLevel.WARN,
  retry: {
    initialRetryTime: 100,
    retries: 8,
  },
  connectionTimeout: 3000,
  requestTimeout: 25000,
});

// Create topics (usually done via admin, not in app code)
const admin = kafka.admin();
await admin.connect();
await admin.createTopics({
  topics: [
    {
      topic: 'orders',
      numPartitions: 6,
      replicationFactor: 3,
    },
    {
      topic: 'user-events',
      numPartitions: 12,
      replicationFactor: 3,
      configEntries: [
        { name: 'retention.ms', value: '604800000' }, // 7 days
        { name: 'cleanup.policy', value: 'delete' },
      ],
    },
  ],
});
await admin.disconnect();

Producer

const producer = kafka.producer({
  allowAutoTopicCreation: false,
  idempotent: true, // Exactly-once semantics
  maxInFlightRequests: 5,
  transactionTimeout: 30000,
});

await producer.connect();

// Send single message
await producer.send({
  topic: 'orders',
  messages: [
    {
      key: 'order-123',              // Determines partition
      value: JSON.stringify({
        orderId: 'order-123',
        userId: 'user-42',
        items: [{ productId: 'prod-1', qty: 2 }],
        total: 59.99,
        createdAt: new Date().toISOString(),
      }),
      headers: {
        'event-type': 'OrderCreated',
        'correlation-id': 'req-abc-123',
        'source': 'order-service',
      },
    },
  ],
});

// Send batch
await producer.sendBatch({
  topicMessages: [
    {
      topic: 'orders',
      messages: orders.map(order => ({
        key: order.id,
        value: JSON.stringify(order),
        headers: { 'event-type': 'OrderCreated' },
      })),
    },
    {
      topic: 'user-events',
      messages: [
        {
          key: 'user-42',
          value: JSON.stringify({ type: 'order_placed', orderId: 'order-123' }),
        },
      ],
    },
  ],
});

// Graceful shutdown
process.on('SIGTERM', async () => {
  await producer.disconnect();
});

// Singleton producer
let producerInstance;

async function getProducer() {
  if (!producerInstance) {
    producerInstance = kafka.producer({ idempotent: true });
    await producerInstance.connect();
  }
  return producerInstance;
}

Consumer

const consumer = kafka.consumer({
  groupId: 'order-processing-group',
  sessionTimeout: 30000,
  heartbeatInterval: 3000,
  maxBytesPerPartition: 1048576, // 1MB
  maxWaitTimeInMs: 5000,
});

await consumer.connect();
await consumer.subscribe({
  topics: ['orders'],
  fromBeginning: false,  // Only new messages
});

await consumer.run({
  autoCommit: true,
  autoCommitInterval: 5000,
  autoCommitThreshold: 100,

  eachMessage: async ({ topic, partition, message, heartbeat }) => {
    const event = JSON.parse(message.value.toString());
    const eventType = message.headers['event-type']?.toString();

    console.log({
      topic,
      partition,
      offset: message.offset,
      key: message.key?.toString(),
      eventType,
      timestamp: message.timestamp,
    });

    try {
      switch (eventType) {
        case 'OrderCreated':
          await handleOrderCreated(event);
          break;
        case 'OrderPaid':
          await handleOrderPaid(event);
          break;
      }

      // Send heartbeat for long processing
      await heartbeat();
    } catch (err) {
      console.error(`Error processing message:`, err);
      // Message will be retried based on consumer config
    }
  },
});

// Batch processing (higher throughput)
await consumer.run({
  eachBatch: async ({ batch, resolveOffset, heartbeat, isRunning, isStale }) => {
    for (const message of batch.messages) {
      if (!isRunning() || isStale()) break;

      const event = JSON.parse(message.value.toString());
      await processEvent(event);
      resolveOffset(message.offset);
      await heartbeat();
    }
  },
});

Consumer Groups and Partition Assignment

Topic: orders (6 partitions)

Consumer Group: order-processors (3 consumers)
  Consumer 1: Partitions [0, 1]
  Consumer 2: Partitions [2, 3]
  Consumer 3: Partitions [4, 5]

If Consumer 3 dies:
  Consumer 1: Partitions [0, 1, 4]  ← rebalanced
  Consumer 2: Partitions [2, 3, 5]  ← rebalanced

Consumer Group: analytics (1 consumer)
  Consumer 1: Partitions [0, 1, 2, 3, 4, 5]  ← reads same data independently

Multiple consumer groups can independently consume the same topic. This is key for event-driven architectures: one topic, many subscribers.

Manual Offset Management

await consumer.run({
  autoCommit: false, // Manual commit

  eachMessage: async ({ topic, partition, message }) => {
    try {
      await processMessage(message);

      // Commit after successful processing
      await consumer.commitOffsets([{
        topic,
        partition,
        offset: (parseInt(message.offset) + 1).toString(),
      }]);
    } catch (err) {
      // Don't commit — message will be redelivered
      console.error('Processing failed, will retry:', err);
    }
  },
});

Common Pitfalls

  1. No message key: Without a key, messages are distributed round-robin. Related events for the same entity may go to different partitions and be processed out of order.
  2. Too few partitions: Partitions are the unit of parallelism. Too few partitions limit consumer scaling.
  3. Consumer processing too slow: If processing takes longer than session timeout, the consumer is considered dead and partitions are rebalanced.
  4. Not handling rebalances: When partitions are reassigned, in-progress messages may be re-delivered. Ensure idempotent processing.

Summary

Kafka provides durable, ordered, replayable event streams via topics and partitions. Producers send messages with keys that determine partition assignment. Consumer groups enable parallel processing with automatic partition assignment. Use kafkajs for the Node.js client, idempotent producers for exactly-once semantics, and manual offset commits for at-least-once processing guarantees. Kafka excels as the backbone for event-driven microservices architectures.

Exercises

    Set up Kafka with Docker Compose. Build a producer that sends order events and a consumer that processes them. Use message keys for partition affinity.

    Hint

    Use docker-compose with confluentinc/cp-kafka image. Set message key to orderId for ordering guarantee per order.

  2. Multi-Consumer Group Processing

    Create two consumer groups consuming the same topic: one for order processing and one for analytics. Verify both groups independently receive all messages.

    Hint

    Create two consumers with different groupId values. Both subscribe to the same topic.

  3. Exactly-Once Processing

    Implement idempotent message processing with manual offset commits. Ensure that even if a consumer restarts mid-batch, no messages are processed twice.

    Hint

    Use autoCommit: false. Track processed message IDs in the database. Only commit offset after database write succeeds.

Common Problems & Solutions

1 Messages for the same entity processed out of order
Cause: Messages are sent without a key, so they land on different partitions. Partitions are consumed independently.
Solution: Always set a message key that groups related messages (e.g., orderId, userId). Messages with the same key go to the same partition and are processed in order.
2 Consumer group constantly rebalancing
Cause: Consumer processing takes too long, exceeding the session timeout. Kafka thinks the consumer is dead.
Solution: Increase session.timeout.ms. Call heartbeat() during long processing. Reduce per-message processing time or increase partition count for more parallelism.
