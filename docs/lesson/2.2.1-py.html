unittest: Test Cases, Assertions, and Runners

Intuition

unittest gives you a structured template for tests: set up state, run the code, check results, clean up. Each test method is isolated so one failure does not affect others.

unittest: Test Cases, Assertions, and Runners

Python's built-in unittest module provides a class-based framework for writing and organizing tests. It uses TestCase classes with assertion methods.

import unittest

def calculate_total(prices: list[float], tax: float = 0.1) -> float:
    subtotal = sum(prices)
    return round(subtotal * (1 + tax), 2)

class TestCalculateTotal(unittest.TestCase):
    def setUp(self):
        self.prices = [10.0, 20.0, 30.0]

    def test_basic_total(self):
        result = calculate_total(self.prices)
        self.assertEqual(result, 66.0)

    def test_zero_tax(self):
        result = calculate_total(self.prices, tax=0)
        self.assertEqual(result, 60.0)

    def test_empty_list(self):
        self.assertEqual(calculate_total([]), 0.0)

    def test_raises_on_invalid(self):
        with self.assertRaises(TypeError):
            calculate_total("not a list")

    def tearDown(self):
        self.prices = None

if __name__ == '__main__':
    unittest.main()

Key Takeaways: - Each test method must start with test_ - setUp / tearDown run before/after each test method - Use specific assertions: assertEqual, assertTrue, assertRaises, assertAlmostEqual - Run with python -m unittest discover to find all tests automatically - setUpClass / tearDownClass run once per class for expensive setup

Exercises

  1. Test a User Class

    Write a TestUser class testing a User class with methods set_email(email) and full_name(). Include tests for valid input, invalid email (raises ValueError), and edge cases.

    Hint

    Use setUp to create a User instance. Use assertRaises as a context manager for the invalid email test.

  2. Test with setUpClass

    Write tests for a database-like class using setUpClass to initialize a shared connection and tearDownClass to close it. Ensure individual tests use setUp for per-test data.

    Hint

    Decorate setUpClass/tearDownClass with @classmethod. Use cls instead of self in those methods.

Common Problems & Solutions

1 Tests are not discovered when running `python -m unittest discover`.
Cause: Test files do not follow the naming convention `test_*.py` or test methods do not start with `test_`.
Solution: Name files `test_*.py` and prefix all test methods with `test_`. Use `-p` flag to customize the pattern if needed.
2 Tests interfere with each other, causing intermittent failures.
Cause: Shared mutable state between tests (class-level variables modified in tests).
Solution: Use `setUp` to create fresh state for each test. Avoid modifying class-level attributes in test methods.
