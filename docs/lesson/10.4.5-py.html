Design a Metrics/Monitoring System (Datadog)

Intuition

A monitoring system is a specialized streaming pipeline where every metric is a time-series append, and the core tension is between retaining high-resolution data for accurate alerting and compressing it for cost-effective long-term storage. Tagging is what makes the system powerful, but tag cardinality is what makes it expensive.

Design a Metrics/Monitoring System (Datadog)

Requirements

  • Ingest millions of time-series data points per second from agents on every host
  • Support multi-dimensional tagging (host, service, region, custom labels)
  • Provide real-time dashboards, alerting with thresholds/anomaly detection
  • Retain high-resolution data for 15 days, downsampled data for 1+ year

Architecture

Agents on each host collect and batch metrics, sending them to a regional intake API. The intake writes to a stream (Kafka) consumed by a storage writer and an alerting evaluator in parallel. Storage uses a purpose-built time-series database with columnar compression. A query engine serves dashboard and alert queries.

Key Components

  • Agent: Collects system and application metrics, pre-aggregates, batches sends
  • Intake/Router: Validates, tags, and routes metrics to Kafka topics by type
  • TSDB Storage: Write-optimized columnar store (e.g., InfluxDB, custom engine) with rollup compaction
  • Alert Evaluator: Continuously evaluates alert rules against the stream; fires notifications

Scaling

Partition by metric name + tag combination. Use rollup/downsampling jobs to compress old data. Separate read and write paths so dashboard queries don't impact ingestion.

Trade-offs

  • Pre-aggregation at the agent reduces volume but loses granularity
  • Pull-based collection (Prometheus) vs. push-based: pull is simpler to debug; push handles ephemeral hosts better
  • Exact alerting on raw data is expensive; evaluating on rolled-up data may miss brief spikes

Exercises

  1. Build a Simple TSDB with Rollups

    Implement an in-memory time-series store that accepts (metric, tags, timestamp, value) tuples. Add a background job that rolls up data older than 1 hour into 1-minute averages.

    Hint

    Store raw points in a sorted list per series. The rollup job groups points by minute, computes avg/min/max, replaces the raw points with the summary.

  2. Implement a Threshold Alert Evaluator

    Build an evaluator that checks a metric every 10 seconds and fires an alert if the value exceeds a threshold for 3 consecutive evaluation windows.

    Hint

    Maintain a counter per alert rule; increment on breach, reset on normal. Fire when counter reaches 3. This avoids alerting on transient spikes.

Common Problems & Solutions

1 High-cardinality tags (e.g., user_id) cause the TSDB to run out of memory
Cause: Each unique tag combination creates a new time series, and unbounded tag values explode the series count
Solution: Enforce a cardinality limit per metric, reject or drop tags exceeding the limit, and educate users to use low-cardinality tags for metrics.
2 Alert evaluator falls behind during ingestion spikes, causing delayed alerts
Cause: The evaluator reads from the same Kafka consumer group as storage and cannot keep up
Solution: Give the alert evaluator its own consumer group with higher priority partitioning, and allow it to evaluate on a sampled or pre-aggregated stream.
