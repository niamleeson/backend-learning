Docker Networking and Volumes

Intuition

Docker networking is like rooms in an office building. Services on the same network are in the same room and can talk directly. Different networks are different rooms with locked doors between them. Volumes are like filing cabinets -- even if you demolish and rebuild the room (container), the cabinet (volume) keeps your files safe.

Docker Networking and Volumes

Containers need to communicate with each other and persist data. Docker networking and volumes handle these two critical concerns.

Docker Networking

# Create a custom network
docker network create app-network

# Containers on the same network reach each other by name
docker run -d --name db --network app-network postgres:16
docker run -d --name web --network app-network -p 8000:8000 myapp

# Compose: isolate with multiple networks
services:
  web:
    networks: [frontend, backend]
  db:
    networks: [backend]  # Not accessible from frontend
  nginx:
    networks: [frontend]

networks:
  frontend:
  backend:

Docker Volumes

# Named volumes (managed by Docker)
docker volume create pgdata
docker run -d -v pgdata:/var/lib/postgresql/data postgres:16

# Bind mounts (map host directory, for development)
docker run -d -v $(pwd)/src:/app/src myapp

services:
  db:
    volumes:
      - pgdata:/var/lib/postgresql/data     # Named volume
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # Bind mount
volumes:
  pgdata:

Network Types

Type Use Case
bridge Default. Containers on same host
host Container shares host network
overlay Multi-host (Docker Swarm)

Common Mistakes

  • Using the default bridge network (no DNS resolution by name)
  • Using bind mounts for database storage in production
  • Forgetting that container filesystems are ephemeral

Key Takeaways

  • Use custom bridge networks so containers resolve each other by name
  • Use named volumes for persistent data, bind mounts for development
  • Isolate services using multiple networks

Exercises

  1. Network Isolation

    Create a Docker Compose setup with three services: nginx (frontend), a Python API (backend), and PostgreSQL (database). Configure networks so nginx can reach the API, the API can reach the database, but nginx cannot directly reach the database.

    Hint

    Create two networks: 'frontend' and 'backend'. Put nginx and API on frontend, API and database on backend.

  2. Volume Backup and Restore

    Write a bash script that backs up a named Docker volume to a tar.gz file and can restore it. Test with a PostgreSQL database volume.

    Hint

    Use a temporary container: docker run --rm -v pgdata:/data -v $(pwd):/backup alpine tar czf /backup/pgdata.tar.gz /data

Common Problems & Solutions

1 Containers can't communicate by service name
Symptoms: ['getaddrinfo failed', 'Name or service not known']
Cause: Using the default bridge network which doesn't provide DNS resolution between containers.
Solution: Create a custom network with `docker network create mynet` and attach containers to it, or use Docker Compose which creates a custom network automatically.
2 Database data lost after docker compose down
Cause: Running `docker compose down -v` which deletes named volumes, or not using a named volume at all.
Solution: Define a named volume in the `volumes:` top-level key and mount it to the database data directory. Only use `-v` flag with `down` when you intentionally want to reset.
3 Bind mount shows empty directory inside container
Symptoms: ["Application can't find files", 'ls inside container shows empty /app']
Cause: The host path doesn't exist or is incorrect. On macOS/Windows, the path may not be shared with Docker.
Solution: Verify the host path exists. On Docker Desktop, check File Sharing settings to ensure the directory is in a shared path.
