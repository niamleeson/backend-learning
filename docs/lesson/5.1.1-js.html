API Versioning Strategies

Intuition

API versioning is like publishing editions of a textbook. Students using the 3rd edition should not have their book suddenly rewritten mid-semester. You release a 4th edition, keep the 3rd available for a while, and eventually stop printing it.

API Versioning Strategies

As your API evolves, you will inevitably need to make breaking changes. API versioning gives consumers a stable contract while you iterate. Getting versioning wrong leads to broken clients, angry developers, and painful migrations.

Why Version Your API?

APIs are contracts. When external or internal consumers depend on your endpoints, changing the shape of a response or removing a field can break their applications. Versioning lets you:

  • Introduce breaking changes without disrupting existing consumers
  • Deprecate old behavior on a timeline
  • Run multiple versions simultaneously during migration periods
  • Communicate change clearly through the version identifier

The Four Main Strategies

1. URI Path Versioning

The most common and visible approach. The version is embedded directly in the URL path.

GET /api/v1/users
GET /api/v2/users

Express implementation:

import express from 'express';

const app = express();

// v1 routes
const v1Router = express.Router();
v1Router.get('/users', (req, res) => {
  res.json({
    users: [
      { id: 1, name: 'Alice', email: 'alice@example.com' }
    ]
  });
});

// v2 routes - different response shape
const v2Router = express.Router();
v2Router.get('/users', (req, res) => {
  res.json({
    data: [
      {
        id: 1,
        firstName: 'Alice',
        lastName: 'Smith',
        contactInfo: { email: 'alice@example.com' }
      }
    ],
    meta: { total: 1, page: 1 }
  });
});

app.use('/api/v1', v1Router);
app.use('/api/v2', v2Router);

NestJS implementation:

// app.module.ts
import { Module } from '@nestjs/common';
import { RouterModule } from '@nestjs/core';
import { UsersV1Module } from './v1/users.module';
import { UsersV2Module } from './v2/users.module';

@Module({
  imports: [
    UsersV1Module,
    UsersV2Module,
    RouterModule.register([
      { path: 'api/v1', module: UsersV1Module },
      { path: 'api/v2', module: UsersV2Module },
    ]),
  ],
})
export class AppModule {}

Alternatively, NestJS supports built-in URI versioning:

// main.ts
import { VersioningType } from '@nestjs/common';

app.enableVersioning({
  type: VersioningType.URI,
  defaultVersion: '1',
  prefix: 'api/v',
});

// users.controller.ts
@Controller('users')
export class UsersController {
  @Version('1')
  @Get()
  findAllV1() { /* ... */ }

  @Version('2')
  @Get()
  findAllV2() { /* ... */ }
}

Pros: Explicit, easy to cache, easy to route, visible in logs. Cons: URL pollution, clients must update URLs to migrate.

2. Header Versioning

Use a custom header to specify the desired version.

GET /api/users
X-API-Version: 2

// Express middleware approach
function versionRouter(req, res, next) {
  const version = req.headers['x-api-version'] || '1';
  req.apiVersion = parseInt(version, 10);
  next();
}

app.use(versionRouter);

app.get('/api/users', (req, res) => {
  if (req.apiVersion === 2) {
    return res.json({ data: [/* v2 shape */], meta: {} });
  }
  return res.json({ users: [/* v1 shape */] });
});

NestJS built-in header versioning:

app.enableVersioning({
  type: VersioningType.HEADER,
  header: 'X-API-Version',
});

Pros: Clean URLs, version not exposed in URL. Cons: Harder to test in a browser, less visible, not cacheable by URL.

3. Query Parameter Versioning

GET /api/users?version=2

app.get('/api/users', (req, res) => {
  const version = parseInt(req.query.version as string) || 1;
  // route to correct handler based on version
});

Pros: Easy to implement, works in browsers. Cons: Easy to forget, pollutes query params, caching complexities.

4. Content Negotiation (Accept Header)

GitHub's approach — use the Accept header with a vendor media type.

GET /api/users
Accept: application/vnd.myapi.v2+json

app.get('/api/users', (req, res) => {
  const accept = req.headers.accept || '';
  const match = accept.match(/application\/vnd\.myapi\.v(\d+)\+json/);
  const version = match ? parseInt(match[1]) : 1;

  if (version === 2) {
    res.type('application/vnd.myapi.v2+json');
    return res.json({ data: [/* v2 */] });
  }
  res.type('application/vnd.myapi.v1+json');
  return res.json({ users: [/* v1 */] });
});

NestJS:

app.enableVersioning({
  type: VersioningType.MEDIA_TYPE,
  key: 'v=',
});
// Client sends: Accept: application/json;v=2

Pros: RESTful purist approach, clean URLs. Cons: Complex, hard to test, not widely understood.

Version Lifecycle and Deprecation

Don't just create versions — plan their lifecycle:

// Deprecation middleware
function deprecationWarning(version: string, sunsetDate: string) {
  return (req, res, next) => {
    res.set('Deprecation', 'true');
    res.set('Sunset', sunsetDate);
    res.set('Link', '</api/v3/docs>; rel="successor-version"');
    console.warn(`Deprecated API v${version} called: ${req.path}`);
    next();
  };
}

app.use('/api/v1', deprecationWarning('1', 'Sat, 01 Jun 2025 00:00:00 GMT'));

Non-Breaking Changes (No Version Bump Needed)

Not every change requires a new version:

  • Adding a new optional field to a response
  • Adding a new endpoint
  • Adding optional query parameters
  • Adding a new enum value (if clients handle unknown values)

Breaking Changes (Version Bump Required)

  • Removing or renaming a field
  • Changing a field's type
  • Changing URL structure
  • Removing an endpoint
  • Making an optional field required

Code Organization Strategies

Shared Service, Separate Controllers

src/
  users/
    users.service.ts        # shared business logic
    v1/
      users.controller.ts   # v1 response shape
      users.dto.ts
    v2/
      users.controller.ts   # v2 response shape
      users.dto.ts

Transformation Layer

// Keep one canonical internal model, transform per version
class UserTransformer {
  static toV1(user: InternalUser) {
    return { id: user.id, name: `${user.firstName} ${user.lastName}` };
  }

  static toV2(user: InternalUser) {
    return {
      id: user.id,
      firstName: user.firstName,
      lastName: user.lastName,
      contactInfo: { email: user.email },
    };
  }
}

Real-World Recommendation

For most teams, URI path versioning is the pragmatic choice. It is explicit, easy to route, easy to cache, and easy to understand. Save content negotiation for when you have a good reason. Many successful APIs (Stripe, Twilio, GitHub) use path versioning or date-based versioning (/v2024-01-01/).

Exercises

  1. Implement URI Path Versioning in Express

    Create an Express app with /api/v1/products and /api/v2/products. V1 returns { products: [...] }. V2 returns { data: [...], meta: { total, page } }. Both should share the same underlying product service.

    Hint

    Use express.Router() for each version and mount them with app.use('/api/v1', v1Router).

  2. Add Deprecation Headers

    Write middleware that adds Deprecation, Sunset, and Link headers to v1 routes. Log a warning every time a deprecated endpoint is called, including the client's user-agent.

    Hint

    Use res.set() for headers and req.headers['user-agent'] for the client info.

  3. NestJS Built-In Versioning

    Set up a NestJS app using VersioningType.URI. Create a UsersController with @Version('1') and @Version('2') decorated methods that return different response shapes.

    Hint

    Call app.enableVersioning() in main.ts before app.listen().

  4. Build a Version-Aware API Gateway

    Create an Express middleware that reads the version from either the URL path, a header, or a query param (in that priority order) and attaches it to the request object. Write tests for all three sources.

    Hint

    Parse the URL for /v{n}/ pattern first, then check X-API-Version header, then ?version= query param.

Common Problems & Solutions

1 Clients break when you add a new version
Cause: Default version is not set, so clients without explicit version get routed to the newest (breaking) version.
Solution: Always set a default version. Clients without a version header or path should get the oldest supported version, not the newest.
2 Code duplication across versions
Cause: Each version has a full copy of controllers and services.
Solution: Share business logic in services. Only the controller/transformer layer should differ between versions. Use a transformation layer to map internal models to version-specific DTOs.
3 Too many active versions to maintain
Cause: No deprecation policy or sunset dates communicated to consumers.
Solution: Publish a versioning policy (e.g., support N-1 versions, 12-month sunset window). Use Deprecation and Sunset headers. Monitor usage of old versions and actively communicate timelines.
