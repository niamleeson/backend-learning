Pub/Sub and Lua Scripting

Intuition

Pub/Sub is like a radio broadcast: publishers transmit on a channel and anyone tuned in receives the message. No recording, no replay. Lua scripting is like handing a recipe card to the Redis chef: instead of shouting instructions one at a time from outside the kitchen (which lets others interrupt between steps), you give the chef the complete recipe and they execute all steps without interruption.

Pub/Sub and Lua Scripting

Introduction

Redis Pub/Sub enables real-time messaging between services, while Lua scripting lets you execute complex atomic operations on the Redis server. Together, they extend Redis beyond simple key-value storage into a messaging system and a programmable data store.

Redis Pub/Sub

Pub/Sub Model:

  Publisher A --->  Channel: "orders"  ---> Subscriber 1
  Publisher B --->  Channel: "orders"  ---> Subscriber 2
                    Channel: "events"  ---> Subscriber 3

  Messages are fire-and-forget (no persistence)
  Subscribers only see messages published after they subscribe

Basic Pub/Sub with ioredis

import Redis from 'ioredis';

// Subscriber (needs a dedicated connection)
const sub = new Redis();

sub.subscribe('orders', 'notifications', (err, count) => {
  console.log(`Subscribed to ${count} channels`);
});

sub.on('message', (channel, message) => {
  console.log(`[${channel}] ${message}`);
  const data = JSON.parse(message);
  // Handle message based on channel
  switch (channel) {
    case 'orders':
      handleOrderEvent(data);
      break;
    case 'notifications':
      handleNotification(data);
      break;
  }
});

// Publisher (can use the regular connection)
const pub = new Redis();

await pub.publish('orders', JSON.stringify({
  type: 'order.created',
  orderId: '123',
  userId: '456',
  total: 99.99,
}));

await pub.publish('notifications', JSON.stringify({
  type: 'welcome',
  userId: '789',
  message: 'Welcome to our platform!',
}));

Important: The subscriber connection cannot be used for other commands while subscribed. Always use a separate Redis connection for subscriptions.

Pattern-Based Subscription

// Subscribe to all channels matching a pattern
sub.psubscribe('order.*', 'user.*');

sub.on('pmessage', (pattern, channel, message) => {
  console.log(`Pattern: ${pattern}, Channel: ${channel}`);
  // pattern: 'order.*', channel: 'order.created'
});

await pub.publish('order.created', JSON.stringify({ orderId: '123' }));
await pub.publish('order.shipped', JSON.stringify({ orderId: '123' }));
await pub.publish('user.signup', JSON.stringify({ userId: '456' }));

Real-World: Event Bus

class RedisEventBus {
  private subscriber: Redis;
  private publisher: Redis;
  private handlers = new Map<string, Function[]>();

  constructor() {
    this.subscriber = new Redis();
    this.publisher = new Redis();

    this.subscriber.on('message', (channel, message) => {
      const handlers = this.handlers.get(channel) || [];
      const data = JSON.parse(message);
      handlers.forEach(handler => handler(data));
    });
  }

  async on(event: string, handler: Function) {
    if (!this.handlers.has(event)) {
      this.handlers.set(event, []);
      await this.subscriber.subscribe(event);
    }
    this.handlers.get(event)!.push(handler);
  }

  async emit(event: string, data: any) {
    await this.publisher.publish(event, JSON.stringify(data));
  }

  async close() {
    await this.subscriber.quit();
    await this.publisher.quit();
  }
}

const bus = new RedisEventBus();
bus.on('order.created', async (data) => {
  await sendConfirmationEmail(data.userId, data.orderId);
});
bus.on('order.created', async (data) => {
  await updateInventory(data.items);
});

await bus.emit('order.created', { orderId: '123', userId: '456', items: [...] });

Lua Scripting

Lua scripts execute atomically on the Redis server. While the script runs, no other Redis command can execute. This is crucial for operations that need to read and write in a single atomic step.

Why Lua?

Without Lua (race condition):
  Client A: GET count -> 10
  Client B: GET count -> 10
  Client A: SET count 11
  Client B: SET count 11  (should be 12!)

With Lua (atomic):
  Script: local count = redis.call('GET', KEYS[1])
          count = count + 1
          redis.call('SET', KEYS[1], count)
          return count
  -- Runs atomically, no interleaving

Basic Lua Scripts

// Simple script
const result = await redis.eval(
  `return redis.call('GET', KEYS[1])`,
  1,            // number of KEYS
  'mykey'       // KEYS[1]
);

// Rate limiter with Lua
const rateLimitScript = `
  local key = KEYS[1]
  local limit = tonumber(ARGV[1])
  local window = tonumber(ARGV[2])

  local current = redis.call('INCR', key)
  if current == 1 then
    redis.call('EXPIRE', key, window)
  end

  if current > limit then
    return 0  -- rate limited
  end
  return 1  -- allowed
`;

async function isRateLimited(userId: string, limit = 100, windowSeconds = 60) {
  const key = `ratelimit:${userId}:${Math.floor(Date.now() / 1000 / windowSeconds)}`;
  const allowed = await redis.eval(rateLimitScript, 1, key, limit, windowSeconds);
  return allowed === 0;
}

Defining Reusable Scripts

// Define script once, Redis caches the compiled version
const redis = new Redis();

redis.defineCommand('acquireLock', {
  numberOfKeys: 1,
  lua: `
    local key = KEYS[1]
    local value = ARGV[1]
    local ttl = tonumber(ARGV[2])

    if redis.call('SET', key, value, 'NX', 'EX', ttl) then
      return 1
    end
    return 0
  `,
});

redis.defineCommand('releaseLock', {
  numberOfKeys: 1,
  lua: `
    local key = KEYS[1]
    local expected = ARGV[1]

    if redis.call('GET', key) == expected then
      redis.call('DEL', key)
      return 1
    end
    return 0
  `,
});

// Usage
const lockId = crypto.randomUUID();
const acquired = await (redis as any).acquireLock('lock:resource', lockId, 30);

if (acquired) {
  try {
    await doExclusiveWork();
  } finally {
    await (redis as any).releaseLock('lock:resource', lockId);
  }
}

Sliding Window Rate Limiter

const slidingWindowScript = `
  local key = KEYS[1]
  local now = tonumber(ARGV[1])
  local window = tonumber(ARGV[2])
  local limit = tonumber(ARGV[3])

  -- Remove old entries
  redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

  -- Count remaining
  local count = redis.call('ZCARD', key)

  if count >= limit then
    return 0  -- blocked
  end

  -- Add current request
  redis.call('ZADD', key, now, now .. '-' .. math.random(1000000))
  redis.call('EXPIRE', key, window)

  return limit - count  -- remaining requests
`;

async function checkRateLimit(identifier: string, limit: number, windowMs: number) {
  const remaining = await redis.eval(
    slidingWindowScript, 1,
    `ratelimit:${identifier}`,
    Date.now(), windowMs, limit
  );
  return { allowed: remaining > 0, remaining: Math.max(0, Number(remaining)) };
}

Exercises

  1. Real-Time Notification System

    Build a notification system using Redis Pub/Sub: publish events from your API (order.created, user.followed, comment.added), subscribe in a WebSocket server, and forward to connected users. Handle reconnection.

    Hint

    Use separate Redis connections for pub and sub. In the subscriber, maintain a map of userId -> WebSocket connections. Re-subscribe on disconnect.

  2. Distributed Rate Limiter

    Implement a sliding window rate limiter using Lua scripting that works across multiple Node.js instances. Support per-user, per-IP, and per-endpoint limits. Return remaining quota in response headers.

    Hint

    Use sorted sets with timestamps as scores. The Lua script atomically cleans old entries, counts current, and adds new. Use X-RateLimit-Remaining and X-RateLimit-Reset headers.

  3. Distributed Lock with Fencing

    Build a distributed lock using Redis that handles: acquisition with timeout, automatic expiration, safe release (only the owner can release), and lock extension for long operations.

    Hint

    Use SET NX EX with a unique lock value. Release with Lua that checks the value before deleting. Extend with Lua that checks ownership before updating TTL.

Common Problems & Solutions

1 Pub/Sub messages are lost when subscriber disconnects
Cause: Redis Pub/Sub is fire-and-forget. Messages are not persisted. If no subscriber is listening, the message is discarded.
Solution: For reliable messaging, use Redis Streams instead of Pub/Sub. Streams persist messages and support consumer groups with acknowledgment. Or use a dedicated message queue (BullMQ, RabbitMQ).
2 Lua script blocks Redis for too long
Cause: Lua scripts run atomically, blocking all other Redis operations. A slow script (complex computation, large dataset iteration) freezes Redis.
Solution: Keep Lua scripts fast (< 5ms). Avoid iterating over large datasets. Use SCAN in application code instead of Lua for large key spaces. Set lua-time-limit in redis.conf.
3 Subscriber connection used for regular commands fails
Cause: Once a connection enters subscribe mode, it can only use SUBSCRIBE, UNSUBSCRIBE, PSUBSCRIBE, PUNSUBSCRIBE, and PING.
Solution: Always use a dedicated Redis connection for subscriptions. Create two connections: one for pub/commands, one for subscriptions.
