Aggregations: GROUP BY, HAVING, COUNT, SUM, AVG

Intuition

Aggregation is like summarizing survey results. GROUP BY sorts the surveys into piles (by city, by age group, etc.), then aggregate functions (COUNT, SUM, AVG) produce one summary number per pile. HAVING removes piles that don't meet your criteria, like discarding groups with too few responses to be statistically significant.

Aggregations: GROUP BY, HAVING, COUNT, SUM, AVG

Introduction

Aggregation is how you go from individual rows to summary data. "How many orders were placed last month?" "What's the average order total per customer?" "Which products generate the most revenue?" These all require aggregate functions combined with GROUP BY.

Aggregate Functions

The five core aggregation functions:

-- COUNT: number of rows
SELECT COUNT(*) FROM orders;                    -- all rows
SELECT COUNT(email) FROM users;                 -- non-NULL emails only
SELECT COUNT(DISTINCT city) FROM users;         -- unique cities

-- SUM: total of a numeric column
SELECT SUM(total) FROM orders;

-- AVG: average value
SELECT AVG(total) FROM orders;

-- MIN / MAX
SELECT MIN(price), MAX(price) FROM products;

In Node.js:

async function getOrderStats() {
  const result = await pool.query(`
    SELECT
      COUNT(*)::int AS total_orders,
      SUM(total)::numeric(10,2) AS revenue,
      AVG(total)::numeric(10,2) AS avg_order,
      MIN(total)::numeric(10,2) AS smallest_order,
      MAX(total)::numeric(10,2) AS largest_order
    FROM orders
    WHERE created_at >= NOW() - INTERVAL '30 days'
  `);
  return result.rows[0];
}
// { total_orders: 142, revenue: '8523.47', avg_order: '60.02', ... }

Note: PostgreSQL returns COUNT as a string (bigint). Cast with ::int for JavaScript numbers.

GROUP BY

GROUP BY splits rows into groups and applies aggregate functions to each group:

SELECT city, COUNT(*) AS user_count
FROM users
GROUP BY city
ORDER BY user_count DESC;

How GROUP BY works:

Raw Data:                   Grouped:                    Result:
| name  | city     |        city     | rows          | city     | count |
|-------|----------|   =>   New York | [Alice, Dan]  | New York | 2     |
| Alice | New York |        Boston   | [Bob]         | Boston   | 1     |
| Bob   | Boston   |        Chicago  | [Carol, Eve]  | Chicago  | 2     |
| Carol | Chicago  |
| Dan   | New York |
| Eve   | Chicago  |

The Golden Rule

Every column in SELECT must either: 1. Appear in the GROUP BY clause, OR 2. Be inside an aggregate function

-- WRONG: 'name' is not grouped or aggregated
SELECT name, city, COUNT(*) FROM users GROUP BY city;

-- CORRECT options:
SELECT city, COUNT(*) FROM users GROUP BY city;
SELECT city, ARRAY_AGG(name) AS names, COUNT(*) FROM users GROUP BY city;

GROUP BY Multiple Columns

SELECT
  EXTRACT(YEAR FROM created_at) AS year,
  EXTRACT(MONTH FROM created_at) AS month,
  COUNT(*) AS order_count,
  SUM(total) AS revenue
FROM orders
GROUP BY
  EXTRACT(YEAR FROM created_at),
  EXTRACT(MONTH FROM created_at)
ORDER BY year DESC, month DESC;

async function getMonthlyRevenue() {
  const result = await pool.query(`
    SELECT
      TO_CHAR(created_at, 'YYYY-MM') AS month,
      COUNT(*)::int AS orders,
      SUM(total)::numeric(10,2) AS revenue,
      AVG(total)::numeric(10,2) AS avg_order
    FROM orders
    GROUP BY TO_CHAR(created_at, 'YYYY-MM')
    ORDER BY month DESC
    LIMIT 12
  `);
  return result.rows;
}

HAVING: Filtering Groups

WHERE filters individual rows BEFORE grouping. HAVING filters groups AFTER aggregation:

-- Find cities with more than 100 users
SELECT city, COUNT(*) AS user_count
FROM users
GROUP BY city
HAVING COUNT(*) > 100
ORDER BY user_count DESC;

Execution Order:

  FROM users
    |
  WHERE (filter rows)          -- happens first
    |
  GROUP BY city                -- group surviving rows
    |
  HAVING COUNT(*) > 100       -- filter groups
    |
  SELECT city, COUNT(*)       -- compute output
    |
  ORDER BY                    -- sort
    |
  LIMIT                       -- truncate

WHERE vs HAVING

-- WHERE: filter rows before grouping
SELECT city, COUNT(*) AS active_users
FROM users
WHERE status = 'active'        -- filters individual rows
GROUP BY city
HAVING COUNT(*) > 50;          -- filters groups

-- You CANNOT use aliases in HAVING:
-- WRONG: HAVING active_users > 50
-- RIGHT: HAVING COUNT(*) > 50

Advanced Aggregation Patterns

COALESCE with Aggregates

Handle NULL results:

const result = await pool.query(`
  SELECT
    COALESCE(SUM(total), 0) AS revenue
  FROM orders
  WHERE user_id = $1
`, [userId]);
// Returns 0 instead of NULL when user has no orders

FILTER Clause (PostgreSQL)

Conditional aggregation without CASE:

SELECT
  COUNT(*) AS total_orders,
  COUNT(*) FILTER (WHERE status = 'completed') AS completed,
  COUNT(*) FILTER (WHERE status = 'pending') AS pending,
  COUNT(*) FILTER (WHERE status = 'cancelled') AS cancelled,
  SUM(total) FILTER (WHERE status = 'completed') AS completed_revenue
FROM orders;

ARRAY_AGG and STRING_AGG

-- Collect values into an array
SELECT
  o.id AS order_id,
  ARRAY_AGG(p.name) AS product_names
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
GROUP BY o.id;

-- Concatenate into a string
SELECT
  category,
  STRING_AGG(name, ', ' ORDER BY name) AS product_list
FROM products
GROUP BY category;

Real-World Example: E-commerce Analytics Dashboard

async function getAnalyticsDashboard(startDate, endDate) {
  const result = await pool.query(`
    SELECT
      COUNT(DISTINCT o.user_id)::int AS unique_customers,
      COUNT(o.id)::int AS total_orders,
      SUM(o.total)::numeric(12,2) AS total_revenue,
      AVG(o.total)::numeric(10,2) AS avg_order_value,
      SUM(o.total)::numeric / COUNT(DISTINCT o.user_id) AS revenue_per_customer,
      COUNT(*) FILTER (WHERE o.status = 'refunded')::int AS refunds,
      ROUND(
        COUNT(*) FILTER (WHERE o.status = 'refunded')::numeric
        / NULLIF(COUNT(*), 0) * 100, 2
      ) AS refund_rate_pct
    FROM orders o
    WHERE o.created_at BETWEEN $1 AND $2
  `, [startDate, endDate]);
  return result.rows[0];
}

Common Pitfalls

  1. COUNT(*) vs COUNT(column): COUNT(*) counts all rows; COUNT(column) skips NULLs.
  2. BigInt from PostgreSQL: COUNT returns bigint. Cast to ::int or handle strings in JS.
  3. Aggregating across JOINs: If you join a one-to-many relationship, counts and sums get inflated. Use COUNT(DISTINCT ...) or aggregate in a subquery first.

Exercises

  1. Sales Report by Category

    Write a query that shows each product category's total revenue, average order value, number of unique customers, and best-selling product. Only include categories with revenue above $1000. Execute from Node.js.

    Hint

    JOIN products with order_items, GROUP BY category, use HAVING for the $1000 filter. For best-selling product, you may need a subquery or window function.

  2. Time-Series Aggregation

    Create a function that returns daily order counts and revenue for the past 30 days, including days with zero orders (fill gaps with 0).

    Hint

    Use generate_series() to create a date range, then LEFT JOIN with orders grouped by date. COALESCE null counts to 0.

  3. Conditional Aggregation Dashboard

    Using the PostgreSQL FILTER clause, build a query that returns, in a single row: total users, active users, inactive users, users created this month, users created last month.

    Hint

    Use COUNT(*) FILTER (WHERE condition) for each metric. For date comparisons, use NOW() - INTERVAL '1 month'.

Common Problems & Solutions

1 Column must appear in GROUP BY clause or be used in an aggregate function
Cause: Selecting a non-aggregated column that isn't in the GROUP BY clause. SQL doesn't know which value to pick from the group.
Solution: Either add the column to GROUP BY, wrap it in an aggregate function (MAX, MIN, ARRAY_AGG), or remove it from SELECT.
2 SUM or COUNT returns inflated numbers after JOIN
Cause: Joining with a one-to-many table multiplies rows before aggregation. If user has 3 orders and 5 reviews, joining both gives 15 rows per user.
Solution: Aggregate in subqueries first, then join the results. Or use COUNT(DISTINCT column) to avoid double-counting.

-- Inflated:\nSELECT u.id, COUNT(o.id), COUNT(r.id) FROM users u\nJOIN orders o ON ... JOIN reviews r ON ...\n-- Fixed:\nSELECT u.id, COUNT(DISTINCT o.id), COUNT(DISTINCT r.id) FROM users u\nJOIN orders o ON ... JOIN reviews r ON ...

3 PostgreSQL returns COUNT as a string, not a number
Symptoms: typeof result.rows[0].count === 'string' // true
Cause: PostgreSQL's COUNT returns bigint, which the pg library serializes as a string to avoid JavaScript's integer precision issues.
Solution: Cast in SQL: `COUNT(*)::int` for values under 2 billion. Or parse in JS: `parseInt(row.count, 10)`. For large counts, use BigInt.
