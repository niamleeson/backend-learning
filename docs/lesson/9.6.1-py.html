Event Sourcing and CQRS in Practice

Intuition

Event Sourcing is like a bank ledger that records every transaction rather than just the current balance. You can always compute the balance by summing all transactions, and you have a complete history of how it got there.

Event Sourcing & CQRS in Practice

Event Sourcing and CQRS (Command Query Responsibility Segregation) are architectural patterns that separate how data is written from how it is read, enabling powerful audit trails, temporal queries, and optimized read models.

Core Concepts

  • Event Sourcing: Instead of storing current state, store a sequence of immutable events that describe every state change. The current state is derived by replaying events from the beginning (or from a snapshot).
  • Event Store: An append-only log of events, typically ordered by aggregate ID and sequence number. Examples include EventStoreDB, PostgreSQL with an events table, or Kafka.
  • CQRS: Separate the write model (commands that produce events) from the read model (projections optimized for queries). Writes go to the event store; reads come from denormalized views.
  • Projections: Materialized views built by processing the event stream. Different projections serve different query needs from the same event data.
  • Snapshots: Periodic captures of aggregate state to avoid replaying the entire event history on every load.

Key Takeaways

  1. Event Sourcing provides a complete audit trail and enables temporal queries ("what did this look like last Tuesday?").
  2. CQRS allows independent optimization and scaling of read and write paths.
  3. These patterns add significant complexity—use them for domains where audit trails, undo, or complex temporal queries are genuine requirements.

Exercises

  1. Build an Event-Sourced Aggregate

    Implement a BankAccount aggregate that processes commands (OpenAccount, Deposit, Withdraw) by emitting events (AccountOpened, MoneyDeposited, MoneyWithdrawn). Store events in PostgreSQL. Reconstruct the account state by replaying events.

    Hint

    Create an events table with columns: aggregate_id, sequence, event_type, data, timestamp. The aggregate's apply method processes each event to update in-memory state. Load by querying events ordered by sequence.

  2. Build a CQRS Read Projection

    Using the event store from the previous exercise, build a projection that maintains a account_balances read table. An event processor listens for new events and updates the denormalized table. Query the projection for account balances instead of replaying events.

    Hint

    Create a background worker that polls for new events using a last_processed_sequence cursor. For each MoneyDeposited event, UPDATE the balance. Use a separate projections_checkpoint table to track progress.

Common Problems & Solutions

1 Event replay takes too long as the number of events grows to millions per aggregate.
Cause: Reconstructing state by replaying all events from the beginning is O(n) in the number of events, which degrades over time.
Solution: Implement periodic snapshots. Store a snapshot of the aggregate state every N events. On load, read the latest snapshot and replay only events since that snapshot.
2 Changing the schema of an event type breaks downstream projections.
Cause: Events are immutable and stored forever. Adding, removing, or renaming fields in event schemas causes deserialization failures in projections.
Solution: Use event versioning and upcasters. Store a version number with each event. When reading an old version, an upcaster transforms it to the current schema before processing. Never delete or modify stored events.
