Input Validation with Zod and End-to-End Type Safety

Intuition

Think of Zod + tRPC like a typed pipe connecting two rooms. Whatever shape you pour into one end is guaranteed to come out the other end in the exact same shape. If someone tries to pour in something that does not fit, it gets rejected immediately at the entrance rather than leaking through and causing a mess in the other room.

Input Validation with Zod and End-to-End Type Safety

tRPC's killer feature is end-to-end type safety. When you define a Zod schema on the server, the client automatically knows the exact shape of inputs and outputs -- with full IDE auto-complete, no code generation needed.

Zod as the Validation Layer

Zod schemas serve double duty in tRPC: runtime validation AND TypeScript type inference.

import { z } from 'zod';

// Define the schema once
const createUserSchema = z.object({
  email: z.string().email('Invalid email format'),
  name: z.string().min(2, 'Name must be at least 2 characters').max(100),
  password: z.string()
    .min(8, 'Password must be at least 8 characters')
    .regex(/[A-Z]/, 'Must contain at least one uppercase letter')
    .regex(/[0-9]/, 'Must contain at least one number'),
  role: z.enum(['USER', 'ADMIN']).default('USER'),
  bio: z.string().max(500).optional(),
  tags: z.array(z.string()).max(10).default([]),
});

// Infer the TypeScript type from the schema
type CreateUserInput = z.infer<typeof createUserSchema>;
// Result: {
//   email: string;
//   name: string;
//   password: string;
//   role: 'USER' | 'ADMIN';
//   bio?: string;
//   tags: string[];
// }

Complex Zod Schemas for tRPC

// Pagination input (reusable)
const paginationSchema = z.object({
  limit: z.number().int().min(1).max(100).default(20),
  cursor: z.string().uuid().optional(),
});

// Filter schema with operators
const postFilterSchema = z.object({
  status: z.enum(['DRAFT', 'PUBLISHED', 'ARCHIVED']).optional(),
  authorId: z.string().uuid().optional(),
  tags: z.array(z.string()).optional(),
  createdAfter: z.coerce.date().optional(),
  createdBefore: z.coerce.date().optional(),
  search: z.string().min(1).max(200).optional(),
});

// Combine schemas
const listPostsInput = paginationSchema.merge(postFilterSchema);

// Use in router
export const postRouter = router({
  list: publicProcedure
    .input(listPostsInput)
    .query(async ({ input, ctx }) => {
      const where: any = {};
      if (input.status) where.status = input.status;
      if (input.authorId) where.authorId = input.authorId;
      if (input.tags) where.tags = { hasSome: input.tags };
      if (input.search) {
        where.OR = [
          { title: { contains: input.search, mode: 'insensitive' } },
          { body: { contains: input.search, mode: 'insensitive' } },
        ];
      }
      if (input.createdAfter || input.createdBefore) {
        where.createdAt = {};
        if (input.createdAfter) where.createdAt.gte = input.createdAfter;
        if (input.createdBefore) where.createdAt.lte = input.createdBefore;
      }

      const posts = await ctx.prisma.post.findMany({
        where,
        take: input.limit + 1,
        ...(input.cursor && { cursor: { id: input.cursor }, skip: 1 }),
        orderBy: { createdAt: 'desc' },
        include: { author: { select: { id: true, name: true } } },
      });

      let nextCursor: string | undefined;
      if (posts.length > input.limit) {
        nextCursor = posts.pop()!.id;
      }

      return { posts, nextCursor };
    }),
});

Output Validation

tRPC also supports output schemas to guarantee the response shape:

const userOutputSchema = z.object({
  id: z.string(),
  email: z.string(),
  name: z.string(),
  // password is NOT included -- filtered from output
  createdAt: z.date(),
});

export const userRouter = router({
  getById: publicProcedure
    .input(z.object({ id: z.string().uuid() }))
    .output(userOutputSchema) // <-- validates and strips output
    .query(async ({ input, ctx }) => {
      const user = await ctx.prisma.user.findUnique({
        where: { id: input.id },
      });
      if (!user) throw new TRPCError({ code: 'NOT_FOUND' });
      return user; // password will be stripped by output validation
    }),
});

The Client Side: Full Type Safety

// client/trpc.ts
import { createTRPCClient, httpBatchLink } from '@trpc/client';
import type { AppRouter } from '../server/routers'; // TYPE-ONLY import

const trpc = createTRPCClient<AppRouter>({
  links: [
    httpBatchLink({
      url: 'http://localhost:4000/trpc',
      headers: () => ({
        authorization: `Bearer ${getToken()}`,
      }),
    }),
  ],
});

// Full auto-complete and type checking!
async function example() {
  // Type-safe query
  const user = await trpc.user.getById.query({ id: '123' });
  console.log(user.name);     // TypeScript knows this is string
  console.log(user.password);  // ERROR: property 'password' does not exist

  // Type-safe mutation
  const newPost = await trpc.post.create.mutate({
    title: 'Hello',       // Required (min 5 chars)
    body: 'World...',     // Required (min 50 chars)
    // tags: 123,          // ERROR: expected string[]
  });

  // Type-safe pagination
  const posts = await trpc.post.list.query({
    limit: 10,
    status: 'PUBLISHED',
    // status: 'INVALID',  // ERROR: not in enum
  });
}

Batch Requests

tRPC automatically batches multiple simultaneous requests:

// These three calls are batched into ONE HTTP request
const [user, posts, comments] = await Promise.all([
  trpc.user.getById.query({ id: '1' }),
  trpc.post.list.query({ limit: 5 }),
  trpc.comment.recent.query({ limit: 10 }),
]);
// HTTP: POST /trpc/user.getById,post.list,comment.recent

Error Handling on the Client

import { TRPCClientError } from '@trpc/client';

try {
  await trpc.user.create.mutate({
    email: 'invalid',
    name: 'A',
    password: 'weak',
  });
} catch (error) {
  if (error instanceof TRPCClientError) {
    console.log(error.message);      // Error message
    console.log(error.data?.code);    // 'BAD_REQUEST'

    // Access Zod validation errors
    if (error.data?.zodError) {
      const fieldErrors = error.data.zodError.fieldErrors;
      // { email: ['Invalid email format'], name: ['Name must be...'] }
      for (const [field, messages] of Object.entries(fieldErrors)) {
        console.log(`${field}: ${messages.join(', ')}`);
      }
    }
  }
}

Transformers for Complex Types

By default, tRPC serializes data as JSON, which loses Date objects and other non-JSON types. Use superjson:

import superjson from 'superjson';

// Server
const t = initTRPC.context<Context>().create({
  transformer: superjson,
});

// Client
const trpc = createTRPCClient<AppRouter>({
  links: [
    httpBatchLink({
      url: 'http://localhost:4000/trpc',
      transformer: superjson,
    }),
  ],
});

// Now Date objects, Maps, Sets, BigInt, etc. work correctly
const post = await trpc.post.getById.query({ id: '1' });
console.log(post.createdAt instanceof Date); // true!

Type Inference Utilities

tRPC provides helper types for extracting input/output types:

import type { inferRouterInputs, inferRouterOutputs } from '@trpc/server';
import type { AppRouter } from '../server/routers';

type RouterInputs = inferRouterInputs<AppRouter>;
type RouterOutputs = inferRouterOutputs<AppRouter>;

// Get specific procedure types
type CreateUserInput = RouterInputs['user']['create'];
type UserOutput = RouterOutputs['user']['getById'];

// Use in components, forms, etc.
function UserForm(props: { onSubmit: (data: CreateUserInput) => void }) {
  // ...
}

Exercises

  1. Build a Type-Safe API with Complex Validation

    Create a tRPC router for a 'products' resource with Zod validation: create (name 3-200 chars, price positive number with 2 decimal max, category from enum, optional tags array max 5), update (all fields optional), and list (with pagination and filters for price range, category, and search). Set up a client and verify full auto-complete works.

    Hint

    Use z.number().positive().multipleOf(0.01) for price and z.enum() for category.

  2. Add Output Validation

    Add .output() schemas to your procedures that strip sensitive fields (like password, internalNotes) from responses. Verify that the client type does not include the stripped fields.

    Hint

    Define a separate output schema that omits sensitive fields. The .output() schema both validates and infers the client-side return type.

  3. Implement Request Batching

    Set up a tRPC client with httpBatchLink. Write a function that fetches a user profile, their recent posts, and their notification count in a single batched request. Verify with network inspection that only one HTTP request is made.

    Hint

    Use Promise.all() with multiple trpc.*.query() calls. The httpBatchLink automatically batches them.

Common Problems & Solutions

1 Date fields are received as strings on the client
Cause: JSON serialization converts Date objects to strings. Without a transformer, tRPC cannot restore them.
Solution: Add superjson as a transformer to both server and client: initTRPC.context().create({ transformer: superjson }) and pass it to the client link.
2 Client types are stale after server changes
Cause: TypeScript's language server has not picked up the latest server types.
Solution: Restart the TypeScript language server in your IDE. In VS Code: Cmd+Shift+P -> 'TypeScript: Restart TS Server'. For monorepos, ensure the server package is correctly referenced in tsconfig.json paths.
3 Zod validation errors are not user-friendly
Cause: Default Zod error messages are technical (e.g., 'Expected string, received number').
Solution: Add custom error messages to every Zod validator: z.string().min(2, 'Name must be at least 2 characters'). Use z.string({ required_error: 'Name is required' }) for required field messages.
