SSH, Permissions, and File System

Intuition

Think of SSH like a secure, invisible tunnel between your computer and a remote server. The key pair works like a physical lock and key: the public key is the lock you install on every door (server) you want to access, and the private key is the only key that opens those locks. Permissions are like a bouncer at a club checking three lists: what the owner can do, what the group can do, and what everyone else can do.

SSH, Permissions, and File System

Introduction

SSH (Secure Shell) is the primary way backend developers connect to remote servers. Understanding SSH, file permissions, and the Linux file system is essential for managing production servers, deploying applications, and securing your infrastructure.

SSH Fundamentals

How SSH Works

┌─────────────┐                          ┌─────────────┐
│   Your Mac   │   Encrypted Tunnel       │  Server      │
│   (client)   │ ◄══════════════════════► │  (sshd)      │
│              │   Port 22 (default)       │              │
│  ~/.ssh/     │                          │  ~/.ssh/      │
│  id_ed25519  │                          │  authorized_  │
│  (private)   │                          │  keys         │
└─────────────┘                          └─────────────┘

SSH uses public-key cryptography. You generate a key pair: the private key stays on your machine (never share it), and the public key goes on every server you want to access.

Generating SSH Keys

# Generate a modern Ed25519 key (recommended)
ssh-keygen -t ed25519 -C "your.email@example.com"

# Or RSA with 4096 bits (for older systems)
ssh-keygen -t rsa -b 4096 -C "your.email@example.com"

# You'll be prompted for:
# 1. File location (default: ~/.ssh/id_ed25519)
# 2. Passphrase (strongly recommended)

This creates two files: - ~/.ssh/id_ed25519 - Your private key (NEVER share this) - ~/.ssh/id_ed25519.pub - Your public key (safe to share)

Connecting to Servers

# Basic connection
ssh username@server-ip

# Specify port
ssh -p 2222 username@server-ip

# Specify key file
ssh -i ~/.ssh/my-server-key username@server-ip

# Execute a remote command without interactive session
ssh user@server "pm2 status"

# Copy your public key to a server
ssh-copy-id username@server-ip
# This adds your public key to the server's ~/.ssh/authorized_keys

SSH Config File

Instead of remembering IPs and usernames, use ~/.ssh/config:

# ~/.ssh/config

Host production
    HostName 203.0.113.50
    User deploy
    Port 22
    IdentityFile ~/.ssh/id_ed25519
    ForwardAgent yes

Host staging
    HostName 203.0.113.51
    User deploy
    IdentityFile ~/.ssh/id_ed25519

Host db-tunnel
    HostName 203.0.113.50
    User deploy
    LocalForward 5433 localhost:5432

Host *
    ServerAliveInterval 60
    ServerAliveCountMax 3
    AddKeysToAgent yes

Now you can simply type:

ssh production          # Instead of ssh deploy@203.0.113.50
ssh staging
ssh db-tunnel           # Opens a tunnel: local port 5433 -> remote port 5432

SSH Tunneling (Port Forwarding)

SSH tunnels are extremely useful for accessing databases and services behind firewalls.

Local Port Forwarding:
┌──────────┐      SSH Tunnel       ┌──────────┐      ┌──────────┐
│  Local    │ ═══════════════════► │  Server   │ ───► │ Database │
│  :5433    │                      │  (jump)   │      │  :5432   │
└──────────┘                      └──────────┘      └──────────┘

# Local port forwarding: access remote PostgreSQL locally
ssh -L 5433:localhost:5432 deploy@production
# Now connect to localhost:5433 to reach the remote database

# Access a remote Redis through a jump server
ssh -L 6380:redis-server:6379 deploy@jump-server

# Dynamic SOCKS proxy (tunnel all traffic)
ssh -D 1080 deploy@production

SCP and File Transfers

# Copy file to server
scp ./deploy.sh production:/home/deploy/

# Copy file from server
scp production:/var/log/app.log ./

# Copy directory recursively
scp -r ./dist/ production:/var/www/app/

# Use rsync for better transfers (resume, delta sync)
rsync -avz --progress ./dist/ production:/var/www/app/
rsync -avz --delete ./dist/ production:/var/www/app/  # Mirror exactly

Linux File Permissions

Understanding Permission Notation

Every file and directory has three permission sets:

-rwxr-xr-- 1 deploy www-data 4096 Jan 15 10:30 server.js
│├─┤├─┤├─┤   │      │
│ │  │  │     │      └── Group
│ │  │  │     └── Owner
│ │  │  └── Others (everyone else)
│ │  └── Group permissions
│ └── Owner permissions
└── File type (- = file, d = directory, l = symlink)

r = read    (4)
w = write   (2)
x = execute (1)

Numeric (Octal) Permissions

Permission    Octal    Meaning
rwx           7        Read + Write + Execute
rw-           6        Read + Write
r-x           5        Read + Execute
r--           4        Read only
-wx           3        Write + Execute
-w-           2        Write only
--x           1        Execute only
---           0        No permissions

Common permission sets:

chmod 755 script.sh     # rwxr-xr-x  Owner: full, Others: read+execute
chmod 644 config.json   # rw-r--r--  Owner: read+write, Others: read
chmod 600 .env          # rw-------  Owner only (secrets!)
chmod 700 .ssh/         # rwx------  Owner only (SSH directory)
chmod 400 id_ed25519    # r--------  Read-only by owner (private key)

Changing Permissions

# Numeric mode
chmod 755 deploy.sh

# Symbolic mode
chmod u+x deploy.sh      # Add execute for user (owner)
chmod g-w config.json     # Remove write for group
chmod o-rwx .env          # Remove all for others
chmod a+r README.md       # Add read for all

# Recursive
chmod -R 755 /var/www/app/

File Ownership

# Change owner
chown deploy server.js
chown deploy:www-data server.js    # Change owner and group

# Change recursively
chown -R deploy:www-data /var/www/app/

# Change group only
chgrp www-data /var/www/app/

Special Permissions

# Sticky bit - only owner can delete files in directory
chmod +t /tmp
chmod 1755 /shared-dir

# SUID - run file as the file owner
chmod u+s /usr/bin/program
chmod 4755 /usr/bin/program

# SGID - new files inherit directory's group
chmod g+s /var/www/shared/
chmod 2755 /var/www/shared/

File System Deep Dive

Important Directories for Backend Devs

/etc/nginx/              # Nginx configuration
/etc/systemd/system/     # systemd service files
/etc/ssl/                # SSL certificates
/var/log/                # Log files
/var/log/nginx/          # Nginx logs
/var/www/                # Web application files
/opt/                    # Third-party applications
/tmp/                    # Temporary files (cleared on reboot)

Symlinks (Symbolic Links)

# Create a symbolic link
ln -s /var/www/app/current /var/www/app/releases/v2.1.0

# This is how many deployment tools work:
# /var/www/app/
# ├── current -> releases/v2.1.0/    (symlink)
# ├── releases/
# │   ├── v2.0.0/
# │   ├── v2.0.1/
# │   └── v2.1.0/                    (latest)
# └── shared/
#     ├── .env
#     └── uploads/

# Remove symlink (not the target!)
rm /var/www/app/current   # or unlink

Inodes and Hard Links

# View inode numbers
ls -i file.txt

# Check inode usage (running out of inodes is a real problem)
df -i

# Hard link (same inode, cannot cross filesystems)
ln original.txt hardlink.txt

/proc File System

The /proc virtual filesystem gives real-time process and system information:

# CPU information
cat /proc/cpuinfo

# Memory information
cat /proc/meminfo

# Process information
cat /proc/1234/status    # Status of PID 1234
cat /proc/1234/cmdline   # Command that started it
ls -la /proc/1234/fd/    # Open file descriptors

# System limits
cat /proc/sys/fs/file-max             # Max open files system-wide
cat /proc/sys/net/core/somaxconn      # Max socket connections

Practical Security Configuration

Securing SSH on a Server

Edit /etc/ssh/sshd_config:

# Disable password authentication (use keys only)
PasswordAuthentication no
ChallengeResponseAuthentication no

# Disable root login
PermitRootLogin no

# Allow only specific users
AllowUsers deploy

# Change default port (optional, security through obscurity)
Port 2222

# Limit authentication attempts
MaxAuthTries 3

# Restart SSH after changes
sudo systemctl restart sshd

Setting Up a Deploy User

# Create a deploy user
sudo adduser deploy

# Add to sudo group (use sparingly)
sudo usermod -aG sudo deploy

# Set up SSH access
sudo mkdir -p /home/deploy/.ssh
sudo chmod 700 /home/deploy/.ssh
# Add your public key
sudo tee /home/deploy/.ssh/authorized_keys << EOF
ssh-ed25519 AAAAC3NzaC... your.email@example.com
EOF
sudo chmod 600 /home/deploy/.ssh/authorized_keys
sudo chown -R deploy:deploy /home/deploy/.ssh

Common Permission Patterns for Node.js Apps

# Application directory
sudo chown -R deploy:deploy /var/www/myapp
sudo chmod -R 755 /var/www/myapp

# Environment file (contains secrets)
chmod 600 /var/www/myapp/.env

# Log directory
sudo mkdir -p /var/log/myapp
sudo chown deploy:deploy /var/log/myapp
chmod 755 /var/log/myapp

# Upload directory (writable by app)
mkdir -p /var/www/myapp/uploads
chmod 755 /var/www/myapp/uploads

# SSL certificates
sudo chmod 600 /etc/ssl/private/server.key
sudo chmod 644 /etc/ssl/certs/server.crt

Common Pitfalls

  1. Leaving private keys with wrong permissions - SSH refuses to use keys with permissions too open. Always chmod 600 private keys.
  2. Using chmod 777 to "fix" permission issues - This is a massive security risk. Diagnose the actual permission needed.
  3. Forgetting to chown after creating files as root - Files created with sudo are owned by root. Your app user cannot read them.
  4. Not configuring SSH key passphrases - If your laptop is stolen, an unprotected private key gives access to all your servers.
  5. Port forwarding without understanding - Make sure you are not exposing internal services publicly.

Exercises

  1. SSH Config Setup

    Create an SSH config file that defines three hosts: 'dev', 'staging', and 'production'. Each should use a different key file and user. Add a wildcard block that sets ServerAliveInterval to 60 seconds. Test that ssh dev would attempt the correct connection (you don't need a real server).

    Hint

    Create ~/.ssh/config with Host blocks. Use 'ssh -v dev' to verify the config is being read correctly (it will fail to connect but show the parsed config).

  2. Permission Audit Script

    Write a bash script that audits a Node.js project directory and reports any security issues: files with 777 permissions, .env files readable by others, private keys with too-open permissions, and directories writable by others.

    Hint

    Use find with -perm flags: find . -perm -o+w for world-writable, find . -name '.env' -perm -o+r for readable .env files.

  3. SSH Tunnel for Database Access

    Document and practice the SSH tunnel command to forward local port 5433 to a remote PostgreSQL database on port 5432. Then connect using psql through the tunnel. Write a script that establishes the tunnel in the background, runs a query, and tears it down.

    Hint

    Use ssh -f -N -L 5433:localhost:5432 user@host to create a background tunnel. Use $! to capture the PID for later cleanup.

Common Problems & Solutions

1 SSH connection refused or times out
Symptoms: ssh: connect to host 203.0.113.50 port 22: Connection refused
Cause: The SSH daemon is not running, a firewall is blocking port 22, or the server IP/port is wrong.
Solution: Check if sshd is running: `sudo systemctl status sshd`. Check firewall: `sudo ufw status` or `sudo iptables -L`. Verify you can reach the port: `nc -zv server-ip 22`. Check security groups in cloud providers (AWS, GCP).
2 Permission denied (publickey) when SSH connecting
Symptoms: Permission denied (publickey,gssapi-keyex,gssapi-with-mic)
Cause: Your public key is not in the server's authorized_keys, key permissions are wrong, or you are using the wrong key.
Solution: Verify key permissions: `chmod 700 ~/.ssh && chmod 600 ~/.ssh/id_ed25519`. Use verbose mode: `ssh -vvv user@host` to see which keys are tried. Ensure your public key is in `~/.ssh/authorized_keys` on the server. Check the server's `/var/log/auth.log` for details.
3 Application cannot read/write files after deployment
Cause: Files were deployed as root or a different user, and the application user does not have the necessary permissions.
Solution: After deploying, fix ownership: `sudo chown -R deploy:deploy /var/www/app`. Ensure the upload directory is writable: `chmod 755 /var/www/app/uploads`. For log files, ensure the app user owns the log directory.
