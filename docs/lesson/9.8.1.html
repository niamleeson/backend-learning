Distributed Tracing with OpenTelemetry

Intuition

Distributed tracing is like a package tracking system. Each service stamps the package with its handling time and passes it along. At the end, you can see every stop the package made, how long each took, and where delays occurred.

Distributed Tracing & OpenTelemetry

In microservices architectures, a single user request may traverse dozens of services. Distributed tracing provides end-to-end visibility by tracking requests as they flow through the system, making it possible to diagnose latency and failures.

Core Concepts

  • Traces and Spans: A trace represents the entire journey of a request. It consists of spans, each representing a unit of work (an HTTP call, a database query, a function execution). Spans have parent-child relationships forming a tree.
  • Context Propagation: Trace context (trace ID, span ID, flags) is propagated across service boundaries via HTTP headers (W3C Trace Context, B3). This links spans from different services into a single trace.
  • OpenTelemetry (OTel): The vendor-neutral standard for generating, collecting, and exporting telemetry data (traces, metrics, logs). Provides SDKs for all major languages and a Collector for data processing and routing.
  • Sampling: Collecting traces for every request is expensive. Head-based sampling decides at the start; tail-based sampling decides after the trace completes (keeping interesting traces like errors or slow requests).
  • Backends: Jaeger, Zipkin, Grafana Tempo, and Datadog receive and visualize traces. OTel Collector can export to multiple backends simultaneously.

Key Takeaways

  1. Distributed tracing is essential once you have more than a handful of services—you cannot debug distributed systems with logs alone.
  2. OpenTelemetry is the industry standard; instrument once and export to any backend.
  3. Start with auto-instrumentation (OTel hooks into HTTP clients, frameworks, databases) and add manual spans for business-critical operations.

Exercises

  1. Instrument a Python Service with OpenTelemetry

    Add OpenTelemetry auto-instrumentation to a Flask application with PostgreSQL and Redis. Export traces to Jaeger. Make a request that hits all three components and view the complete trace in the Jaeger UI.

    Hint

    Install opentelemetry-distro, opentelemetry-exporter-otlp, and run opentelemetry-bootstrap -a install. Start with opentelemetry-instrument --service_name my-app flask run. Configure OTLP exporter to point to Jaeger's collector endpoint.

  2. Add Custom Spans and Attributes

    Beyond auto-instrumentation, add manual spans for business logic: a span for 'validate_order' with attributes like order.total and order.item_count, and a span for 'check_inventory' with inventory.warehouse. View these custom spans in traces.

Common Problems & Solutions

1 Traces are incomplete—spans from some services are missing.
Cause: Context propagation is broken. A service in the chain does not propagate the trace context headers, breaking the trace into disconnected fragments.
Solution: Ensure all services propagate W3C Trace Context headers (`traceparent`, `tracestate`). Check that HTTP clients and servers are instrumented. Use the OTel Collector's debug exporter to verify context is present in all spans.
2 Tracing adds noticeable latency and increases infrastructure costs.
Cause: All requests are being traced (100% sampling), generating massive volumes of trace data that stress the backend and network.
Solution: Implement sampling: start with probabilistic head-based sampling (e.g., 1% of requests). For better results, use tail-based sampling in the OTel Collector to keep 100% of error traces and slow traces while sampling normal traces.
