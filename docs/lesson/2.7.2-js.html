Caching Patterns: Cache-Aside, Write-Through, TTL

Intuition

Caching is like keeping a notepad on your desk with answers to frequently asked questions. When someone asks, you check the notepad first (cache hit = instant answer) instead of walking to the filing cabinet every time (database query = slow). The challenge is knowing when the notepad is outdated and needs updating — that is cache invalidation, famously one of the two hard problems in computer science.

Caching Patterns: Cache-Aside, Write-Through, TTL

Introduction

Caching is the most impactful performance optimization in backend development. A cache hit can serve a response in microseconds instead of the milliseconds (or seconds) required for a database query, API call, or computation. Redis is the most popular caching layer for Node.js applications. Understanding caching patterns helps you decide what to cache, when to invalidate, and how to handle edge cases.

Cache-Aside (Lazy Loading)

The most common pattern. The application checks the cache first. On miss, it fetches from the database and stores in cache.

Cache-Aside Flow:

  Client --> App --> Redis (cache hit?) --YES--> Return cached
                       |NO
                       v
                    Database
                       |
                       v
                    Store in Redis
                       |
                       v
                    Return to client

async function getUserById(userId: string) {
  const cacheKey = `user:${userId}`;

  // 1. Check cache
  const cached = await redis.get(cacheKey);
  if (cached) {
    return JSON.parse(cached);
  }

  // 2. Cache miss - fetch from database
  const user = await prisma.user.findUnique({
    where: { id: userId },
    select: { id: true, name: true, email: true, role: true },
  });

  if (!user) return null;

  // 3. Store in cache with TTL
  await redis.set(cacheKey, JSON.stringify(user), 'EX', 3600); // 1 hour

  return user;
}

Generic Cache-Aside Wrapper

async function cacheable<T>(
  key: string,
  ttlSeconds: number,
  fetcher: () => Promise<T>
): Promise<T> {
  const cached = await redis.get(key);
  if (cached) {
    return JSON.parse(cached) as T;
  }

  const data = await fetcher();

  if (data !== null && data !== undefined) {
    await redis.set(key, JSON.stringify(data), 'EX', ttlSeconds);
  }

  return data;
}

// Usage
const user = await cacheable(
  `user:${userId}`,
  3600,
  () => prisma.user.findUnique({ where: { id: userId } })
);

const products = await cacheable(
  `products:category:${categoryId}:page:${page}`,
  300, // 5 minutes
  () => prisma.product.findMany({
    where: { categoryId },
    take: 20,
    skip: (page - 1) * 20,
  })
);

Write-Through

Every write goes through the cache. The cache is always up-to-date.

async function updateUser(userId: string, data: UpdateUserInput) {
  // 1. Update database
  const user = await prisma.user.update({
    where: { id: userId },
    data,
  });

  // 2. Update cache (write-through)
  await redis.set(`user:${userId}`, JSON.stringify(user), 'EX', 3600);

  return user;
}

Write-Behind (Write-Back)

Writes go to cache first, then asynchronously to the database. Fastest writes but risks data loss.

// Write to Redis immediately, flush to DB periodically
async function trackPageView(pageId: string) {
  // Fast: increment in Redis
  await redis.hincrby(`page_views:${pageId}`, 'count', 1);
  await redis.hset(`page_views:${pageId}`, 'last_viewed', Date.now().toString());
}

// Periodic flush to database
async function flushPageViews() {
  const cursor = '0';
  let keys: string[] = [];
  // Scan for page_view keys
  let c = '0';
  do {
    const [next, found] = await redis.scan(c, 'MATCH', 'page_views:*', 'COUNT', 100);
    c = next;
    keys = keys.concat(found);
  } while (c !== '0');

  for (const key of keys) {
    const data = await redis.hgetall(key);
    const pageId = key.replace('page_views:', '');
    if (data.count) {
      await prisma.page.update({
        where: { id: pageId },
        data: { viewCount: { increment: parseInt(data.count) } },
      });
      await redis.del(key);
    }
  }
}

// Run every minute
setInterval(flushPageViews, 60000);

Cache Invalidation

The hardest problem in caching.

// Simple invalidation: delete the cache key
async function invalidateUser(userId: string) {
  await redis.del(`user:${userId}`);
}

// Pattern-based invalidation
async function invalidateProductCaches(productId: string) {
  // Delete specific cache
  await redis.del(`product:${productId}`);

  // Invalidate all category caches that might include this product
  // (use SCAN, never KEYS)
  let cursor = '0';
  do {
    const [next, keys] = await redis.scan(cursor, 'MATCH', 'products:category:*', 'COUNT', 100);
    cursor = next;
    if (keys.length > 0) await redis.del(...keys);
  } while (cursor !== '0');
}

Preventing Cache Stampede

Cache Stampede:
  1. Popular cache key expires
  2. 1000 concurrent requests all see cache miss
  3. All 1000 hit the database simultaneously
  4. Database overwhelmed

// Solution 1: Locking (only one request rebuilds cache)
async function cacheableWithLock<T>(
  key: string,
  ttl: number,
  fetcher: () => Promise<T>
): Promise<T> {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const lockKey = `lock:${key}`;
  const acquired = await redis.set(lockKey, '1', 'EX', 10, 'NX');

  if (acquired) {
    try {
      const data = await fetcher();
      await redis.set(key, JSON.stringify(data), 'EX', ttl);
      return data;
    } finally {
      await redis.del(lockKey);
    }
  }

  // Another request is rebuilding - wait and retry
  await new Promise(r => setTimeout(r, 100));
  return cacheableWithLock(key, ttl, fetcher);
}

// Solution 2: Stale-While-Revalidate
async function cacheableStaleWhileRevalidate<T>(
  key: string,
  ttl: number,
  staleTtl: number, // extra time to serve stale data
  fetcher: () => Promise<T>
): Promise<T> {
  const raw = await redis.get(key);
  if (raw) {
    const { data, expiresAt } = JSON.parse(raw);
    if (Date.now() > expiresAt) {
      // Stale - trigger background refresh
      refreshInBackground(key, ttl, staleTtl, fetcher);
    }
    return data;
  }
  const data = await fetcher();
  const envelope = { data, expiresAt: Date.now() + ttl * 1000 };
  await redis.set(key, JSON.stringify(envelope), 'EX', ttl + staleTtl);
  return data;
}

Cache Warming

// Pre-populate cache on startup
async function warmCache() {
  const popularProducts = await prisma.product.findMany({
    where: { isActive: true },
    orderBy: { viewCount: 'desc' },
    take: 100,
  });

  const pipeline = redis.pipeline();
  for (const product of popularProducts) {
    pipeline.set(`product:${product.id}`, JSON.stringify(product), 'EX', 3600);
  }
  await pipeline.exec();
  console.log(`Warmed cache with ${popularProducts.length} products`);
}

Exercises

  1. Cache Middleware for Express

    Build an Express middleware that caches GET responses in Redis. Support configurable TTL per route, cache key based on URL + query params, and a way to bypass cache (X-Skip-Cache header). Include cache hit/miss headers in the response.

    Hint

    Intercept res.json() to capture the response body. Use the full URL as the cache key. Set X-Cache: HIT or MISS headers.

  2. Stampede-Protected Cache

    Implement a caching layer that prevents cache stampedes using the locking pattern. Test by simulating 100 concurrent requests for the same cache key with a 1-second database query. Measure how many DB queries actually execute.

    Hint

    Use Redis SET NX EX for locks. Waiting requests should poll with exponential backoff. Measure with a counter that tracks DB calls.

  3. Multi-Level Cache

    Implement a two-level cache: L1 (in-process Map with 100-item LRU) and L2 (Redis). Reads check L1 first, then L2, then database. Writes update both caches. Handle TTL at both levels.

    Hint

    Use a Map with size cap for L1. On eviction from L1, the item might still be in L2. Different TTLs: L1 = 60s, L2 = 3600s.

Common Problems & Solutions

1 Stale data served after database update
Cause: Cache was not invalidated after the database write. The old cached value is still served until TTL expires.
Solution: Invalidate (delete) the cache key after every write. Or use write-through caching that updates cache and database together.
2 Cache stampede crashes the database after a popular key expires
Cause: Many concurrent requests all see a cache miss and hit the database simultaneously.
Solution: Implement cache locking (only one request rebuilds), stale-while-revalidate, or pre-emptive refresh (refresh cache before TTL expires).
3 Redis memory usage keeps growing
Cause: Keys without TTL accumulate. Or cached values are too large. Or too many unique cache keys are created.
Solution: Always set TTL on cache keys. Monitor with `redis-cli INFO memory`. Set maxmemory and maxmemory-policy (allkeys-lru is a good default). Audit key sizes with `redis-cli --bigkeys`.
