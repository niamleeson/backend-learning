REST Principles and Resource Design

Intuition

REST is about modeling your API around resources (things) rather than actions (verbs). The HTTP methods already provide the verbs, so your URLs should describe what you are operating on, creating a natural and predictable interface.

REST Principles and Resource Design

REST (Representational State Transfer) is an architectural style for designing networked APIs. Resources are the core abstraction -- each resource has a URI, and clients interact with resources through standard HTTP methods. Good REST design leads to predictable, scalable, and maintainable APIs.

from flask import Flask, jsonify, request, url_for

app = Flask(__name__)

# Resource: Articles
# Collection: /api/articles
# Individual: /api/articles/<id>
# Sub-resource: /api/articles/<id>/comments

articles_db = {}
next_id = 1

@app.route('/api/articles', methods=['GET'])
def list_articles():
    return jsonify({
        'data': list(articles_db.values()),
        'count': len(articles_db),
        '_links': {'self': url_for('list_articles', _external=True)}
    })

@app.route('/api/articles', methods=['POST'])
def create_article():
    global next_id
    data = request.get_json()
    article = {'id': next_id, 'title': data['title'], 'body': data['body']}
    article['_links'] = {
        'self': url_for('get_article', article_id=next_id, _external=True),
        'collection': url_for('list_articles', _external=True)
    }
    articles_db[next_id] = article
    next_id += 1
    return jsonify(article), 201

@app.route('/api/articles/<int:article_id>', methods=['GET'])
def get_article(article_id):
    article = articles_db.get(article_id)
    if not article:
        return jsonify({'error': 'Not found'}), 404
    return jsonify(article)

Key Takeaways: - Use nouns for resource URIs (/articles, not /getArticles) and let HTTP methods convey actions. - Include hypermedia links (_links) in responses to enable API discoverability (HATEOAS). - Design resources around domain concepts, not database tables.

Exercises

  1. Design a Resource API

    Design and implement REST endpoints for a 'courses' resource with nested 'lessons' sub-resources. Include proper URIs, HTTP methods, status codes, and HATEOAS links.

    Hint

    Use /api/courses for the collection, /api/courses//lessons for nested lessons, and include _links with self and parent references.

  2. Resource Versioning

    Implement two versions of an API (v1 and v2) where v2 returns additional fields. Use URL-based versioning with /api/v1/ and /api/v2/ prefixes.

    Hint

    Create separate blueprints for each version with different url_prefix values, or use a single route that checks the version.

Common Problems & Solutions

1 API returns 200 OK for every response, even errors.
Cause: The endpoint always returns jsonify() without specifying an appropriate HTTP status code.
Solution: Return a tuple (response, status_code) for non-200 responses: return jsonify({'error': 'Not found'}), 404.
2 Deeply nested URLs like /api/users/1/posts/5/comments/3/likes become unwieldy.
Cause: Over-nesting resources makes URLs hard to use and couples resources too tightly.
Solution: Limit nesting to one level. Use /api/comments/3 with a query parameter ?post_id=5 instead of deep nesting.
