Design an E-Commerce Platform (Amazon)

Intuition

Think of a physical department store. The product catalog is the store layout and shelf labels. The shopping cart is literally a cart you push around. Checkout is the cash register. Inventory management is the warehouse in back -- you cannot sell what is not on the shelf or in the back room. The key insight is that browsing (reads) and buying (writes) have very different consistency needs: it is fine if the website shows a product for a few seconds after it sells out, but you must never charge someone for an item you cannot deliver.

Design an E-Commerce Platform (Amazon)

Introduction

Designing an e-commerce platform like Amazon is one of the most common system design interview questions. It tests your ability to handle product catalogs, user sessions, shopping carts, order processing, inventory management, and payments at massive scale. The system must be highly available, partition-tolerant, and eventually consistent in some areas while strongly consistent in others (like inventory and payments).

Step 1: Requirements Gathering

Functional Requirements

  • Product Catalog: Browse, search, and filter products by category, price, ratings
  • Product Detail Pages: View product information, images, reviews, seller info
  • Shopping Cart: Add/remove/update items, persist across sessions
  • Checkout & Orders: Place orders, apply coupons, select shipping, process payment
  • User Accounts: Registration, login, address book, order history
  • Seller Management: Sellers can list products, manage inventory, view orders
  • Reviews & Ratings: Users can rate and review purchased products
  • Recommendations: Show related/suggested products

Non-Functional Requirements

  • Availability: 99.99% uptime (< 53 minutes downtime/year)
  • Latency: Product pages load in < 200ms, search results < 300ms
  • Consistency: Eventual consistency for catalog/reviews, strong consistency for inventory/orders
  • Scalability: Handle 100M+ products, 500M+ users, peak traffic during sales events (10x normal)

Step 2: Capacity Estimation

Assumptions:
- 500M registered users, 50M DAU
- 100M products in catalog
- Average user views 20 pages/day
- 2% of page views result in add-to-cart
- 10% of carts convert to orders
- Average order has 3 items

Traffic:
- Page views: 50M * 20 = 1B/day = ~11,500 QPS
- Peak (10x): ~115,000 QPS
- Cart operations: 1B * 0.02 = 20M/day = ~230 QPS
- Orders: 20M * 0.1 = 2M/day = ~23 QPS
- Search queries: ~30% of page views = 300M/day = ~3,500 QPS

Storage:
- Product data: 100M products * 10KB avg = ~1TB
- Product images: 100M * 5 images * 500KB = ~250TB
- User data: 500M * 2KB = ~1TB
- Order data: 2M/day * 5KB * 365 = ~3.6TB/year
- Reviews: 500M reviews * 1KB = ~500GB

Bandwidth:
- Average page: 2MB (with images from CDN)
- 1B * 2MB = 2PB/day outbound
- CDN offloads ~90%, so origin serves ~200TB/day

Step 4: API Design

Product APIs

// Product Service - Express Router
import express from 'express';
const router = express.Router();

// GET /api/products?category=electronics&minPrice=10&maxPrice=100&page=1&limit=20
router.get('/products', async (req, res) => {
  const { category, minPrice, maxPrice, sort, page = 1, limit = 20 } = req.query;
  const filters = {};
  if (category) filters.categoryId = category;
  if (minPrice || maxPrice) {
    filters.price = {};
    if (minPrice) filters.price.gte = parseFloat(minPrice);
    if (maxPrice) filters.price.lte = parseFloat(maxPrice);
  }

  const [products, total] = await Promise.all([
    prisma.product.findMany({
      where: filters,
      skip: (page - 1) * limit,
      take: parseInt(limit),
      orderBy: sort === 'price_asc' ? { price: 'asc' } : { createdAt: 'desc' },
      include: { images: { take: 1 }, seller: { select: { name: true } } },
    }),
    prisma.product.count({ where: filters }),
  ]);

  res.json({ data: products, meta: { page, limit, total } });
});

// GET /api/products/:id
router.get('/products/:id', async (req, res) => {
  const cacheKey = `product:${req.params.id}`;
  const cached = await redis.get(cacheKey);
  if (cached) return res.json(JSON.parse(cached));

  const product = await prisma.product.findUnique({
    where: { id: req.params.id },
    include: {
      images: true,
      seller: true,
      reviews: { take: 10, orderBy: { createdAt: 'desc' } },
      category: true,
    },
  });

  if (!product) return res.status(404).json({ error: 'Product not found' });

  await redis.setex(cacheKey, 300, JSON.stringify(product)); // 5 min cache
  res.json(product);
});

Cart APIs

// Cart Service - uses Redis for fast access
// POST /api/cart/items
router.post('/cart/items', authenticate, async (req, res) => {
  const { productId, quantity } = req.body;
  const cartKey = `cart:${req.user.id}`;

  // Verify product exists and has stock
  const product = await productService.getProduct(productId);
  if (!product) return res.status(404).json({ error: 'Product not found' });

  const inventory = await inventoryService.getStock(productId);
  if (inventory.available < quantity) {
    return res.status(409).json({ error: 'Insufficient stock' });
  }

  // Store in Redis hash: cart:{userId} -> { productId: JSON(item) }
  const item = { productId, quantity, price: product.price, name: product.name };
  await redis.hset(cartKey, productId, JSON.stringify(item));
  await redis.expire(cartKey, 7 * 24 * 3600); // 7-day TTL

  const cart = await getFullCart(req.user.id);
  res.json(cart);
});

// GET /api/cart
router.get('/cart', authenticate, async (req, res) => {
  const cart = await getFullCart(req.user.id);
  res.json(cart);
});

async function getFullCart(userId) {
  const cartKey = `cart:${userId}`;
  const items = await redis.hgetall(cartKey);
  const cartItems = Object.values(items).map(i => JSON.parse(i));
  const total = cartItems.reduce((sum, i) => sum + i.price * i.quantity, 0);
  return { items: cartItems, total, itemCount: cartItems.length };
}

Order APIs

// Order Service
// POST /api/orders
router.post('/orders', authenticate, async (req, res) => {
  const { shippingAddressId, paymentMethodId, couponCode } = req.body;
  const cart = await getFullCart(req.user.id);

  if (cart.items.length === 0) {
    return res.status(400).json({ error: 'Cart is empty' });
  }

  // Use a database transaction for order creation
  const order = await prisma.$transaction(async (tx) => {
    // 1. Reserve inventory for each item
    for (const item of cart.items) {
      const reserved = await inventoryService.reserve(
        item.productId, item.quantity, tx
      );
      if (!reserved) {
        throw new Error(`Insufficient stock for ${item.name}`);
      }
    }

    // 2. Apply coupon if provided
    let discount = 0;
    if (couponCode) {
      discount = await couponService.validate(couponCode, cart.total, tx);
    }

    // 3. Create order record
    const order = await tx.order.create({
      data: {
        userId: req.user.id,
        status: 'PENDING_PAYMENT',
        subtotal: cart.total,
        discount,
        total: cart.total - discount,
        shippingAddressId,
        items: {
          create: cart.items.map(item => ({
            productId: item.productId,
            quantity: item.quantity,
            priceAtPurchase: item.price,
          })),
        },
      },
      include: { items: true },
    });

    return order;
  });

  // 4. Initiate async payment processing
  await paymentQueue.add('process-payment', {
    orderId: order.id,
    amount: order.total,
    paymentMethodId,
    userId: req.user.id,
  });

  // 5. Clear the cart
  await redis.del(`cart:${req.user.id}`);

  res.status(201).json(order);
});

Step 6: Deep Dive - Inventory Management

Inventory is the hardest part of e-commerce. You must prevent overselling while maintaining high throughput.

Problem: Two users try to buy the last item simultaneously.

User A reads stock = 1    User B reads stock = 1
User A reserves 1         User B reserves 1
Stock goes to -1!         OVERSOLD!

Solution: Pessimistic locking with SELECT FOR UPDATE

// Inventory reservation with pessimistic locking
async function reserveInventory(productId, quantity, tx) {
  // SELECT FOR UPDATE locks the row until transaction commits
  const inventory = await tx.$queryRaw`
    SELECT * FROM inventory
    WHERE product_id = ${productId}
    FOR UPDATE
  `;

  if (!inventory[0] || inventory[0].available_qty < quantity) {
    return false;
  }

  await tx.$executeRaw`
    UPDATE inventory
    SET reserved_qty = reserved_qty + ${quantity},
        updated_at = NOW()
    WHERE product_id = ${productId}
  `;

  return true;
}

// Alternative: Optimistic locking with version column
async function reserveOptimistic(productId, quantity) {
  const result = await prisma.$executeRaw`
    UPDATE inventory
    SET reserved_qty = reserved_qty + ${quantity},
        version = version + 1,
        updated_at = NOW()
    WHERE product_id = ${productId}
      AND (total_qty - reserved_qty) >= ${quantity}
      AND version = ${expectedVersion}
  `;
  return result > 0; // true if row was updated
}

// Search service using Elasticsearch
import { Client } from '@elastic/elasticsearch';
const es = new Client({ node: 'http://elasticsearch:9200' });

async function searchProducts(query, filters, page = 1, limit = 20) {
  const must = [{ multi_match: { query, fields: ['name^3', 'description', 'category'] } }];
  const filter = [];

  if (filters.category) filter.push({ term: { categorySlug: filters.category } });
  if (filters.minPrice || filters.maxPrice) {
    filter.push({ range: { price: {
      ...(filters.minPrice && { gte: filters.minPrice }),
      ...(filters.maxPrice && { lte: filters.maxPrice }),
    } } });
  }
  if (filters.minRating) {
    filter.push({ range: { avgRating: { gte: filters.minRating } } });
  }

  const result = await es.search({
    index: 'products',
    body: {
      from: (page - 1) * limit,
      size: limit,
      query: { bool: { must, filter } },
      aggs: {
        categories: { terms: { field: 'categorySlug', size: 20 } },
        price_ranges: {
          range: { field: 'price', ranges: [
            { to: 25 }, { from: 25, to: 50 },
            { from: 50, to: 100 }, { from: 100 },
          ] },
        },
        avg_rating: { avg: { field: 'avgRating' } },
      },
    },
  });

  return {
    products: result.hits.hits.map(h => ({ id: h._id, ...h._source })),
    facets: result.aggregations,
    total: result.hits.total.value,
  };
}

Caching Strategy

Layer 1: CDN (product images, static assets)  - TTL: hours/days
Layer 2: Redis (product details, categories)   - TTL: 5-15 min
Layer 3: Application cache (hot config data)   - TTL: 1 min

Cache Invalidation:
- Product update -> invalidate product:{id} + search index
- Inventory change -> do NOT cache (always check DB)
- Price change -> invalidate product:{id} + cart recalculation

Handling Flash Sales

// Flash sale with token bucket rate limiting + pre-warmed inventory in Redis
async function handleFlashSale(productId, userId) {
  // 1. Rate limit per user (1 attempt per 5 seconds)
  const rateKey = `flash:rate:${userId}`;
  const attempts = await redis.incr(rateKey);
  if (attempts === 1) await redis.expire(rateKey, 5);
  if (attempts > 1) throw new Error('Too many attempts');

  // 2. Atomic decrement in Redis (pre-loaded inventory count)
  const stockKey = `flash:stock:${productId}`;
  const remaining = await redis.decr(stockKey);

  if (remaining < 0) {
    await redis.incr(stockKey); // restore
    throw new Error('Sold out');
  }

  // 3. Queue the actual order processing
  await orderQueue.add('flash-order', { productId, userId, quantity: 1 });
  return { status: 'reserved', message: 'Processing your order' };
}

Summary

An e-commerce platform requires careful decomposition into services (product, cart, order, inventory, search, payment), each with its own data store optimized for its access pattern. The critical design challenges are inventory management (preventing overselling with locking), search performance (Elasticsearch), cart persistence (Redis), and handling traffic spikes (caching, queuing, rate limiting). Always separate the read path (cached, eventually consistent) from the write path (strongly consistent, transactional).

Common Problems & Solutions

1 Products showing as available but failing at checkout (overselling)
Symptoms: Customer complaints about orders being cancelled after payment. Order fulfillment team reports negative inventory counts in the database.
Cause: The product listing page uses cached data showing available stock, but by the time the user checks out, the actual inventory has been depleted by other concurrent purchases. No locking mechanism prevents the race condition.
Solution: Use pessimistic locking (SELECT FOR UPDATE) or optimistic locking (version column) during the checkout transaction. Keep real-time inventory counts in Redis with atomic DECR operations. Accept that catalog pages may briefly show stale stock -- validate at checkout.
2 Search results are stale or missing newly added products
Cause: Elasticsearch index is not synchronized with the product database. If using periodic batch reindexing, there can be significant lag. If using CDC (Change Data Capture), the pipeline may have failed silently.
Solution: Implement CDC with Kafka/Debezium for near-real-time sync. Add a monitoring dashboard for index lag. Provide a manual reindex endpoint for emergencies. Consider dual-write as fallback (write to both DB and ES, accept occasional inconsistency).
3 Site crashes during flash sales with 10x traffic spike
Cause: The system was designed for normal traffic patterns. Flash sales cause thundering herd on the database, cache stampede when cached items expire simultaneously, and API gateway rate limits being hit.
Solution: Pre-warm caches before the sale. Use Redis atomic operations for inventory instead of database writes. Implement a virtual queue so users wait in line rather than overwhelming the system. Auto-scale application servers based on queue depth. Use circuit breakers on downstream services.
