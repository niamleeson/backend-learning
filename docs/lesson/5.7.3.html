Testing, Documentation, and Production Configuration

Preparing a FastAPI application for production involves comprehensive testing, complete API documentation, environment-based configuration, structured logging, and deployment settings.

from pydantic_settings import BaseSettings
from functools import lru_cache

class Settings(BaseSettings):
    app_name: str = "My API"
    database_url: str
    secret_key: str
    debug: bool = False
    cors_origins: list[str] = ["https://myapp.com"]
    log_level: str = "INFO"

    model_config = {"env_file": ".env", "env_file_encoding": "utf-8"}

@lru_cache
def get_settings() -> Settings:
    return Settings()

# Testing with settings override
import pytest
from httpx import AsyncClient, ASGITransport

@pytest.fixture
def test_settings():
    return Settings(
        database_url="sqlite+aiosqlite:///./test.db",
        secret_key="test-secret",
        debug=True,
    )

@pytest.fixture
async def client(test_settings):
    app.dependency_overrides[get_settings] = lambda: test_settings
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as c:
        yield c
    app.dependency_overrides.clear()

Key Takeaways: - Use pydantic-settings for type-safe configuration from environment variables - @lru_cache ensures settings are loaded once and reused across the application - Override settings in tests using dependency_overrides for isolated testing - Configure uvicorn with --workers, --host, and --port for production deployment

Exercises

  1. Structured Logging Setup

    Configure structured JSON logging for the FastAPI application using structlog or Python's built-in logging with a JSON formatter. Include request ID, user ID, and endpoint path in every log entry via middleware.

    Hint

    Use structlog.configure() with a JSON renderer. In middleware, bind context variables with structlog.contextvars.bind_contextvars(request_id=..., path=...) that will appear in all log entries for that request.

  2. Health Check and Readiness Endpoints

    Create /health (liveness) and /ready (readiness) endpoints. The health endpoint returns 200 if the process is running. The readiness endpoint checks database connectivity and returns 503 if the database is unreachable.

    Hint

    For readiness, attempt a simple query like SELECT 1 in a try/except. Return {"status": "healthy", "database": "connected"} on success or raise HTTPException(503) on failure.

Common Problems & Solutions

1 Environment variables not loading from .env file
Cause: The `python-dotenv` package is not installed, or the `.env` file path is incorrect relative to the working directory.
Solution: Install `python-dotenv` with `pip install python-dotenv`. Use an absolute path for `env_file` or ensure the working directory is correct. Verify the `.env` file format uses `KEY=value` without quotes around values.
2 Application crashes on startup in production but works locally
Cause: Required environment variables are not set in the production environment, and Pydantic Settings raises a validation error for missing required fields.
Solution: Add all required environment variables to your production deployment configuration. Use `Field(default=...)` for optional settings with sensible defaults. Add startup logging that reports which configuration source was loaded.
