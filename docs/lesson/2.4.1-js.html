Intuition

Knex is like having a bilingual assistant who speaks both JavaScript and SQL fluently. You describe what you want in JavaScript, and the assistant writes correct, safe SQL for you — handling dialect differences, parameter escaping, and connection management behind the scenes.

Knex Setup, Configuration, and Connection Pooling

Introduction

Knex.js is a SQL query builder for Node.js that supports PostgreSQL, MySQL, SQLite, and more. It sits between raw SQL (using pg) and a full ORM (like Prisma). Knex gives you programmatic query construction with JavaScript while keeping you close to SQL — you control the queries, but Knex handles parameterization, connection pooling, and dialect differences.

Installing Knex

npm install knex pg

Configuration

// knexfile.js
export default {
  development: {
    client: 'pg',
    connection: {
      host: '127.0.0.1',
      port: 5432,
      database: 'myapp_dev',
      user: 'postgres',
      password: 'secret',
    },
    pool: {
      min: 2,
      max: 10,
    },
    migrations: {
      directory: './db/migrations',
      tableName: 'knex_migrations',
    },
    seeds: {
      directory: './db/seeds',
    },
  },
  production: {
    client: 'pg',
    connection: process.env.DATABASE_URL,
    pool: {
      min: 2,
      max: 20,
    },
    migrations: {
      directory: './db/migrations',
    },
  },
};

Initializing the Knex Instance

// db.js
import Knex from 'knex';
import config from './knexfile.js';

const environment = process.env.NODE_ENV || 'development';
const knex = Knex(config[environment]);

export default knex;

Connection Pooling

Knex manages a pool of database connections automatically:

Connection Pool:

  Node.js App
      |
  +---+---+
  | Knex  |  Pool Manager
  | Pool  |  (min: 2, max: 10)
  +---+---+
   |  |  |
   v  v  v
  [conn1] [conn2] ... [conn10]
      \      |       /
       PostgreSQL Server

- Connections are reused, not created per query
- Pool automatically grows/shrinks between min and max
- Idle connections are kept alive for fast reuse

// Pool configuration options
const knex = Knex({
  client: 'pg',
  connection: { /* ... */ },
  pool: {
    min: 2,                    // minimum connections kept alive
    max: 10,                   // maximum simultaneous connections
    acquireTimeoutMillis: 30000, // timeout waiting for a connection
    createTimeoutMillis: 30000,  // timeout creating a new connection
    idleTimeoutMillis: 30000,    // close idle connections after this
    reapIntervalMillis: 1000,    // check for idle connections interval
    propagateCreateError: false, // don't fail pool on connection error
  },
  // Log queries in development
  debug: process.env.NODE_ENV === 'development',
});

Testing the Connection

async function testConnection() {
  try {
    await knex.raw('SELECT 1+1 AS result');
    console.log('Database connected successfully');
  } catch (err) {
    console.error('Database connection failed:', err.message);
    process.exit(1);
  }
}

testConnection();

Pool Monitoring

// Monitor pool health
function getPoolStatus() {
  const pool = knex.client.pool;
  return {
    used: pool.numUsed(),
    free: pool.numFree(),
    pendingAcquires: pool.numPendingAcquires(),
    pendingCreates: pool.numPendingCreates(),
  };
}

// Log pool status periodically
setInterval(() => {
  console.log('Pool status:', getPoolStatus());
}, 60000);

Pool Sizing Guidelines

Optimal pool size ≈ (number of CPU cores * 2) + number of disks

For a typical server with 4 cores and 1 SSD:
  Pool max = (4 * 2) + 1 = 9 ≈ 10

Common mistakes:
  - Setting max too high (100+) — causes connection overhead
  - Setting min too high — wastes database connections
  - Not setting acquireTimeoutMillis — queries hang forever

Graceful Shutdown

async function shutdown() {
  console.log('Shutting down...');
  await knex.destroy(); // closes all pool connections
  process.exit(0);
}

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);

Using Knex with TypeScript

import Knex, { Knex as KnexType } from 'knex';

interface User {
  id: number;
  name: string;
  email: string;
  created_at: Date;
}

const knex = Knex({ /* config */ });

// Type-safe queries
const users = await knex<User>('users')
  .where('email', 'alice@example.com')
  .first();
// users is User | undefined

Environment-Based Configuration

// Using environment variables
const config = {
  client: 'pg',
  connection: {
    host: process.env.DB_HOST || 'localhost',
    port: parseInt(process.env.DB_PORT || '5432'),
    database: process.env.DB_NAME || 'myapp',
    user: process.env.DB_USER || 'postgres',
    password: process.env.DB_PASSWORD || '',
    ssl: process.env.DB_SSL === 'true'
      ? { rejectUnauthorized: false }
      : false,
  },
};

Exercises

    Access knex.client.pool.numUsed() etc. For the health check, run a simple SELECT 1 query with a timeout.

  3. Connection Wrapper with Logging

    Hint

Common Problems & Solutions

1 KnexTimeoutError: Knex: Timeout acquiring a connection
Cause: All pool connections are in use and no connection became available within the timeout. Usually caused by connection leaks (not releasing connections) or pool size too small for the load.
Solution: Check for unreleased connections (missing `await` on knex transactions, or errors in transaction blocks that skip cleanup). Increase pool max or acquireTimeoutMillis. Monitor with pool.numUsed().
2 Connection refused or ECONNREFUSED
Cause: PostgreSQL is not running, wrong host/port, or firewall blocking the connection.
Solution: Verify PostgreSQL is running: `pg_isready`. Check host, port, and credentials. For Docker, use the container name or host.docker.internal instead of localhost.
3 SSL connection required but not configured
Cause: Production databases often require SSL connections, but Knex defaults to no SSL.
Solution: Add ssl: { rejectUnauthorized: false } to the connection config. For strict SSL, provide the CA certificate.

connection: { ...config, ssl: { rejectUnauthorized: false } }
