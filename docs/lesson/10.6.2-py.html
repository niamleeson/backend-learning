Design an Email Service (Gmail)

Intuition

An email service is a per-user document store with a complex ingestion pipeline, where the fundamental design insight is that emails are written once and read many times, so optimizing the read path (search, threading, labeling) matters more than write throughput. Spam filtering is where most of the engineering complexity actually lives.

Design an Email Service (Gmail)

Requirements

  • Send and receive emails supporting SMTP, IMAP, and POP3 protocols
  • Store billions of emails with full-text search across subject, body, and attachments
  • Provide spam filtering, labels/folders, threaded conversations, and attachment handling
  • Support push notifications for new mail with sub-second delivery

Architecture

Incoming mail arrives via SMTP servers, passes through a spam/virus filter pipeline, and is stored in a distributed mail store indexed per user. A search service indexes email content using an inverted index. Outgoing mail goes through a sending pipeline with DKIM/SPF signing. A notification service pushes new-mail events to connected clients.

Key Components

  • SMTP Gateway: Receives inbound email, performs MX lookup for outbound routing
  • Spam Filter Pipeline: Multi-stage filter (SPF/DKIM check, ML classifier, rule engine)
  • Mail Store: Bigtable-like storage with per-user partitioning and message deduplication
  • Search Index: Per-user inverted index updated asynchronously on mail receipt

Scaling

Partition by user ID; each user's mailbox lives on one shard with replication. Blob storage for large attachments (referenced by pointer in the mail store). Separate the search index into its own cluster.

Trade-offs

  • Per-user partitioning simplifies access patterns but creates hot shards for heavy users
  • Synchronous spam filtering adds latency to delivery; async risks delivering spam first
  • Full-text search of attachments requires OCR/parsing pipelines, adding complexity

Exercises

  1. Build a Threaded Conversation Grouper

    Given a set of emails with In-Reply-To and References headers, group them into conversation threads. Handle cases where intermediate emails are missing.

    Hint

    Build a forest of message IDs using In-Reply-To as parent pointers. For missing intermediates, create placeholder nodes. Merge trees that share References.

  2. Implement a Simple Spam Classifier

    Build a Naive Bayes spam classifier trained on a labeled email dataset. Compute precision and recall on a test set. Integrate it as a filter step in a mock email pipeline.

    Hint

    Tokenize email body, compute P(word|spam) and P(word|ham) from training data. For a new email, multiply probabilities across all words and compare spam vs ham likelihoods.

Common Problems & Solutions

1 Search results are missing recently received emails
Cause: The search index update is asynchronous and lags behind mail delivery during peak hours
Solution: Implement a real-time indexing path for recent emails (last N minutes) that queries the mail store directly, merged with the async index results.
2 Sending reputation degrades, causing outbound emails to land in recipients' spam folders
Cause: Compromised accounts are sending bulk spam through the outbound pipeline, tainting the IP reputation
Solution: Implement outbound rate limits per user, detect anomalous sending patterns, and use dedicated IP pools that isolate transactional mail from user-generated mail.
