Design a Distributed Cache (Memcached)

Intuition

A distributed cache is a partitioned in-memory hash table that trades durability for speed, serving as a buffer between application servers and databases. The key design challenge is maintaining high hit rates while handling node failures without cascading load to the database.

Design a Distributed Cache (Memcached)

Requirements

  • Provide sub-millisecond GET/SET operations for cached data
  • Distribute cached data across multiple nodes with consistent hashing
  • Handle node failures gracefully without losing all cached data
  • Support TTL-based expiration and LRU eviction policies
  • Scale horizontally by adding more cache nodes

Architecture

Clients use a consistent hashing library to route requests directly to the appropriate cache node (no central coordinator). Each cache node stores key-value pairs in a hash table with a doubly-linked list for LRU eviction. A slab allocator manages memory to avoid fragmentation. Monitoring tracks hit rates and eviction counts.

Key Components

  • Client Library: Implements consistent hashing, connection pooling, and failover logic
  • Hash Table: O(1) lookups using chained hashing within each cache node
  • LRU Eviction: Doubly-linked list tracks access order; evicts least recently used items when memory is full
  • Slab Allocator: Pre-allocates memory in size classes to reduce fragmentation
  • TTL Manager: Lazy expiration on access plus periodic background sweep of expired keys

Scaling

Add nodes to the hash ring to increase capacity. Use virtual nodes for balanced distribution. Implement warm-up strategies when adding nodes to avoid cold-start cache misses. Use mcrouter or similar proxy for advanced routing.

Trade-offs

  • No replication means node failure loses that shard's cache (fast rebuild from DB vs. replicated cache cost)
  • Larger items reduce effective cache capacity; compress values to fit more but add CPU overhead

Exercises

  1. Implement an LRU Cache with O(1) Operations

    Build an LRU cache using a hash map and a doubly-linked list. GET moves the item to the head. SET adds to the head and evicts the tail if at capacity. Both operations must be O(1) time complexity.

    Hint

    The hash map stores key -> node pointer. The doubly-linked list orders items by access recency. On GET, unlink the node and move to head. On SET at capacity, remove the tail node and delete from the hash map.

  2. Design a Cache Warming Strategy

    When a new cache node is added to the cluster, design a system that pre-populates it with likely-to-be-requested data to avoid a thundering herd of cache misses hitting the database. Consider using access logs and request sampling.

    Hint

    Replay recent access logs filtered to keys that hash to the new node. Alternatively, during a transition period, read from both old and new nodes, populating the new node on cache misses.

Common Problems & Solutions

1 A cache node crashes and the sudden flood of cache misses overwhelms the database, causing a cascading failure.
Cause: All keys previously served by the crashed node now miss cache and hit the database simultaneously (thundering herd), exceeding its capacity.
Solution: Use consistent hashing so only keys from the failed node redistribute. Implement request coalescing so multiple requests for the same key share a single DB lookup. Add circuit breakers and request queuing to protect the database.
2 A popular item expires from cache and thousands of concurrent requests simultaneously try to rebuild it from the database.
Cause: TTL-based expiration of a hot key causes a stampede where every request cache-misses at the same instant.
Solution: Implement cache stampede protection using a mutex: the first request acquires a lock and rebuilds the cache while others wait or serve stale data. Alternatively, use probabilistic early expiration where items are refreshed before actual TTL.
