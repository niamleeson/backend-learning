Transactions and Connection Pooling

Intuition

Transactions are like an all-or-nothing promise. Connection pooling is like a carpool lane: sharing rides to the database so everyone gets there faster.

Transactions and Connection Pooling

Transactions ensure atomicity: a group of operations either all succeed or all fail. Connection pooling reuses database connections for efficiency.

from sqlalchemy import create_engine

engine = create_engine(
    "postgresql+psycopg2://user:pass@localhost/db",
    pool_size=10,
    max_overflow=20,
    pool_timeout=30,
    pool_recycle=1800
)

# Auto commit/rollback
with engine.begin() as conn:
    conn.execute(text("UPDATE accounts SET balance = balance - 100 WHERE id = 1"))
    conn.execute(text("UPDATE accounts SET balance = balance + 100 WHERE id = 2"))

# Manual transaction control
with engine.connect() as conn:
    trans = conn.begin()
    try:
        conn.execute(text("INSERT INTO orders (...) VALUES (...)"))
        conn.execute(text("UPDATE inventory SET qty = qty - 1 WHERE ..."))
        trans.commit()
    except Exception:
        trans.rollback()
        raise

# Nested savepoints
with engine.begin() as conn:
    conn.execute(text("INSERT INTO logs (msg) VALUES ('start')"))
    sp = conn.begin_nested()
    try:
        conn.execute(text("INSERT INTO risky_table VALUES (...)"))
        sp.commit()
    except Exception:
        sp.rollback()

Key Takeaways: - engine.begin() provides a transaction that auto-commits on success and rolls back on exception. - Connection pools avoid the overhead of creating new TCP connections for every request. - Savepoints (begin_nested) allow partial rollback within a larger transaction.

Exercises

  1. Implement a Transfer Function

    Write a transfer_funds function that debits one account and credits another atomically, rolling back if the source has insufficient balance.

    Hint

    Use engine.begin(), check balance first, raise an exception to trigger rollback if insufficient.

  2. Pool Monitoring

    Write code that prints the current pool status: checked-in, checked-out, and overflow connections.

    Hint

    Use engine.pool.status() or engine.pool.checkedin(), engine.pool.checkedout().

Common Problems & Solutions

1 Connection pool exhausted; requests hang waiting.
Cause: Connections are not returned to the pool due to missing close/context manager.
Solution: Always use context managers (with engine.connect()). Monitor pool with pool events and adjust pool_size.
2 Database reports idle connections that were never closed.
Cause: Connections exceeded pool_recycle time and became stale.
Solution: Set pool_recycle to less than the database's wait_timeout. Enable pool_pre_ping=True for health checks.
