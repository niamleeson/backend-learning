Encryption: Symmetric vs Asymmetric, TLS Handshake

Intuition

Think of encryption like sending a locked box. Symmetric encryption is like Alice and Bob both having a copy of the same padlock key — fast to open, but they had to meet in person first to copy the key. Asymmetric encryption is like Bob sending Alice an open padlock (his public key) — she locks the box with it, and only Bob's unique key can open it. TLS combines both: they use the padlock trick to securely exchange a shared key, then switch to the fast shared-key method for the actual conversation.

Encryption: Symmetric vs Asymmetric, TLS Handshake

Encryption is the backbone of modern security infrastructure. Every HTTPS request, every database credential, every stored password relies on well-implemented encryption. As a backend engineer, understanding encryption is not optional — it is a core competency. In this lesson, we will dissect symmetric and asymmetric encryption, walk through the TLS 1.3 handshake step by step, and write real Python code using the cryptography library.

Symmetric Encryption

Symmetric encryption uses a single shared key for both encryption and decryption. If Alice and Bob both know the secret key, Alice can encrypt a message and Bob can decrypt it with the same key. The fundamental challenge is: how do you securely share that key in the first place?

AES-GCM (Advanced Encryption Standard - Galois/Counter Mode)

AES is the gold standard for symmetric encryption. It operates on 128-bit blocks and supports key sizes of 128, 192, or 256 bits. GCM mode is an authenticated encryption mode, meaning it provides both confidentiality (nobody can read the data) and integrity (nobody can tamper with the data undetected).

AES-GCM produces a ciphertext and an authentication tag. If even a single bit of the ciphertext is modified, decryption will fail. This is critical for preventing bit-flipping attacks.

from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import os

# Generate a 256-bit key
key = AESGCM.generate_key(bit_length=256)

# Create the cipher
aesgcm = AESGCM(key)

# Nonce MUST be unique for every encryption with the same key
# 96 bits (12 bytes) is the recommended size for GCM
nonce = os.urandom(12)

# Optional associated data (authenticated but not encrypted)
associated_data = b"authenticated-header-data"

# Encrypt
plaintext = b"sensitive database credentials"
ciphertext = aesgcm.encrypt(nonce, plaintext, associated_data)

# Decrypt
decrypted = aesgcm.decrypt(nonce, ciphertext, associated_data)
assert decrypted == plaintext

Key points about AES-GCM: - The nonce must NEVER be reused with the same key. Reusing a nonce completely breaks GCM security. - The nonce is not secret — it is typically prepended to the ciphertext for storage. - AES-GCM is hardware-accelerated on modern CPUs via AES-NI instructions, making it extremely fast.

ChaCha20-Poly1305

ChaCha20-Poly1305 is an alternative authenticated encryption scheme designed by Daniel Bernstein. It combines the ChaCha20 stream cipher with the Poly1305 authenticator. Its primary advantage is performance on devices without hardware AES support (such as older mobile devices and IoT hardware).

from cryptography.hazmat.primitives.ciphers.aead import ChaCha20Poly1305
import os

key = ChaCha20Poly1305.generate_key()
cipher = ChaCha20Poly1305(key)
nonce = os.urandom(12)

ciphertext = cipher.encrypt(nonce, b"secret message", b"aad")
plaintext = cipher.decrypt(nonce, ciphertext, b"aad")

ChaCha20-Poly1305 is used heavily in TLS 1.3 and by protocols like WireGuard. On platforms with AES-NI, AES-GCM is typically faster. On platforms without it, ChaCha20 wins.

The ECB Mode Disaster

One of the most common encryption mistakes is using ECB (Electronic Codebook) mode. ECB encrypts each block independently, which means identical plaintext blocks produce identical ciphertext blocks. This leaks structural information about the data.

The classic demonstration is encrypting a bitmap image with ECB — the outline of the image remains clearly visible in the ciphertext. Never use ECB mode. Always use authenticated encryption modes like GCM or use CBC with HMAC.

# NEVER DO THIS — ECB mode leaks patterns
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes

key = os.urandom(32)
cipher = Cipher(algorithms.AES(key), modes.ECB())  # DANGEROUS
encryptor = cipher.encryptor()
# Identical blocks produce identical ciphertext — pattern leakage!

Asymmetric Encryption

Asymmetric encryption uses a key pair: a public key and a private key. Data encrypted with the public key can only be decrypted with the private key, and vice versa. This solves the key distribution problem inherent in symmetric encryption.

RSA (Rivest-Shamir-Adleman)

RSA is the most widely known asymmetric algorithm. Its security is based on the difficulty of factoring large prime numbers. RSA keys are typically 2048 or 4096 bits.

from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives import hashes

# Generate RSA key pair
private_key = rsa.generate_private_key(
    public_exponent=65537,
    key_size=2048,
)
public_key = private_key.public_key()

# Encrypt with public key
message = b"secret api key"
ciphertext = public_key.encrypt(
    message,
    padding.OAEP(
        mgf=padding.MGF1(algorithm=hashes.SHA256()),
        algorithm=hashes.SHA256(),
        label=None,
    ),
)

# Decrypt with private key
plaintext = private_key.decrypt(
    ciphertext,
    padding.OAEP(
        mgf=padding.MGF1(algorithm=hashes.SHA256()),
        algorithm=hashes.SHA256(),
        label=None,
    ),
)
assert plaintext == message

RSA is slow compared to symmetric encryption — roughly 1,000 times slower for equivalent data sizes. This is why RSA is never used to encrypt bulk data. Instead, it is used to encrypt symmetric keys or to create digital signatures.

ECDSA (Elliptic Curve Digital Signature Algorithm)

ECDSA provides the same security as RSA with much smaller key sizes. A 256-bit ECDSA key provides roughly the same security as a 3072-bit RSA key. This makes ECDSA the preferred choice for digital signatures in modern systems.

from cryptography.hazmat.primitives.asymmetric import ec
from cryptography.hazmat.primitives import hashes

# Generate ECDSA key pair
private_key = ec.generate_private_key(ec.SECP256R1())
public_key = private_key.public_key()

# Sign a message
message = b"critical transaction data"
signature = private_key.sign(message, ec.ECDSA(hashes.SHA256()))

# Verify the signature (raises InvalidSignature if verification fails)
public_key.verify(signature, message, ec.ECDSA(hashes.SHA256()))

ECDSA is used extensively in TLS certificates, JWT tokens, and blockchain systems. The curve SECP256R1 (also called P-256) is the most widely used.

Key Exchange: Diffie-Hellman and ECDHE

The Diffie-Hellman key exchange protocol allows two parties to establish a shared secret over an insecure channel without ever transmitting the secret itself. This is one of the most elegant algorithms in cryptography.

How Diffie-Hellman Works

  1. Alice and Bob agree on public parameters (a large prime p and a generator g).
  2. Alice picks a secret integer a, computes A = g^a mod p, and sends A to Bob.
  3. Bob picks a secret integer b, computes B = g^b mod p, and sends B to Alice.
  4. Alice computes s = B^a mod p.
  5. Bob computes s = A^b mod p.
  6. Both arrive at the same shared secret s = g^(ab) mod p.

An eavesdropper sees A and B but cannot compute s without knowing a or b — this is the discrete logarithm problem.

ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)

ECDHE is the elliptic curve variant of Diffie-Hellman with ephemeral keys (new keys generated for each session). This is the key exchange mechanism used in TLS 1.3.

from cryptography.hazmat.primitives.asymmetric import ec
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.kdf.hkdf import HKDF

# Alice generates ephemeral key pair
alice_private = ec.generate_private_key(ec.SECP256R1())
alice_public = alice_private.public_key()

# Bob generates ephemeral key pair
bob_private = ec.generate_private_key(ec.SECP256R1())
bob_public = bob_private.public_key()

# Both derive the same shared secret
alice_shared = alice_private.exchange(ec.ECDH(), bob_public)
bob_shared = bob_private.exchange(ec.ECDH(), alice_public)
assert alice_shared == bob_shared

# Derive a usable symmetric key from the shared secret
derived_key = HKDF(
    algorithm=hashes.SHA256(),
    length=32,
    salt=None,
    info=b"handshake data",
).derive(alice_shared)

The "ephemeral" part is critical. Because new keys are generated for every session, compromising a server's long-term private key does not allow an attacker to decrypt past sessions. This property is called Perfect Forward Secrecy (PFS).

The TLS 1.3 Handshake — Step by Step

TLS 1.3 (defined in RFC 8446) dramatically simplified the handshake compared to TLS 1.2. It completes in just one round trip (1-RTT), down from two.

Step 1: Client Hello

The client sends: - Supported TLS versions (only 1.3) - Supported cipher suites (e.g., TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256) - Key share: the client's ECDHE public key (speculative — the client guesses which curve the server will pick) - Supported signature algorithms - A random 32-byte nonce

Step 2: Server Hello

The server responds with: - Chosen cipher suite - Server's ECDHE public key (key share) - A random 32-byte nonce

At this point, both sides can compute the shared secret using ECDHE. The handshake encryption keys are derived immediately.

Step 3: Encrypted Extensions, Certificate, CertificateVerify, Finished (Server)

All subsequent messages are encrypted using the handshake keys: - EncryptedExtensions: Additional server parameters. - Certificate: The server's X.509 certificate chain (proving its identity). - CertificateVerify: A signature over the handshake transcript using the server's private key. This proves the server owns the certificate. - Finished: A MAC over the entire handshake transcript, confirming the handshake was not tampered with.

Step 4: Finished (Client)

The client verifies the server's certificate chain against trusted CAs, checks the CertificateVerify signature, and sends its own Finished message.

Step 5: Application Data

Both sides derive application traffic keys from the shared secret and begin exchanging encrypted application data.

TLS 1.3 Cipher Suites

TLS 1.3 removed all insecure cipher suites. Only five are defined:

Cipher Suite Encryption Hash
TLS_AES_128_GCM_SHA256 AES-128-GCM SHA-256
TLS_AES_256_GCM_SHA384 AES-256-GCM SHA-384
TLS_CHACHA20_POLY1305_SHA256 ChaCha20-Poly1305 SHA-256
TLS_AES_128_CCM_SHA256 AES-128-CCM SHA-256
TLS_AES_128_CCM_8_SHA256 AES-128-CCM-8 SHA-256

All TLS 1.3 cipher suites use ECDHE for key exchange and provide Perfect Forward Secrecy by default. There is no option to disable PFS in TLS 1.3 — this was a deliberate design decision.

Perfect Forward Secrecy (PFS)

PFS ensures that if a server's long-term private key is compromised, previously recorded encrypted sessions remain secure. This is achieved by using ephemeral keys for each session.

Without PFS (as was common in TLS 1.2 with RSA key exchange), an attacker who records encrypted traffic and later obtains the server's private key can decrypt all recorded sessions. With PFS via ECDHE, each session uses a unique key pair that is discarded after the session ends. There is nothing to compromise retroactively.

PFS is mandatory in TLS 1.3. If you are still running TLS 1.2, ensure your server prefers ECDHE cipher suites.

Encryption at Rest vs. in Transit

Encryption in transit protects data as it moves between systems — TLS is the primary mechanism. This prevents eavesdropping and man-in-the-middle attacks.

Encryption at rest protects data stored on disk — in databases, file systems, backups, and logs. Common approaches include:

  • Full-disk encryption (LUKS, BitLocker): Transparent to applications, protects against physical theft.
  • Database-level encryption (TDE — Transparent Data Encryption): The database engine encrypts data files.
  • Application-level encryption: The application encrypts specific fields before storing them. This provides the strongest guarantees because the database never sees plaintext.

Using Fernet for Application-Level Encryption

Fernet is a high-level symmetric encryption recipe from the cryptography library. It uses AES-128-CBC with HMAC-SHA256 for authentication. It handles nonce generation, padding, and authentication automatically.

from cryptography.fernet import Fernet

# Generate a key (store this securely — e.g., in a KMS)
key = Fernet.generate_key()
f = Fernet(key)

# Encrypt
token = f.encrypt(b"user SSN: 123-45-6789")

# Decrypt
plaintext = f.decrypt(token)
# b'user SSN: 123-45-6789'

# Fernet tokens include a timestamp — you can enforce TTL
try:
    f.decrypt(token, ttl=60)  # Reject if token is older than 60 seconds
except Exception:
    print("Token expired")

Fernet is excellent for encrypting configuration secrets, PII fields in a database, or short-lived tokens. For large data (files, streams), use AES-GCM directly.

Key Management

The hardest part of encryption is not the algorithm — it is key management. Where do you store the encryption key? If the key is hardcoded in source code or stored alongside the encrypted data, encryption provides zero value.

Best practices: - Use a Key Management Service (KMS) such as AWS KMS, Google Cloud KMS, or HashiCorp Vault. - Use envelope encryption: encrypt data with a data encryption key (DEK), then encrypt the DEK with a key encryption key (KEK) stored in KMS. - Rotate keys periodically. Fernet supports key rotation with MultiFernet.

from cryptography.fernet import Fernet, MultiFernet

# Key rotation: new key first, old key second
old_key = Fernet(Fernet.generate_key())
new_key = Fernet(Fernet.generate_key())
multi = MultiFernet([new_key, old_key])

# Encrypt with the first (newest) key
token = multi.encrypt(b"data")

# Decrypt tries each key in order
plaintext = multi.decrypt(token)

# Re-encrypt old tokens with the new key
rotated_token = multi.rotate(token)

Common Mistakes and Pitfalls

1. Using ECB Mode

As discussed above, ECB leaks patterns. Always use authenticated encryption (GCM, CCM, or ChaCha20-Poly1305).

2. Hardcoded Keys

# NEVER DO THIS
SECRET_KEY = b"my-super-secret-key-12345678901"  # Hardcoded in source

Hardcoded keys end up in version control, CI logs, and Docker images. Use environment variables, secret managers, or KMS.

3. Weak Random Number Generation

import random

# NEVER use Python's random module for cryptography
key = bytes([random.randint(0, 255) for _ in range(32)])  # INSECURE

# ALWAYS use os.urandom or secrets module
import os
import secrets

key = os.urandom(32)       # Cryptographically secure
token = secrets.token_hex(32)  # Cryptographically secure

Python's random module uses a Mersenne Twister PRNG, which is deterministic and predictable. For any security-sensitive operation, use os.urandom() or the secrets module.

4. Nonce Reuse in GCM

Reusing a nonce with the same key in GCM mode allows an attacker to recover the authentication key and forge ciphertexts. For AES-GCM, either use a counter-based nonce (if you can guarantee uniqueness) or use random 96-bit nonces (safe for up to ~2^32 encryptions per key).

5. Not Verifying Certificates

import requests

# NEVER DO THIS in production
response = requests.get("https://api.example.com", verify=False)  # INSECURE

# Always verify certificates
response = requests.get("https://api.example.com", verify=True)  # Default

Disabling certificate verification defeats the purpose of TLS entirely. It makes the connection vulnerable to man-in-the-middle attacks.

6. Using Deprecated Algorithms

Avoid MD5, SHA-1, DES, 3DES, and RC4. These are all broken or deprecated. Use SHA-256 or SHA-384 for hashing, AES-256 for symmetric encryption, and ECDSA with P-256 or Ed25519 for signatures.

Putting It All Together: A Real-World Example

Here is a complete example of encrypting sensitive user data before storing it in a database, with proper key management patterns.

import os
import json
import base64
from cryptography.hazmat.primitives.ciphers.aead import AESGCM

class FieldEncryptor:
    """Encrypts individual database fields using AES-256-GCM."""

    def __init__(self, key: bytes):
        if len(key) != 32:
            raise ValueError("Key must be 256 bits (32 bytes)")
        self._aesgcm = AESGCM(key)

    def encrypt_field(self, plaintext: str, context: str = "") -> str:
        """Encrypt a string field. Context provides additional authenticated data."""
        nonce = os.urandom(12)
        ciphertext = self._aesgcm.encrypt(
            nonce,
            plaintext.encode("utf-8"),
            context.encode("utf-8") if context else None,
        )
        # Encode nonce + ciphertext as base64 for storage
        payload = base64.b64encode(nonce + ciphertext).decode("ascii")
        return payload

    def decrypt_field(self, encrypted: str, context: str = "") -> str:
        """Decrypt a string field."""
        raw = base64.b64decode(encrypted)
        nonce = raw[:12]
        ciphertext = raw[12:]
        plaintext = self._aesgcm.decrypt(
            nonce,
            ciphertext,
            context.encode("utf-8") if context else None,
        )
        return plaintext.decode("utf-8")

# Usage
key = os.urandom(32)  # In production, load from KMS
encryptor = FieldEncryptor(key)

# Encrypt SSN with user_id as authenticated context
encrypted_ssn = encryptor.encrypt_field(
    "123-45-6789",
    context="user:42"
)
print(f"Encrypted: {encrypted_ssn}")

# Decrypt
ssn = encryptor.decrypt_field(encrypted_ssn, context="user:42")
print(f"Decrypted: {ssn}")

The context parameter binds the ciphertext to a specific user, preventing an attacker from swapping encrypted fields between database rows.

Summary

  • Symmetric encryption (AES-GCM, ChaCha20-Poly1305) is fast and used for bulk data encryption. It requires a shared key.
  • Asymmetric encryption (RSA, ECDSA) solves the key distribution problem but is slow. It is used for key exchange and digital signatures.
  • ECDHE provides ephemeral key exchange, enabling Perfect Forward Secrecy.
  • TLS 1.3 uses ECDHE + AEAD ciphers, completes in one round trip, and mandates PFS.
  • Encryption at rest requires careful key management — use a KMS, not hardcoded keys.
  • Always use authenticated encryption, never ECB mode, never random for cryptographic operations, and never disable certificate verification.

Common Problems & Solutions

1 AES-GCM decryption raises InvalidTag exception even though the key and data appear correct
Symptoms: cryptography.exceptions.InvalidTag raised during decrypt(). The error message provides no detail about which parameter is wrong, by design — leaking that information would help attackers.
Cause: The most common causes are: (1) nonce mismatch — the nonce used for decryption differs from the one used during encryption, often due to incorrect slicing when the nonce is prepended to the ciphertext, (2) associated authenticated data (AAD) mismatch — the AAD passed during decryption does not match what was used during encryption, or (3) the ciphertext was truncated or corrupted during storage or transmission (e.g., improper base64 encoding/decoding).
Solution: Verify that you are extracting the nonce correctly (first 12 bytes for GCM). Ensure the AAD is identical during encryption and decryption — even a difference in encoding (str vs bytes, trailing newline) will cause failure. Add logging to print the lengths of nonce, ciphertext, and AAD at both encrypt and decrypt time to detect mismatches.

Common mistake: forgetting to separate nonce from ciphertext

raw = base64.b64decode(stored_value) nonce = raw[:12] # First 12 bytes ciphertext = raw[12:] # Everything after

Also verify AAD matches exactly

aad_encrypt = f"user:{user_id}".encode() # During encryption aad_decrypt = f"user:{user_id}".encode() # Must be identical

2 TLS handshake fails with 'certificate verify failed' error when connecting to an internal service
Symptoms: ssl.SSLCertVerificationError: [SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed: unable to get local issuer certificate. Or requests.exceptions.SSLError with a similar message.
Cause: The internal service uses a certificate signed by a private/corporate Certificate Authority (CA) that is not in the system's default trust store. Python's ssl module and the requests library only trust CAs in the default bundle (typically from certifi or the OS trust store).
Solution: Add the internal CA's root certificate to the trust store. For requests, pass verify='/path/to/internal-ca-bundle.pem'. For ssl, use context.load_verify_locations('/path/to/ca.pem'). Never set verify=False in production — instead, properly distribute and trust the internal CA certificate.

import requests import ssl

For requests library

response = requests.get( "https://internal-api.corp.com", verify="/etc/ssl/certs/internal-ca.pem" )

For ssl module

ctx = ssl.create_default_context() ctx.load_verify_locations("/etc/ssl/certs/internal-ca.pem")

You can also set the environment variable

export REQUESTS_CA_BUNDLE=/etc/ssl/certs/internal-ca.pem

3 Encrypted data in the database is significantly larger than expected, causing storage and performance issues
Cause: Each encrypted field includes overhead: 12 bytes for the nonce, 16 bytes for the GCM authentication tag, and base64 encoding adds ~33% size increase. If you are encrypting many small fields (e.g., individual columns), the overhead multiplies. Additionally, if Fernet is used, it adds even more overhead due to its token format (version byte, timestamp, IV, padding, HMAC).
Solution: For many small fields, consider encrypting a JSON object containing all sensitive fields as a single blob rather than encrypting each field individually. This amortizes the overhead. If size is critical, use raw binary storage instead of base64 encoding (saves ~33%). Evaluate whether all fields truly need application-level encryption or if database-level TDE is sufficient for some.

Instead of encrypting each field separately:

enc_ssn = encrypt(ssn) # 11 chars -> ~56 bytes enc_dob = encrypt(dob) # 10 chars -> ~56 bytes enc_address = encrypt(address) # 50 chars -> ~100 bytes

Total: ~212 bytes for 71 chars of data

Encrypt as a single JSON blob:

sensitive = json.dumps({"ssn": ssn, "dob": dob, "address": address}) enc_blob = encrypt(sensitive) # 71 chars -> ~140 bytes

Total: ~140 bytes — significant savings
