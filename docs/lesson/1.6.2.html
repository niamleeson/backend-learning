Virtual Environments: venv, pip, pip-tools

Intuition

Virtual environments are isolated copies of Python's site-packages directory. Activating one just adjusts PATH so that python and pip point to the isolated copy.

Virtual environments isolate project dependencies, preventing conflicts between projects. venv is built-in; pip installs packages into the active environment.

# Create and activate a virtual environment
# $ python -m venv .venv
# $ source .venv/bin/activate    (macOS/Linux)
# $ .venv\Scripts\activate       (Windows)

# Install packages
# $ pip install requests flask
# $ pip freeze > requirements.txt
# $ pip install -r requirements.txt

# pip-tools for pinned, reproducible dependencies
# $ pip install pip-tools
# $ echo "flask" > requirements.in
# $ pip-compile requirements.in    # generates requirements.txt with pins
# $ pip-sync requirements.txt      # installs exactly what's pinned

import sys
print(sys.prefix)  # shows the active venv path
  • Always work inside a virtual environment to avoid polluting the system Python.
  • pip freeze captures all installed packages and versions for reproducibility.
  • pip-tools separates intent (requirements.in) from pinned output (requirements.txt).
  • Deactivate with deactivate; delete the .venv directory to remove the environment entirely.

Exercises

  1. Create and Inspect a Venv

    Create a virtual environment, activate it, install requests, then write a script that prints sys.prefix and requests.__version__.

    Hint

    Use python -m venv test_env, activate it, then pip install requests.

  2. Pin Dependencies with pip-tools

    Create a requirements.in with flask and requests, run pip-compile, and examine the generated requirements.txt to see all transitive dependencies.

    Hint

    Install pip-tools, write the .in file, run pip-compile requirements.in.

Common Problems & Solutions

1 Running `pip install` outside a virtual environment installs packages globally.
Cause: Without an active venv, pip targets the system or user site-packages.
Solution: Always activate a venv first, or use `python -m pip` with the venv's Python binary.
2 `requirements.txt` works on your machine but fails on CI.
Cause: Unpinned dependencies (`requests>=2.0`) may resolve to different versions.
Solution: Use `pip freeze` or `pip-compile` to pin exact versions for reproducibility.
