Project Setup: Blog API with Auth and CRUD

Intuition

A blog API is the classic full-stack exercise because it covers every CRUD operation, authentication, authorization (only the author can edit), and relationships between resources. Building one end-to-end solidifies your understanding of how all the pieces connect.

Project: Blog API with Auth and CRUD

This capstone project ties together Flask, SQLAlchemy, Marshmallow, and JWT authentication into a complete blog API. You will build user registration, login, and authenticated CRUD operations for posts and comments.

from flask import Flask
from flask_smorest import Api
from flask_sqlalchemy import SQLAlchemy
from flask_jwt_extended import JWTManager
from flask_migrate import Migrate

db = SQLAlchemy()
jwt = JWTManager()
migrate = Migrate()

def create_app():
    app = Flask(__name__)
    app.config.update(
        API_TITLE='Blog API', API_VERSION='v1',
        OPENAPI_VERSION='3.0.3',
        SQLALCHEMY_DATABASE_URI='sqlite:///blog.db',
        JWT_SECRET_KEY='change-me-in-production'
    )
    db.init_app(app)
    jwt.init_app(app)
    migrate.init_app(app, db)

    from .auth import auth_blp
    from .posts import posts_blp
    api = Api(app)
    api.register_blueprint(auth_blp)
    api.register_blueprint(posts_blp)
    return app

# models.py
class User(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    username = db.Column(db.String(80), unique=True, nullable=False)
    password_hash = db.Column(db.String(128), nullable=False)
    posts = db.relationship('Post', backref='author', lazy='dynamic')

class Post(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    title = db.Column(db.String(200), nullable=False)
    body = db.Column(db.Text, nullable=False)
    user_id = db.Column(db.Integer, db.ForeignKey('user.id'), nullable=False)
    comments = db.relationship('Comment', backref='post', cascade='all, delete-orphan')

class Comment(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    body = db.Column(db.Text, nullable=False)
    user_id = db.Column(db.Integer, db.ForeignKey('user.id'), nullable=False)
    post_id = db.Column(db.Integer, db.ForeignKey('post.id'), nullable=False)

Key Takeaways: - Use passlib or werkzeug.security to hash passwords; never store plaintext. - Protect create/update/delete endpoints with @jwt_required() and verify ownership with get_jwt_identity(). - Structure the project with separate modules for auth, posts, comments, and shared models.

Exercises

  1. User Registration and Login

    Implement POST /auth/register (hash password, create user, return 201) and POST /auth/login (verify password, return JWT tokens). Include input validation with Marshmallow.

    Hint

    Use werkzeug.security.generate_password_hash() and check_password_hash() for password handling.

  2. Authenticated Post CRUD

    Build endpoints for creating, reading, updating, and deleting blog posts. Create and update require JWT authentication. Update and delete must verify the current user is the post author.

    Hint

    Compare get_jwt_identity() with post.user_id before allowing updates or deletes. Return 403 if they do not match.

Common Problems & Solutions

1 Password hash exceeds the database column length limit.
Cause: The password_hash column is defined as String(128) but some hashing algorithms (bcrypt, argon2) produce longer hashes.
Solution: Use String(256) or String(512) for the password_hash column to accommodate different hash algorithms and their output lengths.
2 Users can edit or delete posts created by other users.
Cause: The update/delete endpoint does not verify that the authenticated user owns the resource.
Solution: Add an ownership check: if post.user_id != get_jwt_identity(), abort(403). Apply this check consistently to all mutation endpoints.
