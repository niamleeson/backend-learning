Engine, Connection, and Execution

Intuition

The Engine is like a factory that hands out database connections from a pool. You borrow one, do your work, and return it -- the pool handles the lifecycle.

SQLAlchemy Core: Engine, Connection, Execution

The Engine is the starting point of SQLAlchemy. It manages the connection pool and provides the interface for executing SQL statements.

from sqlalchemy import create_engine, text

engine = create_engine(
    "postgresql+psycopg2://user:pass@localhost:5432/mydb",
    pool_size=5,
    max_overflow=10,
    echo=True
)

# Modern execution pattern (2.0 style)
with engine.connect() as conn:
    result = conn.execute(
        text("SELECT id, name FROM users WHERE active = :active"),
        {"active": True}
    )
    for row in result:
        print(row.id, row.name)
    conn.execute(
        text("UPDATE users SET last_login = now() WHERE id = :id"),
        {"id": 42}
    )
    conn.commit()

# Auto-commit block
with engine.begin() as conn:
    conn.execute(text("INSERT INTO logs (msg) VALUES (:m)"), {"m": "startup"})

Key Takeaways: - create_engine does not open a connection immediately; connections are lazy and pooled. - Always use text() with bound parameters (:param) to prevent SQL injection. - engine.begin() provides a transaction context that auto-commits or rolls back.

Common Problems & Solutions

1 QueuePool limit reached; connection timeout.
Cause: Connections are not being returned to the pool; missing context manager or explicit close.
Solution: Always use 'with engine.connect()' to ensure connections are returned. Increase pool_size if needed.
2 Changes are not persisted to the database.
Cause: Using engine.connect() without calling conn.commit() after write operations.
Solution: Call conn.commit() explicitly or use engine.begin() which auto-commits on context exit.
