Connection Pooling with PgBouncer

Intuition

Connection pooling is like a car-sharing service. Instead of everyone buying their own car (connection) that sits idle 90% of the time, a shared pool of cars serves everyone. PgBouncer is like adding a parking garage (external pooler) between the apartment building (app) and the highway (database) — residents don't each need a parking spot at the highway.

Connection Pooling with PgBouncer

Introduction

Every database query requires a connection. Creating a new PostgreSQL connection involves TCP handshake, TLS negotiation, authentication, and memory allocation — taking 50-100ms and consuming ~10MB of memory per connection. Connection pooling reuses existing connections, dramatically improving performance and allowing your application to handle far more concurrent requests than available database connections.

The Problem

Without Pooling:
100 concurrent requests → 100 DB connections → DB overwhelmed

PostgreSQL default max_connections = 100
Each connection: ~10MB RAM
100 connections: ~1GB RAM just for connections

With Pooling:
100 concurrent requests → 20 DB connections (pooled) → DB handles easily

Node.js Built-In Connection Pooling

Prisma and most Node.js database drivers include built-in connection pooling:

// Prisma connection pool configuration
// DATABASE_URL="postgresql://user:pass@host:5432/db?connection_limit=20&pool_timeout=10"

import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient({
  datasources: {
    db: {
      url: process.env.DATABASE_URL,
      // Connection pool settings are in the URL
    },
  },
  log: ['query', 'info', 'warn', 'error'],
});

// Knex.js connection pool
import knex from 'knex';

const db = knex({
  client: 'pg',
  connection: process.env.DATABASE_URL,
  pool: {
    min: 2,           // Minimum connections to maintain
    max: 20,          // Maximum connections
    idleTimeoutMillis: 30000,    // Close idle connections after 30s
    createTimeoutMillis: 5000,   // Max time to create new connection
    acquireTimeoutMillis: 10000, // Max time to acquire from pool
    reapIntervalMillis: 1000,    // How often to check for idle connections
  },
});

// pg (node-postgres) pool
import pg from 'pg';

const pool = new pg.Pool({
  connectionString: process.env.DATABASE_URL,
  max: 20,
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 5000,
});

// Monitor pool events
pool.on('connect', () => console.log('New connection created'));
pool.on('acquire', () => console.log('Connection acquired'));
pool.on('remove', () => console.log('Connection removed'));
pool.on('error', (err) => console.error('Pool error:', err));

PgBouncer: External Connection Pooler

PgBouncer sits between your application and PostgreSQL, managing a shared pool of connections:

Without PgBouncer:
┌──────────┐  20 conn   ┌────────────┐
│ App 1    │────────────>│            │
└──────────┘             │            │
┌──────────┐  20 conn   │ PostgreSQL │  60 total connections
│ App 2    │────────────>│            │
└──────────┘             │            │
┌──────────┐  20 conn   │            │
│ App 3    │────────────>│            │
└──────────┘             └────────────┘

With PgBouncer:
┌──────────┐             ┌───────────┐             ┌────────────┐
│ App 1    │──20 conn──> │           │             │            │
└──────────┘             │           │  20 conn    │            │
┌──────────┐             │ PgBouncer │────────────>│ PostgreSQL │
│ App 2    │──20 conn──> │           │  (shared)   │            │
└──────────┘             │           │             │            │
┌──────────┐             │           │             │            │
│ App 3    │──20 conn──> │           │             │            │
└──────────┘             └───────────┘             └────────────┘
                          60 app connections        20 DB connections

PgBouncer Modes

┌──────────────────────────────────────────────────────────────┐
│ Mode               │ Connection Reuse   │ Feature Support    │
├────────────────────┼────────────────────┼────────────────────┤
│ session            │ Per client session │ Full SQL support   │
│ (default)          │ (released on       │ Prepared stmts OK  │
│                    │  disconnect)       │                    │
├────────────────────┼────────────────────┼────────────────────┤
│ transaction        │ Per transaction    │ Most SQL           │
│ (recommended)      │ (released after    │ No prepared stmts  │
│                    │  COMMIT/ROLLBACK)  │ across txns        │
├────────────────────┼────────────────────┼────────────────────┤
│ statement          │ Per statement      │ No multi-statement │
│ (most aggressive)  │ (released after    │ No transactions    │
│                    │  each query)       │                    │
└──────────────────────────────────────────────────────────────┘

PgBouncer Configuration

; /etc/pgbouncer/pgbouncer.ini
[databases]
myapp = host=postgres-primary port=5432 dbname=myapp

[pgbouncer]
listen_port = 6432
listen_addr = 0.0.0.0

; Pool mode
pool_mode = transaction

; Pool sizing
default_pool_size = 20
min_pool_size = 5
max_client_conn = 200        ; Max frontend connections
max_db_connections = 50      ; Max backend connections per database

; Timeouts
server_idle_timeout = 600    ; Close idle backend connections after 10 min
client_idle_timeout = 0      ; No timeout for idle frontend connections
query_timeout = 30           ; Kill queries running longer than 30s
query_wait_timeout = 120     ; Max time to wait for a connection from pool

; Authentication
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt

; Logging
log_connections = 1
log_disconnections = 1
log_pooler_errors = 1
stats_period = 60

Docker Compose with PgBouncer

version: '3.8'
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: app
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data
    # Only accessible via PgBouncer, not directly
    expose:
      - "5432"

  pgbouncer:
    image: edoburu/pgbouncer
    environment:
      DATABASE_URL: postgres://app:secret@postgres:5432/myapp
      POOL_MODE: transaction
      DEFAULT_POOL_SIZE: 20
      MAX_CLIENT_CONN: 200
      MAX_DB_CONNECTIONS: 50
    ports:
      - "6432:5432"
    depends_on:
      - postgres

  app:
    build: .
    environment:
      # App connects to PgBouncer, NOT directly to PostgreSQL
      DATABASE_URL: postgres://app:secret@pgbouncer:5432/myapp
    depends_on:
      - pgbouncer

volumes:
  pgdata:

Connecting Node.js to PgBouncer

// Prisma with PgBouncer
// schema.prisma - add pgbouncer=true to enable compatible mode
// DATABASE_URL="postgresql://app:secret@pgbouncer:6432/myapp?pgbouncer=true&connection_limit=10"

const prisma = new PrismaClient();

// Knex with PgBouncer
const db = knex({
  client: 'pg',
  connection: 'postgresql://app:secret@pgbouncer:6432/myapp',
  pool: {
    min: 2,
    max: 10, // Per-instance limit; PgBouncer manages the real pool
  },
});

Monitoring Connection Pools

// Monitor Prisma connection pool
const prisma = new PrismaClient({
  log: [
    { emit: 'event', level: 'query' },
  ],
});

prisma.$on('query', (e) => {
  if (e.duration > 1000) {
    console.warn(`Slow query (${e.duration}ms): ${e.query}`);
  }
});

// Monitor pg Pool
app.get('/admin/pool-stats', async (req, res) => {
  res.json({
    totalCount: pool.totalCount,     // Total connections in pool
    idleCount: pool.idleCount,       // Idle connections
    waitingCount: pool.waitingCount, // Clients waiting for connection
  });
});

// PgBouncer stats via SQL
async function getPgBouncerStats() {
  const client = new pg.Client('postgresql://pgbouncer:6432/pgbouncer');
  await client.connect();

  const pools = await client.query('SHOW POOLS');
  const stats = await client.query('SHOW STATS');
  const clients = await client.query('SHOW CLIENTS');

  await client.end();
  return { pools: pools.rows, stats: stats.rows, clients: clients.rows };
}

Pool Sizing Guidelines

Rule of thumb for PostgreSQL:
  max_connections = (CPU cores * 2) + effective_io_concurrency

For a 4-core server: max_connections = (4 * 2) + 2 = 10

PgBouncer allows many app connections mapped to fewer DB connections:
  App instances: 10 (each with pool of 20)
  PgBouncer max_client_conn: 200
  PgBouncer max_db_connections: 20
  PostgreSQL max_connections: 25 (20 + overhead)

Common Pitfalls

  1. Too many connections: More connections != better performance. PostgreSQL performs worse with too many active connections due to context switching.
  2. Prepared statements with PgBouncer transaction mode: PgBouncer in transaction mode doesn't support prepared statements across transactions. Use pgbouncer=true in Prisma.
  3. Connection leak: Not releasing connections back to the pool (forgetting to call client.release() with pg). Use try/finally.
  4. Pool timeout too short: Under load, pool exhaustion causes query failures. Set appropriate acquire timeout.

Summary

Connection pooling is essential for production Node.js applications. Application-level pooling (Prisma, Knex, pg.Pool) manages connections within a single process. PgBouncer provides external pooling that works across multiple application instances, dramatically reducing the number of PostgreSQL connections needed. Use transaction mode for the best connection reuse, and size your pool based on CPU cores rather than expected concurrent requests.

Common Problems & Solutions

1 Application hangs under load with 'connection pool timeout' errors
Symptoms: Application works fine under low load but hangs or returns errors under high load.
Cause: All connections in the pool are in use and new queries cannot acquire a connection within the timeout period.
Solution: Increase pool size, optimize slow queries that hold connections too long, or add PgBouncer. Check for connection leaks (unreleased connections).
2 Prisma errors with PgBouncer: 'prepared statement does not exist'
Cause: PgBouncer in transaction mode doesn't support named prepared statements across transactions, which Prisma uses by default.
Solution: Add `?pgbouncer=true` to your DATABASE_URL. This tells Prisma to use protocol-level prepared statements compatible with PgBouncer.
