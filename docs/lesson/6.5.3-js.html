SSL/TLS Termination and HTTPS Setup

Intuition

SSL/TLS is like putting your letter (HTTP request) in a tamper-proof, addressed envelope before mailing it. Without encryption, anyone who handles the mail (routers, ISPs, attackers on public WiFi) can read the contents. TLS termination at Nginx is like having a secure mailroom at the building entrance -- the mailroom opens the encrypted envelope and passes the letter internally through the secure building (private network) to the right office (Node.js).

SSL/TLS Termination and HTTPS Setup

Why HTTPS is Non-Negotiable

HTTPS encrypts traffic between clients and your server, preventing eavesdropping, tampering, and impersonation. Modern browsers mark HTTP sites as "Not Secure." Search engines penalize HTTP. Many APIs (geolocation, service workers, HTTP/2) require HTTPS.

How TLS Works

Client                                   Server
  │                                        │
  │──── ClientHello (supported ciphers) ──►│
  │                                        │
  │◄── ServerHello (chosen cipher) ────────│
  │◄── Certificate (public key) ───────────│
  │                                        │
  │──── Verify certificate with CA ────────│
  │──── Key Exchange ──────────────────────►│
  │                                        │
  │◄════ Encrypted Connection ════════════►│
  │      (symmetric encryption)            │
  │                                        │

TLS Termination at Nginx

Client ──HTTPS──► Nginx ──HTTP──► Node.js
                  (TLS           (internal
                  terminates     network,
                  here)          no encryption
                                 needed)

Let's Encrypt with Certbot (Free SSL)

Installation and First Certificate

# Install Certbot
sudo apt update
sudo apt install certbot python3-certbot-nginx

# Get a certificate (automatic Nginx configuration)
sudo certbot --nginx -d myapp.com -d www.myapp.com

# Or standalone mode (when Nginx is not yet configured)
sudo certbot certonly --standalone -d myapp.com

# Certificate files are stored in:
# /etc/letsencrypt/live/myapp.com/fullchain.pem  (certificate + chain)
# /etc/letsencrypt/live/myapp.com/privkey.pem    (private key)

Auto-Renewal

# Test renewal
sudo certbot renew --dry-run

# Certbot installs a timer/cron automatically, but verify:
sudo systemctl status certbot.timer

# Or add manual cron
# 0 0 1,15 * * certbot renew --quiet --post-hook "systemctl reload nginx"

Certbot with Docker

# docker-compose.yml
services:
  nginx:
    image: nginx:1.25-alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/conf.d:/etc/nginx/conf.d:ro
      - certbot-conf:/etc/letsencrypt:ro
      - certbot-www:/var/www/certbot:ro

  certbot:
    image: certbot/certbot
    volumes:
      - certbot-conf:/etc/letsencrypt
      - certbot-www:/var/www/certbot
    command: certonly --webroot -w /var/www/certbot
             --email admin@myapp.com -d myapp.com
             --agree-tos --no-eff-email

volumes:
  certbot-conf:
  certbot-www:

# nginx/conf.d/default.conf
# First, get the certificate with HTTP only:
server {
    listen 80;
    server_name myapp.com;

    # Certbot challenge
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

# Then add HTTPS server block:
server {
    listen 443 ssl http2;
    server_name myapp.com;

    ssl_certificate /etc/letsencrypt/live/myapp.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.com/privkey.pem;

    # Include recommended SSL settings
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;

    location / {
        proxy_pass http://api:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

SSL Configuration Best Practices

server {
    listen 443 ssl http2;
    server_name myapp.com;

    # Certificate
    ssl_certificate /etc/letsencrypt/live/myapp.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.com/privkey.pem;

    # Protocol versions (disable TLS 1.0 and 1.1)
    ssl_protocols TLSv1.2 TLSv1.3;

    # Cipher suites (modern configuration)
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;

    # OCSP Stapling (faster certificate verification)
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/letsencrypt/live/myapp.com/chain.pem;
    resolver 8.8.8.8 8.8.4.4 valid=300s;
    resolver_timeout 5s;

    # Session settings
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;

    # Security headers
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
    add_header X-Frame-Options DENY always;
    add_header X-Content-Type-Options nosniff always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Content-Security-Policy "default-src 'self'" always;
}

Self-Signed Certificates (Development)

# Generate self-signed certificate for local development
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/ssl/private/selfsigned.key \
  -out /etc/ssl/certs/selfsigned.crt \
  -subj "/C=US/ST=State/L=City/O=Dev/CN=localhost"

# Generate for multiple domains (SAN)
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout localhost.key \
  -out localhost.crt \
  -subj "/CN=localhost" \
  -addext "subjectAltName=DNS:localhost,DNS:*.localhost,IP:127.0.0.1"

HTTPS in Node.js (Direct, Without Nginx)

// For cases where you need Node.js to handle TLS directly
import https from 'https';
import fs from 'fs';
import express from 'express';

const app = express();

// ... routes ...

const options = {
  key: fs.readFileSync('/etc/ssl/private/server.key'),
  cert: fs.readFileSync('/etc/ssl/certs/server.crt'),
};

https.createServer(options, app).listen(443, () => {
  console.log('HTTPS server running on port 443');
});

// Redirect HTTP to HTTPS
import http from 'http';
http.createServer((req, res) => {
  res.writeHead(301, { Location: `https://${req.headers.host}${req.url}` });
  res.end();
}).listen(80);

Testing SSL Configuration

# Test with curl
curl -vI https://myapp.com 2>&1 | grep -E "SSL|TLS|subject|issuer|expire"

# Check certificate details
openssl s_client -connect myapp.com:443 -servername myapp.com < /dev/null 2>/dev/null | openssl x509 -noout -dates -subject -issuer

# Check certificate expiration
echo | openssl s_client -servername myapp.com -connect myapp.com:443 2>/dev/null | openssl x509 -noout -enddate

# SSL Labs grade (online)
# https://www.ssllabs.com/ssltest/analyze.html?d=myapp.com

Certificate Monitoring Script

#!/bin/bash
# check-ssl.sh - Monitor certificate expiration

DOMAIN="myapp.com"
WARN_DAYS=30

EXPIRY=$(echo | openssl s_client -servername "$DOMAIN" -connect "$DOMAIN:443" 2>/dev/null | openssl x509 -noout -enddate | cut -d= -f2)

EXPIRY_EPOCH=$(date -d "$EXPIRY" +%s 2>/dev/null || date -j -f "%b %d %T %Y %Z" "$EXPIRY" +%s)
NOW_EPOCH=$(date +%s)
DAYS_LEFT=$(( (EXPIRY_EPOCH - NOW_EPOCH) / 86400 ))

echo "Certificate for $DOMAIN expires in $DAYS_LEFT days ($EXPIRY)"

if [[ $DAYS_LEFT -lt $WARN_DAYS ]]; then
    echo "WARNING: Certificate expiring soon!"
    # Send alert (email, Slack, etc.)
    exit 1
fi

HSTS (HTTP Strict Transport Security)

# Tell browsers to ALWAYS use HTTPS for this domain
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;

# max-age: how long (seconds) browsers should remember to use HTTPS
# includeSubDomains: applies to all subdomains
# preload: submit to browser preload list (hstspreload.org)

Common Pitfalls

  1. Not auto-renewing certificates - Let's Encrypt certificates expire in 90 days. Always set up auto-renewal with certbot.
  2. Mixed content - HTTPS page loading HTTP resources (images, scripts) causes browser warnings. Ensure all resources use HTTPS or protocol-relative URLs.
  3. Forgetting to redirect HTTP to HTTPS - Users typing your domain without 'https://' get the insecure version. Always redirect.
  4. Not setting HSTS - Without HSTS, the first visit can be intercepted before the redirect. HSTS tells browsers to always use HTTPS.
  5. Self-signed certificates in production - Browsers will show scary warnings. Always use Let's Encrypt or a CA-signed certificate for production.

Common Problems & Solutions

1 Certbot fails with 'connection refused' during challenge
Cause: Port 80 is not open in the firewall/security group, or another service is using port 80.
Solution: Ensure port 80 is open: `sudo ufw allow 80`. Check if Nginx is running and listening on port 80. If using standalone mode, stop Nginx first. For AWS, check the security group allows inbound port 80.
2 Browser shows 'Your connection is not private' after setting up SSL
Cause: Using a self-signed certificate, certificate does not match the domain, or the certificate chain is incomplete.
Solution: In production, use Let's Encrypt (free and trusted). Ensure the domain in the certificate matches the URL. Use fullchain.pem (not just cert.pem) to include the full certificate chain.
3 Mixed content warnings after enabling HTTPS
Cause: The page is served over HTTPS but includes resources (images, scripts, API calls) loaded over HTTP.
Solution: Update all resource URLs to use HTTPS or protocol-relative URLs (//example.com/resource). Add Content-Security-Policy header: `upgrade-insecure-requests`. Search your codebase for 'http://' references.
