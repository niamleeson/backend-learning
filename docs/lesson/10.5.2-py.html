Design a Ride-Sharing Service (Uber)

Intuition

A ride-sharing system is a real-time geospatial matching problem where the core challenge is maintaining a constantly-updating spatial index of drivers and querying it fast enough to match riders before drivers move significantly. The cell-based architecture ensures that a failure in one city never cascades to another.

Design a Ride-Sharing Service (Uber)

Requirements

  • Match riders with nearby drivers in real time (under 5 seconds)
  • Track live driver locations with updates every 3-5 seconds
  • Handle surge pricing based on supply/demand ratios per zone
  • Support trip lifecycle: request, match, pickup, in-progress, complete, payment

Architecture

Drivers continuously report GPS locations to a Location Service that indexes them in a geospatial data structure. When a rider requests a trip, the Matching Service queries nearby available drivers, ranks them, and dispatches. A Trip Service manages the state machine for each ride. A separate Pricing Service computes dynamic fares.

Key Components

  • Location Service: Geohash or QuadTree index of active driver positions in Redis/memory
  • Matching Service: Finds k-nearest available drivers, scores by ETA, rating, direction
  • Trip Service: State machine (requested -> matched -> en_route -> in_trip -> completed)
  • Pricing Service: Computes base fare + surge multiplier based on zone demand/supply ratio

Scaling

Partition geographically by city or geohash prefix. Location updates are write-heavy; use in-memory stores. Matching is compute-heavy; scale independently. Use cell-based architecture for fault isolation per city.

Trade-offs

  • Geohash is simpler but creates edge-of-cell boundary issues; QuadTree adapts to density but is harder to distribute
  • Greedy matching (nearest driver) is fast but globally suboptimal; batch matching optimizes overall wait time but adds latency
  • Frequent location updates improve accuracy but increase bandwidth and processing cost

Exercises

  1. Build a Geohash-Based Nearest Driver Finder

    Index 10,000 simulated driver locations using geohashes. Given a rider location, find the 5 nearest available drivers by expanding geohash rings. Measure query latency.

    Hint

    Start with the rider's geohash cell, check for drivers. If fewer than 5 found, expand to the 8 neighboring cells. Repeat until enough candidates, then sort by actual distance.

  2. Implement a Simple Surge Pricing Model

    Divide a city into a grid of zones. Track rider requests and available drivers per zone. Compute a surge multiplier when demand/supply ratio exceeds 1.5x.

    Hint

    Use a sliding window counter for requests and a point-in-time count for available drivers per zone. Multiplier = max(1.0, demand/supply * scaling_factor).

Common Problems & Solutions

1 Riders at geohash cell boundaries get matched with far-away drivers while nearby ones in the adjacent cell are ignored
Cause: The search only checks the rider's own geohash cell and misses drivers just across the border
Solution: Always search the target cell plus all adjacent cells (9-cell neighborhood), then filter by actual Haversine distance.
2 Driver location updates are lost during peak hours causing stale matches
Cause: The location service write throughput is saturated and starts dropping UDP packets
Solution: Use a partitioned in-memory store (e.g., Redis Cluster) for location writes, and switch to TCP with backpressure to avoid silent data loss.
