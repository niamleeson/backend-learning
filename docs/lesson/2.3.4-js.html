Triggers, Views, and Stored Procedures

Intuition

Views are like bookmarks for complex queries — you save the query once and reuse it by name. Triggers are like automatic alarms: when something changes in the database, they fire and do something. Stored procedures are like recipes stored in the kitchen (database) itself, rather than in a cookbook (application) that someone has to bring in from outside.

Triggers, Views, and Stored Procedures

Introduction

Triggers, views, and stored procedures (functions) move logic from your Node.js application into the database itself. This can improve performance, ensure consistency regardless of which client accesses the database, and simplify application code. However, it also makes debugging harder and ties your logic to a specific database. Understanding when and how to use these features is a key part of database mastery.

Views

A view is a named query that acts like a virtual table:

-- Create a view
CREATE VIEW active_products AS
SELECT p.id, p.name, p.price, c.name AS category
FROM products p
JOIN categories c ON p.category_id = c.id
WHERE p.is_active = true AND p.stock_quantity > 0;

-- Use it like a table
SELECT * FROM active_products WHERE price < 50;
SELECT category, COUNT(*) FROM active_products GROUP BY category;

// Views simplify Node.js queries
async function getActiveProducts(maxPrice) {
  const result = await pool.query(
    'SELECT * FROM active_products WHERE price <= $1 ORDER BY name',
    [maxPrice]
  );
  return result.rows;
}

Materialized Views

Pre-computed views stored on disk — faster reads, but stale data:

CREATE MATERIALIZED VIEW product_stats AS
SELECT
  p.id,
  p.name,
  COALESCE(AVG(r.rating), 0)::numeric(3,2) AS avg_rating,
  COUNT(r.id)::int AS review_count,
  COALESCE(SUM(oi.quantity), 0)::int AS total_sold
FROM products p
LEFT JOIN reviews r ON p.id = r.product_id
LEFT JOIN order_items oi ON p.id = oi.product_id
GROUP BY p.id, p.name;

-- Create index on materialized view
CREATE UNIQUE INDEX idx_product_stats_id ON product_stats (id);

-- Refresh (can be done concurrently to avoid locking)
REFRESH MATERIALIZED VIEW CONCURRENTLY product_stats;

// Refresh periodically
async function refreshProductStats() {
  await pool.query('REFRESH MATERIALIZED VIEW CONCURRENTLY product_stats');
}

// Run every 5 minutes
setInterval(refreshProductStats, 5 * 60 * 1000);

Triggers

Triggers automatically execute a function when a table event occurs:

-- Trigger function (must return TRIGGER)
CREATE OR REPLACE FUNCTION update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Attach trigger to table
CREATE TRIGGER set_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION update_timestamp();

Common Trigger Patterns

Audit Log

CREATE TABLE audit_log (
  id serial PRIMARY KEY,
  table_name text NOT NULL,
  record_id text NOT NULL,
  action text NOT NULL, -- INSERT, UPDATE, DELETE
  old_data jsonb,
  new_data jsonb,
  changed_by text,
  changed_at timestamptz DEFAULT NOW()
);

CREATE OR REPLACE FUNCTION audit_trigger()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO audit_log (table_name, record_id, action, old_data, new_data, changed_by)
  VALUES (
    TG_TABLE_NAME,
    COALESCE(NEW.id::text, OLD.id::text),
    TG_OP,
    CASE WHEN TG_OP != 'INSERT' THEN to_jsonb(OLD) END,
    CASE WHEN TG_OP != 'DELETE' THEN to_jsonb(NEW) END,
    current_setting('app.current_user', true)
  );
  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

-- Apply to any table
CREATE TRIGGER users_audit
  AFTER INSERT OR UPDATE OR DELETE ON users
  FOR EACH ROW EXECUTE FUNCTION audit_trigger();

// Set the current user for audit trails
async function queryAsUser(sql, params, userId) {
  const client = await pool.connect();
  try {
    await client.query("SET LOCAL app.current_user = $1", [userId]);
    const result = await client.query(sql, params);
    return result;
  } finally {
    client.release();
  }
}

Maintain Aggregate Counts

-- Keep a cached count on the parent table
ALTER TABLE categories ADD COLUMN product_count integer DEFAULT 0;

CREATE OR REPLACE FUNCTION update_category_count()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    UPDATE categories SET product_count = product_count + 1
    WHERE id = NEW.category_id;
  ELSIF TG_OP = 'DELETE' THEN
    UPDATE categories SET product_count = product_count - 1
    WHERE id = OLD.category_id;
  ELSIF TG_OP = 'UPDATE' AND OLD.category_id != NEW.category_id THEN
    UPDATE categories SET product_count = product_count - 1
    WHERE id = OLD.category_id;
    UPDATE categories SET product_count = product_count + 1
    WHERE id = NEW.category_id;
  END IF;
  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER products_category_count
  AFTER INSERT OR UPDATE OR DELETE ON products
  FOR EACH ROW EXECUTE FUNCTION update_category_count();

Stored Procedures / Functions

-- Function: returns a value
CREATE OR REPLACE FUNCTION get_user_stats(p_user_id uuid)
RETURNS TABLE (
  total_orders bigint,
  total_spent numeric,
  avg_order numeric,
  member_since timestamptz
) AS $$
BEGIN
  RETURN QUERY
  SELECT
    COUNT(o.id),
    COALESCE(SUM(o.total), 0),
    COALESCE(AVG(o.total), 0),
    u.created_at
  FROM users u
  LEFT JOIN orders o ON u.id = o.user_id
  WHERE u.id = p_user_id
  GROUP BY u.created_at;
END;
$$ LANGUAGE plpgsql;

async function getUserStats(userId) {
  const result = await pool.query(
    'SELECT * FROM get_user_stats($1)',
    [userId]
  );
  return result.rows[0];
}

Procedure (no return value, supports transactions)

CREATE OR REPLACE PROCEDURE transfer_funds(
  p_from_account uuid,
  p_to_account uuid,
  p_amount numeric
)
LANGUAGE plpgsql AS $$
BEGIN
  IF p_amount <= 0 THEN
    RAISE EXCEPTION 'Amount must be positive';
  END IF;

  UPDATE accounts SET balance = balance - p_amount
  WHERE id = p_from_account AND balance >= p_amount;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'Insufficient funds';
  END IF;

  UPDATE accounts SET balance = balance + p_amount
  WHERE id = p_to_account;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'Destination account not found';
  END IF;
END;
$$;

async function transferFunds(fromId, toId, amount) {
  try {
    await pool.query('CALL transfer_funds($1, $2, $3)', [fromId, toId, amount]);
    return { success: true };
  } catch (err) {
    return { success: false, error: err.message };
  }
}

When to Use Database Logic vs Application Logic

Use database logic (triggers/functions) for:
  - Data integrity (updated_at, audit logs)
  - Operations that MUST happen regardless of client
  - Performance-critical operations avoiding round trips
  - Aggregate maintenance (cached counts)

Use application logic for:
  - Business rules that change frequently
  - Complex logic requiring external services
  - Logic that needs unit testing
  - Cross-database portability

Exercises

  1. Build an Audit System

    Create a generic audit trigger that logs all INSERT, UPDATE, and DELETE operations on any table. Include the old data, new data, timestamp, and user who made the change. Apply it to 3 different tables.

    Hint

    Use TG_TABLE_NAME, TG_OP, to_jsonb(OLD), to_jsonb(NEW). Set the user via SET LOCAL app.current_user.

  2. Materialized View Dashboard

    Create a materialized view that pre-computes a dashboard summary: total users, total orders, revenue, average order value, top products. Write a refresh function and a Node.js endpoint that reads from it.

    Hint

    Use CONCURRENTLY for refresh to avoid locking. Create a unique index on the materialized view for concurrent refresh to work.

  3. Stored Procedure: Order Processing

    Write a stored procedure that processes an order: validates stock, decrements inventory, creates order and order_items, and returns the order. Handle out-of-stock errors. Call it from Node.js.

    Hint

    Use RAISE EXCEPTION for errors. Use SELECT FOR UPDATE to lock inventory rows during the transaction.

Common Problems & Solutions

1 Trigger causes infinite recursion
Cause: A trigger on table A updates table A, which fires the trigger again, creating an infinite loop.
Solution: Use a condition to prevent recursion: check if the relevant column actually changed (`IF OLD.col IS DISTINCT FROM NEW.col`). Or use `pg_trigger_depth()` to detect recursive calls.
2 Materialized view refresh blocks reads
Cause: Regular REFRESH MATERIALIZED VIEW takes an exclusive lock on the view.
Solution: Use `REFRESH MATERIALIZED VIEW CONCURRENTLY`. This requires a UNIQUE INDEX on the view and takes longer, but allows reads during refresh.
3 Stored procedure logic is hard to test and debug
Cause: PL/pgSQL doesn't have the same tooling as Node.js (no debugger, no unit testing framework).
Solution: Keep database functions simple and focused. Complex business logic belongs in the application. Use pgTAP for database testing if needed.
