Strawberry: Python-First GraphQL Library

Intuition

If traditional GraphQL libraries (like Graphene) are like writing HTML by hand, Strawberry is like using a modern framework that generates the HTML from your Python classes. You define types in idiomatic Python and Strawberry handles the GraphQL schema translation.

Strawberry: Python-First GraphQL Library

Strawberry is a modern Python GraphQL library that uses Python type hints and dataclasses for schema definition. It integrates natively with FastAPI, Django, and Flask.

FastAPI Integration

import strawberry
from strawberry.fastapi import GraphQLRouter
from fastapi import FastAPI

@strawberry.type
class Book:
    id: int
    title: str
    isbn: str
    price: float

# In-memory store for demonstration
books_db = [
    Book(id=1, title="Clean Code", isbn="978-0132350884", price=33.99),
    Book(id=2, title="Designing Data-Intensive Apps", isbn="978-1449373320", price=39.99),
]

@strawberry.input
class BookInput:
    title: str
    isbn: str
    price: float

@strawberry.type
class Query:
    @strawberry.field
    def books(self) -> list[Book]:
        return books_db

    @strawberry.field
    def book(self, id: int) -> Book | None:
        return next((b for b in books_db if b.id == id), None)

@strawberry.type
class Mutation:
    @strawberry.mutation
    def add_book(self, input: BookInput) -> Book:
        new_book = Book(id=len(books_db) + 1, **vars(input))
        books_db.append(new_book)
        return new_book

schema = strawberry.Schema(query=Query, mutation=Mutation)
app = FastAPI()
app.include_router(GraphQLRouter(schema), prefix="/graphql")

Custom Scalars and Enums

import enum
from datetime import datetime

@strawberry.enum
class Genre(enum.Enum):
    FICTION = "fiction"
    NONFICTION = "nonfiction"
    TECHNICAL = "technical"

@strawberry.type
class BookWithMeta:
    title: str
    genre: Genre
    published_at: datetime  # Strawberry handles datetime serialization

Common Mistakes

  • Forgetting to use @strawberry.input for mutation arguments
  • Not using async resolvers when calling async data sources
  • Returning raw dictionaries instead of typed Strawberry objects

Key Takeaways

  • Strawberry uses Python type hints for schema generation -- no SDL strings needed
  • @strawberry.input for mutation inputs, @strawberry.type for output types
  • Built-in GraphiQL IDE at /graphql for interactive exploration

Exercises

  1. Build a Strawberry CRUD API

    Create a complete Strawberry GraphQL API for a movie database with Movie and Director types. Include queries for listing and searching movies, and mutations for adding, updating, and deleting movies. Use @strawberry.input for mutation inputs.

    Hint

    Hint

    Create a class extending strawberry.permission.BasePermission. Override the has_permission method to check info.context['request'].headers.

Common Problems & Solutions

1 Strawberry schema does not include fields from parent class
Symptoms: ['Missing fields in GraphiQL schema explorer', 'Type errors when querying inherited fields']
Cause: Using regular Python inheritance instead of Strawberry's type system. Strawberry requires @strawberry.type on all types in the hierarchy.
Solution: Ensure all parent classes are also decorated with @strawberry.type, or use strawberry.experimental.pydantic to convert Pydantic models.
2 Async resolvers block the event loop
Cause: Calling synchronous database queries inside async resolvers without offloading to a thread.
Solution: Use async database drivers (asyncpg, aiosqlite) in async resolvers, or use sync resolvers which Strawberry runs in a thread pool automatically.
