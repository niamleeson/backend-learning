RBAC and ABAC: Role and Attribute-Based Access Control

Intuition

Think of RBAC like a building where each person gets a colored badge — red for executives, blue for engineers, green for visitors — and doors open based on badge color. ABAC adds smart locks that also check the time of day, whether you are carrying approved equipment, and whether the room belongs to your department. Together, the badge gets you through the front door quickly, while the smart lock handles the nuanced decisions at each room.

RBAC and ABAC: Role and Attribute-Based Access Control

Access control is the gatekeeper of every backend system. Once you have authenticated a user (confirmed who they are), you need authorization — determining what they are allowed to do. Two of the most widely adopted authorization models are Role-Based Access Control (RBAC) and Attribute-Based Access Control (ABAC). In this lesson we will build both from scratch, understand when to use each, and learn how to combine them for production-grade systems.

1. RBAC Fundamentals

Core Concepts

RBAC assigns permissions not to individual users but to roles, and users are then assigned one or more roles. This creates a clean separation:

Concept Description
User An authenticated identity (person, service account)
Role A named collection of permissions (e.g., admin, editor, viewer)
Permission A specific action on a resource (e.g., articles:write, users:delete)
Role Assignment The mapping of a user to one or more roles

The key insight is indirection: instead of managing thousands of individual user-permission mappings, you manage a handful of roles.

Role Hierarchies

In many systems, roles form a hierarchy. An admin inherits everything an editor can do, and an editor inherits everything a viewer can do:

admin
  └── editor
        └── viewer

This is sometimes called RBAC1 (hierarchical RBAC) in the NIST model. We will implement this pattern shortly.

Database Schema for RBAC

A solid relational schema is the backbone of any RBAC implementation. Here is a design that supports role hierarchies and many-to-many relationships:

-- Core tables
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(100) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE roles (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL,
    description TEXT,
    parent_role_id INTEGER REFERENCES roles(id)  -- hierarchy
);

CREATE TABLE permissions (
    id SERIAL PRIMARY KEY,
    resource VARCHAR(100) NOT NULL,   -- e.g., 'articles', 'users'
    action VARCHAR(50) NOT NULL,      -- e.g., 'read', 'write', 'delete'
    UNIQUE(resource, action)
);

-- Junction tables
CREATE TABLE user_roles (
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    role_id INTEGER REFERENCES roles(id) ON DELETE CASCADE,
    granted_at TIMESTAMP DEFAULT NOW(),
    PRIMARY KEY (user_id, role_id)
);

CREATE TABLE role_permissions (
    role_id INTEGER REFERENCES roles(id) ON DELETE CASCADE,
    permission_id INTEGER REFERENCES permissions(id) ON DELETE CASCADE,
    PRIMARY KEY (role_id, permission_id)
);

The parent_role_id column on the roles table lets you express hierarchy. When checking permissions, you walk up the chain.

Resolving Permissions with Hierarchy

A recursive query retrieves all permissions for a user, including inherited ones:

WITH RECURSIVE role_chain AS (
    -- Start with the user's directly assigned roles
    SELECT r.id, r.parent_role_id
    FROM roles r
    JOIN user_roles ur ON ur.role_id = r.id
    WHERE ur.user_id = :user_id

    UNION

    -- Walk down the hierarchy (children inherit parent permissions)
    SELECT r.id, r.parent_role_id
    FROM roles r
    JOIN role_chain rc ON r.id = rc.parent_role_id
)
SELECT DISTINCT p.resource, p.action
FROM role_chain rc
JOIN role_permissions rp ON rp.role_id = rc.id
JOIN permissions p ON p.id = rp.permission_id;

2. Implementing RBAC in Python

Flask Example with Decorators

Decorators provide a clean, declarative way to protect endpoints:

from functools import wraps
from flask import Flask, g, jsonify, request

app = Flask(__name__)


def get_user_permissions(user_id: str) -> set:
    """Fetch all permissions for a user (including inherited).
    In production this would query the database or a cache."""
    # Simulated permission store
    permission_store = {
        "alice": {"articles:read", "articles:write", "articles:delete", "users:read"},
        "bob": {"articles:read", "articles:write"},
        "charlie": {"articles:read"},
    }
    return permission_store.get(user_id, set())


def require_permission(resource: str, action: str):
    """Decorator that checks if the current user has the required permission."""
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            user_id = g.get("user_id")
            if not user_id:
                return jsonify({"error": "Authentication required"}), 401

            permissions = get_user_permissions(user_id)
            required = f"{resource}:{action}"

            if required not in permissions:
                return jsonify({
                    "error": "Forbidden",
                    "detail": f"Missing permission: {required}"
                }), 403

            return f(*args, **kwargs)
        return decorated_function
    return decorator


@app.before_request
def extract_user():
    # In production, this comes from a JWT or session
    g.user_id = request.headers.get("X-User-Id")


@app.route("/articles", methods=["GET"])
@require_permission("articles", "read")
def list_articles():
    return jsonify([{"id": 1, "title": "Hello World"}])


@app.route("/articles", methods=["POST"])
@require_permission("articles", "write")
def create_article():
    return jsonify({"id": 2, "title": "New Article"}), 201


@app.route("/articles/<int:article_id>", methods=["DELETE"])
@require_permission("articles", "delete")
def delete_article(article_id):
    return jsonify({"deleted": article_id})

FastAPI Example with Dependencies

FastAPI's dependency injection system offers an even more elegant approach:

from fastapi import FastAPI, Depends, HTTPException, Request, status
from typing import Set

app = FastAPI()


async def get_current_user(request: Request) -> dict:
    user_id = request.headers.get("X-User-Id")
    if not user_id:
        raise HTTPException(status_code=401, detail="Not authenticated")
    # In production: decode JWT, look up session, etc.
    return {"user_id": user_id}


async def get_permissions(user: dict = Depends(get_current_user)) -> Set[str]:
    """Resolve all permissions for the authenticated user."""
    # In production: query DB or Redis cache
    store = {
        "alice": {"articles:read", "articles:write", "articles:delete"},
        "bob": {"articles:read", "articles:write"},
    }
    return store.get(user["user_id"], set())


class PermissionChecker:
    """Callable dependency that checks for a specific permission."""

    def __init__(self, resource: str, action: str):
        self.required = f"{resource}:{action}"

    async def __call__(self, permissions: Set[str] = Depends(get_permissions)):
        if self.required not in permissions:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"Missing permission: {self.required}",
            )


@app.get("/articles", dependencies=[Depends(PermissionChecker("articles", "read"))])
async def list_articles():
    return [{"id": 1, "title": "Hello World"}]


@app.post("/articles", dependencies=[Depends(PermissionChecker("articles", "write"))])
async def create_article():
    return {"id": 2, "title": "New Article"}

The PermissionChecker class is a reusable dependency. It leverages FastAPI's Depends system so the permission resolution is cached within a single request.

3. ABAC: Going Beyond Roles

Why ABAC?

RBAC works well for coarse-grained control ("editors can edit articles"), but it struggles with fine-grained, context-dependent rules like:

  • "Users can only edit articles they authored."
  • "Documents can only be accessed during business hours."
  • "Managers can approve expenses under $10,000; directors can approve any amount."

Adding a new role for every such rule leads to role explosion. ABAC solves this by evaluating attributes at decision time.

The ABAC Model

ABAC policies evaluate four categories of attributes:

Category Examples
Subject user.role, user.department, user.clearance_level
Object (Resource) resource.owner, resource.classification, resource.created_at
Action action.type (read, write, delete)
Environment env.time, env.ip_address, env.request_count

A policy engine evaluates rules against these attributes and returns Allow or Deny.

Implementing a Simple ABAC Engine in Python

from dataclasses import dataclass, field
from typing import Any, Callable, List
from datetime import datetime


@dataclass
class AccessRequest:
    """Encapsulates all attributes for an access decision."""
    subject: dict      # {"id": "alice", "role": "editor", "department": "engineering"}
    resource: dict      # {"type": "article", "owner": "alice", "status": "draft"}
    action: str         # "read", "write", "delete"
    environment: dict = field(default_factory=lambda: {
        "time": datetime.utcnow(),
        "ip_address": "0.0.0.0",
    })


@dataclass
class Policy:
    """A single ABAC policy rule."""
    name: str
    description: str
    effect: str  # "allow" or "deny"
    condition: Callable[[AccessRequest], bool]
    priority: int = 0  # higher priority policies are evaluated first


class PolicyEngine:
    """Evaluates access requests against a set of policies.

    Strategy: deny-by-default. At least one 'allow' policy must match,
    and no 'deny' policy may match.
    """

    def __init__(self):
        self.policies: List[Policy] = []

    def add_policy(self, policy: Policy):
        self.policies.append(policy)
        self.policies.sort(key=lambda p: p.priority, reverse=True)

    def evaluate(self, request: AccessRequest) -> bool:
        allow_matched = False

        for policy in self.policies:
            try:
                if policy.condition(request):
                    if policy.effect == "deny":
                        # Explicit deny always wins
                        return False
                    elif policy.effect == "allow":
                        allow_matched = True
            except Exception:
                # If a policy evaluation fails, skip it (fail-closed)
                continue

        return allow_matched


# --- Define policies ---

engine = PolicyEngine()

# Policy 1: Editors can read and write articles
engine.add_policy(Policy(
    name="editor-article-access",
    description="Editors can read and write articles",
    effect="allow",
    condition=lambda req: (
        req.subject.get("role") == "editor"
        and req.resource.get("type") == "article"
        and req.action in ("read", "write")
    ),
))

# Policy 2: Users can only edit their own articles
engine.add_policy(Policy(
    name="owner-only-write",
    description="Deny writes to articles you do not own",
    effect="deny",
    priority=10,  # Higher priority than the allow rule
    condition=lambda req: (
        req.action == "write"
        and req.resource.get("type") == "article"
        and req.subject.get("id") != req.resource.get("owner")
        and req.subject.get("role") != "admin"  # admins bypass
    ),
))

# Policy 3: No access outside business hours (9-17 UTC)
engine.add_policy(Policy(
    name="business-hours-only",
    description="Deny all access outside business hours",
    effect="deny",
    priority=100,
    condition=lambda req: (
        req.environment["time"].hour < 9
        or req.environment["time"].hour >= 17
    ),
))


# --- Evaluate ---

request = AccessRequest(
    subject={"id": "alice", "role": "editor", "department": "content"},
    resource={"type": "article", "owner": "alice", "status": "draft"},
    action="write",
    environment={"time": datetime(2026, 3, 17, 14, 30), "ip_address": "10.0.0.5"},
)

print(engine.evaluate(request))  # True — alice is an editor writing her own article

request_other = AccessRequest(
    subject={"id": "bob", "role": "editor", "department": "content"},
    resource={"type": "article", "owner": "alice", "status": "draft"},
    action="write",
    environment={"time": datetime(2026, 3, 17, 14, 30), "ip_address": "10.0.0.6"},
)

print(engine.evaluate(request_other))  # False — bob is not the owner

This engine is deliberately simple. Production policy engines like Open Policy Agent (OPA) or AWS Cedar use declarative policy languages, but the fundamental model is the same.

4. Integrating ABAC into FastAPI

Let us wire the ABAC engine into a real endpoint:

from fastapi import FastAPI, Depends, HTTPException, Request
from datetime import datetime

app = FastAPI()

# Assume `engine` is the PolicyEngine defined above


async def get_current_user(request: Request) -> dict:
    # Simplified; in production decode a JWT
    return {
        "id": request.headers.get("X-User-Id", "anonymous"),
        "role": request.headers.get("X-User-Role", "viewer"),
        "department": request.headers.get("X-User-Dept", "unknown"),
    }


def abac_check(action: str):
    """Factory that returns a dependency performing an ABAC check."""
    async def dependency(
        request: Request,
        user: dict = Depends(get_current_user),
    ):
        # Build the resource dict from path params or DB lookup
        article_id = request.path_params.get("article_id")
        resource = await fetch_article_metadata(article_id) if article_id else {}
        resource.setdefault("type", "article")

        access_request = AccessRequest(
            subject=user,
            resource=resource,
            action=action,
            environment={
                "time": datetime.utcnow(),
                "ip_address": request.client.host if request.client else "0.0.0.0",
            },
        )

        if not engine.evaluate(access_request):
            raise HTTPException(status_code=403, detail="Access denied by policy")

    return dependency


@app.put("/articles/{article_id}", dependencies=[Depends(abac_check("write"))])
async def update_article(article_id: int):
    return {"updated": article_id}

The beauty of this approach is that endpoint code stays clean. All authorization logic lives in policies, and you can add, remove, or modify policies without touching handlers.

5. Combining RBAC and ABAC

In practice, most mature systems use a hybrid model:

  1. RBAC handles the coarse-grained layer: "Does this user have the editor role?" This is fast and cacheable.
  2. ABAC handles the fine-grained layer: "Is this editor the owner of this particular resource?" This requires runtime attribute resolution.

A clean architecture applies them sequentially:

async def hybrid_check(resource: str, action: str):
    """First check RBAC, then ABAC."""
    async def dependency(
        request: Request,
        user: dict = Depends(get_current_user),
        permissions: set = Depends(get_permissions),
    ):
        # Step 1: RBAC — fast, cached check
        required_perm = f"{resource}:{action}"
        if required_perm not in permissions:
            raise HTTPException(status_code=403, detail="Role lacks permission")

        # Step 2: ABAC — contextual check
        article_id = request.path_params.get("article_id")
        resource_meta = await fetch_article_metadata(article_id) if article_id else {}

        access_request = AccessRequest(
            subject=user,
            resource=resource_meta,
            action=action,
            environment={"time": datetime.utcnow()},
        )

        if not engine.evaluate(access_request):
            raise HTTPException(status_code=403, detail="Policy denied access")

    return dependency

This two-phase approach is both performant (RBAC short-circuits early) and expressive (ABAC handles the edge cases).

6. Real-World Scenarios

Admin Panels

Admin panels often use a straightforward RBAC model with three to five roles:

  • Super Admin: Full access, can manage other admins.
  • Admin: Can manage content and users, but cannot change system settings.
  • Moderator: Can review and approve/reject content.
  • Support: Read-only access to user data for troubleshooting.

The key design decision is whether roles are global or scoped. A global admin has power everywhere, while a scoped admin might only manage a specific section.

Multi-Tenant Applications

Multi-tenancy introduces a critical constraint: users must never access data from another tenant. This is where ABAC shines:

# Tenant isolation policy
engine.add_policy(Policy(
    name="tenant-isolation",
    description="Users can only access resources in their own tenant",
    effect="deny",
    priority=1000,  # Highest priority — always checked
    condition=lambda req: (
        req.subject.get("tenant_id") != req.resource.get("tenant_id")
    ),
))

This single policy replaces what would otherwise be hundreds of tenant-specific roles.

Database Schema for Multi-Tenant RBAC

Extend the earlier schema with tenant scoping:

CREATE TABLE tenant_user_roles (
    tenant_id UUID REFERENCES tenants(id),
    user_id UUID REFERENCES users(id),
    role_id INTEGER REFERENCES roles(id),
    PRIMARY KEY (tenant_id, user_id, role_id)
);

Now a user can be an admin in Tenant A but only a viewer in Tenant B.

7. Performance Considerations

Authorization is checked on every request, so performance matters:

  1. Cache RBAC lookups. A user's roles and permissions change infrequently. Cache them in Redis with a TTL of 5-15 minutes. Invalidate on role changes.
import redis
import json

r = redis.Redis()

def get_cached_permissions(user_id: str) -> set:
    cache_key = f"permissions:{user_id}"
    cached = r.get(cache_key)
    if cached:
        return set(json.loads(cached))

    permissions = fetch_permissions_from_db(user_id)
    r.setex(cache_key, 600, json.dumps(list(permissions)))  # 10 min TTL
    return permissions


def invalidate_permissions(user_id: str):
    r.delete(f"permissions:{user_id}")
  1. Minimize ABAC attribute lookups. Fetch resource metadata lazily — only when the RBAC check passes.
  2. Pre-compute where possible. If an ABAC policy depends on the count of records a user owns, maintain a counter rather than running COUNT(*) on every request.

8. Common Pitfalls

Pitfall 1: Checking Roles Instead of Permissions

Never write if user.role == "admin". Always check permissions. Roles are an organizational abstraction; permissions are the actual capabilities. If you hardcode role names into business logic, adding a new role requires code changes everywhere.

# Bad
if current_user.role == "admin":
    delete_article(article_id)

# Good
if "articles:delete" in current_user.permissions:
    delete_article(article_id)

Pitfall 2: Forgetting Deny-by-Default

Your system should deny access unless an explicit allow rule matches. A missing policy should mean no access, not full access. This is the principle of least privilege.

Pitfall 3: Mixing Authorization into Business Logic

Keep authorization checks separate from business logic. Use decorators, middleware, or dependency injection. When authorization rules are scattered throughout your codebase, they become impossible to audit.

Pitfall 4: Not Logging Access Decisions

Always log authorization decisions — especially denials. This is critical for security audits, debugging, and compliance:

import logging

logger = logging.getLogger("authorization")

def evaluate_with_logging(engine: PolicyEngine, request: AccessRequest) -> bool:
    result = engine.evaluate(request)
    logger.info(
        "access_decision",
        extra={
            "subject": request.subject.get("id"),
            "action": request.action,
            "resource_type": request.resource.get("type"),
            "resource_id": request.resource.get("id"),
            "decision": "allow" if result else "deny",
        },
    )
    return result

9. When to Use Which

Scenario Recommended Model
Simple app with a few fixed roles RBAC
Need to enforce ownership rules ABAC (or hybrid)
Multi-tenant SaaS Hybrid RBAC+ABAC
Compliance-heavy (healthcare, finance) ABAC with policy engine
Microservices with centralized auth RBAC with an external policy service

Start with RBAC. It covers 80% of use cases. Add ABAC policies only when you encounter rules that cannot be cleanly expressed as role-permission mappings. The hybrid approach gives you the best of both worlds: simplicity where possible, power where necessary.

Summary

RBAC and ABAC are complementary, not competing. RBAC provides a stable, cacheable, easy-to-understand permission model. ABAC adds the ability to make context-sensitive decisions based on any attribute of the user, resource, action, or environment. A well-architected system layers them: RBAC for the broad strokes, ABAC for the fine details. Combined with caching, logging, and deny-by-default policies, this hybrid model can scale from a simple admin panel to a complex multi-tenant platform.

Common Problems & Solutions

1 Users with the correct role are still getting 403 Forbidden errors
Symptoms: Permissions work for new users or after cache expiry but not immediately after role changes. Restarting the application or clearing Redis fixes the issue temporarily.
Cause: Role-permission mappings are cached, and the cache was not invalidated after a role's permissions were updated. The user's cached permission set is stale and missing the newly granted permission.
Solution: Invalidate the permission cache whenever role-permission mappings change. Use a pattern like `permissions:{user_id}` for per-user caches and invalidate all affected users when a role is modified. Alternatively, use short TTLs (e.g., 5 minutes) so stale entries expire quickly.

def update_role_permissions(role_id, new_permissions): # Update DB db.execute("DELETE FROM role_permissions WHERE role_id = %s", (role_id,)) for perm_id in new_permissions: db.execute("INSERT INTO role_permissions VALUES (%s, %s)", (role_id, perm_id))

# Invalidate caches for all users with this role
affected_users = db.fetch_all(
    "SELECT user_id FROM user_roles WHERE role_id = %s", (role_id,)
)
for user in affected_users:
    redis_client.delete(f"permissions:{user['user_id']}")
2 ABAC policies allow access when they should deny, or deny when they should allow
Cause: Policy evaluation order and conflict resolution are misconfigured. When multiple policies match — some allowing, some denying — the engine does not have a clear precedence strategy, leading to unpredictable results.
Solution: Adopt an explicit conflict resolution strategy. The most common and secure approach is 'deny overrides': if any deny policy matches, access is denied regardless of allow policies. Assign priorities to policies and ensure deny rules have higher priority. Always log which policies matched to aid debugging.

Ensure deny policies have higher priority

engine.add_policy(Policy( name="deny-outside-hours", effect="deny", priority=100, # High priority condition=lambda req: req.environment["time"].hour < 9, ))

engine.add_policy(Policy( name="allow-editors", effect="allow", priority=10, # Lower priority than deny condition=lambda req: req.subject.get("role") == "editor", ))

3 Authorization checks cause significant latency on every request
Symptoms: API response times increase by 50-200ms under load. Database monitoring shows high query volume on the roles, permissions, and user_roles tables.
Cause: Each request triggers multiple database queries: one to resolve the user's roles, another to resolve role permissions (possibly with recursive hierarchy traversal), and additional queries to fetch resource attributes for ABAC evaluation.
Solution: Layer your caching strategy. Cache RBAC results (roles and permissions) in Redis with a TTL since they change infrequently. For ABAC, only fetch resource attributes after RBAC passes (fail fast). Pre-compute commonly needed attributes. Consider embedding essential claims (roles, tenant_id) directly into the JWT to eliminate the user-roles lookup entirely.
