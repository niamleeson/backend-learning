Automated Deployment Strategies

Intuition

Deployment strategies are like updating a restaurant's menu. Blue-green is having two kitchens -- cook the new menu in kitchen B while kitchen A serves customers, then swap. Rolling update is changing one dish at a time. Canary is offering the new dish to 10% of tables first to see if they like it before rolling it out to everyone.

Automated Deployment Strategies

CI gets your code tested; CD gets it to production safely. Different strategies trade off between speed, risk, and complexity.

Deployment Strategies

Blue-Green: Two identical environments. Deploy to the inactive one, test it, switch traffic, keep the old one for rollback.

Rolling Update: Update instances one at a time, keeping the service available throughout.

Canary: Route a small percentage of traffic to the new version first.

GitHub Actions CD Pipeline

name: Deploy
on:
  push:
    branches: [main]

jobs:
  test:
    uses: ./.github/workflows/ci.yml

  deploy:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    environment: production
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to server
        env:
          SSH_KEY: ${{ secrets.DEPLOY_SSH_KEY }}
        run: |
          echo "$SSH_KEY" > key.pem && chmod 600 key.pem
          ssh -i key.pem deploy@server 'cd /opt/app && \
            git pull origin main && \
            docker compose up -d --build && \
            docker compose exec web python manage.py migrate'
      - name: Health check
        run: |
          sleep 10
          curl -f https://myapp.com/health || exit 1
      - name: Notify on failure
        if: failure()
        run: curl -X POST ${{ secrets.SLACK_WEBHOOK }} -d '{"text":"Deploy failed!"}'

Common Mistakes

  • Deploying directly to production without a staging environment
  • No rollback plan -- always keep the previous version ready
  • Not running database migrations before switching traffic

Key Takeaways

  • Start with blue-green and evolve to canary as needed
  • Always include health checks and automatic rollback
  • Use GitHub Environments with required reviewers for production

Exercises

  1. Build a CD Pipeline

    Create a GitHub Actions workflow that deploys to a staging server on pushes to develop and to production on pushes to main. Include health checks and Slack notifications for failures.

    Hint

    Use the environment key with protection rules. Use if conditions to select the target.

  2. Implement Blue-Green with Docker Compose

    Write a deployment script that runs two Docker Compose stacks (blue and green), deploys to the inactive one, runs smoke tests, and switches nginx upstream to the new stack.

    Hint

    Use two compose files with different port mappings and an nginx config that you reload with the new upstream.

  3. Rollback Script

    Create a rollback mechanism that keeps the last 3 deployed versions as tagged Docker images and can instantly rollback by switching the running container to a previous tag.

    Hint

    Tag images with git SHA or timestamps. Use docker tag and docker compose up with the previous tag.

Common Problems & Solutions

1 Deployment succeeds but application is broken
Symptoms: ['Users report errors after deploy', 'No alerts triggered despite broken app']
Cause: No post-deployment health checks or smoke tests.
Solution: Add a health check step after deployment: `curl -f https://myapp.com/health || rollback`. Implement a `/health` endpoint that verifies database connectivity and critical dependencies.
2 Database migration breaks rolling deployment
Symptoms: ['Some requests fail with column not found', '500 errors during rolling update']
Cause: The new code requires a schema change that is incompatible with the old code still running.
Solution: Use backward-compatible migrations: add new columns as nullable, deploy code that handles both schemas, then clean up. Never rename or remove columns in the same deploy.
