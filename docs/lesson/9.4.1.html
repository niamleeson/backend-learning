Read Replicas and Master-Slave Replication

Intuition

Replication is like having scribes copy a master document. Reads can be served from any copy, but all edits must go through the original. The scribes are always slightly behind, so recently edited pages might not be in their copies yet.

Read Replicas & Master-Slave Replication

Database replication creates copies of your data across multiple servers, improving read scalability, fault tolerance, and geographic distribution. Master-slave (primary-replica) replication is the most common pattern.

Core Concepts

  • Primary (Master): The single authoritative node that accepts all writes. Changes are recorded in a write-ahead log (WAL) and streamed to replicas.
  • Replicas (Slaves): Read-only copies that receive and apply changes from the primary. Applications route read queries to replicas to offload the primary.
  • Replication Lag: The delay between a write on the primary and its appearance on replicas. This creates a window of eventual consistency that applications must handle.
  • Synchronous vs Asynchronous Replication: Synchronous replication waits for at least one replica to confirm before acknowledging the write (stronger consistency, higher latency). Asynchronous replication acknowledges immediately (lower latency, risk of data loss on primary failure).
  • Failover: When the primary fails, a replica is promoted to primary. Automatic failover tools (Patroni, pg_auto_failover) detect failures and orchestrate promotion.

Key Takeaways

  1. Read replicas linearly scale read capacity but do not help with write-heavy workloads.
  2. Design your application to tolerate replication lag—use primary reads for consistency-critical operations.
  3. Test your failover process regularly; automatic failover is only reliable if it has been practiced.

Exercises

  1. Set Up PostgreSQL Streaming Replication

    Configure a PostgreSQL primary and one streaming replica using Docker. Write data to the primary and verify it appears on the replica. Measure replication lag using pg_stat_replication and pg_last_wal_replay_lsn().

    Hint

    On the primary, set wal_level=replica, max_wal_senders=3. Create a replication user. On the replica, use pg_basebackup to initialize, then configure primary_conninfo in postgresql.conf.

  2. Implement Read/Write Splitting in Django

    Configure Django's database router to send all writes to the primary and reads to a replica. Handle the case where a user writes data and immediately reads it back by routing post-write reads to the primary for a short window.

    Hint

    Create a custom DATABASE_ROUTERS class. Use middleware to set a flag after POST requests that forces reads from the primary for the next 5 seconds (store the timestamp in the session).

Common Problems & Solutions

1 Users see stale data immediately after making an update.
Cause: The application reads from a replica that has not yet received the latest write due to replication lag.
Solution: Implement 'read-your-writes' consistency: after a write, route subsequent reads from the same user to the primary for a short window (e.g., 5 seconds), then fall back to replicas.
2 Replication lag grows steadily over time and the replica falls increasingly behind.
Cause: The replica cannot keep up with the write rate, often due to insufficient I/O capacity or long-running queries on the replica blocking WAL replay.
Solution: Ensure replicas have adequate I/O throughput. Set `hot_standby_feedback = on` and `max_standby_streaming_delay` appropriately. Cancel long-running replica queries that block replay.
