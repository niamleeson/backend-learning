gRPC vs REST: When to Use Which

Intuition

REST is like sending letters -- universally understood, human-readable, works everywhere. gRPC is like a dedicated phone line between offices -- faster, more efficient, but requires both sides to have the right equipment. Most companies use letters for external communication and phone lines internally.

gRPC vs REST: When to Use Which

Choosing between gRPC and REST depends on your use case. Understanding the trade-offs helps you pick the right tool and even use them together in a polyglot architecture.

When to Use REST

# REST: public APIs, browser clients, simplicity
from fastapi import FastAPI

app = FastAPI()

@app.get("/products/{id}")
def get_product(id: int):
    """Easy to test: curl http://localhost:8000/products/1"""
    return {"id": id, "name": "Widget", "price": 9.99}

When to Use gRPC

# gRPC: internal microservices, high throughput, streaming
# service RecommendationService {
#   rpc GetRecommendations(UserRequest) returns (stream Product);
# }

class RecommendationServicer(recommendation_pb2_grpc.RecommendationServiceServicer):
    def GetRecommendations(self, request, context):
        """Streams product recommendations in real time"""
        for product in ml_model.recommend(request.user_id):
            yield recommendation_pb2.Product(
                id=product.id, name=product.name, score=product.score
            )

Hybrid Architecture

# Common pattern: REST gateway for external clients, gRPC internally
# Browser -> REST API Gateway -> gRPC Microservices

@app.get("/recommendations/{user_id}")
async def get_recommendations(user_id: int):
    """REST endpoint that proxies to gRPC service"""
    channel = grpc.aio.insecure_channel('recommendation-svc:50051')
    stub = recommendation_pb2_grpc.RecommendationServiceStub(channel)
    results = []
    async for product in stub.GetRecommendations(
        recommendation_pb2.UserRequest(user_id=user_id)
    ):
        results.append({"id": product.id, "name": product.name})
    return {"recommendations": results}

Common Mistakes

  • Using gRPC for public APIs consumed by web browsers without a proxy
  • Using REST for internal high-throughput microservice communication where latency matters
  • Choosing one exclusively when a hybrid approach is optimal

Key Takeaways

  • REST: public APIs, browser clients, human-readable, widely understood
  • gRPC: internal services, high performance, streaming, strong contracts
  • The best architectures often use both: REST at the edge, gRPC internally

Exercises

  1. Benchmark REST vs gRPC

    Implement the same API (list 1000 items) as both a REST/JSON endpoint and a gRPC service. Measure serialization size and response time for both. Document your findings.

    Hint

    Use sys.getsizeof() or len() on serialized data to compare sizes. Use time.perf_counter() for timing. Run 100 iterations and average the results.

  2. REST-to-gRPC Gateway

    Build a FastAPI REST endpoint that acts as a gateway to a gRPC backend service. The REST endpoint accepts JSON, translates it to a protobuf message, calls the gRPC service, and returns JSON.

    Hint

    Use the generated protobuf stubs to create request messages from the JSON body. Convert the gRPC response back to a dict using MessageToDict from google.protobuf.json_format.

Common Problems & Solutions

1 Cannot call gRPC service from browser JavaScript
Symptoms: ['CORS errors from browser', 'Protocol errors when calling gRPC from fetch()']
Cause: Browsers do not support HTTP/2 trailers, which gRPC requires for its protocol.
Solution: Use grpc-web with an Envoy proxy that translates between grpc-web and standard gRPC. Alternatively, expose a REST gateway that proxies to gRPC services.
2 JSON REST API too slow for inter-service communication
Cause: JSON serialization overhead and HTTP/1.1 connection limitations create latency in high-throughput service-to-service calls.
Solution: Switch internal service communication to gRPC with protobuf. Benchmark before and after. Keep REST for external-facing endpoints.
