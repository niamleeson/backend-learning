Intuition

Think of NestJS like a well-designed office building. The CLI is the architect who creates the blueprints and room layout. Modules are floors, controllers are reception desks that direct visitors, and services are the employees who do the actual work. NestJS Architecture and CLI Setup

Introduction

NestJS is a progressive Node.js framework for building efficient, reliable, and scalable server-side applications. Unlike Express, which is minimalist and unopinionated, NestJS provides a full application architecture out of the box -- inspired by Angular's module system and powered by TypeScript decorators.

If Express is a blank canvas, NestJS is a well-organized toolbox with designated compartments. It enforces structure through modules, controllers, providers, and decorators, making large codebases maintainable and testable.

Why NestJS?

Before diving in, let's understand what problems NestJS solves:

  1. Structure at scale -- Express apps tend to become spaghetti as they grow. NestJS enforces organization.
  2. Dependency Injection -- Built-in DI container makes testing and modularity first-class citizens.
  3. TypeScript-first -- Designed from the ground up for TypeScript, with full decorator support.
  4. Platform agnostic -- Can run on Express or Fastify under the hood.
  5. Ecosystem -- Official packages for WebSockets, GraphQL, microservices, CQRS, and more.

Architecture Overview

NestJS applications follow a layered architecture:

+--------------------------------------------------+
|                   Client Request                  |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|              Middleware (optional)                 |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|           Guards (auth / authorization)           |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|         Interceptors (before handler)             |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|      Pipes (validation / transformation)          |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|              Controller (route handler)           |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|         Service / Provider (business logic)       |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|         Interceptors (after handler)              |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|       Exception Filters (error handling)          |
+--------------------------------------------------+
          |
          v
+--------------------------------------------------+
|                  Client Response                  |
+--------------------------------------------------+

This is called the request pipeline. Every incoming request passes through these layers in order, giving you fine-grained control at each stage.

Installing the NestJS CLI

The NestJS CLI is the recommended way to scaffold and manage projects:

npm install -g @nestjs/cli
nest --version

Creating a New Project

nest new my-api

The CLI will ask you to choose a package manager (npm, yarn, or pnpm). Once complete, you get a fully structured project.

The Entry Point: main.ts

import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  await app.listen(3000);
}
bootstrap();

NestFactory.create() takes a root module and creates an application instance. Under the hood, NestJS sets up Express (by default) and wires together all modules, controllers, and providers.

Switching to Fastify

import { NestFactory } from '@nestjs/core';
import { FastifyAdapter, NestFastifyApplication } from '@nestjs/platform-fastify';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create<NestFastifyApplication>(
    AppModule,
    new FastifyAdapter(),
  );
  await app.listen(3000, '0.0.0.0');
}
bootstrap();

The Root Module: app.module.ts

import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';

@Module({
  imports: [],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}

The @Module() decorator takes a metadata object with: - imports -- Other modules this module depends on - controllers -- Controllers that belong to this module - providers -- Services/providers available within this module - exports -- Providers to make available to importing modules

The Controller: app.controller.ts

import { Controller, Get } from '@nestjs/common';
import { AppService } from './app.service';

@Controller()
export class AppController {
  constructor(private readonly appService: AppService) {}

  @Get()
  getHello(): string {
    return this.appService.getHello();
  }
}

Key points: - @Controller() marks a class as a controller - @Get() maps the method to a GET request at the base route - The AppService is injected via the constructor (dependency injection)

The Service: app.service.ts

import { Injectable } from '@nestjs/common';

@Injectable()
export class AppService {
  getHello(): string {
    return 'Hello World!';
  }
}

@Injectable() marks the class as a provider that can be managed by the NestJS DI container.

CLI Generators

The CLI can generate components for you:

# Generate a module
nest generate module users
# shorthand: nest g mo users

# Generate a controller
nest generate controller users
# shorthand: nest g co users

# Generate a service
nest generate service users
# shorthand: nest g s users

# Generate a full CRUD resource
nest generate resource products
# shorthand: nest g res products

The resource generator is especially powerful -- it creates a module, controller, service, DTOs, and entity file all at once, with full CRUD boilerplate.

nest-cli.json Configuration

{
  "$schema": "https://json.schemastore.org/nest-cli",
  "collection": "@nestjs/schematics",
  "sourceRoot": "src",
  "compilerOptions": {
    "deleteOutDir": true
  }
}

Key options: - sourceRoot -- Where your source code lives - compilerOptions.deleteOutDir -- Clean the dist folder before builds - generateOptions -- Default options for generators (e.g., always generate spec files)

Running the Application

npm run build
npm run start:prod

The start:dev command uses ts-node and watches for file changes, restarting automatically.

Understanding Decorators

NestJS relies heavily on TypeScript decorators. Decorators are functions that attach metadata to classes, methods, or parameters:

// Class decorator
@Controller('users')
export class UsersController {

  // Method decorator
  @Get(':id')
  // Parameter decorator
  findOne(@Param('id') id: string) {
    return `User #${id}`;
  }
}

Make sure experimentalDecorators and emitDecoratorMetadata are enabled in your tsconfig.json (the CLI sets these up for you).

Global Prefix and Versioning

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // All routes will be prefixed with /api
  app.setGlobalPrefix('api');

  // Enable URI versioning
  app.enableVersioning({
    type: VersioningType.URI,
  });

  await app.listen(3000);
}

CORS Configuration

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  app.enableCors({
    origin: ['http://localhost:3001'],
    methods: ['GET', 'POST', 'PUT', 'DELETE', 'PATCH'],
    credentials: true,
  });

  await app.listen(3000);
}

Common Pitfalls

  1. Forgetting to register a provider -- If you create a service but don't add it to a module's providers array, NestJS can't inject it.
  2. Circular module imports -- Module A imports Module B which imports Module A. Use forwardRef() to break cycles.
  3. Missing decorator metadata -- If emitDecoratorMetadata is false in tsconfig, DI will silently fail.
  4. Not using the CLI -- Manually creating files is error-prone. The CLI ensures correct naming, placement, and module registration.

Summary

NestJS provides an opinionated, well-structured framework for building Node.js backends. The CLI scaffolds projects with best practices baked in. The architecture revolves around modules, controllers, and providers, connected through dependency injection and organized through decorators.

Common Problems & Solutions

1 Nest can't resolve dependencies of the UsersService
Symptoms: Error: Nest can't resolve dependencies of the UsersService (?). Please make sure that the argument UsersRepository at index [0] is available in the UsersModule context.
Cause: The service depends on another provider that isn't available in the current module's context. Either the dependency wasn't added to the module's providers array, or it lives in another module that wasn't imported.
Solution: Check that all dependencies are registered as providers in the module. If the dependency is in another module, make sure that module exports the provider AND is imported by the current module.
2 Decorator metadata is not being emitted
Symptoms: Dependency injection silently fails or throws cryptic errors about undefined parameters.
Cause: The tsconfig.json is missing `experimentalDecorators` or `emitDecoratorMetadata` compiler options.
Solution: Ensure your tsconfig.json contains: `"experimentalDecorators": true` and `"emitDecoratorMetadata": true`. The NestJS CLI sets these automatically, but they can be lost during config changes.
3 Application starts but routes return 404
Cause: Controllers are not registered in any module, or the module containing the controller is not imported into the root AppModule.
Solution: Verify the controller is listed in the `controllers` array of its module, and that module is in the `imports` array of AppModule (or a module that is transitively imported).
