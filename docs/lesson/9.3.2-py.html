RabbitMQ: Exchanges, Queues, and Routing

Intuition

Think of RabbitMQ like a post office. Exchanges are sorting machines that read the address (routing key) on each letter (message) and place it in the correct mailbox (queue). Different exchange types represent different sorting rules.

RabbitMQ: Exchanges, Queues & Routing

RabbitMQ is a robust, open-source message broker implementing the AMQP protocol. Its exchange-queue-binding model provides flexible routing that supports a wide range of messaging patterns.

Core Concepts

  • Exchanges: Receive messages from producers and route them to queues based on bindings and routing keys. Types include direct (exact match), fanout (broadcast), topic (pattern match), and headers.
  • Queues: Ordered message buffers where consumers pick up work. Queues can be durable (survive broker restarts), exclusive (single consumer), or auto-delete.
  • Bindings: Rules linking exchanges to queues with a routing key pattern. A queue can bind to multiple exchanges, and an exchange can route to multiple queues.
  • Acknowledgments: Consumers ACK messages after processing. Unacknowledged messages are requeued on consumer failure, ensuring at-least-once delivery.
  • Prefetch Count: Controls how many unacknowledged messages a consumer can hold. Tuning this balances throughput against fair distribution across consumers.

Key Takeaways

  1. Choose the exchange type based on your routing needs: direct for point-to-point, fanout for pub/sub, topic for pattern-based routing.
  2. Always use durable queues and persistent messages for important workloads to survive broker restarts.
  3. Set appropriate prefetch counts to prevent fast consumers from starving slow ones.

Exercises

  1. Build a Topic-Based Routing System

    Create a RabbitMQ setup with a topic exchange. Publish messages with routing keys like order.created, order.shipped, payment.completed. Create queues bound with patterns: order.* for the order service and # for an audit logger.

    Hint

    Use pika in Python. Declare a topic exchange with channel.exchange_declare(exchange='events', exchange_type='topic'). Bind queues with patterns using channel.queue_bind(queue=q, exchange='events', routing_key='order.*').

  2. Implement Publisher Confirms

    Write a reliable publisher that uses RabbitMQ publisher confirms to ensure messages are durably stored before proceeding. Handle negative acknowledgments by retrying failed publishes.

    Hint

    Enable confirms with channel.confirm_delivery(). Use basic_publish in a try/except block catching pika.exceptions.UnroutableError. Track pending confirms with a dictionary keyed by delivery tag.

Common Problems & Solutions

1 Messages are lost when RabbitMQ is restarted.
Cause: Queues and/or messages are not configured as durable/persistent.
Solution: Declare queues with `durable=True` and publish messages with `delivery_mode=2` (persistent). This writes messages to disk, surviving broker restarts.
2 One consumer is overwhelmed while others are idle.
Cause: The prefetch count is too high, causing RabbitMQ to dispatch a large batch of messages to a single consumer before it finishes processing them.
Solution: Set `channel.basic_qos(prefetch_count=1)` or a small number to ensure fair dispatch. Messages are only sent to consumers that have acknowledged previous ones.
