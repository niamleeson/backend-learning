Protocol Buffers and Service Definitions

Intuition

A .proto file is like a blueprint shared between a manufacturer and an assembly line. Both sides agree on exactly what a 'User' looks like -- its fields, their types, and their order. The blueprint generates custom packaging machines (generated code) that can pack and unpack data at maximum speed.

Protocol Buffers and Service Definitions

Protocol Buffers (protobuf) is Google's language-neutral, platform-neutral serialization format. Combined with gRPC, it defines both the data structures and service interfaces for remote procedure calls.

Defining Messages

# user.proto
syntax = "proto3";

package userservice;

message User {
  int32 id = 1;
  string name = 2;
  string email = 3;
  UserStatus status = 4;
  repeated string roles = 5;
}

enum UserStatus {
  ACTIVE = 0;
  INACTIVE = 1;
  SUSPENDED = 2;
}

message GetUserRequest {
  int32 id = 1;
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
}

message UserList {
  repeated User users = 1;
  int32 total = 2;
}

Defining Services

# user_service.proto
service UserService {
  rpc GetUser (GetUserRequest) returns (User);
  rpc CreateUser (CreateUserRequest) returns (User);
  rpc ListUsers (ListUsersRequest) returns (UserList);
  rpc DeleteUser (GetUserRequest) returns (Empty);
}

message ListUsersRequest {
  int32 page = 1;
  int32 page_size = 2;
}

message Empty {}

Generate Python Code

# Using generated code
import user_pb2

user = user_pb2.User(
    id=1,
    name="Alice",
    email="alice@example.com",
    status=user_pb2.ACTIVE,
    roles=["admin", "editor"]
)
serialised = user.SerializeToString()  # Binary format, very compact

Common Mistakes

  • Reusing field numbers after removing a field (use reserved instead)
  • Using proto2 syntax when proto3 is standard for new projects
  • Not versioning .proto files alongside application code

Key Takeaways

  • Protobuf is 3-10x smaller and faster than JSON for serialization
  • Field numbers are permanent identifiers -- never reuse them
  • The .proto file is the single source of truth for your API contract

Exercises

  1. Design a Proto Schema

    Write a .proto file for an e-commerce order system with Order, OrderItem, and Address messages. Include an OrderService with RPCs for CreateOrder, GetOrder, and ListOrders. Use enums for order status.

    Hint

    Use nested messages for OrderItem inside Order. Use 'repeated' for lists. Define an enum OrderStatus with values like PENDING, CONFIRMED, SHIPPED, DELIVERED.

  2. Proto Evolution Exercise

    Given an existing User proto, add new fields (phone_number, created_at) while maintaining backward compatibility. Then remove an old field safely using 'reserved'. Demonstrate that old serialized data still deserializes correctly.

    Hint

    Add new fields with new field numbers (never reuse old numbers). Use 'reserved 4;' to prevent reuse of removed field numbers. Use google.protobuf.Timestamp for dates.

Common Problems & Solutions

1 Deserialization fails after schema update
Symptoms: ['Corrupted data after deployment', 'Type mismatch errors in proto deserialization']
Cause: A field number was reused for a different field type after the old field was removed.
Solution: Never reuse field numbers. Use the 'reserved' keyword to mark removed field numbers and names: reserved 4, 7; reserved "old_field";
2 Generated Python code not found at import time
Cause: The protoc compilation step was not run, or output files are in the wrong directory.
Solution: Add protoc generation to your build process. Use a Makefile or setup.py to run: python -m grpc_tools.protoc -I. --python_out=./generated --grpc_python_out=./generated *.proto
