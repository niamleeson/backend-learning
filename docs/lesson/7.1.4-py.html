Shell Scripting Basics for Backend Devs

Intuition

A shell script is like a recipe card for your kitchen (server). Instead of remembering every step each time you cook (deploy), you write it down once. set -euo pipefail is your safety checklist -- stop immediately if any step goes wrong rather than serving a half-cooked dish.

Shell Scripting Basics for Backend Devs

Shell scripts automate repetitive tasks: deployments, database migrations, health checks, and environment setup.

Script Fundamentals

#!/usr/bin/env bash
set -euo pipefail  # Exit on error, undefined vars, pipe failures

APP_NAME="myapp"
DEPLOY_DIR="/opt/${APP_NAME}"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
echo "Deploying ${APP_NAME} at ${TIMESTAMP}"

Conditionals and Loops

if command -v docker &> /dev/null; then
    echo "Docker is installed"
else
    echo "Docker not found. Installing..."
    curl -fsSL https://get.docker.com | sh
fi

SERVERS=("web1" "web2" "web3")
for server in "${SERVERS[@]}"; do
    ssh deploy@"${server}" "cd /opt/app && git pull && sudo systemctl restart app"
done

A Simple Deploy Script

#!/usr/bin/env bash
set -euo pipefail
APP_DIR="/opt/myapp"

[ -d "$APP_DIR" ] || { echo "ERROR: App dir not found" >&2; exit 1; }
tar -czf "/opt/backups/backup_$(date +%F).tar.gz" "$APP_DIR"

cd "$APP_DIR"
git pull origin main
source venv/bin/activate
pip install -r requirements.txt
sudo systemctl restart myapp
echo "Deployment complete!"

Functions and Error Handling

log() { echo "[$(date '+%H:%M:%S')] $*"; }
die() { echo "ERROR: $*" >&2; exit 1; }

check_health() {
    local status
    status=$(curl -s -o /dev/null -w '%{http_code}' "$1")
    [ "$status" -eq 200 ] || die "Health check failed: HTTP $status"
}

Common Mistakes

  • Forgetting set -euo pipefail, causing scripts to silently continue after errors
  • Not quoting variables: $VAR vs "$VAR" -- unquoted vars break on spaces

Key Takeaways

  • Always start scripts with set -euo pipefail
  • Quote all variable expansions to handle spaces in paths
  • Use functions to keep scripts organized and reusable

Exercises

  1. Environment Setup Script

    Write a bash script that sets up a Python development environment: checks for Python 3.10+, creates a virtual environment, installs dependencies from requirements.txt, copies .env.example to .env if .env doesn't exist, and runs migrations.

    Hint

    Use python3 --version, compare versions, and check file existence with [ -f .env ].

  2. Health Check & Alerting Script

    Write a script that pings a list of URLs, checks for HTTP 200, and logs failures with timestamps. Make it suitable for running via cron every minute.

    Hint

    Use curl -s -o /dev/null -w '%{http_code}' and loop through an array of URLs.

  3. Rolling Deployment Script

    Write a deploy script that deploys to 3 servers one at a time, runs a health check after each, and rolls back if any server fails.

    Hint

    Use a for loop, ssh for remote commands, a health check function, and trap for cleanup.

Common Problems & Solutions

1 Script works interactively but fails in cron or CI
Symptoms: ['command not found errors in cron logs', 'Scripts pass locally but fail in GitHub Actions']
Cause: The script depends on environment variables or PATH entries that exist in an interactive shell but not in cron/CI environments.
Solution: Use absolute paths for all commands. Source required env files explicitly. Add `export PATH=/usr/local/bin:$PATH` at the top.
2 Script breaks on filenames with spaces
Cause: Unquoted variable expansions undergo word splitting.
Solution: Always double-quote variables: use `"$file"` not `$file`. Use `"${array[@]}"` for arrays.

Wrong

for f in $files; do

Right

for f in "${files[@]}"; do
