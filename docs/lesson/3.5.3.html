Relationships: One-to-Many, Many-to-Many, Back Populates

Intuition

Relationships are bridges between models. Once defined, you can walk from an author to their books or a student to their courses without writing SQL.

Relationships: One-to-Many, Many-to-Many

SQLAlchemy relationships define how models reference each other, enabling navigation between related objects without manual joins.

from sqlalchemy import ForeignKey, Table, Column, Integer
from sqlalchemy.orm import Mapped, mapped_column, relationship

class Author(Base):
    __tablename__ = 'authors'
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100))
    books: Mapped[list['Book']] = relationship(back_populates='author')

class Book(Base):
    __tablename__ = 'books'
    id: Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str] = mapped_column(String(200))
    author_id: Mapped[int] = mapped_column(ForeignKey('authors.id'))
    author: Mapped['Author'] = relationship(back_populates='books')

# Many-to-Many
student_course = Table('student_course', Base.metadata,
    Column('student_id', Integer, ForeignKey('students.id'), primary_key=True),
    Column('course_id', Integer, ForeignKey('courses.id'), primary_key=True)
)

class Student(Base):
    __tablename__ = 'students'
    id: Mapped[int] = mapped_column(primary_key=True)
    courses: Mapped[list['Course']] = relationship(
        secondary=student_course, back_populates='students'
    )

Key Takeaways: - back_populates links both sides so changes propagate bidirectionally. - One-to-many uses a ForeignKey on the child. Many-to-many uses a secondary association table. - cascade='all, delete-orphan' on the parent auto-deletes children when the parent is removed.

Exercises

  1. Association Object Pattern

    Model a student-course relationship where enrollment has extra data (grade, enrolled_at) using an association object instead of a plain junction table.

    Hint

    Create an Enrollment model with student_id, course_id FKs, plus grade and enrolled_at columns.

  2. Self-Referential Relationship

    Create an Employee model with a manager_id that references the same table, and a relationship to access direct reports.

    Hint

    manager_id ForeignKey('employees.id'), reports = relationship('Employee', back_populates='manager').

Common Problems & Solutions

1 Accessing a relationship attribute triggers unexpected SQL queries.
Cause: Default lazy loading issues a SELECT per access, causing N+1 query problems.
Solution: Use joinedload() or selectinload() in your query options, or set lazy='selectin' on the relationship.
2 Deleting a parent fails due to foreign key constraint.
Cause: Child rows still reference the parent and no cascade is configured.
Solution: Add cascade='all, delete-orphan' to the relationship or ON DELETE CASCADE on the foreign key.
