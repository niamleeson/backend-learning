Partitioning and Query Optimization at Scale

Intuition

Partitioning is like organizing a filing cabinet by year. When someone asks for a 2024 invoice, you open only the 2024 drawer instead of searching through every drawer. The documents are still all in one cabinet logically, but physically separated for efficiency.

Partitioning & Query Optimization at Scale

As tables grow to millions or billions of rows, query performance degrades without proper partitioning and optimization. Table partitioning divides large tables into smaller, manageable pieces while maintaining a unified interface.

Core Concepts

  • Range Partitioning: Divide data by ranges of a column (e.g., monthly partitions for time-series data). Excellent for time-based queries and data lifecycle management.
  • List Partitioning: Divide data by discrete values (e.g., by region or status). Useful when queries consistently filter by a categorical column.
  • Hash Partitioning: Distribute rows evenly across partitions using a hash function. Prevents hot spots but does not support range queries on the partition key.
  • Partition Pruning: The query planner skips irrelevant partitions, dramatically reducing I/O. Effective only when the partition key appears in the WHERE clause.
  • Index Strategy at Scale: Partial indexes, covering indexes, and BRIN indexes reduce storage and improve performance on large tables. EXPLAIN ANALYZE is your primary tool for optimization.

Key Takeaways

  1. Partition tables when they exceed tens of millions of rows and queries have natural partition key filters.
  2. Partition pruning only works when the partition key is in the query's WHERE clause—design queries accordingly.
  3. Combine partitioning with proper indexing; partitioning alone does not replace indexes.

Exercises

  1. Implement Range Partitioning in PostgreSQL

    Create a logs table partitioned by month using PostgreSQL declarative partitioning. Insert 10 million rows spanning 12 months. Compare query performance with EXPLAIN ANALYZE for date-filtered queries on the partitioned table versus an equivalent non-partitioned table.

    Hint

    Use CREATE TABLE logs (...) PARTITION BY RANGE (created_at) then CREATE TABLE logs_2024_01 PARTITION OF logs FOR VALUES FROM ('2024-01-01') TO ('2024-02-01'). Check for 'Partition Pruning' in the EXPLAIN output.

  2. Optimize Slow Queries with EXPLAIN ANALYZE

    Given a slow query on a large table, use EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) to identify the bottleneck. Create appropriate indexes (B-tree, GIN, or BRIN), adjust the query plan, and achieve at least a 10x improvement.

    Hint

    Look for sequential scans on large tables, nested loop joins with high row counts, and sort operations. Add indexes on columns in WHERE, JOIN, and ORDER BY clauses. Consider partial indexes if the query filters a small subset.

Common Problems & Solutions

1 Queries on a partitioned table are not faster and partition pruning is not happening.
Cause: The WHERE clause does not reference the partition key, or uses a function on the partition key that prevents pruning (e.g., `WHERE EXTRACT(month FROM created_at) = 1`).
Solution: Ensure queries filter directly on the partition key column with literal values or parameters. Use `WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01'` instead of functions on the column.
2 INSERTs become slow as the number of partitions grows to hundreds.
Cause: PostgreSQL's partition routing has overhead proportional to the number of partitions. Very large partition counts degrade planning time.
Solution: Keep partition count under a few hundred. Use coarser partitions (monthly instead of daily) or implement a two-level partitioning scheme. Detach old partitions and archive them.
