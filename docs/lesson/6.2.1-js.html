Dockerfile: Building Node.js Application Images

Intuition

A Dockerfile is like a recipe for creating a sealed, portable lunchbox. You list every ingredient (dependencies) and step (build commands) needed. Multi-stage builds are like using a full kitchen to cook (build stage) but only packing the finished dish into the lunchbox (production stage) -- you do not need the oven and pots in the lunchbox.

Dockerfile: Building Node.js Application Images

Why Docker for Node.js?

Docker packages your application with its exact runtime environment - Node.js version, system dependencies, npm packages - into a portable image. This eliminates "works on my machine" problems and ensures identical behavior across development, CI, staging, and production.

Image Layers

┌─────────────────────────────┐
│  COPY . .                   │  Layer 5 (changes often)
├─────────────────────────────┤
│  RUN npm ci                 │  Layer 4 (changes with package.json)
├─────────────────────────────┤
│  COPY package*.json ./      │  Layer 3
├─────────────────────────────┤
│  WORKDIR /app               │  Layer 2
├─────────────────────────────┤
│  node:20-alpine             │  Layer 1 (base, rarely changes)
└─────────────────────────────┘

Each layer is cached. If a layer hasn't changed,
Docker reuses the cached version.

Writing Dockerfiles for Node.js

Basic Dockerfile

# Dockerfile
FROM node:20-alpine

# Create app directory
WORKDIR /app

# Copy package files first (better caching)
COPY package.json package-lock.json ./

# Install dependencies
RUN npm ci

# Copy application source
COPY . .

# Build TypeScript (if applicable)
RUN npm run build

# Expose port (documentation only)
EXPOSE 3000

# Start the application
CMD ["node", "dist/server.js"]

Production-Optimized Dockerfile

# ============================
# Stage 1: Build
# ============================
FROM node:20-alpine AS builder

WORKDIR /app

# Copy dependency files
COPY package.json package-lock.json ./

# Install ALL dependencies (including devDependencies for building)
RUN npm ci

# Copy source code
COPY tsconfig.json ./
COPY src/ ./src/
COPY prisma/ ./prisma/

# Generate Prisma client
RUN npx prisma generate

# Build TypeScript
RUN npm run build

# Remove dev dependencies
RUN npm ci --production && npm cache clean --force

# ============================
# Stage 2: Production
# ============================
FROM node:20-alpine AS production

# Add non-root user
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup

# Install security updates
RUN apk update && apk upgrade && rm -rf /var/cache/apk/*

# Install dumb-init for proper signal handling
RUN apk add --no-cache dumb-init

WORKDIR /app

# Copy built artifacts from builder stage
COPY --from=builder --chown=appuser:appgroup /app/dist ./dist
COPY --from=builder --chown=appuser:appgroup /app/node_modules ./node_modules
COPY --from=builder --chown=appuser:appgroup /app/package.json ./
COPY --from=builder --chown=appuser:appgroup /app/prisma ./prisma

# Set environment
ENV NODE_ENV=production
ENV PORT=3000

# Switch to non-root user
USER appuser

# Expose port
EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# Use dumb-init as PID 1 for proper signal handling
ENTRYPOINT ["dumb-init", "--"]

# Start the application
CMD ["node", "dist/server.js"]

Key Dockerfile Instructions

FROM

# Use specific version (always pin versions in production)
FROM node:20.11.0-alpine3.19

# Bad: uses latest, unpredictable
FROM node:latest

# Alpine is smaller (~50MB vs ~350MB for debian-based)
FROM node:20-alpine

# Slim is a middle ground
FROM node:20-slim

COPY vs ADD

# COPY - simple file/directory copy (preferred)
COPY package.json ./
COPY src/ ./src/

# ADD - has extra features (auto-extract archives, URLs)
ADD archive.tar.gz /app/  # Auto-extracts
# Prefer COPY unless you need ADD's features

RUN - Reducing Layers

# Bad: creates 3 layers
RUN apk update
RUN apk add curl
RUN rm -rf /var/cache/apk/*

# Good: single layer, smaller image
RUN apk update && \
    apk add --no-cache curl && \
    rm -rf /var/cache/apk/*

ENV vs ARG

# ARG - only available during build
ARG NODE_VERSION=20
FROM node:${NODE_VERSION}-alpine

ARG BUILD_DATE
LABEL build-date=$BUILD_DATE

# ENV - available during build AND in running container
ENV NODE_ENV=production
ENV PORT=3000

ENTRYPOINT vs CMD

# CMD - default command, easily overridden
CMD ["node", "server.js"]
# docker run myapp                    -> node server.js
# docker run myapp node repl          -> node repl (overridden)

# ENTRYPOINT - always runs, CMD becomes arguments
ENTRYPOINT ["node"]
CMD ["server.js"]
# docker run myapp                    -> node server.js
# docker run myapp script.js          -> node script.js

# Best practice for production:
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "dist/server.js"]

Essential Docker Commands

# Build an image
docker build -t myapp:latest .
docker build -t myapp:1.0.0 -f Dockerfile.production .

# Run a container
docker run -d -p 3000:3000 --name myapp myapp:latest
docker run -d -p 3000:3000 --env-file .env --name myapp myapp:latest
docker run --rm -it myapp:latest sh  # Interactive shell

# Container management
docker ps                   # Running containers
docker ps -a                # All containers (including stopped)
docker logs myapp           # View logs
docker logs -f myapp        # Follow logs
docker exec -it myapp sh    # Shell into running container
docker stop myapp           # Graceful stop (SIGTERM)
docker kill myapp           # Force stop (SIGKILL)
docker rm myapp             # Remove container

# Image management
docker images               # List images
docker rmi myapp:latest     # Remove image
docker system prune -a      # Remove ALL unused images and containers

# Debugging
docker inspect myapp        # Container details
docker stats                # Resource usage
docker top myapp            # Running processes

Signal Handling with dumb-init

Node.js as PID 1 in a container does not handle signals properly by default. dumb-init fixes this:

Without dumb-init:
docker stop myapp
  └─> SIGTERM sent to PID 1 (node)
      └─> Node.js doesn't forward to children
          └─> After 10s, Docker sends SIGKILL (ungraceful)

With dumb-init:
docker stop myapp
  └─> SIGTERM sent to PID 1 (dumb-init)
      └─> dumb-init forwards SIGTERM to node (PID 2)
          └─> Node.js runs graceful shutdown handlers
              └─> Clean exit

Common Pitfalls

  1. Running as root - Always create and use a non-root user. Root in container = root on host (in many configurations).
  2. Using npm install instead of npm ci - npm ci gives deterministic builds from lockfile. npm install may update packages.
  3. Not leveraging layer caching - Copy package.json BEFORE copying source code. Dependencies change less often than code.
  4. Including dev dependencies in production - Use multi-stage builds or npm ci --production.
  5. Not using .dockerignore - Without it, node_modules and .git are copied into the image, bloating it.
  6. Not pinning base image versions - node:latest today may be different tomorrow.

Exercises

  1. Build a Production Dockerfile

    Create a multi-stage Dockerfile for a TypeScript Express application with Prisma. The build stage should compile TypeScript and generate the Prisma client. The production stage should be based on Alpine, run as a non-root user, include a health check, and use dumb-init.

    Hint

    Use 'FROM node:20-alpine AS builder' for the build stage and a fresh 'FROM node:20-alpine' for production. Copy only dist/, node_modules/, and prisma/ from the builder.

  2. Optimize Image Size

    Start with a basic Dockerfile that produces a ~1GB image (using node:20, copying everything, including devDependencies). Optimize it step by step: switch to Alpine, add .dockerignore, use multi-stage build, and production-only dependencies. Document the size at each step.

    Hint

    Use 'docker images' to check sizes after each optimization. Alpine alone should cut the base by ~300MB. Multi-stage should remove build tools and devDependencies.

  3. Docker Layer Caching Experiment

    Build a Dockerfile two ways: (1) COPY everything then npm install, (2) COPY package*.json first, npm ci, then COPY the rest. Make a change to a source file and rebuild both. Measure and compare build times to demonstrate caching benefits.

    Hint

    Use 'time docker build .' to measure. With proper caching, only the COPY . . and later layers rebuild when source changes.

Common Problems & Solutions

1 Docker build fails with 'npm ERR! code ERESOLVE' or dependency errors
Cause: The package-lock.json is out of sync with package.json, or there are platform-specific dependencies (e.g., bcrypt native modules).
Solution: Delete package-lock.json and node_modules locally, run `npm install` to regenerate the lockfile, then rebuild. For native modules, ensure the Alpine image has build tools: `RUN apk add --no-cache python3 make g++` or use pre-built alternatives (bcryptjs instead of bcrypt).
2 Container runs but cannot connect to database
Cause: localhost inside the container refers to the container itself, not the host machine. The database is on the host or another container.
Solution: Use Docker networking. If database is in another container, use the container name as hostname. If on host, use `host.docker.internal` (Docker Desktop) or `--network host` (Linux). Best practice: use Docker Compose with a shared network.
3 SIGTERM not reaching Node.js process, container takes 10 seconds to stop
Cause: Node.js as PID 1 does not handle signals properly. Or CMD is in shell form (`CMD node server.js`) which wraps in /bin/sh.
Solution: Use exec form for CMD: `CMD ["node", "server.js"]`. Better: install and use dumb-init as ENTRYPOINT. Also ensure your Node.js code handles SIGTERM: `process.on('SIGTERM', () => server.close())`.
