EXPLAIN ANALYZE and Query Optimization

Intuition

EXPLAIN ANALYZE is like having X-ray vision into the database's decision-making process. Just as a doctor uses an X-ray to see what's happening inside the body before prescribing treatment, you use EXPLAIN ANALYZE to see what's happening inside a query before optimizing it. Guessing at the problem without looking at the plan is like prescribing medicine without a diagnosis.

EXPLAIN ANALYZE and Query Optimization

Introduction

Writing correct SQL is one thing. Writing fast SQL is another. PostgreSQL's EXPLAIN and EXPLAIN ANALYZE commands reveal exactly how the database executes your query — showing the plan it chose, the costs it estimated, and (with ANALYZE) the actual time each step took. This is your primary tool for diagnosing and fixing slow queries.

EXPLAIN vs EXPLAIN ANALYZE

-- EXPLAIN: shows the planned execution without running the query
EXPLAIN SELECT * FROM users WHERE email = 'alice@example.com';

-- EXPLAIN ANALYZE: actually runs the query and shows real timing
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'alice@example.com';

-- EXPLAIN with more detail
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT * FROM users WHERE email = 'alice@example.com';

Warning: EXPLAIN ANALYZE actually executes the query. For DELETE or UPDATE, wrap in a transaction and rollback:

BEGIN;
EXPLAIN ANALYZE DELETE FROM orders WHERE created_at < '2020-01-01';
ROLLBACK;

Reading an Execution Plan

async function explainQuery(sql, params = []) {
  const result = await pool.query(
    `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) ${sql}`,
    params
  );
  return result.rows[0]['QUERY PLAN'][0];
}

const plan = await explainQuery(
  'SELECT * FROM orders WHERE user_id = $1 ORDER BY created_at DESC LIMIT 10',
  [42]
);
console.log(JSON.stringify(plan, null, 2));

Key Plan Nodes

Seq Scan           -- Reads every row in the table (full table scan)
Index Scan         -- Uses an index to find rows, then fetches from table
Index Only Scan    -- Gets all data from the index itself (fastest)
Bitmap Index Scan  -- Builds a bitmap of matching rows, then fetches
Nested Loop        -- For each row in outer, scan inner (small datasets)
Hash Join          -- Build hash table from one side, probe with other
Merge Join         -- Sort both sides, then merge (pre-sorted data)
Sort               -- In-memory or on-disk sorting
Aggregate          -- GROUP BY / COUNT / SUM processing

Reading Plan Output

Limit  (cost=0.43..12.50 rows=10 width=64) (actual time=0.05..0.12 rows=10 loops=1)
  ->  Index Scan using idx_orders_user_date on orders
        (cost=0.43..156.20 rows=128 width=64) (actual time=0.04..0.10 rows=10 loops=1)
        Index Cond: (user_id = 42)
        Buffers: shared hit=4

Planning Time: 0.15 ms
Execution Time: 0.18 ms

How to read this:

cost=0.43..12.50    Estimated startup..total cost (arbitrary units)
rows=10             Estimated rows returned
width=64            Estimated average row width in bytes
actual time=0.05..0.12  Real time in milliseconds (start..end)
rows=10             Actual rows returned
loops=1             Number of times this node executed
Buffers: shared hit=4   Pages read from cache (no disk I/O)

Common Performance Problems and Fixes

1. Sequential Scan on Large Table

-- Problem: Seq Scan on 10M row table
EXPLAIN ANALYZE
SELECT * FROM logs WHERE user_id = 42;

-- Output: Seq Scan on logs (actual time=0.01..4521.32 rows=15 loops=1)
--         Filter: (user_id = 42)
--         Rows Removed by Filter: 9999985

-- Fix: Add an index
CREATE INDEX idx_logs_user_id ON logs (user_id);

-- After: Index Scan using idx_logs_user_id (actual time=0.03..0.12 rows=15 loops=1)

2. Index Not Used Due to Function

-- Problem: index on email, but using LOWER()
EXPLAIN ANALYZE
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';
-- Seq Scan! Index on email doesn't help because LOWER() transforms the value

-- Fix: functional index
CREATE INDEX idx_users_email_lower ON users (LOWER(email));

3. N+1 Query Pattern

// TERRIBLE: N+1 queries
const users = await pool.query('SELECT * FROM users LIMIT 100');
for (const user of users.rows) {
  const orders = await pool.query(
    'SELECT * FROM orders WHERE user_id = $1', [user.id]
  );
  // 101 queries total!
}

// FIXED: Single JOIN query
const result = await pool.query(`
  SELECT u.*, json_agg(o.*) AS orders
  FROM users u
  LEFT JOIN orders o ON u.id = o.user_id
  GROUP BY u.id
  LIMIT 100
`);
// 1 query!

4. Missing LIMIT on Large Result Sets

// Fetching 1M rows into Node.js memory
const result = await pool.query('SELECT * FROM huge_table');
// Crashes with out-of-memory

// Fix: Use cursors for large result sets
const client = await pool.connect();
try {
  await client.query('BEGIN');
  const cursor = await client.query(
    'DECLARE my_cursor CURSOR FOR SELECT * FROM huge_table'
  );
  while (true) {
    const batch = await client.query('FETCH 1000 FROM my_cursor');
    if (batch.rows.length === 0) break;
    await processBatch(batch.rows);
  }
  await client.query('CLOSE my_cursor');
  await client.query('COMMIT');
} finally {
  client.release();
}

Query Optimization Checklist

1. [ ] Run EXPLAIN ANALYZE on slow queries
2. [ ] Check for Seq Scans on large tables -> add indexes
3. [ ] Check for Sort nodes -> add index matching ORDER BY
4. [ ] Check estimated vs actual rows -> run ANALYZE
5. [ ] Look for Nested Loop with large outer -> consider Hash Join
6. [ ] Check Buffers for disk reads -> increase shared_buffers
7. [ ] Verify parameters are passed as $1 not concatenated
8. [ ] Remove unnecessary columns from SELECT
9. [ ] Add LIMIT when you don't need all rows
10.[ ] Check for N+1 patterns in application code

Monitoring Slow Queries

// Enable pg_stat_statements extension
await pool.query('CREATE EXTENSION IF NOT EXISTS pg_stat_statements');

// Find slowest queries
async function getSlowestQueries(limit = 20) {
  const result = await pool.query(`
    SELECT
      query,
      calls,
      mean_exec_time::numeric(10,2) AS avg_ms,
      total_exec_time::numeric(10,2) AS total_ms,
      rows
    FROM pg_stat_statements
    ORDER BY mean_exec_time DESC
    LIMIT $1
  `, [limit]);
  return result.rows;
}

ANALYZE Command

Update table statistics so the planner makes better decisions:

-- Analyze a specific table
ANALYZE users;

-- Analyze entire database
ANALYZE;

-- Check when a table was last analyzed
SELECT relname, last_analyze, last_autoanalyze
FROM pg_stat_user_tables
WHERE relname = 'users';

Exercises

  1. Query Plan Detective

    Create a table with 1 million rows, run 5 different queries, and for each: (a) predict what the plan will look like, (b) run EXPLAIN ANALYZE, (c) compare prediction to reality, (d) optimize if needed.

    Hint

    Use generate_series to create test data. Try queries with and without indexes, with functions on indexed columns, with ORDER BY, with JOINs.

  2. Build a Slow Query Monitor

    Write a Node.js function that wraps pool.query, measures execution time, and logs any query taking over a configurable threshold (e.g., 100ms). Include the query text, parameters, duration, and row count.

    Hint

    Monkey-patch pool.query or create a wrapper that records Date.now() before and after. Store slow queries in an array or log file.

  3. N+1 Query Detector

    Write a utility that detects N+1 query patterns by tracking query templates. If the same parameterized query is executed more than N times within a short window, log a warning with the query and count.

    Hint

    Maintain a Map keyed by query text. Reset the window periodically. Alert when count exceeds threshold.

Common Problems & Solutions

1 EXPLAIN estimates are wildly wrong (estimated 1 row, actual 100,000)
Cause: Table statistics are outdated. The planner uses statistics to estimate row counts, and after major data changes, the stats can be stale.
Solution: Run `ANALYZE tablename` to update statistics. Configure autovacuum to run ANALYZE more frequently if data changes rapidly.
2 Query plan changes between environments (fast in dev, slow in production)
Cause: Different data volumes lead to different plan choices. With 100 rows, a Seq Scan is optimal. With 10M rows, an Index Scan is needed but the planner might not choose it if stats are wrong.
Solution: Test with production-sized data. Use pg_stat_statements to monitor actual query performance. Ensure ANALYZE runs regularly in production.
3 Adding an index doesn't improve query performance
Cause: The planner may still prefer a sequential scan if: (1) the table is small, (2) the query returns most rows, (3) statistics suggest low selectivity, or (4) the query uses a function on the indexed column.
Solution: Check the selectivity: if your WHERE matches > 10-20% of rows, an index scan may truly be slower. For function usage, create a functional index.
