Partitioning and Query Optimization at Scale

Intuition

Partitioning is like organizing a filing cabinet. Instead of one giant drawer with millions of papers (one huge table), you split them into drawers by year (range partitioning). When looking for a 2024 invoice, you only search the 2024 drawer, ignoring all others. This is partition pruning — skipping irrelevant drawers entirely.

Partitioning and Query Optimization at Scale

Introduction

As tables grow to millions or billions of rows, even indexed queries slow down. Table partitioning splits a large table into smaller physical pieces while maintaining a single logical table interface. Combined with query optimization techniques, partitioning keeps your database responsive at scale.

Range Partitioning in PostgreSQL

-- Create partitioned table
CREATE TABLE orders (
    id BIGSERIAL,
    user_id INTEGER NOT NULL,
    total DECIMAL(10,2) NOT NULL,
    status VARCHAR(20) NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- Create partitions
CREATE TABLE orders_2024_q1 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');

CREATE TABLE orders_2024_q2 PARTITION OF orders
    FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');

CREATE TABLE orders_2024_q3 PARTITION OF orders
    FOR VALUES FROM ('2024-07-01') TO ('2024-10-01');

CREATE TABLE orders_2024_q4 PARTITION OF orders
    FOR VALUES FROM ('2024-10-01') TO ('2025-01-01');

-- Future partitions (create in advance!)
CREATE TABLE orders_2025_q1 PARTITION OF orders
    FOR VALUES FROM ('2025-01-01') TO ('2025-04-01');

-- Indexes on partitioned tables
CREATE INDEX idx_orders_user_id ON orders (user_id);
CREATE INDEX idx_orders_status ON orders (status);
-- Indexes are automatically created on each partition

Automatic Partition Creation

// Script to create partitions automatically
async function createQuarterlyPartition(prisma, year, quarter) {
  const startMonth = (quarter - 1) * 3 + 1;
  const endMonth = startMonth + 3;

  const startDate = `${year}-${String(startMonth).padStart(2, '0')}-01`;
  const endDate = endMonth > 12
    ? `${year + 1}-01-01`
    : `${year}-${String(endMonth).padStart(2, '0')}-01`;

  const partitionName = `orders_${year}_q${quarter}`;

  await prisma.$executeRawUnsafe(`
    CREATE TABLE IF NOT EXISTS ${partitionName} PARTITION OF orders
    FOR VALUES FROM ('${startDate}') TO ('${endDate}')
  `);

  console.log(`Created partition ${partitionName}: ${startDate} to ${endDate}`);
}

// Run monthly to ensure partitions exist ahead of time
async function ensurePartitions() {
  const now = new Date();
  const year = now.getFullYear();
  const quarter = Math.ceil((now.getMonth() + 1) / 3);

  // Create current and next 2 quarters
  for (let i = 0; i < 3; i++) {
    let q = quarter + i;
    let y = year;
    if (q > 4) { q -= 4; y += 1; }
    await createQuarterlyPartition(prisma, y, q);
  }
}

Partition Pruning

PostgreSQL automatically skips irrelevant partitions when the query includes the partition key:

-- Only scans orders_2024_q3 partition (partition pruning)
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE created_at >= '2024-07-15' AND created_at < '2024-08-01';

-- Scans ALL partitions (no pruning - missing partition key)
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 42;

-- Scans only matching partitions
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE created_at >= '2024-01-01' AND created_at < '2024-07-01'
  AND user_id = 42;

Query Optimization at Scale

EXPLAIN ANALYZE

async function analyzeQuery(prisma, query) {
  const plan = await prisma.$queryRawUnsafe(`EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) ${query}`);
  return plan[0]['QUERY PLAN'][0];
}

// Express endpoint for query analysis
app.post('/admin/explain', async (req, res) => {
  const plan = await analyzeQuery(prisma, req.body.query);

  res.json({
    executionTime: plan['Execution Time'],
    planningTime: plan['Planning Time'],
    plan: plan.Plan,
    warnings: identifyIssues(plan.Plan),
  });
});

function identifyIssues(plan) {
  const issues = [];

  if (plan['Node Type'] === 'Seq Scan' && plan['Actual Rows'] > 10000) {
    issues.push(`Sequential scan on ${plan['Relation Name']} (${plan['Actual Rows']} rows). Consider adding an index.`);
  }

  if (plan['Actual Rows'] / (plan['Plan Rows'] || 1) > 10) {
    issues.push('Row estimate was off by 10x+. Run ANALYZE on the table.');
  }

  // Check children recursively
  if (plan.Plans) {
    for (const child of plan.Plans) {
      issues.push(...identifyIssues(child));
    }
  }

  return issues;
}

Effective Indexing

-- Composite index for common query patterns
CREATE INDEX idx_orders_user_status_created
ON orders (user_id, status, created_at DESC);

-- Partial index for common filter
CREATE INDEX idx_orders_active
ON orders (user_id, created_at DESC)
WHERE status = 'active';

-- Covering index (includes all needed columns)
CREATE INDEX idx_orders_covering
ON orders (user_id, created_at DESC)
INCLUDE (total, status);
-- Query can be answered entirely from the index (index-only scan)

-- BRIN index for time-series data (very small, works with partitions)
CREATE INDEX idx_orders_created_brin
ON orders USING BRIN (created_at) WITH (pages_per_range = 32);

Batch Processing Large Datasets

// BAD: Loading millions of rows into memory
const allOrders = await prisma.order.findMany({
  where: { status: 'pending' },
}); // Out of memory!

// GOOD: Cursor-based batch processing
async function* batchIterator(prisma, batchSize = 1000) {
  let cursor = undefined;

  while (true) {
    const batch = await prisma.order.findMany({
      where: { status: 'pending' },
      take: batchSize,
      skip: cursor ? 1 : 0,
      cursor: cursor ? { id: cursor } : undefined,
      orderBy: { id: 'asc' },
    });

    if (batch.length === 0) break;

    yield batch;
    cursor = batch[batch.length - 1].id;
  }
}

// Process in batches
for await (const batch of batchIterator(prisma, 1000)) {
  await Promise.all(
    batch.map(order => processOrder(order))
  );
  console.log(`Processed batch of ${batch.length} orders`);
}

// GOOD: Raw SQL with server-side cursor for huge datasets
async function processWithCursor(pool) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    await client.query(
      "DECLARE order_cursor CURSOR FOR SELECT * FROM orders WHERE status = 'pending' ORDER BY id"
    );

    let done = false;
    while (!done) {
      const result = await client.query('FETCH 1000 FROM order_cursor');
      if (result.rows.length === 0) {
        done = true;
      } else {
        await processBatch(result.rows);
      }
    }

    await client.query('CLOSE order_cursor');
    await client.query('COMMIT');
  } finally {
    client.release();
  }
}

Materialized Views

-- Precompute expensive aggregations
CREATE MATERIALIZED VIEW order_daily_stats AS
SELECT
    DATE(created_at) AS date,
    COUNT(*) AS order_count,
    SUM(total) AS revenue,
    AVG(total) AS avg_order_value,
    COUNT(DISTINCT user_id) AS unique_customers
FROM orders
GROUP BY DATE(created_at);

-- Index the materialized view
CREATE UNIQUE INDEX idx_order_daily_stats_date
ON order_daily_stats (date);

-- Refresh periodically
REFRESH MATERIALIZED VIEW CONCURRENTLY order_daily_stats;

// Scheduled refresh with BullMQ
await scheduledQueue.add('refresh-materialized-views', {}, {
  repeat: { pattern: '0 * * * *' }, // Every hour
  jobId: 'refresh-mat-views',
});

const worker = new Worker('scheduled-tasks', async (job) => {
  if (job.name === 'refresh-materialized-views') {
    await prisma.$executeRaw`
      REFRESH MATERIALIZED VIEW CONCURRENTLY order_daily_stats
    `;
    return { refreshed: true };
  }
}, { connection });

Common Pitfalls

  1. Queries without partition key: If your WHERE clause doesn't include the partition key, PostgreSQL scans ALL partitions. Always include the partition key.
  2. Too many partitions: Having thousands of small partitions adds planning overhead. Aim for partitions with millions of rows each.
  3. Missing indexes on partitions: Indexes must be created on the partitioned table (they are automatically applied to partitions in PostgreSQL 11+).
  4. Not archiving old partitions: Old partitions consume space. Detach and archive partitions that are no longer needed.

Summary

Table partitioning splits large tables into manageable pieces, enabling partition pruning for faster queries and easier data lifecycle management. Range partitioning is ideal for time-series data, list partitioning for categorical data, and hash partitioning for even distribution. Combine partitioning with proper indexing, batch processing, and materialized views for optimal performance at scale.

Common Problems & Solutions

1 Inserts failing with 'no partition of relation found for row'
Cause: A new row's partition key value doesn't match any existing partition range (e.g., new year started but partition wasn't created).
Solution: Create partitions ahead of time using a scheduled job. Add a default partition as a safety net.

CREATE TABLE orders_default PARTITION OF orders DEFAULT;

2 Partitioned queries not faster than unpartitioned
Cause: Queries don't include the partition key in WHERE clause, so PostgreSQL scans all partitions (no pruning).
Solution: Always include the partition key (e.g., created_at for time-partitioned tables) in WHERE clauses. Check with EXPLAIN that partition pruning is happening.
