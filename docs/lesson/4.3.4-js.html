Exception Filters: Custom Error Handling

Intuition

Exception filters are like emergency rooms at a hospital. Different ERs specialize in different problems -- the cardiac unit handles heart issues (DatabaseExceptionFilter), the general ER handles everything else (AllExceptionsFilter). The patient (exception) is routed to the most appropriate ER, treated (transformed into a proper response), and discharged (sent to the client).

Exception Filters: Custom Error Handling

Introduction

Exception filters handle all unhandled exceptions in your NestJS application. When an exception is thrown anywhere in the request pipeline -- from a controller, service, guard, pipe, or interceptor -- an exception filter catches it and transforms it into a user-friendly HTTP response.

The Default Exception Filter

NestJS has a built-in exception filter that handles all exceptions. For HttpException and its subclasses, it returns the exception's status code and message. For unrecognized exceptions, it returns a generic 500 Internal Server Error.

Throwing Exceptions

@Injectable()
export class UsersService {
  async findOne(id: string): Promise<User> {
    const user = await this.usersRepo.findById(id);

    if (!user) {
      throw new NotFoundException(`User with ID "${id}" not found`);
    }

    return user;
  }

  async create(dto: CreateUserDto): Promise<User> {
    const existing = await this.usersRepo.findByEmail(dto.email);
    if (existing) {
      throw new ConflictException('Email already registered');
    }
    return this.usersRepo.create(dto);
  }
}

Creating Custom Exception Filters

Basic Exception Filter

import {
  ExceptionFilter, Catch, ArgumentsHost, HttpException, HttpStatus,
} from '@nestjs/common';
import { Request, Response } from 'express';

@Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();
    const status = exception.getStatus();
    const exceptionResponse = exception.getResponse();

    const errorResponse = {
      statusCode: status,
      timestamp: new Date().toISOString(),
      path: request.url,
      method: request.method,
      message:
        typeof exceptionResponse === 'string'
          ? exceptionResponse
          : (exceptionResponse as any).message || 'An error occurred',
    };

    response.status(status).json(errorResponse);
  }
}

Catch-All Exception Filter

@Catch() // No argument = catches everything
export class AllExceptionsFilter implements ExceptionFilter {
  private readonly logger = new Logger('ExceptionFilter');

  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();

    let status = HttpStatus.INTERNAL_SERVER_ERROR;
    let message = 'Internal server error';
    let details: any = undefined;

    if (exception instanceof HttpException) {
      status = exception.getStatus();
      const exResponse = exception.getResponse();
      message =
        typeof exResponse === 'string'
          ? exResponse
          : (exResponse as any).message;
      details = typeof exResponse === 'object' ? exResponse : undefined;
    } else if (exception instanceof Error) {
      message = exception.message;

      // Log the full error for internal exceptions
      this.logger.error(
        `Unhandled exception: ${exception.message}`,
        exception.stack,
      );
    }

    const errorResponse = {
      statusCode: status,
      timestamp: new Date().toISOString(),
      path: request.url,
      method: request.method,
      message,
      // Only include stack trace in development
      ...(process.env.NODE_ENV !== 'production' && exception instanceof Error
        ? { stack: exception.stack }
        : {}),
    };

    response.status(status).json(errorResponse);
  }
}

Database Exception Filter

@Catch(QueryFailedError) // TypeORM exception
export class DatabaseExceptionFilter implements ExceptionFilter {
  private readonly logger = new Logger('DatabaseException');

  catch(exception: QueryFailedError, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();

    let status = HttpStatus.INTERNAL_SERVER_ERROR;
    let message = 'Database error';

    // PostgreSQL error codes
    const pgError = exception as any;
    switch (pgError.code) {
      case '23505': // unique_violation
        status = HttpStatus.CONFLICT;
        message = 'Resource already exists';
        break;
      case '23503': // foreign_key_violation
        status = HttpStatus.BAD_REQUEST;
        message = 'Referenced resource not found';
        break;
      case '23502': // not_null_violation
        status = HttpStatus.BAD_REQUEST;
        message = `Missing required field: ${pgError.column}`;
        break;
      default:
        this.logger.error(`Database error: ${exception.message}`, exception.stack);
    }

    response.status(status).json({
      statusCode: status,
      timestamp: new Date().toISOString(),
      path: request.url,
      message,
    });
  }
}

Custom Exception Classes

// exceptions/business.exception.ts
export class BusinessException extends HttpException {
  constructor(
    public readonly errorCode: string,
    message: string,
    status: HttpStatus = HttpStatus.BAD_REQUEST,
  ) {
    super({ errorCode, message, statusCode: status }, status);
  }
}

export class InsufficientFundsException extends BusinessException {
  constructor(required: number, available: number) {
    super(
      'INSUFFICIENT_FUNDS',
      `Insufficient funds: required ${required}, available ${available}`,
      HttpStatus.PAYMENT_REQUIRED,
    );
  }
}

export class RateLimitExceededException extends BusinessException {
  constructor(retryAfter: number) {
    super(
      'RATE_LIMIT_EXCEEDED',
      `Rate limit exceeded. Retry after ${retryAfter} seconds`,
      HttpStatus.TOO_MANY_REQUESTS,
    );
  }
}

Filter for Business Exceptions

@Catch(BusinessException)
export class BusinessExceptionFilter implements ExceptionFilter {
  catch(exception: BusinessException, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const status = exception.getStatus();

    response.status(status).json({
      statusCode: status,
      errorCode: exception.errorCode,
      message: exception.message,
      timestamp: new Date().toISOString(),
    });
  }
}

Binding Exception Filters

// Method level
@Post()
@UseFilters(HttpExceptionFilter)
create(@Body() dto: CreateUserDto) {}

// Controller level
@Controller('users')
@UseFilters(HttpExceptionFilter)
export class UsersController {}

// Global (main.ts -- no DI)
app.useGlobalFilters(new AllExceptionsFilter());

// Global with DI (preferred)
@Module({
  providers: [
    { provide: APP_FILTER, useClass: AllExceptionsFilter },
    { provide: APP_FILTER, useClass: DatabaseExceptionFilter },
  ],
})
export class AppModule {}

Filter Execution Order

When multiple filters are bound, more specific filters run first:

Exception thrown
  -> Method-level @Catch(SpecificException) filter
  -> Controller-level filter
  -> Global filter
  -> Default NestJS exception handler

The first filter that matches the exception type handles it. If no filter catches it, the default handler takes over.

Logging and Monitoring in Filters

@Catch()
export class GlobalExceptionFilter implements ExceptionFilter {
  private readonly logger = new Logger('GlobalExceptionFilter');

  constructor(private readonly httpAdapterHost: HttpAdapterHost) {}

  catch(exception: unknown, host: ArgumentsHost) {
    const { httpAdapter } = this.httpAdapterHost;
    const ctx = host.switchToHttp();
    const request = ctx.getRequest();

    const status =
      exception instanceof HttpException
        ? exception.getStatus()
        : HttpStatus.INTERNAL_SERVER_ERROR;

    // Structured logging for monitoring
    this.logger.error({
      statusCode: status,
      path: httpAdapter.getRequestUrl(request),
      method: httpAdapter.getRequestMethod(request),
      message: exception instanceof Error ? exception.message : 'Unknown error',
      stack: exception instanceof Error ? exception.stack : undefined,
      timestamp: new Date().toISOString(),
    });

    const responseBody = {
      statusCode: status,
      timestamp: new Date().toISOString(),
      path: httpAdapter.getRequestUrl(request),
      message:
        status === HttpStatus.INTERNAL_SERVER_ERROR
          ? 'Internal server error'
          : exception instanceof HttpException
            ? exception.message
            : 'Unknown error',
    };

    httpAdapter.reply(ctx.getResponse(), responseBody, status);
  }
}

RFC 7807 Problem Details

Implement the standard error response format:

interface ProblemDetails {
  type: string;        // URI identifying the problem type
  title: string;       // Short human-readable summary
  status: number;      // HTTP status code
  detail?: string;     // Human-readable explanation
  instance?: string;   // URI identifying the specific occurrence
}

@Catch()
export class ProblemDetailsFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();

    const status =
      exception instanceof HttpException
        ? exception.getStatus()
        : HttpStatus.INTERNAL_SERVER_ERROR;

    const problemDetails: ProblemDetails = {
      type: `https://api.example.com/errors/${status}`,
      title: HttpStatus[status] || 'Internal Server Error',
      status,
      detail: exception instanceof Error ? exception.message : undefined,
      instance: request.url,
    };

    response
      .status(status)
      .header('Content-Type', 'application/problem+json')
      .json(problemDetails);
  }
}

Common Pitfalls

  1. Exposing internal errors in production -- Stack traces and database error details should never reach the client in production.
  2. Catching too broadly -- A @Catch() filter with no argument catches everything, including intentional exceptions from guards and pipes. Be specific when possible.
  3. Not logging unhandled exceptions -- Internal server errors should always be logged with full stack traces for debugging.
  4. Filter order confusion -- More specific filters should be registered before generic ones.

Exercises

  1. Build a Production Exception Filter

    Create a global exception filter that: returns structured error responses (with errorCode, message, timestamp), logs errors with stack traces, hides internal details in production, includes correlation/request IDs for tracing, and handles both HttpException and unknown exceptions.

    Hint

    Use the Logger class for logging. Check process.env.NODE_ENV to determine what to expose. Generate or extract a correlation ID from the X-Request-Id header.

  2. Database Error Handler

    Create a specialized exception filter for database errors (PostgreSQL error codes). Map unique violations to 409 Conflict, foreign key violations to 400 Bad Request, and connection errors to 503 Service Unavailable. Include the constraint name in the error message for debugging.

    Hint

    PostgreSQL errors have a code property (e.g., '23505' for unique violation). Check the constraint and detail properties for additional context.

  3. Implement RFC 7807 Problem Details

    Create an exception filter that formats all errors according to RFC 7807 (Problem Details for HTTP APIs). Set the Content-Type to 'application/problem+json'. Create custom exception classes that carry problem type URIs.

    Hint

    Define a ProblemDetails interface with type, title, status, detail, instance. Set the Content-Type header explicitly. Map common HTTP status codes to descriptive type URIs.

Common Problems & Solutions

1 Custom exception filter doesn't catch exceptions from guards or pipes
Cause: The filter is bound at the method or controller level, but guards and pipes run outside that scope.
Solution: Register the filter globally using APP_FILTER in a module, or in main.ts with app.useGlobalFilters(). Global filters catch exceptions from the entire request pipeline.
2 Exception filter can't inject dependencies
Cause: The filter is instantiated with `new MyFilter()` in main.ts or @UseFilters(), bypassing the DI container.
Solution: Register the filter as a provider: `{ provide: APP_FILTER, useClass: MyFilter }`. Inject HttpAdapterHost instead of directly accessing Express response for platform agnosticism.
3 Two exception filters conflict -- wrong one handles the exception
Cause: Multiple filters registered globally where a generic @Catch() filter is registered before a specific one.
Solution: Register specific filters before generic ones. NestJS checks filters in registration order and uses the first one that matches the exception type.
