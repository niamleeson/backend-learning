Design a Distributed Message Queue (Kafka)

Intuition

Think of a message queue as a massive, distributed notebook with numbered pages. Writers add pages at the end (append-only), readers bookmark where they stopped (offsets), and multiple reading groups can independently track their own progress through the same notebook without interfering with each other.

Design a Distributed Message Queue (Kafka)

Introduction

A distributed message queue is the central nervous system of modern data-intensive applications. It decouples producers from consumers, absorbs traffic spikes, guarantees delivery, and enables real-time data pipelines. In this lesson we design a system inspired by Apache Kafka — the industry standard for high-throughput, fault-tolerant event streaming.

Requirements

Functional Requirements

  • Producers publish messages to named topics
  • Consumers subscribe to topics and receive messages in order
  • Messages are persisted durably (survive broker restarts)
  • Support consumer groups — multiple consumers share the load within a group, but each message is delivered to exactly one consumer per group
  • Support message retention by time or size (e.g., keep 7 days or 100 GB)
  • Consumers can replay messages from any offset
  • At-least-once delivery guarantee by default; support exactly-once semantics

Non-Functional Requirements

  • Throughput: 500,000 messages/second sustained
  • Latency: p99 < 10 ms for produce, < 50 ms for end-to-end
  • Durability: zero message loss for acknowledged writes
  • Availability: 99.99% uptime
  • Horizontal scalability: add brokers to increase capacity linearly

Capacity Estimation

Messages/sec:        500,000
Average message size: 1 KB
Data ingestion rate:  500,000 * 1 KB = ~500 MB/s
Replication factor:   3
Actual write rate:    500 MB/s * 3 = 1.5 GB/s
Retention:            7 days
Storage per day:      500 MB/s * 86,400 = ~43 TB/day
Total storage (7d):   43 TB * 7 = ~300 TB (before replication)
With replication:     300 TB * 3 = ~900 TB

                      Round up to 80 for headroom

Core Concepts

Topics and Partitions

A topic is a named stream of records. Each topic is divided into partitions — ordered, immutable sequences of records. Partitions enable parallelism: each partition can be consumed by one consumer in a group.

Topic: user-events (3 partitions)

Partition 0: [msg0] [msg3] [msg6] [msg9]  ...
Partition 1: [msg1] [msg4] [msg7] [msg10] ...
Partition 2: [msg2] [msg5] [msg8] [msg11] ...

Each message has an offset (sequential ID within partition)

Commit Log

Each partition is backed by an append-only commit log on disk. This is the key insight that makes Kafka fast — sequential disk writes are faster than random memory access on modern hardware.

Segment files on disk:

partition-0/
  00000000000000000000.log    (0 - 1,000,000)
  00000000000001000000.log    (1,000,001 - 2,000,000)
  00000000000002000000.log    (2,000,001 - ...)
  00000000000000000000.index  (offset -> position mapping)
  00000000000000000000.timeindex (timestamp -> offset mapping)

API Design

Producer API

import express from 'express';
const app = express();
app.use(express.json());

// POST /api/v1/topics/:topicName/messages
// Publish a message to a topic
app.post('/api/v1/topics/:topicName/messages', async (req, res) => {
  const { topicName } = req.params;
  const { key, value, headers } = req.body;
  // key is used for partition assignment (hash(key) % numPartitions)
  // value is the message payload (Buffer or JSON)
  // headers are optional key-value metadata

  const result = await messageQueue.produce({
    topic: topicName,
    key,
    value: Buffer.from(JSON.stringify(value)),
    headers,
    timestamp: Date.now(),
  });

  res.status(201).json({
    topic: topicName,
    partition: result.partition,
    offset: result.offset,
    timestamp: result.timestamp,
  });
});

// POST /api/v1/topics/:topicName/messages/batch
// Publish a batch of messages
app.post('/api/v1/topics/:topicName/messages/batch', async (req, res) => {
  const { topicName } = req.params;
  const { messages } = req.body; // Array of { key, value, headers }

  const results = await messageQueue.produceBatch({
    topic: topicName,
    messages: messages.map(m => ({
      key: m.key,
      value: Buffer.from(JSON.stringify(m.value)),
      headers: m.headers,
      timestamp: Date.now(),
    })),
  });

  res.status(201).json({ results });
});

Consumer API

// POST /api/v1/consumers
// Register a consumer in a group
app.post('/api/v1/consumers', async (req, res) => {
  const { groupId, topics, autoCommit } = req.body;

  const consumer = await messageQueue.createConsumer({
    groupId,
    topics,
    autoCommit: autoCommit ?? true,
    sessionTimeout: 30000,
    heartbeatInterval: 3000,
  });

  res.status(201).json({
    consumerId: consumer.id,
    groupId,
    assignedPartitions: consumer.assignments,
  });
});

// GET /api/v1/consumers/:consumerId/messages
// Poll for new messages
app.get('/api/v1/consumers/:consumerId/messages', async (req, res) => {
  const { consumerId } = req.params;
  const maxMessages = parseInt(req.query.max) || 100;
  const timeoutMs = parseInt(req.query.timeout) || 5000;

  const messages = await messageQueue.poll(consumerId, {
    maxMessages,
    timeoutMs,
  });

  res.json({
    messages: messages.map(m => ({
      topic: m.topic,
      partition: m.partition,
      offset: m.offset,
      key: m.key,
      value: JSON.parse(m.value.toString()),
      timestamp: m.timestamp,
      headers: m.headers,
    })),
  });
});

// POST /api/v1/consumers/:consumerId/offsets
// Commit consumer offsets
app.post('/api/v1/consumers/:consumerId/offsets', async (req, res) => {
  const { consumerId } = req.params;
  const { offsets } = req.body;
  // offsets: [{ topic, partition, offset }]

  await messageQueue.commitOffsets(consumerId, offsets);
  res.status(204).send();
});

Topic Management API

// POST /api/v1/topics
app.post('/api/v1/topics', async (req, res) => {
  const { name, partitions, replicationFactor, retentionMs, retentionBytes } = req.body;

  const topic = await messageQueue.createTopic({
    name,
    partitions: partitions || 12,
    replicationFactor: replicationFactor || 3,
    config: {
      'retention.ms': retentionMs || 7 * 24 * 60 * 60 * 1000,
      'retention.bytes': retentionBytes || -1, // unlimited
      'segment.bytes': 1073741824, // 1 GB segments
    },
  });

  res.status(201).json(topic);
});

Database Schema

The message queue primarily uses an append-only log on the filesystem, but metadata is stored in a coordination service. Here is the metadata schema:

-- Topic metadata
CREATE TABLE topics (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            VARCHAR(255) UNIQUE NOT NULL,
  partition_count INT NOT NULL DEFAULT 12,
  replication_factor INT NOT NULL DEFAULT 3,
  retention_ms    BIGINT NOT NULL DEFAULT 604800000,
  retention_bytes BIGINT DEFAULT -1,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Partition assignments (which broker owns which partition)
CREATE TABLE partition_assignments (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  topic_id        UUID NOT NULL REFERENCES topics(id),
  partition_num   INT NOT NULL,
  leader_broker   VARCHAR(255) NOT NULL,
  replica_brokers TEXT[] NOT NULL,  -- array of broker IDs
  isr             TEXT[] NOT NULL,  -- in-sync replicas
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (topic_id, partition_num)
);

-- Consumer group offsets
CREATE TABLE consumer_offsets (
  group_id        VARCHAR(255) NOT NULL,
  topic_id        UUID NOT NULL REFERENCES topics(id),
  partition_num   INT NOT NULL,
  committed_offset BIGINT NOT NULL DEFAULT 0,
  metadata        VARCHAR(255),
  committed_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (group_id, topic_id, partition_num)
);

-- Consumer group membership
CREATE TABLE consumer_group_members (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  group_id        VARCHAR(255) NOT NULL,
  consumer_id     VARCHAR(255) NOT NULL,
  assigned_partitions JSONB NOT NULL DEFAULT '[]',
  last_heartbeat  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  session_timeout_ms INT NOT NULL DEFAULT 30000,
  joined_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (group_id, consumer_id)
);

CREATE INDEX idx_consumer_group_heartbeat
  ON consumer_group_members(group_id, last_heartbeat);

-- Broker registry
CREATE TABLE brokers (
  id              VARCHAR(255) PRIMARY KEY,
  host            VARCHAR(255) NOT NULL,
  port            INT NOT NULL,
  rack            VARCHAR(50),
  status          VARCHAR(20) NOT NULL DEFAULT 'active',
  last_heartbeat  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  registered_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

Deep Dive: Key Design Decisions

Why Append-Only Logs?

Sequential I/O on modern SSDs achieves 500+ MB/s, while random I/O achieves ~50 MB/s. By only appending to logs, we exploit the disk's strongest access pattern.

Zero-Copy Transfer

When a consumer reads messages, the broker uses sendfile() to transfer data directly from the OS page cache to the network socket, bypassing user space entirely.

Traditional:   Disk -> Kernel Buffer -> User Buffer -> Socket Buffer -> NIC
Zero-copy:     Disk -> Kernel Buffer ─────────────────────────────> NIC

Consumer Group Rebalancing

When a consumer joins or leaves a group, partitions must be reassigned. This is the rebalancing protocol:

1. Consumer C3 joins Group-A
2. Coordinator detects membership change
3. All consumers in Group-A pause
4. Coordinator runs assignment algorithm:
   - Range: P0,P1 -> C1; P2,P3 -> C2; P4,P5 -> C3
   - Round Robin: P0->C1, P1->C2, P2->C3, P3->C1, P4->C2, P5->C3
5. Each consumer receives its new assignment
6. Consumers resume from last committed offsets

Scaling Strategies

  • More partitions = more parallelism (but more file handles, more rebalancing overhead)
  • More brokers = more storage and throughput capacity
  • Tiered storage: move old segments to S3/GCS for cheap long-term retention
  • Compression: producer-side compression (gzip, lz4, zstd) reduces network and storage by 60-80%
  • Batching: producers accumulate messages and send in batches to reduce network round trips

Trade-offs

Decision Pros Cons
Append-only log Incredible write throughput, simple Compaction needed for key-based topics
Partition-level ordering Scales horizontally No global ordering across partitions
Pull-based consumers Consumer controls pace, supports replay Requires polling, slightly higher latency
Replication Durability, fault tolerance 3x storage, replication lag
Consumer group rebalancing Automatic scaling Stop-the-world pauses during rebalance
At-least-once default Simpler, higher throughput Consumers must be idempotent

Failure Handling

  • Broker failure: Leader partition fails over to an ISR (in-sync replica). Unclean leader election can cause data loss.
  • Consumer failure: Session timeout triggers rebalance; partitions reassigned to remaining consumers.
  • Network partition: Split-brain risk. Use min.insync.replicas=2 with acks=all to prevent writes when quorum is lost.
  • Disk failure: Replicas on other brokers take over. Replace disk and re-replicate.

Summary

A distributed message queue like Kafka uses append-only commit logs for extreme write throughput, partitioning for horizontal scale, replication for durability, and consumer groups for parallel processing. The key design insight is treating the log as the fundamental storage abstraction — everything else (replay, retention, consumer coordination) follows naturally from this choice.

Exercises

  1. Build a Single-Partition Message Queue

    Implement a simple message queue in Node.js using an append-only file as the commit log. Support produce (append message), consume (read from offset), and commit offset operations. Use fs.appendFile for writes and fs.createReadStream for reads.

    Hint

    Store each message as a line of JSON. Track offsets as byte positions in the file. Use readline to parse messages from a given byte offset.

  2. Implement Consumer Groups

    Extend your message queue to support consumer groups. Create a topic with 4 partitions and 2 consumers in a group. Messages with the same key should always go to the same partition. When a consumer disconnects, its partitions should be redistributed.

    Hint

    Use a hash function on the message key to determine the partition. Store consumer-partition assignments in a Map. Implement a simple round-robin rebalance when membership changes.

  3. Benchmark Batching vs Single Writes

    Write a producer that sends 100,000 messages. Compare throughput and latency when sending one message at a time versus batching 500 messages per write. Measure messages/second and p99 latency for both approaches.

    Hint

    Use process.hrtime.bigint() for precise timing. Accumulate messages in an array and flush when the batch is full or a timeout (e.g., 5ms) expires.

  4. Design a Dead Letter Queue

    Implement a DLQ mechanism: if a consumer fails to process a message after 3 retries, move it to a special dead-letter topic. Include the original topic, partition, offset, error message, and retry count in the DLQ message.

    Hint

    Wrap your consumer processing in a retry loop with exponential backoff. After max retries, publish to a {topic}-dlq topic with enriched metadata.

Common Problems & Solutions

1 Consumer group rebalancing causes processing delays and duplicate messages
Symptoms: Periodic latency spikes in consumer metrics. Duplicate processing of messages after rebalances. Consumer lag spikes visible in monitoring.
Cause: When a consumer joins or leaves, all consumers in the group must pause, revoke their partitions, and receive new assignments. During this window, no messages are processed, and uncommitted messages will be re-delivered.
Solution: Use cooperative incremental rebalancing (KIP-429) instead of eager rebalancing. Configure longer session timeouts to avoid unnecessary rebalances from transient issues. Use static group membership where consumers have a persistent identity.
2 Producer throughput degrades as the number of partitions increases
Symptoms: Network throughput is low despite high CPU usage on the producer. Many small TCP packets visible in network monitoring.
Cause: Each partition requires a separate network connection and in-flight request buffer. The producer's linger.ms and batch.size settings may not be tuned for the partition count, causing many small requests.
Solution: Increase linger.ms to allow larger batches. Use compression to reduce network overhead. Limit the number of partitions per topic to what is actually needed for consumer parallelism. Use a connection pool with max connections per broker.
3 Messages are delivered out of order within a partition
Cause: When retries are enabled and max.in.flight.requests.per.connection > 1, a failed batch can be retried after a subsequent batch succeeds, causing reordering.
Solution: Set max.in.flight.requests.per.connection to 1 for strict ordering (sacrifices throughput). Alternatively, enable idempotent producer mode which handles ordering automatically with sequence numbers.
