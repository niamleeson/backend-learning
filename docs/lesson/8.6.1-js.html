Service Decomposition and Bounded Contexts

Intuition

Bounded contexts are like departments in a company. HR, Engineering, and Finance all have a concept of 'employee' but track different attributes (salary history vs. GitHub commits vs. tax withholdings). Each department owns its own records and communicates with others through memos (events) or phone calls (APIs), not by reaching into each other's filing cabinets (shared database).

Service Decomposition and Bounded Contexts

Introduction

Microservices architecture decomposes a monolithic application into small, independently deployable services. The key challenge is deciding where to draw service boundaries. Domain-Driven Design (DDD) provides the answer through bounded contexts — well-defined areas of the business domain where a particular model applies.

Monolith vs Microservices

Bounded Contexts

A bounded context is a boundary within which a domain model is defined and consistent. The same real-world concept may have different representations in different contexts.

E-Commerce Example:

┌─ User Context ─────────┐  ┌─ Order Context ──────────┐
│                         │  │                          │
│ User {                  │  │ Customer {               │
│   id, name, email,      │  │   id, name,              │
│   password, preferences,│  │   shippingAddress,        │
│   loginHistory          │  │   billingAddress          │
│ }                       │  │ }                        │
│                         │  │                          │
│ Cares about: auth,      │  │ Cares about: who ordered │
│ profile, preferences    │  │ and where to ship        │
└─────────────────────────┘  └──────────────────────────┘

Same person, different models!
User Context doesn't need shippingAddress.
Order Context doesn't need password or loginHistory.

Decomposition Strategies

1. Decompose by Business Capability

Business Capabilities → Services:

┌─────────────────────────────────────────────────────┐
│ Business Capability      │ Service                   │
├──────────────────────────┼───────────────────────────┤
│ User Management          │ User Service              │
│ Product Catalog          │ Catalog Service           │
│ Order Management         │ Order Service             │
│ Payment Processing       │ Payment Service           │
│ Shipping & Delivery      │ Shipping Service          │
│ Notifications            │ Notification Service      │
│ Analytics & Reporting    │ Analytics Service         │
│ Inventory Management     │ Inventory Service         │
└──────────────────────────┴───────────────────────────┘

2. Decompose by Subdomain

DDD Subdomain Classification:

Core Domain (competitive advantage):
  - Product Recommendation Engine
  - Pricing Strategy
  → Most investment, custom development

Supporting Domain (necessary but not differentiating):
  - Inventory Management
  - Shipping Coordination
  → Build or buy, moderate investment

Generic Domain (common across industries):
  - User Authentication
  - Payment Processing
  - Email Sending
  → Buy/use SaaS (Auth0, Stripe, SendGrid)

Service Communication

Synchronous (Request-Response):
  ┌──────┐  HTTP/gRPC  ┌──────┐
  │Svc A │────────────>│Svc B │
  │      │<────────────│      │
  └──────┘  Response    └──────┘

Asynchronous (Event-Driven):
  ┌──────┐   Event    ┌──────────┐   Event   ┌──────┐
  │Svc A │──────────>│  Message  │─────────>│Svc B │
  │      │           │  Broker   │          │      │
  └──────┘           │(Kafka/MQ) │          └──────┘
                     └──────────┘

// Synchronous: Order Service calls User Service
class OrderService {
  constructor(userServiceClient) {
    this.userServiceClient = userServiceClient;
  }

  async createOrder(userId, items) {
    // Sync call to validate user
    const user = await this.userServiceClient.getUser(userId);
    if (!user) throw new Error('User not found');

    const order = await this.orderRepo.create({
      customerId: userId,
      customerName: user.name,
      items,
      status: 'PENDING',
    });

    // Async: publish event for other services
    await this.eventBus.publish('order.created', {
      orderId: order.id,
      customerId: userId,
      items,
      total: order.total,
    });

    return order;
  }
}

// HTTP client for service-to-service calls
class UserServiceClient {
  constructor(baseUrl) {
    this.baseUrl = baseUrl;
  }

  async getUser(userId) {
    const response = await fetch(`${this.baseUrl}/api/users/${userId}`, {
      headers: { 'X-Service': 'order-service' },
      signal: AbortSignal.timeout(3000), // 3s timeout
    });

    if (response.status === 404) return null;
    if (!response.ok) throw new Error(`User service error: ${response.status}`);
    return response.json();
  }
}

Data Ownership

Each service owns its data. No shared databases.

WRONG: Shared Database
┌──────┐  ┌──────┐  ┌──────┐
│Svc A │  │Svc B │  │Svc C │
└──┬───┘  └──┬───┘  └──┬───┘
   │         │         │
   └─────────┼─────────┘
             ▼
    ┌────────────────┐
    │ Shared Database │  ← Tight coupling!
    └────────────────┘

RIGHT: Database Per Service
┌──────┐  ┌──────┐  ┌──────┐
│Svc A │  │Svc B │  │Svc C │
└──┬───┘  └──┬───┘  └──┬───┘
   │         │         │
   ▼         ▼         ▼
┌──────┐  ┌──────┐  ┌──────┐
│DB A  │  │DB B  │  │DB C  │
└──────┘  └──────┘  └──────┘

// Data duplication via events (eventual consistency)

// User Service publishes events when user data changes
class UserService {
  async updateProfile(userId, data) {
    const user = await this.userRepo.update(userId, data);

    // Publish event for other services to update their copies
    await this.eventBus.publish('user.profile.updated', {
      userId: user.id,
      name: user.name,
      email: user.email,
      // Only fields other services need
    });

    return user;
  }
}

// Order Service maintains its own copy of customer data
class OrderEventHandler {
  async handleUserUpdated(event) {
    // Update the local customer record
    await this.customerRepo.upsert({
      where: { externalUserId: event.userId },
      update: { name: event.name, email: event.email },
      create: {
        externalUserId: event.userId,
        name: event.name,
        email: event.email,
      },
    });
  }
}

When NOT to Use Microservices

Start with a monolith when:
  - Small team (< 10 developers)
  - Early-stage product (requirements unclear)
  - Simple domain
  - No need for independent deployment
  - No polyglot requirement

Move to microservices when:
  - Teams need to deploy independently
  - Different scaling requirements per feature
  - Clear domain boundaries have emerged
  - Organizational growth demands it
  - Monolith deployments are too risky/slow

Common Pitfalls

  1. Distributed monolith: Services that must be deployed together and share databases — all the complexity of microservices with none of the benefits.
  2. Too many services too early: Start with a well-structured monolith. Extract services as boundaries become clear.
  3. Synchronous chains: Service A calls B calls C calls D — latency compounds and one failure cascades.
  4. Shared database: Two services using the same database table creates tight coupling.

Summary

Service decomposition requires identifying bounded contexts — areas of the business with distinct models and responsibilities. Each service owns its data, communicates via APIs or events, and can be deployed independently. Use DDD to identify core, supporting, and generic domains. Start with a modular monolith and extract services when organizational and technical pressure demands it.

Exercises

  1. Domain Decomposition

    Take an e-commerce monolith with users, products, orders, payments, shipping, and notifications. Identify bounded contexts, define service boundaries, and specify which data each service owns.

    Hint

    Group by business capability. Identify which entities belong to which context. Note where the same concept (e.g., 'user') has different models in different contexts.

  2. Service Communication Design

    For an order creation flow involving User, Inventory, Payment, and Notification services, design both synchronous (validate user, check inventory) and asynchronous (payment, notification) communication patterns.

    Hint

    Synchronous for queries that block the flow. Asynchronous for actions that can happen eventually. Draw a sequence diagram.

  3. Event-Driven Data Sync

    Implement event-based data synchronization between a User Service and Order Service. When a user updates their address, the Order Service updates its local customer record.

    Hint

    User Service publishes 'user.updated' events. Order Service subscribes and updates its customer table.

Common Problems & Solutions

1 Services are tightly coupled and must deploy together
Cause: Services share a database or have synchronous dependency chains that break when any service changes.
Solution: Each service must own its data. Use events for data propagation. Design APIs with backward compatibility (versioning).
2 Microservices slower than the monolith
Cause: What was a function call in the monolith is now a network call. Latency adds up across service chains.
Solution: Minimize synchronous service-to-service calls. Use async events where possible. Denormalize data to avoid cross-service queries. Add caching.
3 Difficult to trace requests across services
Cause: A single user request triggers calls across multiple services, making debugging difficult.
Solution: Implement distributed tracing with OpenTelemetry. Use correlation IDs that propagate across all service calls. Centralize logs.
