Entity-Relationship Diagrams and Schema Design

Intuition

Designing a database schema is like designing the floor plan of a building before construction. You decide which rooms (tables) you need, what goes in each room (columns), and how rooms connect to each other (relationships). Changing the floor plan after the building is up is possible but expensive — so getting it right early saves enormous effort later.

Entity-Relationship Diagrams and Schema Design

Introduction

Before writing CREATE TABLE statements, you need to model your domain. Entity-Relationship (ER) diagrams are the standard tool for visualizing database structure — showing tables, columns, data types, and the relationships between them. Good schema design is the foundation of a maintainable backend.

Entities, Attributes, and Relationships

Entity:       A thing you want to store (User, Order, Product)
Attribute:    A property of an entity (name, email, price)
Relationship: How entities connect (User places Order)

Relationship Types:
  One-to-One   (1:1)  User <-> Profile
  One-to-Many  (1:N)  User <-> Orders
  Many-to-Many (M:N)  Students <-> Courses

ASCII ER Diagram Notation

One-to-Many (User has many Orders):

  +--------+       +--------+
  | users  |       | orders |
  +--------+       +--------+
  | id  PK |<------| id  PK |
  | name   |  1  N | user_id FK |
  | email  |       | total  |
  +--------+       +--------+

Many-to-Many (Students enroll in Courses):

  +----------+     +-------------+     +---------+
  | students |     | enrollments |     | courses |
  +----------+     +-------------+     +---------+
  | id    PK |<----| student_id FK|    | id   PK |
  | name     |     | course_id FK |--->| title   |
  +----------+     | enrolled_at  |    | credits |
                   | grade        |    +---------+
                   +-------------+
                   (junction table)

Schema Design Process

1. Identify entities (nouns in requirements)
2. Identify attributes for each entity
3. Identify relationships (verbs connecting entities)
4. Determine cardinality (1:1, 1:N, M:N)
5. Create junction tables for M:N relationships
6. Add primary keys, foreign keys, constraints
7. Choose data types
8. Add indexes for common queries

Real-World Schema: E-Commerce Platform

// Build the schema using raw pg queries
async function createEcommerceSchema() {
  await pool.query(`
    -- Users
    CREATE TABLE users (
      id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
      email varchar(320) UNIQUE NOT NULL,
      name text NOT NULL,
      password_hash text NOT NULL,
      role text NOT NULL DEFAULT 'customer'
        CHECK (role IN ('customer', 'admin', 'vendor')),
      created_at timestamptz DEFAULT NOW(),
      updated_at timestamptz DEFAULT NOW()
    );

    -- Addresses (1:N with users)
    CREATE TABLE addresses (
      id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
      user_id uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
      label text DEFAULT 'home',
      street text NOT NULL,
      city text NOT NULL,
      state text,
      postal_code text NOT NULL,
      country char(2) NOT NULL,
      is_default boolean DEFAULT false,
      UNIQUE (user_id, label)
    );

    -- Categories (self-referencing for hierarchy)
    CREATE TABLE categories (
      id serial PRIMARY KEY,
      name text NOT NULL,
      slug text UNIQUE NOT NULL,
      parent_id integer REFERENCES categories(id),
      description text
    );

    -- Products
    CREATE TABLE products (
      id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
      name text NOT NULL,
      slug text UNIQUE NOT NULL,
      description text,
      price numeric(10, 2) NOT NULL CHECK (price >= 0),
      compare_at_price numeric(10, 2),
      sku varchar(100) UNIQUE,
      stock_quantity integer NOT NULL DEFAULT 0 CHECK (stock_quantity >= 0),
      category_id integer REFERENCES categories(id),
      is_active boolean DEFAULT true,
      metadata jsonb DEFAULT '{}',
      created_at timestamptz DEFAULT NOW()
    );

    -- Product Images (1:N)
    CREATE TABLE product_images (
      id serial PRIMARY KEY,
      product_id uuid NOT NULL REFERENCES products(id) ON DELETE CASCADE,
      url text NOT NULL,
      alt_text text,
      sort_order integer DEFAULT 0
    );

    -- Orders
    CREATE TABLE orders (
      id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
      user_id uuid NOT NULL REFERENCES users(id),
      status text NOT NULL DEFAULT 'pending'
        CHECK (status IN ('pending','processing','shipped','delivered','cancelled')),
      shipping_address_id uuid REFERENCES addresses(id),
      subtotal numeric(10, 2) NOT NULL,
      tax numeric(10, 2) NOT NULL DEFAULT 0,
      shipping numeric(10, 2) NOT NULL DEFAULT 0,
      total numeric(10, 2) GENERATED ALWAYS AS (subtotal + tax + shipping) STORED,
      notes text,
      created_at timestamptz DEFAULT NOW(),
      updated_at timestamptz DEFAULT NOW()
    );

    -- Order Items (junction between orders and products)
    CREATE TABLE order_items (
      id serial PRIMARY KEY,
      order_id uuid NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
      product_id uuid NOT NULL REFERENCES products(id),
      quantity integer NOT NULL CHECK (quantity > 0),
      unit_price numeric(10, 2) NOT NULL,
      total numeric(10, 2) GENERATED ALWAYS AS (quantity * unit_price) STORED
    );

    -- Reviews (M:N between users and products)
    CREATE TABLE reviews (
      id serial PRIMARY KEY,
      user_id uuid NOT NULL REFERENCES users(id),
      product_id uuid NOT NULL REFERENCES products(id),
      rating integer NOT NULL CHECK (rating BETWEEN 1 AND 5),
      title text,
      body text,
      created_at timestamptz DEFAULT NOW(),
      UNIQUE (user_id, product_id) -- one review per user per product
    );

    -- Product Tags (M:N)
    CREATE TABLE tags (
      id serial PRIMARY KEY,
      name text UNIQUE NOT NULL
    );

    CREATE TABLE product_tags (
      product_id uuid REFERENCES products(id) ON DELETE CASCADE,
      tag_id integer REFERENCES tags(id) ON DELETE CASCADE,
      PRIMARY KEY (product_id, tag_id)
    );
  `);
}

Design Patterns

Soft Deletes

ALTER TABLE users ADD COLUMN deleted_at timestamptz;

-- "Delete" a user
UPDATE users SET deleted_at = NOW() WHERE id = $1;

-- Query only active users
SELECT * FROM users WHERE deleted_at IS NULL;

-- Partial index for active records
CREATE INDEX idx_active_users ON users (email) WHERE deleted_at IS NULL;

Audit Timestamps

-- Auto-update updated_at with a trigger
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at();

Polymorphic Associations

-- Comments on multiple entity types
CREATE TABLE comments (
  id serial PRIMARY KEY,
  user_id uuid NOT NULL REFERENCES users(id),
  commentable_type text NOT NULL CHECK (commentable_type IN ('post', 'product', 'review')),
  commentable_id uuid NOT NULL,
  body text NOT NULL,
  created_at timestamptz DEFAULT NOW()
);

CREATE INDEX idx_comments_target
  ON comments (commentable_type, commentable_id);

Schema Naming Conventions

Tables:      plural, snake_case         (users, order_items)
Columns:     singular, snake_case       (first_name, created_at)
Primary key: id                         (not user_id)
Foreign key: <singular_table>_id        (user_id, product_id)
Junction:    <table1>_<table2>          (product_tags, student_courses)
Indexes:     idx_<table>_<columns>      (idx_users_email)
Constraints: <table>_<column>_<type>    (users_email_unique)

Exercises

  1. Design a Social Media Schema

    Design a database schema for a social media platform supporting: users with profiles, posts (text + images), comments, likes (on posts and comments), follows (user follows user), and hashtags. Draw the ER diagram and write CREATE TABLE statements.

    Hint

    Users, posts, comments, likes, follows, hashtags, post_images are all entities. Likes and follows are junction tables. Use polymorphic association or separate tables for post_likes and comment_likes.

  2. Schema Review

    Given a schema with intentional design flaws (missing FKs, wrong data types, missing indexes, naming inconsistencies), identify all issues and write ALTER TABLE statements to fix them.

    Hint

    Check for: missing NOT NULL, wrong numeric types for money, missing indexes on FKs, timestamp vs timestamptz, inconsistent naming.

  3. Multi-Tenant Schema Design

    Design a schema for a SaaS app where each tenant (company) has its own users, projects, and tasks. Consider three approaches: (a) shared tables with tenant_id, (b) schema per tenant, (c) database per tenant. Implement approach (a).

    Hint

    Add tenant_id to every table. Create a row-level security policy. Add composite indexes starting with tenant_id.

Common Problems & Solutions

1 Foreign key column has no index, causing slow JOINs and deletes
Cause: PostgreSQL creates indexes for PRIMARY KEY and UNIQUE constraints, but NOT for FOREIGN KEY columns.
Solution: Always create indexes on foreign key columns: `CREATE INDEX idx_orders_user_id ON orders (user_id)`. This speeds up JOINs and cascading deletes.
2 Many-to-many relationship requires a junction table but data is duplicated
Cause: Trying to model M:N without a junction table, storing comma-separated IDs or arrays.
Solution: Create a proper junction table with composite primary key and foreign keys to both sides. This enables proper constraints, indexing, and additional attributes on the relationship.
3 Schema changes require downtime and are risky
Cause: No migration system in place. Schema changes are applied manually.
Solution: Use a migration tool (Prisma Migrate, Knex migrations, or raw SQL files with version numbers). Test migrations against a copy of production data before applying.
