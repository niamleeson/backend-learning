Building a Real-Time Chat Application

Intuition

A chat server is essentially a post office that delivers letters instantly: it keeps a directory of connected users and their mailboxes, stamps each message with metadata, and makes sure everyone in the conversation gets a copy.

Building a Real-Time Chat Application

A real-time chat application combines WebSockets for live messaging, user authentication, message persistence, and a connection manager to route messages between participants.

from fastapi import FastAPI, WebSocket, WebSocketDisconnect
from datetime import datetime
import json

app = FastAPI()

class ChatManager:
    def __init__(self):
        self.connections: dict[str, WebSocket] = {}
        self.messages: list[dict] = []

    async def connect(self, username: str, ws: WebSocket):
        await ws.accept()
        self.connections[username] = ws
        await self.broadcast_system(f"{username} joined the chat")

    async def disconnect(self, username: str):
        self.connections.pop(username, None)
        await self.broadcast_system(f"{username} left the chat")

    async def broadcast_system(self, message: str):
        msg = {"type": "system", "content": message,
               "timestamp": datetime.utcnow().isoformat()}
        await self._send_all(msg)

    async def handle_message(self, username: str, content: str):
        msg = {"type": "message", "sender": username,
               "content": content,
               "timestamp": datetime.utcnow().isoformat()}
        self.messages.append(msg)
        await self._send_all(msg)

    async def _send_all(self, msg: dict):
        for ws in self.connections.values():
            await ws.send_json(msg)

chat = ChatManager()

@app.websocket("/chat/{username}")
async def chat_endpoint(websocket: WebSocket, username: str):
    await chat.connect(username, websocket)
    try:
        while True:
            data = await websocket.receive_text()
            await chat.handle_message(username, data)
    except WebSocketDisconnect:
        await chat.disconnect(username)

Key Takeaways: - Map usernames to WebSocket connections for directed messaging - Distinguish system messages (join/leave) from user messages with a type field - Store messages for history and replay when new users connect - Use send_json() for structured message delivery

Exercises

  1. Message History on Connect

    When a new user connects to the chat, send them the last 50 messages as a 'history' event before they start receiving live messages. Format the history as a single JSON array.

    Hint

    After accept() and before the receive loop, send await ws.send_json({"type": "history", "messages": self.messages[-50:]}) to deliver the backlog.

  2. Direct Messaging

    Extend the chat to support direct messages. If a user sends a message starting with @username, deliver it only to that user (and the sender). Show an error if the target user is not connected.

    Hint

    Parse the message in handle_message. If it starts with @, extract the target username, look it up in self.connections, and send only to that connection and the sender's connection.

Common Problems & Solutions

1 Duplicate messages received when reconnecting quickly
Cause: The old WebSocket connection has not been cleaned up before the new one registers, resulting in two entries for the same username.
Solution: In the `connect` method, check if the username already exists in the connections dict. If so, close the old connection before registering the new one. Use `await old_ws.close()` before updating.
2 Message ordering inconsistent across clients
Cause: Messages are sent to each client sequentially; network latency variations cause different delivery orders.
Solution: Include a server-assigned sequence number or timestamp in each message. Clients should order messages by this field rather than by arrival time. This ensures consistent ordering regardless of delivery speed.
