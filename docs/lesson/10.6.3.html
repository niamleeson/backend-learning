Design a Logging/Analytics Pipeline

Intuition

A logging pipeline is a high-throughput ETL system where the key insight is that logs have a natural time dimension that dictates everything: partitioning, retention, and query patterns. The collection tier (Kafka) is the shock absorber that prevents bursty log producers from overwhelming the indexing layer.

Design a Logging/Analytics Pipeline

Requirements

  • Ingest terabytes of log data daily from thousands of services and hosts
  • Support structured and unstructured log formats with schema evolution
  • Provide near real-time search (under 30 seconds from emission to searchable)
  • Retain logs for 30-90 days with archival to cold storage for compliance

Architecture

Log agents on each host tail log files and ship events to a collection tier (Kafka/Fluentd). A stream processing layer enriches, parses, and routes logs by type. Hot logs go to an indexing cluster (Elasticsearch/OpenSearch) for search. Cold logs are archived to S3/GCS in columnar format (Parquet) for batch analytics.

Key Components

  • Log Agent: Filebeat/Fluentd on each host; handles backpressure and local buffering
  • Collection Tier: Kafka topics partitioned by service; decouples producers from consumers
  • Stream Processor: Flink/Logstash that parses, enriches (adds host/region tags), and routes
  • Index Cluster: Elasticsearch with time-based indices and ILM (Index Lifecycle Management) for rollover and deletion

Scaling

Time-based index partitioning (one index per day) for easy retention management. Scale Kafka partitions and ES data nodes independently. Use sampling for very high-volume debug logs.

Trade-offs

  • Indexing everything enables ad-hoc search but is expensive; selective indexing saves cost but requires knowing what to keep upfront
  • Shorter index retention reduces cost but limits incident investigation windows
  • Structured logging is easier to query but requires upfront schema discipline from developers

Exercises

  1. Build a Log Pipeline with Backpressure

    Create a pipeline: log generator -> in-memory buffer -> batch writer to disk. Implement backpressure so the generator slows down when the buffer is 80% full.

    Hint

    Use a bounded blocking queue as the buffer. When the queue reaches 80% capacity, have the producer sleep or drop low-priority logs. The consumer batches and writes every N messages or T seconds.

  2. Implement Time-Based Index Rotation

    Simulate an indexing system that creates a new index daily, automatically deletes indices older than 7 days, and redirects all writes to the current day's index.

    Hint

    Use an alias pointing to the current index. A scheduled job creates tomorrow's index, switches the alias, and deletes the index from 8 days ago.

Common Problems & Solutions

1 Elasticsearch cluster goes red during log bursts from a single service
Cause: One service emits 100x normal log volume during an incident, saturating the index write queue
Solution: Implement per-service rate limiting in the stream processor, and use separate Kafka topics or ES indices for high-volume services to isolate impact.
2 Log searches return incomplete results for recent events
Cause: The Elasticsearch refresh interval is set to 30 seconds, so recently indexed logs are not yet searchable
Solution: Reduce the refresh interval to 5 seconds for hot indices (accepting higher resource usage), or provide a separate real-time tail view that reads directly from Kafka.
