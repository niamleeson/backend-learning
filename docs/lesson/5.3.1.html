async def vs def: When to Use Each

Intuition

The event loop is like a single-lane highway: async def lets cars merge smoothly, but one stalled car blocks everything. Plain def sends blocking work to a side road (thread pool) so the highway stays clear.

async def vs def: When to Use Each

FastAPI supports both async and sync endpoint functions. Choosing correctly between async def and def impacts performance. Async endpoints run on the event loop; sync endpoints are run in a thread pool.

import asyncio
import httpx
import time
from fastapi import FastAPI

app = FastAPI()

# Use async def for I/O-bound work with async libraries
@app.get("/async-fetch")
async def async_fetch():
    async with httpx.AsyncClient() as client:
        resp = await client.get("https://httpbin.org/delay/1")
    return {"status": resp.status_code}

# Use def for CPU-bound or blocking I/O work
@app.get("/sync-compute")
def sync_compute():
    # Runs in a thread pool, won't block the event loop
    total = sum(i * i for i in range(1_000_000))
    return {"total": total}

# WRONG: blocking call in async def blocks the event loop
@app.get("/bad-example")
async def bad_example():
    time.sleep(5)  # This blocks the entire server!
    return {"message": "Don't do this"}

Key Takeaways: - Use async def when you await async libraries (httpx, aiofiles, asyncpg) - Use plain def for blocking I/O or CPU-bound work; FastAPI runs it in a thread pool - Never use blocking calls like time.sleep() or requests.get() inside async def - When in doubt, use def as it is safer against accidental blocking

Exercises

  1. Benchmark Async vs Sync

    Create two endpoints that each make 5 sequential HTTP calls to a slow endpoint. One uses async def with httpx.AsyncClient, the other uses def with requests. Use time.perf_counter() to measure and return the elapsed time.

    Hint

    For the async version, use await in a loop. For the bonus challenge, use asyncio.gather() to make all 5 calls concurrently and compare the timing.

  2. Run Blocking Code in Async Endpoint

    Create an async def endpoint that needs to call a blocking function (simulated with time.sleep(2)). Use asyncio.to_thread() or loop.run_in_executor() to offload it without blocking the event loop.

    Hint

    Use result = await asyncio.to_thread(blocking_function, arg1, arg2) to run the blocking function in a separate thread from within an async endpoint.

Common Problems & Solutions

1 Server becomes unresponsive under concurrent requests
Cause: A blocking operation like `time.sleep()`, `requests.get()`, or heavy computation is inside an `async def` endpoint, blocking the event loop.
Solution: Move blocking code to a `def` endpoint (FastAPI auto-threads it) or use `asyncio.to_thread()` to offload it. Replace blocking libraries with their async equivalents.
2 async def endpoint is slower than expected
Cause: Using `await` sequentially for independent operations instead of running them concurrently with `asyncio.gather()`.
Solution: Use `results = await asyncio.gather(coro1(), coro2(), coro3())` to run independent async operations concurrently instead of awaiting them one by one.
