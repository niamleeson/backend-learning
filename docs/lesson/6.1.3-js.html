systemd, cron, and Process Management

Intuition

systemd is like a super-reliable babysitter for your application. You tell it exactly how to start your app, and it watches over it 24/7 - restarting it if it crashes, starting it when the server boots, and keeping logs of everything. cron is like an alarm clock that triggers tasks on a schedule, while signals are like tapping someone on the shoulder (SIGTERM) versus pushing them out the door (SIGKILL).

systemd, cron, and Process Management

Introduction

In production, your Node.js application needs to start automatically on boot, restart if it crashes, run scheduled tasks, and be manageable by operations teams. This lesson covers the Linux tools that make this possible: systemd for service management, cron for scheduled tasks, and general process management techniques.

systemd: The Modern Service Manager

systemd is the init system and service manager for most modern Linux distributions. It manages the lifecycle of services (daemons) on your system.

Creating a Service for Your Node.js App

Create a service file at /etc/systemd/system/myapp.service:

[Unit]
Description=My Node.js Application
Documentation=https://github.com/yourorg/myapp
After=network.target postgresql.service
Wants=postgresql.service

[Service]
Type=simple
User=deploy
Group=deploy
WorkingDirectory=/var/www/myapp
ExecStart=/usr/bin/node dist/server.js
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure
RestartSec=5
StartLimitBurst=5
StartLimitIntervalSec=60

# Environment
Environment=NODE_ENV=production
Environment=PORT=3000
EnvironmentFile=/var/www/myapp/.env

# Security hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/var/www/myapp/uploads /var/log/myapp
PrivateTmp=true

# Logging
StandardOutput=journal
StandardError=journal
SyslogIdentifier=myapp

# Resource limits
LimitNOFILE=65535
MemoryMax=512M
CPUQuota=80%

[Install]
WantedBy=multi-user.target

Managing Services

# Reload systemd after creating/modifying service files
sudo systemctl daemon-reload

# Start, stop, restart
sudo systemctl start myapp
sudo systemctl stop myapp
sudo systemctl restart myapp
sudo systemctl reload myapp     # Graceful reload (if supported)

# Enable/disable auto-start on boot
sudo systemctl enable myapp
sudo systemctl disable myapp

# Check status
sudo systemctl status myapp

# View logs
sudo journalctl -u myapp              # All logs
sudo journalctl -u myapp -f           # Follow (like tail -f)
sudo journalctl -u myapp --since today
sudo journalctl -u myapp --since "2024-01-15 10:00" --until "2024-01-15 11:00"
sudo journalctl -u myapp -n 100       # Last 100 lines
sudo journalctl -u myapp -p err       # Only errors

# List all services
systemctl list-units --type=service
systemctl list-units --type=service --state=running

Service Dependencies

# Ensure PostgreSQL starts before your app
[Unit]
After=postgresql.service redis.service
Requires=postgresql.service         # Hard dependency - if PG fails, app fails
Wants=redis.service                 # Soft dependency - app starts even if Redis fails

# Start multiple instances
# myapp@.service (template)
[Service]
ExecStart=/usr/bin/node dist/server.js
Environment=PORT=%i

# Then: systemctl start myapp@3000 myapp@3001 myapp@3002

Understanding Restart Policies

# Restart policies:
Restart=no                 # Never restart (default)
Restart=on-failure         # Restart on non-zero exit code (RECOMMENDED)
Restart=always             # Always restart (even on clean exit)
Restart=on-abnormal        # Restart on signal/timeout
Restart=on-abort           # Restart on signal only

# Restart timing
RestartSec=5               # Wait 5 seconds before restarting
StartLimitBurst=5          # Max 5 restarts...
StartLimitIntervalSec=60   # ...within 60 seconds, then give up

cron: Scheduled Tasks

cron executes commands on a schedule. Essential for backups, cleanup tasks, report generation, and periodic data processing.

Cron Schedule Format

┌───────────── minute (0-59)
│ ┌───────────── hour (0-23)
│ │ ┌───────────── day of month (1-31)
│ │ │ ┌───────────── month (1-12)
│ │ │ │ ┌───────────── day of week (0-7, 0 and 7 = Sunday)
│ │ │ │ │
* * * * *  command to execute

Common Schedules

# Edit crontab for current user
crontab -e

# List current user's crontab
crontab -l

# Common schedules
* * * * *     # Every minute
*/5 * * * *   # Every 5 minutes
0 * * * *     # Every hour (at minute 0)
0 0 * * *     # Every day at midnight
0 0 * * 0     # Every Sunday at midnight
0 2 * * *     # Every day at 2 AM
0 0 1 * *     # First day of every month
30 4 * * 1-5  # Weekdays at 4:30 AM

Real-World Cron Examples for Node.js

# Database backup every day at 2 AM
0 2 * * * /var/www/myapp/scripts/backup-db.sh >> /var/log/myapp/backup.log 2>&1

# Clean up temp files every hour
0 * * * * find /var/www/myapp/tmp -type f -mmin +60 -delete

# Run a Node.js script to send digest emails every morning
0 8 * * * cd /var/www/myapp && /usr/bin/node scripts/send-digest.js >> /var/log/myapp/digest.log 2>&1

# Rotate application logs weekly
0 0 * * 0 /usr/sbin/logrotate /etc/logrotate.d/myapp

# Health check every 5 minutes
*/5 * * * * curl -sf http://localhost:3000/health || systemctl restart myapp

# Certificate renewal (Let's Encrypt)
0 0 1,15 * * certbot renew --quiet && systemctl reload nginx

Cron Best Practices

# Always redirect output to a log file
0 2 * * * /path/to/script.sh >> /var/log/cron-job.log 2>&1

# Use full paths (cron has minimal PATH)
0 * * * * /usr/bin/node /var/www/myapp/scripts/task.js

# Set environment variables at the top of crontab
SHELL=/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin
NODE_ENV=production
MAILTO=admin@example.com

# Lock file to prevent overlapping runs
*/5 * * * * flock -n /tmp/mytask.lock /var/www/myapp/scripts/long-task.sh

systemd Timers (Modern Alternative to cron)

# /etc/systemd/system/myapp-backup.timer
[Unit]
Description=Daily database backup

[Timer]
OnCalendar=*-*-* 02:00:00
Persistent=true           # Run missed executions after downtime
RandomizedDelaySec=300     # Random delay up to 5 min (avoid thundering herd)

[Install]
WantedBy=timers.target

# /etc/systemd/system/myapp-backup.service
[Unit]
Description=Database backup job

[Service]
Type=oneshot
User=deploy
ExecStart=/var/www/myapp/scripts/backup-db.sh

# Enable and start the timer
sudo systemctl enable myapp-backup.timer
sudo systemctl start myapp-backup.timer

# Check timer status
systemctl list-timers --all

Process Management

Understanding Signals

Signal     Number   Meaning                  Node.js Handler
─────────────────────────────────────────────────────────────
SIGTERM    15       Graceful shutdown         process.on('SIGTERM', ...)
SIGINT     2        Interrupt (Ctrl+C)        process.on('SIGINT', ...)
SIGKILL    9        Force kill (uncatchable)  Cannot be handled
SIGHUP     1        Reload configuration      process.on('SIGHUP', ...)
SIGUSR2    12       User-defined              Used by nodemon, PM2

Graceful Shutdown in Node.js

// server.js - Production-ready signal handling
const server = app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});

function gracefulShutdown(signal) {
  console.log(`Received ${signal}. Starting graceful shutdown...`);

  // Stop accepting new connections
  server.close(() => {
    console.log('HTTP server closed.');

    // Close database connections
    prisma.$disconnect().then(() => {
      console.log('Database disconnected.');

      // Close Redis
      redis.quit().then(() => {
        console.log('Redis disconnected.');
        process.exit(0);
      });
    });
  });

  // Force shutdown after 30 seconds
  setTimeout(() => {
    console.error('Forced shutdown after timeout');
    process.exit(1);
  }, 30000);
}

process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
process.on('SIGINT', () => gracefulShutdown('SIGINT'));

// Handle uncaught errors
process.on('uncaughtException', (err) => {
  console.error('Uncaught Exception:', err);
  gracefulShutdown('uncaughtException');
});

process.on('unhandledRejection', (reason) => {
  console.error('Unhandled Rejection:', reason);
  // Don't exit - just log. Or exit if you prefer fail-fast.
});

Putting It All Together: Production Deployment

Here is a complete example of deploying a Node.js app with systemd:

#!/bin/bash
# deploy.sh - Production deployment script

set -euo pipefail

APP_DIR=/var/www/myapp
DEPLOY_USER=deploy

echo "=== Deploying application ==="

# Pull latest code
cd $APP_DIR
git pull origin main

# Install dependencies
npm ci --production

# Build TypeScript
npm run build

# Run database migrations
npx prisma migrate deploy

# Restart with zero downtime
sudo systemctl reload-or-restart myapp

# Verify health
sleep 5
if curl -sf http://localhost:3000/health > /dev/null; then
    echo "Deployment successful!"
else
    echo "Health check failed! Rolling back..."
    git checkout HEAD~1
    npm ci --production
    npm run build
    sudo systemctl restart myapp
    exit 1
fi

Common Pitfalls

  1. Not using daemon-reload after editing service files - systemd caches unit files. Always run sudo systemctl daemon-reload.
  2. Cron jobs failing silently - Always redirect output: >> /var/log/job.log 2>&1.
  3. Not handling SIGTERM in Node.js - Your app gets 30 seconds to shut down before SIGKILL. Use that time to close connections.
  4. Running Node.js as root - Always create a dedicated user. Use cap_net_bind_service if you need port 80.
  5. Overlapping cron jobs - Use flock to prevent a slow job from running concurrently with itself.

Exercises

  1. Create a systemd Service

    Write a complete systemd service file for a Node.js Express application. It should: run as the 'deploy' user, auto-restart on failure with a 5-second delay, load environment from a .env file, set a memory limit of 512MB, and log to journald. Enable it to start on boot.

    Hint

    Use the [Unit], [Service], and [Install] sections. Key directives: User, WorkingDirectory, ExecStart, Restart, RestartSec, EnvironmentFile, MemoryMax.

  2. Cron Job Suite

    Write a crontab with the following jobs: (1) database backup at 2 AM daily, (2) log cleanup every Sunday, (3) health check every 5 minutes that restarts the service if it fails, (4) SSL certificate renewal on the 1st and 15th of each month. Include proper logging for each job.

    Hint

    Use crontab -e. Remember to redirect both stdout and stderr. Use flock for the backup job to prevent overlapping runs.

  3. Graceful Shutdown Implementation

    Write a Node.js server that handles SIGTERM and SIGINT. On receiving these signals, it should: (1) stop accepting new connections, (2) wait for existing requests to complete (up to 30 seconds), (3) close database connections, and (4) exit. Test by sending signals with kill.

    Hint

    Use server.close() to stop accepting connections. Track active connections with a counter. Use setTimeout as a safety net for forced shutdown.

Common Problems & Solutions

1 systemd service keeps restarting in a loop then stops
Symptoms: Status shows 'failed' and 'start-limit-hit'
Cause: The service failed too many times within the StartLimitIntervalSec window, so systemd stopped trying.
Solution: Check logs with `journalctl -u myapp -n 50`. Fix the root cause (missing env vars, wrong path, port conflict). Reset the failure counter: `sudo systemctl reset-failed myapp`. Adjust StartLimitBurst and StartLimitIntervalSec if the restarts are expected during startup.
2 Cron job works manually but not in cron
Cause: Cron runs with a minimal environment (limited PATH, no shell profile loaded).
Solution: Use full paths to all executables (`/usr/bin/node` not just `node`). Set PATH at the top of crontab. Set the working directory with `cd /path && command`. Add logging to capture errors: `>> /var/log/job.log 2>&1`.
3 Node.js app not releasing port after restart
Symptoms: Error: listen EADDRINUSE :::3000
Cause: The old process did not shut down gracefully, and the port is still held by a lingering connection in TIME_WAIT state.
Solution: Handle SIGTERM properly in your Node.js code. In your server, ensure server.close() is called. If the issue persists, check for child processes: `pgrep -P old_pid`. As a last resort, use SO_REUSEADDR (Express does this by default).
