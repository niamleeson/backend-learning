Design a Notification Service

Intuition

A notification service is an event-driven pipeline that transforms a notification intent into multi-channel deliveries while respecting user preferences. The key challenge is reliable delivery at scale without overwhelming downstream providers or annoying users with duplicates.

Design a Notification Service

Requirements

  • Support push notifications, SMS, email, and in-app channels
  • Handle millions of notifications per day with at-least-once delivery
  • Support user preferences, templates, scheduling, and priority levels
  • Provide delivery tracking and analytics

Architecture

A notification API accepts requests and publishes to a message queue (Kafka). Channel-specific workers consume messages and dispatch via third-party providers (APNs, FCM, Twilio, SendGrid). A preference service filters and routes notifications. A tracking service records delivery status.

Key Components

  • API Gateway: Accepts notification requests, validates, and enqueues
  • Message Queue: Kafka topics partitioned by channel type for parallel processing
  • Channel Workers: Dedicated consumers for each channel with provider-specific adapters
  • Preference Service: Stores user opt-in/out, quiet hours, and channel preferences
  • Template Engine: Renders notification content from templates with variable substitution

Scaling

Scale workers independently per channel based on volume. Use Kafka partitions for parallelism. Batch SMS and email sends for efficiency. Use priority queues to ensure critical notifications (2FA codes) are not delayed by marketing blasts.

Trade-offs

  • At-least-once delivery may cause duplicates; exactly-once adds complexity and latency
  • Third-party provider dependencies require fallback providers and circuit breakers

Exercises

  1. Design a Priority Queue System

    Implement a notification prioritization system with three levels: critical (OTP/2FA), high (order updates), and normal (marketing). Ensure critical notifications are processed within seconds even when millions of normal notifications are queued.

    Hint

    Use separate Kafka topics or queue lanes per priority level with dedicated consumer groups. Allocate more consumer instances to higher-priority topics.

  2. Build a Rate-Limited Provider Adapter

    Create an adapter for an SMS provider that enforces rate limits (e.g., 100 messages/second). Buffer excess messages and retry with exponential backoff when the provider returns 429 or 503 errors.

    Hint

    Use a token bucket for outgoing rate control and a dead-letter queue for messages that fail after max retries.

Common Problems & Solutions

1 Users receive duplicate notifications for the same event, especially during retries.
Cause: Worker crashes after sending but before acknowledging the message, causing the queue to redeliver the same notification.
Solution: Implement idempotency by generating a unique notification ID per event and checking a deduplication store (Redis with TTL) before sending. Track sent notification IDs to skip duplicates.
2 Notification delivery latency spikes during marketing campaign blasts, delaying critical OTP notifications.
Cause: All notifications share the same queue and worker pool, so a flood of low-priority messages starves high-priority ones.
Solution: Separate queues by priority with dedicated worker pools. Reserve capacity for critical notifications. Implement backpressure on low-priority queues during high load.
