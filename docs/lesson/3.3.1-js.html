REST Principles and Resource Design

Intuition

A REST API is like a well-organized library. Each book (resource) has a unique catalog number (URI). You can browse the catalog (GET collection), check out a specific book (GET by ID), donate a new book (POST), replace a damaged copy (PUT), add a bookmark (PATCH), or remove a book (DELETE). The librarian (server) does not remember who you are between visits (stateless) — you show your library card (auth token) every time.

REST Principles and Resource Design

Introduction

REST (Representational State Transfer) is an architectural style for designing networked applications. While many developers equate REST with "JSON over HTTP," true REST encompasses a set of constraints and principles that, when followed, produce APIs that are scalable, maintainable, and intuitive. In this lesson you will learn REST's core principles, how to model resources, and how to design URLs and relationships that make your API a pleasure to use.

REST Constraints

Roy Fielding defined six constraints in his 2000 dissertation:

1. Client-Server Separation

The client and server are independent. The server does not know about the UI, and the client does not know about data storage.

2. Statelessness

Each request contains all the information the server needs. The server does not store client session state between requests.

STATEFUL (non-REST):
  Client: "Log me in"          -> Server stores session
  Client: "Get my profile"     -> Server looks up session

STATELESS (REST):
  Client: "Get my profile"     -> Includes auth token in every request
         Authorization: Bearer eyJhbG...

3. Cacheability

Responses must indicate whether they are cacheable. Proper caching reduces server load and improves performance.

4. Uniform Interface

All resources are accessed through a consistent, standardized interface (HTTP methods, status codes, URIs).

5. Layered System

The client cannot tell whether it is connected directly to the server or through intermediaries (load balancers, caches, gateways).

6. Code on Demand (Optional)

Servers can extend client functionality by sending executable code (rarely used in APIs).

Resources and URIs

In REST, everything is a resource — an entity that can be identified, retrieved, and manipulated. Resources are identified by URIs (Uniform Resource Identifiers).

Resource Naming Conventions

GOOD (nouns, plural, lowercase, kebab-case):
  GET    /api/users
  GET    /api/users/42
  GET    /api/blog-posts
  GET    /api/blog-posts/7/comments

BAD (verbs, singular, camelCase, actions in URL):
  GET    /api/getUsers
  GET    /api/User/42
  GET    /api/blogPosts
  POST   /api/createBlogPost
  POST   /api/users/42/delete

Key Rules

  1. Use nouns, not verbs — HTTP methods provide the verbs
  2. Use plural nouns — /users not /user
  3. Use kebab-case — /blog-posts not /blogPosts
  4. Nest for relationships — /users/42/posts (posts belonging to user 42)
  5. Avoid deep nesting — maximum 2-3 levels deep

Resource Hierarchy

/api/users                      # Collection of users
/api/users/42                   # Single user
/api/users/42/posts             # Posts by user 42
/api/users/42/posts/7           # Post 7 by user 42
/api/users/42/posts/7/comments  # Comments on post 7

# Avoid deeper nesting. Instead:
/api/comments?postId=7          # Query parameter alternative

HTTP Methods as CRUD

HTTP Method  | CRUD Operation | Collection (/users) | Item (/users/42)
-------------|----------------|---------------------|------------------
GET          | Read           | List all users       | Get user 42
POST         | Create         | Create a new user    | (rarely used)
PUT          | Replace        | (rarely used)        | Replace user 42
PATCH        | Update         | (rarely used)        | Update user 42
DELETE       | Delete         | (rarely used)        | Delete user 42

Implementation in Express

import { Router } from 'express';

const router = Router();

// GET /api/users — List users
router.get('/', async (req, res) => {
  const users = await userService.findAll(req.query);
  res.json({ data: users, total: users.length });
});

// GET /api/users/:id — Get single user
router.get('/:id', async (req, res) => {
  const user = await userService.findById(req.params.id);
  if (!user) return res.status(404).json({ error: 'User not found' });
  res.json({ data: user });
});

// POST /api/users — Create user
router.post('/', async (req, res) => {
  const user = await userService.create(req.body);
  res.status(201).json({ data: user });
});

// PUT /api/users/:id — Replace user (full update)
router.put('/:id', async (req, res) => {
  const user = await userService.replace(req.params.id, req.body);
  res.json({ data: user });
});

// PATCH /api/users/:id — Partial update
router.patch('/:id', async (req, res) => {
  const user = await userService.update(req.params.id, req.body);
  res.json({ data: user });
});

// DELETE /api/users/:id — Delete user
router.delete('/:id', async (req, res) => {
  await userService.delete(req.params.id);
  res.status(204).send();
});

PUT vs PATCH

// PUT: Replace the entire resource (all fields required)
// Client sends the complete new representation
PUT /api/users/42
{
  "name": "Alice Smith",      // Required
  "email": "alice@new.com",   // Required
  "role": "admin",            // Required
  "bio": "Updated bio"        // Required (even if unchanged)
}

// PATCH: Partial update (only changed fields)
PATCH /api/users/42
{
  "email": "alice@new.com"    // Only the field being changed
}

Consistent Response Format

Adopt a standard envelope for all responses:

// Success response (single item)
{
  "success": true,
  "data": {
    "id": 42,
    "name": "Alice",
    "email": "alice@example.com"
  }
}

// Success response (collection)
{
  "success": true,
  "data": [...],
  "meta": {
    "total": 150,
    "page": 1,
    "limit": 20,
    "totalPages": 8
  }
}

// Error response
{
  "success": false,
  "error": {
    "message": "User not found",
    "statusCode": 404
  }
}

Filtering, Sorting, and Pagination

Use query parameters for collection modifiers:

GET /api/posts?status=published&author=42           # Filtering
GET /api/posts?sort=createdAt&order=desc             # Sorting
GET /api/posts?page=2&limit=20                       # Pagination
GET /api/posts?fields=id,title,createdAt             # Sparse fieldsets
GET /api/posts?search=express+middleware              # Full-text search

Relationships and Sub-Resources

// A user's posts
GET /api/users/42/posts

// Create a post for user 42
POST /api/users/42/posts

// When the relationship is the primary focus:
GET /api/posts?authorId=42              // Alternative flat approach

// Actions that do not map to CRUD:
POST /api/users/42/activate             // OK for non-CRUD actions
POST /api/orders/99/cancel              // OK for state transitions
POST /api/emails/send                   // OK for process triggers

Idempotency

An operation is idempotent if calling it multiple times produces the same result as calling it once:

GET    /users/42      # Idempotent (safe, no side effects)
PUT    /users/42      # Idempotent (same body -> same result)
DELETE /users/42      # Idempotent (deleting twice = same end state)
PATCH  /users/42      # NOT always idempotent (increment operations)
POST   /users         # NOT idempotent (creates duplicate)

This matters for retries. If a client is unsure whether a request succeeded, it can safely retry idempotent operations.

Common Pitfalls

Using Verbs in URLs

BAD:  POST /api/users/create
      GET  /api/users/getAll
      POST /api/users/42/delete

GOOD: POST   /api/users           (create)
      GET    /api/users           (getAll)
      DELETE /api/users/42        (delete)

Inconsistent Plural/Singular

BAD:  /api/user/42/post
GOOD: /api/users/42/posts

Summary

REST is more than "JSON over HTTP" — it is an architectural style based on resources, statelessness, and a uniform interface. Design resources as nouns in plural form, use HTTP methods for operations, maintain consistent response envelopes, and use query parameters for filtering and pagination. These conventions make your API predictable and self-documenting.

Exercises

  1. Design a REST API for an E-Commerce Platform

    Design the URL structure for an e-commerce API with resources: products, categories, users, orders, and reviews. Define the HTTP methods and expected responses for each endpoint. Include relationships (a product belongs to a category, an order has many items).

    Hint

    Start with the main resources: /api/products, /api/categories. Use nesting for tight relationships: /api/products/:id/reviews. Use query params for loose relationships: /api/orders?userId=42.

  2. Implement a Complete CRUD Router

    Build an Express router for a 'tasks' resource with all CRUD operations, proper status codes, input validation, and consistent response envelopes. Store tasks in memory. Support filtering by status (?status=done) and sorting (?sort=dueDate&order=asc).

    Hint

    Use 200 for reads, 201 for creates, 204 for deletes. Wrap responses in { success: true, data: ... }. Parse and validate query parameters for filtering.

  3. PUT vs PATCH Implementation

    Implement both PUT and PATCH for a user resource. PUT should require all fields and replace the entire object. PATCH should accept partial updates and merge with existing data. Write tests that demonstrate the difference.

    Hint

    For PUT, validate that all required fields are present. For PATCH, use Object.assign() or spread to merge. PUT with missing fields should return 400.

Common Problems & Solutions

1 API feels awkward — many endpoints have verbs in URLs
Cause: Thinking in terms of actions rather than resources. The developer is mapping function names to URLs instead of using HTTP methods.
Solution: Reframe: instead of POST /createUser, think 'I am creating a new user resource' -> POST /users. The HTTP method IS the verb. Resource URLs should be nouns.
2 Deeply nested URLs become unwieldy (/users/42/posts/7/comments/3/likes)
Cause: Over-nesting resources to represent every relationship in the URL hierarchy.
Solution: Limit nesting to 2 levels maximum. Use query parameters for deeper relationships: GET /likes?commentId=3. Or provide a direct route: GET /comments/3/likes.
