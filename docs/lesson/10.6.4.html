Design a Recommendation Engine

Intuition

A recommendation engine is a two-stage system: a broad candidate generation phase that retrieves hundreds of plausible items, followed by a precise ranking phase that scores and orders them using richer features. The cold-start problem is solved not by one technique but by gracefully falling back through a hierarchy of signals: collaborative, content-based, then popularity.

Design a Recommendation Engine

Requirements

  • Generate personalized recommendations for millions of users across items (products, content, people)
  • Combine collaborative filtering, content-based, and popularity signals
  • Serve recommendations with <100ms latency; refresh daily with near real-time boosts
  • Handle cold-start for new users and new items

Architecture

An offline pipeline trains models on user-item interaction data (views, clicks, purchases) using collaborative filtering (matrix factorization, ALS) and content features. Pre-computed recommendations per user are stored in a serving layer. An online layer re-ranks results using real-time signals (session context, recency) before serving.

Key Components

  • Feature Store: Stores user profiles, item features, interaction history for training and serving
  • Training Pipeline: Spark/TensorFlow jobs that train CF and content-based models on interaction matrices
  • Candidate Generation: Retrieves top-N candidates per user from pre-computed lists or ANN (approximate nearest neighbor) index
  • Ranking Service: Re-ranks candidates with a lightweight model using real-time features

Scaling

Pre-compute and cache recommendations per user. Use ANN indexes (FAISS, ScaNN) for real-time candidate retrieval from embedding space. Separate candidate generation (recall) from ranking (precision) to manage compute.

Trade-offs

  • Collaborative filtering is powerful but suffers cold-start; content-based handles new items but lacks serendipity
  • Pre-computed recs are fast but stale; real-time models are fresh but expensive
  • Optimizing for engagement may create filter bubbles; diversity injection hurts click-through but improves satisfaction

Exercises

  1. Implement Matrix Factorization with ALS

    Build a basic Alternating Least Squares recommender on the MovieLens 100K dataset. Split into train/test, compute RMSE, and generate top-10 recommendations for a user.

    Hint

    Initialize user and item factor matrices randomly. Alternate: fix items, solve for users (least squares); fix users, solve for items. Repeat for 10-20 iterations.

  2. Build a Two-Stage Recommend-and-Rank Pipeline

    Implement candidate generation using item-item similarity (cosine on feature vectors) and a ranking stage that re-orders candidates by a weighted score combining similarity, recency, and popularity.

    Hint

    For candidate generation, pre-compute item-item similarity matrix and retrieve top-50 similar items to user's recent interactions. For ranking, combine scores as: alphasimilarity + betarecency + gamma*popularity.

Common Problems & Solutions

1 Recommendations become stale and repetitive for active users
Cause: The offline model only retrains daily, so today's interactions are not reflected in recommendations
Solution: Add a real-time boosting layer that promotes items similar to the user's current session activity, and penalize already-seen items.
2 New items never get recommended because they lack interaction data
Cause: Collaborative filtering cannot score items with no user interactions (cold-start problem)
Solution: Use content-based features (title, category, description embeddings) to place new items in the embedding space, and inject them into candidate generation with an exploration budget.
