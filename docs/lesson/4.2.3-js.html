Circular Dependencies and forwardRef

Intuition

Circular dependencies are like two people trying to go through a revolving door at the same time from opposite sides -- they block each other. forwardRef() is like one person saying 'I'll let you go first, and I'll catch up.' But the best solution is often to find a different door (extract shared logic) so no one blocks anyone.

Circular Dependencies and forwardRef

Introduction

Circular dependencies occur when two or more classes depend on each other, forming a cycle. In NestJS, this can happen at both the module level (Module A imports Module B which imports Module A) and the provider level (Service A injects Service B which injects Service A). While circular dependencies are sometimes unavoidable, they're often a design smell that should be resolved through restructuring.

Types of Circular Dependencies

Module-Level Circular Dependency

UsersModule <------> AuthModule
    |                     |
imports AuthModule   imports UsersModule

UsersModule imports AuthModule for authentication, and AuthModule imports UsersModule to look up users during login.

Provider-Level Circular Dependency

UsersService <------> AuthService
    |                      |
injects AuthService   injects UsersService

UsersService needs AuthService to hash passwords, and AuthService needs UsersService to find users.

The Problem

Without intervention, NestJS can't resolve circular dependencies:

// This FAILS:
@Module({
  imports: [AuthModule], // AuthModule isn't fully defined yet
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}

@Module({
  imports: [UsersModule], // UsersModule isn't fully defined yet
  providers: [AuthService],
  exports: [AuthService],
})
export class AuthModule {}

Error:

Error: A circular dependency has been detected.

Solution: forwardRef()

forwardRef() is a function that returns a reference to a class/module that hasn't been defined yet. NestJS resolves it lazily.

Module-Level forwardRef

import { Module, forwardRef } from '@nestjs/common';

@Module({
  imports: [forwardRef(() => AuthModule)],
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}

@Module({
  imports: [forwardRef(() => UsersModule)],
  providers: [AuthService],
  exports: [AuthService],
})
export class AuthModule {}

Both sides must use forwardRef().

Provider-Level forwardRef

import { Injectable, Inject, forwardRef } from '@nestjs/common';

@Injectable()
export class UsersService {
  constructor(
    @Inject(forwardRef(() => AuthService))
    private authService: AuthService,
  ) {}

  async createUser(dto: CreateUserDto) {
    const hashedPassword = await this.authService.hashPassword(dto.password);
    // ... create user with hashed password
  }
}

@Injectable()
export class AuthService {
  constructor(
    @Inject(forwardRef(() => UsersService))
    private usersService: UsersService,
  ) {}

  async validateUser(email: string, password: string) {
    const user = await this.usersService.findByEmail(email);
    // ... validate password
  }
}

Again, both sides must use forwardRef() with @Inject().

When forwardRef is Needed vs. When to Refactor

Acceptable Circular Dependencies

Sometimes two closely related modules legitimately need each other:

// Auth needs Users to validate credentials
// Users needs Auth to hash passwords on creation
// These are tightly coupled by nature

Better Solution: Extract Shared Logic

Often, a circular dependency indicates a missing abstraction. Extract the shared functionality into a third module:

BEFORE (circular):
  UsersModule <--> AuthModule

AFTER (no cycle):
  UsersModule --> PasswordModule <-- AuthModule

// password/password.module.ts
@Module({
  providers: [PasswordService],
  exports: [PasswordService],
})
export class PasswordModule {}

@Injectable()
export class PasswordService {
  async hash(password: string): Promise<string> {
    return bcrypt.hash(password, 10);
  }

  async compare(password: string, hash: string): Promise<boolean> {
    return bcrypt.compare(password, hash);
  }
}

// users/users.module.ts -- no circular dependency
@Module({
  imports: [PasswordModule],
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}

// auth/auth.module.ts -- no circular dependency
@Module({
  imports: [UsersModule, PasswordModule],
  providers: [AuthService],
  exports: [AuthService],
})
export class AuthModule {}

Use Events to Break Cycles

Another pattern is to use events instead of direct dependencies:

import { EventEmitter2 } from '@nestjs/event-emitter';

@Injectable()
export class UsersService {
  constructor(private eventEmitter: EventEmitter2) {}

  async createUser(dto: CreateUserDto) {
    const user = await this.usersRepo.create(dto);

    // Instead of calling AuthService directly:
    this.eventEmitter.emit('user.created', { userId: user.id });

    return user;
  }
}

@Injectable()
export class AuthService {
  @OnEvent('user.created')
  async handleUserCreated(payload: { userId: string }) {
    // Set up default auth settings for the new user
    await this.createDefaultPermissions(payload.userId);
  }
}

No circular dependency -- both services are decoupled through events.

Use an Interface + Injection Token

// Shared interface
export interface IUsersService {
  findByEmail(email: string): Promise<User | null>;
}

export const USERS_SERVICE = Symbol('USERS_SERVICE');

// Auth module depends on the interface, not the concrete class
@Injectable()
export class AuthService {
  constructor(
    @Inject(USERS_SERVICE) private usersService: IUsersService,
  ) {}
}

// Users module provides the implementation
@Module({
  providers: [
    UsersService,
    { provide: USERS_SERVICE, useExisting: UsersService },
  ],
  exports: [USERS_SERVICE],
})
export class UsersModule {}

Detecting Circular Dependencies

NestJS Tells You

NestJS throws a clear error at startup:

[Nest] ERROR  A circular dependency has been detected.
Please, make sure that each side of a bidirectional
relationship is decorated with "forwardRef()".

Visual Inspection

Draw your module dependency graph. If you can trace a path from a module back to itself, there's a cycle:

A -> B -> C -> A  (CYCLE!)
A -> B -> C       (OK)
A -> B, A -> C    (OK)

Nest CLI Debug

# Start with debug logging to see module resolution
nest start --debug

Multiple Circular Dependencies

If you have a chain: A -> B -> C -> A, you need forwardRef at the point that breaks the cycle:

// Break the cycle at C -> A
@Module({
  imports: [forwardRef(() => AModule)],
})
export class CModule {}

@Module({
  imports: [forwardRef(() => CModule)],
})
export class AModule {}

forwardRef Limitations

  1. Both sides must use forwardRef -- If only one side uses it, resolution still fails.
  2. Doesn't fix design problems -- It patches the symptom, not the cause. Prefer refactoring.
  3. Makes the dependency graph harder to understand -- Future developers may not realize why forwardRef is there.
  4. Can mask deeper issues -- Multiple circular dependencies often indicate poor module boundaries.

Decision Flowchart

Is there a circular dependency?
  |
  +-- No --> You're fine
  |
  +-- Yes --> Can you extract shared logic into a new module?
        |
        +-- Yes --> Do that (preferred)
        |
        +-- No --> Can you use events to decouple?
              |
              +-- Yes --> Use @nestjs/event-emitter
              |
              +-- No --> Use forwardRef() on BOTH sides

Common Pitfalls

  1. Only one side uses forwardRef -- Both the module/provider and its counterpart must use forwardRef().
  2. Using forwardRef as a default -- It should be a last resort, not a habit. Refactor first.
  3. Deeply nested circular chains -- If A -> B -> C -> D -> A, the dependency graph is too tangled. Major refactoring is needed.
  4. Mixing module-level and provider-level forwardRef -- If the circular dependency is between modules, use forwardRef in the imports arrays. If between providers within the same module, use forwardRef in the @Inject() decorators.

    Hint

  2. Refactor to Eliminate Circular Dependency

Common Problems & Solutions

1 forwardRef() is used but circular dependency error persists
Cause: Only one side of the circular relationship uses forwardRef(). Both sides must use it.
Solution: Add forwardRef() to BOTH modules' imports arrays (for module-level circularity) or to BOTH services' @Inject() decorators (for provider-level circularity).

// WRONG: only one side\n@Module({ imports: [forwardRef(() => AuthModule)] })\nexport class UsersModule {}\n\n@Module({ imports: [UsersModule] }) // Missing forwardRef!\nexport class AuthModule {}\n\n// CORRECT: both sides\n@Module({ imports: [forwardRef(() => AuthModule)] })\nexport class UsersModule {}\n\n@Module({ imports: [forwardRef(() => UsersModule)] })\nexport class AuthModule {}

2 Provider is undefined when using forwardRef
Symptoms: The injected service is undefined or throws 'Cannot read property of undefined' at runtime.
Cause: The provider-level forwardRef is used in the constructor type but not with @Inject(). TypeScript's type system doesn't preserve the forwardRef at the type level.
Solution: Always use @Inject(forwardRef(() => ServiceClass)) alongside the parameter type annotation.
3 Application works with forwardRef but tests fail
Cause: The test module doesn't replicate the forwardRef setup, or mock providers aren't configured correctly for the circular dependency.
Solution: In test modules, you don't need forwardRef because you provide mocks directly. Use useValue mocks for both services and remove the circular dependency entirely in tests.
