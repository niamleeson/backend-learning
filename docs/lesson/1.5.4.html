Context Managers and the with Statement

Intuition

Context managers are Python's way of saying 'do this, and no matter what happens, always clean up.' They replace error-prone try/finally patterns with a clean protocol.

Context managers automate resource setup and teardown using the with statement. They guarantee cleanup even when exceptions occur.

# Built-in: file handling
with open("data.txt") as f:
    content = f.read()
# f is automatically closed here, even on error

# Custom context manager using __enter__ / __exit__
class Timer:
    def __enter__(self):
        import time
        self.start = time.perf_counter()
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        import time
        self.elapsed = time.perf_counter() - self.start
        print(f"Elapsed: {self.elapsed:.4f}s")
        return False  # do not suppress exceptions

with Timer() as t:
    sum(range(1_000_000))
  • __enter__ runs at the start of with and its return value is bound by as.
  • __exit__ always runs at the end, receiving exception info (or None if no error).
  • Returning True from __exit__ suppresses the exception — use with extreme caution.
  • The contextlib.contextmanager decorator lets you write context managers as generators.

Exercises

  1. Temporary Directory Context Manager

    Write a context manager TempDir that creates a temporary directory on enter and deletes it on exit. Return the directory path from __enter__.

    Hint

    Use tempfile.mkdtemp() in __enter__ and shutil.rmtree() in __exit__.

  2. Generator-Based Context Manager

    Rewrite the Timer class above as a generator-based context manager using @contextlib.contextmanager.

    Hint

    Use yield to split setup and teardown. Everything before yield is __enter__, after is __exit__.

Common Problems & Solutions

1 Returning `True` from `__exit__` silently swallows all exceptions.
Cause: `True` tells Python the exception was handled and should not propagate.
Solution: Return `False` (or nothing) from `__exit__` unless you intentionally want to suppress specific exceptions.
2 Forgetting `as` in `with open('f') as fh:` — using the file handle outside the `with` block.
Cause: After the `with` block ends, the file is closed; reads/writes will raise `ValueError`.
Solution: Do all file operations inside the `with` block.
