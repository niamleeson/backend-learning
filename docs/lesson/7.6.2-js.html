Graceful Degradation and Health Checks

Intuition

Think of graceful degradation like a restaurant during a power outage. The kitchen gas stove still works, so they can serve hot food, but the blender is down so no smoothies. They tell customers 'limited menu tonight' rather than closing entirely. Health checks are like the health inspector: liveness checks that the building exists, and readiness checks that the kitchen is clean and stocked.

Graceful Degradation and Health Checks

The Difference Between Crashing and Degrading

When a dependency fails, your application has two choices: crash entirely or continue serving requests at reduced functionality. Graceful degradation is the art of designing your system so that partial failures lead to partial -- not total -- outages.

Health checks are the mechanism that lets load balancers, orchestrators (Kubernetes), and monitoring systems know whether your application can handle traffic and, importantly, how well it can handle traffic.

Graceful Degradation Strategies

Strategy 1: Feature Flags and Fallbacks

When a non-critical service fails, disable the feature instead of failing the entire request.

│    Product          Fails!              Slow (3s)      │
│    data             ┌───────┐           ┌─────────┐    │
│    returned         │Return │           │ Return  │    │
│                     │empty  │           │ cached  │    │
│                     │array  │           │ reviews │    │
│                     └───────┘           └─────────┘    │
│                                                        │
│  Result: Page loads with product data, no recs,        │
│  and slightly stale reviews. User is unblocked.        │
└────────────────────────────────────────────────────────┘

const CircuitBreaker = require('opossum');

class ProductPageService {
  constructor(productService, recommendationService, reviewService) {
    this.productService = productService;

    // Non-critical services get circuit breakers with fallbacks
    this.recBreaker = new CircuitBreaker(
      (productId) => recommendationService.getRecommendations(productId),
      { timeout: 2000, errorThresholdPercentage: 50, resetTimeout: 15000 }
    );
    this.recBreaker.fallback(() => []);  // Return empty array

    this.reviewBreaker = new CircuitBreaker(
      (productId) => reviewService.getReviews(productId),
      { timeout: 3000, errorThresholdPercentage: 50, resetTimeout: 15000 }
    );
    this.reviewBreaker.fallback((productId) =>
      this.getCachedReviews(productId)
    );
  }

  async getProductPage(productId) {
    // Core data -- if this fails, the whole request fails
    const product = await this.productService.getProduct(productId);

    // Non-critical data -- fetched concurrently with fallbacks
    const [recommendations, reviews] = await Promise.allSettled([
      this.recBreaker.fire(productId),
      this.reviewBreaker.fire(productId),
    ]);

    return {
      product,
      recommendations:
        recommendations.status === 'fulfilled'
          ? recommendations.value
          : [],
      reviews:
        reviews.status === 'fulfilled'
          ? reviews.value
          : [],
      _degraded: {
        recommendations: recommendations.status === 'rejected',
        reviews: reviews.status === 'rejected',
      },
    };
  }

  async getCachedReviews(productId) {
    // Return from local cache
    const cached = await this.cache.get(`reviews:${productId}`);
    return cached || [];
  }
}

Strategy 2: Load Shedding

When your system is overloaded, it is better to reject some requests cleanly than to serve all of them poorly:

class LoadShedder {
  constructor({ maxConcurrent = 100, maxQueueSize = 50 }) {
    this.maxConcurrent = maxConcurrent;
    this.maxQueueSize = maxQueueSize;
    this.activeRequests = 0;
    this.queuedRequests = 0;
  }

  middleware() {
    return (req, res, next) => {
      if (this.activeRequests >= this.maxConcurrent) {
        if (this.queuedRequests >= this.maxQueueSize) {
          // Shed load: reject with 503
          return res.status(503).json({
            error: 'Service overloaded',
            retryAfter: 5,
          });
        }
        this.queuedRequests++;
      }

      this.activeRequests++;
      res.on('finish', () => {
        this.activeRequests--;
      });
      next();
    };
  }
}

const shedder = new LoadShedder({ maxConcurrent: 200, maxQueueSize: 100 });
app.use(shedder.middleware());

Strategy 3: Reduced Functionality Mode

Switch entire subsystems to a degraded mode:

const degradedFeatures = new Map();

function setDegraded(feature, isDegraded) {
  degradedFeatures.set(feature, isDegraded);
}

function isDegraded(feature) {
  return degradedFeatures.get(feature) || false;
}

// Middleware to auto-degrade based on circuit breaker state
function autoDegradation(breakerMap) {
  for (const [feature, breaker] of breakerMap) {
    breaker.on('open', () => setDegraded(feature, true));
    breaker.on('close', () => setDegraded(feature, false));
  }
}

// Usage in route
app.get('/search', async (req, res) => {
  if (isDegraded('search-service')) {
    // Return cached/popular results instead of live search
    const popular = await getCachedPopularResults();
    return res.json({
      results: popular,
      _notice: 'Showing popular results. Full search temporarily unavailable.',
    });
  }

  const results = await searchService.search(req.query.q);
  res.json({ results });
});

Health Checks

Why Health Checks Matter

Health checks serve three purposes:

  1. Liveness -- Is the process running and not deadlocked?
  2. Readiness -- Can the application handle traffic? (database connected, caches warm, etc.)
  3. Startup -- Has the application finished initializing?

Health Checks with @godaddy/terminus

Terminus is the standard library for graceful shutdown and health checks in Node.js HTTP servers:

npm install @godaddy/terminus

const http = require('http');
const express = require('express');
const { createTerminus } = require('@godaddy/terminus');
const { PrismaClient } = require('@prisma/client');
const Redis = require('ioredis');

const app = express();
const prisma = new PrismaClient();
const redis = new Redis();

// Your Express routes
app.get('/', (req, res) => res.json({ status: 'ok' }));

const server = http.createServer(app);

// Health check function: checks all dependencies
async function onHealthCheck() {
  const checks = {};

  // Check database connectivity
  try {
    await prisma.$queryRaw`SELECT 1`;
    checks.database = { status: 'healthy' };
  } catch (err) {
    checks.database = { status: 'unhealthy', error: err.message };
    throw new Error('Database health check failed');
  }

  // Check Redis connectivity
  try {
    await redis.ping();
    checks.redis = { status: 'healthy' };
  } catch (err) {
    checks.redis = { status: 'unhealthy', error: err.message };
    throw new Error('Redis health check failed');
  }

  // Check disk space (simplified)
  const { freemem, totalmem } = require('os');
  const memUsagePercent = ((totalmem() - freemem()) / totalmem()) * 100;
  checks.memory = {
    status: memUsagePercent < 90 ? 'healthy' : 'warning',
    usagePercent: memUsagePercent.toFixed(1),
  };

  return checks;
}

// Graceful shutdown: clean up resources
async function onSignal() {
  console.log('Server is shutting down...');
  await prisma.$disconnect();
  await redis.quit();
  console.log('Cleanup complete.');
}

// Before shutdown: stop accepting new requests
async function beforeShutdown() {
  // Wait for load balancer to deregister this instance
  console.log('Waiting for load balancer deregistration...');
  await new Promise((resolve) => setTimeout(resolve, 5000));
}

createTerminus(server, {
  healthChecks: {
    '/healthz': onHealthCheck,      // Liveness probe
    '/readyz': onHealthCheck,       // Readiness probe
    verbatim: true,                 // Return the checks object as JSON
  },
  onSignal,
  beforeShutdown,
  timeout: 30000,                   // Force shutdown after 30s
  logger: console.error,
});

server.listen(3000, () => {
  console.log('Server listening on port 3000');
});

Custom Health Check Endpoints (Without Terminus)

If you prefer building health checks manually:

const express = require('express');
const app = express();

// Simple liveness check
app.get('/healthz', (req, res) => {
  res.status(200).json({ status: 'alive', timestamp: new Date().toISOString() });
});

// Detailed readiness check
app.get('/readyz', async (req, res) => {
  const results = {
    status: 'ready',
    timestamp: new Date().toISOString(),
    checks: {},
  };

  let isReady = true;

  // Check database
  try {
    const start = Date.now();
    await prisma.$queryRaw`SELECT 1`;
    const latency = Date.now() - start;
    results.checks.database = {
      status: 'up',
      latencyMs: latency,
    };
    if (latency > 1000) {
      results.checks.database.status = 'degraded';
    }
  } catch (err) {
    results.checks.database = { status: 'down', error: err.message };
    isReady = false;
  }

  // Check Redis
  try {
    const start = Date.now();
    await redis.ping();
    const latency = Date.now() - start;
    results.checks.redis = { status: 'up', latencyMs: latency };
  } catch (err) {
    results.checks.redis = { status: 'down', error: err.message };
    isReady = false;
  }

  // Check external API
  try {
    const start = Date.now();
    await axios.get('https://api.payment.com/health', { timeout: 2000 });
    results.checks.paymentApi = { status: 'up', latencyMs: Date.now() - start };
  } catch (err) {
    results.checks.paymentApi = { status: 'down' };
    // Not marking isReady = false because payment is non-critical
    // for readiness (we have a circuit breaker + fallback)
  }

  // Check circuit breaker states
  results.checks.circuitBreakers = {
    userService: userBreaker.opened ? 'open' : 'closed',
    paymentService: paymentBreaker.opened ? 'open' : 'closed',
  };

  results.status = isReady ? 'ready' : 'not_ready';
  res.status(isReady ? 200 : 503).json(results);
});

Deep vs Shallow Health Checks

A shallow health check just confirms the process is running. A deep health check verifies all dependencies:

// Shallow: for liveness probes (should NEVER call external services)
app.get('/healthz', (req, res) => {
  res.status(200).send('OK');
});

// Deep: for readiness probes (checks dependencies)
app.get('/readyz', async (req, res) => {
  // ...check database, Redis, external APIs
});

Important: Liveness probes should be shallow. If your liveness probe depends on an external database and that database goes down, Kubernetes will restart your pod -- which solves nothing since the database is still down. This creates a restart loop.

         ┌───────────────────┐          ┌───────────────────┐
         │   Liveness Probe   │          │   Liveness Probe   │
         │   checks DB ────X  │          │   returns 200 OK   │
         │                    │          │   (no ext. deps)   │
         │   DB down =>       │          │                    │
         │   probe fails =>   │          │   Readiness Probe  │
         │   pod restarts =>  │          │   checks DB ────X  │
         │   DB still down => │          │                    │
         │   probe fails =>   │          │   DB down =>       │
         │   restart loop!    │          │   pod removed from │
         │                    │          │   load balancer     │

Graceful Shutdown

When your application receives a termination signal (SIGTERM from Kubernetes, Ctrl+C in development), it should:

  1. Stop accepting new connections
  2. Finish processing in-flight requests
  3. Close database connections and other resources
  4. Exit cleanly
const server = app.listen(3000);

let isShuttingDown = false;

// Middleware to reject new requests during shutdown
app.use((req, res, next) => {
  if (isShuttingDown) {
    res.set('Connection', 'close');
    return res.status(503).json({ error: 'Server is shutting down' });
  }
  next();
});

async function gracefulShutdown(signal) {
  console.log(`Received ${signal}. Starting graceful shutdown...`);
  isShuttingDown = true;

  // Give load balancer time to remove this instance
  await new Promise((resolve) => setTimeout(resolve, 5000));

  // Stop accepting new connections
  server.close(async () => {
    console.log('HTTP server closed.');

    try {
      // Close database connections
      await prisma.$disconnect();
      console.log('Database disconnected.');

      // Close Redis
      await redis.quit();
      console.log('Redis disconnected.');

      // Close message queue connections
      await messageQueue.close();
      console.log('Message queue disconnected.');

      process.exit(0);
    } catch (err) {
      console.error('Error during shutdown:', err);
      process.exit(1);
    }
  });

  // Force shutdown after timeout
  setTimeout(() => {
    console.error('Forced shutdown after timeout');
    process.exit(1);
  }, 30000);
}

process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
process.on('SIGINT', () => gracefulShutdown('SIGINT'));

Dependency Health Monitoring

Track the health of each dependency and use it to make degradation decisions:

class DependencyMonitor {
  constructor() {
    this.dependencies = new Map();
  }

  register(name, checkFn, { critical = true, intervalMs = 10000 } = {}) {
    const dep = {
      name,
      checkFn,
      critical,
      status: 'unknown',
      lastCheck: null,
      lastError: null,
      consecutiveFailures: 0,
    };

    this.dependencies.set(name, dep);

    // Periodic health checking
    setInterval(async () => {
      try {
        const start = Date.now();
        await checkFn();
        dep.status = 'healthy';
        dep.latencyMs = Date.now() - start;
        dep.consecutiveFailures = 0;
        dep.lastError = null;
      } catch (err) {
        dep.status = 'unhealthy';
        dep.consecutiveFailures++;
        dep.lastError = err.message;
      }
      dep.lastCheck = new Date().toISOString();
    }, intervalMs);
  }

  isSystemHealthy() {
    for (const dep of this.dependencies.values()) {
      if (dep.critical && dep.status === 'unhealthy') {
        return false;
      }
    }
    return true;
  }

  getReport() {
    const report = {};
    for (const [name, dep] of this.dependencies) {
      report[name] = {
        status: dep.status,
        critical: dep.critical,
        lastCheck: dep.lastCheck,
        consecutiveFailures: dep.consecutiveFailures,
        latencyMs: dep.latencyMs,
        lastError: dep.lastError,
      };
    }
    return report;
  }
}

// Setup
const monitor = new DependencyMonitor();

monitor.register('postgres', async () => {
  await prisma.$queryRaw`SELECT 1`;
}, { critical: true, intervalMs: 5000 });

monitor.register('redis', async () => {
  await redis.ping();
}, { critical: true, intervalMs: 5000 });

monitor.register('email-service', async () => {
  await axios.get('https://email-service.internal/healthz', { timeout: 2000 });
}, { critical: false, intervalMs: 30000 });

// Use in health endpoint
app.get('/readyz', (req, res) => {
  const isHealthy = monitor.isSystemHealthy();
  res.status(isHealthy ? 200 : 503).json({
    status: isHealthy ? 'ready' : 'not_ready',
    dependencies: monitor.getReport(),
  });
});

Common Problems & Solutions

1 Liveness probe depends on the database and causes restart loops
Cause: The liveness health check queries the database. When the database goes down, the liveness probe fails, Kubernetes restarts the pod, the new pod also cannot reach the database, the probe fails again, and the pod enters CrashLoopBackOff.
Solution: Separate liveness from readiness. Liveness probes should only verify the process is running (return 200 with no external calls). Readiness probes check dependencies. When the database is down, the readiness probe fails and the pod is removed from the load balancer, but it is not restarted. Once the database recovers, the readiness probe passes and the pod is added back.
2 Graceful shutdown is interrupted, leaving database connections open
Cause: The SIGTERM handler does not wait for in-flight requests to complete before closing connections, or there is no forced shutdown timeout so the process hangs indefinitely waiting for a stuck connection.
Solution: Use @godaddy/terminus or implement a two-phase shutdown: first stop accepting new connections (server.close()), then wait for in-flight requests to drain, then close database and Redis connections. Always set a maximum timeout (e.g., 30 seconds) and force process.exit(1) if cleanup has not finished.
3 Health check endpoint exposes sensitive internal information
Cause: The health check returns detailed error messages, stack traces, or internal service URLs that could aid an attacker.
Solution: Return detailed health information only for internal health check paths (e.g., behind a VPC or protected by auth). Public-facing health endpoints should return minimal information: just the status code (200 or 503) and a simple status string. Use different endpoints for different audiences: /healthz (public, minimal) and /internal/health (detailed, internal only).
