Graceful Degradation and Health Checks

Intuition

Think of graceful degradation like a restaurant during a kitchen equipment failure. If the oven breaks, the restaurant does not close entirely — it stops offering baked dishes and promotes stovetop meals instead. Health checks are like the host at the front door who knows which tables are available: they direct new guests only to sections that are ready to serve, and if a section is being cleaned, no one is seated there until it is ready again.

Graceful Degradation and Health Checks

In production systems, failure is not a question of if but when. Databases go down, external APIs become unresponsive, memory runs out, and network partitions occur. The difference between a resilient system and a fragile one lies in how it behaves when things go wrong. Graceful degradation ensures your application continues to provide value — even if reduced — rather than failing catastrophically. Health checks give orchestrators and load balancers the information they need to route traffic intelligently and restart unhealthy instances.

This lesson covers the strategies, patterns, and implementation details you need to build backend services that bend rather than break.

What Is Graceful Degradation?

Graceful degradation is the practice of designing systems to continue operating at reduced functionality when some component fails. Instead of returning a 500 error to every user when a recommendation engine goes down, you serve a static list of popular items. Instead of blocking all writes when the cache is unavailable, you write directly to the database and accept higher latency.

The key principle: partial service is almost always better than no service.

Degradation Strategies

1. Fallback Responses

The simplest form of degradation is returning a pre-computed or default response when the primary data source is unavailable.

import redis
import json
from typing import Optional

class ProductRecommendationService:
    def __init__(self, redis_client: redis.Redis):
        self.redis_client = redis_client
        self.default_recommendations = [
            {"id": "prod_001", "name": "Best Seller A", "score": 0.95},
            {"id": "prod_002", "name": "Best Seller B", "score": 0.90},
            {"id": "prod_003", "name": "Best Seller C", "score": 0.85},
        ]

    def get_recommendations(self, user_id: str) -> dict:
        try:
            cached = self.redis_client.get(f"recs:{user_id}")
            if cached:
                return {
                    "recommendations": json.loads(cached),
                    "source": "personalized",
                    "degraded": False,
                }
        except redis.ConnectionError:
            pass  # Fall through to default
        except redis.TimeoutError:
            pass  # Fall through to default

        return {
            "recommendations": self.default_recommendations,
            "source": "default",
            "degraded": True,
        }

Notice the degraded flag in the response. This is important — downstream consumers and monitoring systems need to know when they are receiving degraded data.

2. Feature Flags for Degradation

Feature flags let you toggle functionality on and off without deploying new code. During incidents, you can disable expensive features to reduce load.

class FeatureFlags:
    """Simple in-memory feature flag system with remote override support."""

    def __init__(self):
        self._flags = {
            "enable_recommendations": True,
            "enable_search_autocomplete": True,
            "enable_real_time_notifications": True,
            "enable_image_processing": True,
            "read_only_mode": False,
        }

    def is_enabled(self, flag_name: str, default: bool = False) -> bool:
        return self._flags.get(flag_name, default)

    def set_flag(self, flag_name: str, value: bool):
        self._flags[flag_name] = value


feature_flags = FeatureFlags()


def handle_product_page(product_id: str) -> dict:
    product = get_product(product_id)  # Core functionality, always attempted

    response = {"product": product, "degraded_features": []}

    if feature_flags.is_enabled("enable_recommendations"):
        try:
            response["recommendations"] = get_recommendations(product_id)
        except Exception:
            response["degraded_features"].append("recommendations")
            response["recommendations"] = []
    else:
        response["degraded_features"].append("recommendations")
        response["recommendations"] = []

    if feature_flags.is_enabled("enable_search_autocomplete"):
        response["autocomplete_enabled"] = True
    else:
        response["autocomplete_enabled"] = False
        response["degraded_features"].append("autocomplete")

    return response

3. Read-Only Mode

When your write path is compromised (database primary is down, message queue is full), switching to read-only mode preserves the ability to serve existing data.

from functools import wraps
from flask import Flask, jsonify, request

app = Flask(__name__)

class AppState:
    read_only = False

def reject_writes_in_read_only(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        if AppState.read_only and request.method in ("POST", "PUT", "PATCH", "DELETE"):
            return jsonify({
                "error": "Service is in read-only mode",
                "code": "READ_ONLY_MODE",
                "retry_after": 300,
            }), 503
        return f(*args, **kwargs)
    return decorated

@app.route("/api/orders", methods=["GET", "POST"])
@reject_writes_in_read_only
def orders():
    if request.method == "GET":
        return jsonify({"orders": get_all_orders()})
    else:
        order = create_order(request.json)
        return jsonify(order), 201

@app.route("/admin/read-only", methods=["POST"])
def toggle_read_only():
    AppState.read_only = request.json.get("enabled", False)
    return jsonify({"read_only": AppState.read_only})

Health Checks: The Foundation of Observability

Health checks are HTTP endpoints that report the status of your service. Orchestrators like Kubernetes, load balancers like NGINX or AWS ALB, and monitoring systems poll these endpoints to make routing and restart decisions.

The Three Types of Probes

Kubernetes defines three probe types, and this model has become the industry standard even outside Kubernetes:

Probe Question It Answers Failure Action
Liveness Is the process alive and not deadlocked? Restart the container
Readiness Can this instance handle traffic right now? Remove from load balancer
Startup Has the app finished initializing? Wait (don't check liveness yet)

Getting these wrong has serious consequences. If your liveness probe checks database connectivity and the database goes down, Kubernetes will restart all your pods — making things dramatically worse.

The Golden Rule of Liveness Probes

Liveness probes should only check if the process itself is healthy. They should NOT check external dependencies. A liveness probe failure causes a restart, and restarting a pod won't fix a broken database.

Implementing Health Checks in Flask

import time
import threading
import psycopg2
import redis
from flask import Flask, jsonify

app = Flask(__name__)

# Track application state
class HealthState:
    started = False
    start_time = time.time()
    last_deadlock_check = time.time()
    is_shutting_down = False


# --- Liveness Probe ---
@app.route("/health/live")
def liveness():
    """Only checks if the process is alive and responsive.
    Does NOT check external dependencies."""
    if HealthState.is_shutting_down:
        return jsonify({"status": "shutting_down"}), 503

    # Optional: check that the main thread isn't deadlocked
    # by verifying a background heartbeat is recent
    if time.time() - HealthState.last_deadlock_check > 30:
        return jsonify({"status": "deadlocked"}), 503

    return jsonify({
        "status": "alive",
        "uptime_seconds": int(time.time() - HealthState.start_time),
    }), 200


# --- Readiness Probe ---
@app.route("/health/ready")
def readiness():
    """Checks if the service can handle requests.
    Includes dependency checks."""
    checks = {}
    overall_healthy = True

    # Check database
    try:
        conn = psycopg2.connect(
            "postgresql://user:pass@db:5432/myapp",
            connect_timeout=2
        )
        conn.execute("SELECT 1")
        conn.close()
        checks["database"] = {"status": "healthy", "latency_ms": 5}
    except Exception as e:
        checks["database"] = {"status": "unhealthy", "error": str(e)}
        overall_healthy = False

    # Check Redis cache
    try:
        r = redis.Redis(host="redis", port=6379, socket_timeout=2)
        r.ping()
        checks["cache"] = {"status": "healthy"}
    except Exception as e:
        checks["cache"] = {"status": "unhealthy", "error": str(e)}
        # Cache might be optional — decide if this makes us unready
        # overall_healthy = False  # Uncomment if cache is critical

    status_code = 200 if overall_healthy else 503
    return jsonify({
        "status": "ready" if overall_healthy else "not_ready",
        "checks": checks,
    }), status_code


# --- Startup Probe ---
@app.route("/health/startup")
def startup():
    """Checks if the app has finished initializing."""
    if HealthState.started:
        return jsonify({"status": "started"}), 200
    return jsonify({"status": "starting"}), 503


# Background heartbeat to detect deadlocks
def heartbeat_loop():
    while True:
        HealthState.last_deadlock_check = time.time()
        time.sleep(10)

threading.Thread(target=heartbeat_loop, daemon=True).start()

Implementing Health Checks in FastAPI

FastAPI's dependency injection and async support make health checks particularly elegant.

import asyncio
import time
from contextlib import asynccontextmanager
from enum import Enum
from fastapi import FastAPI, Depends
from pydantic import BaseModel
import asyncpg
import aioredis


class HealthStatus(str, Enum):
    HEALTHY = "healthy"
    UNHEALTHY = "unhealthy"
    DEGRADED = "degraded"


class ComponentHealth(BaseModel):
    name: str
    status: HealthStatus
    latency_ms: float | None = None
    error: str | None = None


class HealthResponse(BaseModel):
    status: HealthStatus
    components: list[ComponentHealth]
    timestamp: float


class DependencyChecker:
    def __init__(self):
        self.db_pool: asyncpg.Pool | None = None
        self.redis: aioredis.Redis | None = None
        self.started = False

    async def check_database(self) -> ComponentHealth:
        start = time.monotonic()
        try:
            async with self.db_pool.acquire(timeout=2) as conn:
                await conn.fetchval("SELECT 1")
            latency = (time.monotonic() - start) * 1000
            return ComponentHealth(
                name="database",
                status=HealthStatus.HEALTHY,
                latency_ms=round(latency, 2),
            )
        except Exception as e:
            return ComponentHealth(
                name="database",
                status=HealthStatus.UNHEALTHY,
                error=str(e),
            )

    async def check_redis(self) -> ComponentHealth:
        start = time.monotonic()
        try:
            await asyncio.wait_for(self.redis.ping(), timeout=2)
            latency = (time.monotonic() - start) * 1000
            return ComponentHealth(
                name="cache",
                status=HealthStatus.HEALTHY,
                latency_ms=round(latency, 2),
            )
        except Exception as e:
            return ComponentHealth(
                name="cache",
                status=HealthStatus.UNHEALTHY,
                error=str(e),
            )

    async def check_all(self) -> HealthResponse:
        results = await asyncio.gather(
            self.check_database(),
            self.check_redis(),
        )

        # Determine overall status
        statuses = [r.status for r in results]
        if all(s == HealthStatus.HEALTHY for s in statuses):
            overall = HealthStatus.HEALTHY
        elif any(s == HealthStatus.UNHEALTHY for s in statuses):
            # If only optional deps are down, we're degraded not unhealthy
            critical_unhealthy = any(
                r.status == HealthStatus.UNHEALTHY
                for r in results
                if r.name == "database"  # database is critical
            )
            overall = HealthStatus.UNHEALTHY if critical_unhealthy else HealthStatus.DEGRADED
        else:
            overall = HealthStatus.DEGRADED

        return HealthResponse(
            status=overall,
            components=list(results),
            timestamp=time.time(),
        )


checker = DependencyChecker()

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup
    checker.db_pool = await asyncpg.create_pool("postgresql://user:pass@db/myapp")
    checker.redis = aioredis.from_url("redis://redis:6379")
    checker.started = True
    yield
    # Shutdown
    await checker.db_pool.close()
    await checker.redis.close()

app = FastAPI(lifespan=lifespan)

@app.get("/health/live")
async def liveness():
    return {"status": "alive"}

@app.get("/health/ready", response_model=HealthResponse)
async def readiness():
    health = await checker.check_all()
    if health.status == HealthStatus.UNHEALTHY:
        from fastapi.responses import JSONResponse
        return JSONResponse(content=health.model_dump(), status_code=503)
    return health

@app.get("/health/startup")
async def startup():
    if checker.started:
        return {"status": "started"}
    from fastapi.responses import JSONResponse
    return JSONResponse(content={"status": "starting"}, status_code=503)

Circuit Breakers for Degradation

Circuit breakers prevent cascading failures by stopping calls to a failing dependency after a threshold of errors. They transition through three states: Closed (normal operation), Open (all calls rejected immediately), and Half-Open (a small number of test calls allowed).

import time
from enum import Enum
from threading import Lock


class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"


class CircuitBreaker:
    def __init__(
        self,
        name: str,
        failure_threshold: int = 5,
        recovery_timeout: float = 30.0,
        half_open_max_calls: int = 3,
    ):
        self.name = name
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.half_open_max_calls = half_open_max_calls

        self._state = CircuitState.CLOSED
        self._failure_count = 0
        self._success_count = 0
        self._last_failure_time = 0
        self._half_open_calls = 0
        self._lock = Lock()

    @property
    def state(self) -> CircuitState:
        with self._lock:
            if self._state == CircuitState.OPEN:
                if time.time() - self._last_failure_time > self.recovery_timeout:
                    self._state = CircuitState.HALF_OPEN
                    self._half_open_calls = 0
            return self._state

    def can_execute(self) -> bool:
        state = self.state
        if state == CircuitState.CLOSED:
            return True
        if state == CircuitState.HALF_OPEN:
            with self._lock:
                if self._half_open_calls < self.half_open_max_calls:
                    self._half_open_calls += 1
                    return True
            return False
        return False  # OPEN

    def record_success(self):
        with self._lock:
            if self._state == CircuitState.HALF_OPEN:
                self._success_count += 1
                if self._success_count >= self.half_open_max_calls:
                    self._state = CircuitState.CLOSED
                    self._failure_count = 0
                    self._success_count = 0
            else:
                self._failure_count = 0

    def record_failure(self):
        with self._lock:
            self._failure_count += 1
            self._last_failure_time = time.time()
            if self._state == CircuitState.HALF_OPEN:
                self._state = CircuitState.OPEN
            elif self._failure_count >= self.failure_threshold:
                self._state = CircuitState.OPEN


# Usage with degradation
recommendation_breaker = CircuitBreaker("recommendations", failure_threshold=5)

def get_recommendations_with_fallback(user_id: str) -> dict:
    if not recommendation_breaker.can_execute():
        return {"items": get_popular_items(), "source": "fallback"}

    try:
        result = call_recommendation_service(user_id)
        recommendation_breaker.record_success()
        return {"items": result, "source": "personalized"}
    except Exception:
        recommendation_breaker.record_failure()
        return {"items": get_popular_items(), "source": "fallback"}

Load Shedding

Load shedding is the practice of intentionally dropping requests when the system is overloaded, to protect the requests you can serve. It is better to successfully serve 80% of requests and reject 20% than to serve 0% because the system collapsed.

import time
from collections import deque
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse

app = FastAPI()

class LoadShedder:
    def __init__(self, max_concurrent: int = 100, max_queue_time_ms: float = 500):
        self.max_concurrent = max_concurrent
        self.max_queue_time_ms = max_queue_time_ms
        self._active_requests = 0
        self._lock = __import__("threading").Lock()

    def try_acquire(self) -> bool:
        with self._lock:
            if self._active_requests >= self.max_concurrent:
                return False
            self._active_requests += 1
            return True

    def release(self):
        with self._lock:
            self._active_requests -= 1


shedder = LoadShedder(max_concurrent=100)

@app.middleware("http")
async def load_shedding_middleware(request: Request, call_next):
    request_start = time.monotonic()

    if not shedder.try_acquire():
        return JSONResponse(
            status_code=503,
            content={
                "error": "Service overloaded",
                "code": "LOAD_SHED",
                "retry_after": 5,
            },
            headers={"Retry-After": "5"},
        )

    try:
        response = await call_next(request)

        # Check if time in queue was too long (request already stale)
        elapsed_ms = (time.monotonic() - request_start) * 1000
        if elapsed_ms > shedder.max_queue_time_ms:
            return JSONResponse(
                status_code=503,
                content={"error": "Request timeout in queue"},
                headers={"Retry-After": "5"},
            )

        return response
    finally:
        shedder.release()

Health Check Aggregation Patterns

In microservice architectures, you often need a unified view of system health. The aggregator pattern collects health from multiple services and presents a single dashboard.

import asyncio
import aiohttp
from dataclasses import dataclass


@dataclass
class ServiceEndpoint:
    name: str
    url: str
    critical: bool = True


SERVICES = [
    ServiceEndpoint("user-service", "http://user-svc:8000/health/ready", critical=True),
    ServiceEndpoint("order-service", "http://order-svc:8000/health/ready", critical=True),
    ServiceEndpoint("notification-service", "http://notif-svc:8000/health/ready", critical=False),
    ServiceEndpoint("analytics-service", "http://analytics-svc:8000/health/ready", critical=False),
]


async def check_service(session: aiohttp.ClientSession, svc: ServiceEndpoint) -> dict:
    try:
        async with session.get(svc.url, timeout=aiohttp.ClientTimeout(total=3)) as resp:
            body = await resp.json()
            return {
                "name": svc.name,
                "status": "healthy" if resp.status == 200 else "unhealthy",
                "critical": svc.critical,
                "details": body,
            }
    except Exception as e:
        return {
            "name": svc.name,
            "status": "unhealthy",
            "critical": svc.critical,
            "error": str(e),
        }


async def aggregate_health() -> dict:
    async with aiohttp.ClientSession() as session:
        results = await asyncio.gather(
            *[check_service(session, svc) for svc in SERVICES]
        )

    critical_healthy = all(
        r["status"] == "healthy" for r in results if r["critical"]
    )
    all_healthy = all(r["status"] == "healthy" for r in results)

    if all_healthy:
        overall = "healthy"
    elif critical_healthy:
        overall = "degraded"
    else:
        overall = "unhealthy"

    return {
        "status": overall,
        "services": results,
    }

Real-World Degradation Examples

Netflix

Netflix is famous for its approach to graceful degradation. When the personalization service is down, users see a generic trending list. When a regional CDN fails, traffic is rerouted to another region with slightly higher latency. Their Chaos Monkey tool randomly kills services in production to ensure degradation paths are always tested.

Amazon

Amazon's retail site has hundreds of microservices rendering different page sections. If the "customers also bought" service is slow, the page renders without that section. Each section has a timeout budget (e.g., 200ms), and any component that exceeds it is dropped from the response.

Twitter/X

During traffic spikes (major events), Twitter historically shed non-essential features like the "Who to Follow" sidebar, trending topic personalization, and real-time notification counts. Core functionality — posting and reading tweets — was prioritized.

Best Practices Checklist

  1. Keep liveness probes simple. Return 200 if the process is alive. Nothing more.
  2. Use readiness probes for dependency checks. This removes the pod from the load balancer without restarting it.
  3. Set appropriate timeouts. Health check timeouts should be much shorter than your normal request timeouts. A 2-3 second timeout is typical.
  4. Include latency in health responses. A database that responds in 5 seconds is technically alive but effectively unusable.
  5. Distinguish critical from optional dependencies. Not every dependency needs to make you unready.
  6. Add the Retry-After header when returning 503. Well-behaved clients will respect it.
  7. Test your degradation paths. Use chaos engineering to verify fallbacks work. Untested fallback code is unreliable code.
  8. Monitor degraded state. Track metrics on how often you serve degraded responses. If it is constant, you have a reliability problem, not a successful degradation.
  9. Implement graceful shutdown. When your process receives SIGTERM, stop accepting new requests, finish in-flight ones, then exit. This prevents dropped connections during deployments.
import signal
import sys

def graceful_shutdown(signum, frame):
    HealthState.is_shutting_down = True
    # Give in-flight requests time to complete
    time.sleep(10)
    sys.exit(0)

signal.signal(signal.SIGTERM, graceful_shutdown)

Summary

Graceful degradation and health checks are two sides of the same coin. Health checks communicate your service's current capability to the infrastructure, while degradation strategies define what your service does when that capability is reduced. Together, they form the backbone of resilient backend systems. A service that degrades gracefully under pressure earns far more trust than one that promises perfection but shatters at the first sign of trouble.

Common Problems & Solutions

1 Kubernetes keeps restarting pods even though the application code is fine
Symptoms: Pods show CrashLoopBackOff status. Pod restart counts are high. All pods restart simultaneously during a database outage. kubectl describe pod shows liveness probe failed messages.
Cause: The liveness probe checks external dependencies (database, cache, external API). When the dependency goes down, the liveness probe fails, triggering a pod restart. But restarting the pod cannot fix an external dependency, creating a crash loop that makes the outage worse.
Solution: Liveness probes should only verify that the application process itself is healthy — it can respond to HTTP requests and is not deadlocked. Move all dependency checks to the readiness probe. A readiness failure removes the pod from the load balancer without restarting it, which is the correct behavior when a dependency is down.

BAD: liveness probe checks database

@app.route('/health/live') def liveness(): db.execute('SELECT 1') # If DB is down, pod restarts! return 'OK'

GOOD: liveness only checks process health

@app.route('/health/live') def liveness(): return jsonify({'status': 'alive'}), 200

Dependency checks go in readiness

@app.route('/health/ready') def readiness(): try: db.execute('SELECT 1') return jsonify({'status': 'ready'}), 200 except Exception: return jsonify({'status': 'not_ready'}), 503

2 Health check endpoint causes cascading failures under load
Cause: The health check endpoint performs expensive operations like full database queries, connection pool creation, or comprehensive test suites. Under high load, the health checks consume resources needed by actual application requests, or they time out because the application is already under stress, causing the orchestrator to take the instance out of service — increasing load on remaining instances and cascading the failure.
Solution: Keep health checks lightweight. Use connection pool pings instead of full queries. Cache health check results for a few seconds so repeated probe calls do not multiply load. Set a separate, shorter timeout for health check operations than for normal requests. Consider using a dedicated thread or connection for health checks that is isolated from the main request pool.

BAD: expensive health check

@app.route('/health/ready') def readiness(): # Creates new connection each time conn = psycopg2.connect(DATABASE_URL) conn.execute('SELECT count(*) FROM large_table') conn.close() return 'OK'

3 Degraded responses are served permanently and nobody notices
Cause: Fallback responses work silently. There is no monitoring, alerting, or logging when the system enters degraded mode. The system appears healthy from the outside while serving stale or generic data indefinitely.
Solution: Instrument every degradation path with metrics and alerts. Track the percentage of degraded responses per endpoint. Set alerts for when degradation exceeds a threshold or duration. Include a 'degraded' flag in API responses so clients and monitoring dashboards can detect it. Log every state transition (healthy to degraded, degraded to healthy) with context about which dependency triggered it.
