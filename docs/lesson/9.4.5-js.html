Design a Metrics/Monitoring System (Datadog)

Intuition

Think of a metrics system like a network of weather stations. Each station (agent) measures temperature, wind, and humidity (metrics) every few seconds and sends readings to a central weather service (ingestion pipeline). The service stores fine-grained data for recent days but only daily averages for years past (downsampling). Meteorologists (users) view dashboards of charts and set up storm warnings (alerts) that trigger when thresholds are crossed. The key challenge is that there are millions of stations reporting simultaneously.

Design a Metrics/Monitoring System (Datadog)

Introduction

A metrics and monitoring system collects, stores, and visualizes time-series data from applications and infrastructure. Systems like Datadog, Prometheus, Grafana, and New Relic are essential for operating production services. This design problem tests your understanding of time-series databases, data ingestion at scale, aggregation pipelines, alerting, and dashboard rendering.

Step 1: Requirements Gathering

Functional Requirements

  • Ingest metrics from thousands of hosts and services (CPU, memory, disk, custom app metrics)
  • Support multiple metric types: counters, gauges, histograms, timers
  • Store metrics with configurable retention (7 days at 1-second granularity, 1 year at 1-minute, 5 years at 1-hour)
  • Query metrics with filters, grouping, and aggregation functions (avg, sum, max, min, percentiles)
  • Create dashboards with real-time charts
  • Define alert rules that trigger notifications when thresholds are breached
  • Support tagging/labeling for multi-dimensional queries (e.g., host=web-1, region=us-east)

Non-Functional Requirements

  • Ingestion rate: 10 million data points per second
  • Query latency: < 500ms for dashboard queries (last 1 hour, aggregated)
  • Storage: Retain data for 5 years with progressive downsampling
  • Availability: 99.9% for ingestion, 99.95% for queries
  • Freshness: Metrics visible within 10 seconds of emission

Out of Scope

  • Log aggregation (separate system)
  • Distributed tracing
  • Synthetic monitoring

Step 2: Capacity Estimation

Assumptions:
- 100,000 hosts reporting metrics
- Each host reports 100 metrics every 10 seconds
- 100,000 x 100 / 10 = 1 million data points/second (average)
- Peak: 10 million data points/second
- Each data point: metric_name + tags + timestamp + value = ~150 bytes

Ingestion Bandwidth:
- 10M points/s x 150 bytes = 1.5 GB/s inbound
- Daily: 1.5 GB/s x 86,400 = ~130 TB/day raw

Storage (with compression and downsampling):
- Raw (10s granularity, 7 days): 130 TB/day x 7 = 910 TB
- With compression (10x): 91 TB
- 1-minute rollups (1 year): 130 TB / 6 x 365 / 10 = ~800 TB
- 1-hour rollups (5 years): ~70 TB
- Total: ~1 PB

Query Load:
- 10,000 concurrent dashboard users
- Each dashboard: 20 charts, each chart queries 1-3 metrics
- Refresh every 30 seconds
- 10,000 x 20 x 2 / 30 = ~13,000 queries/second

Unique time series (cardinality):
- 100,000 hosts x 100 metrics x 10 tag combinations = 100 million time series

Time Series Identity

A unique time series is identified by (metric_name, sorted_tags). This is the "cardinality" of the system:

function timeSeriesId(metric, tags) {
  const sortedTags = Object.entries(tags)
    .sort(([a], [b]) => a.localeCompare(b))
    .map(([k, v]) => `${k}=${v}`)
    .join(',');
  return `${metric}{${sortedTags}}`;
}

// Example:
// http.request.duration{host=web-42,method=GET,path=/api/users,region=us-east-1,service=user-api,status_code=200}

Step 6: Metrics Agent

import os from 'node:os';

class MetricsAgent {
  constructor(endpoint, tags = {}) {
    this.endpoint = endpoint;
    this.defaultTags = tags;
    this.buffer = [];
    this.flushIntervalMs = 10000; // Flush every 10 seconds
    this.maxBufferSize = 1000;
  }

  // Gauge: current value (e.g., CPU usage, memory)
  gauge(metric, value, tags = {}) {
    this.buffer.push({
      metric,
      type: 'gauge',
      value,
      timestamp: Math.floor(Date.now() / 1000),
      tags: { ...this.defaultTags, ...tags },
    });
    this._maybeFlush();
  }

  // Counter: monotonically increasing value (e.g., request count)
  increment(metric, value = 1, tags = {}) {
    this.buffer.push({
      metric,
      type: 'counter',
      value,
      timestamp: Math.floor(Date.now() / 1000),
      tags: { ...this.defaultTags, ...tags },
    });
    this._maybeFlush();
  }

  // Histogram: distribution of values (e.g., request latency)
  histogram(metric, value, tags = {}) {
    this.buffer.push({
      metric,
      type: 'histogram',
      value,
      timestamp: Math.floor(Date.now() / 1000),
      tags: { ...this.defaultTags, ...tags },
    });
    this._maybeFlush();
  }

  // System metrics collector
  collectSystemMetrics() {
    const cpus = os.cpus();
    const totalMem = os.totalmem();
    const freeMem = os.freemem();
    const loadAvg = os.loadavg();

    this.gauge('system.cpu.count', cpus.length);
    this.gauge('system.memory.total', totalMem);
    this.gauge('system.memory.used', totalMem - freeMem);
    this.gauge('system.memory.usage_pct', ((totalMem - freeMem) / totalMem) * 100);
    this.gauge('system.load.1m', loadAvg[0]);
    this.gauge('system.load.5m', loadAvg[1]);
    this.gauge('system.load.15m', loadAvg[2]);
  }

  start() {
    // Collect system metrics every 10 seconds
    setInterval(() => this.collectSystemMetrics(), this.flushIntervalMs);
    // Flush buffer periodically
    setInterval(() => this.flush(), this.flushIntervalMs);
  }

  _maybeFlush() {
    if (this.buffer.length >= this.maxBufferSize) {
      this.flush();
    }
  }

  async flush() {
    if (this.buffer.length === 0) return;

    const batch = this.buffer.splice(0, this.buffer.length);
    try {
      await fetch(`${this.endpoint}/api/v1/metrics`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ metrics: batch }),
      });
    } catch (err) {
      // Put failed metrics back in buffer (with limit to prevent memory growth)
      if (this.buffer.length < 10000) {
        this.buffer.unshift(...batch);
      }
      console.error('Failed to flush metrics:', err.message);
    }
  }
}

// Usage in an Express app
const metrics = new MetricsAgent('http://metrics-gateway:8080', {
  host: os.hostname(),
  service: 'user-api',
  region: process.env.REGION,
});
metrics.start();

// Middleware to track request metrics
function metricsMiddleware(req, res, next) {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    const tags = {
      method: req.method,
      path: req.route?.path || req.path,
      status_code: String(res.statusCode),
    };
    metrics.increment('http.requests.total', 1, tags);
    metrics.histogram('http.request.duration_ms', duration, tags);
  });
  next();
}

Write Worker (Batch Writer)

import { Kafka } from 'kafkajs';

class MetricsWriter {
  constructor() {
    this.batch = new Map(); // seriesId -> [datapoints]
    this.batchSize = 10000;
    this.flushIntervalMs = 5000;
  }

  async start() {
    const kafka = new Kafka({ brokers: ['kafka:9092'] });
    const consumer = kafka.consumer({ groupId: 'metrics-writer' });
    await consumer.connect();
    await consumer.subscribe({ topic: 'metrics-ingest' });

    // Periodic flush
    setInterval(() => this.flush(), this.flushIntervalMs);

    await consumer.run({
      eachMessage: async ({ message }) => {
        const point = JSON.parse(message.value.toString());
        const seriesId = message.key.toString();

        if (!this.batch.has(seriesId)) {
          this.batch.set(seriesId, []);
        }
        this.batch.get(seriesId).push({
          timestamp: point.timestamp,
          value: point.value,
        });

        if (this.getTotalPoints() >= this.batchSize) {
          await this.flush();
        }
      },
    });
  }

  getTotalPoints() {
    let total = 0;
    for (const points of this.batch.values()) {
      total += points.length;
    }
    return total;
  }

  async flush() {
    if (this.batch.size === 0) return;

    const currentBatch = this.batch;
    this.batch = new Map();

    // Write to time-series database
    // In production: write to ClickHouse, TimescaleDB, or custom TSDB
    const insertValues = [];
    for (const [seriesId, points] of currentBatch) {
      for (const point of points) {
        insertValues.push({
          series_id: seriesId,
          timestamp: new Date(point.timestamp * 1000),
          value: point.value,
        });
      }
    }

    // Batch insert (using TimescaleDB/PostgreSQL as example)
    await prisma.$executeRaw`
      INSERT INTO metrics_raw (series_id, timestamp, value)
      SELECT * FROM unnest(
        ${insertValues.map(v => v.series_id)}::text[],
        ${insertValues.map(v => v.timestamp)}::timestamptz[],
        ${insertValues.map(v => v.value)}::double precision[]
      )
    `;
  }
}

Downsampling (Rollup)

class Downsampler {
  // Run periodically to create rollups
  async createOneMinuteRollups() {
    // Aggregate raw 10-second data into 1-minute buckets
    await prisma.$executeRaw`
      INSERT INTO metrics_1m (series_id, bucket, avg_value, min_value, max_value, sum_value, count)
      SELECT
        series_id,
        time_bucket('1 minute', timestamp) AS bucket,
        AVG(value) AS avg_value,
        MIN(value) AS min_value,
        MAX(value) AS max_value,
        SUM(value) AS sum_value,
        COUNT(*) AS count
      FROM metrics_raw
      WHERE timestamp >= NOW() - INTERVAL '10 minutes'
        AND timestamp < NOW() - INTERVAL '5 minutes'
      GROUP BY series_id, bucket
      ON CONFLICT (series_id, bucket) DO NOTHING
    `;
  }

  async createOneHourRollups() {
    await prisma.$executeRaw`
      INSERT INTO metrics_1h (series_id, bucket, avg_value, min_value, max_value, sum_value, count)
      SELECT
        series_id,
        time_bucket('1 hour', bucket) AS bucket,
        AVG(avg_value) AS avg_value,
        MIN(min_value) AS min_value,
        MAX(max_value) AS max_value,
        SUM(sum_value) AS sum_value,
        SUM(count) AS count
      FROM metrics_1m
      WHERE bucket >= NOW() - INTERVAL '2 hours'
        AND bucket < NOW() - INTERVAL '1 hour'
      GROUP BY series_id, time_bucket('1 hour', bucket)
      ON CONFLICT (series_id, bucket) DO NOTHING
    `;
  }
}

Step 9: Alerting Engine

class AlertEngine {
  constructor(queryService, notifier) {
    this.queryService = queryService;
    this.notifier = notifier;
    this.evaluationIntervalMs = 30000; // Evaluate every 30 seconds
  }

  async start() {
    setInterval(() => this.evaluateAllRules(), this.evaluationIntervalMs);
  }

  async evaluateAllRules() {
    const rules = await prisma.alertRule.findMany({
      where: { isEnabled: true },
    });

    // Evaluate rules in parallel with concurrency limit
    const CONCURRENCY = 20;
    for (let i = 0; i < rules.length; i += CONCURRENCY) {
      const batch = rules.slice(i, i + CONCURRENCY);
      await Promise.all(batch.map(rule => this.evaluateRule(rule)));
    }
  }

  async evaluateRule(rule) {
    try {
      const endTime = new Date();
      const startTime = new Date(endTime.getTime() - rule.windowSeconds * 1000);

      const result = await this.queryService.execute({
        metric: rule.metricName,
        aggregation: rule.aggregation,
        filters: rule.filters,
        groupBy: rule.groupBy || [],
        startTime,
        endTime,
        granularity: `${rule.windowSeconds}s`, // Single bucket for the window
      });

      for (const series of result) {
        const value = series.datapoints[0]?.[1];
        if (value === undefined) continue;

        const isTriggered = this.checkCondition(value, rule.operator, rule.threshold);

        if (isTriggered) {
          await this.fireAlert(rule, value, series.tags);
        } else {
          await this.resolveAlert(rule, series.tags);
        }
      }
    } catch (err) {
      console.error(`Failed to evaluate rule ${rule.id}:`, err.message);
    }
  }

  checkCondition(value, operator, threshold) {
    switch (operator) {
      case '>': return value > threshold;
      case '<': return value < threshold;
      case '>=': return value >= threshold;
      case '<=': return value <= threshold;
      case '==': return Math.abs(value - threshold) < 0.001;
      default: return false;
    }
  }

  async fireAlert(rule, value, tags) {
    // Check if already firing (avoid duplicate alerts)
    const existing = await prisma.alertIncident.findFirst({
      where: { ruleId: rule.id, status: 'firing' },
    });

    if (!existing) {
      const incident = await prisma.alertIncident.create({
        data: {
          ruleId: rule.id,
          status: 'firing',
          triggeredValue: value,
        },
      });

      // Send notification
      await this.notifier.send({
        channels: rule.notifyChannels,
        severity: rule.severity,
        title: `Alert: ${rule.name}`,
        message: `${rule.metricName} ${rule.operator} ${rule.threshold} (current: ${value.toFixed(2)})`,
        tags,
        incidentId: incident.id,
      });
    }
  }

  async resolveAlert(rule, tags) {
    const incident = await prisma.alertIncident.findFirst({
      where: { ruleId: rule.id, status: 'firing' },
    });

    if (incident) {
      await prisma.alertIncident.update({
        where: { id: incident.id },
        data: { status: 'resolved', resolvedAt: new Date() },
      });

      await this.notifier.send({
        channels: rule.notifyChannels,
        severity: 'info',
        title: `Resolved: ${rule.name}`,
        message: `Alert resolved after ${Math.floor((Date.now() - incident.triggeredAt.getTime()) / 1000)}s`,
        tags,
      });
    }
  }
}

Query Scaling

Shard the time-series database by time range and by series_id:

Time-based sharding:
  Shard A: Last 24 hours (hot, SSD, in-memory cache)
  Shard B: 1-7 days ago (warm, SSD)
  Shard C: 7-30 days ago (cold, HDD)
  Shard D: 30+ days (archive, compressed, HDD)

Series-based sharding:
  Shard 1: series_id % 10 == 0 (hash-based distribution)
  Shard 2: series_id % 10 == 1
  ...

Cardinality Management

High cardinality (too many unique tag combinations) is the biggest operational challenge:

// Enforce cardinality limits
class CardinalityGuard {
  constructor(redis, maxPerMetric = 100000) {
    this.redis = redis;
    this.maxPerMetric = maxPerMetric;
  }

  async checkAndRecord(metricName, tagsHash) {
    const key = `cardinality:${metricName}`;
    const current = await this.redis.scard(key);

    if (current >= this.maxPerMetric) {
      const isMember = await this.redis.sismember(key, tagsHash);
      if (!isMember) {
        // New series would exceed limit - drop it
        console.warn(`Cardinality limit reached for ${metricName}: ${current}`);
        return false;
      }
    }

    await this.redis.sadd(key, tagsHash);
    return true;
  }
}

Compression Techniques

Time-series data compresses extremely well because consecutive timestamps and values are similar:

  • Delta-of-delta for timestamps: If interval is regular (10s), store only the deviation from expected
  • XOR encoding for values: Store XOR of consecutive values (often just a few bits differ)
  • Gorilla compression: Facebook's algorithm achieves 12x compression on real-world metrics

Summary

A metrics and monitoring system ingests millions of data points per second from agents running on monitored hosts. Data flows through a message buffer (Kafka), gets written in batches to a time-series database (TimescaleDB or ClickHouse), and is progressively downsampled from high-resolution (10s) to lower resolution (1m, 1h) for long-term storage. A query engine supports multi-dimensional aggregation for dashboard rendering, while an alerting engine periodically evaluates rules against recent data and triggers notifications. The biggest operational challenges are managing time-series cardinality and scaling write throughput.

Common Problems & Solutions

1 Ingestion pipeline falls behind, causing stale dashboards and delayed alerts
Symptoms: Dashboards show data gaps or delays. Alerts fire minutes after the actual threshold breach. Kafka consumer lag metric is growing continuously.
Cause: Write workers cannot keep up with the incoming data rate. Kafka consumer lag grows, meaning metrics are ingested but not yet written to the TSDB.
Solution: Scale write workers horizontally (more consumer instances with more Kafka partitions). Use batch inserts instead of individual writes. Pre-aggregate on the agent side (send 1-minute rollups instead of raw points). Monitor Kafka consumer lag as a key operational metric.
2 Query latency degrades as the number of unique time series grows (high cardinality explosion)
Symptoms: Dashboard queries that were fast become increasingly slow over days/weeks. Series registry table grows to hundreds of millions of rows. Memory usage on TSDB nodes keeps climbing.
Cause: A tag with unbounded values (like user_id, request_id, or full URL path) creates millions of unique time series, making index lookups and aggregation slow.
Solution: Enforce cardinality limits per metric (e.g., max 100K unique series). Reject or drop metrics that exceed the limit. Educate users to avoid high-cardinality tags. Use pre-aggregation for high-cardinality dimensions.
3 Alerts are flapping (firing and resolving rapidly in alternation)
Cause: The metric value oscillates around the threshold, causing the alert to fire and resolve on every evaluation cycle. This floods notification channels with noise.
Solution: Implement hysteresis: use a higher threshold to fire and a lower threshold to resolve (or vice versa). Add a minimum firing duration (e.g., must be above threshold for 3 consecutive evaluations). Support 'for' clauses like Prometheus (e.g., fire only if condition is true for 5 minutes).
