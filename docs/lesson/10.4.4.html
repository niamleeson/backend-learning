Design a Content Delivery Network (CDN)

Intuition

A CDN is a hierarchical caching system where the fundamental trade-off is freshness vs. latency: the closer content is cached to the user, the faster it is served, but the harder it is to keep current. The shield tier is the key architectural insight that collapses thousands of edge misses into a single origin request.

Design a Content Delivery Network (CDN)

Requirements

  • Serve static and dynamic content from edge locations closest to users
  • Support cache invalidation, TTL-based expiry, and origin shielding
  • Handle TLS termination, DDoS mitigation, and HTTP/2+ at the edge
  • Deliver 99.99% availability with sub-50ms latency for cached content

Architecture

A global network of PoPs (Points of Presence) each running reverse proxy/cache servers. DNS-based or Anycast routing directs users to the nearest PoP. On cache miss, the edge fetches from a mid-tier shield cache before hitting the origin, reducing origin load. A control plane pushes configuration and invalidation commands.

Key Components

  • Edge Cache Servers: Nginx/Varnish nodes with SSD-backed LRU caches
  • Shield/Mid-Tier Cache: Aggregates cache misses from multiple edges to protect origin
  • DNS/Anycast Router: Directs users to the optimal PoP based on latency and health
  • Control Plane: Distributes config, purge commands, and TLS certificates

Scaling

Add PoPs in underserved regions. Use tiered caching (edge -> shield -> origin) to absorb traffic spikes. Horizontally scale edge nodes within a PoP behind a load balancer.

Trade-offs

  • Push vs. pull caching: push guarantees freshness but wastes bandwidth on unpopular content
  • Long TTLs improve hit rate but delay content updates
  • Anycast simplifies routing but complicates debugging and failover

Exercises

  1. Simulate Tiered Caching

    Build a simulation with 5 edge caches, 1 shield cache, and 1 origin. Generate a Zipfian request distribution and measure hit rates at each tier as you vary TTL.

    Hint

    Model each cache as an LRU map with a max size. On miss, escalate to the next tier. Track hits/misses at each level separately.

  2. Implement Cache Invalidation via Purge Propagation

    Design a purge system where the control plane issues an invalidation that reaches all edge nodes within 5 seconds. Test with 10 simulated edge nodes.

    Hint

    Use a pub/sub fanout (e.g., Redis Pub/Sub or a message queue topic) where each edge subscribes and immediately evicts the purged key on receipt.

Common Problems & Solutions

1 Origin server overwhelmed after a mass cache purge
Cause: All edges simultaneously refetch the purged content, creating a thundering herd on origin
Solution: Use request coalescing at the shield tier: only one request goes to origin while others wait, and implement stale-while-revalidate to serve old content briefly.
2 Users in a specific region get stale content indefinitely
Cause: The PoP in that region has a misconfigured TTL override ignoring purge commands
Solution: Add health checks and config version verification in the control plane; flag PoPs running stale configurations and force a config resync.
