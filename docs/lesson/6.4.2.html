API Contracts and Contract Testing

Intuition

A contract test is like a prenuptial agreement between services. Before they start working together, both sides agree on exactly what they will provide and expect. If either side breaks the agreement, the test catches it immediately -- no surprises in production.

API Contracts and Contract Testing

Contract testing verifies that an API provider and its consumers agree on the API's shape and behavior, catching integration issues without full end-to-end tests.

Schema-Based Contract Testing with Schemathesis

# Install: pip install schemathesis
# Run against your OpenAPI spec:
# schemathesis run http://localhost:8000/openapi.json

# Or write programmatic tests:
import schemathesis

schema = schemathesis.from_url("http://localhost:8000/openapi.json")

@schema.parametrize()
def test_api_contract(case):
    response = case.call()
    case.validate_response(response)

Consumer-Driven Contract Tests

import pytest
from pydantic import BaseModel, ValidationError

# The contract: what the consumer expects
class UserContract(BaseModel):
    id: int
    name: str
    email: str

def test_user_endpoint_fulfills_contract(client):
    """Consumer-driven: verify the provider returns what we expect"""
    response = client.get("/users/1")
    assert response.status_code == 200

    # Validate the response matches the consumer's expected contract
    try:
        user = UserContract(**response.json())
    except ValidationError as e:
        pytest.fail(f"API response broke the contract: {e}")

    assert user.id == 1
    assert isinstance(user.name, str)
    assert "@" in user.email

def test_error_contract(client):
    """Verify error responses also follow the contract"""
    response = client.get("/users/99999")
    assert response.status_code == 404
    body = response.json()
    assert "type" in body  # RFC 7807
    assert "title" in body
    assert "status" in body

Provider Verification

import json
from fastapi.testclient import TestClient

def test_provider_matches_openapi_spec(app):
    """Verify the actual API matches its OpenAPI spec"""
    client = TestClient(app)
    spec = client.get("/openapi.json").json()

    for path, methods in spec["paths"].items():
        for method, details in methods.items():
            if method == "get":
                response = client.get(path.replace("{id}", "1"))
                assert response.status_code in [
                    int(code) for code in details["responses"].keys()
                ]

Common Mistakes

  • Only testing happy paths and ignoring error response contracts
  • Not versioning contracts alongside the API
  • Running contract tests only in CI instead of also during local development

Key Takeaways

  • Contract tests catch breaking changes before deployment
  • Schemathesis auto-generates test cases from your OpenAPI spec
  • Consumer-driven contracts ensure providers do not break their clients

Exercises

  1. Set Up Schemathesis

    Install schemathesis and run it against an existing FastAPI application. Fix any issues it discovers (missing response schemas, undocumented error codes, validation gaps). Document what it found.

    Hint

    Run: schemathesis run http://localhost:8000/openapi.json --checks all. Review failures and add missing response models and error handlers to your FastAPI app.

  2. Consumer-Driven Contract Suite

    Write a contract test suite where a 'frontend' consumer defines Pydantic models for what it expects from 3 API endpoints. The tests validate that the actual API responses conform to these models, including error cases.

    Hint

    Create separate Pydantic models representing the consumer's expectations (not reusing the provider's models). Test each endpoint's response against these consumer models.

  3. Breaking Change Detector

    Build a script that compares two versions of an OpenAPI spec (old and new) and reports breaking changes: removed endpoints, removed fields, changed types, new required fields.

    Hint

    Load both specs as dicts. Compare paths, methods, and schema properties. A removed path or a new required field is a breaking change; an added optional field is not.

Common Problems & Solutions

1 Deployment breaks a downstream consumer service
Symptoms: ['Consumer service starts throwing deserialization errors after provider deploys', 'Integration test suite does not catch the issue']
Cause: A field was renamed in the API response, but no contract test verified the consumer's expectations.
Solution: Implement consumer-driven contract tests. Each consumer defines its expected response shape, and the provider's CI runs these tests before deployment.
2 Schemathesis reports hundreds of failures
Cause: The OpenAPI spec has permissive schemas (e.g., no field constraints) that allow invalid inputs the server actually rejects.
Solution: Tighten the OpenAPI spec: add min/max constraints, required fields, enums, and pattern validations. The spec should accurately reflect what the server actually accepts.
