Design a Ticket Booking System (Ticketmaster)

Intuition

Think of buying concert tickets like entering a physical box office that only has 5 windows. When a hot show goes on sale, thousands of people show up. Instead of a mob rushing the windows, security sets up a line (the virtual queue). They hand out numbered tickets so everyone knows their position. Every few minutes, a batch of people is let into the lobby (admitted). In the lobby, you have 15 minutes to pick your seats at a kiosk (the hold). The kiosk puts a sticky note on your chosen seats so nobody else can grab them. If you do not pay within 15 minutes, the sticky notes are removed and those seats go back to available. This is exactly how the system works: the queue shapes traffic, holds are temporary claims, and the final purchase is the only permanent action.

Design a Ticket Booking System (Ticketmaster)

Introduction

Designing a ticket booking system like Ticketmaster is a fascinating system design problem because it combines the challenges of e-commerce with extreme concurrency and time pressure. When a popular concert goes on sale, millions of users compete for a limited number of tickets within seconds. Unlike hotel rooms where availability is spread across dates, event tickets create massive thundering herd problems at a single point in time. The system must handle these spikes gracefully, prevent double-selling, and provide a fair user experience.

Step 1: Requirements Gathering

Functional Requirements

  • Event Browsing: Browse/search events by location, date, category, artist
  • Venue & Seating: Display venue maps with seat sections, view from seat
  • Ticket Selection: Select specific seats or best-available in a section
  • Temporary Hold: Reserve selected tickets for a limited time (e.g., 10 minutes)
  • Checkout: Purchase held tickets with payment processing
  • Queue System: Virtual waiting room for high-demand events
  • Ticket Transfer: Transfer/resell tickets to other users
  • QR Codes: Generate unique QR codes for entry

Non-Functional Requirements

  • Scalability: Handle 10M+ concurrent users for a single popular event
  • Latency: Seat map load < 500ms, seat selection < 200ms
  • Consistency: Absolutely no double-selling of the same seat
  • Fairness: Users who arrive first should get tickets first (queue system)
  • Availability: 99.99% uptime, especially during on-sale events

Step 2: Capacity Estimation

Assumptions:
- 500K events/year, average capacity 5,000 seats
- Top 1% of events (5,000) generate 90% of traffic
- Popular event on-sale: 10M users in queue, 50K seats
- On-sale window: first 5 minutes see 80% of traffic
- Normal operations: 1M daily active users

Peak Traffic (popular event on-sale):
- Concurrent users: 10M
- Queue check requests: 10M * 1 req/2sec = 5M QPS
- Seat map requests: 500K/min (users entering from queue)
- Seat hold requests: 50K in 5 min = ~170 TPS
- Checkout: 50K in 10 min = ~83 TPS

Normal Traffic:
- Search/browse: 1M * 5 pages = 5M/day = ~58 QPS
- Ticket purchases: 100K/day = ~1.2 TPS

Storage:
- Events: 500K * 10KB = ~5GB
- Venue/seats: 10K venues * 50K seats * 200 bytes = ~100GB
- Tickets: 500K events * 5K seats * 500 bytes = ~1.25TB/year
- Orders: 100K/day * 2KB * 365 = ~73GB/year

Step 4: API Design

Virtual Queue System

import express from 'express';
import crypto from 'node:crypto';
const router = express.Router();

// POST /api/events/:eventId/queue/join
router.post('/events/:eventId/queue/join', authenticate, async (req, res) => {
  const { eventId } = req.params;

  // Check if event is in on-sale mode
  const event = await eventService.getEvent(eventId);
  if (!event || event.status !== 'ON_SALE') {
    return res.status(400).json({ error: 'Event is not on sale' });
  }

  // Check if user is already in queue
  const existing = await redis.zscore(`queue:${eventId}`, req.user.id);
  if (existing !== null) {
    return res.json({
      queueToken: await redis.hget(`queue:tokens:${eventId}`, req.user.id),
      position: await getQueuePosition(eventId, req.user.id),
      estimatedWait: await getEstimatedWait(eventId, req.user.id),
    });
  }

  // Add to sorted set with timestamp as score
  const timestamp = Date.now();
  const queueToken = crypto.randomUUID();

  await redis.multi()
    .zadd(`queue:${eventId}`, timestamp, req.user.id)
    .hset(`queue:tokens:${eventId}`, req.user.id, queueToken)
    .exec();

  const position = await getQueuePosition(eventId, req.user.id);

  res.status(201).json({
    queueToken,
    position,
    estimatedWait: position * 2, // ~2 seconds per person
    totalInQueue: await redis.zcard(`queue:${eventId}`),
  });
});

// GET /api/events/:eventId/queue/status
router.get('/events/:eventId/queue/status', authenticate, async (req, res) => {
  const { eventId } = req.params;

  const position = await getQueuePosition(eventId, req.user.id);
  if (position === null) {
    return res.status(404).json({ error: 'Not in queue' });
  }

  // Check if user has been admitted
  const admitted = await redis.sismember(`queue:admitted:${eventId}`, req.user.id);

  res.json({
    position: admitted ? 0 : position,
    admitted,
    estimatedWait: admitted ? 0 : position * 2,
    ticketsRemaining: await getTicketsRemaining(eventId),
  });
});

async function getQueuePosition(eventId, userId) {
  const rank = await redis.zrank(`queue:${eventId}`, userId);
  return rank !== null ? rank + 1 : null;
}

// Queue processor: admits users in batches
async function processQueue(eventId, batchSize = 100) {
  const ticketsRemaining = await getTicketsRemaining(eventId);
  if (ticketsRemaining <= 0) return;

  // Pop next batch from queue
  const nextUsers = await redis.zrange(
    `queue:${eventId}`, 0, batchSize - 1
  );

  if (nextUsers.length === 0) return;

  // Mark as admitted
  const pipeline = redis.pipeline();
  for (const userId of nextUsers) {
    pipeline.sadd(`queue:admitted:${eventId}`, userId);
    pipeline.zrem(`queue:${eventId}`, userId);
  }
  // Set admission expiry (15 minutes to complete purchase)
  for (const userId of nextUsers) {
    pipeline.setex(`queue:admission:${eventId}:${userId}`, 900, '1');
  }
  await pipeline.exec();
}

Seat Selection & Hold

// POST /api/events/:eventId/tickets/hold
router.post('/events/:eventId/tickets/hold', authenticate, async (req, res) => {
  const { eventId } = req.params;
  const { seatIds, sectionId, quantity } = req.body;

  // Verify user is admitted from queue
  const admitted = await redis.sismember(
    `queue:admitted:${eventId}`, req.user.id
  );
  if (!admitted) {
    return res.status(403).json({ error: 'Not admitted from queue' });
  }

  // Release any existing holds by this user
  await releaseUserHolds(eventId, req.user.id);

  let selectedSeats;

  if (seatIds && seatIds.length > 0) {
    // Specific seat selection
    selectedSeats = seatIds;
  } else {
    // Best available in section
    selectedSeats = await findBestAvailable(eventId, sectionId, quantity);
    if (selectedSeats.length < quantity) {
      return res.status(409).json({ error: 'Not enough seats available' });
    }
  }

  // Attempt atomic hold using Redis
  const holdId = crypto.randomUUID();
  const holdExpiry = 600; // 10 minutes
  const results = [];

  for (const seatId of selectedSeats) {
    // SET NX ensures only one user can hold a seat
    const held = await redis.set(
      `hold:${eventId}:${seatId}`,
      JSON.stringify({ holdId, userId: req.user.id, seatId }),
      'EX', holdExpiry,
      'NX'
    );
    results.push({ seatId, held: held === 'OK' });
  }

  const successfulHolds = results.filter(r => r.held);
  const failedHolds = results.filter(r => !r.held);

  // If not all seats could be held, release the ones that were
  if (failedHolds.length > 0 && seatIds) {
    for (const hold of successfulHolds) {
      await redis.del(`hold:${eventId}:${hold.seatId}`);
    }
    return res.status(409).json({
      error: 'Some seats are no longer available',
      unavailableSeats: failedHolds.map(h => h.seatId),
    });
  }

  // Store hold reference for user
  await redis.setex(
    `user:hold:${eventId}:${req.user.id}`,
    holdExpiry,
    JSON.stringify({ holdId, seatIds: successfulHolds.map(h => h.seatId) })
  );

  res.json({
    holdId,
    seats: successfulHolds.map(h => h.seatId),
    expiresIn: holdExpiry,
    expiresAt: new Date(Date.now() + holdExpiry * 1000).toISOString(),
  });
});

async function findBestAvailable(eventId, sectionId, quantity) {
  // Get all seats in section that are not held or sold
  const allSeats = await prisma.seat.findMany({
    where: { eventId, sectionId, status: 'AVAILABLE' },
    orderBy: [{ row: 'asc' }, { number: 'asc' }],
  });

  // Filter out seats with active holds in Redis
  const available = [];
  for (const seat of allSeats) {
    const held = await redis.exists(`hold:${eventId}:${seat.id}`);
    if (!held) available.push(seat.id);
    if (available.length >= quantity) break;
  }

  return available;
}

Checkout

// POST /api/events/:eventId/tickets/purchase
router.post('/events/:eventId/tickets/purchase',
  authenticate, async (req, res) => {
  const { eventId } = req.params;
  const { holdId, paymentMethodId } = req.body;

  // 1. Verify the hold exists and belongs to this user
  const holdData = await redis.get(
    `user:hold:${eventId}:${req.user.id}`
  );
  if (!holdData) {
    return res.status(410).json({ error: 'Hold expired. Please select seats again.' });
  }

  const hold = JSON.parse(holdData);
  if (hold.holdId !== holdId) {
    return res.status(400).json({ error: 'Invalid hold' });
  }

  // 2. Verify all seats are still held by this user
  for (const seatId of hold.seatIds) {
    const seatHold = await redis.get(`hold:${eventId}:${seatId}`);
    if (!seatHold) {
      return res.status(410).json({ error: 'Hold expired for some seats' });
    }
    const parsed = JSON.parse(seatHold);
    if (parsed.userId !== req.user.id) {
      return res.status(409).json({ error: 'Seat hold conflict' });
    }
  }

  // 3. Get pricing
  const seats = await prisma.seat.findMany({
    where: { id: { in: hold.seatIds } },
    include: { section: true },
  });
  const totalPrice = seats.reduce((sum, s) => sum + parseFloat(s.section.price), 0);
  const serviceFee = totalPrice * 0.15;
  const grandTotal = totalPrice + serviceFee;

  // 4. Process payment
  const payment = await paymentService.charge({
    amount: grandTotal,
    currency: 'USD',
    paymentMethodId,
    metadata: { eventId, seatIds: hold.seatIds },
  });

  if (!payment.success) {
    return res.status(402).json({ error: 'Payment failed', reason: payment.reason });
  }

  // 5. Create order and tickets in database
  const order = await prisma.$transaction(async (tx) => {
    // Mark seats as sold
    await tx.seat.updateMany({
      where: { id: { in: hold.seatIds } },
      data: { status: 'SOLD', soldAt: new Date() },
    });

    // Create order
    const order = await tx.order.create({
      data: {
        userId: req.user.id,
        eventId,
        totalPrice: grandTotal,
        paymentId: payment.id,
        status: 'CONFIRMED',
        tickets: {
          create: seats.map(seat => ({
            seatId: seat.id,
            barcode: crypto.randomUUID(),
            status: 'VALID',
          })),
        },
      },
      include: { tickets: { include: { seat: true } } },
    });

    return order;
  });

  // 6. Clean up holds
  const pipeline = redis.pipeline();
  for (const seatId of hold.seatIds) {
    pipeline.del(`hold:${eventId}:${seatId}`);
  }
  pipeline.del(`user:hold:${eventId}:${req.user.id}`);
  await pipeline.exec();

  // 7. Send confirmation
  await notificationQueue.add('ticket-confirmation', {
    orderId: order.id,
    userId: req.user.id,
  });

  res.status(201).json(order);
});

Step 6: Deep Dive - Virtual Queue

The virtual queue is the most critical component for high-demand events.

On-Sale Flow:

1. Before on-sale time:
   Users land on event page -> redirected to waiting room

2. At on-sale time:
   All waiting users get random positions (fair lottery)
   New arrivals join end of queue

3. Queue processing:
   Batch of N users admitted every few seconds
   Admitted users have 15 min to select + purchase

4. After purchase/timeout:
   Slot freed, next batch admitted

┌────────────────────────────────────────────────┐
│                Virtual Queue                    │
│                                                │
│  Waiting Room ──> Queue ──> Admitted ──> Shop  │
│  (pre-sale)      (FIFO)    (15 min)    (buy)  │
│                                                │
│  10M users       Sorted    500 active   Seat   │
│  waiting         by time   at a time    map +  │
│                                         hold   │
└────────────────────────────────────────────────┘

// Fair queue initialization: randomize positions for users who joined before on-sale
async function initializeQueue(eventId) {
  const waitingRoom = await redis.smembers(`waiting:${eventId}`);

  // Shuffle for fairness
  const shuffled = waitingRoom.sort(() => Math.random() - 0.5);

  // Add to sorted set with sequential scores
  const pipeline = redis.pipeline();
  shuffled.forEach((userId, index) => {
    pipeline.zadd(`queue:${eventId}`, index, userId);
  });
  await pipeline.exec();

  // Clean up waiting room
  await redis.del(`waiting:${eventId}`);

  // Start queue processor
  startQueueProcessor(eventId);
}

function startQueueProcessor(eventId) {
  const interval = setInterval(async () => {
    const remaining = await getTicketsRemaining(eventId);
    if (remaining <= 0) {
      clearInterval(interval);
      return;
    }
    await processQueue(eventId, 100); // admit 100 users per batch
  }, 3000); // every 3 seconds
}

Summary

A ticket booking system has two distinct modes: normal operations (low traffic, standard CRUD) and on-sale events (extreme traffic spike, thundering herd). The virtual queue absorbs the spike, admitting users in controlled batches. Redis handles the ephemeral state (queue positions, seat holds) with automatic expiration. The database handles the permanent state (orders, tickets). The key patterns are: virtual queue for traffic shaping, Redis SET NX for atomic seat holds, time-limited holds with automatic release, and database transactions for final purchase. Always separate the hot path (queue polling) from the critical path (payment processing).

Exercises

  1. Build a Virtual Queue

    Implement a virtual queue system using Redis sorted sets. Support join, check position, and batch admission. Test with 10,000 simulated users joining the queue and being admitted in batches of 50 every 2 seconds. Track metrics: queue depth over time, average wait time, admission rate.

    Hint

    Use ZADD with timestamp scores for ordering. Use ZRANGE + ZREM for batch admission. Use SISMEMBER for checking admission status. Run the queue processor as a setInterval loop.

  2. Atomic Seat Hold with Redis

    Build a seat hold service that uses Redis SET NX to ensure only one user can hold a seat at a time. Implement: hold acquisition (10 min TTL), hold release, hold extension, and automatic expiry. Test with 100 concurrent users trying to hold the same seat.

    Hint

    Use redis.set(key, value, 'EX', 600, 'NX') for atomic hold. Return failure immediately if NX fails. Use a Lua script if you need to atomically hold multiple seats.

  3. Seat Map Availability Service

    Create an API that returns real-time seat availability for a venue section. Combine database state (SOLD) with Redis state (HELD) to produce the current availability view. Implement a WebSocket endpoint that pushes availability updates as seats are held or released.

    Hint

    Query the database for sold seats, then check Redis for held seats, then merge. Use a Redis pub/sub channel to broadcast hold/release events to WebSocket subscribers.

Common Problems & Solutions

1 Seat holds expiring during checkout, causing users to lose their tickets
Symptoms: Users report completing payment but receiving 'seats no longer available' errors. High cart abandonment rate on the checkout page.
Cause: The 10-minute hold timer expires while the user is entering payment information or the payment processor is slow. The seats return to available and another user grabs them.
Solution: Extend the hold when checkout begins (add 5 more minutes). Show a countdown timer on the client. Use a shorter initial hold (7 min for browsing) and extend on checkout initiation. Allow the user to re-hold the same seats if they expired within the last 60 seconds.
2 Queue position jumps erratically or users get pushed back in the queue
Cause: Using a Redis sorted set score based on timestamp, but clock skew between multiple application servers causes inconsistent ordering. Or the queue position is calculated incorrectly when batches are admitted.
Solution: Use a single atomic counter (INCR) for position assignment instead of timestamps. Alternatively, use a centralized time source. Calculate position as ZRANK relative to the remaining queue, not absolute position.
3 System crashes under load when a popular event goes on sale
Cause: Queue polling (every 2 seconds per user) creates massive read load on Redis. Without proper rate limiting, 10M users polling creates 5M QPS. The API gateway and Redis cannot handle the load.
Solution: Use longer polling intervals with jitter (2-5 seconds, randomized). Cache queue positions in local memory for 1 second. Use CDN for the waiting room page. Implement WebSocket for push-based position updates instead of client polling. Pre-warm Redis cluster before on-sale events.
