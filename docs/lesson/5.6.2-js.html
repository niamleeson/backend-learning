Service Layer and Domain Model

Intuition

Think of a restaurant. The waiter (controller) takes your order and brings food, but does not cook. The chef (service) coordinates the cooking process, decides when to plate, and orchestrates the kitchen. The recipe (domain model) contains the cooking rules -- you cannot serve raw chicken, you must season before grilling. Each has a clear role, and swapping out the waiter does not change the recipe.

Service Layer and Domain Model

The Service Layer pattern organizes business logic into a dedicated layer between controllers and repositories. The Domain Model represents your business entities with both data and behavior. Together, they form the backbone of a well-structured backend.

Architecture Overview

+------------------+
| Controllers      |  HTTP/GraphQL/gRPC concerns
| (thin)           |  Parsing, validation, response formatting
+--------+---------+
         |
+--------v---------+
| Service Layer    |  Business logic, orchestration
| (thick)          |  Authorization, workflows, domain rules
+--------+---------+
         |
+--------v---------+
| Domain Model     |  Business entities and value objects
| (core)           |  Self-contained business rules
+--------+---------+
         |
+--------v---------+
| Repository Layer |  Data access
| (adapter)        |  Database operations
+------------------+

Domain Model: Entities with Behavior

Instead of anemic objects (data-only), domain entities encapsulate business rules:

// domain/entities/post.entity.ts
export class PostEntity {
  constructor(
    public readonly id: string,
    public title: string,
    public body: string,
    public status: PostStatus,
    public readonly authorId: string,
    public tags: string[],
    public readonly createdAt: Date,
    public updatedAt: Date,
    public publishedAt: Date | null,
  ) {}

  // Business rule: only drafts can be published
  publish(): void {
    if (this.status !== PostStatus.DRAFT) {
      throw new DomainError(`Cannot publish a post with status '${this.status}'`);
    }
    this.status = PostStatus.PUBLISHED;
    this.publishedAt = new Date();
    this.updatedAt = new Date();
  }

  // Business rule: only published posts can be archived
  archive(): void {
    if (this.status !== PostStatus.PUBLISHED) {
      throw new DomainError('Only published posts can be archived');
    }
    this.status = PostStatus.ARCHIVED;
    this.updatedAt = new Date();
  }

  // Business rule: cannot unpublish archived posts
  unpublish(): void {
    if (this.status !== PostStatus.PUBLISHED) {
      throw new DomainError('Only published posts can be unpublished');
    }
    this.status = PostStatus.DRAFT;
    this.publishedAt = null;
    this.updatedAt = new Date();
  }

  updateContent(title?: string, body?: string, tags?: string[]): void {
    if (this.status === PostStatus.ARCHIVED) {
      throw new DomainError('Cannot edit archived posts');
    }
    if (title) this.title = title;
    if (body) this.body = body;
    if (tags) this.tags = tags;
    this.updatedAt = new Date();
  }

  isOwnedBy(userId: string): boolean {
    return this.authorId === userId;
  }

  get isPublished(): boolean {
    return this.status === PostStatus.PUBLISHED;
  }

  get wordCount(): number {
    return this.body.split(/\s+/).length;
  }

  get readTimeMinutes(): number {
    return Math.ceil(this.wordCount / 200);
  }
}

// Value object
export class Email {
  private readonly value: string;

  constructor(email: string) {
    if (!this.isValid(email)) {
      throw new DomainError(`Invalid email: ${email}`);
    }
    this.value = email.toLowerCase().trim();
  }

  private isValid(email: string): boolean {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
  }

  toString(): string {
    return this.value;
  }

  equals(other: Email): boolean {
    return this.value === other.value;
  }
}

Service Layer

Services orchestrate business operations using domain entities and repositories:

// services/post.service.ts
export class PostService {
  constructor(
    private postRepo: IPostRepository,
    private userRepo: IUserRepository,
    private eventBus: IEventBus,
    private notificationService: INotificationService,
  ) {}

  async createPost(authorId: string, data: CreatePostInput): Promise<PostEntity> {
    // Verify author exists
    const author = await this.userRepo.findById(authorId);
    if (!author) throw new NotFoundError('User', authorId);

    // Create domain entity
    const post = new PostEntity(
      generateId(),
      data.title,
      data.body,
      PostStatus.DRAFT,
      authorId,
      data.tags || [],
      new Date(),
      new Date(),
      null,
    );

    // Persist
    const saved = await this.postRepo.create(post);

    // Side effects
    this.eventBus.emit(new PostCreatedEvent(saved));

    return saved;
  }

  async publishPost(postId: string, userId: string): Promise<PostEntity> {
    const post = await this.getPostOrThrow(postId);

    // Authorization
    if (!post.isOwnedBy(userId)) {
      throw new ForbiddenError('You can only publish your own posts');
    }

    // Domain logic (the entity enforces its own rules)
    post.publish();

    // Persist
    const updated = await this.postRepo.update(post.id, {
      status: post.status,
      publishedAt: post.publishedAt,
      updatedAt: post.updatedAt,
    });

    // Side effects
    this.eventBus.emit(new PostPublishedEvent(updated));
    await this.notificationService.notifyFollowers(
      post.authorId,
      `Published: ${post.title}`
    );

    return updated;
  }

  async updatePost(
    postId: string,
    userId: string,
    data: UpdatePostInput
  ): Promise<PostEntity> {
    const post = await this.getPostOrThrow(postId);

    if (!post.isOwnedBy(userId)) {
      throw new ForbiddenError('You can only edit your own posts');
    }

    // Domain logic handles validation
    post.updateContent(data.title, data.body, data.tags);

    return this.postRepo.update(post.id, {
      title: post.title,
      body: post.body,
      tags: post.tags,
      updatedAt: post.updatedAt,
    });
  }

  async deletePost(postId: string, userId: string, userRole: string): Promise<void> {
    const post = await this.getPostOrThrow(postId);

    // Admins can delete any post, others only their own
    if (!post.isOwnedBy(userId) && userRole !== 'ADMIN') {
      throw new ForbiddenError('Cannot delete this post');
    }

    await this.postRepo.delete(postId);
    this.eventBus.emit(new PostDeletedEvent(post));
  }

  private async getPostOrThrow(id: string): Promise<PostEntity> {
    const post = await this.postRepo.findById(id);
    if (!post) throw new NotFoundError('Post', id);
    return post;
  }
}

Thin Controllers

Controllers should be thin -- just handle HTTP and delegate to services:

@Controller('api/posts')
export class PostsController {
  constructor(private postService: PostService) {}

  @Post()
  @UseGuards(JwtAuthGuard)
  async create(@Body() dto: CreatePostDto, @CurrentUser() user: AuthUser) {
    return this.postService.createPost(user.id, dto);
  }

  @Post(':id/publish')
  @UseGuards(JwtAuthGuard)
  async publish(@Param('id') id: string, @CurrentUser() user: AuthUser) {
    return this.postService.publishPost(id, user.id);
  }

  @Put(':id')
  @UseGuards(JwtAuthGuard)
  async update(
    @Param('id') id: string,
    @Body() dto: UpdatePostDto,
    @CurrentUser() user: AuthUser,
  ) {
    return this.postService.updatePost(id, user.id, dto);
  }
}

Anemic vs Rich Domain Model

Anemic Model (anti-pattern):          Rich Domain Model:
+------------------+                  +------------------+
| Post             |                  | PostEntity       |
| - id: string     |                  | - id: string     |
| - title: string  |                  | - title: string  |
| - status: string |                  | - status: string |
+------------------+                  +------------------+
  Just data,                          | + publish()      |
  no behavior                         | + archive()      |
                                      | + updateContent()|
+------------------+                  | + isOwnedBy()    |
| PostService      |                  | + wordCount      |
| + publish()      |                  +------------------+
| + archive()      |                    Entity enforces
| + validate()     |                    its own rules
+------------------+
  All logic in service

The rich domain model keeps business rules close to the data they operate on, making the code more maintainable and self-documenting.

Service Composition

Complex operations may coordinate multiple services:

// services/order.service.ts
export class OrderService {
  constructor(
    private orderRepo: IOrderRepository,
    private inventoryService: InventoryService,
    private paymentService: PaymentService,
    private emailService: EmailService,
  ) {}

  async placeOrder(userId: string, items: OrderItem[]): Promise<Order> {
    // 1. Check inventory
    for (const item of items) {
      const available = await this.inventoryService.checkStock(item.productId);
      if (available < item.quantity) {
        throw new BadRequestError(`Insufficient stock for ${item.productId}`);
      }
    }

    // 2. Create order
    const order = new OrderEntity(generateId(), userId, items);
    const savedOrder = await this.orderRepo.create(order);

    // 3. Reserve inventory
    await this.inventoryService.reserve(items);

    // 4. Process payment
    try {
      await this.paymentService.charge(userId, order.total);
      order.markAsPaid();
      await this.orderRepo.update(order.id, { status: order.status });
    } catch (err) {
      // Compensating action: release inventory
      await this.inventoryService.release(items);
      order.markAsFailed();
      await this.orderRepo.update(order.id, { status: order.status });
      throw err;
    }

    // 5. Send confirmation
    await this.emailService.sendOrderConfirmation(userId, savedOrder);

    return savedOrder;
  }
}

Exercises

  1. Build a Rich Domain Model for an Order System

    Create an OrderEntity with states: CREATED, PAID, SHIPPED, DELIVERED, CANCELLED, REFUNDED. Implement methods for pay(), ship(), deliver(), cancel(), and refund() with proper state transition validation. Write tests for all valid and invalid transitions.

    Hint

    Use a state machine approach: each method checks the current state and only transitions if valid. Throw DomainError for invalid transitions.

  2. Implement a Service Layer with Side Effects

    Build a PostService that uses a PostRepository (interface), EventBus, and NotificationService. Implement createPost, publishPost, and deletePost with proper authorization, domain validation, and event emission. Write tests using in-memory implementations.

    Hint

    Inject all dependencies as interfaces. The EventBus can be a simple in-memory implementation for testing.

  3. Refactor from Anemic to Rich Model

    Take a service with all business logic in service methods operating on plain data objects. Refactor it to use a rich domain model where entities encapsulate their own validation and state transitions. Keep the service for orchestration only.

    Hint

    Move each validation check from the service into the corresponding entity method. The service should call entity methods, not implement business rules.

Common Problems & Solutions

1 Service becomes a 'god class' with hundreds of methods
Cause: All business logic for a domain area is placed in a single service.
Solution: Split into focused services: PostQueryService, PostCommandService, PostPublishingService. Use CQRS to separate read and write operations.
2 Circular dependency between services
Cause: ServiceA depends on ServiceB and ServiceB depends on ServiceA.
Solution: Extract the shared logic into a third service. Use events for loose coupling: instead of ServiceA calling ServiceB, ServiceA emits an event that ServiceB listens to.
3 Domain entities have no access to repositories
Cause: By design, domain entities should not depend on infrastructure. But some domain logic needs data from other entities.
Solution: The service layer is responsible for loading dependencies. Pass all needed data into domain methods as parameters rather than having the entity fetch data itself.
