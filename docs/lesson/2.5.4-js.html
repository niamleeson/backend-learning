Prisma Migrate: Creating and Running Migrations

Intuition

Prisma Migrate is like a GPS navigation system for database changes. You set your destination (the schema you want), and it calculates the route (migration SQL) from where you are (current database state) to where you want to be. In development, it drives for you. In production, it follows the pre-planned route exactly.

Prisma Migrate: Creating and Running Migrations

Introduction

Prisma Migrate creates SQL migration files from changes in your Prisma schema. Unlike Knex migrations where you write SQL/JS manually, Prisma compares your schema to the current database state and generates the migration SQL automatically. This schema-first workflow means your Prisma schema is always the source of truth.

Migration Workflow

1. Edit schema.prisma
2. Run: npx prisma migrate dev --name description
3. Prisma diffs schema vs database
4. Generates SQL migration file
5. Applies migration to database
6. Regenerates Prisma Client

  schema.prisma ---> diff ---> migration.sql ---> database
       |                                            |
       +--- prisma generate ---> @prisma/client ----+

Creating Your First Migration

# Generate and apply migration
npx prisma migrate dev --name init

This creates:

prisma/
  migrations/
    20240115120000_init/
      migration.sql
    migration_lock.toml

-- prisma/migrations/20240115120000_init/migration.sql
-- CreateEnum
CREATE TYPE "Role" AS ENUM ('CUSTOMER', 'ADMIN', 'VENDOR');

-- CreateTable
CREATE TABLE "users" (
    "id" TEXT NOT NULL,
    "email" VARCHAR(320) NOT NULL,
    "name" TEXT NOT NULL,
    "password_hash" TEXT NOT NULL,
    "role" "Role" NOT NULL DEFAULT 'CUSTOMER',
    "created_at" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "updated_at" TIMESTAMP(3) NOT NULL,

    CONSTRAINT "users_pkey" PRIMARY KEY ("id")
);

-- CreateIndex
CREATE UNIQUE INDEX "users_email_key" ON "users"("email");

Adding Schema Changes

// Add a new field to User
model User {
  // ... existing fields
  phone    String?   // NEW
  avatar   String?   // NEW
}

npx prisma migrate dev --name add_phone_and_avatar

Generated SQL:

-- AlterTable
ALTER TABLE "users" ADD COLUMN "phone" TEXT;
ALTER TABLE "users" ADD COLUMN "avatar" TEXT;

Common Migration Scenarios

Adding a New Table with Relations

model Product {
  id         String   @id @default(uuid())
  name       String
  price      Decimal  @db.Decimal(10, 2)
  categoryId String?  @map("category_id")
  category   Category? @relation(fields: [categoryId], references: [id])
  @@map("products")
  @@index([categoryId])
}

model Category {
  id       String    @id @default(uuid())
  name     String    @unique
  products Product[]
  @@map("categories")
}

Renaming a Column

model User {
  // Change 'name' to 'fullName'
  fullName String @map("full_name") // rename the column
}

npx prisma migrate dev --name rename_name_to_full_name
# Prisma might generate DROP + ADD instead of RENAME
# Check the SQL and edit if needed!

Important: Prisma may generate a DROP COLUMN + ADD COLUMN instead of ALTER COLUMN RENAME. Always review the generated SQL for destructive changes!

Adding NOT NULL to Existing Column

// Step 1: Add the column as nullable with a default

// Step 2: Backfill data
// Create a custom migration:
npx prisma migrate dev --name backfill_timezone --create-only

UPDATE "users" SET "timezone" = 'UTC' WHERE "timezone" IS NULL;
ALTER TABLE "users" ALTER COLUMN "timezone" SET NOT NULL;

Migration Commands

# Development: generate + apply + regenerate client
npx prisma migrate dev --name description

# Generate migration WITHOUT applying (for review)
npx prisma migrate dev --name description --create-only

# Production: apply pending migrations
npx prisma migrate deploy

# Reset database (drops all data!)
npx prisma migrate reset

# Check migration status
npx prisma migrate status

# Resolve failed migration
npx prisma migrate resolve --applied "20240115120000_init"

Production Deployment

// package.json
{
  "scripts": {
    "build": "tsc",
    "start": "node dist/index.js",
    "migrate:deploy": "prisma migrate deploy",
    "postinstall": "prisma generate"
  }
}

# CI/CD pipeline
steps:
  - name: Install dependencies
    run: npm ci
  - name: Run migrations
    run: npx prisma migrate deploy
    env:
      DATABASE_URL: ${{ secrets.DATABASE_URL }}
  - name: Start application
    run: npm start

migrate dev vs migrate deploy:

  migrate dev (development):
    - Creates migration files
    - Applies them
    - Regenerates client
    - Can reset database
    - Interactive prompts

  migrate deploy (production):
    - ONLY applies existing migration files
    - No file generation
    - No interactive prompts
    - Never resets database
    - Safe for CI/CD

Seeding with Prisma

// prisma/seed.ts
import { PrismaClient } from '@prisma/client';
const prisma = new PrismaClient();

async function main() {
  // Clear existing data
  await prisma.orderItem.deleteMany();
  await prisma.order.deleteMany();
  await prisma.user.deleteMany();

  // Seed users
  const alice = await prisma.user.create({
    data: {
      email: 'alice@example.com',
      name: 'Alice',
      passwordHash: '$2b$10$...',
      role: 'ADMIN',
    },
  });

  console.log('Seeded:', { alice });
}

main()
  .catch(console.error)
  .finally(() => prisma.$disconnect());

// package.json
{
  "prisma": {
    "seed": "tsx prisma/seed.ts"
  }
}

npx prisma db seed
# Or automatically with: npx prisma migrate reset

Exercises

  1. Migration Evolution

    Starting from an empty schema, create 5 sequential migrations that: (1) create users table, (2) create posts table with user FK, (3) add tags and post_tags tables, (4) add a 'views' counter to posts, (5) rename 'views' to 'view_count'. Run migrate:latest, then migrate:rollback, then migrate:latest again.

    Hint

    Use --create-only for step 5 to review the SQL. Prisma might try to drop and recreate instead of rename — edit the SQL manually.

  2. Safe NOT NULL Migration

    You have a users table with 100k rows and need to add a non-nullable 'timezone' column. Write a migration that: adds the column as nullable, backfills with 'UTC', then makes it NOT NULL. Test with real data.

    Hint

    Use --create-only, then edit the SQL to include the UPDATE statement before the NOT NULL alteration.

  3. CI/CD Migration Pipeline

    Write a script that: checks migration status, applies pending migrations, runs the seed if the database is empty, and starts the application. Handle errors gracefully (failed migrations should not start the app).

    Hint

    Use npx prisma migrate status to check, npx prisma migrate deploy to apply. Check exit codes.

Common Problems & Solutions

1 Migration generates DROP COLUMN instead of RENAME
Cause: Prisma doesn't always detect renames. It sees the old column disappear and a new one appear.
Solution: Use `--create-only` to generate the migration file without applying it. Edit the SQL to use `ALTER TABLE ... RENAME COLUMN` instead of DROP + ADD. Then apply with `npx prisma migrate dev`.
2 Drift detected: database schema differs from migration history
Cause: Someone modified the database directly (outside of migrations), or `prisma db push` was used.
Solution: Run `npx prisma migrate dev` which will detect the drift and either reset or create a fix migration. For production, run `npx prisma migrate resolve` to mark the issue as resolved.
3 Failed migration leaves database in inconsistent state
Cause: Migration partially applied before failing. PostgreSQL wraps DDL in transactions, so this is rare, but can happen with data migrations.
Solution: Use `npx prisma migrate resolve --rolled-back '20240115_migration_name'` to mark it as rolled back. Fix the migration SQL, then re-apply.
