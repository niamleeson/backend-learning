API Gateway and Service Discovery

Intuition

An API gateway is like a hotel concierge who knows all the services in the city—you tell them what you need, and they route you there. Service discovery is the concierge's constantly updated directory of businesses and their addresses.

API Gateway & Service Discovery

In a microservices architecture, clients need a single entry point to reach many services, and services need to find each other dynamically. API gateways and service discovery solve these complementary problems.

Core Concepts

  • API Gateway: A reverse proxy that serves as the single entry point for all client requests. It handles routing, authentication, rate limiting, request transformation, response aggregation, and TLS termination.
  • Backend for Frontend (BFF): A variant where each client type (web, mobile, IoT) gets its own gateway tailored to its needs, reducing over-fetching and simplifying client logic.
  • Service Discovery: The mechanism by which services find each other's network locations. Can be client-side (the client queries a registry and load-balances itself) or server-side (a load balancer queries the registry).
  • Service Registry: A database of available service instances and their locations. Examples: Consul, Eureka, etcd, Kubernetes DNS.
  • Health Checks: Services register themselves and send periodic heartbeats. The registry deregisters unhealthy instances so traffic is not routed to them.

Key Takeaways

  1. An API gateway simplifies client interactions but can become a bottleneck and single point of failure—deploy it with high availability.
  2. Service discovery eliminates hard-coded URLs and enables dynamic scaling, deployments, and failover.
  3. In Kubernetes, service discovery is built in via DNS and the Services abstraction, reducing the need for external registries.

Exercises

  1. Build an API Gateway with Kong or Traefik

    Deploy Kong or Traefik as an API gateway in front of three microservices. Configure path-based routing (/users to user-service, /orders to order-service), rate limiting (100 req/min per IP), and JWT authentication.

    Hint

    For Kong: install plugins with curl -X POST http://kong:8001/services/{service}/plugins --data 'name=rate-limiting' --data 'config.minute=100'. For Traefik: use Docker labels or file-based configuration.

  2. Implement Service Discovery with Consul

    Set up a Consul cluster and register three service instances. Write a Python client that queries Consul's HTTP API to discover healthy instances and implements client-side load balancing with round-robin.

    Hint

    Register with PUT /v1/agent/service/register. Discover with GET /v1/health/service/{name}?passing=true. Parse the response to extract IP and port for each healthy instance.

Common Problems & Solutions

1 The API gateway becomes a bottleneck, adding significant latency to all requests.
Cause: The gateway is performing too many operations (authentication, transformation, logging) synchronously, or it is under-provisioned for the traffic volume.
Solution: Scale the gateway horizontally behind a network load balancer. Offload heavy operations (logging, analytics) to async pipelines. Cache authentication tokens at the gateway. Consider splitting into BFF gateways per client type.
2 Services intermittently fail to discover each other, causing 503 errors.
Cause: The service registry has stale entries from crashed instances that did not deregister, or health check intervals are too long.
Solution: Configure aggressive health check intervals (5-10 seconds) with deregister-after timeout. Implement client-side caching of discovered endpoints with short TTLs and fallback to last-known-good addresses.
