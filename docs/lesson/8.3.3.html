Certificates, Certificate Authorities, and mTLS

Intuition

Think of certificates like passports. A Certificate Authority is like a government passport office — it verifies your identity and issues an official document (certificate) that others trust because they trust the issuing authority. Mutual TLS is like two people at a border crossing both showing their passports to each other before being allowed to proceed.

Certificates, Certificate Authorities, and mTLS

In the previous lessons, we explored TLS handshakes and symmetric/asymmetric cryptography. Now we turn to the trust infrastructure that makes secure communication possible at scale: X.509 certificates, Certificate Authorities (CAs), and mutual TLS (mTLS). These are the building blocks that let two parties on the internet — or two microservices in your data center — verify each other's identity before exchanging sensitive data.

X.509 Certificate Structure

An X.509 certificate is a standardized digital document that binds a public key to an identity. Think of it as a cryptographically signed ID card. Every TLS connection you make — whether browsing the web or calling an internal API — relies on X.509 certificates.

Here is the high-level structure of an X.509v3 certificate:

Field Description
Version Almost always v3 (value 2 in ASN.1 encoding)
Serial Number Unique identifier assigned by the CA
Signature Algorithm Algorithm the CA used to sign (e.g., SHA-256 with RSA)
Issuer Distinguished Name (DN) of the CA that signed this cert
Validity Not Before and Not After timestamps
Subject Distinguished Name of the entity this cert represents
Subject Public Key Info The public key and its algorithm
Extensions v3 extensions like Subject Alternative Names (SANs), Key Usage, Basic Constraints
Signature The CA's digital signature over all the above fields

The Subject Alternative Name (SAN) extension is particularly important in modern TLS. It lists the DNS names and/or IP addresses the certificate is valid for. Browsers and clients check SANs rather than the older Common Name (CN) field.

Let's inspect a certificate programmatically using Python's cryptography library:

from cryptography import x509
from cryptography.hazmat.backends import default_backend

# Load a PEM-encoded certificate
with open("server.crt", "rb") as f:
    cert = x509.load_pem_x509_certificate(f.read(), default_backend())

print(f"Subject: {cert.subject}")
print(f"Issuer: {cert.issuer}")
print(f"Serial Number: {cert.serial_number}")
print(f"Not Valid Before: {cert.not_valid_before_utc}")
print(f"Not Valid After: {cert.not_valid_after_utc}")
print(f"Signature Algorithm: {cert.signature_algorithm_oid._name}")

# Extract Subject Alternative Names
try:
    san_ext = cert.extensions.get_extension_for_class(x509.SubjectAlternativeName)
    print(f"SANs: {san_ext.value.get_values_for_type(x509.DNSName)}")
except x509.ExtensionNotFound:
    print("No SAN extension found")

Certificate Chains and How CAs Work

A Certificate Authority (CA) is a trusted entity that issues certificates. The trust model is hierarchical:

  1. Root CA: A self-signed certificate that is pre-installed in operating systems and browsers as a trust anchor. Root CAs are kept offline for security.
  2. Intermediate CA: A certificate signed by the root CA (or another intermediate). These handle day-to-day issuance.
  3. Leaf (End-Entity) Certificate: The certificate your server presents, signed by an intermediate CA.

When a client connects to a server, it receives the leaf certificate and any intermediate certificates. The client then builds a chain of trust:

Leaf Cert (your server)
  └── signed by Intermediate CA
        └── signed by Root CA (trusted, in local store)

The client validates each link: it checks the signature, ensures the cert is within its validity period, verifies the issuer matches the next certificate's subject, and confirms the root is in its trust store. If any link fails, the connection is rejected.

Why Intermediates?

If a root CA's private key is compromised, every certificate it ever issued becomes suspect. By using intermediates, the root key stays offline in a hardware security module (HSM). If an intermediate is compromised, only that intermediate's certificates need to be revoked — the root remains safe.

Certificate Signing Requests (CSRs)

To obtain a certificate from a CA, you generate a Certificate Signing Request (CSR). A CSR contains your public key and identity information, signed with your private key (to prove you possess it). The CA verifies your identity, then issues a certificate.

Here is how to generate a private key and CSR with the cryptography library:

from cryptography import x509
from cryptography.x509.oid import NameOID
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.hazmat.primitives.asymmetric import rsa

# Generate a private key
private_key = rsa.generate_private_key(
    public_exponent=65537,
    key_size=2048,
)

# Save the private key
with open("server.key", "wb") as f:
    f.write(private_key.private_bytes(
        encoding=serialization.Encoding.PEM,
        format=serialization.PrivateFormat.TraditionalOpenSSL,
        encryption_algorithm=serialization.NoEncryption(),
    ))

# Build the CSR
csr = (
    x509.CertificateSigningRequestBuilder()
    .subject_name(x509.Name([
        x509.NameAttribute(NameOID.COUNTRY_NAME, "US"),
        x509.NameAttribute(NameOID.STATE_OR_PROVINCE_NAME, "California"),
        x509.NameAttribute(NameOID.ORGANIZATION_NAME, "My Backend Service"),
        x509.NameAttribute(NameOID.COMMON_NAME, "api.example.com"),
    ]))
    .add_extension(
        x509.SubjectAlternativeName([
            x509.DNSName("api.example.com"),
            x509.DNSName("api-internal.example.com"),
        ]),
        critical=False,
    )
    .sign(private_key, hashes.SHA256())
)

# Save the CSR
with open("server.csr", "wb") as f:
    f.write(csr.public_bytes(serialization.Encoding.PEM))

print("CSR generated successfully")

Self-Signed vs CA-Signed Certificates

Self-signed certificates are certificates where the issuer and subject are the same entity — you sign your own certificate with your own private key. They are free and instant to create, making them useful for:

  • Local development and testing
  • Internal infrastructure where you control the trust store
  • Acting as your own root CA

However, self-signed certs are not trusted by default by browsers or other clients because no independent third party has vouched for the identity.

CA-signed certificates are issued by a recognized Certificate Authority. The CA verifies your identity (at varying levels of rigor) before signing. These are trusted automatically because the CA's root certificate is in the client's trust store.

Here is how to create a self-signed certificate for development:

import datetime
from cryptography import x509
from cryptography.x509.oid import NameOID
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.hazmat.primitives.asymmetric import rsa

# Generate CA private key
ca_key = rsa.generate_private_key(public_exponent=65537, key_size=4096)

# Build self-signed CA certificate
ca_name = x509.Name([
    x509.NameAttribute(NameOID.COMMON_NAME, "My Dev CA"),
    x509.NameAttribute(NameOID.ORGANIZATION_NAME, "Dev Team"),
])

ca_cert = (
    x509.CertificateBuilder()
    .subject_name(ca_name)
    .issuer_name(ca_name)  # self-signed: issuer == subject
    .public_key(ca_key.public_key())
    .serial_number(x509.random_serial_number())
    .not_valid_before(datetime.datetime.now(datetime.timezone.utc))
    .not_valid_after(datetime.datetime.now(datetime.timezone.utc) + datetime.timedelta(days=3650))
    .add_extension(
        x509.BasicConstraints(ca=True, path_length=None),
        critical=True,
    )
    .add_extension(
        x509.KeyUsage(
            digital_signature=True, key_cert_sign=True, crl_sign=True,
            content_commitment=False, key_encipherment=False,
            data_encipherment=False, key_agreement=False,
            encipher_only=False, decipher_only=False,
        ),
        critical=True,
    )
    .sign(ca_key, hashes.SHA256())
)

with open("ca.crt", "wb") as f:
    f.write(ca_cert.public_bytes(serialization.Encoding.PEM))

print("Self-signed CA certificate created")

Let's Encrypt and the ACME Protocol

Let's Encrypt revolutionized TLS adoption by providing free, automated certificates. It uses the ACME (Automatic Certificate Management Environment) protocol (RFC 8555) to verify domain ownership and issue certificates without human intervention.

The ACME flow works like this:

  1. Your ACME client (e.g., certbot) contacts the Let's Encrypt server and requests a certificate for api.example.com.
  2. Let's Encrypt responds with a challenge — typically an HTTP-01 challenge (place a specific file at http://api.example.com/.well-known/acme-challenge/TOKEN) or a DNS-01 challenge (create a specific DNS TXT record).
  3. Your client completes the challenge, proving you control the domain.
  4. Let's Encrypt verifies the challenge and issues the certificate.
  5. The certificate is valid for 90 days, encouraging automated renewal.

For backend services, you can use the acme Python library to automate this programmatically, but in practice most teams use certbot or integrate with their cloud provider's certificate management (e.g., AWS ACM, GCP Certificate Manager).

Certificate Pinning

Certificate pinning is a technique where a client is configured to accept only specific certificates or public keys, rather than trusting any certificate signed by any CA in the trust store. This defends against compromised CAs or misissued certificates.

There are two approaches:

  • Certificate pinning: Pin the exact certificate (fragile — breaks on renewal)
  • Public key pinning: Pin the public key (survives certificate renewal if you reuse the key)
import hashlib
import ssl
import socket

def connect_with_pin(hostname, port, expected_pin):
    """Connect to a server and verify its public key pin."""
    context = ssl.create_default_context()

    with socket.create_connection((hostname, port)) as sock:
        with context.wrap_socket(sock, server_hostname=hostname) as ssock:
            # Get the server's certificate in DER format
            der_cert = ssock.getpeercert(binary_form=True)

            # Compute SHA-256 hash of the certificate
            cert_hash = hashlib.sha256(der_cert).hexdigest()

            if cert_hash != expected_pin:
                raise ssl.SSLError(
                    f"Certificate pin mismatch! "
                    f"Expected {expected_pin}, got {cert_hash}"
                )

            print(f"Pin verified for {hostname}")
            return ssock

Warning: Certificate pinning can cause outages if you pin the wrong thing or forget to update pins before certificate rotation. Many organizations have moved away from HTTP Public Key Pinning (HPKP) for this reason. Use it judiciously, primarily for high-security service-to-service communication.

Mutual TLS (mTLS)

In standard TLS, only the server presents a certificate. The client verifies the server's identity, but the server has no cryptographic proof of the client's identity. Mutual TLS (mTLS) adds a second step: the server also requests and verifies a certificate from the client.

mTLS is the gold standard for service-to-service authentication in microservice architectures. Instead of relying on API keys or tokens that can be stolen, each service has its own certificate issued by an internal CA. The identity is tied to the cryptographic key pair.

How mTLS Works

  1. Client connects to server and initiates TLS handshake.
  2. Server sends its certificate; client verifies it.
  3. Server sends a CertificateRequest message.
  4. Client sends its own certificate.
  5. Server verifies the client's certificate against its trusted CA.
  6. Both sides derive session keys and begin encrypted communication.

Implementing mTLS in Python

Let's build a complete mTLS example. First, we need to generate certificates for both server and client, signed by our CA.

import datetime
from cryptography import x509
from cryptography.x509.oid import NameOID
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.hazmat.primitives.asymmetric import rsa


def generate_signed_cert(ca_cert, ca_key, common_name, san_names, filename_prefix):
    """Generate a certificate signed by our CA."""
    # Generate key for the entity
    entity_key = rsa.generate_private_key(public_exponent=65537, key_size=2048)

    # Build the certificate
    subject = x509.Name([
        x509.NameAttribute(NameOID.COMMON_NAME, common_name),
    ])

    cert = (
        x509.CertificateBuilder()
        .subject_name(subject)
        .issuer_name(ca_cert.subject)
        .public_key(entity_key.public_key())
        .serial_number(x509.random_serial_number())
        .not_valid_before(datetime.datetime.now(datetime.timezone.utc))
        .not_valid_after(
            datetime.datetime.now(datetime.timezone.utc) + datetime.timedelta(days=365)
        )
        .add_extension(
            x509.SubjectAlternativeName(
                [x509.DNSName(name) for name in san_names]
            ),
            critical=False,
        )
        .add_extension(
            x509.BasicConstraints(ca=False, path_length=None),
            critical=True,
        )
        .sign(ca_key, hashes.SHA256())  # Signed by the CA
    )

    # Write key and cert to files
    with open(f"{filename_prefix}.key", "wb") as f:
        f.write(entity_key.private_bytes(
            encoding=serialization.Encoding.PEM,
            format=serialization.PrivateFormat.TraditionalOpenSSL,
            encryption_algorithm=serialization.NoEncryption(),
        ))

    with open(f"{filename_prefix}.crt", "wb") as f:
        f.write(cert.public_bytes(serialization.Encoding.PEM))

    return cert, entity_key


# Assume ca_cert and ca_key from the self-signed CA example above
# Generate server certificate
generate_signed_cert(
    ca_cert, ca_key,
    common_name="my-server",
    san_names=["localhost", "my-server.internal"],
    filename_prefix="server",
)

# Generate client certificate
generate_signed_cert(
    ca_cert, ca_key,
    common_name="my-client-service",
    san_names=["my-client-service.internal"],
    filename_prefix="client",
)

Now, the mTLS server:

import ssl
import socket

def create_mtls_server(host="localhost", port=8443):
    """Create a server that requires client certificates."""
    context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)

    # Load the server's certificate and private key
    context.load_cert_chain(certfile="server.crt", keyfile="server.key")

    # Require client certificates and specify the trusted CA
    context.verify_mode = ssl.CERT_REQUIRED
    context.load_verify_locations(cafile="ca.crt")

    # Security hardening
    context.minimum_version = ssl.TLSVersion.TLSv1_3

    with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as sock:
        sock.bind((host, port))
        sock.listen(5)
        print(f"mTLS server listening on {host}:{port}")

        with context.wrap_socket(sock, server_side=True) as ssock:
            conn, addr = ssock.accept()
            with conn:
                # Extract the client's certificate information
                client_cert = conn.getpeercert()
                client_cn = dict(
                    x[0] for x in client_cert["subject"]
                ).get("commonName", "unknown")

                print(f"Client authenticated: {client_cn}")

                data = conn.recv(1024)
                print(f"Received: {data.decode()}")
                conn.sendall(b"Hello from mTLS server!")


create_mtls_server()

And the mTLS client:

import ssl
import socket

def create_mtls_client(host="localhost", port=8443):
    """Connect to a server using mutual TLS."""
    context = ssl.SSLContext(ssl.PROTOCOL_TLS_CLIENT)

    # Load the client's certificate and private key
    context.load_cert_chain(certfile="client.crt", keyfile="client.key")

    # Trust our internal CA
    context.load_verify_locations(cafile="ca.crt")

    with socket.create_connection((host, port)) as sock:
        with context.wrap_socket(sock, server_hostname="localhost") as ssock:
            print(f"Connected with: {ssock.version()}")

            ssock.sendall(b"Hello from mTLS client!")
            response = ssock.recv(1024)
            print(f"Server response: {response.decode()}")


create_mtls_client()

The critical line in the server is context.verify_mode = ssl.CERT_REQUIRED. Without this, the server will not ask the client for a certificate, and you are back to regular one-way TLS.

mTLS in Production: Service Meshes

In modern cloud-native architectures, tools like Istio, Linkerd, and Consul Connect handle mTLS transparently. They inject sidecar proxies that manage certificate issuance, rotation, and mTLS negotiation automatically. Your application code communicates over plain HTTP to the local sidecar, and the sidecar handles mTLS to the destination sidecar. This is called transparent mTLS and dramatically reduces the complexity for application developers.

Certificate Revocation

Certificates sometimes need to be invalidated before their expiration date — for example, if a private key is compromised. There are two main revocation mechanisms:

Certificate Revocation Lists (CRLs)

A CRL is a signed list of revoked certificate serial numbers published by the CA. Clients download the CRL periodically and check if a certificate's serial number appears on it.

Drawbacks: CRLs can grow large, and there is a window between revocation and the next CRL update where a revoked cert is still accepted.

Online Certificate Status Protocol (OCSP)

OCSP allows clients to query the CA in real-time: "Is this specific certificate still valid?" The CA responds with good, revoked, or unknown.

OCSP Stapling improves on this by having the server periodically fetch its own OCSP response and "staple" it to the TLS handshake, so the client does not need to contact the CA separately. This improves both performance and privacy.

import ssl

# Enable OCSP stapling on a server context
context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
context.load_cert_chain(certfile="server.crt", keyfile="server.key")

# Check for OCSP on the client side
client_context = ssl.create_default_context()
# Python's ssl module has limited built-in OCSP support;
# for production use, consider the 'ocspchecker' or
# 'cryptography' library for OCSP request building:

from cryptography.x509 import ocsp
from cryptography.x509.ocsp import OCSPRequestBuilder
from cryptography.hazmat.primitives import hashes

# Build an OCSP request
ocsp_request = (
    OCSPRequestBuilder()
    .add_certificate(cert, ca_cert, hashes.SHA256())
    .build()
)

# The serialized request would be sent via HTTP POST to the
# CA's OCSP responder URL (found in the cert's AIA extension)
ocsp_request_data = ocsp_request.public_bytes(serialization.Encoding.DER)

Security Best Practices

Here is a checklist for production certificate management:

  1. Use short-lived certificates (90 days or less) and automate renewal.
  2. Never commit private keys to version control.
  3. Use at least 2048-bit RSA keys or 256-bit ECDSA keys.
  4. Always include SANs — do not rely solely on CN.
  5. Set minimum_version = TLSv1_2 (or TLSv1_3 if possible) in your SSL context.
  6. Rotate certificates and keys before expiration, with overlap periods.
  7. Store private keys in HSMs or secret management systems (e.g., HashiCorp Vault, AWS Secrets Manager).
  8. Monitor certificate expiration with alerting (e.g., Prometheus with x509_cert_expiry).
  9. For mTLS, use a dedicated internal CA — do not use a public CA for service identities.
  10. Implement OCSP stapling on your servers to handle revocation checking efficiently.

Putting It All Together

Certificates and CAs are the trust foundation of TLS. Standard TLS gives you server authentication — you know you are talking to the right server. mTLS extends this to mutual authentication — both sides prove their identity cryptographically. In a microservices world, mTLS replaces fragile shared secrets and API keys with strong, rotatable, auditable cryptographic identities.

The key concepts to internalize:

  • X.509 certificates bind public keys to identities and are signed by a trusted CA.
  • Certificate chains build trust from a leaf cert up to a trusted root.
  • CSRs are how you request a certificate from a CA without exposing your private key.
  • Let's Encrypt and ACME automated free certificate issuance for the public web.
  • mTLS requires both client and server to present certificates, enabling strong service-to-service authentication.
  • Revocation (CRL/OCSP) handles compromised certificates before they expire.

Mastering these concepts is essential for building secure backend systems, whether you are exposing public APIs or orchestrating internal microservices.

Common Problems & Solutions

1 SSL handshake fails with 'certificate verify failed' when connecting to an internal service using mTLS
Symptoms: ssl.SSLCertVerificationError: [SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed: unable to get local issuer certificate
Cause: The client does not have the CA certificate that signed the server's certificate in its trust store, or the server is not sending the full certificate chain (missing intermediate certificates).
Solution: Ensure the client loads the correct CA certificate with load_verify_locations(). On the server side, make sure the cert chain file includes both the leaf certificate and all intermediate certificates. Verify the chain independently with OpenSSL.

Client fix: explicitly load your internal CA

context = ssl.SSLContext(ssl.PROTOCOL_TLS_CLIENT) context.load_verify_locations(cafile='/path/to/internal-ca.crt')

Server fix: use a chain file that includes intermediates

Concatenate: cat server.crt intermediate.crt > server-chain.crt

context.load_cert_chain( certfile='server-chain.crt', keyfile='server.key' )

2 mTLS server accepts connections without client certificates
Symptoms: Clients can connect successfully even without presenting a certificate, defeating the purpose of mutual authentication.
Cause: The server's SSL context has verify_mode set to ssl.CERT_NONE (the default) or ssl.CERT_OPTIONAL instead of ssl.CERT_REQUIRED.
Solution: Set verify_mode to ssl.CERT_REQUIRED and ensure load_verify_locations is called with the CA that signed client certificates.

Incorrect: clients are not required to authenticate

context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER) context.load_cert_chain('server.crt', 'server.key')

context.verify_mode defaults to ssl.CERT_NONE

Correct: require and verify client certificates

context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER) context.load_cert_chain('server.crt', 'server.key') context.verify_mode = ssl.CERT_REQUIRED # This is the key line context.load_verify_locations(cafile='client-ca.crt')

3 Certificate works in browser but fails in Python requests/urllib
Symptoms: requests.exceptions.SSLError or urllib.error.URLError with 'hostname mismatch' even though the certificate is valid.
Cause: The certificate's Subject Alternative Name (SAN) does not include the hostname you are connecting to. Browsers may fall back to checking the Common Name (CN), but Python's ssl module strictly requires a SAN match as per RFC 6125.
Solution: Regenerate the certificate with the correct hostnames listed in the SAN extension. Always include all hostnames and IP addresses your service will be accessed by.

When generating the CSR or certificate, include all access names in SAN

from cryptography import x509

san = x509.SubjectAlternativeName([ x509.DNSName('api.example.com'), x509.DNSName('api-internal.example.com'), x509.DNSName('localhost'), # For local dev x509.IPAddress(ipaddress.IPv4Address('127.0.0.1')), # IP access ])

builder = builder.add_extension(san, critical=False)
