Structured Logging with structlog and JSON

Intuition

Think of structured logging like the difference between a handwritten diary and a spreadsheet. A diary entry says 'Sold 3 red widgets to John for $45 on Tuesday' — it is readable but hard to analyze at scale. A spreadsheet row with columns for quantity, color, product, customer, amount, and date lets you instantly sort, filter, and aggregate thousands of entries. structlog turns your application's diary entries into spreadsheet rows that your observability tools can query in milliseconds.

Structured Logging with structlog and JSON

Logging is one of those aspects of software engineering that separates production-ready systems from toy projects. When your application handles thousands of requests per second across dozens of microservices, the difference between structured and unstructured logging becomes the difference between diagnosing an incident in minutes versus hours.

Why Structured Logging Matters

Traditional logging looks like this:

import logging

logger = logging.getLogger(__name__)
logger.info("User 12345 placed order 67890 for $99.99")

This produces a human-readable string:

2026-03-17 10:23:45,123 INFO User 12345 placed order 67890 for $99.99

Now imagine you need to answer: "How many orders did user 12345 place in the last hour?" or "What is the average order value across all users?" You would need to parse that string with regex or string splitting — fragile, slow, and error-prone.

Structured logging emits log entries as key-value pairs, typically serialized as JSON:

{
  "event": "order_placed",
  "user_id": 12345,
  "order_id": 67890,
  "amount": 99.99,
  "currency": "USD",
  "timestamp": "2026-03-17T10:23:45.123Z",
  "level": "info"
}

With structured logs, tools like Elasticsearch, Datadog, Splunk, and Loki can index individual fields. You can filter by user_id, aggregate by amount, and alert on specific event types — all without writing a single regex.

The Core Advantages

  1. Machine-parseable: Log aggregation systems can index and query fields directly.
  2. Consistent schema: Every log entry follows a predictable structure.
  3. Rich context: You can attach arbitrary metadata (request IDs, trace IDs, user context) without string formatting gymnastics.
  4. Better correlation: Tracking a request across services becomes trivial with bound context.
  5. Easier alerting: Set up alerts on specific fields and values rather than pattern matching.

Limitations of Python's stdlib logging

Python's built-in logging module is powerful but was designed in an era of unstructured, text-based logs. While you can make it produce JSON output (with custom formatters), the ergonomics are clunky:

import logging
import json

class JSONFormatter(logging.Formatter):
    def format(self, record):
        log_entry = {
            "timestamp": self.formatTime(record),
            "level": record.levelname,
            "message": record.getMessage(),
            "logger": record.name,
            "module": record.module,
        }
        # Adding extra fields requires convention and discipline
        if hasattr(record, "user_id"):
            log_entry["user_id"] = record.user_id
        return json.dumps(log_entry)

handler = logging.StreamHandler()
handler.setFormatter(JSONFormatter())
logger = logging.getLogger("myapp")
logger.addHandler(handler)
logger.setLevel(logging.INFO)

# Passing extra context is awkward
logger.info("Order placed", extra={"user_id": 12345})

The problems compound quickly: there is no built-in way to bind context that persists across log calls, the extra dict is easy to forget, and building a processor pipeline requires significant custom code. This is where structlog shines.

Introduction to structlog

structlog is a Python library that makes structured logging natural and powerful. It wraps loggers (its own or stdlib's) with a processor pipeline architecture that transforms log events through a chain of functions.

Install it:

pip install structlog

Basic Usage

import structlog

log = structlog.get_logger()

log.info("order_placed", user_id=12345, order_id=67890, amount=99.99)

With default configuration, this outputs a colorized, human-readable format in development. But the real power comes from configuring it for production.

Configuring structlog for Production

A production-ready structlog configuration typically looks like this:

import structlog
import logging
import sys

def configure_logging(json_output: bool = True, log_level: str = "INFO"):
    """Configure structlog for production use."""

    shared_processors = [
        structlog.contextvars.merge_contextvars,
        structlog.stdlib.add_log_level,
        structlog.stdlib.add_logger_name,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
        structlog.processors.UnicodeDecoder(),
    ]

    if json_output:
        # Production: JSON output
        renderer = structlog.processors.JSONRenderer()
    else:
        # Development: colorized console output
        renderer = structlog.dev.ConsoleRenderer(colors=True)

    structlog.configure(
        processors=[
            *shared_processors,
            structlog.stdlib.ProcessorFormatter.wrap_for_formatter,
        ],
        logger_factory=structlog.stdlib.LoggerFactory(),
        wrapper_class=structlog.stdlib.BoundLogger,
        cache_logger_on_first_use=True,
    )

    formatter = structlog.stdlib.ProcessorFormatter(
        processors=[
            structlog.stdlib.ProcessorFormatter.remove_processors_meta,
            renderer,
        ],
        foreign_pre_chain=shared_processors,
    )

    handler = logging.StreamHandler(sys.stdout)
    handler.setFormatter(formatter)

    root_logger = logging.getLogger()
    root_logger.handlers.clear()
    root_logger.addHandler(handler)
    root_logger.setLevel(getattr(logging, log_level.upper()))

This configuration achieves several things: it integrates with Python's stdlib logging so third-party libraries' log output also gets structured, it uses ISO timestamps for consistent time formatting, and it switches between human-readable and JSON output based on the environment.

Understanding the Processor Pipeline

The processor pipeline is structlog's most powerful concept. Each processor is a callable that receives the logger, the method name, and an event dict, and returns a (potentially modified) event dict:

def add_environment(logger, method_name, event_dict):
    """Add deployment environment to every log entry."""
    event_dict["environment"] = os.environ.get("APP_ENV", "development")
    return event_dict

def add_hostname(logger, method_name, event_dict):
    """Add hostname for identifying which instance emitted the log."""
    import socket
    event_dict["hostname"] = socket.gethostname()
    return event_dict

def censor_sensitive_data(logger, method_name, event_dict):
    """Remove or mask sensitive fields before they reach output."""
    sensitive_keys = {"password", "token", "secret", "api_key", "ssn", "credit_card"}
    for key in list(event_dict.keys()):
        if key in sensitive_keys:
            event_dict[key] = "***REDACTED***"
        elif key == "email" and isinstance(event_dict[key], str):
            parts = event_dict[key].split("@")
            if len(parts) == 2:
                event_dict[key] = f"{parts[0][:2]}***@{parts[1]}"
    return event_dict

You add these to your processor chain:

structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,
        add_environment,
        add_hostname,
        censor_sensitive_data,
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
    # ... rest of config
)

Processors execute in order, so place sensitive data filtering before the renderer but after context merging.

Binding Context

One of structlog's killer features is context binding. You can create loggers with persistent context that flows through all subsequent log calls.

Instance-Level Binding

import structlog

log = structlog.get_logger()

# Bind context for this logger instance
request_log = log.bind(request_id="abc-123", user_id=42)

request_log.info("processing_request", path="/api/orders")
# Output includes request_id and user_id automatically

request_log.info("database_query", table="orders", duration_ms=45)
# Same context, different event

Context Variables (Thread/Async Safe)

For web applications, you typically want context bound per-request. structlog's contextvars integration handles this correctly for both threaded and async code:

import structlog
from structlog.contextvars import bind_contextvars, clear_contextvars, unbind_contextvars

# In your middleware or request handler setup:
def logging_middleware(request, call_next):
    clear_contextvars()  # Clean slate for each request

    bind_contextvars(
        request_id=request.headers.get("X-Request-ID", str(uuid.uuid4())),
        user_id=getattr(request.state, "user_id", None),
        method=request.method,
        path=request.url.path,
    )

    response = call_next(request)

    return response

Now every log call anywhere in the request handling chain automatically includes the request ID, user ID, method, and path — without passing a logger instance around.

Here is a complete FastAPI example:

import uuid
import time
import structlog
from structlog.contextvars import bind_contextvars, clear_contextvars
from fastapi import FastAPI, Request
from starlette.middleware.base import BaseHTTPMiddleware

app = FastAPI()
log = structlog.get_logger()

class LoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        clear_contextvars()

        request_id = request.headers.get("X-Request-ID", str(uuid.uuid4()))
        start_time = time.perf_counter()

        bind_contextvars(
            request_id=request_id,
            method=request.method,
            path=request.url.path,
            client_ip=request.client.host,
        )

        log.info("request_started")

        try:
            response = await call_next(request)
            duration_ms = (time.perf_counter() - start_time) * 1000
            log.info(
                "request_completed",
                status_code=response.status_code,
                duration_ms=round(duration_ms, 2),
            )
            response.headers["X-Request-ID"] = request_id
            return response
        except Exception as exc:
            duration_ms = (time.perf_counter() - start_time) * 1000
            log.error(
                "request_failed",
                error=str(exc),
                error_type=type(exc).__name__,
                duration_ms=round(duration_ms, 2),
            )
            raise

app.add_middleware(LoggingMiddleware)

@app.get("/api/orders/{order_id}")
async def get_order(order_id: int):
    log.info("fetching_order", order_id=order_id)
    # Every log call here automatically has request_id, method, path
    order = await fetch_order_from_db(order_id)
    log.info("order_fetched", order_id=order_id, status=order.status)
    return order

Correlation IDs Across Services

In a microservice architecture, a single user action might trigger calls across multiple services. Correlation IDs (also called trace IDs) let you follow a request's journey.

import httpx
import structlog
from structlog.contextvars import get_contextvars

log = structlog.get_logger()

async def call_payment_service(order_id: int, amount: float):
    """Call the payment service, propagating the correlation ID."""
    ctx = get_contextvars()  # Get current bound context
    request_id = ctx.get("request_id", str(uuid.uuid4()))

    log.info("calling_payment_service", order_id=order_id, amount=amount)

    async with httpx.AsyncClient() as client:
        response = await client.post(
            "http://payment-service/api/charge",
            json={"order_id": order_id, "amount": amount},
            headers={
                "X-Request-ID": request_id,
                "X-Correlation-ID": request_id,
            },
        )

    log.info(
        "payment_service_response",
        status_code=response.status_code,
        order_id=order_id,
    )
    return response.json()

The receiving service extracts X-Correlation-ID from the incoming request and binds it to its own logging context. Now you can search your log aggregation tool for a single correlation ID and see the entire trace across all services.

JSON Output and Log Aggregation

JSON logs are the lingua franca of modern observability stacks. When structlog is configured with JSONRenderer, each log line is a single JSON object on one line (JSONL format), which is what tools like Filebeat, Fluentd, and the Datadog agent expect.

import structlog

structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(sort_keys=True),
    ],
    wrapper_class=structlog.stdlib.BoundLogger,
    logger_factory=structlog.PrintLoggerFactory(),
)

log = structlog.get_logger()
log.info("server_started", port=8080, workers=4)

Output:

{"event": "server_started", "level": "info", "port": 8080, "timestamp": "2026-03-17T10:30:00.000000Z", "workers": 4}

Custom JSON Serialization

Sometimes you need to handle types that are not JSON-serializable by default:

import datetime
import decimal
import uuid
import json

def custom_json_serializer(obj, **kwargs):
    """Handle non-standard types in log entries."""
    def default(o):
        if isinstance(o, datetime.datetime):
            return o.isoformat()
        if isinstance(o, datetime.date):
            return o.isoformat()
        if isinstance(o, decimal.Decimal):
            return float(o)
        if isinstance(o, uuid.UUID):
            return str(o)
        if isinstance(o, set):
            return list(o)
        if isinstance(o, bytes):
            return o.decode("utf-8", errors="replace")
        raise TypeError(f"Object of type {type(o).__name__} is not JSON serializable")

    return json.dumps(obj, default=default, **kwargs)

structlog.configure(
    processors=[
        # ... other processors
        structlog.processors.JSONRenderer(serializer=custom_json_serializer),
    ],
    # ...
)

Integrating with stdlib Logging

Many third-party libraries (SQLAlchemy, urllib3, celery) use Python's stdlib logging. The ProcessorFormatter ensures those logs also go through your structlog pipeline:

import logging
import structlog

# This was shown in the configuration section above.
# The key line is `foreign_pre_chain`, which processes
# log records from non-structlog loggers.

# After configuration, stdlib loggers are captured:
sqlalchemy_logger = logging.getLogger("sqlalchemy.engine")
sqlalchemy_logger.setLevel(logging.INFO)
# SQLAlchemy's logs now appear as structured JSON

This is critical because in production, you want ALL logs — from your code and from libraries — in the same structured format.

Sensitive Data Filtering

Logging sensitive data (PII, authentication tokens, credit card numbers) is a compliance and security risk. Build filtering into your processor pipeline so it is impossible to accidentally log secrets:

import re

SENSITIVE_PATTERNS = {
    "credit_card": re.compile(r"\b\d{4}[- ]?\d{4}[- ]?\d{4}[- ]?\d{4}\b"),
    "ssn": re.compile(r"\b\d{3}-\d{2}-\d{4}\b"),
    "bearer_token": re.compile(r"Bearer\s+[A-Za-z0-9\-._~+/]+=*"),
}

SENSITIVE_KEYS = {
    "password", "passwd", "secret", "token", "api_key",
    "apikey", "authorization", "credit_card", "ssn",
    "social_security", "private_key",
}

def filter_sensitive_data(logger, method_name, event_dict):
    """Recursively scrub sensitive data from log events."""
    def scrub(key, value):
        if isinstance(key, str) and key.lower() in SENSITIVE_KEYS:
            return "[REDACTED]"
        if isinstance(value, str):
            for pattern_name, pattern in SENSITIVE_PATTERNS.items():
                if pattern.search(value):
                    value = pattern.sub(f"[REDACTED_{pattern_name.upper()}]", value)
        if isinstance(value, dict):
            return {k: scrub(k, v) for k, v in value.items()}
        if isinstance(value, (list, tuple)):
            return [scrub(key, item) for item in value]
        return value

    return {k: scrub(k, v) for k, v in event_dict.items()}

Performance Considerations

Logging happens on every request, often multiple times. Poor logging performance can significantly impact your application.

1. Cache Loggers

Always set cache_logger_on_first_use=True in your structlog configuration. This avoids re-resolving the processor chain on every log call.

2. Avoid Expensive Computation in Log Calls

# Bad: str() is called even if log level is above DEBUG
log.debug("user_data", data=str(large_object))

# Good: use lazy evaluation or check the level
if log.isEnabledFor(logging.DEBUG):
    log.debug("user_data", data=str(large_object))

3. Use orjson for Faster Serialization

import orjson

def orjson_serializer(obj, **kwargs):
    return orjson.dumps(obj, option=orjson.OPT_NON_STR_KEYS).decode("utf-8")

structlog.configure(
    processors=[
        # ...
        structlog.processors.JSONRenderer(serializer=orjson_serializer),
    ],
    # ...
)

orjson is typically 3-10x faster than the standard library's json module.

4. Async Log Shipping

Do not send logs over the network synchronously in the request path. Write to stdout/stderr and let a sidecar (Filebeat, Fluentd) ship them, or use a background queue.

5. Log Sampling for High-Volume Events

import random

def sample_debug_logs(logger, method_name, event_dict):
    """Only emit 10% of debug logs to reduce volume."""
    if method_name == "debug" and random.random() > 0.1:
        raise structlog.DropEvent
    return event_dict

Exception Logging

structlog integrates cleanly with exception handling:

import structlog

log = structlog.get_logger()

try:
    result = process_payment(order_id=123, amount=49.99)
except PaymentGatewayError as exc:
    log.error(
        "payment_failed",
        order_id=123,
        error=str(exc),
        error_type=type(exc).__name__,
        exc_info=True,  # Includes full traceback in the log
    )
    raise

You can also add a processor that formats exceptions as structured data:

structlog.configure(
    processors=[
        # ... other processors
        structlog.processors.format_exc_info,  # Converts exc_info to a string field
        structlog.processors.JSONRenderer(),
    ],
)

Logging Best Practices for Production

Use Event Names, Not Sentences

# Bad
log.info("The user has successfully logged in")

# Good
log.info("user_login_success", user_id=user.id, login_method="oauth")

Event names should be snake_case identifiers that are easy to search and aggregate.

Log at the Right Level

  • DEBUG: Detailed diagnostic information useful during development.
  • INFO: Confirmation that things are working as expected (request started/completed, job processed).
  • WARNING: Something unexpected happened but the application can continue (deprecated API used, retry triggered).
  • ERROR: An operation failed and needs attention (payment failed, database connection lost).
  • CRITICAL: The application cannot continue (configuration missing, cannot bind to port).

Include Timing Information

import time

start = time.perf_counter()
result = await slow_database_query()
duration_ms = (time.perf_counter() - start) * 1000

log.info("db_query_completed", query="get_user_orders", duration_ms=round(duration_ms, 2))

Do Not Log and Raise (Usually)

# Bad: logs the error AND re-raises, causing duplicate log entries
# when a higher-level handler also logs it
try:
    do_something()
except ValueError as e:
    log.error("something_failed", error=str(e))
    raise  # The caller might log this again

# Better: either log OR raise, unless you are adding unique context
try:
    do_something()
except ValueError as e:
    # Add context and re-raise — let the boundary handler log it
    raise ValueError(f"Failed processing order {order_id}") from e

Structured Logging in Tests

Capture logs in tests to verify correct logging behavior:

import structlog
from structlog.testing import capture_logs

def test_order_processing_logs_completion():
    with capture_logs() as cap_logs:
        process_order(order_id=123, user_id=42)

    assert any(
        log["event"] == "order_processed" and log["order_id"] == 123
        for log in cap_logs
    )

Putting It All Together

Here is a complete, production-ready logging setup:

# app/logging_config.py
import os
import sys
import logging
import structlog

def setup_logging():
    is_production = os.environ.get("APP_ENV") == "production"
    log_level = os.environ.get("LOG_LEVEL", "INFO").upper()

    shared_processors = [
        structlog.contextvars.merge_contextvars,
        filter_sensitive_data,
        add_environment,
        structlog.stdlib.add_log_level,
        structlog.stdlib.add_logger_name,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
        structlog.processors.format_exc_info,
        structlog.processors.UnicodeDecoder(),
    ]

    if is_production:
        renderer = structlog.processors.JSONRenderer()
    else:
        renderer = structlog.dev.ConsoleRenderer(colors=True)

    structlog.configure(
        processors=[
            *shared_processors,
            structlog.stdlib.ProcessorFormatter.wrap_for_formatter,
        ],
        logger_factory=structlog.stdlib.LoggerFactory(),
        wrapper_class=structlog.stdlib.BoundLogger,
        cache_logger_on_first_use=True,
    )

    formatter = structlog.stdlib.ProcessorFormatter(
        processors=[
            structlog.stdlib.ProcessorFormatter.remove_processors_meta,
            renderer,
        ],
        foreign_pre_chain=shared_processors,
    )

    handler = logging.StreamHandler(sys.stdout)
    handler.setFormatter(formatter)

    root = logging.getLogger()
    root.handlers.clear()
    root.addHandler(handler)
    root.setLevel(getattr(logging, log_level))

    # Quiet down noisy libraries
    logging.getLogger("urllib3").setLevel(logging.WARNING)
    logging.getLogger("httpx").setLevel(logging.WARNING)

Call setup_logging() at application startup — before any other module imports create loggers — and every log statement in your entire application, including third-party libraries, flows through your structured pipeline.

Summary

Structured logging with structlog transforms logging from a debugging afterthought into a first-class observability tool. By emitting JSON-formatted, context-rich log events, you enable powerful querying, alerting, and correlation across distributed systems. The processor pipeline architecture makes it easy to add cross-cutting concerns like sensitive data filtering, environment tagging, and performance sampling without modifying individual log call sites. Combined with correlation IDs and proper integration with stdlib logging, structlog provides a production-grade logging foundation for Python backend services.

Common Problems & Solutions

1 Log context from one request leaks into another request's logs
Symptoms: Log entries for one user show another user's ID, or a request's logs contain a request_id from a completely different request.
Cause: Context variables are not cleared at the start of each request. When using structlog's contextvars, bound values persist for the lifetime of the context (which may be reused across requests in async frameworks or thread pools).
Solution: Always call `structlog.contextvars.clear_contextvars()` at the very start of each request handling cycle, before binding any new values. This ensures a clean slate. ```python async def dispatch(self, request, call_next): clear_contextvars() # Critical: reset context first bind_contextvars( request_id=str(uuid.uuid4()), user_id=get_user_id(request), ) response = await call_next(request) return response ```

Broken middleware - missing clear_contextvars

async def dispatch(self, request, call_next): bind_contextvars(request_id=str(uuid.uuid4())) # Previous request's user_id is still bound! response = await call_next(request) return response

2 Third-party library logs appear as unstructured text while application logs are structured JSON
Symptoms: Mixed log output: some lines are JSON objects and others are plain text with the default stdlib format. Log aggregation tools fail to parse certain lines.
Cause: structlog is configured but the stdlib logging root logger is not set up to use structlog's ProcessorFormatter. Libraries like SQLAlchemy, celery, and urllib3 use stdlib logging directly and bypass structlog entirely.
Solution: Configure a `structlog.stdlib.ProcessorFormatter` with `foreign_pre_chain` and attach it to the root stdlib logger's handler. The `foreign_pre_chain` processes log records from non-structlog loggers through your processor pipeline. ```python formatter = structlog.stdlib.ProcessorFormatter( processors=[ structlog.stdlib.ProcessorFormatter.remove_processors_meta, structlog.processors.JSONRenderer(), ], foreign_pre_chain=[ structlog.contextvars.merge_contextvars, structlog.stdlib.add_log_level, structlog.processors.TimeStamper(fmt='iso'), ], ) handler = logging.StreamHandler(sys.stdout) handler.setFormatter(formatter) root_logger = logging.getLogger() root_logger.handlers.clear() root_logger.addHandler(handler) ```
3 JSONRenderer raises TypeError for non-serializable objects
Symptoms: TypeError: Object of type datetime is not JSON serializable — the log call crashes instead of emitting a log entry, potentially causing request failures if the exception is not caught.
Cause: Log entries contain Python objects that the default json.dumps cannot serialize, such as datetime objects, Decimal values, UUID instances, SQLAlchemy models, or custom classes.
Solution: Provide a custom serializer to JSONRenderer that handles your application's types. For maximum performance, use `orjson` which handles many types natively. ```python import json import datetime import decimal import uuid def safe_serializer(obj, **kwargs): def default(o): if isinstance(o, (datetime.datetime, datetime.date)): return o.isoformat() if isinstance(o, decimal.Decimal): return float(o) if isinstance(o, uuid.UUID): return str(o) if hasattr(o, '__dict__'): return str(o) # Fallback for unknown objects raise TypeError(f'Not serializable: {type(o)}') return json.dumps(obj, default=default, **kwargs) structlog.configure( processors=[ # ... structlog.processors.JSONRenderer(serializer=safe_serializer), ], ) ```
