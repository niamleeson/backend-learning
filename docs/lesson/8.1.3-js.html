Horizontal vs Vertical Scaling

Intuition

Vertical scaling is like upgrading to a bigger moving truck when you have too much stuff. Horizontal scaling is like hiring more trucks. One mega-truck has a size limit and if it breaks down you are stuck, but a fleet of trucks can grow without limit and survive individual breakdowns.

Horizontal vs Vertical Scaling

Introduction

When your application cannot handle the current load, you have two fundamental options: make your existing server bigger (vertical scaling) or add more servers (horizontal scaling). This decision shapes your entire system architecture.

Vertical Scaling (Scaling Up)

Vertical scaling means adding more resources to a single machine: more CPU cores, more RAM, faster disks.

Before:                    After:
┌──────────┐              ┌──────────────────┐
│ 2 CPU    │              │ 16 CPU           │
│ 4GB RAM  │     ──>      │ 64GB RAM         │
│ 100GB SSD│              │ 1TB NVMe SSD     │
└──────────┘              └──────────────────┘
  $40/month                 $640/month

Advantages: - No code changes required - No distributed system complexity - Simpler deployment and operations - Data consistency is trivial (single machine)

Disadvantages: - Hardware limits (you cannot buy a 1000-core machine) - Cost grows non-linearly (2x resources often costs 4x) - Single point of failure - Downtime during upgrades

For Node.js, vertical scaling is limited because a single Node.js process only uses one CPU core. You need the cluster module or PM2 to use multiple cores:

import cluster from 'node:cluster';
import os from 'node:os';

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length;
  console.log(`Primary ${process.pid}: spawning ${numCPUs} workers`);

  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} died. Restarting...`);
    cluster.fork();
  });
} else {
  // Each worker runs the Express app
  const app = express();
  app.get('/', (req, res) => {
    res.json({ pid: process.pid, uptime: process.uptime() });
  });
  app.listen(3000);
}

Horizontal Scaling (Scaling Out)

Horizontal scaling means adding more machines (instances) to distribute the load.

Before:                    After:
┌──────────┐              ┌──────────┐  ┌──────────┐  ┌──────────┐
│ Server 1 │              │ Server 1 │  │ Server 2 │  │ Server 3 │
│ 4 CPU    │     ──>      │ 4 CPU    │  │ 4 CPU    │  │ 4 CPU    │
│ 8GB RAM  │              │ 8GB RAM  │  │ 8GB RAM  │  │ 8GB RAM  │
└──────────┘              └──────────┘  └──────────┘  └──────────┘
  $80/month                 $240/month (linear cost scaling)

Advantages: - Virtually unlimited scaling - Linear cost growth - No single point of failure - Can scale down during low traffic (cost savings)

Disadvantages: - Application must be stateless (or use shared state) - Distributed system complexity - Need load balancer, service discovery - Data consistency challenges

Making Node.js Apps Horizontally Scalable

The key requirement for horizontal scaling is statelessness. Your server must not store any request-dependent data in memory.

// BAD: In-memory state prevents horizontal scaling
const sessions = new Map();
const rateLimit = new Map();
const cache = new Map();

app.post('/login', (req, res) => {
  sessions.set(req.body.userId, { loggedIn: true });
  // This session only exists on THIS server instance
});

// GOOD: External state stores
import Redis from 'ioredis';
const redis = new Redis(process.env.REDIS_URL);

app.post('/login', async (req, res) => {
  const sessionId = generateSessionId();
  await redis.setex(`session:${sessionId}`, 86400, JSON.stringify({
    userId: req.body.userId,
    loggedIn: true,
  }));
  res.cookie('sessionId', sessionId, { httpOnly: true });
});

Stateless Checklist

┌────────────────────────────────────────────────────────┐
│ State Type          │ Move From Memory To              │
├─────────────────────┼──────────────────────────────────┤
│ Sessions            │ Redis / JWT tokens               │
│ Rate limiting       │ Redis (shared counters)          │
│ Cache               │ Redis / Memcached                │
│ File uploads        │ S3 / Cloud Storage               │
│ WebSocket state     │ Redis Pub/Sub for cross-instance │
│ Scheduled jobs      │ BullMQ (Redis-backed)            │
└────────────────────────────────────────────────────────┘

Scaling the Database

Application servers are easy to scale horizontally. Databases are harder:

           Application Layer (easy to scale)
         ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
         │App 1 │  │App 2 │  │App 3 │  │App 4 │
         └──┬───┘  └──┬───┘  └──┬───┘  └──┬───┘
            │         │         │         │
            ▼         ▼         ▼         ▼
         ┌────────────────────────────────────┐
         │        Database (hard to scale)     │
         │  ┌────────┐    ┌──────────────┐    │
         │  │ Primary │───>│ Read Replica │    │
         │  │ (write) │   │  (read only) │    │
         │  └────────┘    └──────────────┘    │
         └────────────────────────────────────┘

Database scaling strategies include: - Read replicas: Route reads to replicas, writes to primary - Sharding: Split data across multiple databases - Caching layer: Redis reduces database load - Connection pooling: PgBouncer manages connections efficiently

// Read replica routing with Prisma
import { PrismaClient } from '@prisma/client';

const writePrisma = new PrismaClient({
  datasources: { db: { url: process.env.DATABASE_PRIMARY_URL } },
});

const readPrisma = new PrismaClient({
  datasources: { db: { url: process.env.DATABASE_REPLICA_URL } },
});

// Route reads to replica, writes to primary
async function getUsers(page, limit) {
  return readPrisma.user.findMany({ skip: (page - 1) * limit, take: limit });
}

async function createUser(data) {
  return writePrisma.user.create({ data });
}

Auto-Scaling

Cloud platforms provide auto-scaling based on metrics:

                    Auto-Scaling Group
     ┌──────────────────────────────────────────┐
     │                                          │
     │  ┌──────┐  ┌──────┐  ┌──────┐           │
     │  │App 1 │  │App 2 │  │App 3 │  + more   │
     │  └──────┘  └──────┘  └──────┘  as needed │
     │                                          │
     └──────────────────────────────────────────┘
                      │
     Scaling Rules:   │
     CPU > 70% ──────> Add instance
     CPU < 30% ──────> Remove instance
     Request count > 1000/s ──> Add instance

For containerized Node.js apps with Kubernetes:

# kubernetes HPA (Horizontal Pod Autoscaler)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: node-api
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: node-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70

When to Use Which

┌─────────────────────────────────────────────────────────────┐
│ Scenario                        │ Strategy                  │
├─────────────────────────────────┼───────────────────────────┤
│ Early stage startup             │ Vertical first            │
│ Predictable, moderate growth    │ Vertical + read replicas  │
│ Unpredictable spikes            │ Horizontal + auto-scaling │
│ Global user base                │ Horizontal + multi-region │
│ CPU-bound workload              │ Horizontal (more cores)   │
│ Memory-bound workload           │ Vertical (more RAM)       │
│ Database bottleneck             │ Caching + read replicas   │
└─────────────────────────────────────────────────────────────┘

Real-World Scaling Path

Most successful applications follow this path:

Stage 1: Single Server
  App + DB on one machine
  ↓
Stage 2: Separate DB
  App server + Managed DB (RDS)
  ↓
Stage 3: Add Caching
  App + Redis + DB
  ↓
Stage 4: Horizontal App Scaling
  Load Balancer + 2-3 App servers + Redis + DB
  ↓
Stage 5: Database Scaling
  App cluster + Redis Cluster + DB Primary + Read Replicas
  ↓
Stage 6: Microservices + Sharding
  Service mesh + Message queues + Sharded DB

Common Pitfalls

  1. Premature horizontal scaling: Adding complexity before you need it. Start vertical, move horizontal when you hit limits.
  2. Forgetting shared state: In-memory caches, sessions, or rate limiters break when you add a second server.
  3. Not testing with multiple instances: Always test locally with 2+ instances behind a load balancer before deploying.
  4. Ignoring database as bottleneck: Scaling app servers while the database is the real bottleneck wastes money.

Summary

Vertical scaling is simpler but has hard limits and creates single points of failure. Horizontal scaling is more complex but offers virtually unlimited capacity and better reliability. Most production systems use both: vertical scaling for databases and horizontal scaling for stateless application servers. The key to horizontal scaling in Node.js is making your application stateless by moving all shared state to external stores like Redis.

Exercises

  1. Statelessness Audit

    Take an existing Express application and identify all in-memory state (sessions, caches, counters). Refactor each to use Redis via ioredis, making the app horizontally scalable.

    Hint

    Search for new Map(), new Set(), module-level variables, and global.*. Replace with Redis GET/SET operations.

  2. Multi-Instance Testing

    Run 3 instances of a Node.js app on different ports with PM2 or the cluster module. Put Nginx in front. Verify that sessions work correctly across instances using a shared Redis store.

    Hint

    Use pm2 start app.js -i 3 or configure Nginx upstream with 3 ports. Use connect-redis for session storage.

  3. Auto-Scaling Simulation

    Build a simple auto-scaler that monitors CPU usage of child processes and spawns/kills workers based on thresholds. Use the cluster module to manage workers.

    Hint

    Use os.loadavg() or process.cpuUsage() to monitor load. Fork new workers when load > 70%, kill extras when load < 30%.

Common Problems & Solutions

1 Application works with one instance but breaks with multiple
Cause: In-memory state (sessions, caches, rate limiters) is not shared between instances. Each instance has its own isolated memory.
Solution: Move all shared state to Redis. Use JWT for stateless authentication. Use BullMQ for distributed job processing.
2 Adding more app servers does not improve performance
Cause: The bottleneck is the database, not the application servers. All new servers are still waiting on the same overloaded database.
Solution: Add a caching layer (Redis) to reduce DB queries. Add read replicas for read-heavy workloads. Optimize slow queries with EXPLAIN ANALYZE.
3 WebSocket connections dropping when scaling horizontally
Cause: WebSocket connections are stateful and tied to a specific server. New connections may route to a different server than where the room/channel state exists.
Solution: Use Redis Pub/Sub as a message broker between instances. Libraries like `@socket.io/redis-adapter` handle this automatically.

import { createAdapter } from '@socket.io/redis-adapter';\nimport { createClient } from 'redis';\nconst pubClient = createClient({ url: REDIS_URL });\nconst subClient = pubClient.duplicate();\nio.adapter(createAdapter(pubClient, subClient));
