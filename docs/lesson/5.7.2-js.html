Dependency Inversion in Practice

Intuition

Dependency Inversion is like hiring through a job description instead of calling a specific person. Your company (high-level module) needs 'someone who can handle accounting' (abstraction/interface), not 'specifically Jane from ABC Corp' (concrete implementation). This way, you can hire anyone who meets the job requirements, and you can easily swap them out without restructuring the company.

Dependency Inversion in Practice

The Dependency Inversion Principle (DIP) states: high-level modules should not depend on low-level modules; both should depend on abstractions. This is the foundation of testable, flexible architecture.

The Problem Without DIP

// BAD: High-level depends on low-level
import { PrismaClient } from '@prisma/client';  // low-level detail
import nodemailer from 'nodemailer';            // low-level detail

class OrderService {
  private prisma = new PrismaClient();  // tightly coupled
  private mailer = nodemailer.createTransport(/* ... */); // tightly coupled

  async createOrder(data: any) {
    const order = await this.prisma.order.create({ data });  // can't test without DB
    await this.mailer.sendMail({ /* ... */ });  // sends real email in tests!
    return order;
  }
}

Problems: - Cannot test without a real database - Cannot test without sending real emails - Cannot switch from Prisma to Drizzle without rewriting - Cannot switch from nodemailer to SendGrid without rewriting

The Solution: Depend on Abstractions

Without DIP:                    With DIP:
OrderService                    OrderService
    |                               |
    +-> PrismaClient            IOrderRepository <---+
    +-> nodemailer              IEmailService <---+   |
                                                  |   |
                           SendGridAdapter -------+   |
                           PrismaOrderRepo -----------+

// GOOD: Both depend on abstractions

// 1. Define abstractions (interfaces)
interface IOrderRepository {
  create(data: CreateOrderData): Promise<Order>;
  findById(id: string): Promise<Order | null>;
  update(id: string, data: Partial<Order>): Promise<Order>;
}

interface IEmailService {
  send(to: string, subject: string, body: string): Promise<void>;
}

interface IPaymentGateway {
  charge(amount: number, paymentMethodId: string): Promise<PaymentResult>;
  refund(chargeId: string): Promise<RefundResult>;
}

// 2. High-level module depends on abstractions
class OrderService {
  constructor(
    private orderRepo: IOrderRepository,     // abstraction
    private emailService: IEmailService,     // abstraction
    private paymentGateway: IPaymentGateway, // abstraction
  ) {}

  async createOrder(userId: string, items: OrderItem[]): Promise<Order> {
    const order = await this.orderRepo.create({ userId, items, status: 'CREATED' });
    await this.emailService.send(userId, 'Order Created', `Order #${order.id}`);
    return order;
  }

  async payOrder(orderId: string, paymentMethodId: string): Promise<Order> {
    const order = await this.orderRepo.findById(orderId);
    if (!order) throw new NotFoundError('Order', orderId);

    const result = await this.paymentGateway.charge(order.total, paymentMethodId);
    if (!result.success) throw new PaymentError(result.error);

    return this.orderRepo.update(orderId, { status: 'PAID', chargeId: result.chargeId });
  }
}

Implementing Adapters

// Low-level: Prisma implementation
class PrismaOrderRepository implements IOrderRepository {
  constructor(private prisma: PrismaClient) {}

  async create(data: CreateOrderData): Promise<Order> {
    return this.prisma.order.create({ data });
  }
  async findById(id: string): Promise<Order | null> {
    return this.prisma.order.findUnique({ where: { id } });
  }
  async update(id: string, data: Partial<Order>): Promise<Order> {
    return this.prisma.order.update({ where: { id }, data });
  }
}

// Low-level: SendGrid implementation
class SendGridEmailService implements IEmailService {
  constructor(private apiKey: string) {}
  async send(to: string, subject: string, body: string): Promise<void> {
    // SendGrid API call
  }
}

// Low-level: Stripe implementation
class StripePaymentGateway implements IPaymentGateway {
  constructor(private stripe: Stripe) {}
  async charge(amount: number, paymentMethodId: string): Promise<PaymentResult> {
    const intent = await this.stripe.paymentIntents.create({
      amount: Math.round(amount * 100),
      currency: 'usd',
      payment_method: paymentMethodId,
      confirm: true,
    });
    return { success: true, chargeId: intent.id };
  }
  async refund(chargeId: string): Promise<RefundResult> {
    await this.stripe.refunds.create({ payment_intent: chargeId });
    return { success: true };
  }
}

Testing with Fakes

// Test doubles
class FakeOrderRepository implements IOrderRepository {
  private orders: Order[] = [];
  private counter = 1;

  async create(data: CreateOrderData): Promise<Order> {
    const order = { id: String(this.counter++), ...data, total: 0, createdAt: new Date() };
    this.orders.push(order);
    return order;
  }
  async findById(id: string): Promise<Order | null> {
    return this.orders.find(o => o.id === id) || null;
  }
  async update(id: string, data: Partial<Order>): Promise<Order> {
    const idx = this.orders.findIndex(o => o.id === id);
    this.orders[idx] = { ...this.orders[idx], ...data };
    return this.orders[idx];
  }
}

class FakeEmailService implements IEmailService {
  sentEmails: Array<{ to: string; subject: string; body: string }> = [];

  async send(to: string, subject: string, body: string): Promise<void> {
    this.sentEmails.push({ to, subject, body });
  }
}

class FakePaymentGateway implements IPaymentGateway {
  shouldFail = false;
  charges: Array<{ amount: number; id: string }> = [];

  async charge(amount: number, paymentMethodId: string): Promise<PaymentResult> {
    if (this.shouldFail) return { success: false, error: 'Card declined' };
    const id = `ch_${Date.now()}`;
    this.charges.push({ amount, id });
    return { success: true, chargeId: id };
  }
  async refund(chargeId: string): Promise<RefundResult> {
    return { success: true };
  }
}

// Tests are fast, isolated, and deterministic
describe('OrderService', () => {
  let service: OrderService;
  let orderRepo: FakeOrderRepository;
  let emailService: FakeEmailService;
  let paymentGateway: FakePaymentGateway;

  beforeEach(() => {
    orderRepo = new FakeOrderRepository();
    emailService = new FakeEmailService();
    paymentGateway = new FakePaymentGateway();
    service = new OrderService(orderRepo, emailService, paymentGateway);
  });

  it('sends confirmation email on order creation', async () => {
    await service.createOrder('user-1', [{ productId: 'p1', quantity: 2 }]);
    expect(emailService.sentEmails).toHaveLength(1);
    expect(emailService.sentEmails[0].to).toBe('user-1');
  });

  it('handles payment failure gracefully', async () => {
    const order = await service.createOrder('user-1', []);
    paymentGateway.shouldFail = true;
    await expect(service.payOrder(order.id, 'pm_123')).rejects.toThrow('Card declined');
  });
});

NestJS Dependency Injection

NestJS's DI system makes dependency inversion natural:

// Define injection tokens
export const ORDER_REPOSITORY = Symbol('ORDER_REPOSITORY');
export const EMAIL_SERVICE = Symbol('EMAIL_SERVICE');
export const PAYMENT_GATEWAY = Symbol('PAYMENT_GATEWAY');

// Module wiring
@Module({
  providers: [
    OrderService,
    {
      provide: ORDER_REPOSITORY,
      useClass: PrismaOrderRepository,
    },
    {
      provide: EMAIL_SERVICE,
      useFactory: (config: ConfigService) => {
        const provider = config.get('EMAIL_PROVIDER');
        if (provider === 'sendgrid') return new SendGridEmailService(config.get('SENDGRID_KEY'));
        if (provider === 'ses') return new SesEmailService(config.get('AWS_REGION'));
        return new ConsoleEmailService(); // dev fallback
      },
      inject: [ConfigService],
    },
    {
      provide: PAYMENT_GATEWAY,
      useClass: StripePaymentGateway,
    },
  ],
})
export class OrderModule {}

// Service uses injection tokens
@Injectable()
export class OrderService {
  constructor(
    @Inject(ORDER_REPOSITORY) private orderRepo: IOrderRepository,
    @Inject(EMAIL_SERVICE) private emailService: IEmailService,
    @Inject(PAYMENT_GATEWAY) private paymentGateway: IPaymentGateway,
  ) {}
}

Manual DI (Without Framework)

You do not need a DI framework. Constructor injection works fine:

// composition-root.ts
function createApp() {
  const prisma = new PrismaClient();
  const redis = new Redis();

  // Create driven adapters
  const orderRepo = new PrismaOrderRepository(prisma);
  const emailService = new SendGridEmailService(process.env.SENDGRID_KEY!);
  const paymentGateway = new StripePaymentGateway(stripe);
  const cacheService = new RedisCacheService(redis);

  // Create services (inject abstractions)
  const orderService = new OrderService(orderRepo, emailService, paymentGateway);
  const productService = new ProductService(productRepo, cacheService);

  // Create controllers (inject services)
  const orderController = new OrderController(orderService);
  const productController = new ProductController(productService);

  // Wire routes
  const app = express();
  app.post('/orders', (req, res) => orderController.create(req, res));
  app.get('/products', (req, res) => productController.list(req, res));

  return app;
}

Exercises

  1. Refactor Direct Dependencies to Interfaces

    Take a service that directly imports PrismaClient, nodemailer, and stripe. Refactor it to depend on interfaces instead. Create production implementations and test fakes for each.

    Hint

    Extract an interface for each dependency. Move the constructor to accept interfaces. Create concrete classes that implement each interface.

  2. Build a Pluggable Notification System

    Create an INotificationService interface with send(userId, message) method. Implement: EmailNotification, SMSNotification, PushNotification, and SlackNotification adapters. Write a service that uses the interface and tests that work with any adapter.

    Hint

    Each adapter implements the same interface. The service does not know which adapter it uses.

  3. NestJS Multi-Provider Module

    Create a NestJS module that configures different adapter implementations based on environment (development, staging, production). In dev, use in-memory repositories and console email. In production, use Prisma and SendGrid.

    Hint

    Use useFactory providers that read ConfigService to decide which implementation to instantiate.

Common Problems & Solutions

1 Too many interfaces for simple operations
Cause: Every class has an extracted interface, even when there is only one implementation and no testing benefit.
Solution: Extract interfaces only when you have multiple implementations or need to mock in tests. Not every class needs an interface.
2 Constructor parameter lists become very long
Cause: A service depends on many abstractions, leading to 6+ constructor parameters.
Solution: This often indicates the service has too many responsibilities. Split it into smaller, focused services. Alternatively, group related dependencies into a single context object.
3 Wiring dependencies manually is tedious
Cause: Without a DI framework, you manually construct and inject every dependency in the composition root.
Solution: Use a DI container (NestJS, tsyringe, or inversify) for automatic resolution. Or accept manual wiring as a trade-off for simplicity and transparency.
