Common Table Expressions (CTEs)

Intuition

CTEs are like naming the intermediate steps in a math problem. Instead of one giant equation, you write 'Let A = ..., Let B = ... using A, Final answer = ... using B'. Each step has a clear name and purpose, making the whole solution easier to understand and debug.

Common Table Expressions (CTEs)

Introduction

Common Table Expressions (CTEs) are named temporary result sets that exist only within the scope of a single query. Defined with the WITH keyword, they make complex queries readable, reusable, and maintainable. CTEs are often called "named subqueries" and they are the single best tool for taming complex SQL.

Basic Syntax

WITH cte_name AS (
  SELECT ...
)
SELECT * FROM cte_name;

Simple CTE Example

-- Without CTE (hard to read)
SELECT name, total_spent
FROM (
  SELECT user_id, SUM(total) AS total_spent
  FROM orders
  GROUP BY user_id
  HAVING SUM(total) > 1000
) high_spenders
JOIN users ON users.id = high_spenders.user_id;

-- With CTE (much clearer)
WITH high_spenders AS (
  SELECT user_id, SUM(total) AS total_spent
  FROM orders
  GROUP BY user_id
  HAVING SUM(total) > 1000
)
SELECT u.name, hs.total_spent
FROM users u
JOIN high_spenders hs ON u.id = hs.user_id;

Multiple CTEs

Chain multiple CTEs separated by commas:

async function getComprehensiveReport() {
  const result = await pool.query(`
    WITH monthly_revenue AS (
      SELECT
        DATE_TRUNC('month', created_at) AS month,
        SUM(total) AS revenue,
        COUNT(*) AS order_count
      FROM orders
      WHERE created_at >= NOW() - INTERVAL '12 months'
      GROUP BY DATE_TRUNC('month', created_at)
    ),
    monthly_users AS (
      SELECT
        DATE_TRUNC('month', created_at) AS month,
        COUNT(*) AS new_users
      FROM users
      WHERE created_at >= NOW() - INTERVAL '12 months'
      GROUP BY DATE_TRUNC('month', created_at)
    )
    SELECT
      TO_CHAR(mr.month, 'YYYY-MM') AS month,
      mr.revenue::numeric(12,2),
      mr.order_count::int,
      COALESCE(mu.new_users, 0)::int AS new_users,
      (mr.revenue / NULLIF(mr.order_count, 0))::numeric(10,2) AS avg_order_value
    FROM monthly_revenue mr
    LEFT JOIN monthly_users mu ON mr.month = mu.month
    ORDER BY mr.month DESC
  `);
  return result.rows;
}

CTEs Referencing Other CTEs

Each CTE can reference any CTE defined before it:

WITH
  -- Step 1: Get order totals per user
  user_totals AS (
    SELECT user_id, SUM(total) AS total_spent, COUNT(*) AS order_count
    FROM orders
    GROUP BY user_id
  ),
  -- Step 2: Categorize users (references user_totals)
  user_tiers AS (
    SELECT
      user_id,
      total_spent,
      order_count,
      CASE
        WHEN total_spent >= 10000 THEN 'platinum'
        WHEN total_spent >= 5000 THEN 'gold'
        WHEN total_spent >= 1000 THEN 'silver'
        ELSE 'bronze'
      END AS tier
    FROM user_totals
  ),
  -- Step 3: Summarize tiers (references user_tiers)
  tier_summary AS (
    SELECT
      tier,
      COUNT(*) AS user_count,
      AVG(total_spent) AS avg_spend
    FROM user_tiers
    GROUP BY tier
  )
SELECT * FROM tier_summary ORDER BY avg_spend DESC;

Recursive CTEs

Recursive CTEs can reference themselves, enabling traversal of tree/graph structures:

-- Organizational hierarchy
WITH RECURSIVE org_tree AS (
  -- Base case: top-level managers (no manager)
  SELECT id, name, manager_id, 1 AS depth, name::text AS path
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- Recursive case: employees with managers
  SELECT e.id, e.name, e.manager_id, t.depth + 1,
         t.path || ' > ' || e.name
  FROM employees e
  JOIN org_tree t ON e.manager_id = t.id
)
SELECT * FROM org_tree ORDER BY path;

async function getOrgChart(rootId = null) {
  const result = await pool.query(`
    WITH RECURSIVE org_tree AS (
      SELECT id, name, manager_id, 1 AS depth,
             ARRAY[id] AS path
      FROM employees
      WHERE ($1::int IS NULL AND manager_id IS NULL) OR id = $1

      UNION ALL

      SELECT e.id, e.name, e.manager_id, t.depth + 1,
             t.path || e.id
      FROM employees e
      JOIN org_tree t ON e.manager_id = t.id
      WHERE t.depth < 10  -- prevent infinite loops
    )
    SELECT id, name, manager_id, depth,
           REPEAT('  ', depth - 1) || name AS indented_name
    FROM org_tree
    ORDER BY path
  `, [rootId]);
  return result.rows;
}

Recursive CTE execution:

Iteration 1 (base):    CEO
                        |
Iteration 2:           VP-Eng        VP-Sales
                       |    |           |
Iteration 3:          Dev1  Dev2      Sales1

Each iteration adds one more level.
Stops when UNION ALL produces no new rows.

Recursive CTE for Graph Traversal

-- Find all ancestors of a category
WITH RECURSIVE ancestors AS (
  SELECT id, name, parent_id
  FROM categories
  WHERE id = 42  -- starting category

  UNION ALL

  SELECT c.id, c.name, c.parent_id
  FROM categories c
  JOIN ancestors a ON c.id = a.parent_id
)
SELECT * FROM ancestors;

CTE Performance Considerations

In PostgreSQL 12+, CTEs are inlined by default (the optimizer can merge them into the main query). If you need the CTE to act as an optimization barrier, use MATERIALIZED:

-- Inlined (default in PG 12+, optimizer can push filters down)
WITH user_stats AS (
  SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id
)
SELECT * FROM user_stats WHERE cnt > 10;

-- Materialized (computed once, stored in memory)
WITH user_stats AS MATERIALIZED (
  SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id
)
SELECT * FROM user_stats WHERE cnt > 10;

CTE with INSERT, UPDATE, DELETE (Writeable CTEs)

PostgreSQL allows DML in CTEs:

async function archiveOldOrders(daysOld) {
  const result = await pool.query(`
    WITH archived AS (
      DELETE FROM orders
      WHERE created_at < NOW() - INTERVAL '${daysOld} days'
        AND status = 'completed'
      RETURNING *
    )
    INSERT INTO orders_archive
    SELECT * FROM archived
    RETURNING id
  `);
  return result.rows.length;
}

This atomically moves rows from one table to another — the DELETE and INSERT happen in a single operation.

Real-World Example: Funnel Analysis

async function getFunnelAnalysis(startDate, endDate) {
  const result = await pool.query(`
    WITH
      visitors AS (
        SELECT COUNT(DISTINCT session_id) AS cnt
        FROM page_views
        WHERE created_at BETWEEN $1 AND $2
      ),
      signups AS (
        SELECT COUNT(*) AS cnt
        FROM users
        WHERE created_at BETWEEN $1 AND $2
      ),
      first_orders AS (
        SELECT COUNT(DISTINCT user_id) AS cnt
        FROM orders
        WHERE created_at BETWEEN $1 AND $2
        AND user_id IN (
          SELECT id FROM users WHERE created_at BETWEEN $1 AND $2
        )
      ),
      repeat_orders AS (
        SELECT COUNT(DISTINCT user_id) AS cnt
        FROM (
          SELECT user_id, COUNT(*) AS order_count
          FROM orders
          WHERE created_at BETWEEN $1 AND $2
          GROUP BY user_id
          HAVING COUNT(*) >= 2
        ) multi
      )
    SELECT
      v.cnt AS visitors,
      s.cnt AS signups,
      f.cnt AS first_purchase,
      r.cnt AS repeat_purchase,
      ROUND(s.cnt::numeric / NULLIF(v.cnt, 0) * 100, 2) AS signup_rate,
      ROUND(f.cnt::numeric / NULLIF(s.cnt, 0) * 100, 2) AS purchase_rate,
      ROUND(r.cnt::numeric / NULLIF(f.cnt, 0) * 100, 2) AS repeat_rate
    FROM visitors v, signups s, first_orders f, repeat_orders r
  `, [startDate, endDate]);
  return result.rows[0];
}

Exercises

  1. Build a Multi-Step Analytics Query

    Using multiple CTEs, build a query that: (1) calculates each user's total spending, (2) assigns them to a tier (platinum/gold/silver/bronze), (3) computes the average spending per tier, and (4) joins back to list users who are above their tier's average.

    Hint

    Create 3 CTEs: user_spending, user_tiers, tier_averages. Each references the previous one. Final SELECT joins user_tiers with tier_averages.

  2. Recursive Category Breadcrumbs

    Given a categories table with id, name, parent_id, write a recursive CTE that generates a breadcrumb path for each category (e.g., 'Electronics > Computers > Laptops'). Build a Node.js function around it.

    Hint

    Start with root categories (parent_id IS NULL), UNION ALL to join children. Concatenate names with ' > ' using string_agg or path concatenation.

  3. Writeable CTE: Soft Delete with Archival

    Write a single query using a writeable CTE that: (1) selects users who haven't logged in for 2 years, (2) inserts them into an archived_users table, (3) deletes them from users, and (4) returns the count of archived users.

    Hint

    Use WITH deleted AS (DELETE FROM users WHERE ... RETURNING *) INSERT INTO archived_users SELECT * FROM deleted.

Common Problems & Solutions

1 Recursive CTE runs indefinitely or returns too many rows
Cause: Circular references in the data (e.g., A -> B -> C -> A) or missing termination condition.
Solution: Add a depth counter and WHERE clause to limit recursion: `WHERE depth < 20`. Also check your data for circular references with a separate query.
2 CTE is unexpectedly slow compared to a subquery
Cause: In PostgreSQL < 12, CTEs are always materialized, preventing the optimizer from pushing predicates into the CTE.
Solution: Upgrade to PostgreSQL 12+ where CTEs are inlined by default. Or add the filter inside the CTE instead of outside it.
3 CTE column names conflict with main query columns
Cause: Using the same column names in the CTE and the tables in the main query without aliases.
Solution: Always use table aliases and explicitly name CTE columns: `WITH stats(user_id, total) AS (...)`. Reference as `stats.user_id`.
