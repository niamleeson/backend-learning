BullMQ: Jobs, Workers, Queues, and Scheduling

Intuition

BullMQ is like a restaurant's order ticket system. Waiters (API endpoints) write order tickets (jobs) and hang them on the rail (Redis queue). Cooks (workers) pull tickets and prepare dishes independently. If a dish fails, the ticket goes back on the rail for another attempt.

BullMQ: Jobs, Workers, Queues, and Scheduling

Introduction

BullMQ is the premier job queue library for Node.js, built on top of Redis. It enables you to offload time-consuming tasks — sending emails, processing images, generating reports — from your request handlers into background workers. This pattern is essential for building responsive, scalable backend systems.

Why Message Queues?

Without Queue (Synchronous):
User ──> API ──> Send Email (3s) ──> Generate PDF (5s) ──> Response (8s total)

With Queue (Asynchronous):
User ──> API ──> Enqueue Jobs ──> Response (50ms)
                      │
                      ▼
              Background Workers:
              Worker 1: Send Email (3s)
              Worker 2: Generate PDF (5s)

BullMQ Architecture

┌──────────┐     ┌───────────┐     ┌──────────────┐
│ Producer │────>│   Redis   │────>│   Worker(s)  │
│ (API)    │     │  (Queue)  │     │  (Consumers) │
└──────────┘     └───────────┘     └──────────────┘
     │                                    │
     │  add job to queue                  │  process jobs
     │                                    │
     ▼                                    ▼
  Job States: waiting ──> active ──> completed
                                 └──> failed ──> waiting (retry)

Setup

import { Queue, Worker, QueueScheduler, QueueEvents } from 'bullmq';
import Redis from 'ioredis';

// Shared Redis connection config
const connection = {
  host: process.env.REDIS_HOST || 'localhost',
  port: parseInt(process.env.REDIS_PORT) || 6379,
  maxRetriesPerRequest: null, // Required for BullMQ
};

// Create a queue
const emailQueue = new Queue('email', { connection });
const pdfQueue = new Queue('pdf-generation', { connection });

Adding Jobs (Producer)

// Simple job
await emailQueue.add('welcome-email', {
  to: 'user@example.com',
  subject: 'Welcome!',
  template: 'welcome',
  data: { name: 'Alice' },
});

// Job with options
await emailQueue.add('order-confirmation', {
  to: 'user@example.com',
  orderId: 'ORD-123',
}, {
  priority: 1,                    // Lower = higher priority
  delay: 5000,                    // Delay 5 seconds before processing
  attempts: 3,                    // Retry up to 3 times
  backoff: {
    type: 'exponential',          // exponential or fixed
    delay: 1000,                  // Initial delay: 1s, then 2s, 4s
  },
  removeOnComplete: {
    age: 3600,                    // Keep completed jobs for 1 hour
    count: 1000,                  // Keep last 1000 completed jobs
  },
  removeOnFail: {
    age: 86400,                   // Keep failed jobs for 24 hours
  },
});

// Bulk add
await emailQueue.addBulk([
  { name: 'digest', data: { userId: 1 }, opts: { priority: 5 } },
  { name: 'digest', data: { userId: 2 }, opts: { priority: 5 } },
  { name: 'digest', data: { userId: 3 }, opts: { priority: 5 } },
]);

From Express Route

import express from 'express';
const app = express();

app.post('/api/orders', authenticate, async (req, res) => {
  // Create order in DB (fast)
  const order = await prisma.order.create({
    data: { ...req.body, userId: req.user.id },
  });

  // Offload slow tasks to queues
  await Promise.all([
    emailQueue.add('order-confirmation', {
      orderId: order.id,
      userId: req.user.id,
    }),
    pdfQueue.add('invoice', {
      orderId: order.id,
    }),
    emailQueue.add('notify-warehouse', {
      orderId: order.id,
      items: order.items,
    }, { delay: 60000 }), // Wait 1 min (allow cancellation window)
  ]);

  // Respond immediately
  res.status(201).json({ order, message: 'Order placed successfully' });
});

Processing Jobs (Worker)

// worker.js - Run as a separate process
import { Worker } from 'bullmq';

const emailWorker = new Worker('email', async (job) => {
  console.log(`Processing job ${job.id}: ${job.name}`);

  switch (job.name) {
    case 'welcome-email':
      await sendWelcomeEmail(job.data);
      break;
    case 'order-confirmation':
      await sendOrderConfirmation(job.data);
      break;
    case 'notify-warehouse':
      await notifyWarehouse(job.data);
      break;
    default:
      throw new Error(`Unknown job name: ${job.name}`);
  }

  // Return value stored in job.returnvalue
  return { sent: true, timestamp: new Date().toISOString() };
}, {
  connection,
  concurrency: 5,           // Process 5 jobs simultaneously
  limiter: {
    max: 100,                // Max 100 jobs
    duration: 60000,         // Per 60 seconds (rate limiting)
  },
});

// Event handlers
emailWorker.on('completed', (job, result) => {
  console.log(`Job ${job.id} completed:`, result);
});

emailWorker.on('failed', (job, err) => {
  console.error(`Job ${job.id} failed:`, err.message);
  // Alert if final retry
  if (job.attemptsMade >= job.opts.attempts) {
    alertOps(`Email job permanently failed: ${job.id}`);
  }
});

emailWorker.on('error', (err) => {
  console.error('Worker error:', err);
});

Job Progress and Events

// Worker reporting progress
const pdfWorker = new Worker('pdf-generation', async (job) => {
  const { orderId } = job.data;

  await job.updateProgress(10);
  const orderData = await fetchOrderData(orderId);

  await job.updateProgress(30);
  const html = await renderTemplate(orderData);

  await job.updateProgress(60);
  const pdf = await generatePDF(html);

  await job.updateProgress(90);
  const url = await uploadToS3(pdf);

  await job.updateProgress(100);
  return { pdfUrl: url };
}, { connection });

// Monitoring with QueueEvents
const queueEvents = new QueueEvents('pdf-generation', { connection });

queueEvents.on('progress', ({ jobId, data }) => {
  console.log(`Job ${jobId} progress: ${data}%`);
});

queueEvents.on('completed', ({ jobId, returnvalue }) => {
  console.log(`Job ${jobId} completed:`, returnvalue);
});

queueEvents.on('failed', ({ jobId, failedReason }) => {
  console.error(`Job ${jobId} failed:`, failedReason);
});

// API endpoint to check job status
app.get('/api/jobs/:jobId', async (req, res) => {
  const job = await pdfQueue.getJob(req.params.jobId);
  if (!job) return res.status(404).json({ error: 'Job not found' });

  const state = await job.getState();
  res.json({
    id: job.id,
    name: job.name,
    state,
    progress: job.progress,
    result: job.returnvalue,
    failedReason: job.failedReason,
    attemptsMade: job.attemptsMade,
    timestamp: job.timestamp,
  });
});

Job Flows (Parent-Child)

import { FlowProducer } from 'bullmq';

const flowProducer = new FlowProducer({ connection });

// Create a job flow: parent waits for all children
const flow = await flowProducer.add({
  name: 'process-order',
  queueName: 'orders',
  data: { orderId: 'ORD-123' },
  children: [
    {
      name: 'validate-payment',
      queueName: 'payments',
      data: { orderId: 'ORD-123', amount: 99.99 },
    },
    {
      name: 'check-inventory',
      queueName: 'inventory',
      data: { orderId: 'ORD-123', items: ['PROD-1', 'PROD-2'] },
    },
    {
      name: 'generate-invoice',
      queueName: 'pdf-generation',
      data: { orderId: 'ORD-123' },
    },
  ],
});

// Parent job only runs after ALL children complete
const orderWorker = new Worker('orders', async (job) => {
  // Access children results
  const childrenValues = await job.getChildrenValues();
  console.log('All children completed:', childrenValues);

  // Finalize order processing
  await prisma.order.update({
    where: { id: job.data.orderId },
    data: { status: 'PROCESSED' },
  });
}, { connection });

NestJS Integration

// queue.module.ts
import { BullModule } from '@nestjs/bullmq';

@Module({
  imports: [
    BullModule.forRoot({
      connection: {
        host: 'localhost',
        port: 6379,
      },
    }),
    BullModule.registerQueue({ name: 'email' }),
  ],
  providers: [EmailProcessor],
  exports: [BullModule],
})
export class QueueModule {}

// email.processor.ts
import { Processor, WorkerHost } from '@nestjs/bullmq';
import { Job } from 'bullmq';

@Processor('email')
export class EmailProcessor extends WorkerHost {
  async process(job: Job) {
    switch (job.name) {
      case 'welcome':
        return this.sendWelcome(job.data);
      case 'reset-password':
        return this.sendPasswordReset(job.data);
    }
  }
}

// order.service.ts
import { InjectQueue } from '@nestjs/bullmq';
import { Queue } from 'bullmq';

@Injectable()
export class OrderService {
  constructor(@InjectQueue('email') private emailQueue: Queue) {}

  async createOrder(data) {
    const order = await this.prisma.order.create({ data });
    await this.emailQueue.add('order-confirmation', { orderId: order.id });
    return order;
  }
}

Common Pitfalls

  1. Not handling worker crashes: Use autorun: true (default) and handle the error event. BullMQ automatically re-adds active jobs that crash.
  2. Forgetting removeOnComplete: Without it, Redis memory grows unbounded with completed job data.
  3. Running workers in the API process: Workers should be separate processes so CPU-heavy jobs don't block request handling.
  4. Not setting maxRetriesPerRequest to null: BullMQ requires this ioredis option or connections fail.

Summary

BullMQ provides robust background job processing for Node.js via Redis. Producers add jobs to queues, workers process them asynchronously with retry logic, rate limiting, and priority ordering. Use job flows for complex multi-step workflows, QueueEvents for monitoring, and Bull Board for visual management. Always run workers as separate processes from your API server.

Exercises

  1. Email Queue System

    Build an Express API that accepts email requests and processes them through BullMQ. Include welcome emails, password resets, and order confirmations with different priorities and retry strategies.

    Hint

    Create separate job names for each email type. Use priority: 1 for password resets, priority: 5 for welcome emails. Set 3 retries with exponential backoff.

  2. Job Progress Tracking

    Create a PDF generation queue where jobs report progress. Build an API endpoint that returns job status and a simple polling client that shows progress updates.

    Hint

    Use job.updateProgress(percent) in the worker. Create GET /api/jobs/:id that returns state and progress. Client polls every 2 seconds.

  3. Flow Producer Pipeline

    Implement an order processing pipeline using FlowProducer with parent-child jobs: validate payment, check inventory, generate invoice. Parent job runs only after all children complete.

    Hint

    Use FlowProducer.add() with children array. In parent worker, call job.getChildrenValues() to get results.

  4. Rate-Limited API Calls

    Build a worker that calls an external API with rate limiting (max 10 requests per second). Use BullMQ's built-in limiter to respect the API rate limit.

    Hint

    Set worker limiter: { max: 10, duration: 1000 }. This ensures no more than 10 jobs process per second.

Common Problems & Solutions

1 Jobs stuck in 'active' state and never complete
Symptoms: Dashboard shows growing number of active jobs. No completed or failed events firing.
Cause: Worker process crashed while processing a job. The job remains in active state because no worker acknowledged completion or failure.
Solution: BullMQ automatically moves stalled jobs back to waiting. Ensure `stalledInterval` is set (default 30s). Check worker error handlers and ensure async functions don't silently swallow errors.
2 Redis memory growing unbounded
Cause: Completed and failed jobs are kept in Redis forever. With high job throughput, this consumes all available memory.
Solution: Set removeOnComplete and removeOnFail options when adding jobs. Use age-based AND count-based limits.

await queue.add('job', data, {\n removeOnComplete: { age: 3600, count: 1000 },\n removeOnFail: { age: 86400, count: 5000 },\n});

3 Worker not processing jobs despite jobs being in queue
Cause: ioredis connection configured with maxRetriesPerRequest (default 20), which BullMQ requires to be null.
Solution: Set maxRetriesPerRequest: null in the Redis connection options passed to BullMQ.

const connection = { host: 'localhost', port: 6379, maxRetriesPerRequest: null };
