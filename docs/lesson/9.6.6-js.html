Design a Multi-Tenant SaaS Platform

Intuition

Think of a multi-tenant SaaS platform like an apartment building. All tenants share the same building (infrastructure), elevators (API gateway), and utilities (shared services). Each apartment has its own lock (authentication), its own space (data isolation), and different floor plans based on what they pay (plan tiers). The building manager (admin) ensures that a loud party in one apartment (noisy neighbor) does not disturb everyone else through noise limits (rate limiting) and soundproofing (resource isolation).

Design a Multi-Tenant SaaS Platform

Introduction

Multi-tenancy is the foundational architecture for Software-as-a-Service (SaaS) products. A multi-tenant system serves many customers (tenants) from a shared infrastructure while maintaining strict data isolation, per-tenant customization, and fair resource allocation. Think of platforms like Slack, Salesforce, Shopify, or any B2B SaaS product. The design challenge is balancing cost efficiency (shared resources) with isolation (each tenant's data and performance must be independent).

Requirements

Functional Requirements

  • Support thousands of tenants (organizations) on shared infrastructure
  • Each tenant has its own users, data, configurations, and billing plan
  • Strict data isolation — Tenant A must never see Tenant B's data
  • Per-tenant customization — custom branding, feature flags, integrations
  • Role-based access control (RBAC) within each tenant
  • Support multiple billing tiers (Free, Pro, Enterprise) with different resource limits
  • Tenant onboarding — self-service signup, provisioning, and data import
  • Tenant offboarding — data export and complete deletion (GDPR compliance)
  • Admin dashboard for tenant management, usage metrics, and billing

Non-Functional Requirements

  • 50,000 tenants, ranging from 5 users to 100,000 users
  • Total users: 10 million
  • Noisy neighbor prevention — one tenant's heavy usage must not degrade others
  • Data residency — support storing data in specific regions (EU, US, APAC)
  • 99.99% availability
  • Tenant provisioning time: < 30 seconds
  • Zero data leakage between tenants (security critical)

API Design

Tenant Context Middleware

import express from 'express';
const app = express();

// Tenant identification middleware — runs on every request
function tenantContext(req, res, next) {
  // Strategy 1: Subdomain-based
  const subdomain = req.hostname.split('.')[0];
  // Strategy 2: Header-based
  const headerTenant = req.headers['x-tenant-id'];
  // Strategy 3: JWT claim
  const jwtTenant = req.user?.tenantId;

  const tenantSlug = subdomain || headerTenant || jwtTenant;
  if (!tenantSlug) {
    return res.status(400).json({ error: 'Tenant identification required' });
  }

  // Look up tenant configuration (cached in Redis)
  const tenant = await tenantCache.get(tenantSlug);
  if (!tenant) {
    return res.status(404).json({ error: 'Tenant not found' });
  }
  if (tenant.status !== 'active') {
    return res.status(403).json({ error: `Tenant is ${tenant.status}` });
  }

  // Attach tenant context to request
  req.tenant = {
    id: tenant.id,
    slug: tenant.slug,
    plan: tenant.plan,
    dbSchema: tenant.dbSchema,       // for schema-per-tenant
    dbConnectionString: tenant.dbUrl, // for db-per-tenant
    features: tenant.features,
    limits: tenant.limits,
    region: tenant.region,
  };

  next();
}

app.use(tenantContext);

Tenant-Scoped Data Access

import { PrismaClient } from '@prisma/client';

// Shared database approach with tenant_id column
// CRITICAL: Every query MUST include tenant_id filter
class TenantScopedRepository {
  constructor(prisma, tenantId) {
    this.prisma = prisma;
    this.tenantId = tenantId;
  }

  async findProjects(options = {}) {
    return this.prisma.project.findMany({
      where: {
        tenantId: this.tenantId, // ALWAYS scope by tenant
        ...options.where,
      },
      orderBy: options.orderBy || { createdAt: 'desc' },
      take: options.limit || 50,
      skip: options.offset || 0,
    });
  }

  async createProject(data) {
    return this.prisma.project.create({
      data: {
        ...data,
        tenantId: this.tenantId, // ALWAYS set tenant
      },
    });
  }

  async findProjectById(id) {
    const project = await this.prisma.project.findFirst({
      where: {
        id,
        tenantId: this.tenantId, // Prevent cross-tenant access
      },
    });
    return project;
  }
}

// Usage in route handler
app.get('/api/v1/projects', async (req, res) => {
  const repo = new TenantScopedRepository(prisma, req.tenant.id);
  const projects = await repo.findProjects({
    limit: parseInt(req.query.limit) || 50,
    offset: parseInt(req.query.offset) || 0,
  });
  res.json({ projects });
});

Row-Level Security with PostgreSQL

// Using PostgreSQL Row-Level Security (RLS) as a safety net
// Even if application code has a bug, the DB enforces isolation

const setupRLS = async (prisma) => {
  await prisma.$executeRaw`
    -- Enable RLS on the projects table
    ALTER TABLE projects ENABLE ROW LEVEL SECURITY;

    -- Policy: users can only see their tenant's projects
    CREATE POLICY tenant_isolation_policy ON projects
      USING (tenant_id = current_setting('app.current_tenant_id')::uuid);

    -- Policy: inserts must match current tenant
    CREATE POLICY tenant_insert_policy ON projects
      FOR INSERT
      WITH CHECK (tenant_id = current_setting('app.current_tenant_id')::uuid);
  `;
};

// Middleware to set tenant context at the database level
async function setDbTenantContext(prisma, tenantId) {
  await prisma.$executeRaw`
    SET LOCAL app.current_tenant_id = ${tenantId};
  `;
}

// Usage in transaction
app.get('/api/v1/projects', async (req, res) => {
  const projects = await prisma.$transaction(async (tx) => {
    await tx.$executeRaw`
      SET LOCAL app.current_tenant_id = ${req.tenant.id};
    `;
    // RLS automatically filters — no tenant_id in WHERE needed
    // (but you should still include it for defense in depth)
    return tx.project.findMany({
      where: { tenantId: req.tenant.id },
      take: 50,
    });
  });

  res.json({ projects });
});

app.post('/api/v1/admin/tenants', async (req, res) => {
  const { name, slug, plan, adminEmail, region } = req.body;

  // Validate slug uniqueness
  const existing = await tenantService.findBySlug(slug);
  if (existing) {
    return res.status(409).json({ error: 'Slug already taken' });
  }

  // Create tenant with all associated resources
  const tenant = await prisma.$transaction(async (tx) => {
    // 1. Create tenant record
    const t = await tx.tenant.create({
      data: {
        name,
        slug,
        plan: plan || 'free',
        region: region || 'us-east-1',
        status: 'provisioning',
        limits: getPlanLimits(plan || 'free'),
        features: getPlanFeatures(plan || 'free'),
      },
    });

    // 2. Create admin user
    await tx.user.create({
      data: {
        tenantId: t.id,
        email: adminEmail,
        role: 'admin',
      },
    });

    // 3. Create default resources (workspace, project, etc.)
    await tx.workspace.create({
      data: {
        tenantId: t.id,
        name: 'Default Workspace',
        isDefault: true,
      },
    });

    return t;
  });

  // Async provisioning (DNS, SSL cert, etc.)
  await provisioningQueue.add('provision-tenant', {
    tenantId: tenant.id,
    slug: tenant.slug,
    region: tenant.region,
  });

  res.status(201).json({
    id: tenant.id,
    slug: tenant.slug,
    status: 'provisioning',
    adminEmail,
    dashboardUrl: `https://${slug}.app.example.com`,
  });
});

// PUT /api/v1/admin/tenants/:tenantId/plan
// Upgrade/downgrade tenant plan
app.put('/api/v1/admin/tenants/:tenantId/plan', async (req, res) => {
  const { tenantId } = req.params;
  const { plan } = req.body;

  const tenant = await tenantService.findById(tenantId);
  if (!tenant) return res.status(404).json({ error: 'Tenant not found' });

  // Check if downgrade is possible (within new limits)
  if (isPlanDowngrade(tenant.plan, plan)) {
    const usage = await usageMeter.getTenantUsage(tenantId);
    const newLimits = getPlanLimits(plan);
    const violations = checkLimitViolations(usage, newLimits);
    if (violations.length > 0) {
      return res.status(422).json({
        error: 'Cannot downgrade — current usage exceeds new plan limits',
        violations,
      });
    }
  }

  const updated = await tenantService.updatePlan(tenantId, {
    plan,
    limits: getPlanLimits(plan),
    features: getPlanFeatures(plan),
  });

  // Update billing subscription
  await billingService.updateSubscription(tenant.stripeCustomerId, plan);

  // Invalidate tenant cache
  await tenantCache.invalidate(tenant.slug);

  res.json(updated);
});

Rate Limiting per Tenant

import Redis from 'ioredis';
const redis = new Redis();

const PLAN_RATE_LIMITS = {
  free:       { requests: 100,   window: 60 },   // 100 req/min
  pro:        { requests: 1000,  window: 60 },   // 1000 req/min
  enterprise: { requests: 10000, window: 60 },   // 10000 req/min
};

async function tenantRateLimiter(req, res, next) {
  const { id: tenantId, plan } = req.tenant;
  const limits = PLAN_RATE_LIMITS[plan];
  const key = `ratelimit:${tenantId}:${Math.floor(Date.now() / 1000 / limits.window)}`;

  const current = await redis.incr(key);
  if (current === 1) {
    await redis.expire(key, limits.window);
  }

  res.setHeader('X-RateLimit-Limit', limits.requests);
  res.setHeader('X-RateLimit-Remaining', Math.max(0, limits.requests - current));
  res.setHeader('X-RateLimit-Reset', Math.ceil(Date.now() / 1000 / limits.window) * limits.window);

  if (current > limits.requests) {
    return res.status(429).json({
      error: 'Rate limit exceeded',
      retryAfter: limits.window,
    });
  }

  next();
}

app.use(tenantRateLimiter);

Scaling Strategies

  • Tenant-aware routing: Route large tenants to dedicated application server pools to prevent noisy-neighbor issues.
  • Database sharding by tenant: Group tenants onto database shards. Move high-traffic tenants to dedicated shards.
  • Connection pooling: Use PgBouncer or similar to pool connections across tenants. Avoid per-tenant connection pools for small tenants.
  • Caching with tenant namespacing: All Redis keys prefixed with tenant:{id}: to prevent key collisions.
  • CDN with tenant-specific cache keys: Cache static assets per tenant domain.
  • Regional deployment: Deploy full stack copies in EU, US, APAC. Route tenants to their designated region.

Trade-offs

Decision Pros Cons
Shared schema (tenant_id) Simple, cheap, easy migrations Must never forget WHERE tenant_id =, noisy neighbor risk
Schema-per-tenant Good isolation, easy per-tenant backup Schema migrations across thousands of schemas
DB-per-tenant Best isolation, independent scaling Connection overhead, migration coordination
PostgreSQL RLS Database-enforced isolation safety net Slight performance overhead, complexity
Subdomain routing Clean URLs, easy tenant identification SSL wildcard cert needed, DNS management
Tiered isolation (hybrid) Right-sized for each tenant's needs Complex codebase, multiple code paths

Noisy Neighbor Prevention

Strategies:

1. Per-tenant rate limiting (API layer)
   - Free: 100 req/min
   - Pro: 1,000 req/min
   - Enterprise: 10,000 req/min

2. Per-tenant connection limits (DB layer)
   - Free: 5 connections
   - Pro: 20 connections
   - Enterprise: 100 connections

3. Per-tenant query timeout
   - Free: 5 second max query time
   - Pro: 30 second max
   - Enterprise: 120 second max

4. Background job queues per priority
   - Free: low-priority shared queue
   - Pro: medium-priority queue
   - Enterprise: dedicated queue with reserved workers

5. Storage quotas
   - Free: 1 GB
   - Pro: 100 GB
   - Enterprise: Unlimited (metered)

Summary

Designing a multi-tenant SaaS platform requires careful decisions across every layer: tenant identification (subdomain, header, JWT), data isolation (shared schema with tenant_id, separate schemas, separate databases), resource management (rate limiting, connection pooling, quotas), and customization (feature flags, branding, plan-based limits). The key insight is that one size does not fit all — use a tiered isolation model where small tenants share everything (cheap) and enterprise tenants get dedicated resources (isolated). Row-Level Security in PostgreSQL provides a critical safety net against data leakage bugs. Every query, every cache key, and every background job must be tenant-scoped.

Common Problems & Solutions

1 A developer forgot to include tenant_id in a database query, exposing Tenant B's data to Tenant A
Symptoms: Users report seeing data that does not belong to their organization. Security audit reveals queries without tenant_id filter.
Cause: In a shared-schema model, every query must include a tenant_id filter. Without it, the query returns data across all tenants. This is the most critical bug in multi-tenant systems.
Solution: Implement defense in depth: (1) Use PostgreSQL Row-Level Security as a database-level safety net, (2) Create a tenant-scoped repository layer that automatically injects tenant_id into every query, (3) Add integration tests that verify cross-tenant isolation for every endpoint, (4) Use Prisma middleware to log warnings when queries lack tenant_id.
2 One enterprise tenant's heavy batch import is degrading performance for all other tenants
Symptoms: All tenants experience elevated API response times. Database connection pool at 100% utilization. Monitoring shows one tenant generating 80% of all queries.
Cause: The large tenant's import generates thousands of concurrent database queries that consume all available connections and CPU, causing connection pool exhaustion and slow queries for everyone else.
Solution: Implement per-tenant connection limits at the PgBouncer level. Route the enterprise tenant's batch operations through a dedicated database replica or shard. Use separate job queues for batch operations with concurrency limits per tenant. Implement query timeouts that vary by plan tier.
3 Schema migrations take hours because they must be applied across 50,000 tenant schemas
Cause: In a schema-per-tenant architecture, each ALTER TABLE must be executed against every tenant schema sequentially. A single migration multiplied by 50,000 schemas takes unreasonably long.
Solution: Migrate to a shared-schema model for small tenants (tenant_id column with RLS). For schema-per-tenant, use parallel migration execution with configurable concurrency. Implement online schema migrations (pt-online-schema-change or pg_repack) that do not lock tables. Batch migrations during low-traffic windows. Consider using zero-downtime migration patterns (expand-contract).
