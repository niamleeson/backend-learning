Node.js Cluster Module: Multi-Core Utilization

Intuition

The cluster module is like a restaurant manager (primary process) who hires multiple chefs (workers). Each chef has their own workstation but they all serve the same dining room (port). If a chef calls in sick, the manager hires a replacement. During a menu change, the manager trains new chefs one at a time while the old ones finish their current orders (zero-downtime restart).

Node.js Cluster Module: Multi-Core Utilization

Introduction

Node.js runs JavaScript on a single thread. On a modern 8-core server, a single Node.js process uses only 12.5% of available CPU. The cluster module lets you fork multiple worker processes that share the same server port, utilizing all available CPU cores.

The Problem

Single Process:
┌────────────────────────────────────────┐
│  8-Core Server                         │
│  ┌──────┐                              │
│  │Node.js│  Core 1: 100% busy          │
│  │Process│  Core 2-8: IDLE             │
│  └──────┘                              │
│  Only 1/8 of CPU utilized             │
└────────────────────────────────────────┘

With Cluster Module:
┌────────────────────────────────────────┐
│  8-Core Server                         │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐  │
│  │Worker│ │Worker│ │Worker│ │Worker│  │
│  │  1   │ │  2   │ │  3   │ │  4   │  │
│  └──────┘ └──────┘ └──────┘ └──────┘  │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐  │

Basic Cluster Setup

import cluster from 'node:cluster';
import os from 'node:os';
import express from 'express';

const numCPUs = os.cpus().length;

if (cluster.isPrimary) {
  console.log(`Primary process ${process.pid} starting ${numCPUs} workers`);

  // Fork workers
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  // Handle worker exit
  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} died (code: ${code}, signal: ${signal})`);
    // Replace dead worker
    console.log('Starting a new worker...');
    cluster.fork();
  });

  // Monitor workers
  cluster.on('online', (worker) => {
    console.log(`Worker ${worker.process.pid} is online`);
  });

} else {
  // Workers share the same port
  const app = express();

  app.get('/', (req, res) => {
    res.json({
      pid: process.pid,
      uptime: process.uptime(),
    });
  });

  app.get('/heavy', (req, res) => {
    // CPU-intensive work — only blocks THIS worker
    let sum = 0;
    for (let i = 0; i < 1e8; i++) sum += i;
    res.json({ result: sum, pid: process.pid });
  });

  app.listen(3000, () => {
    console.log(`Worker ${process.pid} listening on port 3000`);
  });
}

Graceful Shutdown and Zero-Downtime Restart

if (cluster.isPrimary) {
  const workers = [];

  function forkWorker() {
    const worker = cluster.fork();
    workers.push(worker);
    return worker;
  }

  // Initial fork
  for (let i = 0; i < numCPUs; i++) {
    forkWorker();
  }

  // Zero-downtime restart
  process.on('SIGUSR2', async () => {
    console.log('Received SIGUSR2 - starting zero-downtime restart');

    const currentWorkers = [...Object.values(cluster.workers)];

    for (const worker of currentWorkers) {
      // Start new worker
      const newWorker = forkWorker();

      // Wait for new worker to be ready
      await new Promise((resolve) => {
        newWorker.on('listening', resolve);
      });

      // Gracefully shut down old worker
      worker.send('shutdown');
      worker.disconnect();

      await new Promise((resolve) => {
        const timeout = setTimeout(() => {
          worker.kill('SIGKILL');
          resolve();
        }, 10000); // Force kill after 10s

        worker.on('exit', () => {
          clearTimeout(timeout);
          resolve();
        });
      });
    }

    console.log('Zero-downtime restart complete');
  });

  // Replace crashed workers
  cluster.on('exit', (worker, code, signal) => {
    if (code !== 0 && !worker.exitedAfterDisconnect) {
      console.log(`Worker ${worker.process.pid} crashed. Restarting...`);
      forkWorker();
    }
  });

} else {
  const app = express();
  const server = app.listen(3000);

  // Handle graceful shutdown
  process.on('message', (msg) => {
    if (msg === 'shutdown') {
      console.log(`Worker ${process.pid}: shutting down gracefully`);

      // Stop accepting new connections
      server.close(() => {
        console.log(`Worker ${process.pid}: all connections closed`);
        process.exit(0);
      });

      // Force close after timeout
      setTimeout(() => {
        console.log(`Worker ${process.pid}: forcing shutdown`);
        process.exit(1);
      }, 5000);
    }
  });
}

Inter-Process Communication (IPC)

Workers can communicate with the primary process via IPC:

if (cluster.isPrimary) {
  // Aggregate metrics from all workers
  const metrics = { totalRequests: 0, errors: 0 };

  cluster.on('message', (worker, message) => {
    if (message.type === 'metric') {
      metrics.totalRequests += message.requests || 0;
      metrics.errors += message.errors || 0;
    }
  });

  // Broadcast to all workers
  function broadcast(message) {
    for (const worker of Object.values(cluster.workers)) {
      worker.send(message);
    }
  }

  // Periodically request metrics
  setInterval(() => {
    broadcast({ type: 'report-metrics' });
  }, 10000);

  // Expose metrics endpoint on a different port
  const adminApp = express();
  adminApp.get('/metrics', (req, res) => {
    res.json({
      ...metrics,
      workers: Object.values(cluster.workers).map(w => ({
        pid: w.process.pid,
        state: w.state,
      })),
    });
  });
  adminApp.listen(3001);

} else {
  let requestCount = 0;
  let errorCount = 0;

  const app = express();

  app.use((req, res, next) => {
    requestCount++;
    res.on('finish', () => {
      if (res.statusCode >= 500) errorCount++;
    });
    next();
  });

  // Report metrics when asked
  process.on('message', (msg) => {
    if (msg.type === 'report-metrics') {
      process.send({
        type: 'metric',
        requests: requestCount,
        errors: errorCount,
      });
      requestCount = 0;
      errorCount = 0;
    }
  });
}

PM2: Cluster Management Made Easy

PM2 provides cluster mode without writing cluster code:

// ecosystem.config.js
module.exports = {
  apps: [{
    name: 'api',
    script: './dist/server.js',
    instances: 'max',        // Use all CPU cores
    exec_mode: 'cluster',    // Enable cluster mode
    max_memory_restart: '500M',
    env: {
      NODE_ENV: 'production',
      PORT: 3000,
    },
    // Graceful shutdown
    kill_timeout: 5000,
    listen_timeout: 10000,
    // Zero-downtime reload
    wait_ready: true,
    // Logging
    error_file: './logs/error.log',
    out_file: './logs/out.log',
    merge_logs: true,
  }],
};

# Start in cluster mode
pm2 start ecosystem.config.js

# Zero-downtime reload
pm2 reload api

# Scale up/down
pm2 scale api 4   # Set to 4 instances
pm2 scale api +2  # Add 2 more instances

For PM2 graceful shutdown:

// In your app
const server = app.listen(3000, () => {
  // Tell PM2 we're ready
  process.send?.('ready');
});

process.on('SIGINT', () => {
  server.close(() => {
    process.exit(0);
  });
});

Shared State Caveat

Each worker has its own memory space. In-memory state is NOT shared:

// BAD: This counter is per-worker, not global
let requestCount = 0;
app.use((req, res, next) => {
  requestCount++; // Only counts requests for THIS worker
  next();
});

// GOOD: Use Redis for shared state
app.use(async (req, res, next) => {
  await redis.incr('global:request-count');
  next();
});

Common Pitfalls

  1. Assuming shared memory: Each worker is a separate process. Use Redis or IPC for shared state.
  2. Not handling worker crashes: Without restart logic, workers silently die and reduce capacity.
  3. Running more workers than CPU cores: Extra workers cause context switching overhead. Match worker count to core count.
  4. Not implementing graceful shutdown: Killing workers immediately drops in-flight requests. Always drain connections first.

Summary

The cluster module enables Node.js to utilize all CPU cores by forking worker processes that share a server port. The primary process manages workers, handles restarts, and can coordinate via IPC. PM2 provides production-ready cluster management with zero-downtime reloads. Remember that each worker has isolated memory — use Redis for shared state.

Exercises

  1. Clustered Express Server

    Build an Express app using the cluster module that spawns workers equal to CPU count. Add automatic worker replacement on crash and a metrics endpoint on the primary process.

    Hint

    Use cluster.on('exit') to detect crashes and cluster.fork() to replace. Track worker PIDs and states.

  2. Zero-Downtime Restart

    Implement zero-downtime restart triggered by SIGUSR2 signal. New workers start before old ones shut down, ensuring no dropped requests.

    Hint

    Start new worker, wait for 'listening' event, then disconnect old worker. Use server.close() for graceful shutdown.

  3. PM2 Production Deployment

    Configure an ecosystem.config.js for a NestJS application with cluster mode, memory limits, graceful shutdown, and log management. Test zero-downtime reload.

    Hint

    Set exec_mode: 'cluster', instances: 'max', wait_ready: true. Use process.send('ready') in your app.

Common Problems & Solutions

1 Only one worker receiving all requests
Cause: On some platforms (notably Windows), the OS scheduling may favor one worker. Linux distributes connections more evenly.
Solution: On Linux, the default round-robin scheduling works well. On other platforms, consider using a reverse proxy (Nginx) for more even distribution.
2 Session data lost between requests
Cause: Requests are routed to different workers, and sessions are stored in worker memory.
Solution: Use Redis-backed sessions (connect-redis) or stateless JWT authentication. Never store sessions in worker memory.
3 Memory usage multiplied by worker count
Cause: Each worker is a full Node.js process with its own V8 heap. 8 workers each using 200MB = 1.6GB total.
Solution: Monitor per-worker memory usage. Set max_memory_restart in PM2. Optimize application memory usage before adding workers.
