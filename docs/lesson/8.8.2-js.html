Distributed Consensus: Raft and Paxos

Intuition

Distributed consensus is like a group of friends trying to decide on a restaurant over a flaky group chat. Raft's approach: one person becomes the 'organizer' (leader) by getting a majority vote. The organizer proposes a restaurant, and everyone agrees if the organizer is still in charge. If the organizer goes offline, the group votes for a new organizer. The key rule: decisions only count if a majority participates.

Distributed Consensus: Raft and Paxos

Introduction

Distributed consensus is the problem of getting multiple nodes in a distributed system to agree on a value. This is fundamental to leader election, configuration management, and replicated state machines. Raft and Paxos are the two most important consensus algorithms.

Why Consensus?

Problem: Who is the leader?

Node 1: "I'm the leader!"     ← Split brain
Node 2: "I'm the leader!"     ← Both accept writes
Node 3: "Who's the leader?"   ← Confusion

Solution: Consensus algorithm ensures
  ONE leader is elected and ALL nodes agree

Without consensus, distributed systems face: - Split brain: Multiple nodes think they are the leader - Data loss: Conflicting writes on different "leaders" - Inconsistency: Nodes diverge and cannot reconcile

Raft Consensus Algorithm

Raft was designed to be understandable. It breaks consensus into three sub-problems: leader election, log replication, and safety.

Raft Node States

┌──────────┐    timeout     ┌────────────┐   wins election  ┌──────────┐
│ Follower │───────────────>│ Candidate  │────────────────>│  Leader  │
│          │                │            │                  │          │
│          │<───────────────│            │<─────────────────│          │
│          │  discovers     │            │  discovers       │          │
│          │  current       │            │  higher term     │          │
│          │  leader        │            │  leader          │          │
└──────────┘                └────────────┘                  └──────────┘

Leader Election

Term 1: Node 1 is leader
  Node 1 (Leader) ──heartbeat──> Node 2 (Follower)
  Node 1 (Leader) ──heartbeat──> Node 3 (Follower)

Node 1 crashes! Heartbeat stops.

Term 2: Election
  Node 2: "Timeout! No heartbeat. Starting election for Term 2"
  Node 2 ──RequestVote──> Node 3: "Vote for me?"
  Node 3: "OK, you have my vote"
  Node 2: "I have 2/3 votes (majority). I am the new leader!"

Term 2: Node 2 is leader
  Node 2 (Leader) ──heartbeat──> Node 3 (Follower)
  (Node 1 is down)

Node 1 comes back:
  Node 1: "I was leader of Term 1"
  Node 2: "I'm leader of Term 2 (higher term)"
  Node 1: "OK, I'll be a follower"

Simplified Raft Implementation (Educational)

class RaftNode {
  constructor(id, peers) {
    this.id = id;
    this.peers = peers; // Other node URLs
    this.state = 'follower'; // follower, candidate, leader
    this.currentTerm = 0;
    this.votedFor = null;
    this.log = [];
    this.commitIndex = 0;

    // Timeouts
    this.electionTimeout = this._randomTimeout(150, 300);
    this.heartbeatInterval = 50;

    this.resetElectionTimer();
  }

  _randomTimeout(min, max) {
    return min + Math.random() * (max - min);
  }

  resetElectionTimer() {
    clearTimeout(this.electionTimer);
    this.electionTimer = setTimeout(
      () => this.startElection(),
      this.electionTimeout
    );
  }

  async startElection() {
    this.state = 'candidate';
    this.currentTerm++;
    this.votedFor = this.id;
    let votesReceived = 1; // Vote for self

    console.log(`Node ${this.id}: Starting election for term ${this.currentTerm}`);

    const votePromises = this.peers.map(async (peer) => {
      try {
        const response = await fetch(`${peer}/request-vote`, {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({
            term: this.currentTerm,
            candidateId: this.id,
            lastLogIndex: this.log.length - 1,
            lastLogTerm: this.log[this.log.length - 1]?.term || 0,
          }),
          signal: AbortSignal.timeout(1000),
        });
        return response.json();
      } catch {
        return { voteGranted: false };
      }
    });

    const results = await Promise.all(votePromises);
    votesReceived += results.filter(r => r.voteGranted).length;

    const majority = Math.floor((this.peers.length + 1) / 2) + 1;

    if (this.state === 'candidate' && votesReceived >= majority) {
      this.becomeLeader();
    } else {
      this.resetElectionTimer();
    }
  }

  becomeLeader() {
    this.state = 'leader';
    console.log(`Node ${this.id}: Became leader for term ${this.currentTerm}`);

    // Send periodic heartbeats
    this.heartbeatTimer = setInterval(
      () => this.sendHeartbeats(),
      this.heartbeatInterval
    );
  }

  async sendHeartbeats() {
    for (const peer of this.peers) {
      try {
        await fetch(`${peer}/append-entries`, {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({
            term: this.currentTerm,
            leaderId: this.id,
            entries: [],
            leaderCommit: this.commitIndex,
          }),
          signal: AbortSignal.timeout(500),
        });
      } catch {
        // Peer may be down
      }
    }
  }

  // Handle vote request
  handleVoteRequest({ term, candidateId, lastLogIndex, lastLogTerm }) {
    if (term > this.currentTerm) {
      this.currentTerm = term;
      this.state = 'follower';
      this.votedFor = null;
    }

    const voteGranted =
      term >= this.currentTerm &&
      (this.votedFor === null || this.votedFor === candidateId);

    if (voteGranted) {
      this.votedFor = candidateId;
      this.resetElectionTimer();
    }

    return { term: this.currentTerm, voteGranted };
  }

  // Handle heartbeat / log entries
  handleAppendEntries({ term, leaderId, entries }) {
    if (term >= this.currentTerm) {
      this.currentTerm = term;
      this.state = 'follower';
      this.resetElectionTimer();
    }

    return { term: this.currentTerm, success: term >= this.currentTerm };
  }
}

Using etcd for Leader Election in Node.js

import { Etcd3 } from 'etcd3';

const client = new Etcd3({ hosts: 'http://etcd:2379' });

class LeaderElection {
  constructor(client, electionName, instanceId) {
    this.client = client;
    this.electionName = electionName;
    this.instanceId = instanceId;
    this.isLeader = false;
  }

  async start() {
    const lease = this.client.lease(10); // 10 second TTL
    lease.on('lost', () => {
      this.isLeader = false;
      console.log('Lost leadership, re-electing...');
      this.start();
    });

    const key = `election/${this.electionName}`;

    try {
      // Try to become leader (atomic put-if-absent)
      await this.client.if(key, 'Create', '==', 0)
        .then(this.client.put(key).value(this.instanceId).lease(lease.grant))
        .commit();

      this.isLeader = true;
      console.log(`${this.instanceId}: I am the leader!`);
    } catch {
      // Someone else is leader — watch for changes
      this.isLeader = false;
      console.log(`${this.instanceId}: Following. Watching for leader change...`);

      const watcher = await this.client.watch().key(key).create();
      watcher.on('delete', () => {
        watcher.cancel();
        this.start(); // Re-try election
      });
    }
  }
}

// Usage
const election = new LeaderElection(client, 'scheduler', 'instance-1');
await election.start();

// Only leader runs scheduled tasks
if (election.isLeader) {
  runScheduledTasks();
}

Common Pitfalls

  1. Implementing your own consensus: Use established implementations (etcd, ZooKeeper, Consul). Rolling your own is extremely error-prone.
  2. Not handling split brain: If two nodes both think they are leader, data corruption can occur. Always verify leadership before performing leader-only operations.
  3. Network partition between minority and majority: The minority partition cannot elect a leader (no majority). Ensure your cluster has an odd number of nodes.
  4. Not testing failure scenarios: Consensus algorithms are only as good as their implementation. Test node crashes, network partitions, and slow nodes.

Summary

Distributed consensus enables multiple nodes to agree on values, leader election, and state machine replication. Raft is the preferred algorithm for its understandability — it separates leader election from log replication and guarantees safety through term-based voting. In practice, use established consensus systems (etcd, Consul, ZooKeeper) rather than implementing your own. Consensus is the foundation of reliable distributed systems, from database replication to service discovery.

Common Problems & Solutions

1 Two nodes both acting as leader (split brain)
Cause: Network partition isolated the old leader from the majority. The majority elected a new leader, but the old leader doesn't know and continues operating.
Solution: Leaders must verify their lease/term before performing write operations. Use fencing tokens that increase with each leadership change. Old leaders with stale tokens are rejected.
2 Cluster cannot elect a leader
Cause: Even number of nodes causes tied votes. Or too many nodes are down, preventing a majority.
Solution: Always use an odd number of nodes (3, 5, 7). Ensure at least (N/2 + 1) nodes are healthy for the cluster to function.
