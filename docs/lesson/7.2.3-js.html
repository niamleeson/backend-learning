RBAC and ABAC: Role and Attribute-Based Access Control

Intuition

RBAC is like a building access card system: your card says you are an "Engineer" and that gives you access to the engineering floor, the lab, and the cafeteria. ABAC is like a smart building that also checks the time of day, whether you booked the conference room, and whether you have completed your safety training before letting you into the lab. Most real buildings use both: the card gets you in the door (RBAC), but certain rooms have additional rules (ABAC).

RBAC and ABAC: Role and Attribute-Based Access Control

Why Access Control Matters

Authentication answers "Who are you?" Authorization answers "What are you allowed to do?" Even after you know a user is Alice, you need to determine whether Alice can delete a blog post, approve a purchase order, or access the admin dashboard.

Getting authorization wrong leads to some of the most dangerous security vulnerabilities: users accessing other users' data, regular users performing admin actions, or sensitive operations being exposed to the public.

RBAC: Role-Based Access Control

RBAC is the most widely used authorization model. The core idea is simple: instead of assigning permissions directly to users, you create roles (like "admin", "editor", "viewer") and assign permissions to those roles. Then you assign roles to users.

RBAC Data Model

┌──────────┐       ┌──────────┐       ┌──────────────┐
│  Users    │──M:N──│  Roles   │──M:N──│  Permissions  │
│           │       │          │       │               │
│  alice    │       │  admin   │       │  posts:create  │
│  bob      │       │  editor  │       │  posts:read    │
│  charlie  │       │  viewer  │       │  posts:update  │
│           │       │          │       │  posts:delete  │
│           │       │          │       │  users:manage  │
└──────────┘       └──────────┘       └──────────────┘

alice  -> admin   -> posts:create, posts:read, posts:update,
                     posts:delete, users:manage
bob    -> editor  -> posts:create, posts:read, posts:update
charlie -> viewer -> posts:read

Database Schema

// prisma/schema.prisma equivalent in SQL
// Using a permission-string approach (simpler)

const knex = require('knex');

async function createRBACTables(knex) {
  // Roles table
  await knex.schema.createTable('roles', (table) => {
    table.increments('id').primary();
    table.string('name').unique().notNullable();   // 'admin', 'editor'
    table.string('description');
    table.timestamps(true, true);
  });

  // Permissions table
  await knex.schema.createTable('permissions', (table) => {
    table.increments('id').primary();
    table.string('resource').notNullable();   // 'posts', 'users'
    table.string('action').notNullable();     // 'create', 'read', 'update', 'delete'
    table.string('description');
    table.unique(['resource', 'action']);
  });

  // Role-Permission mapping (M:N)
  await knex.schema.createTable('role_permissions', (table) => {
    table.integer('role_id').references('id').inTable('roles').onDelete('CASCADE');
    table.integer('permission_id').references('id').inTable('permissions').onDelete('CASCADE');
    table.primary(['role_id', 'permission_id']);
  });

  // User-Role mapping (M:N)
  await knex.schema.createTable('user_roles', (table) => {
    table.integer('user_id').references('id').inTable('users').onDelete('CASCADE');
    table.integer('role_id').references('id').inTable('roles').onDelete('CASCADE');
    table.primary(['user_id', 'role_id']);
  });
}

RBAC Implementation

// rbac/rbac-service.js
class RBACService {
  constructor(db) {
    this.db = db;
    this.permissionCache = new Map(); // Cache: userId -> Set<permission>
  }

  // Get all permissions for a user (through their roles)
  async getUserPermissions(userId) {
    // Check cache first
    if (this.permissionCache.has(userId)) {
      return this.permissionCache.get(userId);
    }

    const rows = await this.db('user_roles')
      .join('role_permissions', 'user_roles.role_id', 'role_permissions.role_id')
      .join('permissions', 'role_permissions.permission_id', 'permissions.id')
      .where('user_roles.user_id', userId)
      .select('permissions.resource', 'permissions.action');

    const permissions = new Set(
      rows.map(r => `${r.resource}:${r.action}`)
    );

    // Cache for 5 minutes
    this.permissionCache.set(userId, permissions);
    setTimeout(() => this.permissionCache.delete(userId), 5 * 60 * 1000);

    return permissions;
  }

  // Check if a user has a specific permission
  async hasPermission(userId, resource, action) {
    const permissions = await this.getUserPermissions(userId);
    return permissions.has(`${resource}:${action}`);
  }

  // Check if a user has a specific role
  async hasRole(userId, roleName) {
    const role = await this.db('user_roles')
      .join('roles', 'user_roles.role_id', 'roles.id')
      .where({ 'user_roles.user_id': userId, 'roles.name': roleName })
      .first();
    return !!role;
  }

  // Assign a role to a user
  async assignRole(userId, roleName) {
    const role = await this.db('roles').where({ name: roleName }).first();
    if (!role) throw new Error(`Role not found: ${roleName}`);

    await this.db('user_roles').insert({
      user_id: userId,
      role_id: role.id
    }).onConflict(['user_id', 'role_id']).ignore();

    // Invalidate cache
    this.permissionCache.delete(userId);
  }

  // Remove a role from a user
  async removeRole(userId, roleName) {
    const role = await this.db('roles').where({ name: roleName }).first();
    if (!role) throw new Error(`Role not found: ${roleName}`);

    await this.db('user_roles').where({
      user_id: userId,
      role_id: role.id
    }).delete();

    this.permissionCache.delete(userId);
  }
}

module.exports = { RBACService };

Express Middleware for RBAC

// rbac/middleware.js
function requirePermission(resource, action) {
  return async (req, res, next) => {
    if (!req.user) {
      return res.status(401).json({ error: 'Authentication required' });
    }

    const allowed = await req.rbac.hasPermission(req.user.id, resource, action);

    if (!allowed) {
      return res.status(403).json({
        error: 'Forbidden',
        message: `You do not have permission to ${action} ${resource}`
      });
    }

    next();
  };
}

function requireRole(...roles) {
  return async (req, res, next) => {
    if (!req.user) {
      return res.status(401).json({ error: 'Authentication required' });
    }

    const userRoles = await Promise.all(
      roles.map(role => req.rbac.hasRole(req.user.id, role))
    );

    if (!userRoles.some(Boolean)) {
      return res.status(403).json({
        error: 'Forbidden',
        message: `Required role: ${roles.join(' or ')}`
      });
    }

    next();
  };
}

module.exports = { requirePermission, requireRole };

// routes/posts.js
const express = require('express');
const { requirePermission, requireRole } = require('../rbac/middleware');

const router = express.Router();

// Anyone with posts:read can list posts

Hierarchical Roles

In many systems, roles inherit from other roles. An admin has all editor permissions, and an editor has all viewer permissions.

// rbac/hierarchical-rbac.js
const ROLE_HIERARCHY = {
  admin: ['editor', 'viewer'],
  editor: ['viewer'],
  viewer: []
};

class HierarchicalRBAC {
  // Get all roles including inherited ones
  getEffectiveRoles(role) {
    const roles = new Set([role]);
    const queue = [role];

    while (queue.length > 0) {
      const current = queue.shift();
      const inherited = ROLE_HIERARCHY[current] || [];
      for (const parent of inherited) {
        if (!roles.has(parent)) {
          roles.add(parent);
          queue.push(parent);
        }
      }
    }

    return roles;
  }
}

// admin -> gets {admin, editor, viewer}
// editor -> gets {editor, viewer}
// viewer -> gets {viewer}

ABAC: Attribute-Based Access Control

RBAC works well for many scenarios, but it breaks down when access decisions depend on context: "Editors can edit posts, but only their own posts" or "Managers can approve expenses under $10,000, but directors can approve any amount."

ABAC evaluates policies based on attributes of the subject (user), resource (what they are accessing), action (what they want to do), and environment (time, location, IP).

ABAC Concepts

┌─────────────────────────────────────────────────────────┐
│                    ABAC Policy Engine                     │
│                                                          │
│  Subject Attributes  │  Resource Attributes               │
│  - role: "editor"    │  - type: "post"                    │
│  - department: "eng" │  - owner_id: 42                    │
│  - clearance: 3      │  - status: "draft"                 │
│                      │  - classification: "internal"       │
│  Action Attributes   │  Environment Attributes             │
│  - type: "update"    │  - time: "2024-03-15T14:30:00Z"    │
│  - fields: ["body"] │  - ip: "192.168.1.100"              │
│                      │  - mfa_verified: true               │
│                                                          │
│  Policy: ALLOW if                                        │
│    subject.role == "editor" AND                          │
│    resource.owner_id == subject.id AND                   │
│    resource.status == "draft"                            │
│                                                          │
│  Result: ALLOW / DENY                                    │
└─────────────────────────────────────────────────────────┘

ABAC Policy Engine

// abac/policy-engine.js
class PolicyEngine {
  constructor() {
    this.policies = [];
  }

  addPolicy(policy) {
    this.policies.push(policy);
  }

  evaluate(context) {
    // Find all policies that apply to this action and resource
    const applicable = this.policies.filter(policy =>
      policy.resource === context.resource.type &&
      policy.action === context.action
    );

    if (applicable.length === 0) {
      return { allowed: false, reason: 'No applicable policy found' };
    }

    // Deny-first: if any policy explicitly denies, deny
    for (const policy of applicable) {
      const result = policy.evaluate(context);
      if (result.effect === 'deny') {
        return { allowed: false, reason: result.reason };
      }
    }

    // Then check for any allow
    for (const policy of applicable) {
      const result = policy.evaluate(context);
      if (result.effect === 'allow') {
        return { allowed: true, reason: result.reason };
      }
    }

    // Default deny
    return { allowed: false, reason: 'No policy granted access' };
  }
}

// Define policies as objects
class Policy {
  constructor({ name, resource, action, effect, condition }) {
    this.name = name;
    this.resource = resource;
    this.action = action;
    this.effect = effect; // 'allow' or 'deny'
    this.condition = condition; // function(context) => boolean
  }

  evaluate(context) {
    const matches = this.condition(context);
    return {
      effect: matches ? this.effect : 'no_match',
      reason: matches ? this.name : null
    };
  }
}

module.exports = { PolicyEngine, Policy };

Defining ABAC Policies

// abac/policies.js
const { PolicyEngine, Policy } = require('./policy-engine');

const engine = new PolicyEngine();

// Policy: Admins can do anything with posts
engine.addPolicy(new Policy({
  name: 'admin-full-access',
  resource: 'post',
  action: '*',
  effect: 'allow',
  condition: (ctx) => ctx.subject.roles.includes('admin')
}));

// Policy: Editors can update their own posts
engine.addPolicy(new Policy({
  name: 'editor-own-posts',
  resource: 'post',
  action: 'update',
  effect: 'allow',
  condition: (ctx) =>
    ctx.subject.roles.includes('editor') &&
    ctx.resource.ownerId === ctx.subject.id
}));

// Policy: Editors can update any post if it is a draft
engine.addPolicy(new Policy({
  name: 'editor-draft-posts',
  resource: 'post',
  action: 'update',
  effect: 'allow',
  condition: (ctx) =>
    ctx.subject.roles.includes('editor') &&
    ctx.resource.status === 'draft'
}));

// Policy: No one can delete published posts (deny overrides allow)
engine.addPolicy(new Policy({
  name: 'no-delete-published',
  resource: 'post',
  action: 'delete',
  effect: 'deny',
  condition: (ctx) => ctx.resource.status === 'published'
}));

// Policy: Managers can approve expenses under their limit
engine.addPolicy(new Policy({
  name: 'manager-expense-approval',
  resource: 'expense',
  action: 'approve',
  effect: 'allow',
  condition: (ctx) =>
    ctx.subject.roles.includes('manager') &&
    ctx.resource.amount <= ctx.subject.approvalLimit
}));

// Policy: Deny access outside business hours for sensitive resources
engine.addPolicy(new Policy({
  name: 'business-hours-only',
  resource: 'financial_report',
  action: 'read',
  effect: 'deny',
  condition: (ctx) => {
    const hour = new Date(ctx.environment.timestamp).getHours();
    return hour < 8 || hour > 18;
  }
}));

module.exports = { engine };

ABAC Middleware

// abac/middleware.js
function abacMiddleware(engine) {
  return (resourceType, action, getResource) => {
    return async (req, res, next) => {
      if (!req.user) {
        return res.status(401).json({ error: 'Authentication required' });
      }

      // Build the ABAC context
      const resource = getResource
        ? await getResource(req)    // Fetch the resource to check attributes
        : { type: resourceType };

      const context = {
        subject: {
          id: req.user.id,
          roles: req.user.roles,
          department: req.user.department,
          approvalLimit: req.user.approvalLimit
        },
        resource: {
          type: resourceType,
          ...resource
        },
        action: action,
        environment: {
          timestamp: new Date().toISOString(),
          ip: req.ip,
          userAgent: req.get('user-agent')
        }
      };

      const decision = engine.evaluate(context);

      if (!decision.allowed) {
        return res.status(403).json({
          error: 'Forbidden',
          reason: decision.reason
        });
      }

      // Attach the resource to avoid re-fetching in the handler
      req.resource = resource;
      next();
    };
  };
}

module.exports = { abacMiddleware };

// routes/posts.js (with ABAC)
const { abacMiddleware } = require('../abac/middleware');
const { engine } = require('../abac/policies');

const authorize = abacMiddleware(engine);

// Fetch the post resource for attribute checking
async function getPost(req) {
  return await PostService.findById(req.params.id);
}

// Update: engine evaluates ownership and status
router.put('/:id',
  authorize('post', 'update', getPost),
  async (req, res) => {
    // req.resource is already loaded by the middleware
    const updated = await PostService.update(req.params.id, req.body);
    res.json(updated);
  }
);

// Delete: engine checks if post is published (deny policy)
router.delete('/:id',
  authorize('post', 'delete', getPost),
  async (req, res) => {
    await PostService.delete(req.params.id);
    res.status(204).end();
  }
);

Combining RBAC and ABAC

In practice, most applications use a hybrid approach: RBAC for coarse-grained access and ABAC for fine-grained rules.

// hybrid-authorization.js
class HybridAuthorization {
  constructor(rbacService, abacEngine) {
    this.rbac = rbacService;
    this.abac = abacEngine;
  }

  async authorize(context) {
    // Step 1: RBAC check (fast, coarse-grained)
    const hasBasePermission = await this.rbac.hasPermission(
      context.subject.id,
      context.resource.type,
      context.action
    );

    if (!hasBasePermission) {
      return {
        allowed: false,
        reason: 'Role does not include this permission'
      };
    }

    // Step 2: ABAC check (contextual, fine-grained)
    const abacResult = this.abac.evaluate(context);

    return abacResult;
  }
}

// Usage:
// 1. RBAC says: "editors can update posts" (general permission)
// 2. ABAC says: "but only if it is their own post or it is a draft"

Seeding Roles and Permissions

// seeds/rbac-seed.js
async function seedRBAC(db) {
  // Create permissions
  const permissions = [
    { resource: 'posts', action: 'create' },
    { resource: 'posts', action: 'read' },
    { resource: 'posts', action: 'update' },
    { resource: 'posts', action: 'delete' },
    { resource: 'posts', action: 'publish' },
    { resource: 'users', action: 'read' },
    { resource: 'users', action: 'create' },
    { resource: 'users', action: 'update' },
    { resource: 'users', action: 'delete' },
    { resource: 'settings', action: 'read' },
    { resource: 'settings', action: 'update' }
  ];

  for (const perm of permissions) {
    await db('permissions').insert(perm).onConflict(['resource', 'action']).ignore();
  }

  // Create roles
  const roles = [
    { name: 'admin', description: 'Full system access' },
    { name: 'editor', description: 'Content management' },
    { name: 'viewer', description: 'Read-only access' }
  ];

  for (const role of roles) {
    await db('roles').insert(role).onConflict('name').ignore();
  }

  // Define role-permission mappings
  const rolePermissions = {
    admin: ['posts:*', 'users:*', 'settings:*'],  // All permissions
    editor: ['posts:create', 'posts:read', 'posts:update', 'posts:publish',
             'users:read'],
    viewer: ['posts:read', 'users:read']
  };

  for (const [roleName, perms] of Object.entries(rolePermissions)) {
    const role = await db('roles').where({ name: roleName }).first();

    for (const perm of perms) {
      if (perm.includes('*')) {
        // Wildcard: assign all permissions for this resource
        const resource = perm.split(':')[0];
        const matchingPerms = await db('permissions').where({ resource });
        for (const p of matchingPerms) {
          await db('role_permissions').insert({
            role_id: role.id, permission_id: p.id
          }).onConflict(['role_id', 'permission_id']).ignore();
        }
      } else {
        const [resource, action] = perm.split(':');
        const p = await db('permissions').where({ resource, action }).first();
        if (p) {
          await db('role_permissions').insert({
            role_id: role.id, permission_id: p.id
          }).onConflict(['role_id', 'permission_id']).ignore();
        }
      }
    }
  }
}

module.exports = { seedRBAC };

Using CASL for Production ABAC

For production Node.js applications, the CASL library provides a battle-tested ABAC implementation.

// casl/abilities.js
const { AbilityBuilder, createMongoAbility } = require('@casl/ability');

function defineAbilitiesFor(user) {
  const { can, cannot, build } = new AbilityBuilder(createMongoAbility);

  if (user.roles.includes('admin')) {
    can('manage', 'all');  // Admin can do everything
  }

  if (user.roles.includes('editor')) {
    can('read', 'Post');
    can('create', 'Post');
    can('update', 'Post', { authorId: user.id });          // Own posts only
    can('update', 'Post', { status: 'draft' });            // Any draft
    cannot('delete', 'Post', { status: 'published' });     // No deleting published
  }

  if (user.roles.includes('viewer')) {
    can('read', 'Post', { status: 'published' });          // Only published
  }

  // Everyone can read their own profile
  can('read', 'User', { id: user.id });
  can('update', 'User', { id: user.id });

  return build();
}

// Usage in Express
app.use((req, res, next) => {
  if (req.user) {
    req.ability = defineAbilitiesFor(req.user);
  }
  next();
});

router.put('/posts/:id', async (req, res) => {
  const post = await PostService.findById(req.params.id);

  if (!req.ability.can('update', post)) {
    return res.status(403).json({ error: 'Cannot update this post' });
  }

  const updated = await PostService.update(post.id, req.body);
  res.json(updated);
});

Common Pitfalls

  1. Checking roles instead of permissions - Avoid if (user.role === 'admin') scattered through code. Check permissions instead so you can remap them without code changes.
  2. Forgetting the default deny - Always deny by default and explicitly grant access.
  3. Client-side only authorization - Hiding a button is NOT security. Always enforce on the server.
  4. Role explosion - Too many fine-grained roles become unmanageable. That is when ABAC is needed.
  5. Not invalidating cached permissions - When roles change, cached permission lookups become stale.

Exercises

Include endpoints to: create roles, create permissions, assign permissions to roles, assign roles to users, and check if a user has a specific permission. Use SQLite with Knex for storage.

    Hint

  2. ABAC Policy Engine

    Build a policy engine that evaluates authorization decisions based on subject, resource, action, and environment attributes. Define at least 5 policies including: ownership-based access, time-based restrictions, and deny-overrides. Write comprehensive tests for each policy.

    Hint

  3. Hybrid RBAC + ABAC Blog Authorization

Common Problems & Solutions

1 User reports they cannot access a resource despite having the correct role
Cause: The permission cache is stale. The user was recently assigned a new role but the cached permissions still reflect their old roles. Or there is a typo in the resource/action string (e.g., 'post' vs 'posts').
Solution: Invalidate the permission cache whenever roles or permissions change. Use consistent naming conventions for resources and actions. Add logging to the authorization middleware that shows exactly which permission was checked and what the user's effective permissions are.
2 Role explosion: the system has 50+ roles and they are hard to manage
Cause: The team created very specific roles like 'marketing-blog-editor' and 'engineering-blog-editor' instead of using attributes. This happens when RBAC is stretched beyond its design limits.
Solution: Migrate to a hybrid RBAC + ABAC model. Reduce roles to broad categories (admin, editor, viewer) and use ABAC policies for fine-grained rules based on department, ownership, or resource attributes. Consider using a library like CASL to define abilities declaratively.
3 Authorization check passes but the user sees a 403 on some resources and not others
Cause: The ABAC policy depends on resource attributes (like ownership or status) that differ between resources. The permission check works for resources the user owns but fails for others, and the error message does not explain which attribute caused the denial.
Solution: Add detailed logging to your policy engine that records which policy matched and which condition failed. Return specific error messages like 'You can only edit your own posts' instead of generic 'Forbidden'. Implement a 'dry run' mode that explains why access was denied.
