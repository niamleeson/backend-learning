EXPLAIN ANALYZE and Query Optimization

Intuition

EXPLAIN ANALYZE is like an X-ray of your query. It shows exactly how the database decided to find your data and where it spent the most time.

EXPLAIN ANALYZE: Query Optimization

EXPLAIN ANALYZE executes a query and shows the actual execution plan, including row estimates, timing, and which indexes were used.

-- Basic EXPLAIN ANALYZE
EXPLAIN ANALYZE
SELECT u.name, COUNT(o.id)
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.status = 'active'
GROUP BY u.name;

-- Verbose output with buffers
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM products
WHERE category = 'electronics'
ORDER BY price;

Key plan nodes to understand: - Seq Scan: full table scan, no index used - Index Scan / Index Only Scan: using an index - Hash Join / Nested Loop / Merge Join: join strategies - Sort: explicit sorting step

Key Takeaways: - Compare estimated vs actual rows; large mismatches indicate stale statistics (run ANALYZE). - BUFFERS shows cache hit ratio; high shared hit means data is in memory. - Look for the most expensive node (highest actual time) and optimize that first.

Exercises

  1. Read a Query Plan

    Run EXPLAIN ANALYZE on a JOIN query with a WHERE clause. Identify whether an index scan or sequential scan is used and note the actual vs estimated rows.

    Hint

    Look for 'Seq Scan' or 'Index Scan' nodes. Compare 'rows=' (estimated) with 'actual rows='.

  2. Optimize a Slow Query

    Given a query with a Seq Scan on a large table, add an appropriate index and verify the plan changes to Index Scan.

    Hint

    Create an index on the filtered column, rerun EXPLAIN ANALYZE, and confirm the scan type changed.

Common Problems & Solutions

1 Estimated rows are wildly different from actual rows.
Cause: Table statistics are outdated after bulk inserts or deletes.
Solution: Run ANALYZE on the table to refresh statistics. Adjust default_statistics_target for high-cardinality columns.
2 Query uses Nested Loop Join and is very slow.
Cause: Nested Loop is O(n*m) and poor for large tables without an index on the inner relation.
Solution: Ensure the inner table has an index on the join column. The planner may switch to Hash or Merge Join.
