Docker Compose: Multi-Container Applications

Intuition

Docker Compose is like a stage director's script. Each service (actor) has its entrance cue (depends_on), costume (image/build), and position (ports). The director (Compose) ensures everyone appears on stage in the right order and can talk to each other by name.

Docker Compose: Multi-Container Applications

Real applications need multiple services: a web app, database, cache, worker. Docker Compose defines and runs multi-container setups from a single YAML file.

docker-compose.yml

services:
  web:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      db:
        condition: service_healthy
    volumes:
      - .:/app
    restart: unless-stopped

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d mydb"]
      interval: 5s
      retries: 5

  redis:
    image: redis:7-alpine

  worker:
    build: .
    command: celery -A tasks worker --loglevel=info
    depends_on:
      - db
      - redis

volumes:
  pgdata:

Essential Commands

docker compose up -d              # Start all services in background
docker compose logs -f web        # Follow logs for web service
docker compose exec web bash      # Shell into running container
docker compose down -v            # Stop and remove containers + volumes
docker compose ps                 # List running services

Common Mistakes

  • Using depends_on without health checks -- the DB container starts but Postgres isn't ready
  • Hardcoding passwords in docker-compose.yml committed to git
  • Mounting volumes in production (use only for development)

Key Takeaways

  • Use health checks with depends_on conditions for reliable startup order
  • Named volumes persist data across container restarts
  • Services communicate via service names as hostnames (e.g., db, redis)

Exercises

  1. Full-Stack Compose Setup

    Create a docker-compose.yml for a FastAPI app with PostgreSQL, Redis, and a Celery worker. Include health checks for Postgres, proper depends_on ordering, and a named volume for database persistence.

    Hint

    Use the pg_isready command for Postgres healthcheck and service names for inter-container networking.

  2. Development vs Production Compose

    Create a base docker-compose.yml and a docker-compose.override.yml for development (with volume mounts and debug mode) and a docker-compose.prod.yml for production (without volume mounts, with restart policies).

    Hint

    Use docker compose -f docker-compose.yml -f docker-compose.prod.yml up for production.

  3. Compose with Environment Files

    Refactor a docker-compose.yml to use an .env file for all secrets and configuration values. Ensure the .env file is in .gitignore and create a .env.example template.

    Hint

    Use env_file directive or ${VARIABLE} syntax in docker-compose.yml.

Common Problems & Solutions

1 Web container crashes because database isn't ready
Symptoms: ['Connection refused to port 5432', 'FATAL: the database system is starting up']
Cause: depends_on only waits for the container to start, not for the service inside to be ready.
Solution: Add a healthcheck to the database service and use `depends_on: db: condition: service_healthy`.
2 Data lost when running docker compose down
Cause: Using `docker compose down -v` removes named volumes containing database data.
Solution: Use `docker compose down` without `-v` to preserve volumes. Only use `-v` when you intentionally want to reset data.
3 Cannot connect to service from host machine
Symptoms: ['curl: (7) Failed to connect to localhost port 8000']
Cause: Port mapping is missing or the service is binding to 127.0.0.1 inside the container.
Solution: Ensure the service binds to `0.0.0.0` inside the container and that `ports` mapping is defined in docker-compose.yml.
