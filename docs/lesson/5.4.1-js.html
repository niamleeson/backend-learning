Protocol Buffers and Service Definitions

Intuition

Protocol Buffers are like a precisely engineered shipping container system. Instead of wrapping each item in variable-size gift wrap (JSON), every item goes into a numbered slot in a standardized container. Both the sender and receiver have the same container blueprint (.proto file), so there is never confusion about what goes where.

Protocol Buffers and Service Definitions

Protocol Buffers (protobuf) is Google's language-neutral data serialization format used by gRPC. It is smaller, faster, and more strictly typed than JSON. Understanding protobuf is the foundation for building gRPC services.

Why Protocol Buffers?

JSON:  { "name": "Alice", "age": 30, "active": true }
       46 bytes, text parsing required

Protobuf: [binary encoding]
       ~15 bytes, direct memory mapping, schema-enforced

Protobuf advantages: - 3-10x smaller than JSON - 20-100x faster serialization/deserialization - Strongly typed with a schema (.proto file) - Backward/forward compatible with field numbering - Code generation for any language

Proto3 Syntax

// user.proto
syntax = "proto3";

package myapp.user;

// Import other proto files
import "google/protobuf/timestamp.proto";
import "google/protobuf/empty.proto";

// Message definitions (like TypeScript interfaces)
message User {
  string id = 1;              // field number 1
  string email = 2;           // field number 2
  string name = 3;
  string bio = 4;
  Role role = 5;
  google.protobuf.Timestamp created_at = 6;
  repeated string tags = 7;   // array of strings
  optional string avatar_url = 8; // explicitly optional
}

// Enums
enum Role {
  ROLE_UNSPECIFIED = 0;  // proto3 requires 0 value
  ROLE_USER = 1;
  ROLE_ADMIN = 2;
  ROLE_MODERATOR = 3;
}

// Nested messages
message Address {
  string street = 1;
  string city = 2;
  string state = 3;
  string zip = 4;
  string country = 5;
}

message UserProfile {
  User user = 1;
  Address address = 2;
  map<string, string> metadata = 3;  // key-value pairs
}

Field Numbers Are Critical

Field numbers are the wire identifier. They must: - Never be reused after deletion (use reserved) - Stay stable across versions - Use 1-15 for frequently used fields (1 byte encoding) - Use 16-2047 for less common fields (2 byte encoding)

message Post {
  reserved 4, 8;              // these numbers cannot be reused
  reserved "old_field_name";  // these names cannot be reused

  string id = 1;
  string title = 2;
  string body = 3;
  // field 4 was deleted (old 'summary' field)
  string author_id = 5;
  PostStatus status = 6;
}

Service Definitions

gRPC services are defined in proto files:

// service.proto
syntax = "proto3";

package myapp.blog;

import "google/protobuf/empty.proto";
import "google/protobuf/timestamp.proto";

// Request/Response messages
message CreatePostRequest {
  string title = 1;
  string body = 2;
  repeated string tags = 3;
}

message GetPostRequest {
  string id = 1;
}

message ListPostsRequest {
  int32 page_size = 1;
  string page_token = 2;  // cursor for pagination
  PostStatus status_filter = 3;
}

message ListPostsResponse {
  repeated Post posts = 1;
  string next_page_token = 2;
  int32 total_count = 3;
}

message Post {
  string id = 1;
  string title = 2;
  string body = 3;
  string author_id = 4;
  PostStatus status = 5;
  repeated string tags = 6;
  google.protobuf.Timestamp created_at = 7;
  google.protobuf.Timestamp updated_at = 8;
}

message UpdatePostRequest {
  string id = 1;
  optional string title = 2;
  optional string body = 3;
  repeated string tags = 4;
  optional PostStatus status = 5;
}

message DeletePostRequest {
  string id = 1;
}

enum PostStatus {
  POST_STATUS_UNSPECIFIED = 0;
  POST_STATUS_DRAFT = 1;
  POST_STATUS_PUBLISHED = 2;
  POST_STATUS_ARCHIVED = 3;
}

// Service definition
service BlogService {
  // Unary RPC: one request, one response
  rpc CreatePost(CreatePostRequest) returns (Post);
  rpc GetPost(GetPostRequest) returns (Post);
  rpc UpdatePost(UpdatePostRequest) returns (Post);
  rpc DeletePost(DeletePostRequest) returns (google.protobuf.Empty);
  rpc ListPosts(ListPostsRequest) returns (ListPostsResponse);

  // Server streaming: one request, stream of responses
  rpc WatchPosts(ListPostsRequest) returns (stream Post);

  // Client streaming: stream of requests, one response
  rpc BatchCreatePosts(stream CreatePostRequest) returns (ListPostsResponse);

  // Bidirectional streaming: stream both ways
  rpc Chat(stream ChatMessage) returns (stream ChatMessage);
}

message ChatMessage {
  string sender_id = 1;
  string text = 2;
  google.protobuf.Timestamp timestamp = 3;
}

Generating Code for Node.js

Use grpc-tools and ts-proto for TypeScript-friendly code generation:

npm install @grpc/grpc-js @grpc/proto-loader
# For code generation:
npm install -D grpc-tools grpc_tools_node_protoc_ts
# Or for ts-proto (better TypeScript support):
npm install -D ts-proto

Using @grpc/proto-loader (Dynamic Loading)

import * as grpc from '@grpc/grpc-js';
import * as protoLoader from '@grpc/proto-loader';

const packageDefinition = protoLoader.loadSync('./protos/service.proto', {
  keepCase: true,
  longs: String,
  enums: String,
  defaults: true,
  oneofs: true,
  includeDirs: ['./protos'],
});

const proto = grpc.loadPackageDefinition(packageDefinition) as any;
const BlogService = proto.myapp.blog.BlogService;

Using ts-proto (Static Code Generation)

# Install protoc compiler and ts-proto
npx buf generate
# or directly:
protoc --plugin=./node_modules/.bin/protoc-gen-ts_proto \
  --ts_proto_out=./src/generated \
  --ts_proto_opt=outputServices=grpc-js \
  --ts_proto_opt=esModuleInterop=true \
  ./protos/service.proto

This generates TypeScript types and service interfaces:

// generated/service.ts (auto-generated)
export interface Post {
  id: string;
  title: string;
  body: string;
  authorId: string;
  status: PostStatus;
  tags: string[];
  createdAt: Date | undefined;
  updatedAt: Date | undefined;
}

export interface BlogServiceServer extends grpc.UntypedServiceImplementation {
  createPost: grpc.handleUnaryCall<CreatePostRequest, Post>;
  getPost: grpc.handleUnaryCall<GetPostRequest, Post>;
  listPosts: grpc.handleUnaryCall<ListPostsRequest, ListPostsResponse>;
  watchPosts: grpc.handleServerStreamingCall<ListPostsRequest, Post>;
}

Schema Evolution and Compatibility

Protobuf supports backward-compatible evolution:

// Version 1
message User {
  string id = 1;
  string name = 2;
}

// Version 2 (backward compatible)
message User {
  string id = 1;
  string name = 2;
  string email = 3;     // NEW: old clients ignore this
  int32 age = 4;        // NEW: old clients ignore this
}

// Version 3 (removing a field - use reserved)
message User {
  reserved 2;           // 'name' removed, number reserved
  reserved "name";
  string id = 1;
  string first_name = 5;
  string last_name = 6;
  string email = 3;
  int32 age = 4;
}

Rules for backward compatibility: - Never change a field number - Never change a field type - Add new fields with new numbers - Use reserved for removed fields - Never reuse a field number

Exercises

  1. Design a Proto Schema for an E-Commerce Service

    Write a .proto file for an e-commerce service with messages for Product (id, name, description, price, category, stock), Order (id, user_id, items, total, status), and OrderItem (product_id, quantity, unit_price). Define a ProductService and OrderService with CRUD operations.

    Hint

    Use nested messages for OrderItem inside Order. Use enums for OrderStatus and ProductCategory. Remember to start enum values at 0 with UNSPECIFIED.

  2. Practice Schema Evolution

    Start with a simple User message (id, name, email). Evolve it to add phone, address (nested message), and roles (repeated enum). Then remove the 'name' field and split it into first_name and last_name. Use reserved correctly throughout.

    Hint

    When removing a field, add both the field number and field name to reserved statements.

  3. Generate TypeScript Types from Proto

    Set up ts-proto code generation for your .proto files. Write a script that generates TypeScript interfaces and gRPC service definitions. Verify the generated types match your expectations.

    Hint

    Use protoc with --ts_proto_out and --ts_proto_opt=outputServices=grpc-js for gRPC-compatible output.

Common Problems & Solutions

1 Field number reuse causes data corruption
Cause: A deleted field's number was reused for a new field with a different type. Old messages are deserialized incorrectly.
Solution: Always use 'reserved' for deleted field numbers. Never reuse a field number, even if the old field was removed long ago.
2 Proto file imports fail with 'file not found'
Cause: The protoc compiler cannot find imported .proto files (like google/protobuf/timestamp.proto).
Solution: Use --proto_path (-I) to specify include directories. For Google well-known types, install them: npm install google-proto-files, then add the include path.
3 Enum values collide across messages
Cause: Proto3 enums are scoped to the package, not the message. Two enums in the same package with the same value name conflict.
Solution: Prefix enum values with the enum name: POST_STATUS_DRAFT, USER_ROLE_ADMIN. This is the protobuf convention.
