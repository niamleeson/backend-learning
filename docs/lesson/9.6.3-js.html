Design a Logging/Analytics Pipeline

Intuition

Think of a logging pipeline as a river system. Thousands of tributaries (services) feed into a main river (Kafka). The river flows through treatment plants (processors) that filter and enrich the water. Some water goes to a fast-access reservoir (Elasticsearch for real-time queries), some to a large lake (ClickHouse for analytics), and some to long-term underground storage (S3 for archival). Sensors along the way trigger alarms (alerts) when pollution levels spike.

Design a Logging/Analytics Pipeline

Introduction

Every production system generates massive volumes of logs, metrics, and events. A logging and analytics pipeline collects, transports, processes, stores, and queries this data to power real-time dashboards, alerting, debugging, and business intelligence. Think of systems like the ELK stack (Elasticsearch, Logstash, Kibana), Datadog, Splunk, or a custom analytics platform. This design problem is at the heart of observability engineering.

Requirements

Functional Requirements

  • Ingest structured and unstructured logs from thousands of services
  • Ingest application metrics (counters, gauges, histograms)
  • Ingest user events (page views, clicks, purchases) for business analytics
  • Real-time streaming queries (last 5 minutes of errors)
  • Historical batch queries (aggregate events over the last 30 days)
  • Alerting based on thresholds (error rate > 5%, latency p99 > 2s)
  • Dashboards with real-time and historical visualizations
  • Log retention policies (hot: 7 days, warm: 30 days, cold: 1 year)

Non-Functional Requirements

  • Ingest 10 million events/second at peak
  • Query latency: < 1s for real-time, < 30s for historical aggregations
  • Storage: handle 50 TB/day of raw data
  • Data loss tolerance: < 0.01% (best-effort for metrics, durable for business events)
  • 99.9% availability for ingestion (never drop data silently)

Capacity Estimation

Events/second:          10,000,000 (peak)
Average event size:     500 bytes
Ingestion bandwidth:    10M * 500B = 5 GB/s
Daily raw storage:      5 GB/s * 86,400 = ~430 TB/day
With compression (10x): ~43 TB/day
Monthly storage:        43 TB * 30 = ~1.3 PB/month

Query load:
  Real-time queries:    1000 queries/second
  Historical queries:   100 queries/second
  Dashboard refreshes:  500 queries/second (cached)

Index size (Elasticsearch):
  Assuming 1:1 index-to-data ratio: ~43 TB/day of indices
  7-day hot tier: ~300 TB of indices

API Design

Event Ingestion API

import express from 'express';
import { Kafka } from 'kafkajs';
import { v4 as uuidv4 } from 'uuid';

const app = express();
app.use(express.json({ limit: '5mb' }));

const kafka = new Kafka({ brokers: ['kafka-1:9092', 'kafka-2:9092'] });
const producer = kafka.producer({ allowAutoTopicCreation: false });

// POST /api/v1/ingest/events
// Batch event ingestion endpoint
app.post('/api/v1/ingest/events', async (req, res) => {
  const { events } = req.body;

  if (!Array.isArray(events) || events.length === 0) {
    return res.status(400).json({ error: 'events array required' });
  }
  if (events.length > 1000) {
    return res.status(400).json({ error: 'Max 1000 events per batch' });
  }

  const enrichedEvents = events.map(event => ({
    id: event.id || uuidv4(),
    type: event.type,
    source: event.source,
    timestamp: event.timestamp || Date.now(),
    data: event.data,
    metadata: {
      ...event.metadata,
      ingestTimestamp: Date.now(),
      ingestServer: process.env.HOSTNAME,
      ip: req.ip,
      userAgent: req.headers['user-agent'],
    },
  }));

  const messages = enrichedEvents.map(event => ({
    key: event.source,  // partition by source for ordering
    value: JSON.stringify(event),
    headers: {
      eventType: event.type,
      source: event.source,
    },
  }));

  await producer.send({
    topic: `events-${categorizeEventType(events[0].type)}`,
    messages,
    acks: 1,  // leader ack only for low latency
  });

  res.status(202).json({
    accepted: enrichedEvents.length,
    ids: enrichedEvents.map(e => e.id),
  });
});

// POST /api/v1/ingest/logs
// Structured log ingestion
app.post('/api/v1/ingest/logs', async (req, res) => {
  const { logs } = req.body;

  const messages = logs.map(log => ({
    key: log.service,
    value: JSON.stringify({
      ...log,
      '@timestamp': log.timestamp || new Date().toISOString(),
      ingestTimestamp: Date.now(),
    }),
  }));

  await producer.send({
    topic: 'logs-raw',
    messages,
    acks: 1,
  });

  res.status(202).json({ accepted: logs.length });
});

Query API

import { Client as ElasticsearchClient } from '@elastic/elasticsearch';
const es = new ElasticsearchClient({ node: 'http://elasticsearch:9200' });

// POST /api/v1/query/logs
// Search logs with filters
app.post('/api/v1/query/logs', async (req, res) => {
  const {
    query,
    service,
    level,
    startTime,
    endTime,
    limit,
    offset,
    sort,
  } = req.body;

  const must = [];
  if (query) must.push({ query_string: { query } });
  if (service) must.push({ term: { 'service.keyword': service } });
  if (level) must.push({ term: { 'level.keyword': level } });
  if (startTime || endTime) {
    must.push({
      range: {
        '@timestamp': {
          ...(startTime && { gte: startTime }),
          ...(endTime && { lte: endTime }),
        },
      },
    });
  }

  const result = await es.search({
    index: 'logs-*',
    body: {
      query: { bool: { must } },
      sort: [{ '@timestamp': { order: sort || 'desc' } }],
      from: offset || 0,
      size: limit || 100,
      highlight: {
        fields: { message: {} },
      },
    },
  });

  res.json({
    total: result.hits.total.value,
    logs: result.hits.hits.map(hit => ({
      ...hit._source,
      _highlight: hit.highlight,
    })),
  });
});

// POST /api/v1/query/analytics
// Aggregation query for analytics
app.post('/api/v1/query/analytics', async (req, res) => {
  const { eventType, groupBy, metric, interval, startTime, endTime } = req.body;

  const result = await es.search({
    index: 'events-*',
    body: {
      query: {
        bool: {
          must: [
            { term: { 'type.keyword': eventType } },
            { range: { timestamp: { gte: startTime, lte: endTime } } },
          ],
        },
      },
      aggs: {
        over_time: {
          date_histogram: {
            field: 'timestamp',
            fixed_interval: interval || '1h',
          },
          aggs: {
            metric_value: {
              [metric.type]: { field: metric.field },
            },
            ...(groupBy && {
              grouped: {
                terms: { field: `${groupBy}.keyword`, size: 20 },
              },
            }),
          },
        },
      },
      size: 0,
    },
  });

  res.json({
    buckets: result.aggregations.over_time.buckets.map(b => ({
      timestamp: b.key_as_string,
      count: b.doc_count,
      value: b.metric_value.value,
      groups: b.grouped?.buckets,
    })),
  });
});

// POST /api/v1/alerts/rules
app.post('/api/v1/alerts/rules', async (req, res) => {
  const { name, query, condition, window, channels } = req.body;
  // condition: { operator: '>', threshold: 5, metric: 'count' }
  // window: '5m'
  // channels: ['slack:#ops', 'pagerduty:team-backend']

  const rule = await alertService.createRule({
    name,
    query,
    condition,
    windowSeconds: parseDuration(window),
    channels,
    createdBy: req.user.id,
  });

  res.status(201).json(rule);
});

Database Schema

-- Alert rules
CREATE TABLE alert_rules (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            VARCHAR(255) NOT NULL,
  query           TEXT NOT NULL,       -- Log search query
  condition_op    VARCHAR(10) NOT NULL, -- '>', '<', '>=', '<=', '=='
  condition_val   FLOAT NOT NULL,      -- Threshold value
  condition_metric VARCHAR(50) NOT NULL, -- 'count', 'avg', 'p99'
  window_seconds  INT NOT NULL,        -- Evaluation window
  cooldown_seconds INT NOT NULL DEFAULT 300,
  channels        TEXT[] NOT NULL,     -- Notification channels
  is_active       BOOLEAN NOT NULL DEFAULT TRUE,
  created_by      UUID NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Alert incidents
CREATE TABLE alert_incidents (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  rule_id         UUID NOT NULL REFERENCES alert_rules(id),
  status          VARCHAR(20) NOT NULL DEFAULT 'firing',
  triggered_value FLOAT NOT NULL,
  threshold_value FLOAT NOT NULL,
  triggered_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  resolved_at     TIMESTAMPTZ,
  acknowledged_by UUID,
  acknowledged_at TIMESTAMPTZ
);

CREATE INDEX idx_alert_incidents_rule ON alert_incidents(rule_id, triggered_at DESC);
CREATE INDEX idx_alert_incidents_status ON alert_incidents(status)
  WHERE status = 'firing';

-- Dashboard configurations
CREATE TABLE dashboards (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            VARCHAR(255) NOT NULL,
  description     TEXT,
  panels          JSONB NOT NULL DEFAULT '[]',
  owner_id        UUID NOT NULL,
  is_public       BOOLEAN NOT NULL DEFAULT FALSE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Data source connections
CREATE TABLE data_sources (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            VARCHAR(255) NOT NULL,
  type            VARCHAR(50) NOT NULL, -- 'elasticsearch', 'prometheus', 'clickhouse'
  connection_url  VARCHAR(1024) NOT NULL,
  credentials     JSONB,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Retention policies
CREATE TABLE retention_policies (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  index_pattern   VARCHAR(255) NOT NULL, -- 'logs-*', 'events-*'
  hot_days        INT NOT NULL DEFAULT 7,
  warm_days       INT NOT NULL DEFAULT 30,
  cold_days       INT NOT NULL DEFAULT 365,
  delete_after    INT DEFAULT NULL,   -- NULL = keep forever
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

Scaling Strategies

  • Horizontal ingestion: Stateless ingestion servers behind a load balancer; scale by adding instances.
  • Kafka as the buffer: Absorbs burst traffic. If processors slow down, Kafka retains data for reprocessing.
  • Index lifecycle management (ILM): Automatically roll indices daily, shrink, force-merge, then move to warm/cold nodes.
  • Sampling: For extreme volumes, sample logs at 10% for indexing but keep 100% in cold storage (S3). Preserve 100% of ERROR and above.
  • Column-oriented storage: Use ClickHouse or Parquet files for analytical queries on historical data — 10-100x faster than Elasticsearch for aggregations.

Trade-offs

Decision Pros Cons
Kafka as central bus Decoupling, replay, buffering Operational complexity, cost
Elasticsearch for hot logs Fast full-text search, Kibana Expensive at scale, GC issues
ClickHouse for analytics Blazing aggregation speed Limited full-text search
Agent-based collection Reliable, buffered, compressed Must deploy/manage agents on every host
Sampling high-volume logs Dramatically reduces cost May miss rare but important events
Tiered storage Cost-effective retention Query across tiers is complex

Summary

A logging and analytics pipeline is a multi-stage system: collection (agents), ingestion (API + validation), transport (Kafka), processing (stream + batch), storage (hot/warm/cold tiers), and query (search + aggregation + alerting). The key design principles are: never lose data at ingestion (buffer aggressively), use tiered storage to manage costs, separate real-time from batch workloads, and make the ingestion path as simple and fast as possible while deferring complex processing to downstream consumers.

Exercises

  1. Build a Log Ingestion Service

    Create a Node.js Express service that accepts JSON log entries via HTTP POST, validates the schema, enriches them with a server timestamp and request metadata, and writes them to a local file in JSONL format (one JSON object per line). Implement batching — accumulate up to 100 logs or 1 second before flushing to disk.

    Hint

    Use a buffer array and setInterval for time-based flushing. Use fs.appendFile in write mode. Implement a flush function that JSON-stringifies each log on a separate line and writes the batch atomically.

  2. Implement a Real-Time Log Tail

    Build a WebSocket endpoint that streams new log entries to connected clients in real time. Clients should be able to subscribe with filters (e.g., service=UserService AND level=ERROR). Only matching logs should be sent to each client.

    Hint

    Use the 'ws' npm package. Store connected clients with their filter criteria. When a new log is ingested, evaluate it against all client filters and send to matching clients. Parse filter expressions into a function.

  3. Build a Simple Alert Engine

    Create an alerting system that evaluates rules against a sliding window of logs. Support rules like 'if ERROR count > 10 in 5 minutes, alert'. Implement cooldown so the same alert does not fire repeatedly. Send alerts to a console output or a webhook.

    Hint

    Use a circular buffer or array of timestamped events per rule. Every 30 seconds, evaluate each rule against the window. Track last alert time per rule for cooldown. Use node-fetch to call webhooks.

Common Problems & Solutions

1 Log ingestion service becomes a bottleneck during traffic spikes, causing log loss
Symptoms: HTTP 503 errors from the ingestion API during traffic spikes. Missing log entries for peak traffic periods. Agent retry queues growing.
Cause: The ingestion service writes synchronously to downstream systems (Elasticsearch directly) and cannot absorb sudden traffic increases. Backpressure from slow consumers causes the ingestion API to timeout.
Solution: Always put a message queue (Kafka) between ingestion and storage. The ingestion service should only validate and enqueue — never write directly to the storage layer. Implement local disk buffering in agents as a fallback. Use acks=1 on Kafka for speed.
2 Elasticsearch cluster runs out of disk space and becomes read-only
Symptoms: Elasticsearch returns 403 Forbidden with 'cluster_block_exception' on write requests. No new logs appear in Kibana.
Cause: Index lifecycle management is not configured. Old indices are not deleted or moved to cheaper storage. The cluster hits the low-watermark (85% disk) and stops accepting writes.
Solution: Implement ILM policies that rollover indices daily, force-merge after 1 day, move to warm after 7 days, delete after 30 days. Monitor disk usage with alerts at 70% and 80%. Use curator or ILM to automate index management.
3 Dashboard queries for 30-day aggregations take over 60 seconds
Cause: Elasticsearch is querying across hundreds of daily indices with billions of documents. Aggregation queries require scanning all matching documents.
Solution: Pre-compute rollup aggregations (hourly/daily summaries) and store them in a dedicated analytics store (ClickHouse or TimescaleDB). Dashboard queries hit the rollup tables instead of raw data. For common queries, cache results in Redis with a 5-minute TTL.
