JSONB: Storing and Querying Semi-Structured Data

Intuition

JSONB is like having a filing cabinet where most drawers have neatly labeled folders (regular columns), but one special drawer accepts anything — notes, photos, receipts — in any format (JSONB). You can still search through it, but it's best for things that don't fit the standard folder structure.

JSONB: Storing and Querying Semi-Structured Data

Introduction

PostgreSQL's jsonb type gives you the flexibility of a document database (like MongoDB) while keeping all the power of a relational database — transactions, JOINs, indexes, and SQL. It's the sweet spot when you need structured tables for core data and flexible documents for metadata, settings, or API payloads.

JSON vs JSONB

json:   Stores exact text, preserves whitespace and key order
        Slower queries (must parse on every access)

jsonb:  Binary format, decomposes into key/value pairs
        Faster queries (pre-parsed)
        Supports indexes
        Does NOT preserve key order or duplicate keys

Always use jsonb unless you need to preserve exact formatting.

Creating Tables with JSONB

CREATE TABLE events (
  id serial PRIMARY KEY,
  type text NOT NULL,
  data jsonb NOT NULL DEFAULT '{}',
  metadata jsonb,
  created_at timestamptz DEFAULT NOW()
);

CREATE TABLE user_preferences (
  user_id integer PRIMARY KEY REFERENCES users(id),
  settings jsonb NOT NULL DEFAULT '{}'
);

Inserting JSONB Data

// From Node.js, pass objects directly — pg handles serialization
await pool.query(`
  INSERT INTO events (type, data, metadata)
  VALUES ($1, $2, $3)
`, [
  'page_view',
  { url: '/products/42', referrer: 'google.com', device: 'mobile' },
  { ip: '192.168.1.1', user_agent: 'Mozilla/5.0' }
]);

// Nested objects work too
await pool.query(`
  INSERT INTO user_preferences (user_id, settings)
  VALUES ($1, $2)
`, [1, {
  theme: 'dark',
  notifications: { email: true, push: false, sms: false },
  language: 'en',
  timezone: 'America/New_York'
}]);

Querying JSONB

Extracting Values

-- -> returns jsonb, ->> returns text
SELECT
  data -> 'url' AS url_json,       -- "/products/42" (jsonb, with quotes)
  data ->> 'url' AS url_text,      -- /products/42 (text, no quotes)
  data -> 'nested' -> 'key' AS deep_json,
  data -> 'nested' ->> 'key' AS deep_text
FROM events;

-- #> and #>> for path-based access
SELECT
  settings #> '{notifications, email}' AS email_json,
  settings #>> '{notifications, email}' AS email_text
FROM user_preferences;

async function getUserSettings(userId) {
  const result = await pool.query(`
    SELECT
      settings ->> 'theme' AS theme,
      settings ->> 'language' AS language,
      settings #>> '{notifications, email}' AS email_notifications
    FROM user_preferences
    WHERE user_id = $1
  `, [userId]);
  return result.rows[0];
}

Filtering with JSONB

-- Containment operator @> (most common, indexable)
SELECT * FROM events WHERE data @> '{"device": "mobile"}';

-- Existence operators
SELECT * FROM events WHERE data ? 'referrer';        -- key exists
SELECT * FROM events WHERE data ?| ARRAY['a', 'b'];  -- any key exists
SELECT * FROM events WHERE data ?& ARRAY['a', 'b'];  -- all keys exist

-- Comparison on extracted values
SELECT * FROM events WHERE data ->> 'url' LIKE '/products%';
SELECT * FROM events
WHERE (data ->> 'score')::int > 90;

Aggregating JSONB

-- Build JSON from rows
SELECT json_agg(row_to_json(u)) AS users_json
FROM (SELECT id, name, email FROM users LIMIT 10) u;

-- Build a JSON object
SELECT jsonb_object_agg(key, value)
FROM user_preferences, jsonb_each(settings);

-- Expand JSONB arrays
SELECT id, jsonb_array_elements(data -> 'tags') AS tag
FROM posts;

-- Expand JSONB objects
SELECT id, key, value
FROM events, jsonb_each(data) AS kv(key, value);

Updating JSONB

-- Set a top-level key
UPDATE user_preferences
SET settings = jsonb_set(settings, '{theme}', '"light"')
WHERE user_id = 1;

-- Set a nested key
UPDATE user_preferences
SET settings = jsonb_set(settings, '{notifications, push}', 'true')
WHERE user_id = 1;

-- Remove a key
UPDATE user_preferences
SET settings = settings - 'deprecated_field'
WHERE user_id = 1;

-- Merge (shallow) with ||
UPDATE user_preferences
SET settings = settings || '{"language": "fr", "new_field": true}'
WHERE user_id = 1;

async function updateUserSettings(userId, updates) {
  // Merge new settings into existing
  const result = await pool.query(`
    UPDATE user_preferences
    SET settings = settings || $2::jsonb
    WHERE user_id = $1
    RETURNING settings
  `, [userId, JSON.stringify(updates)]);
  return result.rows[0]?.settings;
}

await updateUserSettings(1, { theme: 'dark', language: 'es' });

Indexing JSONB

-- GIN index: supports @>, ?, ?|, ?& operators
CREATE INDEX idx_events_data ON events USING gin (data);

-- GIN with jsonb_path_ops: smaller, only supports @>
CREATE INDEX idx_events_data_path ON events USING gin (data jsonb_path_ops);

-- B-tree index on specific extracted value
CREATE INDEX idx_events_url ON events ((data ->> 'url'));

-- Composite: type + extracted jsonb field
CREATE INDEX idx_events_type_device ON events (type, (data ->> 'device'));

When to Use JSONB vs Columns

Use regular columns when:
- Data has a consistent, known structure
- You need constraints (NOT NULL, CHECK, FK)
- You query/filter on the field frequently
- The field is used in JOINs

Use JSONB when:
- Structure varies between rows
- Schema changes frequently
- You're storing third-party API responses
- You need key-value settings/preferences
- The data is read as a whole blob

Real-World Pattern: Audit Log with JSONB

async function logAuditEvent(userId, action, details) {
  await pool.query(`
    INSERT INTO audit_log (user_id, action, details, created_at)
    VALUES ($1, $2, $3, NOW())
  `, [userId, action, details]);
}

// Usage
await logAuditEvent(1, 'user.update', {
  field: 'email',
  old_value: 'old@example.com',
  new_value: 'new@example.com',
  ip_address: '192.168.1.1'
});

// Query audit log
async function getAuditLog(filters) {
  const result = await pool.query(`
    SELECT
      al.action,
      al.details,
      al.created_at,
      u.name AS user_name
    FROM audit_log al
    JOIN users u ON al.user_id = u.id
    WHERE al.action = $1
      AND al.details @> $2
    ORDER BY al.created_at DESC
    LIMIT 50
  `, [filters.action, JSON.stringify({ field: filters.field })]);
  return result.rows;
}

Common Problems & Solutions

1 JSONB queries are slow despite having a GIN index
Cause: Using ->> operator in WHERE clause instead of @> containment. Expression-based queries (data ->> 'key' = 'value') don't use GIN indexes.
Solution: Rewrite using containment: `WHERE data @> '{"key": "value"}'` which uses the GIN index. Or create a B-tree expression index on the specific path.

-- Slow (no GIN index use):\nSELECT * FROM events WHERE data ->> 'type' = 'click';\n-- Fast (uses GIN index):\nSELECT * FROM events WHERE data @> '{"type": "click"}';

2 Cannot compare JSONB extracted numbers directly
Cause: The ->> operator returns text, not numbers. `data ->> 'score' > 90` compares strings, not numbers.
Solution: Cast the extracted value: `(data ->> 'score')::int > 90`. Or use the -> operator which returns jsonb and compare with a jsonb literal.
3 jsonb_set doesn't create intermediate objects for nested paths
Cause: jsonb_set requires the parent path to exist. Setting '{a, b, c}' fails if 'a.b' doesn't exist.
Solution: Use `jsonb_set` with `create_if_missing` parameter (true by default in PG 16+). For older versions, build the path incrementally or use the || merge operator with the full nested structure.
