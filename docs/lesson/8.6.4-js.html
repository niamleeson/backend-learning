Saga Pattern and Distributed Transactions

Intuition

A saga is like planning a wedding involving multiple vendors. The choreography approach is like each vendor knowing what to do when they hear the previous vendor finished (florist delivers after venue confirms). The orchestration approach is like having a wedding planner who calls each vendor in order and handles cancellations if something goes wrong. If the caterer cancels last-minute, the planner cancels the venue and florist (compensating actions).

Saga Pattern and Distributed Transactions

Introduction

In a monolith, a single database transaction guarantees that all operations succeed or all fail (ACID). In microservices, each service has its own database, making cross-service transactions impossible with traditional ACID. The Saga pattern manages distributed transactions by coordinating a sequence of local transactions across services, with compensating actions for rollback.

The Problem

Monolith Transaction (ACID):
  BEGIN TRANSACTION
    deduct_inventory()
    charge_payment()
    create_shipment()
  COMMIT  ← All or nothing

Microservices (No distributed transaction):
  Inventory Service: deduct_inventory()  ← SUCCESS
  Payment Service: charge_payment()      ← FAILS!
  Shipping Service: never called

  Problem: inventory was deducted but payment failed
  Result: inconsistent state across services!

Saga Types

Choreography-Based Saga

Each service publishes events and reacts to events from other services. No central coordinator.

Order Created
    │
    ▼
┌─────────────┐  inventory.reserved  ┌─────────────┐
│ Inventory   │─────────────────────>│ Payment     │
│ Service     │                      │ Service     │
└─────────────┘                      └──────┬──────┘
                                            │
                              payment.charged │
                                            ▼
                                     ┌─────────────┐
                                     │ Shipping    │
                                     │ Service     │
                                     └─────────────┘

Failure & Compensation:
Payment fails → publishes payment.failed
Inventory Service listens → releases reserved inventory
Order Service listens → marks order as failed

// Choreography implementation with events

// Order Service
class OrderService {
  async createOrder(data) {
    const order = await this.orderRepo.create({
      ...data,
      status: 'PENDING',
    });

    // Publish event to start saga
    await this.eventBus.publish('order.created', {
      orderId: order.id,
      items: data.items,
      userId: data.userId,
      total: order.total,
    });

    return order;
  }

  // Listen for saga completion/failure events
  @EventPattern('payment.charged')
  async handlePaymentCharged(data) {
    await this.orderRepo.update(data.orderId, { status: 'PAID' });
  }

  @EventPattern('payment.failed')
  async handlePaymentFailed(data) {
    await this.orderRepo.update(data.orderId, {
      status: 'PAYMENT_FAILED',
      failureReason: data.reason,
    });
  }

  @EventPattern('shipping.created')
  async handleShippingCreated(data) {
    await this.orderRepo.update(data.orderId, { status: 'SHIPPED' });
  }
}

// Inventory Service
class InventoryService {
  @EventPattern('order.created')
  async handleOrderCreated(data) {
    try {
      await this.inventoryRepo.reserve(data.items);
      await this.eventBus.publish('inventory.reserved', {
        orderId: data.orderId,
        items: data.items,
      });
    } catch (err) {
      await this.eventBus.publish('inventory.failed', {
        orderId: data.orderId,
        reason: err.message,
      });
    }
  }

  // Compensating action
  @EventPattern('payment.failed')
  async handlePaymentFailed(data) {
    await this.inventoryRepo.release(data.items);
    console.log(`Released inventory for order ${data.orderId}`);
  }
}

// Payment Service
class PaymentService {
  @EventPattern('inventory.reserved')
  async handleInventoryReserved(data) {
    try {
      const charge = await this.paymentGateway.charge(data.total);
      await this.eventBus.publish('payment.charged', {
        orderId: data.orderId,
        chargeId: charge.id,
      });
    } catch (err) {
      await this.eventBus.publish('payment.failed', {
        orderId: data.orderId,
        items: data.items,
        reason: err.message,
      });
    }
  }
}

Orchestration-Based Saga

A central Saga Orchestrator coordinates the transaction steps.

                  ┌────────────────────┐
                  │  Saga Orchestrator │
                  └────────┬───────────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
         1. Reserve   2. Charge    3. Ship
              │            │            │
              ▼            ▼            ▼
         ┌────────┐  ┌────────┐  ┌────────┐
         │Inventory│  │Payment │  │Shipping│
         └────────┘  └────────┘  └────────┘

On failure at step 2:
  Orchestrator calls compensate on step 1 (release inventory)

class OrderSagaOrchestrator {
  constructor(inventoryClient, paymentClient, shippingClient, orderRepo) {
    this.steps = [
      {
        name: 'reserve_inventory',
        execute: (ctx) => inventoryClient.reserve(ctx.items),
        compensate: (ctx) => inventoryClient.release(ctx.items),
      },
      {
        name: 'charge_payment',
        execute: (ctx) => paymentClient.charge(ctx.userId, ctx.total),
        compensate: (ctx) => paymentClient.refund(ctx.chargeId),
      },
      {
        name: 'create_shipment',
        execute: (ctx) => shippingClient.createShipment(ctx.orderId, ctx.items),
        compensate: (ctx) => shippingClient.cancelShipment(ctx.shipmentId),
      },
    ];
    this.orderRepo = orderRepo;
  }

  async execute(orderData) {
    const context = { ...orderData };
    const completedSteps = [];

    try {
      for (const step of this.steps) {
        console.log(`Executing step: ${step.name}`);
        const result = await step.execute(context);

        // Merge result into context for subsequent steps
        Object.assign(context, result);
        completedSteps.push(step);

        // Update order status
        await this.orderRepo.update(context.orderId, {
          sagaStep: step.name,
          status: 'PROCESSING',
        });
      }

      // All steps succeeded
      await this.orderRepo.update(context.orderId, { status: 'COMPLETED' });
      return { success: true, context };

    } catch (err) {
      console.error(`Saga failed at step: ${completedSteps.length}`, err);

      // Compensate in reverse order
      for (const step of completedSteps.reverse()) {
        try {
          console.log(`Compensating: ${step.name}`);
          await step.compensate(context);
        } catch (compErr) {
          // Log compensation failure — may need manual intervention
          console.error(`Compensation failed for ${step.name}:`, compErr);
          await this.alertOps(step.name, compErr);
        }
      }

      await this.orderRepo.update(context.orderId, {
        status: 'FAILED',
        failureReason: err.message,
      });

      return { success: false, error: err.message };
    }
  }
}

// Usage with BullMQ for reliability
const sagaQueue = new Queue('order-sagas', { connection });

// API creates the saga job
app.post('/api/orders', async (req, res) => {
  const order = await orderRepo.create({ ...req.body, status: 'PENDING' });
  await sagaQueue.add('execute-order-saga', {
    orderId: order.id,
    ...req.body,
  });
  res.status(202).json({ orderId: order.id, status: 'PENDING' });
});

// Worker processes sagas
const sagaWorker = new Worker('order-sagas', async (job) => {
  const saga = new OrderSagaOrchestrator(
    inventoryClient, paymentClient, shippingClient, orderRepo
  );
  return saga.execute(job.data);
}, {
  connection,
  concurrency: 5,
  attempts: 1, // Saga handles its own retries
});

Saga State Machine

// Persistent saga state for recovery
class SagaStateMachine {
  constructor(redis) {
    this.redis = redis;
  }

  async startSaga(sagaId, type, data) {
    const saga = {
      id: sagaId,
      type,
      data,
      status: 'STARTED',
      currentStep: 0,
      completedSteps: [],
      createdAt: Date.now(),
    };
    await this.redis.set(`saga:${sagaId}`, JSON.stringify(saga), 'EX', 86400);
    return saga;
  }

  async advanceStep(sagaId, stepName, result) {
    const saga = JSON.parse(await this.redis.get(`saga:${sagaId}`));
    saga.currentStep++;
    saga.completedSteps.push({ name: stepName, result, completedAt: Date.now() });
    await this.redis.set(`saga:${sagaId}`, JSON.stringify(saga), 'EX', 86400);
    return saga;
  }

  async failSaga(sagaId, error) {
    const saga = JSON.parse(await this.redis.get(`saga:${sagaId}`));
    saga.status = 'COMPENSATING';
    saga.error = error;
    await this.redis.set(`saga:${sagaId}`, JSON.stringify(saga), 'EX', 86400);
    return saga;
  }

  async completeSaga(sagaId) {
    const saga = JSON.parse(await this.redis.get(`saga:${sagaId}`));
    saga.status = 'COMPLETED';
    saga.completedAt = Date.now();
    await this.redis.set(`saga:${sagaId}`, JSON.stringify(saga), 'EX', 86400);
    return saga;
  }
}

Choreography vs Orchestration

┌────────────────┬──────────────────┬──────────────────┐
│                │ Choreography     │ Orchestration    │
├────────────────┼──────────────────┼──────────────────┤
│ Coupling       │ Loose            │ Centralized      │
│ Complexity     │ Distributed      │ Centralized      │
│ Debugging      │ Hard to trace    │ Easy to trace    │
│ Single failure │ No               │ Orchestrator     │
│   point        │                  │                  │
│ Best for       │ Simple sagas     │ Complex sagas    │
│                │ (2-3 steps)      │ (4+ steps)       │
│ Testing        │ Integration only │ Unit testable    │
└────────────────┴──────────────────┴──────────────────┘

Common Pitfalls

  1. Not making compensations idempotent: A compensation may run multiple times. Make it safe to repeat (use idempotency keys).
  2. Compensation that can also fail: If "refund payment" fails during compensation, you need manual intervention. Always log and alert on compensation failures.
  3. Missing saga timeout: A saga stuck mid-execution wastes resources and leaves data inconsistent. Set timeouts and auto-compensate.
  4. Using sagas where you don't need them: If two operations can be in the same database, use a regular transaction. Sagas are for cross-service only.

Summary

The Saga pattern manages distributed transactions across microservices through a sequence of local transactions with compensating actions for rollback. Choreography uses events and is simpler but harder to debug. Orchestration uses a central coordinator and is easier to manage for complex flows. Always make saga steps and compensations idempotent, persist saga state for recovery, and handle compensation failures with alerts.

Exercises

  1. Orchestrated Order Saga

    Implement a saga orchestrator for order processing: reserve inventory, charge payment, create shipment. Include compensating actions that run in reverse on failure.

    Hint

    Define steps as objects with execute and compensate functions. Track completed steps. On failure, compensate in reverse order.

  2. Choreography-Based Saga

    Build 3 NestJS microservices that communicate via events for an order saga. Each service publishes success/failure events that other services react to.

    Hint

    Use Redis or RabbitMQ transport. Each service has @EventPattern handlers for both success and failure events.

  3. Saga Recovery

    Implement persistent saga state in Redis. Build a recovery worker that detects stalled sagas (no progress for 5 minutes) and automatically triggers compensation.

    Hint

    Store saga state with timestamps. A scheduled job checks for sagas stuck in non-terminal states past timeout.

Common Problems & Solutions

1 Saga leaves data in inconsistent state after partial failure
Cause: Compensation action failed (e.g., refund API returned error), leaving the saga incomplete.
Solution: Retry compensating actions with exponential backoff. Log all compensation failures. Have a manual intervention process for unrecoverable cases. Send alerts to operations team.
2 Events processed out of order in choreography saga
Cause: Message broker doesn't guarantee ordering, so 'payment.charged' arrives before 'inventory.reserved'.
Solution: Each step should verify prerequisites before executing. Store saga state and only proceed when all required prior events have been received.
