Design a Real-Time Collaborative Editor (Google Docs)

Intuition

Real-time collaboration is a distributed consensus problem disguised as a text editing feature: every client has a local copy and the system must ensure all copies converge to the same state despite concurrent edits arriving in different orders. OT achieves this by transforming operations against each other; CRDTs achieve it by making operations inherently commutative.

Design a Real-Time Collaborative Editor (Google Docs)

Requirements

  • Multiple users edit the same document simultaneously with <100ms perceived latency
  • All users converge to the same document state regardless of edit order
  • Support rich text formatting, comments, cursor presence, and revision history
  • Handle offline edits that sync when connectivity is restored

Architecture

Clients apply edits locally for instant feedback, then send operations to a collaboration server. The server uses Operational Transformation (OT) or a CRDT to merge concurrent edits deterministically. Transformed operations are broadcast to all other clients. A persistence layer stores the document and its full operation history.

Key Components

  • Client Editor: Applies local edits optimistically; queues operations for server sync
  • OT/CRDT Engine: Transforms concurrent operations to preserve intent (OT) or uses conflict-free merge (CRDT)
  • Collaboration Server: Serializes operations, assigns global order, broadcasts to clients
  • Document Store: Persists document snapshots and operation logs for history/undo

Scaling

Partition by document: each document is handled by one server (sticky sessions). Use sharding across millions of documents. Take periodic snapshots to truncate the operation log.

Trade-offs

  • OT is well-proven (Google Docs) but requires a central server for ordering; CRDTs enable P2P but are more complex and memory-heavy
  • Storing every operation enables full revision history but grows storage linearly
  • Optimistic local apply provides instant UX but requires complex conflict resolution

Exercises

  1. Implement Basic Operational Transformation

    Build an OT engine for a plain-text document supporting insert(pos, char) and delete(pos) operations. Write a transform function that correctly handles two concurrent inserts at different positions.

    Hint

    For two inserts: if op1.pos <= op2.pos, shift op2.pos right by 1 (and vice versa). For insert vs delete: if insert.pos <= delete.pos, shift delete.pos right by 1.

  2. Build a Collaborative Editing Server

    Create a WebSocket server that accepts operations from multiple clients, applies OT to resolve conflicts, and broadcasts the transformed operations to all connected clients.

    Hint

    Maintain a server-side operation log. When a client sends an op based on revision N, transform it against all operations after N in the log before applying and broadcasting.

Common Problems & Solutions

1 Documents diverge after a rapid sequence of concurrent edits from 3+ users
Cause: The OT transform function does not handle the composition of three-way concurrent transforms correctly
Solution: Ensure the server imposes a total order on operations: each client must acknowledge the server's version before sending the next op, and transform against the full server history.
2 Operation log grows unboundedly for long-lived documents, slowing down new client joins
Cause: New clients must replay the entire operation history to reconstruct the document
Solution: Periodically snapshot the document state and truncate the log. New clients load the latest snapshot and only replay operations after it.
