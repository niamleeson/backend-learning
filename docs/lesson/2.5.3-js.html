Relations: One-to-Many, Many-to-Many, and Nested Writes

Intuition

Nested writes in Prisma are like filling out a form that automatically creates entries in related filing cabinets. When you create a user with orders and order items, it is like submitting a single form that the system breaks down and files in three different cabinets, all atomically. If any filing fails, all forms are returned and nothing is stored.

Relations: One-to-Many, Many-to-Many, and Nested Writes

Introduction

Prisma's relation handling is one of its most powerful features. Nested writes let you create, update, or delete related records in a single operation, all wrapped in an automatic transaction. This eliminates the need for manual transaction management in most CRUD scenarios.

One-to-Many: Creating Related Records

// Create user with orders in one operation
const user = await prisma.user.create({
  data: {
    email: 'alice@example.com',
    name: 'Alice',
    passwordHash: 'hashed...',
    orders: {
      create: [
        {
          total: new Prisma.Decimal('49.99'),
          status: 'DELIVERED',
          items: {
            create: [
              { productId: 'prod-1', quantity: 2, unitPrice: new Prisma.Decimal('9.99') },
              { productId: 'prod-2', quantity: 1, unitPrice: new Prisma.Decimal('30.01') },
            ],
          },
        },
      ],
    },
  },
  include: {
    orders: { include: { items: true } },
  },
});
// Creates user + order + 2 order items in a single transaction!

Nested Write Operations

// All nested write operations:
await prisma.user.update({
  where: { id: userId },
  data: {
    orders: {
      create: { /* ... */ },          // Create new related records
      createMany: { data: [/* ... */] }, // Create many at once
      connect: { id: orderId },        // Link existing record
      connectOrCreate: {               // Link or create
        where: { id: orderId },
        create: { /* ... */ },
      },
      disconnect: { id: orderId },     // Unlink (set FK to null)
      set: [{ id: id1 }, { id: id2 }], // Replace all connections
      update: {                        // Update related record
        where: { id: orderId },
        data: { status: 'SHIPPED' },
      },
      updateMany: {                    // Update many related
        where: { status: 'PENDING' },
        data: { status: 'CANCELLED' },
      },
      delete: { id: orderId },         // Delete related record
      deleteMany: { status: 'CANCELLED' }, // Delete many related
    },
  },
});

Many-to-Many Operations

// Create post with tags (implicit M:N)
const post = await prisma.post.create({
  data: {
    title: 'Getting Started with Prisma',
    body: 'Content here...',
    tags: {
      connectOrCreate: [
        { where: { name: 'prisma' }, create: { name: 'prisma' } },
        { where: { name: 'nodejs' }, create: { name: 'nodejs' } },
        { where: { name: 'database' }, create: { name: 'database' } },
      ],
    },
  },
  include: { tags: true },
});

// Add tags to existing post
await prisma.post.update({
  where: { id: postId },
  data: {
    tags: {
      connect: [{ name: 'typescript' }],      // add existing tag
      disconnect: [{ name: 'database' }],      // remove tag
    },
  },
});

// Replace all tags
await prisma.post.update({
  where: { id: postId },
  data: {
    tags: {
      set: [{ name: 'prisma' }, { name: 'typescript' }], // replace all
    },
  },
});

Explicit M:N with Extra Fields

// Enrollment with grade (explicit junction)
const enrollment = await prisma.enrollment.create({
  data: {
    student: { connect: { id: studentId } },
    course: { connect: { id: courseId } },
    enrolledAt: new Date(),
  },
});

// Update grade
await prisma.enrollment.update({
  where: {
    studentId_courseId: { studentId, courseId },
  },
  data: { grade: 'A' },
});

// Find student's courses with grades
const studentCourses = await prisma.enrollment.findMany({
  where: { studentId },
  include: {
    course: { select: { title: true, credits: true } },
  },
  orderBy: { enrolledAt: 'desc' },
});

Querying Relations

// Filter by related records
const highSpenders = await prisma.user.findMany({
  where: {
    orders: {
      some: {
        total: { gt: new Prisma.Decimal('500') },
        status: 'DELIVERED',
      },
    },
  },
});

// Users with NO orders
const inactiveUsers = await prisma.user.findMany({
  where: { orders: { none: {} } },
});

// Users where ALL orders are delivered
const loyalCustomers = await prisma.user.findMany({
  where: { orders: { every: { status: 'DELIVERED' } } },
});

// Count related records
const usersWithCounts = await prisma.user.findMany({
  select: {
    id: true,
    name: true,
    _count: {
      select: { orders: true, reviews: true },
    },
  },
  orderBy: {
    orders: { _count: 'desc' },
  },
});
// [{ id: '...', name: 'Alice', _count: { orders: 15, reviews: 8 } }, ...]

Transactions

// Interactive transaction (for complex logic)
const order = await prisma.$transaction(async (tx) => {
  // Check stock
  const product = await tx.product.findUniqueOrThrow({
    where: { id: productId },
  });

  if (product.stock < quantity) {
    throw new Error('Insufficient stock');
  }

  // Decrement stock
  await tx.product.update({
    where: { id: productId },
    data: { stock: { decrement: quantity } },
  });

  // Create order
  return tx.order.create({
    data: {
      userId,
      total: new Prisma.Decimal(product.price.toNumber() * quantity),
      items: {
        create: { productId, quantity, unitPrice: product.price },
      },
    },
    include: { items: true },
  });
});

// Sequential transaction (batch operations)
const [updatedUser, newOrder] = await prisma.$transaction([
  prisma.user.update({ where: { id: userId }, data: { lastOrderAt: new Date() } }),
  prisma.order.create({ data: { userId, total: 99.99 } }),
]);

Real-World Pattern: Repository with Prisma

class UserRepository {
  constructor(private prisma: PrismaClient) {}

  async findById(id: string) {
    return this.prisma.user.findUnique({
      where: { id },
      select: {
        id: true, name: true, email: true, role: true,
        _count: { select: { orders: true } },
      },
    });
  }

  async findWithOrders(id: string, orderLimit = 10) {
    return this.prisma.user.findUnique({
      where: { id },
      include: {
        orders: {
          take: orderLimit,
          orderBy: { createdAt: 'desc' },
          include: { items: { include: { product: true } } },
        },
      },
    });
  }

  async createWithAddress(data: CreateUserInput) {
    return this.prisma.user.create({
      data: {
        email: data.email,
        name: data.name,
        passwordHash: data.passwordHash,
        addresses: data.address ? {
          create: data.address,
        } : undefined,
      },
      include: { addresses: true },
    });
  }
}

Exercises

    Hint

    Use prisma.$transaction for the multi-step logic. Nested writes handle the order + items creation. Separate update calls for stock decrement.

  2. Tag Management System

    Build functions for: addTags(postId, tagNames[]) that creates tags if they don't exist and connects them, removeTags(postId, tagNames[]), replaceTags(postId, tagNames[]), and getPostsByTag(tagName). Use connectOrCreate and set.

    Hint

    Use connectOrCreate for adding tags. Use set: [] for replacing all tags. Use disconnect for removing specific tags.

  3. Nested Query Explorer

    Write 10 different queries that use relation filters (some, every, none), nested includes, relation counts (_count), and ordering by relation count. Document what SQL each generates.

    Hint

    Enable query logging: new PrismaClient({ log: ['query'] }). Each Prisma operation generates one or more SQL queries.

Common Problems & Solutions

1 Nested create fails with foreign key constraint error
Cause: Creating a related record that references a non-existent parent. Or the relation fields in the schema don't match the actual database foreign keys.
Solution: Use `connect` to link existing records, `connectOrCreate` to link-or-create. Verify schema matches DB with `npx prisma db pull`.
2 Nested writes generate too many queries
Cause: Prisma uses separate queries for relations, not JOINs. A deeply nested include with many relations can generate dozens of queries.
Solution: Use `select` to limit fetched fields. For performance-critical paths, use `prisma.$queryRaw` with manual JOINs. Consider denormalization for read-heavy paths.
3 disconnect doesn't work on required relations
Cause: You can't set a required (non-nullable) foreign key to null, which is what disconnect does.
Solution: For required relations, use `delete` instead of `disconnect`. Or make the relation optional in the schema if disconnection should be allowed.
