Pagination Patterns: Offset, Cursor, Keyset

Intuition

Offset pagination is like saying 'skip the first 100 people in line and give me the next 20.' If people join or leave the line, you will count wrong. Cursor pagination is like saying 'start from the person wearing the red hat and give me the next 20' -- no matter how the line changes, you always start from the right spot.

Pagination Patterns: Offset, Cursor, Keyset

Pagination is how you break large datasets into manageable chunks. Choosing the wrong pagination strategy can lead to duplicate results, missed records, or terrible performance at scale.

Why Pagination Matters

Returning all records in a single response is a non-starter for production APIs:

  • Memory: Serializing 100,000 records can exhaust server memory
  • Bandwidth: Clients on mobile cannot handle multi-megabyte responses
  • Database: SELECT * FROM users with a million rows will time out
  • UX: Users do not need all data at once

1. Offset-Based Pagination

The simplest and most common approach. Uses page and limit (or offset and limit).

GET /api/posts?page=3&limit=20
-- translates to: SELECT * FROM posts ORDER BY id LIMIT 20 OFFSET 40

// Express implementation
app.get('/api/posts', async (req, res) => {
  const page = Math.max(1, parseInt(req.query.page as string) || 1);
  const limit = Math.min(100, Math.max(1, parseInt(req.query.limit as string) || 20));
  const offset = (page - 1) * limit;

  const [posts, total] = await Promise.all([
    db.query('SELECT * FROM posts ORDER BY created_at DESC LIMIT $1 OFFSET $2', [limit, offset]),
    db.query('SELECT COUNT(*) FROM posts'),
  ]);

  res.json({
    data: posts.rows,
    pagination: {
      page,
      limit,
      total: parseInt(total.rows[0].count),
      totalPages: Math.ceil(parseInt(total.rows[0].count) / limit),
    },
  });
});

With Prisma:

async function getPosts(page: number, limit: number) {
  const [posts, total] = await Promise.all([
    prisma.post.findMany({
      skip: (page - 1) * limit,
      take: limit,
      orderBy: { createdAt: 'desc' },
    }),
    prisma.post.count(),
  ]);

  return { posts, total, page, limit };
}

The Problem with Offset Pagination

Page 1: [A, B, C, D, E]   (OFFSET 0, LIMIT 5)
-- Someone inserts record Z at the top --
Page 2: [E, F, G, H, I]   (OFFSET 5, LIMIT 5)
                ^--- E appears on both pages!

Offset also has a performance problem:

-- For page 10000 with 20 per page:
SELECT * FROM posts ORDER BY id LIMIT 20 OFFSET 199980;
-- Database must scan and discard 199,980 rows!

2. Cursor-Based Pagination

Uses an opaque cursor (usually a base64-encoded identifier) to mark position.

GET /api/posts?limit=20
--> returns: { data: [...], cursors: { next: "eyJpZCI6NDJ9" } }

GET /api/posts?limit=20&after=eyJpZCI6NDJ9
--> returns next page

interface CursorPaginationResult<T> {
  data: T[];
  cursors: {
    next: string | null;
    previous: string | null;
    hasMore: boolean;
  };
}

function encodeCursor(data: Record<string, any>): string {
  return Buffer.from(JSON.stringify(data)).toString('base64url');
}

function decodeCursor(cursor: string): Record<string, any> {
  return JSON.parse(Buffer.from(cursor, 'base64url').toString('utf-8'));
}

app.get('/api/posts', async (req, res) => {
  const limit = Math.min(100, parseInt(req.query.limit as string) || 20);
  const afterCursor = req.query.after as string | undefined;

  let whereClause = '';
  const params: any[] = [limit + 1]; // fetch one extra to check hasMore

  if (afterCursor) {
    const { id } = decodeCursor(afterCursor);
    whereClause = 'WHERE id < $2';
    params.push(id);
  }

  const result = await db.query(
    `SELECT * FROM posts ${whereClause} ORDER BY id DESC LIMIT $1`,
    params
  );

  const hasMore = result.rows.length > limit;
  const posts = result.rows.slice(0, limit);

  res.json({
    data: posts,
    cursors: {
      next: hasMore ? encodeCursor({ id: posts[posts.length - 1].id }) : null,
      previous: afterCursor || null,
      hasMore,
    },
  });
});

With Prisma cursor pagination:

async function getPostsCursor(limit: number, cursor?: number) {
  const posts = await prisma.post.findMany({
    take: limit + 1,
    ...(cursor && {
      cursor: { id: cursor },
      skip: 1, // skip the cursor itself
    }),
    orderBy: { id: 'desc' },
  });

  const hasMore = posts.length > limit;
  const data = posts.slice(0, limit);

  return {
    data,
    nextCursor: hasMore ? data[data.length - 1].id : null,
  };
}

3. Keyset Pagination (Seek Method)

Similar to cursor, but the key values are passed directly (not encoded). Most performant for large datasets.

GET /api/posts?limit=20&created_before=2024-01-15T10:30:00Z&id_before=1234

app.get('/api/posts', async (req, res) => {
  const limit = Math.min(100, parseInt(req.query.limit as string) || 20);
  const createdBefore = req.query.created_before as string | undefined;
  const idBefore = req.query.id_before as string | undefined;

  let query = 'SELECT * FROM posts';
  const params: any[] = [limit + 1];

  if (createdBefore && idBefore) {
    // Composite keyset for deterministic ordering
    query += ' WHERE (created_at, id) < ($2, $3)';
    params.push(createdBefore, parseInt(idBefore));
  }

  query += ' ORDER BY created_at DESC, id DESC LIMIT $1';

  const result = await db.query(query, params);
  const hasMore = result.rows.length > limit;
  const posts = result.rows.slice(0, limit);
  const lastPost = posts[posts.length - 1];

  res.json({
    data: posts,
    next: hasMore
      ? {
          created_before: lastPost.created_at,
          id_before: lastPost.id,
        }
      : null,
  });
});

Keyset pagination requires a composite index for performance:

CREATE INDEX idx_posts_created_id ON posts (created_at DESC, id DESC);

Comparison

+------------------+-----------+-------------+------------+----------+
| Feature          |  Offset   |   Cursor    |   Keyset   |
+------------------+-----------+-------------+------------+
| Jump to page N   |   Yes     |    No       |    No      |
| Consistent view  |   No      |    Yes      |    Yes     |
| Performance      |  O(offset)|    O(1)*    |   O(1)*    |
| Handles inserts  |   Badly   |    Well     |   Well     |
| Complexity       |   Low     |   Medium    |   Medium   |
| Bidirectional    |   Yes     |    Yes      |   Harder   |
+------------------+-----------+-------------+------------+
* With proper index

Real-World Patterns

Relay-Style Cursor Pagination (GraphQL)

The Relay specification defines a standard cursor format used heavily in GraphQL:

interface Edge<T> {
  node: T;
  cursor: string;
}

interface PageInfo {
  hasNextPage: boolean;
  hasPreviousPage: boolean;
  startCursor: string | null;
  endCursor: string | null;
}

interface Connection<T> {
  edges: Edge<T>[];
  pageInfo: PageInfo;
  totalCount: number;
}

function toConnection<T extends { id: number }>(
  items: T[],
  hasMore: boolean,
  hasPrevious: boolean
): Connection<T> {
  const edges = items.map(item => ({
    node: item,
    cursor: encodeCursor({ id: item.id }),
  }));

  return {
    edges,
    pageInfo: {
      hasNextPage: hasMore,
      hasPreviousPage: hasPrevious,
      startCursor: edges.length > 0 ? edges[0].cursor : null,
      endCursor: edges.length > 0 ? edges[edges.length - 1].cursor : null,
    },
    totalCount: -1, // often omitted or computed separately
  };
}

When to Use What

  • Offset: Admin dashboards, small datasets, when users need "page 47 of 200"
  • Cursor: Infinite scroll UIs, feeds, mobile apps, real-time data
  • Keyset: High-performance APIs, large datasets, when you control the sort order

For most new APIs, cursor-based pagination is the best default choice. It performs well, handles concurrent writes gracefully, and is the standard in GraphQL.

Exercises

  1. Implement All Three Pagination Styles

    Build an Express API for a 'products' resource that supports all three pagination styles via a query parameter: ?style=offset, ?style=cursor, ?style=keyset. Each should return the same data but with appropriate navigation metadata.

    Hint

    Create three separate query builder functions and select based on the style param.

  2. Relay-Style Cursor Pagination

    Implement a GraphQL-compatible Relay connection type with edges, nodes, cursors, and pageInfo (hasNextPage, hasPreviousPage, startCursor, endCursor). Support both 'first/after' (forward) and 'last/before' (backward) pagination.

    Hint

    For backward pagination, reverse the sort order, fetch, then reverse the results back.

  3. Benchmark Offset vs Keyset

    Seed a PostgreSQL table with 1 million rows. Write a script that fetches page 1, page 100, page 1000, and page 50000 using both offset and keyset pagination. Measure and compare the response times.

    Hint

    Use console.time/console.timeEnd and make sure you have an appropriate index for keyset queries.

Common Problems & Solutions

1 Duplicate or missing items when using offset pagination
Cause: Records are inserted or deleted between page fetches, shifting the offset boundary.
Solution: Switch to cursor-based pagination, or if offset is required, add a 'snapshot' timestamp filter: WHERE created_at <= :snapshot_time.
2 Extremely slow responses on deep pages with offset
Cause: OFFSET N requires the database to scan and discard N rows before returning results.
Solution: Switch to keyset/cursor pagination. If offset is required, set a maximum page limit (e.g., max 1000 pages) and suggest search/filter instead.
3 Cursor pagination does not support jumping to a specific page
Cause: Cursors are sequential; you can only move forward or backward from a known position.
Solution: Accept this trade-off. For UIs that need page jumping, use offset. For infinite scroll or feeds, cursors are superior. You can also provide a 'total count' endpoint separately.
