systemd, cron, and Process Management

Intuition

systemd is like a building manager who ensures all the tenants (services) are in their offices when the building opens and restarts anyone who crashes. Cron is like a timer that rings at set intervals to remind you to do recurring chores -- take out the trash every night, check the mailbox every hour.

systemd, cron, and Process Management

In production, your Python applications run as background services. systemd manages services, cron schedules recurring tasks, and understanding process management prevents resource leaks.

systemd Service Files

# /etc/systemd/system/myapp.service
[Unit]
Description=My Python Web App
After=network.target postgresql.service

[Service]
User=deploy
WorkingDirectory=/opt/myapp
ExecStart=/opt/myapp/venv/bin/gunicorn -w 4 -b 0.0.0.0:8000 app:create_app()
Restart=always
RestartSec=5
EnvironmentFile=/opt/myapp/.env

[Install]
WantedBy=multi-user.target

# Managing services
sudo systemctl start myapp      # Start the service
sudo systemctl enable myapp     # Start on boot
sudo systemctl status myapp     # Check status
sudo journalctl -u myapp -f     # Follow service logs

Cron Jobs

# Edit crontab
crontab -e

# Format: minute hour day month weekday command
*/5 * * * *  /opt/myapp/venv/bin/python /opt/myapp/tasks/cleanup.py
0 2 * * *    /usr/bin/pg_dump mydb > /backups/db_$(date +\%F).sql
0 */6 * * *  /opt/myapp/venv/bin/python /opt/myapp/tasks/sync_data.py >> /var/log/sync.log 2>&1

Process Management

ps aux | grep python            # List Python processes
top -p $(pgrep -d, gunicorn)    # Monitor specific processes
kill -SIGTERM 12345             # Graceful shutdown
kill -SIGKILL 12345             # Force kill (last resort)
nohup python long_task.py &     # Run in background, survive logout

Common Mistakes

  • Forgetting Restart=always in service files, causing downtime after crashes
  • Not redirecting cron output, losing error messages silently
  • Using SIGKILL instead of SIGTERM, preventing graceful cleanup

Key Takeaways

  • Use systemd for long-running services, cron for scheduled tasks
  • Always check journalctl for service troubleshooting
  • Prefer SIGTERM over SIGKILL to allow graceful shutdown

Exercises

  1. Create a systemd Service

    Write a systemd service file for a FastAPI application that uses Uvicorn, restarts on failure, starts after the network is available, and loads environment variables from a .env file.

    Hint

    Use ExecStart with uvicorn, set Restart=on-failure, After=network.target, and EnvironmentFile.

  2. Scheduled Database Backup

    Set up a cron job that runs pg_dump every night at 2 AM, compresses the output with gzip, names the file with the current date, and keeps only the last 7 backups.

    Hint

    Combine pg_dump, gzip, date formatting, and find -mtime +7 -delete in a small shell script called by cron.

  3. Process Monitoring Script

    Write a bash script that checks if a gunicorn process is running. If not, restart the systemd service and send an alert to a log file with a timestamp.

    Hint

    Use pgrep gunicorn, check the exit code, and use systemctl restart if needed.

Common Problems & Solutions

1 systemd service fails to start
Symptoms: ['Active: failed (Result: exit-code)', 'Main process exited, code=exited, status=1/FAILURE']
Cause: Incorrect ExecStart path, missing dependencies, or permission issues.
Solution: Check logs with `sudo journalctl -u myapp -n 50 --no-pager`. Verify the ExecStart binary path exists and is executable. Ensure the User has permissions to the WorkingDirectory.
2 Cron job runs but produces no output or effect
Symptoms: ['Script works manually but not via cron']
Cause: Cron runs with a minimal environment -- PATH and other variables differ from your shell.
Solution: Use absolute paths for all commands in cron. Redirect output: `>> /var/log/myjob.log 2>&1`. Set PATH at the top of the crontab.
