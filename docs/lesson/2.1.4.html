Mypy Configuration and CI Integration

Intuition

Mypy configuration is like setting the strictness level on a spell checker. Start strict on new code, relax rules on legacy modules, and run it in CI so nothing slips through.

Mypy Configuration and CI Integration

Mypy is the most widely used static type checker for Python. Proper configuration and CI integration ensures type errors are caught before code is merged.

# mypy.ini or pyproject.toml configuration
# [mypy] in mypy.ini, or [tool.mypy] in pyproject.toml

# pyproject.toml example:
# [tool.mypy]
# python_version = "3.11"
# strict = true
# warn_return_any = true
# warn_unused_configs = true
# disallow_untyped_defs = true
#
# [tool.mypy.overrides]
# module = "third_party_lib.*"
# ignore_missing_imports = true

# Per-file ignores and inline overrides
def legacy_function(data):  # type: ignore[no-untyped-def]
    return data

# Gradual typing: start strict on new code
# CI integration (GitHub Actions)
# - name: Type check
#   run: mypy src/ --strict --no-error-summary

Key Takeaways: - Start with --strict for new projects; adopt gradually for legacy code - Use pyproject.toml [tool.mypy] for project-wide config - Override settings per module with [[tool.mypy.overrides]] - Use type: ignore[error-code] sparingly and always specify the error code - Integrate mypy into CI so type errors block merges

Exercises

  1. Configure Strict Mypy

    Create a pyproject.toml with [tool.mypy] set to strict mode. Add overrides to ignore missing imports for requests and numpy. Run mypy on a sample module.

    Hint

    Use [[tool.mypy.overrides]] with module = ["requests.*", "numpy.*"] and ignore_missing_imports = true.

  2. Add Mypy to CI

    Write a GitHub Actions workflow step that runs mypy with --strict on the src/ directory and fails the build on any type error.

    Hint

    Add a step after install: run: pip install mypy && mypy src/ --strict with continue-on-error: false.

Common Problems & Solutions

1 mypy reports `Cannot find implementation or library stub` for third-party packages.
Cause: The package lacks type stubs and mypy cannot infer types from it.
Solution: Install type stubs (`pip install types-requests`) or add `ignore_missing_imports = true` for that module in mypy config.
2 mypy is too slow on large codebases, causing CI timeouts.
Cause: mypy rechecks all files every run without caching.
Solution: Use `mypy --incremental` (default) and cache `.mypy_cache/` between CI runs. Consider `mypy daemon` for local development.
