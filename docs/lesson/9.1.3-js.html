Design a Notification Service

Intuition

A notification service works like a post office. Letters (notifications) arrive, get sorted by destination type (email, SMS, push -- like domestic, international, express), placed into separate processing lines, and handed off to delivery trucks (providers like SendGrid, Twilio). If one truck breaks down, the other routes keep flowing. And just like real mail, you can track whether it was delivered, opened, or returned to sender.

Design a Notification Service

Introduction

A notification service delivers messages to users across multiple channels: push notifications, email, SMS, and in-app notifications. Companies like Uber, Slack, and Amazon send billions of notifications daily. This system must handle multiple delivery channels, user preferences, templating, scheduling, and reliable delivery at scale -- making it an excellent system design interview question that tests your understanding of async processing, message queues, and system integration.

Step 1: Requirements Gathering

Functional Requirements

  1. Send notifications through multiple channels: push (iOS/Android), email, SMS, in-app
  2. Support user notification preferences (which channels, quiet hours, frequency caps)
  3. Template system for notification content with variable substitution
  4. Priority levels: critical (immediate), high, normal, low (can be batched)
  5. Scheduling: send notifications at a specific time or after a delay
  6. Delivery tracking: sent, delivered, read, failed, bounced
  7. Rate limiting per user to prevent notification fatigue

Non-Functional Requirements

  1. High throughput: handle 10M+ notifications per day
  2. Reliable delivery: at-least-once delivery guarantee
  3. Low latency for critical notifications (<1 second end-to-end)
  4. Pluggable providers: switch email/SMS providers without code changes
  5. Horizontally scalable to handle traffic spikes (e.g., flash sales)

Step 2: Capacity Estimation

Traffic Estimates

  • Daily notifications: 10M (across all channels)
  • Peak: 500 notifications/second (normal), 5,000/second during spikes
  • Channel breakdown: 40% push, 30% email, 10% SMS, 20% in-app
  • Average notification payload: 1 KB

Storage Estimates

  • Notification records: 10M/day x 1 KB = 10 GB/day
  • 1 year retention: 10 GB x 365 = 3.6 TB
  • User preferences: 50M users x 500 bytes = 25 GB
  • Templates: negligible (~1000 templates x 5 KB = 5 MB)

Queue Throughput

  • Average: ~120 messages/second into the queue
  • Peak burst: 5,000 messages/second
  • Worker processing time per notification: ~200ms (including external API call)
  • Workers needed at peak: 5,000 x 0.2 = 1,000 concurrent workers

Component Responsibilities

  • Notification Service: The orchestration layer. Validates input, resolves user preferences, renders templates, deduplicates, and routes to the correct channel queue.
  • Priority Router: Sends critical notifications to high-priority queues with more workers; batches low-priority notifications.
  • Channel Queues: Separate BullMQ queues per channel for independent scaling and failure isolation.
  • Channel Workers: Process queued notifications by calling external providers (SendGrid, Twilio, APNs, FCM).
  • Delivery Tracking: Records delivery status and processes webhooks from providers (bounce, delivered, opened).

Express Route Implementation

import express from 'express';
import { Queue } from 'bullmq';

const router = express.Router();
const notificationQueue = new Queue('notifications', { connection: redisConnection });

router.post('/api/v1/notifications', async (req, res) => {
  const { user_id, template_id, channels, priority, data, schedule_at, idempotency_key } = req.body;

  // Idempotency check
  if (idempotency_key) {
    const existing = await redis.get(`idempotency:${idempotency_key}`);
    if (existing) return res.status(200).json(JSON.parse(existing));
  }

  // Validate template exists
  const template = await prisma.notificationTemplate.findUnique({
    where: { id: template_id }
  });
  if (!template) return res.status(400).json({ error: 'Invalid template_id' });

  // Resolve user preferences
  const preferences = await prisma.notificationPreference.findUnique({
    where: { userId: user_id }
  });

  // Determine target channels
  const targetChannels = resolveChannels(channels, preferences, template.category);

  // Create notification record
  const notification = await prisma.notification.create({
    data: {
      userId: user_id,
      templateId: template_id,
      priority,
      channels: targetChannels,
      data,
      status: 'queued'
    }
  });

  // Enqueue for processing
  const jobOptions = {
    priority: priorityToNumber(priority),
    delay: schedule_at ? new Date(schedule_at).getTime() - Date.now() : 0,
    attempts: 3,
    backoff: { type: 'exponential', delay: 5000 }
  };

  await notificationQueue.add('send', {
    notificationId: notification.id,
    userId: user_id,
    templateId: template_id,
    channels: targetChannels,
    data,
    priority
  }, jobOptions);

  const response = {
    notification_id: notification.id,
    status: 'queued',
    channels: targetChannels
  };

  // Store idempotency key
  if (idempotency_key) {
    await redis.setex(`idempotency:${idempotency_key}`, 86400, JSON.stringify(response));
  }

  res.status(202).json(response);
});

Step 5: Template System

// Templates stored in the database with Handlebars-style placeholders
const renderTemplate = (template, data) => {
  let rendered = template;
  for (const [key, value] of Object.entries(data)) {
    rendered = rendered.replaceAll(`{{${key}}}`, value);
  }
  return rendered;
};

  processed_at    TIMESTAMP
);

CREATE INDEX idx_notifications_user ON notifications(user_id, created_at DESC);
CREATE INDEX idx_notifications_status ON notifications(status) WHERE status IN ('queued', 'processing');

-- Per-channel delivery status
CREATE TABLE notification_deliveries (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  notification_id UUID REFERENCES notifications(id),
  channel         VARCHAR(10) NOT NULL,   -- push, email, sms, in_app
  provider        VARCHAR(50),            -- sendgrid, twilio, fcm
  status          VARCHAR(20) NOT NULL DEFAULT 'pending',
  provider_msg_id VARCHAR(200),           -- external message ID for tracking
  sent_at         TIMESTAMP,
  delivered_at    TIMESTAMP,
  read_at         TIMESTAMP,
  failed_at       TIMESTAMP,
  error_message   TEXT,
  retry_count     INT DEFAULT 0
);

Step 7: Worker Implementation

import { Worker } from 'bullmq';

// Channel-specific provider abstraction
const providers = {
  push: new PushProvider(),    // wraps FCM + APNs
  email: new EmailProvider(),  // wraps SendGrid / SES
  sms: new SmsProvider(),      // wraps Twilio
  in_app: new InAppProvider()  // writes to DB + WebSocket
};

const worker = new Worker('notifications', async (job) => {
  const { notificationId, userId, templateId, channels, data, priority } = job.data;

  // Check quiet hours
  const prefs = await getPreferences(userId);
  if (isQuietHours(prefs) && priority !== 'critical') {
    // Re-queue for after quiet hours
    const resumeAt = getQuietHoursEnd(prefs);
    await notificationQueue.add('send', job.data, {
      delay: resumeAt - Date.now()
    });
    return;
  }

  // Send to each channel in parallel
  const results = await Promise.allSettled(
    channels.map(async (channel) => {
      const template = await getTemplate(templateId, channel);
      const rendered = renderTemplate(template.body, data);
      const subject = template.subject ? renderTemplate(template.subject, data) : null;

      const delivery = await prisma.notificationDelivery.create({
        data: { notificationId, channel, status: 'sending' }
      });

      try {
        const result = await providers[channel].send(userId, {
          subject,
          body: rendered,
          data
        });

        await prisma.notificationDelivery.update({
          where: { id: delivery.id },
          data: {
            status: 'sent',
            sentAt: new Date(),
            providerMsgId: result.messageId
          }
        });
      } catch (err) {
        await prisma.notificationDelivery.update({
          where: { id: delivery.id },
          data: {
            status: 'failed',
            failedAt: new Date(),
            errorMessage: err.message
          }
        });
        throw err; // triggers BullMQ retry
      }
    })
  );

  // Update overall notification status
  const allSucceeded = results.every(r => r.status === 'fulfilled');
  await prisma.notification.update({
    where: { id: notificationId },
    data: {
      status: allSucceeded ? 'sent' : 'partial_failure',
      processedAt: new Date()
    }
  });
}, {
  connection: redisConnection,
  concurrency: 50
});

Step 8: Scaling Strategies

Separate Queues per Channel

Isolate channels so a Twilio outage does not block push notifications:

const queues = {
  push: new Queue('notifications:push', { connection: redisConnection }),
  email: new Queue('notifications:email', { connection: redisConnection }),
  sms: new Queue('notifications:sms', { connection: redisConnection }),
  in_app: new Queue('notifications:in_app', { connection: redisConnection })
};

Notification Deduplication

Prevent duplicate sends using idempotency keys:

async function isDuplicate(idempotencyKey) {
  const result = await redis.set(
    `dedup:${idempotencyKey}`,
    '1',
    'EX', 86400,
    'NX'  // Only set if not exists
  );
  return result === null; // null means key already existed
}

Batching for Low-Priority Notifications

Digest emails and batch low-priority notifications:

// Collect low-priority notifications and send as a batch
const digestWorker = new Worker('notifications:digest', async (job) => {
  const { userId } = job.data;
  const pending = await prisma.notification.findMany({
    where: { userId, priority: 'low', status: 'queued' }
  });

  if (pending.length > 0) {
    await providers.email.sendDigest(userId, pending);
  }
});

// Schedule digest processing every hour
const digestScheduler = new Queue('notifications:digest');
await digestScheduler.upsertJobScheduler('digest-all', {
  every: 3600000
});

Provider Failover

Implement automatic failover when a provider goes down:

class EmailProvider {
  constructor() {
    this.providers = [new SendGridProvider(), new SESProvider()];
    this.currentIndex = 0;
  }

  async send(userId, content) {
    for (let i = 0; i < this.providers.length; i++) {
      const index = (this.currentIndex + i) % this.providers.length;
      try {
        return await this.providers[index].send(userId, content);
      } catch (err) {
        if (i === this.providers.length - 1) throw err;
        // Try next provider
      }
    }
  }
}

Summary

A notification service is fundamentally an async message processing pipeline. The key architectural decisions are: separating queues per channel for failure isolation, implementing at-least-once delivery with idempotency for correctness, using a template system for content management, respecting user preferences and quiet hours, and building provider abstraction layers that support automatic failover. The system must handle both real-time critical alerts and batched low-priority digests, requiring a priority-aware routing layer.

Exercises

  1. Build a Multi-Channel Notification Worker

    Implement a BullMQ worker that processes notification jobs. Each job specifies a user_id, template_id, and channels array. The worker should render the template, send to each channel using mock provider functions, record delivery status in the database, and handle partial failures (some channels succeed, others fail). Use Prisma for database operations.

    Hint

    Use Promise.allSettled() to send to all channels in parallel without short-circuiting on failure. Check each result's status to determine if it was 'fulfilled' or 'rejected' and update delivery records accordingly.

  2. Implement User Preference Resolution

    Write a resolveChannels() function that takes requested channels, user preferences, and notification category as input and returns the final list of channels to send on. It must filter out disabled channels, respect category opt-outs, check quiet hours (skip non-critical during quiet hours), and handle the 'daily_digest' email frequency preference.

    Hint

    Parse quiet hours using the user's timezone (use luxon or date-fns-tz). For daily_digest, queue the email notification instead of sending immediately. Return an object with { immediate: ['push'], deferred: ['email'] }.

  3. Notification Deduplication System

    Build an idempotency layer that prevents duplicate notifications. Use Redis SET with NX (set-if-not-exists) to track idempotency keys. Write tests that verify: (1) the first request succeeds, (2) a retry with the same key returns the cached response, (3) keys expire after 24 hours.

    Hint

    Use redis.set(key, value, 'EX', 86400, 'NX') which atomically sets the key only if it does not exist. If the return value is null, the key already existed (duplicate). Store the original response JSON as the value.

  4. Provider Circuit Breaker

    Implement a circuit breaker pattern for notification providers. When a provider (e.g., SendGrid) fails 5 times in 60 seconds, the circuit opens and automatically routes to a fallback provider (e.g., SES). After 30 seconds, the circuit enters half-open state and lets one test request through. Track circuit state in Redis for distributed correctness.

    Hint

    Store failure counts and circuit state in Redis with TTLs. Use three states: CLOSED (normal), OPEN (all requests go to fallback), HALF_OPEN (one test request allowed). Use redis.incr() with pexpire for failure counting.

Common Problems & Solutions

1 Users receive duplicate notifications for the same event
Cause: Retries from the message queue (due to transient failures or worker crashes) cause the same notification to be processed multiple times, and there is no deduplication check.
Solution: Implement idempotency using a unique key per notification event (e.g., 'order_shipped:order_12345'). Before processing, check if the key exists in Redis. Use SET NX to atomically claim the key. Also check the notification_deliveries table for existing records.
2 SMS provider outage causes push notifications to back up
Cause: All notification channels share a single queue. When SMS workers slow down or fail, the queue grows and all channels are affected.
Solution: Use separate queues per channel (notifications:push, notifications:email, notifications:sms, notifications:in_app). This provides failure isolation -- an SMS outage has zero impact on push notification delivery.
3 Users complain about receiving notifications at 3 AM
Cause: The notification service does not account for user timezones when evaluating quiet hours. Quiet hours of 22:00-08:00 are evaluated in UTC instead of the user's local time.
Solution: Store the user's timezone in preferences. When checking quiet hours, convert the current UTC time to the user's local time before comparison. Use a library like luxon: DateTime.now().setZone(userTimezone).hour to get the local hour.
