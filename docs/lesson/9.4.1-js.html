Design Search Autocomplete (Google)

Intuition

Think of autocomplete like a librarian who has memorized the most popular book titles. When you say "har...", they immediately suggest "Harry Potter" without searching every shelf. The trie is like an organized card catalog where following letters down a path instantly narrows options, and the librarian has pre-sorted the most popular completions for every prefix they know.

Design Search Autocomplete (Google)

Introduction

Search autocomplete (also called typeahead or search suggestions) is the feature that predicts what a user is about to type and shows real-time suggestions as they enter characters. Google processes over 8.5 billion searches per day, and autocomplete must respond within 100ms for every keystroke. This problem tests your ability to design a system that handles extreme read throughput with ultra-low latency.

Step 1: Requirements Gathering

Functional Requirements

  • As a user types in a search box, the system returns the top 5-10 suggestions matching the prefix
  • Suggestions are ranked by popularity (search frequency)
  • Suggestions update periodically to reflect trending queries
  • Support for multiple languages
  • Filter out offensive or inappropriate suggestions

Non-Functional Requirements

  • Latency: < 100ms p99 response time
  • Availability: 99.99% uptime
  • Scale: 10 billion queries per day, suggestions for every keystroke
  • Freshness: Trending topics appear within minutes
  • Consistency: Eventual consistency is acceptable

Out of Scope

  • Personalized suggestions (per-user history)
  • Spell correction
  • The actual search results page

Step 2: Capacity Estimation

Assumptions:
- 500 million daily active users (DAU)
- Average 6 searches per user per day = 3 billion searches/day
- Average 5 characters typed per search (with autocomplete reducing typing)
- Each keystroke triggers a suggestion request

Query Volume:
- 3 billion searches x 5 keystrokes = 15 billion suggestion requests/day
- 15 billion / 86,400 seconds = ~174,000 requests/second (QPS)
- Peak QPS (3x average) = ~520,000 QPS

Storage:
- Unique search queries: ~500 million
- Average query length: 20 characters = 20 bytes
- Frequency counter: 8 bytes
- Total raw data: 500M x 28 bytes = ~14 GB
- Trie structure overhead (3-5x): ~70 GB
- With replication (3x): ~210 GB

Bandwidth:
- Average response: 10 suggestions x 20 chars = 200 bytes + overhead = ~500 bytes
- 174,000 QPS x 500 bytes = ~87 MB/s outbound
- Peak: ~260 MB/s outbound

Step 4: Core Data Structure - The Trie

A trie (prefix tree) is the ideal data structure for autocomplete. Each node represents a character, and paths from root to node form prefixes.

Root
├── t
│   ├── h
│   │   └── e [freq: 50000]
│   │       ├── r
│   │       │   └── e [freq: 12000]
│   │       └── n [freq: 8000]
│   ├── r
│   │   ├── u
│   │   │   └── m
│   │   │       └── p [freq: 25000]
│   │   └── e
│   │       └── n
│   │           └── d [freq: 15000]
│   └── o
│       └── p [freq: 9000]
└── ...

Query: "tr" -> Returns: ["trump", "trend", "tree", ...]

Trie Implementation in JavaScript

class TrieNode {
  constructor() {
    this.children = new Map();
    this.isEnd = false;
    this.frequency = 0;
    // Cache top-k suggestions at each node for O(1) lookup
    this.topSuggestions = [];
  }
}

class AutocompleteTrie {
  constructor(k = 10) {
    this.root = new TrieNode();
    this.k = k; // number of suggestions to return
  }

  insert(word, frequency) {
    let node = this.root;
    for (const char of word) {
      if (!node.children.has(char)) {
        node.children.set(char, new TrieNode());
      }
      node = node.children.get(char);
    }
    node.isEnd = true;
    node.frequency = frequency;
  }

  // Pre-compute top-k suggestions at every node
  buildTopSuggestions(node = this.root, prefix = '') {
    const suggestions = [];
    this._collectAll(node, prefix, suggestions);
    suggestions.sort((a, b) => b.frequency - a.frequency);
    node.topSuggestions = suggestions.slice(0, this.k);

    for (const [char, child] of node.children) {
      this.buildTopSuggestions(child, prefix + char);
    }
  }

  _collectAll(node, prefix, results) {
    if (node.isEnd) {
      results.push({ word: prefix, frequency: node.frequency });
    }
    for (const [char, child] of node.children) {
      this._collectAll(child, prefix + char, results);
    }
  }

  // O(L) lookup where L = prefix length
  getSuggestions(prefix) {
    let node = this.root;
    for (const char of prefix) {
      if (!node.children.has(char)) return [];
      node = node.children.get(char);
    }
    return node.topSuggestions;
  }
}

// Usage
const trie = new AutocompleteTrie(5);
trie.insert('the', 50000);
trie.insert('there', 12000);
trie.insert('then', 8000);
trie.insert('trump', 25000);
trie.insert('trend', 15000);
trie.buildTopSuggestions();

console.log(trie.getSuggestions('tr'));
// [{ word: 'trump', frequency: 25000 }, { word: 'trend', frequency: 15000 }]

Step 5: API Design

import express from 'express';

const app = express();

// GET /api/v1/suggestions?q=prefix&limit=10&lang=en
app.get('/api/v1/suggestions', async (req, res) => {
  const { q, limit = 10, lang = 'en' } = req.query;

  if (!q || q.length === 0) {
    return res.json({ suggestions: [] });
  }

  if (q.length > 100) {
    return res.status(400).json({ error: 'Query too long' });
  }

  const sanitizedQuery = q.toLowerCase().trim();
  const results = await suggestionService.getSuggestions(
    sanitizedQuery,
    parseInt(limit),
    lang
  );

  res.json({
    query: sanitizedQuery,
    suggestions: results.map(r => ({
      text: r.word,
      score: r.frequency,
    })),
  });
});

// POST /api/v1/queries (data collection - async, fire-and-forget)

Client-Side Optimization

// Debounce requests - don't send on every keystroke
function debounce(fn, delay) {
  let timer;
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

const fetchSuggestions = debounce(async (prefix) => {
  const res = await fetch(
    `/api/v1/suggestions?q=${encodeURIComponent(prefix)}&limit=5`
  );
  const data = await res.json();
  renderSuggestions(data.suggestions);
}, 150); // 150ms debounce

// Also: abort previous in-flight requests
let controller;
async function fetchWithAbort(prefix) {
  controller?.abort();
  controller = new AbortController();
  const res = await fetch(`/api/v1/suggestions?q=${prefix}`, {
    signal: controller.signal,
  });
  return res.json();
}

// Aggregator worker - processes Kafka messages
import { Kafka } from 'kafkajs';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();
const kafka = new Kafka({ brokers: ['kafka:9092'] });
const consumer = kafka.consumer({ groupId: 'query-aggregator' });

async function runAggregator() {
  await consumer.connect();
  await consumer.subscribe({ topic: 'search-queries' });

  // Batch aggregation in memory
  const batchCounts = new Map();
  let lastFlush = Date.now();

  await consumer.run({
    eachMessage: async ({ message }) => {
      const { query } = JSON.parse(message.value.toString());
      const normalized = query.toLowerCase().trim();
      batchCounts.set(normalized, (batchCounts.get(normalized) || 0) + 1);

      // Flush to DB every 60 seconds or every 10,000 entries
      if (Date.now() - lastFlush > 60000 || batchCounts.size > 10000) {
        await flushToDatabase(batchCounts);
        batchCounts.clear();
        lastFlush = Date.now();
      }
    },
  });
}

async function flushToDatabase(counts) {
  const entries = Array.from(counts.entries());
  await prisma.$transaction(
    entries.map(([query, count]) =>
      prisma.$executeRaw`
        INSERT INTO query_frequencies (query_text, frequency)
        VALUES (${query}, ${count})
        ON CONFLICT (query_text, language)
        DO UPDATE SET
          frequency = query_frequencies.frequency + ${count},
          last_updated = NOW()
      `
    )
  );
}

runAggregator();

Step 8: Serving Layer with Redis

For production, we cache trie lookups in Redis for sub-millisecond responses:

import Redis from 'ioredis';

const redis = new Redis({ host: 'redis-cluster', port: 6379 });

class RedisSuggestionService {
  constructor(redis) {
    this.redis = redis;
    this.TTL = 3600; // 1 hour cache
  }

  // Store suggestions using sorted sets
  async populateSuggestions(prefix, suggestions) {
    const key = `autocomplete:${prefix}`;
    const pipeline = this.redis.pipeline();
    pipeline.del(key);
    for (const { word, frequency } of suggestions) {
      pipeline.zadd(key, frequency, word);
    }
    pipeline.expire(key, this.TTL);
    await pipeline.exec();
  }

  // Retrieve top-k suggestions
  async getSuggestions(prefix, limit = 10) {
    const key = `autocomplete:${prefix}`;
    // ZREVRANGE returns highest scores first
    const results = await this.redis.zrevrange(key, 0, limit - 1, 'WITHSCORES');

    const suggestions = [];
    for (let i = 0; i < results.length; i += 2) {
      suggestions.push({
        word: results[i],
        frequency: parseInt(results[i + 1]),
      });
    }
    return suggestions;
  }
}

Step 9: Scaling Strategies

Sharding by Prefix

Shard A: a-f prefixes  ->  Trie Server Cluster A (3 replicas)
Shard B: g-m prefixes  ->  Trie Server Cluster B (3 replicas)
Shard C: n-s prefixes  ->  Trie Server Cluster C (3 replicas)
Shard D: t-z prefixes  ->  Trie Server Cluster D (3 replicas)

Multi-Layer Caching

Layer 1: Browser cache (Cache-Control headers, 5 min TTL)
    |
Layer 2: CDN edge cache (popular prefixes like "wh", "ho")
    |
Layer 3: Application-level LRU cache (in-memory, per-server)
    |
Layer 4: Redis distributed cache
    |
Layer 5: Trie store (source of truth)

Edge Deployment

Deploy suggestion servers in edge locations close to users to minimize network latency:

US-East User ──> US-East Edge Server ──> US-East Redis Replica
EU User ──────> EU Edge Server ────────> EU Redis Replica
Asia User ────> Asia Edge Server ──────> Asia Redis Replica

Step 10: Trade-offs and Decisions

Decision Option A Option B Recommendation
Data structure Trie (prefix tree) Sorted set in Redis Trie for complex ranking; Redis sorted sets for simpler systems
Storage In-memory trie Redis cluster In-memory for lowest latency; Redis for shared state
Update frequency Real-time Batch (every 15 min) Batch for most queries; real-time for trending
Ranking Pure frequency ML-based scoring Frequency first; add ML later for personalization
Sharding By prefix range By consistent hash Prefix range for locality; consistent hash for even distribution

Handling Trending Topics

Use a two-layer ranking: a base layer from historical frequency (updated hourly) and a trending layer from recent spikes (updated every few minutes). Combine with a weighted formula:

function computeScore(historicalFreq, recentFreq, hoursAgo) {
  const decayFactor = Math.exp(-0.1 * hoursAgo);
  const trendBoost = recentFreq * decayFactor * 10;
  return historicalFreq + trendBoost;
}

Content Filtering

Maintain a blocklist of offensive terms. Filter at both the trie-building stage and at query time as a safety net. Use a Bloom filter for fast membership checks.

Summary

Search autocomplete requires a read-heavy architecture optimized for sub-100ms latency. The core data structure is a trie with pre-computed top-k suggestions at each node. A data collection pipeline aggregates query frequencies, and a periodic trie builder creates snapshots that are distributed to serving nodes. Multi-layer caching (browser, CDN, in-memory, Redis) absorbs the massive read load, while sharding by prefix range distributes the trie across multiple servers.

Common Problems & Solutions

1 Autocomplete suggestions are stale and don't reflect trending topics
Symptoms: A major news event occurs but related search suggestions don't appear for hours. Users notice autocomplete is "behind" compared to competitors.
Cause: The trie is rebuilt too infrequently (e.g., daily) and the data pipeline has too much lag between query collection and trie updates.
Solution: Implement a two-tier system: a base trie rebuilt hourly from historical data, plus a real-time trending layer using a sliding window counter in Redis. Merge results at query time with a weighted scoring formula that boosts recent spikes.
2 Memory usage on trie servers keeps growing and eventually causes OOM crashes
Cause: The trie stores every unique query ever seen, including long-tail queries with very low frequency. Rare queries consume memory without providing useful suggestions.
Solution: Set a minimum frequency threshold (e.g., 10 searches/week) for trie inclusion. Periodically prune low-frequency entries. Use compact trie representations (compressed/Patricia trie) to reduce node count.
3 Latency spikes during trie rebuilds
Cause: The trie is rebuilt in-place on the serving node, causing garbage collection pressure and contention during the build process.
Solution: Build the new trie offline on a separate builder service. Serialize it to a binary snapshot, distribute the snapshot to serving nodes, and atomically swap the old trie for the new one. This ensures zero-downtime updates.
