Pagination, Filtering, and Search

Intuition

Pagination, filtering, and search are the three pillars of data retrieval that transform a toy API into a production-ready one. Without them, every list endpoint becomes a ticking time bomb as your data grows.

Pagination, Filtering, and Search

Production APIs need efficient data retrieval strategies. Pagination prevents loading entire tables into memory, filtering lets clients narrow results, and search enables text-based queries. Implementing these correctly improves both performance and user experience.

from flask import Flask, request, jsonify, url_for
from flask_sqlalchemy import SQLAlchemy

db = SQLAlchemy()
app = Flask(__name__)
app.config['SQLALCHEMY_DATABASE_URI'] = 'sqlite:///app.db'
db.init_app(app)

class Post(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    title = db.Column(db.String(200), nullable=False)
    body = db.Column(db.Text)
    category = db.Column(db.String(50))
    published = db.Column(db.Boolean, default=False)

@app.route('/api/posts')
def list_posts():
    # Pagination parameters
    page = request.args.get('page', 1, type=int)
    per_page = min(request.args.get('per_page', 20, type=int), 100)

    # Build query with filters
    query = Post.query

    category = request.args.get('category')
    if category:
        query = query.filter_by(category=category)

    published = request.args.get('published')
    if published is not None:
        query = query.filter_by(published=published.lower() == 'true')

    # Full-text search (simple LIKE approach)
    search = request.args.get('q')
    if search:
        query = query.filter(
            Post.title.ilike(f'%{search}%') |
            Post.body.ilike(f'%{search}%')
        )

    # Sort
    sort = request.args.get('sort', 'id')
    order = request.args.get('order', 'asc')
    sort_col = getattr(Post, sort, Post.id)
    query = query.order_by(sort_col.desc() if order == 'desc' else sort_col.asc())

    # Execute paginated query
    pagination = query.paginate(page=page, per_page=per_page, error_out=False)

    return jsonify({
        'data': [{'id': p.id, 'title': p.title, 'category': p.category}
                 for p in pagination.items],
        'meta': {
            'page': page, 'per_page': per_page,
            'total': pagination.total, 'pages': pagination.pages
        }
    })

Key Takeaways: - Always cap per_page with a maximum to prevent clients from requesting enormous result sets. - Build queries incrementally by chaining filter conditions based on which parameters are present. - For production search, consider PostgreSQL full-text search or Elasticsearch instead of LIKE queries.

Exercises

  1. Cursor-Based Pagination

    Implement cursor-based pagination using the last item's ID as the cursor instead of page numbers. Return a 'next_cursor' in the response that clients use for the next page.

    Hint

    Use Post.query.filter(Post.id > cursor).limit(per_page).all() and set next_cursor to the last item's ID. This is more efficient than OFFSET for large datasets.

  2. Advanced Filtering

    Build a filtering system that supports operators: ?price_gte=10&price_lte=50&name_contains=widget. Parse these query parameters and build SQLAlchemy filter conditions dynamically.

    Hint

    Parse parameter names to extract field and operator (e.g., 'price_gte' -> field='price', op='gte'). Map operators to SQLAlchemy comparators: gte -> column >= value.

Common Problems & Solutions

1 Pagination performance degrades significantly on large tables with high page numbers.
Cause: OFFSET-based pagination requires the database to scan and skip all rows before the offset, which gets slower as the offset increases.
Solution: Switch to cursor-based (keyset) pagination using WHERE id > last_seen_id ORDER BY id LIMIT n. This uses index scans and has consistent performance regardless of page depth.
2 LIKE '%search%' queries are extremely slow on large tables.
Cause: Leading wildcard (%) prevents the database from using indexes, resulting in full table scans.
Solution: Use PostgreSQL full-text search (tsvector/tsquery), add a GIN index, or use a dedicated search engine like Elasticsearch for complex search requirements.
