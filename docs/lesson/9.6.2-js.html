Design an Email Service (Gmail)

Intuition

Think of an email service like a postal system for a city of 1 billion people. Each person has a private mailbox (sharded storage). The post office (SMTP gateway) receives mail, a sorting facility (spam filter + routing) processes it, and letter carriers (delivery workers) place it in the right mailbox. The key insight: you organize everything around individual mailboxes, not around letters.

Design an Email Service (Gmail)

Introduction

Email is one of the oldest and most widely used distributed systems on the internet. Designing a large-scale email service like Gmail involves handling billions of messages daily, supporting full-text search across massive mailboxes, managing attachments efficiently, and maintaining real-time notifications. This is a rich system design problem that touches storage, search, queueing, and push notifications.

Requirements

Functional Requirements

  • Users can send emails to one or many recipients (including CC, BCC)
  • Users can receive emails and see them in their inbox
  • Support folders (Inbox, Sent, Drafts, Trash, Spam, custom labels)
  • Full-text search across all emails
  • Support attachments up to 25 MB per email
  • Push notifications for new email arrival
  • Support threading — group related emails into conversations
  • Mark emails as read/unread, star, archive
  • Spam and phishing detection

Non-Functional Requirements

  • 1 billion active users
  • 300 billion emails sent/received per day (across all users)
  • Average mailbox: 10 GB (growing)
  • Search latency: < 500 ms for full-text search
  • Availability: 99.99%
  • Durability: zero data loss for stored email

Capacity Estimation

Users:                  1 billion
Emails/day:             300 billion (~3,500 per second per user is wrong; 
                        avg ~100 emails/user/day, many are B2B/automated)
Emails/second:          300B / 86,400 = ~3.5 million emails/sec
Avg email size:         50 KB (without attachments)
With attachments (20%): 50 KB * 0.8 + 500 KB * 0.2 = 140 KB avg
Storage/day:            3.5M/s * 140 KB * 86,400 = ~42 PB/day
Storage/year:           ~15 EB (exabytes)
Bandwidth (ingress):    3.5M/s * 140 KB = ~490 GB/s

Simplified per-user:
  Emails/day:           100
  Storage growth/day:   100 * 140 KB = 14 MB/day
  Storage growth/year:  ~5 GB/year

API Design

Send Email

import express from 'express';
import multer from 'multer';
const app = express();
const upload = multer({ limits: { fileSize: 25 * 1024 * 1024 } });

// POST /api/v1/emails/send
app.post('/api/v1/emails/send', upload.array('attachments', 10), async (req, res) => {
  const { to, cc, bcc, subject, body, bodyHtml, inReplyTo, threadId } = req.body;
  const attachments = req.files || [];

  // Validate recipients
  const allRecipients = [
    ...parseRecipients(to),
    ...parseRecipients(cc),
    ...parseRecipients(bcc),
  ];
  if (allRecipients.length === 0) {
    return res.status(400).json({ error: 'At least one recipient required' });
  }
  if (allRecipients.length > 500) {
    return res.status(400).json({ error: 'Maximum 500 recipients' });
  }

  // Upload attachments to object store
  const attachmentRefs = await Promise.all(
    attachments.map(file => attachmentStore.upload({
      buffer: file.buffer,
      filename: file.originalname,
      contentType: file.mimetype,
      size: file.size,
    }))
  );

  // Create email record
  const email = await emailService.create({
    from: req.user.email,
    to: parseRecipients(to),
    cc: parseRecipients(cc),
    bcc: parseRecipients(bcc),
    subject,
    body,
    bodyHtml,
    attachments: attachmentRefs,
    threadId: threadId || generateThreadId(),
    inReplyTo,
    messageId: generateMessageId(req.user.domain),
  });

  // Enqueue for async delivery
  await messageQueue.produce('email-send', {
    key: email.id,
    value: email,
  });

  res.status(202).json({
    id: email.id,
    messageId: email.messageId,
    status: 'queued',
  });
});

Read Emails (Inbox)

// GET /api/v1/emails?folder=inbox&page=1&limit=50
app.get('/api/v1/emails', async (req, res) => {
  const { folder, page, limit, q } = req.query;
  const userId = req.user.id;

  let emails;
  if (q) {
    // Full-text search via Elasticsearch
    emails = await searchService.searchEmails(userId, {
      query: q,
      folder,
      from: parseInt(page - 1) * parseInt(limit),
      size: parseInt(limit) || 50,
    });
  } else {
    emails = await emailService.listByFolder(userId, {
      folder: folder || 'inbox',
      page: parseInt(page) || 1,
      limit: parseInt(limit) || 50,
    });
  }

  res.json({
    emails: emails.items.map(e => ({
      id: e.id,
      threadId: e.threadId,
      from: e.from,
      to: e.to,
      subject: e.subject,
      snippet: e.body.substring(0, 200),
      isRead: e.isRead,
      isStarred: e.isStarred,
      labels: e.labels,
      hasAttachments: e.attachments.length > 0,
      receivedAt: e.receivedAt,
    })),
    pagination: {
      page: emails.page,
      limit: emails.limit,
      total: emails.total,
      hasMore: emails.hasMore,
    },
  });
});

Thread View

// GET /api/v1/emails/threads/:threadId
app.get('/api/v1/emails/threads/:threadId', async (req, res) => {
  const { threadId } = req.params;
  const userId = req.user.id;

  const thread = await emailService.getThread(userId, threadId);
  if (!thread) return res.status(404).json({ error: 'Thread not found' });

  // Mark all unread messages in thread as read
  await emailService.markThreadRead(userId, threadId);

  res.json({
    threadId,
    subject: thread.subject,
    participants: thread.participants,
    messageCount: thread.messages.length,
    messages: thread.messages.map(m => ({
      id: m.id,
      from: m.from,
      to: m.to,
      cc: m.cc,
      body: m.body,
      bodyHtml: m.bodyHtml,
      attachments: m.attachments.map(a => ({
        id: a.id,
        filename: a.filename,
        size: a.size,
        contentType: a.contentType,
        downloadUrl: a.downloadUrl,
      })),
      sentAt: m.sentAt,
    })),
  });
});

Database Schema

-- User accounts
CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         VARCHAR(255) UNIQUE NOT NULL,
  display_name  VARCHAR(255) NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  storage_used  BIGINT NOT NULL DEFAULT 0,
  storage_limit BIGINT NOT NULL DEFAULT 15737418240, -- 15 GB
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Emails (denormalized for read performance)
-- In production this would be a wide-column store like Bigtable or Cassandra
CREATE TABLE emails (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  message_id    VARCHAR(512) UNIQUE NOT NULL,  -- RFC 5322 Message-ID
  thread_id     VARCHAR(255) NOT NULL,
  from_addr     VARCHAR(255) NOT NULL,
  to_addrs      TEXT[] NOT NULL,
  cc_addrs      TEXT[] DEFAULT '{}',
  bcc_addrs     TEXT[] DEFAULT '{}',
  subject       TEXT NOT NULL,
  body_text     TEXT,
  body_html     TEXT,
  size_bytes    INT NOT NULL,
  in_reply_to   VARCHAR(512),
  has_attachments BOOLEAN NOT NULL DEFAULT FALSE,
  spam_score    FLOAT DEFAULT 0.0,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_emails_thread ON emails(thread_id, created_at);
CREATE INDEX idx_emails_message_id ON emails(message_id);

-- User-email relationship (per-user view of emails)
CREATE TABLE user_emails (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL REFERENCES users(id),
  email_id      UUID NOT NULL REFERENCES emails(id),
  folder        VARCHAR(50) NOT NULL DEFAULT 'inbox',
  labels        TEXT[] DEFAULT '{}',
  is_read       BOOLEAN NOT NULL DEFAULT FALSE,
  is_starred    BOOLEAN NOT NULL DEFAULT FALSE,
  is_draft      BOOLEAN NOT NULL DEFAULT FALSE,
  is_spam       BOOLEAN NOT NULL DEFAULT FALSE,
  is_trash      BOOLEAN NOT NULL DEFAULT FALSE,
  snoozed_until TIMESTAMPTZ,
  received_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (user_id, email_id)
);

CREATE INDEX idx_user_emails_folder ON user_emails(user_id, folder, received_at DESC);
CREATE INDEX idx_user_emails_labels ON user_emails USING GIN(labels);
CREATE INDEX idx_user_emails_unread ON user_emails(user_id, folder)
  WHERE is_read = FALSE;

-- Attachments metadata
CREATE TABLE attachments (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email_id      UUID NOT NULL REFERENCES emails(id),
  filename      VARCHAR(512) NOT NULL,
  content_type  VARCHAR(255) NOT NULL,
  size_bytes    INT NOT NULL,
  storage_key   VARCHAR(1024) NOT NULL, -- S3/GCS key
  checksum      VARCHAR(64) NOT NULL,   -- SHA-256
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_attachments_email ON attachments(email_id);

-- Labels (user-defined)
CREATE TABLE labels (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL REFERENCES users(id),
  name          VARCHAR(100) NOT NULL,
  color         VARCHAR(7),
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (user_id, name)
);

Scaling Strategies

  • Sharding by user ID: Each user's mailbox lives on a deterministic shard. This is the primary scaling axis.
  • Attachment deduplication: Hash attachments; if the same file is sent to 1000 people, store it once.
  • Search index partitioning: Each user's search index is colocated with their email shard.
  • Read replicas: Heavy read workload (inbox refresh) served by replicas; writes go to the leader.
  • CDN for attachments: Large files served from edge locations.
  • Tiered storage: Recent emails on SSD, older emails on cheaper HDD/object storage.

Trade-offs

Decision Pros Cons
Shard by user ID Queries are single-shard Cross-user queries (admin) require scatter-gather
Separate attachment store (S3) Dedup, CDN, cheap storage Extra hop to fetch, eventual consistency
Async delivery via queue Absorbs spikes, retry logic Email not instant — slight delay
Elasticsearch for search Powerful full-text search Another system to maintain, index lag
Denormalized user_emails table Fast reads per user Data duplication across recipients

Summary

Designing an email service at scale requires separating the write path (sending/receiving) from the read path (inbox/search), using message queues for reliable async processing, object storage for attachments, full-text search engines for queries, and sharding by user for horizontal scale. The key insight is that email is fundamentally a per-user system — almost every query is scoped to a single user, making user-based sharding natural and effective.

Exercises

  1. Build an Email Send Pipeline

    Create a Node.js service with Express that accepts an email send request, validates it, stores the email in a database (SQLite or PostgreSQL), and enqueues it for delivery using BullMQ. Create a worker that processes the queue, simulates SMTP delivery, and updates the email status.

    Hint

    Use BullMQ with Redis for the queue. Create separate 'send' and 'delivery-status' queues. The worker should simulate delivery with a random delay and occasionally fail to test retry logic.

  2. Implement Email Threading

    Build a threading system that groups emails into conversations. Use the In-Reply-To and References headers to link emails. Given a set of 20 emails, correctly group them into threads and display them in chronological order within each thread.

    Hint

    Parse the Message-ID, In-Reply-To, and References headers. Build a tree structure where each email is a node. Use the References header as a fallback when In-Reply-To is missing.

  3. Full-Text Search with Elasticsearch

    Set up Elasticsearch and index 1000 sample emails. Implement search that supports: keyword search in subject and body, filtering by date range, filtering by sender, and highlighting matched terms in results.

    Hint

    Use the @elastic/elasticsearch Node.js client. Create a mapping with text fields for subject and body, keyword fields for from/to, and date fields. Use multi_match queries with highlighting.

Common Problems & Solutions

1 Inbox loading is slow for users with 100,000+ emails
Symptoms: Inbox API response time > 2 seconds for power users. Database CPU spikes during inbox loads.
Cause: The inbox query scans too many rows even with pagination because the index on (user_id, folder, received_at) is not selective enough, or the query plan does a sort on disk for non-indexed label combinations.
Solution: Ensure a covering index on (user_id, folder, received_at DESC) that includes the columns needed for the inbox list view. Consider materialized views for common folder views. Use cursor-based pagination (WHERE received_at < :lastSeen) instead of OFFSET-based pagination.
2 Search index falls behind during high email volume periods
Symptoms: Users search for a recently received email and cannot find it. Search index consumer lag grows during peak hours.
Cause: The search indexer cannot keep up with the email ingestion rate. Elasticsearch bulk indexing is bottlenecked by JVM garbage collection or disk I/O.
Solution: Increase the search indexer consumer parallelism. Use Elasticsearch bulk API with optimal batch sizes (5-15 MB per batch). Implement backpressure — if the indexing queue grows too large, temporarily buffer in Kafka and replay. Consider separate hot/warm Elasticsearch clusters.
3 Attachment storage costs are growing faster than user base
Cause: The same attachment (e.g., a company-wide PDF) is stored once per recipient instead of being deduplicated. Deleted emails still have their attachments stored.
Solution: Implement content-addressable storage: hash each attachment and store it once. Use reference counting to track how many emails reference each attachment. Run a garbage collector that deletes unreferenced attachments after a grace period.
