Standardized Error Responses and Validation

Intuition

Validation is like airport security. Every piece of luggage (request) goes through the same scanners (schemas) before entering the terminal (route handlers). If something is wrong, you get a clear, specific explanation of what failed and why — not just a vague 'access denied.'

Standardized Error Responses and Validation

Introduction

Inconsistent error responses are one of the most common complaints about APIs. When every endpoint returns errors in a different format, clients cannot reliably parse failures. This lesson covers standardized error response formats, input validation strategies using Zod and express-validator, and how to build a validation pipeline that produces clear, actionable error messages.

The Problem with Inconsistent Errors

// Inconsistent error formats across endpoints:
{ "error": "Not found" }
{ "message": "Invalid email", "code": 400 }
{ "errors": ["Name required", "Email invalid"] }
{ "status": "error", "reason": "Unauthorized" }

// Clients need special handling for each endpoint!

Designing a Standard Error Format

Simple Format

interface ApiErrorResponse {
  success: false;
  error: {
    message: string;
    statusCode: number;
    code?: string;           // Machine-readable error code
    details?: unknown;        // Validation errors, etc.
  };
}

RFC 7807 Problem Details (Industry Standard)

interface ProblemDetails {
  type: string;       // URI identifying the error type
  title: string;      // Short human-readable summary
  status: number;     // HTTP status code
  detail?: string;    // Human-readable explanation
  instance?: string;  // URI identifying this occurrence
  [key: string]: any; // Extension fields
}

// Example response:
{
  "type": "https://api.example.com/errors/validation",
  "title": "Validation Error",
  "status": 422,
  "detail": "The request body contains invalid fields",
  "instance": "/api/users",
  "errors": [
    { "field": "email", "message": "Must be a valid email address" },
    { "field": "name", "message": "Must be between 2 and 100 characters" }
  ]
}

Implementing Standardized Error Responses

// src/utils/ApiError.ts
export class ApiError extends Error {
  constructor(
    public statusCode: number,
    message: string,
    public code?: string,
    public details?: unknown
  ) {
    super(message);
    this.name = 'ApiError';
  }

  toJSON(): object {
    return {
      success: false,
      error: {
        message: this.message,
        statusCode: this.statusCode,
        ...(this.code && { code: this.code }),
        ...(this.details && { details: this.details }),
      }
    };
  }
}

// src/middleware/errorHandler.ts
export function errorHandler(err: Error, req: Request, res: Response, next: NextFunction) {
  if (err instanceof ApiError) {
    return res.status(err.statusCode).json(err.toJSON());
  }

  // Zod validation errors
  if (err.name === 'ZodError') {
    return res.status(422).json({
      success: false,
      error: {
        message: 'Validation failed',
        statusCode: 422,
        code: 'VALIDATION_ERROR',
        details: (err as any).errors,
      }
    });
  }

  // Default
  console.error('Unexpected error:', err);
  res.status(500).json({
    success: false,
    error: {
      message: process.env.NODE_ENV === 'production'
        ? 'Internal server error'
        : err.message,
      statusCode: 500,
      code: 'INTERNAL_ERROR',
    }
  });
}

Input Validation with Zod

Zod is a TypeScript-first schema validation library that provides both validation and type inference:

npm install zod

Creating Validation Schemas

import { z } from 'zod';

// User creation schema
const createUserSchema = z.object({
  name: z.string()
    .min(2, 'Name must be at least 2 characters')
    .max(100, 'Name must be at most 100 characters')
    .trim(),
  email: z.string()
    .email('Must be a valid email address')
    .toLowerCase(),
  password: z.string()
    .min(8, 'Password must be at least 8 characters')
    .regex(/[A-Z]/, 'Password must contain at least one uppercase letter')
    .regex(/[0-9]/, 'Password must contain at least one number'),
  role: z.enum(['user', 'admin', 'editor']).default('user'),
  age: z.number().int().min(13).max(150).optional(),
  website: z.string().url().optional(),
});

// TypeScript type is inferred automatically
type CreateUserInput = z.infer<typeof createUserSchema>;
// { name: string; email: string; password: string; role: 'user' | 'admin' | 'editor'; age?: number; website?: string; }

// Update schema (all fields optional)
const updateUserSchema = createUserSchema.partial().omit({ password: true });

// Query parameters schema
const listUsersQuerySchema = z.object({
  page: z.string().transform(Number).pipe(z.number().int().min(1)).default('1'),
  limit: z.string().transform(Number).pipe(z.number().int().min(1).max(100)).default('20'),
  sort: z.enum(['name', 'email', 'createdAt']).default('createdAt'),
  order: z.enum(['asc', 'desc']).default('desc'),
  search: z.string().optional(),
  role: z.enum(['user', 'admin', 'editor']).optional(),
});

Validation Middleware

// src/middleware/validate.ts
import { z, ZodSchema, ZodError } from 'zod';

interface ValidationSchemas {
  body?: ZodSchema;
  query?: ZodSchema;
  params?: ZodSchema;
}

export function validate(schemas: ValidationSchemas) {
  return (req: Request, res: Response, next: NextFunction) => {
    const errors: Array<{ location: string; field: string; message: string }> = [];

    for (const [location, schema] of Object.entries(schemas)) {
      if (!schema) continue;

      const data = (req as any)[location];
      const result = schema.safeParse(data);

      if (!result.success) {
        for (const issue of result.error.issues) {
          errors.push({
            location,
            field: issue.path.join('.'),
            message: issue.message,
          });
        }
      } else {
        // Replace with parsed/transformed data
        (req as any)[location] = result.data;
      }
    }

    if (errors.length > 0) {
      return res.status(422).json({
        success: false,
        error: {
          message: 'Validation failed',
          statusCode: 422,
          code: 'VALIDATION_ERROR',
          details: errors,
        }
      });
    }

    next();
  };
}

Using Validation in Routes

import { validate } from '../middleware/validate';
import { createUserSchema, updateUserSchema, listUsersQuerySchema } from './users.schemas';

const router = Router();

router.get('/',
  validate({ query: listUsersQuerySchema }),
  async (req, res) => {
    // req.query is now validated and typed
    const { page, limit, sort, order, search, role } = req.query;
    const users = await userService.findAll({ page, limit, sort, order, search, role });
    res.json({ success: true, data: users });
  }
);

router.post('/',
  validate({ body: createUserSchema }),
  async (req, res) => {
    // req.body is validated, trimmed, lowercased
    const user = await userService.create(req.body);
    res.status(201).json({ success: true, data: user });
  }
);

router.patch('/:id',
  validate({
    params: z.object({ id: z.string().regex(/^\d+$/, 'ID must be numeric') }),
    body: updateUserSchema,
  }),
  async (req, res) => {
    const user = await userService.update(req.params.id, req.body);
    res.json({ success: true, data: user });
  }
);

Complex Validation Patterns

Cross-Field Validation

const dateRangeSchema = z.object({
  startDate: z.string().datetime(),
  endDate: z.string().datetime(),
}).refine(
  data => new Date(data.startDate) < new Date(data.endDate),
  { message: 'End date must be after start date', path: ['endDate'] }
);

const passwordChangeSchema = z.object({
  currentPassword: z.string(),
  newPassword: z.string().min(8),
  confirmPassword: z.string(),
}).refine(
  data => data.newPassword === data.confirmPassword,
  { message: 'Passwords do not match', path: ['confirmPassword'] }
).refine(
  data => data.currentPassword !== data.newPassword,
  { message: 'New password must be different', path: ['newPassword'] }
);

Discriminated Unions

const notificationSchema = z.discriminatedUnion('type', [
  z.object({
    type: z.literal('email'),
    email: z.string().email(),
    subject: z.string(),
  }),
  z.object({
    type: z.literal('sms'),
    phone: z.string().regex(/^\+\d{10,15}$/),
    message: z.string().max(160),
  }),
  z.object({
    type: z.literal('push'),
    deviceToken: z.string(),
    title: z.string(),
  }),
]);

Common Pitfalls

Validating Only the Request Body

// WRONG: only validating body
router.get('/users', validate({ body: schema }), handler);

// CORRECT: validate query params for GET, params for route params
router.get('/users', validate({ query: listSchema }), handler);
router.get('/users/:id', validate({ params: idSchema }), handler);
router.post('/users', validate({ body: createSchema }), handler);

Exposing Internal Error Details

// WRONG: leaks Prisma error details
catch (err) {
  res.status(500).json({ error: err.message });
  // "Unique constraint failed on the fields: (`email`)"
}

// CORRECT: translate to user-friendly message
catch (err) {
  if (err.code === 'P2002') { // Prisma unique constraint
    throw new ApiError(409, 'A user with this email already exists', 'DUPLICATE_EMAIL');
  }
  throw err;
}

Summary

Standardized error responses and robust input validation are non-negotiable for production APIs. Adopt a consistent error format (ideally RFC 7807), validate all input with Zod schemas, create reusable validation middleware, and translate internal errors to user-friendly messages. This makes your API predictable for clients and secure against malicious input.

Exercises

  1. Implement RFC 7807 Error Responses

    Create an error handling system that returns RFC 7807 Problem Details format. Build custom error classes that produce compliant responses with type URIs, titles, status codes, and detail messages. Include validation error extensions.

    Hint

    Set Content-Type to 'application/problem+json'. The 'type' field should be a URI (can be a URL to your error documentation). Include an 'errors' extension field for validation details.

  2. Complete Zod Validation Pipeline

    Build a validation middleware system with Zod schemas for a blog API: create/update post (title, content, tags, publishedAt), query parameters for listing (pagination, filtering by tag, date range), and route parameters (numeric ID). Include cross-field validation.

    Hint

    Use z.string().transform(Number) for query params that should be numbers. Use .refine() for cross-field validation like date ranges. Use .partial() for update schemas.

  3. Error Code Registry

    Create an error code system where each error has a unique code (USER_NOT_FOUND, DUPLICATE_EMAIL, INVALID_TOKEN, etc.). Map these codes to status codes and default messages. Build middleware that translates internal errors (Prisma, JWT) to your error codes.

    Hint

    Create a Map or object that maps error codes to { statusCode, message } pairs. Create a factory function: ApiError.fromCode('DUPLICATE_EMAIL', { email: 'a@b.com' }).

Common Problems & Solutions

1 Validation errors show cryptic internal messages instead of user-friendly ones
Cause: Using default error messages from the validation library without customizing them.
Solution: Always provide custom error messages in Zod schemas: z.string().min(2, 'Name must be at least 2 characters'). Map technical field names to user-friendly labels.
2 Client cannot distinguish between different error types programmatically
Cause: All errors return the same format with only a human-readable message. No machine-readable error codes.
Solution: Include a 'code' field (e.g., 'VALIDATION_ERROR', 'RESOURCE_NOT_FOUND', 'DUPLICATE_RESOURCE') that clients can switch on. Consistent codes enable programmatic error handling.
