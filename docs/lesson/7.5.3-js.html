Prometheus: Metrics Collection and Grafana Dashboards

Intuition

Prometheus metrics are like the dashboard instruments in a car. The speedometer (gauge) shows your current speed, the odometer (counter) tracks total distance traveled and only goes up, and the trip computer (histogram) tells you the distribution of your speeds over time -- average, fastest, slowest. Just as you would not drive a car without a dashboard, you should not run a production service without metrics. The /metrics endpoint is your dashboard, Prometheus is the computer recording everything, and Grafana is the fancy display screen.

Prometheus: Metrics Collection and Grafana Dashboards

Why Application Metrics Matter

Logs tell you what happened. Metrics tell you how your system is performing right now. While logs are invaluable for debugging specific incidents, metrics give you the high-level view: how many requests per second, what is the 99th percentile latency, how much memory is being used, how many database connections are active.

Prometheus is the industry-standard metrics system. It uses a pull-based model: your application exposes a /metrics endpoint, and Prometheus scrapes it periodically.

┌────────────┐    scrape /metrics    ┌────────────────┐    query     ┌─────────────┐
│  Your App  │ ◄──────────────────── │   Prometheus    │ ◄────────── │   Grafana    │
│  (Node.js) │    every 15-30s       │   (time-series  │             │ (dashboards) │
│            │                       │    database)    │             │              │
│ /metrics   │                       │                │             │              │
└────────────┘                       └────────────────┘             └─────────────┘

     Expose                              Store &                      Visualize
     metrics                             aggregate                    & alert

Installing prom-client

npm install prom-client

prom-client is the official Prometheus client library for Node.js.

Metric Types

Prometheus has four core metric types:

┌────────────────┬──────────────────────────────────────────────────┐
│ Type           │ Description and Use Case                         │
├────────────────┼──────────────────────────────────────────────────┤
│ Counter        │ Monotonically increasing value                   │
│                │ Use: total requests, errors, bytes sent          │
│                │ Operations: inc()                                │
├────────────────┼──────────────────────────────────────────────────┤
│ Gauge          │ Value that can go up or down                     │
│                │ Use: active connections, queue size, temperature │
│                │ Operations: set(), inc(), dec()                  │
├────────────────┼──────────────────────────────────────────────────┤
│ Histogram      │ Samples observations in configurable buckets     │
│                │ Use: request latency, response sizes             │
│                │ Operations: observe()                            │
│                │ Provides: count, sum, bucket distributions       │
├────────────────┼──────────────────────────────────────────────────┤
│ Summary        │ Calculates quantiles over sliding time window    │
│                │ Use: request latency (when you need percentiles) │
│                │ Operations: observe()                            │
│                │ Provides: count, sum, quantile values            │
└────────────────┴──────────────────────────────────────────────────┘

Basic Setup with Default Metrics

import express from 'express';
import client from 'prom-client';

// Collect default Node.js metrics (CPU, memory, event loop, GC)
const register = new client.Registry();
client.collectDefaultMetrics({
  register,
  prefix: 'myapp_',    // prefix all default metrics
  gcDurationBuckets: [0.001, 0.01, 0.1, 1, 2, 5],
});

const app = express();

// Expose metrics endpoint
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});

app.listen(3000);

Default metrics include: - process_cpu_seconds_total -- CPU usage - process_resident_memory_bytes -- Memory usage - nodejs_heap_size_total_bytes -- V8 heap size - nodejs_active_handles_total -- Active libuv handles - nodejs_eventloop_lag_seconds -- Event loop delay - nodejs_gc_duration_seconds -- Garbage collection duration

Counter: Tracking Total Requests and Errors

import client from 'prom-client';

const register = new client.Registry();

// Total HTTP requests counter
const httpRequestsTotal = new client.Counter({
  name: 'http_requests_total',
  help: 'Total number of HTTP requests',
  labelNames: ['method', 'route', 'status_code'],
  registers: [register],
});

// Total errors counter
const errorsTotal = new client.Counter({
  name: 'errors_total',
  help: 'Total number of errors',
  labelNames: ['type', 'module'],
  registers: [register],
});

// Usage in middleware
app.use((req, res, next) => {
  res.on('finish', () => {
    httpRequestsTotal.inc({
      method: req.method,
      route: req.route?.path || req.path,
      status_code: res.statusCode,
    });

    if (res.statusCode >= 500) {
      errorsTotal.inc({ type: 'http_5xx', module: 'api' });
    }
  });
  next();
});

Histogram: Request Latency Distribution

Histograms are the most important metric type for understanding API performance:

const httpRequestDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'Duration of HTTP requests in seconds',
  labelNames: ['method', 'route', 'status_code'],
  buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10],
  registers: [register],
});

// Middleware to time requests
app.use((req, res, next) => {
  const end = httpRequestDuration.startTimer();

  res.on('finish', () => {
    end({
      method: req.method,
      route: req.route?.path || req.path,
      status_code: res.statusCode,
    });
  });

  next();
});

// Custom operation timing
const dbQueryDuration = new client.Histogram({
  name: 'db_query_duration_seconds',
  help: 'Duration of database queries in seconds',
  labelNames: ['operation', 'table'],
  buckets: [0.001, 0.005, 0.01, 0.05, 0.1, 0.5, 1, 5],
  registers: [register],
});

async function findUser(id) {
  const end = dbQueryDuration.startTimer();
  try {
    const user = await db.user.findUnique({ where: { id } });
    end({ operation: 'findUnique', table: 'users' });
    return user;
  } catch (err) {
    end({ operation: 'findUnique', table: 'users' });
    throw err;
  }
}

Summary: Percentile Calculation

const requestDurationSummary = new client.Summary({
  name: 'http_request_duration_summary_seconds',
  help: 'Request duration percentiles',
  labelNames: ['method', 'route'],
  percentiles: [0.5, 0.9, 0.95, 0.99],
  maxAgeSeconds: 300,  // 5-minute sliding window
  ageBuckets: 5,
  registers: [register],
});

Histograms vs Summaries: - Histograms: aggregatable across instances, uses more memory, server-side percentile calculation - Summaries: per-instance percentiles, cannot be aggregated, client-side calculation - Recommendation: use Histograms for most cases

Complete Express Metrics Middleware

import express from 'express';
import client from 'prom-client';

const register = new client.Registry();
client.collectDefaultMetrics({ register, prefix: 'myapp_' });

// Metrics
const httpRequestsTotal = new client.Counter({
  name: 'myapp_http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'route', 'status_code'],
  registers: [register],
});

const httpRequestDuration = new client.Histogram({
  name: 'myapp_http_request_duration_seconds',
  help: 'HTTP request duration in seconds',
  labelNames: ['method', 'route', 'status_code'],
  buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10],
  registers: [register],
});

const httpRequestSize = new client.Histogram({
  name: 'myapp_http_request_size_bytes',
  help: 'HTTP request body size in bytes',
  labelNames: ['method', 'route'],
  buckets: [100, 1000, 10000, 100000, 1000000],
  registers: [register],
});

const activeRequests = new client.Gauge({
  name: 'myapp_http_active_requests',
  help: 'Number of active HTTP requests',
  registers: [register],
});

// Metrics middleware
function metricsMiddleware(req, res, next) {
  activeRequests.inc();
  const end = httpRequestDuration.startTimer();

  // Track request size
  const contentLength = parseInt(req.headers['content-length'] || '0', 10);
  if (contentLength > 0) {
    httpRequestSize.observe(
      { method: req.method, route: req.route?.path || req.path },
      contentLength
    );
  }

  res.on('finish', () => {
    const route = req.route?.path || req.path;
    const labels = {
      method: req.method,
      route,
      status_code: res.statusCode,
    };

    httpRequestsTotal.inc(labels);
    end(labels);
    activeRequests.dec();
  });

  next();
}

// Application
const app = express();
app.use(express.json());
app.use(metricsMiddleware);

// Metrics endpoint (do NOT apply metricsMiddleware to this route)
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});

// Business routes
app.get('/api/users', async (req, res) => {
  const users = await findUsers();
  res.json(users);
});

app.listen(3000);

Useful PromQL Queries for Grafana

# Request rate (requests per second)
rate(myapp_http_requests_total[5m])

# Error rate (percentage of 5xx responses)
sum(rate(myapp_http_requests_total{status_code=~"5.."}[5m]))
/
sum(rate(myapp_http_requests_total[5m])) * 100

# 95th percentile latency
histogram_quantile(0.95, rate(myapp_http_request_duration_seconds_bucket[5m]))

# 99th percentile latency by route
histogram_quantile(0.99,
  sum by (route, le) (rate(myapp_http_request_duration_seconds_bucket[5m]))
)

# Average request duration
rate(myapp_http_request_duration_seconds_sum[5m])
/
rate(myapp_http_request_duration_seconds_count[5m])

# Memory usage
myapp_process_resident_memory_bytes / 1024 / 1024  # in MB

# Event loop lag
myapp_nodejs_eventloop_lag_seconds

# Active connections
myapp_http_active_requests

# Requests per second by status code
sum by (status_code) (rate(myapp_http_requests_total[5m]))

The RED Method for API Monitoring

The RED method defines three essential metrics for every service:

┌─────────────────────────────────────────────────────────┐
│                  The RED Method                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  R - Rate:     How many requests per second?            │
│                rate(http_requests_total[5m])             │
│                                                         │
│  E - Errors:   How many of those requests are failing?  │
│                rate(http_requests_total{status=~"5.."}   │
│                [5m])                                     │
│                                                         │
│  D - Duration: How long do the requests take?           │
│                histogram_quantile(0.99,                  │
│                rate(http_request_duration_seconds_bucket │
│                [5m]))                                    │
│                                                         │
│  These three metrics tell you 80% of what you need      │
│  to know about your service's health.                   │
└─────────────────────────────────────────────────────────┘

Label Best Practices

// GOOD: Low cardinality labels
httpRequestsTotal.inc({ method: 'GET', route: '/api/users', status_code: 200 });

// BAD: High cardinality labels (creates too many time series)
httpRequestsTotal.inc({
  method: 'GET',
  url: '/api/users/a1b2c3d4-e5f6-7890',  // unique per request!
  userId: 'u_abc123',                      // unique per user!
});

// Rule of thumb:
// - method: ~5 values (GET, POST, PUT, DELETE, PATCH)
// - route: ~20-50 values (your route patterns)
// - status_code: ~10-20 values (200, 201, 400, 404, 500...)
// - NEVER use user IDs, request IDs, or timestamps as labels

Summary

Prometheus with prom-client gives your Node.js application production-grade observability. Use counters for totals (requests, errors), gauges for current values (connections, queue size), and histograms for distributions (latency, response size). Follow the RED method for core API metrics, use low-cardinality labels, expose a /metrics endpoint, and visualize everything in Grafana. This combination is the industry standard for monitoring backend services.

Exercises

Expose a /metrics endpoint. Create a load test script using a loop of fetch() calls to generate traffic, then verify the metrics output.

    Hint

    Use histogram.startTimer() which returns a function that records the duration when called. Use route patterns (req.route?.path) instead of full URLs to keep label cardinality low.

  2. Custom Business Metrics Dashboard

    Add business-level metrics to a sample e-commerce API: orders placed (counter with payment method label), order value (histogram with dollar amount buckets), active shopping carts (gauge), and cache hit ratio (gauge). Write a script that simulates traffic and generates realistic metric data. Document the PromQL queries you would use in Grafana for each metric.

    Hint

    For the cache hit ratio, maintain two counters (hits and total) and calculate the ratio in PromQL: rate(cache_hits[5m]) / rate(cache_total[5m]).

  3. Prometheus + Grafana Stack

    Set up a complete monitoring stack with Docker Compose: your Node.js app with prom-client, Prometheus scraping the /metrics endpoint, and Grafana with a pre-configured dashboard. The dashboard should show: request rate, error rate percentage, p50/p95/p99 latency, memory usage, event loop lag, and active connections. Export the Grafana dashboard JSON.

    Hint

    Use Grafana's provisioning to auto-configure the Prometheus data source. Use histogram_quantile() for latency percentiles. Set appropriate time ranges (5m for rates, 1m for gauges).

  4. Alerting Rules

    Write Prometheus alerting rules for: error rate above 5% for 5 minutes, p99 latency above 2 seconds for 10 minutes, memory usage above 80% of limit, and event loop lag above 100ms. Configure these in a prometheus rules file and test them by intentionally degrading your application.

    Hint

    Use recording rules to pre-compute expensive PromQL. Alert rules use the for clause to avoid flapping. Use labels and annotations for alert metadata.

Common Problems & Solutions

1 Prometheus scrape target shows as DOWN in the targets page
Cause: Prometheus cannot reach the /metrics endpoint. Common reasons: wrong hostname/port in prometheus.yml, Docker networking issues (using localhost instead of container name), or the app is not exposing /metrics.
Solution: Verify the app serves /metrics by curling it directly: `curl http://localhost:3000/metrics`. In Docker Compose, use the service name as the hostname (e.g., 'app:3000' not 'localhost:3000'). Check Prometheus logs for connection errors.
2 High cardinality causing Prometheus to run out of memory
Cause: Labels contain high-cardinality values like user IDs, request IDs, or full URLs with path parameters (e.g., /api/users/abc123 instead of /api/users/:id).
Solution: Replace dynamic path segments with route patterns: use req.route?.path (e.g., '/api/users/:id') instead of req.originalUrl. Never use user IDs, session IDs, or timestamps as label values. Monitor Prometheus' own metrics: `prometheus_tsdb_head_series` shows total time series count.
3 Histogram percentiles are inaccurate or show unexpected values
Cause: The histogram bucket boundaries do not cover the actual range of observed values. If most requests take 50-200ms but your smallest bucket is 1s, all observations land in the same bucket.
Solution: Choose bucket boundaries that match your expected distribution. For HTTP latency: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10]. Use histogram_quantile() with rate() for accurate percentile calculation: `histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))`.
