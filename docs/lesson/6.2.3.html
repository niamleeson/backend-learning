Subscriptions, DataLoaders, and N+1 Prevention

Intuition

The N+1 problem is like a teacher taking attendance by walking to each student's house one at a time. A DataLoader is like the teacher sending one group text: 'Students 1, 5, 12, and 23, please confirm attendance.' One round trip instead of many.

Subscriptions, DataLoaders, and N+1 Prevention

GraphQL subscriptions enable real-time updates, while DataLoaders solve the notorious N+1 query problem that plagues naive GraphQL implementations.

Subscriptions with Strawberry

import strawberry
import asyncio
from typing import AsyncGenerator

class Notification:
    message: str

@strawberry.type
class Subscription:
    @strawberry.subscription
    async def notifications(self, channel: str) -> AsyncGenerator[Notification, None]:
        # In production, use Redis Pub/Sub or similar

            msg = await get_next_message(channel)

DataLoader for N+1 Prevention

from strawberry.dataloader import DataLoader

async def load_users(user_ids: list[int]) -> list[User]:
    # Single batch query instead of N individual queries
    users = await db.fetch_all(
        "SELECT * FROM users WHERE id = ANY(:ids)",
        {"ids": user_ids}
    )
    user_map = {u.id: u for u in users}
    return [user_map.get(uid) for uid in user_ids]

user_loader = DataLoader(load_fn=load_users)

@strawberry.type
class Post:
    id: int
    title: str
    author_id: int

    @strawberry.field
    async def author(self, info) -> User:
        # This is called N times but batched into 1 query
        return await info.context["user_loader"].load(self.author_id)

Common Mistakes

  • Creating DataLoader instances globally instead of per-request (stale cache)
  • Returning results from the batch function in a different order than the input keys
  • Not handling missing keys in the batch function (returning None for missing IDs)

Key Takeaways

  • DataLoaders batch and cache individual loads within a single request
  • Always create new DataLoader instances per request to avoid stale data
  • The batch function must return results in the same order as input keys

Exercises

  1. Implement a DataLoader

    Create a GraphQL API with Post and Author types. When querying a list of posts with their authors, ensure that the author resolver uses a DataLoader to batch all author lookups into a single database query. Log queries to verify batching.

    Hint

    The load_fn receives a list of IDs and must return a list of results in the same order. Use a dict for O(1) lookups after the batch query.

  2. Real-Time Chat Subscription

    Build a chat application with a GraphQL subscription that pushes new messages to connected clients. Use an async generator and an in-memory message queue. Include a mutation to send messages.

    Hint

    Use asyncio.Queue as a simple message broker. Each subscription creates a new queue that receives messages when the mutation publishes them.

  3. Detect N+1 Queries

    Write a middleware or test that detects N+1 query patterns by counting database calls during a single GraphQL request. Raise a warning if the count exceeds a threshold.

    Hint

    Wrap your database client to increment a counter on each query. Check the counter after each request and log a warning if it exceeds len(results) + 1.

Common Problems & Solutions

1 N+1 queries make GraphQL API extremely slow
Symptoms: ['Response time scales linearly with list size', 'Database connection pool exhaustion under load']
Cause: Each nested resolver issues its own database query. A list of 50 posts with authors results in 51 queries (1 for posts + 50 for authors).
Solution: Use DataLoader to batch and deduplicate database calls. All author IDs are collected and fetched in a single query.

Without DataLoader: 51 queries

Query 1: SELECT * FROM posts LIMIT 50

Query 2-51: SELECT * FROM users WHERE id = :id (x50)

With DataLoader: 2 queries

Query 1: SELECT * FROM posts LIMIT 50

Query 2: SELECT * FROM users WHERE id = ANY(:ids)

2 DataLoader returns wrong results for some keys
Cause: The batch function returns results in a different order than the input keys, or does not handle missing keys.
Solution: Build a lookup dictionary from the batch results, then return a list comprehension that maps each input key to its result: [user_map.get(id) for id in ids].
