API Keys and Rate Limiting

Intuition

API keys are like building access cards that identify who is making requests, while rate limiting is like a turnstile that controls how fast people can enter. Together they answer both 'who are you?' and 'are you going too fast?'.

API Keys and Rate Limiting

API keys provide simple authentication for service-to-service communication, while rate limiting protects your API from abuse. Together, they form a basic but effective security layer.

from fastapi import FastAPI, Depends, HTTPException, Security
from fastapi.security import APIKeyHeader
from collections import defaultdict
import time

app = FastAPI()
api_key_header = APIKeyHeader(name="X-API-Key")

VALID_API_KEYS = {"key-abc123": "service-a", "key-xyz789": "service-b"}

async def verify_api_key(api_key: str = Security(api_key_header)):
    if api_key not in VALID_API_KEYS:
        raise HTTPException(status_code=403, detail="Invalid API key")
    return VALID_API_KEYS[api_key]

# Simple in-memory rate limiter
rate_limit_store: dict[str, list[float]] = defaultdict(list)

async def rate_limiter(
    api_key: str = Security(api_key_header)
):
    now = time.time()
    window = 60  # 1 minute
    max_requests = 10
    requests = rate_limit_store[api_key]
    rate_limit_store[api_key] = [t for t in requests if now - t < window]
    if len(rate_limit_store[api_key]) >= max_requests:
        raise HTTPException(status_code=429, detail="Rate limit exceeded")
    rate_limit_store[api_key].append(now)

@app.get("/data", dependencies=[Depends(rate_limiter)])
async def get_data(client: str = Depends(verify_api_key)):
    return {"client": client, "data": [1, 2, 3]}

Key Takeaways: - APIKeyHeader extracts keys from request headers; APIKeyQuery from query params - Rate limiting with sliding windows prevents burst abuse - In-memory rate limiting works for single instances; use Redis for distributed systems - Return 429 status code with Retry-After header for rate-limited requests

Exercises

  1. Tiered Rate Limiting

    Implement rate limiting with different tiers based on the API key: free keys get 10 requests/minute, premium keys get 100 requests/minute. Store the tier information alongside the API key and apply the correct limit.

    Hint

    Extend VALID_API_KEYS to store {"client": "name", "tier": "premium", "limit": 100}. In the rate limiter, look up the tier-specific limit for the current API key.

  2. Redis-Based Rate Limiter

    Replace the in-memory rate limiter with a Redis-based sliding window implementation using redis.asyncio. Use a sorted set with timestamps as scores to track requests per key.

    Hint

    Use ZADD to add request timestamps, ZREMRANGEBYSCORE to remove old entries, and ZCARD to count requests in the window. Wrap it in a FastAPI dependency.

Common Problems & Solutions

1 Rate limiter resets on server restart
Cause: The rate limit state is stored in memory using a Python dictionary, which is lost when the process restarts.
Solution: Use an external store like Redis to persist rate limit counters. Redis TTL can automatically expire old entries, and it supports atomic operations for accurate counting across multiple server instances.
2 API key visible in server logs and browser history
Cause: The API key is being passed as a query parameter instead of a header, making it visible in URLs logged by proxies and browsers.
Solution: Use `APIKeyHeader` instead of `APIKeyQuery` to transmit keys in headers. Headers are not logged by default in most web servers and don't appear in browser history.
