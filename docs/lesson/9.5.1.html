Service Decomposition and Bounded Contexts

Intuition

Bounded contexts are like departments in a company. The word 'order' means different things in Sales (a customer purchase) and Manufacturing (a production batch). Each department has its own processes and vocabulary, with clear interfaces for communication.

Service Decomposition & Bounded Contexts

Microservices architecture starts with properly decomposing a system into services. Domain-Driven Design's bounded contexts provide the intellectual framework for deciding where to draw service boundaries.

Core Concepts

  • Bounded Context: A boundary within which a domain model is consistent and unified. Each bounded context has its own ubiquitous language—the same word (e.g., "account") can mean different things in different contexts.
  • Decomposition Strategies: Decompose by business capability (payments, inventory, shipping), by subdomain (core, supporting, generic), or by team ownership (Conway's Law).
  • Context Mapping: Define relationships between bounded contexts: shared kernel, customer-supplier, conformist, anti-corruption layer, or separate ways.
  • Anti-Corruption Layer (ACL): A translation layer that prevents one service's domain model from leaking into another. Essential when integrating with legacy systems or third-party APIs.
  • Strangler Fig Pattern: Incrementally migrate from a monolith by routing new features to microservices while the monolith handles existing functionality, gradually replacing it.

Key Takeaways

  1. Service boundaries should align with bounded contexts, not technical layers (no "database service" or "auth service" that everything depends on).
  2. If two teams must constantly coordinate to ship features, their services are probably too tightly coupled or the boundary is wrong.
  3. Start with a monolith, identify bounded contexts, and extract services only when the benefits outweigh the distributed system complexity.

Exercises

  1. Map Bounded Contexts for an E-Commerce System

    Given an e-commerce monolith, identify at least 5 bounded contexts (e.g., Catalog, Ordering, Payments, Shipping, Identity). For each, define the key entities, their relationships within the context, and the integration events between contexts.

    Hint

    Start with Event Storming: list all domain events (OrderPlaced, PaymentProcessed, ItemShipped), then group them by context. Entities like Product exist in multiple contexts but with different attributes.

  2. Implement an Anti-Corruption Layer

    Build an ACL that translates between a legacy REST API (with camelCase fields, XML responses, and custom error codes) and your internal domain model (with snake_case, JSON, and standard exceptions). Include mapping, validation, and error translation.

    Hint

    Create an adapter class with methods like get_user(id) that calls the legacy API, parses XML, maps fields to your domain model, and translates errors. Use the Repository or Gateway pattern.

Common Problems & Solutions

1 Two services frequently change together—a feature requires coordinated deployments.
Cause: The service boundary cuts through a bounded context, creating tight coupling. Or shared database tables create implicit coupling.
Solution: Re-evaluate the boundary using domain analysis. Consider merging the services or restructuring so each owns its data completely. Replace shared databases with APIs or events.
2 Extracting a microservice from the monolith causes data inconsistency between the two systems.
Cause: The monolith and new service both read/write the same database tables during the migration, leading to split-brain scenarios.
Solution: Use the Strangler Fig pattern: route traffic gradually, use change data capture (CDC) to sync data during the transition, and enforce a single writer per entity at all times.
