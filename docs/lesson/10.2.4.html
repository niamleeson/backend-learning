Design a Search Engine (Google)

Intuition

A search engine is a pipeline that transforms the web into an inverted index and uses ranking algorithms to surface the most relevant documents for any query. The inverted index is the fundamental data structure that makes sub-second search over billions of documents possible.

Design a Search Engine (Google)

Requirements

  • Crawl billions of web pages and build a searchable index
  • Return relevant results for keyword queries in under 200ms
  • Rank results by relevance using link analysis and content signals
  • Handle typos, synonyms, and natural language queries
  • Update the index continuously as web content changes

Architecture

A web crawler discovers and fetches pages. A document processor extracts text, normalizes content, and builds an inverted index. The index is sharded across thousands of machines. A query engine parses queries, searches relevant shards, merges results, and ranks them. A PageRank-like algorithm provides authority scores.

Key Components

  • Web Crawler: Distributed crawler with URL frontier, politeness policies, and deduplication
  • Inverted Index: Maps terms to sorted lists of (document_id, term_frequency, positions)
  • Query Parser: Handles boolean operators, phrases, spelling correction, and query expansion
  • Ranking Engine: Combines TF-IDF, PageRank, freshness, and user engagement signals
  • Index Sharding: Partition index by document ID (document-partitioned) or by term (term-partitioned)

Scaling

Document-partitioned indexing allows each shard to independently score documents. Fan-out queries to all shards and merge top-K results. Use tiered indexing: a fast tier for popular pages and a full tier for comprehensive results. Cache frequent queries.

Trade-offs

  • Document-partitioned sharding enables independent scoring but requires querying all shards; term-partitioned reduces fan-out but complicates ranking
  • Fresher index requires more crawling resources; stale index saves resources but returns outdated results

Exercises

  1. Build an Inverted Index

    Create a system that processes a collection of documents, tokenizes them, removes stop words, applies stemming, and builds an inverted index. Support boolean AND/OR queries and phrase matching using positional indexes.

    Hint

    Store each index entry as: term -> [(doc_id, [position1, position2, ...]), ...]. For phrase queries, check that positions in consecutive terms differ by exactly 1.

  2. Implement a Simple PageRank

    Build a simplified PageRank algorithm over a graph of web pages. Iteratively compute authority scores where each page distributes its rank equally among outgoing links. Include a damping factor of 0.85. Run until convergence.

    Hint

    Initialize all pages with rank 1/N. Each iteration: new_rank = (1-d)/N + d * sum(rank[source]/outlinks[source] for source in inlinks). Repeat until the maximum rank change is below a threshold.

Common Problems & Solutions

1 Query latency increases as the index grows, with tail latencies exceeding 2 seconds for rare term queries.
Cause: Rare terms have long posting lists spread across many shards, requiring full fan-out and merging of large result sets.
Solution: Use tiered indexing with a small fast tier containing popular documents queried first. Only hit the full index if the fast tier has insufficient results. Cache posting lists for frequent terms in memory.
2 The web crawler gets trapped in infinite loops or spider traps, wasting resources on low-value pages.
Cause: Dynamic websites generate infinite URL variations (e.g., calendar pages, session IDs in URLs) that appear as unique pages.
Solution: Implement URL normalization to canonicalize URLs. Set a maximum depth per domain. Use a URL deduplication bloom filter. Detect and blacklist spider traps by monitoring crawl patterns per domain.
