Threading: Threads, Locks, and the GIL

Intuition

Threads are like cooks sharing one kitchen. The GIL means only one cook can use the stove at a time, but while one waits for water to boil (I/O), another can chop vegetables.

Threading: Threads, Locks, and the GIL

Threads let you run multiple tasks concurrently within a single process. Python's GIL limits true parallelism for CPU-bound work, but threads excel at I/O-bound tasks.

import threading
import time

counter = 0
lock = threading.Lock()

def increment(n: int) -> None:
    global counter
    for _ in range(n):
        with lock:  # Prevent race conditions
            counter += 1

threads = []
for _ in range(4):
    t = threading.Thread(target=increment, args=(100_000,))
    threads.append(t)
    t.start()

for t in threads:
    t.join()

print(f"Counter: {counter}")  # Always 400000 with lock

# Using threading.Event for signaling
stop_event = threading.Event()

def worker(event: threading.Event) -> None:
    while not event.is_set():
        print("Working...")
        time.sleep(0.5)

t = threading.Thread(target=worker, args=(stop_event,))
t.start()
time.sleep(2)
stop_event.set()  # Signal the thread to stop
t.join()

Key Takeaways: - The GIL prevents multiple threads from executing Python bytecode simultaneously - Threads are ideal for I/O-bound work (network, file, database operations) - Always use Lock when multiple threads access shared mutable state - Use Event, Condition, and Semaphore for thread coordination - thread.join() blocks until the thread completes

Exercises

    Implement a SafeCounter class with increment(), decrement(), and value property. Use a Lock internally. Verify thread safety by spawning 10 threads.

    Hint

    Use threading.Lock() as an instance variable. Wrap every read and write of the internal count in with self._lock:.

  2. Producer-Consumer with Queue

    Use threading.Thread and queue.Queue to implement a producer-consumer pattern. One producer adds items, two consumers process them.

    Hint

    queue.Queue is thread-safe. Use queue.get() (blocks until item available) and queue.task_done(). Signal completion with sentinel values.

Common Problems & Solutions

1 Race condition: counter value is less than expected without a lock.
Cause: Multiple threads read-modify-write the same variable, and operations interleave unpredictably.
Solution: Protect shared mutable state with `threading.Lock()`. Use `with lock:` context manager for automatic release.
2 Deadlock: program hangs forever with multiple locks.
Cause: Two threads each hold one lock and wait for the other, creating a circular dependency.
Solution: Always acquire locks in a consistent global order. Use `lock.acquire(timeout=5)` to detect deadlocks, or use `threading.RLock` for reentrant locking.
