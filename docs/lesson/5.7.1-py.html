Project: Full CRUD API with Auth and Pagination

Intuition

A production CRUD API is like a well-organized filing cabinet: authentication checks who can open it, authorization controls which drawers they access, and pagination ensures you only pull one folder at a time instead of dumping everything on the floor.

Full CRUD API with Authentication and Pagination

This capstone combines everything: a complete REST API with CRUD operations, JWT authentication, role-based access, and cursor-based pagination for production-ready data access.

from fastapi import FastAPI, Depends, Query, HTTPException, status
from pydantic import BaseModel

app = FastAPI(title="Task Manager API", version="1.0.0")

class TaskCreate(BaseModel):
    title: str
    description: str | None = None
    priority: int = 1

class TaskResponse(BaseModel):
    id: int
    title: str
    description: str | None
    priority: int
    owner_id: int
    model_config = {"from_attributes": True}

class PaginatedResponse(BaseModel):
    items: list[TaskResponse]
    total: int
    page: int
    pages: int

@app.get("/tasks", response_model=PaginatedResponse)
async def list_tasks(
    page: int = Query(1, ge=1),
    size: int = Query(20, ge=1, le=100),
    user=Depends(get_current_user),
    db=Depends(get_db),
):
    offset = (page - 1) * size
    query = select(Task).where(Task.owner_id == user.id)
    total = await db.scalar(select(func.count()).select_from(query))
    tasks = (await db.execute(query.offset(offset).limit(size))).scalars().all()
    return PaginatedResponse(
        items=tasks, total=total, page=page, pages=-(-total // size)
    )

Key Takeaways: - Pagination prevents loading entire datasets; use offset/limit or cursor-based approaches - Always scope queries to the authenticated user for data isolation - Return total count and page metadata for frontend pagination controls - Use dependency injection to layer auth, DB session, and pagination cleanly

Exercises

  1. Cursor-Based Pagination

    Replace offset pagination with cursor-based pagination using the task ID as the cursor. Return a next_cursor in the response that clients pass to get the next page. This performs better for large datasets.

    Hint

    Use select(Task).where(Task.id > cursor).limit(size + 1). If you get size + 1 results, there is a next page and the cursor is the last item's ID. Return only size items.

  2. Bulk Operations

    Add endpoints for bulk create (POST /tasks/bulk with a list of tasks), bulk update (change priority for multiple task IDs), and bulk delete (delete multiple tasks by ID list). Ensure all operations are atomic.

    Hint

    Use session.add_all() for bulk create. For bulk update, use update(Task).where(Task.id.in_(ids)).values(priority=new_priority). Wrap each bulk operation in a transaction.

Common Problems & Solutions

1 Offset pagination returns duplicate or missing items
Cause: Items are being inserted or deleted between paginated requests, causing the offset to shift and skip or repeat items.
Solution: Use cursor-based pagination with a stable sort key (like auto-incrementing ID or created_at timestamp). This anchors the position to a specific record rather than a numeric offset.
2 User can access other users' tasks by guessing IDs
Cause: The endpoint fetches tasks by ID without checking if the task belongs to the authenticated user.
Solution: Always include an ownership check in queries: `select(Task).where(Task.id == task_id, Task.owner_id == user.id)`. Return 404 instead of 403 to avoid revealing that the resource exists.
