Design a Chat System (WhatsApp)

Intuition

A chat system is a real-time message routing network where persistent connections enable instant delivery and a store-and-forward mechanism handles offline users. The core challenge is maintaining millions of stateful WebSocket connections while ensuring no messages are lost.

Design a Chat System (WhatsApp)

Requirements

  • Support 1:1 and group messaging with real-time delivery
  • Handle online/offline status, read receipts, and typing indicators
  • Store message history and sync across devices
  • Support media sharing (images, videos, documents)
  • Scale to billions of messages per day

Architecture

Clients maintain persistent WebSocket connections to chat servers. A connection manager tracks which user is connected to which server. Messages are routed through a message service that stores them in a database and forwards to recipients. Offline messages are queued for later delivery. Media is uploaded to object storage with CDN.

Key Components

  • WebSocket Servers: Maintain persistent bidirectional connections with clients
  • Connection Manager: Maps user IDs to WebSocket server addresses (Redis)
  • Message Service: Persists messages and routes them to recipient connections
  • Group Service: Manages group membership and fans out messages to members
  • Media Service: Handles upload, compression, thumbnail generation, and CDN distribution

Scaling

Partition WebSocket servers by user ID ranges. Use Kafka for async message fanout in large groups. Shard message storage by conversation ID. Use a push notification fallback when WebSocket is disconnected.

Trade-offs

  • WebSockets provide real-time delivery but require stateful connections and sticky routing
  • Fan-out on write (pre-deliver to all members) is fast for reads but expensive for large groups

Exercises

  1. Implement a Message Sync Protocol

    Design a protocol where clients track the last message sequence number they received. On reconnection, the client sends its last sequence number and the server sends all missed messages. Handle the case where history is too large by paginating.

    Hint

    Assign monotonically increasing sequence numbers per conversation. Store messages in order and use the sequence number as a cursor for efficient range queries.

  2. Design Group Message Fanout

    Build a system that delivers a message to all members of a group chat. For small groups (< 100), fan out on write. For large groups (100+), fan out on read. Implement both strategies and a routing decision layer.

    Hint

    For write fanout, insert a message copy into each member's inbox. For read fanout, store one copy and have members pull from the group timeline on demand.

Common Problems & Solutions

1 Messages are delivered out of order or lost when the WebSocket server crashes mid-delivery.
Cause: The server acknowledged receipt from the sender but crashed before forwarding to the recipient, and the message was only in memory.
Solution: Persist messages to the database before acknowledging to the sender. Use message sequence numbers per conversation to detect gaps. Clients request missing messages on reconnection.
2 Group chats with thousands of members cause massive write amplification and high latency.
Cause: Fan-out on write creates a database entry per member per message, overwhelming write capacity for large groups.
Solution: Switch to fan-out on read for groups above a threshold. Store one copy of the message and have members pull updates. Use a hybrid approach where active members get push delivery and inactive members pull on open.
