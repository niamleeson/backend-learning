ioredis: Async Client, Pipelining, and Cluster Mode

Intuition

Pipelining is like sending a shopping list to the store instead of making a separate trip for each item. You hand over the whole list, the store gathers everything, and sends it all back in one delivery. Cluster mode is like having multiple stores in different neighborhoods, each responsible for certain products, so no single store gets overwhelmed.

ioredis: Async Client, Pipelining, and Cluster Mode

Introduction

ioredis is the most popular Redis client for Node.js. Beyond basic get/set operations, it provides pipelining (batch commands), cluster support, sentinel support, and advanced features that are essential for production Redis usage.

Pipelining

Pipelining sends multiple commands to Redis without waiting for each response, dramatically reducing network round trips.

import Redis from 'ioredis';
const redis = new Redis();

// Without pipelining: 100 round trips
for (let i = 0; i < 100; i++) {
  await redis.set(`key:${i}`, `value:${i}`); // each waits for response
}

// With pipelining: 1 round trip for all 100 commands
const pipeline = redis.pipeline();
for (let i = 0; i < 100; i++) {
  pipeline.set(`key:${i}`, `value:${i}`);
}
const results = await pipeline.exec();
// results: [[null, 'OK'], [null, 'OK'], ...]
// Each entry: [error, result]

Without Pipeline:           With Pipeline:
  CMD1 -> wait -> resp      CMD1 |
  CMD2 -> wait -> resp      CMD2 |-> all sent at once
  CMD3 -> wait -> resp      CMD3 |
  ~300ms (3 round trips)    RESP1, RESP2, RESP3 (all at once)
                            ~3ms (1 round trip)

Pipeline with Mixed Commands

const pipe = redis.pipeline();
pipe.get('user:1:name');
pipe.hgetall('user:1:profile');
pipe.zrevrange('user:1:scores', 0, 4, 'WITHSCORES');
pipe.scard('user:1:followers');
pipe.ttl('session:user:1');

const results = await pipe.exec();
const [name, profile, scores, followerCount, sessionTtl] = results!.map(([err, val]) => val);

Transactions (MULTI/EXEC)

Transactions ensure a group of commands execute atomically:

const result = await redis.multi()
  .set('key1', 'val1')
  .set('key2', 'val2')
  .incr('counter')
  .exec();
// All commands execute atomically
// result: [[null, 'OK'], [null, 'OK'], [null, 42]]

// With optimistic locking (WATCH)
async function transferPoints(fromUser: string, toUser: string, amount: number) {
  const fromKey = `points:${fromUser}`;
  const toKey = `points:${toUser}`;

  // Watch keys for changes
  await redis.watch(fromKey, toKey);

  const fromPoints = parseInt(await redis.get(fromKey) || '0');
  if (fromPoints < amount) {
    await redis.unwatch();
    throw new Error('Insufficient points');
  }

  // If any watched key changed since WATCH, EXEC returns null
  const result = await redis.multi()
    .decrby(fromKey, amount)
    .incrby(toKey, amount)
    .exec();

  if (!result) {
    throw new Error('Transaction aborted - retry');
  }
}

Redis Cluster

import Redis from 'ioredis';

const cluster = new Redis.Cluster([
  { host: 'redis-node-1', port: 6379 },
  { host: 'redis-node-2', port: 6379 },
  { host: 'redis-node-3', port: 6379 },
], {
  redisOptions: {
    password: process.env.REDIS_PASSWORD,
  },
  scaleReads: 'slave', // read from replicas
  clusterRetryStrategy(times) {
    return Math.min(times * 100, 3000);
  },
});

// Usage is the same as single-node Redis
await cluster.set('key', 'value');
const val = await cluster.get('key');

Connection Events and Error Handling

const redis = new Redis({
  retryStrategy(times) {
    if (times > 10) return null; // stop retrying
    return Math.min(times * 200, 5000);
  },
  reconnectOnError(err) {
    return err.message.includes('READONLY');
  },
  lazyConnect: true, // don't connect until first command
  enableOfflineQueue: true, // queue commands while disconnected
});

redis.on('connect', () => console.log('Connected to Redis'));
redis.on('ready', () => console.log('Redis ready'));
redis.on('error', (err) => console.error('Redis error:', err));
redis.on('close', () => console.log('Redis connection closed'));
redis.on('reconnecting', (ms) => console.log(`Reconnecting in ${ms}ms`));

await redis.connect(); // needed with lazyConnect: true

Performance Best Practices

// 1. Use pipelines for batch operations
const pipe = redis.pipeline();
ids.forEach(id => pipe.get(`user:${id}`));
const results = await pipe.exec();

// 2. Use MGET/MSET for simple batch get/set
const values = await redis.mget(...keys);

// 3. Avoid KEYS in production (use SCAN)
let cursor = '0';
do {
  const [next, keys] = await redis.scan(cursor, 'MATCH', 'prefix:*', 'COUNT', 100);
  cursor = next;
  // process keys
} while (cursor !== '0');

// 4. Set maxmemory and eviction policy
// redis.conf: maxmemory 256mb, maxmemory-policy allkeys-lru

// 5. Monitor with INFO
const info = await redis.info('memory');
console.log(info); // used_memory, maxmemory, etc.

Graceful Shutdown

async function shutdown() {
  console.log('Closing Redis connections...');
  await redis.quit(); // wait for pending commands, then close
  // NOT redis.disconnect() which drops pending commands
  process.exit(0);
}

process.on('SIGTERM', shutdown);

Common Problems & Solutions

1 Pipeline errors are silently swallowed
Cause: Pipeline exec() returns an array where each entry is [error, result]. If you don't check the error, failures are silent.
Solution: Always check for errors in pipeline results: `results.forEach(([err, val]) => { if (err) throw err; })`. Or write a helper that throws on any error.
2 CROSSSLOT error in Redis Cluster
Cause: A multi-key operation (MGET, pipeline) accesses keys on different hash slots in a cluster. Redis Cluster requires all keys in a transaction to be on the same node.
Solution: Use hash tags to force keys to the same slot: `{user:1}:name` and `{user:1}:email` will be on the same node because the hash is computed from `user:1` only.
3 Connection pool exhaustion under high load
Cause: Not enough connections for the number of concurrent operations. Or connections are leaked by not handling errors.
Solution: For single-node: ioredis uses a single connection by default (fine for most cases due to pipelining). For cluster: each node gets its own connection. If you need more throughput, create multiple Redis instances.
