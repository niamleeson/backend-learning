Apollo Server: Resolvers, Context, and Directives

Intuition

Think of resolvers as a team of specialized workers on an assembly line. The parent resolver hands off its result to child resolvers, who each do their specific job (fetch comments, look up the author, etc.). Context is the shared toolbox everyone on the line has access to -- the wrench (database), the badge scanner (auth), and the parts catalog (DataLoader).

Apollo Server: Resolvers, Context, and Directives

Apollo Server is the most popular GraphQL server implementation for Node.js. This lesson covers advanced resolver patterns, context management for authentication, and custom directives for reusable schema logic.

Resolver Architecture

Every field in a GraphQL schema has a resolver. Resolvers follow a specific signature:

type Resolver = (
  parent: any,       // Result from the parent resolver
  args: any,         // Arguments passed to the field
  context: Context,  // Shared per-request context
  info: GraphQLResolveInfo // Query AST and metadata
) => any;

Default Resolvers

Apollo uses default resolvers for scalar fields. If your User type has a name field, and the parent resolver returns { name: 'Alice' }, Apollo auto-resolves name without an explicit resolver.

// You do NOT need this:
const resolvers = {
  User: {
    name: (parent) => parent.name, // Apollo does this automatically
  },
};

// You ONLY need resolvers for:
// 1. Computed fields
// 2. Relationship fields that require DB queries
// 3. Fields that transform the parent's data

Context: Per-Request Shared State

Context is created fresh for each request. It is the place for: - Authenticated user info - Database connections - DataLoaders - Request-scoped services

import { ApolloServer } from '@apollo/server';
import { expressMiddleware } from '@apollo/server/express4';
import jwt from 'jsonwebtoken';

interface Context {
  userId: string | null;
  userRole: string | null;
  prisma: PrismaClient;
  loaders: ReturnType<typeof createLoaders>;
}

const server = new ApolloServer<Context>({
  typeDefs,
  resolvers,
});

await server.start();

app.use(
  '/graphql',
  express.json(),
  expressMiddleware(server, {
    context: async ({ req }): Promise<Context> => {
      // Extract user from JWT token
      let userId: string | null = null;
      let userRole: string | null = null;

      const token = req.headers.authorization?.replace('Bearer ', '');
      if (token) {
        try {
          const decoded = jwt.verify(token, process.env.JWT_SECRET!) as any;
          userId = decoded.sub;
          userRole = decoded.role;
        } catch {
          // Invalid token - continue as unauthenticated
        }
      }

      return {
        userId,
        userRole,
        prisma,
        loaders: createLoaders(prisma),
      };
    },
  })
);

Using Context in Resolvers

const resolvers = {
  Query: {
    me: async (_parent, _args, context: Context) => {
      if (!context.userId) return null;
      return context.prisma.user.findUnique({
        where: { id: context.userId },
      });
    },

    adminDashboard: async (_parent, _args, context: Context) => {
      if (context.userRole !== 'ADMIN') {
        throw new GraphQLError('Admin access required', {
          extensions: { code: 'FORBIDDEN' },
        });
      }
      // return dashboard data
    },
  },

  Mutation: {
    createPost: async (_parent, args, context: Context) => {
      requireAuth(context);
      return context.prisma.post.create({
        data: { ...args.input, authorId: context.userId },
      });
    },
  },
};

function requireAuth(context: Context): asserts context is Context & { userId: string } {
  if (!context.userId) {
    throw new GraphQLError('Authentication required', {
      extensions: { code: 'UNAUTHENTICATED' },
    });
  }
}

Custom Directives

Directives add reusable behavior to schema fields. They are declared in SDL and implemented in code.

@auth Directive

# Schema
directive @auth(requires: Role = USER) on FIELD_DEFINITION

enum Role {
  USER
  ADMIN
  MODERATOR
}

type Query {
  publicPosts: [Post!]!
  me: User @auth
  adminUsers: [User!]! @auth(requires: ADMIN)
}

type Mutation {
  createPost(input: CreatePostInput!): Post! @auth
  deleteUser(id: ID!): Boolean! @auth(requires: ADMIN)
}

Implementation using schema transforms:

import { mapSchema, getDirective, MapperKind } from '@graphql-tools/utils';
import { defaultFieldResolver, GraphQLSchema } from 'graphql';

function authDirectiveTransformer(schema: GraphQLSchema): GraphQLSchema {
  return mapSchema(schema, {
    [MapperKind.OBJECT_FIELD]: (fieldConfig) => {
      const authDirective = getDirective(schema, fieldConfig, 'auth')?.[0];
      if (!authDirective) return fieldConfig;

      const requiredRole = authDirective.requires || 'USER';
      const originalResolve = fieldConfig.resolve || defaultFieldResolver;

      fieldConfig.resolve = async (parent, args, context, info) => {
        if (!context.userId) {
          throw new GraphQLError('Authentication required', {
            extensions: { code: 'UNAUTHENTICATED' },
          });
        }

        const roleHierarchy = ['USER', 'MODERATOR', 'ADMIN'];
        const userLevel = roleHierarchy.indexOf(context.userRole);
        const requiredLevel = roleHierarchy.indexOf(requiredRole);

        if (userLevel < requiredLevel) {
          throw new GraphQLError(`Role '${requiredRole}' required`, {
            extensions: { code: 'FORBIDDEN' },
          });
        }

        return originalResolve(parent, args, context, info);
      };

      return fieldConfig;
    },
  });
}

// Apply the transformer
import { makeExecutableSchema } from '@graphql-tools/schema';

let schema = makeExecutableSchema({ typeDefs, resolvers });
schema = authDirectiveTransformer(schema);

const server = new ApolloServer({ schema });

type User {

  firstName: String!

@cacheControl Directive

Apollo's built-in caching directive:

type Post @cacheControl(maxAge: 60) {
  id: ID!
  title: String!
  body: String!
  viewCount: Int! @cacheControl(maxAge: 10) # changes frequently
}

type Query {
  publicPosts: [Post!]! @cacheControl(maxAge: 300)
  me: User @cacheControl(maxAge: 0, scope: PRIVATE)
}

Apollo Server Plugins

Plugins hook into the request lifecycle:

import { ApolloServerPlugin } from '@apollo/server';

const loggingPlugin: ApolloServerPlugin<Context> = {
  async requestDidStart(requestContext) {
    const start = Date.now();
    const operationName = requestContext.request.operationName;

    console.log(`GraphQL request started: ${operationName}`);

    return {
      async didResolveOperation(ctx) {
        console.log(`Operation: ${ctx.operation.operation} ${operationName}`);
      },

      async didEncounterErrors(ctx) {
        for (const err of ctx.errors) {
          console.error(`GraphQL Error: ${err.message}`, {
            path: err.path,
            code: err.extensions?.code,
          });
        }
      },

      async willSendResponse() {
        const duration = Date.now() - start;
        console.log(`GraphQL response: ${operationName} (${duration}ms)`);
      },
    };
  },
};

// Query complexity plugin
const complexityPlugin: ApolloServerPlugin<Context> = {
  async requestDidStart() {
    return {
      async didResolveOperation(ctx) {
        // Calculate complexity based on query depth and fields
        const complexity = calculateComplexity(ctx.document, ctx.operation);
        if (complexity > 1000) {
          throw new GraphQLError(
            `Query complexity ${complexity} exceeds maximum of 1000`,
            { extensions: { code: 'QUERY_TOO_COMPLEX' } }
          );
        }
      },
    };
  },
};

const server = new ApolloServer({
  typeDefs,
  resolvers,
  plugins: [loggingPlugin, complexityPlugin],
});

Query Depth Limiting

import depthLimit from 'graphql-depth-limit';

const server = new ApolloServer({
  typeDefs,
  resolvers,
  validationRules: [depthLimit(5)], // Max 5 levels of nesting
});

Organizing Resolvers

For large schemas, split resolvers by type:

src/
  graphql/
    schema.graphql
    resolvers/
      index.ts       # merges all resolvers
      user.ts
      post.ts
      comment.ts
    directives/
      auth.ts
      cache.ts
    loaders/
      index.ts

// resolvers/index.ts
import { merge } from 'lodash';
import { userResolvers } from './user';
import { postResolvers } from './post';
import { commentResolvers } from './comment';
import { dateTimeResolver } from './scalars';

export const resolvers = merge(
  userResolvers,
  postResolvers,
  commentResolvers,
  { DateTime: dateTimeResolver }
);

Custom Scalars

import { GraphQLScalarType, Kind } from 'graphql';

const dateTimeResolver = new GraphQLScalarType({
  name: 'DateTime',
  description: 'ISO 8601 date-time string',
  serialize(value: Date) {
    return value.toISOString();
  },
  parseValue(value: string) {
    const date = new Date(value);
    if (isNaN(date.getTime())) throw new Error('Invalid DateTime');
    return date;
  },
  parseLiteral(ast) {
    if (ast.kind === Kind.STRING) {
      const date = new Date(ast.value);
      if (isNaN(date.getTime())) throw new Error('Invalid DateTime');
      return date;
    }
    throw new Error('DateTime must be a string');
  },
});

Exercises

  1. Build a Context with Authentication

    Set up Apollo Server with Express and implement a context function that extracts a JWT token from the Authorization header, verifies it, and attaches the user ID and role to the context. Create a 'me' query that returns the current user.

    Hint

    Use jwt.verify() in the context function. Catch errors silently so unauthenticated requests still work for public queries.

  2. Implement an @auth Directive

    Create a custom @auth directive that supports role-based access. Decorate sensitive fields with @auth(requires: ADMIN). Test that unauthenticated users get an error and that regular users cannot access admin-only fields.

    Hint

    Use mapSchema from @graphql-tools/utils to transform field resolvers. Wrap the original resolver with an auth check.

  3. Create a Logging Plugin

    Build an Apollo Server plugin that logs operation name, type (query/mutation), duration, and any errors for every request. Format the output as structured JSON suitable for a log aggregation system.

    Hint

    Use requestDidStart to capture the start time and return lifecycle hooks (willSendResponse, didEncounterErrors).

  4. Implement a DateTime Custom Scalar

    Create a DateTime scalar that serializes JS Date objects to ISO strings, parses ISO string inputs back to Date objects, and validates the format. Register it in your resolver map.

    Hint

    Implement serialize, parseValue, and parseLiteral methods on a new GraphQLScalarType instance.

Common Problems & Solutions

1 Context is undefined or missing fields in resolvers
Cause: The context function in expressMiddleware is not returning the expected object, or an async operation inside it fails silently.
Solution: Add error logging inside the context function. Make sure it always returns an object, even when authentication fails. Check that you are passing the context function to expressMiddleware, not to the ApolloServer constructor.
2 Directive transformer does not apply to mutations
Cause: The MapperKind used in mapSchema does not cover the MUTATION type.
Solution: MapperKind.OBJECT_FIELD applies to all object types including Query and Mutation. Make sure the directive is declared in SDL with 'on FIELD_DEFINITION' and applied to mutation fields.
3 Resolver changes do not take effect
Cause: Apollo Server caches the schema at startup. Hot reloading requires special configuration.
Solution: Restart the server after resolver changes. For development, use nodemon or tsx --watch to auto-restart on file changes.
