Custom Providers: useValue, useFactory, useExisting

Intuition

Custom providers are like hiring strategies. useClass is saying 'hire someone with these qualifications.' useValue is 'here's a contractor I already have.' useFactory is 'let me run an interview process and pick the right person based on criteria.' useExisting is 'just give them the same person who's already on that other team.'

Custom Providers: useValue, useFactory, useExisting

Introduction

The standard @Injectable() class pattern covers most use cases, but NestJS's DI system is far more flexible. Custom providers let you inject configuration objects, create instances conditionally, swap implementations for testing, connect to external services, and more. There are four custom provider types: useClass, useValue, useFactory, and useExisting.

Provider Syntax Recap

Every provider has two parts: 1. Token -- How consumers request the dependency (a class, string, or symbol) 2. Value -- What the container provides (a class instance, a value, or a factory result)

{
  provide: TOKEN,      // What consumers ask for
  useClass: SomeClass, // What the container gives them
}

useClass: Alternative Implementation

useClass lets you swap the actual class behind a token:

// Interface (abstract class in TS for DI)
export abstract class PaymentGateway {
  abstract charge(amount: number, currency: string): Promise<PaymentResult>;
  abstract refund(transactionId: string): Promise<RefundResult>;
}

// Implementations
@Injectable()
export class StripeGateway extends PaymentGateway {
  async charge(amount: number, currency: string) {
    // Stripe API call
    return { success: true, transactionId: 'stripe_123' };
  }
  async refund(transactionId: string) {
    return { success: true };
  }
}

@Injectable()
export class PayPalGateway extends PaymentGateway {
  async charge(amount: number, currency: string) {
    // PayPal API call
    return { success: true, transactionId: 'paypal_456' };
  }
  async refund(transactionId: string) {
    return { success: true };
  }
}

Switch implementations based on environment:

@Module({
  providers: [
    {
      provide: PaymentGateway,
      useClass:
        process.env.PAYMENT_PROVIDER === 'paypal'
          ? PayPalGateway
          : StripeGateway,
    },
  ],
  exports: [PaymentGateway],
})
export class PaymentModule {}

Consumers always inject PaymentGateway -- they never know which implementation they get:

@Injectable()
export class OrdersService {
  constructor(private payment: PaymentGateway) {}

  async checkout(order: Order) {
    return this.payment.charge(order.total, order.currency);
  }
}

useValue: Static Values and Mocks

useValue provides a fixed value -- objects, primitives, or mock instances:

Configuration Objects

export interface AppConfig {
  apiUrl: string;
  maxRetries: number;
  timeout: number;
}

@Module({
  providers: [
    {
      provide: 'APP_CONFIG',
      useValue: {
        apiUrl: 'https://api.example.com',
        maxRetries: 3,
        timeout: 5000,
      } as AppConfig,
    },
  ],
})
export class AppModule {}

External Library Instances

import Redis from 'ioredis';

@Module({
  providers: [
    {
      provide: 'REDIS_CLIENT',
      useValue: new Redis({
        host: 'localhost',
        port: 6379,
      }),
    },
  ],
  exports: ['REDIS_CLIENT'],
})
export class RedisModule {}

Mock Providers for Testing

const mockUsersService = {
  findAll: jest.fn().mockResolvedValue([{ id: '1', name: 'Test' }]),
  findOne: jest.fn().mockResolvedValue({ id: '1', name: 'Test' }),
  create: jest.fn().mockResolvedValue({ id: '2', name: 'New' }),
};

const module = await Test.createTestingModule({
  controllers: [UsersController],
  providers: [
    {
      provide: UsersService,
      useValue: mockUsersService,
    },
  ],
}).compile();

useFactory: Dynamic Providers

useFactory uses a factory function to create the provider. The function can be async, accept injected dependencies, and contain logic:

Basic Factory

providers: [
  {
    provide: 'DATABASE_CONNECTION',
    useFactory: async () => {
      const connection = await createConnection({
        type: 'postgres',
        host: 'localhost',
        port: 5432,
        database: 'mydb',
      });
      return connection;
    },
  },
]

Factory with Injected Dependencies

providers: [
  ConfigService,
  {
    provide: 'DATABASE_CONNECTION',
    useFactory: async (config: ConfigService) => {
      const connection = await createConnection({
        type: 'postgres',
        host: config.get('DB_HOST'),
        port: config.get<number>('DB_PORT'),
        database: config.get('DB_NAME'),
        username: config.get('DB_USER'),
        password: config.get('DB_PASS'),
        ssl: config.get('NODE_ENV') === 'production',
      });
      return connection;
    },
    inject: [ConfigService], // Dependencies for the factory
  },
]

The inject array maps positionally to the factory function's parameters.

Conditional Factory

providers: [
  {
    provide: 'CACHE_SERVICE',
    useFactory: (config: ConfigService) => {
      const cacheType = config.get('CACHE_TYPE');

      switch (cacheType) {
        case 'redis':
          return new RedisCacheService(config.get('REDIS_URL'));
        case 'memcached':
          return new MemcachedService(config.get('MEMCACHED_URL'));
        default:
          return new InMemoryCacheService();
      }
    },
    inject: [ConfigService],
  },
]

Async Factory with Initialization

providers: [
  {
    provide: 'ELASTICSEARCH_CLIENT',
    useFactory: async (config: ConfigService) => {
      const client = new Client({
        node: config.get('ELASTICSEARCH_URL'),
      });

      // Verify connection on startup
      try {
        await client.ping();
        console.log('Elasticsearch connected');
      } catch (error) {
        console.error('Elasticsearch connection failed:', error);
        throw error; // Prevents app from starting
      }

      return client;
    },
    inject: [ConfigService],
  },
]

useExisting: Provider Aliases

useExisting creates an alias -- a different token that resolves to the same instance:

@Injectable()
export class LoggerService {
  log(message: string) { console.log(message); }
  error(message: string) { console.error(message); }
  warn(message: string) { console.warn(message); }
}

@Module({
  providers: [
    LoggerService,
    {
      provide: 'LOGGER',
      useExisting: LoggerService, // Same instance as LoggerService
    },
    {
      provide: 'APP_LOGGER',
      useExisting: LoggerService, // Same instance again
    },
  ],
  exports: [LoggerService, 'LOGGER', 'APP_LOGGER'],
})
export class LoggingModule {}

All three tokens resolve to the exact same LoggerService instance.

Real-World Alias Pattern: Interface Segregation

// Define narrow interfaces
export abstract class UserReader {
  abstract findById(id: string): Promise<User>;
  abstract findByEmail(email: string): Promise<User>;
}

export abstract class UserWriter {
  abstract create(data: CreateUserDto): Promise<User>;
  abstract update(id: string, data: UpdateUserDto): Promise<User>;
}

// Single implementation
@Injectable()
export class UsersService extends UserReader implements UserWriter {
  async findById(id: string) { /* ... */ }
  async findByEmail(email: string) { /* ... */ }
  async create(data: CreateUserDto) { /* ... */ }
  async update(id: string, data: UpdateUserDto) { /* ... */ }
}

// Register with aliases
@Module({
  providers: [
    UsersService,
    { provide: UserReader, useExisting: UsersService },
    { provide: UserWriter, useExisting: UsersService },
  ],
  exports: [UserReader, UserWriter],
})
export class UsersModule {}

Now consumers can request only the capabilities they need:

@Injectable()
export class ReportService {
  // Can only read users -- can't accidentally mutate
  constructor(private users: UserReader) {}
}

@Injectable()
export class AdminService {
  // Can read and write
  constructor(
    private userReader: UserReader,
    private userWriter: UserWriter,
  ) {}
}

Combining Custom Provider Patterns

@Module({
  providers: [
    // Standard class
    UsersService,

    // Configuration value
    { provide: 'API_VERSION', useValue: '2.0' },

    // Conditional implementation
    {
      provide: 'STORAGE',
      useClass: process.env.STORAGE === 's3' ? S3Storage : LocalStorage,
    },

    // Async factory with dependencies
    {
      provide: 'DB_POOL',
      useFactory: async (config: ConfigService) => {
        return createPool(config.get('DATABASE_URL'));
      },
      inject: [ConfigService],
    },

    // Alias
    { provide: 'PRIMARY_STORAGE', useExisting: 'STORAGE' },
  ],
})
export class AppModule {}

Common Pitfalls

  1. Forgetting inject in useFactory -- The factory receives undefined for each missing dependency. Always list all dependencies in the inject array.
  2. useValue with class instances -- The instance is created immediately (not lazily). Use useFactory for lazy or async initialization.
  3. useExisting with wrong token -- The referenced provider must exist in the same module or be imported. Otherwise you get a resolution error.
  4. String token collisions -- Two modules might use the same string 'CONFIG'. Use Symbols or namespaced strings like 'payments.config'.

Exercises

  1. Build a Strategy Pattern with useClass

    Create a NotificationModule with an abstract NotificationSender class and three implementations: EmailSender, SmsSender, PushSender. Use useClass to switch between them based on an environment variable. Write a NotificationService that uses whichever implementation is configured.

    Hint

    Define the abstract class with abstract send() method. Create three implementations. Use process.env.NOTIFICATION_TYPE to pick the useClass value.

  2. Create an Async Database Factory

    Build a useFactory provider that creates a database connection pool. The factory should read config from ConfigService, create the pool, run a test query to verify connectivity, and log the connection info. If the connection fails, it should retry 3 times before throwing.

    Hint

    Make the factory async. Use a for loop for retries with a delay between attempts. The inject array should include [ConfigService].

  3. Implement Interface Segregation with useExisting

    Create a ProductsService that implements both ProductReader and ProductWriter abstract classes. Register aliases so that the catalog module can only read products while the admin module can read and write.

    Hint

    Register ProductsService as the main provider, then use useExisting for ProductReader and ProductWriter aliases. Export only the interfaces, not the concrete class.

Common Problems & Solutions

1 Factory provider receives undefined dependencies
Cause: The `inject` array is missing or doesn't match the factory function's parameter order.
Solution: Ensure every dependency the factory needs is listed in the `inject` array in the correct order. The array maps positionally to function parameters.

// WRONG: inject is missing\n{\n provide: 'DB',\n useFactory: (config: ConfigService) => { / config is undefined / },\n}\n\n// CORRECT:\n{\n provide: 'DB',\n useFactory: (config: ConfigService) => { / works / },\n inject: [ConfigService],\n}

2 useExisting provider throws 'not found in the current context'
Cause: The referenced provider (the alias target) is not registered in the same module or imported from another module.
Solution: Ensure the target provider is registered in the same module's providers array, or import the module that exports it.
3 useValue provider instance is shared and mutated unexpectedly
Cause: useValue provides the same object reference to all consumers. If one consumer mutates it, all others see the change.
Solution: Use useFactory to create a new instance for each consumer, or use Object.freeze() on the useValue object to prevent mutations.
