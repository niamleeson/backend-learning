Testing Pipelines: Lint, Type-Check, Test, Coverage

Intuition

A testing pipeline is like airport security. Each checkpoint catches different problems: the lint check catches formatting issues (metal detector), type checking catches type mismatches (X-ray machine), unit tests catch logic bugs (baggage screening), and integration tests ensure everything works together (the final boarding check). Only code that passes ALL checkpoints makes it to production.

Testing Pipelines: Lint, Type-Check, Test, Coverage

The Quality Gate Pipeline

A testing pipeline enforces code quality at every push. It acts as an automated code reviewer that catches bugs, style issues, type errors, and regressions before they reach production.

Complete Testing Pipeline

# .github/workflows/test.yml
name: Test Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  NODE_VERSION: '20'

jobs:
  # ================================
  # Stage 1: Code Quality (fast)
  # ================================
  lint:
    name: Lint & Format
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'

      - run: npm ci

      # ESLint
      - name: Run ESLint
        run: npx eslint . --format=json --output-file=eslint-report.json
        continue-on-error: true

      - name: Annotate ESLint results
        if: always()
        uses: ataylorme/eslint-annotate-action@v3
        with:
          report-json: eslint-report.json

      # Prettier
      - name: Check formatting
        run: npx prettier --check "src/**/*.{ts,js,json}"

  typecheck:
    name: Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
      - run: npm ci
      - run: npx prisma generate     # Generate types
      - name: TypeScript check
        run: npx tsc --noEmit --pretty

  # ================================
  # Stage 2: Unit Tests (medium)
  # ================================
  unit-tests:
    name: Unit Tests
    needs: [lint, typecheck]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
      - run: npm ci
      - run: npx prisma generate

      - name: Run unit tests
        run: npx jest --testPathPattern='unit' --coverage --coverageReporters=json-summary --coverageReporters=text
        env:
          NODE_ENV: test

      - name: Upload unit test coverage
        uses: actions/upload-artifact@v4
        with:
          name: unit-coverage
          path: coverage/

  # ================================
  # Stage 3: Integration Tests (slow)
  # ================================
  integration-tests:
    name: Integration Tests
    needs: [lint, typecheck]
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_DB: test_db
          POSTGRES_USER: test_user
          POSTGRES_PASSWORD: test_pass
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
      redis:
        image: redis:7-alpine
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
      - run: npm ci

      - name: Setup database
        run: npx prisma migrate deploy
        env:
          DATABASE_URL: postgresql://test_user:test_pass@localhost:5432/test_db

      - name: Seed test data
        run: npx prisma db seed
        env:
          DATABASE_URL: postgresql://test_user:test_pass@localhost:5432/test_db

      - name: Run integration tests
        run: npx jest --testPathPattern='integration' --coverage --forceExit
        env:
          DATABASE_URL: postgresql://test_user:test_pass@localhost:5432/test_db
          REDIS_URL: redis://localhost:6379
          NODE_ENV: test

  # ================================
  # Stage 4: Coverage Report
  # ================================
  coverage:
    name: Coverage Report
    needs: [unit-tests, integration-tests]
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    steps:
      - uses: actions/checkout@v4

      - name: Download unit coverage
        uses: actions/download-artifact@v4
        with:
          name: unit-coverage
          path: coverage/

      - name: Coverage threshold check
        run: |
          COVERAGE=$(cat coverage/coverage-summary.json | jq '.total.lines.pct')
          echo "Line coverage: ${COVERAGE}%"
          if (( $(echo "$COVERAGE < 80" | bc -l) )); then
            echo "::error::Coverage ${COVERAGE}% is below 80% threshold"
            exit 1
          fi

      - name: Comment coverage on PR
        uses: marocchino/sticky-pull-request-comment@v2
        with:
          header: coverage
          message: |
            ## Test Coverage Report
            | Metric | Coverage |
            |--------|----------|
            | Lines  | ${{ env.LINE_COV }}% |
            | Branch | ${{ env.BRANCH_COV }}% |

ESLint Configuration for CI

// eslint.config.js (flat config, ESLint 9+)
import eslint from '@eslint/js';
import tseslint from 'typescript-eslint';

export default tseslint.config(
  eslint.configs.recommended,
  ...tseslint.configs.recommended,
  {
    rules: {
      '@typescript-eslint/no-unused-vars': 'error',
      '@typescript-eslint/explicit-function-return-type': 'warn',
      '@typescript-eslint/no-explicit-any': 'warn',
      'no-console': ['warn', { allow: ['warn', 'error'] }],
    },
  },
  {
    ignores: ['dist/', 'node_modules/', 'coverage/'],
  }
);

Jest Configuration for CI

// jest.config.ts
import type { Config } from 'jest';

const config: Config = {
  preset: 'ts-jest',
  testEnvironment: 'node',
  roots: ['<rootDir>/src'],
  moduleNameMapper: {
    '^@/(.*)$': '<rootDir>/src/$1',
  },
  coverageDirectory: 'coverage',
  coveragePathIgnorePatterns: [
    '/node_modules/',
    '/dist/',
    '/__tests__/',
    '/test/',
  ],
  coverageThreshold: {
    global: {
      branches: 70,
      functions: 80,
      lines: 80,
      statements: 80,
    },
  },
  projects: [
    {
      displayName: 'unit',
      testMatch: ['**/*.unit.test.ts'],
      testEnvironment: 'node',
    },
    {
      displayName: 'integration',
      testMatch: ['**/*.integration.test.ts'],
      testEnvironment: 'node',
      globalSetup: '<rootDir>/test/setup.ts',
      globalTeardown: '<rootDir>/test/teardown.ts',
    },
  ],
};

export default config;

Pre-Commit Hooks with Husky + lint-staged

// package.json
{
  "scripts": {
    "prepare": "husky",
    "lint": "eslint .",
    "lint:fix": "eslint . --fix",
    "format": "prettier --write 'src/**/*.{ts,js,json}'",
    "typecheck": "tsc --noEmit"
  },
  "lint-staged": {
    "*.{ts,js}": [
      "eslint --fix",
      "prettier --write"
    ],
    "*.{json,md}": [
      "prettier --write"
    ]
  }
}

# Setup husky
npx husky init

# .husky/pre-commit
npx lint-staged

# .husky/pre-push
npm run typecheck && npm test

  }

Common Pitfalls

  1. Running all checks sequentially - Lint, typecheck, and tests can run in parallel since they are independent. Use separate jobs.
  2. Not failing the pipeline on lint errors - Lint warnings that never become errors are ignored. Set strict rules.
  3. Skipping integration tests in CI - Unit tests alone miss real bugs. Always test with real databases in CI.
  4. Coverage thresholds that are too high or too low - 80% line coverage is a good starting point. 100% is rarely practical or necessary.
  5. Not caching between pipeline runs - Each run installs fresh dependencies without caching, wasting minutes.

Exercises

  1. Build a Quality Gate

    Create a complete GitHub Actions workflow with four parallel stages: lint (ESLint + Prettier), typecheck (tsc), unit tests (Jest), and integration tests (Jest + PostgreSQL service container). Add a final coverage stage that checks for 80% minimum coverage and comments the result on the PR.

    Hint

    Use 'needs' for stage dependencies. Run lint and typecheck in parallel, then tests after both pass. Use actions/upload-artifact to share coverage between jobs.

  2. Pre-Commit Hook Setup

    Set up Husky with lint-staged to run ESLint and Prettier on staged files before each commit, and TypeScript type-checking before each push. Verify that commits with lint errors are rejected.

    Hint

    Use 'npx husky init' to set up. Configure lint-staged in package.json. Create .husky/pre-commit and .husky/pre-push files.

  3. Coverage Trend Tracking

    Modify the CI workflow to fail if coverage drops more than 2% compared to the main branch. Store the main branch coverage in a GitHub Actions artifact and compare it in PR builds.

    Hint

    On main pushes, upload coverage-summary.json as an artifact. On PRs, download the main branch artifact and compare the values using jq.

Common Problems & Solutions

1 ESLint passes locally but fails in CI
Cause: Different ESLint versions (local vs CI) or different configurations loaded. The CI environment may have stricter settings.
Solution: Pin ESLint version in package.json. Ensure .eslintrc is committed. Run `npm ci` (not `npm install`) in CI for deterministic installs. Check that .eslintignore covers generated files.
2 Integration tests flaky (pass sometimes, fail sometimes)
Cause: Tests depend on timing, shared database state, or ordering. Parallel test execution can cause conflicts.
Solution: Use unique database names per test run. Clean up data in beforeEach/afterEach. Use --runInBand for integration tests. Add --forceExit to prevent hanging. Increase timeouts for slow CI environments.
3 Coverage report shows 0% even though tests pass
Cause: Jest is running compiled JavaScript but coverage is configured for TypeScript source, or coverage path patterns are wrong.
Solution: Ensure ts-jest is configured properly. Check coveragePathIgnorePatterns is not accidentally excluding your source. Use `--collectCoverageFrom='src/**/*.ts'` explicitly.
