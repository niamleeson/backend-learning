HTTP Methods, Status Codes, and Content Negotiation

Intuition

HTTP is a conversation with rules. The method is what you want to do (read, create, update, delete), the status code is the answer (success, your fault, my fault), and headers are the metadata — like the tone of voice, expiration dates, and formatting preferences in a business letter.

HTTP Methods, Status Codes, and Content Negotiation

Introduction

HTTP is the foundation of every web API. While Express abstracts many details, understanding HTTP methods, status codes, headers, and content negotiation deeply is what separates a competent API developer from one who just "makes things work." This lesson covers the full spectrum of HTTP semantics you need for production API development.

HTTP Methods in Depth

GET — Retrieve Resources

// Safe (no side effects) and idempotent
// MUST NOT have a request body (per HTTP spec)
router.get('/api/users', async (req, res) => {
  const users = await userService.findAll();
  res.json({ data: users });
});

// Conditional GET with ETag
router.get('/api/users/:id', async (req, res) => {
  const user = await userService.findById(req.params.id);
  if (!user) return res.status(404).json({ error: 'Not found' });

  const etag = generateETag(user);
  if (req.get('If-None-Match') === etag) {
    return res.status(304).send(); // Not Modified
  }

  res.set('ETag', etag);
  res.json({ data: user });
});

POST — Create Resources

// NOT idempotent — calling twice creates two resources
router.post('/api/users', async (req, res) => {
  const user = await userService.create(req.body);

  // Return 201 Created with the new resource
  res.status(201)
     .set('Location', `/api/users/${user.id}`) // Location header
     .json({ data: user });
});

PUT — Replace Resources

// Idempotent — same request always produces same result
// Requires the COMPLETE resource representation
router.put('/api/users/:id', async (req, res) => {
  const { name, email, role } = req.body;

  // Validate all required fields are present
  if (!name || !email || !role) {
    return res.status(400).json({
      error: 'PUT requires all fields: name, email, role'
    });
  }

  const user = await userService.replace(req.params.id, { name, email, role });

  if (!user) {
    // PUT can create if resource does not exist (upsert)
    const newUser = await userService.create({ id: req.params.id, name, email, role });
    return res.status(201).json({ data: newUser });
  }

  res.json({ data: user }); // 200 OK
});

PATCH — Partial Update

// NOT guaranteed idempotent
router.patch('/api/users/:id', async (req, res) => {
  const user = await userService.findById(req.params.id);
  if (!user) return res.status(404).json({ error: 'Not found' });

  // Only update provided fields
  const updated = await userService.update(req.params.id, req.body);
  res.json({ data: updated });
});

DELETE — Remove Resources

// Idempotent — deleting an already-deleted resource is fine
router.delete('/api/users/:id', async (req, res) => {
  const deleted = await userService.delete(req.params.id);

  if (!deleted) {
    // Two common approaches:
    return res.status(404).json({ error: 'Not found' });
    // OR: return 204 (idempotent — end state is the same)
  }

  res.status(204).send(); // No Content
});

HEAD — Metadata Only

// Same as GET but without the response body
// Express handles HEAD automatically for GET routes
// Useful for checking if a resource exists or getting headers
router.head('/api/users/:id', async (req, res) => {
  const exists = await userService.exists(req.params.id);
  res.status(exists ? 200 : 404).send();
});

OPTIONS — Describe Capabilities

// Usually handled by CORS middleware
// Can provide custom capability info
router.options('/api/users', (req, res) => {
  res.set('Allow', 'GET, POST, OPTIONS');
  res.status(204).send();
});

HTTP Status Codes

2xx — Success

200 OK              // Default success. GET, PUT, PATCH
201 Created         // Resource created. POST (include Location header)
202 Accepted        // Request accepted but processing not complete (async)
204 No Content      // Success with no body. DELETE, PUT with no response body

3xx — Redirection

301 Moved Permanently    // Resource permanently moved (cacheable)
302 Found                // Temporary redirect
304 Not Modified         // Conditional GET — client cache is still valid
307 Temporary Redirect   // Same as 302 but preserves HTTP method
308 Permanent Redirect   // Same as 301 but preserves HTTP method

4xx — Client Errors

400 Bad Request          // Malformed request, validation error
401 Unauthorized         // No authentication or invalid credentials
403 Forbidden            // Authenticated but not authorized
404 Not Found            // Resource does not exist
405 Method Not Allowed   // HTTP method not supported for this route
409 Conflict             // Conflicts with current state (duplicate, version mismatch)
410 Gone                 // Resource permanently removed (unlike 404, won't return)
415 Unsupported Media Type // Wrong Content-Type header
422 Unprocessable Entity // Syntactically valid but semantically invalid
429 Too Many Requests    // Rate limit exceeded

5xx — Server Errors

500 Internal Server Error  // Unexpected server error
502 Bad Gateway            // Invalid response from upstream server
503 Service Unavailable    // Server overloaded or in maintenance
504 Gateway Timeout        // Upstream server did not respond in time

Choosing the Right Status Code

Decision tree for common scenarios:

Request succeeded?
  Yes -> Did it create something?
           Yes -> 201 Created
           No  -> Is there a response body?
                    Yes -> 200 OK
                    No  -> 204 No Content
  No  -> Is it the client's fault?
           Yes -> Is the request malformed?
                    Yes -> 400 Bad Request
                    No  -> Is it an auth problem?
                             Not logged in -> 401 Unauthorized
                             No permission -> 403 Forbidden
                             Not found     -> 404 Not Found
                             Duplicate     -> 409 Conflict
                             Validation    -> 422 Unprocessable Entity
           No  -> 500 Internal Server Error

Content Negotiation

Content negotiation lets clients and servers agree on the response format:

Accept Header

router.get('/api/users/:id', async (req, res) => {
  const user = await userService.findById(req.params.id);
  if (!user) return res.status(404).json({ error: 'Not found' });

  res.format({
    'application/json': () => {
      res.json({ data: user });
    },
    'text/csv': () => {
      res.set('Content-Type', 'text/csv');
      res.send(`id,name,email\n${user.id},${user.name},${user.email}`);
    },
    default: () => {
      res.status(406).json({ error: 'Not Acceptable' });
    }
  });
});

Content-Type Header

The client tells the server what format the request body is in:

// Handle different request content types
app.use(express.json());                          // application/json
app.use(express.urlencoded({ extended: true }));  // application/x-www-form-urlencoded
app.use(express.text());                          // text/plain

router.post('/api/data', (req, res) => {
  if (req.is('json')) {
    // Handle JSON body
  } else if (req.is('urlencoded')) {
    // Handle form body
  } else {
    res.status(415).json({ error: 'Unsupported Media Type' });
  }
});

Caching Headers

// Cache-Control
router.get('/api/products', async (req, res) => {
  const products = await productService.findAll();

  res.set('Cache-Control', 'public, max-age=300'); // Cache for 5 min
  res.json({ data: products });
});

// No caching for dynamic/private data
router.get('/api/profile', auth, async (req, res) => {
  res.set('Cache-Control', 'private, no-cache');
  res.json({ data: req.user });
});

// ETag for conditional requests
router.get('/api/products/:id', async (req, res) => {
  const product = await productService.findById(req.params.id);
  const etag = `"${product.updatedAt.getTime()}"`;

  res.set('ETag', etag);

  if (req.get('If-None-Match') === etag) {
    return res.status(304).send();
  }

  res.json({ data: product });
});

Common Pitfalls

Using 200 for Everything

// WRONG: always returning 200
res.status(200).json({ error: 'User not found' }); // Should be 404
res.status(200).json({ error: 'Validation failed' }); // Should be 400/422

// CORRECT: use appropriate status codes
res.status(404).json({ error: 'User not found' });
res.status(422).json({ error: 'Validation failed', details: [...] });

401 vs 403

// 401 Unauthorized: "Who are you?" (no/invalid credentials)
// 403 Forbidden: "I know who you are, but you can't do this"

// No token provided or token is invalid
res.status(401).json({ error: 'Authentication required' });

// Valid token but user lacks permission
res.status(403).json({ error: 'Admin access required' });

Summary

HTTP methods define operations (GET=read, POST=create, PUT=replace, PATCH=update, DELETE=remove). Status codes communicate outcomes precisely (2xx success, 4xx client error, 5xx server error). Content negotiation lets clients and servers agree on data formats. Use these semantics correctly and your API becomes self-documenting and predictable.

Exercises

  1. Status Code Decision Handler

    Implement a complete set of CRUD routes for a 'projects' resource that uses correct status codes for every scenario: 200 for successful reads, 201 for creates with Location header, 204 for deletes, 404 for missing resources, 400 for validation errors, 409 for duplicate names.

    Hint

    For 201 responses, set the Location header: res.set('Location', /api/projects/${id}). For 204, use res.status(204).send() with no body.

  2. ETag-Based Conditional Requests

    Implement ETag support for a resource endpoint. Generate ETags from resource data, check If-None-Match headers on GET requests (return 304), and check If-Match headers on PUT/PATCH requests (return 412 Precondition Failed if mismatched).

    Hint

    Generate ETag from a hash of the resource data or updatedAt timestamp. Compare with req.get('If-None-Match') for GET and req.get('If-Match') for PUT/PATCH.

  3. Content Negotiation Endpoint

    Build a GET /api/reports/:id endpoint that returns data in JSON, CSV, or XML format based on the Accept header. Use res.format() for negotiation. Return 406 Not Acceptable for unsupported formats.

    Hint

    Use res.format({ 'application/json': fn, 'text/csv': fn, 'application/xml': fn, default: fn }).

Common Problems & Solutions

1 Client receives 200 OK but the operation actually failed
Cause: The API returns 200 for all responses, embedding the actual status in the response body. This breaks HTTP semantics and confuses clients.
Solution: Use proper HTTP status codes. 4xx for client errors, 5xx for server errors. HTTP clients and tools rely on status codes for error detection.
2 PUT request fails with 'Missing required fields' even though only one field changed
Cause: PUT replaces the entire resource. Unlike PATCH, it requires the complete representation.
Solution: Use PATCH for partial updates. If using PUT, the client must send all fields. Document this clearly in your API documentation.
