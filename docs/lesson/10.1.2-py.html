Design a Rate Limiter

Intuition

A rate limiter is a gatekeeper that counts requests in time windows and rejects excess traffic to protect backend services from overload. The fundamental tension is between precision (exact counts) and performance (minimal latency overhead per request).

Design a Rate Limiter

Requirements

  • Limit requests per user/IP/API-key within configurable time windows
  • Support multiple algorithms: fixed window, sliding window, token bucket, leaky bucket
  • Return appropriate HTTP 429 responses with retry-after headers
  • Operate in a distributed environment with minimal latency overhead

Architecture

A rate limiter middleware intercepts requests before they reach the application. It checks counters stored in a fast in-memory store (Redis). Configuration rules are stored in a database and cached locally. The middleware returns pass/reject decisions in under 1ms.

Key Components

  • Rules Engine: Configurable rules per endpoint, user tier, or API key stored in a config DB
  • Counter Store: Redis with atomic INCR and TTL operations for tracking request counts
  • Middleware: Lightweight interceptor deployed as a sidecar or library in the API gateway
  • Response Handler: Returns 429 status with X-RateLimit-Remaining and Retry-After headers

Scaling

Use Redis Cluster for partitioned counter storage. Deploy rate limiter logic at the API gateway level for centralized enforcement. Use local counters with periodic sync for approximate but faster limiting.

Trade-offs

  • Centralized counters are accurate but add network latency; local counters are fast but may allow brief bursts above limits
  • Token bucket allows bursts up to bucket size; sliding window log is precise but uses more memory

Exercises

  1. Implement a Sliding Window Rate Limiter

    Build a rate limiter that uses the sliding window log algorithm. Track timestamps of each request in a sorted set. For each incoming request, remove entries outside the window, count remaining entries, and accept or reject.

    Hint

    Use Redis ZRANGEBYSCORE to remove old entries and ZCARD to count current entries, all within a single pipeline for atomicity.

  2. Build a Token Bucket Implementation

    Create a token bucket rate limiter where tokens refill at a fixed rate. Each request consumes one token. If the bucket is empty, reject the request. Support configurable bucket size and refill rate per user.

    Hint

    Instead of a background refill thread, calculate tokens lazily: tokens = min(max_tokens, last_tokens + (now - last_refill) * refill_rate).

Common Problems & Solutions

1 Rate limits are not enforced accurately in a multi-node deployment, allowing users to exceed limits by hitting different servers.
Cause: Each server maintains its own local counters without synchronization, so a user sending requests to N servers effectively gets N times the intended limit.
Solution: Use a centralized Redis instance for counter storage with atomic Lua scripts for check-and-increment. Alternatively, use sticky sessions to route a user to the same server.
2 Redis becomes a single point of failure, and its downtime disables all rate limiting.
Cause: The system depends entirely on Redis availability for every rate limit decision with no fallback mechanism.
Solution: Implement a local in-memory fallback rate limiter that activates when Redis is unavailable. Use Redis Sentinel or Cluster for high availability, and consider allowing requests through (fail-open) during brief outages.
