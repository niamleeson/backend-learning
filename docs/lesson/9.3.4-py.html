Task Scheduling and Periodic Tasks

Intuition

Celery Beat is like a cron job manager that drops work orders into a task queue on a schedule. The workers do not care whether the task was triggered by a user request or by the scheduler—they just process what is in the queue.

Task Scheduling & Periodic Tasks

Many backend systems require work to happen on a schedule—generating reports, sending digest emails, cleaning up expired data, or syncing with external services. Celery Beat and other scheduling tools provide reliable periodic task execution.

Core Concepts

  • Celery Beat: A scheduler that sends tasks to the broker at defined intervals. It runs as a separate process alongside workers. Supports crontab-style schedules and fixed intervals.
  • Crontab Schedules: Express complex schedules like "every weekday at 8am" or "every 15 minutes during business hours" using familiar crontab syntax.
  • Dynamic Scheduling: Store schedules in a database (using django-celery-beat) so they can be modified at runtime without restarting the scheduler.
  • Single Execution Guarantee: Ensure scheduled tasks run exactly once per interval, even in multi-node deployments. Use distributed locks or a single Beat instance to prevent duplicate scheduling.
  • Timezone Handling: Always configure Celery's timezone (timezone = 'UTC') and handle DST transitions correctly. Schedule business-logic tasks in UTC and convert for display.

Key Takeaways

  1. Run only one Celery Beat instance to prevent duplicate task scheduling. Use a distributed lock if running multiple for high availability.
  2. Use crontab schedules for time-of-day requirements and interval schedules for fixed-frequency tasks.
  3. Monitor scheduled tasks for missed runs, excessive duration, and failures.

Exercises

  1. Set Up Periodic Tasks with Celery Beat

    Configure Celery Beat with three periodic tasks: a health check every 30 seconds, a report generator daily at midnight UTC, and a cleanup task every Sunday at 3am. Verify they execute on schedule by checking the worker logs.

    Hint

    In your Celery config, use beat_schedule = {'health-check': {'task': 'tasks.health_check', 'schedule': 30.0}, 'daily-report': {'task': 'tasks.generate_report', 'schedule': crontab(hour=0, minute=0)}}.

  2. Implement Dynamic Scheduling with django-celery-beat

    Install django-celery-beat, migrate the database, and create an admin interface for managing periodic tasks. Add a periodic task through the Django admin, verify it runs, then modify its schedule at runtime without restarting any processes.

    Hint

    Set CELERY_BEAT_SCHEDULER = 'django_celery_beat.schedulers:DatabaseScheduler' in Django settings. Create PeriodicTask and IntervalSchedule objects through the admin or ORM.

Common Problems & Solutions

1 Periodic tasks run twice at each scheduled interval.
Cause: Multiple Celery Beat instances are running (e.g., one per server in a multi-node deployment), each sending the same tasks.
Solution: Ensure only one Beat instance runs. Use a distributed lock with `celery-singleton` or a sidecar like `celery-redbeat` that stores the schedule in Redis with leader election.
2 A scheduled report task runs for 45 minutes, overlapping with its next scheduled run every 30 minutes.
Cause: The task execution time exceeds the schedule interval, causing concurrent runs.
Solution: Use a distributed lock within the task to skip execution if a previous run is still active. Alternatively, set `CELERY_ACKS_LATE=True` with `task_reject_on_worker_lost=True`, or increase the interval to exceed the worst-case execution time.
