Distributed Tracing with OpenTelemetry for Node.js

Intuition

Distributed tracing is like a package tracking system. When you ship a package (request), it gets a tracking number (trace ID). At each facility (service) it passes through, a scan is recorded (span). You can follow the package's entire journey — see where it was, how long each stop took, and where it got stuck — all from one tracking number.

Distributed Tracing with OpenTelemetry for Node.js

Introduction

In a microservices architecture, a single user request can traverse 5-10 services. When something goes wrong — a slow response, an error, a timeout — you need to trace the request across all services to find the root cause. Distributed tracing with OpenTelemetry gives you this visibility by propagating trace context across service boundaries.

Why Distributed Tracing?

Without tracing:
  User: "The checkout is slow"
  Developer: Checks API Gateway logs... looks fine (200ms)
  Developer: Checks Order Service logs... looks fine (150ms)
  Developer: Checks Payment Service logs... AH HA! (3500ms)
  Developer: Checks Payment Service code... calls Fraud Service
  Developer: Checks Fraud Service logs... (2800ms for ML model)

  Time to debug: 45 minutes

With tracing:
  User: "The checkout is slow"
  Developer: Opens trace in Jaeger/Zipkin
  Developer: Sees full request waterfall instantly:

  API Gateway ─────────────────────── 4200ms
    └─ Order Service ──────────────── 3800ms
         └─ Payment Service ───────── 3500ms
              └─ Fraud Service ────── 2800ms ← BOTTLENECK
                   └─ ML Model ────── 2500ms

  Time to debug: 2 minutes

Setup

// tracing.js - MUST be imported BEFORE any other modules
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { OTLPMetricExporter } from '@opentelemetry/exporter-metrics-otlp-http';
import { PeriodicExportingMetricReader } from '@opentelemetry/sdk-metrics';
import { Resource } from '@opentelemetry/resources';
import { ATTR_SERVICE_NAME, ATTR_SERVICE_VERSION } from '@opentelemetry/semantic-conventions';

const sdk = new NodeSDK({
  resource: new Resource({
    [ATTR_SERVICE_NAME]: process.env.SERVICE_NAME || 'order-service',
    [ATTR_SERVICE_VERSION]: process.env.SERVICE_VERSION || '1.0.0',
    environment: process.env.NODE_ENV || 'development',
  }),

  // Trace exporter
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT || 'http://otel-collector:4318/v1/traces',
  }),

  // Metric exporter
  metricReader: new PeriodicExportingMetricReader({
    exporter: new OTLPMetricExporter({
      url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT || 'http://otel-collector:4318/v1/metrics',
    }),
    exportIntervalMillis: 30000,
  }),

  // Auto-instrumentation for common libraries
  instrumentations: [
    getNodeAutoInstrumentations({
      '@opentelemetry/instrumentation-express': { enabled: true },
      '@opentelemetry/instrumentation-http': { enabled: true },
      '@opentelemetry/instrumentation-pg': { enabled: true },
      '@opentelemetry/instrumentation-redis': { enabled: true },
      '@opentelemetry/instrumentation-ioredis': { enabled: true },
    }),
  ],
});

sdk.start();

// Graceful shutdown
process.on('SIGTERM', () => {
  sdk.shutdown().then(() => process.exit(0));
});

export default sdk;

Manual Spans

import { trace, SpanStatusCode, context } from '@opentelemetry/api';

const tracer = trace.getTracer('order-service');

// Add custom spans for business logic
async function processOrder(orderData) {
  return tracer.startActiveSpan('processOrder', async (span) => {
    try {
      span.setAttribute('order.id', orderData.orderId);
      span.setAttribute('order.total', orderData.total);
      span.setAttribute('order.items_count', orderData.items.length);

      // Child span for validation
      const validationResult = await tracer.startActiveSpan('validateOrder', async (validationSpan) => {
        const result = await validateItems(orderData.items);
        validationSpan.setAttribute('validation.items_checked', result.itemsChecked);
        validationSpan.end();
        return result;
      });

      // Child span for payment
      const paymentResult = await tracer.startActiveSpan('chargePayment', async (paymentSpan) => {
        paymentSpan.setAttribute('payment.amount', orderData.total);
        paymentSpan.setAttribute('payment.method', orderData.paymentMethod);

        const result = await paymentService.charge(orderData);
        paymentSpan.setAttribute('payment.charge_id', result.chargeId);
        paymentSpan.end();
        return result;
      });

      span.setAttribute('order.status', 'completed');
      span.setStatus({ code: SpanStatusCode.OK });
      return { success: true, chargeId: paymentResult.chargeId };

    } catch (err) {
      span.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
      span.recordException(err);
      throw err;
    } finally {
      span.end();
    }
  });
}

Context Propagation Across Services

import { context, propagation, trace } from '@opentelemetry/api';

// Service A: Make HTTP call to Service B (auto-instrumented)
// The HTTP instrumentation automatically injects trace context headers
const response = await fetch('http://order-service:3002/api/orders', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(orderData),
});
// Headers like 'traceparent' are added automatically

// For manual propagation (e.g., message queues):
function injectTraceContext(carrier) {
  propagation.inject(context.active(), carrier);
  return carrier;
}

function extractTraceContext(carrier) {
  return propagation.extract(context.active(), carrier);
}

// Producer: Inject into message headers
async function publishEvent(topic, data) {
  const headers = {};
  injectTraceContext(headers);

  await producer.send({
    topic,
    messages: [{
      value: JSON.stringify(data),
      headers, // Contains traceparent header
    }],
  });
}

// Consumer: Extract from message headers
async function handleMessage(message) {
  const parentContext = extractTraceContext(message.headers);

  // Create child span in the extracted context
  context.with(parentContext, () => {
    tracer.startActiveSpan('processMessage', async (span) => {
      span.setAttribute('messaging.topic', message.topic);
      await processEvent(JSON.parse(message.value));
      span.end();
    });
  });
}

Custom Metrics

import { metrics } from '@opentelemetry/api';

const meter = metrics.getMeter('order-service');

// Counter
const orderCounter = meter.createCounter('orders.created', {
  description: 'Number of orders created',
  unit: 'orders',
});

// Histogram
const orderValueHistogram = meter.createHistogram('orders.value', {
  description: 'Distribution of order values',
  unit: 'USD',
});

// UpDownCounter (can go up and down)
const activeOrders = meter.createUpDownCounter('orders.active', {
  description: 'Number of currently active orders',
});

// Usage
app.post('/api/orders', async (req, res) => {
  const order = await createOrder(req.body);

  orderCounter.add(1, {
    'order.type': order.type,
    'order.region': order.region,
  });

  orderValueHistogram.record(order.total, {
    'order.type': order.type,
  });

  activeOrders.add(1);

  res.json(order);
});

Common Pitfalls

  1. Importing tracing after other modules: OpenTelemetry must be initialized before importing Express, pg, ioredis, etc. The auto-instrumentation patches these modules at import time.
  2. Not ending spans: Unclosed spans leak memory and produce incomplete traces. Always use try/finally.
  3. Too many spans: Creating a span for every function call creates noise. Span meaningful operations only.
  4. Not propagating context in async boundaries: Use context.with() when crossing async boundaries like message queues.

Summary

OpenTelemetry provides a unified standard for distributed tracing, metrics, and logging. Auto-instrumentation captures traces for Express, HTTP, PostgreSQL, and Redis with zero code changes. Manual spans add business-context to traces. Context propagation links spans across services via HTTP headers or message queue headers. Use Jaeger for trace visualization and Prometheus for metrics. Always initialize OpenTelemetry before importing application modules.

Common Problems & Solutions

1 Traces show disconnected spans (no parent-child relationship)
Cause: Trace context is not being propagated between services. Services create independent root spans.
Solution: Ensure HTTP auto-instrumentation is enabled (it propagates traceparent header). For message queues, manually inject/extract trace context.
2 Auto-instrumentation not capturing database queries
Cause: OpenTelemetry SDK was initialized after the database client was imported.
Solution: Import and initialize the tracing module as the very first import in your application entry point, before any other modules.
