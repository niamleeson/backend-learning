Autogenerate, Branching, and Migration Strategies

Intuition

Autogenerate is like a diff tool for your database schema. Branching lets multiple developers evolve the schema in parallel, merging when ready.

Autogenerate and Branching Strategies

Alembic can auto-detect differences between your models and the database, and it supports branching and merging for team workflows.

# Autogenerate migration from model changes
# $ alembic revision --autogenerate -m "add posts table"

# Autogenerate detects:
# - New/dropped tables, added/removed columns
# - Changed column types, nullable, server_default
# - Added/removed indexes and foreign keys

# Autogenerate does NOT detect:
# - Table or column renames (seen as drop + add)
# - Changes to CHECK constraints or triggers

# Branch management
# $ alembic heads              # show all branch heads
# $ alembic merge -m "merge"   # merge two branches

# versions/abc_branch1.py
revision = 'abc'
down_revision = 'base_rev'

# versions/def_branch2.py
revision = 'def'
down_revision = 'base_rev'  # same parent = branch

# Merge migration
# $ alembic merge abc def -m "merge branches"
# Creates: down_revision = ('abc', 'def')

Key Takeaways: - --autogenerate compares model metadata to the live database and generates migration ops. - Always review autogenerated migrations; they may miss renames or produce incorrect diffs. - Use alembic merge when two developers create migrations from the same base revision.

Exercises

  1. Autogenerate and Review

    Add a new model to your project, run autogenerate, and review the generated migration for correctness before running it.

    Hint

    alembic revision --autogenerate -m 'description', then inspect the generated upgrade/downgrade functions.

  2. Resolve a Branch Conflict

    Simulate two developers creating migrations from the same base. Use alembic merge to resolve the multiple heads.

    Hint

    Create two revisions with the same down_revision, run alembic heads, then alembic merge.

Common Problems & Solutions

1 Autogenerate creates drop + create instead of rename.
Cause: Alembic cannot detect renames; it sees the old table gone and a new one added.
Solution: Manually edit the migration to use op.rename_table() instead of drop + create.
2 Multiple heads error when running alembic upgrade head.
Cause: Two migration branches exist without a merge migration.
Solution: Run alembic merge -m 'merge' to create a merge point, then upgrade.
