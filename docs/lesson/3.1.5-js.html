Configuration Management and Environment Variables

Intuition

Configuration is like the settings panel on a machine. The machine (your code) stays the same, but the settings (environment variables) change depending on where it is installed — the factory floor (development), the testing lab (staging), or the customer site (production).

Configuration Management and Environment Variables

Introduction

Every real application needs different settings for different environments: database URLs, API keys, feature flags, port numbers. Hardcoding these values is a critical anti-pattern — it leads to secrets leaking into version control, painful deployments, and code that cannot adapt to its environment. In this lesson you will learn how to properly manage configuration in an Express application using environment variables, the dotenv library, and structured configuration modules.

Environment Variables Basics

Environment variables are key-value pairs set outside your application code, typically by the operating system or deployment platform:

# Setting env vars in the shell
export PORT=3000
export DATABASE_URL="postgresql://user:pass@localhost:5432/mydb"
export JWT_SECRET="super-secret-key"
export NODE_ENV="development"

# Or inline when running the app
PORT=3000 NODE_ENV=production node dist/server.js

In Node.js, access them via process.env:

const port = process.env.PORT;         // "3000" (always a string!)
const dbUrl = process.env.DATABASE_URL;
const env = process.env.NODE_ENV;       // "development" | "production"

Using dotenv

In development, you typically store environment variables in a .env file. The dotenv package loads them into process.env:

npm install dotenv

Create .env in your project root:

# .env
PORT=3000
NODE_ENV=development
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/myapp
JWT_SECRET=my-development-secret-key
JWT_EXPIRES_IN=1h
CORS_ORIGIN=http://localhost:5173
LOG_LEVEL=debug
REDIS_URL=redis://localhost:6379

Critical: Add .env to .gitignore:

# .gitignore
.env
.env.local
.env.*.local

Load it as early as possible in your application:

// src/server.ts
import dotenv from 'dotenv';
dotenv.config(); // Load .env BEFORE importing anything else

import app from './app';
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});

Creating a Configuration Module

Rather than scattering process.env calls throughout your codebase, centralize configuration in a typed module:

// src/config/index.ts
import dotenv from 'dotenv';

dotenv.config();

function requireEnv(key: string): string {
  const value = process.env[key];
  if (!value) {
    throw new Error(`Missing required environment variable: ${key}`);
  }
  return value;
}

export const config = {
  env: process.env.NODE_ENV || 'development',
  port: parseInt(process.env.PORT || '3000', 10),
  isProduction: process.env.NODE_ENV === 'production',
  isDevelopment: process.env.NODE_ENV !== 'production',

  database: {
    url: requireEnv('DATABASE_URL'),
    poolMin: parseInt(process.env.DB_POOL_MIN || '2', 10),
    poolMax: parseInt(process.env.DB_POOL_MAX || '10', 10),
  },

  jwt: {
    secret: requireEnv('JWT_SECRET'),
    expiresIn: process.env.JWT_EXPIRES_IN || '1h',
    refreshExpiresIn: process.env.JWT_REFRESH_EXPIRES_IN || '7d',
  },

  cors: {
    origin: process.env.CORS_ORIGIN || 'http://localhost:5173',
  },

  redis: {
    url: process.env.REDIS_URL || 'redis://localhost:6379',
  },

  logging: {
    level: process.env.LOG_LEVEL || 'info',
  },
} as const;

Now import and use the config everywhere:

// src/app.ts
import { config } from './config';
import cors from 'cors';

const app = express();
app.use(cors({ origin: config.cors.origin }));

// src/modules/auth/auth.service.ts
import { config } from '../../config';
import jwt from 'jsonwebtoken';

function generateToken(userId: number) {
  return jwt.sign({ userId }, config.jwt.secret, {
    expiresIn: config.jwt.expiresIn,
  });
}

Validation with Zod

For robust configuration, validate environment variables at startup using Zod:

// src/config/env.ts
import { z } from 'zod';
import dotenv from 'dotenv';

dotenv.config();

const envSchema = z.object({
  NODE_ENV: z.enum(['development', 'production', 'test']).default('development'),
  PORT: z.string().transform(Number).default('3000'),
  DATABASE_URL: z.string().url(),
  JWT_SECRET: z.string().min(32, 'JWT_SECRET must be at least 32 characters'),
  JWT_EXPIRES_IN: z.string().default('1h'),
  CORS_ORIGIN: z.string().url().default('http://localhost:5173'),
  REDIS_URL: z.string().optional(),
  LOG_LEVEL: z.enum(['error', 'warn', 'info', 'debug']).default('info'),
});

type Env = z.infer<typeof envSchema>;

function validateEnv(): Env {
  const result = envSchema.safeParse(process.env);

  if (!result.success) {
    console.error('Invalid environment variables:');
    console.error(result.error.format());
    process.exit(1);
  }

  return result.data;
}

export const env = validateEnv();

This approach catches misconfiguration at startup, not at runtime when a user hits a broken code path.

Environment-Specific Configuration

// src/config/index.ts
import { env } from './env';

const baseConfig = {
  port: env.PORT,
  jwt: { secret: env.JWT_SECRET, expiresIn: env.JWT_EXPIRES_IN },
};

const developmentConfig = {
  ...baseConfig,
  logging: { level: 'debug' as const, prettyPrint: true },
  cors: { origin: '*' },
};

const productionConfig = {
  ...baseConfig,
  logging: { level: 'info' as const, prettyPrint: false },
  cors: { origin: env.CORS_ORIGIN },
};

const testConfig = {
  ...baseConfig,
  logging: { level: 'error' as const, prettyPrint: false },
  cors: { origin: '*' },
};

const configs = {
  development: developmentConfig,
  production: productionConfig,
  test: testConfig,
};

export const config = configs[env.NODE_ENV];

Multiple .env Files

For different environments, use multiple env files:

.env                # Default / shared values
.env.development    # Development overrides
.env.test           # Test overrides
.env.production     # Production overrides (not committed!)
.env.example        # Template showing required variables (committed)

// Load environment-specific .env file
import dotenv from 'dotenv';
const envFile = `.env.${process.env.NODE_ENV || 'development'}`;
dotenv.config({ path: envFile });
dotenv.config(); // Fallback to .env for defaults

Create a .env.example template and commit it:

# .env.example - Copy to .env and fill in values
PORT=3000
NODE_ENV=development
DATABASE_URL=postgresql://user:password@localhost:5432/dbname
JWT_SECRET=change-me-to-a-long-random-string
CORS_ORIGIN=http://localhost:5173

Using Configuration in Express

// src/server.ts
import { config } from './config';
import app from './app';

const server = app.listen(config.port, () => {
  console.log(`[${config.env}] Server running on port ${config.port}`);
});

// Graceful shutdown
const gracefulShutdown = (signal: string) => {
  console.log(`${signal} received. Starting graceful shutdown...`);
  server.close(() => {
    console.log('Server closed');
    process.exit(0);
  });

  // Force close after 10 seconds
  setTimeout(() => {
    console.error('Forced shutdown after timeout');
    process.exit(1);
  }, 10000);
};

process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
process.on('SIGINT', () => gracefulShutdown('SIGINT'));

Security Considerations

NEVER DO THIS:

  // Hardcoded secrets in source code
  const JWT_SECRET = "my-secret-key";

  // Logging secrets
  console.log("Config:", process.env);

  // Committing .env files to git

  // Using the same secrets across environments

Best practices:

  1. Never commit secrets to version control
  2. Use different secrets per environment
  3. Rotate secrets periodically
  4. Use a secrets manager in production (AWS Secrets Manager, HashiCorp Vault)
  5. Validate at startup so misconfigurations fail fast
  6. Use strong, random values for secrets (not dictionary words)

Generate a strong secret:

node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"

Common Pitfalls

1. Loading dotenv Too Late

// WRONG: config module loaded before dotenv
import { config } from './config'; // process.env not yet populated!
import dotenv from 'dotenv';
dotenv.config();

// CORRECT: load dotenv first
import dotenv from 'dotenv';
dotenv.config();
import { config } from './config';

2. Treating Environment Variables as Non-Strings

// WRONG: comparing string to number
if (process.env.PORT === 3000) { ... } // Always false!

// CORRECT: parse or compare as string
if (parseInt(process.env.PORT || '0', 10) === 3000) { ... }

3. Not Providing Defaults for Optional Variables

// WRONG: crashes if LOG_LEVEL not set
const level = process.env.LOG_LEVEL.toUpperCase();

// CORRECT: provide default
const level = (process.env.LOG_LEVEL || 'info').toUpperCase();

Summary

Configuration management is about separating what changes between environments from what stays the same. Use dotenv for local development, centralize configuration in a typed module, validate with Zod at startup, never commit secrets, and fail fast on missing required values. These practices prevent an entire class of deployment and security bugs.

Exercises

  1. Build a Validated Config Module

    Create a configuration module using Zod that validates: PORT (number, default 3000), DATABASE_URL (required URL string), JWT_SECRET (required, minimum 32 chars), NODE_ENV (enum: development/production/test), and LOG_LEVEL (enum with default 'info'). The app should exit with a clear error message if validation fails.

    Hint

    Use z.string().transform(Number) for PORT. Use safeParse and check result.success. Log result.error.format() for readable error messages.

  2. Multi-Environment Configuration

    Set up a config system that loads .env.development, .env.test, or .env.production based on NODE_ENV, with .env as a fallback. Create a .env.example template. Test by switching NODE_ENV and verifying different values load.

    Hint

    Call dotenv.config() twice: first with the environment-specific path, then with no path for defaults. Earlier values are not overwritten.

  3. Graceful Shutdown Handler

    Implement a graceful shutdown for your Express server that listens for SIGTERM and SIGINT, stops accepting new connections, waits for in-flight requests to complete (with a timeout), closes database connections, and then exits.

    Hint

    Use server.close(callback) to stop accepting connections. Set a setTimeout for forced exit. Chain cleanup operations (db disconnect, etc.) before calling process.exit(0).

Common Problems & Solutions

1 Environment variables are undefined even though .env file exists
Cause: dotenv.config() was called after modules that read process.env, or the .env file is not in the project root directory.
Solution: Call dotenv.config() as the very first thing in your entry file, before any imports that read config. If .env is not in the root, pass the path: dotenv.config({ path: './config/.env' }).
2 Application works locally but fails in production with missing env vars
Cause: The .env file is not deployed to production (correctly), but the required environment variables are not set in the production environment.
Solution: Set environment variables through your deployment platform (Docker env, Kubernetes secrets, AWS Parameter Store). Use a .env.example file to document required variables. Validate at startup with Zod to catch this immediately.
3 Secrets appear in logs or error messages
Cause: Logging process.env or the full config object, or including secrets in error messages.
Solution: Never log the entire process.env or config object. Redact sensitive fields in logging. Use a config structure that separates public config from secrets.
