Webhooks and Event Schemas

Intuition

Webhooks are like push notifications for servers. Instead of constantly checking (polling) whether something happened, you register your phone number and get a call when the event occurs. The caller will keep trying if you do not answer.

Webhooks & Event Schemas

Webhooks are HTTP callbacks that enable real-time integration between systems. Combined with well-defined event schemas, they provide a lightweight alternative to full message brokers for inter-system event notification.

Core Concepts

  • Webhooks: When an event occurs, the source system sends an HTTP POST request to a pre-registered URL with event data in the body. The receiver processes the event and returns a 2xx status.
  • Event Schema Design: Define clear, versioned schemas for webhook payloads. Include event type, timestamp, unique event ID, schema version, and the payload. Use JSON Schema, Avro, or Protobuf for validation.
  • Delivery Guarantees: Webhooks are inherently at-least-once. If the receiver fails to respond with 2xx, the sender retries. Receivers must be idempotent to handle duplicate deliveries.
  • Security: Verify webhook authenticity using HMAC signatures. The sender signs the payload with a shared secret; the receiver verifies the signature before processing.
  • Reliability Patterns: Implement retry with exponential backoff on the sender side. On the receiver side, accept and queue webhooks quickly (respond 200 immediately) and process asynchronously.

Key Takeaways

  1. Webhooks are simple to implement but hard to make reliable—plan for retries, idempotency, and signature verification from day one.
  2. Schema versioning is essential; changing a webhook payload without versioning breaks all subscribers.
  3. Accept webhooks fast and process async—never do heavy work in the webhook handler before responding.

Exercises

  1. Build a Webhook System with HMAC Verification

    Create a webhook sender that signs payloads with HMAC-SHA256 and a shared secret. Build a receiver that verifies the signature header, rejects invalid requests, and processes valid events. Include retry logic on the sender side with exponential backoff.

    Hint

    Sender: signature = hmac.new(secret, body, hashlib.sha256).hexdigest(), send as X-Signature header. Receiver: recompute the signature and compare with hmac.compare_digest() to prevent timing attacks.

  2. Design a Versioned Event Schema

    Define a JSON Schema for webhook events with required fields: event_id, event_type, version, timestamp, data. Create two versions of an order.completed event where v2 adds a shipping_address field. Implement a handler that processes both versions.

    Hint

    Use jsonschema to validate incoming payloads. Route to version-specific handlers based on the version field. The v1 handler uses default values for fields added in v2.

Common Problems & Solutions

1 Webhook delivery fails repeatedly and the sender gives up, causing lost events.
Cause: The receiver's endpoint is intermittently down or too slow, and the sender's retry window is too short.
Solution: Implement a generous retry policy (e.g., 5 retries over 24 hours with exponential backoff). Provide a webhook event log API so receivers can replay missed events. Consider a dead letter mechanism for permanently failed deliveries.
2 Webhook receiver processes the same event multiple times, causing duplicate charges.
Cause: The receiver processed the event but returned a timeout or 5xx error, causing the sender to retry and the receiver to process it again.
Solution: Store the `event_id` in a database before processing. Check for the event ID at the start of the handler and skip if already processed. Return 200 immediately and process asynchronously.
