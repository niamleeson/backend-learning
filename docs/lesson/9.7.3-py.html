Idempotency and Distributed Transactions

Intuition

Idempotency is like an elevator button—pressing it five times has the same effect as pressing it once. The outbox pattern is like writing a letter and putting it in your outbox tray as part of finishing your task, rather than walking to the mailbox separately and risking you forget.

Idempotency & Distributed Transactions

In distributed systems, network failures, retries, and message duplication are facts of life. Idempotency ensures operations can be safely retried, and distributed transaction patterns manage consistency across service boundaries.

Core Concepts

  • Idempotency: An operation is idempotent if performing it multiple times has the same effect as performing it once. GET, PUT, and DELETE are naturally idempotent in HTTP; POST is not without explicit design.
  • Idempotency Keys: A unique identifier (UUID, request ID) attached to each request. The server stores processed keys and returns the cached result for duplicates instead of re-executing.
  • Two-Phase Commit (2PC): A coordinator asks all participants to prepare (vote yes/no), then commits or aborts based on unanimous agreement. Provides atomicity but is blocking—if the coordinator fails after prepare, participants are stuck.
  • Outbox Pattern: Instead of publishing events directly, write events to an outbox table in the same database transaction as the business data. A separate process reads the outbox and publishes to the message broker, ensuring atomicity.
  • Change Data Capture (CDC): Use the database's transaction log (WAL) to capture changes and publish them as events. Tools like Debezium read the log and push changes to Kafka, ensuring no events are lost.

Key Takeaways

  1. Make all state-changing operations idempotent—it is the single most important pattern for distributed system reliability.
  2. Avoid 2PC in microservices; prefer the Saga pattern with compensating transactions for distributed coordination.
  3. The outbox pattern solves the dual-write problem (writing to a database and publishing an event) by making it a single atomic operation.

Exercises

  1. Implement an Idempotent API Endpoint

    Build a payment processing API that accepts an Idempotency-Key header. Store the key and result in PostgreSQL. For duplicate requests, return the stored result without re-processing. Handle concurrent duplicate requests with database-level locking.

    Hint

    Use INSERT ... ON CONFLICT (idempotency_key) DO NOTHING to atomically detect duplicates. Use SELECT FOR UPDATE or advisory locks to prevent concurrent processing of the same key.

  2. Implement the Outbox Pattern with Debezium

    Set up a Django application that writes business data and outbox events in the same transaction. Configure Debezium to capture changes from the outbox table's WAL and publish them to Kafka. Verify that every database write produces exactly one Kafka event.

    Hint

    Create an outbox table with id, aggregate_type, aggregate_id, event_type, payload. In your Django view, use transaction.atomic() to write both business data and the outbox row. Configure Debezium's PostgreSQL connector with table.include.list=public.outbox.

Common Problems & Solutions

1 Idempotency key storage grows unboundedly and slows down lookups.
Cause: Processed idempotency keys are never cleaned up, accumulating millions of rows over time.
Solution: Add a `created_at` timestamp and run a periodic job that deletes keys older than the retry window (e.g., 24-48 hours). Add an index on `idempotency_key` for fast lookups. Consider using Redis with TTL for short-lived keys.
2 Events are published to Kafka but the corresponding database transaction was rolled back.
Cause: The application publishes to Kafka directly (not via outbox) and the subsequent database write fails, creating a dual-write inconsistency.
Solution: Use the outbox pattern: write events to a database table within the same transaction as the business data. A separate process (or CDC tool like Debezium) reads the outbox and publishes to Kafka, ensuring atomicity.
