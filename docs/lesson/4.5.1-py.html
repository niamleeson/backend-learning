Sessions and Cookies

Intuition

Sessions give your stateless HTTP application the illusion of memory -- they let you recognize returning users and carry state across requests. Cookies are the transport mechanism, and signing ensures nobody tampers with the cargo.

Sessions and Cookies

HTTP is stateless, so sessions and cookies provide a mechanism to persist data across requests. Flask uses signed cookies for sessions by default -- the session data is stored client-side but cryptographically signed to prevent tampering. For server-side sessions, extensions like Flask-Session store data on the server.

from flask import Flask, session, redirect, url_for, request, jsonify
from datetime import timedelta

app = Flask(__name__)
app.config['SECRET_KEY'] = 'super-secret-key-change-in-production'
app.config['PERMANENT_SESSION_LIFETIME'] = timedelta(days=7)

@app.route('/login', methods=['POST'])
def login():
    data = request.get_json()
    # In production, verify credentials against a database
    if data.get('username') == 'admin' and data.get('password') == 'secret':
        session.permanent = True
        session['user_id'] = 1
        session['username'] = data['username']
        session['role'] = 'admin'
        return jsonify({'message': 'Logged in', 'username': data['username']})
    return jsonify({'error': 'Invalid credentials'}), 401

@app.route('/profile')
def profile():
    if 'user_id' not in session:
        return jsonify({'error': 'Not authenticated'}), 401
    return jsonify({
        'user_id': session['user_id'],
        'username': session['username'],
        'role': session['role']
    })

@app.route('/logout', methods=['POST'])
def logout():
    session.clear()
    return jsonify({'message': 'Logged out'})

@app.route('/preferences', methods=['POST'])
def set_preferences():
    data = request.get_json()
    resp = jsonify({'message': 'Preferences saved'})
    resp.set_cookie('theme', data.get('theme', 'light'),
                    max_age=60*60*24*365, httponly=True,
                    samesite='Lax', secure=True)
    return resp

Key Takeaways: - Flask's default session uses signed cookies; data is visible to the client but tamper-proof. - Set session.permanent = True and PERMANENT_SESSION_LIFETIME to control expiration. - Use httponly, secure, and samesite flags on cookies to enhance security.

Exercises

  1. Shopping Cart with Sessions

    Build a simple shopping cart using Flask sessions. Implement endpoints to add items, view the cart, remove items, and clear the cart. Store cart data in the session.

    Hint

    Initialize session['cart'] as an empty list on first access using session.setdefault('cart', []). Remember that sessions need to be marked as modified if you mutate a nested object.

  2. Server-Side Sessions

    Install Flask-Session and configure it to store sessions in the filesystem or Redis instead of cookies. Compare the cookie size and behavior with the default session implementation.

    Hint

    Install flask-session, set SESSION_TYPE='filesystem' or 'redis', and initialize Session(app). The session interface is the same; only the storage backend changes.

Common Problems & Solutions

1 Session data is lost between requests even though session values are set.
Cause: SECRET_KEY is not configured, so Flask cannot sign the session cookie and silently drops the session.
Solution: Set app.config['SECRET_KEY'] to a strong random string. Without it, sessions will not persist.
2 Modifying a mutable object in the session (like appending to a list) does not persist.
Cause: Flask only detects session changes when you reassign top-level session keys. Mutating a nested list or dict is not detected.
Solution: After modifying a nested object, set session.modified = True to force Flask to save the session.
