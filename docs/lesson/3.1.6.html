Common Table Expressions (CTEs)

Intuition

CTEs are like naming intermediate variables in a math proof. Instead of one giant nested expression, you break it into labeled steps that build on each other.

Common Table Expressions (CTEs)

CTEs use the WITH clause to define temporary named result sets that make complex queries readable and composable.

-- Basic CTE
WITH active_users AS (
    SELECT id, name, email
    FROM users
    WHERE status = 'active'
)
SELECT au.name, COUNT(o.id) AS order_count
FROM active_users au
JOIN orders o ON au.id = o.user_id
GROUP BY au.name;

-- Chained CTEs
WITH monthly AS (
    SELECT DATE_TRUNC('month', created_at) AS month,
           SUM(amount) AS revenue
    FROM sales GROUP BY 1
),
growth AS (
    SELECT month, revenue,
           LAG(revenue) OVER (ORDER BY month) AS prev
    FROM monthly
)
SELECT month, revenue,
       ROUND((revenue - prev) / prev * 100, 1) AS pct_growth
FROM growth;

Key Takeaways: - CTEs improve readability by breaking complex logic into named steps. - Multiple CTEs can be chained with commas. Each can reference previously defined CTEs. - Recursive CTEs use WITH RECURSIVE for tree or graph traversal.

Exercises

  1. Multi-Step Sales Report

    Write chained CTEs: first compute daily totals, then weekly averages, then return weeks where the average exceeds $1000.

    Hint

    CTE1 groups by date, CTE2 groups CTE1 by week with AVG, outer query filters HAVING avg > 1000.

  2. Recursive Org Chart

    Use WITH RECURSIVE to traverse an employees table (id, name, manager_id) and list all reports under a given manager.

    Hint

    Base case: WHERE manager_id = :target. Recursive step: JOIN employees ON manager_id = cte.id.

Common Problems & Solutions

1 Recursive CTE runs infinitely and times out.
Cause: Circular references in the data or missing termination condition.
Solution: Add a depth counter and WHERE depth < N to cap recursion, or use CYCLE detection in PostgreSQL 14+.
2 CTE performance is worse than a subquery.
Cause: In older PostgreSQL versions, CTEs act as optimization fences preventing predicate pushdown.
Solution: Use PostgreSQL 12+ which inlines non-recursive CTEs, or add MATERIALIZED/NOT MATERIALIZED hints.
