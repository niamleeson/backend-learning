Custom Exceptions

Intuition

Custom exceptions turn vague runtime failures into named, structured errors that callers can handle precisely — they are your API's error contract.

Custom exceptions let you define domain-specific error types, making your code's failure modes explicit and catchable at the right granularity.

class AppError(Exception):
    """Base exception for the application."""

class ValidationError(AppError):
    def __init__(self, field, message):
        self.field = field
        self.message = message
        super().__init__(f"{field}: {message}")

class NotFoundError(AppError):
    pass

def get_user(user_id):
    if not isinstance(user_id, int):
        raise ValidationError("user_id", "must be an integer")
    if user_id <= 0:
        raise NotFoundError(f"User {user_id} not found")
    return {"id": user_id, "name": "Alice"}

try:
    get_user("abc")
except ValidationError as e:
    print(f"Validation failed on {e.field}: {e.message}")
except AppError:
    print("General app error")
  • Create a base exception for your application; all custom exceptions inherit from it.
  • Add attributes (like field, code) to carry structured error context.
  • Always call super().__init__() so the exception message is set properly.
  • Catch the base exception as a fallback after catching specific subtypes.

Exercises

  1. HTTP Error Hierarchy

    Create a base HttpError with a status_code attribute, then subclasses NotFound(404), Unauthorized(401), and ServerError(500). Each should set the code in __init__.

    Hint

    Pass status_code to __init__, store it as self.status_code, and call super().__init__().

  2. Retry on Custom Exception

    Write a decorator retry(times) that retries a function up to times if it raises a TransientError (custom exception), then re-raises on final failure.

    Hint

    Wrap the function call in a for loop with try/except, re-raising after exhausting attempts.

Common Problems & Solutions

1 Forgetting `super().__init__()` leads to empty exception messages when printed or logged.
Cause: The base `Exception.__init__` sets `self.args`, which `str()` and `repr()` rely on.
Solution: Always call `super().__init__(message)` in your custom `__init__`.
2 Raising generic `Exception` instead of a custom type makes it impossible to catch selectively.
Cause: Callers cannot distinguish between your error and unrelated `Exception` subclasses.
Solution: Define and raise domain-specific exception classes.
