Configuration: ConfigModule and Environment Variables

Intuition

ConfigModule is like a company's HR policies document. Instead of each employee (service) having their own set of rules, there is one centralized, validated source of truth. When someone needs a policy detail, they check the official document (ConfigService) rather than passing notes around informally (hardcoded values).

Configuration: ConfigModule and Environment Variables

Introduction

Every application needs configuration -- database URLs, API keys, feature flags, port numbers. Hardcoding these values is a security risk and makes deployment across environments impossible. NestJS provides @nestjs/config, a wrapper around the dotenv library, with type safety, validation, and namespaced configurations.

Installation

npm i @nestjs/config

Basic Setup

// app.module.ts
import { Module } from '@nestjs/common';
import { ConfigModule } from '@nestjs/config';

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true, // Available everywhere without importing
    }),
  ],
})
export class AppModule {}

Create a .env file in the project root:

DATABASE_URL=postgresql://user:pass@localhost:5432/mydb
JWT_SECRET=super-secret-key
PORT=3000
NODE_ENV=development

Using ConfigService

import { Injectable } from '@nestjs/common';
import { ConfigService } from '@nestjs/config';

@Injectable()
export class AppService {
  constructor(private config: ConfigService) {}

  getDatabaseUrl(): string {
    return this.config.get<string>('DATABASE_URL');
  }

  getPort(): number {
    return this.config.get<number>('PORT', 3000); // Default value
  }

  isProduction(): boolean {
    return this.config.get('NODE_ENV') === 'production';
  }
}

Using in main.ts

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  const config = app.get(ConfigService);

  const port = config.get<number>('PORT', 3000);
  await app.listen(port);
  console.log(`Application running on port ${port}`);
}

Configuration Namespaces

For larger applications, organize configuration into namespaces:

// config/database.config.ts
import { registerAs } from '@nestjs/config';

export default registerAs('database', () => ({
  host: process.env.DB_HOST || 'localhost',
  port: parseInt(process.env.DB_PORT, 10) || 5432,
  name: process.env.DB_NAME || 'mydb',
  username: process.env.DB_USERNAME || 'postgres',
  password: process.env.DB_PASSWORD || 'postgres',
  ssl: process.env.DB_SSL === 'true',
}));

// config/jwt.config.ts
import { registerAs } from '@nestjs/config';

export default registerAs('jwt', () => ({
  secret: process.env.JWT_SECRET,
  expiresIn: process.env.JWT_EXPIRES_IN || '1h',
  refreshExpiresIn: process.env.JWT_REFRESH_EXPIRES_IN || '7d',
}));

// config/app.config.ts
import { registerAs } from '@nestjs/config';

export default registerAs('app', () => ({
  port: parseInt(process.env.PORT, 10) || 3000,
  environment: process.env.NODE_ENV || 'development',
  apiPrefix: process.env.API_PREFIX || 'api',
  corsOrigin: process.env.CORS_ORIGIN || '*',
}));

Register the namespaces:

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      load: [databaseConfig, jwtConfig, appConfig],
    }),
  ],
})
export class AppModule {}

Access namespaced config:

@Injectable()
export class DatabaseService {
  constructor(private config: ConfigService) {}

  getConnectionOptions() {
    return {
      host: this.config.get<string>('database.host'),
      port: this.config.get<number>('database.port'),
      database: this.config.get<string>('database.name'),
    };
  }
}

Strongly Typed Namespace Injection

For even better type safety, inject the namespace directly:

import { Inject, Injectable } from '@nestjs/common';
import { ConfigType } from '@nestjs/config';
import databaseConfig from './config/database.config';

@Injectable()
export class DatabaseService {
  constructor(
    @Inject(databaseConfig.KEY)
    private dbConfig: ConfigType<typeof databaseConfig>,
  ) {}

  connect() {
    // Full type inference -- no string keys needed
    console.log(this.dbConfig.host); // string
    console.log(this.dbConfig.port); // number
  }
}

Environment Validation with Joi

Validate environment variables at startup to fail fast:

npm i joi

import * as Joi from 'joi';

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      validationSchema: Joi.object({
        NODE_ENV: Joi.string()
          .valid('development', 'production', 'test')
          .default('development'),
        PORT: Joi.number().default(3000),
        DATABASE_URL: Joi.string().required(),
        JWT_SECRET: Joi.string().required().min(32),
        REDIS_URL: Joi.string().uri().optional(),
      }),
      validationOptions: {
        allowUnknown: true, // Don't reject unknown env vars
        abortEarly: false,  // Report all errors, not just the first
      },
    }),
  ],
})
export class AppModule {}

If validation fails, the app crashes on startup with a clear error:

Error: Config validation error:
"JWT_SECRET" is required
"DATABASE_URL" is required

Environment-Specific Files

ConfigModule.forRoot({
  envFilePath: [
    `.env.${process.env.NODE_ENV}.local`, // .env.development.local
    `.env.${process.env.NODE_ENV}`,        // .env.development
    '.env.local',                           // .env.local
    '.env',                                 // .env
  ],
})

Files are loaded in order. The first file found with a given variable wins.

Custom Configuration Factory

For complex configuration that needs computation:

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      load: [
        () => ({
          database: {
            url: process.env.DATABASE_URL,
            poolSize: process.env.NODE_ENV === 'production' ? 20 : 5,
            logging: process.env.NODE_ENV !== 'production',
          },
          cache: {
            ttl: parseInt(process.env.CACHE_TTL, 10) || 300,
            max: parseInt(process.env.CACHE_MAX, 10) || 100,
          },
        }),
      ],
    }),
  ],
})
export class AppModule {}

Expanding Variables

Enable variable expansion to reference other env vars:

# .env
BASE_URL=http://localhost
PORT=3000
APP_URL=${BASE_URL}:${PORT}

ConfigModule.forRoot({
  expandVariables: true,
})

Type-Safe Configuration Interface

Create a configuration interface for your entire app:

// config/configuration.interface.ts
export interface AppConfiguration {
  port: number;
  environment: string;
  database: {
    host: string;
    port: number;
    name: string;
    username: string;
    password: string;
  };
  jwt: {
    secret: string;
    expiresIn: string;
  };
  redis: {
    url: string;
    ttl: number;
  };
}

// config/configuration.ts
export default (): AppConfiguration => ({
  port: parseInt(process.env.PORT, 10) || 3000,
  environment: process.env.NODE_ENV || 'development',
  database: {
    host: process.env.DB_HOST || 'localhost',
    port: parseInt(process.env.DB_PORT, 10) || 5432,
    name: process.env.DB_NAME || 'mydb',
    username: process.env.DB_USERNAME || 'postgres',
    password: process.env.DB_PASSWORD || '',
  },
  jwt: {
    secret: process.env.JWT_SECRET || 'dev-secret',
    expiresIn: process.env.JWT_EXPIRES_IN || '1h',
  },
  redis: {
    url: process.env.REDIS_URL || 'redis://localhost:6379',
    ttl: parseInt(process.env.CACHE_TTL, 10) || 300,
  },
});

Security Best Practices

  1. Never commit .env files -- Add .env* to .gitignore (except .env.example).
  2. Provide .env.example -- Document all required variables without values.
  3. Validate on startup -- Fail fast with Joi validation.
  4. Use secrets managers in production -- AWS Secrets Manager, Vault, etc.
  5. Rotate secrets -- JWT secrets, API keys, database passwords.
# .env.example (commit this)
NODE_ENV=development
PORT=3000
DATABASE_URL=
JWT_SECRET=
REDIS_URL=

Common Pitfalls

  1. Missing isGlobal: true -- Without this, you must import ConfigModule in every module that needs ConfigService.
  2. Using process.env directly -- Bypasses NestJS's config system, loses validation and defaults. Always use ConfigService.
  3. Forgetting to install @nestjs/config -- The ConfigModule/ConfigService don't come with the core package.
  4. Type coercion -- process.env values are always strings. Always parseInt() for numbers and compare explicitly for booleans.

Exercises

  1. Set Up Namespaced Configuration

    Create a NestJS app with three config namespaces: app (port, environment), database (host, port, name, credentials), and mail (smtp host, port, from address). Use registerAs() for each. Inject the typed config into relevant services.

    Hint

    Create three files in a config/ folder. Use @Inject(databaseConfig.KEY) with ConfigType<typeof databaseConfig> for type-safe injection.

  2. Add Joi Validation

    Add comprehensive Joi validation to your ConfigModule. Include required fields (DATABASE_URL, JWT_SECRET), type checking (PORT must be a number), enum validation (NODE_ENV must be development/production/test), and minimum lengths (JWT_SECRET >= 32 chars).

    Hint

    Use validationOptions: { abortEarly: false } to see all validation errors at once. Test by removing required env vars.

  3. Environment-Specific Configs

    Set up environment-specific config files (.env.development, .env.production, .env.test) with different values. Create a service that logs the current configuration on startup to verify the correct file is loaded.

    Hint

    Use the envFilePath array option. Set NODE_ENV before starting: NODE_ENV=production npm run start.

Common Problems & Solutions

1 ConfigService returns undefined for all values
Cause: The `.env` file is not in the project root directory, or `ConfigModule.forRoot()` was not called.
Solution: Ensure .env is in the root (same level as package.json). Verify ConfigModule.forRoot() is in AppModule's imports. Check the envFilePath option if using custom locations.
2 ConfigService is not available in another module
Cause: ConfigModule was not configured with `isGlobal: true`, and the target module doesn't import ConfigModule.
Solution: Add `isGlobal: true` to ConfigModule.forRoot() options, or add `ConfigModule` to the imports array of each module that needs ConfigService.
3 Config validation passes but values are wrong
Cause: Multiple .env files exist and the wrong one takes precedence, or variable expansion is not enabled.
Solution: Check the envFilePath order (first match wins). Enable `expandVariables: true` if using variable references like ${VAR}. Log config values at startup to debug.
