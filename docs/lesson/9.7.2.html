Distributed Consensus: Raft and Paxos

Intuition

Distributed consensus is like a jury deliberation where members must reach a majority vote, even if some jurors leave the room temporarily. Raft simplifies this by electing a foreman (leader) who proposes verdicts and counts votes.

Distributed Consensus: Raft & Paxos

Distributed consensus is the problem of getting multiple nodes to agree on a single value, even when some nodes fail or messages are lost. Raft and Paxos are the two most important algorithms that solve this problem.

Core Concepts

  • Consensus Problem: Multiple nodes must agree on the same value (e.g., who is the leader, what is the committed log entry) despite failures. This is fundamental to replicated state machines, leader election, and distributed databases.
  • Paxos: The original consensus algorithm by Leslie Lamport. Uses proposers, acceptors, and learners across two phases (prepare/promise and accept/accepted). Correct but notoriously difficult to understand and implement.
  • Raft: Designed as an understandable alternative to Paxos. Separates consensus into three sub-problems: leader election, log replication, and safety. Used by etcd, Consul, CockroachDB, and TiKV.
  • Leader Election: In Raft, nodes start as followers. If a follower receives no heartbeat, it becomes a candidate and requests votes. The candidate with a majority becomes the leader for that term.
  • Log Replication: The leader receives client requests, appends them to its log, and replicates to followers. An entry is committed when a majority of nodes have it. Committed entries are guaranteed to be durable.

Key Takeaways

  1. Consensus requires a majority (quorum) of nodes to be available—a 3-node cluster tolerates 1 failure, 5 nodes tolerate 2.
  2. Raft is the practical choice for most systems due to its understandability and well-tested implementations.
  3. Consensus adds latency (majority must acknowledge) but provides the strongest guarantees for critical metadata and configuration.

Exercises

  1. Visualize Raft with the Interactive Simulator

    Use the Raft visualization at thesecretlivesofdata.com/raft to understand leader election, log replication, and network partitions. Then, deploy a 3-node etcd cluster and force a leader election by stopping the current leader. Observe the election process in the logs.

    Hint

    Run etcdctl endpoint status --cluster to identify the leader. Stop it with docker stop etcd-leader. Watch the remaining nodes' logs for raft.node: ... elected leader messages. Verify the cluster is still functional.

  2. Implement Simplified Raft Leader Election

    Build a simplified Raft leader election in Python using UDP sockets for 3 nodes. Implement heartbeats, election timeouts with randomization, vote requests, and term numbers. Demonstrate that a leader is elected and re-elected when the current leader stops.

    Hint

    Each node has a random election timeout between 150-300ms. On timeout without heartbeat, increment term, vote for self, and request votes from peers. On receiving a majority, become leader and send heartbeats every 50ms.

Common Problems & Solutions

1 A 3-node consensus cluster becomes unavailable when 2 nodes fail simultaneously.
Cause: Raft requires a majority quorum (2 of 3) to operate. With 2 nodes down, no quorum can be formed and the cluster cannot elect a leader or commit entries.
Solution: For higher fault tolerance, use a 5-node cluster (tolerates 2 failures). For 3-node clusters, ensure nodes are in different failure domains (availability zones). Implement automated recovery to bring failed nodes back quickly.
2 Consensus cluster has high latency for writes due to cross-datacenter replication.
Cause: Nodes are spread across geographically distant data centers. Each write requires a majority acknowledgment, which includes at least one cross-datacenter round trip.
Solution: Place a majority of nodes in the primary data center for lower write latency (sacrificing some fault tolerance). Use techniques like Raft learners (non-voting replicas) in remote data centers for reads. Consider Multi-Paxos or Flexible Paxos for optimized quorum configurations.
