Design a Distributed Task Scheduler

Intuition

A distributed task scheduler is essentially a time-indexed work queue with distributed coordination to ensure every task fires exactly when it should without duplicates. The hardest part is making the dispatcher layer itself fault-tolerant so that no scheduled task is ever lost or double-fired.

Design a Distributed Task Scheduler

Requirements

  • Schedule one-time and recurring tasks (cron-like) across a cluster
  • Guarantee at-least-once execution with distributed locking
  • Support priorities, retries with backoff, and dead-letter handling
  • Scale to millions of scheduled tasks with sub-second trigger accuracy

Architecture

Tasks are persisted in a durable store (e.g., MySQL or Cassandra) partitioned by scheduled time. A dispatcher layer polls for due tasks, acquires distributed locks, and dispatches work to a pool of executors. A heartbeat mechanism detects executor failures and re-queues tasks.

Key Components

  • Task Store: Time-partitioned table holding task metadata, schedule, retries
  • Dispatcher: Leader-elected or sharded process that scans for due tasks
  • Executor Pool: Stateless workers that run task payloads; report success/failure
  • Dead-Letter Queue: Captures tasks that exceed max retries for investigation

Scaling

Partition the timeline into buckets; assign each bucket to a dispatcher shard. Use consistent hashing to rebalance when dispatchers join or leave. Executors auto-scale based on queue depth.

Trade-offs

  • At-least-once is simpler but requires idempotent task handlers
  • Polling vs. push: polling is simpler but adds latency; push via message queue is more responsive
  • Storing tasks in a relational DB gives strong consistency; NoSQL gives better write throughput

Exercises

  1. Build a Time-Bucketed Dispatcher

    Implement a dispatcher that partitions upcoming tasks into 1-second buckets, scans the current bucket, and dispatches tasks to worker threads. Measure trigger accuracy.

    Hint

    Use a sorted data structure (e.g., a priority queue or Redis sorted set with score = timestamp) and poll the current-second bucket every 200ms.

  2. Add Retry with Exponential Backoff

    Extend your scheduler so that failed tasks are retried up to 3 times with exponential backoff (1s, 2s, 4s). After max retries, move to a dead-letter store.

    Hint

    On failure, compute the next retry time as now + base * 2^attempt, update the task's scheduled time, and re-insert it into the task store.

Common Problems & Solutions

1 Tasks are triggered twice when a dispatcher fails over mid-execution
Cause: The lock was released on failover before the executor finished, so the new dispatcher re-dispatched the task
Solution: Use fencing tokens: the executor checks that its token is still valid before committing results; stale tokens are rejected.
2 Hot time buckets cause dispatcher lag during bulk-scheduled events
Cause: Thousands of tasks share the same scheduled second, overwhelming a single dispatcher shard
Solution: Sub-partition hot buckets by task ID hash and spread them across multiple dispatcher shards.
