Service Layer and Domain Model

Intuition

The domain model is like a board game's rulebook -- it defines what moves are legal, how scoring works, and when the game ends. The service layer is the game master who sets up the board, manages turns, and enforces the rules. The game master does not invent new rules; they apply the existing ones.

Service Layer and Domain Model

The Service Layer defines the application's boundary and orchestrates domain operations. The Domain Model encapsulates business rules and logic in plain Python objects, independent of frameworks and databases.

Domain Model

from dataclasses import dataclass, field
from datetime import datetime
from typing import Optional

@dataclass
class Money:
    amount: float
    currency: str = "USD"

    def __add__(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise ValueError("Cannot add different currencies")
        return Money(self.amount + other.amount, self.currency)

@dataclass
class OrderLine:
    product_id: int
    quantity: int
    unit_price: Money

    @property
    def total(self) -> Money:
        return Money(self.unit_price.amount * self.quantity, self.unit_price.currency)

@dataclass
class Order:
    id: Optional[int] = None
    customer_id: int = 0
    lines: list[OrderLine] = field(default_factory=list)
    status: str = "draft"
    created_at: datetime = field(default_factory=datetime.utcnow)

    def add_line(self, product_id: int, quantity: int, unit_price: Money):
        if self.status != "draft":
            raise ValueError("Cannot modify a submitted order")
        self.lines.append(OrderLine(product_id, quantity, unit_price))

    @property
    def total(self) -> Money:
        return Money(sum(line.total.amount for line in self.lines))

    def submit(self):
        if not self.lines:
            raise ValueError("Cannot submit an empty order")
        self.status = "submitted"

Service Layer

class OrderService:
    def __init__(self, uow: AbstractUnitOfWork):
        self.uow = uow

    def create_order(self, customer_id: int, items: list[dict]) -> Order:
        with self.uow:
            customer = self.uow.customers.get(customer_id)
            if not customer:
                raise ValueError("Customer not found")

            order = Order(customer_id=customer_id)
            for item in items:
                product = self.uow.products.get(item["product_id"])
                order.add_line(product.id, item["quantity"], product.price)

            order.submit()
            self.uow.orders.add(order)
        return order

Common Mistakes

  • Putting business rules in the service layer instead of the domain model
  • Making domain objects depend on the database or framework imports
  • Anemic domain model: entities are just data holders with no behavior

Key Takeaways

  • Domain models contain business rules and are framework-agnostic
  • The service layer orchestrates use cases by coordinating domain objects and repositories
  • Keep the service layer thin; push logic into the domain model

Exercises

  1. Build a Rich Domain Model

    Create domain models for a bank account system: Account, Transaction, and Money value objects. Accounts should enforce rules like minimum balance, daily withdrawal limits, and preventing overdrafts. No framework dependencies.

    Hint

    Use dataclasses. Put validation in methods like withdraw() and deposit(). Raise domain exceptions (InsufficientFunds, DailyLimitExceeded) for rule violations.

  2. Service Layer Orchestration

    Write a TransferService that transfers money between two accounts. It should use a UoW to ensure atomicity and call domain model methods for the actual business logic.

    Hint

    The service loads both accounts via repositories, calls source.withdraw(amount) and target.deposit(amount), then commits. All validation is in the domain model.

Common Problems & Solutions

1 Business rules are scattered across the codebase
Symptoms: ['Same validation duplicated in multiple places', 'Inconsistent behavior between API and background jobs']
Cause: Validation and business logic live in route handlers, serializers, and database triggers instead of in domain objects.
Solution: Centralize business rules in domain model methods. Route handlers should only parse input and call the service layer. The service layer coordinates domain objects.
2 Anemic domain model: entities have no behavior
Cause: Domain classes are just data containers (like Pydantic models) with all logic in service functions.
Solution: Move behavior into domain entities. Instead of service.calculate_order_total(order), make it order.total. Instead of service.can_cancel(order), make it order.can_cancel().
