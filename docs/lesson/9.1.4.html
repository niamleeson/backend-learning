CDNs, DNS, and Edge Computing

Intuition

A CDN is like a chain of local libraries that stock copies of popular books so you do not have to travel to the central archive. Edge computing is like placing librarians at each branch who can answer questions directly.

CDNs, DNS & Edge Computing

Content Delivery Networks, DNS infrastructure, and edge computing work together to bring data and computation closer to users, reducing latency and improving reliability for globally distributed applications.

Core Concepts

  • CDN Architecture: CDNs cache content at Points of Presence (PoPs) worldwide. When a user requests a resource, they are routed to the nearest PoP via anycast or GeoDNS.
  • Cache Hierarchy: CDNs use tiered caching—edge nodes, regional shields, and origin servers—to maximize cache hit rates while minimizing origin load.
  • DNS as a Load Balancer: DNS can route traffic geographically (GeoDNS), by latency, or by weighted records. Services like Route 53 and Cloudflare offer health-checked DNS failover.
  • Edge Computing: Running serverless functions at CDN edge locations (Cloudflare Workers, Lambda@Edge) allows request manipulation, authentication, A/B testing, and personalization without hitting the origin.
  • Cache Control: HTTP headers like Cache-Control, ETag, and Vary control what gets cached, for how long, and under what conditions.

Key Takeaways

  1. CDNs are not just for static assets—modern CDNs can cache dynamic content, run edge functions, and act as WAFs.
  2. DNS is a critical piece of infrastructure; misconfigured TTLs or missing health checks can cause outages.
  3. Edge computing shifts logic closer to users but introduces complexity in deployment, debugging, and state management.

Exercises

  1. Set Up Cloudflare CDN with Cache Rules

    Configure a Cloudflare-proxied domain with custom cache rules: cache static assets for 30 days, HTML pages for 5 minutes, and bypass cache for API endpoints. Verify behavior using curl -I to inspect cache headers.

    Hint

    Use Cloudflare Page Rules or Cache Rules. Look for cf-cache-status: HIT in response headers to confirm caching is working.

  2. Build an Edge Function

    Using Cloudflare Workers or AWS Lambda@Edge, write an edge function that performs geo-based routing: redirect users from the EU to an EU-specific origin, and users from other regions to the default origin.

    Hint

    In Cloudflare Workers, access request.cf.country to determine the user's location. Use Response.redirect() or fetch() to route to the appropriate origin.

Common Problems & Solutions

1 Users see stale content after a deployment, even after purging the CDN cache.
Cause: Browser caching with long `max-age` values means users continue serving cached content locally, regardless of CDN purge.
Solution: Use content-hashed filenames (e.g., `app.3f2a1b.js`) for static assets so new deployments get new URLs. Set short `max-age` with `s-maxage` for CDN-level caching.
2 CDN cache hit rate is very low despite high traffic.
Cause: High-cardinality query parameters or `Vary` headers create too many cache keys, fragmenting the cache.
Solution: Normalize query parameters at the CDN level, strip unnecessary parameters, and reduce `Vary` header values to only what is needed (e.g., `Vary: Accept-Encoding`).
