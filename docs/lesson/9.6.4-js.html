Design a Recommendation Engine

Intuition

Imagine a personal shopper in a massive department store with 50 million products. They cannot look at every item for each customer. Instead, they quickly grab items from sections the customer usually shops in (candidate generation), evaluate each item based on the customer's preferences and what similar customers bought (scoring), then curate a small selection that is diverse and appropriate (filtering). The magic is doing all of this in under 100 milliseconds.

Design a Recommendation Engine

Introduction

Recommendation engines power the core experience of platforms like YouTube, Netflix, Spotify, Amazon, and TikTok. They analyze user behavior, item characteristics, and interaction patterns to suggest content that users are most likely to engage with. Designing a recommendation system at scale involves real-time data pipelines, machine learning model serving, feature stores, A/B testing infrastructure, and careful handling of cold-start problems.

Requirements

Functional Requirements

  • Generate personalized recommendations for each user ("Recommended for You")
  • Support multiple recommendation types: home feed, "similar items", "users also liked", "trending"
  • Update recommendations in near real-time as users interact with content
  • Support content-based filtering (recommend items similar to what user liked)
  • Support collaborative filtering (recommend items that similar users liked)
  • Handle the cold-start problem for new users and new items
  • Support A/B testing of different recommendation algorithms
  • Provide explainability ("Because you watched X")

Non-Functional Requirements

  • 500 million daily active users
  • Recommendation API latency: p99 < 100 ms
  • Generate candidate pool of 10,000 items, rank to top 100, serve top 20
  • Update user profile with new interactions within 5 minutes
  • Support 100,000 recommendation requests/second

Capacity Estimation

DAU:                     500 million
Recommendation requests: 500M * 10 sessions * 5 pages = 25 billion/day
Requests/second:         25B / 86,400 = ~290,000 req/s (peak 2-3x)
Peak requests/second:    ~600,000-900,000 req/s

User profiles:           500 million * 5 KB avg = ~2.5 TB
Item catalog:            50 million items * 2 KB = ~100 GB
Interaction events/day:  500M * 200 interactions = 100 billion events/day
Interaction storage/day: 100B * 100 bytes = ~10 TB/day

Model serving:
  Candidate generation:  ~5 ms per request
  Ranking model:         ~20 ms per request (batch of 1000)
  Feature lookup:        ~5 ms per request
  Total pipeline:        ~50 ms budget

API Design

Recommendation API

import express from 'express';
const app = express();

// GET /api/v1/recommendations/feed
// Personalized home feed
app.get('/api/v1/recommendations/feed', async (req, res) => {
  const userId = req.user.id;
  const { limit, cursor, context } = req.query;
  // context: 'homepage', 'after_purchase', 'search_empty'

  // 1. Determine which experiment variant this user is in
  const experiment = await abTestService.getVariant(userId, 'rec-algo-v3');

  // 2. Get user features from feature store
  const userFeatures = await featureStore.getUserFeatures(userId);

  // 3. Generate candidates from multiple sources
  const [cfCandidates, cbCandidates, trending, popular] = await Promise.all([
    collaborativeFilter.getCandidates(userId, userFeatures, 5000),
    contentBasedFilter.getCandidates(userId, userFeatures, 3000),
    trendingService.getCandidates(userFeatures.region, 1000),
    popularService.getCandidates(userFeatures.region, 1000),
  ]);

  // 4. Merge and deduplicate candidates
  const allCandidates = deduplicateCandidates([
    ...cfCandidates,
    ...cbCandidates,
    ...trending,
    ...popular,
  ]);

  // 5. Fetch item features for all candidates
  const itemFeatures = await featureStore.getItemFeaturesBatch(
    allCandidates.map(c => c.itemId)
  );

  // 6. Score with ranking model
  const scored = await rankingModel.predict(
    experiment.modelVersion,
    allCandidates.map(c => ({
      userId,
      itemId: c.itemId,
      userFeatures,
      itemFeatures: itemFeatures[c.itemId],
      candidateSource: c.source,
      context: context || 'homepage',
    }))
  );

  // 7. Apply business rules (filter seen items, enforce diversity)
  const filtered = applyBusinessRules(scored, {
    excludeSeen: true,
    maxPerCategory: 3,
    minDiversity: 0.4,
    userId,
  });

  // 8. Paginate
  const pageSize = parseInt(limit) || 20;
  const results = filtered.slice(0, pageSize);

  // 9. Log impression for training data
  await eventLogger.logImpression({
    userId,
    items: results.map(r => r.itemId),
    experiment: experiment.variant,
    modelVersion: experiment.modelVersion,
    timestamp: Date.now(),
  });

  res.json({
    recommendations: results.map(r => ({
      itemId: r.itemId,
      score: r.score,
      reason: r.explanation, // "Because you liked X"
      source: r.candidateSource,
      metadata: r.itemMetadata,
    })),
    experiment: {
      id: experiment.id,
      variant: experiment.variant,
    },
    nextCursor: results[results.length - 1]?.itemId,
  });
});

// GET /api/v1/recommendations/similar/:itemId
// Similar items ("More like this")
app.get('/api/v1/recommendations/similar/:itemId', async (req, res) => {
  const { itemId } = req.params;
  const { limit } = req.query;

  // Use pre-computed item-to-item similarity from embedding space
  const similar = await similarityIndex.getNearest(itemId, {
    limit: parseInt(limit) || 20,
    minScore: 0.5,
  });

  res.json({
    sourceItem: itemId,
    recommendations: similar.map(s => ({
      itemId: s.itemId,
      similarityScore: s.score,
      metadata: s.itemMetadata,
    })),
  });
});

User Interaction Tracking

// POST /api/v1/interactions
// Track user interactions for recommendation feedback
app.post('/api/v1/interactions', async (req, res) => {
  const { itemId, type, duration, context } = req.body;
  // type: 'view', 'click', 'like', 'dislike', 'purchase', 'share', 'skip'

  const interaction = {
    userId: req.user.id,
    itemId,
    type,
    duration,
    context,
    timestamp: Date.now(),
    sessionId: req.headers['x-session-id'],
  };

  // Fire and forget to Kafka for async processing
  await eventProducer.send('user-interactions', {
    key: req.user.id,
    value: interaction,
  });

  // Update real-time user profile (for near real-time recs)
  await userProfileUpdater.addInteraction(interaction);

  res.status(202).json({ status: 'accepted' });
});

Database Schema

-- User profiles (features for recommendation)
CREATE TABLE user_profiles (
  user_id          UUID PRIMARY KEY,
  embedding        VECTOR(128),           -- User embedding from CF model
  preferred_categories TEXT[] DEFAULT '{}',
  avg_interaction_duration FLOAT DEFAULT 0,
  interaction_count INT DEFAULT 0,
  last_active_at   TIMESTAMPTZ,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Item catalog
CREATE TABLE items (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title            VARCHAR(512) NOT NULL,
  description      TEXT,
  category         VARCHAR(100) NOT NULL,
  tags             TEXT[] DEFAULT '{}',
  embedding        VECTOR(128),           -- Item embedding
  popularity_score FLOAT DEFAULT 0,
  quality_score    FLOAT DEFAULT 0,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_items_category ON items(category);
CREATE INDEX idx_items_embedding ON items USING ivfflat (embedding vector_cosine_ops);

-- User interactions (append-only event log)
-- In production, use Cassandra or a time-series database
CREATE TABLE user_interactions (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          UUID NOT NULL,
  item_id          UUID NOT NULL,
  interaction_type VARCHAR(20) NOT NULL,
  duration_seconds FLOAT,
  context          JSONB,
  session_id       VARCHAR(255),
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_interactions_user ON user_interactions(user_id, created_at DESC);
CREATE INDEX idx_interactions_item ON user_interactions(item_id, created_at DESC);

-- Pre-computed recommendations (materialized)
CREATE TABLE precomputed_recommendations (
  user_id          UUID NOT NULL,
  rec_type         VARCHAR(50) NOT NULL, -- 'feed', 'trending', etc.
  item_ids         UUID[] NOT NULL,
  scores           FLOAT[] NOT NULL,
  model_version    VARCHAR(50) NOT NULL,
  computed_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  expires_at       TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (user_id, rec_type)
);

-- A/B test experiments
CREATE TABLE experiments (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name             VARCHAR(255) UNIQUE NOT NULL,
  variants         JSONB NOT NULL,  -- [{name, modelVersion, trafficPct}]
  status           VARCHAR(20) NOT NULL DEFAULT 'draft',
  start_date       TIMESTAMPTZ,
  end_date         TIMESTAMPTZ,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Experiment assignments
CREATE TABLE experiment_assignments (
  experiment_id    UUID NOT NULL REFERENCES experiments(id),
  user_id          UUID NOT NULL,
  variant          VARCHAR(50) NOT NULL,
  assigned_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (experiment_id, user_id)
);

Scaling Strategies

  • Pre-compute: For most users, generate recommendations offline every few hours. Serve from cache. Only compute in real-time for active users.
  • Approximate Nearest Neighbors (ANN): Use libraries like FAISS or Annoy for embedding similarity search in sub-millisecond time.
  • Feature Store (Redis): Pre-compute and cache user and item features. Eliminates expensive feature computation at request time.
  • Model serving: Use TensorFlow Serving or ONNX Runtime with GPU batching for efficient model inference.
  • Caching: Cache popular item recommendations (trending, popular in your region) at the CDN/API Gateway level.
  • Sharding by user: Each recommendation server handles a subset of users, keeping relevant data in memory.

Trade-offs

Decision Pros Cons
Pre-computed recs Very fast serving (cache hit) Stale recommendations, storage cost
Real-time ranking Fresh, context-aware Higher latency, compute cost
Collaborative filtering Captures taste patterns Cold-start problem for new users
Content-based filtering No cold-start for items Filter bubble, limited serendipity
Hybrid approach Best of both worlds Complex to implement and tune
Embedding similarity (ANN) Fast, scalable Approximate — may miss some relevant items

Cold-Start Strategies

New User (no interaction history):
  1. Serve popular/trending items
  2. Ask preferences during onboarding
  3. Use demographic features (age, location)
  4. Quickly incorporate first few interactions

New Item (no interaction data):
  1. Use content features (title, description, category)
  2. Boost new items in "explore" slots
  3. Show to a random sample of users to gather initial signals
  4. Use the item's creator/brand's historical performance

Summary

A recommendation engine at scale is a multi-stage pipeline: candidate generation narrows millions of items to thousands, a ranking model scores each candidate, business rules filter and diversify, and the top results are served. The offline pipeline trains models and pre-computes features, while the online pipeline assembles features and scores candidates in real-time. The key challenges are handling cold-start, maintaining freshness, ensuring diversity, and serving recommendations within a strict latency budget.

Exercises

  1. Build a Collaborative Filter

    Implement a simple user-based collaborative filtering algorithm in Node.js. Given a matrix of user-item ratings, find the K most similar users to a target user (using cosine similarity), then recommend items those similar users liked that the target user has not seen.

    Hint

    Represent ratings as a sparse matrix (Map of Maps). Compute cosine similarity between user vectors. For each candidate item, compute a weighted average of similar users' ratings.

  2. Content-Based Recommender with TF-IDF

    Build a content-based recommender that uses TF-IDF vectors of item descriptions to find similar items. Given an item, return the 10 most similar items based on cosine similarity of their TF-IDF vectors.

    Hint

    Tokenize descriptions, compute term frequency and inverse document frequency, represent each item as a vector. Use dot product / (magnitude * magnitude) for cosine similarity.

  3. A/B Test Framework

    Build a simple A/B testing framework that deterministically assigns users to experiment variants based on a hash of their user ID. Support multiple concurrent experiments with configurable traffic splits. Track impression and click metrics per variant.

    Hint

    Hash the userId + experimentId with crypto.createHash('md5'). Map the hash to a value 0-99 and use traffic percentage ranges to determine the variant. Store metrics in a Map keyed by experimentId + variant.

  4. Real-Time User Profile Updater

    Build a service that consumes user interaction events from a queue (simulate with an EventEmitter) and updates a user profile in Redis. The profile should include: last 50 interactions, preferred categories (top 5), interaction count, and a running average session duration.

    Hint

    Use Redis sorted sets for recent interactions (ZADD with timestamp as score, ZREMRANGEBYRANK to keep last 50). Use a Redis hash for aggregated features. Compute preferred categories from the last 200 interactions.

Common Problems & Solutions

1 Recommendations become stale and repetitive for active users
Symptoms: Users complain about seeing the same recommendations. Engagement metrics (CTR) decline over the course of a session.
Cause: Pre-computed recommendations are regenerated on a batch schedule (every 6 hours), but active users exhaust the recommendation set within minutes and keep seeing the same items.
Solution: Implement a hybrid approach: serve pre-computed recommendations as the base, but layer in a real-time re-ranking step that incorporates the user's latest interactions. Maintain a 'seen' set per user session and filter out already-seen items. Add a diversity penalty to reduce repetition.
2 New items never get recommended because they have no interaction data
Symptoms: New items have zero impressions days after being added. Item catalog coverage (% of items ever recommended) is below 10%.
Cause: The collaborative filtering model requires interaction data to generate item embeddings. New items have no interactions and are never recommended, creating a chicken-and-egg problem.
Solution: Reserve 5-10% of recommendation slots as 'explore' positions filled with new/under-explored items. Use content-based features to generate initial embeddings for new items. Implement a multi-armed bandit (epsilon-greedy or Thompson sampling) to balance exploitation of known-good items with exploration of new ones.
3 Ranking model inference takes too long (>200ms) causing recommendation API timeouts
Cause: The ranking model scores each candidate individually. With 10,000 candidates, the total inference time exceeds the latency budget. The model is also too large to fit in GPU memory for batch inference.
Solution: Use a two-stage ranking approach: a lightweight model (logistic regression or small neural net) quickly scores all 10,000 candidates and selects the top 500, then a heavy model (deep neural network) precisely scores only those 500. Batch inference requests. Use ONNX Runtime for optimized CPU inference.
