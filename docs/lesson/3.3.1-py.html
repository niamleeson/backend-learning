Normalization: 1NF through 3NF and Beyond

Intuition

Normalization means every fact is stored in exactly one place. If you change a department name, you update one row instead of thousands.

Normalization: 1NF through 3NF

Normalization organizes tables to reduce data redundancy and improve integrity. Each normal form builds on the previous one.

-- UNNORMALIZED: repeating groups
-- orders(id, customer, item1, item2, item3) -- BAD

-- 1NF: atomic values, no repeating groups
CREATE TABLE order_items (
    order_id INTEGER REFERENCES orders(id),
    product_id INTEGER REFERENCES products(id),
    quantity INTEGER,
    PRIMARY KEY (order_id, product_id)
);

-- 2NF: remove partial dependencies
-- product_id -> product_name is partial dep, move to products

-- 3NF: remove transitive dependencies
-- employees(id, dept_id, dept_name) -- dept_name depends on dept_id
CREATE TABLE departments (id SERIAL PRIMARY KEY, name TEXT);
CREATE TABLE employees (
    id SERIAL PRIMARY KEY,
    dept_id INTEGER REFERENCES departments(id)
);

Key Takeaways: - 1NF: each cell holds a single atomic value; rows are uniquely identifiable. - 2NF: every non-key column depends on the entire primary key (eliminates partial dependencies). - 3NF: non-key columns depend only on the key, not on other non-key columns.

Exercises

  1. Identify Normal Form Violations

    Given a table with columns (student_id, course_id, student_name, course_name, grade), identify which normal form it violates and fix it.

    Hint

    student_name depends only on student_id (partial dep). Split into students, courses, and enrollments tables.

  2. Normalize an Order System

    Take a flat orders CSV with repeated customer info and product info. Design a normalized 3NF schema with proper foreign keys.

    Hint

    Create customers, products, orders, and order_items tables. Each entity gets its own table.

Common Problems & Solutions

1 Over-normalized schema requires too many JOINs and queries are slow.
Cause: Breaking every attribute into separate tables creates excessive join overhead.
Solution: Denormalize strategically for read-heavy workloads. Use materialized views to cache joined results.
2 Data anomalies: updating a customer name requires changing many rows.
Cause: Customer data is duplicated across the orders table (not normalized).
Solution: Extract customer data into a separate table and reference it with a foreign key.
