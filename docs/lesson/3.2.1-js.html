Middleware Anatomy: req, res, next

Intuition

Middleware is like an assembly line in a factory. Each station (middleware function) inspects or modifies the product (request) as it moves down the belt. A station can stamp it (add data), reject it (send error response), or wave it through (call next). The product only reaches the final station (route handler) if every station passes it along.

Middleware Anatomy: req, res, next

Introduction

Middleware is the backbone of Express.js. Every piece of logic that sits between an incoming request and the final response is middleware — parsing JSON bodies, checking authentication, logging requests, handling errors. Understanding how middleware works is the key to understanding Express itself.

A middleware function has access to three things: the request object (req), the response object (res), and the next function. Together, they form a pipeline — a chain of functions that each request passes through.

What Is Middleware?

A middleware function is any function with the signature:

(req: Request, res: Response, next: NextFunction) => void

It can: 1. Execute any code 2. Modify the req and res objects 3. End the request-response cycle (send a response) 4. Call next() to pass control to the next middleware

import express, { Request, Response, NextFunction } from 'express';

const app = express();

// A simple middleware that logs every request
const logger = (req: Request, res: Response, next: NextFunction) => {
  console.log(`${req.method} ${req.path}`);
  next(); // Pass control to the next middleware
};

app.use(logger);

app.get('/', (req, res) => {
  res.json({ message: 'Hello' });
});

The Middleware Pipeline

Express executes middleware in the order it is registered. Think of it as a conveyor belt:

Request arrives
    |
    v
[Middleware 1] -- next() -->  [Middleware 2] -- next() -->  [Route Handler]
    ^                              ^                              |
    |                              |                              v
    |                              |                         Response sent
    |                              |
    +---- If no next(), request hangs here

const app = express();

app.use((req, res, next) => {
  console.log('1. First middleware');
  next();
});

app.use((req, res, next) => {
  console.log('2. Second middleware');
  next();
});

app.get('/', (req, res) => {
  console.log('3. Route handler');
  res.json({ message: 'Hello' });
});

// Request to GET / prints:
// 1. First middleware
// 2. Second middleware
// 3. Route handler

The next() Function

next() is the mechanism that passes control to the next matching middleware or route handler. It has three behaviors:

1. next() — Continue to Next Middleware

app.use((req, res, next) => {
  console.log('Processing...');
  next(); // Continue to the next middleware
});

2. next(error) — Skip to Error Handler

app.use((req, res, next) => {
  try {
    // Some operation that might fail
    const data = JSON.parse('invalid json');
    next();
  } catch (err) {
    next(err); // Skip all remaining middleware, go to error handler
  }
});

3. next('route') — Skip to Next Route

app.get('/users/:id',
  (req, res, next) => {
    if (req.params.id === '0') {
      next('route'); // Skip remaining handlers for this route
    } else {
      next(); // Continue to the next handler
    }
  },
  (req, res) => {
    res.json({ user: { id: req.params.id } }); // Skipped if id is 0
  }
);

app.get('/users/:id', (req, res) => {
  res.json({ message: 'Guest user' }); // Handles id=0
});

Types of Middleware

Application-Level Middleware

Bound to the app instance with app.use() or app.METHOD():

// Runs for ALL requests
app.use((req, res, next) => {
  req.requestTime = Date.now();
  next();
});

// Runs only for GET requests to /api/*
app.get('/api/*', (req, res, next) => {
  console.log('API request received');
  next();
});

Router-Level Middleware

Bound to a Router instance:

const router = Router();

// Runs for all requests to this router
router.use((req, res, next) => {
  console.log('Router middleware');
  next();
});

router.get('/users', (req, res) => {
  res.json({ users: [] });
});

Error-Handling Middleware

Has four parameters — the extra first parameter is the error:

app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  console.error(err.stack);
  res.status(500).json({ error: 'Something went wrong' });
});

Express identifies error handlers by the four-parameter signature. If you omit any parameter, it becomes regular middleware.

Middleware Execution Flow

 Request: GET /api/users
    |
    v
 app.use(express.json())        -- parses body, calls next()
    |
    v
 app.use(cors())                -- sets CORS headers, calls next()
    |
    v
 app.use(requestLogger)         -- logs request, calls next()
    |
    v
 app.use('/api', authMiddleware) -- checks auth, calls next() or sends 401
    |
    v
 router.get('/users', handler)  -- sends response
    |
    v
 (if error was passed to next)
    |
    v
 app.use(errorHandler)          -- handles error, sends error response

Modifying req and res

Middleware commonly attaches data to req for downstream handlers:

// Extend the Request type
declare global {
  namespace Express {
    interface Request {
      requestId: string;
      startTime: number;
      user?: { id: number; role: string };
    }
  }
}

// Add request ID
const addRequestId = (req: Request, res: Response, next: NextFunction) => {
  req.requestId = crypto.randomUUID();
  res.set('X-Request-Id', req.requestId);
  next();
};

// Add timing
const addTiming = (req: Request, res: Response, next: NextFunction) => {
  req.startTime = Date.now();

  // Hook into response finish to log timing
  res.on('finish', () => {
    const duration = Date.now() - req.startTime;
    console.log(`${req.method} ${req.path} ${res.statusCode} ${duration}ms`);
  });

  next();
};

app.use(addRequestId);
app.use(addTiming);

Conditional Middleware

Apply middleware based on conditions:

// Only in development
if (process.env.NODE_ENV === 'development') {
  app.use(morgan('dev'));
}

// Factory function for conditional middleware
function unless(middleware: Function, ...paths: string[]) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (paths.includes(req.path)) {
      return next(); // Skip middleware for these paths
    }
    middleware(req, res, next);
  };
}

// Auth on all routes except /login and /register
app.use(unless(authMiddleware, '/login', '/register', '/health'));

Async Middleware

Express 4.x does not catch async errors automatically. You must handle them:

// WRONG: unhandled rejection if fetchUser throws
app.use(async (req, res, next) => {
  req.user = await fetchUser(req.headers.authorization);
  next();
});

// CORRECT: wrap in try-catch
app.use(async (req: Request, res: Response, next: NextFunction) => {
  try {
    req.user = await fetchUser(req.headers.authorization);
    next();
  } catch (error) {
    next(error);
  }
});

// BETTER: create a wrapper utility
function asyncHandler(fn: (req: Request, res: Response, next: NextFunction) => Promise<void>) {
  return (req: Request, res: Response, next: NextFunction) => {
    fn(req, res, next).catch(next);
  };
}

app.use(asyncHandler(async (req, res, next) => {
  req.user = await fetchUser(req.headers.authorization);
  next();
}));

Note: Express 5 (currently in beta) handles async errors natively.

Common Pitfalls

1. Forgetting to Call next()

// WRONG: request hangs forever
app.use((req, res, next) => {
  console.log('Logging...');
  // Forgot next()! Request never reaches route handler.
});

// CORRECT
app.use((req, res, next) => {
  console.log('Logging...');
  next();
});

2. Calling next() After Sending a Response

// WRONG: sends response then continues pipeline
app.use((req, res, next) => {
  res.json({ data: 'from middleware' });
  next(); // Next handler might try to send another response!
});

// CORRECT: either send a response OR call next(), not both
app.use((req, res, next) => {
  if (unauthorized) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  next();
});

3. Wrong Order of Middleware Registration

// WRONG: route registered before body parser
app.post('/api/data', (req, res) => {
  res.json(req.body); // undefined!
});
app.use(express.json());

// CORRECT: body parser before routes
app.use(express.json());
app.post('/api/data', (req, res) => {
  res.json(req.body); // properly parsed
});

Summary

Middleware is the fundamental building block of Express applications. Each middleware function receives req, res, and next. It can transform the request, send a response, or pass control downstream. The order of registration determines execution order. Always call next() unless you are ending the request, always handle async errors, and remember that error-handling middleware needs four parameters.

Exercises

  1. Request Timer Middleware

    Write middleware that records the start time on the request object, then listens for the response 'finish' event to calculate and log the total request duration in milliseconds. Format: 'GET /api/users 200 42ms'.

    Hint

    Use Date.now() to record start time. Listen with res.on('finish', () => { ... }) to calculate duration when the response completes.

  2. Request ID Middleware

    Create middleware that generates a unique request ID (using crypto.randomUUID()), attaches it to req.requestId, and adds it as an X-Request-Id response header. Downstream handlers should be able to access req.requestId.

    Hint

    Import crypto from 'node:crypto'. Set the header with res.set('X-Request-Id', id). Extend the Express Request interface for TypeScript.

  3. Async Handler Wrapper

    Write an asyncHandler utility function that wraps async route handlers and automatically catches rejected promises, forwarding errors to next(). Test it with a route that throws an error and verify the error handler catches it.

    Hint

    The wrapper returns a function (req, res, next) that calls the original function and chains .catch(next) on the returned promise.

  4. Conditional Middleware

    Create a middleware factory function skipFor(middleware, paths) that runs the given middleware for all routes EXCEPT those in the paths array. Use it to skip authentication for /health and /login.

    Hint

    Return a new middleware function that checks if req.path is in the paths array. If yes, call next() directly. Otherwise, call middleware(req, res, next).

Common Problems & Solutions

1 Request hangs and eventually times out
Cause: A middleware function does not call next() and does not send a response, so the request is stuck in the pipeline.
Solution: Ensure every middleware either calls next() to continue or sends a response (res.json(), res.send(), etc.). Add a timeout middleware as a safety net.
2 Error handler middleware is never called
Cause: The error handler does not have exactly four parameters (err, req, res, next), so Express treats it as regular middleware.
Solution: Error handling middleware MUST have four parameters: (err, req, res, next). Even if you do not use next, include it in the signature. TypeScript: (err: Error, req: Request, res: Response, next: NextFunction).
3 Async middleware errors crash the server instead of being handled
Cause: Express 4.x does not catch rejected promises from async middleware. An unhandled rejection occurs.
Solution: Wrap async middleware in try-catch and call next(error) in the catch block. Or use an asyncHandler wrapper: fn(req, res, next).catch(next).
