Circuit Breakers and Retry Patterns

Intuition

Think of a circuit breaker like a smart traffic light at a road that frequently floods. Normally, the light is green (closed state) and cars pass through. When the road floods (the downstream service fails), the light turns red (open state), preventing cars from driving into floodwater and getting stuck. Periodically, the light turns yellow (half-open state) to let one car test if the road has drained — if it gets through, the light goes green again; if not, it stays red. Meanwhile, exponential backoff with jitter is like asking people to return at randomly staggered times rather than having everyone rush back the moment the road reopens.

Circuit Breakers and Retry Patterns

In distributed systems, failure is not a matter of if but when. Services go down, networks partition, databases become overloaded, and third-party APIs become unresponsive. Without proper resilience patterns, a single failing dependency can cascade through your entire system, turning a minor issue into a full-scale outage. Circuit breakers and retry patterns are two of the most critical tools in a backend engineer's resilience toolkit.

The Problem: Cascading Failures

Imagine a microservices architecture where Service A calls Service B, which calls Service C. If Service C becomes slow (responding in 30 seconds instead of 100ms), Service B's threads pile up waiting for responses. Service B's response times degrade, causing Service A's threads to pile up as well. Within minutes, your entire system is effectively down — not because anything crashed, but because resources are exhausted waiting for a single slow dependency.

This is a cascading failure, and it is one of the most dangerous failure modes in distributed systems.

Naive Retries: Making Things Worse

The first instinct when a request fails is to retry it. But naive retries can amplify the problem dramatically:

import requests

def call_service(url):
    """Naive retry — DO NOT use this in production."""
    max_retries = 3
    for attempt in range(max_retries):
        try:
            response = requests.get(url, timeout=5)
            response.raise_for_status()
            return response.json()
        except requests.RequestException:
            if attempt == max_retries - 1:
                raise
            # Immediate retry with no delay!
            continue

If a service is handling 1,000 requests per second and each request retries 3 times on failure, the failing service now receives 3,000 requests per second — exactly when it is least able to handle them. This is called a retry storm, and it can prevent a struggling service from ever recovering.

The Circuit Breaker Pattern

The circuit breaker pattern, popularized by Michael Nygard in Release It! and later by Netflix's Hystrix library, draws a direct analogy from electrical circuit breakers. When a circuit detects dangerous conditions, it "trips" to prevent further damage.

Three States

A circuit breaker operates in three states:

1. Closed (Normal Operation) Requests flow through normally. The circuit breaker monitors for failures. When failures exceed a configured threshold (e.g., 5 failures in 60 seconds, or a 50% failure rate), the circuit breaker transitions to the Open state.

2. Open (Failing Fast) All requests are immediately rejected without calling the downstream service. This gives the failing service time to recover and prevents resource exhaustion in the calling service. After a configured timeout period (e.g., 30 seconds), the circuit breaker transitions to Half-Open.

3. Half-Open (Testing Recovery) A limited number of requests are allowed through to test whether the downstream service has recovered. If these requests succeed, the circuit breaker transitions back to Closed. If they fail, it transitions back to Open.

Custom Circuit Breaker Implementation

Let us build a circuit breaker from scratch to understand the internals:

import time
import threading
from enum import Enum
from typing import Callable, Any, Optional
from functools import wraps


class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"


class CircuitBreakerError(Exception):
    """Raised when the circuit breaker is open."""
    pass


class CircuitBreaker:
    def __init__(
        self,
        failure_threshold: int = 5,
        recovery_timeout: float = 30.0,
        half_open_max_calls: int = 1,
        expected_exceptions: tuple = (Exception,),
    ):
        self._failure_threshold = failure_threshold
        self._recovery_timeout = recovery_timeout
        self._half_open_max_calls = half_open_max_calls
        self._expected_exceptions = expected_exceptions

        self._state = CircuitState.CLOSED
        self._failure_count = 0
        self._last_failure_time: Optional[float] = None
        self._half_open_calls = 0
        self._lock = threading.Lock()

    @property
    def state(self) -> CircuitState:
        with self._lock:
            if self._state == CircuitState.OPEN:
                # Check if recovery timeout has elapsed
                if time.time() - self._last_failure_time >= self._recovery_timeout:
                    self._state = CircuitState.HALF_OPEN
                    self._half_open_calls = 0
            return self._state

    def _handle_success(self):
        with self._lock:
            if self._state == CircuitState.HALF_OPEN:
                self._half_open_calls += 1
                if self._half_open_calls >= self._half_open_max_calls:
                    self._state = CircuitState.CLOSED
                    self._failure_count = 0
            elif self._state == CircuitState.CLOSED:
                self._failure_count = 0

    def _handle_failure(self):
        with self._lock:
            self._failure_count += 1
            self._last_failure_time = time.time()
            if self._state == CircuitState.HALF_OPEN:
                self._state = CircuitState.OPEN
            elif self._failure_count >= self._failure_threshold:
                self._state = CircuitState.OPEN

    def call(self, func: Callable, *args, **kwargs) -> Any:
        current_state = self.state

        if current_state == CircuitState.OPEN:
            raise CircuitBreakerError(
                f"Circuit breaker is OPEN. Call rejected. "
                f"Recovery in {self._recovery_timeout - (time.time() - self._last_failure_time):.1f}s"
            )

        try:
            result = func(*args, **kwargs)
            self._handle_success()
            return result
        except self._expected_exceptions as e:
            self._handle_failure()
            raise

    def __call__(self, func: Callable) -> Callable:
        """Use as a decorator."""
        @wraps(func)
        def wrapper(*args, **kwargs):
            return self.call(func, *args, **kwargs)
        return wrapper


# Usage as a decorator
service_b_breaker = CircuitBreaker(
    failure_threshold=5,
    recovery_timeout=30.0,
    expected_exceptions=(requests.RequestException,),
)

@service_b_breaker
def call_service_b(payload):
    response = requests.post("http://service-b/api/process", json=payload, timeout=5)
    response.raise_for_status()
    return response.json()


# Usage in application code
try:
    result = call_service_b({"order_id": 12345})
except CircuitBreakerError:
    # Return cached data, default value, or graceful degradation
    result = get_cached_result(12345)
except requests.RequestException:
    # The request was attempted but failed
    result = handle_request_failure()

Retry Patterns Done Right

Exponential Backoff with Jitter

When retries are necessary, exponential backoff prevents retry storms by increasing the delay between attempts. Adding jitter (randomness) prevents synchronized retries from multiple clients:

import random
import time
from typing import Callable, TypeVar, Any

T = TypeVar("T")


def retry_with_backoff(
    func: Callable[..., T],
    max_retries: int = 3,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
    exponential_base: float = 2.0,
    jitter: bool = True,
    retryable_exceptions: tuple = (Exception,),
    *args,
    **kwargs,
) -> T:
    """
    Retry a function with exponential backoff and optional jitter.

    Delay formula: min(max_delay, base_delay * exponential_base ** attempt)
    With jitter:  random uniform between 0 and the computed delay
    """
    last_exception = None

    for attempt in range(max_retries + 1):
        try:
            return func(*args, **kwargs)
        except retryable_exceptions as e:
            last_exception = e
            if attempt == max_retries:
                break

            # Calculate delay with exponential backoff
            delay = min(max_delay, base_delay * (exponential_base ** attempt))

            # Add full jitter (recommended by AWS)
            if jitter:
                delay = random.uniform(0, delay)

            print(f"Attempt {attempt + 1} failed: {e}. Retrying in {delay:.2f}s...")
            time.sleep(delay)

    raise last_exception


# Usage
result = retry_with_backoff(
    requests.get,
    max_retries=3,
    base_delay=1.0,
    retryable_exceptions=(requests.ConnectionError, requests.Timeout),
    url="http://service-b/api/data",
    timeout=5,
)

The jitter strategies commonly discussed are: - Full Jitter: random(0, delay) — recommended by most experts, spreads retries most evenly - Equal Jitter: delay/2 + random(0, delay/2) — guarantees a minimum wait - Decorrelated Jitter: min(max_delay, random(base_delay, previous_delay * 3)) — each retry is independent of the base calculation

Retry Budgets

A retry budget limits the total number of retries across all requests as a percentage of total traffic. This prevents retry storms even when many requests fail simultaneously:

import time
import threading
from collections import deque


class RetryBudget:
    """
    Limits retries to a percentage of recent requests.
    For example, a 20% budget means if 100 requests were made
    in the last minute, only 20 retries are allowed.
    """

    def __init__(self, budget_percent: float = 20.0, window_seconds: float = 60.0):
        self._budget_percent = budget_percent
        self._window_seconds = window_seconds
        self._requests: deque = deque()   # timestamps of all requests
        self._retries: deque = deque()    # timestamps of retries
        self._lock = threading.Lock()

    def _clean_window(self):
        cutoff = time.time() - self._window_seconds
        while self._requests and self._requests[0] < cutoff:
            self._requests.popleft()
        while self._retries and self._retries[0] < cutoff:
            self._retries.popleft()

    def record_request(self):
        with self._lock:
            self._requests.append(time.time())

    def can_retry(self) -> bool:
        with self._lock:
            self._clean_window()
            total_requests = len(self._requests)
            total_retries = len(self._retries)
            if total_requests == 0:
                return True
            allowed_retries = total_requests * (self._budget_percent / 100.0)
            return total_retries < allowed_retries

    def record_retry(self):
        with self._lock:
            self._retries.append(time.time())


# Usage
budget = RetryBudget(budget_percent=20.0, window_seconds=60.0)

def make_request_with_budget(url: str) -> dict:
    budget.record_request()
    try:
        return requests.get(url, timeout=5).json()
    except requests.RequestException:
        if budget.can_retry():
            budget.record_retry()
            time.sleep(1)
            return requests.get(url, timeout=5).json()
        else:
            raise RuntimeError("Retry budget exhausted")

The Bulkhead Pattern

The bulkhead pattern (named after ship compartments that prevent flooding from spreading) isolates different dependencies so that a failure in one does not consume all resources:

import concurrent.futures
from typing import Callable, Any


class Bulkhead:
    """
    Limits concurrent calls to a dependency using a thread pool.
    If the pool is exhausted, new calls are rejected immediately
    rather than queuing and consuming more resources.
    """

    def __init__(self, name: str, max_concurrent: int = 10, max_queue: int = 5):
        self.name = name
        self._executor = concurrent.futures.ThreadPoolExecutor(
            max_workers=max_concurrent,
            thread_name_prefix=f"bulkhead-{name}",
        )
        self._semaphore = threading.Semaphore(max_concurrent + max_queue)

    def execute(self, func: Callable, *args, timeout: float = 30.0, **kwargs) -> Any:
        if not self._semaphore.acquire(blocking=False):
            raise RuntimeError(
                f"Bulkhead '{self.name}' is full. Rejecting request."
            )
        try:
            future = self._executor.submit(func, *args, **kwargs)
            return future.result(timeout=timeout)
        except concurrent.futures.TimeoutError:
            raise TimeoutError(
                f"Bulkhead '{self.name}' call timed out after {timeout}s"
            )
        finally:
            self._semaphore.release()


# Separate bulkheads for different dependencies
db_bulkhead = Bulkhead("database", max_concurrent=20, max_queue=10)
api_bulkhead = Bulkhead("external-api", max_concurrent=5, max_queue=2)

# If the external API is slow, it only exhausts its own bulkhead
# The database calls continue working normally
db_result = db_bulkhead.execute(query_database, "SELECT * FROM users")
api_result = api_bulkhead.execute(call_external_api, payload)

Timeout Strategies

Timeouts are the foundation of all resilience patterns. Without timeouts, circuit breakers and retries are meaningless because a single hung connection can block resources indefinitely.

import signal


class TimeoutError(Exception):
    pass


def with_timeout(seconds: float):
    """Decorator to enforce a hard timeout on any function (Unix only)."""
    def decorator(func):
        def handler(signum, frame):
            raise TimeoutError(f"Function {func.__name__} timed out after {seconds}s")

        @wraps(func)
        def wrapper(*args, **kwargs):
            old_handler = signal.signal(signal.SIGALRM, handler)
            signal.setitimer(signal.ITIMER_REAL, seconds)
            try:
                return func(*args, **kwargs)
            finally:
                signal.setitimer(signal.ITIMER_REAL, 0)
                signal.signal(signal.SIGALRM, old_handler)
        return wrapper
    return decorator


# Layered timeouts — each layer has its own, progressively shorter
# Overall request: 10s
# Circuit breaker: 8s
# Individual HTTP call: 5s
# Database query: 3s

A key principle: set timeouts at every layer, and make inner timeouts shorter than outer ones. If your HTTP client has a 5-second timeout, your API gateway should have a slightly higher timeout (e.g., 7 seconds) to account for processing overhead.

Combining Circuit Breakers with Retries

The most robust approach combines both patterns. Retries handle transient errors while the circuit breaker handles sustained failures:

from tenacity import retry, stop_after_attempt, wait_exponential_jitter, retry_if_exception_type
import pybreaker

# Circuit breaker wraps the raw call
service_breaker = pybreaker.CircuitBreaker(
    fail_max=5,
    reset_timeout=30,
    name="payment-service",
)


@service_breaker
def _call_payment_service(order_id: int, amount: float) -> dict:
    """Raw call protected by circuit breaker."""
    response = requests.post(
        "http://payment-service/api/charge",
        json={"order_id": order_id, "amount": amount},
        timeout=5,
    )
    response.raise_for_status()
    return response.json()


# Retry wraps the circuit breaker call
# IMPORTANT: Do NOT retry on CircuitBreakerError — the circuit is open for a reason
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential_jitter(initial=1, max=10),
    retry=retry_if_exception_type((ConnectionError, TimeoutError)),
)
def charge_payment(order_id: int, amount: float) -> dict:
    """Retryable call with circuit breaker protection."""
    try:
        return _call_payment_service(order_id, amount)
    except pybreaker.CircuitBreakerError:
        # Don't retry — fall back immediately
        return {"status": "pending", "message": "Payment service unavailable"}


# Application code
def process_order(order_id: int, amount: float):
    result = charge_payment(order_id, amount)
    if result.get("status") == "pending":
        # Queue for later processing
        enqueue_payment_retry(order_id, amount)
    return result

Notice the critical design decision: retries wrap the circuit breaker, not the other way around, and we never retry when the circuit is open. Retrying against an open circuit is wasteful since we already know the dependency is down.

Real-World Lessons from Netflix Hystrix

Netflix's Hystrix library (now in maintenance mode, succeeded by Resilience4j) established several principles that remain best practices:

  1. Fail fast: When the circuit is open, return immediately. Do not queue requests.
  2. Fallbacks everywhere: Every protected call should have a fallback — cached data, default values, or a simplified response.
  3. Monitor everything: Track success rate, latency percentiles, circuit breaker state changes, and thread pool utilization.
  4. Isolate by dependency: Each external dependency gets its own circuit breaker and thread pool (bulkhead).
  5. Request collapsing: Batch multiple requests to the same dependency to reduce load.
# Production-grade resilient service client
class ResilientServiceClient:
    def __init__(self, service_name: str, base_url: str):
        self.service_name = service_name
        self.base_url = base_url

        self.circuit_breaker = pybreaker.CircuitBreaker(
            fail_max=5,
            reset_timeout=30,
            name=service_name,
        )
        self.bulkhead = Bulkhead(service_name, max_concurrent=10)
        self.retry_budget = RetryBudget(budget_percent=20.0)
        self._cache = {}  # Simple cache for fallbacks

    def get(self, path: str, cache_key: str = None, **kwargs) -> dict:
        self.retry_budget.record_request()
        url = f"{self.base_url}{path}"

        try:
            return self.bulkhead.execute(
                self._protected_get, url, timeout=10, **kwargs
            )
        except pybreaker.CircuitBreakerError:
            return self._fallback(cache_key, "circuit_open")
        except Exception as e:
            if self.retry_budget.can_retry():
                self.retry_budget.record_retry()
                try:
                    return self.bulkhead.execute(
                        self._protected_get, url, timeout=10, **kwargs
                    )
                except Exception:
                    return self._fallback(cache_key, "retry_failed")
            return self._fallback(cache_key, str(e))

    @property
    def _protected_get(self):
        @self.circuit_breaker
        def _get(url, **kwargs):
            resp = requests.get(url, timeout=5, **kwargs)
            resp.raise_for_status()
            return resp.json()
        return _get

    def _fallback(self, cache_key: str, reason: str) -> dict:
        if cache_key and cache_key in self._cache:
            return {**self._cache[cache_key], "_fallback": True, "_reason": reason}
        return {"error": "Service unavailable", "_fallback": True, "_reason": reason}

Testing Resilience Patterns

Resilience patterns must be tested — otherwise you are only hoping they work when disaster strikes:

import pytest
from unittest.mock import patch, MagicMock


def test_circuit_breaker_opens_after_threshold():
    breaker = CircuitBreaker(failure_threshold=3, recovery_timeout=10)

    def failing_func():
        raise ConnectionError("Connection refused")

    # First 3 calls fail and trip the breaker
    for _ in range(3):
        with pytest.raises(ConnectionError):
            breaker.call(failing_func)

    # 4th call should be rejected by the circuit breaker
    assert breaker.state == CircuitState.OPEN
    with pytest.raises(CircuitBreakerError):
        breaker.call(failing_func)


def test_circuit_breaker_recovers():
    breaker = CircuitBreaker(failure_threshold=2, recovery_timeout=1)

    def failing_func():
        raise ConnectionError("Connection refused")

    # Trip the breaker
    for _ in range(2):
        with pytest.raises(ConnectionError):
            breaker.call(failing_func)

    assert breaker.state == CircuitState.OPEN

    # Wait for recovery timeout
    time.sleep(1.1)

    assert breaker.state == CircuitState.HALF_OPEN

    # Successful call should close the circuit
    breaker.call(lambda: "success")
    assert breaker.state == CircuitState.CLOSED


def test_retry_budget_limits_retries():
    budget = RetryBudget(budget_percent=10.0, window_seconds=60.0)

    # Record 10 requests
    for _ in range(10):
        budget.record_request()

    # Budget allows 10% = 1 retry
    assert budget.can_retry() is True
    budget.record_retry()

    # Second retry should be denied
    assert budget.can_retry() is False

import random

def chaos_middleware(func):

Summary of Best Practices

  1. Always set timeouts — on every network call, at every layer.
  2. Use exponential backoff with full jitter for retries.
  3. Implement retry budgets to prevent retry storms across your fleet.
  4. Deploy circuit breakers for every external dependency.
  5. Never retry on open circuits — respect the circuit breaker's decision.
  6. Use bulkheads to isolate failures between dependencies.
  7. Provide fallbacks — cached data, defaults, or degraded functionality.
  8. Monitor and alert on circuit breaker state changes.
  9. Test your resilience patterns with chaos engineering and fault injection.
  10. Combine patterns thoughtfully: retries for transient errors, circuit breakers for sustained failures, bulkheads for isolation, and timeouts as the foundation for all of them.

Common Problems & Solutions

1 Circuit breaker never closes again after opening
Symptoms: The circuit breaker opens correctly on failures but all subsequent requests are permanently rejected with CircuitBreakerError, even after the downstream service recovers. Logs show the breaker entering HALF_OPEN but never returning to CLOSED.
Cause: The half-open state is not properly detecting successful calls. Common causes include: (1) the success handler checks for CLOSED state instead of HALF_OPEN, (2) the recovery timeout check has an off-by-one error, or (3) a race condition between state checking and request execution in multithreaded code.
Solution: Ensure your state transition logic explicitly handles the HALF_OPEN -> CLOSED transition on success. Use proper locking around state changes. Verify the recovery timeout comparison uses >= rather than > to avoid edge cases. Add logging on every state transition to trace the issue.

Bug: checking for wrong state in success handler

def _handle_success(self): if self._state == CircuitState.CLOSED: # Wrong! Should check HALF_OPEN self._failure_count = 0

Fix:

def _handle_success(self): with self._lock: if self._state == CircuitState.HALF_OPEN: self._state = CircuitState.CLOSED self._failure_count = 0 elif self._state == CircuitState.CLOSED: self._failure_count = 0

2 Retry storms overwhelming a recovering service
Symptoms: After a downstream service restarts, it immediately becomes overloaded again. Traffic graphs show periodic spikes corresponding to retry waves. The service enters a cycle of recovery and failure.
Cause: Retries are configured per-request without a global budget. When a service recovers, all backed-up clients retry simultaneously. Additionally, if all clients use identical backoff delays (no jitter), retries synchronize and hit the service in waves.
Solution: Implement a retry budget that limits retries to a percentage of total traffic (typically 10-20%). Always use full jitter with exponential backoff to decorrelate retry timing across clients. Consider adding a circuit breaker that prevents retries entirely when the failure rate is high enough.
3 Bulkhead thread pool silently dropping requests under load
Symptoms: Requests to a specific dependency time out even though the dependency is healthy. Thread pool utilization shows 100%. Memory usage grows steadily because futures are queuing without bound. Other dependencies served by the same application remain responsive (confirming bulkhead isolation is working for them).
Cause: The ThreadPoolExecutor's default behavior when the work queue is full depends on the configuration. If max_workers is too low or the queue is unbounded, you either reject too aggressively or accumulate unbounded memory usage. Additionally, if the timeout on future.result() is not set, threads can hang indefinitely.
Solution: Always set explicit timeouts on future.result(). Use a semaphore to implement admission control before submitting to the executor. Size your bulkhead based on the expected latency of the dependency: max_concurrent = target_throughput * p99_latency. Monitor the rejection rate and tune accordingly.

Problem: no admission control, no timeout

executor = ThreadPoolExecutor(max_workers=10) future = executor.submit(call_service) # Queues unboundedly result = future.result() # Blocks forever if the call hangs

Fix: semaphore for admission control + timeout

semaphore = threading.Semaphore(15) # 10 workers + 5 queue if not semaphore.acquire(blocking=False): raise RuntimeError("Bulkhead full") try: future = executor.submit(call_service) result = future.result(timeout=10) finally: semaphore.release()
