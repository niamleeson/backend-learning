Multiprocessing: Processes, Pools, and Shared State

Intuition

Multiprocessing is like opening separate kitchens (processes) instead of sharing one. Each cook works independently with their own tools, and you pass ingredients through a window (serialization).

Multiprocessing: Processes, Pools, and Shared State

Multiprocessing bypasses the GIL by running code in separate OS processes, enabling true parallelism for CPU-bound work.

from multiprocessing import Process, Pool, Value, Lock
import os

def cpu_intensive(n: int) -> int:
    """Simulate CPU-bound work."""
    return sum(i * i for i in range(n))

# Using Pool for parallel map
def main_pool():
    with Pool(processes=4) as pool:
        results = pool.map(cpu_intensive, [10**6] * 4)
        print(f"Results: {[r % 1000 for r in results]}")

# Shared state between processes
def increment_shared(counter: Value, lock: Lock, n: int):
    for _ in range(n):
        with lock:
            counter.value += 1

def main_shared():
    counter = Value('i', 0)  # shared integer
    lock = Lock()
    procs = [Process(target=increment_shared, args=(counter, lock, 10000))
             for _ in range(4)]
    for p in procs:
        p.start()
    for p in procs:
        p.join()
    print(f"Counter: {counter.value}")  # 40000

if __name__ == '__main__':
    main_pool()

Key Takeaways: - Each process has its own memory space and GIL (true parallelism) - Pool.map distributes work across processes easily - Use Value and Array for simple shared state between processes - Data must be picklable to pass between processes - Always guard process creation with if __name__ == '__main__':

    Hint

    Hint

    Manager().dict() creates a proxy dict that is safe for multi-process access. Pass it as an argument to each process.

Common Problems & Solutions

1 `RuntimeError: An attempt has been made to start a new process before the current process has finished bootstrapping`.
Cause: Process creation code is not guarded by `if __name__ == '__main__':` on Windows/macOS.
Solution: Always wrap multiprocessing code in `if __name__ == '__main__':` to prevent recursive process spawning.
2 `PicklingError: Can't pickle local object` when passing arguments to a process.
Cause: Lambda functions, inner functions, or certain objects cannot be serialized (pickled) for inter-process communication.
Solution: Use top-level named functions instead of lambdas. For complex objects, implement `__reduce__` or use `multiprocessing.Manager`.
