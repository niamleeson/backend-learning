Controllers: Routes, Parameters, and DTOs

Intuition

Controllers are like restaurant waiters. They take your order (request), pass it to the kitchen (service), and bring back the food (response). They don't cook -- they route. DTOs are the order forms: structured, validated, ensuring the kitchen gets exactly what it needs.

Controllers: Routes, Parameters, and DTOs

Introduction

Controllers in NestJS are responsible for handling incoming HTTP requests and returning responses. They are the entry point for your API endpoints, but they should not contain business logic -- that belongs in services. A controller's job is to receive a request, delegate to a service, and return the result.

Creating a Controller

nest g controller users

This generates users.controller.ts and a spec file, and registers the controller in UsersModule.

Basic Controller

import { Controller, Get, Post, Put, Delete, Param, Body, Query } from '@nestjs/common';
import { UsersService } from './users.service';

@Controller('users')
export class UsersController {
  constructor(private readonly usersService: UsersService) {}

  @Get()
  findAll() {
    return this.usersService.findAll();
  }

  @Get(':id')
  findOne(@Param('id') id: string) {
    return this.usersService.findOne(id);
  }

  @Post()
  create(@Body() createUserDto: CreateUserDto) {
    return this.usersService.create(createUserDto);
  }

  @Put(':id')
  update(@Param('id') id: string, @Body() updateUserDto: UpdateUserDto) {
    return this.usersService.update(id, updateUserDto);
  }

  @Delete(':id')
  remove(@Param('id') id: string) {
    return this.usersService.remove(id);
  }
}

Route Decorators

NestJS provides decorators for all HTTP methods:

Decorator HTTP Method
@Get() GET
@Post() POST
@Put() PUT
@Patch() PATCH
@Delete() DELETE
@Options() OPTIONS
@Head() HEAD
@All() All methods

Route Paths

@Controller('users')
export class UsersController {
  @Get()              // GET /users
  findAll() {}

  @Get('active')      // GET /users/active
  findActive() {}

  @Get(':id')         // GET /users/123
  findOne() {}

  @Get(':id/posts')   // GET /users/123/posts
  findUserPosts() {}
}

Wildcard Routes

@Get('ab*cd')  // Matches abcd, ab_cd, abecd, etc.
findWildcard() {}

Parameter Decorators

NestJS provides decorators to extract data from the request:

import {
  Controller, Get, Post, Param, Query, Body, Headers,
  Ip, Req, Res, HttpCode, Header, Redirect,
} from '@nestjs/common';
import { Request, Response } from 'express';

@Controller('users')
export class UsersController {

  // Route parameters
  @Get(':id')
  findOne(@Param('id') id: string) {
    return `User ${id}`;
  }

  // Multiple route parameters
  @Get(':userId/posts/:postId')
  findUserPost(
    @Param('userId') userId: string,
    @Param('postId') postId: string,
  ) {
    return `User ${userId}, Post ${postId}`;
  }

  // All params as an object
  @Get(':id')
  findOneAll(@Param() params: { id: string }) {
    return `User ${params.id}`;
  }

  // Query parameters: GET /users?page=1&limit=10
  @Get()
  findAll(
    @Query('page') page: string,
    @Query('limit') limit: string,
  ) {
    return `Page ${page}, Limit ${limit}`;
  }

  // Request body
  @Post()
  create(@Body() body: CreateUserDto) {
    return body;
  }

  // Specific body field
  @Post()
  createName(@Body('name') name: string) {
    return name;
  }

  // Headers
  @Get()
  getWithAuth(@Headers('authorization') auth: string) {
    return `Auth: ${auth}`;
  }

  // IP address
  @Get('ip')
  getIp(@Ip() ip: string) {
    return `Your IP: ${ip}`;
  }

  // Raw request/response (escape hatch)
  @Get('raw')
  getRaw(@Req() req: Request, @Res() res: Response) {
    res.status(200).json({ message: 'Raw response' });
  }
}

Important: When you inject @Res(), NestJS no longer handles the response automatically. You must call res.json() or res.send() yourself. To access the response object while keeping automatic handling, use @Res({ passthrough: true }).

Data Transfer Objects (DTOs)

DTOs define the shape of data flowing into your API. They are plain TypeScript classes (not interfaces) because they exist at runtime and can be used with validation pipes.

Creating DTOs

// dto/create-user.dto.ts
export class CreateUserDto {
  name: string;
  email: string;
  password: string;
  age?: number;
}

// dto/update-user.dto.ts
import { PartialType } from '@nestjs/mapped-types';
import { CreateUserDto } from './create-user.dto';

export class UpdateUserDto extends PartialType(CreateUserDto) {}

PartialType() makes all fields optional -- perfect for update operations where you only send changed fields.

DTOs with Validation

Install validation packages:

npm i class-validator class-transformer

import { IsString, IsEmail, IsOptional, IsInt, Min, Max, MinLength } from 'class-validator';

export class CreateUserDto {
  @IsString()
  @MinLength(2)
  name: string;

  @IsEmail()
  email: string;

  @IsString()
  @MinLength(8)
  password: string;

  @IsOptional()
  @IsInt()
  @Min(0)
  @Max(150)
  age?: number;
}

Enable validation globally in main.ts:

import { ValidationPipe } from '@nestjs/common';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(new ValidationPipe({
    whitelist: true,           // Strip properties not in DTO
    forbidNonWhitelisted: true, // Throw error on extra properties
    transform: true,            // Auto-transform payloads to DTO instances
  }));
  await app.listen(3000);
}

With transform: true, path params and query params are automatically converted to the types specified in your DTOs.

Mapped Types

NestJS provides utility types for creating DTOs from existing ones:

import { PartialType, PickType, OmitType, IntersectionType } from '@nestjs/mapped-types';

// All fields optional
class UpdateUserDto extends PartialType(CreateUserDto) {}

// Only specific fields
class UserEmailDto extends PickType(CreateUserDto, ['email'] as const) {}

// All fields except some
class CreateUserWithoutPasswordDto extends OmitType(CreateUserDto, ['password'] as const) {}

// Combine two DTOs
class ExtendedUserDto extends IntersectionType(CreateUserDto, AdditionalInfoDto) {}

HTTP Status Codes and Response Headers

import { Controller, Post, HttpCode, Header, Redirect } from '@nestjs/common';

@Controller('users')
export class UsersController {

  // Custom status code (default for POST is 201)
  @Post()
  @HttpCode(200)
  create(@Body() dto: CreateUserDto) {
    return this.usersService.create(dto);
  }

  // Custom response header
  @Post()
  @Header('Cache-Control', 'none')
  createWithHeader(@Body() dto: CreateUserDto) {
    return this.usersService.create(dto);
  }

  // Redirect
  @Get('old-endpoint')
  @Redirect('https://example.com/new-endpoint', 301)
  redirectOld() {}

  // Dynamic redirect
  @Get('docs')
  @Redirect('https://docs.nestjs.com', 302)
  getDocs(@Query('version') version: string) {
    if (version === 'v5') {
      return { url: 'https://docs.nestjs.com/v5/' };
    }
  }
}

Async Controllers

Controllers can return promises or observables. NestJS handles both transparently:

@Controller('users')
export class UsersController {
  // Async/await
  @Get()
  async findAll(): Promise<User[]> {
    return this.usersService.findAll();
  }

  // Observable (RxJS)
  @Get()
  findAll(): Observable<User[]> {
    return this.usersService.findAll();
  }
}

Request Payload Patterns

Pagination Query DTO

import { IsOptional, IsInt, Min, Max } from 'class-validator';
import { Type } from 'class-transformer';

export class PaginationQueryDto {
  @IsOptional()
  @Type(() => Number)
  @IsInt()
  @Min(1)
  page?: number = 1;

  @IsOptional()
  @Type(() => Number)
  @IsInt()
  @Min(1)
  @Max(100)
  limit?: number = 20;
}

@Get()
findAll(@Query() query: PaginationQueryDto) {
  // query.page and query.limit are numbers with defaults
  return this.usersService.findAll(query.page, query.limit);
}

Common Pitfalls

  1. Route order matters -- Static routes must come before parameterized routes. @Get('active') must be defined before @Get(':id'), otherwise 'active' is treated as an ID.
  2. String params -- All route params and query params come as strings by default. Use transform: true in ValidationPipe or @Type(() => Number) to auto-convert.
  3. Using interfaces for DTOs -- Interfaces are erased at compile time. Use classes so class-validator decorators work at runtime.
  4. Injecting @Res() breaks automatic response -- When you inject the raw response object, you take over response handling completely.

Exercises

  1. Build a Products Controller

    Create a ProductsController with full CRUD operations. Implement DTOs for CreateProductDto (name, price, category, description) and UpdateProductDto. Add validation decorators and pagination support for the findAll endpoint.

    Hint

    Use @nestjs/mapped-types for PartialType. Add class-validator decorators. Use a PaginationQueryDto with @Type(() => Number) for query params.

  2. Nested Route Parameters

    Create a controller that handles routes like /users/:userId/posts/:postId/comments/:commentId. Extract all three parameters and return them in a structured response.

    Hint

    Use multiple @Param() decorators or @Param() without arguments to get the full params object.

  3. API Versioning

    Set up URI versioning in main.ts. Create two versions of a users endpoint: v1 returns basic user info (id, name), v2 returns extended info (id, name, email, createdAt). Both should use the same service.

    Hint

    Call app.enableVersioning({ type: VersioningType.URI }) in main.ts. Use @Version('1') and @Version('2') on controller methods.

  4. Custom Response Headers and Status Codes

    Create an endpoint that returns a 202 Accepted status for async operations, sets a custom X-Request-Id header, and includes a Location header pointing to a status-check URL.

    Hint

    Use @HttpCode(202) and @Header(). For dynamic headers, inject @Res({ passthrough: true }) and call res.header().

Common Problems & Solutions

1 Parameterized route catches requests meant for a static route
Cause: The route `@Get(':id')` is defined before `@Get('profile')`, so 'profile' is interpreted as an id parameter.
Solution: Always define static routes before parameterized routes in the controller. Move `@Get('profile')` above `@Get(':id')`.

// WRONG order:\n@Get(':id') // catches /users/profile\nfindOne() {}\n\n@Get('profile') // never reached\ngetProfile() {}\n\n// CORRECT order:\n@Get('profile') // matched first\ngetProfile() {}\n\n@Get(':id') // only non-matching routes fall through\nfindOne() {}

2 Query parameters are always strings
Symptoms: Type errors or unexpected behavior when doing arithmetic on query parameters.
Cause: HTTP query parameters are inherently strings. Without transformation, `@Query('page')` returns '1' (string) not 1 (number).
Solution: Enable `transform: true` in the global ValidationPipe, and use `@Type(() => Number)` from class-transformer in your DTO class.
3 Validation decorators are ignored on DTOs
Cause: The DTO is defined as an interface instead of a class, or the global ValidationPipe is not enabled.
Solution: Use classes (not interfaces) for DTOs. Add `app.useGlobalPipes(new ValidationPipe())` in main.ts. Ensure class-validator and class-transformer are installed.
