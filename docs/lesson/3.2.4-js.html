Writing Custom Middleware

Intuition

Writing custom middleware is like designing customs checkpoints at a border. Each checkpoint has a specific job — check passports (authentication), verify visas (authorization), scan luggage (validation). You design the rules, and every traveler (request) must pass through them in order.

Writing Custom Middleware

Introduction

While third-party middleware covers common needs, every real application requires custom middleware tailored to its specific requirements. Custom middleware lets you implement cross-cutting concerns — authentication, authorization, rate limiting, request validation, logging enrichment, and more — in reusable, composable functions.

Middleware Function Signature

Every Express middleware follows the same pattern:

import { Request, Response, NextFunction } from 'express';

function myMiddleware(req: Request, res: Response, next: NextFunction): void {
  // 1. Do something with req and/or res
  // 2. Either call next() to continue or send a response to stop
}

Middleware Factory Pattern

Most real-world middleware needs configuration. Use a factory function that returns middleware:

// Factory: returns a configured middleware function
function rateLimit(options: { windowMs: number; max: number }) {
  const requests = new Map<string, { count: number; resetTime: number }>();

  return (req: Request, res: Response, next: NextFunction) => {
    const key = req.ip || 'unknown';
    const now = Date.now();
    const record = requests.get(key);

    if (!record || now > record.resetTime) {
      requests.set(key, { count: 1, resetTime: now + options.windowMs });
      return next();
    }

    if (record.count >= options.max) {
      return res.status(429).json({
        error: 'Too many requests',
        retryAfter: Math.ceil((record.resetTime - now) / 1000),
      });
    }

    record.count++;
    next();
  };
}

// Usage
app.use('/api', rateLimit({ windowMs: 60_000, max: 100 }));

Common Custom Middleware Examples

Authentication Middleware

import jwt from 'jsonwebtoken';
import { config } from '../config';

interface JwtPayload {
  userId: number;
  email: string;
  role: string;
}

export function authenticate(req: Request, res: Response, next: NextFunction) {
  const authHeader = req.get('authorization');

  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'No token provided' });
  }

  const token = authHeader.slice(7);

  try {
    const payload = jwt.verify(token, config.jwt.secret) as JwtPayload;
    req.user = { id: payload.userId, email: payload.email, role: payload.role };
    next();
  } catch (error) {
    if (error instanceof jwt.TokenExpiredError) {
      return res.status(401).json({ error: 'Token expired' });
    }
    return res.status(401).json({ error: 'Invalid token' });
  }
}

Authorization (Role-Based)

export function authorize(...allowedRoles: string[]) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!req.user) {
      return res.status(401).json({ error: 'Not authenticated' });
    }

    if (!allowedRoles.includes(req.user.role)) {
      return res.status(403).json({
        error: 'Insufficient permissions',
        required: allowedRoles,
        current: req.user.role,
      });
    }

    next();
  };
}

// Usage: only admins and editors can create posts
router.post('/posts', authenticate, authorize('admin', 'editor'), createPost);

Request Validation

import { z, ZodSchema } from 'zod';

export function validate(schema: {
  body?: ZodSchema;
  query?: ZodSchema;
  params?: ZodSchema;
}) {
  return (req: Request, res: Response, next: NextFunction) => {
    const errors: Record<string, string[]> = {};

    if (schema.body) {
      const result = schema.body.safeParse(req.body);
      if (!result.success) {
        errors.body = result.error.errors.map(e => `${e.path.join('.')}: ${e.message}`);
      } else {
        req.body = result.data;
      }
    }

    if (schema.query) {
      const result = schema.query.safeParse(req.query);
      if (!result.success) {
        errors.query = result.error.errors.map(e => `${e.path.join('.')}: ${e.message}`);
      }
    }

    if (schema.params) {
      const result = schema.params.safeParse(req.params);
      if (!result.success) {
        errors.params = result.error.errors.map(e => `${e.path.join('.')}: ${e.message}`);
      }
    }

    if (Object.keys(errors).length > 0) {
      return res.status(400).json({ error: 'Validation failed', details: errors });
    }

    next();
  };
}

// Usage
const createUserSchema = {
  body: z.object({
    name: z.string().min(2).max(100),
    email: z.string().email(),
    password: z.string().min(8),
  }),
};

router.post('/users', validate(createUserSchema), createUser);

Request Logging with Context

import crypto from 'node:crypto';

export function requestContext(req: Request, res: Response, next: NextFunction) {
  const requestId = req.get('x-request-id') || crypto.randomUUID();
  const startTime = process.hrtime.bigint();

  req.requestId = requestId;
  res.set('X-Request-Id', requestId);

  res.on('finish', () => {
    const duration = Number(process.hrtime.bigint() - startTime) / 1_000_000;
    console.log(JSON.stringify({
      requestId,
      method: req.method,
      path: req.path,
      status: res.statusCode,
      duration: `${duration.toFixed(2)}ms`,
      userAgent: req.get('user-agent'),
      ip: req.ip,
      userId: req.user?.id,
    }));
  });

  next();
}

Response Time Header

export function responseTime(req: Request, res: Response, next: NextFunction) {
  const start = process.hrtime();

  res.on('finish', () => {
    const [seconds, nanoseconds] = process.hrtime(start);
    const ms = seconds * 1000 + nanoseconds / 1_000_000;
    // Header already sent, but this is logged
  });

  // Set header before response is sent
  const originalJson = res.json.bind(res);
  res.json = function (body: any) {
    const [seconds, nanoseconds] = process.hrtime(start);
    const ms = seconds * 1000 + nanoseconds / 1_000_000;
    res.set('X-Response-Time', `${ms.toFixed(2)}ms`);
    return originalJson(body);
  };

  next();
}

API Key Middleware

export function requireApiKey(req: Request, res: Response, next: NextFunction) {
  const apiKey = req.get('x-api-key') || req.query.apiKey;

  if (!apiKey) {
    return res.status(401).json({ error: 'API key required' });
  }

  // In production, validate against a database
  const validKeys = new Set(process.env.API_KEYS?.split(',') || []);

  if (!validKeys.has(apiKey as string)) {
    return res.status(403).json({ error: 'Invalid API key' });
  }

  next();
}

Async Middleware

Always handle async errors properly:

// Utility wrapper for async middleware
export function asyncHandler(
  fn: (req: Request, res: Response, next: NextFunction) => Promise<any>
) {
  return (req: Request, res: Response, next: NextFunction) => {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}

// Async middleware using the wrapper
export const loadUser = asyncHandler(async (req, res, next) => {
  const userId = req.params.userId;
  const user = await userService.findById(parseInt(userId, 10));

  if (!user) {
    return res.status(404).json({ error: 'User not found' });
  }

  req.user = user;
  next();
});

Composing Middleware

Combine multiple middleware into a single unit:

// Compose multiple middleware into one
function compose(...middlewares: Array<(req: Request, res: Response, next: NextFunction) => void>) {
  return (req: Request, res: Response, next: NextFunction) => {
    let index = 0;

    function run(err?: any): void {
      if (err) return next(err);
      if (index >= middlewares.length) return next();
      const middleware = middlewares[index++];
      try {
        middleware(req, res, run);
      } catch (e) {
        next(e);
      }
    }

    run();
  };
}

// Usage
const protectedRoute = compose(authenticate, authorize('admin'));
router.delete('/users/:id', protectedRoute, deleteUser);

Or simply use arrays — Express flattens them:

const protectedRoute = [authenticate, authorize('admin')];
router.delete('/users/:id', ...protectedRoute, deleteUser);

Testing Custom Middleware

import { describe, it, expect, vi } from 'vitest';

describe('authenticate middleware', () => {
  it('should reject requests without a token', () => {
    const req = { get: vi.fn().mockReturnValue(undefined) } as any;
    const res = {
      status: vi.fn().mockReturnThis(),
      json: vi.fn().mockReturnThis(),
    } as any;
    const next = vi.fn();

    authenticate(req, res, next);

    expect(res.status).toHaveBeenCalledWith(401);
    expect(res.json).toHaveBeenCalledWith({ error: 'No token provided' });
    expect(next).not.toHaveBeenCalled();
  });

  it('should call next() with a valid token', () => {
    const token = jwt.sign({ userId: 1, email: 'a@b.com', role: 'user' }, config.jwt.secret);
    const req = { get: vi.fn().mockReturnValue(`Bearer ${token}`) } as any;
    const res = {} as any;
    const next = vi.fn();

    authenticate(req, res, next);

    expect(next).toHaveBeenCalled();
    expect(req.user).toEqual({ id: 1, email: 'a@b.com', role: 'user' });
  });
});

Common Pitfalls

1. Not Using Return

// WRONG: execution continues after sending 401
function auth(req, res, next) {
  if (!req.user) {
    res.status(401).json({ error: 'Unauthorized' });
  }
  next(); // Called even after sending 401!
}

// CORRECT
function auth(req, res, next) {
  if (!req.user) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  next();
}

2. Swallowing Errors in Async Middleware

// WRONG: error disappears
async function loadData(req, res, next) {
  try {
    req.data = await fetchData();
    next();
  } catch (err) {
    // Error is swallowed! No response, no next(err)
    console.log(err);
  }
}

// CORRECT: forward error
async function loadData(req, res, next) {
  try {
    req.data = await fetchData();
    next();
  } catch (err) {
    next(err); // Forward to error handler
  }
}

Summary

Custom middleware lets you implement reusable logic that runs across your application. Use factory functions for configurable middleware, always handle async errors, use return before early responses, and compose middleware for clean route definitions. Test middleware in isolation by mocking req, res, and next.

Exercises

  1. Rate Limiter with Sliding Window

    Implement a rate limiting middleware factory that uses a sliding window algorithm. It should accept options for windowMs, max requests, and an optional key generator function. Return appropriate 429 responses with Retry-After headers.

    Hint

    Store timestamps of requests per key in an array. On each request, filter out timestamps older than windowMs. If the remaining count exceeds max, reject the request.

  2. Request Validation Middleware with Zod

    Create a validate() middleware factory that accepts Zod schemas for body, query, and params. It should validate all three, collect errors, and return a structured 400 error response. On success, replace req.body/query/params with the parsed (and coerced) values.

    Hint

    Use schema.safeParse() to validate without throwing. Map error.errors to readable messages. Use result.data as the sanitized value.

  3. Audit Log Middleware

    Write middleware that logs write operations (POST, PUT, PATCH, DELETE) to an in-memory audit log with: timestamp, user ID, method, path, request body, and response status code. Expose a GET /audit endpoint to view the log.

    Hint

    Use res.on('finish', ...) to capture the response status code after the handler completes. Only log if req.method is not GET or HEAD.

  4. Circuit Breaker Middleware

    Implement a simple circuit breaker middleware that tracks failures for downstream service calls. After 5 consecutive failures, the circuit opens and rejects requests with 503 for 30 seconds before allowing a single test request through.

    Hint

    Use a closure to maintain state: failure count, circuit state (closed/open/half-open), and last failure time. Reset on success.

Common Problems & Solutions

1 Middleware runs but next handler never receives the modified request data
Cause: The middleware assigns to a local variable instead of modifying req directly, or assigns to a property that downstream code is not reading.
Solution: Assign directly to req: req.user = userData. Ensure TypeScript declarations extend the Express Request interface. Verify downstream code reads the same property name.
2 Middleware factory creates shared state between requests
Cause: Variables declared outside the returned middleware function are shared across all requests (closures). This is intentional for caches/counters but problematic for per-request state.
Solution: For per-request state, assign to the req object inside the middleware. For shared state (rate limiting), use the closure intentionally with proper cleanup.
