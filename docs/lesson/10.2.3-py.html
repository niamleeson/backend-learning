Design a Key-Value Store (Redis/DynamoDB)

Intuition

A distributed key-value store is a partitioned, replicated hash map that uses consistent hashing for data distribution and quorum protocols for tunable consistency. The CAP theorem trade-off between consistency and availability is controlled by adjusting quorum parameters.

Design a Key-Value Store (Redis/DynamoDB)

Requirements

  • Support GET, PUT, DELETE operations with single-digit millisecond latency
  • Handle billions of key-value pairs across a distributed cluster
  • Provide tunable consistency (strong or eventual) and high availability
  • Support automatic partitioning, replication, and failure recovery

Architecture

Data is partitioned across nodes using consistent hashing with virtual nodes. Each partition is replicated to N nodes for durability. A coordinator node handles client requests and manages quorum reads/writes. An anti-entropy protocol (Merkle trees) detects and repairs inconsistencies between replicas.

Key Components

  • Consistent Hash Ring: Maps keys to nodes with virtual nodes for balanced distribution
  • Replication Manager: Copies data to N successor nodes on the hash ring
  • Quorum Protocol: Configurable W (write) and R (read) quorum values (W + R > N for strong consistency)
  • Gossip Protocol: Nodes exchange heartbeats to detect failures and propagate membership changes
  • Merkle Trees: Per-partition hash trees for efficient replica synchronization

Scaling

Add nodes and rebalance partitions with minimal data movement. Use virtual nodes to handle heterogeneous hardware. Implement hinted handoff for temporary failure handling. Use SSTables and LSM trees for efficient disk storage.

Trade-offs

  • W=1 gives fast writes but risks data loss; W=N ensures durability but is slow and less available
  • LSM trees optimize writes but reads may check multiple levels; B-trees optimize reads but have write amplification

Exercises

  1. Implement Consistent Hashing with Virtual Nodes

    Build a consistent hashing ring that maps keys to nodes. Support adding and removing nodes with minimal key reassignment. Use 150 virtual nodes per physical node to ensure even distribution.

    Hint

    Hash each virtual node ID to a position on the ring. To find the node for a key, hash the key and walk clockwise to the first virtual node. Map that virtual node back to its physical node.

  2. Build a Quorum Read/Write System

    Implement a coordinator that sends write requests to N replicas and waits for W acknowledgments. For reads, query N replicas, wait for R responses, and return the value with the highest version number. Make W and R configurable.

    Hint

    Use async requests to all replicas in parallel. Collect responses and proceed once the quorum threshold is met. Attach vector clocks or timestamps to values for version comparison.

Common Problems & Solutions

1 After adding new nodes to the cluster, some keys become temporarily unavailable or return stale data.
Cause: During rebalancing, key ownership transfers between nodes, but data migration has not completed, leaving gaps in the new owner's data.
Solution: Implement a two-phase rebalancing: first copy data to new owners while old owners continue serving, then atomically switch ownership once migration is verified. Use Merkle trees to verify data completeness.
2 Concurrent writes to the same key on different replicas create conflicting versions that cannot be automatically resolved.
Cause: With eventual consistency (W=1), two clients can write different values to different replicas simultaneously, creating sibling values.
Solution: Use vector clocks to track causal relationships between versions. When siblings are detected, return all conflicting versions to the client for application-level resolution, or use last-write-wins with synchronized timestamps.
