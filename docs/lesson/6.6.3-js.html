Docker Swarm vs Kubernetes: Choosing an Orchestrator

Intuition

Docker Swarm is like a family-run restaurant chain with 3-5 locations. The owner (manager node) knows every employee, every dish, and can coordinate everything easily. Kubernetes is like a multinational fast-food corporation with hundreds of locations, regional managers, automated supply chains, and complex logistics. Both serve food (run containers), but the level of complexity and capability is vastly different. Pick the one that matches the scale of your business.

Docker Swarm vs Kubernetes: Choosing an Orchestrator

Why Container Orchestration?

When you move beyond a single server running Docker Compose, you need an orchestrator to manage containers across multiple machines, handle failover, scaling, networking, and updates. The two main options are Docker Swarm (simpler) and Kubernetes (more powerful).

Docker Swarm

Setting Up a Swarm

# Initialize swarm on manager node
docker swarm init --advertise-addr 192.168.1.100

# Output gives you a join token for workers:
# docker swarm join --token SWMTKN-1-xxx 192.168.1.100:2377

# On worker nodes:
docker swarm join --token SWMTKN-1-xxx 192.168.1.100:2377

# List nodes
docker node ls

Deploying Services

# Create a service
docker service create \
  --name myapp-api \
  --replicas 3 \
  --publish 3000:3000 \
  --env NODE_ENV=production \
  --update-delay 10s \
  --update-parallelism 1 \
  myorg/myapp:1.0.0

# Scale
docker service scale myapp-api=5

# Update (rolling)
docker service update \
  --image myorg/myapp:2.0.0 \
  --update-delay 10s \
  --update-failure-action rollback \
  myapp-api

# Rollback
docker service rollback myapp-api

# View services
docker service ls
docker service ps myapp-api
docker service logs myapp-api -f

Docker Stack (Swarm's Compose)

# docker-stack.yml
version: '3.8'

services:
  api:
    image: myorg/myapp:1.0.0
    deploy:
      replicas: 3
      update_config:
        parallelism: 1
        delay: 10s
        failure_action: rollback
      rollback_config:
        parallelism: 1
        delay: 5s
      restart_policy:
        condition: on-failure
        delay: 5s
        max_attempts: 3
      resources:
        limits:
          cpus: '0.5'
          memory: 512M
        reservations:
          cpus: '0.25'
          memory: 256M
      placement:
        constraints:
          - node.role == worker
    ports:
      - "3000:3000"
    environment:
      NODE_ENV: production
      DATABASE_URL: postgresql://user:pass@postgres:5432/mydb
    networks:
      - app-network
    secrets:
      - db-password
      - jwt-secret

  postgres:
    image: postgres:16-alpine
    deploy:
      replicas: 1
      placement:
        constraints:
          - node.role == manager
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db-password
    networks:
      - app-network
    secrets:
      - db-password

  redis:
    image: redis:7-alpine
    deploy:
      replicas: 1
    command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru
    networks:
      - app-network

  nginx:
    image: nginx:1.25-alpine
    deploy:
      replicas: 2
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    networks:
      - app-network

networks:
  app-network:
    driver: overlay              # Overlay network for multi-node

volumes:
  pgdata:

secrets:
  db-password:
    external: true
  jwt-secret:
    external: true

# Create secrets
echo "super-secret" | docker secret create db-password -
echo "jwt-secret-key" | docker secret create jwt-secret -

# Deploy stack
docker stack deploy -c docker-stack.yml myapp

# List stacks
docker stack ls

# List stack services
docker stack services myapp

# Remove stack
docker stack rm myapp

Decision Framework

Choose Docker Swarm When:

1. Small to medium team (2-10 developers)
2. < 20 services to orchestrate
3. Already using Docker Compose (easy migration)
4. Need a quick setup (minutes, not days)
5. Limited DevOps resources
6. Single-region deployment
7. Simple scaling needs

Choose Kubernetes When:

1. Large team or multiple teams
2. 20+ microservices
3. Need auto-scaling (HPA, VPA, cluster autoscaler)
4. Multi-region / multi-cloud deployment
5. Advanced networking (service mesh, network policies)
6. Complex deployment strategies (canary, A/B testing)
7. Rich ecosystem needed (Helm, Operators, CRDs)
8. Hiring (K8s skills are more marketable)

Choose Neither When:

1. Single server is sufficient
2. < 5 containers
3. PM2 + Nginx handles your needs
4. Docker Compose on a single host works fine
5. Using a PaaS (Heroku, Railway, Render)
6. Serverless (Lambda, Cloud Functions) fits better

From Docker Compose to Swarm

# Your docker-compose.yml works with minor additions:
# 1. Add 'deploy' sections
# 2. Change network driver to 'overlay'
# 3. Use 'docker stack deploy' instead of 'docker compose up'

docker stack deploy -c docker-compose.yml myapp

Swarm Concept          Kubernetes Equivalent
──────────────────────────────────────────
Service                Deployment + Service
Stack                  Namespace + multiple resources
docker-stack.yml       Multiple YAML manifests (or Helm chart)
Overlay network        Pod networking (CNI)
Swarm secrets          Kubernetes Secrets
Config                 ConfigMap
Node constraints       Node affinity / tolerations

Common Pitfalls

  1. Starting with Kubernetes when Swarm would suffice - K8s is powerful but complex. If Swarm meets your needs, use it. You can always migrate later.
  2. Running stateful services (databases) in orchestrators - Databases prefer stable hosts. Use managed services (RDS, Cloud SQL) unless you have expertise with StatefulSets.
  3. Not considering managed K8s - Self-managing a K8s control plane is hard. Use EKS (AWS), GKE (GCP), or AKS (Azure).
  4. Ignoring the operational cost - Kubernetes requires ongoing maintenance: upgrades, security patches, monitoring, and an on-call team.
  5. Resume-driven development - Choosing K8s because it looks good on a resume rather than because the project needs it.

Exercises

  1. Docker Swarm Deployment

    Set up a 3-node Docker Swarm cluster (1 manager, 2 workers) using Docker Machine or VMs. Deploy a Node.js application stack with PostgreSQL and Redis. Test rolling updates and rollbacks. Verify that the application survives a worker node going down.

    Hint

    Use 'docker swarm init' on the manager. Join workers with the token. Deploy with 'docker stack deploy'. Simulate failure by stopping Docker on a worker node.

  2. Swarm to K8s Migration Plan

    Take a docker-stack.yml file and convert it to equivalent Kubernetes manifests: Deployment, Service, ConfigMap, Secret, and Ingress. Document the mapping between Swarm and K8s concepts.

    Hint

    Each Swarm service becomes a K8s Deployment + Service pair. Stack secrets become K8s Secrets. Deploy config maps for environment variables. Overlay network is handled by K8s CNI automatically.

  3. Architecture Decision Document

    Write a decision document for a hypothetical project: 3 microservices, 2 developers, expecting 1000 requests/day initially, growing to 100,000/day in a year. Evaluate: Docker Compose, Docker Swarm, and Kubernetes. Recommend a starting point and migration path.

    Hint

    Consider team size, operational complexity, scaling needs, and timeline. Start simple (Compose or Swarm) and plan migration triggers (when to move to K8s).

Common Problems & Solutions

1 Docker Swarm services not reachable across nodes
Cause: The overlay network is not properly configured, or ports required for Swarm communication are blocked by the firewall.
Solution: Ensure these ports are open between all Swarm nodes: TCP 2377 (cluster management), TCP/UDP 7946 (node communication), UDP 4789 (overlay network). Check: `docker network ls` for overlay networks. Verify services are on the same network.
2 Kubernetes cluster is too expensive
Cause: Running K8s control plane nodes + worker nodes + managed service fees adds up, especially for small applications.
Solution: Consider alternatives: Use a managed K8s with smaller node pools, use spot/preemptible instances for workers, right-size your nodes, or switch to Docker Swarm or a PaaS for cost-sensitive workloads. EKS alone costs ~$72/month for the control plane.
3 Cannot decide between Swarm and Kubernetes
Cause: Analysis paralysis from trying to pick the 'right' tool before understanding the actual requirements.
Solution: Start with the simplest option that works (often Docker Compose on a single server). Move to Swarm when you need multi-node. Move to Kubernetes when you need auto-scaling, complex networking, or a rich ecosystem. The migration path from Compose -> Swarm is very smooth.
