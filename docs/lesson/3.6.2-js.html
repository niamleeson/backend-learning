Pagination, Filtering, and Search

Intuition

Pagination is like reading a book — you do not load the entire book into your brain at once; you read one page at a time. Filtering is the table of contents that lets you jump to relevant chapters. Search is the index at the back that points you to specific topics across all chapters.

Pagination, Filtering, and Search

Introduction

Any API that returns collections of data needs pagination, filtering, and search. Without pagination, a request for "all posts" could return millions of records, destroying performance. Without filtering and search, clients must fetch everything and filter client-side. This lesson covers implementing robust, efficient collection endpoints with cursor-based and offset-based pagination, multi-field filtering, and full-text search.

Offset-Based Pagination

The simplest pagination approach uses page and limit (or skip and take):

// GET /api/posts?page=2&limit=20
router.get('/', asyncHandler(async (req, res) => {
  const page = Math.max(1, parseInt(req.query.page as string) || 1);
  const limit = Math.min(100, Math.max(1, parseInt(req.query.limit as string) || 20));
  const skip = (page - 1) * limit;

  const [data, total] = await Promise.all([
    prisma.post.findMany({
      skip,
      take: limit,
      orderBy: { createdAt: 'desc' },
      include: { author: { select: { id: true, name: true } } },
    }),
    prisma.post.count(),
  ]);

  res.json({
    success: true,
    data,
    meta: {
      total,
      page,
      limit,
      totalPages: Math.ceil(total / limit),
      hasNextPage: page * limit < total,
      hasPrevPage: page > 1,
    },
  });
}));

Pros and Cons

Offset Pagination:
  + Simple to implement
  + Allows jumping to any page
  + Easy for UI page numbers
  - Performance degrades on high offsets (OFFSET 100000 is slow)
  - Inconsistent if data changes between pages (items shift)

Cursor-Based Pagination

Uses a cursor (typically the last item's ID or timestamp) to fetch the next page:

// GET /api/posts?cursor=42&limit=20
router.get('/', asyncHandler(async (req, res) => {
  const limit = Math.min(100, parseInt(req.query.limit as string) || 20);
  const cursor = req.query.cursor ? parseInt(req.query.cursor as string) : undefined;

  const posts = await prisma.post.findMany({
    take: limit + 1, // Fetch one extra to determine hasNextPage
    ...(cursor && {
      cursor: { id: cursor },
      skip: 1, // Skip the cursor item itself
    }),
    orderBy: { id: 'desc' },
    include: { author: { select: { id: true, name: true } } },
  });

  const hasNextPage = posts.length > limit;
  const data = hasNextPage ? posts.slice(0, -1) : posts;
  const nextCursor = hasNextPage ? data[data.length - 1].id : null;

  res.json({
    success: true,
    data,
    meta: {
      hasNextPage,
      nextCursor,
      limit,
    },
  });
}));

Cursor Pagination:
  + Consistent performance regardless of position
  + No missing/duplicate items when data changes
  + Works well with infinite scroll UIs
  - Cannot jump to arbitrary pages
  - More complex implementation

Filtering

Basic Filtering

// GET /api/posts?published=true&authorId=42&tag=typescript
router.get('/', asyncHandler(async (req, res) => {
  const { published, authorId, tag } = req.query;

  const where: Prisma.PostWhereInput = {};

  if (published !== undefined) {
    where.published = published === 'true';
  }
  if (authorId) {
    where.authorId = parseInt(authorId as string, 10);
  }
  if (tag) {
    where.tags = { some: { name: tag as string } };
  }

  const data = await prisma.post.findMany({
    where,
    orderBy: { createdAt: 'desc' },
  });

  res.json({ success: true, data });
}));

Advanced Filtering with Operators

// GET /api/posts?createdAfter=2025-01-01&createdBefore=2025-12-31&minComments=5
router.get('/', asyncHandler(async (req, res) => {
  const { createdAfter, createdBefore, minComments } = req.query;

  const where: Prisma.PostWhereInput = {
    published: true,
    ...(createdAfter && { createdAt: { gte: new Date(createdAfter as string) } }),
    ...(createdBefore && {
      createdAt: {
        ...(where.createdAt as any),
        lte: new Date(createdBefore as string),
      },
    }),
    ...(minComments && {
      comments: { some: {} }, // At least one comment
    }),
  };

  // ... pagination and response
}));

Sorting

// GET /api/posts?sort=title&order=asc
const sortField = req.query.sort as string || 'createdAt';
const sortOrder = req.query.order as string || 'desc';

const allowedSortFields = ['createdAt', 'title', 'updatedAt'];
const orderBy = allowedSortFields.includes(sortField)
  ? { [sortField]: sortOrder === 'asc' ? 'asc' : 'desc' }
  : { createdAt: 'desc' };

const posts = await prisma.post.findMany({ orderBy });

Full-Text Search

Simple Search with LIKE/ILIKE

// GET /api/posts?search=express middleware
router.get('/', asyncHandler(async (req, res) => {
  const search = req.query.search as string;

  const where: Prisma.PostWhereInput = {
    published: true,
    ...(search && {
      OR: [
        { title: { contains: search, mode: 'insensitive' } },
        { content: { contains: search, mode: 'insensitive' } },
        { tags: { some: { name: { contains: search, mode: 'insensitive' } } } },
      ],
    }),
  };

  // ... query and response
}));

PostgreSQL Full-Text Search

// For better search, use PostgreSQL's built-in full-text search
const posts = await prisma.$queryRaw`
  SELECT id, title, slug, excerpt,
    ts_rank(to_tsvector('english', title || ' ' || content), plainto_tsquery('english', ${search})) as rank
  FROM posts
  WHERE to_tsvector('english', title || ' ' || content) @@ plainto_tsquery('english', ${search})
    AND published = true
  ORDER BY rank DESC
  LIMIT ${limit} OFFSET ${skip}
`;

Building a Reusable Query Builder

// src/utils/queryBuilder.ts
import { Prisma } from '@prisma/client';

export function buildPagination(query: { page?: number; limit?: number }) {
  const page = Math.max(1, query.page || 1);
  const limit = Math.min(100, Math.max(1, query.limit || 20));
  return { skip: (page - 1) * limit, take: limit, page, limit };
}

export function buildPostFilters(query: Record<string, any>): Prisma.PostWhereInput {
  const where: Prisma.PostWhereInput = {};

  if (query.published !== undefined) where.published = query.published;
  if (query.authorId) where.authorId = query.authorId;
  if (query.tag) where.tags = { some: { name: query.tag } };
  if (query.search) {
    where.OR = [
      { title: { contains: query.search, mode: 'insensitive' } },
      { content: { contains: query.search, mode: 'insensitive' } },
    ];
  }
  if (query.createdAfter) {
    where.createdAt = { ...(where.createdAt as any || {}), gte: new Date(query.createdAfter) };
  }
  if (query.createdBefore) {
    where.createdAt = { ...(where.createdAt as any || {}), lte: new Date(query.createdBefore) };
  }

  return where;
}

export function buildSort(sort?: string, order?: string, allowedFields: string[] = []) {
  if (sort && allowedFields.includes(sort)) {
    return { [sort]: order === 'asc' ? 'asc' : 'desc' };
  }
  return { createdAt: 'desc' as const };
}

export function buildMeta(total: number, page: number, limit: number) {
  return {
    total,
    page,
    limit,
    totalPages: Math.ceil(total / limit),
    hasNextPage: page * limit < total,
    hasPrevPage: page > 1,
  };
}

Common Pitfalls

Not Limiting Maximum Page Size

// WRONG: client can request limit=1000000
const limit = parseInt(req.query.limit as string) || 20;

// CORRECT: enforce maximum
const limit = Math.min(100, Math.max(1, parseInt(req.query.limit as string) || 20));

Allowing Sort by Any Field

// WRONG: allows sorting by any column (potential SQL injection with raw queries)
const orderBy = { [req.query.sort]: 'asc' };

// CORRECT: whitelist allowed sort fields
const allowedFields = ['createdAt', 'title', 'updatedAt'];
const sort = allowedFields.includes(req.query.sort as string) ? req.query.sort : 'createdAt';

Summary

Pagination, filtering, and search are essential for any collection endpoint. Use offset pagination for simple cases with page numbers, cursor pagination for high-performance infinite scroll. Validate and whitelist all filter and sort parameters. Build reusable query utilities to keep your route handlers clean.

Exercises

  1. Implement Both Pagination Styles

    Build a posts endpoint that supports both offset (?page=2&limit=20) and cursor (?cursor=42&limit=20) pagination based on a ?mode=offset|cursor query parameter. Include complete metadata in responses.

    Hint

    Check the mode query parameter. For offset, use skip/take. For cursor, use cursor/skip(1)/take(limit+1). Each mode returns different meta fields.

  2. Multi-Tag Filtering

    Implement tag filtering that supports both AND and OR logic: ?tags=node,express&tagMode=and (posts with ALL tags) vs ?tags=node,express&tagMode=or (posts with ANY tag).

    Hint

    For AND: use tags: { every: { name: { in: tagArray } } } but note Prisma limitations. For OR: use tags: { some: { name: { in: tagArray } } }. For strict AND, you may need raw SQL.

  3. Search with Ranking

    Implement a search endpoint that ranks results by relevance. Title matches should rank higher than content matches. Use PostgreSQL full-text search with ts_rank, or implement a simple scoring system in JavaScript.

    Hint

    For simple scoring: search both fields, assign score 10 for title match and 1 for content match, sort by score descending.

Common Problems & Solutions

1 Pagination returns duplicate or missing items when data changes
Cause: Offset-based pagination shifts items when rows are inserted or deleted between page requests.
Solution: Use cursor-based pagination for data that changes frequently. Or use a consistent sort key (like ID) that does not change when new items are added.
2 Search is very slow on large tables
Cause: LIKE '%search%' queries cannot use standard B-tree indexes and require a full table scan.
Solution: Use PostgreSQL full-text search indexes (GIN index on tsvector columns). Or use a dedicated search engine like Elasticsearch or Meilisearch for complex search needs.
