Saga Pattern and Distributed Transactions

Intuition

A saga is like planning a vacation with separate bookings for flight, hotel, and car. If the hotel booking fails, you cancel the flight. Each booking is independent, but you have a plan for undoing each step if something goes wrong later.

Saga Pattern & Distributed Transactions

In a microservices architecture, a single business operation often spans multiple services, each with its own database. Traditional ACID transactions do not work across service boundaries. The Saga pattern provides a way to manage distributed transactions through a sequence of local transactions with compensating actions.

Core Concepts

  • Saga: A sequence of local transactions where each step has a corresponding compensating transaction. If step N fails, compensating transactions for steps N-1 through 1 are executed to undo partial work.
  • Choreography-Based Saga: Each service listens for events and decides what to do next. No central coordinator. Simple for small flows but hard to understand and debug for complex workflows.
  • Orchestration-Based Saga: A central orchestrator directs the saga, telling each service what to do. Easier to understand and monitor but introduces a single point of coordination.
  • Compensating Transactions: The undo operation for each step. Not always a simple rollback—e.g., compensating a "ship order" step might mean "create return shipment" rather than "undo shipping."
  • Semantic Locks: Reserve resources during the saga to prevent conflicts. For example, mark an order as "pending" while the saga is in progress to prevent concurrent modifications.

Key Takeaways

  1. Sagas provide eventual consistency, not immediate consistency—design your UI and APIs to handle intermediate states.
  2. Orchestration is preferred for complex workflows; choreography works for simple 2-3 step flows.
  3. Every saga step must have a well-defined compensating action; if compensation is impossible, that step should be the last in the saga.

Exercises

  1. Implement an Orchestration-Based Saga

    Build a saga orchestrator for an order workflow: (1) Reserve inventory, (2) Process payment, (3) Create shipment. If any step fails, execute compensating transactions for previous steps. Use a state machine to track saga progress.

    Hint

    Model the saga as a state machine with states: STARTED, INVENTORY_RESERVED, PAYMENT_PROCESSED, SHIPMENT_CREATED, COMPLETED, COMPENSATING, FAILED. Store the state in a database and transition on each step's success or failure.

  2. Build a Choreography-Based Saga

    Implement the same order workflow using choreography. Each service publishes events and subscribes to others: OrderService emits OrderCreated, InventoryService listens and emits InventoryReserved or InventoryFailed, etc. Handle the compensation chain through failure events.

    Hint

    Each service subscribes to both success and failure events from upstream services. On PaymentFailed, InventoryService subscribes and releases the reservation. Use a correlation ID to link all events in a saga.

Common Problems & Solutions

1 A saga gets stuck in an intermediate state and never completes or compensates.
Cause: A service crashed after acknowledging but before publishing its result event, or a message was lost.
Solution: Implement saga timeouts: if a saga does not progress within a threshold, trigger compensation automatically. Use the outbox pattern to ensure event publishing is atomic with the local transaction.
2 Compensating transactions fail, leaving the system in an inconsistent state.
Cause: Compensating actions can fail too (e.g., payment refund service is down), and there is no mechanism to retry them.
Solution: Implement compensation retries with exponential backoff. Log failed compensations to a dead letter queue for manual resolution. Design compensations to be idempotent so they can be safely retried.
