Creating and Running Migrations

Intuition

Migrations are like numbered instructions for building your database. You can fast-forward to any point or rewind step by step.

Creating and Running Migrations

Migrations are scripts that evolve your database schema. Each migration has an upgrade (apply) and downgrade (revert) function.

# Generate a migration
# $ alembic revision -m "add users table"

from alembic import op
import sqlalchemy as sa

def upgrade():
    op.create_table(
        'users',
        sa.Column('id', sa.Integer, primary_key=True),
        sa.Column('name', sa.String(100), nullable=False),
        sa.Column('email', sa.String(255), unique=True),
        sa.Column('created_at', sa.DateTime, server_default=sa.func.now())
    )
    op.create_index('idx_users_email', 'users', ['email'])

def downgrade():
    op.drop_index('idx_users_email')
    op.drop_table('users')

# Run migrations
# $ alembic upgrade head       # apply all pending
# $ alembic upgrade +1         # apply next one
# $ alembic downgrade -1       # revert last one
# $ alembic current            # show current revision
# $ alembic history            # show all revisions

Key Takeaways: - Each migration gets a unique revision ID. Alembic tracks the current version in an alembic_version table. - Always write both upgrade() and downgrade() for reversibility. - Use op.add_column, op.drop_column, op.alter_column, op.create_index for schema changes.

Exercises

  1. Write a Migration Manually

    Create a migration that adds a 'phone' column to the users table and a partial index on it where phone IS NOT NULL.

    Hint

    op.add_column('users', sa.Column('phone', sa.String(20))), op.create_index with postgresql_where.

  2. Data Migration

    Write a migration that adds a 'full_name' column and populates it from existing 'first_name' and 'last_name' columns.

    Hint

    Add column first, then use op.execute("UPDATE users SET full_name = first_name || ' ' || last_name").

Common Problems & Solutions

1 Migration fails midway, leaving the database in a broken state.
Cause: Some operations like CREATE INDEX CONCURRENTLY cannot run inside a transaction.
Solution: Test migrations on a staging database first. For non-transactional DDL, use op.execute() outside the transaction.
2 Downgrade does not fully revert the upgrade.
Cause: Downgrade function is incomplete or does not reverse data changes.
Solution: Always test downgrade after upgrade. Include data rollback logic, not just schema rollback.
