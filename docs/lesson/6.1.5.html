Standardized Error Responses (RFC 7807)

Intuition

Without RFC 7807, every API invents its own error format -- one uses 'message', another uses 'error', another nests them differently. It is like every hospital using a different form for patient records. RFC 7807 is the standardized medical form: everyone knows where to find the diagnosis (title), the details (detail), and the reference code (type).

Standardized Error Responses (RFC 7807)

RFC 7807 (Problem Details for HTTP APIs) defines a standard format for error responses, making it easier for clients to parse and handle errors consistently across different APIs.

RFC 7807 Error Response

from fastapi import FastAPI, HTTPException, Request
from fastapi.responses import JSONResponse

app = FastAPI()

class ProblemDetail(Exception):
    def __init__(self, status: int, title: str, detail: str,
                 type_uri: str = "about:blank", instance: str = None,
                 **kwargs):
        self.status = status
        self.title = title
        self.detail = detail
        self.type_uri = type_uri
        self.instance = instance
        self.extra = kwargs

@app.exception_handler(ProblemDetail)
async def problem_detail_handler(request: Request, exc: ProblemDetail):
    content = {
        "type": exc.type_uri,
        "title": exc.title,
        "status": exc.status,
        "detail": exc.detail,
    }
    if exc.instance:
        content["instance"] = exc.instance
    content.update(exc.extra)
    return JSONResponse(
        status_code=exc.status,
        content=content,
        media_type="application/problem+json"
    )

@app.get("/accounts/{account_id}/withdraw")
def withdraw(account_id: int, amount: float):
    balance = 50.0  # fetched from DB
    if amount > balance:
        raise ProblemDetail(
            status=422,
            title="Insufficient Funds",
            detail=f"Account {account_id} has {balance}, cannot withdraw {amount}",
            type_uri="https://api.example.com/errors/insufficient-funds",
            instance=f"/accounts/{account_id}/withdraw",
            balance=balance,
            requested=amount,
        )

Common Mistakes

  • Returning error responses with inconsistent shapes across endpoints
  • Using generic 500 errors without meaningful detail
  • Leaking stack traces or internal details in production error responses

Key Takeaways

  • RFC 7807 provides: type, title, status, detail, and instance fields
  • Use the application/problem+json content type for error responses
  • You can extend the format with custom fields relevant to the error domain

Exercises

  1. Implement a Problem Detail Exception Handler

    Create a FastAPI exception handler that catches all HTTPExceptions and converts them to RFC 7807 format with the correct content type. Include a 'type' URI that links to your error documentation.

    Hint

    Register a global exception handler with @app.exception_handler(HTTPException). Map status codes to type URIs and titles using a dictionary.

  2. Validation Error Formatter

    Override FastAPI's default RequestValidationError handler to return RFC 7807 responses. Include an 'errors' extension field that lists each validation failure with its location (body/query/path) and message.

    Hint

    Import RequestValidationError from fastapi.exceptions. Use exc.errors() to get the list of validation errors and restructure them.

  3. Error Catalog System

    Build an error catalog where each error type is defined once (with type URI, default title, and status code), and handlers reference errors by code. This ensures consistency across all endpoints.

    Hint

    Create an ErrorCatalog class or dictionary mapping error codes to ProblemDetail defaults. Endpoints raise errors by code, and the handler fills in the standard fields.

Common Problems & Solutions

1 Client error handling is fragile and breaks between endpoints
Symptoms: ['Client code has endpoint-specific error parsing logic', "Frontend shows 'Unknown error' for unrecognized error shapes"]
Cause: Each endpoint returns errors in a different format -- some use {"error": ...}, others use {"message": ...}, others return plain strings.
Solution: Adopt RFC 7807 across the entire API. Use a global exception handler so all errors follow the same structure regardless of the endpoint.
2 Sensitive internal details leaked in error responses
Cause: Stack traces, database queries, or internal paths are included in production error messages.
Solution: In production, return only the RFC 7807 standard fields with user-safe messages. Log the full details server-side. Use different error detail levels for development vs production.
