Design Search Autocomplete (Google)

Intuition

Search autocomplete is fundamentally a prefix-lookup problem where pre-computing the top results at every trie node turns an expensive traversal into an O(1) read. The real challenge is keeping those pre-computed results fresh as query popularity shifts in real time.

Design Search Autocomplete (Google)

Requirements

  • Return top-k suggestions as user types each character
  • Suggestions ranked by popularity, freshness, and personalization
  • Latency under 100ms per keystroke; support billions of queries/day
  • Handle multi-language input and trending queries in near real-time

Architecture

A Trie (prefix tree) serves as the core data structure, stored in-memory across distributed nodes. An offline pipeline aggregates query logs, computes frequency scores, and rebuilds the trie periodically. A real-time layer captures trending queries via stream processing and merges them into the trie.

Key Components

  • Trie Nodes: Store top-k results at each prefix to avoid traversal at query time
  • Data Collection Service: Aggregates raw query logs into frequency tables
  • Ranking Service: Blends popularity, recency, and user context into a final score
  • Zookeeper/Consistent Hashing: Partition tries by prefix range across nodes

Scaling

Shard tries alphabetically or by hash prefix. Use read replicas and CDN edge caching for hot prefixes. Replicate across regions for low latency.

Trade-offs

  • Pre-computed top-k per node trades storage for speed
  • Periodic rebuild vs. real-time update: freshness vs. complexity
  • Personalization increases relevance but adds privacy and storage cost

Exercises

  1. Build a Basic Trie with Top-K

    Implement a trie that stores the top-5 suggestions at each node. Insert a corpus of 10,000 queries with frequencies and benchmark prefix lookup latency.

    Hint

    At each node keep a min-heap of size k; when inserting, bubble up frequency changes from leaf to root.

  2. Simulate Trending Query Injection

    Create a stream simulator that emits 'trending' queries at high frequency. Merge these into your trie in near real-time and verify that suggestions update within 5 seconds.

    Hint

    Use a separate in-memory buffer for trending queries and merge it with the main trie on a short timer, replacing entries only when the trending score exceeds existing ones.

Common Problems & Solutions

1 Stale suggestions persist for hours after a trending event ends
Cause: The offline trie rebuild runs on a long schedule and the real-time layer only adds, never demotes entries
Solution: Implement a TTL or exponential decay on trending scores so entries naturally expire, and run incremental trie updates every few minutes.
2 Memory usage spikes when the full trie is loaded on every node
Cause: No sharding; each node holds the entire prefix space
Solution: Partition the trie by prefix ranges using consistent hashing and route queries to the responsible shard.
