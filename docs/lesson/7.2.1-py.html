Dockerfile: Building Python Application Images

Intuition

A Dockerfile is like flat-pack furniture instructions. Each instruction (FROM, RUN, COPY) is a step, and Docker caches each step as a layer. If step 5 changes but steps 1-4 haven't, Docker reuses the cached layers and only rebuilds from step 5 onward -- like skipping furniture steps you've already completed.

Dockerfile: Building Python Application Images

Docker packages your application and all its dependencies into a portable container image. A Dockerfile is the recipe for building that image.

Basic Python Dockerfile

FROM python:3.12-slim

# Set environment variables
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

# Set working directory
WORKDIR /app

# Install system dependencies
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# Install Python dependencies (cached layer)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY . .

# Create non-root user
RUN adduser --disabled-password --gecos '' appuser
USER appuser

# Expose port and set entrypoint
EXPOSE 8000
CMD ["gunicorn", "-w", "4", "-b", "0.0.0.0:8000", "app:create_app()"]

Building and Running

# Build the image
docker build -t myapp:latest .

# Run the container
docker run -d -p 8000:8000 --name myapp \
  -e DATABASE_URL=postgresql://... \
  myapp:latest

# View logs
docker logs -f myapp

.dockerignore

__pycache__
*.pyc
.git
.env
venv/
.pytest_cache
node_modules

Common Mistakes

  • Copying code before installing dependencies, busting the pip install cache on every code change
  • Running containers as root -- always create and switch to a non-root user
  • Not using .dockerignore, resulting in bloated images with .git and pycache

Key Takeaways

  • Order Dockerfile instructions from least to most frequently changing for optimal caching
  • Use python:3.x-slim instead of the full image to reduce size by ~600MB
  • Always set PYTHONUNBUFFERED=1 so logs appear immediately in docker logs

Exercises

  1. Build a FastAPI Docker Image

    Write a Dockerfile for a FastAPI application that uses Uvicorn, installs from a pyproject.toml, creates a non-root user, and exposes port 8000. Build and verify it runs correctly.

    Hint

    Use pip install . if using pyproject.toml. Use CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0"].

  2. Layer Caching Optimization

    Given a Dockerfile that copies all code first then installs dependencies, refactor it to maximize layer caching. Measure the build time difference.

    Hint

    Copy requirements.txt and pip install BEFORE copying the rest of the application code.

Common Problems & Solutions

1 Docker build fails with 'pip install' errors
Symptoms: ["error: command 'gcc' failed", 'ModuleNotFoundError during build']
Cause: Missing system-level build dependencies (gcc, libffi-dev, etc.) needed to compile Python packages.
Solution: Add a RUN step before pip install to install build dependencies: `RUN apt-get update && apt-get install -y gcc libffi-dev`.
2 Container runs but logs don't appear
Symptoms: ['docker logs shows nothing', 'Logs only appear when container stops']
Cause: Python buffers stdout by default inside containers.
Solution: Set `ENV PYTHONUNBUFFERED=1` in the Dockerfile or pass `-e PYTHONUNBUFFERED=1` at runtime.
3 Image size is over 1GB
Cause: Using the full python base image and not cleaning up apt caches.
Solution: Use `python:3.12-slim` as the base, add `--no-install-recommends` to apt-get, and `rm -rf /var/lib/apt/lists/*` after installing. Use multi-stage builds for even smaller images.

docker images myapp # Check image size
