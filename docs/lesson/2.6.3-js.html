Drizzle Migrations and Drizzle Kit

Intuition

Drizzle Kit is like a translator that watches your TypeScript schema and writes the SQL migration letters for you. You can trust the translations and send them as-is, or review and tweak them before sending. Unlike a black-box translator, you see every word and can change anything.

Drizzle Migrations and Drizzle Kit

Introduction

Drizzle Kit is the CLI companion to Drizzle ORM that handles migrations. Like Prisma Migrate, it diffs your TypeScript schema against the database and generates SQL migration files. But unlike Prisma, Drizzle Kit gives you the raw SQL files which you can edit freely before applying.

Drizzle Kit Setup

npm install -D drizzle-kit

// drizzle.config.ts
import { defineConfig } from 'drizzle-kit';

export default defineConfig({
  schema: './src/db/schema.ts',
  out: './drizzle',           // migration output directory
  dialect: 'postgresql',
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
  verbose: true,
  strict: true,
});

Generating Migrations

# Generate migration from schema changes
npx drizzle-kit generate

# This creates:
# drizzle/
#   0000_initial.sql
#   0001_add_products.sql
#   meta/
#     _journal.json
#     0000_snapshot.json

Example generated migration:

-- drizzle/0000_initial.sql
CREATE TYPE "role" AS ENUM ('customer', 'admin', 'vendor');

CREATE TABLE IF NOT EXISTS "users" (
  "id" uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  "email" varchar(320) NOT NULL,
  "name" text NOT NULL,
  "password_hash" text NOT NULL,
  "role" "role" NOT NULL DEFAULT 'customer',
  "created_at" timestamp with time zone NOT NULL DEFAULT now(),
  "updated_at" timestamp with time zone NOT NULL DEFAULT now(),
  "deleted_at" timestamp with time zone,
  CONSTRAINT "users_email_unique" UNIQUE("email")
);

CREATE UNIQUE INDEX "idx_users_email" ON "users" USING btree ("email");

Applying Migrations

// src/db/migrate.ts
import { drizzle } from 'drizzle-orm/node-postgres';
import { migrate } from 'drizzle-orm/node-postgres/migrator';
import { Pool } from 'pg';

async function runMigrations() {
  const pool = new Pool({ connectionString: process.env.DATABASE_URL });
  const db = drizzle(pool);

  console.log('Running migrations...');
  await migrate(db, { migrationsFolder: './drizzle' });
  console.log('Migrations complete.');

  await pool.end();
}

runMigrations().catch(console.error);

# Run migration script
npx tsx src/db/migrate.ts

Drizzle Kit Commands

# Generate migration files from schema diff
npx drizzle-kit generate

# Push schema directly to database (no migration files - dev only)
npx drizzle-kit push

# Pull schema from existing database (introspection)
npx drizzle-kit pull

# Open Drizzle Studio (visual database browser)
npx drizzle-kit studio

# Check for schema drift
npx drizzle-kit check

# Drop migration
npx drizzle-kit drop

Drizzle Studio

npx drizzle-kit studio
# Opens at https://local.drizzle.studio

Drizzle Studio provides a web UI for browsing and editing data, similar to Prisma Studio.

Migration Workflow

Development:
  1. Edit schema.ts
  2. npx drizzle-kit generate  (create migration SQL)
  3. Review the generated SQL (edit if needed!)
  4. npx tsx src/db/migrate.ts  (apply migration)
  5. Test your changes

Production:
  1. Migrations are committed to git
  2. CI/CD runs: npx tsx src/db/migrate.ts
  3. Only applies pending migrations
  4. Uses drizzle/__drizzle_migrations table to track state

Quick development (no migration files):
  1. Edit schema.ts
  2. npx drizzle-kit push  (applies directly, no migration file)
  3. Use for prototyping only!

Editing Generated Migrations

Unlike Prisma, Drizzle encourages editing migration files:

-- Drizzle generated a DROP + ADD, but we want RENAME
-- Original:
-- ALTER TABLE "users" DROP COLUMN "name";
-- ALTER TABLE "users" ADD COLUMN "full_name" text NOT NULL;

-- Edited:
ALTER TABLE "users" RENAME COLUMN "name" TO "full_name";

Seeding

// src/db/seed.ts
import { db } from './index';
import { users, products, categories } from './schema';

async function seed() {
  console.log('Seeding database...');

  // Clear tables
  await db.delete(orderItems);
  await db.delete(orders);
  await db.delete(products);
  await db.delete(categories);
  await db.delete(users);

  // Seed users
  const [admin] = await db.insert(users).values({
    email: 'admin@example.com',
    name: 'Admin User',
    passwordHash: '$2b$10$...',
    role: 'admin',
  }).returning();

  // Seed categories
  const [electronics] = await db.insert(categories).values({
    name: 'Electronics',
    slug: 'electronics',
  }).returning();

  // Seed products
  await db.insert(products).values([
    { name: 'Laptop', slug: 'laptop', price: '999.99', categoryId: electronics.id, stock: 50 },
    { name: 'Mouse', slug: 'mouse', price: '29.99', categoryId: electronics.id, stock: 200 },
  ]);

  console.log('Seeding complete.');
  process.exit(0);
}

seed().catch(console.error);

Drizzle vs Prisma Migrations Comparison

Feature             Drizzle Kit          Prisma Migrate
-----------------------------------------------------------
Schema language     TypeScript           .prisma DSL
Generated output    Raw SQL files        Raw SQL files
Editable SQL        Yes (encouraged)     Yes (possible)
Introspection       drizzle-kit pull     prisma db pull
Direct push         drizzle-kit push     prisma db push
Visual browser      Drizzle Studio       Prisma Studio
Code generation     No (types from TS)   Yes (required)
Migration runner    Custom script        prisma migrate deploy

Exercises

  1. Full Migration Lifecycle

    Create a schema with 4 tables, generate initial migration, apply it, then make 3 schema changes (add column, add table, add index). Generate and apply each migration separately. Test rollback by reversing changes.

    Hint

    Use drizzle-kit generate after each schema change. Review the SQL before applying. Drizzle doesn't auto-generate down migrations, so plan reversals manually.

  2. Introspect and Migrate

    Use drizzle-kit pull to introspect an existing database. Clean up the generated schema (add proper relation names, fix types). Then make a change and generate a migration.

    Hint

    Pull generates schema.ts from the database. You'll need to add relations manually. After cleanup, changes will generate proper migration diffs.

  3. CI/CD Migration Script

    Write a robust migration script that: checks database connectivity, runs pending migrations with a timeout, logs the results, and exits with appropriate codes. Integrate into a Docker entrypoint.

    Hint

    Wrap migrate() in a try-catch. Set a connection timeout. Log migration names as they run. Exit with code 1 on failure.

Common Problems & Solutions

1 drizzle-kit generate produces no output
Cause: The schema file path in drizzle.config.ts doesn't match the actual file location, or there are no changes from the last snapshot.
Solution: Verify the schema path in drizzle.config.ts. Check the meta/ folder for the latest snapshot. If starting fresh, delete the drizzle/ folder and regenerate.
2 Migration fails with 'relation already exists'
Cause: The migration was partially applied previously, or the database was modified outside of migrations.
Solution: Check the __drizzle_migrations table to see what's been applied. For a fresh start, drop the table and re-run all migrations. For production, manually resolve by editing the migration or marking it as applied.
3 Drizzle generates destructive migration (DROP COLUMN) for a rename
Cause: Drizzle Kit can't detect renames — it sees a column disappear and a new one appear.
Solution: Use `--custom` flag or edit the generated SQL to use ALTER TABLE RENAME COLUMN instead of DROP + ADD. Always review generated SQL before applying.
