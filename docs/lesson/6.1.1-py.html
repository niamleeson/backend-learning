API Versioning Strategies

Intuition

Think of API versioning like editions of a textbook. Students using the 3rd edition can keep studying from it, but new students get the 4th edition with updated content. You retire old editions only after giving everyone time to switch.

API Versioning Strategies

As your API evolves, you need a strategy to introduce breaking changes without disrupting existing clients. There are three primary versioning approaches, each with distinct trade-offs.

URI Path Versioning

The most common and visible approach:

from fastapi import FastAPI

app = FastAPI()

@app.get("/v1/users/{user_id}")
def get_user_v1(user_id: int):
    return {"id": user_id, "name": "Alice"}

@app.get("/v2/users/{user_id}")
def get_user_v2(user_id: int):
    return {"id": user_id, "full_name": "Alice Smith", "email": "alice@example.com"}

Header-Based Versioning

Clients specify the version via a custom header or Accept header:

from fastapi import Header, HTTPException

@app.get("/users/{user_id}")
def get_user(user_id: int, x_api_version: str = Header(default="1")):
    if x_api_version == "2":
        return {"id": user_id, "full_name": "Alice Smith"}
    return {"id": user_id, "name": "Alice"}

Query Parameter Versioning

@app.get("/users/{user_id}")
def get_user(user_id: int, version: int = 1):
    if version == 2:
        return {"id": user_id, "full_name": "Alice Smith"}
    return {"id": user_id, "name": "Alice"}

Common Mistakes

  • Versioning too aggressively: every minor change does not need a new version
  • Not deprecating old versions with a clear sunset timeline
  • Mixing versioning strategies within the same API

Key Takeaways

  • URI versioning is the simplest and most discoverable
  • Header versioning keeps URLs clean but is harder to test in a browser
  • Always communicate deprecation schedules to consumers

Exercises

  1. Implement URI Versioning

    Create a FastAPI application with two versioned endpoints: /v1/products and /v2/products. The v2 endpoint should include additional fields (category, rating) compared to v1.

    Hint

    Use separate route decorators for each version. Consider using a shared data layer so both versions query the same data but serialize differently.

  2. Header-Based Version Router

    Build middleware that reads an X-API-Version header and routes requests to different handler functions based on the version number. Return a 400 error for unsupported versions.

    Hint

    Use FastAPI's dependency injection with Header() to extract the version, then dispatch to version-specific logic using a dictionary mapping.

  3. Version Deprecation Warning

    Add a response header 'Sunset' with an RFC 7231 date and a 'Deprecation' header to all v1 endpoints, warning clients that v1 will be retired.

    Hint

    Use a FastAPI middleware or a Response dependency to inject the Sunset and Deprecation headers into every v1 response.

Common Problems & Solutions

1 Clients break when deploying a new version
Symptoms: ['400/500 errors from client apps after deployment', 'Deserialization failures in mobile/frontend apps']
Cause: Breaking changes were introduced in the same URL path without versioning, so existing clients receive unexpected response shapes.
Solution: Always version your API before introducing breaking changes. Keep the old version running alongside the new one until clients migrate.
2 Maintaining too many API versions becomes unmanageable
Cause: No deprecation policy was established, so every version ever released is still supported.
Solution: Define a deprecation policy (e.g., support N-1 versions). Communicate sunset dates via headers and documentation. Monitor version usage to know when it is safe to remove old versions.
