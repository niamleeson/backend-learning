Logging: Basics to Production Setup

Intuition

Logging gives you a severity-based, configurable output system that can be tuned per-module without changing code — far superior to scattered print statements.

Python's logging module provides a flexible framework for emitting log messages from your application, replacing ad-hoc print() calls with structured, leveled output.

import logging

# Basic configuration
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)

logger = logging.getLogger(__name__)

logger.debug("Detailed trace info")     # hidden at INFO level
logger.info("Server started on port 8080")
logger.warning("Disk usage above 80%%")
logger.error("Failed to connect to DB")
logger.exception("Unhandled error")     # includes traceback
  • Use getLogger(__name__) to create per-module loggers; this follows the module hierarchy.
  • Levels: DEBUG < INFO < WARNING < ERROR < CRITICAL. Set the threshold at the root.
  • logger.exception() automatically attaches the current traceback — use it inside except blocks.
  • In production, add FileHandler or RotatingFileHandler instead of logging to stdout.

Exercises

  1. Multi-Handler Setup

    Configure a logger that writes DEBUG+ to a file and WARNING+ to the console using two different handlers with different formatters.

    Hint

    Create a FileHandler and a StreamHandler, set levels on each, and add both to the logger.

  2. Structured Log Context

    Write a function that logs a user action including user_id and action as structured fields using the extra parameter.

    Hint

    Pass extra={'user_id': uid, 'action': act} and include %(user_id)s in the format string.

Common Problems & Solutions

1 Calling `logging.basicConfig()` multiple times has no effect after the first call.
Cause: `basicConfig` is a no-op if the root logger already has handlers configured.
Solution: Call `basicConfig` once at startup, or pass `force=True` (Python 3.8+) to reconfigure.
2 Log messages appear twice in the console.
Cause: Both the root logger and a child logger have handlers attached, causing duplicate output.
Solution: Add handlers only to the root logger, or set `propagate = False` on child loggers.
