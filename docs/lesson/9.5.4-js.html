Design a Real-Time Collaborative Editor (Google Docs)

Intuition

Imagine two people writing on the same whiteboard from different rooms, each with a camera feed of the board. With OT, a coordinator in the middle watches both cameras and adjusts each person's instructions so they do not overwrite each other — 'You wrote at position 5, but the other person already inserted text at position 3, so your position is actually 6 now.' With CRDTs, each person's marker has a unique invisible ID embedded in every letter, so the whiteboard itself knows how to merge any combination of edits without a coordinator. CRDTs are like self-organizing text.

Design a Real-Time Collaborative Editor (Google Docs)

Introduction

A real-time collaborative editor allows multiple users to simultaneously edit the same document with changes appearing instantly for all participants. Google Docs serves over 1 billion users and handles millions of concurrent editing sessions. This is widely considered one of the hardest system design problems because it demands conflict-free concurrent editing, real-time synchronization, rich text formatting, offline support, and version history — all while maintaining a responsive user experience.

This problem tests your deep understanding of distributed systems concurrency, CRDTs vs Operational Transformation, real-time communication patterns, and the challenges of building a system where consistency and responsiveness must coexist.

Requirements

Functional Requirements

  1. Create and manage documents — Create, rename, delete, share documents
  2. Rich text editing — Bold, italic, headings, lists, tables, images, links
  3. Real-time collaboration — Multiple users edit simultaneously; changes appear within 100ms
  4. Cursor and selection presence — See other users' cursors, selections, and names in real-time
  5. Comments and suggestions — Add comments anchored to text ranges; suggest edits
  6. Version history — View all changes with timestamps and authors; restore any version
  7. Offline editing — Continue editing offline; sync when reconnected
  8. Access control — Owner, editor, commenter, viewer permission levels
  9. Conflict resolution — Concurrent edits never cause data loss or corruption

Non-Functional Requirements

  1. Real-time latency — Edits visible to collaborators within 100-200ms
  2. Consistency — All users converge to the same document state (eventual consistency)
  3. Durability — Zero data loss; every keystroke must eventually persist
  4. Scalability — Support 50M concurrent documents, up to 100 editors per document
  5. Availability — 99.99% uptime

Capacity Estimation

Total users:                    1 billion
Daily active users:             300 million
Concurrent editing sessions:    50 million documents
Avg collaborators per document: 2-3 (up to 100)
Concurrent WebSocket connections: 80 million

Operations:
  - Keystrokes per active user: 3/sec average
  - Total operations: 80M * 3 = 240M operations/sec
  - After batching (50ms windows): 80M * 1 batch/50ms = 80M batched ops/sec
  - With compression: ~40M messages/sec over WebSockets

Storage:
  - Avg document size: 50KB (text + formatting)
  - Total documents: 5 billion
  - Document storage: 5B * 50KB = 250 TB
  - Version history (ops log): ~10x document size = 2.5 PB

Bandwidth:
  - Avg operation message: 100 bytes
  - Inbound: 240M * 100B = 24 GB/sec
  - Outbound (fan-out to collaborators): 24GB * 2 avg = 48 GB/sec
  - Total: ~72 GB/sec

WebSocket servers:
  - Each server handles ~50K connections
  - Servers needed: 80M / 50K = 1,600 WebSocket servers

Core Implementation: Operational Transformation vs CRDT

Approach 1: Operational Transformation (OT)

OT is the approach Google Docs uses. A central server transforms concurrent operations against each other to maintain consistency.

// Operational Transformation engine
class OTServer {
  constructor(documentId) {
    this.documentId = documentId;
    this.version = 0;
    this.operations = [];  // operation history
    this.document = '';     // current document state
  }

  // Receive an operation from a client
  receiveOperation(clientOp, clientVersion) {
    // Transform against all operations that happened since client's version
    let transformedOp = clientOp;
    for (let i = clientVersion; i < this.version; i++) {
      transformedOp = this.transform(transformedOp, this.operations[i]);
    }

    // Apply to server document
    this.document = this.apply(this.document, transformedOp);
    this.operations.push(transformedOp);
    this.version++;

    return { op: transformedOp, version: this.version };
  }

  // Transform operation A against operation B
  // When two users type at different positions simultaneously
  transform(opA, opB) {
    if (opA.type === 'insert' && opB.type === 'insert') {
      if (opA.position <= opB.position) {
        return opA; // A is before B, no change needed
      } else {
        // A is after B, shift A's position by B's length
        return { ...opA, position: opA.position + opB.text.length };
      }
    }

    if (opA.type === 'insert' && opB.type === 'delete') {
      if (opA.position <= opB.position) {
        return opA;
      } else if (opA.position > opB.position + opB.length) {
        return { ...opA, position: opA.position - opB.length };
      } else {
        return { ...opA, position: opB.position };
      }
    }

    if (opA.type === 'delete' && opB.type === 'insert') {
      if (opA.position >= opB.position) {
        return { ...opA, position: opA.position + opB.text.length };
      }
      return opA;
    }

    if (opA.type === 'delete' && opB.type === 'delete') {
      if (opA.position >= opB.position + opB.length) {
        return { ...opA, position: opA.position - opB.length };
      }
      if (opA.position + opA.length <= opB.position) {
        return opA;
      }
      // Overlapping deletes — complex case
      const start = Math.max(opA.position, opB.position);
      const end = Math.min(opA.position + opA.length, opB.position + opB.length);
      const overlap = Math.max(0, end - start);
      return {
        ...opA,
        position: Math.min(opA.position, opB.position),
        length: opA.length - overlap,
      };
    }

    return opA;
  }

  apply(doc, op) {
    if (op.type === 'insert') {
      return doc.slice(0, op.position) + op.text + doc.slice(op.position);
    }
    if (op.type === 'delete') {
      return doc.slice(0, op.position) + doc.slice(op.position + op.length);
    }
    return doc;
  }
}

Approach 2: CRDT with Yjs (Modern Approach)

import * as Y from 'yjs';
import { WebSocketServer } from 'ws';
import Redis from 'ioredis';

const pub = new Redis();
const sub = new Redis();

class CollaborationService {
  constructor() {
    this.documents = new Map(); // docId -> Y.Doc
    this.clients = new Map();    // docId -> Set<WebSocket>
  }

  // Initialize or load a document
  async getDocument(docId) {
    if (this.documents.has(docId)) {
      return this.documents.get(docId);
    }

    const ydoc = new Y.Doc();

    // Load latest snapshot from storage
    const snapshot = await this.loadLatestSnapshot(docId);
    if (snapshot) {
      Y.applyUpdate(ydoc, new Uint8Array(snapshot.content));
    }

    // Apply any operations after the snapshot
    const ops = await this.loadOperationsSince(docId, snapshot?.version || 0);
    for (const op of ops) {
      Y.applyUpdate(ydoc, new Uint8Array(op.op_data));
    }

    // Listen for local changes to persist
    ydoc.on('update', (update, origin) => {
      this.persistOperation(docId, update);
      this.broadcastToOtherServers(docId, update, origin);
    });

    this.documents.set(docId, ydoc);
    return ydoc;
  }

  // Handle new WebSocket connection
  async handleConnection(ws, docId, userId) {
    const ydoc = await this.getDocument(docId);

    // Add to client set
    if (!this.clients.has(docId)) {
      this.clients.set(docId, new Set());
    }
    this.clients.get(docId).add(ws);

    // Send current document state
    const stateVector = Y.encodeStateAsUpdate(ydoc);
    ws.send(JSON.stringify({
      type: 'sync',
      data: Array.from(stateVector),
      version: ydoc.clientID,
    }));

    // Send current presence info
    const presence = await this.getPresence(docId);
    ws.send(JSON.stringify({ type: 'presence', users: presence }));

    // Add this user to presence
    await this.addPresence(docId, userId);

    // Handle incoming messages
    ws.on('message', async (raw) => {
      const message = JSON.parse(raw);

      switch (message.type) {
        case 'update': {
          // Apply CRDT update from this client
          const update = new Uint8Array(message.data);
          Y.applyUpdate(ydoc, update, ws);

          // Broadcast to other local clients
          this.broadcastToLocal(docId, ws, message);
          break;
        }
        case 'cursor': {
          // Broadcast cursor position
          this.broadcastToLocal(docId, ws, {
            type: 'cursor',
            user_id: userId,
            ...message,
          });
          await pub.publish(`doc:${docId}:cursor`, JSON.stringify({
            user_id: userId,
            ...message,
          }));
          break;
        }
      }
    });

    ws.on('close', async () => {
      this.clients.get(docId)?.delete(ws);
      await this.removePresence(docId, userId);

      // If no more clients for this doc, schedule cleanup
      if (this.clients.get(docId)?.size === 0) {
        setTimeout(() => this.maybeUnloadDocument(docId), 30000);
      }
    });
  }

  // Broadcast to clients on THIS server
  broadcastToLocal(docId, sender, message) {
    const clients = this.clients.get(docId);
    if (!clients) return;

    const data = JSON.stringify(message);
    for (const client of clients) {
      if (client !== sender && client.readyState === 1) {
        client.send(data);
      }
    }
  }

  // Broadcast to collaboration servers via Redis Pub/Sub
  async broadcastToOtherServers(docId, update, origin) {
    if (origin === 'remote') return; // Don't re-broadcast remote updates
    await pub.publish(`doc:${docId}:update`, JSON.stringify({
      data: Array.from(update),
      server_id: process.env.SERVER_ID,
    }));
  }

  // Persist operation to append-only log
  async persistOperation(docId, update) {
    await prisma.operations.create({
      data: {
        document_id: docId,
        version: await this.getNextVersion(docId),
        author_id: 0, // resolved from WebSocket context
        op_type: 'crdt_update',
        op_data: Buffer.from(update),
      },
    });
  }

  // Periodically snapshot the document
  async createSnapshot(docId) {
    const ydoc = this.documents.get(docId);
    if (!ydoc) return;

    const state = Y.encodeStateAsUpdate(ydoc);
    await prisma.document_snapshots.create({
      data: {
        document_id: docId,
        version: await this.getCurrentVersion(docId),
        content: Buffer.from(state),
        content_type: 'crdt-yjs',
        size_bytes: state.length,
      },
    });
  }

  // Presence management via Redis
  async addPresence(docId, userId) {
    await redis.sadd(`presence:${docId}`, userId.toString());
    await redis.expire(`presence:${docId}`, 300);
  }

  async removePresence(docId, userId) {
    await redis.srem(`presence:${docId}`, userId.toString());
  }

  async getPresence(docId) {
    return redis.smembers(`presence:${docId}`);
  }
}

Offline Support

// Client-side: offline queue using IndexedDB
class OfflineManager {
  constructor(ydoc, docId) {
    this.ydoc = ydoc;
    this.docId = docId;
    this.pendingUpdates = [];
    this.isOnline = navigator.onLine;

    // Buffer updates when offline
    ydoc.on('update', (update) => {
      if (!this.isOnline) {
        this.pendingUpdates.push(update);
        this.saveToIndexedDB(update);
      }
    });

    window.addEventListener('online', () => this.sync());
    window.addEventListener('offline', () => { this.isOnline = false; });
  }

  async sync() {
    this.isOnline = true;
    // Send all buffered updates
    for (const update of this.pendingUpdates) {
      websocket.send(JSON.stringify({
        type: 'update',
        data: Array.from(update),
      }));
    }
    this.pendingUpdates = [];
    await this.clearIndexedDB();
  }
}

Snapshot every 100 operations to bound reconstruction time.
Old operations can be archived to cold storage after snapshot.

Comment Anchoring

One tricky aspect is keeping comments anchored to the right text as the document changes:

// Comment anchors use CRDT-relative positions
// instead of absolute character offsets
class CommentAnchor {
  // Convert absolute offset to CRDT-relative position
  static fromOffset(ydoc, offset) {
    const ytext = ydoc.getText('content');
    // Y.createRelativePositionFromTypeIndex creates a position
    // that moves with the text as edits happen around it
    return Y.createRelativePositionFromTypeIndex(ytext, offset);
  }

  // Convert CRDT-relative position back to absolute offset
  static toOffset(ydoc, relativePosition) {
    const absPos = Y.createAbsolutePositionFromRelativePosition(
      relativePosition,
      ydoc
    );
    return absPos ? absPos.index : null; // null if text was deleted
  }
}

Summary

A real-time collaborative editor is built on three pillars: (1) a concurrency control mechanism (OT or CRDT) that ensures all users converge to the same document state regardless of the order operations arrive, (2) a WebSocket-based real-time transport layer that delivers operations to collaborators within 100-200ms, and (3) a persistence layer using an append-only operations log with periodic snapshots for fast document loading. CRDT-based approaches (Yjs) are increasingly preferred over OT for new systems because they handle offline editing natively and do not require a central transformation server. Documents should be sharded across collaboration servers using consistent hashing on document ID, with Redis Pub/Sub bridging cross-server communication. Comment anchoring requires CRDT-relative positions that survive concurrent edits. The system must handle 240M+ operations per second through aggressive batching, compression, and efficient binary encoding of updates.

    Use Y.XmlFragment for rich text. Bind to Quill with y-quill. The y-websocket package provides a ready-made WebSocket server. Snapshot with Y.encodeStateAsUpdate() every 30 seconds.

  3. Operations Log with Snapshot Recovery

    Implement a document persistence system that stores every operation in an append-only PostgreSQL table and creates snapshots every 100 operations. Build a recovery function that loads the latest snapshot and replays subsequent operations to reconstruct the document.

    Hint

    Use a BIGSERIAL version column. Query: SELECT * FROM operations WHERE document_id = $1 AND version > $2 ORDER BY version ASC. Measure reconstruction time with and without snapshots.

Common Problems & Solutions

1 Document takes 10+ seconds to load because the operations log has millions of entries
Cause: Without snapshots, loading a document requires replaying every operation since creation. A heavily-edited document can have millions of operations.
Solution: Implement periodic snapshotting — every 100-500 operations, serialize the current document state and store it. On load, fetch the latest snapshot and replay only the operations since that snapshot. This bounds loading time to O(snapshot_interval) operations regardless of total document history. Archive old operations to cold storage.
2 Cursor positions jump around when multiple users are typing near each other
Cause: Cursor positions are stored as absolute character offsets. When another user inserts text before the cursor, the offset becomes stale and points to the wrong position.
Solution: Use CRDT-relative positions for cursor tracking instead of absolute offsets. In Yjs, use Y.createRelativePositionFromTypeIndex() which creates a position that is stable relative to the surrounding characters, not the absolute offset. This position automatically adjusts as text is inserted or deleted around it.
3 WebSocket server crashes when a viral document gets 500+ simultaneous editors
Cause: Each editor generates ~3 operations/second. With 500 editors, that is 1,500 ops/sec inbound, each of which must be broadcast to 499 other clients — resulting in ~750,000 outbound messages/second on a single server.
Solution: Implement operation batching: collect operations in a 50ms window and send them as a single batch. This reduces message count by 10-50x. For extremely hot documents, implement a fan-out tree: the primary server sends updates to 5 relay servers, each of which handles 100 clients. Rate-limit cursor updates to 5/second per user. Consider read-only mode for viewers to eliminate their operation overhead.
