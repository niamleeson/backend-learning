Full-Text Search in PostgreSQL

Intuition

Full-text search is like the difference between scanning every page of every book in a library (LIKE '%word%') and using the library's catalog system that has already indexed every word, removed filler words, and knows the root form of every term. The catalog (tsvector) takes space to maintain, but finding things is orders of magnitude faster.

Full-Text Search in PostgreSQL

Introduction

Full-text search (FTS) in PostgreSQL lets you search natural-language text efficiently without an external search engine like Elasticsearch. For many applications, PostgreSQL's built-in FTS is more than sufficient and eliminates the complexity of maintaining a separate search infrastructure.

Core Concepts

Document -> tsvector: Pre-processed searchable representation
Query   -> tsquery:  Parsed search query

tsvector: 'quick' 'brown' 'fox' 'jump' 'lazi' 'dog'
           (stemmed, lowercased, stop words removed)

tsquery:  'quick & fox'  (search for both words)

tsvector and tsquery

-- Convert text to tsvector
SELECT to_tsvector('english', 'The quick brown fox jumps over the lazy dog');
-- Result: 'brown':3 'dog':9 'fox':4 'jump':5 'lazi':8 'quick':2
-- Note: "the" and "over" removed (stop words), "jumps" -> "jump" (stemming)

-- Convert search to tsquery
SELECT to_tsquery('english', 'quick & fox');
-- Result: 'quick' & 'fox'

-- Matching
SELECT to_tsvector('english', 'The quick brown fox')
    @@ to_tsquery('english', 'quick & fox');
-- Result: true

Basic Full-Text Search

-- Simple search
SELECT title, body
FROM articles
WHERE to_tsvector('english', title || ' ' || body)
   @@ to_tsquery('english', 'postgresql & performance');

async function searchArticles(searchTerm) {
  // plainto_tsquery is safer for user input (handles special chars)
  const result = await pool.query(`
    SELECT
      id,
      title,
      ts_headline('english', body, plainto_tsquery('english', $1),
        'MaxWords=50, MinWords=20, StartSel=<mark>, StopSel=</mark>'
      ) AS snippet,
      ts_rank(
        to_tsvector('english', title || ' ' || body),
        plainto_tsquery('english', $1)
      ) AS rank
    FROM articles
    WHERE to_tsvector('english', title || ' ' || body)
       @@ plainto_tsquery('english', $1)
    ORDER BY rank DESC
    LIMIT 20
  `, [searchTerm]);
  return result.rows;
}

Search Query Types

-- to_tsquery: strict syntax, supports operators
to_tsquery('english', 'cat & dog')           -- AND
to_tsquery('english', 'cat | dog')           -- OR
to_tsquery('english', '!cat')                -- NOT
to_tsquery('english', 'cat <-> dog')         -- FOLLOWED BY (phrase)
to_tsquery('english', 'super <2> hero')      -- within 2 words

-- plainto_tsquery: simple, user-friendly (treats all words as AND)
plainto_tsquery('english', 'quick brown fox')
-- Result: 'quick' & 'brown' & 'fox'

-- phraseto_tsquery: phrase matching
phraseto_tsquery('english', 'quick brown fox')
-- Result: 'quick' <-> 'brown' <-> 'fox'

-- websearch_to_tsquery: Google-like syntax (PG 11+)
websearch_to_tsquery('english', '"quick fox" -lazy')
-- Result: 'quick' <-> 'fox' & !'lazi'

async function advancedSearch(query) {
  // websearch_to_tsquery handles Google-like syntax safely
  const result = await pool.query(`
    SELECT id, title,
      ts_rank_cd(search_vector, websearch_to_tsquery('english', $1)) AS rank
    FROM articles
    WHERE search_vector @@ websearch_to_tsquery('english', $1)
    ORDER BY rank DESC
    LIMIT 20
  `, [query]);
  return result.rows;
}

// Users can search like:
// "javascript tutorial"       -> phrase match
// javascript -beginner        -> exclude beginner
// javascript OR typescript    -> either term

Stored tsvector Column (Performance)

Recomputing tsvector on every query is expensive. Store it as a column:

-- Add a generated tsvector column
ALTER TABLE articles ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body, '')), 'B')
  ) STORED;

-- Create a GIN index on it
CREATE INDEX idx_articles_search ON articles USING gin (search_vector);

Weighted Search

Weights allow you to rank title matches higher than body matches:

Weights:
A = 1.0  (highest - title)
B = 0.4  (body)
C = 0.2  (metadata)
D = 0.1  (lowest - comments)

async function weightedSearch(query) {
  const result = await pool.query(`
    SELECT
      id,
      title,
      ts_rank(search_vector, websearch_to_tsquery('english', $1)) AS rank
    FROM articles
    WHERE search_vector @@ websearch_to_tsquery('english', $1)
    ORDER BY rank DESC
    LIMIT 20
  `, [query]);
  return result.rows;
}

Highlighting Results

async function searchWithHighlights(query) {
  const result = await pool.query(`
    SELECT
      id,
      ts_headline('english', title, websearch_to_tsquery('english', $1),
        'StartSel=<b>, StopSel=</b>, MaxFragments=0'
      ) AS highlighted_title,
      ts_headline('english', body, websearch_to_tsquery('english', $1),
        'StartSel=<b>, StopSel=</b>, MaxWords=60, MinWords=30, MaxFragments=3, FragmentDelimiter= ... '
      ) AS highlighted_body,
      ts_rank(search_vector, websearch_to_tsquery('english', $1)) AS rank
    FROM articles
    WHERE search_vector @@ websearch_to_tsquery('english', $1)
    ORDER BY rank DESC
    LIMIT 20
  `, [query]);
  return result.rows;
}

Autocomplete / Prefix Search

-- Prefix matching with :*
SELECT title FROM articles
WHERE search_vector @@ to_tsquery('english', 'postg:*');
-- Matches: PostgreSQL, Postgres, postgraduate...

-- For fast autocomplete, use trigram index
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_articles_title_trgm ON articles USING gin (title gin_trgm_ops);

SELECT title FROM articles
WHERE title ILIKE '%postg%'
ORDER BY similarity(title, 'postg') DESC
LIMIT 10;

async function autocomplete(prefix) {
  const result = await pool.query(`
    SELECT DISTINCT title
    FROM articles
    WHERE title ILIKE $1
    ORDER BY similarity(title, $2) DESC
    LIMIT 10
  `, [`%${prefix}%`, prefix]);
  return result.rows.map(r => r.title);
}

When to Use PostgreSQL FTS vs Elasticsearch

PostgreSQL FTS is great when:
- Search is a feature, not the product
- Dataset < 10M documents
- You want transactional consistency
- Simpler ops (no separate cluster)
- Budget/team constraints

Elasticsearch/Meilisearch when:
- Search IS the product
- Need faceted search, fuzzy matching, "did you mean?"
- Dataset > 10M documents with complex queries
- Need real-time indexing at high write volume
- Need multi-language analysis

Common Problems & Solutions

1 Full-text search returns no results for partial words
Cause: tsvector stores stemmed complete words, not substrings. 'post' won't match 'postgresql' with standard tsquery.
Solution: Use prefix matching with :* operator: `to_tsquery('post:*')`. For substring matching, use pg_trgm extension with ILIKE and GIN trigram index.
2 Search is slow on large tables
Cause: Computing to_tsvector on every query, or missing GIN index on the tsvector column.
Solution: Store the tsvector as a generated column and create a GIN index on it. This precomputes the search vector and enables fast index lookups.
3 Stop words like 'the', 'a', 'is' in search queries cause empty results
Cause: Stop words are removed by the text search configuration. Searching for only stop words produces an empty tsquery.
Solution: Use the 'simple' configuration for exact matching: `to_tsvector('simple', text)`. Or combine with the 'english' config for best of both worlds.
