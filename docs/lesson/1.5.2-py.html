Built-in Exception Hierarchy

Intuition

The exception hierarchy is a class tree. Catching a parent type catches all its children, so choosing the right level in the tree determines how broad or narrow your error handling is.

Python's exception classes form a hierarchy rooted at BaseException. Understanding this tree helps you catch exceptions at the right level of specificity.

# Key hierarchy (simplified)
# BaseException
#  +-- SystemExit
#  +-- KeyboardInterrupt
#  +-- Exception
#       +-- ArithmeticError
#       |    +-- ZeroDivisionError
#       |    +-- OverflowError
#       +-- LookupError
#       |    +-- IndexError
#       |    +-- KeyError
#       +-- OSError
#       |    +-- FileNotFoundError
#       |    +-- PermissionError
#       +-- ValueError
#       +-- TypeError
#       +-- AttributeError
#       +-- StopIteration

# Catching a parent catches all children
try:
    d = {}
    d["missing"]
except LookupError as e:
    print(type(e).__name__)  # KeyError
  • BaseException is the root; Exception is the root for all "normal" errors.
  • SystemExit and KeyboardInterrupt inherit from BaseException, not Exception — so except Exception won't catch them.
  • Catch the most specific exception possible to avoid masking bugs.
  • Use except (ExcA, ExcB) to handle multiple exception types with the same handler.

Exercises

  1. Exception Ancestry

    Write a function exception_chain(exc_class) that returns a list of class names from the given exception class up to BaseException using __mro__.

    Hint

    Use exc_class.__mro__ and filter for subclasses of BaseException.

  2. Selective Catching

    Write code that handles FileNotFoundError differently from other OSError subclasses. Print 'missing' for file-not-found and 'os error' for everything else.

    Hint

    Put the more specific except FileNotFoundError before except OSError.

Common Problems & Solutions

1 Catching `Exception` accidentally suppresses `StopIteration`, breaking iterators.
Cause: `StopIteration` inherits from `Exception`, so a broad except can swallow it.
Solution: Avoid wrapping iterator code in broad `except Exception` blocks, or explicitly re-raise `StopIteration`.
2 Order of except clauses matters: a parent class before a child makes the child unreachable.
Cause: Python matches except clauses top-to-bottom; the first match wins.
Solution: Always put more specific (child) exception handlers before broader (parent) ones.
