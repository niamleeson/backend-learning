JOINs: INNER, LEFT, RIGHT, FULL, CROSS

Intuition

JOINs are like matching puzzle pieces. INNER JOIN only keeps pieces that connect perfectly. LEFT JOIN keeps all pieces from the left side, even if nothing connects to them (you just see empty gaps). CROSS JOIN throws all pieces into a blender and creates every possible pairing.

JOINs: INNER, LEFT, RIGHT, FULL, CROSS

Introduction

Real-world data is spread across multiple tables. A users table links to an orders table, which links to order_items, which links to products. JOINs let you combine rows from two or more tables based on a related column, and they are the single most important SQL concept for backend developers to master.

Why Normalize Data?

Instead of one giant table:

| order_id | user_name | user_email         | product_name | price |
|----------|-----------|--------------------| -------------|-------|
| 1        | Alice     | alice@example.com  | Widget       | 9.99  |
| 2        | Alice     | alice@example.com  | Gadget       | 19.99 |

We split into separate tables:

users:          orders:              products:
+----+-------+  +----+---------+---+  +----+--------+-------+
| id | name  |  | id | user_id |...|  | id | name   | price |
+----+-------+  +----+---------+---+  +----+--------+-------+
| 1  | Alice |  | 1  | 1       |   |  | 1  | Widget | 9.99  |
| 2  | Bob   |  | 2  | 1       |   |  | 2  | Gadget | 19.99 |
+----+-------+  +----+---------+---+  +----+--------+-------+

This avoids data duplication and inconsistency. JOINs reassemble this data when needed.

INNER JOIN

Returns only rows that have matching values in both tables:

SELECT users.name, orders.id AS order_id, orders.total
FROM users
INNER JOIN orders ON users.id = orders.user_id;

Venn Diagram:

  users         orders
  +-----+     +-----+
  |     |XXXXX|     |
  |     |XXXXX|     |
  +-----+     +-----+
        ^^^^^^^
     Only matching
     rows returned

In Node.js:

async function getUserOrders(userId) {
  const result = await pool.query(`
    SELECT
      u.name AS user_name,
      o.id AS order_id,
      o.total,
      o.created_at
    FROM users u
    INNER JOIN orders o ON u.id = o.user_id
    WHERE u.id = $1
    ORDER BY o.created_at DESC
  `, [userId]);
  return result.rows;
}

Key behavior: If a user has no orders, they won't appear in the results at all. If an order has no matching user (orphaned), it also won't appear.

LEFT JOIN (LEFT OUTER JOIN)

Returns all rows from the left table, and matched rows from the right table. If no match, right columns are NULL:

SELECT u.name, o.id AS order_id, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

Result:
| name  | order_id | total |
|-------|----------|-------|
| Alice | 1        | 29.99 |
| Alice | 2        | 49.99 |
| Bob   | NULL     | NULL  |   <-- Bob has no orders

Use case: "Show me all users and their orders, including users who haven't ordered anything."

Finding Rows Without Matches

A powerful pattern — find users with NO orders:

SELECT u.name, u.email
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;

This is often faster than using NOT IN or NOT EXISTS subqueries.

RIGHT JOIN (RIGHT OUTER JOIN)

Returns all rows from the right table, and matched rows from the left. Functionally, it's a LEFT JOIN with the tables swapped:

-- These two are equivalent:
SELECT * FROM users u RIGHT JOIN orders o ON u.id = o.user_id;
SELECT * FROM orders o LEFT JOIN users u ON u.id = o.user_id;

In practice, most developers just reorder their tables and use LEFT JOIN. RIGHT JOIN is rarely used.

FULL OUTER JOIN

Returns all rows from both tables. Where there's no match, NULLs fill in:

SELECT u.name, o.id AS order_id
FROM users u
FULL OUTER JOIN orders o ON u.id = o.user_id;

| name  | order_id |
|-------|----------|
| Alice | 1        |
| Alice | 2        |
| Bob   | NULL     |  <-- user with no orders
| NULL  | 5        |  <-- orphaned order

Use case: Data integrity checks, finding orphaned records.

CROSS JOIN

Produces the Cartesian product — every row from the left table paired with every row from the right:

SELECT colors.name AS color, sizes.name AS size
FROM colors
CROSS JOIN sizes;

If colors has 3 rows and sizes has 4 rows, the result has 12 rows. Use sparingly!

Use case: Generating all combinations (e.g., product variants, schedules).

Multi-Table JOINs

You can chain JOINs:

async function getOrderDetails(orderId) {
  const result = await pool.query(`
    SELECT
      o.id AS order_id,
      u.name AS customer,
      p.name AS product,
      oi.quantity,
      oi.unit_price,
      oi.quantity * oi.unit_price AS line_total
    FROM orders o
    INNER JOIN users u ON o.user_id = u.id
    INNER JOIN order_items oi ON o.id = oi.order_id
    INNER JOIN products p ON oi.product_id = p.id
    WHERE o.id = $1
  `, [orderId]);
  return result.rows;
}

Join Flow:

  orders ---> users        (get customer name)
    |
    +-------> order_items   (get line items)
                  |
                  +-------> products  (get product names)

Self JOIN

Join a table to itself — useful for hierarchical data:

-- Employees and their managers
SELECT
  e.name AS employee,
  m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

JOIN Performance Tips

  1. Always JOIN on indexed columns. Foreign keys should have indexes.
  2. Filter early: Put conditions in the WHERE clause (or the ON clause for outer joins) to reduce rows before joining.
  3. Be careful with LEFT JOIN + WHERE on the right table: This effectively converts it to an INNER JOIN.
-- This is WRONG if you want a true LEFT JOIN:
SELECT u.*, o.*
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.status = 'active';  -- filters out NULLs, making it an INNER JOIN!

-- Correct: put the condition in the ON clause
SELECT u.*, o.*
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'active';

Real-World Example: Dashboard Query

async function getDashboardData() {
  const result = await pool.query(`
    SELECT
      u.id,
      u.name,
      u.email,
      COUNT(o.id) AS total_orders,
      COALESCE(SUM(o.total), 0) AS total_spent,
      MAX(o.created_at) AS last_order_date
    FROM users u
    LEFT JOIN orders o ON u.id = o.user_id
    GROUP BY u.id, u.name, u.email
    ORDER BY total_spent DESC
    LIMIT 50
  `);
  return result.rows;
}

This combines LEFT JOIN with aggregation to build a user summary — even users with zero orders appear (thanks to LEFT JOIN and COALESCE).

Exercises

  1. Multi-Table Report

    Given tables users, orders, order_items, and products, write a query that returns each user's name, total number of distinct products they've purchased, and their total spending. Include users who haven't placed any orders.

    Hint

    Use LEFT JOIN to chain all four tables. Use COUNT(DISTINCT product_id) for unique products, COALESCE for handling NULLs, and GROUP BY on user columns.

  2. Find Orphaned Records

    Write queries to find: (a) orders that reference non-existent users, (b) products that have never been ordered. Implement both as Node.js functions.

    Hint

    Use LEFT JOIN with IS NULL check on the joined table's primary key.

  3. Self-JOIN Hierarchy

    Given a categories table with columns id, name, parent_id, write a query to display each category alongside its parent category name. Then extend it to show the full path (grandparent > parent > child) for up to 3 levels.

    Hint

    Use multiple LEFT JOINs on the same table with different aliases: LEFT JOIN categories parent ON c.parent_id = parent.id LEFT JOIN categories grandparent ON parent.parent_id = grandparent.id.

  4. JOIN Type Comparison

    Write 4 versions of a query joining students and enrollments: INNER, LEFT, RIGHT, and FULL OUTER. For each, explain in a comment which students/enrollments would be missing and why.

    Hint

    INNER: only enrolled students. LEFT: all students including unenrolled. RIGHT: all enrollments including those with deleted students. FULL: everything from both sides.

Common Problems & Solutions

1 LEFT JOIN behaves like INNER JOIN — rows with no match are missing
Cause: Adding a WHERE condition on the right table's columns filters out NULL rows. `WHERE orders.status = 'active'` removes rows where orders is NULL.
Solution: Move the filter condition into the ON clause: `LEFT JOIN orders ON users.id = orders.user_id AND orders.status = 'active'`. This preserves unmatched left rows.

-- Wrong:\nSELECT * FROM users u LEFT JOIN orders o ON u.id = o.user_id WHERE o.status = 'active';\n-- Right:\nSELECT * FROM users u LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'active';

2 Query returns duplicate rows or inflated counts
Cause: Joining tables with a one-to-many relationship creates row multiplication. If a user has 3 orders, they appear 3 times.
Solution: Use GROUP BY with aggregate functions (COUNT, SUM) or use DISTINCT. For counts, be careful: `COUNT(*)` counts the multiplied rows, `COUNT(DISTINCT o.id)` counts unique orders.
3 CROSS JOIN produces millions of rows and crashes the application
Cause: CROSS JOIN creates the Cartesian product (rows_a * rows_b). Two tables of 10,000 rows each produce 100 million rows.
Solution: Only use CROSS JOIN on small tables or with LIMIT. For accidental cross joins, ensure you have proper ON conditions in your JOINs.
