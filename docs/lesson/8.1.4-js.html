CDNs, DNS, and Edge Computing

Intuition

Think of a CDN like a chain of local libraries. Instead of everyone driving to the main library downtown (origin server), each neighborhood has a branch (edge node) with copies of popular books (cached content). DNS is the phone book that tells you which branch is closest.

CDNs, DNS, and Edge Computing

Introduction

Before a request even reaches your Node.js server, it passes through layers of internet infrastructure: DNS resolves the domain, CDNs serve cached content, and edge nodes can run code close to users. Understanding these layers lets you dramatically reduce latency and server load.

Content Delivery Networks (CDNs)

A CDN is a geographically distributed network of servers that caches and serves content from locations close to users.

Without CDN:                        With CDN:

User (Tokyo)                        User (Tokyo)
    │                                   │
    │  200ms round trip                 │  20ms
    │                                   ▼
    ▼                              CDN Edge (Tokyo)
Server (US-East)                   Cache HIT? ──> Return cached
                                   Cache MISS?
                                        │  200ms (one time)
                                        ▼
                                   Origin Server (US-East)

What CDNs Cache

┌─────────────────────────────────────────────────────────┐
│ Content Type     │ Cacheable? │ TTL Recommendation       │
├──────────────────┼────────────┼──────────────────────────┤
│ Static files     │ Always     │ 1 year (with hash names) │
│ Images, fonts    │ Always     │ 1 year                   │
│ API responses    │ Sometimes  │ Seconds to minutes       │
│ HTML pages       │ Depends    │ Short TTL or stale-while │
│ User-specific    │ Never*     │ Use Vary header          │
│ WebSocket        │ Never      │ Pass through             │
└─────────────────────────────────────────────────────────┘

Setting Cache Headers in Express

import express from 'express';
const app = express();

// Static assets: cache for 1 year
app.use('/static', express.static('public', {
  maxAge: '1y',
  immutable: true,
}));

// API response caching
app.get('/api/products', async (req, res) => {
  const products = await getProducts();

  // Cache at CDN for 60s, browser for 0s
  res.set('Cache-Control', 'public, s-maxage=60, max-age=0');
  // Stale content served while revalidating
  res.set('Cache-Control', 'public, s-maxage=60, stale-while-revalidate=30');

  res.json(products);
});

// Private data: never cache at CDN
app.get('/api/profile', authenticate, async (req, res) => {
  res.set('Cache-Control', 'private, no-store');
  res.json(await getProfile(req.user.id));
});

Cache Invalidation Strategies

Strategy 1: URL Versioning (Best for static assets)
  /static/app.a1b2c3d4.js  (hash in filename)
  New deploy = new hash = new URL = CDN fetches from origin

Strategy 2: TTL-Based (Good for API responses)
  Cache-Control: s-maxage=60
  Content may be stale for up to 60 seconds

Strategy 3: Purge API (For immediate invalidation)
  POST /cdn/purge { urls: ['/api/products'] }
  CloudFront, Fastly, Cloudflare all support purge APIs

Strategy 4: Stale-While-Revalidate (Best of both)
  Cache-Control: s-maxage=60, stale-while-revalidate=300
  Serve stale content immediately, refresh in background

DNS Deep Dive

DNS translates human-readable domains to IP addresses, but it also enables sophisticated traffic routing.

DNS Record Types

┌──────────┬───────────────────────────────────────────────┐
│ Record   │ Purpose                                       │
├──────────┼───────────────────────────────────────────────┤
│ A        │ Domain -> IPv4 address                        │
│ AAAA     │ Domain -> IPv6 address                        │
│ CNAME    │ Domain -> another domain (alias)              │
│ MX       │ Mail server for the domain                    │
│ TXT      │ Arbitrary text (SPF, DKIM, verification)      │
│ NS       │ Nameserver for the domain                     │
│ SRV      │ Service location (host + port)                │
└──────────┴───────────────────────────────────────────────┘

DNS-Based Load Balancing

api.example.com
  │
  ├── A Record: 93.184.216.34  (US-East server)
  ├── A Record: 198.51.100.42  (US-West server)
  └── A Record: 203.0.113.99   (EU server)

DNS round-robin: returns records in rotating order

Geo-DNS Routing

Route users to the closest datacenter:

User in Tokyo     ──> api.example.com ──> 103.5.140.x  (Asia server)
User in London    ──> api.example.com ──> 185.70.40.x  (EU server)
User in New York  ──> api.example.com ──> 54.239.28.x  (US server)

Services like AWS Route 53, Cloudflare DNS, and Google Cloud DNS support geo-routing, latency-based routing, weighted routing, and failover routing.

DNS TTL and Caching

// DNS caching in Node.js
import dns from 'node:dns';
import { Resolver } from 'node:dns/promises';

// Node.js does NOT cache DNS by default
// Use cacheable-lookup for DNS caching
import CacheableLookup from 'cacheable-lookup';

const cacheable = new CacheableLookup();
// Install globally for all HTTP requests
cacheable.install(require('node:http').globalAgent);
cacheable.install(require('node:https').globalAgent);

// Or use with specific requests
import https from 'node:https';
https.get('https://api.example.com/data', {
  lookup: cacheable.lookup,
}, (res) => { /* ... */ });

Edge Computing

Edge computing runs your code at CDN edge locations, closer to users. Instead of just caching static content, you run logic at the edge.

Traditional:
User (Tokyo) ──200ms──> Origin (US-East) ──> Process ──> Respond

Edge Computing:
User (Tokyo) ──20ms──> Edge Worker (Tokyo) ──> Process ──> Respond
                            │
                            │ Only if needed:
                            └──200ms──> Origin (US-East)

What Runs at the Edge

Good for Edge:                    Keep at Origin:
- A/B testing                     - Database writes
- Auth token validation           - Complex business logic
- Request routing                 - Transactions
- Geolocation-based content       - Real-time collaboration
- Rate limiting                   - Heavy computation
- Header manipulation             - Session management
- URL rewrites/redirects
- Bot detection

Edge Functions with Cloudflare Workers

// Cloudflare Worker example
export default {
  async fetch(request, env) {
    const url = new URL(request.url);

    // Geo-based routing at the edge
    const country = request.headers.get('CF-IPCountry');

    if (url.pathname.startsWith('/api/')) {
      // Route API requests to nearest origin
      const origins = {
        US: 'https://us-api.example.com',
        EU: 'https://eu-api.example.com',
        default: 'https://api.example.com',
      };
      const origin = origins[country] || origins.default;
      return fetch(`${origin}${url.pathname}${url.search}`, request);
    }

    // Edge-side A/B testing
    if (url.pathname === '/') {
      const bucket = request.headers.get('CF-Connecting-IP')
        .split('.').reduce((a, b) => a + parseInt(b), 0) % 2;

      return fetch(
        bucket === 0
          ? 'https://origin.example.com/variant-a'
          : 'https://origin.example.com/variant-b',
        request
      );
    }

    return fetch(request);
  },
};

Edge-Side Rate Limiting

// Rate limiting at the edge using Cloudflare Durable Objects
export class RateLimiter {
  constructor(state) {
    this.state = state;
  }

  async fetch(request) {
    const ip = request.headers.get('CF-Connecting-IP');
    const key = `rate:${ip}`;

    let count = (await this.state.storage.get(key)) || 0;
    count++;

    if (count > 100) {
      return new Response('Rate limited', { status: 429 });
    }

    await this.state.storage.put(key, count);
    // Auto-expire after 60 seconds
    this.state.storage.setAlarm(Date.now() + 60000);

    return fetch(request);
  }
}

Putting It All Together

A production request flow with CDN, DNS, and edge:

User types api.example.com/products
    │
    ▼
DNS Resolution (Route 53 - geo-routing)
    │ Returns CDN edge IP closest to user
    ▼
CDN Edge Node (Cloudflare/CloudFront)
    │
    ├── Static asset? ──> Serve from edge cache
    │
    ├── API with cache? ──> Check edge cache
    │       │ HIT  ──> Return cached response
    │       │ MISS ──> Forward to origin
    │
    ├── Edge Worker? ──> Run edge function
    │       │ A/B test, auth check, rate limit
    │       │ May respond without hitting origin
    │
    └── Forward to origin
            │
            ▼
    Load Balancer (ALB/Nginx)
            │
            ▼
    Node.js Application Server
            │
            ▼
    Redis Cache ──> PostgreSQL

Common Pitfalls

  1. Caching authenticated responses at CDN: User A sees User B's data. Always use Cache-Control: private for user-specific data.
  2. Long DNS TTL during migration: Users hit old servers for hours. Lower TTL before migration, restore after.
  3. Not using Vary header: CDN serves wrong content type or language. Use Vary: Accept-Encoding, Accept-Language.
  4. Edge function cold starts: First request to an edge function may be slow. Keep functions small and avoid heavy imports.

Summary

CDNs, DNS, and edge computing form the outer layers of your system architecture. CDNs cache content close to users, DNS routes traffic to the right servers, and edge computing runs logic at the network edge. Together they can reduce latency from hundreds of milliseconds to single-digit milliseconds for many requests, while significantly reducing origin server load.

Exercises

  1. Cache Header Strategy

    Design and implement a cache header strategy for an Express API with: static assets, public product listings, private user profiles, and real-time notifications. Set appropriate Cache-Control headers for each.

    Hint

    Use s-maxage for CDN caching, max-age for browser caching, private for user data, and no-store for real-time data.

  2. DNS Resolution Profiler

    Build a Node.js script that resolves a domain name and measures the time for DNS lookup, TCP connection, TLS handshake, and first byte. Compare with and without DNS caching.

    Hint

    Use the node:perf_hooks module and http.request with socket events: lookup, connect, secureConnect, response.

  3. Edge Function Prototype

    Create a simple HTTP proxy in Node.js that simulates edge behavior: check a local cache first, serve stale content with background refresh, and forward cache misses to an origin server.

    Hint

    Use a Map as the local cache with TTL tracking. On cache hit with expired TTL, return stale data and fetch fresh data in background.

Common Problems & Solutions

1 Users in different regions seeing each other's cached data
Cause: CDN is caching responses without considering the `Vary` header or authentication state. A cached response for User A is served to User B.
Solution: Set `Cache-Control: private, no-store` for authenticated endpoints. Use `Vary: Authorization` if caching per-user at CDN (not recommended). Only cache truly public, non-personalized responses.
2 Stale content persists after deployment
Cause: CDN caches have long TTLs and old content is served until the cache expires.
Solution: Use content-hashed filenames for static assets (Webpack/Vite do this automatically). For API responses, use short TTLs with `stale-while-revalidate`. Use CDN purge APIs for immediate invalidation.
3 DNS changes not propagating
Cause: High TTL on DNS records means resolvers worldwide have cached the old IP address.
Solution: Lower DNS TTL to 60-300 seconds before making changes. Wait for the old TTL to expire before making the DNS change. After propagation, restore the TTL to a higher value (3600+).
