Multi-Stage Builds and Image Optimization

Intuition

Multi-stage builds are like building a house. You need scaffolding, cranes, and power tools during construction (build stage), but once the house is finished, you remove all the construction equipment. The final image is just the house itself (runtime), not the entire construction site. Each stage is a clean room where you pick only what you need for the next stage.

Multi-Stage Builds and Image Optimization

Why Image Size Matters

Every megabyte in your Docker image impacts: pull times (especially in CI/CD), storage costs, startup speed, and attack surface. A typical unoptimized Node.js image can be 1GB+. With proper optimization, you can reduce it to under 150MB.

Multi-Stage Builds In Depth

The Problem

# Single-stage: everything in one image
FROM node:20-alpine

WORKDIR /app
COPY . .
RUN npm ci                    # Installs ALL dependencies (including dev)
RUN npm run build             # TypeScript compiler, needed only at build time

CMD ["node", "dist/server.js"]

# Result: image contains TypeScript, dev tools, test frameworks, etc.
# None of these are needed at runtime!

The Solution: Multi-Stage Builds

# ============================
# Stage 1: Dependencies
# ============================
FROM node:20-alpine AS deps

WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

# ============================
# Stage 2: Build
# ============================
FROM node:20-alpine AS builder

WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .

# Build TypeScript
RUN npm run build

# Generate Prisma client
RUN npx prisma generate

# ============================
# Stage 3: Production dependencies only
# ============================
FROM node:20-alpine AS prod-deps

WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --production

# ============================
# Stage 4: Production runtime
# ============================
FROM node:20-alpine AS runner

RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001 -G nodejs

RUN apk add --no-cache dumb-init

WORKDIR /app

# Copy ONLY what's needed to run
COPY --from=prod-deps --chown=nodejs:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nodejs:nodejs /app/dist ./dist
COPY --from=builder --chown=nodejs:nodejs /app/prisma ./prisma
COPY --chown=nodejs:nodejs package.json ./

ENV NODE_ENV=production
USER nodejs
EXPOSE 3000

ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "dist/server.js"]

Layer Caching Optimization

Order Matters

# SLOW: Any source change invalidates npm ci cache
COPY . .
RUN npm ci
RUN npm run build

# FAST: npm ci only reruns when package files change
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

Using BuildKit Cache Mounts

# syntax=docker/dockerfile:1

FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./

# Cache npm packages between builds
RUN --mount=type=cache,target=/root/.npm \
    npm ci

COPY . .
RUN npm run build

# Enable BuildKit
export DOCKER_BUILDKIT=1
docker build .

# Or use docker buildx
docker buildx build .

Alpine-Specific Optimizations

Handling Native Dependencies

FROM node:20-alpine AS builder

# Install build dependencies for native modules
RUN apk add --no-cache python3 make g++

WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

# In production stage, no build tools needed
FROM node:20-alpine
# Native modules are already compiled in node_modules
COPY --from=builder /app/node_modules ./node_modules

Avoiding Common Alpine Issues

# Problem: some packages need glibc (Alpine uses musl)
# Solution: Install compatibility layer if needed
RUN apk add --no-cache libc6-compat

# Problem: timezone data missing in Alpine
RUN apk add --no-cache tzdata
ENV TZ=UTC

Advanced Optimization Techniques

Removing Unnecessary Files from node_modules

FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --production

# Remove unnecessary files from node_modules
RUN find node_modules -name "*.md" -delete && \
    find node_modules -name "*.map" -delete && \
    find node_modules -name "*.ts" -not -name "*.d.ts" -delete && \
    find node_modules -name "LICENSE*" -delete && \
    find node_modules -name "CHANGELOG*" -delete && \
    rm -rf node_modules/*/{test,tests,__tests__,spec,specs} && \
    rm -rf node_modules/*/{example,examples,doc,docs}

Using node-prune

RUN npm ci --production && \
    npx node-prune          # Removes test files, docs, etc. from node_modules

Distroless Images

Google's distroless images contain only the runtime -- no shell, no package manager, no OS utilities. Maximum security.

FROM node:20-alpine AS builder
WORKDIR /app
COPY . .
RUN npm ci && npm run build && npm ci --production

# Distroless: no shell, no package manager, minimal attack surface
FROM gcr.io/distroless/nodejs20-debian12
COPY --from=builder /app/dist /app/dist
COPY --from=builder /app/node_modules /app/node_modules
COPY --from=builder /app/package.json /app/
WORKDIR /app
CMD ["dist/server.js"]

# Build for both AMD64 and ARM64 (Apple Silicon + Intel + AWS Graviton)
docker buildx create --name mybuilder --use

  --platform linux/amd64,linux/arm64 \
  -t myorg/myapp:latest \
  --push .

CI/CD Cache Optimization

# GitHub Actions with Docker layer caching
- name: Build and push
  uses: docker/build-push-action@v5
  with:
    context: .
    push: true
    tags: myorg/myapp:latest
    cache-from: type=gha         # GitHub Actions cache
    cache-to: type=gha,mode=max

Security Scanning

# Scan image for vulnerabilities

# Snyk
snyk container test myapp:latest

Common Pitfalls

  1. Not using multi-stage builds - The single biggest optimization. Separating build from runtime can halve your image size.
  2. Installing dev dependencies in production stage - Always use npm ci --production or separate dependency stages.
  3. Cache-busting with COPY . . - Place frequently changing operations (COPY source) after rarely changing ones (npm ci).
  4. Not scanning for vulnerabilities - Alpine images are smaller but can still have CVEs. Scan regularly.
  5. Ignoring build context size - A large .dockerignore-less context slows down every build. Always check what is being sent.

Exercises

  1. Size Optimization Challenge

    Start with a Dockerfile using node:20 (Debian) with a single stage. Measure its size. Then optimize in steps: (1) switch to Alpine, (2) add .dockerignore, (3) implement multi-stage build, (4) prune node_modules. Document the size at each step and calculate total reduction.

    Hint

    Use 'docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}"' to compare. Tag each iteration differently.

  2. BuildKit Cache Mounts

    Modify your Dockerfile to use BuildKit cache mounts for npm. Benchmark build times with and without cache mounts by building, making a small change to package.json (add a comment), and rebuilding.

    Hint

    Add '# syntax=docker/dockerfile:1' at the top. Use 'RUN --mount=type=cache,target=/root/.npm npm ci'. Enable with DOCKER_BUILDKIT=1.

  3. Security Audit

    Build your optimized image, then scan it with 'docker scout' or Trivy. Fix any HIGH or CRITICAL vulnerabilities by updating base images or packages. Document the before/after vulnerability count.

    Hint

    Install trivy with brew. Run 'trivy image myapp:latest'. Often, simply updating the Alpine base image or specific npm packages resolves CVEs.

Common Problems & Solutions

1 Multi-stage build fails: file not found in COPY --from
Cause: The file path in the builder stage does not match what you are trying to copy, or the build step did not produce output.
Solution: Add a debug step in the builder stage: `RUN ls -la /app/dist/`. Ensure the build command actually produces output in the expected directory. Check that WORKDIR is set correctly in both stages.
2 Native npm packages crash in Alpine production stage
Cause: Native modules (bcrypt, sharp, etc.) compiled in the builder stage may link against libraries not present in the production stage.
Solution: Ensure the production stage has necessary runtime libraries: `RUN apk add --no-cache libc6-compat`. For sharp, add `vips-dev`. Better yet, use pure-JS alternatives (bcryptjs) or install native deps in the production stage too.
3 Build takes forever despite 'no changes'
Cause: Docker build context is too large (node_modules, .git, etc. being sent to daemon), or cache is not being utilized due to layer ordering.
Solution: Check .dockerignore is properly excluding node_modules and .git. Verify layer order: package.json copy and npm ci should come BEFORE source code copy. Use `docker build --progress=plain .` to see which layers are rebuilding.
