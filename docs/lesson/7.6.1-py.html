Terraform Basics: Providers, Resources, State

Intuition

Terraform is like a blueprint for a building. You describe what rooms (resources) you want, and Terraform is the contractor who builds exactly that. If you change the blueprint (add a room), the contractor only builds the new room without tearing down the existing ones. The state file is the contractor's notebook tracking what has been built so far.

Terraform Basics: Providers, Resources, State

Terraform defines infrastructure as code. Instead of clicking through cloud consoles, you declare resources in .tf files and Terraform creates them.

Core Setup

terraform {
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }
  backend "s3" {
    bucket = "myapp-terraform-state"
    key    = "prod/terraform.tfstate"
    region = "us-east-1"
  }
}
provider "aws" { region = var.aws_region }

Defining Resources

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.instance_type
  tags = { Name = "web-server", Environment = var.environment }
}

resource "aws_db_instance" "main" {
  allocated_storage   = 20
  engine              = "postgres"
  instance_class      = "db.t3.micro"
  db_name             = "myapp"
  username            = var.db_username
  password            = var.db_password
  skip_final_snapshot = true
}

variable "aws_region" { default = "us-east-1" }
variable "instance_type" { default = "t3.micro" }
output "web_public_ip" { value = aws_instance.web.public_ip }

Terraform Workflow

terraform init      # Download providers, initialize backend
terraform plan      # Preview changes (dry run)
terraform apply     # Apply changes
terraform destroy   # Tear down all managed resources

Common Mistakes

  • Storing state locally instead of a remote backend (risks loss and conflicts)
  • Hardcoding secrets in .tf files
  • Running apply without reviewing plan first

Key Takeaways

  • Always use a remote backend (S3 + DynamoDB lock) for teams
  • Run terraform plan before every apply
  • Use variables and outputs for reusable configurations

Exercises

  1. Provision a Web Stack

    Write Terraform configuration to create an EC2 instance, an RDS PostgreSQL database, and an S3 bucket. Use variables for instance type, region, and database credentials. Output the EC2 public IP and RDS endpoint.

    Hint

    Use aws_instance, aws_db_instance, and aws_s3_bucket resources. Reference variables with var.name.

  2. Remote State Setup

    Configure Terraform to use an S3 backend for remote state storage with DynamoDB for state locking. Create the S3 bucket and DynamoDB table first, then migrate local state to remote.

    Hint

    Create the S3 bucket and DynamoDB table manually or with a separate Terraform config. Use terraform init -migrate-state.

  3. Terraform Modules

    Refactor a monolithic Terraform configuration into reusable modules: one for networking (VPC, subnets), one for compute (EC2), and one for database (RDS).

    Hint

    Create a modules/ directory with subdirectories. Use module blocks in root main.tf to call them.

Common Problems & Solutions

1 Terraform state is out of sync with actual infrastructure
Symptoms: ["terraform plan shows changes you didn't make", 'Error: resource already exists']
Cause: Someone modified resources manually through the AWS console.
Solution: Run `terraform plan` to see drift. Use `terraform import` to bring manually created resources under Terraform management. Use `terraform refresh` to update state from actual infrastructure.
2 State lock error prevents terraform apply
Symptoms: ['Error: Error locking state', 'Lock Info: ID: xxxx-xxxx']
Cause: A previous terraform operation crashed without releasing the DynamoDB lock.
Solution: Use `terraform force-unlock ` with the lock ID from the error message. Verify no other operation is genuinely running first.
