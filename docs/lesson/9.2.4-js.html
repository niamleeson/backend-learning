Design a Search Engine (Google)

Intuition

A search engine is like a massive library with a card catalog. The web crawler is a team of librarians constantly visiting bookstores (websites), bringing back books (pages), and noting what each book is about. The indexer builds the card catalog (inverted index) -- under each subject, it lists which books mention it. When a patron (user) asks a question, the librarian does not read every book. Instead, they look up the subject in the card catalog, find matching books, and rank them by quality (PageRank) and relevance (TF-IDF). The most popular questions are posted on a bulletin board (cache) so the librarian does not even need to check the catalog.

Design a Search Engine (Google)

Introduction

A web search engine is arguably the most complex software system ever built. It crawls billions of web pages, indexes their content, and returns the most relevant results in under 200 milliseconds. In a system design interview, you are not expected to replicate Google -- but you must demonstrate understanding of the core components: web crawling, indexing, ranking, and serving. We will focus on the backend infrastructure and data systems that make search work.

Step 1: Requirements Gathering

Functional Requirements

  • Web crawling: Discover and download web pages continuously
  • Indexing: Build a searchable index from crawled content
  • Search queries: Accept a text query and return ranked results
  • Autocomplete: Suggest queries as the user types
  • Snippet generation: Show relevant excerpts from matching pages
  • Freshness: New and updated content appears in results quickly

Non-Functional Requirements

  • Latency: Search results returned in < 200ms (p95)
  • Scale: Index 10B+ web pages, serve 100K queries/sec
  • Availability: 99.99% uptime for search serving
  • Freshness: Breaking news indexed within minutes
  • Relevance: Results should be highly relevant (measured by click-through rate)

Step 2: Capacity Estimation

Web pages indexed:     10 billion
Avg page size:         50 KB (text content after stripping HTML)
Raw content:           10B * 50KB = 500 TB
Inverted index size:   ~20% of raw content = 100 TB
Total index (with metadata): ~200 TB

Queries per second:    100,000 QPS (peak: 300K)
Crawl rate:            1 billion pages/day (recrawl + new)
Bandwidth for crawling: 1B * 50KB / 86400 = ~580 GB/s

Serving infrastructure:
  Each query touches ~100 index shards
  Each shard returns top 10 results in ~50ms
  Merge + rank in ~50ms
  Total: ~200ms

Step 4: API Design

import express from 'express';
const router = express.Router();

// Search endpoint
router.get('/api/v1/search', async (req, res) => {
  const {
    q,                    // search query
    page = 1,             // page number
    limit = 10,           // results per page
    lang = 'en',          // language filter
    freshness,            // 'hour', 'day', 'week', 'month'
    safe = 'moderate',    // safe search level
  } = req.query;

  if (!q || q.trim().length === 0) {
    return res.status(400).json({ error: 'Query is required' });
  }

  const startTime = process.hrtime.bigint();

  // 1. Parse and analyze query
  const parsedQuery = queryParser.parse(q, { lang });
  // { terms: ['distributed', 'systems'], phrases: ['cap theorem'],
  //   filters: { site: null, filetype: null }, intent: 'informational' }

  // 2. Check query cache
  const cacheKey = `search:${lang}:${q}:${page}`;
  const cached = await redis.get(cacheKey);
  if (cached) {
    const elapsed = Number(process.hrtime.bigint() - startTime) / 1e6;
    const result = JSON.parse(cached);
    result.meta.cached = true;
    result.meta.took_ms = elapsed;
    return res.json(result);
  }

  // 3. Fan out to index shards
  const shardResults = await queryCoordinator.fanOut(parsedQuery, {
    offset: (page - 1) * limit,
    limit,
    freshness,
  });

  // 4. Merge and re-rank results
  const rankedResults = ranker.merge(shardResults, parsedQuery);

  // 5. Generate snippets
  const resultsWithSnippets = await Promise.all(
    rankedResults.map(async (result) => ({
      title: result.title,
      url: result.url,
      snippet: snippetGenerator.generate(result.content, parsedQuery.terms),
      favicon: result.favicon,
      publishedAt: result.publishedAt,
    }))
  );

  const elapsed = Number(process.hrtime.bigint() - startTime) / 1e6;

  const response = {
    results: resultsWithSnippets,
    meta: {
      query: q,
      total: shardResults.totalHits,
      page,
      limit,
      took_ms: elapsed,
      cached: false,
    },
  };

  // Cache for 5 minutes
  await redis.setex(cacheKey, 300, JSON.stringify(response));

  res.json(response);
});

Step 5: Database Schema / Data Structures

The Inverted Index

The inverted index is the core data structure. It maps every word to the list of documents containing it.

Forward Index (what a page contains):
  doc_1: ["the", "cat", "sat", "on", "the", "mat"]
  doc_2: ["the", "dog", "sat", "on", "the", "log"]

Inverted Index (which pages contain a word):
  "cat"  -> [(doc_1, pos:1, tf:1)]
  "sat"  -> [(doc_1, pos:2, tf:1), (doc_2, pos:2, tf:1)]
  "dog"  -> [(doc_2, pos:1, tf:1)]
  "mat"  -> [(doc_1, pos:5, tf:1)]
  "log"  -> [(doc_2, pos:5, tf:1)]
  "the"  -> [(doc_1, pos:0, tf:2), (doc_2, pos:0, tf:2)]
  "on"   -> [(doc_1, pos:3, tf:1), (doc_2, pos:3, tf:1)]

class InvertedIndex {
  constructor() {
    this.index = new Map();       // term -> PostingList
    this.documents = new Map();   // docId -> document metadata
    this.docCount = 0;
  }

  addDocument(docId, content, metadata) {
    const terms = this.tokenize(content);
    const termFreqs = new Map();

    terms.forEach((term, position) => {
      if (!termFreqs.has(term)) termFreqs.set(term, { freq: 0, positions: [] });
      const entry = termFreqs.get(term);
      entry.freq++;
      entry.positions.push(position);
    });

    for (const [term, { freq, positions }] of termFreqs) {
      if (!this.index.has(term)) {
        this.index.set(term, []);
      }
      this.index.get(term).push({
        docId,
        termFrequency: freq,
        positions,
      });
    }

    this.documents.set(docId, {
      ...metadata,
      totalTerms: terms.length,
    });
    this.docCount++;
  }

  search(query, limit = 10) {
    const queryTerms = this.tokenize(query);
    const scores = new Map();

    for (const term of queryTerms) {
      const postings = this.index.get(term) || [];
      const idf = Math.log(this.docCount / (postings.length + 1));

      for (const posting of postings) {
        const doc = this.documents.get(posting.docId);
        const tf = posting.termFrequency / doc.totalTerms;
        const tfidf = tf * idf;

        const current = scores.get(posting.docId) || 0;
        scores.set(posting.docId, current + tfidf);
      }
    }

    return [...scores.entries()]
      .sort((a, b) => b[1] - a[1])
      .slice(0, limit)
      .map(([docId, score]) => ({
        docId,
        score,
        ...this.documents.get(docId),
      }));
  }

  tokenize(text) {
    return text
      .toLowerCase()
      .replace(/[^a-z0-9\s]/g, '')
      .split(/\s+/)
      .filter(t => t.length > 1 && !STOP_WORDS.has(t));
  }
}

Step 6: Deep Dive -- PageRank and Ranking

PageRank Algorithm

PageRank treats the web as a graph where pages are nodes and hyperlinks are edges. A page's rank is proportional to the number and quality of pages linking to it.

function computePageRank(graph, damping = 0.85, iterations = 20) {
  const N = graph.nodeCount;
  const ranks = new Map();

  // Initialize all ranks equally
  for (const node of graph.nodes()) {
    ranks.set(node, 1 / N);
  }

  for (let i = 0; i < iterations; i++) {
    const newRanks = new Map();

    for (const node of graph.nodes()) {
      let incomingRank = 0;

      for (const inLink of graph.incomingLinks(node)) {
        const outDegree = graph.outgoingLinks(inLink).length;
        incomingRank += ranks.get(inLink) / outDegree;
      }

      newRanks.set(node, (1 - damping) / N + damping * incomingRank);
    }

    // Update ranks
    for (const [node, rank] of newRanks) {
      ranks.set(node, rank);
    }
  }

  return ranks;
}

Modern search engines combine many signals:

Step 7: Scaling Strategies

  1. Index sharding: Partition the index across hundreds of machines by document ID. Each shard holds a fraction of all documents.
  2. Query fan-out: A query is sent to ALL shards in parallel. Each shard returns its top K results. The coordinator merges and re-ranks.
  3. Index replication: Each shard is replicated 3+ times for availability. Queries are load-balanced across replicas.
  4. Tiered indexing: Hot tier (recent, popular pages) on fast SSDs. Cold tier (old, rarely accessed pages) on cheaper storage.
  5. Caching: Cache popular queries (the top 1000 queries account for a large percentage of traffic). Cache at multiple levels: result cache, posting list cache, document cache.
  6. Incremental indexing: Instead of rebuilding the entire index, apply delta updates from the crawl stream.

Common Problems & Solutions

1 Search latency spikes during index updates
Cause: Updating the inverted index in-place while serving queries causes lock contention and cache invalidation.
Solution: Use an immutable segment-based approach. Build new index segments offline, then atomically swap them into the serving path. Old segments are garbage collected after all in-flight queries complete. This is how Lucene/Elasticsearch work.
2 Stale results for rapidly changing pages (e.g., news sites)
Cause: The batch crawl and index pipeline has a multi-hour delay between crawling a page and it appearing in search results.
Solution: Implement a real-time indexing pipeline: high-priority URLs (news sites, social media) are crawled every few minutes via a streaming pipeline (Kafka). Updates are applied to a small real-time index that is merged with the main index at query time.
3 Crawler overloading target websites
Cause: No rate limiting per domain. The crawler sends hundreds of concurrent requests to the same domain.
Solution: Implement per-domain rate limiting in the URL frontier. Respect robots.txt crawl-delay directives. Use a domain-based queue where each domain has a minimum interval between requests (e.g., 1 request per second per domain).
