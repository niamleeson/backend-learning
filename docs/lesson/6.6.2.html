Dependency Inversion in Practice

Intuition

Dependency inversion is like a power outlet standard. Appliances (high-level) do not care if electricity comes from solar panels, coal, or wind (low-level). Both sides depend on the outlet standard (abstraction). You can switch power sources without rewiring your house.

Dependency Inversion in Practice

The Dependency Inversion Principle (DIP) states that high-level modules should not depend on low-level modules; both should depend on abstractions. In Python, this means your business logic depends on interfaces, not concrete implementations.

Without Dependency Inversion

# BAD: High-level module depends on low-level details
from sqlalchemy.orm import Session
import smtplib

class OrderService:
    def __init__(self):
        self.db = Session()  # Concrete dependency
        self.smtp = smtplib.SMTP("smtp.gmail.com")  # Concrete dependency

    def place_order(self, order_data):
        self.db.add(Order(**order_data))
        self.db.commit()
        self.smtp.sendmail("from@x.com", order_data["email"], "Order placed")

With Dependency Inversion

# GOOD: Both high and low level depend on abstractions
from abc import ABC, abstractmethod

class OrderRepository(ABC):
    @abstractmethod
    def save(self, order) -> None: ...

class Notifier(ABC):
    @abstractmethod
    def notify(self, recipient: str, message: str) -> None: ...

class OrderService:
    def __init__(self, repo: OrderRepository, notifier: Notifier):
        self.repo = repo
        self.notifier = notifier

    def place_order(self, order_data):
        order = Order(**order_data)
        self.repo.save(order)
        self.notifier.notify(order_data["email"], "Order placed")

Dependency Injection with FastAPI

from fastapi import FastAPI, Depends

def get_order_repo() -> OrderRepository:
    session = get_db_session()
    return SQLAlchemyOrderRepository(session)

def get_notifier() -> Notifier:
    return EmailNotifier(smtp_host=settings.SMTP_HOST)

def get_order_service(
    repo: OrderRepository = Depends(get_order_repo),
    notifier: Notifier = Depends(get_notifier),
) -> OrderService:
    return OrderService(repo=repo, notifier=notifier)

@app.post("/orders")
def create_order(data: OrderCreate, service: OrderService = Depends(get_order_service)):
    return service.place_order(data.dict())

Overriding for Tests

def test_place_order():
    fake_repo = InMemoryOrderRepository()
    fake_notifier = FakeNotifier()  # Records calls instead of sending emails

    service = OrderService(repo=fake_repo, notifier=fake_notifier)
    service.place_order({"email": "test@x.com", "item": "book"})

    assert len(fake_repo.orders) == 1
    assert fake_notifier.last_message == "Order placed"

Common Mistakes

  • Using dependency injection without actual abstractions (injecting concrete classes)
  • Creating god-factory functions that wire everything at import time
  • Over-applying DIP to simple utility functions that do not need abstraction

Key Takeaways

  • DIP makes code testable: swap real dependencies with fakes in tests
  • FastAPI's Depends() is a built-in dependency injection framework
  • Abstract base classes in Python serve as the 'interfaces' for DIP

Exercises

  1. Refactor to Use Dependency Inversion

    Take a service class that directly instantiates a database connection, an HTTP client, and a logger. Refactor it so all three are injected as abstract dependencies. Create fake implementations for testing.

    Hint

    Define abstract base classes for each dependency. Change the constructor to accept them as parameters. In tests, pass fake/mock implementations.

  2. FastAPI Dependency Injection Chain

    Build a FastAPI app with a 3-level dependency chain: Config -> Repository -> Service. Override the config dependency in tests to use a test database. Verify the entire chain resolves correctly.

    Hint

    Use app.dependency_overrides[get_config] = get_test_config in tests. Each Depends() function receives its own dependencies.

Common Problems & Solutions

1 Cannot unit test service because it creates its own database connection
Symptoms: ['Tests require a running database', 'Tests are slow because they hit real external services']
Cause: The service class instantiates dependencies internally instead of receiving them via constructor injection.
Solution: Apply dependency inversion: define abstract interfaces for external dependencies and inject concrete implementations through the constructor or FastAPI's Depends().

Untestable

class UserService: def init(self): self.db = PostgresDB() # hardcoded

Testable

class UserService: def init(self, db: DatabasePort): # injected self.db = db

2 Dependency injection container becomes complex and hard to follow
Cause: Too many levels of dependency injection with complex factory functions.
Solution: Use a composition root pattern: wire all dependencies in one place (e.g., a create_app() function). Keep the dependency graph shallow. Use FastAPI's Depends() for request-scoped dependencies.
