Client-Server Model and Request Lifecycle

Intuition

Think of a web request like sending a letter: DNS is looking up the address, TCP is establishing a reliable postal route, and HTTP is the actual letter and reply. Each step adds latency but ensures your message arrives correctly.

Client-Server Model & Request Lifecycle

The client-server model is the foundational architecture of the modern web. When a user types a URL or clicks a link, a complex chain of events unfolds that routes the request from their browser to the correct server and back again.

Core Concepts

  • DNS Resolution: The browser resolves the domain name to an IP address through the DNS hierarchy, checking local cache, recursive resolvers, and authoritative nameservers.
  • TCP/TLS Handshake: A TCP connection is established via a three-way handshake (SYN, SYN-ACK, ACK). For HTTPS, a TLS handshake negotiates encryption parameters.
  • HTTP Request/Response Cycle: The client sends an HTTP request with method, headers, and optional body. The server processes it through middleware, routing, business logic, and returns a response with status code, headers, and body.
  • Connection Management: HTTP/1.1 uses keep-alive connections; HTTP/2 multiplexes streams over a single connection; HTTP/3 uses QUIC over UDP.

Key Takeaways

  1. Every web request traverses multiple network layers before reaching application code.
  2. Understanding the full lifecycle helps you diagnose latency, debug failures, and optimize performance at each stage.
  3. Modern protocols like HTTP/2 and HTTP/3 reduce overhead by multiplexing and eliminating head-of-line blocking.

Exercises

  1. Trace a Request with curl

    Use curl -v --trace-time https://example.com to observe DNS resolution, TCP handshake, TLS negotiation, and HTTP exchange. Document each phase with its timing.

    Hint

    Look for lines starting with ' Connected to', ' TLSv1.3', '> GET', and '< HTTP' to identify each phase.

  2. Build a Minimal HTTP Server

    Using Python's socket module (not http.server), build a TCP server that accepts connections, parses raw HTTP request lines, and returns a properly formatted HTTP response with status line, headers, and body.

    Hint

    Read bytes from the socket until you find '\r\n\r\n' to detect the end of headers. Respond with 'HTTP/1.1 200 OK\r\nContent-Length: ...\r\n\r\n...'.

Common Problems & Solutions

1 Requests intermittently time out despite the server being healthy.
Cause: DNS resolution is slow or returning stale records, causing the client to attempt connections to unreachable IPs.
Solution: Implement DNS caching at the client or application level, reduce DNS TTLs during migrations, and use health-checked DNS providers like Route 53 or Cloudflare.
2 High latency on first requests but subsequent requests are fast.
Cause: Cold TCP and TLS handshakes add round trips. Without connection reuse, every request pays this cost.
Solution: Enable HTTP keep-alive, use HTTP/2 for multiplexing, and consider TLS session resumption or 0-RTT with TLS 1.3.
