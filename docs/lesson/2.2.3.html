Mocking: unittest.mock, patch, MagicMock

Intuition

Mocking is like using a stunt double in a movie. The real actor (dependency) stays safe while the double (mock) takes the hit, and you control exactly what the double does.

Mocking: unittest.mock, patch, and MagicMock

Mocking lets you replace real objects with controlled substitutes during testing, isolating the code under test from external dependencies.

from unittest.mock import patch, MagicMock, PropertyMock
import requests

def fetch_user(user_id: int) -> dict:
    response = requests.get(f"https://api.example.com/users/{user_id}")
    response.raise_for_status()
    return response.json()

# Patching an external call
@patch('requests.get')
def test_fetch_user(mock_get):
    mock_response = MagicMock()
    mock_response.json.return_value = {"id": 1, "name": "Alice"}
    mock_response.raise_for_status.return_value = None
    mock_get.return_value = mock_response

    result = fetch_user(1)
    assert result == {"id": 1, "name": "Alice"}
    mock_get.assert_called_once_with("https://api.example.com/users/1")

# Using patch as context manager
def test_fetch_user_error():
    with patch('requests.get') as mock_get:
        mock_get.side_effect = requests.ConnectionError
        try:
            fetch_user(1)
        except requests.ConnectionError:
            pass
        mock_get.assert_called_once()

Key Takeaways: - Patch where the object is used, not where it is defined - MagicMock auto-creates attributes and methods on access - Use side_effect to raise exceptions or return different values per call - assert_called_once_with() verifies exact call arguments - Prefer dependency injection over mocking when possible for cleaner tests

Exercises

  1. Mock a Database Layer

    Write a UserService class that depends on a UserRepository. Mock the repository in tests to verify UserService.get_active_users() filters correctly without a real database.

    Hint

    Create the mock with MagicMock(spec=UserRepository) to ensure only real methods can be called. Set return_value on the mock's method.

  2. Test Retry Logic with side_effect

    Write a function that retries an API call up to 3 times. Use side_effect = [ConnectionError, ConnectionError, mock_response] to simulate two failures then success.

    Hint

    Pass a list to side_effect to return different values on consecutive calls. Assert the mock was called exactly 3 times.

Common Problems & Solutions

1 Mock does not take effect: the real function still runs during the test.
Cause: Patching the wrong import path. You must patch where the name is looked up, not where it is defined.
Solution: If `module_a` imports `requests.get`, patch `module_a.requests.get`, not `requests.get` directly.
2 MagicMock silently accepts any attribute access, hiding real bugs.
Cause: By default, MagicMock creates attributes dynamically, so typos in method names go undetected.
Solution: Use `MagicMock(spec=RealClass)` to restrict the mock to only the real class's attributes and methods.
