Subqueries and Correlated Subqueries

Intuition

A subquery is like asking a colleague a question in the middle of your work. A regular subquery is asking once and using the answer everywhere. A correlated subquery is asking a different question for every item you're reviewing — it works, but it's much slower because your colleague has to look things up each time.

Subqueries and Correlated Subqueries

Introduction

A subquery is a query nested inside another query. They allow you to break complex problems into manageable steps, use the result of one query as input to another, and express logic that's difficult or impossible with JOINs alone. Mastering subqueries is essential for writing sophisticated SQL.

Types of Subqueries

Subquery Types:

1. Scalar Subquery     -> Returns a single value
2. Row Subquery        -> Returns a single row
3. Table Subquery      -> Returns multiple rows/columns
4. Correlated Subquery -> References the outer query

Scalar Subqueries

Returns a single value — used in SELECT, WHERE, or HAVING:

-- Find users whose order total exceeds the average
SELECT name, email
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.total > (SELECT AVG(total) FROM orders);

In Node.js:

async function getAboveAverageOrders() {
  const result = await pool.query(`
    SELECT
      u.name,
      o.id AS order_id,
      o.total,
      (SELECT AVG(total)::numeric(10,2) FROM orders) AS avg_total
    FROM users u
    JOIN orders o ON u.id = o.user_id
    WHERE o.total > (SELECT AVG(total) FROM orders)
    ORDER BY o.total DESC
  `);
  return result.rows;
}

Scalar Subquery in SELECT

SELECT
  u.name,
  u.email,
  (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS order_count,
  (SELECT MAX(total) FROM orders o WHERE o.user_id = u.id) AS max_order
FROM users u;

Warning: Scalar subqueries in SELECT execute once per row — this can be very slow on large tables. Prefer JOINs with GROUP BY for performance.

Table Subqueries with IN

-- Find users who have placed at least one order over $100
SELECT name, email
FROM users
WHERE id IN (
  SELECT DISTINCT user_id
  FROM orders
  WHERE total > 100
);

NOT IN

-- Find users who have never placed an order
SELECT name, email
FROM users
WHERE id NOT IN (
  SELECT user_id FROM orders WHERE user_id IS NOT NULL
);

Critical pitfall: If the subquery returns ANY NULL value, NOT IN returns no rows! Always add WHERE column IS NOT NULL in the subquery, or better, use NOT EXISTS.

EXISTS and NOT EXISTS

EXISTS returns true if the subquery returns at least one row:

-- Find users with at least one order (same as IN but often faster)
SELECT u.name, u.email
FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.user_id = u.id
);

-- Find users with NO orders (safe alternative to NOT IN)
SELECT u.name, u.email
FROM users u
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.user_id = u.id
);

EXISTS stops scanning as soon as it finds one match, making it efficient.

Correlated Subqueries

A correlated subquery references columns from the outer query, meaning it runs once for each row of the outer query:

-- Find each user's most recent order
SELECT u.name, o.id, o.total, o.created_at
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.created_at = (
  SELECT MAX(o2.created_at)
  FROM orders o2
  WHERE o2.user_id = u.id  -- references outer query
);

Correlated Subquery Execution:

For each row in outer query:
  +---> Run inner query using outer row's values
  |     Return result
  +---> Evaluate WHERE condition
  |
  Next row...

This is O(n * m) in the worst case!

Lateral Joins (PostgreSQL alternative to correlated subqueries)

PostgreSQL's LATERAL keyword lets you reference the outer query in a JOIN, which is often cleaner and faster:

-- Get each user's 3 most recent orders
SELECT u.name, recent.*
FROM users u
CROSS JOIN LATERAL (
  SELECT o.id, o.total, o.created_at
  FROM orders o
  WHERE o.user_id = u.id
  ORDER BY o.created_at DESC
  LIMIT 3
) recent;

async function getUsersWithRecentOrders(limit = 3) {
  const result = await pool.query(`
    SELECT u.name, u.email, recent.order_id, recent.total, recent.created_at
    FROM users u
    CROSS JOIN LATERAL (
      SELECT o.id AS order_id, o.total, o.created_at
      FROM orders o
      WHERE o.user_id = u.id
      ORDER BY o.created_at DESC
      LIMIT $1
    ) recent
    ORDER BY u.name, recent.created_at DESC
  `, [limit]);
  return result.rows;
}

Subqueries in FROM (Derived Tables)

Use a subquery as a virtual table:

SELECT category, avg_price
FROM (
  SELECT
    category,
    AVG(price)::numeric(10,2) AS avg_price,
    COUNT(*) AS product_count
  FROM products
  GROUP BY category
) AS category_stats
WHERE product_count > 5
ORDER BY avg_price DESC;

Performance: Subquery vs JOIN

// Often the planner optimizes these identically, but not always.

// Subquery approach
const sub = await pool.query(`
  SELECT * FROM users
  WHERE id IN (SELECT user_id FROM orders WHERE total > 100)
`);

// JOIN approach (usually preferred)
const join = await pool.query(`
  SELECT DISTINCT u.*
  FROM users u
  JOIN orders o ON u.id = o.user_id
  WHERE o.total > 100
`);

// EXISTS approach (often fastest for "does a match exist?")
const exists = await pool.query(`
  SELECT * FROM users u
  WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.total > 100)
`);

General guidance: - For "row exists" checks: use EXISTS - For fetching related data: use JOIN - For complex multi-step logic: use CTEs (next lesson) - Avoid correlated subqueries in SELECT on large tables

Common Problems & Solutions

1 NOT IN returns zero rows even when matches clearly exist
Cause: The subquery returns one or more NULL values. In SQL, `x NOT IN (1, 2, NULL)` evaluates to UNKNOWN (not TRUE) for every x, so no rows pass the filter.
Solution: Use NOT EXISTS instead: `WHERE NOT EXISTS (SELECT 1 FROM table WHERE ...)`, or add `WHERE column IS NOT NULL` to the subquery.

-- Broken: NOT IN with possible NULLs\nSELECT * FROM users WHERE id NOT IN (SELECT user_id FROM orders);\n-- Fixed: NOT EXISTS\nSELECT * FROM users u WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

2 Correlated subquery makes the query extremely slow
Cause: The subquery executes once per row of the outer query, leading to O(n*m) performance on large tables.
Solution: Rewrite using a JOIN with GROUP BY, a LATERAL join, or a CTE. LATERAL is especially useful when you need top-N per group.
3 Subquery returns more than one row when a single value is expected
Cause: Using a subquery that returns multiple rows where a scalar (single value) is expected, such as in `WHERE col = (subquery)`.
Solution: Use `IN` instead of `=` for multi-row results, or add `LIMIT 1` / aggregate function to the subquery to guarantee a single value.
