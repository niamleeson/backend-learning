Numeric Types: int, float, complex

Intuition

Integers are exact because Python allocates as much memory as needed. Floats are approximations stored in a fixed number of bits, like trying to write 1/3 in decimal, you always lose something.

Python provides three numeric types: int for whole numbers (unlimited precision), float for decimals (IEEE 754 double precision), and complex for complex arithmetic.

# Integers have arbitrary precision
big = 10 ** 100  # googol, no overflow!
hex_val = 0xFF   # 255
bin_val = 0b1010 # 10

# Floats have precision limits
print(0.1 + 0.2)           # 0.30000000000000004
print(0.1 + 0.2 == 0.3)    # False!

from decimal import Decimal
print(Decimal('0.1') + Decimal('0.2') == Decimal('0.3'))  # True

# Complex numbers
z = 3 + 4j
print(z.real, z.imag)       # 3.0 4.0
print(abs(z))               # 5.0 (magnitude)
  • Python ints never overflow; they grow as large as memory allows
  • Never compare floats with ==; use math.isclose() instead
  • Use Decimal for financial calculations requiring exact precision
  • Underscores improve readability: 1_000_000 is valid

Exercises

  1. Float Comparison

    Write a function almost_equal(a, b) that returns True if two floats are within 1e-9 of each other. Test with 0.1 + 0.2 and 0.3.

    Hint

    Use abs(a - b) < 1e-9 or math.isclose(a, b).

  2. Base Conversions

    Given the integer 255, print its binary, octal, and hexadecimal representations using built-in functions.

    Hint

    Use bin(), oct(), and hex().

Common Problems & Solutions

1 Unexpected result: `0.1 + 0.2 != 0.3`
Cause: IEEE 754 floating-point cannot represent 0.1 exactly in binary.
Solution: Use `math.isclose(0.1 + 0.2, 0.3)` or the `decimal` module for exact arithmetic.
2 TypeError: can't convert complex to float
Cause: Calling `float()` on a complex number is not allowed since there's no single real equivalent.
Solution: Access `.real` or `.imag` attributes explicitly: `float(z.real)`.
