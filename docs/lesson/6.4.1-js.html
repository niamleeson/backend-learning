AWS Fundamentals: EC2, RDS, S3

Intuition

Think of AWS like renting an apartment building for your application. EC2 is the apartment itself (the server where your code runs). RDS is the building's secure storage room (managed database -- the building handles maintenance, backups, and security). S3 is like a self-storage facility across the street (virtually unlimited space for files). IAM is the key card system that controls who can access what.

AWS Fundamentals: EC2, RDS, S3

Why AWS for Backend Developers

Amazon Web Services (AWS) is the most widely used cloud platform. Understanding its core services -- EC2 for compute, RDS for databases, and S3 for storage -- is essential for deploying and operating Node.js applications at scale.

EC2 (Elastic Compute Cloud)

EC2 provides virtual servers (instances) in the cloud where you run your Node.js applications.

Launching an EC2 Instance

# Using AWS CLI
aws ec2 run-instances \
  --image-id ami-0c55b159cbfafe1f0 \    # Amazon Linux 2023
  --instance-type t3.small \
  --key-name my-key-pair \
  --security-group-ids sg-12345678 \
  --subnet-id subnet-12345678 \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=myapp-server}]' \
  --user-data file://setup-script.sh

User Data Script (runs on first boot)

#!/bin/bash
# setup-script.sh - Automated EC2 setup

# Update system
yum update -y

# Install Node.js 20
curl -fsSL https://rpm.nodesource.com/setup_20.x | bash -
yum install -y nodejs

# Install PM2
npm install -g pm2

# Install Nginx
amazon-linux-extras install nginx1 -y
systemctl enable nginx
systemctl start nginx

# Create deploy user
useradd -m deploy
mkdir -p /home/deploy/.ssh
cp /root/.ssh/authorized_keys /home/deploy/.ssh/
chown -R deploy:deploy /home/deploy/.ssh
chmod 700 /home/deploy/.ssh

# Create app directory
mkdir -p /var/www/myapp
chown deploy:deploy /var/www/myapp

# Configure PM2 startup
su - deploy -c "pm2 startup systemd -u deploy --hp /home/deploy"

Security Groups (Firewall)

# Create security group
aws ec2 create-security-group \
  --group-name myapp-sg \
  --description "Security group for myapp" \
  --vpc-id vpc-12345678

# Allow SSH (from your IP only)
aws ec2 authorize-security-group-ingress \
  --group-id sg-12345678 \
  --protocol tcp \
  --port 22 \
  --cidr 203.0.113.50/32

# Allow HTTP/HTTPS (from anywhere)
aws ec2 authorize-security-group-ingress \
  --group-id sg-12345678 \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0

aws ec2 authorize-security-group-ingress \
  --group-id sg-12345678 \
  --protocol tcp \
  --port 443 \
  --cidr 0.0.0.0/0

RDS (Relational Database Service)

RDS is a managed database service. It handles backups, patching, replication, and failover so you do not have to.

Creating a PostgreSQL RDS Instance

aws rds create-db-instance \
  --db-instance-identifier myapp-db \
  --db-instance-class db.t3.small \
  --engine postgres \
  --engine-version 16.1 \
  --master-username myapp_admin \
  --master-user-password "$DB_PASSWORD" \
  --allocated-storage 20 \
  --storage-type gp3 \
  --vpc-security-group-ids sg-db-12345678 \
  --db-subnet-group-name myapp-db-subnet \
  --backup-retention-period 7 \
  --multi-az \
  --storage-encrypted \
  --no-publicly-accessible

Connecting from Node.js

// config/database.ts
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient({
  datasources: {
    db: {
      // RDS endpoint from AWS console or environment variable
      url: process.env.DATABASE_URL,
      // Example: postgresql://myapp_admin:password@myapp-db.abc123.us-east-1.rds.amazonaws.com:5432/myapp
    },
  },
});

export default prisma;

RDS Best Practices

1. NEVER make RDS publicly accessible
   └── Keep in private subnet, access via EC2 or VPN

2. Enable Multi-AZ for production
   └── Automatic failover to standby in another AZ

3. Enable automated backups
   └── Point-in-time recovery up to 35 days

4. Use read replicas for read-heavy workloads
   └── Route read queries to replicas

5. Enable encryption at rest
   └── Uses AWS KMS keys

6. Use IAM authentication (not passwords)
   └── Short-lived tokens instead of static passwords

S3 (Simple Storage Service)

S3 stores files (objects) like user uploads, images, backups, and static assets.

S3 Operations with AWS SDK

// services/storage.ts
import {
  S3Client,
  PutObjectCommand,
  GetObjectCommand,
  DeleteObjectCommand,
} from '@aws-sdk/client-s3';
import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

const s3 = new S3Client({
  region: process.env.AWS_REGION || 'us-east-1',
});

const BUCKET = process.env.S3_BUCKET!;

// Upload a file
export async function uploadFile(
  key: string,
  body: Buffer,
  contentType: string
): Promise<string> {
  await s3.send(
    new PutObjectCommand({
      Bucket: BUCKET,
      Key: key,
      Body: body,
      ContentType: contentType,
      ServerSideEncryption: 'AES256',
    })
  );
  return `https://${BUCKET}.s3.amazonaws.com/${key}`;
}

// Generate pre-signed URL for direct upload
export async function getUploadUrl(
  key: string,
  contentType: string,
  expiresIn = 3600
): Promise<string> {
  const command = new PutObjectCommand({
    Bucket: BUCKET,
    Key: key,
    ContentType: contentType,
  });
  return getSignedUrl(s3, command, { expiresIn });
}

// Generate pre-signed URL for download
export async function getDownloadUrl(
  key: string,
  expiresIn = 3600
): Promise<string> {
  const command = new GetObjectCommand({
    Bucket: BUCKET,
    Key: key,
  });
  return getSignedUrl(s3, command, { expiresIn });
}

// Delete a file
export async function deleteFile(key: string): Promise<void> {
  await s3.send(
    new DeleteObjectCommand({
      Bucket: BUCKET,
      Key: key,
    })
  );
}

Express File Upload to S3

// routes/upload.ts
import { Router } from 'express';
import multer from 'multer';
import { v4 as uuidv4 } from 'uuid';
import path from 'path';
import { uploadFile } from '../services/storage';

const router = Router();
const upload = multer({
  storage: multer.memoryStorage(),
  limits: { fileSize: 10 * 1024 * 1024 }, // 10MB
  fileFilter: (req, file, cb) => {
    const allowed = ['.jpg', '.jpeg', '.png', '.gif', '.webp'];
    const ext = path.extname(file.originalname).toLowerCase();
    if (allowed.includes(ext)) {
      cb(null, true);
    } else {
      cb(new Error('Invalid file type'));
    }
  },
});

router.post('/upload', upload.single('image'), async (req, res) => {
  if (!req.file) {
    return res.status(400).json({ error: 'No file provided' });
  }

  const ext = path.extname(req.file.originalname);
  const key = `uploads/${uuidv4()}${ext}`;

  const url = await uploadFile(key, req.file.buffer, req.file.mimetype);

  res.json({ url, key });
});

export default router;

Common Pitfalls

  1. Making RDS publicly accessible - Always place databases in private subnets. Access through EC2 or VPN only.
  2. Hardcoding AWS credentials - Use IAM roles for EC2, environment variables for local dev. Never commit credentials.
  3. Not enabling S3 versioning - Without versioning, deleted or overwritten files are gone forever.
  4. Using t2/t3 burstable instances in production without understanding burst credits - Once credits run out, CPU is throttled. Monitor CPU credit balance.
  5. Not setting up billing alerts - Always create a budget alert to avoid surprise charges.

Exercises

  1. Deploy Node.js to EC2

    Launch a t3.micro EC2 instance, SSH into it, install Node.js and PM2, deploy a simple Express API, and configure a security group to allow HTTP traffic. Verify you can reach the API from your browser using the public IP.

    Hint

    Use the AWS Free Tier. Create a key pair first. Install Node.js via NodeSource. Use PM2 to keep the app running. Open port 3000 in the security group.

  2. S3 File Upload API

    Build an Express endpoint that accepts file uploads via multer, stores them in S3 with a unique key, and returns the S3 URL. Implement a second endpoint that generates pre-signed URLs for direct client-to-S3 uploads (bypassing your server).

    Hint

    Use @aws-sdk/client-s3 and @aws-sdk/s3-request-presigner. For pre-signed URLs, generate a PutObject signed URL with expiration.

  3. RDS Connection Setup

    Create a free-tier RDS PostgreSQL instance, configure it in a VPC with a private subnet, and connect to it from an EC2 instance using Prisma. Run migrations and verify the database is working.

    Hint

    The EC2 and RDS security groups must allow traffic between them. Use SSH tunnel from your local machine to EC2 to manage the database with a GUI tool.

Common Problems & Solutions

1 Cannot connect to RDS from EC2
Cause: The RDS security group does not allow inbound traffic from the EC2 security group on port 5432.
Solution: Edit the RDS security group to allow inbound PostgreSQL (port 5432) from the EC2 instance's security group ID. Ensure both are in the same VPC. Verify the RDS endpoint is correct (not the IP, which can change).
2 S3 upload fails with AccessDenied
Cause: The IAM user/role does not have permission to write to the S3 bucket, or the bucket policy denies the action.
Solution: Check the IAM policy attached to your EC2 role or IAM user. Ensure it has s3:PutObject permission on the correct bucket/prefix. Check the bucket policy for explicit denies. Use AWS CloudTrail to see the exact authorization failure.
3 EC2 instance runs out of memory and becomes unresponsive
Cause: Node.js application consuming too much memory on a small instance, or a memory leak in the application.
Solution: Add swap space: `dd if=/dev/zero of=/swapfile bs=1M count=2048 && mkswap /swapfile && swapon /swapfile`. Set Node.js memory limit: `--max-old-space-size=512`. Upgrade to a larger instance type. Use PM2's max_memory_restart to auto-restart on memory threshold.
