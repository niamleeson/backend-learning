OAuth2 Flows: Authorization Code, Client Credentials, PKCE

Intuition

Think of OAuth2 like a hotel key card system. When you check in (authenticate), the front desk (authorization server) gives you a key card (access token) that only opens your room and the pool (scopes), not every room in the hotel. PKCE is like writing a secret number on a piece of paper at check-in and having to recite it when you pick up your key — it proves you are the same person who started the check-in process, even if someone overheard your reservation number.

OAuth2 Flows: Authorization Code, Client Credentials, PKCE

OAuth2 is the industry-standard protocol for authorization. It enables third-party applications to obtain limited access to a web service, either on behalf of a resource owner (a user) or on behalf of the application itself. Understanding OAuth2 deeply is essential for any backend engineer building APIs, integrating with external services, or securing microservices.

The OAuth2 Specification Overview

OAuth2 (RFC 6749) defines four roles:

  1. Resource Owner — The user who owns the data and can grant access to it.
  2. Client — The application requesting access to the resource owner's data.
  3. Authorization Server — The server that authenticates the resource owner and issues tokens (e.g., Auth0, Okta, Keycloak, or your own).
  4. Resource Server — The API server hosting the protected resources.

The protocol works by issuing tokens instead of sharing credentials. The client never sees the user's password. Instead, it receives a token with limited scope and lifetime.

Token Types

OAuth2 uses two primary token types:

Access Tokens

An access token is a credential used to access protected resources. It represents the authorization granted to the client. Access tokens are typically short-lived (minutes to hours) and are sent with every API request.

# Sending an access token with a request
import httpx

response = httpx.get(
    "https://api.example.com/users/me",
    headers={"Authorization": f"Bearer {access_token}"}
)

Access tokens can be opaque strings (random identifiers that the resource server must validate against the authorization server) or self-contained JWTs (JSON Web Tokens) that carry claims and can be verified locally.

Refresh Tokens

A refresh token is a long-lived credential used to obtain new access tokens without requiring the user to re-authenticate. Refresh tokens are stored securely on the server side and are never sent to the resource server.

# Using a refresh token to get a new access token
def refresh_access_token(refresh_token: str, client_id: str, client_secret: str) -> dict:
    response = httpx.post(
        "https://auth.example.com/oauth/token",
        data={
            "grant_type": "refresh_token",
            "refresh_token": refresh_token,
            "client_id": client_id,
            "client_secret": client_secret,
        }
    )
    response.raise_for_status()
    return response.json()  # {"access_token": "...", "expires_in": 3600, ...}

Scopes

Scopes limit what an access token can do. They represent specific permissions. For example, a GitHub OAuth app might request read:user and repo scopes. The authorization server presents these to the user during consent, and the issued token is limited to the granted scopes.

# Requesting specific scopes during authorization
scopes = "read:user repo:status notifications"
auth_url = (
    f"https://github.com/login/oauth/authorize"
    f"?client_id={CLIENT_ID}"
    f"&redirect_uri={REDIRECT_URI}"
    f"&scope={scopes}"
    f"&state={state}"
)

Scopes follow the principle of least privilege: always request only the permissions your application actually needs.

Flow 1: Authorization Code Flow

The Authorization Code flow is the most common and most secure flow for server-side web applications. It involves a front-channel (browser) and a back-channel (server-to-server) exchange.

When to Use

  • Server-side web applications where the client can securely store a client secret
  • Any scenario where a user needs to grant permission to a third-party app

Step-by-Step Sequence

Here is the complete sequence of the Authorization Code flow described step by step:

  1. User clicks "Login with Provider" — The client application redirects the user's browser to the authorization server's /authorize endpoint.
  2. Authorization Server authenticates the user — The user logs in (if not already) and sees a consent screen listing the requested scopes.
  3. User grants consent — The authorization server redirects the browser back to the client's redirect_uri with an authorization code and the original state parameter.
  4. Client exchanges the code for tokens — The client's backend server sends the authorization code, along with its client_id and client_secret, to the authorization server's /token endpoint (back-channel, server-to-server).
  5. Authorization Server returns tokens — The server responds with an access token, optionally a refresh token, token type, and expiration.
  6. Client uses the access token — The client includes the access token in API requests to the resource server.
User        Client (Browser)      Client (Server)      Auth Server       Resource Server
 |               |                       |                   |                   |
 |--click login->|                       |                   |                   |
 |               |---redirect to auth--->|                   |                   |
 |               |                       |                   |                   |
 |<--------------+--- /authorize --------+------------------>|                   |
 |               |                       |                   |                   |
 |--login+consent-->                     |                   |                   |
 |               |                       |                   |                   |
 |<--redirect with code+state-----------/|                   |                   |
 |               |                       |                   |                   |
 |---code------->|---code to server----->|                   |                   |
 |               |                       |---POST /token---->|                   |
 |               |                       |<--access_token----|                   |
 |               |                       |                   |                   |
 |               |                       |---GET /api ------>|------------------>|
 |               |                       |<--protected data--|<------------------|

Full Implementation

import httpx
import secrets
from urllib.parse import urlencode
from fastapi import FastAPI, Request, HTTPException
from fastapi.responses import RedirectResponse

app = FastAPI()

# Configuration
CLIENT_ID = "your-client-id"
CLIENT_SECRET = "your-client-secret"
AUTH_SERVER = "https://auth.example.com"
REDIRECT_URI = "https://yourapp.com/callback"

# In production, use a proper session store (Redis, database)
pending_states: dict[str, bool] = {}


@app.get("/login")
async def login():
    """Step 1: Redirect user to authorization server."""
    state = secrets.token_urlsafe(32)
    pending_states[state] = True

    params = urlencode({
        "response_type": "code",
        "client_id": CLIENT_ID,
        "redirect_uri": REDIRECT_URI,
        "scope": "openid profile email",
        "state": state,
    })
    return RedirectResponse(f"{AUTH_SERVER}/authorize?{params}")


@app.get("/callback")
async def callback(code: str, state: str):
    """Step 3-5: Receive code and exchange for tokens."""
    # Verify state to prevent CSRF attacks
    if state not in pending_states:
        raise HTTPException(status_code=400, detail="Invalid state parameter")
    del pending_states[state]

    # Exchange authorization code for tokens (back-channel)
    async with httpx.AsyncClient() as client:
        token_response = await client.post(
            f"{AUTH_SERVER}/oauth/token",
            data={
                "grant_type": "authorization_code",
                "code": code,
                "redirect_uri": REDIRECT_URI,
                "client_id": CLIENT_ID,
                "client_secret": CLIENT_SECRET,
            },
            headers={"Content-Type": "application/x-www-form-urlencoded"},
        )

    if token_response.status_code != 200:
        raise HTTPException(status_code=400, detail="Token exchange failed")

    tokens = token_response.json()
    access_token = tokens["access_token"]
    refresh_token = tokens.get("refresh_token")
    expires_in = tokens["expires_in"]

    # Store tokens securely in your session/database
    # NEVER expose the access token to the browser directly
    return {"message": "Login successful", "expires_in": expires_in}


async def fetch_user_profile(access_token: str) -> dict:
    """Step 6: Use the access token to call protected APIs."""
    async with httpx.AsyncClient() as client:
        response = await client.get(
            f"{AUTH_SERVER}/userinfo",
            headers={"Authorization": f"Bearer {access_token}"},
        )
        response.raise_for_status()
        return response.json()

The State Parameter

The state parameter is critical for security. It prevents Cross-Site Request Forgery (CSRF) attacks. The client generates a random, unguessable value, stores it in the session, and sends it with the authorization request. When the callback arrives, the client verifies the returned state matches. Without this check, an attacker could craft a malicious authorization response and trick the user into linking the attacker's account.

Flow 2: Client Credentials Flow

The Client Credentials flow is used for service-to-service communication where no user is involved. The client authenticates directly with the authorization server using its own credentials.

When to Use

  • Microservice-to-microservice communication
  • Backend daemons or cron jobs accessing APIs
  • Machine-to-machine (M2M) scenarios
  • Any case where the application acts on its own behalf, not on behalf of a user

Step-by-Step Sequence

  1. Client sends credentials — The client sends its client_id and client_secret directly to the authorization server's /token endpoint.
  2. Authorization server validates — The server verifies the client credentials.
  3. Authorization server returns an access token — No refresh token is typically issued since the client can always re-authenticate.
  4. Client uses the access token — The client calls the resource server with the token.
Client (Service A)         Auth Server           Resource Server (Service B)
      |                         |                         |
      |---POST /token---------->|                         |
      |   (client_id + secret)  |                         |
      |<--access_token----------|                         |
      |                         |                         |
      |---GET /api (Bearer)---------------------------->  |
      |<--response-----------------------------------------|

Full Implementation

import httpx
import time
from dataclasses import dataclass


@dataclass
class TokenCache:
    access_token: str
    expires_at: float


class ServiceClient:
    """A client that uses Client Credentials flow for service-to-service auth."""

    def __init__(self, client_id: str, client_secret: str, token_url: str):
        self.client_id = client_id
        self.client_secret = client_secret
        self.token_url = token_url
        self._token_cache: TokenCache | None = None

    async def _get_access_token(self) -> str:
        """Fetch or return cached access token."""
        # Return cached token if still valid (with 30-second buffer)
        if self._token_cache and self._token_cache.expires_at > time.time() + 30:
            return self._token_cache.access_token

        async with httpx.AsyncClient() as client:
            response = await client.post(
                self.token_url,
                data={
                    "grant_type": "client_credentials",
                    "client_id": self.client_id,
                    "client_secret": self.client_secret,
                    "scope": "service:read service:write",
                },
                headers={"Content-Type": "application/x-www-form-urlencoded"},
            )
            response.raise_for_status()
            data = response.json()

        self._token_cache = TokenCache(
            access_token=data["access_token"],
            expires_at=time.time() + data["expires_in"],
        )
        return self._token_cache.access_token

    async def call_service(self, url: str) -> dict:
        """Make an authenticated request to another service."""
        token = await self._get_access_token()
        async with httpx.AsyncClient() as client:
            response = await client.get(
                url,
                headers={"Authorization": f"Bearer {token}"},
            )
            response.raise_for_status()
            return response.json()


# Usage
order_service = ServiceClient(
    client_id="order-service",
    client_secret="super-secret-key",
    token_url="https://auth.internal.example.com/oauth/token",
)

# In an async context:
# user_data = await order_service.call_service("https://user-service.internal/api/users/123")

Notice the token caching strategy. The Client Credentials flow does not return refresh tokens — instead, you cache the access token and re-authenticate when it expires. The 30-second buffer ensures you do not use a token that is about to expire.

Flow 3: Authorization Code Flow with PKCE

PKCE (Proof Key for Code Exchange, pronounced "pixy") was originally designed for mobile and single-page applications that cannot securely store a client secret. It is now recommended for ALL authorization code flows, even confidential clients, as an additional layer of security.

The Problem PKCE Solves

In mobile apps and SPAs, the client secret cannot be kept secret — it would be embedded in the app binary or JavaScript bundle. Without a secret, an attacker who intercepts the authorization code (via a malicious app registered for the same custom URL scheme, for example) could exchange it for tokens. PKCE prevents this by binding the token exchange to the original authorization request.

How PKCE Works

  1. Client generates a code_verifier — A cryptographically random string (43-128 characters).
  2. Client derives a code_challenge — The SHA-256 hash of the verifier, base64url-encoded.
  3. Client sends code_challenge with the authorization request.
  4. Authorization server stores the challenge — Associated with the authorization code it issues.
  5. Client sends code_verifier with the token exchange request.
  6. Authorization server verifies — It hashes the verifier and compares it to the stored challenge. If they match, the code is legitimate.

An attacker who intercepts the authorization code does not have the code_verifier (it was never transmitted over the front channel), so they cannot complete the token exchange.

User         Client (SPA/Mobile)          Auth Server             Resource Server
 |                |                            |                        |
 |                |--generate code_verifier--->|                        |
 |                |--derive code_challenge     |                        |
 |                |                            |                        |
 |--click login-->|                            |                        |
 |                |---/authorize-------------->|                        |
 |                |   + code_challenge         |                        |
 |                |   + challenge_method=S256  |                        |
 |                |                            |                        |
 |<--login + consent screen--------------------|                        |
 |--grant-------->|                            |                        |
 |                |<--redirect + code----------|                        |
 |                |                            |                        |
 |                |---POST /token------------->|                        |
 |                |   + code                   |                        |
 |                |   + code_verifier          |                        |
 |                |                            |--verify:               |
 |                |                            |  SHA256(verifier)      |
 |                |                            |  == challenge?         |
 |                |<--access_token-------------|                        |
 |                |                            |                        |
 |                |---GET /api (Bearer)------->|----------------------->|
 |                |<--data---------------------|<-----------------------|

Full Implementation

import hashlib
import base64
import secrets
import httpx
from urllib.parse import urlencode


def generate_pkce_pair() -> tuple[str, str]:
    """Generate a PKCE code verifier and code challenge."""
    # Generate a cryptographically random code verifier (43-128 chars)
    code_verifier = secrets.token_urlsafe(64)  # ~86 chars

    # Derive the code challenge using SHA-256
    digest = hashlib.sha256(code_verifier.encode("ascii")).digest()
    code_challenge = base64.urlsafe_b64encode(digest).rstrip(b"=").decode("ascii")

    return code_verifier, code_challenge


class PKCEAuthClient:
    """OAuth2 client using Authorization Code + PKCE for public clients."""

    def __init__(self, client_id: str, auth_server: str, redirect_uri: str):
        self.client_id = client_id
        self.auth_server = auth_server
        self.redirect_uri = redirect_uri
        # In a real app, store these in the user's session
        self._code_verifier: str | None = None
        self._state: str | None = None

    def get_authorization_url(self, scopes: list[str]) -> str:
        """Build the authorization URL with PKCE challenge."""
        self._code_verifier, code_challenge = generate_pkce_pair()
        self._state = secrets.token_urlsafe(32)

        params = urlencode({
            "response_type": "code",
            "client_id": self.client_id,
            "redirect_uri": self.redirect_uri,
            "scope": " ".join(scopes),
            "state": self._state,
            "code_challenge": code_challenge,
            "code_challenge_method": "S256",
        })
        return f"{self.auth_server}/authorize?{params}"

    async def exchange_code(self, code: str, state: str) -> dict:
        """Exchange authorization code for tokens using PKCE verifier."""
        if state != self._state:
            raise ValueError("State mismatch — possible CSRF attack")

        if not self._code_verifier:
            raise ValueError("No code verifier found — start auth flow first")

        async with httpx.AsyncClient() as client:
            response = await client.post(
                f"{self.auth_server}/oauth/token",
                data={
                    "grant_type": "authorization_code",
                    "code": code,
                    "redirect_uri": self.redirect_uri,
                    "client_id": self.client_id,
                    # No client_secret needed! PKCE replaces it.
                    "code_verifier": self._code_verifier,
                },
            )
            response.raise_for_status()
            return response.json()


# Usage example
pkce_client = PKCEAuthClient(
    client_id="mobile-app-id",
    auth_server="https://auth.example.com",
    redirect_uri="myapp://callback",
)

# Step 1: Get URL and open in browser/webview
auth_url = pkce_client.get_authorization_url(["openid", "profile"])
print(f"Open this URL: {auth_url}")

Token Storage Best Practices

How and where you store tokens is just as important as how you obtain them.

Server-Side Applications

  • Store tokens in your database or session store (e.g., Redis, PostgreSQL), encrypted at rest.
  • Never expose access tokens to the browser. Use a session cookie (HttpOnly, Secure, SameSite=Strict) that maps to the token on the server.
  • Encrypt refresh tokens before storing them in the database.
from cryptography.fernet import Fernet

# Generate a key once and store it securely (e.g., in env vars / vault)
ENCRYPTION_KEY = Fernet.generate_key()
cipher = Fernet(ENCRYPTION_KEY)

def store_refresh_token(user_id: str, refresh_token: str):
    encrypted = cipher.encrypt(refresh_token.encode())
    # Save encrypted token to database
    db.execute(
        "UPDATE users SET refresh_token = %s WHERE id = %s",
        (encrypted, user_id)
    )

def retrieve_refresh_token(user_id: str) -> str:
    row = db.execute("SELECT refresh_token FROM users WHERE id = %s", (user_id,))
    return cipher.decrypt(row["refresh_token"]).decode()

Single-Page Applications (SPAs)

  • Use the Backend-for-Frontend (BFF) pattern — the SPA communicates with its own backend, which handles OAuth and stores tokens server-side.
  • If you must store tokens in the browser, use in-memory storage (JavaScript variables) rather than localStorage or sessionStorage, which are vulnerable to XSS.
  • Never store refresh tokens in the browser. Use token rotation if you must.

Mobile Applications

  • Use platform-specific secure storage: Keychain (iOS) or Keystore/EncryptedSharedPreferences (Android).
  • Never store tokens in plain-text files or shared preferences.

Common Implementation Mistakes

1. Not Validating the State Parameter

Skipping state validation opens your application to CSRF attacks. An attacker can initiate an OAuth flow with their own account and trick a victim into completing it, linking the attacker's identity to the victim's session.

2. Storing Client Secrets in Frontend Code

Client secrets in JavaScript bundles or mobile app binaries are not secret. Use PKCE for public clients instead of pretending the secret is safe.

3. Using the Implicit Flow

The Implicit flow (response_type=token) returns the access token directly in the URL fragment. This is deprecated because tokens in URLs appear in browser history, referrer headers, and server logs. Always use Authorization Code + PKCE instead.

4. Overly Broad Scopes

Requesting more permissions than needed violates the principle of least privilege and makes users less likely to grant consent. Request only what you need.

5. Not Implementing Token Refresh Logic

Access tokens expire. If your application does not handle token refresh gracefully, users will encounter unexpected 401 errors. Implement proactive refresh (before expiration) rather than reactive refresh (after a 401).

async def make_authenticated_request(url: str, token_store: dict) -> dict:
    """Example of proactive token refresh."""
    # Check if token is about to expire (within 60 seconds)
    if time.time() > token_store["expires_at"] - 60:
        new_tokens = await refresh_access_token(
            token_store["refresh_token"],
            CLIENT_ID,
            CLIENT_SECRET,
        )
        token_store["access_token"] = new_tokens["access_token"]
        token_store["expires_at"] = time.time() + new_tokens["expires_in"]
        if "refresh_token" in new_tokens:
            token_store["refresh_token"] = new_tokens["refresh_token"]

    async with httpx.AsyncClient() as client:
        response = await client.get(
            url,
            headers={"Authorization": f"Bearer {token_store['access_token']}"},
        )
        response.raise_for_status()
        return response.json()

6. Not Using HTTPS

OAuth2 relies on TLS for transport security. All authorization endpoints, redirect URIs, and token endpoints must use HTTPS. Tokens transmitted over HTTP can be intercepted trivially.

7. Accepting Redirect URI Wildcards

Your authorization server should enforce exact redirect URI matching. Allowing wildcards or partial matches enables open redirect attacks where an attacker can steal authorization codes.

Choosing the Right Flow

Scenario Flow Notes
Server-side web app Authorization Code Client secret stored securely on server
Single-page application Authorization Code + PKCE No client secret; use BFF pattern ideally
Mobile application Authorization Code + PKCE Use custom URL schemes or universal links
Service-to-service Client Credentials No user involved; machine-to-machine
IoT/limited input devices Device Authorization User authorizes on separate device (RFC 8628)

Summary

OAuth2 provides a flexible, token-based authorization framework. The Authorization Code flow is the workhorse for user-facing applications, keeping secrets on the server and exchanging short-lived codes for tokens. The Client Credentials flow serves machine-to-machine scenarios with straightforward credential exchange. PKCE extends the Authorization Code flow to public clients by cryptographically binding the token exchange to the original request, eliminating the need for a client secret. In all cases, proper token storage, scope management, state validation, and HTTPS usage are non-negotiable for a secure implementation.

Common Problems & Solutions

1 Authorization code exchange returns 'invalid_grant' error
Symptoms: After the user authorizes and is redirected back to your callback URL, the POST request to the /token endpoint returns a 400 status with error 'invalid_grant'.
Cause: This typically occurs because: (1) the authorization code has already been used (codes are single-use), (2) the code has expired (usually within 30-60 seconds), (3) the redirect_uri in the token request does not exactly match the one used in the authorization request, or (4) the code was issued to a different client_id.
Solution: Ensure the redirect_uri matches exactly (including trailing slashes and URL encoding). Exchange the code immediately upon receiving it. Never attempt to reuse a code. Log the exact redirect_uri sent in both the authorize and token requests and compare them character by character.

Common mistake: mismatched redirect URIs

Authorization request uses:

redirect_uri = "https://myapp.com/callback"

But token exchange uses:

redirect_uri = "https://myapp.com/callback/" # Trailing slash = different URI!

Fix: Use a constant for redirect_uri

REDIRECT_URI = "https://myapp.com/callback"

Use REDIRECT_URI in BOTH the authorize and token requests

2 PKCE verification fails with 'invalid_grant' despite correct implementation
Symptoms: The token exchange fails when using PKCE even though you are sending the code_verifier. The authorization server returns 'invalid_grant' or 'code_verifier mismatch'.
Cause: The most common cause is incorrect base64url encoding of the code challenge. Standard base64 uses '+' and '/' characters, but base64url uses '-' and '_'. Additionally, padding characters ('=') must be stripped. Another cause is the code_verifier and code_challenge being generated in different sessions (e.g., the verifier was lost between generating the authorization URL and handling the callback).
Solution: Use base64.urlsafe_b64encode (not base64.b64encode) and strip the trailing '=' padding. Ensure the code_verifier is persisted in the user's session between the authorization request and the callback. Verify by manually computing SHA256 of your verifier and comparing to the challenge.

import hashlib import base64

code_verifier = "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"

WRONG: standard base64

wrong_challenge = base64.b64encode( hashlib.sha256(code_verifier.encode()).digest() ).decode()

CORRECT: base64url with padding stripped

correct_challenge = base64.urlsafe_b64encode( hashlib.sha256(code_verifier.encode()).digest() ).rstrip(b'=').decode()

print(f"Wrong: {wrong_challenge}") print(f"Correct: {correct_challenge}")

3 Access tokens are leaked in server logs or browser history
Symptoms: Security audit reveals access tokens appearing in web server access logs, browser history, or referrer headers. Users report unauthorized access to their accounts.
Cause: This happens when using the deprecated Implicit flow (tokens in URL fragments), passing tokens as URL query parameters instead of headers, or logging full request URLs/headers without redaction.
Solution: Always use the Authorization Code flow (with PKCE for public clients) instead of the Implicit flow. Send tokens in the Authorization header, never as query parameters. Configure your logging middleware to redact the Authorization header. Use response_type=code, never response_type=token.

BAD: Token in query parameter (appears in logs)

response = httpx.get(f"https://api.example.com/data?access_token={token}")

GOOD: Token in Authorization header

response = httpx.get( "https://api.example.com/data", headers={"Authorization": f"Bearer {token}"} )

Logging middleware that redacts sensitive headers

import logging

def log_request(request): headers = dict(request.headers) if "authorization" in headers: headers["authorization"] = "Bearer [REDACTED]" logging.info(f"{request.method} {request.url} headers={headers}")
