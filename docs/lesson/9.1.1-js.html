Design a URL Shortener (TinyURL)

Intuition

Think of a URL shortener like a coat check at a theater. You hand over your bulky coat (long URL), receive a small numbered ticket (short code), and when you return with the ticket, you get your coat back instantly. The system needs to be fast at both issuing tickets and retrieving coats, and it must never give two people the same ticket number.

Design a URL Shortener (TinyURL)

Introduction

A URL shortener takes a long URL and creates a compact alias that redirects to the original. Services like TinyURL, Bitly, and short.io handle billions of redirects daily. This is one of the most commonly asked system design interview questions because it touches on hashing, database design, caching, scalability, and analytics -- all in a deceptively simple-looking system.

Step 1: Requirements Gathering

Functional Requirements

  1. Given a long URL, generate a unique short URL
  2. When a user visits the short URL, redirect them to the original long URL
  3. Users can optionally specify a custom short alias
  4. Short links expire after a configurable time (default: 5 years)
  5. Analytics: track click count, referrer, and geographic data

Non-Functional Requirements

  1. The system should be highly available -- a broken redirect is a terrible user experience
  2. Redirection must happen in real-time with minimal latency (< 100ms)
  3. Short URLs should not be predictable (guessable) for security
  4. The system should scale to handle 100M+ URLs and 10B+ redirects per month

Step 2: Capacity Estimation

Traffic Estimates

  • New URL shortenings: 100M per month (~40 per second)
  • Read-to-write ratio: 100:1 (redirects dominate)
  • Redirects: 10B per month (~4,000 per second)
  • Peak redirects: ~8,000 per second (2x average)

Storage Estimates

  • Average URL length: 200 bytes
  • Short URL key: 7 characters = 7 bytes
  • Metadata (timestamps, user_id): ~100 bytes
  • Per record: ~500 bytes
  • 100M URLs/month x 12 months x 5 years = 6B URLs
  • Total storage: 6B x 500 bytes = 3 TB

Bandwidth

  • Incoming (writes): 40 req/s x 500 bytes = 20 KB/s
  • Outgoing (reads): 4,000 req/s x 500 bytes = 2 MB/s

Cache

  • If we cache 20% of hot URLs: 6B x 0.20 x 500 bytes = 600 GB
  • In practice, URL popularity follows a power law, so caching the top 20% covers 80%+ of traffic

Component Responsibilities

  • Load Balancer: Distributes traffic across API servers. Uses round-robin or least-connections.
  • API Servers: Stateless Node.js instances handling URL creation and redirection.
  • Redis Cache: Stores frequently accessed short-to-long URL mappings for sub-millisecond lookups.
  • PostgreSQL: Persistent storage for all URL mappings, user data, and metadata.
  • Analytics Queue: Decouples click tracking from the redirect hot path.
  • Analytics Worker: Processes click events asynchronously for dashboards and reporting.

Express Route Implementation

import express from 'express';
import { createShortUrl, resolveShortUrl, getStats } from './url.service';

const router = express.Router();

// Create short URL
router.post('/api/v1/urls', async (req, res) => {
  const { long_url, custom_alias, expires_at } = req.body;

  if (!long_url) {
    return res.status(400).json({ error: 'long_url is required' });
  }

  try {
    const result = await createShortUrl({ long_url, custom_alias, expires_at });
    res.status(201).json(result);
  } catch (err) {
    if (err.message === 'ALIAS_TAKEN') {
      return res.status(409).json({ error: 'Custom alias already in use' });
    }
    res.status(500).json({ error: 'Internal server error' });
  }
});

// Redirect
router.get('/:shortCode', async (req, res) => {
  const { shortCode } = req.params;

  const longUrl = await resolveShortUrl(shortCode);
  if (!longUrl) {
    return res.status(404).json({ error: 'Short URL not found' });

  trackClick(shortCode, req.headers, req.ip).catch(() => {});

Step 5: Short Code Generation

This is the core algorithmic challenge. We need to generate unique, compact identifiers.

Approach 1: Base62 Encoding of Auto-Increment ID

const BASE62_CHARS = '0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz';

function encodeBase62(num) {
  if (num === 0) return BASE62_CHARS[0];
  let result = '';
  while (num > 0) {
    result = BASE62_CHARS[num % 62] + result;
    num = Math.floor(num / 62);
  }
  return result;
}

// 7-character Base62 can encode 62^7 = 3.5 trillion unique URLs
// encodeBase62(1000000) => "4C92"
  • Pros: Simple, sequential, guaranteed unique
  • Cons: Predictable (users can guess next URL), single point of failure for ID generation

Approach 2: MD5/SHA256 Hash + Truncation

import crypto from 'crypto';

function generateShortCode(longUrl) {
  const hash = crypto.createHash('md5').update(longUrl).digest('base64url');
  return hash.substring(0, 7); // Take first 7 characters
}
  • Pros: Deterministic (same URL gives same code), no counter needed
  • Cons: Collision risk -- must handle collisions in the database

Approach 3: Pre-Generated Key Service (Recommended)

// A separate Key Generation Service (KGS) pre-generates unique keys
// and stores them in a database. API servers fetch keys in batches.

class KeyGenerationService {
  constructor(redisClient, batchSize = 1000) {
    this.redis = redisClient;
    this.localKeys = [];
    this.batchSize = batchSize;
  }

  async getKey() {
    if (this.localKeys.length === 0) {
      // Atomically pop a batch of pre-generated keys from Redis
      const keys = await this.redis.spop('available_keys', this.batchSize);
      this.localKeys = keys;
    }
    return this.localKeys.pop();
  }
}
  • Pros: No collisions, fast (keys are ready), not predictable
  • Cons: Requires a separate service, keys need pre-generation

  id            BIGSERIAL PRIMARY KEY,

Step 7: Scaling Strategies

Caching Layer

Since reads massively outnumber writes (100:1), caching is critical:

import Redis from 'ioredis';

const redis = new Redis();
const CACHE_TTL = 60 * 60 * 24; // 24 hours

async function resolveShortUrl(shortCode) {
  // 1. Check cache first
  const cached = await redis.get(`url:${shortCode}`);
  if (cached) return cached;

  // 2. Cache miss -> query database
  const url = await prisma.url.findUnique({
    where: { shortCode },
    select: { longUrl: true, expiresAt: true }
  });

  if (!url || (url.expiresAt && url.expiresAt < new Date())) {
    return null;
  }

  // 3. Populate cache
  await redis.setex(`url:${shortCode}`, CACHE_TTL, url.longUrl);
  return url.longUrl;
}

Database Sharding

At billions of URLs, a single PostgreSQL instance will not suffice. Shard by the first character(s) of the short code:

Sharding Strategy:
  short_code[0] in [0-9, A-F] -> Shard 1
  short_code[0] in [G-V]      -> Shard 2
  short_code[0] in [W-Z, a-m] -> Shard 3
  short_code[0] in [n-z]      -> Shard 4

Alternatively, use consistent hashing on the short code for more even distribution.

Rate Limiting

Prevent abuse by limiting URL creation per IP or user:

import rateLimit from 'express-rate-limit';

const createUrlLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100,                  // 100 URLs per window
  message: { error: 'Too many URLs created, try again later' }
});

router.post('/api/v1/urls', createUrlLimiter, createUrlHandler);

301 vs 302 Redirects

This is a critical trade-off:

  • 301 (Moved Permanently): Browser caches the redirect. Reduces server load but you lose analytics visibility since subsequent visits never hit your server.
  • 302 (Found / Temporary): Browser always hits your server first. More server load but you can track every click.

Most URL shorteners use 302 to preserve analytics accuracy, even though 301 is more semantically correct.

What Happens When a Key Collides?

With the hash approach, two different URLs might produce the same 7-character code. Handle it with retry logic:

async function createShortUrl(longUrl, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    const code = generateShortCode(longUrl + (i > 0 ? String(i) : ''));
    try {
      await prisma.url.create({ data: { shortCode: code, longUrl } });
      return code;
    } catch (err) {
      if (err.code === 'P2002') continue; // Unique constraint violation
      throw err;
    }
  }
  throw new Error('Failed to generate unique short code');
}

Summary

A URL shortener is a read-heavy system where the core challenge is generating unique, compact identifiers efficiently. The key architectural decisions are: choosing an ID generation strategy (pre-generated keys win for scale), caching aggressively with Redis for the redirect hot path, decoupling analytics into an async pipeline, and picking the right redirect status code based on your analytics requirements. At extreme scale, shard the database by short code prefix and use a NoSQL store for the simple key-value lookups that dominate the workload.

Common Problems & Solutions

1 Short codes are predictable and users can enumerate all URLs
Cause: Using sequential auto-increment IDs with Base62 encoding produces sequential short codes (aB3, aB4, aB5...), making it trivial to guess the next URL.
Solution: Use a pre-generated random key service, or hash-based generation. If using auto-increment, apply a bijective scrambling function (e.g., multiply by a large prime modulo a larger prime) before Base62 encoding to destroy sequentiality.
2 Database becomes a bottleneck under heavy redirect traffic
Cause: Every redirect performs a database lookup. At 4,000+ requests per second, the database connection pool is exhausted and queries queue up.
Solution: Add a Redis caching layer in front of the database. Since URL mappings are immutable (the long URL never changes), they are perfect for aggressive caching with long TTLs. Cache hit rates above 90% are typical.
3 Click counts are inaccurate under high concurrency
Cause: Multiple concurrent requests read the same click_count, increment it, and write back, causing lost updates (classic race condition).
Solution: Use atomic database increments (UPDATE urls SET click_count = click_count + 1), or better, decouple click counting into an async pipeline using a message queue and aggregate counts periodically.
