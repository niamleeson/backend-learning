Error Handling Middleware and Global Error Handler

Intuition

Error handling middleware is like the safety net under a trapeze act. No matter where a performer (route handler) falls, the net (error handler) catches them. But you must position the net below all the performers (register after routes), and it must be wide enough to catch everything (handle all error types).

Error Handling Middleware and Global Error Handler

Introduction

Error handling is one of the most critical aspects of building a production API. Poor error handling leads to cryptic responses, information leaks, and crashes. Express provides a specific pattern for error handling middleware that, when used properly, gives you centralized, consistent, and secure error management.

Express Error Handling Basics

Express error-handling middleware has four parameters. This signature is how Express distinguishes error handlers from regular middleware:

import { Request, Response, NextFunction } from 'express';

// Regular middleware: 3 params
app.use((req: Request, res: Response, next: NextFunction) => { ... });

// Error handling middleware: 4 params (err is first)
app.use((err: Error, req: Request, res: Response, next: NextFunction) => { ... });

How Errors Reach the Error Handler

Three ways errors enter the error handling pipeline:

1. Calling next(error) explicitly
   |
2. Throwing in synchronous code
   |
3. Promise rejection in async handlers (Express 5 only;
   Express 4 needs a wrapper)

         |
         v
  Regular middleware skipped
         |
         v
  Error handler middleware
  (err, req, res, next)

1. Calling next(error)

app.get('/users/:id', (req, res, next) => {
  const id = parseInt(req.params.id, 10);
  if (isNaN(id)) {
    const error = new Error('Invalid user ID');
    return next(error); // Jumps to error handler
  }
  res.json({ user: { id } });
});

2. Throwing in Synchronous Code

// Express catches synchronous throws automatically
app.get('/sync-error', (req, res) => {
  throw new Error('Something broke!'); // Caught by Express
});

3. Async Errors (Require Handling)

// WRONG: Express 4 does NOT catch async errors
app.get('/async-error', async (req, res) => {
  const data = await failingOperation(); // Unhandled rejection!
});

// CORRECT: wrap in try-catch
app.get('/async-error', async (req, res, next) => {
  try {
    const data = await failingOperation();
    res.json(data);
  } catch (error) {
    next(error);
  }
});

// BETTER: use a wrapper
const asyncHandler = (fn: Function) =>
  (req: Request, res: Response, next: NextFunction) =>
    Promise.resolve(fn(req, res, next)).catch(next);

app.get('/async-error', asyncHandler(async (req: Request, res: Response) => {
  const data = await failingOperation();
  res.json(data);
}));

Custom Error Classes

Create a hierarchy of error classes for different error types:

// src/utils/ApiError.ts
export class ApiError extends Error {
  public readonly statusCode: number;
  public readonly isOperational: boolean;
  public readonly details?: any;

  constructor(
    statusCode: number,
    message: string,
    options?: { isOperational?: boolean; details?: any }
  ) {
    super(message);
    this.statusCode = statusCode;
    this.isOperational = options?.isOperational ?? true;
    this.details = options?.details;
    this.name = this.constructor.name;

    // Capture stack trace (excludes constructor from trace)
    Error.captureStackTrace(this, this.constructor);
  }
}

export class NotFoundError extends ApiError {
  constructor(resource: string, id?: string | number) {
    const message = id
      ? `${resource} with ID ${id} not found`
      : `${resource} not found`;
    super(404, message);
  }
}

export class ValidationError extends ApiError {
  constructor(details: Record<string, string[]>) {
    super(400, 'Validation failed', { details });
  }
}

export class UnauthorizedError extends ApiError {
  constructor(message = 'Authentication required') {
    super(401, message);
  }
}

export class ForbiddenError extends ApiError {
  constructor(message = 'Insufficient permissions') {
    super(403, message);
  }
}

export class ConflictError extends ApiError {
  constructor(message: string) {
    super(409, message);
  }
}

Using custom errors in route handlers:

import { NotFoundError, ValidationError } from '../utils/ApiError';

app.get('/users/:id', asyncHandler(async (req: Request, res: Response) => {
  const user = await userService.findById(req.params.id);
  if (!user) {
    throw new NotFoundError('User', req.params.id);
  }
  res.json({ user });
}));

app.post('/users', asyncHandler(async (req: Request, res: Response) => {
  const existing = await userService.findByEmail(req.body.email);
  if (existing) {
    throw new ConflictError('Email already registered');
  }
  const user = await userService.create(req.body);
  res.status(201).json({ user });
}));

Building the Global Error Handler

// src/middleware/errorHandler.ts
import { Request, Response, NextFunction } from 'express';
import { ApiError } from '../utils/ApiError';

interface ErrorResponse {
  success: false;
  error: {
    message: string;
    statusCode: number;
    details?: any;
    stack?: string;
  };
}

export function errorHandler(
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction
): void {
  // Log the error
  console.error(`[ERROR] ${req.method} ${req.path}:`, {
    message: err.message,
    stack: err.stack,
    requestId: req.requestId,
  });

  // Known operational errors
  if (err instanceof ApiError) {
    const response: ErrorResponse = {
      success: false,
      error: {
        message: err.message,
        statusCode: err.statusCode,
        ...(err.details && { details: err.details }),
        ...(process.env.NODE_ENV === 'development' && { stack: err.stack }),
      },
    };
    res.status(err.statusCode).json(response);
    return;
  }

  // JSON parse errors (from express.json())
  if (err instanceof SyntaxError && 'body' in err) {
    res.status(400).json({
      success: false,
      error: {
        message: 'Invalid JSON in request body',
        statusCode: 400,
      },
    });
    return;
  }

  // Unknown / unexpected errors
  const statusCode = 500;
  const response: ErrorResponse = {
    success: false,
    error: {
      message: process.env.NODE_ENV === 'production'
        ? 'Internal server error'
        : err.message,
      statusCode,
      ...(process.env.NODE_ENV === 'development' && { stack: err.stack }),
    },
  };

  res.status(statusCode).json(response);
}

Registration Order

The error handler MUST be registered after all routes:

const app = express();

// Middleware
app.use(express.json());
app.use(cors());

// Routes
app.use('/api/users', userRoutes);
app.use('/api/posts', postRoutes);

// 404 handler (for routes that do not match)
app.use((req: Request, res: Response) => {
  res.status(404).json({
    success: false,
    error: { message: `Route ${req.method} ${req.path} not found`, statusCode: 404 }
  });
});

// Global error handler (MUST be last middleware)
app.use(errorHandler);

Handling Unhandled Rejections and Uncaught Exceptions

// src/server.ts
process.on('unhandledRejection', (reason: Error) => {
  console.error('Unhandled Rejection:', reason);
  // In production, you might want to shut down gracefully
});

process.on('uncaughtException', (error: Error) => {
  console.error('Uncaught Exception:', error);
  // MUST exit — the process is in an undefined state
  process.exit(1);
});

Multiple Error Handlers

You can chain error handlers for different concerns:

// Log all errors
function errorLogger(err: Error, req: Request, res: Response, next: NextFunction) {
  console.error(`[${new Date().toISOString()}] Error:`, {
    method: req.method,
    path: req.path,
    error: err.message,
    stack: err.stack,
  });
  next(err); // Pass to next error handler
}

// Send error response
function errorResponder(err: Error, req: Request, res: Response, next: NextFunction) {
  if (err instanceof ApiError) {
    res.status(err.statusCode).json({ success: false, error: { message: err.message } });
    return;
  }
  next(err); // Pass unknown errors to fallback
}

// Fallback for unknown errors
function errorFallback(err: Error, req: Request, res: Response, next: NextFunction) {
  res.status(500).json({ success: false, error: { message: 'Internal server error' } });
}

// Register in order
app.use(errorLogger);
app.use(errorResponder);
app.use(errorFallback);

Security: What NOT to Expose

// WRONG: leaks internal details in production
res.status(500).json({
  error: err.message,              // "ECONNREFUSED 127.0.0.1:5432"
  stack: err.stack,                // Full stack trace
  query: 'SELECT * FROM users...',  // SQL query
});

// CORRECT: sanitize in production
const isProduction = process.env.NODE_ENV === 'production';
res.status(500).json({
  success: false,
  error: {
    message: isProduction ? 'Internal server error' : err.message,
    ...(isProduction ? {} : { stack: err.stack }),
  }
});

Common Pitfalls

Defining Error Handler With Wrong Parameter Count

// WRONG: 3 params - Express treats this as regular middleware
app.use((err, req, res) => {
  res.status(500).json({ error: err.message });
});

// CORRECT: Must have exactly 4 params
app.use((err, req, res, next) => {
  res.status(500).json({ error: err.message });
});

Error Handler Registered Before Routes

// WRONG: error handler never receives route errors
app.use(errorHandler);
app.use('/api', apiRoutes);

// CORRECT: error handler after ALL routes
app.use('/api', apiRoutes);
app.use(errorHandler);

Summary

Express error handling relies on four-parameter middleware functions registered after all routes. Create custom error classes for different HTTP error types, use an async handler wrapper to catch promise rejections, sanitize error output in production, and handle process-level errors for robustness. A well-designed error handling system makes your API predictable, debuggable, and secure.

Exercises

  1. Custom Error Hierarchy

    Create a set of custom error classes: ApiError (base), NotFoundError, ValidationError, UnauthorizedError, ForbiddenError, and ConflictError. Each should set the appropriate HTTP status code. Write a global error handler that uses instanceof to format appropriate responses.

    Hint

    Extend the built-in Error class. Use Error.captureStackTrace(this, this.constructor) for clean stack traces. The error handler should check instanceof ApiError first.

  2. Async Error Wrapper

    Write an asyncHandler utility that wraps async route handlers and catches rejected promises. Apply it to 5 routes that perform async operations. Verify that thrown errors reach the global error handler.

    Hint

    The wrapper returns (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next). Test by making a route throw an error and checking the response.

  3. Environment-Aware Error Responses

    Build an error handler that returns detailed error information (message, stack, request details) in development but returns only safe, generic messages in production. Include request ID in all error responses for correlation.

    Hint

    Check process.env.NODE_ENV to determine detail level. Always include the request ID (from middleware or X-Request-Id header). Never expose stack traces or internal error messages in production.

Common Problems & Solutions

1 Error handler middleware is never triggered
Cause: The error handler has 3 parameters instead of 4, or it is registered before routes, or async errors are not being forwarded with next(error).
Solution: Ensure the error handler has exactly 4 parameters: (err, req, res, next). Register it after all routes and 404 handlers. Wrap async handlers with try-catch and call next(error).
2 Stack trace and internal details exposed to users in production
Cause: The error handler sends err.message and err.stack regardless of environment.
Solution: Check NODE_ENV and only include detailed error information in development. In production, return a generic 'Internal server error' message for 500 errors.
3 Express crashes on unhandled promise rejection instead of returning a 500 error
Cause: An async route handler rejects without try-catch, and Express 4 does not catch async errors.
Solution: Use an asyncHandler wrapper for all async route handlers. Also add process.on('unhandledRejection', ...) as a safety net. Consider upgrading to Express 5 which handles this natively.
