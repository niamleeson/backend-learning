Refresh Tokens and Token Revocation

Intuition

Think of access tokens as day passes to a building and refresh tokens as membership cards. The day pass (access token) gets you in quickly but expires at closing time. When it expires, you show your membership card (refresh token) at the front desk to get a new day pass. If your membership card is reported stolen, the front desk can immediately invalidate it and issue you a new one.

Refresh Tokens and Token Revocation

Introduction

Short-lived access tokens (15 minutes) are secure but inconvenient -- users would need to log in every 15 minutes. Refresh tokens solve this by providing a long-lived token (days/weeks) that can be exchanged for new access tokens. This lesson covers implementing a complete refresh token system with token revocation.

Token Architecture

+-----------------------------------+
|          Access Token             |
|  - Short-lived (15 min)           |
|  - Sent with every request        |
|  - Stateless (JWT)                |
|  - Contains user claims           |
+-----------------------------------+

+-----------------------------------+
|         Refresh Token             |
|  - Long-lived (7 days)            |
|  - Used only to get new tokens    |
|  - Stored in database (stateful)  |
|  - Can be revoked                 |
+-----------------------------------+

Token Flow

1. Login:
   Client -> POST /auth/login { email, password }
   Server -> { accessToken (15m), refreshToken (7d) }

2. API Requests:
   Client -> GET /api/users (Authorization: Bearer <accessToken>)
   Server -> 200 { data }

3. Access Token Expires:
   Client -> GET /api/users (Authorization: Bearer <expired>)
   Server -> 401 Unauthorized

4. Refresh:
   Client -> POST /auth/refresh { refreshToken }
   Server -> { accessToken (new, 15m), refreshToken (new, 7d) }

5. Logout:
   Client -> POST /auth/logout { refreshToken }
   Server -> Revokes refresh token in database

Token Service

// auth/token.service.ts
import { Injectable, UnauthorizedException } from '@nestjs/common';
import { JwtService } from '@nestjs/jwt';
import { ConfigService } from '@nestjs/config';
import { PrismaService } from '../prisma/prisma.service';
import { v4 as uuidv4 } from 'uuid';
import * as crypto from 'crypto';

@Injectable()
export class TokenService {
  constructor(
    private jwtService: JwtService,
    private config: ConfigService,
    private prisma: PrismaService,
  ) {}

  async generateTokens(user: { id: string; email: string; roles: string[] }) {
    const payload = {
      sub: user.id,
      email: user.email,
      roles: user.roles,
    };

    const [accessToken, refreshToken] = await Promise.all([
      this.jwtService.signAsync(payload, {
        secret: this.config.get('JWT_SECRET'),
        expiresIn: '15m',
      }),
      this.createRefreshToken(user.id),
    ]);

    return { accessToken, refreshToken };
  }

  private async createRefreshToken(userId: string): Promise<string> {
    const token = crypto.randomBytes(64).toString('hex');

    await this.prisma.refreshToken.create({
      data: {
        token: this.hashToken(token),
        userId,
        expiresAt: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000), // 7 days
      },
    });

    return token;
  }

  async refreshTokens(refreshToken: string, userAgent?: string) {
    const hashedToken = this.hashToken(refreshToken);

    const storedToken = await this.prisma.refreshToken.findUnique({
      where: { token: hashedToken },
      include: { user: true },
    });

    // Validation checks
    if (!storedToken) {
      throw new UnauthorizedException('Invalid refresh token');
    }

    if (storedToken.revokedAt) {
      // Token reuse detected! Revoke all tokens for this user
      await this.revokeAllUserTokens(storedToken.userId);
      throw new UnauthorizedException(
        'Refresh token has been revoked. All sessions have been terminated.',
      );
    }

    if (storedToken.expiresAt < new Date()) {
      throw new UnauthorizedException('Refresh token has expired');
    }

    // Rotate: revoke old token, create new pair
    const user = storedToken.user;

    // Revoke the current refresh token
    await this.prisma.refreshToken.update({
      where: { id: storedToken.id },
      data: { revokedAt: new Date() },
    });

    // Generate new token pair
    return this.generateTokens({
      id: user.id,
      email: user.email,
      roles: user.roles,
    });
  }

  async revokeToken(refreshToken: string): Promise<void> {
    const hashedToken = this.hashToken(refreshToken);

    await this.prisma.refreshToken.updateMany({
      where: { token: hashedToken, revokedAt: null },
      data: { revokedAt: new Date() },
    });
  }

  async revokeAllUserTokens(userId: string): Promise<void> {
    await this.prisma.refreshToken.updateMany({
      where: { userId, revokedAt: null },
      data: { revokedAt: new Date() },
    });
  }

  private hashToken(token: string): string {
    return crypto.createHash('sha256').update(token).digest('hex');
  }

  // Cleanup expired tokens (run as a cron job)
  async cleanupExpiredTokens(): Promise<number> {
    const result = await this.prisma.refreshToken.deleteMany({
      where: {
        OR: [
          { expiresAt: { lt: new Date() } },
          {
            revokedAt: { not: null },
            revokedAt: { lt: new Date(Date.now() - 30 * 24 * 60 * 60 * 1000) },
          },
        ],
      },
    });
    return result.count;
  }
}

Updated Auth Service

// auth/auth.service.ts
@Injectable()
export class AuthService {
  constructor(
    private usersService: UsersService,
    private tokenService: TokenService,
  ) {}

  async login(user: any) {
    const tokens = await this.tokenService.generateTokens(user);
    return { ...tokens, user: { id: user.id, email: user.email, name: user.name } };
  }

  async refresh(refreshToken: string) {
    return this.tokenService.refreshTokens(refreshToken);
  }

  async logout(refreshToken: string) {
    await this.tokenService.revokeToken(refreshToken);
    return { message: 'Logged out successfully' };
  }

  async logoutAll(userId: string) {
    await this.tokenService.revokeAllUserTokens(userId);
    return { message: 'All sessions terminated' };
  }
}

Token Rotation Security

Refresh token rotation means every time a refresh token is used, a new one is issued and the old one is revoked. This limits the window of vulnerability if a token is stolen.

Legitimate Use:
  RT1 -> revoke RT1, issue RT2
  RT2 -> revoke RT2, issue RT3

Token Theft (attacker uses stolen RT1 after user used it):
  RT1 (already revoked) -> DETECTED! Revoke ALL tokens for user

This is why we check revokedAt -- reuse of a revoked token indicates theft.

Access Token Revocation (Blacklisting)

For immediate access token revocation (e.g., user banned), use a blacklist in Redis:

@Injectable()
export class TokenBlacklistService {
  constructor(@Inject('REDIS_CLIENT') private redis: Redis) {}

  async blacklist(token: string, expiresIn: number): Promise<void> {
    const jti = this.extractJti(token);
    await this.redis.setex(`blacklist:${jti}`, expiresIn, '1');
  }

  async isBlacklisted(token: string): Promise<boolean> {
    const jti = this.extractJti(token);
    const result = await this.redis.get(`blacklist:${jti}`);
    return result !== null;
  }

  private extractJti(token: string): string {
    const payload = JSON.parse(
      Buffer.from(token.split('.')[1], 'base64').toString(),
    );
    return payload.jti;
  }
}

Add to JwtStrategy:

async validate(payload: JwtPayload, done: Function) {
  const isBlacklisted = await this.blacklistService.isBlacklisted(
    payload.jti,
  );
  if (isBlacklisted) {
    throw new UnauthorizedException('Token has been revoked');
  }
  return { id: payload.sub, email: payload.email, roles: payload.roles };
}

Cleanup Cron Job

import { Injectable } from '@nestjs/common';
import { Cron, CronExpression } from '@nestjs/schedule';

@Injectable()
export class TokenCleanupService {
  constructor(private tokenService: TokenService) {}

  @Cron(CronExpression.EVERY_DAY_AT_MIDNIGHT)
  async handleCleanup() {
    const count = await this.tokenService.cleanupExpiredTokens();
    console.log(`Cleaned up ${count} expired refresh tokens`);
  }
}

Cookie-Based Token Storage

For web applications, httpOnly cookies are more secure than localStorage:

@Post('login')
@HttpCode(HttpStatus.OK)
async login(@Request() req: any, @Res({ passthrough: true }) res: Response) {
  const result = await this.authService.login(req.user);

  // Set refresh token as httpOnly cookie
  res.cookie('refresh_token', result.refreshToken, {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'strict',
    path: '/auth/refresh',
    maxAge: 7 * 24 * 60 * 60 * 1000, // 7 days
  });

  // Only return access token in response body
  return {
    accessToken: result.accessToken,
    user: result.user,
  };
}

Common Pitfalls

  1. Storing refresh tokens as plain text -- Always hash refresh tokens in the database. If the database is compromised, plain tokens grant access.
  2. No token rotation -- Without rotation, a stolen refresh token can be used indefinitely until it expires.
  3. Not detecting token reuse -- When a revoked token is reused, it indicates theft. Revoke all user tokens as a security measure.
  4. Refresh tokens that never expire -- Always set an expiration. 7-30 days is typical. Require re-authentication after that.

Exercises

  1. Implement Refresh Token Rotation

    Build a complete refresh token system with: token generation on login, token refresh endpoint with rotation (old token revoked, new pair issued), token reuse detection (revoke all user tokens if a revoked token is reused), and logout endpoint that revokes the refresh token.

    Hint

    Store refresh tokens in the database with a hashed token column, userId, expiresAt, and revokedAt. On refresh, check if the token was already revoked. If so, revoke ALL tokens for that user (security measure).

  2. Add Access Token Blacklisting

    Implement a Redis-based access token blacklist. When a user logs out or is banned, add their current access token to the blacklist. Modify JwtStrategy to check the blacklist on every request. Set TTL to match the token's remaining lifetime.

    Hint

    Calculate remaining TTL from the token's exp claim: exp - Math.floor(Date.now() / 1000). Store in Redis with that TTL so it auto-cleans.

  3. Session Management Dashboard

    Create endpoints for users to: list all their active sessions (from refresh tokens with user-agent/IP), revoke a specific session, and revoke all other sessions (keep current). Include last-used timestamps.

    Hint

    Store userAgent and ipAddress when creating refresh tokens. Add a lastUsedAt column updated on each refresh. Create a GET /auth/sessions endpoint that returns active (non-revoked, non-expired) tokens.

Common Problems & Solutions

1 Refresh token reuse detected falsely
Cause: The client retries a failed refresh request, but the first attempt succeeded and rotated the token. The retry uses the now-revoked token.
Solution: Implement a short grace period (e.g., 10 seconds) where a recently revoked token is still accepted. Or ensure the client handles the refresh response atomically and doesn't retry.
2 Refresh tokens table grows indefinitely
Cause: Expired and revoked tokens are never cleaned up from the database.
Solution: Implement a scheduled job (using @nestjs/schedule) that runs daily to delete tokens where expiresAt < now() or where revokedAt is more than 30 days ago.
3 User can't be logged out immediately (access token still valid)
Cause: JWTs are stateless -- once issued, they can't be invalidated until they expire. The refresh token is revoked, but the access token is still valid for up to 15 minutes.
Solution: Implement access token blacklisting with Redis. On logout, add the access token's JTI to a Redis set with TTL matching the token's remaining lifetime.
