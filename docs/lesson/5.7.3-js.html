Structuring a Large Node.js Backend Project

Intuition

Structuring a large project is like organizing a city. The anti-pattern is like putting all restaurants on one street, all houses on another, and all offices on a third -- everyone has to commute everywhere. Feature-based structure is like neighborhoods where each one has its own shops, homes, and offices. People mostly stay in their neighborhood (module) and only travel to others when necessary.

Structuring a Large Node.js Backend Project

As Node.js backends grow beyond a few endpoints, ad-hoc file organization becomes unmanageable. This lesson covers proven project structures for scaling from startup prototype to enterprise application.

Shared Module Patterns

Utility Functions

// shared/utils/pagination.ts
export interface PaginationOptions {
  page: number;
  limit: number;
}

export interface PaginatedResult<T> {
  data: T[];
  meta: {
    page: number;
    limit: number;
    total: number;
    totalPages: number;
    hasNext: boolean;
    hasPrev: boolean;
  };
}

export function paginate<T>(
  items: T[],
  total: number,
  options: PaginationOptions
): PaginatedResult<T> {
  const totalPages = Math.ceil(total / options.limit);
  return {
    data: items,
    meta: {
      page: options.page,
      limit: options.limit,
      total,
      totalPages,
      hasNext: options.page < totalPages,
      hasPrev: options.page > 1,
    },
  };
}

Configuration Management

// config/app.config.ts
import { z } from 'zod';

const envSchema = z.object({
  NODE_ENV: z.enum(['development', 'staging', 'production']).default('development'),
  PORT: z.coerce.number().default(3000),
  DATABASE_URL: z.string().url(),
  REDIS_URL: z.string().url(),
  JWT_SECRET: z.string().min(32),
  JWT_EXPIRATION: z.string().default('15m'),
  CORS_ORIGINS: z.string().transform(s => s.split(',')),
  LOG_LEVEL: z.enum(['debug', 'info', 'warn', 'error']).default('info'),
});

export type AppConfig = z.infer<typeof envSchema>;

export function loadConfig(): AppConfig {
  const result = envSchema.safeParse(process.env);
  if (!result.success) {
    console.error('Invalid environment configuration:');
    console.error(result.error.format());
    process.exit(1);
  }
  return result.data;
}

Module Dependency Rules

+--------+     +--------+     +--------+
| Order  |---->| User   |     | Post   |
| Module |     | Module |<----| Module |
+--------+     +--------+     +--------+
    |               |               |
    v               v               v
+----------------------------------------+
|           Shared Module                |
| (database, auth, utils, errors)        |
+----------------------------------------+

Rules:
1. Modules can depend on Shared
2. Modules can depend on other modules' exported services
3. No circular dependencies between modules
4. Shared depends on nothing application-specific

Key Principles

  1. Colocation: Keep related files together (tests next to code)
  2. Feature isolation: Each module is self-contained
  3. Clear boundaries: Modules expose explicit public APIs via barrel exports
  4. Dependency direction: Dependencies flow toward the domain core
  5. Configuration validation: Validate env vars at startup, not at runtime
  6. Consistent naming: Use a naming convention and stick to it
  7. Index files for public API: Each module has an index.ts that exports only what other modules need

Exercises

  1. Restructure a Flat Project

    Take a project with everything in flat folders (controllers/, services/, models/) and restructure it into feature-based modules. Each module should be self-contained with its own controller, service, repository, DTOs, and tests. Create barrel exports for each module.

    Hint

    Create a modules/ directory. Move each feature's files into its own folder. Add index.ts barrel exports. Update import paths.

  2. Validate Configuration at Startup

    Create a configuration module using Zod that validates all required environment variables at application startup. If any are missing or invalid, the app should print a clear error message and exit immediately.

    Hint

    Use z.object() with all env vars. Call envSchema.safeParse(process.env) in a loadConfig() function called from main.ts.

  3. Enforce Module Boundaries

    Set up ESLint rules (or TypeScript project references) that prevent circular dependencies between modules. Module A should not be able to import from Module B's internal files -- only from its barrel export.

    Hint

    Use eslint-plugin-import with no-restricted-imports rule, or use TypeScript project references with composite: true.

  4. Create a Monorepo Template

    Set up a monorepo using npm workspaces (or pnpm/yarn workspaces) with three packages: shared (types and utils), api (Express or NestJS app), and worker (background job processor). The api and worker packages should import from shared.

    Hint

    Use 'workspaces' in root package.json. Reference shared package by name in api and worker dependencies.

Common Problems & Solutions

1 Circular dependencies between modules
Cause: Module A imports from Module B, and Module B imports from Module A.
Solution: Extract the shared dependency into a third module or use events for loose coupling. If UserModule and OrderModule both need each other, extract shared types into a Shared module.
2 The shared/ folder becomes a dumping ground
Cause: Developers put anything that does not clearly belong to a module into shared/.
Solution: Be strict about what goes in shared/: only truly cross-cutting concerns (auth, logging, database, error handling). If something is used by only 2 modules, consider keeping it in one and importing from the other.
3 Import paths are extremely long and fragile
Cause: Deep nesting results in imports like '../../../shared/utils/pagination.ts'.
Solution: Configure TypeScript path aliases in tsconfig.json: { "paths": { "@shared/*": ["src/shared/*"], "@modules/*": ["src/modules/*"] } }. Use these aliases in imports.
