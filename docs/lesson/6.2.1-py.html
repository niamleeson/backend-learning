GraphQL Fundamentals: Schema, Queries, and Mutations

Intuition

REST is like a set menu at a restaurant -- you get what the chef prepared. GraphQL is like a buffet -- you pick exactly what you want from what is available. The schema is the menu board showing everything on offer, and your query is your plate selection.

GraphQL Fundamentals: Schema, Queries, and Mutations

GraphQL is a query language for APIs that lets clients request exactly the data they need. Unlike REST where the server decides the response shape, GraphQL puts clients in control.

Schema Definition

import strawberry
from typing import Optional

@strawberry.type
class User:
    id: int
    name: str
    email: str
    posts: list["Post"]

@strawberry.type
class Post:
    id: int
    title: str
    content: str
    author: User

@strawberry.type
class Query:
    @strawberry.field
    def user(self, id: int) -> Optional[User]:
        return get_user_by_id(id)

    @strawberry.field
    def posts(self, limit: int = 10) -> list[Post]:
        return get_posts(limit=limit)

@strawberry.type
class Mutation:
    @strawberry.mutation
    def create_post(self, title: str, content: str, author_id: int) -> Post:
        return create_new_post(title, content, author_id)

schema = strawberry.Schema(query=Query, mutation=Mutation)

Client Query Examples

# Client requests only what it needs
query = """
{
  user(id: 1) {
    name
    posts {
      title
    }
  }
}
"""

# Mutation
mutation = """
mutation {
  createPost(title: "Hello", content: "World", authorId: 1) {
    id
    title
  }
}
"""

Common Mistakes

  • Treating GraphQL like REST: creating one query per resource instead of leveraging nested resolution
  • Not defining input types for complex mutations
  • Ignoring query complexity and allowing unbounded nested queries

Key Takeaways

  • GraphQL uses a single endpoint with a strongly typed schema
  • Queries read data; mutations modify data
  • Clients request exactly the fields they need, solving over-fetching and under-fetching

Exercises

  1. Design a GraphQL Schema

    Design a GraphQL schema for a bookstore with Book, Author, and Review types. Books have authors and reviews. Include queries to fetch a single book, list books by author, and a mutation to add a review.

    Hint

    Define the types with their relationships first. Use strawberry.field for queries that accept filter arguments like author_id.

  2. Query vs Mutation Separation

    Build a small task management API with GraphQL. Implement queries for listing tasks (with filtering by status) and mutations for creating, updating, and deleting tasks. Follow the CQRS principle.

    Hint

    Keep Query resolvers read-only and Mutation resolvers for writes. Use strawberry.enum for task status (TODO, IN_PROGRESS, DONE).

Common Problems & Solutions

1 Over-fetching still occurs with GraphQL
Symptoms: ["Database queries SELECT * even when client requests only 'name'", 'Slow responses despite clients requesting minimal fields']
Cause: Resolvers fetch all fields from the database regardless of what the client requested.
Solution: Use the 'info' parameter in resolvers to inspect which fields were requested, and query only those columns from the database.
2 Deeply nested queries cause server overload
Cause: No query depth or complexity limits, allowing clients to send queries like user { posts { author { posts { author ... } } } }.
Solution: Implement query depth limiting and complexity analysis. Strawberry supports extensions for this. Set a maximum depth of 5-7 levels.
