Sharding Strategies: Hash, Range, and Directory

Intuition

Sharding is like splitting a library's book collection across multiple buildings. Hash sharding assigns books by a formula (author name hash) — evenly distributed but you cannot browse alphabetically. Range sharding puts A-H in Building 1, I-P in Building 2, Q-Z in Building 3 — easy to browse a range but some letters have more books. Directory sharding has a central catalog telling you which building has which book — flexible but you always need to check the catalog first.

Sharding Strategies: Hash, Range, and Directory

Introduction

When a single database server cannot handle your data volume or write throughput — even with read replicas — you need to split data across multiple database servers. This is sharding (also called horizontal partitioning). Each shard holds a subset of the data, and the application routes queries to the correct shard.

When Do You Need Sharding?

Scaling Path:
1. Single DB            → Handles up to ~10K queries/sec
2. + Read Replicas      → Handles read-heavy loads
3. + Caching (Redis)    → Reduces DB load 80-95%
4. + Connection Pooling → Better connection management
5. → SHARDING          → When writes are the bottleneck
                          or data exceeds single-server capacity

Do NOT shard until you have exhausted options 1-4. Sharding adds enormous complexity.

Sharding Architecture

┌──────────┐
│   App    │
│  Server  │
└────┬─────┘
     │ Shard Key: user_id = 42
     │ Route to: Shard 2 (42 % 3 = 0... but hash gives 2)
     │
     ▼
┌──────────┐   ┌──────────┐   ┌──────────┐
│ Shard 0  │   │ Shard 1  │   │ Shard 2  │
│ Users    │   │ Users    │   │ Users    │
│ 1-33%    │   │ 34-66%   │   │ 67-100%  │
│ of data  │   │ of data  │   │ of data  │
└──────────┘   └──────────┘   └──────────┘

Hash-Based Sharding

The shard key is hashed and the result determines which shard stores the data.

import crypto from 'node:crypto';

class HashShardRouter {
  constructor(shardCount, shardConnections) {
    this.shardCount = shardCount;
    this.shards = shardConnections; // Array of PrismaClient instances
  }

  getShard(shardKey) {
    const hash = crypto
      .createHash('md5')
      .update(String(shardKey))
      .digest('hex');
    const shardIndex = parseInt(hash.substring(0, 8), 16) % this.shardCount;
    return this.shards[shardIndex];
  }

  // Simple modulo (less even distribution but simpler)
  getShardSimple(numericKey) {
    return this.shards[numericKey % this.shardCount];
  }
}

// Setup
import { PrismaClient } from '@prisma/client';

const shards = [
  new PrismaClient({ datasources: { db: { url: process.env.SHARD_0_URL } } }),
  new PrismaClient({ datasources: { db: { url: process.env.SHARD_1_URL } } }),
  new PrismaClient({ datasources: { db: { url: process.env.SHARD_2_URL } } }),
];

const router = new HashShardRouter(3, shards);

// Usage
async function getUser(userId) {
  const shard = router.getShard(userId);
  return shard.user.findUnique({ where: { id: userId } });
}

async function createUser(data) {
  const userId = generateUserId();
  const shard = router.getShard(userId);
  return shard.user.create({ data: { ...data, id: userId } });
}

Pros: Even data distribution, simple routing logic. Cons: Range queries across shards are expensive. Adding/removing shards requires data migration.

Range-Based Sharding

Data is split by ranges of the shard key.

┌────────────────────────────────────────────┐
│ Shard Key Range  │ Shard  │ Example        │
├──────────────────┼────────┼────────────────┤
│ user_id 1-1M     │ Shard 0│ Early users    │
│ user_id 1M-2M    │ Shard 1│ Growth phase   │
│ user_id 2M-3M    │ Shard 2│ Recent users   │
├──────────────────┼────────┼────────────────┤
│ date < 2024      │ Shard 0│ Historical     │
│ date 2024-2025   │ Shard 1│ Recent         │
│ date >= 2025     │ Shard 2│ Current        │
└────────────────────────────────────────────┘

class RangeShardRouter {
  constructor(ranges) {
    // ranges: [{ min, max, shard }]
    this.ranges = ranges.sort((a, b) => a.min - b.min);
  }

  getShard(shardKey) {
    for (const range of this.ranges) {
      if (shardKey >= range.min && shardKey < range.max) {
        return range.shard;
      }
    }
    // Default to last shard for overflow
    return this.ranges[this.ranges.length - 1].shard;
  }
}

const router = new RangeShardRouter([
  { min: 0, max: 1000000, shard: shards[0] },
  { min: 1000000, max: 2000000, shard: shards[1] },
  { min: 2000000, max: Infinity, shard: shards[2] },
]);

// Time-based sharding
class TimeRangeShardRouter {
  constructor(shards) {
    this.shards = shards;
  }

  getShard(timestamp) {
    const date = new Date(timestamp);
    const year = date.getFullYear();

    if (year < 2024) return this.shards[0]; // Archive
    if (year < 2025) return this.shards[1]; // Previous year
    return this.shards[2];                   // Current
  }
}

Pros: Range queries within a shard are efficient. Easy to understand and implement. Cons: Uneven distribution (hotspots on recent data). New shards fill faster than old ones.

Directory-Based Sharding

A lookup table maps each shard key to its shard location.

┌──────────────────────────────────┐
│ Directory Service (Redis/DB)     │
│                                  │
│ user:1    → shard-0              │
│ user:2    → shard-1              │
│ user:3    → shard-0              │
│ user:4    → shard-2              │
│ tenant:A  → shard-0              │
│ tenant:B  → shard-1              │
└──────────────────────────────────┘

class DirectoryShardRouter {
  constructor(redis, shards) {
    this.redis = redis;
    this.shards = shards;
    this.defaultShard = shards[0];
  }

  async getShard(entityType, entityId) {
    const key = `shard_map:${entityType}:${entityId}`;
    const shardId = await this.redis.get(key);

    if (shardId !== null) {
      return this.shards[parseInt(shardId)];
    }

    // Assign to least-loaded shard
    const shardIndex = await this.assignShard(entityType, entityId);
    return this.shards[shardIndex];
  }

  async assignShard(entityType, entityId) {
    // Find shard with least data
    const counts = await Promise.all(
      this.shards.map((shard, i) =>
        this.redis.get(`shard_count:${i}`).then(c => ({ index: i, count: parseInt(c) || 0 }))
      )
    );

    const leastLoaded = counts.sort((a, b) => a.count - b.count)[0];

    await this.redis.pipeline()
      .set(`shard_map:${entityType}:${entityId}`, leastLoaded.index)
      .incr(`shard_count:${leastLoaded.index}`)
      .exec();

    return leastLoaded.index;
  }

  // Migrate entity to different shard
  async migrateShard(entityType, entityId, targetShardIndex) {
    const key = `shard_map:${entityType}:${entityId}`;
    const currentShardIndex = parseInt(await this.redis.get(key));

    // Copy data from source to target
    const sourceShard = this.shards[currentShardIndex];
    const targetShard = this.shards[targetShardIndex];

    const data = await sourceShard[entityType].findUnique({
      where: { id: entityId },
    });

    await targetShard[entityType].create({ data });
    await sourceShard[entityType].delete({ where: { id: entityId } });

    // Update directory
    await this.redis.set(key, targetShardIndex);
  }
}

Pros: Flexible placement, easy rebalancing, supports multi-tenant. Cons: Directory is a single point of failure. Extra lookup on every query.

Cross-Shard Queries

The biggest challenge with sharding — queries that span multiple shards:

class ShardedUserService {
  constructor(router) {
    this.router = router;
  }

  // Single-shard query (fast)
  async getUser(userId) {
    const shard = this.router.getShard(userId);
    return shard.user.findUnique({ where: { id: userId } });
  }

  // Cross-shard query (slow but sometimes necessary)
  async searchUsers(query) {
    // Query all shards in parallel
    const results = await Promise.all(
      this.router.shards.map(shard =>
        shard.user.findMany({
          where: { name: { contains: query, mode: 'insensitive' } },
          take: 20,
        })
      )
    );

    // Merge and sort results
    return results
      .flat()
      .sort((a, b) => a.name.localeCompare(b.name))
      .slice(0, 20);
  }

  // Cross-shard aggregation
  async countAllUsers() {
    const counts = await Promise.all(
      this.router.shards.map(shard => shard.user.count())
    );
    return counts.reduce((sum, count) => sum + count, 0);
  }

  // Cross-shard pagination (complex!)
  async listUsers(page, limit) {
    // Fetch extra from each shard to ensure correct global ordering
    const perShardLimit = limit * 2;
    const perShardSkip = 0; // Cannot simply divide skip by shard count

    const results = await Promise.all(
      this.router.shards.map(shard =>
        shard.user.findMany({
          take: perShardLimit,
          orderBy: { createdAt: 'desc' },
        })
      )
    );

    return results
      .flat()
      .sort((a, b) => b.createdAt - a.createdAt)
      .slice((page - 1) * limit, page * limit);
  }
}

Choosing a Shard Key

The shard key determines everything about your sharding strategy:

┌──────────────────────────────────────────────────────────┐
│ Shard Key Criteria:                                      │
│                                                          │
│ 1. High cardinality (many unique values)                 │
│ 2. Even distribution (no hotspots)                       │
│ 3. Query isolation (most queries hit one shard)          │
│ 4. Stable (doesn't change for an entity)                 │
│                                                          │
│ Good shard keys:                                         │
│   - user_id (for user-centric apps)                      │
│   - tenant_id (for multi-tenant SaaS)                    │
│   - geographic region (for geo-distributed apps)         │
│                                                          │
│ Bad shard keys:                                          │
│   - created_at (hot shard for recent data)               │
│   - status (few values, uneven distribution)             │
│   - country (popular countries create hotspots)           │
└──────────────────────────────────────────────────────────┘

Common Pitfalls

  1. Sharding too early: Sharding adds massive complexity. Exhaust vertical scaling, read replicas, and caching first.
  2. Wrong shard key: Changing the shard key later requires migrating all data. Choose carefully upfront.
  3. Cross-shard JOINs: Sharding makes JOINs across shards extremely expensive. Denormalize or use eventual consistency.
  4. Unbalanced shards: One shard gets much more data/traffic than others. Monitor shard sizes and rebalance.

Summary

Hash-based sharding provides even distribution but makes range queries difficult. Range-based sharding supports efficient range queries but can create hotspots. Directory-based sharding offers the most flexibility but adds a lookup overhead. Choose your shard key based on query patterns and data distribution. Avoid sharding until you have exhausted simpler scaling options.

Common Problems & Solutions

1 One shard receiving significantly more traffic than others
Cause: The shard key has low cardinality or skewed distribution (e.g., sharding by country where 80% of users are in one country).
Solution: Choose a high-cardinality shard key like user_id. Use consistent hashing for better distribution. Monitor per-shard metrics and rebalance if needed.
2 Adding a new shard requires migrating massive amounts of data
Cause: Hash-based sharding with simple modulo: changing shard count (N to N+1) changes the shard assignment for most keys.
Solution: Use consistent hashing to minimize key redistribution. Or use directory-based sharding where you can assign new entities to new shards without moving existing data.
3 Cross-shard queries are extremely slow
Cause: Queries that need data from multiple shards require scatter-gather: querying all shards and merging results in the application.
Solution: Design the shard key so most queries only hit one shard. Denormalize data to avoid cross-shard joins. Use a search index (Elasticsearch) for cross-shard search queries.
