Kubernetes Introduction: Pods, Services, Deployments

Intuition

Kubernetes is like a fleet management system for shipping containers. You tell it 'I need 3 copies of my API running at all times' (Deployment), and K8s handles the rest: placing containers on ships (nodes), replacing ones that fall overboard (pod restart), balancing cargo (load distribution), and routing delivery trucks to the right dock (Services/Ingress). You describe the desired state, and Kubernetes continuously works to match reality to your description.

Kubernetes Introduction: Pods, Services, Deployments

What is Kubernetes?

Kubernetes (K8s) is a container orchestration platform that automates deployment, scaling, and management of containerized applications. It is the industry standard for running containers in production at scale.

When to Use Kubernetes

Use Kubernetes when:                 Skip Kubernetes when:
────────────────────                 ─────────────────────
- Multiple microservices             - Single monolith
- Need auto-scaling                  - Small team (< 5 devs)
- Multi-region deployment            - Simple deployment needs
- High availability required         - Budget constraints
- Team has K8s expertise             - PM2 + Nginx is sufficient
- 10+ containers to manage           - < 5 containers

Core Concepts

Pods

A Pod is the smallest deployable unit -- one or more containers that share networking and storage.

# pod.yaml (rarely created directly, use Deployments instead)
apiVersion: v1
kind: Pod
metadata:
  name: myapp
  labels:
    app: myapp
spec:
  containers:
    - name: api
      image: myorg/myapp:1.0.0
      ports:
        - containerPort: 3000
      env:
        - name: NODE_ENV
          value: "production"
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: myapp-secrets
              key: database-url
      resources:
        requests:
          memory: "128Mi"
          cpu: "100m"
        limits:
          memory: "512Mi"
          cpu: "500m"
      livenessProbe:
        httpGet:
          path: /health
          port: 3000
        initialDelaySeconds: 10
        periodSeconds: 15
      readinessProbe:
        httpGet:
          path: /ready
          port: 3000
        initialDelaySeconds: 5
        periodSeconds: 10

Deployments

Deployments manage Pod replicas with rolling updates and rollbacks.

# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-api
  labels:
    app: myapp
    component: api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      component: api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1        # At most 1 pod down during update
      maxSurge: 1              # At most 1 extra pod during update
  template:
    metadata:
      labels:
        app: myapp
        component: api
    spec:
      containers:
        - name: api
          image: myorg/myapp:1.0.0
          ports:
            - containerPort: 3000
          env:
            - name: NODE_ENV
              value: "production"
            - name: PORT
              value: "3000"
          envFrom:
            - configMapRef:
                name: myapp-config
            - secretRef:
                name: myapp-secrets
          resources:
            requests:
              memory: "256Mi"
              cpu: "200m"
            limits:
              memory: "512Mi"
              cpu: "500m"
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 15
            periodSeconds: 20
          readinessProbe:
            httpGet:
              path: /ready
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 10
      imagePullSecrets:
        - name: registry-credentials

Services

Services provide stable networking for Pods (which have dynamic IPs).

# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-api
spec:
  selector:
    app: myapp
    component: api
  type: ClusterIP            # Internal only (default)
  ports:
    - port: 80               # Service port
      targetPort: 3000       # Pod port
      protocol: TCP

---
# External access via LoadBalancer
apiVersion: v1
kind: Service
metadata:
  name: myapp-api-public
spec:
  selector:
    app: myapp
    component: api
  type: LoadBalancer          # Creates cloud load balancer
  ports:
    - port: 80
      targetPort: 3000

Service Types

ClusterIP (default):
  Internal only. Other pods can reach it by service name.
  myapp-api:80 -> pods:3000

NodePort:
  Exposes on each node's IP at a static port.
  <NodeIP>:30080 -> pods:3000

LoadBalancer:
  Creates external cloud load balancer.
  external-ip:80 -> pods:3000

ExternalName:
  Maps to a DNS name (CNAME).
  mydb -> rds-endpoint.amazonaws.com

---
# secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: myapp-secrets
type: Opaque
stringData:                    # Plain text (K8s encodes to base64)
  database-url: "postgresql://user:pass@postgres:5432/mydb"
  jwt-secret: "super-secret-key"
  redis-url: "redis://redis:6379"

Ingress (HTTP Routing)

# ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - api.myapp.com
      secretName: myapp-tls
  rules:
    - host: api.myapp.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: myapp-api
                port:
                  number: 80

Horizontal Pod Autoscaler

# hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80

# Cluster info
kubectl cluster-info
kubectl get nodes

# Deployments
kubectl apply -f deployment.yaml
kubectl get deployments

Common Pitfalls

  1. Not setting resource requests and limits - Without limits, a single pod can consume all node resources. Without requests, the scheduler cannot make good placement decisions.
  2. Missing health checks - Without liveness and readiness probes, K8s cannot detect unhealthy pods or know when a pod is ready for traffic.
  3. Storing secrets in ConfigMaps - ConfigMaps are not encrypted. Use Secrets (still base64, but can be encrypted at rest) or external secret managers.
  4. Not using namespaces - Without namespaces, all resources are in 'default'. Use namespaces to isolate environments (dev, staging, production).
  5. Over-engineering - Kubernetes adds complexity. If you have 1-3 services, PM2 + Nginx or Docker Compose may be simpler and more appropriate.

Exercises

  1. Deploy Node.js to Minikube

    Set up a local Kubernetes cluster with Minikube. Create a Deployment for a Node.js application with 3 replicas, a ClusterIP Service, and an Ingress. Verify you can access the application through the Ingress.

    Hint

    Start with 'minikube start'. Enable the ingress addon: 'minikube addons enable ingress'. Build your Docker image inside Minikube: 'eval $(minikube docker-env) && docker build -t myapp:latest .'

  2. Rolling Update and Rollback

    Deploy version 1.0 of your application, then update to version 2.0 using a rolling update strategy. Monitor the rollout with kubectl rollout status. Then simulate a bad deployment and practice rolling back.

    Hint

    Update the image tag in the deployment: 'kubectl set image deployment/myapp api=myapp:2.0'. Rollback with: 'kubectl rollout undo deployment/myapp'. View history: 'kubectl rollout history deployment/myapp'.

  3. Complete Application Stack

    Create Kubernetes manifests for a full application: Node.js API Deployment + Service, PostgreSQL StatefulSet + Service, Redis Deployment + Service, ConfigMap for app config, Secrets for database credentials, and an HPA for the API.

    Hint

    Use StatefulSet for PostgreSQL (needs stable storage). Use PersistentVolumeClaim for database storage. Use envFrom to inject ConfigMap and Secret into pods.

Common Problems & Solutions

1 Pod stuck in CrashLoopBackOff
Cause: The container starts and then crashes repeatedly. Usually a misconfiguration, missing environment variable, or application error.
Solution: Check logs: `kubectl logs pod-name --previous` (shows logs from the crashed container). Check events: `kubectl describe pod pod-name`. Common causes: wrong CMD/ENTRYPOINT, missing env vars, failed database connection. Fix the issue and redeploy.
2 Pod stuck in Pending state
Cause: Kubernetes cannot schedule the pod. Usually insufficient resources (CPU/memory) or no matching nodes.
Solution: Check events: `kubectl describe pod pod-name`. Look for 'Insufficient cpu' or 'Insufficient memory'. Reduce resource requests, add nodes, or use cluster autoscaler. Check node taints and pod tolerations.
3 Service cannot reach pods (connection refused)
Cause: The service selector does not match the pod labels, or the targetPort does not match the container port.
Solution: Verify labels match: `kubectl get pods --show-labels`. Check the service endpoints: `kubectl get endpoints service-name`. If endpoints are empty, the selector is wrong. Verify targetPort matches the container's port.
