Design a News Feed (Twitter/Instagram)

Intuition

A news feed is like a personalized newspaper. For regular columnists (normal users), their articles are pre-printed and bundled into each subscriber's paper at the printing press (fan-out on write). For headline news from major figures (celebrities), the articles are pulled from the wire service and inserted into your paper right when you pick it up (fan-out on read). The editor (ranking algorithm) decides the front-page order based on what you find most interesting.

Design a News Feed (Twitter/Instagram)

Introduction

A news feed aggregates and displays content from people and pages a user follows, ranked by relevance or sorted by time. Twitter's home timeline, Instagram's feed, Facebook's News Feed, and LinkedIn's feed are all variants of this system. The core challenge is the fan-out problem: when a user with millions of followers posts content, how do you efficiently deliver that content to every follower's personalized feed? This question tests your ability to reason about read vs. write trade-offs, caching strategies, and ranking systems.

Step 1: Requirements Gathering

Functional Requirements

  1. Users can create posts (text, images, videos)
  2. Users can follow/unfollow other users
  3. The home feed shows posts from followed users, sorted by relevance or time
  4. Feed supports pagination (infinite scroll)
  5. Like, comment, and share on posts
  6. Content ranking based on engagement, recency, and user preferences

Non-Functional Requirements

  1. Feed generation should be fast: < 200ms latency
  2. High availability: feed must always be accessible
  3. Eventual consistency is acceptable (a new post can take a few seconds to appear in followers' feeds)
  4. Scale to 500M DAU with 10K+ posts per second
  5. Feed should not show duplicate posts

Step 2: Capacity Estimation

Traffic Estimates

  • DAU: 500M users
  • Average feed refreshes per user per day: 10
  • Feed reads: 5B per day (~58,000 per second)
  • New posts per day: 200M (~2,300 per second)
  • Average followers per user: 200 (but highly skewed -- celebrities have millions)

Storage Estimates

  • Post metadata: ~1 KB per post
  • Daily post storage: 200M x 1 KB = 200 GB/day
  • Feed cache per user: 500 post IDs x 8 bytes = 4 KB
  • Feed cache for all users: 500M x 4 KB = 2 TB (fits in distributed Redis)

Fan-Out Estimates

  • Average user post: 200 followers = 200 fan-out operations
  • Celebrity post (1M followers): 1M fan-out operations
  • If a celebrity posts, writing to 1M feed caches takes time

Step 3: The Fan-Out Problem

This is THE central design challenge. Two fundamental approaches:

Approach 1: Fan-Out on Write (Push Model)

When a user creates a post, immediately push it to every follower's feed cache.

User A posts
     |
     v
+------------------+
| Fan-out Service  | --- writes to ---> Follower 1's feed cache
|                  | --- writes to ---> Follower 2's feed cache
|                  | --- writes to ---> Follower 3's feed cache
|                  | --- writes to ---> ... (all N followers)
+------------------+

When Follower 1 opens feed:
  -> Just read from pre-computed feed cache (FAST!)
  • Pros: Feed reads are very fast (pre-computed), simple read path
  • Cons: Slow for users with millions of followers, wasted work for inactive followers

Approach 2: Fan-Out on Read (Pull Model)

When a user opens their feed, fetch recent posts from all followed users on the fly.

Follower 1 opens feed
     |
     v
+------------------+
| Feed Service     | --- fetch from ---> User A's posts
|                  | --- fetch from ---> User B's posts
|                  | --- fetch from ---> User C's posts
|                  | --- merge + rank ----> Return feed
+------------------+
  • Pros: No fan-out delay, no wasted work, always fresh
  • Cons: Slow reads (must query N followed users), high latency

Approach 3: Hybrid (Recommended)

Use fan-out on write for regular users, fan-out on read for celebrities:

                    Post Created
                         |
                         v
              +----------+-----------+
              | Is author a celebrity |
              | (> 10K followers)?   |
              +----------+-----------+
                    |           |
                   No          Yes
                    |           |
                    v           v
           Fan-out on      Store in author's
           Write (push     post timeline only
           to all          (pull at read time)
           followers)

    Feed Read Time:
    +-----------------------------------+
    | Pre-computed feed (from push)     |
    | + Recent posts from celebrities   |
    |   user follows (fetched on read)  |
    | = Merged, ranked, returned        |
    +-----------------------------------+

Express Implementation

import express from 'express';
import { Queue } from 'bullmq';

const router = express.Router();
const fanoutQueue = new Queue('feed-fanout', { connection: redisConnection });

// Create post
router.post('/api/v1/posts', auth, async (req, res) => {
  const { content, media_urls, visibility } = req.body;
  const authorId = req.user.id;

  const post = await prisma.post.create({
    data: { authorId, content, mediaUrls: media_urls, visibility }
  });

  // Trigger async fan-out
  const followerCount = await prisma.follow.count({
    where: { followingId: authorId }
  });

  if (followerCount <= 10000) {
    // Regular user: fan-out on write
    await fanoutQueue.add('fanout', {
      postId: post.id,
      authorId,
      timestamp: post.createdAt.getTime()
    });
  }
  // Celebrity: no fan-out, their posts are fetched on read

  res.status(201).json(post);
});

// Get home feed
router.get('/api/v1/feed', auth, async (req, res) => {
  const userId = req.user.id;
  const cursor = req.query.cursor;
  const limit = Math.min(parseInt(req.query.limit) || 20, 50);

  // 1. Get pre-computed feed from cache
  const cachedPostIds = await getFeedFromCache(userId, cursor, limit);

  // 2. Get recent posts from followed celebrities (fan-out on read)
  const celebrityPosts = await getCelebrityPosts(userId, cursor, limit);

  // 3. Merge and rank
  const allPostIds = mergeAndDeduplicate(cachedPostIds, celebrityPosts);
  const posts = await enrichPosts(allPostIds, userId);
  const rankedPosts = await rankPosts(posts, userId);

  const nextCursor = encodeCursor(rankedPosts[rankedPosts.length - 1]);

  res.json({
    posts: rankedPosts.slice(0, limit),
    next_cursor: nextCursor,
    has_more: rankedPosts.length > limit
  });
});

Step 6: Fan-Out Worker

import { Worker } from 'bullmq';
import Redis from 'ioredis';

const redis = new Redis();

const fanoutWorker = new Worker('feed-fanout', async (job) => {
  const { postId, authorId, timestamp } = job.data;

  // Get all follower IDs
  const followers = await getFollowerIds(authorId);

  // Batch update follower feed caches
  const BATCH_SIZE = 1000;
  for (let i = 0; i < followers.length; i += BATCH_SIZE) {
    const batch = followers.slice(i, i + BATCH_SIZE);
    const pipeline = redis.pipeline();

    for (const followerId of batch) {
      const feedKey = `feed:${followerId}`;
      // Add post to follower's feed (sorted set, scored by timestamp)
      pipeline.zadd(feedKey, timestamp, postId);
      // Trim to keep only latest 1000 posts
      pipeline.zremrangebyrank(feedKey, 0, -1001);
    }

    await pipeline.exec();
  }
}, {
  connection: redisConnection,
  concurrency: 10
});

Feed Cache Structure (Redis Sorted Set)

Key: feed:user_123
Score: timestamp (or ranking score)
Member: post_id

  feed:user_123 = {
    post_xyz: 1710756000,   // most recent
    post_abc: 1710755000,
    post_def: 1710754000,
    ...                     // up to 1000 posts
  }

Query: ZREVRANGE feed:user_123 0 19  -> top 20 posts by time
Cursor: ZREVRANGEBYSCORE feed:user_123 1710755000 -inf LIMIT 0 20

Step 8: Ranking and Scoring

A chronological feed is simple but a ranked feed drives much higher engagement:

function calculateFeedScore(post, user) {
  const recencyScore = getRecencyDecay(post.created_at);  // 0-1, exponential decay
  const engagementScore = normalizeEngagement(
    post.likes_count,
    post.comments_count,
    post.shares_count
  );  // 0-1
  const affinityScore = getUserAffinity(user.id, post.author_id);  // 0-1
  const contentTypeBoost = post.media_urls?.length > 0 ? 1.2 : 1.0;

  // Weighted combination
  return (
    0.3 * recencyScore +
    0.3 * engagementScore +
    0.3 * affinityScore +
    0.1 * contentTypeBoost
  );
}

function getRecencyDecay(createdAt) {
  const hoursAgo = (Date.now() - new Date(createdAt).getTime()) / 3600000;
  return Math.exp(-0.05 * hoursAgo); // half-life of ~14 hours
}

function getUserAffinity(userId, authorId) {
  // Based on past interactions: likes, comments, profile visits
  // In production, this is a ML model
  // For interview: mention collaborative filtering, interaction history
  return interactionScore(userId, authorId);
}

Step 9: Scaling Strategies

Cache Warming

When a user's feed cache is empty (cold start), generate it on the fly:

async function getFeedFromCache(userId, cursor, limit) {
  const feedKey = `feed:${userId}`;
  const exists = await redis.exists(feedKey);

  if (!exists) {
    // Cold start: build feed from scratch
    await warmFeedCache(userId);
  }

  if (cursor) {
    const score = decodeCursorScore(cursor);
    return redis.zrevrangebyscore(feedKey, score, '-inf', 'LIMIT', 0, limit);
  }

  return redis.zrevrange(feedKey, 0, limit - 1);
}

async function warmFeedCache(userId) {
  // Get users this person follows
  const following = await prisma.follow.findMany({
    where: { followerId: userId },
    select: { followingId: true }
  });

  // Fetch recent posts from each (limited)
  const recentPosts = await prisma.post.findMany({
    where: {
      authorId: { in: following.map(f => f.followingId) },
      createdAt: { gte: new Date(Date.now() - 7 * 86400000) } // last 7 days
    },
    orderBy: { createdAt: 'desc' },
    take: 500
  });

  // Populate cache
  if (recentPosts.length > 0) {
    const pipeline = redis.pipeline();
    for (const post of recentPosts) {
      pipeline.zadd(`feed:${userId}`, post.createdAt.getTime(), post.id);
    }
    pipeline.expire(`feed:${userId}`, 86400); // 24h TTL
    await pipeline.exec();
  }
}

Sharding the Feed Cache

With 500M users and 4 KB per feed, the total cache is 2 TB. Use Redis Cluster to distribute across nodes:

Redis Cluster (6 nodes):
  Node 1: hash slots 0-2730      (feeds for ~83M users)
  Node 2: hash slots 2731-5460   (feeds for ~83M users)
  Node 3: hash slots 5461-8191   (feeds for ~83M users)
  Node 4-6: replicas of nodes 1-3

Handling Unfollow

When a user unfollows someone, remove that person's posts from the feed cache:

router.delete('/api/v1/users/:userId/follow', auth, async (req, res) => {
  const followerId = req.user.id;
  const followingId = req.params.userId;

  await prisma.follow.delete({
    where: { followerId_followingId: { followerId, followingId } }
  });

  // Remove unfollowed user's posts from feed cache
  const posts = await prisma.post.findMany({
    where: { authorId: parseInt(followingId) },
    select: { id: true },
    take: 1000
  });

  if (posts.length > 0) {
    await redis.zrem(`feed:${followerId}`, ...posts.map(p => p.id));
  }

  res.status(204).send();
});

Step 10: Trade-Offs and Discussion Points

Decision Option A Option B Recommendation
Fan-out On write On read Hybrid (write for normal, read for celebrities)
Feed order Chronological Ranked Ranked (higher engagement) with chronological option
Feed storage Redis sorted set Cassandra Redis for hot feeds, Cassandra for cold
Pagination Offset Cursor Cursor (stable with new inserts)
Post storage PostgreSQL Cassandra PostgreSQL to start, Cassandra at extreme scale
Celebrity threshold 10K followers 100K 10K (tune based on fanout latency metrics)

Why Cursor Pagination for Feeds?

Offset pagination breaks when new items are inserted:

Page 1: items 1-20 (user sees these)
New item inserted at position 1
Page 2: items 21-40 ... but item 20 shifted to position 21
  -> User sees item 20 TWICE (duplicate!)

Cursor pagination:
Page 1: items with score > 0, limit 20, last item score = 1710755000
Page 2: items with score < 1710755000, limit 20
  -> No duplicates regardless of inserts

The Thundering Herd Problem

When a celebrity with 10M followers posts, if using fan-out on write, the system must update 10M feed caches simultaneously. This creates a write spike that can overwhelm Redis:

  • Solution 1: Rate-limit the fan-out (spread over minutes using delayed jobs)
  • Solution 2: Use fan-out on read for celebrities (our hybrid approach)
  • Solution 3: Probabilistic fan-out: only push to recently active followers

Summary

A news feed system's central challenge is the fan-out problem. The hybrid approach -- push for regular users and pull for celebrities -- balances write amplification against read latency. The architecture uses Redis sorted sets for pre-computed feed caches, BullMQ workers for async fan-out, cursor-based pagination for stable scrolling, and a ranking function that combines recency, engagement, and user affinity. At interview time, the key discussion points are: justifying the celebrity threshold, explaining cursor pagination over offset, discussing ranking vs. chronological ordering, and reasoning about cache warming strategies for cold-start users.

Exercises

  1. Implement Fan-Out on Write

    Build a BullMQ worker that processes new posts and pushes them to follower feed caches in Redis. Use Redis sorted sets with timestamps as scores. Implement batched processing (1000 followers per batch) to avoid blocking Redis. Test with a user who has 5000 followers and verify all feeds are updated.

    Hint

    Use redis.pipeline() to batch ZADD commands. For each batch of 1000 followers, create a pipeline that does zadd('feed:followerId', timestamp, postId) for each follower. Use ZREMRANGEBYRANK to trim feeds to the latest 1000 posts.

  2. Build the Feed Retrieval Endpoint

    Implement GET /api/v1/feed with cursor-based pagination. Read post IDs from the user's Redis sorted set, fetch post details from the database, enrich with has_liked status, and return with a cursor for the next page. Handle the cold-start case where the feed cache is empty by building it on the fly from the database.

    Hint

    Use ZREVRANGEBYSCORE with the cursor score as the max for pagination. For cold start, query posts from followed users in the last 7 days. Encode the cursor as base64({ lastScore: timestamp }). Use prisma.findMany with an IN clause for post IDs.

  3. Implement Hybrid Fan-Out

    Modify the post creation flow to use hybrid fan-out. When a regular user (< 10K followers) posts, push to all followers. When a celebrity posts, do not fan out -- instead, modify the feed read path to fetch the celebrity's recent posts and merge them with the cached feed. Test both paths.

    Hint

    On post creation, check followerCount. On feed read, maintain a list of followed celebrities (store in Redis set). For each celebrity, fetch their last 20 posts from the posts table and merge into the feed results by timestamp before pagination.

  4. Add a Simple Ranking Function

    Replace chronological ordering with a ranking function. Score each post based on: recency (exponential decay), engagement (likes + 2comments + 3shares, normalized), and author affinity (how often the user has liked the author's posts). Re-sort the feed by ranking score instead of timestamp. Compare the ranked feed to the chronological feed.

    Hint

    Calculate recencyScore = Math.exp(-0.05 * hoursAgo). For affinity, count likes by the user on the author's posts in the last 30 days, normalize to 0-1. Combine with weights: 0.4recency + 0.3engagement + 0.3*affinity. Sort descending by score.

Common Problems & Solutions

1 Celebrity posts take minutes to appear in followers' feeds
Cause: Fan-out on write for a user with 5M followers requires 5M Redis write operations. Even at 100K ops/second, this takes 50 seconds, during which the fan-out queue is backed up.
Solution: Switch to the hybrid model: do not fan-out for users above a follower threshold (e.g., 10K). Instead, fetch their recent posts at feed-read time and merge with the pre-computed feed. This eliminates the write amplification problem for celebrities.
2 Users see duplicate posts when scrolling through their feed
Cause: Using offset-based pagination (page=1, page=2), and new posts are inserted between page loads, shifting items down and causing the same post to appear on both pages.
Solution: Switch to cursor-based pagination. Use the last post's timestamp (or ranking score) as the cursor. The next page fetches posts with score less than the cursor, which is stable regardless of new insertions.
3 Feed cache misses are slow, causing timeouts for users who have not opened the app in days
Cause: When a user's feed cache has expired (TTL exceeded) or was never populated, the system must build the entire feed from scratch by querying posts from all followed users -- a slow, expensive operation.
Solution: Implement proactive cache warming: when a user opens the app, if the cache is older than 1 hour, start warming in the background. Use a 'stale-while-revalidate' pattern where the stale cache is served immediately while a fresh one is being built. Also, schedule periodic cache warming for frequently active users.
