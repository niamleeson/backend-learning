Testing Pipelines: Lint, Type-Check, Test, Coverage

Intuition

Think of a quality pipeline like airport security. First, a quick visual check (linting) catches obvious issues. Then a scanner (type checking) catches hidden problems. Finally, thorough inspection (tests with coverage) verifies everything works. Each stage is faster to fail than the next, saving everyone time.

Testing Pipelines: Lint, Type-Check, Test, Coverage

A robust CI pipeline enforces code quality through linting, type checking, test coverage thresholds, and security scanning.

Complete Testing Pipeline

name: Code Quality
on: [push, pull_request]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install ruff
      - run: ruff check .
      - run: ruff format --check .

  type-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install -r requirements.txt mypy
      - run: mypy src/ --strict

  test:
    needs: [lint, type-check]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install -r requirements.txt pytest pytest-cov
      - run: pytest --cov=src --cov-report=xml --cov-fail-under=80

Local Configuration

# pyproject.toml
[tool.ruff]
target-version = "py312"
line-length = 88

[tool.mypy]
strict = true

[tool.pytest.ini_options]
addopts = "--cov=src --cov-report=term-missing"

[tool.coverage.report]
fail_under = 80
exclude_lines = ["pragma: no cover", "if TYPE_CHECKING:"]

Common Mistakes

  • Not running lint before tests -- wasting CI time on broken code
  • Setting coverage threshold too high initially (start at 70-80%)

Key Takeaways

  • Run fast checks (lint) first, slow checks (tests) after
  • Use needs: to create job dependencies and fail fast
  • Configure tools in pyproject.toml so local and CI behavior match

Exercises

  1. Build a Quality Gate

    Create a GitHub Actions workflow with separate jobs for ruff linting, mypy type checking, pytest with coverage, and a security scan using pip-audit. Configure job dependencies so tests only run if lint and type-check pass.

    Hint

    Use the needs keyword to create dependencies. Install pip-audit for dependency vulnerability scanning.

  2. Pre-commit Hooks

    Set up pre-commit hooks that run ruff, mypy, and pytest on staged files before each commit. Create a .pre-commit-config.yaml and integrate it with the CI pipeline.

    Hint

    Use the pre-commit framework with ruff-pre-commit and local hooks for mypy.

Common Problems & Solutions

1 mypy fails in CI but passes locally
Symptoms: ["error: Library stubs not installed for 'requests'", 'Module has no attribute']
Cause: Missing type stubs in CI, different mypy versions, or missing dependencies that provide type information.
Solution: Pin the mypy version in requirements. Install all dependencies including type stubs (types-requests, etc.) in CI. Use the same mypy configuration (pyproject.toml) everywhere.
2 Coverage percentage keeps dropping
Symptoms: ['Coverage decreased from 85% to 72%']
Cause: New code is being added without corresponding tests.
Solution: Set `--cov-fail-under=80` in pytest to enforce a minimum threshold. Add coverage reporting to PRs so reviewers can see coverage changes. Use `# pragma: no cover` sparingly for untestable code.
