Docker Swarm vs Kubernetes: Choosing an Orchestrator

Intuition

Choosing between Swarm and Kubernetes is like choosing between a reliable minivan and a semi truck. The minivan (Swarm) is easy to drive, great for small loads, and you can start using it immediately. The semi truck (Kubernetes) can haul massive cargo, has advanced GPS and fleet management, but requires a commercial license and training to operate. Don't rent a semi to move a couch.

Docker Swarm vs Kubernetes: Choosing an Orchestrator

Both orchestrate containers but differ in complexity, features, and use cases.

Docker Swarm

docker swarm init
docker stack deploy -c docker-compose.yml myapp
docker service scale myapp_web=5
docker service ls

services:
  web:
    image: myapp:1.2.0
    deploy:
      replicas: 3
      update_config:
        parallelism: 1
        delay: 10s
      restart_policy:
        condition: on-failure
      resources:
        limits: { cpus: '0.5', memory: 256M }
    ports: ["8000:8000"]
    networks: [app-net]

networks:
  app-net:
    driver: overlay

Comparison

Feature Docker Swarm Kubernetes
Setup Minutes Hours/days
Learning curve Low Steep
Auto-scaling Basic Advanced (HPA)
Ecosystem Small Massive
Best for Small teams, <10 services Large teams, complex apps

Decision Framework

if team_size < 5 and services < 10:
    use = "Docker Swarm"
elif need_multi_cloud or enterprise_scale:
    use = "Kubernetes"
elif want_managed_service:
    use = "ECS/Cloud Run"

Common Mistakes

  • Choosing K8s for a 3-service app (over-engineering)
  • Choosing Swarm when you'll need advanced autoscaling soon
  • Not considering managed services (ECS, Cloud Run)

Key Takeaways

  • Swarm is simpler but has declining community adoption
  • K8s is the industry standard but operationally complex
  • For small-to-medium apps, managed container services are often best

Exercises

  1. Deploy with Docker Swarm

    Initialize a Docker Swarm cluster (single node for learning), deploy a multi-service application using docker stack deploy with a compose file. Scale a service, perform a rolling update, and observe the behavior.

    Hint

    Use docker swarm init, docker stack deploy -c docker-compose.yml myapp, and docker service scale.

  2. Migration Assessment

    Given a docker-compose.yml with 5 services, evaluate what it would take to migrate to Kubernetes. Write the equivalent Kubernetes manifests (Deployments, Services, ConfigMaps) and document the added complexity vs. benefits.

    Hint

    Create a K8s manifest for each service. Note what Docker Compose handles implicitly (networking, DNS) that K8s requires explicit configuration for.

  3. Orchestrator Decision Matrix

    For three scenarios (startup MVP with 2 developers, mid-size SaaS with 20 microservices, enterprise multi-region deployment), evaluate and recommend the best orchestration approach with justification.

    Hint

    Consider team size, service count, scaling needs, budget, and available expertise for each scenario.

Common Problems & Solutions

1 Docker Swarm service stuck at 0/N replicas
Symptoms: ['docker service ls shows 0/3 REPLICAS', 'No container running for the service']
Cause: The image cannot be pulled, or resource constraints prevent scheduling.
Solution: Check service logs: `docker service ps myapp_web --no-trunc` to see error messages. Verify the image exists and is accessible. Check node resources with `docker node ls`.
2 Cannot decide between orchestrators
Symptoms: ['Weeks spent evaluating tools instead of shipping', 'Over-engineered infrastructure for a simple app']
Cause: Analysis paralysis from comparing features without considering actual needs.
Solution: Start with the simplest option that meets your current needs. For most teams: use Docker Compose for development, a managed service (ECS/Cloud Run) for production, and only adopt Kubernetes when you genuinely need its features (auto-scaling, service mesh, multi-cluster).
