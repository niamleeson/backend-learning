CAP Theorem and Consistency Models

Intuition

The CAP theorem is like a delivery service during a storm (network partition). You can either guarantee every package is correct (consistency) or guarantee every customer gets a package (availability), but not both when roads are blocked.

CAP Theorem & Consistency Models

The CAP theorem is a foundational concept in distributed systems that states it is impossible for a distributed data store to simultaneously provide more than two of three guarantees: Consistency, Availability, and Partition tolerance.

Core Concepts

  • CAP Theorem: In the presence of a network partition (P), a system must choose between Consistency (C)—all nodes see the same data—and Availability (A)—every request receives a response. Since network partitions are inevitable, the real choice is between CP and AP.
  • Strong Consistency: After a write completes, all subsequent reads return the updated value. Requires coordination (consensus protocols) that adds latency.
  • Eventual Consistency: After a write, replicas will eventually converge to the same value, but reads may temporarily return stale data. Used by DynamoDB, Cassandra, and DNS.
  • Causal Consistency: Operations that are causally related are seen in the same order by all nodes. Concurrent operations may be seen in different orders. A practical middle ground.
  • Linearizability: The strongest consistency model. Operations appear to execute atomically at some point between invocation and response. Equivalent to a single-copy system.

Key Takeaways

  1. The CAP theorem is about trade-offs during network partitions, not normal operation—most systems can provide both C and A when the network is healthy.
  2. Most real-world systems choose availability (AP) with tunable consistency rather than strict CP, because unavailability has a direct business cost.
  3. Understand which consistency model your application actually needs—strong consistency is expensive and often unnecessary.

Exercises

  1. Simulate CAP Trade-offs

    Build a simple distributed key-value store with three nodes. Simulate a network partition (block communication between nodes). Demonstrate the trade-off: implement a CP mode (reject writes/reads to partitioned nodes) and an AP mode (allow reads/writes but accept stale data).

    Hint

    Use three Flask servers. In CP mode, require a quorum (2 of 3) for reads and writes—reject if a quorum cannot be reached. In AP mode, serve from any available node and reconcile with vector clocks after the partition heals.

  2. Implement Read-Your-Writes Consistency

    Given an eventually consistent database (simulate with Redis replicas and artificial lag), implement a session-level read-your-writes guarantee. After a write, ensure the same session reads from a source that has the latest data.

    Hint

    On write, record the write timestamp in the session. On read, compare the timestamp with the replica's last sync time. If the replica is behind, read from the primary instead.

Common Problems & Solutions

1 Users occasionally see stale product inventory, leading to overselling.
Cause: The inventory system uses eventual consistency, and read replicas lag behind the primary during high write load.
Solution: Use strong consistency (read from primary) for inventory checks during checkout. Apply optimistic concurrency control with version numbers—reject the purchase if the version has changed since the inventory was read.
2 Distributed database becomes unavailable during network partitions between data centers.
Cause: The system is configured for strong consistency (CP) and cannot form a quorum when a data center is unreachable.
Solution: Evaluate if the business can tolerate eventual consistency for certain operations during partitions. Use multi-region active-active with conflict resolution (CRDTs, last-writer-wins) for availability, or accept the unavailability trade-off for critical data.
