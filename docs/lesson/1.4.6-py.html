heapq and bisect: Priority Queues and Sorted Insertion

Intuition

heapq gives you a lightweight priority queue without a separate class, and bisect lets you maintain a sorted list with binary search — both avoid the overhead of full sorting.

The heapq module implements a min-heap on a plain list, while bisect provides binary search for maintaining sorted sequences.

import heapq, bisect

# heapq: priority queue (min-heap)
tasks = []
heapq.heappush(tasks, (3, "low"))
heapq.heappush(tasks, (1, "critical"))
heapq.heappush(tasks, (2, "medium"))
print(heapq.heappop(tasks))  # (1, 'critical')

# nlargest / nsmallest
nums = [15, 3, 78, 42, 6, 91]
print(heapq.nsmallest(3, nums))  # [3, 6, 15]

# bisect: sorted insertion
sorted_list = [10, 20, 30, 40]
bisect.insort(sorted_list, 25)   # [10, 20, 25, 30, 40]
idx = bisect.bisect_left(sorted_list, 25)  # 2
  • heapq is a min-heap; for max-heap, negate values: heappush(h, -val).
  • heapq.nsmallest(k, iterable) is more efficient than sorting when k is small.
  • bisect.insort keeps a list sorted after insertion in O(n) due to shifting, but the binary search itself is O(log n).
  • bisect_left vs bisect_right matters when the value already exists in the list.

Common Problems & Solutions

1 Heap gives smallest element, but you need the largest.
Cause: `heapq` only implements a min-heap; there is no built-in max-heap.
Solution: Push negated values (`-val`) and negate again on pop, or use `heapq.nlargest`.
2 `bisect.insort` seems slow on very large lists.
Cause: While the search is O(log n), the insertion into a list is O(n) due to element shifting.
Solution: For large datasets with frequent insertions, consider a balanced BST via `sortedcontainers.SortedList`.
