Normalization: 1NF through 3NF and Beyond

Intuition

Normalization is like organizing a kitchen. An unnormalized database is like writing every ingredient's nutritional info on every recipe card. If the info changes, you need to update every card. Normalization says: write the nutritional info once in a reference book, and just reference it from the recipe cards. The trade-off is you need to 'look up' the reference book (JOIN) each time you cook.

Normalization: 1NF through 3NF and Beyond

Introduction

Database normalization is the process of organizing tables to reduce data redundancy and prevent update anomalies. It's a systematic approach to schema design that ensures data integrity. While you don't always need to fully normalize (denormalization has its place), understanding normalization is essential for making informed design decisions.

Why Normalize?

Consider this unnormalized table:

orders_flat:
| order_id | customer_name | customer_email    | product_name | product_price | qty |
|----------|--------------|-------------------|--------------|---------------|-----|
| 1        | Alice        | alice@example.com | Widget       | 9.99          | 2   |
| 2        | Alice        | alice@example.com | Gadget       | 19.99         | 1   |
| 3        | Bob          | bob@example.com   | Widget       | 9.99          | 3   |

Problems (anomalies): - Update anomaly: Alice changes her email — you must update every row. Miss one and data is inconsistent. - Insert anomaly: Can't add a new product until someone orders it. - Delete anomaly: If you delete Bob's only order, you lose the fact that Bob exists.

First Normal Form (1NF)

Rule: Each column contains atomic (indivisible) values. No repeating groups or arrays.

VIOLATES 1NF:
| order_id | products                    |
|----------|-----------------------------|
| 1        | Widget, Gadget, Doohickey   |  <-- multiple values in one cell

1NF COMPLIANT:
| order_id | product    |
|----------|------------|
| 1        | Widget     |
| 1        | Gadget     |
| 1        | Doohickey  |

-- Bad: comma-separated values
CREATE TABLE orders_bad (
  id serial PRIMARY KEY,
  product_names text  -- 'Widget, Gadget, Doohickey'
);

-- Good: separate rows
CREATE TABLE order_items (
  order_id integer REFERENCES orders(id),
  product_id integer REFERENCES products(id),
  quantity integer NOT NULL,
  PRIMARY KEY (order_id, product_id)
);

Note: PostgreSQL arrays (text[]) are debatable for 1NF. They're fine for tags or labels but not for data that needs its own attributes.

Second Normal Form (2NF)

Rule: 1NF + every non-key column depends on the ENTIRE primary key (not just part of it).

Only applies to tables with composite primary keys.

VIOLATES 2NF (composite key: order_id + product_id):
| order_id | product_id | product_name | quantity | unit_price |
|----------|------------|--------------|----------|------------|
                          ^^^^^^^^^^^^
                          Depends only on product_id, not the full key

2NF COMPLIANT: Split into two tables:

order_items:                        products:
| order_id | product_id | qty |     | product_id | product_name | unit_price |
|----------|------------|-----|     |------------|--------------|------------|

Third Normal Form (3NF)

Rule: 2NF + no column depends on another non-key column (no transitive dependencies).

VIOLATES 3NF:
| employee_id | department_id | department_name | department_budget |
|-------------|--------------|-----------------|-------------------|
                               ^^^^^^^^^^^^^^^^^  ^^^^^^^^^^^^^^^^
                               These depend on department_id, not employee_id

3NF COMPLIANT:

employees:                              departments:
| employee_id | department_id |         | department_id | name   | budget |
|-------------|--------------|         |---------------|--------|--------|

-- 3NF schema
CREATE TABLE departments (
  id serial PRIMARY KEY,
  name text NOT NULL,
  budget numeric(12, 2)
);

CREATE TABLE employees (
  id serial PRIMARY KEY,
  name text NOT NULL,
  department_id integer REFERENCES departments(id)
);

Normalization Flowchart

  Unnormalized Data
         |
  [Remove repeating groups, ensure atomic values]
         |
       1NF
         |
  [Remove partial dependencies on composite keys]
         |
       2NF
         |
  [Remove transitive dependencies]
         |
       3NF
         |
  (Usually stop here for most applications)

Beyond 3NF: BCNF

Boyce-Codd Normal Form handles edge cases where 3NF doesn't quite eliminate all anomalies:

Rule: Every determinant (column that determines other columns) must be a candidate key.

VIOLATES BCNF:
| student | subject | teacher   |
|---------|---------|-----------|
| Alice   | Math    | Dr. Smith |
| Bob     | Math    | Dr. Smith |

teacher -> subject (a teacher teaches one subject)
But teacher is not a candidate key.

BCNF Fix: Split into teacher_subjects and student_teachers.

When to Denormalize

Normalization optimizes for write consistency. Sometimes you denormalize for read performance:

// Normalized: requires JOIN for every product listing
const normalized = await pool.query(`
  SELECT p.name, p.price, c.name AS category, b.name AS brand
  FROM products p
  JOIN categories c ON p.category_id = c.id
  JOIN brands b ON p.brand_id = b.id
  WHERE p.id = $1
`, [productId]);

// Denormalized: single table read (faster, but data duplication)
const denormalized = await pool.query(`
  SELECT name, price, category_name, brand_name
  FROM products_denormalized
  WHERE id = $1
`, [productId]);

Common denormalization strategies: - Materialized views: Precomputed JOINs, refreshed periodically - Cached aggregates: Store COUNT, SUM in parent table - Duplicate columns: Copy frequently-joined data

-- Materialized view for a denormalized read model
CREATE MATERIALIZED VIEW product_listings AS
SELECT
  p.id, p.name, p.price,
  c.name AS category_name,
  b.name AS brand_name,
  COALESCE(r.avg_rating, 0) AS avg_rating,
  COALESCE(r.review_count, 0) AS review_count
FROM products p
JOIN categories c ON p.category_id = c.id
JOIN brands b ON p.brand_id = b.id
LEFT JOIN (
  SELECT product_id, AVG(rating) AS avg_rating, COUNT(*) AS review_count
  FROM reviews GROUP BY product_id
) r ON p.id = r.product_id;

-- Refresh periodically
REFRESH MATERIALIZED VIEW CONCURRENTLY product_listings;

Common Problems & Solutions

1 Queries require too many JOINs and are slow
Cause: Over-normalization — splitting data into too many small tables when the data is always queried together.
Solution: Consider strategic denormalization: materialized views for read-heavy queries, cached aggregate columns for counts/sums, or embedding frequently-accessed related data.
2 Data becomes inconsistent after updates
Cause: Insufficient normalization — the same data is stored in multiple places without proper constraints, leading to update anomalies.
Solution: Normalize to at least 3NF. Use foreign keys to enforce referential integrity. If denormalized for performance, use triggers or materialized views to keep copies in sync.
3 Comma-separated values in a column make queries impossible
Cause: Violating 1NF by storing multiple values in a single column (e.g., 'tag1,tag2,tag3').
Solution: Create a junction/linking table: `post_tags(post_id, tag_id)`. Or use PostgreSQL arrays with GIN indexes if the values don't need their own attributes.
