Generics, Union, Optional, and TypeVar

Intuition

Generics are like fill-in-the-blank templates for types. TypeVar is the blank, and when you use the code, Python's type checker fills in the blank consistently throughout.

Generics, Union, Optional, and TypeVar

Python's typing module provides powerful constructs for expressing flexible yet precise type relationships. Generics let you write code that works across types while preserving type safety.

from typing import TypeVar, Generic, Union, Optional

T = TypeVar('T')
K = TypeVar('K')
V = TypeVar('V')

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()

def first_or_default(items: list[T], default: T) -> T:
    return items[0] if items else default

# Union and Optional
def parse_int(value: str) -> Optional[int]:
    # Optional[int] is equivalent to Union[int, None]
    try:
        return int(value)
    except ValueError:
        return None

result: Union[str, int] = "hello"  # can be str or int

Key Takeaways: - TypeVar creates generic type variables bound across a function or class - Generic[T] makes a class parameterized by type - Optional[X] is shorthand for Union[X, None] - Use Union[A, B] or A | B (3.10+) for multiple possible types - Bounded TypeVars (TypeVar('T', bound=Base)) restrict to subtypes

Exercises

  1. Build a Generic Cache

    Create a Cache(Generic[K, V]) class with get(key: K) -> Optional[V] and set(key: K, value: V) -> None methods. Test it with Cache[str, int].

    Hint

    Use two TypeVars K and V, store items in dict[K, V], and return None from get when the key is missing.

  2. Generic Pipeline Function

    Write transform(items: list[T], func: Callable[[T], V]) -> list[V] that applies a function to each item and returns the results.

    Hint

    Import Callable from typing. The function maps T -> V, so transform takes list[T] and returns list[V].

Common Problems & Solutions

1 mypy reports 'Incompatible types' when using a `TypeVar` in unrelated positions.
Cause: A single `TypeVar` must resolve to the same type everywhere it appears in a signature.
Solution: Use separate `TypeVar` names (e.g., `T` and `U`) when parameters and return types are independently generic.
2 `Optional[int]` allows `None` to slip through without checks, causing `TypeError` at runtime.
Cause: Optional means the value can be `None`, and code does not guard against it.
Solution: Always narrow with `if result is not None:` before using the value. mypy enforces this in strict mode.
