Scoped Dependencies and Database Sessions

Intuition

Yield dependencies are like context managers for your endpoints: they open a resource, hand it over for use, and guarantee it gets cleaned up whether the request succeeds or fails. This pattern prevents connection leaks.

Scoped Dependencies and Database Sessions

Yield-based dependencies in FastAPI provide setup-and-teardown semantics, perfect for managing database sessions, transactions, and other resources that need cleanup after each request.

from fastapi import FastAPI, Depends
from sqlalchemy.ext.asyncio import (
    create_async_engine, async_sessionmaker, AsyncSession
)

DATABASE_URL = "sqlite+aiosqlite:///./app.db"
engine = create_async_engine(DATABASE_URL)
async_session = async_sessionmaker(engine, expire_on_commit=False)

async def get_db() -> AsyncSession:
    async with async_session() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise

app = FastAPI()

@app.get("/users/{user_id}")
async def get_user(user_id: int, db: AsyncSession = Depends(get_db)):
    result = await db.execute(
        select(User).where(User.id == user_id)
    )
    user = result.scalar_one_or_none()
    if not user:
        raise HTTPException(status_code=404)
    return user

Key Takeaways: - yield dependencies run setup before yield and cleanup after the response - Database sessions should be scoped per-request to avoid shared state - Wrap yield in try/except to handle rollback on errors - async with ensures proper cleanup of async resources

Exercises

  1. Transaction Scope Dependency

    Create two yield dependencies: one that provides a database session and one that wraps the session in an explicit transaction using session.begin(). Chain them so the transaction dependency depends on the session dependency.

    Hint

    The transaction dependency takes session = Depends(get_db), then uses async with session.begin(): yield session. The commit/rollback is handled by the transaction context manager.

  2. Resource Pool Dependency

    Simulate a connection pool with a list of 3 connection objects. Create a yield dependency that acquires a connection (pops from the pool), yields it, and returns it to the pool in the cleanup phase. Test concurrent requests.

    Hint

    Use asyncio.Semaphore(3) to limit concurrency. In the dependency, await semaphore.acquire(), pop a connection, yield it, then push it back and release the semaphore.

Common Problems & Solutions

1 Database session closed before response is serialized
Cause: Lazy-loaded ORM relationships are accessed during response serialization, but the session has already been closed by the yield dependency.
Solution: Use `expire_on_commit=False` in the sessionmaker, or eagerly load relationships in your query using `selectinload()` or `joinedload()` from SQLAlchemy.
2 Yield dependency cleanup code not running
Cause: An exception is raised between the yield and the cleanup code, or the dependency is not properly declared as a generator.
Solution: Always wrap yield in a try/finally block to guarantee cleanup runs. Ensure the function uses `yield` (not `return`) and that FastAPI recognizes it as a generator dependency.
