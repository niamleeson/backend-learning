Task Scheduling and Periodic Tasks with BullMQ

Intuition

BullMQ scheduled jobs are like a shared office calendar that multiple employees can see. Instead of each employee setting their own alarm (cron), they all check the same calendar (Redis). Only one person handles each meeting (job), and if they are sick (crash), someone else picks it up.

Task Scheduling and Periodic Tasks with BullMQ

Introduction

Many backend applications need to run tasks on a schedule: sending daily digest emails, generating weekly reports, cleaning up expired records, or syncing data with external services. BullMQ's repeatable jobs provide a reliable, distributed alternative to cron jobs that works across multiple server instances.

Why Not Just Use cron or setInterval?

cron/setInterval Problems:
  1. Multiple instances = duplicate execution
  2. If server restarts, schedule is lost
  3. No retry on failure
  4. No monitoring or history
  5. No distributed locking

BullMQ Repeatable Jobs:
  1. Redis-backed = only one instance runs the job
  2. Schedule persists across restarts
  3. Built-in retry with backoff
  4. Job history and monitoring via Bull Board
  5. Distributed by design

BullMQ Repeatable Jobs

Cron-Based Scheduling

import { Queue, Worker } from 'bullmq';

const connection = { host: 'localhost', port: 6379, maxRetriesPerRequest: null };
const scheduledQueue = new Queue('scheduled-tasks', { connection });

// Add repeatable jobs (cron syntax)
async function setupSchedules() {
  // Every day at 8:00 AM UTC
  await scheduledQueue.add('daily-digest', {
    type: 'email-digest',
  }, {
    repeat: {
      pattern: '0 8 * * *', // cron: minute hour day month weekday
    },
    jobId: 'daily-digest', // Unique ID prevents duplicates on restart
  });

  // Every hour
  await scheduledQueue.add('cleanup-expired', {
    type: 'cleanup',
  }, {
    repeat: {
      pattern: '0 * * * *',
    },
    jobId: 'cleanup-expired',
  });

  // Every Monday at 9:00 AM
  await scheduledQueue.add('weekly-report', {
    type: 'report',
  }, {
    repeat: {
      pattern: '0 9 * * 1',
    },
    jobId: 'weekly-report',
  });

  // Every 5 minutes
  await scheduledQueue.add('health-check', {
    type: 'health',
  }, {
    repeat: {
      every: 5 * 60 * 1000, // 5 minutes in ms
    },
    jobId: 'health-check',
  });
}

setupSchedules();

Worker Processing

const worker = new Worker('scheduled-tasks', async (job) => {
  console.log(`Running scheduled task: ${job.name} at ${new Date().toISOString()}`);

  switch (job.data.type) {
    case 'email-digest':
      return await sendDailyDigest();
    case 'cleanup':
      return await cleanupExpiredRecords();
    case 'report':
      return await generateWeeklyReport();
    case 'health':
      return await performHealthCheck();
    default:
      throw new Error(`Unknown task type: ${job.data.type}`);
  }
}, {
  connection,
  concurrency: 3,
});

async function sendDailyDigest() {
  const users = await prisma.user.findMany({
    where: { digestEnabled: true },
  });

  let sent = 0;
  for (const user of users) {
    const digest = await buildDigest(user.id);
    await emailService.send(user.email, 'Daily Digest', digest);
    sent++;
  }
  return { usersSent: sent };
}

async function cleanupExpiredRecords() {
  const result = await prisma.$transaction([
    prisma.session.deleteMany({
      where: { expiresAt: { lt: new Date() } },
    }),
    prisma.passwordResetToken.deleteMany({
      where: { expiresAt: { lt: new Date() } },
    }),
    prisma.verificationToken.deleteMany({
      where: { expiresAt: { lt: new Date() } },
    }),
  ]);

  return {
    sessionsDeleted: result[0].count,
    resetTokensDeleted: result[1].count,
    verificationTokensDeleted: result[2].count,
  };
}

Managing Repeatable Jobs

// List all repeatable jobs
const repeatableJobs = await scheduledQueue.getRepeatableJobs();
console.log(repeatableJobs);
// [{ key: 'daily-digest:::0 8 * * *', name: 'daily-digest', ... }]

// Remove a repeatable job
await scheduledQueue.removeRepeatableByKey('daily-digest:::0 8 * * *');

// Or remove by name and options
await scheduledQueue.removeRepeatable('daily-digest', {
  pattern: '0 8 * * *',
});

// Update schedule: remove old, add new
await scheduledQueue.removeRepeatable('daily-digest', {
  pattern: '0 8 * * *',
});
await scheduledQueue.add('daily-digest', { type: 'email-digest' }, {
  repeat: { pattern: '0 9 * * *' }, // Changed to 9 AM
  jobId: 'daily-digest',
});

Delayed Jobs (One-Time Scheduled)

// Schedule a one-time job for the future
await scheduledQueue.add('send-reminder', {
  userId: 'user-123',
  orderId: 'ORD-456',
  message: 'Your order ships tomorrow!',
}, {
  delay: 24 * 60 * 60 * 1000, // 24 hours from now
});

// Schedule at a specific time
const targetTime = new Date('2025-12-25T00:00:00Z');
const delay = targetTime.getTime() - Date.now();
if (delay > 0) {
  await scheduledQueue.add('christmas-promo', {
    type: 'promotion',
    campaign: 'christmas-2025',
  }, {
    delay,
  });
}

Distributed Locking for Scheduled Tasks

Even though BullMQ ensures a repeatable job runs on only one worker, sometimes you need additional locking for tasks that interact with shared resources:

import Redlock from 'redlock';

const redlock = new Redlock([redis], {
  retryCount: 3,
  retryDelay: 200,
});

const worker = new Worker('scheduled-tasks', async (job) => {
  if (job.data.type === 'report') {
    // Acquire distributed lock
    let lock;
    try {
      lock = await redlock.acquire(
        [`lock:report:${job.data.reportType}`],
        30000 // Lock for 30 seconds
      );

      return await generateReport(job.data);
    } finally {
      if (lock) await lock.release();
    }
  }
}, { connection });

NestJS Scheduled Tasks with BullMQ

// scheduler.module.ts
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bullmq';
import { SchedulerService } from './scheduler.service';
import { SchedulerProcessor } from './scheduler.processor';

@Module({
  imports: [
    BullModule.registerQueue({ name: 'scheduled-tasks' }),
  ],
  providers: [SchedulerService, SchedulerProcessor],
})
export class SchedulerModule {}

// scheduler.service.ts
import { Injectable, OnModuleInit } from '@nestjs/common';
import { InjectQueue } from '@nestjs/bullmq';
import { Queue } from 'bullmq';

@Injectable()
export class SchedulerService implements OnModuleInit {
  constructor(
    @InjectQueue('scheduled-tasks') private taskQueue: Queue,
  ) {}

  async onModuleInit() {
    // Remove existing repeatable jobs to avoid duplicates
    const existing = await this.taskQueue.getRepeatableJobs();
    for (const job of existing) {
      await this.taskQueue.removeRepeatableByKey(job.key);
    }

    // Register all scheduled tasks
    await this.taskQueue.add('daily-cleanup', {}, {
      repeat: { pattern: '0 2 * * *' }, // 2 AM daily
    });

    await this.taskQueue.add('hourly-sync', {}, {
      repeat: { every: 3600000 },
    });
  }
}

import { Processor, WorkerHost } from '@nestjs/bullmq';

@Processor('scheduled-tasks')
export class SchedulerProcessor extends WorkerHost {

    switch (job.name) {
      case 'daily-cleanup':
        return this.dailyCleanup();
      case 'hourly-sync':
        return this.hourlySync();

  }

Monitoring Scheduled Tasks

// Health check for scheduled tasks
app.get('/admin/schedules', async (req, res) => {
  const repeatableJobs = await scheduledQueue.getRepeatableJobs();
  const delayed = await scheduledQueue.getDelayedCount();
  const active = await scheduledQueue.getActiveCount();
  const failed = await scheduledQueue.getFailedCount();

  res.json({
    schedules: repeatableJobs.map(job => ({
      name: job.name,
      pattern: job.pattern || `every ${job.every}ms`,
      next: new Date(job.next).toISOString(),
    })),
    stats: { delayed, active, failed },
  });
});

Common Pitfalls

  1. Duplicate repeatable jobs on restart: If you call queue.add with repeat options every time your app starts, you may create duplicates. Always use a unique jobId or clear existing repeatables first.
  2. Long-running scheduled jobs overlapping: A job that takes longer than its interval will have multiple instances running. Use distributed locks or check for active jobs before starting.
  3. Timezone issues with cron: BullMQ cron runs in UTC by default. Use the tz option for timezone-aware scheduling.
  4. Not monitoring scheduled task failures: A failing daily task might go unnoticed for days. Set up alerts on failed count.

Summary

BullMQ repeatable jobs replace unreliable cron jobs with a Redis-backed, distributed scheduling system. They work correctly across multiple server instances, persist through restarts, and integrate with BullMQ's retry, monitoring, and dashboard capabilities. Use cron patterns for calendar-based schedules and every for interval-based schedules. Always set unique jobIds and monitor for failures.

  2. Rate-Limited API Sync

    Create a scheduled task that syncs data from an external API every 15 minutes. Implement rate limiting to avoid hitting the external API's rate limits.

    Hint

    Use repeat every: 900000 (15 min). Use the worker's limiter option to control request rate. Handle 429 responses with delayed retry.

  3. Schedule Management API

    Build an admin API that lists all scheduled tasks, allows pausing/resuming specific schedules, and shows execution history.

    Hint

    Use getRepeatableJobs() to list, removeRepeatableByKey() to pause (then re-add to resume). Use getCompleted() and getFailed() for history.

Common Problems & Solutions

1 Scheduled job runs multiple times simultaneously
Cause: The job's execution time exceeds the repeat interval, so a new instance starts while the previous one is still running.
Solution: Use a distributed lock (Redlock) to ensure only one instance runs at a time. Or increase the interval to exceed the maximum expected execution time.
2 Duplicate repeatable jobs after application restart
Cause: The application adds repeatable jobs on startup without checking if they already exist, creating multiple schedules for the same task.
Solution: Use a unique jobId for each repeatable job. Or call getRepeatableJobs() and removeRepeatableByKey() before adding new ones.
