Indexes: B-tree, Hash, GIN, GiST, BRIN

Intuition

Indexes are like the index at the back of a textbook. Without it, finding every mention of 'PostgreSQL' means reading every page. With it, you look up the term and jump straight to the relevant pages. Different index types are like different reference systems — an alphabetical index (B-tree), a topic map (GIN), or a chapter summary (BRIN) — each optimized for different kinds of lookups.

Indexes: B-tree, Hash, GIN, GiST, BRIN

Introduction

Indexes are the primary tool for making database queries fast. Without indexes, PostgreSQL must scan every row in a table (a sequential scan) to find matching rows. With the right index, it can jump directly to the relevant rows. Understanding when and how to create indexes — and when NOT to — is a core backend engineering skill.

How Indexes Work: The Analogy

Without index (Sequential Scan):
  Read every page in a book to find "PostgreSQL"
  Time: O(n) - proportional to table size

With index (Index Scan):
  Look up "PostgreSQL" in the book's index -> page 42
  Jump directly to page 42
  Time: O(log n) - logarithmic, much faster

B-tree Index (Default)

The most common index type. Works for equality and range queries:

-- Automatically created for PRIMARY KEY and UNIQUE constraints
CREATE TABLE users (
  id serial PRIMARY KEY,         -- B-tree index auto-created
  email varchar(320) UNIQUE,     -- B-tree index auto-created
  name text,
  created_at timestamptz
);

-- Manually create a B-tree index
CREATE INDEX idx_users_created_at ON users (created_at);
CREATE INDEX idx_users_name ON users (name);

B-tree Structure:

         [M]
        /   \
     [D,H]  [R,V]
    / | \   / | \
  [A-C][E-G][I-L] [N-Q][S-U][W-Z]

- Sorted, balanced tree
- O(log n) lookups
- Supports: =, <, >, <=, >=, BETWEEN, IN, IS NULL
- Supports ORDER BY without sorting

Composite (Multi-Column) Indexes

CREATE INDEX idx_orders_user_date
ON orders (user_id, created_at DESC);

-- This index helps with:
SELECT * FROM orders WHERE user_id = 5;                          -- YES (leftmost prefix)
SELECT * FROM orders WHERE user_id = 5 AND created_at > '2024-01-01'; -- YES (both columns)
SELECT * FROM orders WHERE user_id = 5 ORDER BY created_at DESC;      -- YES (covers sort)

-- This index does NOT help with:
SELECT * FROM orders WHERE created_at > '2024-01-01';  -- NO (skips leftmost column)

Key Rule: Composite indexes work left-to-right. The leftmost column must be in the query.

Partial Indexes

Index only a subset of rows:

-- Only index active users (much smaller than indexing all users)
CREATE INDEX idx_active_users ON users (email)
WHERE status = 'active';

-- Only index unprocessed orders
CREATE INDEX idx_pending_orders ON orders (created_at)
WHERE status = 'pending';

Hash Index

Optimized for equality comparisons only:

CREATE INDEX idx_users_email_hash ON users USING hash (email);

-- Works for: WHERE email = 'alice@example.com'
-- Does NOT work for: WHERE email LIKE 'alice%' or ORDER BY email

Hash indexes are smaller than B-tree for equality-only lookups, but B-tree is usually preferred for its versatility.

GIN Index (Generalized Inverted Index)

Designed for values that contain multiple elements: arrays, JSONB, full-text search:

-- For array columns
CREATE INDEX idx_posts_tags ON posts USING gin (tags);
SELECT * FROM posts WHERE tags @> ARRAY['javascript']; -- uses GIN

-- For JSONB
CREATE INDEX idx_events_data ON events USING gin (data);
SELECT * FROM events WHERE data @> '{"type": "click"}'; -- uses GIN

-- For full-text search
CREATE INDEX idx_articles_search ON articles
USING gin (to_tsvector('english', title || ' ' || body));

GIN Structure (Inverted Index):

  Term        -> Row IDs
  ---------------------------
  javascript  -> {1, 3, 7, 12}
  nodejs      -> {1, 5, 7}
  typescript  -> {3, 9, 12}

  Query: tags @> '{javascript, nodejs}'
  Result: intersection of {1,3,7,12} and {1,5,7} = {1, 7}

GiST Index (Generalized Search Tree)

For geometric data, ranges, and full-text search:

-- Range types
CREATE TABLE reservations (
  id serial PRIMARY KEY,
  room_id int,
  during tstzrange NOT NULL
);

CREATE INDEX idx_reservations_during ON reservations USING gist (during);

-- Find overlapping reservations
SELECT * FROM reservations
WHERE during && tstzrange('2024-01-15', '2024-01-20');

BRIN Index (Block Range Index)

Very compact index for naturally ordered data:

-- Perfect for time-series data where rows arrive in order
CREATE INDEX idx_logs_created ON logs USING brin (created_at);

-- Very small (kilobytes vs megabytes for B-tree)
-- Works well when physical row order matches the indexed column

Index Management from Node.js

// Create index concurrently (doesn't lock the table)
await pool.query(`
  CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_orders_user_id
  ON orders (user_id)
`);

// Check existing indexes
const indexes = await pool.query(`
  SELECT
    indexname,
    indexdef,
    pg_size_pretty(pg_relation_size(indexname::regclass)) AS size
  FROM pg_indexes
  WHERE tablename = 'orders'
  ORDER BY pg_relation_size(indexname::regclass) DESC
`);

// Check if a query uses an index
const plan = await pool.query(`
  EXPLAIN (FORMAT JSON)
  SELECT * FROM orders WHERE user_id = 5
`);
console.log(JSON.stringify(plan.rows[0], null, 2));

When NOT to Index

  • Small tables (< 1000 rows): sequential scan is faster
  • Columns with low cardinality and no partial filter (e.g., boolean with 50/50 split)
  • Columns rarely used in WHERE, JOIN, or ORDER BY
  • Tables with heavy write load (each index slows down inserts/updates)

Index Selection Guide

Query Pattern                    Index Type
-----------------------------------------------
col = value                      B-tree (default)
col < / > / BETWEEN              B-tree
col IN (list)                    B-tree
ORDER BY col                     B-tree
array @> value                   GIN
jsonb @> / ? / ?&               GIN
Full-text search (@@)            GIN or GiST
Range overlap (&&)               GiST
Equality only                    Hash
Time-series (naturally ordered)  BRIN

Exercises

  1. Index Audit

    Write a Node.js function that queries pg_indexes and pg_stat_user_indexes to find: (a) unused indexes (idx_scan = 0), (b) duplicate indexes (same columns), and (c) missing indexes (sequential scans on large tables). Output recommendations.

    Hint

    Query pg_stat_user_indexes for idx_scan counts, pg_indexes for definitions, pg_stat_user_tables for seq_scan and table size.

  2. Composite Index Experiment

    Create a table with 1 million rows. Create various composite indexes and use EXPLAIN ANALYZE to verify which query patterns use which indexes. Test: (a) full match, (b) leftmost prefix only, (c) non-leftmost column only.

    Hint

    Use generate_series to create test data. Compare 'Index Scan', 'Index Only Scan', 'Bitmap Index Scan', and 'Seq Scan' in the EXPLAIN output.

  3. GIN Index for Tags

    Create a posts table with a tags text[] column. Insert 100k rows with random tags. Compare query performance WITH and WITHOUT a GIN index for: (a) posts containing a specific tag, (b) posts containing ALL of several tags, (c) posts containing ANY of several tags.

    Hint

    Use @> for 'contains all', && for 'overlaps any'. Time queries with EXPLAIN ANALYZE.

Common Problems & Solutions

1 Index exists but query still does a sequential scan
Cause: Multiple possibilities: (1) table is small enough that seq scan is faster, (2) query doesn't match index columns or uses functions on indexed columns, (3) statistics are outdated.
Solution: Run ANALYZE on the table. Check if the query uses functions on indexed columns (e.g., WHERE LOWER(email) = ...) — you need a functional index. For small tables, seq scan is actually optimal.

-- Functional index for case-insensitive search\nCREATE INDEX idx_users_email_lower ON users (LOWER(email));

2 INSERT/UPDATE performance degrades after adding indexes
Cause: Every index must be updated on each write. Many indexes on a heavily written table compound this overhead.
Solution: Only create indexes that are actually used. Remove unused indexes. For bulk inserts, consider dropping indexes, inserting, then recreating. Use pg_stat_user_indexes to find unused indexes.
3 CREATE INDEX locks the table, blocking writes
Cause: Standard CREATE INDEX acquires an exclusive lock on the table.
Solution: Use CREATE INDEX CONCURRENTLY which builds the index without locking writes. It takes longer but doesn't block DML.

CREATE INDEX CONCURRENTLY idx_name ON table (column);
