Design a Chat System (WhatsApp)

Intuition

A chat system is like a mail room in a massive apartment building. Each resident (user) has a mailbox (WebSocket connection) on a specific floor (server). When someone sends a letter, the mail room (message router) looks up which floor the recipient lives on and delivers it to the right mailbox. If the resident is not home (offline), the letter is held in a safe (pending queue) until they return, and a text message (push notification) is sent to let them know mail is waiting.

Design a Chat System (WhatsApp)

Introduction

A real-time chat system like WhatsApp, Slack, or Telegram must deliver messages between users with minimal latency, support group conversations, handle millions of concurrent connections, and provide reliable message delivery even when users are offline. This is one of the most comprehensive system design questions because it requires reasoning about real-time communication, persistent connections, message ordering, offline delivery, read receipts, and media sharing.

Step 1: Requirements Gathering

Functional Requirements

  1. One-on-one messaging between users
  2. Group chat (up to 500 members)
  3. Message types: text, image, video, file
  4. Delivery status: sent, delivered, read
  5. Online/offline presence indicators
  6. Push notifications for offline users
  7. Message history and search
  8. End-to-end encryption (conceptual)

Non-Functional Requirements

  1. Real-time delivery: messages appear within 100ms for online users
  2. Reliable: no message loss, even during server failures
  3. Ordered: messages in a conversation appear in the correct order
  4. Scalable: support 50M+ concurrent connections
  5. Available: 99.99% uptime

Step 2: Capacity Estimation

Traffic Estimates

  • Daily Active Users (DAU): 500M
  • Average messages per user per day: 40
  • Total messages per day: 20B
  • Messages per second: ~230,000
  • Peak: ~500,000 messages/second
  • Concurrent WebSocket connections: 50M+

Storage Estimates

  • Average text message size: 200 bytes
  • Text messages per day: 20B x 200 bytes = 4 TB/day
  • Media messages (20% of total): 4B messages x 100 KB average = 400 TB/day
  • 5-year text retention: 4 TB x 365 x 5 = 7.3 PB
  • Media stored in object storage (S3) separately

Bandwidth

  • Incoming text: 4 TB/day = ~370 MB/s
  • Incoming media: 400 TB/day = ~37 GB/s (media upload bandwidth)
  • Outgoing: similar to incoming (messages are delivered to recipients)

Key Design Decision: WebSocket Connection Management

Each client maintains a persistent WebSocket connection to a WebSocket server. The critical question is: when Client A sends a message to Client B, how does Server 1 know which server Client B is connected to?

Connection Registry (Redis):
  user:clientB -> ws-server-2:port
  user:clientA -> ws-server-1:port
  user:clientC -> ws-server-3:port

Message Flow:
  1. Client A sends message to WS Server 1
  2. WS Server 1 looks up Client B's server in Redis
  3. WS Server 1 publishes message to Kafka topic
  4. WS Server 2 (subscribed to Kafka) receives message
  5. WS Server 2 delivers message to Client B via WebSocket

WebSocket Server Implementation

import { WebSocketServer } from 'ws';
import Redis from 'ioredis';
import { Kafka } from 'kafkajs';

const wss = new WebSocketServer({ port: 8080 });
const redis = new Redis();
const pub = new Redis();  // separate connection for pub/sub
const sub = new Redis();

const connections = new Map();  // userId -> WebSocket

const kafka = new Kafka({ brokers: ['kafka-1:9092', 'kafka-2:9092'] });
const producer = kafka.producer();
const consumer = kafka.consumer({ groupId: `ws-server-${SERVER_ID}` });

wss.on('connection', async (ws, req) => {
  const userId = authenticateWebSocket(req);
  connections.set(userId, ws);

  // Register this user's WebSocket server in Redis
  await redis.set(`ws:user:${userId}`, SERVER_ID, 'EX', 300);

  // Update presence
  await redis.hset(`presence:${userId}`, 'status', 'online', 'last_seen', Date.now());
  await pub.publish('presence', JSON.stringify({ userId, status: 'online' }));

  ws.on('message', async (raw) => {
    const event = JSON.parse(raw);

    switch (event.type) {
      case 'send_message':
        await handleSendMessage(userId, event.data);
        break;
      case 'typing':
        await handleTyping(userId, event.data);
        break;
      case 'read_receipt':
        await handleReadReceipt(userId, event.data);
        break;
    }
  });

  ws.on('close', async () => {
    connections.delete(userId);
    await redis.del(`ws:user:${userId}`);
    await redis.hset(`presence:${userId}`, 'status', 'offline', 'last_seen', Date.now());
    await pub.publish('presence', JSON.stringify({ userId, status: 'offline' }));
  });
});

async function handleSendMessage(senderId, data) {
  const { conversation_id, content, content_type, client_msg_id } = data;

  // Deduplication check
  const isDup = await redis.set(`dedup:${client_msg_id}`, '1', 'EX', 3600, 'NX');
  if (!isDup) return;

  // Generate server-side message ID and timestamp
  const message = {
    id: generateMessageId(),
    conversationId: conversation_id,
    senderId,
    content,
    contentType: content_type,
    timestamp: new Date().toISOString(),
    status: 'sent'
  };

  // Persist message
  await messageStore.save(message);

  // Get conversation participants
  const participants = await getConversationParticipants(conversation_id);

  // Produce to Kafka for fan-out to recipients
  await producer.send({
    topic: 'chat-messages',
    messages: [{
      key: conversation_id,
      value: JSON.stringify({
        message,
        recipients: participants.filter(id => id !== senderId)
      })
    }]
  });

  // Ack back to sender
  const senderWs = connections.get(senderId);
  if (senderWs) {
    senderWs.send(JSON.stringify({
      type: 'message_ack',
      data: { client_msg_id, message_id: message.id, timestamp: message.timestamp }
    }));
  }
}

Offline Message Delivery

When a user is offline, messages must be stored and delivered when they reconnect:

// In the Kafka consumer (handles message delivery)
await consumer.run({
  eachMessage: async ({ message }) => {
    const { message: chatMsg, recipients } = JSON.parse(message.value);

    for (const recipientId of recipients) {
      const ws = connections.get(recipientId);

      if (ws && ws.readyState === WebSocket.OPEN) {
        // Online: deliver immediately via WebSocket
        ws.send(JSON.stringify({ type: 'new_message', data: chatMsg }));
      } else {
        // Offline: store in pending messages
        await redis.lpush(`pending:${recipientId}`, JSON.stringify(chatMsg));
        // Trigger push notification
        await pushNotificationQueue.add('send', {
          userId: recipientId,
          title: `New message from ${chatMsg.senderName}`,
          body: chatMsg.content.substring(0, 100)
        });
      }
    }
  }
});

// When user reconnects, deliver pending messages
wss.on('connection', async (ws, req) => {
  const userId = authenticateWebSocket(req);
  // ... setup ...

  // Deliver pending messages
  const pending = await redis.lrange(`pending:${userId}`, 0, -1);
  for (const msg of pending.reverse()) {
    ws.send(JSON.stringify({ type: 'new_message', data: JSON.parse(msg) }));
  }
  await redis.del(`pending:${userId}`);
});

Step 6: Database Schema

For a chat system at WhatsApp scale, we use different databases for different data:

Message Store (Cassandra -- optimized for write-heavy, time-series data)

-- Cassandra table: partition by conversation, cluster by timestamp
CREATE TABLE messages (
  conversation_id  UUID,
  message_id       TIMEUUID,
  sender_id        BIGINT,
  content          TEXT,
  content_type     TEXT,        -- text, image, video, file
  media_url        TEXT,
  status           TEXT,
  created_at       TIMESTAMP,
  PRIMARY KEY (conversation_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);

Conversation Metadata (PostgreSQL)

CREATE TABLE conversations (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  type            VARCHAR(10) NOT NULL,  -- direct, group
  name            VARCHAR(200),          -- for group chats
  avatar_url      TEXT,
  created_by      BIGINT REFERENCES users(id),
  created_at      TIMESTAMP DEFAULT NOW(),
  updated_at      TIMESTAMP DEFAULT NOW()
);

CREATE TABLE conversation_members (
  conversation_id UUID REFERENCES conversations(id),
  user_id         BIGINT REFERENCES users(id),
  role            VARCHAR(10) DEFAULT 'member',  -- admin, member
  joined_at       TIMESTAMP DEFAULT NOW(),
  last_read_msg   UUID,
  muted_until     TIMESTAMP,
  PRIMARY KEY (conversation_id, user_id)
);

CREATE INDEX idx_conv_members_user ON conversation_members(user_id);

-- For direct messages, find existing conversation between two users
CREATE TABLE direct_conversations (
  user_id_1       BIGINT,
  user_id_2       BIGINT,
  conversation_id UUID REFERENCES conversations(id),
  PRIMARY KEY (user_id_1, user_id_2)
);

Step 7: Presence System

// Presence is tracked in Redis with TTL-based heartbeats
const PRESENCE_TTL = 60; // seconds

class PresenceService {
  constructor(redis) {
    this.redis = redis;
  }

  async setOnline(userId) {
    await this.redis.setex(`presence:${userId}`, PRESENCE_TTL, 'online');
  }

  async heartbeat(userId) {
    // Client sends heartbeat every 30 seconds
    await this.redis.expire(`presence:${userId}`, PRESENCE_TTL);
  }

  async isOnline(userId) {
    return await this.redis.exists(`presence:${userId}`);
  }

  // Get presence for multiple users (conversation member list)
  async getBulkPresence(userIds) {
    const pipeline = this.redis.pipeline();
    userIds.forEach(id => pipeline.get(`presence:${id}`));
    const results = await pipeline.exec();
    return userIds.map((id, i) => ({
      userId: id,
      online: results[i][1] !== null
    }));
  }

  async setOffline(userId) {
    await this.redis.del(`presence:${userId}`);
    // Store last_seen timestamp
    await this.redis.set(`last_seen:${userId}`, Date.now());
  }
}

Step 8: Scaling Strategies

WebSocket Server Scaling

A single Node.js server can handle ~50K-100K concurrent WebSocket connections. For 50M connections:

50M connections / 50K per server = 1,000 WebSocket servers

Use a load balancer with sticky sessions (based on user ID hash) so reconnections hit the same server when possible.

Message Ordering

Kafka guarantees ordering within a partition. By using conversation_id as the partition key, all messages in a conversation go to the same partition, ensuring order:

await producer.send({
  topic: 'chat-messages',
  messages: [{
    key: conversation_id,  // partition key = conversation
    value: JSON.stringify(message)
  }]
});

Group Message Fan-Out

For a group of 500 members, a single message must be delivered to 499 recipients. Two approaches:

  1. Fan-out on write: When a message is sent, create 499 delivery records and route to each user's server. Pro: fast reads. Con: high write amplification.
  2. Fan-out on read: Store one copy of the message. When each member opens the chat, they pull new messages. Pro: efficient writes. Con: slower reads.

Hybrid approach: fan-out on write for small groups (<100 members), fan-out on read for large groups.

Media Handling

Media files should never flow through the chat servers:

1. Client requests upload URL -> API returns presigned S3 URL
2. Client uploads media directly to S3
3. Client sends message with media_url reference
4. Recipient downloads media directly from S3/CDN

Decision Option A Option B Recommendation
Transport WebSocket Long polling WebSocket (bidirectional, lower overhead)
Message store Cassandra MongoDB Cassandra (better write throughput, time-series)
Message bus Kafka Redis Pub/Sub Kafka (durable, ordered, replayable)
Group delivery Fan-out on write Fan-out on read Hybrid based on group size
Presence Heartbeat + TTL Connection tracking Heartbeat + TTL (simpler, handles crashes)
Encryption Server-side End-to-end E2E for privacy (server cannot read messages)

End-to-End Encryption (Conceptual)

In E2E encryption, the server cannot read message content:

1. Each user has a public/private key pair
2. Sender encrypts message with recipient's public key
3. Server stores encrypted blob -- cannot decrypt it
4. Recipient decrypts with their private key

Challenge: Group chats require the Signal Protocol's
"sender keys" scheme for efficient group encryption.

Summary

A chat system is a real-time, write-heavy, connection-heavy application. The architecture centers on WebSocket servers for persistent connections, Kafka for reliable message routing between servers, Cassandra for high-throughput message storage, and Redis for presence and connection registry. Key design decisions include using conversation_id as the Kafka partition key for message ordering, implementing a heartbeat-based presence system, handling offline delivery with pending message queues and push notifications, and choosing between fan-out strategies for group chats based on group size.

Common Problems & Solutions

1 Messages arrive out of order in group chats
Cause: Messages from different senders are processed by different WebSocket servers and Kafka consumers, leading to non-deterministic delivery order.
Solution: Use conversation_id as the Kafka partition key so all messages in a conversation go to the same partition, which guarantees ordering. On the client, sort messages by server-assigned timestamps. Use TIMEUUID or Snowflake IDs that embed timestamps for natural ordering.
2 WebSocket server runs out of memory with too many connections
Cause: Each WebSocket connection consumes memory for the socket buffer and associated state. A single Node.js process trying to handle 200K+ connections exhausts available RAM.
Solution: Limit connections per server to 50K-100K. Scale horizontally with more WebSocket server instances behind a load balancer. Use sticky sessions so a user's reconnection attempts go to the same server. Monitor memory per connection and tune buffer sizes.
3 Presence status shows users as online when they have actually crashed
Cause: If a client crashes without cleanly closing the WebSocket, the server may not immediately detect the disconnect. TCP keep-alive timeouts can be minutes long.
Solution: Implement application-level heartbeats (ping/pong every 30 seconds). Use Redis key expiry (TTL of 60 seconds) for presence. If a heartbeat is missed, the key expires and the user appears offline. Also configure WebSocket ping intervals on the server side.
