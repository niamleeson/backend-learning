Hashing: bcrypt, argon2, and Password Storage

Intuition

Password hashing is like a meat grinder: you can put a steak in and get ground meat out, but you cannot reassemble the steak from the ground meat. bcrypt and argon2 are intentionally slow grinders -- they take hundreds of milliseconds on purpose -- so that an attacker who steals your ground meat cannot try billions of steaks per second to find a match.

Hashing: bcrypt, argon2, and Password Storage

Introduction

Storing passwords is one of the most security-critical tasks in backend development, and it is astonishingly easy to get wrong. If your database is compromised and you stored passwords in plaintext, every single user account is immediately exposed. Even basic encryption is insufficient because encryption is reversible — anyone with the key can recover every password. Hashing is the correct approach: a one-way mathematical function that converts a password into a fixed-length string from which the original password cannot be recovered.

In this lesson, you will learn how password hashing works, why general-purpose hash functions like SHA-256 are unsuitable for passwords, and how to use purpose-built algorithms like bcrypt and argon2 in Node.js to store passwords securely.

Why Not Store Passwords Directly?

Let us examine the progression of password storage approaches and why each inadequate method fails.

Level 0: Plaintext

Database:
+--------+-----------+
| user   | password  |
+--------+-----------+
| alice  | hunter2   |
| bob    | p@ssw0rd  |
+--------+-----------+

Attacker steals DB -> every password is immediately visible.

Level 1: Encrypted Passwords

Database:
+--------+---------------------------+
| user   | encrypted_password        |
+--------+---------------------------+
| alice  | aGVsbG8gd29ybGQ=          |
| bob    | c2VjcmV0MTIz              |
+--------+---------------------------+

Attacker steals DB + encryption key -> every password is recoverable.

Encryption is a two-way function. If the attacker obtains the key (which is often stored on the same server or in environment variables), all passwords are compromised.

Level 2: Simple Hash (SHA-256)

Database:
+--------+------------------------------------------------------------------+
| user   | hashed_password                                                  |
+--------+------------------------------------------------------------------+
| alice  | b8e9... (SHA-256 of "hunter2")                                   |
| bob    | 5e88... (SHA-256 of "p@ssw0rd")                                  |
+--------+------------------------------------------------------------------+

Problem: identical passwords produce identical hashes.
Attacker precomputes hashes for millions of common passwords (rainbow table).

Level 3: Hash + Salt (Still Not Enough with SHA)

Database:
+--------+--------+----------------------------------------------+
| user   | salt   | hashed_password                              |
+--------+--------+----------------------------------------------+
| alice  | x9f2.. | SHA-256("x9f2.." + "hunter2") = a3c1...      |
| bob    | k7m4.. | SHA-256("k7m4.." + "p@ssw0rd") = f8d2...     |
+--------+--------+----------------------------------------------+

Rainbow tables defeated, but SHA-256 is too fast.
Modern GPUs compute billions of SHA-256 hashes per second.

Level 4: Purpose-Built Password Hashing (bcrypt / argon2)

Database:
+--------+--------------------------------------------------------------+
| user   | hashed_password                                              |
+--------+--------------------------------------------------------------+
| alice  | $2b$12$LJ3m4... (bcrypt with salt embedded, cost=12)         |
| bob    | $2b$12$Kx9p7... (bcrypt with salt embedded, cost=12)         |
+--------+--------------------------------------------------------------+

Salt is embedded in the hash.
Deliberately slow: ~250ms per hash at cost=12.
GPU-resistant (bcrypt) or memory-hard (argon2).

How Password Hashing Works

The general flow for registration and login looks like this:

REGISTRATION:

  User enters password
        |
        v
  Generate random salt
        |
        v
  hash(password + salt) --> hashed_password
        |
        v
  Store hashed_password in DB (salt is embedded)


LOGIN:

  User enters password
        |
        v
  Retrieve hashed_password from DB
        |
        v
  Extract salt from stored hash
        |
        v
  hash(entered_password + extracted_salt)
        |
        v
  Compare new hash with stored hash
        |
        v
  Match? --> Login success
  No match? --> Login failure

bcrypt: The Industry Workhorse

bcrypt has been the standard password hashing algorithm since 1999. It is based on the Blowfish cipher and has three key properties:

  1. Built-in salt -- the salt is generated automatically and embedded in the output
  2. Configurable cost factor -- the saltRounds (or cost) parameter controls how many iterations are performed, making it deliberately slow
  3. GPU resistance -- bcrypt requires frequent memory accesses that make it harder to parallelize on GPUs compared to SHA

Installing bcrypt

npm install bcrypt
npm install -D @types/bcrypt

There are two bcrypt packages commonly used in Node.js: - bcrypt -- native C++ addon, faster, requires build tools (node-gyp) - bcryptjs -- pure JavaScript implementation, no native dependencies, slightly slower

For production, prefer bcrypt. For environments where native compilation is problematic (like some CI pipelines), use bcryptjs.

Hashing a Password

const bcrypt = require('bcrypt');

// The cost factor (saltRounds) determines computational expense
const SALT_ROUNDS = 12;

async function hashPassword(plainPassword) {
  // bcrypt.hash generates a random salt and hashes in one step
  const hashedPassword = await bcrypt.hash(plainPassword, SALT_ROUNDS);
  return hashedPassword;
}

// Example usage
async function demo() {
  const hash = await hashPassword('mySecretPassword');
  console.log(hash);
  // Output: $2b$12$LJ3m4x5p9qRtY1W2Z3v4a.7X8K9L0M1N2O3P4Q5R6S7T8U9V0W1X2
}
demo();

Anatomy of a bcrypt Hash

$2b$12$LJ3m4x5p9qRtY1W2Z3v4a.7X8K9L0M1N2O3P4Q5R6S7T8U9V0W1X2
 ^^  ^^  ^^^^^^^^^^^^^^^^^^^^^^ ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
 |   |   |                      |
 |   |   22-char salt           31-char hash
 |   |
 |   cost factor (2^12 = 4096 iterations)
 |
 algorithm version (2b = current)

The salt and hash are combined into a single string. You never need to store the salt separately.

Verifying a Password

const bcrypt = require('bcrypt');

async function verifyPassword(plainPassword, storedHash) {
  // bcrypt.compare extracts the salt from storedHash,
  // hashes plainPassword with that salt,
  // and compares the result
  const isMatch = await bcrypt.compare(plainPassword, storedHash);
  return isMatch;
}

async function demo() {
  const hash = await bcrypt.hash('mySecretPassword', 12);

  console.log(await verifyPassword('mySecretPassword', hash)); // true
  console.log(await verifyPassword('wrongPassword', hash));    // false
}
demo();

Choosing the Right Cost Factor

The cost factor should make hashing take roughly 250ms on your production hardware. Here is how to benchmark it:

const bcrypt = require('bcrypt');

async function benchmarkBcrypt() {
  const password = 'testPassword123';

  for (let rounds = 8; rounds <= 16; rounds++) {
    const start = Date.now();
    await bcrypt.hash(password, rounds);
    const elapsed = Date.now() - start;
    console.log(`Cost ${rounds}: ${elapsed}ms`);
  }
}
benchmarkBcrypt();

// Typical output on a modern server:
// Cost 8:   ~15ms
// Cost 10:  ~60ms
// Cost 12:  ~250ms   <-- good starting point
// Cost 14:  ~1000ms
// Cost 16:  ~4000ms

Start with cost 12 and increase it as hardware improves. You can rehash passwords at login when you increase the cost factor.

Rehashing on Cost Upgrade

const bcrypt = require('bcrypt');

const CURRENT_COST = 14; // upgraded from 12

async function loginAndUpgradeHash(plainPassword, storedHash) {
  const isValid = await bcrypt.compare(plainPassword, storedHash);
  if (!isValid) return { success: false };

  // Check if the stored hash uses an outdated cost factor
  const storedRounds = bcrypt.getRounds(storedHash);
  let newHash = null;

  if (storedRounds < CURRENT_COST) {
    // Rehash with the new cost factor
    newHash = await bcrypt.hash(plainPassword, CURRENT_COST);
    // Update the hash in the database
  }

  return { success: true, newHash };
}

argon2: The Modern Alternative

Argon2 won the Password Hashing Competition in 2015. It has a key advantage over bcrypt: it is memory-hard, meaning it requires a configurable amount of RAM in addition to CPU time. This makes it even more resistant to GPU and ASIC attacks.

Argon2 comes in three variants: - argon2d -- data-dependent memory access, faster but vulnerable to side-channel attacks - argon2i -- data-independent memory access, safer against side-channel attacks - argon2id -- hybrid (recommended), uses argon2i for the first pass and argon2d for subsequent passes

Installing argon2

npm install argon2

Hashing with argon2

const argon2 = require('argon2');

async function hashWithArgon2(password) {
  const hash = await argon2.hash(password, {
    type: argon2.argon2id,  // recommended variant
    memoryCost: 65536,      // 64 MB of RAM
    timeCost: 3,            // 3 iterations
    parallelism: 4          // 4 threads
  });
  return hash;
}

async function verifyWithArgon2(password, storedHash) {
  return await argon2.verify(storedHash, password);
}

async function demo() {
  const hash = await hashWithArgon2('mySecretPassword');
  console.log(hash);
  // $argon2id$v=19$m=65536,t=3,p=4$c2FsdHNhbHQ$...

  console.log(await verifyWithArgon2('mySecretPassword', hash)); // true
  console.log(await verifyWithArgon2('wrongPassword', hash));    // false
}
demo();

Anatomy of an argon2 Hash

$argon2id$v=19$m=65536,t=3,p=4$c2FsdHNhbHQ$hashvalue
 ^^^^^^^^ ^^^^ ^^^^^^^^^^^^^^^^ ^^^^^^^^^^^^ ^^^^^^^^^
 |        |    |                |             |
 |        |    |                salt (base64) hash (base64)
 |        |    |
 |        |    m=memory, t=iterations, p=parallelism
 |        |
 |        version
 |
 algorithm variant

bcrypt vs argon2: When to Use Which

+------------------+---------------------------+---------------------------+
| Feature          | bcrypt                    | argon2id                  |
+------------------+---------------------------+---------------------------+
| Age              | 1999 (battle-tested)      | 2015 (modern)             |
| Memory-hard      | No                        | Yes (configurable)        |
| GPU resistance   | Moderate                  | Strong                    |
| ASIC resistance  | Low                       | High                      |
| Max password len | 72 bytes                  | No practical limit        |
| Configuration    | Single cost factor        | Memory, time, parallelism |
| Ecosystem        | Universal support         | Growing support           |
| OWASP rec.       | Acceptable                | Preferred                 |
+------------------+---------------------------+---------------------------+

Use argon2id for new projects. It is the OWASP-recommended algorithm. Use bcrypt when argon2 is not available or for compatibility with existing systems.

Building a Complete User Authentication Module

const bcrypt = require('bcrypt');
const { randomBytes } = require('node:crypto');

const SALT_ROUNDS = 12;
const MIN_PASSWORD_LENGTH = 8;

class PasswordService {
  /**
   * Validate password strength before hashing.
   * Returns an array of error messages (empty if valid).
   */
  static validateStrength(password) {
    const errors = [];

    if (password.length < MIN_PASSWORD_LENGTH) {
      errors.push(`Password must be at least ${MIN_PASSWORD_LENGTH} characters`);
    }
    if (!/[A-Z]/.test(password)) {
      errors.push('Password must contain at least one uppercase letter');
    }
    if (!/[a-z]/.test(password)) {
      errors.push('Password must contain at least one lowercase letter');
    }
    if (!/[0-9]/.test(password)) {
      errors.push('Password must contain at least one digit');
    }

    return errors;
  }

  /**
   * Hash a password using bcrypt.
   */
  static async hash(password) {
    return bcrypt.hash(password, SALT_ROUNDS);
  }

  /**
   * Verify a password against a stored hash.
   */
  static async verify(password, storedHash) {
    return bcrypt.compare(password, storedHash);
  }

  /**
   * Check if a stored hash needs to be upgraded.
   */
  static needsRehash(storedHash) {
    const rounds = bcrypt.getRounds(storedHash);
    return rounds < SALT_ROUNDS;
  }

  /**
   * Generate a secure random password reset token.
   */
  static generateResetToken() {
    return randomBytes(32).toString('hex');
  }
}

module.exports = { PasswordService };

Timing Attacks and Constant-Time Comparison

When comparing hashes manually, you must use constant-time comparison to prevent timing attacks. Both bcrypt.compare and argon2.verify already do this internally, but if you ever need to compare tokens or hashes yourself:

const { timingSafeEqual } = require('node:crypto');

function safeCompare(a, b) {
  if (a.length !== b.length) return false;

  const bufA = Buffer.from(a);
  const bufB = Buffer.from(b);

  return timingSafeEqual(bufA, bufB);
}

A naive === comparison returns early on the first mismatched character, which leaks information about how many characters matched. timingSafeEqual always takes the same amount of time regardless of where the difference occurs.

TIMING ATTACK:

Naive comparison: "abc" === "axyz"
  Compare 'a' == 'a' -> match, continue   (~1 unit of time)
  Compare 'b' == 'x' -> mismatch, RETURN  (~2 units of time)

Naive comparison: "abc" === "abd"
  Compare 'a' == 'a' -> match, continue   (~1 unit)
  Compare 'b' == 'b' -> match, continue   (~2 units)
  Compare 'c' == 'd' -> mismatch, RETURN  (~3 units)

Attacker can deduce correct characters by measuring response time!

Constant-time comparison: always checks ALL characters.
  Takes the same time regardless of where differences are.

Common Password Storage Mistakes

1. Using MD5 or SHA for Passwords

// NEVER DO THIS
const crypto = require('node:crypto');
const hash = crypto.createHash('sha256').update(password).digest('hex');
// SHA-256 can compute billions of hashes per second on a GPU

2. Truncating Passwords Before Hashing

// WRONG: silently truncates long passwords
const truncated = password.substring(0, 20);
const hash = await bcrypt.hash(truncated, 12);

// Note: bcrypt itself truncates at 72 bytes.
// If this is a concern, use argon2 or pre-hash with SHA-256:
const preHash = crypto.createHash('sha256').update(password).digest('base64');
const hash2 = await bcrypt.hash(preHash, 12);

3. Logging Passwords

// NEVER log passwords, even in development
console.log(`User login: ${email}, password: ${password}`); // TERRIBLE
console.log(`User login: ${email}`); // Correct

4. Not Using Constant Error Messages

// WRONG: reveals whether email exists
if (!user) return res.status(404).json({ error: 'User not found' });
if (!match) return res.status(401).json({ error: 'Wrong password' });

// CORRECT: same message for both cases
if (!user || !match) {
  return res.status(401).json({ error: 'Invalid credentials' });
}

Pepper: An Additional Layer

A pepper is a secret value stored outside the database (typically in environment variables) that is combined with the password before hashing:

const PEPPER = process.env.PASSWORD_PEPPER; // e.g., a 32-char random string

async function hashWithPepper(password) {
  const pepperedPassword = password + PEPPER;
  return bcrypt.hash(pepperedPassword, 12);
}

async function verifyWithPepper(password, storedHash) {
  const pepperedPassword = password + PEPPER;
  return bcrypt.compare(pepperedPassword, storedHash);
}

If the database is compromised but the application server (where the pepper lives) is not, the attacker cannot crack the passwords even with unlimited computing power.

Summary

Never store passwords in plaintext or use general-purpose hash functions. Use bcrypt (cost 12+) or argon2id with appropriate memory and time parameters. Always use the library's built-in comparison functions for timing-safe verification. Return identical error messages for "user not found" and "wrong password" to prevent user enumeration. Consider adding a pepper for defense in depth. Periodically increase cost factors and rehash passwords at login time.

Common Problems & Solutions

1 bcrypt.hash throws 'Error: data and salt arguments required' even though a password is provided
Cause: The password variable is undefined or null, often because req.body was not parsed (missing express.json() middleware) or the field name in the request does not match the destructured variable name.
Solution: Ensure app.use(express.json()) is registered before the route. Add a check: if (!password) return res.status(400)... before calling bcrypt.hash(). Log typeof password to debug.
2 bcrypt.compare always returns false even with the correct password
Cause: The stored hash was truncated when saved to the database. bcrypt hashes are 60 characters long, but the database column may be defined as VARCHAR(50) or similar, silently truncating the hash.
Solution: Ensure the password column in your database is at least VARCHAR(60) for bcrypt or VARCHAR(128) for argon2. Check the stored hash length: if it is less than 60 characters, the column is too short.
3 Installing the bcrypt package fails with node-gyp errors on macOS or Windows
Cause: The native bcrypt package requires C++ compilation tools (node-gyp, Python, make). These are not always installed on developer machines or CI environments.
Solution: Install build tools: on macOS run 'xcode-select --install', on Windows install windows-build-tools. Alternatively, switch to bcryptjs which is a pure JavaScript implementation: npm install bcryptjs. The API is identical.
