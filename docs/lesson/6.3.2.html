Unary and Streaming RPCs with grpcio

Intuition

Unary RPC is like making a phone call to ask one question and getting one answer. Server streaming is like calling a radio station and listening to a continuous broadcast. Bidirectional streaming is like a live phone conversation where both parties talk freely.

Unary and Streaming RPCs with grpcio

gRPC supports four communication patterns: unary (request-response), server streaming, client streaming, and bidirectional streaming. Each pattern suits different use cases.

Unary RPC (Request-Response)

# Server implementation
import grpc
from concurrent import futures
import user_pb2
import user_pb2_grpc

class UserServicer(user_pb2_grpc.UserServiceServicer):
    def GetUser(self, request, context):
        # request.id is the user ID
        user = db.get_user(request.id)
        if not user:
            context.abort(grpc.StatusCode.NOT_FOUND, "User not found")
        return user_pb2.User(id=user.id, name=user.name, email=user.email)

def serve():
    server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
    user_pb2_grpc.add_UserServiceServicer_to_server(UserServicer(), server)
    server.add_insecure_port('[::]:50051')
    server.start()
    server.wait_for_termination()

Server Streaming

class UserServicer(user_pb2_grpc.UserServiceServicer):
    def ListUsers(self, request, context):
        """Server streams users one at a time"""
        users = db.get_all_users(page_size=request.page_size)
        for user in users:
            yield user_pb2.User(id=user.id, name=user.name, email=user.email)

Client Code

import grpc
import user_pb2
import user_pb2_grpc

channel = grpc.insecure_channel('localhost:50051')
stub = user_pb2_grpc.UserServiceStub(channel)

# Unary call
response = stub.GetUser(user_pb2.GetUserRequest(id=1))
print(response.name)

# Server streaming
for user in stub.ListUsers(user_pb2.ListUsersRequest(page_size=100)):
    print(user.name)

Common Mistakes

  • Forgetting to call context.abort() for errors (returning empty messages instead)
  • Not setting appropriate deadlines on client calls
  • Using insecure channels in production

Key Takeaways

  • Unary RPCs are like REST calls: one request, one response
  • Server streaming is ideal for large result sets or real-time feeds
  • Client streaming is useful for upload scenarios (e.g., file chunks)
  • Bidirectional streaming enables chat-like interactions

Exercises

  1. Build a Unary gRPC Service

    Create a calculator gRPC service with Add, Subtract, Multiply, and Divide RPCs. Handle division by zero with an appropriate gRPC status code. Write both server and client.

    Hint

    Use grpc.StatusCode.INVALID_ARGUMENT for division by zero. Define a CalculatorRequest with two float fields and a CalculatorResponse with a result field.

  2. Server Streaming Log Service

    Build a log streaming service where clients subscribe to a log stream. The server yields log entries as they appear. Implement a filter parameter so clients can subscribe to specific log levels (INFO, WARNING, ERROR).

    Hint

    Use a generator function that yields log entries. Simulate log generation with time.sleep(). Filter entries by the requested log level before yielding.

  3. Client Streaming File Upload

    Implement a file upload RPC where the client streams file chunks and the server assembles them. Return the total bytes received and a checksum in the response.

    Hint

    Define a FileChunk message with bytes data and a filename field. The server accumulates chunks, and after the client finishes streaming, it returns an UploadResponse with the total size.

Common Problems & Solutions

1 gRPC calls hang indefinitely
Symptoms: ['Client application freezes', 'Thread pool exhaustion on client side']
Cause: No deadline or timeout set on the client call, and the server is unresponsive.
Solution: Always set deadlines on gRPC calls: stub.GetUser(request, timeout=5.0). On the server side, check context.is_active() in long-running operations.

Set a 5-second timeout

response = stub.GetUser(request, timeout=5.0)

2 Connection refused when connecting to gRPC server
Cause: Server is listening on localhost but client connects to 0.0.0.0, or port mismatch between server and client.
Solution: Ensure server uses '[::]:50051' to listen on all interfaces. Match the port exactly in the client channel. Use grpc.insecure_channel('localhost:50051') for local development.
