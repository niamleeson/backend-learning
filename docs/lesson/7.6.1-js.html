Circuit Breakers and Retry Patterns

Intuition

A circuit breaker works like the electrical breaker box in your house. When a short circuit (failed service) causes too much current (failed requests), the breaker trips and cuts the connection instantly, protecting the entire house (your application) from catching fire (cascading failure). After a cooldown, you can flip the switch to test if the problem is fixed.

Circuit Breakers and Retry Patterns

Why Services Fail (And Why You Should Expect It)

In a distributed backend system, failures are not exceptional -- they are inevitable. A database goes down, a third-party API times out, a downstream microservice becomes overloaded. If your Node.js application blindly retries every failed request or keeps hammering an unresponsive service, you amplify the problem instead of containing it.

Circuit breakers and retry patterns are the two foundational resilience mechanisms that prevent cascading failures and allow your system to degrade gracefully.

The Cascading Failure Problem

Consider a typical request flow:

┌────────┐     ┌──────────┐     ┌──────────────┐     ┌──────────────┐
│ Client │────>│ API GW   │────>│ Order Service│────>│ Payment Svc  │
└────────┘     └──────────┘     └──────────────┘     └──────────────┘
                                       │                    │
                                       │              (timeout!)
                                       │                    │
                                       ▼                    ▼
                                 Threads blocked      Database overloaded
                                 Memory grows         Connections exhausted
                                 OOM crash            Total outage

When Payment Service becomes slow, Order Service threads pile up waiting for responses. Request queues grow, memory consumption spikes, and eventually Order Service itself crashes. This chain reaction is a cascading failure.

Circuit Breaker Pattern

The Concept

A circuit breaker monitors calls to an external dependency. When failures exceed a threshold, the circuit "opens" and immediately rejects subsequent calls without attempting the operation. After a cooldown period, the circuit enters a "half-open" state, allowing a limited number of test requests through. If those succeed, the circuit "closes" and normal operation resumes.

                    ┌──────────────────────────────────────────┐
                    │         Circuit Breaker States            │
                    │                                          │
                    │   ┌────────┐  failures > threshold       │
                    │   │        │──────────────────┐          │
                    │   │ CLOSED │                  │          │
                    │   │        │<─────┐           │          │
                    │   └────────┘      │           ▼          │
                    │    (normal)    success    ┌────────┐     │
                    │                from       │        │     │
                    │                half-open  │  OPEN  │     │
                    │                   │       │        │     │
                    │                   │       └───┬────┘     │
                    │                   │    (fail  │          │
                    │              ┌────┴───┐ fast) │          │
                    │              │        │       │          │
                    │              │  HALF  │<──────┘          │
                    │              │  OPEN  │  after timeout   │
                    │              │        │                  │
                    │              └────────┘                  │
                    │               (test                      │
                    │                request)                  │
                    └──────────────────────────────────────────┘

CLOSED -- Requests pass through normally. Failures are counted. When the failure count (or failure rate) exceeds a configured threshold within a time window, the circuit trips open.

OPEN -- All requests are immediately rejected with an error (or a fallback response). No calls reach the downstream service. A reset timer begins counting down.

HALF-OPEN -- After the reset timeout, a limited number of probe requests are allowed through. If they succeed, the circuit closes. If they fail, the circuit reopens.

Implementing Circuit Breakers with Opossum

Opossum is the most popular circuit breaker library for Node.js. It wraps any async function and adds circuit breaker semantics.

npm install opossum

Basic Usage

const CircuitBreaker = require('opossum');
const axios = require('axios');

// The function to protect with a circuit breaker
async function fetchUserProfile(userId) {
  const response = await axios.get(
    `https://user-service.internal/users/${userId}`,
    { timeout: 3000 }
  );
  return response.data;
}

// Create a circuit breaker around the function
const breaker = new CircuitBreaker(fetchUserProfile, {
  timeout: 5000,            // Time in ms before a call is considered failed
  errorThresholdPercentage: 50,  // Trip when 50% of requests fail
  resetTimeout: 10000,      // Try again after 10 seconds
  volumeThreshold: 5,       // Minimum requests before tripping
  rollingCountTimeout: 10000,    // Window for calculating error percentage
  rollingCountBuckets: 10,  // Number of buckets in the rolling window
});

// Use the circuit breaker
async function handleGetUser(req, res) {
  try {
    const user = await breaker.fire(req.params.userId);
    res.json(user);
  } catch (err) {
    if (err.message === 'Breaker is open') {
      res.status(503).json({
        error: 'Service temporarily unavailable',
        retryAfter: 10,
      });
    } else {
      res.status(500).json({ error: 'Failed to fetch user profile' });
    }
  }
}

Event Handling and Observability

Opossum emits events you should wire into your logging and metrics:

const pino = require('pino');
const logger = pino();

breaker.on('success', (result) => {
  logger.debug({ service: 'user-service' }, 'Circuit breaker call succeeded');
});

breaker.on('timeout', () => {
  logger.warn({ service: 'user-service' }, 'Circuit breaker call timed out');
});

breaker.on('reject', () => {
  logger.warn({ service: 'user-service' }, 'Circuit breaker rejected call (open)');
});

breaker.on('open', () => {
  logger.error({ service: 'user-service' }, 'Circuit breaker OPENED');
});

breaker.on('halfOpen', () => {
  logger.info({ service: 'user-service' }, 'Circuit breaker HALF-OPEN, testing...');
});

breaker.on('close', () => {
  logger.info({ service: 'user-service' }, 'Circuit breaker CLOSED, service recovered');
});

breaker.on('fallback', (result) => {
  logger.info({ service: 'user-service' }, 'Circuit breaker fallback executed');
});

Fallback Strategies

Instead of returning an error when the circuit is open, you can provide a fallback:

// Return cached data when circuit is open
const NodeCache = require('node-cache');
const cache = new NodeCache({ stdTTL: 300 });

breaker.fallback((userId) => {
  const cached = cache.get(`user:${userId}`);
  if (cached) {
    return { ...cached, _stale: true };
  }
  throw new Error('No cached data available');
});

// Cache successful responses
breaker.on('success', (result) => {
  if (result && result.id) {
    cache.set(`user:${result.id}`, result);
  }
});

Multiple Circuit Breakers

In a real application, create a separate circuit breaker for each external dependency:

// services/circuitBreakers.js
const CircuitBreaker = require('opossum');

const defaultOptions = {
  timeout: 5000,
  errorThresholdPercentage: 50,
  resetTimeout: 15000,
  volumeThreshold: 5,
};

function createBreaker(fn, name, overrides = {}) {
  const breaker = new CircuitBreaker(fn, {
    ...defaultOptions,
    ...overrides,
    name,
  });

  breaker.on('open', () => console.error(`[${name}] Circuit OPENED`));
  breaker.on('close', () => console.info(`[${name}] Circuit CLOSED`));
  breaker.on('halfOpen', () => console.info(`[${name}] Circuit HALF-OPEN`));

  return breaker;
}

const userServiceBreaker = createBreaker(
  fetchUserProfile,
  'user-service',
  { timeout: 3000 }
);

const paymentServiceBreaker = createBreaker(
  processPayment,
  'payment-service',
  { timeout: 10000, errorThresholdPercentage: 30 }
);

const emailServiceBreaker = createBreaker(
  sendEmail,
  'email-service',
  { timeout: 8000, errorThresholdPercentage: 70 }
);

module.exports = {
  userServiceBreaker,
  paymentServiceBreaker,
  emailServiceBreaker,
};

Retry Patterns

Retries handle transient failures -- brief network blips, momentary server overloads, or temporary database locks. The key is retrying intelligently.

Naive Retry (Do NOT Do This)

// BAD: Immediate retries with no backoff
async function fetchWithRetry(url, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await axios.get(url);
    } catch (err) {
      if (i === maxRetries - 1) throw err;
      // Immediately retries -- hammers the server!
    }
  }
}

This approach creates a retry storm: when a server is struggling, hundreds of clients retrying simultaneously makes it worse.

Exponential Backoff with Jitter

The correct approach increases wait time between retries and adds randomness to prevent thundering herd:

Retry Timeline:

  Attempt 1    Attempt 2         Attempt 3                 Attempt 4
     │             │                  │                         │
     ▼             ▼                  ▼                         ▼
─────X───wait──────X─────wait────────X──────────wait───────────X────>
         ~1s            ~2s                    ~4s
     (+ jitter)     (+ jitter)            (+ jitter)

async function retryWithBackoff(fn, options = {}) {
  const {
    maxRetries = 3,
    baseDelay = 1000,
    maxDelay = 30000,
    factor = 2,
    jitter = true,
    retryableErrors = null,  // null means retry all errors
  } = options;

  let lastError;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      lastError = err;

      // Check if the error is retryable
      if (retryableErrors && !retryableErrors(err)) {
        throw err;  // Non-retryable error, fail immediately
      }

      if (attempt === maxRetries) {
        throw err;  // Exhausted all retries
      }

      // Calculate delay with exponential backoff
      let delay = Math.min(baseDelay * Math.pow(factor, attempt), maxDelay);

      // Add jitter to prevent thundering herd
      if (jitter) {
        delay = delay * (0.5 + Math.random() * 0.5);
      }

      console.warn(
        `Attempt ${attempt + 1} failed, retrying in ${Math.round(delay)}ms...`,
        err.message
      );

      await new Promise((resolve) => setTimeout(resolve, delay));
    }
  }
}

// Usage
const user = await retryWithBackoff(
  () => axios.get('https://user-service.internal/users/123'),
  {
    maxRetries: 3,
    baseDelay: 1000,
    retryableErrors: (err) => {
      // Only retry on network errors or 5xx responses
      if (!err.response) return true;  // Network error
      return err.response.status >= 500;
    },
  }
);

Using async-retry

The async-retry package provides a clean API for retries:

npm install async-retry

const retry = require('async-retry');
const axios = require('axios');

async function fetchOrder(orderId) {
  return retry(
    async (bail, attemptNumber) => {
      console.log(`Attempt ${attemptNumber} to fetch order ${orderId}`);

      try {
        const response = await axios.get(
          `https://order-service.internal/orders/${orderId}`
        );
        return response.data;
      } catch (err) {
        // If the error is a 4xx, do NOT retry (client error)
        if (err.response && err.response.status < 500) {
          bail(err);  // bail() stops retrying and throws immediately
          return;
        }
        throw err;  // Throw to trigger a retry
      }
    },
    {
      retries: 4,
      factor: 2,
      minTimeout: 1000,
      maxTimeout: 10000,
      randomize: true,
      onRetry: (err, attempt) => {
        console.warn(`Retry attempt ${attempt}: ${err.message}`);
      },
    }
  );
}

Retry with Circuit Breaker

The ideal combination: retries handle transient glitches, and the circuit breaker prevents retry storms from killing an already-struggling service.

const CircuitBreaker = require('opossum');
const retry = require('async-retry');

// Inner function: retries transient errors
async function fetchWithRetry(userId) {
  return retry(
    async (bail) => {
      try {
        const res = await axios.get(
          `https://user-service.internal/users/${userId}`,
          { timeout: 3000 }
        );
        return res.data;
      } catch (err) {
        if (err.response && err.response.status === 404) {
          bail(err);  // Do not retry 404s
          return;
        }
        throw err;
      }
    },
    { retries: 2, minTimeout: 500, factor: 2, randomize: true }
  );
}

// Outer layer: circuit breaker wraps the retry logic
const userBreaker = new CircuitBreaker(fetchWithRetry, {
  timeout: 12000,  // Account for retry delays
  errorThresholdPercentage: 50,
  resetTimeout: 30000,
  volumeThreshold: 10,
});

userBreaker.fallback((userId) => {
  return { id: userId, name: 'Unknown', _fallback: true };
});

// Usage in route handler
app.get('/users/:id', async (req, res) => {
  try {
    const user = await userBreaker.fire(req.params.id);
    res.json(user);
  } catch (err) {
    res.status(503).json({ error: 'User service unavailable' });
  }
});

│                                                  │
│  Request ──> Circuit Breaker ──> Retry Logic     │
│                    │                   │          │
│              Is circuit open?    Attempt 1 fail? │
│                    │                   │          │

│              NO: pass through    Attempt 2 fail? │

Retry Budget Pattern

A more advanced approach limits the total percentage of requests that can be retries:

class RetryBudget {
  constructor({ windowMs = 60000, maxRetryRatio = 0.1 }) {
    this.windowMs = windowMs;
    this.maxRetryRatio = maxRetryRatio;
    this.requests = [];  // timestamps of all requests
    this.retries = [];   // timestamps of retry requests
  }

  recordRequest() {
    this._cleanup();
    this.requests.push(Date.now());
  }

  canRetry() {
    this._cleanup();
    if (this.requests.length === 0) return true;
    const retryRatio = this.retries.length / this.requests.length;
    return retryRatio < this.maxRetryRatio;
  }

  recordRetry() {
    this._cleanup();
    this.retries.push(Date.now());
    this.requests.push(Date.now());
  }

  _cleanup() {
    const cutoff = Date.now() - this.windowMs;
    this.requests = this.requests.filter((t) => t > cutoff);
    this.retries = this.retries.filter((t) => t > cutoff);
  }
}

const budget = new RetryBudget({ windowMs: 60000, maxRetryRatio: 0.1 });

async function fetchWithBudgetedRetry(fn) {
  budget.recordRequest();
  try {
    return await fn();
  } catch (err) {
    if (budget.canRetry()) {
      budget.recordRetry();
      return await fn();  // One retry attempt
    }
    throw err;  // Budget exhausted, no retry
  }
}

Idempotency and Retries

Retries are only safe for idempotent operations. If you retry a non-idempotent operation (like creating an order), you may create duplicates.

// Use idempotency keys for non-idempotent operations
const { v4: uuidv4 } = require('uuid');

async function createOrder(orderData) {
  const idempotencyKey = uuidv4();

  return retry(
    async () => {
      const response = await axios.post(
        'https://order-service.internal/orders',
        orderData,
        {
          headers: {
            'Idempotency-Key': idempotencyKey,
          },
        }
      );
      return response.data;
    },
    { retries: 3, minTimeout: 1000 }
  );
}

Timeout Patterns

Timeouts are the first line of defense. Without them, a slow dependency can hold your connections indefinitely.

const axios = require('axios');

// Per-request timeout
const client = axios.create({
  baseURL: 'https://api.external.com',
  timeout: 5000,  // 5 second timeout
});

// AbortController for custom timeout logic
async function fetchWithAbort(url, timeoutMs = 5000) {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), timeoutMs);

  try {
    const response = await fetch(url, { signal: controller.signal });
    return await response.json();
  } finally {
    clearTimeout(timeoutId);
  }
}

Bulkhead Pattern

Limit concurrent calls to an external service to prevent one slow dependency from consuming all your resources:

class Bulkhead {
  constructor(maxConcurrent) {
    this.maxConcurrent = maxConcurrent;
    this.running = 0;
    this.queue = [];
  }

  async execute(fn) {
    if (this.running >= this.maxConcurrent) {
      await new Promise((resolve) => this.queue.push(resolve));
    }
    this.running++;
    try {
      return await fn();
    } finally {
      this.running--;
      if (this.queue.length > 0) {
        const next = this.queue.shift();
        next();
      }
    }
  }
}

// Only allow 10 concurrent requests to the payment service
const paymentBulkhead = new Bulkhead(10);

app.post('/checkout', async (req, res) => {
  try {
    const result = await paymentBulkhead.execute(() =>
      paymentServiceBreaker.fire(req.body)
    );
    res.json(result);
  } catch (err) {
    res.status(503).json({ error: 'Payment service overloaded' });
  }
});

Summary

Pattern Purpose When to Use
Circuit Breaker Stop calling a failing service Downstream service is down/slow
Retry with Backoff Handle transient failures Network blips, brief timeouts
Retry Budget Limit retry traffic High-traffic systems
Timeout Bound wait time Every external call
Bulkhead Limit concurrency Protect shared resources
Idempotency Key Safe retries Non-idempotent operations

Combine these patterns in layers: timeouts on every call, retries for transient errors, circuit breakers to protect against sustained outages, and bulkheads to isolate failures.

Common Problems & Solutions

1 Circuit breaker never closes after the downstream service recovers
Cause: The resetTimeout is too long, or the half-open probe requests are failing because the service returns errors on the first request after recovery (e.g., cold start). Another common cause is that the health check endpoint recovers but the actual operation endpoint has not.
Solution: Reduce the resetTimeout to test recovery sooner (e.g., 10-15 seconds instead of 60). Ensure the half-open probe hits a lightweight operation. Monitor the breaker state with events and metrics. In opossum, verify your volumeThreshold is not too high for low-traffic services.
2 Retry storms overload a recovering service and cause it to fail again
Cause: All clients retry simultaneously after their backoff timers expire, or jitter is not implemented. A service that was just recovering gets hit with a spike of retry traffic and goes down again.
Solution: Always use jitter with exponential backoff. Implement a retry budget to cap total retry traffic at 10-20% of normal request volume. Combine retries with a circuit breaker so that when failures reach a threshold, retries stop entirely. Consider adding server-side rate limiting with Retry-After headers.
3 Non-idempotent operations produce duplicate side effects on retry
Cause: Retrying a POST request that creates a resource (order, payment, email) without an idempotency key causes the operation to execute multiple times.
Solution: Generate a unique idempotency key on the client side and include it in the request header. The server must check this key before processing: if the key was already seen, return the cached response. Store idempotency keys in Redis with a TTL. Only retry operations that are naturally idempotent (GET, PUT, DELETE) without extra safeguards.
