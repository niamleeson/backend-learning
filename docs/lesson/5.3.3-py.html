Middleware and Event Handlers

Intuition

Middleware is like a security checkpoint at a building entrance: every visitor passes through it both on the way in and on the way out. Lifespan events are the building's opening and closing procedures, running once each.

Middleware and Event Handlers

Middleware wraps every request/response cycle, enabling cross-cutting concerns like logging, timing, and header injection. Event handlers (lifespan) run code at application startup and shutdown.

import time
from contextlib import asynccontextmanager
from fastapi import FastAPI, Request

resources = {}

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: initialize resources
    resources["db_pool"] = "connected"  # Simulate DB pool
    print("Application started, DB connected")
    yield
    # Shutdown: clean up resources
    resources.clear()
    print("Application shut down, resources cleaned")

app = FastAPI(lifespan=lifespan)

@app.middleware("http")
async def add_timing_header(request: Request, call_next):
    start = time.perf_counter()
    response = await call_next(request)
    duration = time.perf_counter() - start
    response.headers["X-Process-Time"] = f"{duration:.4f}"
    return response

@app.middleware("http")
async def log_requests(request: Request, call_next):
    print(f"{request.method} {request.url.path}")
    response = await call_next(request)
    print(f"Status: {response.status_code}")
    return response

Key Takeaways: - Middleware runs on every request, in reverse declaration order (last declared runs first) - The lifespan context manager replaces deprecated on_event startup/shutdown handlers - Use middleware for logging, CORS, timing, and request ID injection - Avoid heavy computation in middleware as it affects every request

Exercises

  1. Request ID Middleware

    Create middleware that generates a unique UUID for each request, adds it to the request state, includes it in the response headers as X-Request-ID, and logs it. Access the request ID inside an endpoint from request.state.

    Hint

    Use request.state.request_id = str(uuid.uuid4()) in the middleware before calling call_next. Access it in endpoints via request.state.request_id.

  2. Lifespan Resource Manager

    Create a lifespan manager that initializes a connection pool (simulated dict) and a cache (simulated dict) at startup, makes them available via app.state, and cleans them up at shutdown. Test that endpoints can access these resources.

    Hint

    In the lifespan function, set app.state.db_pool = {...} before yield. In endpoints, access it via request.app.state.db_pool.

Common Problems & Solutions

1 Middleware cannot read the request body
Cause: The request body is a stream that can only be consumed once. Reading it in middleware means the endpoint receives an empty body.
Solution: Read the body with `body = await request.body()`, then create a new `Request` scope with the body set, or use `request.state` to pass the parsed body downstream. Alternatively, use a dependency instead of middleware for body inspection.
2 on_event('startup') handler not running
Cause: The deprecated `on_event` decorator conflicts with the newer `lifespan` parameter. When both are used, `lifespan` takes precedence.
Solution: Migrate to the `lifespan` context manager pattern. Move startup code before `yield` and shutdown code after `yield` inside the async context manager.
