Express App Setup and Project Structure

Intuition

Setting up an Express app is like building a house: you need a solid foundation (project structure), plumbing and electrical run in the right order (middleware registration order), and rooms organized by purpose (feature-based modules) rather than by material type.

Express App Setup and Project Structure

Introduction

Express.js is the most widely used web framework for Node.js. It provides a minimal, flexible foundation for building web applications and APIs. Unlike opinionated frameworks that dictate every architectural decision, Express gives you freedom to structure your application however you see fit — which is both its greatest strength and its biggest challenge for newcomers.

In this lesson, you will learn how to set up an Express application from scratch, understand the anatomy of an Express app, and adopt a project structure that scales from a simple API to a production-grade service.

Installing Express

Start by initializing a new Node.js project and installing Express along with TypeScript tooling:

mkdir my-express-api
cd my-express-api
npm init -y
npm install express
npm install -D typescript @types/express @types/node ts-node nodemon

Initialize TypeScript:

npx tsc --init

Update your tsconfig.json with sensible defaults:

{
  "compilerOptions": {
    "target": "ES2020",
    "module": "commonjs",
    "lib": ["ES2020"],
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "declaration": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}

Your First Express Application

Create src/app.ts:

import express, { Request, Response } from 'express';

const app = express();

// Parse JSON request bodies
app.use(express.json());

// A simple health check route
app.get('/health', (req: Request, res: Response) => {
  res.json({ status: 'ok', timestamp: new Date().toISOString() });
});

export default app;

Create src/server.ts to start the server:

import app from './app';

const PORT = process.env.PORT || 3000;

app.listen(PORT, () => {
  console.log(`Server running on http://localhost:${PORT}`);
});

Add scripts to package.json:

{
  "scripts": {
    "dev": "nodemon --exec ts-node src/server.ts",
    "build": "tsc",
    "start": "node dist/server.js"
  }
}

Why Separate app.ts and server.ts?

This separation is a critical best practice. By exporting the app object separately from the server startup logic, you can:

  1. Import app in tests without starting a real server
  2. Reuse the app in serverless environments (AWS Lambda, Vercel)
  3. Control startup order — initialize databases before listening
src/
  app.ts        <-- Express app configuration (middleware, routes)
  server.ts     <-- Server startup (app.listen)

The Express Application Lifecycle

Understanding how Express processes a request is fundamental:

Client Request
      |
      v
+------------------+
|   Express App    |
+------------------+
      |
      v
+-----------------------+
| Middleware Stack       |
| (executed in order)    |
|   1. express.json()   |
|   2. cors()           |
|   3. morgan()         |
|   4. custom logger    |
+-----------------------+
      |
      v
+-----------------------+
| Route Matching        |
| GET /api/users/:id    |
+-----------------------+
      |
      v
+-----------------------+
| Route Handler         |
| (req, res) => { ... } |
+-----------------------+
      |
      v
+-----------------------+
| Error Handler         |
| (if error thrown)      |
+-----------------------+
      |
      v
   Response

Express processes middleware and route handlers in the order they are registered. This ordering is one of the most important concepts to internalize.

Recommended Project Structure

For small to medium APIs, use a feature-based (modular) structure:

src/
  config/
    index.ts            # Central configuration
    database.ts         # DB connection setup
  middleware/
    errorHandler.ts     # Global error handler
    auth.ts             # Authentication middleware
    validate.ts         # Request validation
  modules/
    users/
      users.controller.ts
      users.service.ts
      users.routes.ts
      users.types.ts
    posts/
      posts.controller.ts
      posts.service.ts
      posts.routes.ts
      posts.types.ts
  utils/
    logger.ts
    ApiError.ts
  app.ts
  server.ts

Why Feature-Based?

Compare with the alternative layer-based structure:

# Layer-based (avoid for larger apps)
src/
  controllers/
    usersController.ts
    postsController.ts
  services/
    usersService.ts
    postsService.ts
  routes/
    usersRoutes.ts
    postsRoutes.ts

The layer-based approach forces you to jump between directories when working on a single feature. The feature-based approach keeps related files together, making it easier to reason about a feature and to extract it into a separate service later.

Setting Up a Modular App

Here is a more complete app.ts showing how modules plug in:

import express from 'express';
import cors from 'cors';
import helmet from 'helmet';
import morgan from 'morgan';
import { userRoutes } from './modules/users/users.routes';
import { postRoutes } from './modules/posts/posts.routes';
import { errorHandler } from './middleware/errorHandler';

const app = express();

// Global middleware
app.use(helmet());
app.use(cors());
app.use(morgan('dev'));
app.use(express.json());
app.use(express.urlencoded({ extended: true }));

// Routes
app.use('/api/users', userRoutes);
app.use('/api/posts', postRoutes);

// Health check
app.get('/health', (req, res) => {
  res.json({ status: 'ok' });
});

// 404 handler
app.use((req, res) => {
  res.status(404).json({ error: 'Route not found' });
});

// Global error handler (must be last)
app.use(errorHandler);

export default app;

  }

Common Pitfalls

1. Registering Error Handlers Before Routes

Express error handlers must have four parameters and be registered after all routes:

// WRONG: error handler before routes
app.use(errorHandler);
app.use('/api/users', userRoutes);

// CORRECT: error handler after routes
app.use('/api/users', userRoutes);
app.use(errorHandler);

2. Forgetting express.json()

Without express.json(), req.body will be undefined for JSON requests:

// WRONG: no body parser
app.post('/api/users', (req, res) => {
  console.log(req.body); // undefined!
});

// CORRECT: register body parser first
app.use(express.json());
app.post('/api/users', (req, res) => {
  console.log(req.body); // { name: "Alice" }
});

3. Starting the Server in app.ts

Keep app.listen() in a separate file. Mixing it into app.ts makes testing difficult:

// WRONG: app.ts
const app = express();
app.listen(3000); // Cannot import app without starting server
export default app;

// CORRECT: server.ts
import app from './app';
app.listen(3000);

Summary

Express setup may look simple, but getting the foundation right pays dividends as your application grows. Separate your app from your server, adopt a feature-based structure, register middleware in the correct order, and set up TypeScript from day one.

Exercises

  1. Bootstrap an Express + TypeScript Project

    Create a new Express project from scratch with TypeScript. Set up tsconfig.json, create separate app.ts and server.ts files, add a health check endpoint that returns JSON with status and uptime, and configure nodemon for development.

    Hint

    Use process.uptime() to get the server uptime in seconds. Remember to install @types/express for TypeScript support.

  2. Feature-Based Module Structure

    Refactor a flat Express app into a feature-based structure. Create modules for 'products' and 'categories', each with their own routes, controller, and service files. Wire them into app.ts using Express Router.

    Hint

    Each module's routes file should export a Router instance. Import and mount them with app.use('/api/products', productRoutes).

  3. Environment-Aware Startup

    Modify your server.ts to read PORT from environment variables with a fallback default. Add a startup log that prints the environment (development/production), port, and Node.js version. Ensure the app gracefully handles SIGTERM signals.

    Hint

    Use process.env.NODE_ENV for environment detection. Listen for process.on('SIGTERM', ...) and call server.close() for graceful shutdown.

Common Problems & Solutions

1 req.body is undefined when sending POST requests
Cause: The express.json() middleware is not registered before the route handler, so Express cannot parse the JSON request body.
Solution: Add app.use(express.json()) before any routes that need to read req.body. Also ensure the client sends Content-Type: application/json header.
2 TypeScript errors: 'Cannot find module express'
Cause: The @types/express package is not installed, so TypeScript does not have type definitions for Express.
Solution: Run npm install -D @types/express @types/node. Also check that your tsconfig.json has 'esModuleInterop': true for proper import syntax.
3 Changes to code are not reflected without restarting the server
Cause: Node.js does not hot-reload by default. Running with ts-node or node directly requires manual restarts.
Solution: Use nodemon with ts-node (nodemon --exec ts-node src/server.ts) or tsx watch (tsx watch src/server.ts) for automatic restarts on file changes.
