PostgreSQL Data Types and Type System

Intuition

Data types are contracts with the database about what shape your data takes. Picking the right type means the database can validate, store, and index your data optimally.

PostgreSQL Data Types

PostgreSQL offers a rich set of data types beyond the standard SQL types. Choosing the right type ensures data integrity and query performance.

-- Numeric types
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    price NUMERIC(10, 2),
    weight DOUBLE PRECISION,
    quantity INTEGER DEFAULT 0
);

-- Text, date, and special types
CREATE TABLE events (
    id UUID DEFAULT gen_random_uuid(),
    name VARCHAR(255) NOT NULL,
    description TEXT,
    tags TEXT[],
    metadata JSONB,
    scheduled TIMESTAMPTZ,
    duration INTERVAL,
    location POINT
);

Key Takeaways: - Use NUMERIC for money/exact math, INTEGER/BIGINT for counters, and DOUBLE PRECISION only when approximation is acceptable. - TIMESTAMPTZ stores timezone-aware timestamps; always prefer it over TIMESTAMP. - Arrays, JSONB, UUID, and range types give PostgreSQL flexibility beyond traditional RDBMS.

Exercises

  1. Design a Users Table

    Create a users table with appropriate types: UUID primary key, email, hashed password, login count, bio, and creation timestamp with timezone.

    Hint

    Use UUID with gen_random_uuid(), VARCHAR for email, TEXT for bio, INTEGER for count, TIMESTAMPTZ DEFAULT now().

  2. Array Column Queries

    Create a table with a TEXT[] tags column. Insert rows and query for rows where tags contain 'python' using the ANY operator.

    Hint

    WHERE 'python' = ANY(tags) or use the @> operator with ARRAY['python'].

Common Problems & Solutions

1 Floating point comparison returns unexpected results.
Cause: DOUBLE PRECISION stores approximate values; 0.1 + 0.2 != 0.3.
Solution: Use NUMERIC for exact decimal arithmetic, especially for financial data.
2 TIMESTAMP shows wrong time in different timezones.
Cause: Using TIMESTAMP WITHOUT TIME ZONE loses timezone context.
Solution: Always use TIMESTAMPTZ. Set the session timezone with SET timezone = 'UTC' for consistency.
