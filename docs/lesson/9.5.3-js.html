Design an Online Code Editor (Replit)

Intuition

Think of an online code editor like a shared art studio where each artist (user) gets their own private booth (sandbox container) with supplies (CPU, memory, packages). The canvas (editor) is visible to collaborators through a magic mirror (CRDT replication) that shows everyone's brushstrokes in real-time without conflicts. The building manager (Kubernetes) assigns booths on demand, cleans up after artists leave, and adds more floors to the building when it gets crowded.

Design an Online Code Editor (Replit)

Introduction

An online code editor allows users to write, run, and share code entirely in the browser — no local setup required. Platforms like Replit, CodeSandbox, and StackBlitz handle millions of users running code in dozens of programming languages. This is a fascinating system design problem because it combines real-time collaboration, remote code execution in sandboxed environments, file system management, and terminal streaming.

In a system design interview, this problem tests your understanding of container orchestration, WebSocket-based real-time communication, sandboxed execution, and resource management.

Requirements

Functional Requirements

  1. Create projects (repls) — Users create projects with a chosen language/template
  2. File editor — Create, edit, rename, delete files with syntax highlighting
  3. Code execution — Run code and see output in a terminal panel
  4. Terminal access — Full interactive terminal (shell) connected to the sandbox
  5. Real-time collaboration — Multiple users edit the same project simultaneously (cursors, selections)
  6. Package management — Install dependencies (npm install, pip install)
  7. Share and embed — Share projects via URL, embed in websites
  8. Version history — View and restore previous versions of files
  9. Persistent storage — Projects persist across sessions

Non-Functional Requirements

  1. Low latency editing — Keystrokes must appear in <50ms (local-first)
  2. Execution isolation — User code cannot affect other users or the host system
  3. Resource limits — CPU, memory, disk, and network limits per sandbox
  4. Scalability — Support 100K concurrent active sandboxes
  5. High availability — 99.9% uptime for the editor; execution environments can be recreated

Capacity Estimation

Registered users:              10 million
Daily active users:            500,000
Concurrent editing sessions:   100,000
Concurrent running sandboxes:  50,000

Sandbox resources (per user):
  - CPU: 0.5 vCPU
  - Memory: 512 MB
  - Disk: 1 GB
  - Network: 10 Mbps

Total compute for 50K sandboxes:
  - CPU: 50K * 0.5 = 25,000 vCPUs
  - Memory: 50K * 512MB = 25 TB
  - Disk: 50K * 1GB = 50 TB (mostly SSD)

File operations:
  - File saves: 100K users * 1 save/10s = 10,000 writes/sec
  - File reads: 100K * 1 read/5s = 20,000 reads/sec

WebSocket messages:
  - Keystrokes: 100K users * 5 keystrokes/sec = 500,000 msg/sec
  - Terminal output: 50K sandboxes * 2 msg/sec = 100,000 msg/sec
  - Total: ~600,000 WebSocket messages/sec

Storage:
  - Avg project size: 10 MB
  - Total projects: 50M projects * 10MB = 500 TB
  - Use object storage (S3) for cold projects, SSD for active

API Design

import express from 'express';
const router = express.Router();

// Create a new project (repl)
router.post('/projects', async (req, res) => {
  // POST /api/v1/projects
  // Body: { name, language, template?, description? }
  // Response: { project_id, name, language, url, sandbox_url }
  const project = await projectService.create({
    owner_id: req.user.id,
    name: req.body.name,
    language: req.body.language,
    template: req.body.template || 'blank',
  });
  res.status(201).json(project);
});

// Get project metadata and file tree
router.get('/projects/:projectId', async (req, res) => {
  // Response: { id, name, language, files: [...], collaborators, created_at }
  const project = await projectService.getWithFileTree(req.params.projectId);
  res.json(project);
});

// File operations
router.get('/projects/:projectId/files/*', async (req, res) => {
  // GET /api/v1/projects/:id/files/src/index.js
  const filePath = req.params[0];
  const content = await fileService.readFile(req.params.projectId, filePath);
  res.json({ path: filePath, content });
});

router.put('/projects/:projectId/files/*', async (req, res) => {
  // PUT /api/v1/projects/:id/files/src/index.js
  // Body: { content: string }
  const filePath = req.params[0];
  await fileService.writeFile(req.params.projectId, filePath, req.body.content);
  res.status(204).send();
});

router.post('/projects/:projectId/files', async (req, res) => {
  // Body: { path, content?, type: 'file' | 'directory' }
  await fileService.createFile(req.params.projectId, req.body);
  res.status(201).send();
});

router.delete('/projects/:projectId/files/*', async (req, res) => {
  const filePath = req.params[0];
  await fileService.deleteFile(req.params.projectId, filePath);
  res.status(204).send();
});

// Execute code / start sandbox
router.post('/projects/:projectId/run', async (req, res) => {
  // POST /api/v1/projects/:id/run
  // Body: { command?: string } (optional, defaults to language-specific run command)
  // Response: { sandbox_id, websocket_url }
  const sandbox = await executionService.startSandbox(
    req.params.projectId,
    req.body.command
  );
  res.json({
    sandbox_id: sandbox.id,
    websocket_url: `wss://exec.example.com/ws/${sandbox.id}`,
  });
});

// Stop sandbox
router.post('/projects/:projectId/stop', async (req, res) => {
  await executionService.stopSandbox(req.params.projectId);
  res.status(204).send();
});

// Version history
router.get('/projects/:projectId/versions', async (req, res) => {
  // Response: { versions: [{ id, timestamp, message, author }] }
  const versions = await versionService.listVersions(
    req.params.projectId,
    { cursor: req.query.cursor, limit: parseInt(req.query.limit) || 20 }
  );
  res.json(versions);
});

// Collaboration: add collaborator
router.post('/projects/:projectId/collaborators', async (req, res) => {
  // Body: { user_id, permission: 'read' | 'write' | 'admin' }
  await projectService.addCollaborator(
    req.params.projectId,
    req.body.user_id,
    req.body.permission
  );
  res.status(201).send();
});

// Share / fork
router.post('/projects/:projectId/fork', async (req, res) => {
  const forked = await projectService.fork(req.params.projectId, req.user.id);
  res.status(201).json(forked);
});

export default router;

-- Projects (Repls)
CREATE TABLE projects (
  id              BIGSERIAL PRIMARY KEY,
  owner_id        BIGINT NOT NULL REFERENCES users(id),
  name            VARCHAR(128) NOT NULL,
  slug            VARCHAR(128) NOT NULL,
  language        VARCHAR(32) NOT NULL,
  description     TEXT,
  is_public       BOOLEAN DEFAULT true,
  forked_from     BIGINT REFERENCES projects(id),
  storage_bucket  VARCHAR(255) NOT NULL,   -- S3 path for project files
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (owner_id, slug)
);
CREATE INDEX idx_project_owner ON projects(owner_id, updated_at DESC);
CREATE INDEX idx_project_language ON projects(language, is_public);

-- Collaborators
CREATE TABLE collaborators (
  project_id      BIGINT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  user_id         BIGINT NOT NULL REFERENCES users(id),
  permission      VARCHAR(16) NOT NULL DEFAULT 'write',
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (project_id, user_id)
);

-- File versions (snapshots)
CREATE TABLE file_versions (
  id              BIGSERIAL PRIMARY KEY,
  project_id      BIGINT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  file_path       TEXT NOT NULL,
  content_hash    VARCHAR(64) NOT NULL,    -- SHA-256 of content
  size_bytes      INTEGER NOT NULL,
  author_id       BIGINT REFERENCES users(id),
  created_at      TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_file_version ON file_versions(project_id, file_path, created_at DESC);

-- Project snapshots (point-in-time versions)
CREATE TABLE project_snapshots (
  id              BIGSERIAL PRIMARY KEY,
  project_id      BIGINT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  author_id       BIGINT REFERENCES users(id),
  message         TEXT,
  file_manifest   JSONB NOT NULL,   -- { "src/index.js": "hash1", "package.json": "hash2" }
  created_at      TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_snapshot_project ON project_snapshots(project_id, created_at DESC);

-- Sandbox sessions (active execution environments)
CREATE TABLE sandbox_sessions (
  id              BIGSERIAL PRIMARY KEY,
  project_id      BIGINT NOT NULL REFERENCES projects(id),
  user_id         BIGINT NOT NULL REFERENCES users(id),
  container_id    VARCHAR(128),
  node_hostname   VARCHAR(128),
  status          VARCHAR(16) DEFAULT 'starting', -- starting, running, stopped, crashed
  cpu_limit       DECIMAL(4,2) DEFAULT 0.50,
  memory_limit_mb INTEGER DEFAULT 512,
  started_at      TIMESTAMPTZ DEFAULT NOW(),
  stopped_at      TIMESTAMPTZ
);
CREATE INDEX idx_sandbox_project ON sandbox_sessions(project_id, status);

Core Implementation: Sandbox Execution

import { KubeConfig, CoreV1Api, AppsV1Api } from '@kubernetes/client-node';
import Redis from 'ioredis';

const redis = new Redis();

class ExecutionService {
  constructor() {
    const kc = new KubeConfig();
    kc.loadFromDefault();
    this.k8sCore = kc.makeApiClient(CoreV1Api);
  }

  async startSandbox(projectId, command) {
    // Check if sandbox already running
    const existing = await this.getActiveSandbox(projectId);
    if (existing) return existing;

    // Create a sandboxed container (pod) for this project
    const sandboxId = `sandbox-${projectId}-${Date.now()}`;
    const project = await projectService.get(projectId);

    const podSpec = {
      apiVersion: 'v1',
      kind: 'Pod',
      metadata: {
        name: sandboxId,
        labels: {
          app: 'sandbox',
          project: projectId.toString(),
          language: project.language,
        },
      },
      spec: {
        containers: [{
          name: 'sandbox',
          image: this.getImage(project.language),
          resources: {
            limits: { cpu: '500m', memory: '512Mi' },
            requests: { cpu: '100m', memory: '128Mi' },
          },
          // Mount project files from S3/PVC
          volumeMounts: [{
            name: 'project-files',
            mountPath: '/workspace',
          }],
          // Security: no root, no network escalation
          securityContext: {
            runAsNonRoot: true,
            runAsUser: 1000,
            readOnlyRootFilesystem: false,
            allowPrivilegeEscalation: false,
          },
        }],
        volumes: [{
          name: 'project-files',
          emptyDir: { sizeLimit: '1Gi' },
        }],
        // Auto-terminate after 1 hour of inactivity
        activeDeadlineSeconds: 3600,
        // Restrict network access
        dnsPolicy: 'None',
        dnsConfig: {
          nameservers: ['8.8.8.8'],
        },
      },
    };

    await this.k8sCore.createNamespacedPod('sandboxes', podSpec);

    // Sync project files to the sandbox
    await this.syncProjectFiles(sandboxId, projectId);

    // Record session
    const session = await prisma.sandbox_sessions.create({
      data: {
        project_id: BigInt(projectId),
        user_id: BigInt(project.owner_id),
        container_id: sandboxId,
        status: 'running',
      },
    });

    return { id: session.id, container_id: sandboxId };
  }

  getImage(language) {
    const images = {
      javascript: 'sandbox-node:20-alpine',
      typescript: 'sandbox-node:20-alpine',
      python: 'sandbox-python:3.12-alpine',
      go: 'sandbox-go:1.22-alpine',
      rust: 'sandbox-rust:1.76-alpine',
      java: 'sandbox-java:21-alpine',
    };
    return `registry.example.com/${images[language] || images.javascript}`;
  }

  async stopSandbox(projectId) {
    const session = await this.getActiveSandbox(projectId);
    if (!session) return;

    // Save any changed files back to S3
    await this.persistFiles(session.container_id, projectId);

    // Delete the pod
    await this.k8sCore.deleteNamespacedPod(session.container_id, 'sandboxes');

    await prisma.sandbox_sessions.update({
      where: { id: session.id },
      data: { status: 'stopped', stopped_at: new Date() },
    });
  }
}

Real-Time Collaboration with CRDT

import * as Y from 'yjs';
import { WebSocketServer } from 'ws';
import Redis from 'ioredis';

const pub = new Redis();
const sub = new Redis();

// Each project has a Yjs document
const docs = new Map();

function getOrCreateDoc(projectId) {
  if (!docs.has(projectId)) {
    const ydoc = new Y.Doc();
    docs.set(projectId, ydoc);
  }
  return docs.get(projectId);
}

const wss = new WebSocketServer({ port: 8081 });

wss.on('connection', (ws, req) => {
  const projectId = extractProjectId(req.url);
  const userId = authenticateWS(req);
  const ydoc = getOrCreateDoc(projectId);

  // Send current document state to new client
  const stateVector = Y.encodeStateAsUpdate(ydoc);
  ws.send(JSON.stringify({ type: 'sync', data: Array.from(stateVector) }));

  // Broadcast presence (cursor position, user info)
  pub.publish(`collab:${projectId}`, JSON.stringify({
    type: 'presence',
    user_id: userId,
    action: 'joined',
  }));

  ws.on('message', (message) => {
    const data = JSON.parse(message);

    if (data.type === 'update') {
      // Apply CRDT update from client
      const update = new Uint8Array(data.data);
      Y.applyUpdate(ydoc, update);

      // Broadcast to other clients in this project
      pub.publish(`collab:${projectId}`, JSON.stringify({
        type: 'update',
        data: Array.from(update),
        sender: userId,
      }));
    }

    if (data.type === 'cursor') {
      // Broadcast cursor position
      pub.publish(`collab:${projectId}`, JSON.stringify({
        type: 'cursor',
        user_id: userId,
        file: data.file,
        position: data.position,
        selection: data.selection,
      }));
    }
  });

  ws.on('close', () => {
    pub.publish(`collab:${projectId}`, JSON.stringify({
      type: 'presence',
      user_id: userId,
      action: 'left',
    }));
  });
});

// Subscribe to updates from other collaboration servers
sub.psubscribe('collab:*');
sub.on('pmessage', (pattern, channel, message) => {
  const projectId = channel.split(':')[1];
  const data = JSON.parse(message);

  // Forward to all WebSocket clients for this project (except sender)
  wss.clients.forEach((client) => {
    if (client.projectId === projectId && client.userId !== data.sender) {
      client.send(message);
    }
  });
});

Summary

An online code editor requires three core subsystems: a real-time collaborative editor (using CRDTs like Yjs for conflict-free merging), a sandboxed execution environment (Kubernetes pods or Firecracker microVMs with strict resource limits and network isolation), and a file management layer (S3 for durable storage with in-memory caching for active sessions). WebSocket connections handle bidirectional communication for keystrokes, terminal output, and cursor presence. The biggest scaling challenge is managing hundreds of thousands of concurrent sandboxes — Kubernetes cluster autoscaling with pod preemption for inactive users is the standard approach. Security is paramount: user code must be fully isolated from the host and other users via container security contexts, network policies, and optional kernel-level isolation with gVisor.

Exercises

  1. Build a Code Execution Sandbox

    Create a Node.js service that receives code (JavaScript), spawns it in a child_process with resource limits (timeout, memory), captures stdout/stderr, and returns the result. Prevent dangerous operations like file system access and network calls.

    Hint

    Use child_process.fork() with a timeout. Use Node.js --max-old-space-size flag for memory limits. Create a restricted execution wrapper using vm2 or isolated-vm npm packages.

  2. Real-Time Collaboration with Yjs

    Build a minimal collaborative text editor using Yjs and WebSocket. Two browser tabs should be able to edit the same document simultaneously with changes appearing in real-time. Display remote cursors with different colors.

    Hint

    Use y-websocket for the server transport layer and y-monaco or y-codemirror for editor binding. Use awareness protocol for cursor positions.

  3. File Versioning System

    Implement a file versioning system that stores content-addressable file snapshots (SHA-256 hash as key, content as value in S3). Create project snapshots as a manifest of file hashes. Support listing versions and restoring to any snapshot.

    Hint

    Hash file content with crypto.createHash('sha256'). Store content in S3 with the hash as key (automatic deduplication). Store manifests in PostgreSQL with JSONB.

  4. Terminal Streaming over WebSocket

    Build a WebSocket server that spawns a shell process (bash) in a container and streams the PTY (pseudo-terminal) output to a connected browser client using xterm.js. Handle bidirectional I/O: user input from browser to shell, shell output to browser.

    Hint

    Use the node-pty npm package to create a pseudo-terminal. Pipe pty.onData to WebSocket send, and WebSocket messages to pty.write.

Common Problems & Solutions

1 Sandbox cold start takes 5+ seconds, making the run button feel slow
Cause: Pulling container images, initializing the filesystem, and installing dependencies on every sandbox creation adds significant latency.
Solution: Pre-warm a pool of generic sandbox containers (one per language) that are already running and waiting for assignment. When a user clicks 'Run', assign an idle container from the pool and sync project files into it. Replenish the pool asynchronously. For frequently used templates (Node.js, Python), keep 100+ pre-warmed containers. This reduces cold start from 5s to under 500ms.
2 CRDT document grows unboundedly in memory for long-lived editing sessions
Cause: Yjs stores the full history of operations for conflict resolution. Long editing sessions with many edits create large CRDT documents (100MB+) that consume excessive server memory.
Solution: Periodically compact the CRDT document by creating a fresh snapshot. Every N operations or every M minutes, encode the current state, create a new Y.Doc from that state, and replace the old document. Persist the snapshot to S3. On reconnection, clients load the latest snapshot rather than replaying the full history.
3 Users can escape the sandbox and access host resources
Cause: Standard Docker containers share the host kernel, and privilege escalation vulnerabilities can allow container escape.
Solution: Layer multiple isolation mechanisms: (1) Run containers with non-root user and drop all capabilities, (2) Use seccomp profiles to restrict system calls, (3) Apply AppArmor/SELinux profiles, (4) Use gVisor (user-space kernel) or Firecracker microVMs for kernel-level isolation, (5) Network policies to prevent sandbox-to-sandbox communication. Defense in depth is essential.
