RabbitMQ: Exchanges, Queues, and Routing

Intuition

RabbitMQ is like a postal sorting office. Letters (messages) arrive and go to a sorting desk (exchange). Direct exchange is sorting by exact zip code. Fanout is making photocopies and sending to every branch. Topic exchange is sorting by partial address patterns — all mail for '1xx' streets goes to the north office.

RabbitMQ: Exchanges, Queues, and Routing

Introduction

RabbitMQ is a message broker that implements the Advanced Message Queuing Protocol (AMQP). Unlike BullMQ which is Redis-based and focused on job queues, RabbitMQ is a full-featured message broker with sophisticated routing, multiple exchange types, and guaranteed delivery. It excels in scenarios where messages need to be routed to different consumers based on content or patterns.

RabbitMQ vs BullMQ

┌───────────────────┬───────────────────┬───────────────────┐
│                   │ BullMQ            │ RabbitMQ          │
├───────────────────┼───────────────────┼───────────────────┤
│ Backing store     │ Redis             │ Erlang/OTP        │
│ Protocol          │ Custom (Redis)    │ AMQP 0-9-1        │
│ Use case          │ Job queues        │ Message routing   │
│ Routing           │ Queue name only   │ Exchanges+routing │
│ Priority queues   │ Yes               │ Yes               │
│ Delayed messages  │ Built-in          │ Plugin required   │
│ Clustering        │ Via Redis Cluster │ Built-in          │
│ Message ordering  │ FIFO per queue    │ FIFO per queue    │
│ Persistence       │ Via Redis         │ Built-in          │
└───────────────────┴───────────────────┴───────────────────┘

Setup with amqplib

import amqp from 'amqplib';

async function connect() {
  const connection = await amqp.connect(process.env.RABBITMQ_URL || 'amqp://localhost');
  const channel = await connection.createChannel();

  // Handle connection errors
  connection.on('error', (err) => {
    console.error('RabbitMQ connection error:', err);
  });
  connection.on('close', () => {
    console.log('RabbitMQ connection closed. Reconnecting...');
    setTimeout(connect, 5000);
  });

  return { connection, channel };
}

Exchange Types

1. Direct Exchange

Routes messages to queues with an exact matching routing key.

Producer                  Direct Exchange              Queues
  │                            │
  │ routingKey="error"   ──────┼──> [error queue]    ──> Error handler
  │ routingKey="info"    ──────┼──> [info queue]     ──> Log aggregator
  │ routingKey="warning" ──────┼──> [warning queue]  ──> Alert service

async function setupDirectExchange(channel) {
  const exchange = 'logs.direct';

  await channel.assertExchange(exchange, 'direct', { durable: true });

  // Create queues and bind with routing keys
  const errorQueue = await channel.assertQueue('logs.error', { durable: true });
  const infoQueue = await channel.assertQueue('logs.info', { durable: true });

  await channel.bindQueue(errorQueue.queue, exchange, 'error');
  await channel.bindQueue(infoQueue.queue, exchange, 'info');

  // Publish
  channel.publish(exchange, 'error', Buffer.from(JSON.stringify({
    message: 'Database connection failed',
    timestamp: new Date().toISOString(),
  })), { persistent: true });

  // Consume
  channel.consume(errorQueue.queue, (msg) => {
    const log = JSON.parse(msg.content.toString());
    console.error('ERROR:', log.message);
    channel.ack(msg); // Acknowledge processing
  });
}

2. Fanout Exchange

Broadcasts messages to ALL bound queues (like pub/sub).

Producer ──> Fanout Exchange ──┬──> Queue A (Email service)
                               ├──> Queue B (SMS service)
                               └──> Queue C (Push notification)

async function setupFanoutExchange(channel) {
  const exchange = 'notifications.fanout';
  await channel.assertExchange(exchange, 'fanout', { durable: true });

  // Each service gets its own queue
  const emailQ = await channel.assertQueue('notifications.email', { durable: true });
  const smsQ = await channel.assertQueue('notifications.sms', { durable: true });
  const pushQ = await channel.assertQueue('notifications.push', { durable: true });

  // Bind all queues (routing key ignored for fanout)
  await channel.bindQueue(emailQ.queue, exchange, '');
  await channel.bindQueue(smsQ.queue, exchange, '');
  await channel.bindQueue(pushQ.queue, exchange, '');

  // One publish reaches all three queues
  channel.publish(exchange, '', Buffer.from(JSON.stringify({
    userId: 'user-123',
    event: 'order_completed',
    orderId: 'ORD-456',
  })), { persistent: true });
}

3. Topic Exchange

Routes based on pattern matching with wildcards.

Routing key format: <facility>.<severity>.<region>

  order.created.us    ──> Matches: order.* , order.created.* , *.*.us
  order.cancelled.eu  ──> Matches: order.* , *.cancelled.*
  payment.failed.us   ──> Matches: payment.* , *.failed.* , #.us

  * = exactly one word
  # = zero or more words

async function setupTopicExchange(channel) {
  const exchange = 'events.topic';
  await channel.assertExchange(exchange, 'topic', { durable: true });

  // Order processing service - all order events
  const orderQ = await channel.assertQueue('events.orders', { durable: true });
  await channel.bindQueue(orderQ.queue, exchange, 'order.*');

  // Payment alerts - only failures
  const paymentAlertQ = await channel.assertQueue('events.payment-alerts', { durable: true });
  await channel.bindQueue(paymentAlertQ.queue, exchange, 'payment.failed');

  // US region monitor - everything from US
  const usQ = await channel.assertQueue('events.us-region', { durable: true });
  await channel.bindQueue(usQ.queue, exchange, '*.*.us');

  // Audit log - capture everything
  const auditQ = await channel.assertQueue('events.audit', { durable: true });
  await channel.bindQueue(auditQ.queue, exchange, '#');

  // Publish events
  channel.publish(exchange, 'order.created.us', Buffer.from(JSON.stringify({
    orderId: 'ORD-123',
    region: 'us',
  })));
  // This goes to: orderQ, usQ, auditQ
}

Message Acknowledgment

// Manual acknowledgment (recommended for reliability)
channel.consume('tasks', async (msg) => {
  try {
    const task = JSON.parse(msg.content.toString());
    await processTask(task);
    channel.ack(msg);          // Success - remove from queue
  } catch (err) {
    if (err.retryable) {
      channel.nack(msg, false, true);  // Requeue for retry
    } else {
      channel.nack(msg, false, false); // Don't requeue (goes to DLQ if configured)
    }
  }
}, { noAck: false }); // Manual ack mode

// Prefetch: limit unacknowledged messages per consumer
channel.prefetch(10); // Only deliver 10 unacked messages at a time

Dead Letter Queues

async function setupWithDLQ(channel) {
  // Dead letter exchange and queue
  await channel.assertExchange('dlx', 'direct', { durable: true });
  await channel.assertQueue('dead-letters', { durable: true });
  await channel.bindQueue('dead-letters', 'dlx', 'failed');

  // Main queue with DLQ configuration
  await channel.assertQueue('tasks', {
    durable: true,
    arguments: {
      'x-dead-letter-exchange': 'dlx',
      'x-dead-letter-routing-key': 'failed',
      'x-message-ttl': 300000,    // Messages expire after 5 min
      'x-max-length': 10000,       // Max queue size
    },
  });

  // Failed messages automatically go to dead-letters queue
  // Process dead letters for investigation
  channel.consume('dead-letters', (msg) => {
    const original = JSON.parse(msg.content.toString());
    const reason = msg.properties.headers['x-death']?.[0]?.reason;
    console.error(`Dead letter (reason: ${reason}):`, original);
    // Log to monitoring system, alert ops
    channel.ack(msg);
  });
}

RabbitMQ with NestJS Microservices

// main.ts - Microservice setup
import { NestFactory } from '@nestjs/core';
import { MicroserviceOptions, Transport } from '@nestjs/microservices';

const app = await NestFactory.createMicroservice(AppModule, {
  transport: Transport.RMQ,
  options: {
    urls: ['amqp://localhost:5672'],
    queue: 'orders_queue',
    queueOptions: { durable: true },
    prefetchCount: 10,
  },
});

await app.listen();

// order.controller.ts
import { Controller } from '@nestjs/common';
import { MessagePattern, Payload, Ctx, RmqContext } from '@nestjs/microservices';

@Controller()
export class OrderController {
  @MessagePattern('order_created')
  async handleOrderCreated(@Payload() data: any, @Ctx() context: RmqContext) {
    const channel = context.getChannelRef();
    const originalMsg = context.getMessage();

    try {
      await this.orderService.processOrder(data);
      channel.ack(originalMsg);
    } catch (err) {
      channel.nack(originalMsg, false, true); // Requeue
    }
  }
}

Connection Management

class RabbitMQConnection {
  constructor(url) {
    this.url = url;
    this.connection = null;
    this.channel = null;
  }

  async connect() {
    this.connection = await amqp.connect(this.url);
    this.channel = await this.connection.createConfirmChannel();

    // Publish confirmations
    this.channel.on('return', (msg) => {
      console.error('Message returned (unroutable):', msg);
    });

    this.connection.on('close', () => {
      console.log('Connection closed. Reconnecting...');
      setTimeout(() => this.connect(), 5000);
    });

    return this;
  }

  async publishWithConfirm(exchange, routingKey, data) {
    return new Promise((resolve, reject) => {
      this.channel.publish(
        exchange,
        routingKey,
        Buffer.from(JSON.stringify(data)),
        { persistent: true, mandatory: true },
        (err) => {
          if (err) reject(err);
          else resolve();
        }
      );
    });
  }

  async close() {
    await this.channel?.close();
    await this.connection?.close();
  }
}

Common Pitfalls

  1. Not acknowledging messages: Unacknowledged messages stay in the queue and are redelivered when the consumer disconnects, causing duplicates.
  2. Using one channel for everything: Channels are not thread-safe. Use one channel per consumer/producer thread.
  3. Not setting prefetch: Without prefetch, RabbitMQ dumps all messages to one consumer, leaving others idle.
  4. Losing messages without persistent flag: Messages are stored in memory by default. Use { persistent: true } for durability.

Summary

RabbitMQ provides sophisticated message routing through exchanges (direct, fanout, topic) and queues. It excels at routing messages to different services based on content, unlike BullMQ which is focused on job processing. Use direct exchanges for targeted routing, fanout for broadcasting, and topic for pattern-based routing. Always use manual acknowledgment, configure dead letter queues, and set prefetch counts for reliable message processing.

  3. Dead Letter Queue Handler

    Set up a queue with a dead letter exchange. Implement a consumer that rejects messages after 3 processing attempts. Build a dead letter processor that logs failed messages and alerts operators.

    Hint

    Track attempt count in message headers. Use channel.nack(msg, false, false) to send to DLQ after max attempts.

Common Problems & Solutions

1 Messages being processed multiple times
Cause: Consumer processes the message but crashes before calling `channel.ack()`. RabbitMQ redelivers the unacknowledged message.
Solution: Make message processing idempotent (safe to run multiple times). Use unique message IDs and track processed IDs in the database. Check `msg.fields.redelivered` flag.
2 One consumer gets all messages while others sit idle
Cause: Prefetch count is not set, so RabbitMQ dispatches all messages to the first consumer in round-robin.
Solution: Set `channel.prefetch(1)` or an appropriate number. This ensures RabbitMQ only sends N unacknowledged messages to each consumer.

channel.prefetch(10); // Each consumer gets max 10 unacked messages

3 Messages lost after RabbitMQ restart
Cause: Queue and/or messages were not configured as durable/persistent.
Solution: Declare queues with `{ durable: true }` and publish messages with `{ persistent: true }`. Also enable publisher confirms for guaranteed delivery.
