GitHub Actions: Workflows, Jobs, and Steps

Intuition

GitHub Actions is like setting up dominoes in your repository. When someone pushes code (tips the first domino), a chain reaction begins: checkout the code, install dependencies, run tests, build, and deploy. Each domino (step) must fall successfully for the next one to trigger. If any domino fails, the chain stops and you are notified.

GitHub Actions: Workflows, Jobs, and Steps

What is GitHub Actions?

GitHub Actions is a CI/CD platform built into GitHub. It lets you automate your build, test, and deployment pipeline directly from your repository. Every push, pull request, or scheduled event can trigger automated workflows.

Your First Workflow

# .github/workflows/ci.yml
name: CI Pipeline

# When to run
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

# Environment variables for all jobs
env:
  NODE_VERSION: '20'

jobs:
  # ================================
  # Build and Test
  # ================================
  build-and-test:
    name: Build & Test
    runs-on: ubuntu-latest

    # Service containers (databases for testing)
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_DB: test_db
          POSTGRES_USER: test_user
          POSTGRES_PASSWORD: test_password
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

      redis:
        image: redis:7-alpine
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      # Checkout code
      - name: Checkout repository
        uses: actions/checkout@v4

      # Setup Node.js with caching
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'

      # Install dependencies
      - name: Install dependencies
        run: npm ci

      # Lint
      - name: Lint code
        run: npm run lint

      # Type check
      - name: Type check
        run: npx tsc --noEmit

      # Run tests
      - name: Run tests
        run: npm test -- --coverage
        env:
          DATABASE_URL: postgresql://test_user:test_password@localhost:5432/test_db
          REDIS_URL: redis://localhost:6379
          NODE_ENV: test

      # Upload coverage
      - name: Upload coverage
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage/
          retention-days: 7

      # Build
      - name: Build
        run: npm run build

      # Upload build artifact
      - name: Upload build
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/

Workflow Triggers (on:)

on:
  # Push to specific branches
  push:
    branches: [main, develop]
    paths:
      - 'src/**'               # Only when source changes
      - 'package.json'
    paths-ignore:
      - '**.md'                # Ignore docs changes
      - '.github/**'

  # Pull requests
  pull_request:
    branches: [main]
    types: [opened, synchronize, reopened]

  # Scheduled (cron)
  schedule:
    - cron: '0 2 * * *'       # Daily at 2 AM UTC

  # Manual trigger
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production

  # After another workflow
  workflow_run:
    workflows: ["CI Pipeline"]
    types: [completed]

  # Release published
  release:
    types: [published]

Job Dependencies and Matrix

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint

  test:
    needs: lint              # Runs after lint succeeds
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20, 22]    # Test on multiple versions
        os: [ubuntu-latest]
      fail-fast: false        # Don't cancel other matrix jobs on failure
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      - run: npm ci
      - run: npm test

  deploy:
    needs: [lint, test]      # Runs after BOTH succeed
    if: github.ref == 'refs/heads/main'  # Only on main branch
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        run: echo "Deploying..."

Secrets and Environment Variables

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production    # Links to GitHub environment
    steps:
      - name: Deploy to server
        env:
          SSH_KEY: ${{ secrets.SSH_PRIVATE_KEY }}
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
        run: |
          echo "$SSH_KEY" > key.pem
          chmod 600 key.pem
          ssh -i key.pem deploy@${{ secrets.SERVER_IP }} "cd /var/www/app && ./deploy.sh"

Setting Up Secrets

  1. Go to Repository Settings > Secrets and variables > Actions
  2. Click "New repository secret"
  3. Add name (e.g., DATABASE_URL) and value

Environments with Protection Rules

  1. Go to Repository Settings > Environments
  2. Create environments: staging, production
  3. Add protection rules: required reviewers, wait timer, branch restrictions

Caching

# Automatic caching with setup-node
- uses: actions/setup-node@v4
  with:
    node-version: '20'
    cache: 'npm'           # Caches ~/.npm

# Manual caching (for other things)
- name: Cache Prisma engines
  uses: actions/cache@v4
  with:
    path: node_modules/.prisma
    key: prisma-${{ hashFiles('prisma/schema.prisma') }}

# Cache Docker layers
- name: Cache Docker layers
  uses: actions/cache@v4
  with:
    path: /tmp/.buildx-cache
    key: docker-${{ hashFiles('Dockerfile') }}
    restore-keys: docker-

Reusable Workflows

# .github/workflows/reusable-test.yml
name: Reusable Test Workflow

on:
  workflow_call:
    inputs:
      node-version:
        required: false
        type: string
        default: '20'
    secrets:
      DATABASE_URL:
        required: true

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
          cache: 'npm'
      - run: npm ci
      - run: npm test
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}

# .github/workflows/ci.yml - Calling the reusable workflow
name: CI
on: [push]

jobs:
  test:
    uses: ./.github/workflows/reusable-test.yml
    with:
      node-version: '20'
    secrets:
      DATABASE_URL: ${{ secrets.DATABASE_URL }}

Composite Actions

# .github/actions/setup-project/action.yml
name: Setup Project
description: Install Node.js and dependencies

inputs:
  node-version:
    description: 'Node.js version'
    required: false
    default: '20'

runs:
  using: 'composite'
  steps:
    - uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}
        cache: 'npm'
    - run: npm ci
      shell: bash
    - run: npx prisma generate
      shell: bash

# Usage in a workflow
steps:
  - uses: actions/checkout@v4
  - uses: ./.github/actions/setup-project
    with:
      node-version: '20'
  - run: npm test

Common Pitfalls

  1. Not caching dependencies - Without caching, npm ci runs on every build (~30-60 seconds). Always use setup-node's cache feature.
  2. Running everything in one job - Split lint, test, and build into separate jobs so failures are clear and parallel execution is possible.
  3. Hardcoding secrets in workflows - Never put secrets in YAML. Always use ${{ secrets.NAME }}.
  4. Not using fail-fast: false in matrix - By default, if one matrix job fails, all are cancelled. Set false to see all results.
  5. Ignoring workflow run time - Free GitHub Actions minutes are limited. Optimize with caching, conditional jobs, and path filters.

Exercises

  1. Complete CI Pipeline

    Create a GitHub Actions workflow that: (1) triggers on push to main and PRs, (2) sets up Node.js with npm caching, (3) runs lint, type-check, and tests in parallel jobs, (4) uses PostgreSQL and Redis service containers for integration tests, (5) uploads test coverage as an artifact.

    Hint

    Use the 'services' key for database containers. Set health checks so the job waits for services to be ready. Use 'needs' for job dependencies.

  2. Matrix Testing

    Create a workflow that tests your application across Node.js versions 18, 20, and 22 using a matrix strategy. Each should run independently (fail-fast: false). Add a summary step that reports which versions passed.

    Hint

    Use strategy.matrix.node-version. Reference the version with ${{ matrix.node-version }}.

  3. Reusable Workflow

    Extract the test job into a reusable workflow that accepts node-version as input and DATABASE_URL as a secret. Call it from both a CI workflow and a nightly scheduled workflow.

    Hint

    Use workflow_call trigger for the reusable workflow. Use 'uses: ./.github/workflows/reusable.yml' to call it.

Common Problems & Solutions

1 Tests pass locally but fail in GitHub Actions
Cause: Environment differences: different Node.js version, missing environment variables, different OS (Ubuntu vs macOS), or timing-dependent tests.
Solution: Pin the Node.js version to match local. Set all required environment variables in the workflow. For timing issues, increase timeouts or use retries. Add `--verbose` flags to test commands for better debugging output.
2 Workflow runs are slow (10+ minutes)
Cause: No dependency caching, unnecessary steps, sequential jobs that could be parallel, or large Docker image builds.
Solution: Enable npm caching with setup-node. Run independent jobs in parallel. Use path filters to skip CI on doc-only changes. Cache Docker layers with actions/cache. Use the 'continue-on-error' strategy with matrix builds.
3 Service container (PostgreSQL) not ready when tests start
Cause: The service container started but PostgreSQL inside it is still initializing.
Solution: Add health check options to the service definition: `options: --health-cmd pg_isready --health-interval 10s --health-timeout 5s --health-retries 5`. GitHub Actions will wait for the health check to pass before starting job steps.
