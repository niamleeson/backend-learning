HATEOAS and Hypermedia APIs

Intuition

HATEOAS is like a choose-your-own-adventure book. Instead of memorizing which page to flip to, the book tells you at the bottom of each page: 'To fight the dragon, turn to page 47. To flee, turn to page 23.' The available choices change based on where you are in the story.

HATEOAS and Hypermedia APIs

HATEOAS (Hypermedia As The Engine Of Application State) is the highest maturity level of REST APIs. It means that API responses include links that tell the client what actions are available next, rather than requiring the client to hardcode URLs.

The Richardson Maturity Model

Leonard Richardson described four levels of REST maturity:

+--------------------------------------------------+
| Level 3: Hypermedia Controls (HATEOAS)           |
|   Responses include links to related actions     |
+--------------------------------------------------+
| Level 2: HTTP Verbs                              |
|   Proper use of GET, POST, PUT, DELETE           |
+--------------------------------------------------+
| Level 1: Resources                               |
|   Individual URIs for each resource              |
+--------------------------------------------------+
| Level 0: The Swamp of POX                        |
|   One endpoint, everything is POST               |
+--------------------------------------------------+

Most APIs stop at Level 2. HATEOAS is Level 3.

What Does HATEOAS Look Like?

Without HATEOAS

{
  "id": 42,
  "title": "API Design Patterns",
  "status": "draft"
}

The client must know ahead of time that to publish this post, it should POST /api/posts/42/publish. That knowledge is hardcoded.

With HATEOAS

{
  "id": 42,
  "title": "API Design Patterns",
  "status": "draft",
  "_links": {
    "self": { "href": "/api/posts/42" },
    "edit": { "href": "/api/posts/42", "method": "PUT" },
    "publish": { "href": "/api/posts/42/publish", "method": "POST" },
    "delete": { "href": "/api/posts/42", "method": "DELETE" },
    "author": { "href": "/api/users/7" }
  }
}

Now the client discovers available actions from the response. If the post is already published, the publish link would not appear.

Hypermedia Formats

HAL (Hypertext Application Language)

The most popular format. Uses _links and _embedded.

// Express HAL response builder
interface HalLink {
  href: string;
  method?: string;
  title?: string;
  templated?: boolean;
}

interface HalResource {
  _links: Record<string, HalLink>;
  _embedded?: Record<string, any>;
  [key: string]: any;
}

function toHalPost(post: Post, currentUser?: User): HalResource {
  const links: Record<string, HalLink> = {
    self: { href: `/api/posts/${post.id}` },
    collection: { href: '/api/posts' },
    author: { href: `/api/users/${post.authorId}` },
    comments: { href: `/api/posts/${post.id}/comments` },
  };

  // Conditional links based on state and permissions
  if (post.status === 'draft' && currentUser?.id === post.authorId) {
    links.publish = {
      href: `/api/posts/${post.id}/publish`,
      method: 'POST',
      title: 'Publish this post',
    };
    links.edit = {
      href: `/api/posts/${post.id}`,
      method: 'PUT',
    };
  }

  if (currentUser?.id === post.authorId || currentUser?.role === 'admin') {
    links.delete = {
      href: `/api/posts/${post.id}`,
      method: 'DELETE',
    };
  }

  return {
    id: post.id,
    title: post.title,
    status: post.status,
    createdAt: post.createdAt,
    _links: links,
  };
}

Collection with Pagination Links

function toHalCollection(
  posts: Post[],
  page: number,
  perPage: number,
  total: number,
  currentUser?: User
): HalResource {
  const totalPages = Math.ceil(total / perPage);

  const links: Record<string, HalLink> = {
    self: { href: `/api/posts?page=${page}&per_page=${perPage}` },
    first: { href: `/api/posts?page=1&per_page=${perPage}` },
    last: { href: `/api/posts?page=${totalPages}&per_page=${perPage}` },
  };

  if (page < totalPages) {
    links.next = { href: `/api/posts?page=${page + 1}&per_page=${perPage}` };
  }
  if (page > 1) {
    links.prev = { href: `/api/posts?page=${page - 1}&per_page=${perPage}` };
  }

  return {
    _links: links,
    _embedded: {
      posts: posts.map(p => toHalPost(p, currentUser)),
    },
    total,
    page,
    perPage,
  };
}

// Express route
app.get('/api/posts', async (req, res) => {
  const page = parseInt(req.query.page as string) || 1;
  const perPage = parseInt(req.query.per_page as string) || 20;

  const { posts, total } = await postService.findAll(page, perPage);
  const hal = toHalCollection(posts, page, perPage, total, req.user);

  res.type('application/hal+json').json(hal);
});

JSON:API Format

Another popular standard that includes relationships and links:

function toJsonApi(post: Post) {
  return {
    data: {
      type: 'posts',
      id: String(post.id),
      attributes: {
        title: post.title,
        status: post.status,
        createdAt: post.createdAt,
      },
      relationships: {
        author: {
          links: { related: `/api/users/${post.authorId}` },
          data: { type: 'users', id: String(post.authorId) },
        },
        comments: {
          links: { related: `/api/posts/${post.id}/comments` },
        },
      },
      links: {
        self: `/api/posts/${post.id}`,
      },
    },
  };
}

Implementing HATEOAS in NestJS

import { Injectable } from '@nestjs/common';

@Injectable()
export class HalService {
  wrapResource<T extends { id: number }>(
    resource: T,
    type: string,
    extraLinks: Record<string, HalLink> = {}
  ): HalResource {
    return {
      ...resource,
      _links: {
        self: { href: `/api/${type}/${resource.id}` },
        collection: { href: `/api/${type}` },
        ...extraLinks,
      },
    };
  }

  wrapCollection<T>(
    items: HalResource[],
    type: string,
    page: number,
    perPage: number,
    total: number
  ): HalResource {
    const totalPages = Math.ceil(total / perPage);
    const links: Record<string, HalLink> = {
      self: { href: `/api/${type}?page=${page}&per_page=${perPage}` },
      first: { href: `/api/${type}?page=1&per_page=${perPage}` },
      last: { href: `/api/${type}?page=${totalPages}&per_page=${perPage}` },
    };
    if (page > 1) links.prev = { href: `/api/${type}?page=${page - 1}&per_page=${perPage}` };
    if (page < totalPages) links.next = { href: `/api/${type}?page=${page + 1}&per_page=${perPage}` };

    return { _links: links, _embedded: { [type]: items }, total, page, perPage };
  }
}

State-Driven Links

The real power of HATEOAS is that links change based on resource state:

Order State Machine:

  [created] ---> [paid] ---> [shipped] ---> [delivered]
      |            |             |
      v            v             v
  [cancelled]  [refunded]    [returned]

function getOrderLinks(order: Order, user: User): Record<string, HalLink> {
  const links: Record<string, HalLink> = {
    self: { href: `/api/orders/${order.id}` },
  };

  switch (order.status) {
    case 'created':
      links.pay = { href: `/api/orders/${order.id}/pay`, method: 'POST' };
      links.cancel = { href: `/api/orders/${order.id}/cancel`, method: 'POST' };
      break;
    case 'paid':
      if (user.role === 'admin') {
        links.ship = { href: `/api/orders/${order.id}/ship`, method: 'POST' };
        links.refund = { href: `/api/orders/${order.id}/refund`, method: 'POST' };
      }
      break;
    case 'shipped':
      links.track = { href: `/api/orders/${order.id}/tracking` };
      break;
    case 'delivered':
      links.return = { href: `/api/orders/${order.id}/return`, method: 'POST' };
      links.review = { href: `/api/orders/${order.id}/review`, method: 'POST' };
      break;
  }

  return links;
}

URI Templates

HATEOAS can include templated URIs (RFC 6570):

{
  "_links": {
    "search": {
      "href": "/api/posts{?q,page,per_page}",
      "templated": true
    },
    "postBySlug": {
      "href": "/api/posts/{slug}",
      "templated": true
    }
  }
}

Should You Use HATEOAS?

Use it when: - Building a public API consumed by many third parties - Resources have complex state machines - You want to decouple client navigation from server URL structure - You are building a truly RESTful API (Level 3)

Skip it when: - Internal APIs with a single frontend consumer - You control both client and server - The added complexity is not worth the benefit - You are using GraphQL or tRPC (they have their own paradigms)

Most real-world APIs do not implement full HATEOAS. Many adopt a pragmatic middle ground: include a _links.self on every resource and pagination links on collections, but skip the full state-driven link machinery.

Exercises

  1. Build a HAL Response Wrapper

    Create a reusable HalBuilder class that can wrap any resource with _links and _embedded. Support addLink(rel, href, method?), embed(rel, resources), and build() methods.

    Hint

    Use the builder pattern: each method returns 'this' for chaining.

  2. State-Driven Order API

    Build an Express API for orders with statuses: created, paid, shipped, delivered, cancelled. Each GET /orders/:id response should include only the valid next-action links for the current state.

    Hint

    Create a state machine map: { created: ['pay', 'cancel'], paid: ['ship', 'refund'], ... } and generate links from it.

  3. HAL Collection with Pagination

    Implement a GET /api/articles endpoint that returns a HAL collection with _embedded articles and pagination links (self, first, last, next, prev). Omit next on the last page and prev on the first page.

    Hint

    Calculate totalPages = Math.ceil(total / perPage) and conditionally include next/prev links.

Common Problems & Solutions

1 Clients ignore the _links and hardcode URLs anyway
Cause: Developers find it easier to construct URLs themselves than parse _links from responses.
Solution: Document clearly that URLs may change. Provide a client SDK that navigates via links. Consider whether full HATEOAS is worth the overhead for your use case.
2 Circular references in _embedded cause infinite serialization
Cause: A post embeds its author, and the author embeds their posts.
Solution: Only embed one level deep. Use links (href) for deeper relationships instead of embedding. Define a clear embedding depth policy.
