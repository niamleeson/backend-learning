Apache Kafka: Producers, Consumers, Topics, Partitions

Intuition

Kafka is like a newspaper printing press with multiple editions (partitions). Subscribers (consumers) each get their own copy and read at their own pace. The press keeps back issues (retention) so new subscribers can catch up on old news.

Apache Kafka: Producers, Consumers & Topics

Apache Kafka is a distributed event streaming platform designed for high-throughput, fault-tolerant, real-time data pipelines. It serves as the backbone of event-driven architectures at companies processing billions of events per day.

Core Concepts

  • Topics & Partitions: A topic is a category of messages. Each topic is split into partitions, which are ordered, immutable, append-only logs. Partitions enable parallelism—each can be consumed by a different consumer.
  • Producers: Publish messages to topics. Producers choose the partition via a key (hash-based) or round-robin. Acknowledge modes: acks=0 (fire-and-forget), acks=1 (leader only), acks=all (all replicas).
  • Consumers & Consumer Groups: Consumers read from partitions. A consumer group distributes partitions among its members—each partition is consumed by exactly one consumer in the group, enabling parallel processing.
  • Offsets: Each message has a sequential offset within its partition. Consumers track their offset to know where they left off. Offsets can be committed automatically or manually for exactly-once semantics.
  • Replication: Each partition has a configurable number of replicas across brokers. The leader handles reads and writes; followers replicate. On leader failure, a follower is elected.

Key Takeaways

  1. Kafka provides ordering guarantees within a partition but not across partitions—use the message key to ensure related messages go to the same partition.
  2. Consumer groups enable horizontal scaling of consumers while maintaining per-partition ordering.
  3. Kafka retains messages for a configurable duration (not until consumed), enabling replay and multiple independent consumer groups.

Exercises

  1. Build a Kafka Producer-Consumer Pipeline

    Using confluent-kafka or kafka-python, create a producer that publishes user activity events (page views, clicks) to a multi-partition topic. Build a consumer group with 3 consumers that processes events in parallel. Verify partition assignment and ordering.

    Hint

    Create a topic with 6 partitions: kafka-topics --create --topic user-events --partitions 6 --replication-factor 1. Use user_id as the message key so all events for a user go to the same partition.

  2. Implement Manual Offset Management

    Build a Kafka consumer that processes messages and commits offsets manually after successful processing. Simulate a failure mid-batch and verify that uncommitted messages are redelivered on restart.

    Hint

    Set enable.auto.commit=False. After processing a batch, call consumer.commit(). To simulate failure, process 5 messages, then exit without committing. On restart, the consumer will re-read from the last committed offset.

Common Problems & Solutions

1 Consumer group rebalancing causes duplicate message processing and brief processing pauses.
Cause: When consumers join or leave the group, Kafka triggers a rebalance that reassigns partitions. During rebalance, no messages are processed, and some messages may be re-processed if offsets were not committed.
Solution: Use cooperative sticky rebalancing (`partition.assignment.strategy=cooperative-sticky`) to minimize disruption. Commit offsets more frequently. Implement idempotent processing to handle duplicates safely.
2 Kafka consumers fall behind (consumer lag grows) and cannot keep up with the production rate.
Cause: Too few consumers relative to partitions, slow processing per message, or too few partitions to parallelize consumption.
Solution: Add more partitions and consumers (consumers cannot exceed partition count). Optimize processing logic, batch database writes, or offload heavy processing to a separate service. Monitor lag with `kafka-consumer-groups --describe`.
