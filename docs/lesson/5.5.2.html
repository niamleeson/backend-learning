Repository Pattern and Service Layer

Intuition

The repository pattern is like a librarian who knows how to find and store books. The service layer is the program manager who decides which books to order and what rules to follow. Your endpoints just talk to the manager.

Repository Pattern and Service Layer

The repository pattern abstracts database access behind a clean interface, while the service layer contains business logic. Together, they create a testable, maintainable architecture.

from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select
from fastapi import Depends

# Repository: handles data access only
class UserRepository:
    def __init__(self, session: AsyncSession):
        self.session = session

    async def get_by_id(self, user_id: int):
        result = await self.session.execute(
            select(User).where(User.id == user_id)
        )
        return result.scalar_one_or_none()

    async def create(self, user: User) -> User:
        self.session.add(user)
        await self.session.flush()
        return user

# Service: contains business logic
class UserService:
    def __init__(self, repo: UserRepository):
        self.repo = repo

    async def register(self, data: UserCreate) -> User:
        existing = await self.repo.get_by_email(data.email)
        if existing:
            raise ValueError("Email already registered")
        user = User(**data.model_dump())
        return await self.repo.create(user)

# Dependency wiring
async def get_user_service(db: AsyncSession = Depends(get_db)):
    repo = UserRepository(db)
    return UserService(repo)

Key Takeaways: - Repositories encapsulate all database queries for a given entity - Services coordinate business rules and call one or more repositories - FastAPI's dependency injection wires everything together per-request - This separation makes unit testing straightforward with mock repositories

Exercises

  1. Generic Repository Base

    Create a generic BaseRepository[T] using Python generics that provides common CRUD operations (get, list, create, update, delete) for any SQLAlchemy model. Extend it for specific models like UserRepository and ProductRepository.

    Hint

    Use class BaseRepository(Generic[T]) with a model class variable. Implement methods using select(self.model). Subclasses set model = User and add entity-specific methods.

  2. Service with Multiple Repositories

    Create an OrderService that depends on OrderRepository, ProductRepository, and UserRepository. Implement a place_order method that validates the user exists, checks product stock, creates the order, and updates stock.

    Hint

    The service constructor takes all three repositories. Each step calls the appropriate repository. If any step fails, the session rollback (from the yield dependency) handles cleanup.

Common Problems & Solutions

1 Service layer tightly coupled to SQLAlchemy types
Cause: The service directly imports and uses SQLAlchemy models and session types, making it impossible to test without a database.
Solution: Define repository interfaces using Python protocols or abstract base classes. The service depends on the interface, not the implementation. In tests, provide mock repositories that implement the same interface.
2 Multiple repositories using different database sessions
Cause: Each repository creates its own session through the dependency system, leading to inconsistent transactions.
Solution: Inject the session once at the endpoint level and pass it to all repositories. Or create a Unit of Work pattern that manages a single session shared across repositories within one request.
