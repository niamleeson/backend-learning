Design a Payment System (Stripe)

Intuition

Think of a payment system like a very meticulous accountant. Every time money moves, the accountant writes two entries in an ink-only ledger: one account goes up, another goes down. If they make a mistake, they do not erase -- they write a new correcting entry. The idempotency key is like a receipt number: if you show the same receipt twice, the accountant says 'already processed' instead of recording it again. This is exactly how banks have worked for centuries -- the digital version just needs to handle it at internet speed.

Design a Payment System (Stripe)

Introduction

A payment system is one of the most critical components of any commerce platform. Designing a system like Stripe involves handling money transfers between buyers, sellers, and financial institutions with absolute correctness. A single bug can mean lost money, duplicate charges, or regulatory violations. This design focuses on building a payment processing platform that handles card payments, manages merchant accounts, ensures idempotency, and maintains an audit trail.

Step 1: Requirements Gathering

Functional Requirements

  • Payment Processing: Accept card payments, process charges, handle refunds
  • Merchant Management: Onboard merchants, manage API keys, configure webhooks
  • Payment Methods: Store and tokenize credit/debit cards securely
  • Payouts: Transfer funds from merchant balance to their bank accounts
  • Webhooks: Notify merchants of payment events asynchronously
  • Idempotency: Every payment operation must be safely retryable
  • Ledger: Maintain a complete, immutable record of all financial transactions
  • Dispute Management: Handle chargebacks and disputes

Non-Functional Requirements

  • Consistency: Exactly-once payment processing (no double charges)
  • Availability: 99.999% uptime for payment processing
  • Latency: Process a payment in < 2 seconds
  • Security: PCI DSS Level 1 compliance, encryption at rest and in transit
  • Auditability: Complete transaction history, immutable ledger
  • Regulatory: Support for different currencies, tax calculations, compliance

Step 2: Capacity Estimation

Assumptions:

Traffic:

Storage:
- Transaction record: ~2KB each
- 500M * 2KB = 1TB/month
- Ledger entries: 3-5 per transaction (auth, capture, settle) = ~5TB/month
- 5-year retention: ~300TB

Bandwidth:
- Inbound: ~190 * 2KB = 380KB/s avg
- Outbound (webhooks): ~190 * 1KB * 3 retries = 570KB/s avg

Step 4: API Design

Payment Intent (Two-Phase Commit Pattern)

import express from 'express';
import crypto from 'node:crypto';
const router = express.Router();

// POST /api/payment-intents
// Creates a payment intent (authorization hold)
router.post('/payment-intents', authenticateMerchant, async (req, res) => {
  const { amount, currency, paymentMethodId, metadata, idempotencyKey } = req.body;

  // 1. Idempotency check
  const existingResult = await idempotencyService.get(idempotencyKey);
  if (existingResult) {
    return res.status(200).json(existingResult);
  }

  // 2. Validate payment method
  const paymentMethod = await vaultService.getPaymentMethod(paymentMethodId);
  if (!paymentMethod) {
    return res.status(400).json({ error: 'Invalid payment method' });
  }

  // 3. Create payment intent record
  const paymentIntent = await prisma.paymentIntent.create({
    data: {
      id: `pi_${crypto.randomUUID().replace(/-/g, '')}`,
      merchantId: req.merchant.id,
      amount,
      currency: currency.toUpperCase(),
      status: 'CREATED',
      paymentMethodId,
      metadata: metadata || {},
    },
  });

  // 4. Store idempotency result
  await idempotencyService.set(idempotencyKey, paymentIntent, 86400);

  res.status(201).json(paymentIntent);
});

// POST /api/payment-intents/:id/confirm
// Actually processes the payment
router.post('/payment-intents/:id/confirm', authenticateMerchant, async (req, res) => {
  const { idempotencyKey } = req.body;

  const existingResult = await idempotencyService.get(idempotencyKey);
  if (existingResult) return res.json(existingResult);

  const paymentIntent = await prisma.paymentIntent.findUnique({
    where: { id: req.params.id },
  });

  if (!paymentIntent) return res.status(404).json({ error: 'Not found' });
  if (paymentIntent.status !== 'CREATED') {
    return res.status(400).json({ error: `Cannot confirm: status is ${paymentIntent.status}` });
  }

  try {
    // 5. Process via payment processor (card network)
    const processorResult = await paymentProcessor.authorize({
      amount: paymentIntent.amount,
      currency: paymentIntent.currency,
      paymentMethodToken: paymentIntent.paymentMethodId,
    });

    // 6. Update status and create ledger entries atomically
    const result = await prisma.$transaction(async (tx) => {
      const updated = await tx.paymentIntent.update({
        where: { id: paymentIntent.id },
        data: {
          status: processorResult.authorized ? 'SUCCEEDED' : 'FAILED',
          processorId: processorResult.id,
          failureReason: processorResult.declineReason || null,
        },
      });

      if (processorResult.authorized) {
        // Double-entry ledger
        await tx.ledgerEntry.createMany({
          data: [
            {
              paymentIntentId: paymentIntent.id,
              accountId: paymentIntent.merchantId,
              type: 'CREDIT',
              amount: paymentIntent.amount,
              currency: paymentIntent.currency,
            },
            {
              paymentIntentId: paymentIntent.id,
              accountId: 'PLATFORM_RECEIVABLE',
              type: 'DEBIT',
              amount: paymentIntent.amount,
              currency: paymentIntent.currency,
            },
          ],
        });

        // Update merchant balance
        await tx.merchantBalance.update({
          where: { merchantId: paymentIntent.merchantId },
          data: { pendingBalance: { increment: paymentIntent.amount } },
        });
      }

      return updated;
    });

    // 7. Emit event for webhook delivery
    await kafka.produce('payment.events', {
      type: result.status === 'SUCCEEDED'
        ? 'payment_intent.succeeded'
        : 'payment_intent.failed',
      data: result,
    });

    await idempotencyService.set(idempotencyKey, result, 86400);
    res.json(result);
  } catch (err) {
    // On error, mark as failed
    await prisma.paymentIntent.update({
      where: { id: paymentIntent.id },
      data: { status: 'FAILED', failureReason: err.message },
    });
    res.status(500).json({ error: 'Payment processing failed' });
  }
});

Refund API

// POST /api/refunds
router.post('/refunds', authenticateMerchant, async (req, res) => {
  const { paymentIntentId, amount, reason, idempotencyKey } = req.body;

  const existingResult = await idempotencyService.get(idempotencyKey);
  if (existingResult) return res.json(existingResult);

  const paymentIntent = await prisma.paymentIntent.findUnique({
    where: { id: paymentIntentId },
  });

  if (!paymentIntent || paymentIntent.status !== 'SUCCEEDED') {
    return res.status(400).json({ error: 'Cannot refund this payment' });
  }

  const refundedSoFar = await prisma.refund.aggregate({
    where: { paymentIntentId, status: 'SUCCEEDED' },
    _sum: { amount: true },
  });

  const refundableAmount = paymentIntent.amount - (refundedSoFar._sum.amount || 0);
  const refundAmount = amount || refundableAmount;

  if (refundAmount > refundableAmount) {
    return res.status(400).json({ error: 'Refund exceeds refundable amount' });
  }

  const refund = await prisma.$transaction(async (tx) => {
    const refund = await tx.refund.create({
      data: {
        id: `re_${crypto.randomUUID().replace(/-/g, '')}`,
        paymentIntentId,
        amount: refundAmount,
        currency: paymentIntent.currency,
        reason,
        status: 'PENDING',
      },
    });

    // Reverse ledger entries
    await tx.ledgerEntry.createMany({
      data: [
        {
          refundId: refund.id,
          accountId: paymentIntent.merchantId,
          type: 'DEBIT',
          amount: refundAmount,
          currency: paymentIntent.currency,
        },
        {
          refundId: refund.id,
          accountId: 'PLATFORM_RECEIVABLE',
          type: 'CREDIT',
          amount: refundAmount,
          currency: paymentIntent.currency,
        },
      ],
    });

    await tx.merchantBalance.update({
      where: { merchantId: paymentIntent.merchantId },
      data: { pendingBalance: { decrement: refundAmount } },
    });

    return refund;
  });

  // Process refund with card network asynchronously
  await refundQueue.add('process-refund', { refundId: refund.id });

  await idempotencyService.set(idempotencyKey, refund, 86400);
  res.status(201).json(refund);
});

Step 6: Deep Dive - Idempotency

Idempotency is the single most important property of a payment system. Network failures, timeouts, and retries are inevitable. A payment must succeed exactly once.

Problem:
Client ──> Payment Service ──> Card Network
                              (processes charge)
           <── timeout ──
           (network issue, response lost)

Client retries ──> Payment Service ──> Card Network
                                      (charges AGAIN!)
           DOUBLE CHARGE!

// Idempotency service using Redis
class IdempotencyService {
  constructor(redis) {
    this.redis = redis;
  }

  async get(key) {
    if (!key) return null;
    const result = await this.redis.get(`idempotency:${key}`);
    return result ? JSON.parse(result) : null;
  }

  async set(key, response, ttlSeconds = 86400) {
    if (!key) return;
    await this.redis.setex(
      `idempotency:${key}`,
      ttlSeconds,
      JSON.stringify(response)
    );
  }

  // Lock to prevent concurrent processing of same key
  async acquireLock(key, ttlMs = 30000) {
    const lockKey = `idempotency:lock:${key}`;
    const result = await this.redis.set(lockKey, '1', 'PX', ttlMs, 'NX');
    return result === 'OK';
  }

  async releaseLock(key) {
    await this.redis.del(`idempotency:lock:${key}`);
  }
}

// Usage in middleware
async function idempotencyMiddleware(req, res, next) {
  const key = req.headers['idempotency-key'];
  if (!key) return next();

  // Check if we already have a result
  const existing = await idempotencyService.get(key);
  if (existing) {
    return res.status(existing.statusCode).json(existing.body);
  }

  // Acquire lock to prevent concurrent processing
  const locked = await idempotencyService.acquireLock(key);
  if (!locked) {
    return res.status(409).json({ error: 'Request is being processed' });
  }

  // Wrap res.json to capture the response
  const originalJson = res.json.bind(res);
  res.json = async (body) => {
    await idempotencyService.set(key, {
      statusCode: res.statusCode,
      body,
    });
    await idempotencyService.releaseLock(key);
    return originalJson(body);
  };

  next();
}

Step 7: Deep Dive - Double-Entry Ledger

Every financial transaction creates TWO ledger entries:
- A DEBIT from one account
- A CREDIT to another account

The sum of all debits MUST equal the sum of all credits.

Payment of $100 from customer to merchant:
┌──────────────────┬────────┬────────┐
│ Account          │ Debit  │ Credit │
├──────────────────┼────────┼────────┤
│ Platform Recv.   │ $100   │        │
│ Merchant Balance │        │ $100   │
└──────────────────┴────────┴────────┘

Refund of $100:
┌──────────────────┬────────┬────────┐
│ Account          │ Debit  │ Credit │
├──────────────────┼────────┼────────┤
│ Merchant Balance │ $100   │        │
│ Platform Recv.   │        │ $100   │
└──────────────────┴────────┴────────┘

Platform fee of $2.90 (2.9%):
┌──────────────────┬────────┬────────┐
│ Account          │ Debit  │ Credit │
├──────────────────┼────────┼────────┤
│ Merchant Balance │ $2.90  │        │
│ Platform Revenue │        │ $2.90  │
└──────────────────┴────────┴────────┘

// Ledger service -- append only, never update or delete
class LedgerService {
  constructor(prisma) {
    this.prisma = prisma;
  }

  async recordPayment(tx, paymentIntentId, merchantId, amount, currency) {
    await tx.ledgerEntry.createMany({
      data: [
        {
          paymentIntentId,
          accountId: 'PLATFORM_RECEIVABLE',
          type: 'DEBIT',
          amount,
          currency,
        },
        {
          paymentIntentId,
          accountId: merchantId,
          type: 'CREDIT',
          amount,
          currency,
        },
      ],
    });
  }

  async getBalance(accountId) {
    const result = await this.prisma.$queryRaw`
      SELECT
        COALESCE(SUM(CASE WHEN type = 'CREDIT' THEN amount ELSE 0 END), 0) -
        COALESCE(SUM(CASE WHEN type = 'DEBIT' THEN amount ELSE 0 END), 0)
        AS balance
      FROM ledger_entries
      WHERE account_id = ${accountId}
    `;
    return result[0].balance;
  }

  async verifyIntegrity() {
    // Total debits must equal total credits
    const result = await this.prisma.$queryRaw`
      SELECT
        SUM(CASE WHEN type = 'DEBIT' THEN amount ELSE 0 END) as total_debits,
        SUM(CASE WHEN type = 'CREDIT' THEN amount ELSE 0 END) as total_credits
      FROM ledger_entries
    `;
    return result[0].total_debits === result[0].total_credits;
  }
}

Step 8: Webhook Delivery

// Webhook worker with exponential backoff retries
import { Worker } from 'bullmq';
import crypto from 'node:crypto';

const webhookWorker = new Worker('webhooks', async (job) => {
  const { merchantId, eventType, payload } = job.data;

  const merchant = await prisma.merchant.findUnique({
    where: { id: merchantId },
  });

  if (!merchant?.webhookUrl) return;

  // Sign the payload
  const timestamp = Math.floor(Date.now() / 1000);
  const signedPayload = `${timestamp}.${JSON.stringify(payload)}`;
  const signature = crypto
    .createHmac('sha256', merchant.webhookSecret)
    .update(signedPayload)
    .digest('hex');

  const response = await fetch(merchant.webhookUrl, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Webhook-Signature': `t=${timestamp},v1=${signature}`,
    },
    body: JSON.stringify({ type: eventType, data: payload }),
    signal: AbortSignal.timeout(10000), // 10s timeout
  });

  if (!response.ok) {
    throw new Error(`Webhook delivery failed: ${response.status}`);
  }
}, {
  connection: redisConnection,
  limiter: { max: 100, duration: 1000 }, // rate limit outbound webhooks
});

// Retry with exponential backoff: 1min, 5min, 30min, 2hr, 24hr
webhookWorker.opts.backoff = {
  type: 'exponential',
  delay: 60000,
};

Summary

A payment system demands the highest standards of correctness, security, and auditability. The core patterns are: idempotency keys (exactly-once processing), double-entry ledger (financial correctness), two-phase payment flow (authorize then capture), tokenized card storage (PCI compliance), and async webhook delivery (merchant notification). Shard by merchant for scalability. Partition the ledger by time. Never update or delete ledger entries -- only append new correcting entries.

Common Problems & Solutions

1 Customers are being double-charged for the same purchase
Symptoms: Customer support reports of duplicate charges. Bank statements show two identical charges seconds apart. Transaction logs show two payment records with different IDs but same amount and card.
Cause: Network timeout during payment processing causes the client to retry. Without idempotency, the retry creates a second charge. The card network processes both authorizations as separate transactions.
Solution: Require idempotency keys on all payment creation endpoints. Before processing, check if the key exists in Redis. Use a distributed lock to prevent concurrent processing of the same key. Return the cached result for duplicate requests.
2 Ledger balance does not match actual bank account balance
Cause: A payment was recorded in the ledger but the bank transfer failed, or a bank transfer succeeded but the ledger entry was not created due to application crash between the two operations.
Solution: Use the Saga pattern: record the ledger entry and bank transfer request in the same database transaction. Process the bank transfer asynchronously. Reconcile daily by comparing ledger totals with bank statements. Flag discrepancies for manual review.
3 Webhook deliveries are backing up, causing merchants to miss payment notifications
Cause: One merchant's webhook endpoint is slow or down, and the retry mechanism is consuming all worker capacity. Other merchants' webhooks are delayed because they share the same queue.
Solution: Use separate queues or rate limits per merchant. Implement circuit breakers: after N consecutive failures, stop attempting delivery to that merchant and alert them. Use priority queues so first-attempt deliveries are not blocked by retries.
