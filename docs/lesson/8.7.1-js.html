Event Sourcing and CQRS in Practice

Intuition

Event sourcing is like a bank ledger. Instead of overwriting your balance, every deposit and withdrawal is recorded as a line item. You can always recalculate the balance by replaying the ledger from the beginning, and you have a complete audit trail. CQRS is like having a separate account summary (read model) that is updated whenever a new line is added to the ledger, so you don't have to read the entire ledger every time you want to check your balance.

Event Sourcing and CQRS in Practice

Introduction

Event Sourcing stores the state of an application as a sequence of events rather than just the current state. CQRS (Command Query Responsibility Segregation) separates the read and write models. Together, they enable powerful patterns for audit trails, time travel, and scalable architectures.

Traditional vs Event Sourcing

Traditional (State-Based):
  UPDATE accounts SET balance = 950 WHERE id = 1;
  -- Previous balance is LOST

Event Sourcing:
  Event 1: AccountCreated { id: 1, initialBalance: 1000 }
  Event 2: MoneyDeposited { id: 1, amount: 500 }
  Event 3: MoneyWithdrawn { id: 1, amount: 200 }
  Event 4: MoneyWithdrawn { id: 1, amount: 350 }
  -- Current balance: 1000 + 500 - 200 - 350 = 950
  -- FULL HISTORY preserved

Event Store Implementation

// Event store using PostgreSQL
class EventStore {
  constructor(prisma) {
    this.prisma = prisma;
  }

  async appendEvent(streamId, event, expectedVersion) {
    // Optimistic concurrency control
    const currentVersion = await this.getStreamVersion(streamId);
    if (expectedVersion !== undefined && currentVersion !== expectedVersion) {
      throw new Error(
        `Concurrency conflict: expected version ${expectedVersion}, got ${currentVersion}`
      );
    }

    return this.prisma.event.create({
      data: {
        streamId,
        type: event.type,
        data: event.data,
        metadata: event.metadata || {},
        version: currentVersion + 1,
        timestamp: new Date(),
      },
    });
  }

  async getEvents(streamId, fromVersion = 0) {
    return this.prisma.event.findMany({
      where: {
        streamId,
        version: { gt: fromVersion },
      },
      orderBy: { version: 'asc' },
    });
  }

  async getStreamVersion(streamId) {
    const latest = await this.prisma.event.findFirst({
      where: { streamId },
      orderBy: { version: 'desc' },
    });
    return latest?.version || 0;
  }

  async getAllEvents(fromPosition = 0, limit = 100) {
    return this.prisma.event.findMany({
      where: { id: { gt: fromPosition } },
      orderBy: { id: 'asc' },
      take: limit,
    });
  }
}

Aggregate with Event Sourcing

class OrderAggregate {
  constructor() {
    this.id = null;
    this.status = null;
    this.items = [];
    this.total = 0;
    this.version = 0;
    this.pendingEvents = [];
  }

  // Rebuild state from events
  static fromEvents(events) {
    const aggregate = new OrderAggregate();
    for (const event of events) {
      aggregate.apply(event, false);
    }
    return aggregate;
  }

  // Apply event to update state
  apply(event, isNew = true) {
    switch (event.type) {
      case 'OrderCreated':
        this.id = event.data.orderId;
        this.status = 'PENDING';
        this.items = event.data.items;
        this.total = event.data.total;
        break;
      case 'OrderPaid':
        this.status = 'PAID';
        break;
      case 'OrderShipped':
        this.status = 'SHIPPED';
        this.trackingNumber = event.data.trackingNumber;
        break;
      case 'OrderCancelled':
        this.status = 'CANCELLED';
        this.cancellationReason = event.data.reason;
        break;
      case 'ItemAdded':
        this.items.push(event.data.item);
        this.total += event.data.item.price;
        break;
    }

    this.version++;
    if (isNew) {
      this.pendingEvents.push(event);
    }
  }

  // Command handlers (business logic)
  create(orderId, items, total) {
    if (this.id) throw new Error('Order already exists');
    this.apply({
      type: 'OrderCreated',
      data: { orderId, items, total },
    });
  }

  pay() {
    if (this.status !== 'PENDING') {
      throw new Error(`Cannot pay order in ${this.status} status`);
    }
    this.apply({ type: 'OrderPaid', data: {} });
  }

  ship(trackingNumber) {
    if (this.status !== 'PAID') {
      throw new Error(`Cannot ship order in ${this.status} status`);
    }
    this.apply({
      type: 'OrderShipped',
      data: { trackingNumber },
    });
  }

  cancel(reason) {
    if (['SHIPPED', 'CANCELLED'].includes(this.status)) {
      throw new Error(`Cannot cancel order in ${this.status} status`);
    }
    this.apply({
      type: 'OrderCancelled',
      data: { reason },
    });
  }
}

// Repository
class OrderRepository {
  constructor(eventStore) {
    this.eventStore = eventStore;
  }

  async load(orderId) {
    const events = await this.eventStore.getEvents(`order-${orderId}`);
    if (events.length === 0) return new OrderAggregate();
    return OrderAggregate.fromEvents(events);
  }

  async save(order) {
    for (const event of order.pendingEvents) {
      await this.eventStore.appendEvent(
        `order-${order.id}`,
        event,
        order.version - order.pendingEvents.length
      );
    }
    order.pendingEvents = [];
  }
}

// Usage
const orderRepo = new OrderRepository(eventStore);

async function createOrder(data) {
  const order = new OrderAggregate();
  order.create(data.orderId, data.items, data.total);
  await orderRepo.save(order);
  return order;
}

async function payOrder(orderId) {
  const order = await orderRepo.load(orderId);
  order.pay();
  await orderRepo.save(order);
  return order;
}

CQRS: Separating Reads and Writes

┌──────────┐    Command     ┌────────────────┐
│  Client   │──────────────>│ Command Handler │──> Event Store
│           │               │ (Write Model)   │    (append only)
│           │               └────────────────┘
│           │                                        │
│           │                                   Events│
│           │                                        ▼
│           │    Query      ┌────────────────┐  ┌──────────┐
│           │──────────────>│ Query Handler  │──│ Read DB   │
│           │               │ (Read Model)   │  │(projected)│
└──────────┘               └────────────────┘  └──────────┘

// Read model projection (materializes events into queryable state)
class OrderProjection {
  constructor(prisma) {
    this.prisma = prisma;
  }

  async project(event) {
    switch (event.type) {
      case 'OrderCreated':
        await this.prisma.orderView.create({
          data: {
            id: event.data.orderId,
            status: 'PENDING',
            itemCount: event.data.items.length,
            total: event.data.total,
            createdAt: event.timestamp,
          },
        });
        break;

      case 'OrderPaid':
        await this.prisma.orderView.update({
          where: { id: event.streamId.replace('order-', '') },
          data: { status: 'PAID', paidAt: event.timestamp },
        });
        break;

      case 'OrderShipped':
        await this.prisma.orderView.update({
          where: { id: event.streamId.replace('order-', '') },
          data: {
            status: 'SHIPPED',
            trackingNumber: event.data.trackingNumber,
            shippedAt: event.timestamp,
          },
        });
        break;
    }
  }
}

// Event processor (reads from event store, projects to read DB)
class EventProcessor {
  constructor(eventStore, projections) {
    this.eventStore = eventStore;
    this.projections = projections;
    this.lastPosition = 0;
  }

  async processNewEvents() {
    const events = await this.eventStore.getAllEvents(this.lastPosition, 100);

    for (const event of events) {
      for (const projection of this.projections) {
        await projection.project(event);
      }
      this.lastPosition = event.id;
    }
  }

  // Run continuously
  async start(intervalMs = 1000) {
    setInterval(() => this.processNewEvents(), intervalMs);
  }
}

// Query service (reads from projected read model)
class OrderQueryService {
  constructor(prisma) {
    this.prisma = prisma;
  }

  async getOrder(id) {
    return this.prisma.orderView.findUnique({ where: { id } });
  }

  async listOrders(filter) {
    return this.prisma.orderView.findMany({
      where: filter,
      orderBy: { createdAt: 'desc' },
    });
  }

  async getOrderStats() {
    return this.prisma.orderView.groupBy({
      by: ['status'],
      _count: true,
      _sum: { total: true },
    });
  }
}

Snapshots for Performance

Replaying thousands of events is slow. Snapshots cache the aggregate state periodically:

class SnapshotRepository {
  constructor(redis, eventStore) {
    this.redis = redis;
    this.eventStore = eventStore;
  }

  async load(streamId) {
    // Try snapshot first
    const snapshot = await this.redis.get(`snapshot:${streamId}`);
    let aggregate;
    let fromVersion = 0;

    if (snapshot) {
      const { state, version } = JSON.parse(snapshot);
      aggregate = OrderAggregate.fromSnapshot(state);
      aggregate.version = version;
      fromVersion = version;
    } else {
      aggregate = new OrderAggregate();
    }

    // Replay events since snapshot
    const events = await this.eventStore.getEvents(streamId, fromVersion);
    for (const event of events) {
      aggregate.apply(event, false);
    }

    // Save new snapshot every 50 events
    if (aggregate.version - fromVersion > 50) {
      await this.saveSnapshot(streamId, aggregate);
    }

    return aggregate;
  }

  async saveSnapshot(streamId, aggregate) {
    await this.redis.setex(
      `snapshot:${streamId}`,
      86400,
      JSON.stringify({
        state: aggregate.toSnapshot(),
        version: aggregate.version,
      })
    );
  }
}

When to Use Event Sourcing + CQRS

Good fit:                          Poor fit:
- Audit trail required             - Simple CRUD apps
- Complex business rules           - Low-volume systems
- Need to replay/debug history     - Team unfamiliar with pattern
- Different read/write patterns    - Tight consistency required
- Financial transactions           - Simple data models
- Collaboration/editing systems

Common Pitfalls

  1. Event schema evolution: Changing event structure breaks replay. Use versioned events and upcasters.
  2. Too many events per aggregate: Replay becomes slow. Use snapshots every N events.
  3. Eventual consistency confusion: Read model may lag behind write model. Design UI to handle this.
  4. Over-applying the pattern: Not every entity needs event sourcing. Use it where history and audit matters.

Summary

Event Sourcing stores state as an immutable sequence of events, providing a complete audit trail and the ability to rebuild state at any point in time. CQRS separates the write model (event store) from the read model (projected views), allowing each to be optimized independently. Use snapshots to avoid replaying long event streams. The combination is powerful but adds complexity — apply it selectively to domains where history and auditability are critical.

    Hint

    Listen to events from the store. Switch on event type and upsert into the read model table.

Common Problems & Solutions

1 Event replay takes too long for aggregates with many events
Cause: An aggregate with 10,000+ events requires replaying all of them to rebuild current state.
Solution: Implement periodic snapshots that cache aggregate state. On load, start from latest snapshot and replay only subsequent events.
2 Read model is stale (eventual consistency gap)
Cause: The projection process has not yet processed the latest events. Users see outdated data after a write.
Solution: For read-after-write scenarios, query the event store directly. Or return the expected state optimistically in the write response.
