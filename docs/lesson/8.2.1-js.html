Cache-Aside, Write-Through, and Write-Behind

Intuition

Cache-aside is like checking your notebook before asking the teacher (database) a question, then writing down the answer. Write-through is having a secretary who always updates both your notebook and the official record simultaneously. Write-behind is jotting notes on a whiteboard and having someone transcribe them to the official record later.

Cache-Aside, Write-Through, and Write-Behind

Introduction

Caching is the most impactful performance optimization in backend systems. By storing frequently accessed data in a fast storage layer (like Redis), you can reduce database load by 90%+ and cut response times from hundreds of milliseconds to single-digit milliseconds. But choosing the right caching pattern determines whether your cache helps or creates subtle bugs.

Why Cache?

Without Cache:                      With Cache:
Client ──> Server ──> Database      Client ──> Server ──> Redis (1ms)
              └── 50-200ms                        │ miss?
                                                  └──> Database (50ms)
                                                       └──> Store in Redis

A typical read-heavy application has a 90/10 read/write ratio. Caching reads alone can eliminate 80-95% of database queries.

Cache-Aside (Lazy Loading)

The most common pattern. The application manages the cache explicitly.

Read Flow:
  1. App checks cache
  2. Cache HIT  ──> Return cached data
  3. Cache MISS ──> Query database
  4. Store result in cache
  5. Return data

Write Flow:
  1. Write to database
  2. Invalidate (delete) cache entry

┌────────┐    1. GET     ┌───────┐
│  App   │──────────────>│ Redis │
│        │<──────────────│       │
│        │   2. HIT/MISS └───────┘
│        │
│        │   3. MISS ──> ┌────────┐
│        │──────────────>│   DB   │
│        │<──────────────│        │
│        │   4. Data     └────────┘
│        │
│        │   5. SET      ┌───────┐
│        │──────────────>│ Redis │
└────────┘               └───────┘

Implementation with ioredis

import Redis from 'ioredis';
import { PrismaClient } from '@prisma/client';

const redis = new Redis(process.env.REDIS_URL);
const prisma = new PrismaClient();

// Generic cache-aside helper
async function cacheAside(key, ttlSeconds, fetchFn) {
  // 1. Check cache
  const cached = await redis.get(key);
  if (cached) {
    return JSON.parse(cached);
  }

  // 2. Cache miss - fetch from source
  const data = await fetchFn();

  // 3. Store in cache
  if (data !== null && data !== undefined) {
    await redis.setex(key, ttlSeconds, JSON.stringify(data));
  }

  return data;
}

// Usage
async function getUser(userId) {
  return cacheAside(
    `user:${userId}`,
    3600, // 1 hour TTL
    () => prisma.user.findUnique({ where: { id: userId } })
  );
}

async function getUserPosts(userId, page = 1) {
  return cacheAside(
    `user:${userId}:posts:page:${page}`,
    300, // 5 min TTL
    () => prisma.post.findMany({
      where: { authorId: userId },
      skip: (page - 1) * 20,
      take: 20,
      orderBy: { createdAt: 'desc' },
    })
  );
}

// Invalidation on write
async function updateUser(userId, data) {
  const user = await prisma.user.update({
    where: { id: userId },
    data,
  });

  // Invalidate cache
  await redis.del(`user:${userId}`);
  // Also invalidate related caches
  const postKeys = await redis.keys(`user:${userId}:posts:*`);
  if (postKeys.length > 0) {
    await redis.del(...postKeys);
  }

  return user;
}

Pros: Only caches data that is actually requested; cache misses are lazy-loaded. Cons: Cache miss penalty (first request is slow); potential for stale data between write and invalidation.

Write-Through

Every write goes through the cache first, then to the database. The cache is always up-to-date.

Write Flow:
  1. App writes to cache
  2. Cache writes to database (synchronously)
  3. Return success

Read Flow:
  1. Always read from cache (always up-to-date)

┌────────┐    1. Write   ┌───────┐    2. Write   ┌────────┐
│  App   │──────────────>│ Redis │──────────────>│   DB   │
│        │<──────────────│       │<──────────────│        │
│        │   3. ACK      └───────┘   Sync ACK    └────────┘

Implementation

class WriteThroughCache {
  constructor(redis, prisma) {
    this.redis = redis;
    this.prisma = prisma;
  }

  async getUser(userId) {
    // Always read from cache
    const cached = await this.redis.get(`user:${userId}`);
    if (cached) return JSON.parse(cached);

    // Initial population
    const user = await this.prisma.user.findUnique({
      where: { id: userId },
    });
    if (user) {
      await this.redis.setex(`user:${userId}`, 3600, JSON.stringify(user));
    }
    return user;
  }

  async createUser(data) {
    // Write to DB first (source of truth)
    const user = await this.prisma.user.create({ data });

    // Then update cache
    await this.redis.setex(
      `user:${user.id}`,
      3600,
      JSON.stringify(user)
    );

    return user;
  }

  async updateUser(userId, data) {
    // Write to DB
    const user = await this.prisma.user.update({
      where: { id: userId },
      data,
    });

    // Update cache (not delete — write-through keeps cache warm)
    await this.redis.setex(
      `user:${userId}`,
      3600,
      JSON.stringify(user)
    );

    return user;
  }
}

Pros: Cache is always consistent with DB; no stale data for cached entries; reads are always fast. Cons: Write latency increases (must write to both); caches data that may never be read; higher write amplification.

Write-Behind (Write-Back)

Writes go to the cache first and are asynchronously flushed to the database in batches. This is the fastest write pattern but the most dangerous.

Write Flow:
  1. App writes to cache
  2. Return success immediately
  3. Background worker flushes to DB periodically

┌────────┐    1. Write   ┌───────┐
│  App   │──────────────>│ Redis │
│        │<──────────────│       │
│        │   2. ACK      └───┬───┘
                             │
                     3. Async flush (batched)
                             │
                         ┌───▼───┐
                         │  DB   │
                         └───────┘

Implementation with BullMQ

import { Queue, Worker } from 'bullmq';
import Redis from 'ioredis';

const redis = new Redis(process.env.REDIS_URL);
const writeQueue = new Queue('db-writes', {
  connection: { host: 'localhost', port: 6379 },
});

// Write to cache + enqueue DB write
async function writeBehind(key, data, dbOperation) {
  // 1. Write to cache immediately
  await redis.setex(key, 3600, JSON.stringify(data));

  // 2. Enqueue async DB write
  await writeQueue.add('db-write', {
    key,
    data,
    operation: dbOperation,
    timestamp: Date.now(),
  });
}

// Background worker processes DB writes
const worker = new Worker('db-writes', async (job) => {
  const { key, data, operation } = job.data;

  switch (operation) {
    case 'createUser':
      await prisma.user.create({ data });
      break;
    case 'updateUser':
      await prisma.user.update({
        where: { id: data.id },
        data,
      });
      break;
  }
}, {
  connection: { host: 'localhost', port: 6379 },
  concurrency: 10,
});

// Usage
async function updateUserProfile(userId, profileData) {
  const updatedUser = { id: userId, ...profileData, updatedAt: new Date() };
  await writeBehind(
    `user:${userId}`,
    updatedUser,
    'updateUser'
  );
  return updatedUser; // Return immediately, DB write happens later
}

Pros: Extremely fast writes; batching reduces DB load; great for write-heavy workloads. Cons: Data loss risk if Redis crashes before flush; complex error handling; eventual consistency.

Pattern Comparison

┌──────────────────┬──────────────┬───────────────┬──────────────┐
│                  │ Cache-Aside  │ Write-Through │ Write-Behind │
├──────────────────┼──────────────┼───────────────┼──────────────┤
│ Read latency     │ Miss: slow   │ Always fast   │ Always fast  │
│                  │ Hit: fast    │               │              │
├──────────────────┼──────────────┼───────────────┼──────────────┤
│ Write latency    │ DB speed     │ DB + Cache    │ Cache only   │
├──────────────────┼──────────────┼───────────────┼──────────────┤
│ Consistency      │ Eventual     │ Strong        │ Eventual     │
├──────────────────┼──────────────┼───────────────┼──────────────┤
│ Data loss risk   │ None         │ None          │ Yes (crash)  │
├──────────────────┼──────────────┼───────────────┼──────────────┤
│ Complexity       │ Low          │ Medium        │ High         │
├──────────────────┼──────────────┼───────────────┼──────────────┤
│ Best for         │ General use  │ Read-heavy +  │ Write-heavy  │
│                  │              │ consistency   │ workloads    │
└──────────────────┴──────────────┴───────────────┴──────────────┘

Read-Through Cache

A variation where the cache itself is responsible for loading data on a miss (the application never talks to the DB directly for reads):

class ReadThroughCache {
  constructor(redis, loaders) {
    this.redis = redis;
    this.loaders = loaders; // { 'user': (id) => fetchUser(id) }
  }

  async get(type, id, ttl = 3600) {
    const key = `${type}:${id}`;
    const cached = await this.redis.get(key);

    if (cached) return JSON.parse(cached);

    const loader = this.loaders[type];
    if (!loader) throw new Error(`No loader for type: ${type}`);

    const data = await loader(id);
    if (data) {
      await this.redis.setex(key, ttl, JSON.stringify(data));
    }
    return data;
  }
}

const cache = new ReadThroughCache(redis, {
  user: (id) => prisma.user.findUnique({ where: { id } }),
  post: (id) => prisma.post.findUnique({ where: { id }, include: { author: true } }),
  product: (id) => prisma.product.findUnique({ where: { id } }),
});

// Usage - application never calls DB directly
const user = await cache.get('user', userId);
const post = await cache.get('post', postId);

Hybrid Approach

Most production systems combine patterns:

Cache-Aside for: general API responses, user profiles
Write-Through for: critical data (account balances, inventory)
Write-Behind for: analytics events, activity logs, view counts

// Hybrid cache service
class HybridCacheService {
  // Cache-aside for reads
  async getProduct(productId) {
    return cacheAside(`product:${productId}`, 300, () =>
      prisma.product.findUnique({ where: { id: productId } })
    );
  }

  // Write-through for critical writes
  async updateInventory(productId, quantity) {
    const product = await prisma.product.update({
      where: { id: productId },
      data: { inventory: quantity },
    });
    await redis.setex(`product:${productId}`, 300, JSON.stringify(product));
    return product;
  }

  // Write-behind for non-critical writes
  async trackProductView(productId, userId) {
    await redis.hincrby(`product:${productId}:views`, 'total', 1);
    await viewQueue.add('track-view', { productId, userId });
  }
}

Common Pitfalls

  1. Cache stampede: Multiple requests for the same key hit the DB simultaneously on cache miss. Use locking or probabilistic early expiration.
  2. Caching null results: If a key does not exist in DB, repeated misses hammer the DB. Cache the null result with a short TTL.
  3. Serialization overhead: JSON.stringify/parse on large objects can be expensive. Consider MessagePack or Protocol Buffers.
  4. Key naming collisions: Use consistent key naming: {entity}:{id}:{subresource}.

Summary

Cache-aside is the safest default for most applications. Write-through adds consistency guarantees at the cost of write latency. Write-behind maximizes write performance but introduces data loss risk. Most production systems use a hybrid approach, choosing the right pattern for each type of data based on consistency requirements and read/write ratios.

Common Problems & Solutions

1 Cache and database have different data (stale cache)
Cause: Write operation updated the database but failed to invalidate the cache, or another process updated the DB directly.
Solution: Always invalidate/update cache after DB writes in the same try/catch block. Use pub/sub to notify other instances of cache invalidations. Set reasonable TTLs as a safety net.

async function updateUser(id, data) {\n const user = await prisma.user.update({ where: { id }, data });\n await redis.del(user:${id}); // Always invalidate\n return user;\n}

2 Database overwhelmed after Redis restart
Cause: All cached data is lost when Redis restarts. Every request becomes a cache miss, sending all traffic to the database simultaneously (thundering herd).
Solution: Enable Redis persistence (RDB + AOF). Implement cache warming on startup. Use circuit breaker pattern to limit DB queries during recovery. Stagger TTLs to prevent synchronized expiration.
3 Memory usage in Redis growing unbounded
Cause: Cached entries are created but never expire. No eviction policy is configured, or keys are created with patterns that bypass TTL.
Solution: Always set TTL on cache entries. Configure Redis maxmemory and eviction policy (allkeys-lru recommended). Monitor key count and memory usage.

// redis.conf\nmaxmemory 256mb\nmaxmemory-policy allkeys-lru
