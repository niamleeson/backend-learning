HATEOAS and Hypermedia APIs

Intuition

HATEOAS works like a website for machines. When you visit a webpage, you do not memorize every URL -- you follow links on the page. Similarly, an API response tells the client 'here is your data, and here are the links to what you can do next.'

HATEOAS and Hypermedia APIs

HATEOAS (Hypermedia As The Engine Of Application State) is a REST constraint where the server provides links in responses that tell the client what actions are available next. Instead of hardcoding URLs, clients discover them dynamically.

Basic Hypermedia Response

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Link(BaseModel):
    rel: str
    href: str
    method: str = "GET"

class UserResponse(BaseModel):
    id: int
    name: str
    email: str
    links: list[Link]

@app.get("/users/{user_id}", response_model=UserResponse)
def get_user(user_id: int):
    return UserResponse(
        id=user_id,
        name="Alice",
        email="alice@example.com",
        links=[
            Link(rel="self", href=f"/users/{user_id}"),
            Link(rel="orders", href=f"/users/{user_id}/orders"),
            Link(rel="update", href=f"/users/{user_id}", method="PUT"),
            Link(rel="delete", href=f"/users/{user_id}", method="DELETE"),
        ]
    )

Collection with Pagination Links

@app.get("/users")
def list_users(page: int = 1, size: int = 10):
    users = [{"id": i, "name": f"User {i}"} for i in range(1, size + 1)]
    return {
        "data": users,
        "links": [
            {"rel": "self", "href": f"/users?page={page}&size={size}"},
            {"rel": "next", "href": f"/users?page={page+1}&size={size}"},
            {"rel": "prev", "href": f"/users?page={max(1,page-1)}&size={size}"},
        ]
    }

Common Mistakes

  • Including links that the current user does not have permission to use
  • Using non-standard relation types without documentation
  • Over-engineering: not every API benefits from full HATEOAS

Key Takeaways

  • HATEOAS decouples clients from URL structures, making APIs more evolvable
  • Use standard link relations (self, next, prev) from IANA where possible
  • Hypermedia is most valuable for APIs with complex state workflows

Exercises

  1. Add Hypermedia Links to a CRUD API

    Extend a basic user CRUD API so that every response includes relevant links: self, collection, create, update, delete. The links should change based on the resource state (e.g., a deactivated user should not have a 'deactivate' link).

    Hint

    Build a helper function that generates links conditionally based on the user's status. Return different link sets for active vs inactive users.

  2. Build a Workflow API with HATEOAS

    Model an order processing workflow (created -> paid -> shipped -> delivered) where each state transition is discoverable through hypermedia links. A 'created' order should show a 'pay' link, a 'paid' order should show a 'ship' link, etc.

    Hint

    Use a dictionary mapping each state to its allowed transitions, and generate links dynamically based on the current order state.

Common Problems & Solutions

1 Client breaks when server URL structure changes
Symptoms: ['404 errors after server refactor', 'Client needs code changes whenever the API restructures routes']
Cause: Client hardcoded API URLs instead of following links from responses.
Solution: Have the client discover URLs from the API root and follow 'rel' links. The only URL the client should know is the entry point.
2 Hypermedia links expose unauthorized actions
Cause: Links are generated without checking the requesting user's permissions.
Solution: Filter links based on the authenticated user's roles and permissions. Only include links for actions the current user is authorized to perform.
