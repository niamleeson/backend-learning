Load Balancing: Algorithms and Strategies

Intuition

A load balancer is like a restaurant host who seats customers across available tables. Round robin fills tables in order; least connections seats at the emptiest table; IP hash always gives regulars their favorite table.

Load Balancing: Algorithms and Strategies

Introduction

When a single server cannot handle all incoming traffic, you distribute requests across multiple servers using a load balancer. Load balancing is the backbone of horizontal scaling and is one of the first things you add when scaling a backend system.

Why Load Balancing?

A single Node.js process can handle thousands of concurrent connections, but it has limits:

  • CPU: Single-threaded event loop saturates one core
  • Memory: V8 heap has a practical limit (~1.5-4GB)
  • Reliability: One server means one point of failure
Without Load Balancer:              With Load Balancer:

Client ──> Server (single)          Client ──> Load Balancer ──┬──> Server 1
                                                               ├──> Server 2
           If it dies,                                         └──> Server 3
           everything dies.
                                    If Server 2 dies,
                                    traffic routes to 1 & 3.

Layer 4 vs Layer 7 Load Balancing

┌─────────────────────────────────────────────────────────┐
│ OSI Layer │ Name        │ LB Operates On                │
├───────────┼─────────────┼───────────────────────────────┤
│ Layer 4   │ Transport   │ TCP/UDP: IP + Port            │
│ Layer 7   │ Application │ HTTP: URL, Headers, Cookies   │
└─────────────────────────────────────────────────────────┘

Layer 4 (L4) load balancers route based on IP and port. They are fast because they do not inspect packet contents. Examples: AWS NLB, HAProxy (TCP mode).

Layer 7 (L7) load balancers inspect HTTP content and can route based on URL path, headers, cookies, or request body. Examples: Nginx, AWS ALB, HAProxy (HTTP mode).

L7 Load Balancer Routing:

  /api/users ────────> User Service Cluster
  /api/orders ───────> Order Service Cluster
  /static/* ─────────> CDN / Static Server
  /ws/* ─────────────> WebSocket Server Cluster

Load Balancing Algorithms

1. Round Robin

Distributes requests sequentially across servers.

class RoundRobinBalancer {
  constructor(servers) {
    this.servers = servers;
    this.current = 0;
  }

  getNext() {
    const server = this.servers[this.current];
    this.current = (this.current + 1) % this.servers.length;
    return server;
  }
}

const lb = new RoundRobinBalancer([
  'http://server1:3000',
  'http://server2:3000',
  'http://server3:3000',
]);

// Request 1 -> server1, Request 2 -> server2, Request 3 -> server3, Request 4 -> server1...

Pros: Simple, even distribution when servers are identical. Cons: Ignores server load — a slow server gets the same traffic as a fast one.

2. Weighted Round Robin

Assigns weights to servers based on capacity.

class WeightedRoundRobinBalancer {
  constructor(servers) {
    // servers: [{ url: string, weight: number }]
    this.servers = servers;
    this.pool = [];
    this.current = 0;
    this._buildPool();
  }

  _buildPool() {
    for (const server of this.servers) {
      for (let i = 0; i < server.weight; i++) {
        this.pool.push(server.url);
      }
    }
  }

  getNext() {
    const server = this.pool[this.current];
    this.current = (this.current + 1) % this.pool.length;
    return server;
  }
}

const lb = new WeightedRoundRobinBalancer([
  { url: 'http://large-server:3000', weight: 5 },
  { url: 'http://small-server:3000', weight: 1 },
]);
// large-server gets 5x more traffic

3. Least Connections

Routes to the server with the fewest active connections.

class LeastConnectionsBalancer {
  constructor(servers) {
    this.servers = servers.map(url => ({
      url,
      activeConnections: 0,
    }));
  }

  getNext() {
    const server = this.servers.reduce((min, s) =>
      s.activeConnections < min.activeConnections ? s : min
    );
    server.activeConnections++;
    return server;
  }

  release(serverUrl) {
    const server = this.servers.find(s => s.url === serverUrl);
    if (server) server.activeConnections--;
  }
}

Best for: Requests with variable processing time (some fast, some slow).

4. IP Hash (Sticky Sessions)

Hashes the client IP to consistently route to the same server.

import crypto from 'node:crypto';

class IPHashBalancer {
  constructor(servers) {
    this.servers = servers;
  }

  getNext(clientIP) {
    const hash = crypto
      .createHash('md5')
      .update(clientIP)
      .digest('hex');
    const index = parseInt(hash.substring(0, 8), 16) % this.servers.length;
    return this.servers[index];
  }
}

Use case: Session affinity when sessions are stored in server memory (though you should prefer external session stores like Redis).

5. Consistent Hashing

Advanced form of hashing that minimizes redistribution when servers are added or removed.

class ConsistentHashRing {
  constructor(servers, virtualNodes = 100) {
    this.ring = new Map();
    this.sortedKeys = [];

    for (const server of servers) {
      for (let i = 0; i < virtualNodes; i++) {
        const hash = this._hash(`${server}:${i}`);
        this.ring.set(hash, server);
        this.sortedKeys.push(hash);
      }
    }
    this.sortedKeys.sort((a, b) => a - b);
  }

  _hash(key) {
    const h = crypto.createHash('md5').update(key).digest();
    return h.readUInt32BE(0);
  }

  getServer(key) {
    const hash = this._hash(key);
    for (const nodeHash of this.sortedKeys) {
      if (hash <= nodeHash) return this.ring.get(nodeHash);
    }
    return this.ring.get(this.sortedKeys[0]);
  }
}

Nginx as a Load Balancer for Node.js

The most common production setup for Node.js applications:

# /etc/nginx/conf.d/api.conf
upstream node_backend {
    least_conn;  # Use least connections algorithm

    server 127.0.0.1:3001 weight=3;
    server 127.0.0.1:3002 weight=3;
    server 127.0.0.1:3003 weight=1;

    keepalive 64;  # Connection pooling to backends
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://node_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_cache_bypass $http_upgrade;
    }
}

Health Checks

A load balancer must know which servers are healthy:

// Express health check endpoint
app.get('/health', async (req, res) => {
  try {
    // Check database connectivity
    await prisma.$queryRaw`SELECT 1`;
    // Check Redis connectivity
    await redis.ping();

    res.status(200).json({
      status: 'healthy',
      uptime: process.uptime(),
      memory: process.memoryUsage(),
      timestamp: new Date().toISOString(),
    });
  } catch (err) {
    res.status(503).json({
      status: 'unhealthy',
      error: err.message,
    });
  }
});

Nginx active health checks (requires Nginx Plus or use passive checks):

upstream node_backend {
    server 127.0.0.1:3001 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:3002 max_fails=3 fail_timeout=30s;
}

Session Persistence Strategies

When using load balancers, server-side sessions become problematic:

Strategy 1: Sticky Sessions (IP Hash)
  - Simple but uneven distribution
  - Server failure loses sessions

Strategy 2: Shared Session Store (Recommended)
  ┌─────────┐     ┌──────────┐     ┌─────────┐
  │ Server 1│────>│  Redis   │<────│Server 2 │
  └─────────┘     │ (sessions)│     └─────────┘
                  └──────────┘
  - Any server can handle any request
  - Redis failure is the risk (use Redis Cluster)

Strategy 3: Stateless (JWT)
  - No server-side session
  - Token contains all needed info
  - Largest token size but most scalable

// Shared session store with Redis
import session from 'express-session';
import RedisStore from 'connect-redis';
import Redis from 'ioredis';

const redisClient = new Redis({ host: 'redis-server', port: 6379 });

app.use(session({
  store: new RedisStore({ client: redisClient }),
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: { secure: true, maxAge: 86400000 },
}));

Common Pitfalls

  1. No health checks: Dead servers still receive traffic, causing errors.
  2. Round robin with uneven servers: A weak server becomes a bottleneck.
  3. Sticky sessions without failover: Server crash loses all sessions for those users.
  4. Ignoring connection draining: Removing a server without finishing in-flight requests causes errors.

Summary

Load balancing distributes traffic across multiple servers to improve throughput, reduce latency, and increase reliability. Choose the algorithm that matches your workload: round robin for uniform requests, least connections for variable workloads, IP hash for session affinity. In production, use Nginx or a cloud load balancer with health checks, connection pooling, and graceful draining.

Exercises

  1. Build a Round Robin Proxy

    Create a Node.js HTTP server that acts as a round-robin load balancer, forwarding requests to 3 backend Express servers running on different ports. Include health checks that remove unhealthy servers from the rotation.

    Hint

    Use node:http to create the proxy. Use http.request() to forward to backends. Periodically ping /health on each backend.

  2. Implement Consistent Hashing

    Build a consistent hash ring that maps user IDs to servers. Add and remove servers and verify that only a small percentage of keys are remapped.

    Hint

    Use virtual nodes (100+ per server) for even distribution. Compare key-to-server mapping before and after adding a server.

  3. Nginx Load Balancer Setup

    Configure Nginx to load balance across 3 Node.js instances using least_conn algorithm. Add health checks, connection keepalive, and test failover by stopping one instance.

    Hint

    Use Docker Compose to run Nginx + 3 Node.js containers. Set max_fails=2 fail_timeout=10s for passive health checks.

  4. Compare Algorithm Performance

    Simulate 10,000 requests across 5 servers with varying response times. Compare round robin, weighted round robin, and least connections. Measure average latency and server utilization.

    Hint

    Create mock servers with setTimeout for variable response times. Track per-server request counts and total completion time.

Common Problems & Solutions

1 Uneven load distribution across servers
Cause: Using round robin when servers have different capacities, or some requests are significantly more expensive than others.
Solution: Switch to weighted round robin (assign weights by server capacity) or least connections (automatically adapts to varying request costs).
2 Users losing sessions after load balancer deployment
Cause: Sessions stored in server memory are not accessible when the load balancer routes the next request to a different server.
Solution: Use a shared session store (Redis) or switch to stateless authentication (JWT). Avoid sticky sessions as a long-term solution.
3 Load balancer health checks passing but requests failing
Cause: Health check endpoint returns 200 but the application has a deeper issue (e.g., database connection pool exhausted, disk full).
Solution: Implement deep health checks that verify database connectivity, Redis availability, and disk space. Return 503 if any dependency is unhealthy.

app.get('/health', async (req, res) => {\n const dbOk = await checkDB();\n const redisOk = await checkRedis();\n const status = dbOk && redisOk ? 200 : 503;\n res.status(status).json({ db: dbOk, redis: redisOk });\n});
