Testing the API and Deployment Prep

Intuition

Testing is your safety net that catches regressions before users do, while deployment prep is about hardening your application for the real world. Together they bridge the gap between 'it works on my machine' and 'it works reliably in production.'

Testing API and Deployment Prep

Testing ensures your API works correctly before deployment. Flask provides a test client for sending HTTP requests without a running server. Preparing for deployment involves configuring WSGI servers, environment variables, and production settings.

import pytest
from app import create_app, db

@pytest.fixture
def app():
    app = create_app()
    app.config.update({
        'TESTING': True,
        'SQLALCHEMY_DATABASE_URI': 'sqlite:///:memory:'
    })
    with app.app_context():
        db.create_all()
        yield app
        db.drop_all()

@pytest.fixture
def client(app):
    return app.test_client()

@pytest.fixture
def auth_header(client):
    # Register and login to get a token
    client.post('/auth/register', json={
        'username': 'testuser', 'password': 'testpass123'
    })
    resp = client.post('/auth/login', json={
        'username': 'testuser', 'password': 'testpass123'
    })
    token = resp.get_json()['access_token']
    return {'Authorization': f'Bearer {token}'}

def test_create_post(client, auth_header):
    resp = client.post('/api/posts', json={
        'title': 'Test Post', 'body': 'Hello World'
    }, headers=auth_header)
    assert resp.status_code == 201
    assert resp.get_json()['title'] == 'Test Post'

def test_unauthorized_access(client):
    resp = client.post('/api/posts', json={'title': 'Test'})
    assert resp.status_code == 401

Key Takeaways: - Use app.test_client() for integration tests that exercise the full request/response cycle. - Test both happy paths and error cases: unauthorized access, validation errors, not found. - Deploy with Gunicorn (or uWSGI) behind Nginx; never use Flask's built-in server in production.

Exercises

  1. Comprehensive API Test Suite

    Write pytest tests covering: user registration, login, token refresh, create/read/update/delete posts, authorization checks (editing another user's post returns 403), and pagination parameters.

    Hint

    Use fixtures to set up test users and posts. Test edge cases like empty strings, missing fields, and invalid tokens.

  2. Docker Deployment

    Create a Dockerfile and docker-compose.yml for the blog API with PostgreSQL. Include environment variable configuration, database migration on startup, and a health check endpoint.

    Hint

    Use a docker-compose service for the database and a separate service for the Flask app. Run flask db upgrade in an entrypoint script before starting Gunicorn.

Common Problems & Solutions

1 Tests pass individually but fail when run together.
Cause: Tests share state through the database or module-level variables. Data from one test leaks into another.
Solution: Use a fresh database for each test with the pytest fixture pattern: create_all() in setup, drop_all() in teardown. Use the in-memory SQLite database for speed.
2 Gunicorn workers crash with 'out of memory' on large file uploads.
Cause: Gunicorn's default sync workers load the entire request body into memory. Large uploads consume all available memory.
Solution: Set MAX_CONTENT_LENGTH in Flask to limit upload size. Use Gunicorn's --timeout flag to handle slow uploads, and consider streaming uploads to disk for very large files.
