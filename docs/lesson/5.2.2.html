Sub-Dependencies and Dependency Trees

Intuition

Dependency trees let you build security and data layers like building blocks: token parsing feeds into user lookup, which feeds into permission checking. Each block is testable in isolation and reusable across your entire API.

Sub-Dependencies and Dependency Trees

Dependencies can depend on other dependencies, forming a tree that FastAPI resolves automatically. This enables composable, layered architectures where each layer handles one concern.

from fastapi import FastAPI, Depends, HTTPException, Header

app = FastAPI()

def get_token(authorization: str = Header(...)) -> str:
    if not authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="Invalid scheme")
    return authorization.split(" ")[1]

def get_current_user(token: str = Depends(get_token)) -> dict:
    users = {"abc123": {"id": 1, "name": "Alice", "role": "admin"}}
    user = users.get(token)
    if not user:
        raise HTTPException(status_code=401, detail="Invalid token")
    return user

def require_admin(user: dict = Depends(get_current_user)) -> dict:
    if user["role"] != "admin":
        raise HTTPException(status_code=403, detail="Admin required")
    return user

@app.get("/admin/dashboard")
async def admin_dashboard(admin: dict = Depends(require_admin)):
    return {"message": f"Welcome {admin['name']}", "role": admin["role"]}

Key Takeaways: - Dependencies can call Depends() on other dependencies, forming chains - FastAPI resolves the entire dependency tree before calling the endpoint - If a dependency appears multiple times in a tree, it is called once per request by default - Use use_cache=False in Depends() to force re-evaluation

Exercises

  1. Role-Based Access Control

    Build a dependency chain: get_token -> get_user -> require_role(role). Create a factory function require_role that returns a dependency checking if the user has the specified role. Use it to protect /admin and /editor routes.

    Hint

    Create def require_role(role: str) that returns an inner function taking user = Depends(get_user) and checking user.role == role. Use Depends(require_role("admin")) in the endpoint.

  2. Caching vs Non-Caching Dependencies

    Create a dependency that generates a random request ID. Use it twice in the same endpoint, once with default caching and once with use_cache=False. Print both IDs to demonstrate the difference.

    Hint

    Use Depends(get_request_id) and Depends(get_request_id, use_cache=False). The cached version returns the same value; the non-cached version generates a new one.

Common Problems & Solutions

1 Circular dependency error at startup
Cause: Two dependencies depend on each other directly or indirectly, creating an unresolvable cycle.
Solution: Restructure your dependencies to break the cycle. Extract shared logic into a common base dependency that both can depend on, rather than having them depend on each other.
2 Sub-dependency receiving unexpected None value
Cause: A parent dependency in the chain is not returning a value, or an exception in a sibling dependency is being silently caught.
Solution: Ensure every dependency in the chain has an explicit return statement. Add logging to each dependency to trace the resolution order and identify where None is introduced.
