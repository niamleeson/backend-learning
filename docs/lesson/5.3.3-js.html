tRPC with Express and NestJS

Intuition

Integrating tRPC with an existing Express or NestJS app is like adding a fast lane on an existing highway. The old lanes (REST endpoints) still work fine, but the new fast lane (tRPC) offers a smoother, type-checked ride. Over time, you can route more and more traffic to the fast lane.

tRPC with Express and NestJS

tRPC integrates with existing Express and NestJS applications, so you do not need to rewrite your entire API. You can mount tRPC alongside REST routes and migrate incrementally.

tRPC with Express

Basic Integration

import express from 'express';
import cors from 'cors';
import * as trpcExpress from '@trpc/server/adapters/express';
import { appRouter } from './routers';
import { createContext } from './context';

const app = express();

// Regular Express middleware
app.use(cors());
app.use(express.json());

// Regular REST routes (existing API)
app.get('/api/health', (req, res) => {
  res.json({ status: 'ok' });
});

app.get('/api/legacy/users', async (req, res) => {
  // Old REST endpoint still works
  res.json({ users: [] });
});

// Mount tRPC alongside REST
app.use(
  '/trpc',
  trpcExpress.createExpressMiddleware({
    router: appRouter,
    createContext,
    onError: ({ error, path, type }) => {
      console.error(`tRPC ${type} error on '${path}':`, error.message);
    },
  })
);

app.listen(4000, () => {
  console.log('Server running on port 4000');
  console.log('REST: http://localhost:4000/api/*');
  console.log('tRPC: http://localhost:4000/trpc/*');
});

Sharing Express Middleware with tRPC

You can reuse existing Express middleware (auth, rate limiting, etc.) in tRPC through context:

import rateLimit from 'express-rate-limit';

// Apply rate limiting to tRPC routes
const trpcLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 100,
  standardHeaders: true,
});

app.use('/trpc', trpcLimiter);
app.use(
  '/trpc',
  trpcExpress.createExpressMiddleware({ router: appRouter, createContext })
);

Using Express Request in Context

import { CreateExpressContextOptions } from '@trpc/server/adapters/express';

export function createContext({ req, res }: CreateExpressContextOptions) {
  // Access Express req/res in context
  return {
    req,
    res,
    prisma,
    userId: extractUserId(req),
    ip: req.ip,
    userAgent: req.headers['user-agent'],
  };
}

tRPC with NestJS

Integrating tRPC with NestJS requires a bit more setup since NestJS has its own DI system:

Option 1: Mount as Express Middleware

// trpc/trpc.module.ts
import { Module, NestModule, MiddlewareConsumer } from '@nestjs/common';
import * as trpcExpress from '@trpc/server/adapters/express';
import { appRouter } from './routers';
import { PrismaService } from '../prisma/prisma.service';

@Module({})
export class TrpcModule implements NestModule {
  constructor(private prisma: PrismaService) {}

  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(
        trpcExpress.createExpressMiddleware({
          router: appRouter,
          createContext: ({ req }) => ({
            prisma: this.prisma,
            userId: (req as any).userId || null,
          }),
        })
      )
      .forRoutes('/trpc');
  }
}

Option 2: NestJS-tRPC Integration

// Using @nest-rpc/trpc or custom integration
// trpc/trpc.service.ts
import { Injectable } from '@nestjs/common';
import { initTRPC, TRPCError } from '@trpc/server';
import { z } from 'zod';
import { PrismaService } from '../prisma/prisma.service';
import { UserService } from '../user/user.service';

interface Context {
  prisma: PrismaService;
  userId: string | null;
}

@Injectable()
export class TrpcService {
  private t = initTRPC.context<Context>().create();

  constructor(
    private prisma: PrismaService,
    private userService: UserService,
  ) {}

  get appRouter() {
    return this.t.router({
      user: this.t.router({
        getById: this.t.procedure
          .input(z.object({ id: z.string() }))
          .query(async ({ input }) => {
            return this.userService.findById(input.id);
          }),

        list: this.t.procedure
          .input(z.object({ limit: z.number().default(20) }))
          .query(async ({ input }) => {
            return this.userService.findAll(input.limit);
          }),
      }),
    });
  }

  createContext(req: any): Context {
    return {
      prisma: this.prisma,
      userId: req.userId || null,
    };
  }
}

Full NestJS Controller Approach

// trpc/trpc.controller.ts
import { All, Controller, Req, Res } from '@nestjs/common';
import { Request, Response } from 'express';
import { resolveHTTPResponse } from '@trpc/server/http';
import { TrpcService } from './trpc.service';

@Controller('trpc')
export class TrpcController {
  constructor(private trpcService: TrpcService) {}

  @All('*')
  async handler(@Req() req: Request, @Res() res: Response) {
    const path = req.path.replace('/trpc/', '');
    const context = this.trpcService.createContext(req);

    const response = await resolveHTTPResponse({
      router: this.trpcService.appRouter,
      req: {
        method: req.method,
        headers: req.headers,
        query: req.query,
        body: req.body,
      },
      path,
      createContext: () => context,
    });

    if (response.status) res.status(response.status);
    for (const [key, value] of Object.entries(response.headers || {})) {
      if (typeof value !== 'undefined') {
        res.setHeader(key, value);
      }
    }
    res.json(JSON.parse(response.body || '{}'));
  }
}

Incremental Migration Strategy

You do not have to replace REST with tRPC all at once:

Phase 1: Add tRPC alongside existing REST API
+-------------------------------------------+
|  Express / NestJS App                     |
|  /api/*     -> REST controllers (existing)|
|  /trpc/*    -> tRPC router (new features) |
+-------------------------------------------+

Phase 2: New features use tRPC, migrate old ones gradually
+-------------------------------------------+
|  /api/legacy/*  -> Old REST endpoints     |
|  /trpc/*        -> tRPC (growing)         |
+-------------------------------------------+

Phase 3: Full tRPC (optional)
+-------------------------------------------+
|  /trpc/*        -> All procedures         |
+-------------------------------------------+

// Migration helper: wrap existing service methods
class UserServiceAdapter {
  constructor(private userService: UserService) {}

  createTrpcRouter(t: ReturnType<typeof initTRPC.create>) {
    return t.router({
      getById: t.procedure
        .input(z.object({ id: z.string() }))
        .query(({ input }) => this.userService.findById(input.id)),

      create: t.procedure
        .input(createUserSchema)
        .mutation(({ input }) => this.userService.create(input)),

      update: t.procedure
        .input(updateUserSchema)
        .mutation(({ input }) => this.userService.update(input.id, input)),

      delete: t.procedure
        .input(z.object({ id: z.string() }))
        .mutation(({ input }) => this.userService.delete(input.id)),
    });
  }
}

import { appRouter } from './routers';

const createCaller = createCallerFactory(appRouter);

describe('User Router', () => {

  beforeEach(() => {
    caller = createCaller({
      prisma: prismaMock,
      userId: 'test-user-id',

  });

  it('should get a user by id', async () => {

    expect(result).toEqual(mockUser);
  });

  it('should throw NOT_FOUND for missing user', async () => {
    prismaMock.user.findUnique.mockResolvedValue(null);
    await expect(
      caller.user.getById({ id: 'nonexistent' })
    ).rejects.toThrow('NOT_FOUND');

  it('should validate input', async () => {
    await expect(
      caller.user.create({ email: 'invalid', name: '', password: '123' })
    ).rejects.toThrow(); // Zod validation error

});

Exercises

  1. Add tRPC to an Existing Express App

    Take an existing Express REST API and mount tRPC alongside it at /trpc. Migrate one resource (e.g., users CRUD) to tRPC while keeping the old REST endpoints functional. Share the same database client between both.

    Hint

    Mount tRPC with app.use('/trpc', trpcExpress.createExpressMiddleware(...)). Share the Prisma client via context.

  2. Test tRPC Procedures with createCallerFactory

    Write unit tests for a tRPC router using createCallerFactory. Test queries, mutations, error cases (NOT_FOUND, UNAUTHORIZED), and input validation failures. Mock the Prisma client.

    Hint

    Create a caller with a mock context and call procedures directly: caller.user.getById({ id: '1' }).

  3. NestJS + tRPC Hybrid API

    In a NestJS application, add a TrpcModule that mounts tRPC as middleware. Inject NestJS services (UserService, PostService) into the tRPC router. Ensure that NestJS guards work on the tRPC route.

    Hint

    Implement NestModule.configure() and use consumer.apply(trpcExpress.createExpressMiddleware(...)).forRoutes('/trpc').

Common Problems & Solutions

1 NestJS DI services are undefined in tRPC context
Cause: The tRPC router is created outside the NestJS DI container, so services are not injected.
Solution: Create the tRPC router inside a NestJS provider that receives injected services. Use the TrpcService pattern shown above where the router is built in a method that has access to injected services.
2 CORS errors when calling tRPC from a different frontend domain
Cause: CORS middleware is not configured for the /trpc routes.
Solution: Apply cors() middleware before the tRPC middleware: app.use('/trpc', cors({ origin: 'http://localhost:3000' }), trpcMiddleware).
3 tRPC batch requests timeout with large payloads
Cause: Multiple batched requests exceed the Express body size limit or server timeout.
Solution: Increase body limit: express.json({ limit: '10mb' }). Configure tRPC maxBatchSize: httpBatchLink({ maxURLLength: 2083 }). Consider splitting very large batches.
