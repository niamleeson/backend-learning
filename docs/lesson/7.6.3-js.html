Chaos Engineering Principles

Intuition

Chaos engineering is like a fire drill for your servers. You do not wait for a real fire to find out that the emergency exit is blocked and the fire extinguisher is expired. By running controlled 'fires' regularly, you discover weaknesses while everyone is awake, alert, and ready to respond -- not at 3 AM when the real outage hits.

Chaos Engineering Principles

What Is Chaos Engineering?

Chaos engineering is the discipline of experimenting on a distributed system to build confidence in the system's ability to withstand turbulent conditions in production. Instead of waiting for failures to happen at 3 AM on a Saturday, you deliberately inject failures during business hours with the team ready to respond.

The core idea is counterintuitive: break things on purpose so they do not break by surprise.

Why Chaos Engineering Matters for Node.js Services

Node.js applications are particularly vulnerable to certain failure modes:

  • Single-threaded event loop blocking -- A CPU-intensive operation freezes all request handling
  • Unhandled promise rejections -- A missing catch statement crashes the process
  • Memory leaks -- Closures, event listeners, or caches that grow unbounded
  • Connection pool exhaustion -- Database or Redis connections not being released
  • Cascading timeouts -- One slow upstream service causing all downstream callers to queue up

Chaos engineering lets you discover these weaknesses before your users do.

Step 2: Form a Hypothesis

A chaos experiment hypothesis looks like:

"When the database connection latency increases by 200ms, the system will continue serving 99% of requests within 2 seconds because our connection pool can absorb temporary latency spikes."

"When the payment service returns 500 errors for 30 seconds, the checkout endpoint will return cached pricing or a 'try again later' message, and the error rate for non-checkout endpoints will not increase."

Chaos Experiments for Node.js

Experiment 1: Simulating Downstream Service Failure

Create a chaos middleware that can inject failures into outgoing HTTP calls:

const axios = require('axios');

class ChaosInterceptor {
  constructor() {
    this.experiments = new Map();
  }

  // Register an experiment
  addExperiment(name, config) {
    this.experiments.set(name, {
      enabled: false,
      ...config,
    });
  }

  // Enable/disable experiments via API
  enable(name) {
    const exp = this.experiments.get(name);
    if (exp) exp.enabled = true;
  }

  disable(name) {
    const exp = this.experiments.get(name);
    if (exp) exp.enabled = false;
  }

  // Apply to axios instance
  applyToAxios(axiosInstance) {
    axiosInstance.interceptors.request.use(async (config) => {
      for (const [name, exp] of this.experiments) {
        if (!exp.enabled) continue;
        if (exp.urlPattern && !config.url.match(exp.urlPattern)) continue;

        // Inject latency
        if (exp.latencyMs) {
          await new Promise((r) => setTimeout(r, exp.latencyMs));
        }

        // Inject failure (based on probability)
        if (exp.failureRate && Math.random() < exp.failureRate) {
          throw {
            response: {
              status: exp.statusCode || 500,
              data: { error: `Chaos experiment: ${name}` },
            },
            config,
          };
        }
      }
      return config;
    });
  }
}

// Setup
const chaos = new ChaosInterceptor();

chaos.addExperiment('payment-latency', {
  urlPattern: /payment-service/,
  latencyMs: 3000,
  failureRate: 0,
});

chaos.addExperiment('payment-failure', {
  urlPattern: /payment-service/,
  latencyMs: 0,
  failureRate: 0.5,  // 50% of requests fail
  statusCode: 503,
});

chaos.addExperiment('database-slowdown', {
  urlPattern: /database/,
  latencyMs: 500,
  failureRate: 0.1,
});

const apiClient = axios.create();
chaos.applyToAxios(apiClient);

// Control plane: enable/disable experiments
// ONLY accessible in staging/testing environments
if (process.env.CHAOS_ENABLED === 'true') {
  app.post('/chaos/:experiment/enable', (req, res) => {
    chaos.enable(req.params.experiment);
    res.json({ status: 'enabled', experiment: req.params.experiment });
  });

  app.post('/chaos/:experiment/disable', (req, res) => {
    chaos.disable(req.params.experiment);
    res.json({ status: 'disabled', experiment: req.params.experiment });
  });

  app.get('/chaos/status', (req, res) => {
    const status = {};
    for (const [name, exp] of chaos.experiments) {
      status[name] = { enabled: exp.enabled };
    }
    res.json(status);
  });
}

Experiment 2: Memory Pressure

Test how your application behaves under memory pressure:

// Chaos experiment: gradual memory consumption
class MemoryPressureExperiment {
  constructor() {
    this.buffers = [];
    this.running = false;
  }

  start({ targetMB = 512, rampUpSeconds = 30 }) {
    this.running = true;
    const chunkSize = 1024 * 1024; // 1MB
    const intervalMs = (rampUpSeconds * 1000) / targetMB;

    console.log(
      `[Chaos] Starting memory pressure: ${targetMB}MB over ${rampUpSeconds}s`
    );

    const interval = setInterval(() => {
      if (!this.running || this.buffers.length >= targetMB) {
        clearInterval(interval);
        return;
      }
      this.buffers.push(Buffer.alloc(chunkSize));
      const usedMB = process.memoryUsage().heapUsed / 1024 / 1024;
      console.log(`[Chaos] Memory used: ${usedMB.toFixed(0)}MB`);
    }, intervalMs);
  }

  stop() {
    this.running = false;
    this.buffers = [];
    if (global.gc) {
      global.gc(); // Call with --expose-gc flag
    }
    console.log('[Chaos] Memory pressure released');
  }
}

Experiment 3: Event Loop Blocking

Test what happens when the event loop is blocked (simulating a CPU-intensive operation or synchronous I/O):

class EventLoopBlockExperiment {
  constructor() {
    this.blocking = false;
  }

  start({ blockDurationMs = 100, intervalMs = 1000 }) {
    this.blocking = true;
    console.log(
      `[Chaos] Blocking event loop for ${blockDurationMs}ms every ${intervalMs}ms`
    );

    const blockLoop = () => {
      if (!this.blocking) return;

      // Synchronously block the event loop
      const start = Date.now();
      while (Date.now() - start < blockDurationMs) {
        // Busy wait
      }

      setTimeout(blockLoop, intervalMs);
    };

    setTimeout(blockLoop, intervalMs);
  }

  stop() {
    this.blocking = false;
    console.log('[Chaos] Event loop blocking stopped');
  }
}

// Monitor event loop lag to detect blocking
const monitorEventLoopLag = () => {
  let lastCheck = Date.now();

  setInterval(() => {
    const now = Date.now();
    const lag = now - lastCheck - 1000; // Expected 1000ms interval
    lastCheck = now;

    if (lag > 100) {
      console.warn(`[Monitor] Event loop lag: ${lag}ms`);
    }
  }, 1000);
};

Experiment 4: Connection Pool Exhaustion

// Simulate connection pool exhaustion by holding connections open
class ConnectionPoolExperiment {
  constructor(prisma) {
    this.prisma = prisma;
    this.heldConnections = [];
  }

  async start({ count = 10 }) {
    console.log(`[Chaos] Holding ${count} database connections open`);

    for (let i = 0; i < count; i++) {
      // Start a transaction and do not commit it
      const promise = this.prisma.$transaction(async (tx) => {
        await tx.$queryRaw`SELECT pg_sleep(300)`; // Hold for 5 min
      }).catch(() => {}); // Ignore errors when we abort

      this.heldConnections.push(promise);
    }
  }

  stop() {
    console.log('[Chaos] Releasing held connections');
    // The connections will eventually time out
    this.heldConnections = [];
  }
}

Infrastructure-Level Chaos with Pumba (Docker)

For containerized Node.js applications, use Pumba to inject failures at the container/network level:

# Kill a container
pumba kill --signal SIGTERM myapp-container

# Add network latency (500ms +/- 100ms to all traffic)
pumba netem --duration 60s delay --time 500 --jitter 100 myapp-container

# Packet loss (10% of packets dropped)
pumba netem --duration 60s loss --percent 10 myapp-container

# Limit bandwidth to 1Mbps
pumba netem --duration 60s rate --rate 1mbit myapp-container

Using Toxiproxy for Network Chaos

Toxiproxy sits between your app and its dependencies, letting you inject network-level faults:

const Toxiproxy = require('toxiproxy-node-client');

const toxiproxy = new Toxiproxy('http://localhost:8474');

async function setupProxies() {
  // Create a proxy for PostgreSQL
  const dbProxy = await toxiproxy.createProxy({
    name: 'postgres',
    listen: '0.0.0.0:5433',
    upstream: 'db:5432',
  });

  // Create a proxy for Redis
  const redisProxy = await toxiproxy.createProxy({
    name: 'redis',
    listen: '0.0.0.0:6380',
    upstream: 'redis:6379',
  });

  return { dbProxy, redisProxy };
}

async function runExperiment() {
  const { dbProxy, redisProxy } = await setupProxies();

  // Experiment: Add 1 second of latency to database calls
  console.log('Adding 1s latency to database...');
  await dbProxy.addToxic({
    type: 'latency',
    attributes: { latency: 1000, jitter: 200 },
  });

  // Observe: check p99 latency, error rates
  await new Promise((r) => setTimeout(r, 30000)); // Run for 30s

  // Experiment: Completely cut Redis connection
  console.log('Cutting Redis connection...');
  await redisProxy.addToxic({
    type: 'timeout',
    attributes: { timeout: 0 },  // No data gets through
  });

  // Observe: does the app degrade gracefully?
  await new Promise((r) => setTimeout(r, 30000));

  // Cleanup
  await dbProxy.removeToxics();
  await redisProxy.removeToxics();
  console.log('Experiment complete.');
}

Chaos Experiment Automation

Automate chaos experiments as part of your CI/CD pipeline (in staging):

// chaos-tests/payment-failure.test.js
const axios = require('axios');

describe('Chaos: Payment Service Failure', () => {
  const APP_URL = process.env.APP_URL || 'http://localhost:3000';

  beforeAll(async () => {
    // Enable the chaos experiment
    await axios.post(`${APP_URL}/chaos/payment-failure/enable`);
    // Wait for circuit breaker to detect failures
    await new Promise((r) => setTimeout(r, 5000));
  });

  afterAll(async () => {
    await axios.post(`${APP_URL}/chaos/payment-failure/disable`);
  });

  test('non-payment endpoints remain healthy', async () => {
    const response = await axios.get(`${APP_URL}/products/1`);
    expect(response.status).toBe(200);
  });

  test('checkout returns graceful error, not 500', async () => {
    try {
      await axios.post(`${APP_URL}/checkout`, { productId: 1 });
    } catch (err) {
      // Should get 503, not 500
      expect(err.response.status).toBe(503);
      expect(err.response.data.error).toContain('unavailable');
      expect(err.response.data.retryAfter).toBeDefined();
    }
  });

  test('health check reports degraded state', async () => {
    const response = await axios.get(`${APP_URL}/readyz`);
    // System should still be ready (payment is non-critical)
    expect(response.status).toBe(200);
    expect(response.data.checks.circuitBreakers.paymentService).toBe('open');
  });
});

Principles and Best Practices

The Five Principles of Chaos Engineering

  1. Build a hypothesis around steady-state behavior -- Define what "normal" looks like with metrics before you break anything.

  2. Vary real-world events -- Inject failures that actually happen: network partitions, disk full, high CPU, dependency failures, clock skew.

  3. Run experiments in production -- Staging environments cannot replicate production's traffic patterns, data volumes, and edge cases. Start small in production with blast radius controls.

  4. Automate experiments to run continuously -- Manual one-off experiments are useful for discovery but automated experiments catch regressions.

  5. Minimize blast radius -- Start with a single server or a small percentage of traffic. Have a kill switch to abort immediately.

Safety Controls

class ExperimentRunner {
  constructor({ metricsClient, abortThreshold }) {
    this.metricsClient = metricsClient;
    this.abortThreshold = abortThreshold;
    this.running = false;
  }

  async run(experiment) {
    this.running = true;
    console.log(`Starting experiment: ${experiment.name}`);

    // Take a baseline measurement
    const baseline = await this.metricsClient.getErrorRate();
    console.log(`Baseline error rate: ${baseline}%`);

    // Start the experiment
    await experiment.start();

    // Monitor during experiment
    const monitorInterval = setInterval(async () => {
      const currentErrorRate = await this.metricsClient.getErrorRate();
      console.log(`Current error rate: ${currentErrorRate}%`);

      if (currentErrorRate > this.abortThreshold) {
        console.error(
          `ABORT: Error rate ${currentErrorRate}% exceeds threshold ${this.abortThreshold}%`
        );
        clearInterval(monitorInterval);
        await experiment.stop();
        this.running = false;
      }
    }, 5000);

    // Auto-stop after duration
    setTimeout(async () => {
      if (this.running) {
        clearInterval(monitorInterval);
        await experiment.stop();
        this.running = false;
        console.log(`Experiment ${experiment.name} completed`);
      }
    }, experiment.durationMs);
  }
}

const runner = new ExperimentRunner({
  metricsClient: prometheusClient,
  abortThreshold: 5, // Abort if error rate exceeds 5%
});

Game Days

A Game Day is a structured event where a team runs chaos experiments together. The process:

  1. Prepare -- Choose experiments, set up monitoring dashboards, define abort criteria, notify stakeholders.
  2. Execute -- Run experiments one at a time. Everyone watches the dashboards.
  3. Observe -- Does the system behave as expected? Are alerts firing? Do runbooks work?
  4. Debrief -- Document what happened, what broke, what worked. Create action items.

Game days build team confidence, improve runbooks, and uncover assumptions that only surface under failure.

Chaos engineering is not about breaking things for fun. It is a disciplined practice that builds justified confidence in your system's resilience. Start small, measure everything, and always have a way to stop.

Exercises

  1. Build a Chaos HTTP Interceptor

    Create a configurable axios interceptor that can inject three types of chaos: latency (configurable delay), failure (configurable error rate and status code), and corruption (modify response data randomly). Build a REST API to enable/disable each chaos type at runtime. Only activate the interceptor when CHAOS_ENABLED=true. Test it against a simple Express app and observe how circuit breakers react.

    Hint

    Use axios.interceptors.request.use for latency and failure injection, and axios.interceptors.response.use for response corruption. Store experiment configs in a Map and expose POST endpoints to toggle them.

  2. Chaos Experiment with Toxiproxy

    Set up a Docker Compose environment with your Express app, PostgreSQL, and Redis all routing through Toxiproxy. Write a test suite that: (1) adds 2s latency to PostgreSQL and verifies the app still responds, (2) kills the Redis connection and verifies the app degrades gracefully, (3) adds 30% packet loss to PostgreSQL and measures the increase in p99 latency.

    Hint

    Use the toxiproxy-node-client package to programmatically add and remove toxics. Use the Toxiproxy REST API on port 8474. Create proxies that match your database and Redis ports.

  3. Event Loop Lag Monitor and Experiment

    Build an event loop lag monitoring module using setTimeout drift detection and the perf_hooks monitorEventLoopDelay API. Then create a chaos experiment that intentionally blocks the event loop for configurable durations. Run the experiment and verify: (1) the monitor detects the lag, (2) health checks report degraded state, (3) the application recovers after the experiment ends.

    Hint

    Use perf_hooks.monitorEventLoopDelay() to get a histogram of event loop delays. Compare with a manual approach using setInterval and checking actual vs expected elapsed time. Expose lag metrics on your /metrics endpoint.

  4. Automated Chaos Test Suite

    Write a Jest test suite that runs against your staging environment and performs three chaos experiments: downstream service failure, high latency, and connection pool exhaustion. Each test should verify the application maintains its SLO (p99 < 2s, error rate < 5%). Include setup/teardown to enable/disable experiments and a global abort mechanism if error rate exceeds a safety threshold.

    Hint

    Use beforeAll to enable chaos and afterAll to disable it. Poll the /metrics endpoint during the test to check error rates. Implement a custom Jest reporter or use setInterval with expect() assertions.

Common Problems & Solutions

1 Chaos experiment caused a real production outage
Cause: The blast radius was not limited. The experiment affected all instances/all traffic instead of a small subset. There was no kill switch or the kill switch was too slow.
Solution: Always start with the smallest possible blast radius: one pod, one percent of traffic, one region. Implement an instant kill switch (a feature flag that disables all experiments). Set automatic abort criteria based on metrics thresholds (e.g., abort if error rate > 5%). Never run chaos experiments without monitoring dashboards open and team members on standby.
2 Chaos experiments pass in staging but failures still happen in production
Cause: Staging does not replicate production conditions: lower traffic volume, different data sizes, no real user behavior patterns, different infrastructure configuration, missing edge cases that only appear at scale.
Solution: Gradually move chaos experiments to production with strict blast radius controls. Use traffic mirroring to test with production-like traffic in staging. Ensure staging infrastructure matches production as closely as possible (same database version, same connection pool sizes, same timeouts). Focus on experiments that test the specific differences between staging and production.
3 Team resistance to chaos engineering -- 'Why would we break things on purpose?'
Cause: Lack of understanding of the purpose, fear of causing outages, no executive sponsorship, or no clear process for running experiments safely.
Solution: Start with non-production environments to build confidence. Run a Game Day with the team to demonstrate value. Show examples of real outages that chaos engineering would have prevented. Establish clear safety protocols (blast radius, kill switch, abort criteria). Get buy-in from leadership by framing chaos engineering as risk reduction, not recklessness.
