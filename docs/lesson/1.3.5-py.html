Composition Over Inheritance

Intuition

Composition is building with LEGO bricks: you assemble complex objects from simpler, interchangeable pieces. Inheritance is more like biological evolution, sometimes you inherit traits you didn't want.

Composition models "has-a" relationships by embedding objects inside other objects, offering greater flexibility than inheritance's "is-a" approach.

class Engine:
    def __init__(self, horsepower: int):
        self.horsepower = horsepower
    def start(self) -> str:
        return f"Engine ({self.horsepower}hp) started"

class GPS:
    def navigate(self, destination: str) -> str:
        return f"Navigating to {destination}"

class Car:
    def __init__(self, model: str, hp: int):
        self.model = model
        self.engine = Engine(hp)       # has-a Engine
        self.gps = GPS()               # has-a GPS

    def start(self) -> str:
        return f"{self.model}: {self.engine.start()}"

    def drive_to(self, dest: str) -> str:
        return f"{self.start()} | {self.gps.navigate(dest)}"

car = Car("Tesla", 350)
print(car.drive_to("San Francisco"))
  • Composition is more flexible: swap components at runtime
  • Avoids the fragile base class problem of deep inheritance
  • Use inheritance for true "is-a" relationships (Dog is an Animal)
  • Use composition for "has-a" or "uses-a" relationships (Car has an Engine)
  • Design principle: favor composition over inheritance

Exercises

  1. Notification System

    Create EmailSender and SMSSender classes. Then create a NotificationService that uses composition to hold a list of senders and sends via all of them.

    Hint

    Store senders as self.senders = [EmailSender(), SMSSender()] and iterate in notify().

  2. Refactor Inheritance to Composition

    Given a LoggingHTTPClient(HTTPClient) that inherits from HTTPClient, refactor it to use composition instead, wrapping an HTTPClient instance.

    Hint

    Store self._client = HTTPClient() and delegate calls like self._client.get(url) after logging.

Common Problems & Solutions

1 Too many delegation methods (boilerplate)
Cause: Composition requires manually forwarding methods to the contained object.
Solution: Use `__getattr__` to automatically delegate attribute access, or use a mixin.
2 Tight coupling between composed objects
Cause: The container directly instantiates specific classes instead of accepting interfaces.
Solution: Use dependency injection: pass components as constructor arguments instead of creating them internally.
