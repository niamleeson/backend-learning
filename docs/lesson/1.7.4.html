pathlib: Modern Path Handling

Intuition

pathlib treats paths as objects instead of strings. This means you get methods, properties, and operators instead of scattering os.path calls throughout your code.

pathlib provides an object-oriented interface for filesystem paths, replacing os.path string manipulation with method calls and operator overloading.

from pathlib import Path

# Creating paths
base = Path("/Users/dev/project")
config = base / "config" / "settings.json"  # / operator joins

print(config.name)       # settings.json
print(config.stem)       # settings
print(config.suffix)     # .json
print(config.parent)     # /Users/dev/project/config

# Common operations
if config.exists():
    text = config.read_text(encoding="utf-8")

# Glob for file discovery
for py_file in base.rglob("*.py"):
    print(py_file)

# Create directories
(base / "output").mkdir(parents=True, exist_ok=True)
  • The / operator joins path segments — much cleaner than os.path.join().
  • .read_text() and .write_text() are shortcuts that handle open/close automatically.
  • .rglob(pattern) recursively searches subdirectories; .glob() searches one level.
  • Path objects work with open(), shutil, and most stdlib functions expecting paths.

Common Problems & Solutions

1 Mixing `os.path` strings with `pathlib.Path` objects causes confusion.
Cause: Some older APIs expect strings, not Path objects.
Solution: Wrap strings with `Path()` early. Use `str(path)` if an API requires a string.
2 `.mkdir()` raises `FileExistsError` when the directory already exists.
Cause: Default behavior is to fail if the directory exists.
Solution: Pass `exist_ok=True` and `parents=True` to create parent directories and ignore existing ones.
