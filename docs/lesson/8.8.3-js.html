Idempotency and Distributed Transactions

Intuition

Idempotency is like an elevator button. No matter how many times you press 'Floor 5', you only go to the 5th floor once. The elevator remembers the request and ignores duplicates. The Outbox pattern is like writing a letter and putting it in your outbox tray on your desk. A mail clerk periodically picks up letters and sends them. If the clerk is sick, the letters wait safely in the tray until the clerk returns — nothing is lost.

Idempotency and Distributed Transactions

Introduction

In distributed systems, networks are unreliable, services crash, and messages get duplicated. Idempotency ensures that repeating an operation produces the same result as doing it once. Distributed transactions coordinate operations across multiple services to maintain data consistency. These concepts are essential for building reliable systems that handle failures gracefully.

Idempotency

Why Idempotency Matters

Scenario: Payment processing

Without idempotency:
  Client ──> Payment Service: "Charge $100"
  Payment Service: Charges $100, response lost in network
  Client (timeout): "Charge $100" (retry)
  Payment Service: Charges $100 AGAIN
  Result: Customer charged $200!

With idempotency:
  Client ──> Payment Service: "Charge $100, key=pay-ord-123"
  Payment Service: Charges $100, stores key, response lost
  Client (timeout): "Charge $100, key=pay-ord-123" (retry)
  Payment Service: Key exists! Returns cached result
  Result: Customer charged $100 (correct)

Implementing Idempotency Keys

import Redis from 'ioredis';

const redis = new Redis();

class IdempotencyService {
  // Check and lock idempotency key
  async checkAndLock(key, ttlSeconds = 86400) {
    // Atomic: set if not exists with expiry
    const result = await redis.set(
      `idempotent:${key}`,
      JSON.stringify({ status: 'processing', startedAt: Date.now() }),
      'NX',  // Only if not exists
      'EX',  // With expiry
      ttlSeconds
    );

    if (!result) {
      // Key exists — check if completed
      const existing = await redis.get(`idempotent:${key}`);
      const parsed = JSON.parse(existing);

      if (parsed.status === 'completed') {
        return { duplicate: true, result: parsed.result };
      }

      if (parsed.status === 'processing') {
        // Still processing — could be a crash or slow operation
        const age = Date.now() - parsed.startedAt;
        if (age > 30000) {
          // Stale lock (>30s) — allow retry
          await redis.del(`idempotent:${key}`);
          return this.checkAndLock(key, ttlSeconds);
        }
        return { duplicate: true, processing: true };
      }
    }

    return { duplicate: false };
  }

  async complete(key, result, ttlSeconds = 86400) {
    await redis.setex(
      `idempotent:${key}`,
      ttlSeconds,
      JSON.stringify({ status: 'completed', result, completedAt: Date.now() })
    );
  }

  async fail(key) {
    // Remove lock so operation can be retried
    await redis.del(`idempotent:${key}`);
  }
}

// Express middleware
function idempotent(keyExtractor) {
  const service = new IdempotencyService();

  return async (req, res, next) => {
    const key = keyExtractor(req);
    if (!key) return next();

    const check = await service.checkAndLock(key);

    if (check.duplicate) {
      if (check.processing) {
        return res.status(409).json({ error: 'Request is being processed' });
      }
      return res.status(200).json(check.result);
    }

    // Store service on request for the handler to use
    req.idempotencyKey = key;
    req.idempotencyService = service;

    // Intercept response to store result
    const originalJson = res.json.bind(res);
    res.json = async (body) => {
      if (res.statusCode >= 200 && res.statusCode < 300) {
        await service.complete(key, body);
      } else {
        await service.fail(key);
      }
      return originalJson(body);
    };

    next();
  };
}

// Usage
app.post('/api/payments',
  idempotent(req => req.headers['idempotency-key']),
  async (req, res) => {
    const charge = await stripe.charges.create({
      amount: req.body.amount,
      currency: 'usd',
      customer: req.body.customerId,
    });
    res.json({ chargeId: charge.id, amount: charge.amount });
  }
);

Database-Level Idempotency

// Use unique constraints for natural idempotency
async function processPayment(orderId, amount) {
  try {
    // Unique constraint on order_id prevents double processing
    const payment = await prisma.payment.create({
      data: {
        orderId,     // UNIQUE constraint
        amount,
        status: 'COMPLETED',
      },
    });
    return { success: true, payment };
  } catch (err) {
    if (err.code === 'P2002') {
      // Unique constraint violation — already processed
      const existing = await prisma.payment.findUnique({
        where: { orderId },
      });
      return { success: true, payment: existing, duplicate: true };
    }
    throw err;
  }
}

Distributed Transaction Patterns

Two-Phase Commit (2PC)

Phase 1: Prepare
  Coordinator ──> "Can you commit?" ──> Service A: "Yes"
  Coordinator ──> "Can you commit?" ──> Service B: "Yes"

Phase 2: Commit
  Coordinator ──> "Commit!" ──> Service A: Committed
  Coordinator ──> "Commit!" ──> Service B: Committed

If any service says "No" in Phase 1:
  Coordinator ──> "Rollback!" ──> All services

2PC is rarely used in microservices due to: - Blocking: all participants lock resources during the protocol - Single point of failure: coordinator crash leaves participants hanging - Performance: two round trips minimum

Outbox Pattern

Ensures events are published reliably alongside database writes:

// The outbox pattern: write event to DB in same transaction as data
class OrderService {
  async createOrder(data) {
    return prisma.$transaction(async (tx) => {
      // 1. Create the order
      const order = await tx.order.create({ data });

      // 2. Write event to outbox table (same transaction!)
      await tx.outboxEvent.create({
        data: {
          aggregateId: order.id,
          aggregateType: 'Order',
          eventType: 'OrderCreated',
          payload: JSON.stringify({
            orderId: order.id,
            userId: data.userId,
            items: data.items,
            total: order.total,
          }),
          published: false,
        },
      });

      return order;
    });
    // Both writes succeed or both fail (ACID)
  }
}

// Outbox publisher: polls for unpublished events
class OutboxPublisher {
  constructor(prisma, kafkaProducer) {
    this.prisma = prisma;
    this.producer = kafkaProducer;
  }

  async publishPending() {
    const events = await this.prisma.outboxEvent.findMany({
      where: { published: false },
      orderBy: { createdAt: 'asc' },
      take: 100,
    });

    for (const event of events) {
      try {
        await this.producer.send({
          topic: `${event.aggregateType.toLowerCase()}-events`,
          messages: [{
            key: event.aggregateId,
            value: event.payload,
            headers: { 'event-type': event.eventType },
          }],
        });

        await this.prisma.outboxEvent.update({
          where: { id: event.id },
          data: { published: true, publishedAt: new Date() },
        });
      } catch (err) {
        console.error(`Failed to publish event ${event.id}:`, err);
        break; // Preserve ordering
      }
    }
  }

  start(intervalMs = 1000) {
    setInterval(() => this.publishPending(), intervalMs);
  }
}

Change Data Capture (CDC)

┌──────────┐    ┌──────────┐    ┌───────────┐    ┌──────────┐
│   App    │───>│PostgreSQL│───>│ Debezium  │───>│  Kafka   │
│          │    │          │    │   (CDC)   │    │          │
└──────────┘    └──────────┘    └───────────┘    └──────────┘
                  WAL logs         Reads WAL      Publishes
                                   changes        events

Benefits:
  - No code changes in the app
  - Captures ALL changes (even direct DB modifications)
  - Very low latency
  - No outbox table needed

HTTP Idempotency Semantics

┌──────────────────────────────────────────────────┐
│ Method  │ Idempotent? │ Safe? │ Notes            │
├─────────┼─────────────┼───────┼──────────────────┤
│ GET     │ Yes         │ Yes   │ No side effects  │
│ PUT     │ Yes         │ No    │ Replace resource │
│ DELETE  │ Yes         │ No    │ Delete resource  │
│ HEAD    │ Yes         │ Yes   │ Like GET no body │
│ POST    │ NO          │ No    │ Create resource  │
│ PATCH   │ NO          │ No    │ Partial update   │
└──────────────────────────────────────────────────┘

POST and PATCH need explicit idempotency keys!

Common Pitfalls

  1. Not making APIs idempotent: Any API that can be retried (which is all of them) should be idempotent for non-GET methods.
  2. Idempotency key in request body: Keys should be in headers, not the body. The body might change between retries.
  3. Too-short idempotency TTL: If the key expires before the client retries, the operation will be duplicated.
  4. Outbox table growing unbounded: Clean up published events regularly with a retention policy.

Summary

Idempotency ensures that retried operations produce the same result, essential for reliable distributed systems. Implement it with idempotency keys (Redis or database unique constraints). For distributed transactions, prefer the Saga pattern or Outbox pattern over 2PC. The Outbox pattern ensures reliable event publishing by writing events to a database table in the same transaction as the business data. CDC with Debezium provides an infrastructure-level alternative.

Common Problems & Solutions

1 Duplicate records created from retried POST requests
Cause: POST endpoint is not idempotent. Network timeout causes client to retry, creating a second record.
Solution: Require Idempotency-Key header on POST requests. Check key before processing. Return cached result for duplicates.

const check = await redis.get(idempotent:${key});\nif (check) return res.json(JSON.parse(check));

2 Events published but business data write failed
Cause: Publishing an event to Kafka before the database transaction commits. If the transaction rolls back, the event is orphaned.
Solution: Use the Outbox pattern: write events to a database table in the same transaction. A separate publisher reads and sends them to Kafka.
