Design a File Storage System (Dropbox)

Intuition

A file storage system is a chunk-based synchronization engine that breaks files into blocks, deduplicates them, and only transfers changed chunks across devices. The delta sync mechanism is what makes it bandwidth-efficient and fast.

Design a File Storage System (Dropbox)

Requirements

  • Upload, download, and sync files across multiple devices
  • Support large files (up to several GB) with resumable uploads
  • Detect and sync only changed portions of files (delta sync)
  • Handle concurrent edits with conflict resolution
  • Provide file versioning and sharing capabilities

Architecture

Clients use a sync agent that monitors local file changes and communicates with a metadata service. Files are split into chunks (4MB blocks), deduplicated, and stored in object storage (S3). A metadata database tracks file hierarchy, chunks, and versions. A notification service pushes sync events to connected clients.

Key Components

  • Sync Agent: Client-side daemon that watches file changes and computes deltas
  • Chunking Service: Splits files into fixed or content-defined chunks using rolling hash (Rabin fingerprinting)
  • Metadata Service: Tracks file paths, chunk lists, versions, and permissions in a relational DB
  • Block Storage: S3-compatible object store holding deduplicated chunks
  • Notification Service: Long polling or WebSocket to alert clients of remote changes

Scaling

Chunk deduplication reduces storage by 30-50%. Use content-defined chunking so insertions don't invalidate all chunks. Shard metadata by user ID. Use CDN for frequently downloaded files.

Trade-offs

  • Fixed-size chunks are simpler but wasteful on edits; content-defined chunks optimize delta sync but add CPU cost
  • Strong consistency for metadata ensures correctness but adds latency; eventual consistency risks conflicts

Exercises

  1. Implement Content-Defined Chunking

    Build a chunking algorithm using a rolling hash (e.g., Rabin fingerprint) that splits a file into variable-size chunks at content-defined boundaries. Chunks should average 4MB with min 1MB and max 8MB.

    Hint

    Slide a window over the file computing a rolling hash. When the hash modulo a target value equals zero, cut a chunk boundary. This ensures insertions only affect the adjacent chunk.

  2. Design a Conflict Resolution System

    Implement a system that detects when two clients edit the same file simultaneously. Use vector clocks or version numbers to detect conflicts. When a conflict occurs, save both versions and let the user resolve.

    Hint

    Each file has a version number. On upload, the client sends its base version. If the server's current version differs, a conflict is detected and both versions are preserved with a naming convention.

Common Problems & Solutions

1 Uploading a large 5GB file fails at 80% completion, and the user must restart from the beginning.
Cause: The upload is not chunked or resumable, so any network interruption loses all progress.
Solution: Implement chunked, resumable uploads. Track which chunks have been received server-side. On retry, the client queries for missing chunks and uploads only those. Use multipart upload APIs.
2 A small edit to a large file causes the entire file to be re-uploaded and re-synced to all devices.
Cause: Fixed-offset chunking means a small insertion shifts all subsequent chunk boundaries, making every chunk appear changed.
Solution: Use content-defined chunking with a rolling hash so chunk boundaries depend on content, not position. An insertion only affects the chunk where the edit occurred, minimizing data transfer.
