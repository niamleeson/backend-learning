When to Use Threads vs Processes vs Async

Intuition

Think of threads as multitasking in one room (shared space, GIL bottleneck), processes as separate rooms (true independence, communication cost), and async as one person juggling tasks while waiting.

When to Use Threads vs Processes vs Async

Choosing the right concurrency model depends on whether your workload is I/O-bound or CPU-bound, and your specific requirements for complexity and overhead.

import asyncio
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor
import time

def io_task():
    """Simulated I/O: network call, file read, DB query."""
    time.sleep(0.1)
    return "done"

def cpu_task(n: int) -> int:
    """CPU-bound: number crunching, image processing."""
    return sum(i * i for i in range(n))

async def async_io_task():
    """Async I/O: best for many concurrent connections."""
    await asyncio.sleep(0.1)
    return "done"

# Decision matrix:
# I/O-bound, few tasks     -> threading
# I/O-bound, many tasks    -> asyncio
# CPU-bound                -> multiprocessing
# Mixed                    -> asyncio + ProcessPoolExecutor

async def mixed_workload():
    loop = asyncio.get_event_loop()
    # Run CPU work in process pool from async context
    with ProcessPoolExecutor() as pool:
        result = await loop.run_in_executor(pool, cpu_task, 10**6)
    return result

Key Takeaways: - Threads: I/O-bound, simple cases, shared memory access needed - Processes: CPU-bound, need true parallelism, can tolerate serialization - Async: High-concurrency I/O (thousands of connections), cooperative - Use run_in_executor to offload blocking/CPU work from async code - Threads have lower overhead than processes but are limited by the GIL

Exercises

  1. Benchmark All Three Models

    Write the same task (fetch 10 URLs) using threading, multiprocessing, and asyncio. Measure and compare execution time for each approach.

    Hint

    For asyncio, use aiohttp. For threading, use ThreadPoolExecutor. For multiprocessing, use ProcessPoolExecutor. The I/O task should show async and threads winning.

  2. Mixed Workload Orchestrator

    Build an async function that fetches data from 5 URLs (I/O) and then processes each response with a CPU-heavy function using run_in_executor with a ProcessPoolExecutor.

    Hint

    Use aiohttp for fetching, then loop.run_in_executor(process_pool, cpu_func, data) for each response. Gather all with asyncio.gather().

Common Problems & Solutions

1 Using multiprocessing for I/O-bound tasks is slower than threading.
Cause: Process creation and inter-process communication overhead exceeds the I/O wait time.
Solution: Use threads or async for I/O-bound work. Reserve multiprocessing for CPU-intensive tasks where the GIL is the bottleneck.
2 Mixing asyncio with synchronous blocking calls freezes the event loop.
Cause: Calling `time.sleep()` or `requests.get()` directly in an async function blocks the entire event loop.
Solution: Use `await asyncio.sleep()` for delays. Use `loop.run_in_executor()` for synchronous blocking calls, or use async libraries like `aiohttp`.
