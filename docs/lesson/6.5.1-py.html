Repository Pattern

Intuition

A repository is like a librarian. You ask for a book by title, and the librarian finds it -- you do not need to know whether books are stored on shelves, in a warehouse, or digitally. If the library switches from physical to digital storage, your interaction with the librarian stays the same.

Repository Pattern

The Repository pattern abstracts data access behind a collection-like interface, decoupling your business logic from the database. This makes your code testable, swappable, and clean.

Abstract Repository

from abc import ABC, abstractmethod
from typing import Generic, TypeVar, Optional

T = TypeVar("T")

class AbstractRepository(ABC, Generic[T]):
    @abstractmethod
    def add(self, entity: T) -> T:
        ...

    @abstractmethod
    def get(self, id: int) -> Optional[T]:
        ...

    @abstractmethod
    def list(self) -> list[T]:
        ...

    @abstractmethod
    def update(self, entity: T) -> T:
        ...

    @abstractmethod
    def delete(self, id: int) -> None:
        ...

SQLAlchemy Implementation

from sqlalchemy.orm import Session

class SQLAlchemyUserRepository(AbstractRepository[User]):
    def __init__(self, session: Session):
        self.session = session

    def add(self, user: User) -> User:
        self.session.add(user)
        self.session.flush()
        return user

    def get(self, id: int) -> Optional[User]:
        return self.session.query(User).filter_by(id=id).first()

    def list(self) -> list[User]:
        return self.session.query(User).all()

    def update(self, user: User) -> User:
        self.session.merge(user)
        self.session.flush()
        return user

    def delete(self, id: int) -> None:
        user = self.get(id)
        if user:
            self.session.delete(user)

In-Memory Implementation for Testing

class InMemoryUserRepository(AbstractRepository[User]):
    def __init__(self):
        self._store: dict[int, User] = {}
        self._next_id = 1

    def add(self, user: User) -> User:
        user.id = self._next_id
        self._store[user.id] = user
        self._next_id += 1
        return user

    def get(self, id: int) -> Optional[User]:
        return self._store.get(id)

Common Mistakes

  • Leaking ORM-specific queries into business logic (defeats the purpose)
  • Making repositories too granular (one per table) instead of per aggregate
  • Putting business logic inside repositories

Key Takeaways

  • Repositories hide data access details behind a clean interface
  • Use abstract base classes so you can swap implementations (SQL, in-memory, API)
  • Repositories operate on domain entities, not raw database rows

Exercises

  1. Build a Repository with Two Implementations

    Create an abstract ProductRepository and implement it twice: once with SQLAlchemy and once with an in-memory dictionary. Write a service that uses the repository and demonstrate swapping implementations.

    Hint

    Use dependency injection to pass the repository into the service. In tests, pass InMemoryProductRepository. In production, pass SQLAlchemyProductRepository.

  2. Repository with Query Methods

    Extend the repository pattern with domain-specific query methods: find_by_email, find_active_users, search_by_name. Implement these in both the SQL and in-memory versions.

    Hint

    Add abstract methods to the base class for each query. In SQL, use filter(). In-memory, use list comprehensions over the internal dict.

Common Problems & Solutions

1 Business logic is tightly coupled to the database
Symptoms: ['Cannot test business logic without a running database', 'Changing databases requires rewriting business logic']
Cause: Service functions directly use SQLAlchemy queries instead of going through an abstraction layer.
Solution: Introduce a repository layer. Move all database queries into repository methods. Services call repository methods instead of using the ORM directly.

Bad: ORM in service

def get_active_users(session): return session.query(User).filter(User.active == True).all()

Good: repository abstraction

def get_active_users(user_repo: AbstractRepository): return user_repo.find_active()

2 Repository methods become a dumping ground for every query variant
Cause: Every new feature adds a new specific query method to the repository, making the interface bloated.
Solution: Use the Specification pattern for complex queries. Keep core CRUD in the repository and pass specification objects for filtering: repo.find(spec=ActiveUsersSpec()).
