Constraints: PRIMARY KEY, FOREIGN KEY, UNIQUE, CHECK

Intuition

Constraints are like the guardrails on a mountain road. Your application code (the driver) should stay on the road, but guardrails (constraints) prevent catastrophe when mistakes happen. Without guardrails, one wrong turn corrupts your data silently. With them, you get an immediate error and the data stays safe.

Constraints: PRIMARY KEY, FOREIGN KEY, UNIQUE, CHECK

Introduction

Constraints are rules enforced by the database to ensure data integrity. They are your last line of defense against bad data — application code can have bugs, but database constraints are inviolable. A well-constrained schema catches errors that would otherwise corrupt your data silently.

Why Constraints Matter

Without constraints:
  - Duplicate emails? The app just creates another user.
  - Negative prices? Sure, why not.
  - Order references non-existent user? No problem.
  - Age = -5? The database doesn't care.

With constraints:
  - The database REJECTS invalid data.
  - You get an immediate, clear error.
  - Data integrity is guaranteed at the database level.

PRIMARY KEY

Uniquely identifies each row. Cannot be NULL. Automatically creates an index.

-- Auto-incrementing integer
CREATE TABLE users (
  id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name text NOT NULL
);

-- UUID
CREATE TABLE orders (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  total numeric(10, 2) NOT NULL
);

-- Composite primary key
CREATE TABLE order_items (
  order_id uuid REFERENCES orders(id),
  product_id uuid REFERENCES products(id),
  quantity integer NOT NULL,
  PRIMARY KEY (order_id, product_id)
);

// Handling primary key violations
try {
  await pool.query(
    'INSERT INTO users (id, name) VALUES ($1, $2)',
    [existingId, 'Alice']
  );
} catch (err) {
  if (err.code === '23505') { // unique_violation
    console.log('User with this ID already exists');
  }
}

FOREIGN KEY

Ensures referential integrity — the referenced row must exist.

CREATE TABLE orders (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES users(id),
  -- or more explicitly:
  -- FOREIGN KEY (user_id) REFERENCES users(id)
  total numeric(10, 2) NOT NULL
);

ON DELETE and ON UPDATE Actions

-- CASCADE: delete orders when user is deleted
user_id uuid REFERENCES users(id) ON DELETE CASCADE

-- SET NULL: set to null when user is deleted
user_id uuid REFERENCES users(id) ON DELETE SET NULL

-- RESTRICT (default): prevent user deletion if orders exist
user_id uuid REFERENCES users(id) ON DELETE RESTRICT

-- SET DEFAULT: set to default value when user is deleted
user_id uuid REFERENCES users(id) ON DELETE SET DEFAULT

// Handling foreign key violations
async function createOrder(userId, total) {
  try {
    const result = await pool.query(
      'INSERT INTO orders (user_id, total) VALUES ($1, $2) RETURNING *',
      [userId, total]
    );
    return result.rows[0];
  } catch (err) {
    if (err.code === '23503') { // foreign_key_violation
      throw new Error(`User ${userId} does not exist`);
    }
    throw err;
  }
}

UNIQUE

Ensures no two rows have the same value (NULLs are allowed and don't conflict):

CREATE TABLE users (
  id serial PRIMARY KEY,
  email varchar(320) UNIQUE NOT NULL,
  username text UNIQUE NOT NULL
);

-- Multi-column unique constraint
CREATE TABLE enrollments (
  student_id integer REFERENCES students(id),
  course_id integer REFERENCES courses(id),
  UNIQUE (student_id, course_id)
);

-- Partial unique (only enforce uniqueness for active records)
CREATE UNIQUE INDEX idx_users_email_active
ON users (email) WHERE deleted_at IS NULL;

Upsert with ON CONFLICT

async function upsertUser(email, name) {
  const result = await pool.query(`
    INSERT INTO users (email, name)
    VALUES ($1, $2)
    ON CONFLICT (email) DO UPDATE
    SET name = EXCLUDED.name, updated_at = NOW()
    RETURNING *
  `, [email, name]);
  return result.rows[0];
}

NOT NULL

Prevents NULL values:

CREATE TABLE products (
  id serial PRIMARY KEY,
  name text NOT NULL,                -- required
  description text,                  -- optional (nullable)
  price numeric(10, 2) NOT NULL,     -- required
  stock integer NOT NULL DEFAULT 0   -- required with default
);

Best practice: Make columns NOT NULL by default. Only allow NULL when there's a clear semantic meaning for "no value."

CHECK Constraints

Custom validation rules:

CREATE TABLE products (
  id serial PRIMARY KEY,
  name text NOT NULL CHECK (length(name) >= 1),
  price numeric(10, 2) NOT NULL CHECK (price >= 0),
  compare_at_price numeric(10, 2) CHECK (compare_at_price > price),
  discount_pct integer CHECK (discount_pct BETWEEN 0 AND 100),
  status text CHECK (status IN ('draft', 'active', 'archived'))
);

-- Named constraints (better error messages)
CREATE TABLE events (
  id serial PRIMARY KEY,
  name text NOT NULL,
  starts_at timestamptz NOT NULL,
  ends_at timestamptz NOT NULL,
  CONSTRAINT events_valid_dates CHECK (ends_at > starts_at),
  CONSTRAINT events_future_only CHECK (starts_at > NOW())
);

// Handling check constraint violations
async function createProduct(data) {
  try {
    const result = await pool.query(`
      INSERT INTO products (name, price, discount_pct, status)
      VALUES ($1, $2, $3, $4)
      RETURNING *
    `, [data.name, data.price, data.discount, data.status]);
    return result.rows[0];
  } catch (err) {
    if (err.code === '23514') { // check_violation
      const constraint = err.constraint;
      if (constraint === 'products_price_check') {
        throw new Error('Price must be non-negative');
      }
      if (constraint === 'products_status_check') {
        throw new Error('Invalid status value');
      }
      throw new Error(`Validation failed: ${constraint}`);
    }
    throw err;
  }
}

EXCLUSION Constraints

Prevent overlapping ranges (like room reservations):

CREATE EXTENSION IF NOT EXISTS btree_gist;

CREATE TABLE room_reservations (
  id serial PRIMARY KEY,
  room_id integer NOT NULL,
  during tstzrange NOT NULL,
  EXCLUDE USING gist (
    room_id WITH =,
    during WITH &&
  )
);

-- This prevents overlapping reservations for the same room!
INSERT INTO room_reservations (room_id, during)
VALUES (1, '[2024-01-15 14:00, 2024-01-15 16:00)');

-- This would fail if it overlaps with the above:
INSERT INTO room_reservations (room_id, during)
VALUES (1, '[2024-01-15 15:00, 2024-01-15 17:00)');
-- ERROR: conflicting key value violates exclusion constraint

PostgreSQL Error Codes Reference

// Map PG error codes to user-friendly messages
const PG_ERROR_MAP = {
  '23505': 'A record with this value already exists',    // unique_violation
  '23503': 'Referenced record does not exist',            // foreign_key_violation
  '23502': 'A required field is missing',                 // not_null_violation
  '23514': 'Value does not meet validation requirements', // check_violation
  '23P01': 'Values conflict with existing data',          // exclusion_violation
};

function handlePgError(err) {
  const message = PG_ERROR_MAP[err.code];
  if (message) {
    return { status: 400, error: message, detail: err.detail };
  }
  return { status: 500, error: 'Internal server error' };
}

Exercises

  1. Constraint-Heavy Schema

    Design a bookings system with tables for rooms, guests, and bookings. Add constraints for: no overlapping bookings per room (EXCLUSION), check-out after check-in (CHECK), valid email format (CHECK with regex), positive prices, and a maximum of 4 guests per room.

    Hint

    Use EXCLUSION with btree_gist for non-overlapping dates. Use CHECK with regex (~) for email validation. Use CHECK for date comparison.

  2. Error Handler Middleware

    Write an Express middleware that catches PostgreSQL constraint violations and returns appropriate HTTP status codes and user-friendly error messages. Map each constraint name to a specific message.

    Hint

    Check err.code for PG error codes (23505, 23503, etc.) and err.constraint for the specific constraint name. Return 409 for unique violations, 400 for check violations.

  3. Upsert Patterns

    Implement three upsert patterns: (a) insert or update on email conflict, (b) insert or ignore on duplicate, (c) insert or update only specific columns. Test each from Node.js.

    Hint

    Use ON CONFLICT (column) DO UPDATE SET ... for (a), ON CONFLICT DO NOTHING for (b), and EXCLUDED to reference the incoming row's values.

Common Problems & Solutions

1 Cannot add NOT NULL constraint to column with existing NULL values
Cause: The table already has rows where the column is NULL.
Solution: First backfill the data: `UPDATE table SET col = 'default' WHERE col IS NULL;` Then add the constraint: `ALTER TABLE table ALTER COLUMN col SET NOT NULL;`
2 Foreign key constraint prevents deleting parent records
Cause: Default ON DELETE behavior is RESTRICT, which prevents deletion of referenced rows.
Solution: Choose the appropriate action: ON DELETE CASCADE (auto-delete children), ON DELETE SET NULL (nullify references), or handle in application code by deleting children first.
3 Unique constraint does not prevent duplicate NULL values
Cause: In SQL, NULL != NULL, so the UNIQUE constraint considers each NULL as distinct.
Solution: If you need to prevent multiple NULLs, use a partial unique index: `CREATE UNIQUE INDEX idx ON table (col) WHERE col IS NOT NULL;` Or use COALESCE with a sentinel value.
