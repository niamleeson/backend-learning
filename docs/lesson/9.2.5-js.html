Design a Distributed Cache (Memcached)

Intuition

A distributed cache is like a network of speed-dial phone books. Each phone book (cache node) holds a subset of frequently called numbers (cached values). When you need a number, your personal assistant (client library) knows exactly which phone book to check (consistent hashing). If the number is there, you get it instantly. If not, you look it up in the full directory (database) and write it in the phone book for next time. When a phone book is full, the least recently looked-up number is erased (LRU eviction). If a phone book is lost (node failure), you do not panic -- you can always look numbers up in the full directory. The key insight is that losing a phone book is annoying but not catastrophic, because the directory (database) is the source of truth.

Design a Distributed Cache (Memcached)

Introduction

A distributed cache stores frequently accessed data in memory across multiple machines, dramatically reducing latency and database load. Memcached, Redis Cluster, and similar systems are the backbone of high-performance web applications. This design problem tests your understanding of in-memory data management, cache eviction policies, consistent hashing, cache coherence, and the fundamental trade-offs between speed, consistency, and memory efficiency.

Step 1: Requirements Gathering

Functional Requirements

  • get(key): Retrieve a cached value by key
  • set(key, value, ttl): Store a value with an optional time-to-live
  • delete(key): Remove a cached entry
  • Multi-get: Batch retrieval of multiple keys in one round trip
  • Atomic increment/decrement: For counters
  • LRU eviction: Automatically evict least recently used items when memory is full

Non-Functional Requirements

  • Latency: < 1ms for get/set at p99
  • Throughput: 1M+ operations per second per node
  • Scalability: Horizontally scale by adding cache nodes
  • Availability: 99.99% -- cache unavailability causes a thundering herd on the database
  • Memory efficiency: Maximize useful data per GB of RAM
  • Simplicity: Thin server, smart client

Step 4: API Design

Cache Client (used by application servers)

import crypto from 'node:crypto';
import net from 'node:net';

class DistributedCacheClient {
  constructor(nodes, options = {}) {
    this.hashRing = new ConsistentHashRing(nodes);
    this.connections = new Map();  // node -> connection pool
    this.retries = options.retries || 2;
    this.timeout = options.timeout || 100; // ms

    // Initialize connection pools
    for (const node of nodes) {
      this.connections.set(node, new ConnectionPool(node, {
        maxConnections: 20,
        idleTimeout: 30000,
      }));
    }
  }

  async get(key) {
    const node = this.hashRing.getNode(key);
    const conn = await this.connections.get(node).acquire();

    try {
      const result = await conn.sendCommand(`GET ${key}`, this.timeout);
      return result === 'NOT_FOUND' ? null : JSON.parse(result);
    } finally {
      conn.release();
    }
  }

  async set(key, value, ttl = 0) {
    const node = this.hashRing.getNode(key);
    const conn = await this.connections.get(node).acquire();
    const serialized = JSON.stringify(value);

    try {
      await conn.sendCommand(`SET ${key} ${ttl} ${serialized}`, this.timeout);
    } finally {
      conn.release();
    }
  }

  async delete(key) {
    const node = this.hashRing.getNode(key);
    const conn = await this.connections.get(node).acquire();

    try {
      await conn.sendCommand(`DELETE ${key}`, this.timeout);
    } finally {
      conn.release();
    }
  }

  // Multi-get: batch retrieval across multiple nodes
  async getMulti(keys) {
    // Group keys by target node
    const nodeKeyMap = new Map();
    for (const key of keys) {
      const node = this.hashRing.getNode(key);
      if (!nodeKeyMap.has(node)) nodeKeyMap.set(node, []);
      nodeKeyMap.get(node).push(key);
    }

    // Send requests in parallel (one per node)
    const results = new Map();
    await Promise.all(
      [...nodeKeyMap.entries()].map(async ([node, nodeKeys]) => {
        const conn = await this.connections.get(node).acquire();
        try {
          const batchResult = await conn.sendCommand(
            `MGET ${nodeKeys.join(' ')}`,
            this.timeout
          );
          for (const [key, value] of Object.entries(batchResult)) {
            results.set(key, value ? JSON.parse(value) : null);
          }
        } finally {
          conn.release();
        }
      })
    );

    return results;
  }
}

router.get('/api/v1/cache/:key', async (req, res) => {
  const { key } = req.params;
  const entry = cacheStore.get(key);

  if (!entry) {
    return res.status(404).json({ error: 'Cache miss' });
  }

  // Check TTL
  if (entry.expiresAt && Date.now() > entry.expiresAt) {
    cacheStore.delete(key);
    return res.status(404).json({ error: 'Cache miss (expired)' });
  }

  res.json({ key, value: entry.value, ttl: entry.remainingTTL() });
});

// Set a cached value
router.put('/api/v1/cache/:key', async (req, res) => {
  const { key } = req.params;
  const { value, ttl } = req.body;

  cacheStore.set(key, value, ttl);
  res.json({ stored: true });
});

// Delete a cached value
router.delete('/api/v1/cache/:key', async (req, res) => {
  cacheStore.delete(req.params.key);
  res.json({ deleted: true });
});

// Multi-get
router.post('/api/v1/cache/mget', async (req, res) => {
  const { keys } = req.body;
  const results = {};
  for (const key of keys) {
    const entry = cacheStore.get(key);
    results[key] = entry && !entry.isExpired() ? entry.value : null;
  }
  res.json({ results });
});

Step 5: Core Data Structures

LRU Cache Implementation

The heart of each cache node is an LRU (Least Recently Used) cache that combines a hash map for O(1) lookups with a doubly-linked list for O(1) eviction.

class LRUCache {
  constructor(maxSize) {
    this.maxSize = maxSize;       // Max number of items
    this.map = new Map();         // key -> Node
    this.head = { prev: null };   // Most recently used (dummy)
    this.tail = { next: null };   // Least recently used (dummy)
    this.head.next = this.tail;
    this.tail.prev = this.head;
    this.hits = 0;
    this.misses = 0;
    this.evictionCount = 0;
  }

  get(key) {
    const node = this.map.get(key);
    if (!node) {
      this.misses++;
      return null;
    }

    // Check TTL
    if (node.expiresAt && Date.now() > node.expiresAt) {
      this._remove(node);
      this.map.delete(key);
      this.misses++;
      return null;
    }

    // Move to front (most recently used)
    this._remove(node);
    this._addToFront(node);
    this.hits++;
    return node.value;
  }

  set(key, value, ttl = 0) {
    // Update existing
    if (this.map.has(key)) {
      const node = this.map.get(key);
      node.value = value;
      node.expiresAt = ttl > 0 ? Date.now() + ttl * 1000 : null;
      this._remove(node);
      this._addToFront(node);
      return;
    }

    // Evict if at capacity
    if (this.map.size >= this.maxSize) {
      this._evict();
    }

    // Insert new node
    const node = {
      key,
      value,
      expiresAt: ttl > 0 ? Date.now() + ttl * 1000 : null,
      prev: null,
      next: null,
    };
    this._addToFront(node);
    this.map.set(key, node);
  }

  delete(key) {
    const node = this.map.get(key);
    if (!node) return false;
    this._remove(node);
    this.map.delete(key);
    return true;
  }

  _addToFront(node) {
    node.next = this.head.next;
    node.prev = this.head;
    this.head.next.prev = node;
    this.head.next = node;
  }

  _remove(node) {
    node.prev.next = node.next;
    node.next.prev = node.prev;
  }

  _evict() {
    const lru = this.tail.prev;
    if (lru === this.head) return; // empty
    this._remove(lru);
    this.map.delete(lru.key);
    this.evictionCount++;
  }

  get size() { return this.map.size; }

  hitRate() {
    const total = this.hits + this.misses;
    return total === 0 ? 0 : (this.hits / total * 100).toFixed(2) + '%';
  }
}

Slab Allocation (Memory Efficiency)

Memcached uses slab allocation to reduce memory fragmentation:

class SlabAllocator {
  constructor() {
    // Slab classes with increasing chunk sizes
    this.slabClasses = [
      { chunkSize: 96,    slabs: [] },   // Class 1: up to 96 bytes
      { chunkSize: 120,   slabs: [] },   // Class 2: up to 120 bytes
      { chunkSize: 152,   slabs: [] },   // Class 3: up to 152 bytes
      { chunkSize: 192,   slabs: [] },   // ...
      { chunkSize: 240,   slabs: [] },
      { chunkSize: 304,   slabs: [] },
      { chunkSize: 384,   slabs: [] },
      { chunkSize: 512,   slabs: [] },
      { chunkSize: 1024,  slabs: [] },
      { chunkSize: 2048,  slabs: [] },
      { chunkSize: 4096,  slabs: [] },
      { chunkSize: 8192,  slabs: [] },
    ];
    this.SLAB_PAGE_SIZE = 1024 * 1024; // 1MB per slab page
  }

  findSlabClass(itemSize) {
    return this.slabClasses.find(sc => sc.chunkSize >= itemSize);
  }

  allocate(itemSize) {
    const slabClass = this.findSlabClass(itemSize);
    if (!slabClass) throw new Error(`Item too large: ${itemSize}`);

    // Find a slab page with a free chunk, or allocate a new page
    let page = slabClass.slabs.find(p => p.freeChunks > 0);
    if (!page) {
      const chunksPerPage = Math.floor(this.SLAB_PAGE_SIZE / slabClass.chunkSize);
      page = {
        buffer: Buffer.alloc(this.SLAB_PAGE_SIZE),
        chunkSize: slabClass.chunkSize,
        totalChunks: chunksPerPage,
        freeChunks: chunksPerPage,
        freeList: Array.from({ length: chunksPerPage }, (_, i) => i),
      };
      slabClass.slabs.push(page);
    }

    const chunkIndex = page.freeList.pop();
    page.freeChunks--;
    const offset = chunkIndex * page.chunkSize;
    return { page, offset, chunkIndex };
  }
}

Step 6: Deep Dive -- Cache Invalidation and Consistency

Cache invalidation is famously one of the two hard problems in computer science.

Invalidation Strategies

1. TTL-Based Expiration:
   - Set a TTL on every cache entry
   - Simple and predictable
   - Risk: stale data until TTL expires

2. Write-Through:
   App ──> Write to Cache ──> Write to DB
   - Cache is always up-to-date
   - Higher write latency

3. Write-Behind (Write-Back):
   App ──> Write to Cache ──> (async) Write to DB
   - Lowest write latency
   - Risk: data loss if cache node crashes before DB write

4. Cache-Aside (Lazy Loading):
   Read:  App ──> Check Cache ──> miss ──> Read DB ──> Update Cache
   Write: App ──> Write DB ──> Invalidate Cache
   - Most common pattern
   - Risk: race condition between invalidation and a concurrent read

Thundering Herd Problem

// Problem: When a popular cache key expires, hundreds of concurrent
// requests all miss the cache and hit the database simultaneously.

// Solution: Cache stampede protection with locks
class CacheWithStampedeProtection {
  constructor(cache, db, redis) {
    this.cache = cache;
    this.db = db;
    this.redis = redis;
  }

  async get(key, fetchFn, ttl = 300) {
    // 1. Try cache
    const cached = await this.cache.get(key);
    if (cached !== null) return cached;

    // 2. Try to acquire lock
    const lockKey = `lock:${key}`;
    const locked = await this.redis.set(lockKey, '1', 'NX', 'EX', 5);

    if (locked) {
      // 3. We got the lock -- fetch from DB
      try {
        const value = await fetchFn();
        await this.cache.set(key, value, ttl);
        return value;
      } finally {
        await this.redis.del(lockKey);
      }
    } else {
      // 4. Another request is fetching -- wait and retry
      await new Promise(resolve => setTimeout(resolve, 50));
      return this.get(key, fetchFn, ttl);
    }
  }
}

Early Expiration (Probabilistic)

// Another stampede solution: proactively refresh before TTL expires
async function getWithEarlyExpiry(cache, key, fetchFn, ttl) {
  const entry = await cache.getRaw(key);
  if (!entry) {
    const value = await fetchFn();
    await cache.set(key, value, ttl);
    return value;
  }

  const remainingTTL = entry.expiresAt - Date.now();
  const totalTTL = ttl * 1000;

  // As TTL gets closer to expiry, probabilistically refresh
  // XFetch algorithm: probability increases as expiry approaches
  const probability = Math.exp(-remainingTTL / (totalTTL * 0.1));

  if (Math.random() < probability) {
    // Refresh in background (don't block the current request)
    fetchFn().then(value => cache.set(key, value, ttl)).catch(() => {});
  }

  return entry.value;
}

Step 7: Scaling Strategies

  1. Consistent hashing with virtual nodes: Each physical node has 100-200 virtual nodes on the hash ring for even distribution. Adding a node remaps only 1/N of keys.
  2. Client-side sharding: The client library determines which node to contact. No central router needed.
  3. Connection pooling: Maintain persistent connections between application servers and cache nodes to avoid TCP handshake overhead.
  4. Multi-get optimization: Group keys by target node and issue one batched request per node instead of N individual requests.
  5. Gutter pool: When a cache node fails, route its keys to a dedicated "gutter" pool instead of the database. This prevents thundering herd on node failure.
  6. Monitoring: Track hit rate, eviction rate, memory usage, and latency per node. A dropping hit rate indicates the cache is too small or keys are too short-lived.
Gutter Pool (Facebook's approach):

Normal:  Client ──> Hash Ring ──> Cache Node 3 ──> HIT

Node 3 down:
         Client ──> Hash Ring ──> Cache Node 3 ──> FAIL
                                  │
                                  ▼
                          Gutter Pool Node ──> MISS ──> DB
                          (temporary, short TTL)

  Instead of ALL of Node 3's keys hitting the DB,
  they populate the gutter pool on first miss.
  Subsequent requests hit the gutter pool.

Step 8: Trade-offs

Decision Option A Option B Recommendation
Eviction policy LRU LFU LRU (simpler, good enough for most workloads)
Serialization JSON Protocol Buffers JSON for simplicity, Protobuf for performance
Protocol Text (Memcached ASCII) Binary Binary (less parsing overhead, 20%+ faster)
Replication None (Memcached style) Primary-replica (Redis style) None for pure cache; replicated if data loss is costly
Consistency None (best-effort) Lease-based Lease-based if stale reads cause serious issues
Memory management Node.js heap Slab allocator (native) Slab allocator for large caches (avoids GC pauses)

Why no replication by default? A cache is not the source of truth. If a cache node dies, the worst case is a temporary increase in database load while the cache warms up. Replication doubles memory cost for data that is already stored in the database. The exception is when cache warm-up is too slow or the database cannot handle the thundering herd -- in that case, replicate or use a gutter pool.

LRU vs. LFU: LRU is simpler and works well for most access patterns. LFU (Least Frequently Used) is better when some items are accessed in bursts then never again (LRU would keep them too long). Redis 4.0+ uses an approximated LFU. A good compromise is Segmented LRU (SLRU) which maintains a probationary and a protected segment.

Exercises

  1. Implement an LRU Cache

    Build an LRU cache in Node.js using a Map and a doubly linked list. Support get(key), set(key, value, ttl), and delete(key). All operations must be O(1). Add a stats endpoint that reports hit rate, miss rate, and eviction count. Benchmark with 1M operations.

    Hint

    Use a Map for O(1) key lookup. The doubly linked list maintains access order. On get, move the node to the head. On set, add to head and evict from tail if full.

  2. Build a Distributed Cache Client

    Create a cache client that distributes keys across 3 local cache server instances using consistent hashing. Implement get, set, delete, and multi-get. Test that adding a 4th node only remaps approximately 25% of keys.

    Hint

    Run 3 Express servers on different ports. Use the ConsistentHashRing class from the lesson. For multi-get, group keys by node and send parallel requests.

  3. Implement Cache Stampede Protection

    Build a cache-aside pattern with stampede protection using Redis locks. Simulate a thundering herd by firing 100 concurrent requests for the same cache key after it expires. Compare database hit count with and without stampede protection.

    Hint

    Use Redis SET with NX and EX flags for the lock. Waiting requests should retry after a short delay. Count DB calls with a counter to verify protection works.

  4. Eviction Policy Comparison

    Implement LRU, LFU, and Random eviction policies. Generate a workload with Zipfian distribution (some keys much more popular than others). Measure and compare hit rates across all three policies for the same cache size.

    Hint

    For Zipfian distribution, use the formula: probability of accessing item i is proportional to 1/i^s where s is the skew parameter. Use s=1.0 for typical web workloads.

Common Problems & Solutions

1 Cache hit rate drops significantly after adding a new node
Cause: Using modulo-based hashing (key hash % N). When N changes from 3 to 4 nodes, up to 75% of keys are remapped to different nodes, causing mass cache misses.
Solution: Switch to consistent hashing with virtual nodes. When a node is added to a ring with N nodes, only approximately 1/N of keys are remapped. Use 100+ virtual nodes per physical node for even distribution.
2 Thundering herd on database after cache node failure
Cause: All keys assigned to the failed node result in cache misses. All application servers simultaneously query the database for those keys.
Solution: Implement a gutter pool: a small set of reserve cache nodes. When a primary cache node is unreachable, route its keys to the gutter pool instead of going to the database. Use short TTLs in the gutter pool. Also consider warming the replacement node from a database snapshot.
3 Memory usage on cache nodes is uneven -- some nodes are 95% full while others are 50%
Cause: Poor distribution of keys across the hash ring, or some keys have significantly larger values than others.
Solution: Increase virtual nodes per physical node (150+) for more even distribution. Monitor per-node memory and key counts. Consider value-size-aware routing: route large values to nodes with more available memory. Alternatively, use slab rebalancing to reclaim memory from underused slab classes.
