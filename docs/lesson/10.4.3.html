Design a Web Crawler (Google)

Intuition

A web crawler is a massive producer-consumer system where the URL frontier is the central coordination point, balancing discovery speed against politeness constraints. The key insight is that per-host queuing and rate limiting are not optional niceties but fundamental architectural requirements.

Design a Web Crawler (Google)

Requirements

  • Crawl billions of web pages, respecting robots.txt and politeness policies
  • Detect and handle duplicate content, redirect chains, and spider traps
  • Prioritize pages by importance (PageRank, freshness, domain authority)
  • Store raw HTML and extracted metadata for downstream indexing

Architecture

A URL Frontier holds URLs to crawl, ordered by priority and politeness constraints. Fetcher workers pull URLs, download pages, and pass HTML to a processing pipeline that extracts links, deduplicates content, and feeds new URLs back to the frontier. A central DNS resolver cache reduces lookup latency.

Key Components

  • URL Frontier: Priority queue with per-domain rate limiting (back queue per host)
  • Fetcher Workers: Async HTTP clients that respect crawl-delay directives
  • Content Deduplicator: SimHash or MinHash fingerprint to detect near-duplicates
  • Blob Store: S3 or HDFS for raw HTML; metadata goes to a search index pipeline

Scaling

Partition the frontier by domain hash. Run fetcher workers across regions close to target servers. Use consistent hashing to ensure one domain is crawled by one worker (politeness).

Trade-offs

  • Breadth-first discovers more domains; best-first finds important pages faster
  • Aggressive crawling gets fresher data but risks IP bans
  • Storing all raw HTML is expensive but enables re-processing without re-crawling

Exercises

  1. Implement a Polite URL Frontier

    Build a URL frontier that maintains per-domain back-queues with configurable crawl-delay. Verify that no domain is hit more than once per its specified delay.

    Hint

    Use a map of domain -> queue plus a global priority queue of (next_allowed_time, domain) to decide which domain to serve next.

  2. Near-Duplicate Detection with SimHash

    Implement SimHash fingerprinting on a set of 1,000 HTML documents. Detect pages with >90% similarity and mark them as duplicates.

    Hint

    Convert each document to a set of shingles (3-word n-grams), hash each shingle, then compute a bitwise majority across all hashes to get the fingerprint.

Common Problems & Solutions

1 Crawler gets stuck in a spider trap generating infinite URLs
Cause: Dynamic pages with session IDs or calendar widgets create unbounded URL variations
Solution: Limit crawl depth per domain, normalize URLs by stripping known tracking parameters, and cap the number of URLs stored per host.
2 DNS resolution becomes a bottleneck slowing the entire crawl
Cause: Each fetch does a fresh DNS lookup, and the default resolver cannot handle the query volume
Solution: Run a local DNS cache (e.g., dnsmasq) and batch DNS pre-resolution for queued URLs ahead of fetch time.
