Modules: Feature Modules and Module Graph

Intuition

Modules in NestJS are like departments in a company. Each department (module) has its own employees (providers) and a front desk (controllers). Departments can share specific employees with other departments (exports), but internal staff remain private. The org chart (module graph) shows how departments depend on each other.

Modules: Feature Modules and Module Graph

Introduction

Modules are the fundamental organizational unit in NestJS. Every NestJS application has at least one module -- the root module (AppModule). As your application grows, you create feature modules to encapsulate related functionality, keeping your codebase organized and maintainable.

What is a Module?

A module is a class decorated with @Module(). The decorator provides metadata that NestJS uses to organize the application structure:

import { Module } from '@nestjs/common';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';

@Module({
  controllers: [UsersController],
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}

Module Metadata Properties

Property Purpose
imports Modules whose exported providers are needed in this module
controllers Controllers instantiated within this module
providers Providers (services) instantiated and available in this module
exports Providers from this module available to other importing modules

The Module Graph

NestJS builds a module graph at startup -- a directed graph of module dependencies. Understanding this graph is key to understanding how NestJS resolves dependencies.

                    AppModule
                   /    |    \
                  /     |     \
         UsersModule  AuthModule  ProductsModule
              |          |    \        |
              |          |     \       |
         DatabaseModule  |   JwtModule |
              \__________|____________/
                         |
                   SharedModule

When AuthModule imports UsersModule, it gains access to everything UsersModule exports. Providers that are NOT exported remain private to their module.

Creating Feature Modules

Use the CLI to generate a module:

nest g module users

This creates src/users/users.module.ts and automatically adds it to AppModule's imports.

Example: Users Feature Module

// users/users.module.ts
import { Module } from '@nestjs/common';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';
import { UsersRepository } from './users.repository';

@Module({
  controllers: [UsersController],
  providers: [UsersService, UsersRepository],
  exports: [UsersService], // Only UsersService is accessible outside
})
export class UsersModule {}

// app.module.ts
import { Module } from '@nestjs/common';
import { UsersModule } from './users/users.module';
import { AuthModule } from './auth/auth.module';

@Module({
  imports: [UsersModule, AuthModule],
})
export class AppModule {}

Module Encapsulation

This is one of NestJS's most important concepts. Providers are private to their module by default. Only providers listed in exports are available to other modules.

// orders/orders.module.ts
@Module({
  providers: [OrdersService, OrdersRepository, PricingService],
  exports: [OrdersService], // Only OrdersService is exported
})
export class OrdersModule {}

// shipping/shipping.module.ts
@Module({
  imports: [OrdersModule],
  providers: [ShippingService],
})
export class ShippingModule {}

In ShippingModule, you can inject OrdersService but NOT OrdersRepository or PricingService. This prevents tight coupling between modules.

Shared Modules

When multiple modules need the same provider, create a shared module:

// shared/shared.module.ts
@Module({
  providers: [LoggerService, MailService, CacheService],
  exports: [LoggerService, MailService, CacheService],
})
export class SharedModule {}

Any module that imports SharedModule gets access to all three services. Each service is still a singleton -- the same instance is shared across all importing modules.

Global Modules

If a module is needed everywhere, you can make it global to avoid importing it in every module:

import { Module, Global } from '@nestjs/common';

@Global()
@Module({
  providers: [DatabaseService, ConfigService],
  exports: [DatabaseService, ConfigService],
})
export class CoreModule {}

Once CoreModule is imported in AppModule, DatabaseService and ConfigService are available everywhere without explicit imports.

Warning: Use @Global() sparingly. It breaks the explicit dependency graph and makes it harder to understand where providers come from. Reserve it for truly cross-cutting concerns like configuration and database connections.

Dynamic Modules

Dynamic modules allow configuration at import time. This is how libraries like TypeOrmModule and ConfigModule work:

// database/database.module.ts
import { Module, DynamicModule } from '@nestjs/common';

@Module({})
export class DatabaseModule {
  static forRoot(options: DatabaseOptions): DynamicModule {
    return {
      module: DatabaseModule,
      global: true,
      providers: [
        {
          provide: 'DATABASE_OPTIONS',
          useValue: options,
        },
        DatabaseService,
      ],
      exports: [DatabaseService],
    };
  }

  static forFeature(entities: any[]): DynamicModule {
    const repositories = entities.map((entity) => ({
      provide: `${entity.name}Repository`,
      useFactory: (db: DatabaseService) => db.getRepository(entity),
      inject: [DatabaseService],
    }));

    return {
      module: DatabaseModule,
      providers: repositories,
      exports: repositories,
    };
  }
}

Usage:

// app.module.ts
@Module({
  imports: [
    DatabaseModule.forRoot({
      host: 'localhost',
      port: 5432,
      database: 'mydb',
    }),
  ],
})
export class AppModule {}

// users/users.module.ts
@Module({
  imports: [DatabaseModule.forFeature([User])],
  controllers: [UsersController],
  providers: [UsersService],
})
export class UsersModule {}

The forRoot() / forFeature() pattern is a NestJS convention: - forRoot() -- Configure the module once (usually in AppModule). Often makes the module global. - forFeature() -- Register feature-specific components (entities, repositories) in feature modules.

Async Dynamic Modules

When configuration requires async operations (e.g., fetching secrets from a vault):

@Module({})
export class DatabaseModule {
  static forRootAsync(options: {
    useFactory: (...args: any[]) => Promise<DatabaseOptions> | DatabaseOptions;
    inject?: any[];
    imports?: any[];
  }): DynamicModule {
    return {
      module: DatabaseModule,
      global: true,
      imports: options.imports || [],
      providers: [
        {
          provide: 'DATABASE_OPTIONS',
          useFactory: options.useFactory,
          inject: options.inject || [],
        },
        DatabaseService,
      ],
      exports: [DatabaseService],
    };
  }
}

Usage:

@Module({
  imports: [
    DatabaseModule.forRootAsync({
      imports: [ConfigModule],
      useFactory: async (config: ConfigService) => ({
        host: config.get('DB_HOST'),
        port: config.get<number>('DB_PORT'),
        database: config.get('DB_NAME'),
      }),
      inject: [ConfigService],
    }),
  ],
})
export class AppModule {}

Re-exporting Modules

A module can re-export modules it imports, making their exports available to any module that imports it:

@Module({
  imports: [DatabaseModule, CacheModule],
  exports: [DatabaseModule, CacheModule], // Re-export both modules
})
export class CommonModule {}

Now any module importing CommonModule gets access to both DatabaseModule and CacheModule exports.

Module Organization Best Practices

Folder Structure

src/
  auth/
    auth.module.ts
    auth.controller.ts
    auth.service.ts
    guards/
    strategies/
    dto/
  users/
    users.module.ts
    users.controller.ts
    users.service.ts
    users.repository.ts
    entities/
    dto/
  shared/
    shared.module.ts
    services/
    interceptors/
    filters/
  core/
    core.module.ts       # Global, imported once in AppModule
    database/
    config/
  app.module.ts
  main.ts

Guidelines

  1. One feature, one module -- Group related controllers, services, and entities together.
  2. Export only what's needed -- Keep providers private unless other modules need them.
  3. Use CoreModule for singletons -- Database, config, logging. Import once in AppModule with @Global().
  4. Use SharedModule for utilities -- Helpers, common pipes, interceptors. Import where needed.
  5. Avoid circular imports -- If A needs B and B needs A, extract the shared part into module C.

Module Lifecycle

NestJS initializes modules in dependency order. During startup:

  1. NestJS resolves the module graph starting from AppModule
  2. Modules are initialized bottom-up (leaf modules first)
  3. Providers in each module are instantiated
  4. onModuleInit() hooks are called in dependency order
  5. onApplicationBootstrap() hooks are called after all modules are initialized
import { Module, OnModuleInit } from '@nestjs/common';

@Module({})
export class UsersModule implements OnModuleInit {
  onModuleInit() {
    console.log('UsersModule has been initialized');
  }
}

Common Pitfalls

  1. Forgetting to export -- You add a service to providers but forget to add it to exports. Other modules get a dependency resolution error.
  2. Importing vs. providing -- You can't inject a service from another module by just adding it to your providers. You must import the module that exports it.
  3. Circular module dependencies -- Module A imports Module B which imports Module A. Use forwardRef(): imports: [forwardRef(() => ModuleB)].
  4. Over-using @Global() -- Makes the dependency graph implicit and harder to debug.

Exercises

  1. Build a Multi-Module Application

    Create a NestJS app with four feature modules: UsersModule, ProductsModule, OrdersModule, and a SharedModule. OrdersModule should depend on both UsersModule and ProductsModule. SharedModule should provide a LoggerService used by all modules.

    Hint

    Generate modules with nest g mo. Remember to export services that other modules need. Import SharedModule in each feature module that needs LoggerService.

  2. Create a Dynamic Module

    Build a NotificationModule with a forRoot() method that accepts configuration (provider type: 'email' | 'sms', apiKey). The module should provide a NotificationService that uses the injected configuration.

    Hint

    Use the provide/useValue pattern to inject configuration. The forRoot() method returns a DynamicModule with the config as a provider.

  3. Implement Module Lifecycle Hooks

    Add OnModuleInit and OnApplicationBootstrap lifecycle hooks to three modules. Log messages in each to observe the initialization order. Verify that leaf modules initialize first.

    Hint

    Implement the OnModuleInit and OnApplicationBootstrap interfaces. The order should follow the dependency graph bottom-up.

Common Problems & Solutions

1 Provider from another module cannot be injected
Symptoms: Error: Nest can't resolve dependencies of the SomeService (?). Please make sure that the argument SomeProvider at index [0] is available in the SomeModule context.
Cause: The provider is registered in Module A's providers but not in its exports array. Module B imports Module A but can't access the unexported provider.
Solution: Add the provider to the `exports` array of the module that owns it. Then import that module in the consuming module's `imports` array.
2 Circular dependency between modules
Symptoms: Error: A circular dependency has been detected. Please, make sure that each side of a bidirectional relationships are decorated with 'forwardRef()'.
Cause: Module A imports Module B, and Module B imports Module A, creating a cycle the DI container can't resolve.
Solution: Use `forwardRef()` on both sides: `imports: [forwardRef(() => ModuleB)]`. Better yet, refactor the shared dependency into a third module that both can import.
3 Dynamic module configuration is undefined at runtime
Cause: The async factory function in `forRootAsync()` depends on a service from a module that isn't listed in the dynamic module's `imports`.
Solution: Include the required module in the `imports` array of the `forRootAsync()` options object, not just in the parent module's imports.
