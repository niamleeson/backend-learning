Filtering, Sorting, and Sparse Fieldsets

Intuition

Think of a restaurant menu with customization. Filtering is choosing 'only vegetarian dishes.' Sorting is asking 'order by price, cheapest first.' Sparse fieldsets is saying 'just show me the name and price, skip the description.' Together, they let each customer get exactly what they need.

Filtering, Sorting, and Sparse Fieldsets

Giving clients control over the data they receive reduces bandwidth, simplifies client-side logic, and improves performance. These three features are the building blocks of a flexible query API.

Filtering

from fastapi import FastAPI, Query
from typing import Optional

app = FastAPI()

@app.get("/products")
def list_products(
    category: Optional[str] = None,
    min_price: Optional[float] = Query(None, ge=0),
    max_price: Optional[float] = Query(None, ge=0),
    in_stock: Optional[bool] = None,
):
    query = "SELECT * FROM products WHERE 1=1"
    params = {}
    if category:
        query += " AND category = :category"
        params["category"] = category
    if min_price is not None:
        query += " AND price >= :min_price"
        params["min_price"] = min_price
    if max_price is not None:
        query += " AND price <= :max_price"
        params["max_price"] = max_price
    return db.execute(query, params).fetchall()

Sorting

ALLOWED_SORT_FIELDS = {"name", "price", "created_at"}

@app.get("/products")
def list_products(sort: str = Query("created_at"), order: str = Query("asc")):
    if sort not in ALLOWED_SORT_FIELDS:
        raise HTTPException(400, f"Cannot sort by {sort}")
    if order not in ("asc", "desc"):
        raise HTTPException(400, "Order must be asc or desc")
    return db.execute(f"SELECT * FROM products ORDER BY {sort} {order}").fetchall()

Sparse Fieldsets

@app.get("/products")
def list_products(fields: Optional[str] = None):
    all_fields = {"id", "name", "price", "category", "description"}
    if fields:
        requested = set(fields.split(","))
        selected = requested & all_fields  # whitelist only valid fields
    else:
        selected = all_fields
    cols = ", ".join(selected)
    return db.execute(f"SELECT {cols} FROM products").fetchall()

Common Mistakes

  • Allowing arbitrary sort fields without validation (SQL injection risk)
  • Not whitelisting filterable/sortable columns
  • Returning full objects when the client only needs two fields

Key Takeaways

  • Always validate and whitelist filter/sort fields against an allowlist
  • Sparse fieldsets reduce payload size and improve mobile performance
  • Combine all three for a powerful, flexible query interface

Exercises

  1. Build a Flexible Query Endpoint

    Create a /books endpoint that supports filtering by author and genre, sorting by title or publication_date, and sparse fieldsets via a ?fields=title,author query parameter. Validate all inputs against allowlists.

    Hint

    Use sets for allowlists and intersection to validate fields. Build SQL dynamically with parameterized queries to prevent injection.

  2. Filter Expression Parser

    Implement a simple filter syntax like ?filter=price:gt:10,category:eq:electronics that gets parsed into SQL WHERE clauses. Support operators: eq, gt, lt, gte, lte, like.

    Hint

    Split the filter string by comma, then split each part by colon. Map operators to SQL: gt->>, lt-><, etc. Always use parameterized queries.

Common Problems & Solutions

1 SQL injection through sort parameter
Symptoms: ['Unexpected database errors', 'Data leakage through crafted sort parameters']
Cause: The sort field from the query string was directly interpolated into SQL without validation.
Solution: Always validate sort and filter field names against a whitelist of allowed column names. Never interpolate user input directly into SQL.

DANGEROUS: direct interpolation

f"SELECT * FROM items ORDER BY {request.args['sort']}"

SAFE: whitelist validation

if sort_field not in ALLOWED_FIELDS: raise HTTPException(400, "Invalid sort field")

2 API responses are too large for mobile clients
Cause: Every response returns all fields including large text blobs, even when clients only need a few fields.
Solution: Implement sparse fieldsets so clients can request only the fields they need. Default to a sensible subset for list endpoints.
