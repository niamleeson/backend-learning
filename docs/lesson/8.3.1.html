Hashing: bcrypt, argon2, and Password Storage

Intuition

Think of password hashing like a bank vault door. A regular hash (SHA-256) is like a screen door — it technically blocks the path, but anyone with a little force (GPU power) blows right through. An adaptive hash like bcrypt or argon2 is like a massive vault door with a time-lock mechanism: even if you know exactly how the lock works, each attempt to try a combination takes a minimum amount of time, making it infeasible to try millions of combinations.

Hashing: bcrypt, argon2, and Password Storage

Password storage is one of the most critical responsibilities of any backend system. A single breach exposing poorly hashed passwords can compromise millions of users — not just on your platform, but across every service where they reused that password. In this lesson, we will explore why naive hashing fails, how modern adaptive hashing algorithms work, and how to implement robust password storage in Python.

Why Plain Hashing Fails for Passwords

You might think that running passwords through SHA-256 before storing them is sufficient. After all, cryptographic hash functions are one-way — you cannot reverse them to recover the original input. However, this reasoning has a fatal flaw: passwords have low entropy.

Most human-chosen passwords come from a relatively small space. Common passwords like password123, qwerty, and letmein appear in virtually every breach. Even passwords that seem complex often follow predictable patterns (a capital letter, lowercase letters, a number, a special character at the end).

An attacker who obtains your database of SHA-256 hashes can simply hash every candidate password and compare:

import hashlib

# An attacker's approach against plain SHA-256
def crack_sha256(target_hash, wordlist_path):
    with open(wordlist_path, 'r') as f:
        for word in f:
            candidate = word.strip()
            candidate_hash = hashlib.sha256(candidate.encode()).hexdigest()
            if candidate_hash == target_hash:
                return candidate
    return None

# SHA-256 of 'password123'
target = 'ef92b778bafe771e89245b89ecbc08a44a4e166c06659911881f383d4473e94f'
result = crack_sha256(target, '/usr/share/wordlists/rockyou.txt')
print(f"Cracked: {result}")  # Cracked: password123

Modern GPUs can compute billions of SHA-256 hashes per second. A single NVIDIA RTX 4090 can test over 20 billion MD5 hashes per second. At that rate, the entire rockyou.txt wordlist (14 million passwords) can be checked in under a millisecond.

Rainbow Tables and Precomputation Attacks

Rainbow tables take this attack further by precomputing hash values for enormous dictionaries. An attacker builds a lookup table mapping hashes back to their plaintext inputs:

e10adc3949ba59abbe56e057f20f883e -> 123456
5f4dcc3b5aa765d61d8327deb882cf99 -> password
25d55ad283aa400af464c76d713c07ad -> 12345678

Once built, cracking any hash becomes a simple table lookup — an O(1) operation. Rainbow tables for MD5 and SHA-1 covering all alphanumeric passwords up to 8-9 characters are freely available online.

Salting: The First Line of Defense

A salt is a random value appended (or prepended) to the password before hashing:

import hashlib
import os

def hash_with_salt(password: str) -> tuple[str, str]:
    salt = os.urandom(16)  # 16 bytes of cryptographic randomness
    salted = salt + password.encode()
    hashed = hashlib.sha256(salted).hexdigest()
    return salt.hex(), hashed

def verify_with_salt(password: str, salt_hex: str, expected_hash: str) -> bool:
    salt = bytes.fromhex(salt_hex)
    salted = salt + password.encode()
    return hashlib.sha256(salted).hexdigest() == expected_hash

salt, hashed = hash_with_salt('password123')
print(f"Salt: {salt}")
print(f"Hash: {hashed}")
print(f"Verify: {verify_with_salt('password123', salt, hashed)}")

Salting defeats rainbow tables because each user gets a unique salt. Even if two users have the same password, their hashes will differ. The attacker must now brute-force each hash individually.

However, salting alone is not enough. A salted SHA-256 hash is still computed in nanoseconds. An attacker targeting a single high-value account can still try billions of candidates per second.

Adaptive Hashing: Intentional Slowness

The key insight behind password hashing algorithms is that speed is the enemy. Unlike general-purpose hashing (where speed is a feature), password hashing must be deliberately slow so that brute-force attacks become impractical.

Adaptive hashing algorithms achieve this through a configurable work factor (also called cost factor) that controls how expensive the computation is. As hardware gets faster, you increase the work factor to maintain the same wall-clock time per hash.

bcrypt

bcrypt, designed in 1999 by Niels Provos and David Mazieres, was the first widely adopted adaptive password hashing function. It is based on the Blowfish cipher and has these properties:

  • Built-in salt: 128-bit random salt is generated and embedded in the output.
  • Configurable cost: The work factor is a power of 2. Cost 12 means 2^12 = 4,096 iterations of the internal key expansion.
  • Fixed output: The hash is always 60 characters in the Modular Crypt Format.
  • Maximum input length: bcrypt truncates passwords at 72 bytes. This is an important limitation.
import bcrypt

# Hashing a password with bcrypt
def hash_password_bcrypt(password: str, rounds: int = 12) -> str:
    """Hash a password using bcrypt with the specified cost factor."""
    salt = bcrypt.gensalt(rounds=rounds)
    hashed = bcrypt.hashpw(password.encode('utf-8'), salt)
    return hashed.decode('utf-8')

def verify_password_bcrypt(password: str, hashed: str) -> bool:
    """Verify a password against a bcrypt hash."""
    return bcrypt.checkpw(
        password.encode('utf-8'),
        hashed.encode('utf-8')
    )

# Usage
hashed = hash_password_bcrypt('my_secure_password')
print(f"Hash: {hashed}")
# Output: $2b$12$LJ3m4ys3Lg2VBe5E8IPVR.eTONBkBmGxqFC0krJTOxbMYBNaMVWmu

print(f"Valid: {verify_password_bcrypt('my_secure_password', hashed)}")
print(f"Invalid: {verify_password_bcrypt('wrong_password', hashed)}")

The output format $2b$12$... encodes the algorithm version (2b), cost factor (12), and the salt+hash together. This self-describing format is a best practice — you can always determine how to verify a hash by inspecting it.

scrypt

scrypt (2009, Colin Percival) was designed to be memory-hard in addition to CPU-hard. While bcrypt's cost is primarily computational, scrypt forces the attacker to use large amounts of RAM, making GPU and ASIC attacks more expensive:

import hashlib
import os

def hash_password_scrypt(password: str) -> str:
    salt = os.urandom(16)
    hashed = hashlib.scrypt(
        password.encode('utf-8'),
        salt=salt,
        n=2**14,   # CPU/memory cost (must be power of 2)
        r=8,       # Block size
        p=1,       # Parallelism
        dklen=32   # Output length
    )
    return f"{salt.hex()}${hashed.hex()}"

def verify_password_scrypt(password: str, stored: str) -> bool:
    salt_hex, hash_hex = stored.split('$')
    salt = bytes.fromhex(salt_hex)
    expected = bytes.fromhex(hash_hex)
    hashed = hashlib.scrypt(
        password.encode('utf-8'),
        salt=salt,
        n=2**14, r=8, p=1, dklen=32
    )
    return hashed == expected

argon2: The Modern Standard

Argon2 won the Password Hashing Competition (PHC) in 2015 and is the recommended algorithm for new applications. It comes in three variants:

  • Argon2d: Data-dependent memory access. Maximum resistance to GPU cracking, but vulnerable to side-channel attacks.
  • Argon2i: Data-independent memory access. Resistant to side-channel attacks. Good for password hashing.
  • Argon2id: A hybrid — uses Argon2i for the first pass and Argon2d for subsequent passes. This is the recommended variant.

Argon2 has three tunable parameters: - Time cost (t): Number of iterations. - Memory cost (m): Memory usage in kibibytes. - Parallelism (p): Number of threads.

from argon2 import PasswordHasher, exceptions

# Create a hasher with tuned parameters
ph = PasswordHasher(
    time_cost=3,         # 3 iterations
    memory_cost=65536,   # 64 MB
    parallelism=4,       # 4 threads
    hash_len=32,         # 32-byte output
    salt_len=16,         # 16-byte salt
    type=argon2.Type.ID  # Argon2id variant
)

# Hash a password
hashed = ph.hash('my_secure_password')
print(f"Hash: {hashed}")
# Output: $argon2id$v=19$m=65536,t=3,p=4$c29tZXNhbHQ$...

# Verify a password
try:
    ph.verify(hashed, 'my_secure_password')
    print("Password is correct")
except exceptions.VerifyMismatchError:
    print("Password is incorrect")

# Check if rehashing is needed (parameters changed)
if ph.check_needs_rehash(hashed):
    new_hash = ph.hash('my_secure_password')
    # Update the stored hash in your database

The check_needs_rehash method is particularly valuable — it tells you if a hash was created with older, weaker parameters so you can transparently upgrade on the next successful login.

Work Factor Tuning

Choosing the right work factor is a balance between security and user experience. The goal is to make each hash take 250ms to 1 second on your production hardware. Here is how to benchmark:

import time
import bcrypt
from argon2 import PasswordHasher

def benchmark_bcrypt():
    password = b'benchmark_password_123'
    print("bcrypt benchmarks:")
    for rounds in range(10, 16):
        start = time.perf_counter()
        bcrypt.hashpw(password, bcrypt.gensalt(rounds=rounds))
        elapsed = time.perf_counter() - start
        print(f"  rounds={rounds}: {elapsed:.3f}s")

def benchmark_argon2():
    password = 'benchmark_password_123'
    print("\nargon2id benchmarks:")
    configs = [
        {'time_cost': 2, 'memory_cost': 32768, 'parallelism': 2},
        {'time_cost': 3, 'memory_cost': 65536, 'parallelism': 4},
        {'time_cost': 4, 'memory_cost': 131072, 'parallelism': 4},
        {'time_cost': 5, 'memory_cost': 262144, 'parallelism': 8},
    ]
    for cfg in configs:
        ph = PasswordHasher(**cfg)
        start = time.perf_counter()
        ph.hash(password)
        elapsed = time.perf_counter() - start
        print(f"  t={cfg['time_cost']}, m={cfg['memory_cost']}KB, "
              f"p={cfg['parallelism']}: {elapsed:.3f}s")

benchmark_bcrypt()
benchmark_argon2()

As a starting point for 2025-era hardware: - bcrypt: cost factor 12-14 - argon2id: time_cost=3, memory_cost=65536 (64MB), parallelism=4

Re-benchmark annually and increase parameters as hardware improves.

Timing Attacks on Password Comparison

A subtle but real vulnerability arises when comparing hashes using standard string equality:

# VULNERABLE to timing attack
def verify_naive(stored_hash: str, computed_hash: str) -> bool:
    return stored_hash == computed_hash  # Short-circuits on first mismatch!

Python's == operator for strings returns False as soon as it finds the first differing character. An attacker can measure response times to deduce hash values one character at a time.

The fix is constant-time comparison:

import hmac

# SAFE: constant-time comparison
def verify_safe(stored_hash: str, computed_hash: str) -> bool:
    return hmac.compare_digest(
        stored_hash.encode('utf-8'),
        computed_hash.encode('utf-8')
    )

hmac.compare_digest always compares every byte regardless of where differences occur. The good news is that bcrypt and argon2-cffi both use constant-time comparison internally, so you are protected when using their built-in verify functions. But if you ever roll your own comparison logic, always use hmac.compare_digest.

Using passlib for a Unified Interface

passlib is a comprehensive password hashing library that provides a unified interface across many algorithms and includes built-in best practices:

from passlib.context import CryptContext

# Define a context with preferred and deprecated schemes
pwd_context = CryptContext(
    schemes=['argon2', 'bcrypt'],
    default='argon2',
    argon2__time_cost=3,
    argon2__memory_cost=65536,
    argon2__parallelism=4,
    bcrypt__rounds=12,
    deprecated=['bcrypt'],  # Mark bcrypt as needing upgrade
)

# Hash a new password (uses argon2 by default)
hashed = pwd_context.hash('my_password')
print(hashed)  # $argon2id$v=19$m=65536,t=3,p=4$...

# Verify — works with both argon2 and bcrypt hashes
valid = pwd_context.verify('my_password', hashed)
print(f"Valid: {valid}")

# Check if the hash needs upgrading
valid, new_hash = pwd_context.verify_and_update('my_password', hashed)
if new_hash:
    # The hash was using a deprecated scheme or outdated parameters.
    # Save new_hash to the database.
    print(f"Upgraded hash: {new_hash}")

This pattern is especially powerful for migration — the deprecated list combined with verify_and_update lets you transparently upgrade users from old algorithms to new ones at login time.

Password Storage Best Practices

Here is a complete, production-ready password service class:

import secrets
import hmac
from argon2 import PasswordHasher, exceptions
from argon2 import Type as Argon2Type


class PasswordService:
    """Production-grade password hashing service."""

    def __init__(self):
        self._hasher = PasswordHasher(
            time_cost=3,
            memory_cost=65536,
            parallelism=4,
            hash_len=32,
            salt_len=16,
            type=Argon2Type.ID,
        )
        # Pepper: a secret key stored outside the database
        # Load from environment variable or secrets manager
        self._pepper = self._load_pepper()

    def _load_pepper(self) -> bytes:
        """Load pepper from secure configuration."""
        import os
        pepper = os.environ.get('PASSWORD_PEPPER')
        if not pepper:
            raise RuntimeError('PASSWORD_PEPPER environment variable not set')
        return pepper.encode('utf-8')

    def _apply_pepper(self, password: str) -> str:
        """Apply HMAC-based pepper to password before hashing."""
        return hmac.new(
            self._pepper,
            password.encode('utf-8'),
            'sha256'
        ).hexdigest()

    def hash_password(self, password: str) -> str:
        """Hash a password for storage."""
        peppered = self._apply_pepper(password)
        return self._hasher.hash(peppered)

    def verify_password(self, password: str, hash_str: str) -> bool:
        """Verify a password against a stored hash."""
        peppered = self._apply_pepper(password)
        try:
            self._hasher.verify(hash_str, peppered)
            return True
        except exceptions.VerifyMismatchError:
            return False
        except exceptions.InvalidHashError:
            # Corrupted hash in database
            return False

    def needs_rehash(self, hash_str: str) -> bool:
        """Check if hash needs upgrading to current parameters."""
        return self._hasher.check_needs_rehash(hash_str)

Key points from this implementation:

  1. Pepper: An additional secret applied via HMAC before hashing. Unlike the salt (stored with the hash), the pepper is stored separately (environment variable, HSM, or secrets manager). If the database is stolen but the pepper is not, the hashes are even harder to crack.

  2. HMAC for peppering: We use HMAC-SHA256 rather than simple concatenation to avoid length-extension attacks and to produce uniform-length input to argon2 (which avoids the 72-byte bcrypt truncation issue if you later switch algorithms).

  3. Rehash detection: On every successful login, check needs_rehash and transparently upgrade.

Migrating from Weak Hashes

If you inherit a system using unsalted MD5 or SHA-1, you need a migration strategy. The safest approach is hash wrapping — apply the new algorithm on top of the old hash without requiring users to log in:

import hashlib
from argon2 import PasswordHasher

ph = PasswordHasher(time_cost=3, memory_cost=65536, parallelism=4)

def migrate_md5_hash(old_md5_hex: str) -> str:
    """
    Wrap an existing MD5 hash with argon2id.
    The stored value becomes argon2id(md5(password)).
    """
    return ph.hash(old_md5_hex)

def verify_migrated(password: str, wrapped_hash: str) -> bool:
    """
    Verify against a wrapped hash:
    Compute md5(password), then verify with argon2.
    """
    md5_hex = hashlib.md5(password.encode()).hexdigest()
    try:
        ph.verify(wrapped_hash, md5_hex)
        return True
    except Exception:
        return False

# Step 1: Batch migration (no user action needed)
# For each row: new_hash = migrate_md5_hash(old_md5_hash)

# Step 2: On successful login, re-hash with pure argon2
def login_and_upgrade(password: str, stored_hash: str, is_wrapped: bool):
    if is_wrapped:
        if verify_migrated(password, stored_hash):
            # Upgrade to pure argon2id
            new_hash = ph.hash(password)
            # UPDATE users SET password_hash = new_hash, is_wrapped = FALSE
            return True, new_hash
    else:
        try:
            ph.verify(stored_hash, password)
            return True, None
        except Exception:
            pass
    return False, None

This two-phase approach provides immediate protection (wrapping) while gradually migrating to pure hashes as users log in. For accounts that never log in again, you still have argon2id protecting the weak MD5 hash.

Database Schema Considerations

Store enough metadata to support algorithm evolution:

CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(512) NOT NULL,  -- Generous length for future algorithms
    hash_scheme VARCHAR(50) DEFAULT 'argon2id',
    hash_version INT DEFAULT 1,          -- Your internal versioning
    password_changed_at TIMESTAMPTZ DEFAULT NOW(),
    created_at TIMESTAMPTZ DEFAULT NOW()
);

Using VARCHAR(512) for the hash field gives room for any algorithm. Argon2id hashes are typically around 97 characters, but future algorithms may be longer. The hash_scheme and hash_version columns make migration logic explicit and queryable — you can easily find users still on old schemes.

Summary of Algorithm Recommendations

Scenario Recommended Algorithm Notes
New application Argon2id PHC winner, memory-hard, modern
Legacy system upgrade bcrypt (then argon2id) Widely supported, well-understood
High-security / compliance Argon2id with pepper Add HSM-backed pepper for defense in depth
Constrained environment bcrypt Lower memory requirements than argon2

Never use MD5, SHA-1, SHA-256, or any general-purpose hash function for passwords, even with a salt. They are too fast. Always use an adaptive, purpose-built password hashing algorithm with a work factor tuned to your hardware.

Common Problems & Solutions

1 bcrypt silently truncates long passwords
Symptoms: Users with very long passwords (e.g., generated by password managers) can authenticate with a truncated version of their password. Security audit flags that different passwords produce identical hashes.
Cause: bcrypt has a maximum input length of 72 bytes. Any password longer than 72 bytes is truncated before hashing. This means two passwords that share the same first 72 bytes but differ afterward will produce identical hashes.
Solution: Pre-hash the password with SHA-256 (or HMAC-SHA-256 with a pepper) before passing it to bcrypt. This produces a fixed-length 64-character hex string that is always under the 72-byte limit while preserving the entropy of longer passwords.

import hashlib import bcrypt

def hash_password_safe(password: str) -> str: # Pre-hash to handle arbitrary length passwords pre_hashed = hashlib.sha256(password.encode('utf-8')).hexdigest() return bcrypt.hashpw( pre_hashed.encode('utf-8'), bcrypt.gensalt(rounds=12) ).decode('utf-8')

Without pre-hashing, these would produce the same hash:

long_pw_a = 'a' * 72 + 'DIFFERENT_SUFFIX_A' long_pw_b = 'a' * 72 + 'DIFFERENT_SUFFIX_B'

hash_a = hash_password_safe(long_pw_a) hash_b = hash_password_safe(long_pw_b) assert not bcrypt.checkpw( hashlib.sha256(long_pw_b.encode()).hexdigest().encode(), hash_a.encode() ) # Correctly distinguishes the two passwords

2 Login endpoint becomes a DoS vector due to expensive hashing
Cause: Adaptive hashing is intentionally slow. An attacker can send thousands of login requests with random passwords, forcing the server to spend significant CPU time on hashing. With argon2's memory cost, this can also exhaust RAM.
Solution: Implement rate limiting on the login endpoint (per IP and per account), use a CAPTCHA after failed attempts, and consider placing hash computation behind an async task queue or a dedicated authentication service with its own resource limits. Also set a maximum password length (e.g., 128 characters) to prevent attackers from submitting extremely long inputs.

from fastapi import FastAPI, Request, HTTPException from slowapi import Limiter from slowapi.util import get_remote_address

app = FastAPI() limiter = Limiter(key_func=get_remote_address)

MAX_PASSWORD_LENGTH = 128

@app.post('/login') @limiter.limit('10/minute') # Max 10 attempts per IP per minute async def login(request: Request, email: str, password: str): if len(password) > MAX_PASSWORD_LENGTH: raise HTTPException(400, 'Password too long') # ... proceed with verification

3 Hash verification fails after migrating between systems or languages
Cause: Different bcrypt implementations use different version prefixes ($2a$, $2b$, $2y$). Some libraries only accept specific prefixes. Additionally, character encoding differences (UTF-8 vs. Latin-1) between the original hashing system and the new one cause different byte sequences for non-ASCII passwords.
Solution: Normalize all password inputs to UTF-8 with NFC Unicode normalization before hashing. For bcrypt prefix issues, replace the prefix string when loading hashes (e.g., $2y$ to $2b$). Test cross-platform compatibility thoroughly before migrating.

import unicodedata

def normalize_password(password: str) -> bytes: """Normalize password to NFC UTF-8 for consistent cross-platform hashing.""" normalized = unicodedata.normalize('NFC', password) return normalized.encode('utf-8')

def fix_bcrypt_prefix(hash_str: str) -> str: """Convert PHP-style $2y$ prefix to Python-compatible $2b$ prefix.""" if hash_str.startswith('$2y$'): return '$2b$' + hash_str[4:] return hash_str
