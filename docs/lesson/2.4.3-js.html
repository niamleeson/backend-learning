Transactions and Raw Queries

Intuition

A transaction is like writing a check at a restaurant. Either the check clears (commit) and the restaurant gets paid, or it bounces (rollback) and nothing changes. You don't want a half-cleared check where the money left your account but never reached the restaurant. Transactions guarantee this all-or-nothing behavior.

Transactions and Raw Queries

Introduction

Transactions group multiple database operations into a single atomic unit — either ALL operations succeed, or NONE of them do. This is critical for data consistency. Imagine transferring money between accounts: you must debit one account and credit another. If the credit fails after the debit succeeds, you've lost money. Transactions prevent this.

ACID Properties

A - Atomicity:    All or nothing. If any operation fails, everything rolls back.
C - Consistency:  Database moves from one valid state to another.
I - Isolation:    Concurrent transactions don't interfere with each other.
D - Durability:   Once committed, data survives crashes.

Knex Transactions

Basic Transaction

try {
  const result = await knex.transaction(async (trx) => {
    // All queries use trx instead of knex
    const [order] = await trx('orders')
      .insert({ user_id: userId, status: 'pending', total: 0 })
      .returning('*');

    let total = 0;
    for (const item of items) {
      // Check stock
      const product = await trx('products')
        .where('id', item.productId)
        .first();

      if (product.stock_quantity < item.quantity) {
        throw new Error(`Insufficient stock for ${product.name}`);
      }

      // Create order item
      await trx('order_items').insert({
        order_id: order.id,
        product_id: item.productId,
        quantity: item.quantity,
        unit_price: product.price,
      });

      // Decrement stock
      await trx('products')
        .where('id', item.productId)
        .decrement('stock_quantity', item.quantity);

      total += product.price * item.quantity;
    }

    // Update order total
    const [finalOrder] = await trx('orders')
      .where('id', order.id)
      .update({ total })
      .returning('*');

    return finalOrder;
    // Transaction auto-commits when the callback resolves
  });

  console.log('Order created:', result);
} catch (err) {
  // Transaction auto-rolled back on error
  console.error('Order failed:', err.message);
}

Transaction Flow:

  BEGIN
    |
    +-- INSERT order
    +-- INSERT order_items (loop)
    +-- UPDATE products (decrement stock)
    +-- UPDATE order (set total)
    |
  COMMIT (if all succeed)
    or
  ROLLBACK (if any throw)

Manual Transaction Control

const trx = await knex.transaction();

try {
  await trx('accounts')
    .where('id', fromAccountId)
    .decrement('balance', amount);

  await trx('accounts')
    .where('id', toAccountId)
    .increment('balance', amount);

  await trx('transfers').insert({
    from_account: fromAccountId,
    to_account: toAccountId,
    amount,
  });

  await trx.commit();
} catch (err) {
  await trx.rollback();
  throw err;
}

Savepoints (Nested Transactions)

await knex.transaction(async (trx) => {
  await trx('users').insert({ name: 'Alice' });

  try {
    // Savepoint - partial rollback possible
    await trx.transaction(async (trx2) => {
      await trx2('profiles').insert({ user_id: 1, bio: 'Hello' });
      throw new Error('Profile creation failed');
    });
  } catch (err) {
    // Only the inner transaction rolled back
    // Outer transaction continues
    console.log('Profile failed, but user still created');
  }

  // This still commits
  await trx('logs').insert({ message: 'User created without profile' });
});

Row-Level Locking

await knex.transaction(async (trx) => {
  // FOR UPDATE - lock the row so no one else can modify it
  const product = await trx('products')
    .where('id', productId)
    .forUpdate()  // SELECT ... FOR UPDATE
    .first();

  if (product.stock_quantity >= requestedQuantity) {
    await trx('products')
      .where('id', productId)
      .decrement('stock_quantity', requestedQuantity);
  } else {
    throw new Error('Out of stock');
  }
});

Raw Queries

knex.raw()

// Simple raw query
const result = await knex.raw(
  'SELECT * FROM users WHERE email = ?',
  ['alice@example.com']
);
console.log(result.rows);

// Raw with named bindings
const result2 = await knex.raw(
  'SELECT * FROM users WHERE age > :minAge AND city = :city',
  { minAge: 21, city: 'New York' }
);

Raw Expressions in Query Builder

// Raw in select
await knex('orders')
  .select(
    'user_id',
    knex.raw('EXTRACT(MONTH FROM created_at) AS month'),
    knex.raw('SUM(total)::numeric(12,2) AS revenue')
  )
  .groupBy('user_id', knex.raw('EXTRACT(MONTH FROM created_at)'));

// Raw in where
await knex('users')
  .whereRaw('created_at > NOW() - INTERVAL ?', ['30 days']);

// Raw in orderBy
await knex('products')
  .orderByRaw('CASE WHEN featured = true THEN 0 ELSE 1 END, price ASC');

// Raw in join
await knex('orders')
  .joinRaw('LEFT JOIN LATERAL (SELECT ...) AS recent ON true');

CTEs with Knex

const report = await knex
  .with('monthly_sales', (qb) => {
    qb.select(
      knex.raw("TO_CHAR(created_at, 'YYYY-MM') AS month"),
      knex.raw('SUM(total) AS revenue'),
      knex.raw('COUNT(*) AS order_count')
    )
    .from('orders')
    .where('status', 'completed')
    .groupByRaw("TO_CHAR(created_at, 'YYYY-MM')");
  })
  .select('*')
  .from('monthly_sales')
  .orderBy('month', 'desc');

Transaction Best Practices

1. Keep transactions SHORT - long transactions hold locks
2. Don't do I/O (HTTP calls, file reads) inside transactions
3. Use FOR UPDATE for read-modify-write patterns
4. Handle deadlocks with retry logic
5. Set a statement_timeout for safety

// Retry logic for deadlocks
async function withRetry(fn, maxRetries = 3) {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      const isDeadlock = err.code === '40P01';
      const isSerializationFailure = err.code === '40001';
      if ((isDeadlock || isSerializationFailure) && attempt < maxRetries) {
        const delay = Math.pow(2, attempt) * 100; // exponential backoff
        await new Promise(r => setTimeout(r, delay));
        continue;
      }
      throw err;
    }
  }
}

// Usage
const order = await withRetry(() =>
  knex.transaction(async (trx) => {
    // ... order creation logic
  })
);

Exercises

  1. Transfer Funds

    Implement a transferFunds(fromId, toId, amount) function using a Knex transaction. It should: verify the source account has sufficient funds, debit the source, credit the destination, create a transfer record, and return the transfer details. Handle all error cases.

    Hint

    Use .forUpdate() to lock both accounts. Check balance >= amount. Use trx for all queries. Throw descriptive errors for insufficient funds or missing accounts.

  2. Batch Import with Rollback

    Write a function that imports a CSV of products into the database within a transaction. If any row fails validation (missing name, negative price), roll back the entire import and return which rows failed.

    Hint

    Parse the CSV first, validate all rows, then insert within a transaction. Collect errors for each row and throw if any exist.

  3. Deadlock Recovery

    Create a scenario that intentionally causes a deadlock (two transactions updating the same rows in different orders), then implement retry logic to handle it gracefully.

    Hint

    Start two transactions concurrently: one updates row A then B, the other updates B then A. Catch error code 40P01 and retry with exponential backoff.

Common Problems & Solutions

1 Transaction holds locks for too long, blocking other queries
Cause: Doing slow operations (HTTP calls, file I/O, complex computation) inside a transaction.
Solution: Prepare all data BEFORE starting the transaction. Inside the transaction, only do database operations. Set a statement_timeout: `await trx.raw('SET LOCAL statement_timeout = 5000')`.
2 Deadlock detected
Cause: Two transactions are waiting for each other's locks. Transaction A locks row 1 and wants row 2; Transaction B locks row 2 and wants row 1.
Solution: Always access rows in a consistent order (e.g., by ascending ID). Implement retry logic with exponential backoff for deadlock errors (code 40P01).
3 Transaction silently commits when it should have rolled back
Cause: Using `knex` instead of `trx` for some queries inside the transaction callback. Those queries run outside the transaction.
Solution: Always use the `trx` object passed to the callback, never the global `knex` instance. Lint for this by searching for `knex(` inside transaction callbacks.
