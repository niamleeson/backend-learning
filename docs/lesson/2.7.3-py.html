Optimization Patterns: Caching, Lazy Evaluation, Slots

Intuition

slots removes the overhead backpack each object carries. Caching is a cheat sheet for answers you have already computed. Lazy evaluation is procrastination done right: only do work when someone actually needs the result.

Optimization Patterns: Caching, Lazy Evaluation, and slots

Python optimization focuses on algorithmic improvements, reducing unnecessary work, and minimizing memory overhead through proven patterns.

from functools import lru_cache

# Caching: avoid redundant computation
@lru_cache(maxsize=256)
def expensive_query(user_id: int) -> dict:
    # Simulates a slow database call
    import time; time.sleep(0.01)
    return {"id": user_id, "name": f"User {user_id}"}

# Lazy evaluation: compute only when needed
class LazyProperty:
    def __init__(self, func):
        self.func = func
        self.attr_name = f"_lazy_{func.__name__}"

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        if not hasattr(obj, self.attr_name):
            setattr(obj, self.attr_name, self.func(obj))
        return getattr(obj, self.attr_name)

class Report:
    def __init__(self, data: list[int]):
        self.data = data

    @LazyProperty
    def summary(self) -> dict:
        print("Computing summary...")  # Only runs once
        return {"mean": sum(self.data) / len(self.data),
                "total": sum(self.data)}

# __slots__: reduce memory per instance
class Point:
    __slots__ = ('x', 'y')
    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y
# ~40-50% less memory than a regular class

Key Takeaways: - lru_cache eliminates repeated expensive computations - Lazy evaluation defers work until the result is actually needed - __slots__ eliminates per-instance __dict__, saving significant memory - Use generators instead of lists for large sequences processed once - Profile first, then optimize the proven bottleneck; premature optimization wastes effort

Exercises

  1. Implement a TTL Cache

    Build a cache decorator that stores results for a configurable time-to-live (TTL in seconds). After the TTL, the cached value expires and is recomputed on next access.

    Hint

    Store (result, timestamp) tuples. On access, check if time.time() - timestamp > ttl. If expired, recompute and update.

  2. Slots vs Dict Benchmark

    Create two versions of a class (with and without __slots__). Instantiate 1 million objects of each. Measure memory usage with tracemalloc and access time with timeit.

    Hint

    Use tracemalloc.get_traced_memory() for peak memory. __slots__ should use ~40% less memory. Attribute access is also slightly faster.

Common Problems & Solutions

1 `lru_cache` causes memory to grow unboundedly in long-running processes.
Cause: Cached results are never evicted if `maxsize=None`, or the cache holds references to large objects.
Solution: Set an appropriate `maxsize`. Call `func.cache_clear()` periodically. Consider a TTL cache for time-sensitive data.
2 Cannot add new attributes to a class that uses `__slots__`.
Cause: `__slots__` restricts instances to only the declared attributes, preventing dynamic attribute creation.
Solution: Add all needed attributes to `__slots__`. If you need flexibility, include `'__dict__'` in `__slots__` (but this negates the memory savings).
