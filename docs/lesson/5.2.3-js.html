Subscriptions, DataLoaders, and N+1 Prevention

Intuition

DataLoader is like a mail carrier who waits until everyone on the street has put their letters in the mailbox, then does one trip to the post office with all the letters -- instead of making a separate trip for each letter. Subscriptions are like subscribing to a newspaper: instead of checking the newsstand every hour, the paper is delivered to your door whenever a new edition is printed.

Subscriptions, DataLoaders, and N+1 Prevention

This lesson tackles two critical GraphQL challenges: real-time data with subscriptions and the infamous N+1 query problem solved by DataLoader.

The N+1 Problem

Consider this query:

query {
  posts(limit: 10) {
    title
    author {
      name
    }
  }
}

Without DataLoader, this executes:

1 query:  SELECT * FROM posts LIMIT 10
10 queries: SELECT * FROM users WHERE id = 1
            SELECT * FROM users WHERE id = 2
            SELECT * FROM users WHERE id = 1  (duplicate!)
            SELECT * FROM users WHERE id = 3
            ...etc
= 11 queries total (1 + N)

DataLoader: Batching and Caching

DataLoader collects all individual requests within a single tick of the event loop and batches them into one query.

import DataLoader from 'dataloader';
import { PrismaClient } from '@prisma/client';

// Batch function: receives array of keys, returns array of results in same order
async function batchUsers(prisma: PrismaClient, ids: readonly string[]) {
  const users = await prisma.user.findMany({
    where: { id: { in: [...ids] } },
  });

  // IMPORTANT: Return results in the SAME ORDER as the input ids
  const userMap = new Map(users.map(u => [u.id, u]));
  return ids.map(id => userMap.get(id) || null);
}

// Create loaders (one per request!)
function createLoaders(prisma: PrismaClient) {
  return {
    userLoader: new DataLoader<string, User | null>(
      (ids) => batchUsers(prisma, ids)
    ),

    postsByAuthorLoader: new DataLoader<string, Post[]>(
      async (authorIds) => {
        const posts = await prisma.post.findMany({
          where: { authorId: { in: [...authorIds] } },
        });
        const grouped = new Map<string, Post[]>();
        for (const post of posts) {
          if (!grouped.has(post.authorId)) grouped.set(post.authorId, []);
          grouped.get(post.authorId)!.push(post);
        }
        return authorIds.map(id => grouped.get(id) || []);
      }
    ),

    commentsByPostLoader: new DataLoader<string, Comment[]>(
      async (postIds) => {
        const comments = await prisma.comment.findMany({
          where: { postId: { in: [...postIds] } },
        });
        const grouped = new Map<string, Comment[]>();
        for (const comment of comments) {
          if (!grouped.has(comment.postId)) grouped.set(comment.postId, []);
          grouped.get(comment.postId)!.push(comment);
        }
        return postIds.map(id => grouped.get(id) || []);
      }
    ),
  };
}

Integrating DataLoader with Context

app.use(
  '/graphql',
  express.json(),
  expressMiddleware(server, {
    context: async ({ req }) => {
      // Create NEW loaders for each request (caching is per-request)
      return {
        userId: extractUserId(req),
        prisma,
        loaders: createLoaders(prisma),
      };
    },
  })
);

// Use loaders in resolvers
const resolvers = {
  Post: {
    author: (post, _args, context) => {
      return context.loaders.userLoader.load(post.authorId);
    },
    comments: (post, _args, context) => {
      return context.loaders.commentsByPostLoader.load(post.id);
    },
  },
  User: {
    posts: (user, _args, context) => {
      return context.loaders.postsByAuthorLoader.load(user.id);
    },
  },
};

Now the same query executes:

1 query: SELECT * FROM posts LIMIT 10
1 query: SELECT * FROM users WHERE id IN (1, 2, 3)  -- batched & deduplicated!
= 2 queries total

DataLoader Execution Flow

Tick 1: Resolve post[0].author -> loader.load('user-1')
         Resolve post[1].author -> loader.load('user-2')
         Resolve post[2].author -> loader.load('user-1')  // cached!
         ...all 10 posts queue loads

Tick 2: DataLoader executes batch function
         batchUsers(['user-1', 'user-2', 'user-3'])
         One SQL query, results distributed to all waiting resolvers

GraphQL Subscriptions

Subscriptions enable real-time, server-pushed updates over WebSocket connections.

type Subscription {
  postCreated: Post!
  commentAdded(postId: ID!): Comment!
  userStatusChanged(userId: ID!): UserStatus!
}

type UserStatus {
  userId: ID!
  online: Boolean!
  lastSeen: DateTime!
}

Setting Up Subscriptions with graphql-ws

import { ApolloServer } from '@apollo/server';
import { expressMiddleware } from '@apollo/server/express4';
import { createServer } from 'http';
import { WebSocketServer } from 'ws';
import { useServer } from 'graphql-ws/lib/use/ws';
import { makeExecutableSchema } from '@graphql-tools/schema';
import { PubSub } from 'graphql-subscriptions';

const pubsub = new PubSub();

const schema = makeExecutableSchema({ typeDefs, resolvers });

// HTTP server for queries and mutations
const app = express();
const httpServer = createServer(app);

// WebSocket server for subscriptions
const wsServer = new WebSocketServer({
  server: httpServer,
  path: '/graphql',
});

const wsServerCleanup = useServer(
  {
    schema,
    context: async (ctx) => {
      // Authenticate WebSocket connections
      const token = ctx.connectionParams?.authorization as string;
      const userId = token ? verifyToken(token) : null;
      return { userId, prisma, loaders: createLoaders(prisma) };
    },
    onConnect: async (ctx) => {
      console.log('Client connected');
      // Return false to reject the connection
    },
    onDisconnect: async (ctx) => {
      console.log('Client disconnected');
    },
  },
  wsServer
);

const server = new ApolloServer({
  schema,
  plugins: [
    {
      async serverWillStart() {
        return {
          async drainServer() {
            await wsServerCleanup.dispose();
          },
        };
      },
    },
  ],
});

await server.start();
app.use('/graphql', express.json(), expressMiddleware(server, { context: async ({ req }) => ({ /* ... */ }) }));
httpServer.listen(4000);

Subscription Resolvers with PubSub

const EVENTS = {
  POST_CREATED: 'POST_CREATED',
  COMMENT_ADDED: 'COMMENT_ADDED',
  USER_STATUS_CHANGED: 'USER_STATUS_CHANGED',
};

const resolvers = {
  Subscription: {
    postCreated: {
      subscribe: () => pubsub.asyncIterableIterator([EVENTS.POST_CREATED]),
    },

    commentAdded: {
      subscribe: (_parent, args) => {
        // Filter: only get comments for a specific post
        return pubsub.asyncIterableIterator([`${EVENTS.COMMENT_ADDED}.${args.postId}`]);
      },
    },

    userStatusChanged: {
      subscribe: (_parent, args) => {
        return pubsub.asyncIterableIterator([`${EVENTS.USER_STATUS_CHANGED}.${args.userId}`]);
      },
    },
  },

  Mutation: {
    createPost: async (_parent, args, context) => {
      const post = await context.prisma.post.create({
        data: { ...args.input, authorId: context.userId },
        include: { author: true },
      });

      // Publish event to all subscribers
      pubsub.publish(EVENTS.POST_CREATED, { postCreated: post });

      return post;
    },

    addComment: async (_parent, args, context) => {
      const comment = await context.prisma.comment.create({
        data: {
          text: args.text,
          postId: args.postId,
          authorId: context.userId,
        },
        include: { author: true },
      });

      // Publish to post-specific channel
      pubsub.publish(`${EVENTS.COMMENT_ADDED}.${args.postId}`, {
        commentAdded: comment,
      });

      return comment;
    },
  },
};

Production PubSub with Redis

The in-memory PubSub only works for single-server setups. For production, use Redis:

import { RedisPubSub } from 'graphql-redis-subscriptions';
import Redis from 'ioredis';

const pubsub = new RedisPubSub({
  publisher: new Redis({ host: 'localhost', port: 6379 }),
  subscriber: new Redis({ host: 'localhost', port: 6379 }),
});

Filtering Subscriptions

Use withFilter for fine-grained subscription filtering:

import { withFilter } from 'graphql-subscriptions';

const resolvers = {
  Subscription: {
    commentAdded: {
      subscribe: withFilter(
        () => pubsub.asyncIterableIterator([EVENTS.COMMENT_ADDED]),
        (payload, variables) => {
          // Only send to clients watching this specific post
          return payload.commentAdded.postId === variables.postId;
        }
      ),
    },
  },
};

Common Problems & Solutions

1 DataLoader returns results in wrong order
Cause: The batch function does not return results in the same order as the input keys.
Solution: Create a Map from the batch query results, then map over the input keys to return results in order: keys.map(id => resultMap.get(id) || null).
2 Subscriptions work in development but not in production
Cause: The in-memory PubSub does not work across multiple server instances.
Solution: Use graphql-redis-subscriptions with a shared Redis instance. All server instances publish and subscribe through Redis.
3 DataLoader cache returns stale data after mutations
Cause: DataLoader caches results per request, but if a mutation updates a record and the same request queries it again, the cached (old) version is returned.
Solution: Clear specific DataLoader keys after mutations: context.loaders.userLoader.clear(userId). Or create fresh loaders per request (which is the recommended pattern).
