Cache Invalidation and TTL Strategies

Intuition

TTL is like milk with an expiration date—you throw it out after a set time regardless. Event-driven invalidation is like checking the milk by smelling it—you discard it only when you know it has gone bad.

Cache Invalidation & TTL Strategies

Cache invalidation is one of the hardest problems in computer science. Getting it wrong leads to stale data, inconsistent user experiences, or cache stampedes that overwhelm your database.

Core Concepts

  • Time-To-Live (TTL): The simplest invalidation strategy. Each cache entry expires after a fixed duration. Short TTLs mean fresher data but lower hit rates; long TTLs mean higher hit rates but staler data.
  • Event-Driven Invalidation: When data changes, explicitly delete or update the corresponding cache entry. More complex but ensures freshness.
  • Cache Stampede Prevention: When a popular key expires, hundreds of requests may simultaneously hit the database. Techniques include mutex locks, probabilistic early expiration (XFetch), and background refresh.
  • Versioned Keys: Append a version number to cache keys (e.g., user:42:v3). On update, increment the version so old cached data is naturally abandoned.
  • Tag-Based Invalidation: Group related cache entries with tags. When a dependency changes, invalidate all entries with that tag (e.g., invalidate all product cache entries when inventory updates).

Key Takeaways

  1. TTL is the simplest and most common strategy but often insufficient for rapidly changing data.
  2. Event-driven invalidation provides stronger consistency but requires reliable event delivery.
  3. Always plan for cache stampede protection on high-traffic keys.

Exercises

  1. Implement Cache Stampede Protection

    Write a cache-aside function that uses a Redis lock (SETNX with expiry) to prevent cache stampedes. When a key expires, only one process should fetch from the DB; others wait and retry reading from cache.

    Hint

    Use redis.set(lock_key, 1, nx=True, ex=5) to acquire a lock. If the lock is not acquired, sleep briefly and retry redis.get(cache_key). The lock holder populates the cache and releases the lock.

  2. Build Tag-Based Cache Invalidation

    Implement a system where cache entries can be tagged. Store tag-to-key mappings in Redis sets. Write an invalidate_tag(tag) function that deletes all cache entries associated with the given tag.

    Hint

    When caching, do redis.sadd(f'tag:{tag}', cache_key) for each tag. To invalidate, get all members of tag:{tag}, delete them with redis.delete(*keys), then delete the tag set.

Common Problems & Solutions

1 Database is overwhelmed after a scheduled cache flush.
Cause: All cache entries expire simultaneously, causing a thundering herd of database queries.
Solution: Add random jitter to TTLs (e.g., base TTL + random 0-60 seconds) to spread expiration over time. Use staggered warm-up scripts to pre-populate the cache.
2 Some users see updated data while others see stale data for the same resource.
Cause: Event-driven invalidation only cleared the cache on one node in a multi-node cache cluster.
Solution: Use a pub/sub mechanism (Redis Pub/Sub or a message broker) to broadcast invalidation events to all cache nodes, or use a centralized cache (single Redis cluster) instead of local caches.
