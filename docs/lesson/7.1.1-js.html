OWASP Top 10 for Node.js Developers

Intuition

Think of the OWASP Top 10 as the most common ways a burglar can break into a house. Broken access control is like having locks that any key can open. Injection is like a mailbox slot big enough to reach in and unlock the door from inside. Security misconfiguration is like leaving a window wide open with a sign saying 'come in.'

OWASP Top 10 for Node.js Developers

Why Security Matters for Backend Developers

Every backend application is a target. Whether you are building a startup MVP or a large-scale enterprise system, attackers will probe your API endpoints, attempt to steal user data, and exploit overlooked vulnerabilities. The Open Web Application Security Project (OWASP) maintains a regularly updated list of the ten most critical web application security risks. Understanding these risks is not optional -- it is a fundamental part of writing production-grade backend code.

This lesson walks through the OWASP Top 10 (2021 edition) from the perspective of a Node.js developer, with concrete code examples showing both vulnerable and secure patterns.

A01: Broken Access Control

Broken access control is the number one vulnerability. It happens when users can act outside their intended permissions -- viewing another user's data, modifying records they should not have access to, or escalating privileges.

Vulnerable Pattern: Insecure Direct Object Reference (IDOR)

// VULNERABLE: No authorization check
app.get('/api/users/:id/profile', async (req, res) => {
  const profile = await prisma.user.findUnique({
    where: { id: parseInt(req.params.id) },
  });
  res.json(profile); // Any authenticated user can view ANY profile
});

Secure Pattern: Enforce Ownership

// SECURE: Verify the requesting user owns the resource
app.get('/api/users/:id/profile', authenticate, async (req, res) => {
  const userId = parseInt(req.params.id);

  // Check that the authenticated user is requesting their own data
  // or has admin privileges
  if (req.user.id !== userId && req.user.role !== 'ADMIN') {
    return res.status(403).json({ error: 'Forbidden' });
  }

  const profile = await prisma.user.findUnique({
    where: { id: userId },
  });

  if (!profile) {
    return res.status(404).json({ error: 'User not found' });
  }

  res.json(profile);
});

Key Defenses

  • Deny by default. Every route should require authentication unless explicitly public.
  • Check resource ownership in every handler, not just at the route level.
  • Use middleware to enforce role-based access.
  • Avoid exposing sequential IDs -- use UUIDs instead.
// Middleware for role-based access
function requireRole(...roles) {
  return (req, res, next) => {
    if (!roles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Insufficient permissions' });
    }
    next();
  };
}

app.delete('/api/users/:id', authenticate, requireRole('ADMIN'), async (req, res) => {
  // Only admins reach this code
  await prisma.user.delete({ where: { id: req.params.id } });
  res.status(204).end();
});

A02: Cryptographic Failures

Cryptographic failures occur when sensitive data is not properly protected -- in transit, at rest, or during processing.

+-------------------------------------------------------------------+
|                    Data Protection Checklist                       |
+-------------------------------------------------------------------+
| In Transit:   HTTPS/TLS everywhere, HSTS headers                  |
| At Rest:      Encrypt database, encrypt backups                   |
| Passwords:    bcrypt/argon2, NEVER plaintext or MD5/SHA            |
| Secrets:      Environment variables, never in code                |
| API Keys:     Rotate regularly, scope permissions                 |
+-------------------------------------------------------------------+

// VULNERABLE: Storing passwords with MD5
const crypto = require('node:crypto');
const hash = crypto.createHash('md5').update(password).digest('hex');

// SECURE: Using bcrypt with proper cost factor
const bcrypt = require('bcrypt');
const SALT_ROUNDS = 12;

async function hashPassword(password) {
  return bcrypt.hash(password, SALT_ROUNDS);
}

async function verifyPassword(password, hash) {
  return bcrypt.compare(password, hash);
}

// VULNERABLE: Secrets hardcoded in source
const stripe = require('stripe')('sk_live_abc123xyz');

// SECURE: Secrets from environment
const stripe = require('stripe')(process.env.STRIPE_SECRET_KEY);

A03: Injection

Injection attacks occur when untrusted data is sent to an interpreter as part of a command or query. SQL injection is the most well-known variant, but NoSQL injection, OS command injection, and LDAP injection are all risks.

We cover SQL injection in depth in lesson 7.1.2. Here is a quick preview:

// VULNERABLE: String concatenation in SQL
const query = `SELECT * FROM users WHERE email = '${req.body.email}'`;

// SECURE: Parameterized query
const result = await pool.query(
  'SELECT * FROM users WHERE email = $1',
  [req.body.email]
);

A04: Insecure Design

Insecure design refers to fundamental architectural flaws that cannot be fixed by perfect implementation. It is about missing or ineffective control design.

// INSECURE DESIGN: No rate limiting on password reset
app.post('/api/auth/reset-password', async (req, res) => {
  const token = generateResetToken();
  await sendResetEmail(req.body.email, token);
  res.json({ message: 'Reset email sent' });
});

// SECURE DESIGN: Rate limiting + token expiry + account lockout
const rateLimit = require('express-rate-limit');

const resetLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 3,                    // 3 attempts per window
  message: { error: 'Too many reset attempts, try again later' },
});

app.post('/api/auth/reset-password', resetLimiter, async (req, res) => {
  const token = crypto.randomBytes(32).toString('hex');
  const expiry = new Date(Date.now() + 30 * 60 * 1000); // 30 min expiry

  await prisma.passwordReset.create({
    data: {
      email: req.body.email,
      token: await bcrypt.hash(token, 10),
      expiresAt: expiry,
    },
  });

  await sendResetEmail(req.body.email, token);

  // Always return the same response to prevent email enumeration
  res.json({ message: 'If that email exists, a reset link was sent' });
});

A05: Security Misconfiguration

This is the most common vulnerability in real-world applications. Default configurations, open cloud storage, verbose error messages, and missing security headers all fall here.

// Use helmet to set security headers automatically
const helmet = require('helmet');
const express = require('express');
const app = express();

app.use(helmet());

// helmet sets these headers (and more):
// X-Content-Type-Options: nosniff
// X-Frame-Options: DENY
// X-XSS-Protection: 0 (relies on CSP instead)
// Strict-Transport-Security: max-age=15552000; includeSubDomains
// Content-Security-Policy: default-src 'self'

// VULNERABLE: Leaking stack traces to clients
app.use((err, req, res, next) => {
  res.status(500).json({
    error: err.message,
    stack: err.stack,  // NEVER expose this in production
  });
});

// SECURE: Generic error in production, details in development
app.use((err, req, res, next) => {
  console.error(err.stack); // Log the full error server-side

  const isDev = process.env.NODE_ENV === 'development';
  res.status(err.status || 500).json({
    error: isDev ? err.message : 'Internal Server Error',
    ...(isDev && { stack: err.stack }),
  });
});

A06: Vulnerable and Outdated Components

Node.js projects pull in hundreds of transitive dependencies. A single vulnerable package in your node_modules can compromise your entire application.

# Check for known vulnerabilities
npm audit

# Fix automatically where possible
npm audit fix

# See detailed report
npm audit --json

# Use Snyk for deeper analysis
npx snyk test

// package.json best practices
{
  "scripts": {
    "audit": "npm audit --audit-level=high",
    "preinstall": "npx npm-force-resolutions"
  },
  "overrides": {
    // Force specific version of a transitive dependency
    "minimist": ">=1.2.6"
  }
}

Automate dependency updates with tools like Dependabot or Renovate. Run npm audit in your CI/CD pipeline and fail the build on high-severity vulnerabilities.

A07: Identification and Authentication Failures

// VULNERABLE: Weak session configuration
app.use(session({
  secret: 'keyboard cat',          // Weak, predictable secret
  cookie: {},                       // No secure flags
}));

// SECURE: Proper session configuration
app.use(session({
  secret: process.env.SESSION_SECRET, // Strong, random secret
  name: '__Host-sid',                  // Prefixed cookie name
  resave: false,
  saveUninitialized: false,
  cookie: {
    secure: true,                      // HTTPS only
    httpOnly: true,                    // No JavaScript access
    sameSite: 'strict',                // CSRF protection
    maxAge: 1000 * 60 * 60 * 24,      // 24 hours
  },
  store: new RedisStore({ client: redisClient }), // External store
}));

A08: Software and Data Integrity Failures

This covers situations where code or data is modified without integrity verification. Subresource integrity, CI/CD pipeline security, and deserialization attacks fall here.

// VULNERABLE: Deserializing untrusted data
const obj = JSON.parse(untrustedInput);
// If this is used with eval() or Function(), it becomes code execution

// Also vulnerable: using node-serialize or similar packages
// that support function deserialization
const serialize = require('node-serialize');
const obj = serialize.unserialize(userInput); // Remote code execution!

// SECURE: Validate deserialized data with a schema
const { z } = require('zod');

const UserInputSchema = z.object({
  name: z.string().max(100),
  email: z.string().email(),
  age: z.number().int().min(0).max(150),
});

const parsed = UserInputSchema.safeParse(JSON.parse(untrustedInput));
if (!parsed.success) {
  return res.status(400).json({ errors: parsed.error.issues });
}

A09: Security Logging and Monitoring Failures

If you cannot detect an attack, you cannot respond to it. Most breaches go undetected for months.

const winston = require('winston');

const securityLogger = winston.createLogger({
  level: 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  defaultMeta: { service: 'auth-service' },
  transports: [
    new winston.transports.File({ filename: 'security.log' }),
  ],
});

// Log security-relevant events
function logSecurityEvent(event, details) {
  securityLogger.warn({
    event,
    ...details,
    timestamp: new Date().toISOString(),
  });
}

// Examples of what to log:
app.post('/api/auth/login', async (req, res) => {
  const user = await findUser(req.body.email);

  if (!user || !(await verifyPassword(req.body.password, user.hash))) {
    logSecurityEvent('LOGIN_FAILED', {
      email: req.body.email,
      ip: req.ip,
      userAgent: req.get('User-Agent'),
    });
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  logSecurityEvent('LOGIN_SUCCESS', {
    userId: user.id,
    ip: req.ip,
  });

  // ... issue token
});

What to Log

+----------------------------------+-----------------------------------+
| Always Log                       | Never Log                         |
+----------------------------------+-----------------------------------+
| Login attempts (success + fail)  | Passwords (even hashed)           |
| Authorization failures           | Full credit card numbers          |
| Input validation failures        | Social security numbers           |
| Admin actions                    | Session tokens / JWTs             |
| Data exports / bulk downloads    | API keys or secrets               |
| Account changes (email, role)    | Personal health information       |
+----------------------------------+-----------------------------------+

A10: Server-Side Request Forgery (SSRF)

SSRF occurs when an application fetches a remote resource based on user-supplied input without proper validation. We cover this in detail in lesson 7.1.3.

// VULNERABLE: Fetching arbitrary URLs
app.post('/api/fetch-metadata', async (req, res) => {
  const response = await fetch(req.body.url);
  const data = await response.json();
  res.json(data);
});
// Attacker can request: http://169.254.169.254/latest/meta-data/
// (AWS metadata endpoint - leaks credentials)

// SECURE: Validate and restrict URLs
const { URL } = require('node:url');
const dns = require('node:dns/promises');
const net = require('node:net');

async function isUrlSafe(inputUrl) {
  const parsed = new URL(inputUrl);

  // Only allow HTTPS
  if (parsed.protocol !== 'https:') return false;

  // Resolve DNS and check for private IPs
  const addresses = await dns.resolve4(parsed.hostname);
  for (const addr of addresses) {
    if (net.isIP(addr) && isPrivateIP(addr)) return false;
  }

  return true;
}

function isPrivateIP(ip) {
  const parts = ip.split('.').map(Number);
  return (
    parts[0] === 10 ||
    parts[0] === 127 ||
    (parts[0] === 172 && parts[1] >= 16 && parts[1] <= 31) ||
    (parts[0] === 192 && parts[1] === 168) ||
    (parts[0] === 169 && parts[1] === 254)
  );
}

Putting It All Together: Security Middleware Stack

const express = require('express');
const helmet = require('helmet');
const rateLimit = require('express-rate-limit');
const cors = require('cors');

const app = express();

// 1. Security headers
app.use(helmet());

// 2. CORS - restrict origins
app.use(cors({
  origin: process.env.ALLOWED_ORIGINS?.split(',') || 'https://myapp.com',
  credentials: true,
}));

// 3. Rate limiting
app.use(rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  standardHeaders: true,
  legacyHeaders: false,
}));

// 4. Body parsing with size limits
app.use(express.json({ limit: '10kb' }));

// 5. Remove fingerprinting
app.disable('x-powered-by');

// 6. Trust proxy (if behind reverse proxy)
app.set('trust proxy', 1);

Request Flow Through Security Layers:

Client --> [TLS/HTTPS] --> [Rate Limiter] --> [CORS Check]
   --> [Helmet Headers] --> [Body Parser + Size Limit]
   --> [Authentication] --> [Authorization] --> [Input Validation]
   --> [Route Handler] --> [Response]

Security Audit Checklist

  1. Run npm audit and resolve all high/critical vulnerabilities
  2. Verify all environment secrets are loaded from env vars, not hardcoded
  3. Confirm HTTPS is enforced with HSTS headers
  4. Test that error responses do not leak stack traces in production
  5. Verify rate limiting on authentication endpoints
  6. Check that all database queries use parameterized statements
  7. Confirm security-relevant events are logged
  8. Review CORS configuration to ensure it is not set to * in production
  9. Verify session cookies use secure, httpOnly, and sameSite flags
  10. Check that file uploads are validated for type and size

Exercises

  1. Security Header Audit

    Create an Express application with helmet configured. Use curl or a browser's developer tools to inspect the response headers. Document each security header that helmet sets, what attack it prevents, and what the default value is. Then customize helmet to add a strict Content-Security-Policy that only allows scripts from your own domain.

    Hint

    Use curl -I http://localhost:3000 to see headers. Check helmet's documentation for the CSP directives: helmet({ contentSecurityPolicy: { directives: { ... } } }).

  2. Build a Rate Limiter Middleware

    Implement a custom rate limiting middleware from scratch (without express-rate-limit) that uses an in-memory Map to track request counts per IP address. It should allow 10 requests per minute, return 429 when exceeded, and include a Retry-After header. Add a cleanup function that removes expired entries every 5 minutes.

    Hint

    Use a Map. On each request, check if the IP exists and whether the window has expired. Use setInterval for cleanup.

  3. Dependency Vulnerability Report

    Take any Node.js project with dependencies. Run npm audit --json, parse the output, and create a summary report that groups vulnerabilities by severity (critical, high, moderate, low), lists the affected packages, and identifies which ones are direct vs transitive dependencies.

    Hint

    Pipe npm audit --json to a script or use JSON.parse on the output. The audit JSON has a vulnerabilities object keyed by package name with severity and dependency path information.

  4. Secure Error Handler

    Write an Express error handling middleware that: (1) logs the full error with stack trace to a file using Winston, (2) returns a sanitized error to the client with no stack trace in production, (3) includes a unique error ID in both the log and the response so support can correlate user reports with server logs, and (4) handles known error types (ValidationError, AuthenticationError, NotFoundError) differently.

    Hint

    Generate an error ID with crypto.randomUUID(). Create custom error classes that extend Error with a statusCode property. Switch on err.constructor.name or use instanceof checks.

Common Problems & Solutions

1 Application leaks stack traces and internal paths in production error responses
Cause: The default Express error handler or a custom error handler is not checking NODE_ENV and sends err.stack to the client. Attackers use this information to map your file structure and identify framework versions.
Solution: Always check `process.env.NODE_ENV` in your error handler. In production, return only a generic message and an error correlation ID. Log the full error server-side. Use `app.disable('x-powered-by')` to prevent framework fingerprinting.
2 npm audit reports dozens of vulnerabilities but npm audit fix does not resolve them
Cause: Most unresolvable audit findings come from transitive (indirect) dependencies where the direct dependency has not released a fix yet. Some are also false positives for your usage.
Solution: Use `npm audit --json` to identify which direct dependencies pull in the vulnerable packages. Check if newer major versions of the direct dependency fix the issue. Use the `overrides` field in package.json to force a patched version of the transitive dependency. For false positives, document the finding and add it to an allow-list in your CI pipeline.
3 CORS errors block requests from the frontend in development, so someone sets origin to '*' and it gets shipped to production
Cause: Wildcard CORS (`*`) allows any website to make requests to your API, which is dangerous for APIs that use cookies or authentication. It often gets set during development and forgotten.
Solution: Use environment-specific CORS configuration. In development, allow localhost origins. In production, set an explicit allowlist. Store allowed origins in environment variables: `cors({ origin: process.env.ALLOWED_ORIGINS.split(',') })`. Add a CI check that verifies CORS is not set to '*' in production config files.
