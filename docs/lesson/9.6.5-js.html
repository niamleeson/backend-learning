Design a CI/CD Pipeline (GitHub Actions)

Intuition

Think of a CI/CD system as a factory assembly line manager. When an order arrives (code push), the manager reads the blueprint (YAML workflow), assigns tasks to available workers (runners), some tasks can happen in parallel (lint and test), others must wait for prerequisites (deploy waits for build). The manager monitors every worker in real-time (log streaming), replaces broken workers instantly (ephemeral VMs), and reports the final quality status to the customer (GitHub status check).

Design a CI/CD Pipeline (GitHub Actions)

Introduction

Continuous Integration and Continuous Deployment (CI/CD) is the backbone of modern software delivery. A CI/CD pipeline automatically builds, tests, and deploys code changes, enabling teams to ship software rapidly and reliably. In this lesson we design a system like GitHub Actions — a cloud-hosted CI/CD platform that executes workflows triggered by events (push, pull request, schedule) on ephemeral compute instances.

Requirements

Functional Requirements

  • Users define workflows as YAML configuration files in their repositories
  • Workflows are triggered by events: push, pull request, tag, schedule (cron), manual dispatch
  • Each workflow contains one or more jobs that run in parallel or with dependencies
  • Each job contains sequential steps (shell commands or reusable actions)
  • Jobs run in isolated environments (containers or VMs) with configurable OS and tools
  • Support secrets management for API keys, tokens, and credentials
  • Support caching of dependencies (node_modules, .m2, pip cache)
  • Support artifacts — upload/download build outputs between jobs
  • Real-time log streaming during execution
  • Support matrix builds (test across Node 18, 20, 22 simultaneously)
  • Support self-hosted runners alongside cloud runners

Non-Functional Requirements

  • 10 million workflow runs per day
  • Job startup time: < 30 seconds (from queue to first step execution)
  • Log streaming latency: < 2 seconds
  • 99.95% availability for the control plane
  • Support 100,000 concurrent job executions
  • Zero data leakage between jobs from different repositories

API Design

Workflow Trigger API

import express from 'express';
import crypto from 'node:crypto';
import YAML from 'yaml';

const app = express();
app.use(express.json());

// POST /api/v1/webhooks/github
// Receives GitHub webhook events
app.post('/api/v1/webhooks/github', async (req, res) => {
  // Verify webhook signature
  const signature = req.headers['x-hub-signature-256'];
  const expected = `sha256=${crypto
    .createHmac('sha256', process.env.WEBHOOK_SECRET)
    .update(JSON.stringify(req.body))
    .digest('hex')}`;

  if (!crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))) {
    return res.status(401).json({ error: 'Invalid signature' });
  }

  const event = req.headers['x-github-event'];
  const payload = req.body;

  // Enqueue event for async processing
  await eventQueue.produce('webhook-events', {
    key: `${payload.repository.full_name}`,
    value: {
      event,
      payload,
      receivedAt: Date.now(),
    },
  });

  // Return immediately (webhook timeout is 10s)
  res.status(202).json({ status: 'accepted' });
});

// POST /api/v1/repos/:owner/:repo/actions/workflows/:workflowId/dispatches
// Manually trigger a workflow
app.post('/api/v1/repos/:owner/:repo/actions/workflows/:workflowId/dispatches',
  async (req, res) => {
    const { owner, repo, workflowId } = req.params;
    const { ref, inputs } = req.body;

    const workflow = await workflowService.get(owner, repo, workflowId);
    if (!workflow) return res.status(404).json({ error: 'Workflow not found' });

    // Validate inputs against workflow definition
    const validationErrors = validateInputs(workflow.inputs, inputs);
    if (validationErrors.length > 0) {
      return res.status(422).json({ errors: validationErrors });
    }

    const run = await workflowEngine.trigger({
      owner,
      repo,
      workflowId,
      ref: ref || 'main',
      event: 'workflow_dispatch',
      inputs,
      triggeredBy: req.user.id,
    });

    res.status(201).json({
      id: run.id,
      status: 'queued',
      url: `/api/v1/repos/${owner}/${repo}/actions/runs/${run.id}`,
    });
  }
);

Run Management API

// GET /api/v1/repos/:owner/:repo/actions/runs
// List workflow runs
app.get('/api/v1/repos/:owner/:repo/actions/runs', async (req, res) => {
  const { owner, repo } = req.params;
  const { status, branch, event, page, per_page } = req.query;

  const runs = await runService.list(owner, repo, {
    status,     // 'queued', 'in_progress', 'completed', 'failure'
    branch,
    event,      // 'push', 'pull_request', 'schedule'
    page: parseInt(page) || 1,
    perPage: parseInt(per_page) || 30,
  });

  res.json({
    total_count: runs.total,
    workflow_runs: runs.items.map(run => ({
      id: run.id,
      name: run.workflowName,
      status: run.status,
      conclusion: run.conclusion,
      head_branch: run.branch,
      head_sha: run.commitSha,
      event: run.event,
      created_at: run.createdAt,
      updated_at: run.updatedAt,
      run_started_at: run.startedAt,
      jobs_url: `/api/v1/repos/${owner}/${repo}/actions/runs/${run.id}/jobs`,
      logs_url: `/api/v1/repos/${owner}/${repo}/actions/runs/${run.id}/logs`,
    })),
  });
});

// GET /api/v1/repos/:owner/:repo/actions/runs/:runId/jobs/:jobId/logs
// Stream job logs in real-time via SSE
app.get('/api/v1/repos/:owner/:repo/actions/runs/:runId/jobs/:jobId/logs',
  async (req, res) => {
    const { runId, jobId } = req.params;

    res.setHeader('Content-Type', 'text/event-stream');
    res.setHeader('Cache-Control', 'no-cache');
    res.setHeader('Connection', 'keep-alive');

    // Send existing logs first
    const existingLogs = await logService.getJobLogs(runId, jobId);
    for (const line of existingLogs) {
      res.write(`data: ${JSON.stringify(line)}\n\n`);
    }

    // Subscribe to live log stream
    const unsubscribe = logStream.subscribe(`${runId}:${jobId}`, (line) => {
      res.write(`data: ${JSON.stringify(line)}\n\n`);
    });

    // Clean up on disconnect
    req.on('close', () => {
      unsubscribe();
    });
  }
);

Database Schema

-- Workflow definitions (cached from YAML in repo)
CREATE TABLE workflows (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  repo_owner      VARCHAR(255) NOT NULL,
  repo_name       VARCHAR(255) NOT NULL,
  file_path       VARCHAR(512) NOT NULL,  -- .github/workflows/ci.yml
  name            VARCHAR(255) NOT NULL,
  triggers        JSONB NOT NULL,          -- {push: {branches: ['main']}}
  raw_yaml        TEXT NOT NULL,
  parsed_config   JSONB NOT NULL,
  commit_sha      VARCHAR(40) NOT NULL,
  is_active       BOOLEAN NOT NULL DEFAULT TRUE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (repo_owner, repo_name, file_path)
);

-- Workflow runs
CREATE TABLE workflow_runs (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workflow_id     UUID NOT NULL REFERENCES workflows(id),
  run_number      INT NOT NULL,
  status          VARCHAR(20) NOT NULL DEFAULT 'queued',
  conclusion      VARCHAR(20),  -- 'success', 'failure', 'cancelled', 'skipped'
  event           VARCHAR(50) NOT NULL,
  branch          VARCHAR(255),
  commit_sha      VARCHAR(40) NOT NULL,
  commit_message  TEXT,
  triggered_by    VARCHAR(255),
  inputs          JSONB DEFAULT '{}',
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_runs_workflow ON workflow_runs(workflow_id, created_at DESC);
CREATE INDEX idx_runs_repo_status ON workflow_runs(status, created_at DESC);
CREATE INDEX idx_runs_commit ON workflow_runs(commit_sha);

-- Jobs within a run
CREATE TABLE jobs (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  run_id          UUID NOT NULL REFERENCES workflow_runs(id),
  name            VARCHAR(255) NOT NULL,
  status          VARCHAR(20) NOT NULL DEFAULT 'queued',
  conclusion      VARCHAR(20),
  runner_id       VARCHAR(255),
  runner_os       VARCHAR(50) NOT NULL DEFAULT 'ubuntu-latest',
  runner_labels   TEXT[] DEFAULT '{}',
  depends_on      UUID[] DEFAULT '{}',  -- job IDs this job depends on
  matrix_values   JSONB DEFAULT '{}',
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_jobs_run ON jobs(run_id);
CREATE INDEX idx_jobs_status ON jobs(status) WHERE status = 'queued';

-- Steps within a job
CREATE TABLE steps (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  job_id          UUID NOT NULL REFERENCES jobs(id),
  step_number     INT NOT NULL,
  name            VARCHAR(255) NOT NULL,
  status          VARCHAR(20) NOT NULL DEFAULT 'pending',
  conclusion      VARCHAR(20),
  command         TEXT,           -- shell command or action reference
  action_ref      VARCHAR(512),  -- 'actions/checkout@v4'
  environment     JSONB DEFAULT '{}',
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ
);

CREATE INDEX idx_steps_job ON steps(job_id, step_number);

-- Secrets (encrypted)
CREATE TABLE secrets (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  repo_owner      VARCHAR(255) NOT NULL,
  repo_name       VARCHAR(255) NOT NULL,
  name            VARCHAR(255) NOT NULL,
  encrypted_value BYTEA NOT NULL,
  key_id          VARCHAR(255) NOT NULL,  -- KMS key used for encryption
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (repo_owner, repo_name, name)
);

-- Runner pool
CREATE TABLE runners (
  id              VARCHAR(255) PRIMARY KEY,
  type            VARCHAR(20) NOT NULL,  -- 'cloud', 'self-hosted'
  os              VARCHAR(50) NOT NULL,
  architecture    VARCHAR(20) NOT NULL,  -- 'x64', 'arm64'
  labels          TEXT[] DEFAULT '{}',
  status          VARCHAR(20) NOT NULL DEFAULT 'idle',
  current_job_id  UUID,
  last_heartbeat  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_runners_available ON runners(os, status)
  WHERE status = 'idle';

Deep Dive: Job Execution Lifecycle

1. Event received (git push)
        │
        ▼
2. Match workflow triggers
   (push to 'main' branch matches "on: push: branches: [main]")
        │
        ▼
3. Parse YAML, build job DAG
        │
   ┌────┴──────────────────────────┐
   │  Job: lint      Job: test     │
   │  (no deps)      (no deps)     │
   │      \              /         │
   │       \            /          │
   │        ▼          ▼           │
   │      Job: build               │
   │      (needs: [lint, test])    │
   │            │                  │
   │            ▼                  │
   │      Job: deploy              │
   │      (needs: [build])         │
   └───────────────────────────────┘
        │
        ▼
4. Enqueue jobs with no dependencies (lint, test)
        │
        ▼
5. Runner Allocator:
   - Find idle runner matching labels + OS
   - If none: provision new VM/container (< 30s)
   - Assign job to runner
        │
        ▼
6. Runner executes:
   a. Clone repository at commit SHA
   b. Restore cache (node_modules)
   c. Execute steps sequentially
   d. Stream logs to log service
   e. Upload artifacts
   f. Report status
        │
        ▼
7. On job completion:
   - Check DAG: are all dependencies of 'build' complete?
   - If yes, enqueue 'build'
   - If any dependency failed, skip dependent jobs
        │
        ▼
8. On run completion:
   - Update run status
   - Post status check to GitHub
   - Send notifications
   - Clean up runner (destroy VM or reset container)

Scaling Strategies

  • Autoscaling runner pools: Scale VM pools based on queue depth. Use warm pools of pre-provisioned VMs for fast startup.
  • Multi-region: Deploy runner pools in multiple cloud regions. Route jobs to the nearest region for lower latency on git clones.
  • Cache layers: Use a distributed cache (S3 + local SSD) for dependency caches. Cache docker images in a regional registry.
  • Log aggregation: Stream logs to S3 via Kinesis/Kafka. Serve recent logs from memory, older logs from S3.
  • Database sharding: Shard workflow_runs and jobs tables by repository to distribute write load.
  • Prioritization: Paid plans get priority in the queue. Use a multi-level priority queue.

Trade-offs

Decision Pros Cons
Ephemeral VMs per job Perfect isolation, clean state Slow startup, expensive
Container-based runners Fast startup (seconds) Weaker isolation, shared kernel
Central job queue Simple, fair scheduling Single point of contention at scale
YAML-based workflow config Version controlled, familiar Complex syntax, hard to debug
Pull-based runner architecture Runners behind firewalls work Polling overhead, slight latency
Pre-provisioned warm pool Fast job start (<10s) Idle cost, resource waste

Security Considerations

  • Secrets are encrypted at rest with per-repository KMS keys
  • Secrets are masked in log output (pattern match and redact)
  • Fork PRs run with read-only permissions (no access to secrets)
  • Runners are single-use (destroyed after each job) to prevent cross-job data leakage
  • Network isolation between jobs using firewall rules
  • Rate limiting on API and webhook endpoints

Summary

A CI/CD pipeline like GitHub Actions is an event-driven system with a workflow engine at its core. Events trigger workflow parsing and job DAG construction, a scheduler matches jobs to runners from an autoscaling pool, runners execute steps in isolated environments while streaming logs, and the system orchestrates job dependencies to completion. The key design challenges are fast job startup (warm VM pools), reliable log streaming (event-driven architecture), secure secrets management, and scaling to handle hundreds of thousands of concurrent jobs.

Exercises

  1. Build a Simple Workflow Engine

    Create a Node.js workflow engine that parses a YAML workflow file, builds a job dependency DAG, and executes jobs in the correct order. Jobs with no dependencies run in parallel. If a job fails, dependent jobs are skipped. Each job runs a shell command and reports success/failure.

    Hint

    Use the 'yaml' npm package to parse YAML. Build an adjacency list for the DAG. Use a queue initialized with jobs that have no dependencies. When a job completes, check if its dependents now have all dependencies satisfied.

  2. Implement Real-Time Log Streaming

    Build a log streaming system using Server-Sent Events (SSE). A 'runner' process writes log lines to a Redis Pub/Sub channel. A 'server' process subscribes to the channel and streams lines to connected HTTP clients. Support multiple concurrent viewers of the same job's logs.

    Hint

    Use ioredis for Redis pub/sub. Create an SSE endpoint that subscribes to a Redis channel named after the job ID. Use res.write('data: ...\n\n') for SSE format. Handle client disconnect to clean up subscriptions.

  3. Build a Job Scheduler with Priority Queue

    Implement a multi-level priority job scheduler. Jobs from paid plans get priority 1, free plans get priority 3, and queued time is used as a tiebreaker (FIFO within same priority). Match jobs to runners based on OS and label requirements.

    Hint

    Use a min-heap or sorted array as the priority queue. The priority key is (plan_priority, queue_timestamp). When a runner becomes available, pop the highest-priority matching job from the queue.

Common Problems & Solutions

1 Job startup time exceeds 2 minutes during peak hours
Symptoms: Users see jobs stuck in 'queued' status for minutes. Queue depth spikes during peak hours. User complaints about slow CI.
Cause: The runner pool is exhausted and new VMs must be provisioned on-demand. Cloud provider VM provisioning takes 60-120 seconds. The autoscaler reacts too slowly to sudden spikes.
Solution: Maintain a warm pool of pre-provisioned VMs (10-20% above current demand). Implement predictive autoscaling based on historical patterns (e.g., spike at 9 AM PST on weekdays). Use container-based runners for fast startup when VMs are unavailable. Offer a 'fast start' option using shared runners that are always available.
2 Log streaming disconnects and shows incomplete output
Symptoms: Log viewer shows partial output. Users must refresh to see new logs. Log gaps during long-running steps like builds.
Cause: SSE connections are dropped by load balancer idle timeouts (60 seconds). Long-running steps produce no output for extended periods, causing the connection to be considered idle.
Solution: Send SSE heartbeat comments every 15 seconds to keep connections alive. Implement reconnection logic on the client using the Last-Event-ID header. Store all log lines in durable storage (S3) so clients can fetch missed lines on reconnect. Use a log buffer that replays from the last seen line ID.
3 Secrets are accidentally exposed in log output
Cause: A workflow step echoes environment variables or a debug command prints the process environment, including injected secrets.
Solution: Implement a log redaction layer that pattern-matches all known secret values and replaces them with '***'. Apply redaction at the runner level before logs leave the machine. Additionally, warn users when a step's exit code indicates potential secret exposure. Never pass secrets as command-line arguments (visible in /proc).
