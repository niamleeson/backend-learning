Dead Letter Queues, Retry Strategies, and Idempotency

Intuition

Think of a mail delivery system: retries are the mail carrier trying again tomorrow when no one is home. The dead letter office (DLQ) is where undeliverable mail goes after all attempts. Idempotency is like having a signature log — delivering the same package twice does not result in two deliveries because the recipient already signed for it.

Dead Letter Queues, Retry Strategies, and Idempotency

Introduction

In distributed systems, failures are not exceptional — they are expected. Messages will fail to process due to temporary errors (network timeouts, service outages) and permanent errors (invalid data, business logic violations). Dead letter queues, retry strategies, and idempotent processing form the triad of reliable message handling.

Dead Letter Queues (DLQ)

A dead letter queue captures messages that cannot be processed successfully, preventing them from being lost or retried infinitely.

Normal Flow:
Queue ──> Consumer ──> Process ──> ACK (removed from queue)

Failure Flow:
Queue ──> Consumer ──> Process ──> FAIL
                                     │
                                     ├── Retry (requeue)
                                     │     └── Eventually succeeds or...
                                     │
                                     └── Dead Letter Queue (after max retries)
                                           └── Manual investigation
                                           └── Automated alerting

DLQ with BullMQ

BullMQ doesn't have a separate DLQ — failed jobs stay in the failed state. But you can implement DLQ-like behavior:

import { Queue, Worker } from 'bullmq';

const mainQueue = new Queue('orders', { connection });
const dlq = new Queue('orders-dlq', { connection });

const worker = new Worker('orders', async (job) => {
  try {
    await processOrder(job.data);
  } catch (err) {
    // Check if this is the final attempt
    if (job.attemptsMade >= job.opts.attempts - 1) {
      // Move to DLQ with error context
      await dlq.add('failed-order', {
        originalJob: {
          id: job.id,
          name: job.name,
          data: job.data,
          attemptsMade: job.attemptsMade,
        },
        error: {
          message: err.message,
          stack: err.stack,
        },
        failedAt: new Date().toISOString(),
      });
    }
    throw err; // Re-throw so BullMQ handles retry
  }
}, {
  connection,
  attempts: 3,
  backoff: { type: 'exponential', delay: 1000 },
});

// DLQ processor - alert or attempt remediation
const dlqWorker = new Worker('orders-dlq', async (job) => {
  const { originalJob, error } = job.data;
  console.error(`DLQ: Order ${originalJob.data.orderId} failed: ${error.message}`);

  // Send alert
  await alertOps({
    type: 'dead_letter',
    queue: 'orders',
    jobId: originalJob.id,
    error: error.message,
  });

  // Optionally: attempt automated remediation
  if (error.message.includes('inventory')) {
    await mainQueue.add('orders', originalJob.data, {
      delay: 300000, // Retry in 5 minutes
      attempts: 1,
    });
  }
}, { connection });

Retry Strategies

1. Fixed Delay

// Retry every 5 seconds
await queue.add('job', data, {
  attempts: 5,
  backoff: { type: 'fixed', delay: 5000 },
});
// Retries at: 5s, 5s, 5s, 5s

2. Exponential Backoff

// Exponentially increasing delays
await queue.add('job', data, {
  attempts: 5,
  backoff: { type: 'exponential', delay: 1000 },
});
// Retries at: 1s, 2s, 4s, 8s

3. Custom Backoff with Jitter

// Custom backoff strategy
await queue.add('api-call', data, {
  attempts: 5,
  backoff: { type: 'custom' },
});

// In worker
const worker = new Worker('api-call', processor, {
  connection,
  settings: {
    backoffStrategy: (attemptsMade) => {
      // Exponential backoff with full jitter
      const baseDelay = 1000;
      const maxDelay = 30000;
      const exponential = Math.min(baseDelay * Math.pow(2, attemptsMade), maxDelay);
      const jitter = Math.random() * exponential;
      return Math.floor(jitter);
    },
  },
});

4. Circuit Breaker Pattern with Retry

class CircuitBreaker {
  constructor({ failureThreshold = 5, resetTimeout = 30000 } = {}) {
    this.failureCount = 0;
    this.failureThreshold = failureThreshold;
    this.resetTimeout = resetTimeout;
    this.state = 'CLOSED'; // CLOSED, OPEN, HALF_OPEN
    this.nextAttempt = null;
  }

  async execute(fn) {
    if (this.state === 'OPEN') {
      if (Date.now() < this.nextAttempt) {
        throw new Error('Circuit breaker is OPEN');
      }
      this.state = 'HALF_OPEN';
    }

    try {
      const result = await fn();
      this.onSuccess();
      return result;
    } catch (err) {
      this.onFailure();
      throw err;
    }
  }

  onSuccess() {
    this.failureCount = 0;
    this.state = 'CLOSED';
  }

  onFailure() {
    this.failureCount++;
    if (this.failureCount >= this.failureThreshold) {
      this.state = 'OPEN';
      this.nextAttempt = Date.now() + this.resetTimeout;
    }
  }
}

// Use in worker
const paymentBreaker = new CircuitBreaker({ failureThreshold: 3 });

const worker = new Worker('payments', async (job) => {
  return paymentBreaker.execute(async () => {
    return await paymentGateway.charge(job.data);
  });
}, { connection, attempts: 5, backoff: { type: 'exponential', delay: 2000 } });

Classifying Errors: Retryable vs Permanent

Not all errors should be retried:

class RetryableError extends Error {
  constructor(message) {
    super(message);
    this.retryable = true;
  }
}

class PermanentError extends Error {
  constructor(message) {
    super(message);
    this.retryable = false;
  }
}

const worker = new Worker('orders', async (job) => {
  try {
    const result = await processOrder(job.data);
    return result;
  } catch (err) {
    // Classify the error
    if (err.code === 'ECONNRESET' || err.code === 'ETIMEDOUT') {
      throw new RetryableError(`Network error: ${err.message}`);
    }
    if (err.status === 400 || err.status === 422) {
      // Bad data - retrying won't help
      throw new PermanentError(`Invalid data: ${err.message}`);
    }
    if (err.status === 429) {
      throw new RetryableError('Rate limited');
    }
    if (err.status >= 500) {
      throw new RetryableError(`Server error: ${err.message}`);
    }
    throw err;
  }
}, {
  connection,
  attempts: 5,
  backoff: { type: 'exponential', delay: 1000 },
});

// Skip retries for permanent errors
worker.on('failed', async (job, err) => {
  if (!err.retryable && job.attemptsMade < job.opts.attempts) {
    // Move directly to DLQ instead of retrying
    await job.moveToFailed(err, job.token, true); // true = don't retry
  }
});

Idempotency

An operation is idempotent if performing it multiple times has the same effect as performing it once. This is critical because retries WILL cause duplicate processing.

Without Idempotency:
  Process payment: charge $100
  Retry (duplicate): charge $100 again
  Result: customer charged $200 ← BUG!

With Idempotency:
  Process payment: charge $100 (idempotency key: "pay-ORD123")
  Retry (duplicate): check key "pay-ORD123" already processed
  Result: customer charged $100 ← CORRECT

Idempotency Key Pattern

class IdempotentProcessor {
  constructor(redis) {
    this.redis = redis;
  }

  async processOnce(idempotencyKey, ttl, processFn) {
    // Try to acquire the idempotency lock
    const acquired = await this.redis.set(
      `idempotent:${idempotencyKey}`,
      'processing',
      'NX',    // Only set if not exists
      'EX',    // Expire
      ttl      // TTL in seconds
    );

    if (!acquired) {
      // Already processed or processing
      const result = await this.redis.get(`idempotent:${idempotencyKey}:result`);
      if (result) return JSON.parse(result);
      throw new Error('Duplicate processing detected');
    }

    try {
      const result = await processFn();

      // Store the result for duplicate requests
      await this.redis.setex(
        `idempotent:${idempotencyKey}:result`,
        ttl,
        JSON.stringify(result)
      );
      await this.redis.set(`idempotent:${idempotencyKey}`, 'completed', 'EX', ttl);

      return result;
    } catch (err) {
      // Remove lock so it can be retried
      await this.redis.del(`idempotent:${idempotencyKey}`);
      throw err;
    }
  }
}

// Usage in worker
const idempotent = new IdempotentProcessor(redis);

const worker = new Worker('payments', async (job) => {
  const idempotencyKey = `payment:${job.data.orderId}`;

  return idempotent.processOnce(idempotencyKey, 86400, async () => {
    const charge = await stripe.charges.create({
      amount: job.data.amount,
      currency: 'usd',
      customer: job.data.customerId,
      idempotency_key: idempotencyKey, // Stripe also supports idempotency
    });
    return { chargeId: charge.id, amount: charge.amount };
  });
}, { connection, attempts: 3, backoff: { type: 'exponential', delay: 2000 } });

Database-Level Idempotency

async function processOrderIdempotent(orderId, action) {
  // Use a processed_events table
  return prisma.$transaction(async (tx) => {
    // Check if already processed
    const existing = await tx.processedEvent.findUnique({
      where: { eventKey: `order:${orderId}:${action}` },
    });

    if (existing) {
      console.log(`Already processed: order:${orderId}:${action}`);
      return existing.result;
    }

    // Process the action
    const result = await performAction(tx, orderId, action);

    // Record that we processed it
    await tx.processedEvent.create({
      data: {
        eventKey: `order:${orderId}:${action}`,
        result: JSON.stringify(result),
        processedAt: new Date(),
      },
    });

    return result;
  });
}

Combining All Three Patterns

// Complete reliable message processing setup

const mainQueue = new Queue('order-processing', { connection });
const dlq = new Queue('order-processing-dlq', { connection });
const idempotent = new IdempotentProcessor(redis);

const worker = new Worker('order-processing', async (job) => {
  const { orderId, action, timestamp } = job.data;
  const idempotencyKey = `${orderId}:${action}:${timestamp}`;

  // 1. Idempotency check
  return idempotent.processOnce(idempotencyKey, 86400, async () => {
    // 2. Classify and process
    try {
      switch (action) {
        case 'charge':
          return await chargeOrder(orderId);
        case 'ship':
          return await shipOrder(orderId);
        case 'notify':
          return await notifyCustomer(orderId);
        default:
          throw new PermanentError(`Unknown action: ${action}`);
      }
    } catch (err) {
      // 3. Error classification
      if (err instanceof PermanentError) {
        // Move to DLQ immediately
        await dlq.add('permanent-failure', {
          originalJob: job.data,
          error: err.message,
          failedAt: new Date().toISOString(),
        });
        return { status: 'permanent_failure', error: err.message };
      }
      throw err; // Retryable errors bubble up for retry
    }
  });
}, {
  connection,
  concurrency: 10,
  attempts: 5,
  backoff: { type: 'exponential', delay: 2000 },
});

Common Pitfalls

  1. Retrying permanent errors: Classifying all errors as retryable wastes resources and delays DLQ processing. Always distinguish retryable from permanent errors.
  2. No idempotency on payment processing: Double-charging customers is a severe bug. Every payment operation must be idempotent.
  3. Infinite retry loops: Without max attempts, a poisoned message retries forever. Always set a maximum.
  4. DLQ without monitoring: Messages sitting in the DLQ unnoticed means lost business. Alert on DLQ depth.

Summary

Dead letter queues capture permanently failed messages for investigation. Retry strategies with exponential backoff and jitter handle transient failures gracefully. Idempotency ensures that retried messages produce the same result as the first processing. Together, these three patterns make your message processing reliable in the face of inevitable distributed system failures.

Common Problems & Solutions

1 Customers being charged multiple times
Cause: Payment processing job retried after timeout, but the original payment actually succeeded. The retry creates a second charge.
Solution: Implement idempotency using a unique key per payment (orderId). Check if the payment was already processed before charging. Use Stripe's built-in idempotency_key parameter.

const charge = await stripe.charges.create({\n amount: 1000,\n currency: 'usd',\n idempotency_key: `order-\${orderId}`,\n});

2 DLQ growing indefinitely with no alerts
Cause: Dead letter queue is configured but no one monitors it. Failed messages accumulate silently.
Solution: Set up monitoring on DLQ depth. Alert when count exceeds threshold. Create a scheduled job that checks DLQ size and sends alerts.

// Check DLQ depth every minute\nsetInterval(async () => {\n const waiting = await dlq.getWaitingCount();\n if (waiting > 100) {\n await alertOps({ type: 'dlq_overflow', count: waiting });\n }\n}, 60000);
