Design a Social Graph (Facebook/LinkedIn)

Intuition

A social graph system is an adjacency-list store where the fundamental challenge is that graph traversals touch many shards in a distributed setting, making multi-hop queries expensive. The practical solution is to precompute the most common queries offline and cache aggressively for high-degree nodes.

Design a Social Graph (Facebook/LinkedIn)

Requirements

  • Store billions of nodes (users) and hundreds of billions of edges (connections)
  • Support queries: mutual friends, degrees of separation, friend recommendations
  • Handle asymmetric (follow) and symmetric (friend) relationship types
  • Serve graph traversal queries within 100ms for up to 3 hops

Architecture

An adjacency list representation stored in a distributed key-value or graph database, partitioned by user ID. A graph query engine executes BFS/DFS traversals, with caching for hot nodes (celebrities). An offline pipeline computes friend-of-friend recommendations and preloads them.

Key Components

  • Graph Store: Adjacency lists in TAO-like system (Facebook) or a graph DB (Neo4j/JanusGraph)
  • Cache Layer: In-memory cache of adjacency lists for high-degree nodes
  • Traversal Engine: Executes bounded BFS for mutual friends, shortest path queries
  • Recommendation Pipeline: Offline MapReduce/Spark jobs compute PYMK (People You May Know)

Scaling

Partition by user ID with replication. Co-locate friends' adjacency lists when possible to reduce cross-shard hops. Use bidirectional BFS for shortest path to reduce search space exponentially.

Trade-offs

  • Graph databases are natural but harder to scale; key-value stores scale but require application-level traversal
  • Precomputing recommendations is stale but fast; real-time traversal is fresh but expensive
  • Storing reverse edges doubles storage but is essential for efficient fan-in queries

Exercises

  1. Implement Bidirectional BFS for Mutual Friends

    Build a graph with 100,000 nodes and random edges. Implement bidirectional BFS to find mutual friends between two users and compare performance against naive single-source BFS.

    Hint

    Run BFS simultaneously from both users, expanding the smaller frontier first. Terminate when the two frontiers intersect.

  2. Build a Friend-of-Friend Recommendation Engine

    Given a user, compute the top-10 friend-of-friend suggestions ranked by number of mutual connections. Filter out existing friends.

    Hint

    For each friend of the user, iterate their friends. Use a counter map to tally how many times each candidate appears, then sort by count.

Common Problems & Solutions

1 Graph traversal queries timeout for celebrity accounts with millions of followers
Cause: High-degree nodes cause BFS to explode, touching millions of adjacency lists in a single hop
Solution: Cap the traversal fan-out by sampling edges for high-degree nodes, and serve precomputed results for known celebrity accounts.
2 Cross-shard graph queries have high latency due to network round trips
Cause: Friends are randomly distributed across shards, so a 2-hop query may hit dozens of shards
Solution: Use graph-aware partitioning (e.g., METIS) to co-locate densely connected subgraphs on the same shard, reducing cross-shard hops.
