Design a Social Graph (Facebook/LinkedIn)

Intuition

Think of a social graph like a city road map. Each person is an intersection, and each friendship is a road connecting two intersections. Finding mutual friends is like asking which intersections can be reached directly from both of two given intersections. Finding degrees of separation is like finding the shortest driving route between two points. At city scale you need a good index (GPS/map), not brute-force exploration of every road.

Design a Social Graph (Facebook/LinkedIn)

Introduction

A social graph is the data structure and system behind every social network — it models relationships between people (friends, followers, connections) and powers features like friend suggestions, mutual connections, degree-of-separation queries, and news feed ranking. Companies like Facebook (2.9B+ users), LinkedIn (900M+ members), and Twitter use social graphs at massive scale.

In a system design interview, this problem tests your ability to model graph data at scale, choose between graph databases and relational stores, handle fan-out for relationship queries, and design APIs that feel instantaneous despite billions of edges.

Requirements

Functional Requirements

  1. Add/remove connections — Users can send friend/connection requests, accept them, and remove connections
  2. Query connections — Get a user's direct connections (1st degree)
  3. Mutual connections — Find shared connections between two users
  4. Degree of separation — Find the shortest path between two users (up to 3 degrees)
  5. Friend/connection suggestions — Recommend people the user may know (friends-of-friends)
  6. Search within network — Search for people with filters (company, location, school)
  7. Block users — Prevent specific users from appearing in results or sending requests

Non-Functional Requirements

  1. Low latency — Connection list and mutual friends must load in <200ms
  2. High availability — 99.99% uptime, the social graph is the backbone of the platform
  3. Consistency — Eventually consistent is acceptable for suggestions; strong consistency for connection status
  4. Scalability — Support 500M+ users with average 500 connections each

Capacity Estimation

Users:                 500 million
Avg connections/user:  500
Total edges:           500M * 500 / 2 = 125 billion edges
                       (divide by 2 because each edge is bidirectional)

Storage per edge:      user_id_1 (8B) + user_id_2 (8B) + metadata (16B) = 32 bytes
Total edge storage:    125B * 32B = 4 TB

Storage per user:      user_id (8B) + profile data (2KB avg) = ~2KB
Total user storage:    500M * 2KB = 1 TB

Read QPS:
  - View connections:   50,000 QPS
  - Mutual friends:     20,000 QPS
  - Friend suggestions: 10,000 QPS
  - Search:             30,000 QPS

Write QPS:
  - New connections:    5,000 QPS
  - Remove connections: 500 QPS

Bandwidth:
  - Avg response size:  5KB (list of 20 connections with basic profiles)
  - Read bandwidth:     50,000 * 5KB = 250 MB/s

API Design

import express from 'express';
const router = express.Router();

// Send a connection request
router.post('/connections/request', async (req, res) => {
  // POST /api/v1/connections/request
  // Body: { target_user_id: string }
  // Response: { request_id, status: 'pending', created_at }
  const { target_user_id } = req.body;
  const request = await connectionService.sendRequest(
    req.user.id,
    target_user_id
  );
  res.status(201).json(request);
});

// Accept or reject a connection request
router.put('/connections/request/:requestId', async (req, res) => {
  // PUT /api/v1/connections/request/:requestId
  // Body: { action: 'accept' | 'reject' }
  const { action } = req.body;
  const result = await connectionService.respondToRequest(
    req.params.requestId,
    req.user.id,
    action
  );
  res.json(result);
});

// Get a user's connections (paginated)
router.get('/users/:userId/connections', async (req, res) => {
  // GET /api/v1/users/:userId/connections?cursor=abc&limit=20
  // Response: { connections: [...], next_cursor, total_count }
  const { cursor, limit = 20 } = req.query;
  const connections = await graphService.getConnections(
    req.params.userId,
    { cursor, limit: parseInt(limit) }
  );
  res.json(connections);
});

// Get mutual connections between two users
router.get('/users/:userId/mutual/:otherUserId', async (req, res) => {
  // GET /api/v1/users/:userId/mutual/:otherUserId?limit=20
  // Response: { mutual_connections: [...], count }
  const result = await graphService.getMutualConnections(
    req.params.userId,
    req.params.otherUserId,
    { limit: parseInt(req.query.limit) || 20 }
  );
  res.json(result);
});

// Get degree of separation
router.get('/users/:userId/distance/:otherUserId', async (req, res) => {
  // GET /api/v1/users/:userId/distance/:otherUserId
  // Response: { degree: 2, path: [userId, intermediary, otherUserId] }
  const result = await graphService.getDegreesOfSeparation(
    req.params.userId,
    req.params.otherUserId,
    { maxDegree: 3 }
  );
  res.json(result);
});

// Get connection suggestions (People You May Know)
router.get('/users/:userId/suggestions', async (req, res) => {
  // GET /api/v1/users/:userId/suggestions?limit=20
  // Response: { suggestions: [{ user, mutual_count, reason }] }
  const suggestions = await suggestionService.getSuggestions(
    req.params.userId,
    { limit: parseInt(req.query.limit) || 20 }
  );
  res.json(suggestions);
});

// Block a user
router.post('/users/block', async (req, res) => {
  // POST /api/v1/users/block
  // Body: { blocked_user_id: string }
  await connectionService.blockUser(req.user.id, req.body.blocked_user_id);
  res.status(204).send();
});

export default router;

Database Schema

PostgreSQL — Edge Table Approach

-- Users table (core profile data)
CREATE TABLE users (
  id            BIGSERIAL PRIMARY KEY,
  username      VARCHAR(64) UNIQUE NOT NULL,
  display_name  VARCHAR(128) NOT NULL,
  email         VARCHAR(255) UNIQUE NOT NULL,
  company       VARCHAR(128),
  location      VARCHAR(128),
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- Connections (edges) - store both directions for fast lookup
CREATE TABLE connections (
  user_id_1     BIGINT NOT NULL REFERENCES users(id),
  user_id_2     BIGINT NOT NULL REFERENCES users(id),
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (user_id_1, user_id_2),
  CHECK (user_id_1 < user_id_2)  -- canonical ordering to prevent duplicates
);

-- Adjacency list for fast lookups from either direction
CREATE TABLE adjacency_list (
  user_id       BIGINT NOT NULL REFERENCES users(id),
  friend_id     BIGINT NOT NULL REFERENCES users(id),
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (user_id, friend_id)
);
CREATE INDEX idx_adjacency_user ON adjacency_list(user_id, created_at DESC);

-- Connection requests
CREATE TABLE connection_requests (
  id            BIGSERIAL PRIMARY KEY,
  sender_id     BIGINT NOT NULL REFERENCES users(id),
  receiver_id   BIGINT NOT NULL REFERENCES users(id),
  status        VARCHAR(16) DEFAULT 'pending',  -- pending, accepted, rejected
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at    TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (sender_id, receiver_id)
);
CREATE INDEX idx_request_receiver ON connection_requests(receiver_id, status);

-- Block list
CREATE TABLE blocks (
  blocker_id    BIGINT NOT NULL REFERENCES users(id),
  blocked_id    BIGINT NOT NULL REFERENCES users(id),
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (blocker_id, blocked_id)
);

Graph Database (Neo4j) for Complex Traversals

(User)-[:CONNECTED_TO]->(User)
(User)-[:BLOCKED]->(User)
(User)-[:WORKS_AT]->(Company)
(User)-[:STUDIED_AT]->(School)

Neo4j Cypher queries for common operations:

// Mutual connections
MATCH (a:User {id: $userId1})-[:CONNECTED_TO]-(mutual)-[:CONNECTED_TO]-(b:User {id: $userId2})
RETURN mutual LIMIT 20;

// Shortest path (degree of separation)
MATCH path = shortestPath(
  (a:User {id: $userId1})-[:CONNECTED_TO*..3]-(b:User {id: $userId2})
)
RETURN length(path) AS degree, [n IN nodes(path) | n.id] AS path;

// Friend suggestions (friends of friends, not already connected)
MATCH (me:User {id: $userId})-[:CONNECTED_TO]-(friend)-[:CONNECTED_TO]-(suggestion)
WHERE NOT (me)-[:CONNECTED_TO]-(suggestion)
  AND NOT (me)-[:BLOCKED]-(suggestion)
  AND suggestion.id <> $userId
RETURN suggestion, COUNT(friend) AS mutual_count
ORDER BY mutual_count DESC
LIMIT 20;

Core Implementation

import Redis from 'ioredis';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();
const redis = new Redis({ host: 'redis-cluster', port: 6379 });

class GraphService {
  // Get direct connections with caching
  async getConnections(userId, { cursor, limit }) {
    const cacheKey = `connections:${userId}:${cursor || 'start'}:${limit}`;
    const cached = await redis.get(cacheKey);
    if (cached) return JSON.parse(cached);

    const where = { user_id: BigInt(userId) };
    if (cursor) {
      where.created_at = { lt: new Date(cursor) };
    }

    const edges = await prisma.adjacency_list.findMany({
      where,
      orderBy: { created_at: 'desc' },
      take: limit + 1,
      include: { friend: { select: { id: true, display_name: true, company: true } } },
    });

    const hasMore = edges.length > limit;
    const connections = edges.slice(0, limit);
    const result = {
      connections: connections.map(e => e.friend),
      next_cursor: hasMore ? connections[limit - 1].created_at.toISOString() : null,
      total_count: await prisma.adjacency_list.count({ where: { user_id: BigInt(userId) } }),
    };

    await redis.setex(cacheKey, 300, JSON.stringify(result));
    return result;
  }

  // Mutual connections using set intersection
  async getMutualConnections(userId1, userId2, { limit }) {
    // Use Redis sorted sets for fast intersection
    const key1 = `friends:${userId1}`;
    const key2 = `friends:${userId2}`;

    // Ensure friend sets are loaded
    await this._ensureFriendSet(userId1);
    await this._ensureFriendSet(userId2);

    // ZINTERSTORE computes intersection in Redis
    const tempKey = `mutual:${userId1}:${userId2}`;
    await redis.zinterstore(tempKey, 2, key1, key2);
    await redis.expire(tempKey, 60);

    const mutualIds = await redis.zrange(tempKey, 0, limit - 1);
    const mutualUsers = await prisma.user.findMany({
      where: { id: { in: mutualIds.map(id => BigInt(id)) } },
      select: { id: true, display_name: true, company: true },
    });

    return { mutual_connections: mutualUsers, count: mutualIds.length };
  }

  // Load user's friend list into Redis sorted set
  async _ensureFriendSet(userId) {
    const key = `friends:${userId}`;
    const exists = await redis.exists(key);
    if (exists) return;

    const edges = await prisma.adjacency_list.findMany({
      where: { user_id: BigInt(userId) },
      select: { friend_id: true, created_at: true },
    });

    if (edges.length === 0) return;

    const pipeline = redis.pipeline();
    for (const edge of edges) {
      pipeline.zadd(key, edge.created_at.getTime(), edge.friend_id.toString());
    }
    pipeline.expire(key, 3600);
    await pipeline.exec();
  }
}

class SuggestionService {
  // Friends-of-friends scored by mutual connection count
  async getSuggestions(userId, { limit }) {
    const cacheKey = `suggestions:${userId}`;
    const cached = await redis.get(cacheKey);
    if (cached) return JSON.parse(cached);

    // SQL approach: friends of friends not already connected
    const suggestions = await prisma.$queryRaw`
      SELECT
        u.id, u.display_name, u.company,
        COUNT(DISTINCT f2.user_id) AS mutual_count
      FROM adjacency_list f1
      JOIN adjacency_list f2 ON f1.friend_id = f2.user_id
      JOIN users u ON f2.friend_id = u.id
      WHERE f1.user_id = ${BigInt(userId)}
        AND f2.friend_id != ${BigInt(userId)}
        AND f2.friend_id NOT IN (
          SELECT friend_id FROM adjacency_list WHERE user_id = ${BigInt(userId)}
        )
        AND f2.friend_id NOT IN (
          SELECT blocked_id FROM blocks WHERE blocker_id = ${BigInt(userId)}
        )
      GROUP BY u.id, u.display_name, u.company
      ORDER BY mutual_count DESC
      LIMIT ${limit}
    `;

    const result = { suggestions };
    await redis.setex(cacheKey, 1800, JSON.stringify(result));
    return result;
  }
}

Scaling Strategies

1. Graph Partitioning (Sharding)

┌──────────────────────────────────────────────┐
│           Graph Partition Strategy             │
│                                               │
│  Shard by user_id hash:                       │
│    Shard 0: users 0-99M   (edges within)     │
│    Shard 1: users 100-199M                    │
│    Shard 2: users 200-299M                    │
│    ...                                        │
│                                               │
│  Cross-shard edges stored on BOTH shards     │
│  (duplicate for fast local reads)             │
│                                               │
│  User 50M ──connected──> User 250M           │
│    Stored on Shard 0 AND Shard 2             │
└──────────────────────────────────────────────┘

2. Caching Layers

┌──────────────────────────────────────────────┐
│  L1: In-process LRU (node-lru-cache)         │
│      - Hot user profiles, 10K entries         │
│      - TTL: 60 seconds                        │
│                                               │
│  L2: Redis Cluster                            │
│      - Friend lists as sorted sets            │
│      - Mutual connection results              │
│      - Suggestion lists (pre-computed)        │
│      - TTL: 5-30 minutes                      │
│                                               │
│  L3: Database (PostgreSQL / Neo4j)            │
│      - Source of truth                        │
└──────────────────────────────────────────────┘

3. Pre-computation for Suggestions

Friend suggestions are expensive to compute in real-time. Use a batch pipeline:

┌─────────────┐     ┌───────────────────┐     ┌──────────────┐
│ Scheduled   │────>│ Spark / Worker    │────>│ Redis / DB   │
│ Cron Job    │     │ Compute FoF for   │     │ Store top    │
│ (hourly)    │     │ each active user  │     │ 50 suggest.  │
└─────────────┘     └───────────────────┘     └──────────────┘

4. Bidirectional BFS for Degree of Separation

Instead of a single BFS from user A, run BFS from both A and B simultaneously:

User A ──> BFS layer 1 ──> BFS layer 2
                                  ↕  (meet in the middle)
User B ──> BFS layer 1 ──> BFS layer 2

This reduces traversal from O(b^d) to O(b^(d/2)) where b = branching factor
For b=500, d=3: single BFS = 125M nodes, bidirectional = ~35K nodes

Trade-offs

Decision Option A Option B
Storage PostgreSQL adjacency table Neo4j graph DB
Familiar, mature, SQL joins Native graph traversal, Cypher
Struggles with 3+ degree queries Purpose-built for path queries
Friend list Store both directions Store canonical (user1 < user2)
2x storage, O(1) lookup per user 1x storage, needs UNION query
Suggestions Real-time computation Pre-computed batch
Always fresh, high latency Stale up to 1hr, fast reads
Sharding Hash-based (user_id) Social-cluster-based
Even distribution, cross-shard edges Fewer cross-shard edges, uneven
Mutual friends SQL JOIN at query time Redis ZINTERSTORE
No extra storage, slower Redis memory cost, sub-ms

Summary

A social graph system at scale requires a hybrid approach: PostgreSQL for the source of truth edge storage with a denormalized adjacency list for fast single-hop queries, Redis sorted sets for mutual connection computation and caching, and optionally a graph database like Neo4j for complex multi-hop traversals. Friend suggestions should be pre-computed in batch. Sharding by user_id with cross-shard edge replication keeps reads local. Bidirectional BFS dramatically reduces the cost of degree-of-separation queries.

Common Problems & Solutions

1 Mutual friends query takes 5+ seconds for users with many connections
Cause: SQL JOIN between two large adjacency lists is O(n*m) without proper indexing, and popular users (influencers) can have millions of connections.
Solution: Use Redis sorted sets with ZINTERSTORE for sub-millisecond set intersection. For extremely popular users, pre-compute and cache mutual friend lists for the most common pairs. Add a composite index on (user_id, friend_id) for the SQL fallback path.
2 Friend suggestion computation overloads the database during peak hours
Cause: Computing friends-of-friends in real-time requires multiple JOINs across millions of rows, creating heavy read load on the primary database.
Solution: Move suggestion computation to an offline batch pipeline that runs during off-peak hours. Use read replicas for the computation. Store results in Redis with a 1-hour TTL. Serve suggestions from cache with a fallback to a simpler algorithm (e.g., same company/school) when cache is cold.
3 Adding a connection is slow because it requires updating multiple stores
Cause: A single connection add must write to the connections table, adjacency list (both directions), invalidate caches for both users, update suggestion caches, and update Elasticsearch.
Solution: Make the critical path synchronous (adjacency list insert + cache invalidation) and push secondary updates (suggestions, search index) to a message queue (BullMQ). Return 201 immediately after the database write. Process secondary effects asynchronously.
