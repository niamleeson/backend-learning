NestJS DI Container: Providers, Scopes, and Tokens

Intuition

The DI container is like a central warehouse. Provider tokens are SKU numbers -- unique identifiers for each item. Singleton scope means one item on the shelf that everyone borrows. Request scope means the warehouse builds a fresh item for each order. Transient scope means each department gets its own copy.

NestJS DI Container: Providers, Scopes, and Tokens

Introduction

The NestJS Dependency Injection (DI) container is the engine that powers the entire framework. It manages the creation, lifecycle, and injection of every provider in your application. Understanding how it works -- providers, scopes, tokens, and resolution -- is essential for building robust NestJS applications.

How the DI Container Resolves Dependencies

When NestJS starts, it performs these steps:

1. Parse all @Module() decorators
2. Build the module dependency graph
3. For each module, collect providers and controllers
4. For each provider/controller, inspect constructor parameters
5. Resolve dependencies recursively (depth-first)
6. Instantiate providers in correct order
7. Cache singleton instances

Dependency Resolution Example

@Injectable()
class DatabaseService {}

@Injectable()
class UsersRepository {
  constructor(private db: DatabaseService) {}
}

@Injectable()
class UsersService {
  constructor(private repo: UsersRepository) {}
}

@Controller('users')
class UsersController {
  constructor(private service: UsersService) {}
}

Resolution order:

UsersController needs UsersService
  -> UsersService needs UsersRepository
    -> UsersRepository needs DatabaseService
      -> DatabaseService has no dependencies -> CREATE
    -> CREATE UsersRepository (inject DatabaseService)
  -> CREATE UsersService (inject UsersRepository)
-> CREATE UsersController (inject UsersService)

Provider Tokens

Every provider has a token -- a unique identifier the DI container uses to look up the provider. There are three types of tokens:

1. Class Token (Default)

// The class itself is the token
providers: [UsersService]
// Equivalent to:
providers: [{ provide: UsersService, useClass: UsersService }]

When you inject via constructor type:

constructor(private usersService: UsersService) {}

NestJS reads the TypeScript type metadata to find the token UsersService.

2. String Token

providers: [
  { provide: 'DATABASE_CONNECTION', useValue: connectionInstance },
]

// Inject with @Inject()
constructor(@Inject('DATABASE_CONNECTION') private db: Connection) {}

3. Symbol Token

export const CACHE_MANAGER = Symbol('CACHE_MANAGER');

providers: [
  { provide: CACHE_MANAGER, useFactory: () => new CacheManager() },
]

constructor(@Inject(CACHE_MANAGER) private cache: CacheManager) {}

Symbols are preferred over strings because they're guaranteed unique -- no accidental collisions.

InjectionToken Class

For even better type safety, use InjectionToken:

import { InjectionToken } from '@nestjs/common';

interface DatabaseConfig {
  host: string;
  port: number;
}

export const DB_CONFIG = new InjectionToken<DatabaseConfig>('DB_CONFIG');

// Register
providers: [{ provide: DB_CONFIG, useValue: { host: 'localhost', port: 5432 } }]

// Inject -- the type is inferred
constructor(@Inject(DB_CONFIG) private config: DatabaseConfig) {}

Provider Scopes

By default, all providers in NestJS are singletons -- one instance shared across the entire application. But NestJS supports three scopes:

DEFAULT (Singleton)

@Injectable() // Default scope
export class UsersService {}

// Explicit:
@Injectable({ scope: Scope.DEFAULT })
export class UsersService {}
  • One instance for the entire app
  • Created when the module initializes
  • Lives for the app's lifetime
  • Best for stateless services (most services)

REQUEST Scope

import { Injectable, Scope } from '@nestjs/common';

@Injectable({ scope: Scope.REQUEST })
export class RequestScopedService {
  constructor() {
    // New instance for EVERY request
    console.log('New instance created');
  }
}
  • New instance per incoming request
  • Garbage collected after the request completes
  • Use for request-specific state (e.g., current user context)

TRANSIENT Scope

@Injectable({ scope: Scope.TRANSIENT })
export class TransientService {
  private id = Math.random();

  getId() {
    return this.id; // Different for each consumer
  }
}
  • New instance for each consumer (injection point)
  • If 3 services inject TransientService, each gets a different instance
  • Use for services that need isolated state per consumer

Scope Bubbling

Important: Scope propagates up the dependency chain. If a singleton depends on a request-scoped provider, the singleton becomes request-scoped too.

UsersController (becomes REQUEST-scoped)
    |
    v
UsersService (becomes REQUEST-scoped)
    |
    v
RequestContextService (REQUEST-scoped)  <-- scope bubbles up

This is called scope bubbling and has significant performance implications. Every request creates new instances of the entire chain.

+-----------------------------------------------+
|            Scope Bubbling Diagram              |
|                                                |
|  Singleton --> Singleton --> REQUEST-scoped     |
|     ^              ^              ^            |
|     |              |              |            |
|  Becomes       Becomes       Original          |
|  REQUEST       REQUEST       REQUEST           |
+-----------------------------------------------+

Accessing the Request Object in Request-Scoped Providers

import { Injectable, Scope, Inject } from '@nestjs/common';
import { REQUEST } from '@nestjs/core';
import { Request } from 'express';

@Injectable({ scope: Scope.REQUEST })
export class RequestContextService {
  constructor(@Inject(REQUEST) private request: Request) {}

  getCurrentUser() {
    return this.request.user;
  }

  getCorrelationId() {
    return this.request.headers['x-correlation-id'];
  }
}

Scope Best Practices and Performance

Performance Impact:
  Singleton  -->  Best (one instance, no overhead)
  Transient  -->  Moderate (new instance per injection point)
  Request    -->  Worst (new instance per request, scope bubbling)

Guidelines: 1. Use DEFAULT (singleton) for almost everything 2. Use REQUEST only when you genuinely need per-request state 3. Avoid REQUEST scope for frequently injected services 4. Consider using AsyncLocalStorage instead of REQUEST scope for request context

AsyncLocalStorage Alternative

import { AsyncLocalStorage } from 'async_hooks';

interface RequestContext {
  userId: string;
  correlationId: string;
}

export const requestContext = new AsyncLocalStorage<RequestContext>();

// Middleware to set context
@Injectable()
export class ContextMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction) {
    const context: RequestContext = {
      userId: req.user?.id,
      correlationId: req.headers['x-correlation-id'] as string,
    };
    requestContext.run(context, () => next());
  }
}

// Use anywhere -- no REQUEST scope needed
@Injectable() // Singleton!
export class AuditService {
  log(action: string) {
    const ctx = requestContext.getStore();
    console.log(`User ${ctx?.userId} performed ${action}`);
  }
}

This avoids scope bubbling entirely while still accessing request data.

Inspecting the DI Container

Use ModuleRef to dynamically interact with the container:

import { Injectable, OnModuleInit } from '@nestjs/common';
import { ModuleRef } from '@nestjs/core';

@Injectable()
export class AppService implements OnModuleInit {
  private usersService: UsersService;

  constructor(private moduleRef: ModuleRef) {}

  onModuleInit() {
    // Get a singleton provider
    this.usersService = this.moduleRef.get(UsersService);
  }

  async doSomethingWithRequestScope() {
    // Resolve a request-scoped provider
    const contextId = ContextIdFactory.create();
    const requestService = await this.moduleRef.resolve(
      RequestScopedService,
      contextId,
    );
  }
}

Provider Registration Patterns

Conditional Providers

providers: [
  {
    provide: 'MAILER',
    useFactory: (config: ConfigService) => {
      if (config.get('NODE_ENV') === 'production') {
        return new SesMailer(config.get('AWS_SES_KEY'));
      }
      return new ConsoleMailer(); // Log emails in dev
    },
    inject: [ConfigService],
  },
]

Multi-Providers (Not Natively Supported)

NestJS doesn't have Angular's multi: true, but you can achieve similar results:

// Collect multiple implementations
const EVENT_HANDLERS = Symbol('EVENT_HANDLERS');

providers: [
  UserCreatedHandler,
  UserUpdatedHandler,
  UserDeletedHandler,
  {
    provide: EVENT_HANDLERS,
    useFactory: (...handlers) => handlers,
    inject: [UserCreatedHandler, UserUpdatedHandler, UserDeletedHandler],
  },
]

// Inject as array
constructor(@Inject(EVENT_HANDLERS) private handlers: EventHandler[]) {}

Common Pitfalls

  1. Scope bubbling surprises -- A single REQUEST-scoped provider deep in the dependency tree forces everything above it to be request-scoped.
  2. Circular tokens -- Two providers that reference each other's tokens. Use forwardRef().
  3. String token typos -- 'DATABASE_CONECTION' vs 'DATABASE_CONNECTION'. Use Symbol or const tokens instead.
  4. Performance with REQUEST scope -- Every request creates new instances. Profile before using.

Exercises

  1. Explore Provider Scopes

    Create three services: SingletonService (default), RequestService (REQUEST scope), and TransientService (TRANSIENT scope). Each should log its creation with a random ID. Inject all three into a controller and observe instance creation patterns across multiple requests.

    Hint

    Add console.log('Created instance:', this.id) in each constructor. For singleton, you'll see one log on startup. For request-scoped, one log per request. For transient, one per injection point.

  2. Implement AsyncLocalStorage Context

    Replace a REQUEST-scoped RequestContextService with an AsyncLocalStorage-based approach. Create middleware that sets the context and a singleton service that reads it. Verify the correct context is available in nested async operations.

    Hint

    Create an AsyncLocalStorage instance. In middleware, call als.run(context, next). In services, call als.getStore() to get the current context.

  3. Dynamic Provider Resolution with ModuleRef

    Build a plugin system where plugins are registered as providers. Use ModuleRef to dynamically resolve plugins by name at runtime. Create a PluginManagerService that discovers and executes plugins.

    Hint

    Register plugins with string tokens. Use moduleRef.get('plugin-name') to resolve them. Store plugin names in a registry provider.

Common Problems & Solutions

1 Singleton service has stale data across requests
Symptoms: User A sees User B's data. Race conditions between concurrent requests.
Cause: A singleton service stores request-specific data in instance properties. Since the same instance handles all requests, data from one request leaks into another.
Solution: Never store request-specific state in singleton services. Use REQUEST scope for per-request state, or better yet, use AsyncLocalStorage to avoid scope bubbling.
2 Request-scoped provider causes massive performance degradation
Symptoms: Response times increase significantly. Memory usage spikes under load.
Cause: Scope bubbling forces the entire dependency chain to be recreated on every request, including expensive services like database connections.
Solution: Minimize the use of REQUEST scope. Consider AsyncLocalStorage for request context. If you must use REQUEST scope, keep the dependency chain short.
3 Cannot resolve request-scoped provider outside of request context
Cause: Attempting to inject a REQUEST-scoped provider in a context where there is no HTTP request (e.g., a cron job or startup hook).
Solution: Use `ModuleRef.resolve()` with a manually created `ContextIdFactory.create()` context. Or restructure to not use REQUEST scope in non-request contexts.
