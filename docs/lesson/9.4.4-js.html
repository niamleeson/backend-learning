Design a Content Delivery Network (CDN)

Intuition

Think of a CDN like a chain of convenience stores. Instead of everyone driving to the central warehouse (origin server) for a bottle of water, each neighborhood has a local store (edge PoP) stocked with the most popular items. When a store runs out, it calls the regional distributor (shield), who only goes to the warehouse if they are also out. Cache invalidation is like a product recall: a message goes out to every store to remove the item from shelves.

Design a Content Delivery Network (CDN)

Introduction

A Content Delivery Network (CDN) is a geographically distributed network of servers that caches and delivers content to users from the nearest edge location. CDNs like Cloudflare, AWS CloudFront, and Akamai serve over 50% of all internet traffic. Designing a CDN tests your understanding of caching, distributed systems, network protocols, consistency, and global infrastructure.

Step 1: Requirements Gathering

Functional Requirements

  • Serve static content (images, CSS, JS, videos) from edge locations closest to users
  • Cache content at edge servers with configurable TTL
  • Support cache invalidation (purge specific URLs or patterns)
  • Handle cache misses by fetching from the origin server
  • Support custom domains with SSL/TLS certificates
  • Provide real-time analytics (hit rate, bandwidth, latency)

Non-Functional Requirements

  • Latency: < 50ms for cache hits from edge
  • Availability: 99.99% uptime
  • Scale: Serve 1 million requests per second globally
  • Throughput: 10 Tbps aggregate bandwidth
  • Consistency: Eventual consistency for cache updates (< 30 seconds global propagation)

Out of Scope

  • DDoS protection and WAF (though CDNs often provide this)
  • Dynamic content acceleration
  • Video transcoding

Step 2: Capacity Estimation

Assumptions:
- 1 million requests/second globally
- Average response size: 100 KB (mix of small assets and larger files)
- Cache hit ratio: 95% (well-configured CDN)
- 200 edge locations worldwide
- Each edge location handles ~5,000 RPS

Bandwidth:
- Total: 1M RPS x 100 KB = 100 GB/s = 800 Gbps
- Per edge location: 100 GB/s / 200 = 500 MB/s = 4 Gbps

Storage per edge:
- Hot content (frequently accessed): 10 TB SSD per edge
- Warm content (less frequent): 50 TB HDD per edge
- Total across all edges: 200 x 60 TB = 12 PB
- Unique content (origin): ~500 TB

Origin traffic (cache misses):
- 5% miss rate: 50,000 RPS to origin
- Bandwidth: 50,000 x 100 KB = 5 GB/s to origin

DNS:
- DNS resolution for routing: 1M queries/second
- GeoDNS or Anycast for user-to-edge routing

Step 4: Request Routing

DNS-Based Routing

Client in Tokyo requests: cdn.example.com/image.jpg

1. DNS Query: cdn.example.com
2. GeoDNS returns IP of Tokyo edge PoP
3. Client connects to Tokyo edge
4. Edge checks local cache
   - HIT: Return cached content (< 10ms)
   - MISS: Fetch from shield, then origin

Anycast Routing

Multiple edge servers advertise the same IP address via BGP. Network routing automatically directs traffic to the nearest PoP:

All edge PoPs announce: 203.0.113.1/32

Client in London   ──BGP──>  EU-London PoP    (2ms)
Client in New York ──BGP──>  US-East PoP      (5ms)
Client in Tokyo    ──BGP──>  Asia-Tokyo PoP   (3ms)

Advantage: Works at Layer 3, no DNS dependency
Disadvantage: Less granular control than GeoDNS

Step 6: Edge Server Cache Implementation

import { LRUCache } from 'lru-cache';
import crypto from 'node:crypto';

class EdgeCache {
  constructor() {
    // L1: In-memory LRU cache (hot items)
    this.memoryCache = new LRUCache({
      max: 10000,             // 10,000 items
      maxSize: 2_000_000_000, // 2 GB
      sizeCalculation: (value) => value.body.length,
      ttl: 300_000,           // 5 minute default TTL
    });

    // L2: Disk cache would be handled by Nginx/Varnish in production
  }

  generateCacheKey(req) {
    // Cache key based on URL, query string, and relevant headers
    const parts = [
      req.hostname,
      req.path,
      req.query ? JSON.stringify(req.query) : '',
      req.headers['accept-encoding'] || '',
    ];
    return crypto.createHash('sha256').update(parts.join('|')).digest('hex');
  }

  get(cacheKey) {
    const entry = this.memoryCache.get(cacheKey);
    if (!entry) return null;

    // Check if content is stale but usable (stale-while-revalidate)
    if (entry.staleAt && Date.now() > entry.staleAt) {
      entry.isStale = true;
    }
    return entry;
  }

  set(cacheKey, response, ttl) {
    this.memoryCache.set(cacheKey, {
      body: response.body,
      headers: response.headers,
      statusCode: response.statusCode,
      cachedAt: Date.now(),
      staleAt: Date.now() + ttl * 1000,
      etag: response.headers.etag,
      lastModified: response.headers['last-modified'],
    }, { ttl: ttl * 1000 });
  }

  purge(pattern) {
    // Simple pattern matching for invalidation
    if (pattern.includes('*')) {
      const regex = new RegExp('^' + pattern.replace(/\*/g, '.*') + '$');
      // LRU cache doesn't support iteration well,
      // so in production use a separate index
      this.memoryCache.clear(); // Simplified: clear all
    } else {
      const key = crypto.createHash('sha256').update(pattern).digest('hex');
      this.memoryCache.delete(key);
    }
  }
}

Edge Request Handler

class EdgeServer {
  constructor(cache, shieldUrl) {
    this.cache = cache;
    this.shieldUrl = shieldUrl; // Regional shield/mid-tier
  }

  async handleRequest(req, res) {
    const cacheKey = this.cache.generateCacheKey(req);
    const cached = this.cache.get(cacheKey);

    // Cache HIT
    if (cached && !cached.isStale) {
      res.set('X-Cache', 'HIT');
      res.set('X-Cache-Age', Math.floor((Date.now() - cached.cachedAt) / 1000));
      for (const [key, value] of Object.entries(cached.headers)) {
        res.set(key, value);
      }
      return res.status(cached.statusCode).send(cached.body);
    }

    // Stale content: serve stale and revalidate in background
    if (cached && cached.isStale) {
      res.set('X-Cache', 'STALE');
      // Serve stale content immediately
      for (const [key, value] of Object.entries(cached.headers)) {
        res.set(key, value);
      }
      res.status(cached.statusCode).send(cached.body);

      // Revalidate in background (don't await)
      this.revalidate(req, cacheKey, cached).catch(console.error);
      return;
    }

    // Cache MISS: fetch from shield/origin
    res.set('X-Cache', 'MISS');
    try {
      const origin = await this.fetchFromShield(req);
      const ttl = this.parseTTL(origin.headers);

      if (ttl > 0 && origin.statusCode === 200) {
        this.cache.set(cacheKey, origin, ttl);
      }

      for (const [key, value] of Object.entries(origin.headers)) {
        res.set(key, value);
      }
      res.status(origin.statusCode).send(origin.body);
    } catch (err) {
      res.status(502).json({ error: 'Origin unreachable' });
    }
  }

  parseTTL(headers) {
    // Parse Cache-Control header
    const cc = headers['cache-control'] || '';
    const maxAgeMatch = cc.match(/max-age=(\d+)/);
    if (maxAgeMatch) return parseInt(maxAgeMatch[1]);

    // Fallback to Expires header
    if (headers['expires']) {
      const expiresAt = new Date(headers['expires']).getTime();
      return Math.max(0, Math.floor((expiresAt - Date.now()) / 1000));
    }

    return 3600; // Default 1 hour
  }

  async fetchFromShield(req) {
    const url = `${this.shieldUrl}${req.originalUrl}`;
    const response = await fetch(url, {
      headers: {
        'Host': req.hostname,
        'X-Forwarded-For': req.ip,
        'Accept-Encoding': 'gzip, br',
      },
    });

    return {
      statusCode: response.status,
      headers: Object.fromEntries(response.headers.entries()),
      body: Buffer.from(await response.arrayBuffer()),
    };
  }

  async revalidate(req, cacheKey, staleEntry) {
    const headers = {};
    if (staleEntry.etag) headers['If-None-Match'] = staleEntry.etag;
    if (staleEntry.lastModified) headers['If-Modified-Since'] = staleEntry.lastModified;

    const response = await fetch(`${this.shieldUrl}${req.originalUrl}`, { headers });
    if (response.status === 304) {
      // Not modified, extend TTL
      const ttl = this.parseTTL(Object.fromEntries(response.headers.entries()));
      this.cache.set(cacheKey, staleEntry, ttl);
    } else if (response.status === 200) {
      // Updated content
      const body = Buffer.from(await response.arrayBuffer());
      const ttl = this.parseTTL(Object.fromEntries(response.headers.entries()));
      this.cache.set(cacheKey, { body, headers: Object.fromEntries(response.headers.entries()), statusCode: 200 }, ttl);
    }
  }
}

// Purge propagation via Redis Pub/Sub
import Redis from 'ioredis';

const publisher = new Redis();
const subscriber = new Redis();

// Purge service (control plane)
async function publishPurge(distributionId, paths) {
  const message = JSON.stringify({
    distributionId,
    paths,
    timestamp: Date.now(),
  });
  await publisher.publish('cdn:purge', message);
}

// Edge server (data plane)
subscriber.subscribe('cdn:purge');
subscriber.on('message', (channel, message) => {
  const { distributionId, paths } = JSON.parse(message);
  for (const path of paths) {
    edgeCache.purge(path);
  }
  console.log(`Purged ${paths.length} paths for distribution ${distributionId}`);
});

Step 9: Scaling Strategies

Shield/Mid-Tier Cache

Without a shield layer, each edge PoP independently fetches from origin on a cache miss. With 200 edges, a single content update could cause 200 origin requests. The shield layer collapses these:

Without Shield:                  With Shield:

200 edges x miss = 200           200 edges x miss = 200
origin requests                  shield requests
                                 Shield miss = 1 origin request

Origin load: 200x                Origin load: 1x

Tiered Caching

Tier 1: In-memory (LRU, 2 GB)  - ~1ms, hottest 10K objects
Tier 2: SSD cache (10 TB)       - ~5ms, warm content
Tier 3: Shield (regional)       - ~20ms, regional cache
Tier 4: Origin                   - ~100-500ms, source of truth

Content Compression

Compress at the edge to reduce bandwidth:

import zlib from 'node:zlib';

function compressResponse(body, acceptEncoding) {
  if (acceptEncoding.includes('br')) {
    return { body: zlib.brotliCompressSync(body), encoding: 'br' };
  }
  if (acceptEncoding.includes('gzip')) {
    return { body: zlib.gzipSync(body), encoding: 'gzip' };
  }
  return { body, encoding: 'identity' };
}

Step 10: Trade-offs and Decisions

Decision Option A Option B Recommendation
Routing GeoDNS Anycast BGP Anycast for simplicity and DDoS resilience; GeoDNS for fine-grained control
Cache eviction LRU LFU LRU is simpler and works well for most workloads; LFU for skewed popularity
Purge propagation Push (pub/sub) Pull (polling) Push for speed; pull as fallback
Consistency Eventual (TTL-based) Strong (purge-and-wait) Eventual for performance; strong purge for critical updates
TLS termination Edge Origin Edge for lower latency; re-encrypt to origin for security

Cold Start Problem

When a new edge PoP is deployed, its cache is empty and everything is a miss. Solutions: - Pre-warming: Push popular content to new edges before enabling traffic - Peer-to-peer: Fetch from nearby edges instead of origin - Gradual rollout: Slowly shift traffic to the new PoP

Summary

A CDN distributes content globally by caching it at edge servers close to users. The architecture consists of edge PoPs (point of presence) with multi-tier caching, a shield/mid-tier layer to protect the origin, and a control plane for configuration and cache invalidation. Routing is handled by Anycast BGP or GeoDNS. Key design decisions include cache eviction policy, purge propagation mechanism, and the balance between consistency and performance. A well-designed CDN achieves 95%+ cache hit rates and sub-50ms latency for cached content.

Common Problems & Solutions

1 Cache hit ratio is below 50%, causing excessive origin load
Symptoms: Origin server is overloaded despite having a CDN. X-Cache headers show mostly MISS. Identical content is stored multiple times under different cache keys.
Cause: Cache keys include unnecessary variation (e.g., unique query parameters, tracking cookies, or session headers), causing each request to generate a unique cache key even for identical content.
Solution: Normalize cache keys by stripping tracking parameters (utm_*, fbclid, etc.). Configure which headers and query parameters are included in the cache key. Use the Vary header correctly to avoid over-segmentation.
2 Stale content served after origin update
Cause: Cache TTL is too long and no purge was issued after the content update. Or the purge did not propagate to all edge servers.
Solution: Implement cache busting with content hashes in filenames (e.g., app.abc123.js). For dynamic content, use shorter TTLs with stale-while-revalidate. Set up automatic purge triggers on content deployment.
3 Thundering herd on cache expiration
Cause: A popular item's TTL expires simultaneously on many edge servers, causing all of them to request it from the origin at the same time, overwhelming the origin.
Solution: Implement request coalescing (only one origin request per cache key, others wait). Add jitter to TTLs so they don't expire at the same time. Use stale-while-revalidate to serve stale content while refreshing in the background.
