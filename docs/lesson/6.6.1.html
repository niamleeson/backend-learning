Hexagonal Architecture: Ports and Adapters

Intuition

Think of a phone charger. Your phone (core domain) has a charging port (interface). Whether you plug in a wall adapter, a car adapter, or a portable battery (different adapters), the phone does not care -- it just receives power through the standardized port. Hexagonal architecture gives your application the same flexibility.

Hexagonal Architecture: Ports and Adapters

Hexagonal Architecture (also known as Ports and Adapters) isolates your core business logic from external concerns like databases, APIs, and UIs. The core defines 'ports' (interfaces), and external systems connect via 'adapters.'

Project Structure

# Typical hexagonal structure:
# src/
#   domain/           <- Core business logic (no external dependencies)
#     models.py
#     services.py
#   ports/            <- Interfaces (abstract classes)
#     repositories.py
#     notifications.py
#   adapters/         <- Implementations of ports
#     postgres_repo.py
#     email_notifier.py
#     in_memory_repo.py
#   entrypoints/      <- How the outside world reaches the core
#     api.py
#     cli.py

Defining Ports

# ports/repositories.py
from abc import ABC, abstractmethod
from domain.models import Order

class OrderRepository(ABC):
    @abstractmethod
    def save(self, order: Order) -> None: ...

    @abstractmethod
    def get(self, order_id: int) -> Order: ...

# ports/notifications.py
class NotificationPort(ABC):
    @abstractmethod
    def send(self, recipient: str, message: str) -> None: ...

Implementing Adapters

# adapters/postgres_repo.py
class PostgresOrderRepository(OrderRepository):
    def __init__(self, session):
        self.session = session

    def save(self, order: Order) -> None:
        self.session.add(order)
        self.session.flush()

    def get(self, order_id: int) -> Order:
        return self.session.query(OrderModel).get(order_id)

# adapters/email_notifier.py
import smtplib

class EmailNotifier(NotificationPort):
    def __init__(self, smtp_host: str):
        self.smtp_host = smtp_host

    def send(self, recipient: str, message: str) -> None:
        with smtplib.SMTP(self.smtp_host) as server:
            server.sendmail("noreply@app.com", recipient, message)

Wiring It Together

# entrypoints/api.py
from domain.services import OrderService
from adapters.postgres_repo import PostgresOrderRepository
from adapters.email_notifier import EmailNotifier

order_service = OrderService(
    repo=PostgresOrderRepository(session),
    notifier=EmailNotifier("smtp.example.com")
)

@app.post("/orders")
def create_order(data: OrderCreate):
    return order_service.create(data)

Common Mistakes

  • Importing adapter code in the domain layer (breaks the dependency rule)
  • Not defining ports explicitly, just using concrete classes everywhere
  • Over-abstracting simple applications that do not need this structure

Key Takeaways

  • The domain layer has zero dependencies on external frameworks
  • Ports are interfaces; adapters are implementations
  • You can swap adapters without changing a single line of business logic

Exercises

  1. Refactor to Hexagonal Architecture

    Take a FastAPI application where route handlers directly call SQLAlchemy and send emails. Refactor it into hexagonal architecture: extract ports for the repository and email service, create adapters, and make the domain layer framework-free.

    Hint

    Start by identifying external dependencies in your business logic. Extract each one into an abstract port. Move implementations into adapters. The domain should import nothing from FastAPI or SQLAlchemy.

  2. Multiple Adapter Implementations

    Create a NotificationPort with three adapters: EmailNotifier, SMSNotifier, and SlackNotifier. Demonstrate switching between them via configuration without changing any business logic.

    Hint

    Use a factory function or environment variable to select the adapter at startup. The service layer accepts a NotificationPort and never knows which concrete adapter it receives.

Common Problems & Solutions

1 Domain layer imports SQLAlchemy or Flask directly
Symptoms: ['Cannot test business logic without database', 'Domain models break when upgrading SQLAlchemy']
Cause: No clear boundary between domain and infrastructure. Domain models inherit from SQLAlchemy Base or use Flask's request object.
Solution: Define domain models as plain Python dataclasses. Create separate ORM models in the adapter layer and map between them. The domain layer should have zero external imports.
2 Too many layers and indirection for a simple CRUD app
Cause: Hexagonal architecture was applied to an application that is essentially a thin wrapper around a database.
Solution: Use hexagonal architecture selectively. Simple CRUD endpoints can use direct ORM access. Apply ports and adapters where you have complex business logic or need to swap external systems.
