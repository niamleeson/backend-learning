Indexes: B-tree, Hash, GIN, GiST, BRIN

Intuition

An index is like a book's table of contents. Different index types are like different ways of organizing that TOC: alphabetical, by topic, by date, or by location.

Indexes: B-tree, Hash, GIN, GiST, BRIN

Indexes speed up queries by creating optimized lookup structures. PostgreSQL supports multiple index types, each suited to different access patterns.

-- B-tree (default): equality and range queries
CREATE INDEX idx_users_email ON users (email);

-- Hash: equality only, smaller than B-tree
CREATE INDEX idx_users_status ON users USING hash (status);

-- GIN: arrays, JSONB, full-text search
CREATE INDEX idx_products_tags ON products USING gin (tags);

-- GiST: geometric, range, proximity queries
CREATE INDEX idx_locations ON places USING gist (location);

-- BRIN: large naturally ordered tables
CREATE INDEX idx_logs_created ON logs USING brin (created_at);

-- Partial index
CREATE INDEX idx_active_users ON users (email)
  WHERE status = 'active';

Key Takeaways: - B-tree handles most use cases: equality, range, sorting, and prefix LIKE queries. - GIN excels for multi-valued columns (arrays, JSONB, text search). BRIN is compact for large sequential data. - Partial and covering indexes reduce size and improve specificity.

Exercises

  1. Index Strategy Design

    Given a query that filters by status='active' and orders by created_at, design an optimal composite index.

    Hint

    CREATE INDEX ON table (status, created_at) or a partial index WHERE status = 'active' on created_at.

  2. GIN Index for JSONB

    Create a GIN index on a JSONB column and write a query that finds all rows where metadata contains a key 'role' with value 'admin'.

    Hint

    CREATE INDEX USING gin (metadata); then WHERE metadata @> '{"role": "admin"}'.

Common Problems & Solutions

1 Index exists but query does not use it.
Cause: Wrapping the indexed column in a function, casting types, or the planner estimating a sequential scan is faster.
Solution: Create a functional index matching the expression, or use SET enable_seqscan = off to test. Check with EXPLAIN.
2 Too many indexes slow down writes.
Cause: Each INSERT/UPDATE/DELETE must also update every index on the table.
Solution: Audit unused indexes with pg_stat_user_indexes. Drop indexes with low idx_scan counts.
