Structuring a Large Python Backend Project

Intuition

Structuring a large project is like organizing a hospital. You would not put surgery, pharmacy, and reception in one room. Each department (module) has a clear responsibility, defined interfaces for interaction, and can operate independently. The building layout (project structure) makes it obvious where to find things and how departments connect.

Structuring a Large Python Backend Project

As projects grow beyond a few files, proper structure becomes critical for maintainability. This lesson covers battle-tested patterns for organizing Python backend projects at scale.

Recommended Structure

# project-root/
# ├── src/
# │   └── myapp/
# │       ├── __init__.py
# │       ├── main.py              # App factory, startup
# │       ├── config.py            # Settings with pydantic-settings
# │       ├── domain/              # Business logic, no framework deps
# │       │   ├── models.py
# │       │   ├── events.py
# │       │   └── exceptions.py
# │       ├── services/            # Use case orchestration
# │       │   ├── order_service.py
# │       │   └── user_service.py
# │       ├── ports/               # Abstract interfaces
# │       │   └── repositories.py
# │       ├── adapters/            # External integrations
# │       │   ├── db/
# │       │   │   ├── models.py    # ORM models
# │       │   │   ├── repositories.py
# │       │   │   └── session.py
# │       │   ├── email.py
# │       │   └── redis_cache.py
# │       └── entrypoints/         # API routes, CLI, workers
# │           ├── api/
# │           │   ├── v1/
# │           │   │   ├── users.py
# │           │   │   └── orders.py
# │           │   └── deps.py      # FastAPI dependencies
# │           └── cli.py
# ├── tests/
# │   ├── unit/
# │   ├── integration/
# │   └── conftest.py
# ├── pyproject.toml
# └── Dockerfile

Configuration with Pydantic Settings

# config.py
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    database_url: str = "postgresql://localhost/myapp"
    redis_url: str = "redis://localhost:6379"
    secret_key: str
    debug: bool = False
    allowed_origins: list[str] = ["http://localhost:3000"]

    class Config:
        env_file = ".env"

settings = Settings()

App Factory Pattern

# main.py
from fastapi import FastAPI
from myapp.entrypoints.api.v1 import users, orders
from myapp.adapters.db.session import init_db

def create_app() -> FastAPI:
    app = FastAPI(title="MyApp", version="1.0.0")
    app.include_router(users.router, prefix="/v1/users", tags=["users"])
    app.include_router(orders.router, prefix="/v1/orders", tags=["orders"])

    @app.on_event("startup")
    async def startup():
        await init_db()

    return app

app = create_app()

Common Mistakes

  • Circular imports from poor module organization
  • Putting everything in one giant views.py or models.py file
  • Not using a src/ layout, causing import conflicts with package names

Key Takeaways

  • Use the src/ layout to avoid import ambiguity
  • Separate domain logic from infrastructure from entrypoints
  • Use an app factory for testability and multiple configurations
  • Keep the dependency direction: entrypoints -> services -> domain <- adapters

Exercises

  1. Restructure a Monolithic App

    Take a FastAPI app with all code in main.py (models, routes, database logic, business logic) and restructure it into the hexagonal layout shown above. Verify all tests still pass after restructuring.

    Hint

    Move in small steps: first extract models, then routes, then services. Run tests after each move. Use relative imports within the package.

  2. Multi-Environment Configuration

    Set up a configuration system using pydantic-settings that loads different settings for development, testing, and production. Use .env files and environment variable overrides. Include database URLs, feature flags, and logging levels.

    Hint

    Create a base Settings class and override values via environment variables. Use a .env.test file for test configuration. FastAPI's TestClient can override dependencies.

  3. App Factory with Plugin Architecture

    Build an app factory that accepts a list of plugin modules. Each plugin registers its own routes, middleware, and startup hooks. Demonstrate adding a new feature module without modifying the core app factory.

    Hint

    Define a Plugin protocol with register(app: FastAPI) method. Each plugin module implements this protocol. The factory iterates plugins and calls register().

Common Problems & Solutions

1 Circular import errors as the project grows
Symptoms: ["ImportError: cannot import name 'X' from partially initialized module", 'Errors appear only when running, not in IDE']
Cause: Module A imports from Module B which imports from Module A, creating a dependency cycle.
Solution: Follow the dependency rule: dependencies point inward (entrypoints -> services -> domain). Use TYPE_CHECKING imports for type hints only. Move shared types to a common module.

Fix circular imports with TYPE_CHECKING

from future import annotations from typing import TYPE_CHECKING

if TYPE_CHECKING: from myapp.services.order_service import OrderService

2 Tests cannot import application modules
Cause: Missing __init__.py files, incorrect PYTHONPATH, or not using the src/ layout with proper package installation.
Solution: Use the src/ layout with pyproject.toml. Install the package in editable mode: pip install -e .[dev]. Ensure all directories have __init__.py files.
